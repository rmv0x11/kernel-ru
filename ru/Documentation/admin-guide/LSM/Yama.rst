====
Yama
====

Yama — это модуль безопасности Linux (Linux Security Module), который собирает
общесистемные средства защиты DAC, не обрабатываемые самим базовым ядром. Он
выбирается на этапе сборки с помощью ``CONFIG_SECURITY_YAMA`` и может управляться
во время выполнения через sysctl в ``/proc/sys/kernel/yama``:

ptrace_scope
============

По мере роста популярности Linux он будет становиться всё более крупной мишенью
для вредоносного ПО. Одной из особенно тревожных слабостей интерфейсов процессов
Linux является то, что один пользователь способен изучать память и состояние
выполнения любого из своих процессов. Например, если одно приложение
(например, Pidgin) было скомпрометировано, у атакующего появилась бы возможность
присоединиться к другим работающим процессам (например, Firefox, SSH-сессиям,
агенту GPG и т. д.), чтобы извлечь дополнительные учётные данные и продолжить
расширять область своей атаки, не прибегая к фишингу с участием пользователя.

Это не теоретическая проблема. Атаки `перехвата SSH-сессии
<https://www.blackhat.com/presentations/bh-usa-05/bh-us-05-boileau.pdf>`_
и `внедрения произвольного кода
<https://c-skills.blogspot.com/2007/05/injectso.html>`_ уже существуют и
остаются возможными, если ptrace разрешено работать как прежде. Поскольку ptrace
обычно не используется не-разработчиками и не-администраторами, строителям систем
следует предоставить возможность отключить эту систему отладки.

В качестве решения некоторые приложения используют ``prctl(PR_SET_DUMPABLE, ...)``,
чтобы явно запретить такое присоединение ptrace (например, ssh-agent), но многие
этого не делают. Более общим решением является разрешение ptrace только напрямую
от родительского процесса к дочернему (то есть прямые «gdb EXE» и «strace EXE»
по-прежнему работают) либо при наличии ``CAP_SYS_PTRACE`` (то есть «gdb --pid=PID»
и «strace -p PID» по-прежнему работают от имени root).

В режиме 1 для программ, определивших специфичные для приложения отношения между
отлаживающим процессом и его подчинённым (обработчики аварийных завершений и т. п.),
можно использовать ``prctl(PR_SET_PTRACER, pid, ...)``. Подчинённый может объявить,
какому другому процессу (и его потомкам) разрешено вызывать ``PTRACE_ATTACH`` по
отношению к нему. Для каждого подчинённого в каждый момент времени может существовать
только один такой объявленный отлаживающий процесс. Например, это используется
обработчиками аварийных завершений KDE, Chromium и Firefox, а также Wine для того,
чтобы разрешить процессам Wine выполнять ptrace только друг для друга. Если процесс
желает полностью отключить эти ограничения ptrace, он может вызвать
``prctl(PR_SET_PTRACER, PR_SET_PTRACER_ANY, ...)``, чтобы присоединиться мог любой
иным образом разрешённый процесс (даже находящийся во внешних пространствах имён pid).

Настройки sysctl (доступные для записи только с ``CAP_SYS_PTRACE``):

0 - классические разрешения ptrace:
    процесс может выполнить ``PTRACE_ATTACH`` к любому другому
    процессу, работающему под тем же uid, при условии что тот может быть выгружен
    в дамп (то есть не менял uid, не запускался привилегированным и не вызывал
    ранее ``prctl(PR_SET_DUMPABLE...)``). Аналогично, ``PTRACE_TRACEME`` остаётся
    без изменений.

1 - ограниченный ptrace:
    процесс должен иметь предопределённое отношение
    с подчинённым, к которому он хочет применить ``PTRACE_ATTACH``. По умолчанию
    это отношение — только для его потомков, когда также выполняется приведённый
    выше классический критерий. Чтобы изменить отношение, подчинённый может вызвать
    ``prctl(PR_SET_PTRACER, debugger, ...)``, чтобы объявить разрешённый PID
    отладчика для вызова ``PTRACE_ATTACH`` по отношению к подчинённому.
    Использование ``PTRACE_TRACEME`` остаётся без изменений.

2 - присоединение только для администратора:
    использовать ptrace могут только процессы с ``CAP_SYS_PTRACE``, либо через
    ``PTRACE_ATTACH``, либо через дочерние процессы, вызывающие ``PTRACE_TRACEME``.

3 - запрет присоединения:
    никакие процессы не могут использовать ptrace ни через ``PTRACE_ATTACH``, ни
    через ``PTRACE_TRACEME``. После установки это значение sysctl изменить нельзя.

Исходная логика «только для потомков» была основана на ограничениях в grsecurity.
