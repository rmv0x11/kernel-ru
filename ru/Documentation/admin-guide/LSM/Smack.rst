=====
Smack
=====


    "Good for you, you've decided to clean the elevator!"
    - The Elevator, from Dark Star

Smack — это Simplified Mandatory Access Control Kernel (упрощённое ядро
мандатного контроля доступа). Smack представляет собой реализацию мандатного
контроля доступа на основе ядра, одной из главных целей проектирования которой
является простота.

Smack — не единственная схема мандатного контроля доступа (Mandatory Access
Control), доступная для Linux. Тем, кто только начинает знакомиться с мандатным
контролем доступа, рекомендуется сравнить Smack с другими доступными механизмами,
чтобы определить, какой из них лучше всего подходит для решаемой задачи.

Smack состоит из трёх основных компонентов:

    - Ядро
    - Базовые утилиты, которые полезны, но не обязательны
    - Конфигурационные данные

Компонент ядра в Smack реализован как модуль Linux Security Modules (LSM). Он
требует netlabel и лучше всего работает с файловыми системами, поддерживающими
расширенные атрибуты, хотя поддержка xattr строго не обязательна.
Запускать ядро со Smack под «ванильным» дистрибутивом безопасно.

Ядра со Smack используют опцию CIPSO протокола IP. Некоторые сетевые конфигурации
не переносят опции IP и могут затруднить доступ к системам, которые их используют,
как это делает Smack.

Smack применяется в операционной системе Tizen. Информацию о том, как Smack
используется в Tizen, можно найти на http://wiki.tizen.org.

Текущий git-репозиторий пространства пользователя Smack:

	git://github.com/smack-team/smack.git

Он должен собираться и устанавливаться в большинстве современных дистрибутивов.
В состав smackutil входят пять команд:

chsmack:
	отображает или устанавливает значения расширенного атрибута Smack

smackctl:
	загружает правила доступа Smack

smackaccess:
	сообщает, имеет ли процесс с одной меткой доступ
	к объекту с другой меткой

Две следующие команды устарели с появлением
интерфейсов smackfs/load2 и smackfs/cipso2.

smackload:
	корректно форматирует данные для записи в smackfs/load

smackcipso:
	корректно форматирует данные для записи в smackfs/cipso

В соответствии с замыслом Smack конфигурационные данные минимальны и строго не
обязательны. Самым важным шагом настройки является монтирование
псевдофайловой системы smackfs.
Если установлена smackutil, об этом позаботится стартовый скрипт,
но это можно сделать и вручную.

Добавьте эту строку в ``/etc/fstab``::

    smackfs /sys/fs/smackfs smackfs defaults 0 0

Каталог ``/sys/fs/smackfs`` создаётся ядром.

Smack использует расширенные атрибуты (xattrs) для хранения меток на объектах
файловой системы. Атрибуты хранятся в пространстве имён расширенных атрибутов
security. Чтобы изменить любой из этих атрибутов, процесс должен иметь
``CAP_MAC_ADMIN``.

Smack использует следующие расширенные атрибуты:

SMACK64
	Используется для принятия решений о контроле доступа. Почти во всех случаях
	метка, присваиваемая новому объекту файловой системы, будет меткой
	процесса, который его создал.

SMACK64EXEC
	Метка Smack процесса, который выполняет (exec) файл программы с
	установленным этим атрибутом, будет равна значению данного атрибута.

SMACK64MMAP
	Не разрешать отображать файл в память (mmap) процессом, метка Smack
	которого не разрешает весь тот доступ, который разрешён процессу
	с меткой, содержащейся в этом атрибуте. Это очень частный случай
	использования для разделяемых библиотек.

SMACK64TRANSMUTE
	Может иметь только значение "TRUE". Если этот атрибут присутствует
	на каталоге в момент создания объекта в этом каталоге и
	правило Smack (подробнее ниже), разрешившее доступ на запись
	в каталог, включает режим transmute ("t"), то объект
	получает метку каталога, а не метку
	создающего процесса. Если создаваемый объект является каталогом,
	атрибут SMACK64TRANSMUTE устанавливается и на нём.

SMACK64IPIN
	Этот атрибут доступен только для файловых дескрипторов сокетов.
	Использовать метку Smack из этого атрибута для принятия решений
	о контроле доступа к пакетам, доставляемым в этот сокет.

SMACK64IPOUT
	Этот атрибут доступен только для файловых дескрипторов сокетов.
	Использовать метку Smack из этого атрибута для принятия решений
	о контроле доступа к пакетам, исходящим из этого сокета.

Существует несколько способов установить метку Smack на файл::

    # attr -S -s SMACK64 -V "value" path
    # chsmack -a value path

Процесс может узнать метку Smack, с которой он выполняется, прочитав
``/proc/self/attr/current``. Процесс с ``CAP_MAC_ADMIN``
может установить метку Smack процесса, записав туда.

Большая часть настройки Smack выполняется записью в файлы
файловой системы smackfs. Эта псевдофайловая система монтируется
в ``/sys/fs/smackfs``.

access
	Предоставлен для обратной совместимости. Предпочтителен интерфейс access2,
	который и следует использовать вместо него.
	Этот интерфейс сообщает, имеет ли субъект с указанной
	меткой Smack определённый доступ к объекту с
	указанной меткой Smack. Запишите в этот файл правило доступа
	фиксированного формата. Следующее чтение покажет, был бы ли доступ
	разрешён. Текст будет либо "1", означающим
	доступ, либо "0", означающим отказ.

access2
	Этот интерфейс сообщает, имеет ли субъект с указанной
	меткой Smack определённый доступ к объекту с
	указанной меткой Smack. Запишите в этот файл правило доступа
	длинного формата. Следующее чтение покажет, был бы ли доступ
	разрешён. Текст будет либо "1", означающим
	доступ, либо "0", означающим отказ.

ambient
	Содержит метку Smack, применяемую к непомеченным сетевым
	пакетам.

change-rule
	Этот интерфейс позволяет изменять существующие правила контроля доступа.
	Формат, принимаемый при записи::

		"%s %s %s %s"

	где первая строка — это метка субъекта, вторая — метка
	объекта, третья — доступ, который нужно разрешить, а четвёртая —
	доступ, который нужно запретить. Строки доступа могут содержать только символы
	"rwxat-". Если правило для данной пары субъекта и объекта существует, оно будет
	изменено путём включения прав из третьей строки и отключения
	прав из четвёртой строки. Если такого правила нет, оно будет
	создано с использованием доступа, указанного в третьей и четвёртой строках.

cipso
	Предоставлен для обратной совместимости. Предпочтителен интерфейс cipso2,
	который и следует использовать вместо него.
	Этот интерфейс позволяет назначить метке Smack конкретный
	заголовок CIPSO. Формат, принимаемый при записи::

		"%24s%4d%4d"["%4d"]...

	Первая строка — это фиксированная метка Smack. Первое число —
	это используемый уровень. Второе число — это количество категорий.
	Последующие числа — это категории::

		"level-3-cats-5-19          3   2   5  19"

cipso2
	Этот интерфейс позволяет назначить метке Smack конкретный
	заголовок CIPSO. Формат, принимаемый при записи::

		"%s%4d%4d"["%4d"]...

	Первая строка — это длинная метка Smack. Первое число —
	это используемый уровень. Второе число — это количество категорий.
	Последующие числа — это категории::

		"level-3-cats-5-19   3   2   5  19"

direct
	Содержит уровень CIPSO, используемый для прямого представления
	меток Smack в сетевых пакетах.

doi
	Содержит домен интерпретации (domain of interpretation) CIPSO, используемый
	в сетевых пакетах.

ipv6host
	Этот интерфейс позволяет рассматривать конкретные интернет-адреса IPv6
	как однометочные хосты. Пакеты отправляются однометочным
	хостам только из процессов, имеющих доступ Smack на запись
	к метке хоста. Всем пакетам, полученным от однометочных хостов,
	присваивается указанная метка. Формат, принимаемый при записи::

		"%h:%h:%h:%h:%h:%h:%h:%h label" or
		"%h:%h:%h:%h:%h:%h:%h:%h/%d label".

	Сокращение адреса "::" не поддерживается.
	Если метка равна "-DELETE", совпавшая запись будет удалена.

load
	Предоставлен для обратной совместимости. Предпочтителен интерфейс load2,
	который и следует использовать вместо него.
	Этот интерфейс позволяет задавать правила контроля доступа в дополнение
	к правилам, определённым в системе. Формат, принимаемый
	при записи::

		"%24s%24s%5s"

	где первая строка — это метка субъекта, вторая — метка
	объекта, а третья — запрашиваемый доступ. Строка доступа
	может содержать только символы "rwxat-" и указывает,
	какой вид доступа разрешён. "-" — это заполнитель для
	прав, которые не разрешены. Строка "r-x--" задавала бы
	доступ на чтение и выполнение. Длина меток ограничена 23
	символами.

load2
	Этот интерфейс позволяет задавать правила контроля доступа в дополнение
	к правилам, определённым в системе. Формат, принимаемый
	при записи::

		"%s %s %s"

	где первая строка — это метка субъекта, вторая — метка
	объекта, а третья — запрашиваемый доступ. Строка доступа
	может содержать только символы "rwxat-" и указывает,
	какой вид доступа разрешён. "-" — это заполнитель для
	прав, которые не разрешены. Строка "r-x--" задавала бы
	доступ на чтение и выполнение.

load-self
	Предоставлен для обратной совместимости. Предпочтителен интерфейс load-self2,
	который и следует использовать вместо него.
	Этот интерфейс позволяет определять правила доступа, специфичные
	для процесса. Эти правила учитываются только в том случае, если доступ
	иначе был бы разрешён, и предназначены для наложения дополнительных
	ограничений на процесс. Формат такой же, как для
	интерфейса load.

load-self2
	Этот интерфейс позволяет определять правила доступа, специфичные
	для процесса. Эти правила учитываются только в том случае, если доступ
	иначе был бы разрешён, и предназначены для наложения дополнительных
	ограничений на процесс. Формат такой же, как для
	интерфейса load2.

logging
	Содержит состояние журналирования Smack.

mapped
	Содержит уровень CIPSO, используемый для отображаемого (mapped)
	представления меток Smack в сетевых пакетах.

netlabel
	Этот интерфейс позволяет рассматривать конкретные интернет-адреса
	как однометочные хосты. Пакеты отправляются однометочным
	хостам без заголовков CIPSO, но только из процессов,
	имеющих доступ Smack на запись к метке хоста. Всем пакетам,
	полученным от однометочных хостов, присваивается указанная
	метка. Формат, принимаемый при записи::

		"%d.%d.%d.%d label" or "%d.%d.%d.%d/%d label".

	Если указанная метка равна "-CIPSO", адрес рассматривается
	как хост, поддерживающий заголовки CIPSO.

onlycap
	Содержит метки, которые должны иметь процессы, чтобы CAP_MAC_ADMIN
	и ``CAP_MAC_OVERRIDE`` были действительны. Если этот файл пуст,
	эти права (capabilities) действительны для процессов с любой
	меткой. Значения задаются записью в файл нужных меток, разделённых
	пробелами, или сбрасываются записью в файл "-".

ptrace
	Используется для определения текущей политики ptrace

	0 - default:
	    это политика, которая опирается на правила доступа Smack.
	    Для ``PTRACE_READ`` субъект должен иметь доступ на чтение к
	    объекту. Для ``PTRACE_ATTACH`` требуется доступ на чтение-запись.

	1 - exact:
	    это политика, которая ограничивает ``PTRACE_ATTACH``. Присоединение
	    разрешено только когда метки субъекта и объекта равны.
	    На ``PTRACE_READ`` это не влияет. Может быть переопределено с помощью ``CAP_SYS_PTRACE``.

	2 - draconian:
	    эта политика ведёт себя как 'exact' выше, с тем
	    исключением, что её нельзя переопределить с помощью ``CAP_SYS_PTRACE``.

revoke-subject
	Запись метки Smack здесь устанавливает доступ в '-' для всех правил
	доступа с этой меткой субъекта.

unconfined
	Если ядро сконфигурировано с ``CONFIG_SECURITY_SMACK_BRINGUP``,
	процесс с ``CAP_MAC_ADMIN`` может записать метку в этот интерфейс.
	После этого доступы, в которых участвует эта метка, будут журналироваться,
	а доступ — разрешаться, если иначе он не был бы разрешён. Обратите внимание, что это
	опасно и может разрушить корректную разметку (labeling) вашей системы.
	Это никогда не следует использовать в продакшене.

relabel-self
	Этот интерфейс содержит список меток, на которые процесс может
	перейти, выполнив запись в ``/proc/self/attr/current``.
	Обычно процесс может изменить свою собственную метку на любое допустимое значение, но только
	если у него есть ``CAP_MAC_ADMIN``. Этот интерфейс позволяет процессу без
	``CAP_MAC_ADMIN`` переразметить себя в одну из меток предопределённого списка.
	Процесс без ``CAP_MAC_ADMIN`` может изменить свою метку только один раз. Когда он
	это сделает, этот список будет очищен.
	Значения задаются записью в файл нужных меток, разделённых
	пробелами, или сбрасываются записью в файл "-".

Если вы используете утилиту smackload,
вы можете добавить правила доступа в ``/etc/smack/accesses``. Они имеют вид::

    subjectlabel objectlabel access

access — это комбинация букв rwxatb, которые задают
вид доступа, разрешённый субъекту с subjectlabel к
объекту с objectlabel. Если правила нет, доступ не разрешается.

Мандатный контроль доступа
--------------------------

Компьютерные системы применяют различные схемы для ограничения того, как информация
разделяется между людьми и службами, использующими машину. Некоторые из этих схем
позволяют программе или пользователю решать, каким другим программам или пользователям разрешён
доступ к фрагментам данных. Эти схемы называются механизмами дискреционного
контроля доступа, потому что контроль доступа задаётся по усмотрению
пользователя. Другие схемы не оставляют решение о том, к чему пользователь или
программа может получить доступ, на усмотрение пользователей или программ. Эти схемы называются механизмами
мандатного контроля доступа, потому что у вас нет выбора в отношении пользователей
или программ, имеющих доступ к фрагментам данных.

Bell & LaPadula
---------------

С середины 1980-х годов и до рубежа веков мандатный контроль
доступа (MAC) был очень тесно связан с моделью безопасности Bell & LaPadula —
математическим описанием политики Министерства обороны Соединённых Штатов
по маркировке бумажных документов. MAC в этой форме пользовался популярностью
внутри Capital Beltway и в скандинавских суперкомпьютерных центрах, но
часто упоминался как не отвечающий общим потребностям.

Domain Type Enforcement
-----------------------

Примерно на рубеже веков стала популярной Domain Type Enforcement (DTE).
Эта схема организует пользователей, программы и данные в домены, которые
защищены друг от друга. Эта схема широко применяется в качестве компонента
популярных дистрибутивов Linux. Административные накладные расходы, требуемые для
поддержки этой схемы, и детальное понимание всей системы,
необходимое для обеспечения безопасного отображения доменов, приводят к тому, что в большинстве случаев эта схема
отключается или используется ограниченным образом.

Smack
-----

Smack — это механизм мандатного контроля доступа, спроектированный для предоставления полезного MAC
при избегании подводных камней его предшественников. Ограничения Bell &
LaPadula устраняются предоставлением схемы, в которой доступом можно управлять
в соответствии с требованиями системы и её назначением, а не теми,
что навязаны замысловатой государственной политикой. Сложность Domain Type
Enforcement также устраняется определением механизмов контроля доступа в терминах режимов доступа,
которые уже используются.

Терминология Smack
------------------

Жаргон, используемый при обсуждении Smack, будет знаком тем, кто имел дело
с другими MAC-системами, и не должен быть слишком трудным для усвоения непосвящёнными.
Есть четыре термина, которые используются особым образом и которые
особенно важны:

  Subject (субъект):
	Субъект — это активная сущность в компьютерной системе.
	В Smack субъект — это задача (task), которая, в свою очередь, является базовой единицей
	выполнения.

  Object (объект):
	Объект — это пассивная сущность в компьютерной системе.
	В Smack объектами могут быть файлы всех типов, IPC и задачи.

  Access (доступ):
	Любая попытка субъекта поместить информацию в объект или получить
	информацию из объекта является доступом.

  Label (метка):
	Данные, идентифицирующие характеристики мандатного контроля доступа
	субъекта или объекта.

Эти определения согласуются с традиционным использованием в сообществе
безопасности. Есть также некоторые термины из Linux, которые, вероятно, всплывут:

  Capability (право, capability):
	Задача, обладающая некоторым правом (capability), имеет разрешение
	нарушать тот или иной аспект политики безопасности системы, определяемый
	конкретным правом. Задача, обладающая одним или несколькими
	правами, является привилегированной задачей, тогда как задача без
	прав является непривилегированной задачей.

  Privilege (привилегия):
	Про задачу, которой разрешено нарушать политику безопасности
	системы, говорят, что она обладает привилегией. На момент написания этого текста задача может
	обладать привилегией либо за счёт обладания правами (capabilities), либо за счёт наличия
	эффективного пользователя root.

Основы Smack
------------

Smack — это расширение системы Linux. Оно навязывает дополнительные ограничения
на то, к каким объектам могут получать доступ субъекты, на основе меток, прикреплённых к
каждому из субъектов и объектов.

Метки
~~~~~

Метки Smack — это строки символов ASCII. Они могут иметь длину до 255 символов,
но рекомендуется ограничивать их двадцатью тремя символами.
Односимвольные метки, использующие специальные символы, то есть всё, что
не является буквой или цифрой, зарезервированы для использования командой разработки
Smack. Метки Smack неструктурированы, чувствительны к регистру, и единственная
операция, которая когда-либо над ними выполняется, — это сравнение на равенство. Метки Smack не могут
содержать непечатаемые символы, символы "/" (слеш), "\" (обратный слеш), "'"
(одинарную кавычку) и '"' (двойную кавычку).
Метки Smack не могут начинаться с '-'. Это зарезервировано для специальных опций.

Существует несколько предопределённых меток::

	_ 	Pronounced "floor", a single underscore character.
	^ 	Pronounced "hat", a single circumflex character.
	* 	Pronounced "star", a single asterisk character.
	? 	Pronounced "huh", a single question mark character.
	@ 	Pronounced "web", a single at sign character.

Каждой задаче в системе со Smack присваивается метка. Метка Smack
процесса обычно будет присваиваться механизмом инициализации
системы.

Правила доступа
~~~~~~~~~~~~~~~

Smack использует традиционные режимы доступа Linux. Этими режимами являются чтение,
выполнение, запись и иногда добавление (append). Есть несколько случаев, когда
режим доступа может быть неочевиден. К ним относятся:

  Сигналы:
	Сигнал — это операция записи от субъекта-задачи к
	объекту-задаче.

  Internet Domain IPC:
	Передача пакета считается
	операцией записи от задачи-источника к задаче-получателю.

Smack ограничивает доступ на основе метки, прикреплённой к субъекту, и метки,
прикреплённой к объекту, к которому он пытается получить доступ. Применяемые правила,
по порядку, таковы:

	1. Любой доступ, запрошенный задачей с меткой "*", отклоняется.
	2. Доступ на чтение или выполнение, запрошенный задачей с меткой "^",
	   разрешается.
	3. Доступ на чтение или выполнение, запрошенный к объекту с меткой "_",
	   разрешается.
	4. Любой доступ, запрошенный к объекту с меткой "*", разрешается.
	5. Любой доступ, запрошенный задачей к объекту с такой же
	   меткой, разрешается.
	6. Любой доступ, явно определённый в загруженном
	   наборе правил, разрешается.
	7. Любой другой доступ отклоняется.

Правила доступа Smack
~~~~~~~~~~~~~~~~~~~~~

Благодаря изоляции, обеспечиваемой Smack, разделение доступа является простым. Есть
много интересных случаев, когда желателен ограниченный доступ субъектов к объектам с
различными метками. Один из примеров — знакомая шпионская модель
конфиденциальности, в которой учёный, работающий над сильно засекреченным проектом, мог бы
читать документы более низких уровней секретности, а всё, что она пишет,
будет «рождаться» сильно засекреченным. Для поддержки таких схем Smack включает
механизм задания правил, разрешающих доступ между метками.

Формат правила доступа
~~~~~~~~~~~~~~~~~~~~~~~

Формат правила доступа таков::

	subject-label object-label access

Где subject-label — это метка Smack задачи, object-label — это метка Smack
сущности, к которой осуществляется доступ, а access — это строка, задающая вид
разрешённого доступа. В спецификации доступа ищутся буквы, которые
описывают режимы доступа:

	a: указывает, что следует предоставить доступ на добавление (append).
	r: указывает, что следует предоставить доступ на чтение.
	w: указывает, что следует предоставить доступ на запись.
	x: указывает, что следует предоставить доступ на выполнение.
	t: указывает, что правило запрашивает трансмутацию (transmutation).
	b: указывает, что о правиле следует сообщать для bring-up.

Прописные значения букв спецификации также допускаются.
Спецификации режимов доступа могут идти в любом порядке. Примеры приемлемых правил::

	TopSecret Secret  rx
	Secret    Unclass R
	Manager   Game    x
	User      HR      w
	Snap      Crackle rwxatb
	New       Old     rRrRr
	Closed    Off     -

Примеры неприемлемых правил::

	Top Secret Secret     rx
	Ace        Ace        r
	Odd        spells     waxbeans

Пробелы в метках не допускаются. Поскольку субъект всегда имеет доступ к файлам
с такой же меткой, задание правила для этого случая бессмысленно. В спецификациях
доступа допускаются только допустимые буквы (rwxatbRWXATB) и символ дефиса ('-').
Дефис является заполнителем, поэтому "a-r" — это то же самое,
что "ar". Одиночный дефис используется для указания того, что доступ не должен разрешаться.

Применение правил доступа
~~~~~~~~~~~~~~~~~~~~~~~~~~

Разработчики Linux редко определяют новые виды сущностей, обычно импортируя
схемы и концепции из других систем. Чаще всего эти другие системы являются
вариантами Unix. Unix обладает многими милыми свойствами, но согласованность
моделей контроля доступа не относится к их числу. Smack стремится обрабатывать доступы настолько
единообразно, насколько это разумно, оставаясь при этом в духе лежащего в основе
механизма.

Объекты файловой системы, включая файлы, каталоги, именованные каналы, символические ссылки
и устройства, требуют прав доступа, которые тесно соответствуют тем, что используются при доступе
по битам режима (mode bit). Чтобы открыть файл на чтение, на файле требуется доступ на чтение. Чтобы
выполнить поиск в каталоге, требуется доступ на выполнение. Создание файла с доступом на запись
требует одновременно доступа на чтение и запись к содержащему каталогу. Удаление
файла требует доступа на чтение и запись к файлу и к содержащему
каталогу. Возможно, что пользователь сможет видеть, что файл существует,
но не сможет видеть никакие его атрибуты, в силу того что он имеет доступ на чтение к
содержащему каталогу, но не к файлу с другой меткой. Это
артефакт того, что имя файла является данными в каталоге, а не частью файла.

Если каталог помечен как трансмутирующий (SMACK64TRANSMUTE=TRUE) и
правило доступа, которое позволяет процессу создавать объект в этом каталоге,
включает доступ 't', то метка, присваиваемая новому объекту, будет меткой
каталога, а не создающего процесса. Это значительно облегчает
двум процессам с разными метками совместное использование данных без предоставления
доступа ко всем их файлам.

Объекты IPC, очереди сообщений, наборы семафоров и сегменты памяти существуют в плоских
пространствах имён, и запросы доступа должны соответствовать только рассматриваемому
объекту.

Объекты-процессы отражают задачи в системе, и метка Smack, используемая для доступа к
ним, — это та же метка Smack, которую задача использовала бы для своих собственных попыток
доступа. Отправка сигнала через системный вызов kill() является операцией записи
от отправителя сигнала к получателю. Отладка процесса требует одновременно чтения
и записи. Создание новой задачи является внутренней операцией, которая приводит к двум
задачам с идентичными метками Smack и не требует проверок доступа.

Сокеты — это структуры данных, прикреплённые к процессам, и отправка пакета от
одного процесса к другому требует, чтобы отправитель имел доступ на запись к
получателю. От получателя не требуется иметь доступ на чтение к отправителю.

Установка правил доступа
~~~~~~~~~~~~~~~~~~~~~~~~~

Конфигурационный файл /etc/smack/accesses содержит правила, которые должны быть установлены при
запуске системы. Его содержимое записывается в специальный файл
/sys/fs/smackfs/load2. Правила можно добавлять в любое время, и они вступают в силу
немедленно. Для любой пары меток субъекта и объекта может быть только
одно правило, причём наиболее недавно заданное переопределяет любую более раннюю
спецификацию.

Атрибут задачи
~~~~~~~~~~~~~~

Метку Smack процесса можно прочитать из ``/proc/<pid>/attr/current``.
Процесс может прочитать свою собственную метку Smack из ``/proc/self/attr/current``.
Привилегированный процесс может изменить свою собственную метку Smack записью в
``/proc/self/attr/current``, но не метку другого процесса.

Формат записи: только метка либо метка, за которой следует один из
3 хвостов: ``\n`` (по общему соглашению для интерфейсов ``/proc/...``),
``\0`` (потому что некоторые приложения некорректно его включают),
``\n\0`` (потому что мы считаем, что некоторые приложения могут некорректно его включать).

Атрибут файла
~~~~~~~~~~~~~

Метка Smack объекта файловой системы хранится как расширенный атрибут
с именем SMACK64 на файле. Этот атрибут находится в пространстве имён security. Он может
быть изменён только процессом с привилегией.

Привилегия
~~~~~~~~~~

Процесс с CAP_MAC_OVERRIDE или CAP_MAC_ADMIN является привилегированным.
CAP_MAC_OVERRIDE разрешает процессу доступ к объектам, в котором ему было бы
отказано иначе. CAP_MAC_ADMIN разрешает процессу изменять
данные Smack, включая правила и атрибуты.

Сеть в Smack
~~~~~~~~~~~~

Как упоминалось ранее, Smack навязывает контроль доступа при передаче по
сетевым протоколам. Каждый пакет, отправляемый процессом со Smack, помечается его меткой
Smack. Это делается добавлением тега CIPSO в заголовок IP-пакета. Каждый
полученный пакет, как ожидается, имеет тег CIPSO, идентифицирующий метку, и
если такого тега нет, предполагается окружающая (ambient) сетевая метка. Перед тем как пакет
будет доставлен, выполняется проверка, чтобы определить, что субъект с меткой на
пакете имеет доступ на запись к процессу-получателю, и если это не так,
пакет отбрасывается.

Настройка CIPSO
~~~~~~~~~~~~~~~

Обычно задавать конфигурацию CIPSO не требуется. Значения по умолчанию,
используемые системой, обрабатывают все внутренние случаи. Smack будет составлять значения
меток CIPSO так, чтобы они соответствовали используемым меткам Smack, без вмешательства
администратора. Непомеченным пакетам, поступающим в систему, будет присвоена
окружающая (ambient) метка.

Smack требует настройки в случае, когда могут встречаться пакеты от системы, которая
не использует Smack, но говорит на CIPSO. Обычно это будет система Trusted
Solaris, но существуют и другие, менее широко распространённые системы.
CIPSO предоставляет с каждым пакетом 3 важных значения: домен интерпретации (Domain Of
Interpretation, DOI), уровень и набор категорий. DOI предназначен для идентификации группы
систем, использующих совместимые схемы разметки, и DOI, заданный в
системе со Smack, должен совпадать с DOI удалённой системы, иначе пакеты будут
отброшены. По умолчанию DOI равен 3. Значение можно прочитать из
/sys/fs/smackfs/doi и изменить записью в /sys/fs/smackfs/doi.

Метка и набор категорий отображаются в метку Smack так, как определено в
/etc/smack/cipso.

Отображение Smack/CIPSO имеет вид::

	smack level [category [category]*]

Smack не ожидает, что уровень или наборы категорий связаны каким-либо
конкретным образом, и не предполагает и не назначает доступы на их основе. Несколько
примеров отображений::

	TopSecret 7
	TS:A,B    7 1 2
	SecBDE    5 2 4 6
	RAFTERS   7 12 26

Символы ":" и "," допускаются в метке Smack, но не имеют особого
значения.

Отображение меток Smack в значения CIPSO определяется записью в
/sys/fs/smackfs/cipso2.

В дополнение к явным отображениям Smack поддерживает прямые отображения CIPSO. Один
уровень CIPSO используется для указания того, что набор категорий, передаваемый в пакете,
на самом деле является кодировкой метки Smack. По умолчанию используемый уровень равен 250.
Значение можно прочитать из /sys/fs/smackfs/direct и изменить записью в
/sys/fs/smackfs/direct.

Атрибуты сокетов
~~~~~~~~~~~~~~~~

Есть два атрибута, связанных с сокетами. Эти атрибуты
могут быть установлены только привилегированными задачами, но любая задача может прочитать их для своих собственных
сокетов.

  SMACK64IPIN:
	Метка Smack объекта-задачи. Привилегированная
	программа, которая будет навязывать политику, может установить её в метку star.

  SMACK64IPOUT:
	Метка Smack, передаваемая с исходящими пакетами.
	Привилегированная программа может установить её так, чтобы она соответствовала метке другой
	задачи, с которой она надеется обмениваться данными.

Сокет домена UNIX (UDS) с BSD-адресом функционирует одновременно как файл в
файловой системе и как сокет. Как файл, он несёт атрибут SMACK64. Этот
атрибут не участвует в навязывании безопасности Smack и неизменяемо
присваивается меткой "*".

Исключения netlabel в Smack
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Вы часто будете обнаруживать, что вашему помеченному приложению приходится общаться с внешним,
непомеченным миром. Чтобы сделать это, существует специальный файл /sys/fs/smackfs/netlabel,
куда вы можете добавить некоторые исключения в виде::

	@IP1	   LABEL1 or
	@IP2/MASK  LABEL2

Это означает, что ваше приложение будет иметь непомеченный доступ к @IP1, если оно имеет
доступ на запись к LABEL1, и доступ к подсети @IP2/MASK, если оно имеет доступ
на запись к LABEL2.

Записи в файле /sys/fs/smackfs/netlabel сопоставляются по наибольшей маске
первой, как в бесклассовой маршрутизации IPv4.

Там можно использовать специальную метку '@' и опцию '-CIPSO'::

	@      means Internet, any application with any label has access to it
	-CIPSO means standard CIPSO networking

Если вы не знаете, что такое CIPSO, и не планируете его использовать, вы можете просто сделать::

	echo 127.0.0.1 -CIPSO > /sys/fs/smackfs/netlabel
	echo 0.0.0.0/0 @      > /sys/fs/smackfs/netlabel

Если вы используете CIPSO в своей локальной сети 192.168.0.0/16 и также нуждаетесь в непомеченном
доступе в Интернет, вы можете иметь::

	echo 127.0.0.1      -CIPSO > /sys/fs/smackfs/netlabel
	echo 192.168.0.0/16 -CIPSO > /sys/fs/smackfs/netlabel
	echo 0.0.0.0/0      @      > /sys/fs/smackfs/netlabel

Написание приложений для Smack
------------------------------

Существует три рода приложений, которые будут работать в системе со Smack. То, как
приложение взаимодействует со Smack, определит, что ему придётся делать, чтобы
корректно работать под Smack.

Приложения, не знающие о Smack
------------------------------

Подавляющее большинство приложений не имеют никаких причин заботиться об
уникальных свойствах Smack. Поскольку запуск программы не влияет на
метку Smack, связанную с процессом, единственная забота, которая, вероятно, возникнет, —
имеет ли процесс доступ на выполнение к программе.

Приложения, для которых Smack имеет значение
--------------------------------------------

Некоторые программы можно улучшить, обучив их о Smack, но сами они не принимают
никаких решений о безопасности. Утилита ls(1) — один из примеров такой
программы.

Приложения, навязывающие Smack
------------------------------

Это специальные программы, которые не только знают о Smack, но и участвуют в
навязывании политики системы. В большинстве случаев это программы, которые
настраивают пользовательские сеансы. Существуют также сетевые службы, которые предоставляют информацию
процессам, работающим с различными метками.

Интерфейсы файловой системы
---------------------------

Smack поддерживает метки на объектах файловой системы, используя расширенные атрибуты.
Метку Smack файла, каталога или другого объекта файловой системы можно получить
с помощью getxattr(2)::

	len = getxattr("/", "security.SMACK64", value, sizeof (value));

поместит метку Smack корневого каталога в value. Привилегированный
процесс может установить метку Smack объекта файловой системы с помощью setxattr(2)::

	len = strlen("Rubble");
	rc = setxattr("/foo", "security.SMACK64", "Rubble", len, 0);

установит метку Smack для /foo в "Rubble", если программа имеет соответствующую
привилегию.

Интерфейсы сокетов
------------------

Атрибуты сокета можно прочитать с помощью fgetxattr(2).

Привилегированный процесс может установить метку Smack исходящих пакетов с помощью
fsetxattr(2)::

	len = strlen("Rubble");
	rc = fsetxattr(fd, "security.SMACK64IPOUT", "Rubble", len, 0);

установит метку Smack "Rubble" на пакетах, исходящих из сокета, если
программа имеет соответствующую привилегию::

	rc = fsetxattr(fd, "security.SMACK64IPIN, "*", strlen("*"), 0);

установит метку Smack "*" в качестве метки объекта, относительно которой будут
проверяться входящие пакеты, если программа имеет соответствующую привилегию.

Администрирование
-----------------

Smack поддерживает некоторые опции монтирования:

  smackfsdef=label:
	задаёт метку, присваиваемую файлам, у которых отсутствует
	расширенный атрибут метки Smack.

  smackfsroot=label:
	задаёт метку, присваиваемую корню
	файловой системы, если у него отсутствует расширенный атрибут Smack.

  smackfshat=label:
	задаёт метку, которая должна иметь доступ на чтение ко
	всем меткам, установленным в файловой системе. Пока не навязывается.

  smackfsfloor=label:
	задаёт метку, к которой все метки, установленные в
	файловой системе, должны иметь доступ на чтение. Пока не навязывается.

  smackfstransmute=label:
	ведёт себя в точности как smackfsroot, за исключением того, что также
	устанавливает флаг transmute на корне точки монтирования

Эти опции монтирования применимы ко всем типам файловых систем.

Аудит Smack
-----------

Если вам нужен аудит Smack событий безопасности, вам нужно установить CONFIG_AUDIT
в конфигурации вашего ядра.
По умолчанию все отклонённые события будут проходить аудит. Вы можете изменить это поведение,
записав один символ в файл /sys/fs/smackfs/logging::

	0 : no logging
	1 : log denied (default)
	2 : log accepted
	3 : log denied & accepted

События журналируются как пары 'key=value'; для каждого события вы как минимум получите
субъект, объект, запрошенные права, действие, функцию ядра,
которая инициировала событие, плюс другие пары в зависимости от типа события,
прошедшего аудит.

Режим Bringup
-------------

Режим bringup предоставляет возможности журналирования, которые могут облегчить
настройку приложений и запуск (bringup) системы. Сконфигурируйте ядро с
CONFIG_SECURITY_SMACK_BRINGUP, чтобы включить эти возможности. Когда режим bringup
включён, доступы, которые проходят благодаря правилам, помеченным режимом доступа "b",
будут журналироваться. Когда для процессов вводится новая метка,
правила можно добавлять агрессивно, пометив их "b". Журналирование позволяет
отслеживать, какие правила фактически используются для этой метки.

Ещё одна возможность режима bringup — это опция "unconfined". Запись
метки в /sys/fs/smackfs/unconfined делает субъекты с этой меткой
способными получать доступ к любому объекту, а объекты с этой меткой — доступными
всем субъектам. Любой доступ, который предоставляется потому, что метка unconfined,
журналируется. Эта возможность опасна, поскольку файлы и каталоги могут
создаваться в местах, где они не могли бы, если бы политика навязывалась.
