Логгер oops/panic для блочных устройств pstore block
====================================================

Введение
--------

pstore block (pstore/blk) — это логгер oops/panic, который записывает свои
журналы на блочное и не-блочное устройство до того, как система аварийно
завершит работу. Получить эти файлы журналов можно, смонтировав файловую
систему pstore примерно так::

    mount -t pstore pstore /sys/fs/pstore


Концепции pstore block
----------------------

pstore/blk предоставляет эффективный способ настройки pstore/blk, который
делит все конфигурации на две части: конфигурации для пользователя и
конфигурации для драйвера.

Конфигурации для пользователя определяют, как работает pstore/blk, например
pmsg_size, kmsg_size и так далее. Все они поддерживают как Kconfig, так и
параметры модуля, но параметры модуля имеют приоритет над Kconfig.

Конфигурации для драйвера полностью относятся к блочному и не-блочному
устройству, например total_size блочного устройства и операции
чтения/записи.

Конфигурации для пользователя
-----------------------------

Все эти конфигурации поддерживают как Kconfig, так и параметры модуля, но
параметры модуля имеют приоритет над Kconfig.

Вот пример параметров модуля::

        pstore_blk.blkdev=/dev/mmcblk0p7 pstore_blk.kmsg_size=64 best_effort=y

Подробности по каждой из конфигураций могут быть вам интересны.

blkdev
~~~~~~

Блочное устройство, которое будет использоваться. Чаще всего это раздел
блочного устройства. Оно обязательно для pstore/blk. Оно также используется
для MTD-устройства.

Когда pstore/blk собран как модуль, "blkdev" принимает следующие варианты:

1. /dev/<disk_name> представляет номер устройства диска
#. /dev/<disk_name><decimal> представляет номер устройства раздела — номер
   устройства диска плюс номер раздела
#. /dev/<disk_name>p<decimal> — то же, что и выше; эта форма используется,
   когда имя диска с разделами заканчивается цифрой.

Когда pstore/blk встроен в ядро, "blkdev" принимает следующие варианты:

#. <hex_major><hex_minor> номер устройства в шестнадцатеричном представлении,
   без ведущего 0x, например b302.
#. PARTUUID=00112233-4455-6677-8899-AABBCCDDEEFF представляет уникальный
   идентификатор раздела, если его предоставляет таблица разделов. Этот UUID
   может быть либо UUID формата EFI/GPT, либо ссылаться на раздел MSDOS с
   использованием формата SSSSSSSS-PP, где SSSSSSSS — это дополненное нулями
   шестнадцатеричное представление 32-битной «сигнатуры диска NT»
   ("NT disk signature"), а PP — дополненное нулями шестнадцатеричное
   представление номера раздела, начинающегося с 1.
#. PARTUUID=<UUID>/PARTNROFF=<int> для выбора раздела относительно раздела с
   известным уникальным идентификатором.
#. <major>:<minor> старший (major) и младший (minor) номер устройства,
   разделённые двоеточием.

Для MTD-устройства принимаются следующие варианты:

1. <device name> имя MTD-устройства. Рекомендуется "pstore".
#. <device number> номер MTD-устройства.

kmsg_size
~~~~~~~~~

Размер чанка (chunk) в КБ для фронтенда oops/panic. Он **ДОЛЖЕН** быть
кратен 4. Он необязателен, если журнал oops/panic вас не интересует.

Для фронтенда oops/panic существует несколько чанков в зависимости от
оставшегося пространства за вычетом других фронтендов pstore.

pstore/blk будет вести запись в чанки oops/panic один за другим и всегда
перезаписывает самый старый чанк, если свободных чанков больше не осталось.

pmsg_size
~~~~~~~~~

Размер чанка в КБ для фронтенда pmsg. Он **ДОЛЖЕН** быть кратен 4. Он
необязателен, если журнал pmsg вас не интересует.

В отличие от фронтенда oops/panic, для фронтенда pmsg существует только один
чанк.

Pmsg — это доступный из пространства пользователя объект pstore. Записи в
*/dev/pmsg0* добавляются в чанк. После перезагрузки содержимое доступно в
*/sys/fs/pstore/pmsg-pstore-blk-0*.

console_size
~~~~~~~~~~~~

Размер чанка в КБ для фронтенда console. Он **ДОЛЖЕН** быть кратен 4. Он
необязателен, если журнал console вас не интересует.

Аналогично фронтенду pmsg, для фронтенда console существует только один чанк.

Весь журнал console будет добавляться в чанк. После перезагрузки содержимое
доступно в */sys/fs/pstore/console-pstore-blk-0*.

ftrace_size
~~~~~~~~~~~

Размер чанка в КБ для фронтенда ftrace. Он **ДОЛЖЕН** быть кратен 4. Он
необязателен, если журнал ftrace вас не интересует.

Аналогично фронтенду oops, для фронтенда ftrace существует несколько чанков
в зависимости от числа процессоров (cpu). Размер каждого чанка равен
ftrace_size / processors_count.

Весь журнал ftrace будет добавляться в чанк. После перезагрузки содержимое
объединяется и доступно в */sys/fs/pstore/ftrace-pstore-blk-0*.

Постоянная трассировка функций (persistent function tracing) может быть
полезна для отладки зависаний, связанных с программным или аппаратным

max_reason
~~~~~~~~~~

Ограничение того, какие виды дампов kmsg сохраняются, можно контролировать с
помощью значения ``max_reason``, как определено в ``enum kmsg_dump_reason``
из include/linux/kmsg_dump.h. Например, чтобы сохранять как Oops, так и Panic,
``max_reason`` следует установить в 2 (KMSG_DUMP_OOPS); чтобы сохранять только
Panic, ``max_reason`` следует установить в 1 (KMSG_DUMP_PANIC). Установка этого
значения в 0 (KMSG_DUMP_UNDEF) означает, что фильтрация по причине будет
контролироваться параметром загрузки ``printk.always_kmsg_dump``: если он не
задан, это будет KMSG_DUMP_OOPS, в противном случае — KMSG_DUMP_MAX.

Сжатие и заголовок
------------------

Блочное устройство достаточно велико для несжатых данных oops. На самом деле
мы не рекомендуем сжатие данных, потому что pstore/blk вставляет некоторую
информацию в первую строку данных oops/panic. Например::

        Panic: Total 16 times

Это означает, что это OOPS|Panic в 16-й раз с момента первой загрузки. Иногда
число случаев oops|panic с момента первой загрузки важно для оценки того,
стабильна ли система.

Следующая строка вставляется файловой системой pstore. Например::

        Oops#2 Part1

Это означает, что это OOPS во 2-й раз при последней загрузке.

Чтение данных
-------------

Данные дампа можно прочитать из файловой системы pstore. Формат этих файлов —
``dmesg-pstore-blk-[N]`` для фронтенда oops/panic, ``pmsg-pstore-blk-0`` для
фронтенда pmsg и так далее. Временная метка файла дампа фиксирует время
срабатывания. Чтобы удалить сохранённую запись с блочного устройства, просто
выполните unlink соответствующего файла pstore.

На что обратить внимание в API чтения/записи при panic
------------------------------------------------------

Если при panic ядро не собирается работать намного дольше, задачи не будут
планироваться, и большинство ресурсов ядра окажется недоступно. Это похоже на
однопоточную программу, выполняющуюся на одноядерном компьютере.

Следующие моменты требуют особого внимания для API чтения/записи при panic:

1. **НЕЛЬЗЯ** выделять никакую память.
   Если вам нужна память, просто выделяйте её во время инициализации блочного
   драйвера, а не ожидайте до момента panic.
#. Должно быть с опросом (polled), **НЕ** управляемым прерываниями.
   Планирования задач больше нет. Блочный драйвер должен делать задержку,
   чтобы гарантировать успех записи, но НЕ переходить в сон.
#. **НЕЛЬЗЯ** захватывать никакую блокировку.
   Нет ни других задач, ни каких-либо разделяемых ресурсов; вы можете
   безопасно сломать все блокировки.
#. Просто используйте CPU для передачи.
   Не используйте DMA для передачи, если только вы не уверены, что DMA не
   будет удерживать блокировку.
#. Управляйте регистрами напрямую.
   Пожалуйста, управляйте регистрами напрямую, а не используйте ресурсы ядра
   Linux. Выполняйте I/O-отображение во время инициализации, а не ожидайте до
   момента возникновения panic.
#. При необходимости сбросьте ваше блочное устройство и контроллер.
   Если вы не уверены в состоянии вашего блочного устройства и контроллера в
   момент возникновения panic, вы можете безопасно остановить и сбросить их.

pstore/blk поддерживает psblk_blkdev_info(), которая определена в
*linux/pstore_blk.h*, для получения информации об используемом блочном
устройстве, такой как номер устройства, число секторов и начальный сектор
всего диска.
