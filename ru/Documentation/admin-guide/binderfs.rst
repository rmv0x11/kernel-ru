Файловая система Android binderfs
=================================

Android binderfs — это файловая система для механизма IPC binder в Android. Она
позволяет динамически добавлять и удалять binder-устройства во время работы.
Binder-устройства, расположенные в новом экземпляре binderfs, независимы от
binder-устройств, расположенных в других экземплярах binderfs. Монтирование
нового экземпляра binderfs даёт возможность получить набор приватных
binder-устройств.

Монтирование binderfs
---------------------

Android binderfs может быть смонтирована с помощью::

  mkdir /dev/binderfs
  mount -t binder binder /dev/binderfs

после чего новый экземпляр binderfs появится в ``/dev/binderfs``.
В свежем экземпляре binderfs не будет присутствовать ни одного binder-устройства.
Будет лишь устройство ``binder-control``, которое служит обработчиком запросов
для binderfs. Монтирование другого экземпляра binderfs в другом месте создаст
новый и отдельный экземпляр, независимый от всех остальных монтирований binderfs.
Это идентично поведению, например, ``devpts`` и ``tmpfs``. Файловая система
Android binderfs может монтироваться в пользовательских пространствах имён.

Опции
-----
max
  Экземпляры binderfs могут монтироваться с ограничением на число
  binder-устройств, которые можно выделить. Опция монтирования ``max=<count>``
  служит ограничением для конкретного экземпляра. Если задано ``max=<count>``,
  то в этом экземпляре binderfs можно выделить только ``<count>``
  binder-устройств.

stats
  Использование ``stats=global`` включает глобальную статистику binder.
  ``stats=global`` доступна только для экземпляра binderfs, смонтированного в
  начальном пользовательском пространстве имён. Попытка использовать эту опцию
  для монтирования экземпляра binderfs в другом пользовательском пространстве
  имён вернёт ошибку доступа.

Выделение binder-устройств
--------------------------

.. _ioctl: http://man7.org/linux/man-pages/man2/ioctl.2.html

Чтобы выделить новое binder-устройство в экземпляре binderfs, необходимо
отправить запрос через узел устройства ``binder-control``. Запрос отправляется в
виде `ioctl() <ioctl_>`_.

Программе нужно открыть узел устройства ``binder-control`` и отправить ядру
запрос ``BINDER_CTL_ADD``. Пользователи binderfs должны сообщить ядру, какое имя
должно получить новое binder-устройство. По умолчанию имя может содержать не
более ``BINDERFS_MAX_NAME`` символов, включая завершающий нулевой байт.

Как только запрос сделан через `ioctl() <ioctl_>`_ с передачей ядру ``struct
binder_device`` с именем, ядро выделит новое binder-устройство и вернёт в
структуре старший и младший номер нового устройства (это необходимо, поскольку
binderfs выделяет старший номер устройства динамически). После возврата из
`ioctl() <ioctl_>`_ под /dev/binderfs появится новое binder-устройство с
выбранным именем.

Удаление binder-устройств
-------------------------

.. _unlink: http://man7.org/linux/man-pages/man2/unlink.2.html
.. _rm: http://man7.org/linux/man-pages/man1/rm.1.html

Binder-устройства binderfs можно удалить с помощью `unlink() <unlink_>`_. Это
означает, что для их удаления можно использовать утилиту `rm() <rm_>`_. Обратите
внимание, что устройство ``binder-control`` удалить нельзя, поскольку это сделало
бы экземпляр binderfs непригодным для использования. Устройство
``binder-control`` будет удалено, когда экземпляр binderfs будет размонтирован и
все ссылки на него будут отброшены.

Возможности binder
------------------

Предполагая, что экземпляр binderfs смонтирован в ``/dev/binderfs``,
возможности, поддерживаемые драйвером binder, можно найти в
``/dev/binderfs/features/``. Наличие отдельных файлов можно проверить, чтобы
определить, поддерживается ли конкретная возможность драйвером.

Пример::

        cat /dev/binderfs/features/oneway_spam_detection
        1
