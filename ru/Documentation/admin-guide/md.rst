RAID-массивы
============

Сборка RAID-массивов во время загрузки
--------------------------------------

Инструменты для управления md-устройствами можно найти по адресу
   https://www.kernel.org/pub/linux/utils/raid/


Загрузиться с вашего md-устройства можно с помощью следующих параметров
командной строки ядра:

для старых raid-массивов без постоянных суперблоков::

  md=<md device no.>,<raid level>,<chunk size factor>,<fault level>,dev0,dev1,...,devn

для raid-массивов с постоянными суперблоками::

  md=<md device no.>,dev0,dev1,...,devn

или, чтобы собрать массив с поддержкой разделов::

  md=d<md device no.>,dev0,dev1,...,devn

``md device no.``
+++++++++++++++++

Номер md-устройства

================= =========
``md device no.`` device
================= =========
              0		md0
	      1		md1
	      2		md2
	      3		md3
	      4		md4
================= =========

``raid level``
++++++++++++++

уровень RAID-массива

=============== =============
``raid level``  level
=============== =============
-1		linear mode
0		striped mode
=============== =============

остальные режимы поддерживаются только при наличии постоянных суперблоков

``chunk size factor``
+++++++++++++++++++++

(только для raid-0 и raid-1)

Задаёт размер блока (chunk) как 4k << n.

``fault level``
+++++++++++++++

Полностью игнорируется

``dev0`` to ``devn``
++++++++++++++++++++

например ``/dev/hda1``, ``/dev/hdc1``, ``/dev/sda1``, ``/dev/sdb1``

Возможная строка loadlin (Harald Hoyer <HarryH@Royal.Net>) выглядит так::

	e:\loadlin\loadlin e:\zimage root=/dev/md0 md=0,0,4,0,/dev/hdb2,/dev/hdc3 ro


Автоопределение RAID-массивов во время загрузки
-----------------------------------------------

Когда md встроен в ядро (а не собран как модуль), сканируются разделы
типа 0xfd, которые автоматически собираются в RAID-массивы.
Это автоопределение можно подавить параметром ядра
``raid=noautodetect``.  Начиная с ядра 2.6.9, автоматически определяться
и запускаться во время загрузки могут только диски с суперблоком типа 0.

Параметр ядра ``raid=partitionable`` (или ``raid=part``) означает,
что все автоматически определённые массивы собираются с поддержкой разделов.

Сборка деградировавших/грязных массивов во время загрузки
---------------------------------------------------------

Если массив raid5 или raid6 одновременно грязный и деградировавший, он может
иметь необнаружимое повреждение данных.  Это связано с тем, что факт его
``dirty`` означает, что чётности нельзя доверять, а факт того, что он
деградировал, означает, что некоторые блоки данных отсутствуют и не могут быть
надёжно восстановлены (из-за отсутствия чётности).

По этой причине md обычно отказывается запускать такой массив.  Это требует от
системного администратора предпринять действия, чтобы явно запустить массив
несмотря на возможное повреждение.  Обычно это делается так::

   mdadm --assemble --force ....

Этот вариант фактически недоступен, если на массиве размещена корневая
файловая система.  Чтобы поддержать загрузку с такого массива, md поддерживает
параметр модуля ``start_dirty_degraded``, который при установке в 1 обходит
проверки и позволяет запускать грязные деградировавшие массивы.

Итак, чтобы загрузиться с корневой файловой системой на грязном деградировавшем
raid 5 или 6, используйте::

   md-mod.start_dirty_degraded=1


Форматы суперблоков
-------------------

Драйвер md может поддерживать различные форматы суперблоков.
В настоящее время он поддерживает форматы суперблоков ``0.90.0`` и формат ``md-1``,
появившийся в серии разработки 2.5.

Ядро автоматически определит, какой формат суперблока используется.

Формат суперблока ``0`` обрабатывается иначе, чем остальные, по
историческим причинам — это исходный формат суперблока.


Общие правила — применимы ко всем форматам суперблоков
------------------------------------------------------

Массив ``created`` путём записи соответствующих суперблоков на все
устройства.

Он ``assembled`` путём связывания каждого из этих устройств с
определённым виртуальным md-устройством.  После того как он полностью собран,
к нему можно обращаться.

Массив должен создаваться инструментом пространства пользователя.  Он запишет
суперблоки на все устройства.  Обычно он помечает массив как
``unclean`` или с отсутствием некоторых устройств, чтобы драйвер md в ядре
мог создать соответствующую избыточность (копирование в raid 1, вычисление
чётности в raid 4/5).

Когда массив собирается, он сначала инициализируется с помощью ioctl
SET_ARRAY_INFO.  Он содержит, в частности, старший и младший номер
версии.  Старший номер версии выбирает, какой формат суперблока
будет использоваться.  Младший номер может использоваться для тонкой настройки
обработки формата, например для указания, где на каждом устройстве искать
суперблок.

Затем каждое устройство добавляется с помощью ioctl ADD_NEW_DISK.  Он
предоставляет, в частности, старший и младший номер, идентифицирующие
добавляемое устройство.

Массив запускается с помощью ioctl RUN_ARRAY.

После запуска можно добавлять новые устройства.  На них должен быть записан
соответствующий суперблок, после чего они передаются с помощью
ADD_NEW_DISK.

Устройства, которые отказали или ещё не активны, можно отсоединить от
массива с помощью HOT_REMOVE_DISK.


Специфические правила для массивов с суперблоком формата 0 и массивов без суперблока (непостоянных)
---------------------------------------------------------------------------------------------------

Массив можно ``created``, описав массив (уровень, размер блока (chunksize)
и т.д.) в ioctl SET_ARRAY_INFO.  При этом должны выполняться ``major_version==0`` и
``raid_disks != 0``.

Затем можно добавлять неинициализированные устройства с помощью ADD_NEW_DISK.
Структура, передаваемая в ADD_NEW_DISK, должна указывать состояние устройства
и его роль в массиве.

После запуска с помощью RUN_ARRAY неинициализированные запасные (spare)
устройства можно добавлять с помощью HOT_ADD_DISK.


MD-устройства в sysfs
---------------------

md-устройства появляются в sysfs (``/sys``) как обычные блочные устройства,
например::

   /sys/block/md0

Каждое ``md``-устройство будет содержать подкаталог с именем ``md``, который
содержит дополнительную md-специфичную информацию об устройстве.

Все md-устройства содержат:

  level
     текстовый файл, указывающий ``raid level``, например raid0, raid1,
     raid5, linear, multipath, faulty.
     Если уровень raid ещё не установлен (массив всё ещё собирается),
     значение будет отражать то, что в него было записано, что может быть
     именем вроде приведённых выше, либо числом, таким как
     ``0``, ``5`` и т.д.

  raid_disks
     текстовый файл с простым числом, указывающим количество устройств
     в полностью работоспособном массиве.  Если это ещё не известно, файл
     будет пустым.  Если массив изменяет размер, он будет содержать
     новое количество устройств.
     Некоторые уровни raid позволяют устанавливать это значение, пока массив
     активен.  Это перенастроит массив.  В противном случае его можно
     устанавливать только при сборке массива.
     Изменение этого атрибута не будет разрешено, если оно
     уменьшило бы размер массива.  Чтобы уменьшить количество дисков
     в, например, raid5, размер массива сначала должен быть уменьшен путём
     установки атрибута ``array_size``.

  chunk_size
     Это размер в байтах для ``chunks`` (блоков), и он имеет значение только для
     уровней raid, в которых задействована чередующаяся запись (striping)
     (0,4,5,6,10). Адресное пространство массива концептуально разделено на
     блоки (chunks), и последовательные блоки записываются чередованием на
     соседние устройства.
     Размер должен быть не менее PAGE_SIZE (4k) и должен быть степенью
     числа 2.  Это можно установить только при сборке массива.

  layout
     ``layout`` (компоновка) массива для конкретного уровня.  Это
     просто число, которое по-разному интерпретируется разными
     уровнями.  Его можно записывать при сборке массива.

  array_size
     Это можно использовать для искусственного ограничения доступного места в
     массиве, чтобы оно было меньше, чем фактически доступно на совокупности
     устройств.  Запись числа (в килобайтах), которое меньше
     доступного размера, установит этот размер.  Любая перенастройка
     массива (например, добавление устройств) не приведёт к изменению размера.
     Запись слова ``default`` приведёт к тому, что фактический размер
     массива станет равным тому размеру, который реально доступен на основе
     ``level``, ``chunk_size`` и ``component_size``.

     Это можно использовать для уменьшения размера массива перед уменьшением
     количества устройств в raid4/5/6 или для поддержки внешних
     форматов метаданных, которые предписывают такое усечение.

  logical_block_size
     Настраивает логический размер блока массива в байтах. Этот атрибут
     поддерживается только для метаданных 1.x. Запишите значение до запуска
     массива. Итоговый LBS массива использует максимум из этой
     настройки и LBS всех совокупных устройств. Обратите внимание, что
     LBS не может превышать PAGE_SIZE, пока RAID не поддерживает folio.
     WARNING: Массивы, созданные на новом ядре, не могут быть собраны на старом
     ядре из-за проверки заполнения (padding). Установите параметр модуля 'check_new_feature'
     в false, чтобы обойти это, но может произойти потеря данных.

  reshape_position
     Это либо ``none``, либо номер сектора в пределах устройств
     массива, до которого дошёл ``reshape``.  Если это установлено, три
     атрибута, упомянутых выше (raid_disks, chunk_size, layout), могут
     потенциально иметь по 2 значения — старое и новое.  Если эти
     значения различаются, чтение атрибута возвращает::

        new (old)

     а запись повлияет на значение ``new``, оставив ``old``
     неизменным.

  component_size
     Для массивов с избыточностью данных (т.е. не raid0, linear, faulty,
     multipath) все компоненты должны быть одного размера — или, по крайней мере,
     должен существовать размер, под который они все предоставляют место.  Это ключевая
     часть геометрии массива.  Он измеряется в секторах
     и может быть прочитан отсюда.  Запись в это значение может изменить размер
     массива, если это поддерживает personality (raid1, raid5, raid6),
     и если компонентные диски достаточно велики.

  metadata_version
     Это указывает формат, используемый для записи метаданных
     о массиве.  Он может быть 0.90 (традиционный формат), 1.0, 1.1,
     1.2 (более новый формат в различных расположениях) или ``none``, указывая, что
     ядро вообще не управляет метаданными.
     Также он может быть ``external:`` за которым следует строка, которая
     устанавливается пространством пользователя.  Это указывает, что метаданными управляет
     программа пространства пользователя.  Любой отказ устройства или другое событие, которое
     требует обновления метаданных, приведёт к приостановке активности массива
     до тех пор, пока событие не будет подтверждено.

  resync_start
     Точка, в которой должна начаться повторная синхронизация (resync).  Если resync не нужен,
     это будет очень большое число (или ``none`` начиная с 2.6.30-rc1).  При
     создании массива оно по умолчанию равно 0, хотя запуск массива как
     ``clean`` установит его гораздо большим.

  new_dev
     В этот файл можно писать, но нельзя читать.  Записываемое значение должно
     быть номером блочного устройства в виде major:minor, например 8:0.
     Это приведёт к присоединению данного устройства к массиву, если оно
     доступно.  Затем оно появится по адресу md/dev-XXX (в зависимости от
     имени устройства), после чего возможна дальнейшая настройка.

  safe_mode_delay
     Когда md-массив не видел запросов на запись в течение определённого периода
     времени, он помечается как ``clean``.  Когда поступает следующий запрос
     на запись, массив помечается как ``dirty`` до начала записи.
     Это называется ``safe_mode``.
     ``Определённый период`` управляется этим файлом, который хранит
     период в виде числа секунд.  По умолчанию это 200 мс (0.200).
     Запись значения 0 отключает safemode.

  array_state
     Этот файл содержит одно слово, которое описывает текущее
     состояние массива.  Во многих случаях состояние можно установить,
     записав слово для желаемого состояния, однако некоторые состояния
     не могут быть установлены явно, а некоторые переходы не разрешены.

     Select/poll работает на этом файле.  Уведомляются обо всех изменениях,
     кроме переходов между Active_idle и active (которые могут быть частыми и не
     очень интересны).  Переход active->active_idle
     сообщается, если метаданные управляются внешне.

     clear
         Нет устройств, нет размера, нет уровня

         Запись эквивалентна ioctl STOP_ARRAY

     inactive
         Может иметь некоторые настройки, но массив не активен;
         весь ввод-вывод приводит к ошибке

         При записи не разрушает массив, а просто останавливает его

     suspended (пока не поддерживается)
         Все запросы ввода-вывода будут блокироваться. Массив можно перенастроить.

         Запись этого, если принята, будет блокировать до тех пор, пока массив не успокоится

     readonly
         resync не может происходить.  суперблоки не записываются.

         Запросы на запись завершаются с ошибкой

     read-auto
         как readonly, но ведёт себя как ``clean`` при запросе на запись.

     clean
         нет ожидающих записей, но в остальном активен.

         При записи в неактивный массив запускается без resync

         Если поступает запрос на запись, то
         если метаданные известны, помечается ``dirty`` и переключается в ``active``.
         если не известны, блокируется и переключается в write-pending

         Если записано в активный массив, у которого есть ожидающие записи, то завершается с ошибкой.
     active
         полностью активен: могут происходить ввод-вывод и resync.
         При записи в неактивный массив запускается с resync

     write-pending
         clean, но записи заблокированы в ожидании записи ``active``.

     active-idle
         как active, но какое-то время не было записей (safe_mode_delay).

  consistency_policy
     Это указывает, как массив поддерживает согласованность в случае непредвиденного
     выключения. Может быть:

     none
       Массив не имеет информации об избыточности, например raid0, linear.

     resync
       Выполняется полная повторная синхронизация, и вся избыточность регенерируется при
       запуске массива после нечистого выключения.

     bitmap
       Resync с помощью битовой карты намерения записи (write-intent bitmap).

     journal
       Для raid4/5/6 устройство журнала используется для журналирования транзакций и их повтора
       после нечистого выключения.

     ppl
       Только для raid5, журнал частичной чётности (Partial Parity Log) используется для закрытия
       дыры записи (write hole) и устранения resync.

     Допустимыми значениями при записи в этот файл являются ``ppl`` и ``resync``,
     используемые для включения и отключения PPL.

  uuid
     Это указывает UUID массива в следующем формате:
     xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx

  bitmap_type
     [RW] При чтении этот файл отобразит текущую и доступные
     битовые карты для этого массива. Текущая активная битовая карта будет заключена
     в скобки []. Запись имени или ID битовой карты в этот файл переключит
     управление этим массивом на новую битовую карту. Обратите внимание, что запись новой
     битовой карты для созданного массива запрещена.

     none
         Нет битовой карты
     bitmap
         Внутренняя битовая карта по умолчанию
     llbitmap
         Внутренняя битовая карта без блокировок (lockless)

Если bitmap_type не равен none, то дополнительные атрибуты битовой карты bitmap/xxx или
llbitmap/xxx будут созданы после события KOBJ_CHANGE md-устройства.

Если bitmap_type равен bitmap, то md-устройство также будет содержать:

  bitmap/location
     Это указывает, где хранится битовая карта намерения записи (write-intent bitmap)
     для массива.

     Может быть одним из ``none``, ``file`` или ``[+-]N``.
     ``file`` в дальнейшем может быть расширено до ``file:/file/name``
     ``[+-]N`` означает столько-то секторов от начала метаданных.

     Это реплицируется на всех устройствах.  Для массивов с внешне
     управляемыми метаданными смещение отсчитывается от начала
     устройства.

  bitmap/chunksize
     Размер в байтах блока (chunk), который будет представлен
     одним битом.  Для RAID456 это часть отдельного
     устройства. Для RAID10 это часть массива.  Для RAID1 это
     и то и другое (что в данном случае одно и то же).

  bitmap/time_base
     Время в секундах между проверками битов в битовой карте на предмет
     очистки. В текущей реализации бит будет очищен
     спустя время от 2 до 3 раз ``time_base`` после того, как все охватываемые блоки
     заведомо синхронизированы.

  bitmap/backlog
     Когда write-mostly устройства активны в RAID1, запросы на запись
     к этим устройствам выполняются в фоновом режиме — файловая система (или
     другой пользователь устройства) не должна их ждать.
     ``backlog`` устанавливает ограничение на количество одновременных фоновых
     записей.  Если их больше указанного, новые записи будут
     синхронными.

  bitmap/metadata
     Это может быть либо ``internal``, либо ``external``.

     ``internal``
       это значение по умолчанию и означает, что метаданные для битовой карты
       хранятся в первых 256 байтах выделенного пространства и
       управляются модулем md.

     ``external``
       означает, что метаданные битовой карты управляются внешне по отношению к
       ядру (т.е. некоторой программой пространства пользователя)

  bitmap/can_clear
     Это либо ``true``, либо ``false``.  Если ``true``, то биты в
     битовой карте будут очищаться, когда считается, что соответствующие блоки
     синхронизированы.  Если ``false``, биты никогда не будут очищаться.
     Это автоматически устанавливается в ``false``, если запись происходит на
     деградировавшем массиве или если массив становится деградировавшим во время записи.
     Когда метаданные управляются внешне, это значение должно устанавливаться в true
     после того, как массив становится недеградировавшим, и этот факт был
     записан в метаданные.

Если bitmap_type равен llbitmap, то md-устройство также будет содержать:

  llbitmap/bits
     Это только для чтения, показывает состояние битов битовой карты, количество каждого
     значения.

  llbitmap/metadata
     Это только для чтения, показывает метаданные битовой карты, включая chunksize, chunkshift,
     chunks, offset и daemon_sleep.

  llbitmap/daemon_sleep
     Это для чтения-записи, время в секундах, через которое функция демона будет
     запускаться для очистки грязных битов.

  llbitmap/barrier_idle
     Это для чтения-записи, время в секундах, на которое страничный барьер будет простаивать,
     что означает очистку грязных битов на странице.

По мере добавления компонентных устройств в md-массив они появляются в каталоге ``md``
в виде новых каталогов с именами::

      dev-XXX

где ``XXX`` — это имя, которое ядро знает для устройства, например hdb1.
Каждый каталог содержит:

      block
        символическую ссылку на блочное устройство в /sys/block, например::

	     /sys/block/md0/md/dev-hdb1/block -> ../../../../block/hdb/hdb1

      super
        Файл, содержащий образ суперблока, прочитанного с этого устройства или
        записанного на него.

      state
	Файл, фиксирующий текущее состояние устройства в массиве,
	которое может быть списком значений, разделённых запятыми:

	      faulty
				устройство было исключено из активного использования из-за
				обнаруженной неисправности, либо у него есть неподтверждённые
				плохие блоки (bad blocks)

	      in_sync
				устройство является полностью синхронизированным членом массива

	      writemostly
				устройство будет подвергаться запросам на чтение
				только при отсутствии других вариантов.

				Это применимо только к raid1-массивам.

	      blocked
				устройство отказало, и отказ ещё не был
				подтверждён обработчиком метаданных.

				Записи, которые были бы записаны на это устройство, если бы
				оно не было неисправным, блокируются.

	      spare
				устройство работает, но не является полноправным членом.

				Сюда входят запасные (spare) устройства, которые находятся в процессе
				восстановления на них данных

	      write_error
				устройство когда-либо сталкивалось с ошибкой записи.

	      want_replacement
				устройство (в основном) работает, но, вероятно,
				должно быть заменено — либо из-за ошибок, либо
				по запросу пользователя.

	      replacement
				устройство является заменой для другого активного
				устройства с тем же raid_disk.


	Этот список может расширяться в будущем.

	В этот файл можно писать.

	Запись ``faulty`` симулирует отказ устройства.

	Запись ``remove`` удаляет устройство из массива.

	Запись ``writemostly`` устанавливает флаг writemostly.

	Запись ``-writemostly`` сбрасывает флаг writemostly.

	Запись ``blocked`` устанавливает флаг ``blocked``.

	Запись ``-blocked`` сбрасывает флаги ``blocked`` и позволяет записям
	завершиться и, возможно, симулирует ошибку.

	Запись ``in_sync`` устанавливает флаг in_sync.

	Запись ``write_error`` устанавливает флаг writeerrorseen.

	Запись ``-write_error`` сбрасывает флаг writeerrorseen.

	Запись ``want_replacement`` разрешена в любое время, кроме как для
	устройства-замены или запасного устройства.  Она устанавливает флаг.

	Запись ``-want_replacement`` разрешена в любое время.  Она сбрасывает
	флаг.

	Запись ``replacement`` или ``-replacement`` разрешена только до
	запуска массива.  Она устанавливает или сбрасывает флаг.


	Этот файл реагирует на select/poll. Любое изменение ``faulty``
	или ``blocked`` вызывает событие.

      errors
	Приблизительный счётчик ошибок чтения, которые были обнаружены на
	этом устройстве, но не привели к исключению устройства из
	массива (либо потому что они были исправлены, либо потому что они
	произошли, пока массив был только для чтения).  При использовании метаданных
	версии 1 это значение сохраняется между перезапусками массива.

	Это значение можно записывать при сборке массива, тем самым
	предоставляя текущий счётчик для массивов с метаданными, управляемыми
	пространством пользователя.

      slot
        Это указывает роль, которую устройство имеет в массиве.  Это будет
	либо ``none``, если устройство не активно в массиве
        (т.е. является запасным или отказало), либо целое число, меньшее
	значения ``raid_disks`` для массива, указывающее, какую позицию
	оно сейчас занимает.  Это можно установить только при сборке
	массива.  Устройство, для которого это установлено, считается работающим.

      offset
        Это указывает расположение на устройстве (в секторах от
        начала), где будут храниться данные из массива.  Любая часть
        устройства до этого смещения не трогается, если только она не
        используется для хранения метаданных (форматы 1.1 и 1.2).

      size
        Объём устройства после смещения, который можно использовать
        для хранения данных.  Обычно это будет то же, что и
	component_size.  Это можно записывать при сборке
        массива.  Если записать значение, меньшее текущего component_size,
        оно будет отклонено.

      recovery_start
        Когда устройство не ``in_sync``, это фиксирует количество
	секторов от начала устройства, которые заведомо
	корректны.  Обычно это ноль, но во время операции
	восстановления оно будет неуклонно увеличиваться, и если восстановление
	прерывается, восстановление этого значения может позволить восстановлению
	избежать повторения более ранних блоков.  С метаданными v1.x это
	значение сохраняется и восстанавливается автоматически.

	Это можно установить в любой момент, когда устройство не является активным членом
	массива — либо до того, как массив активирован, либо до того, как
	установлен ``slot``.

	Установка этого в ``none`` эквивалентна установке ``in_sync``.
	Установка любого другого значения также сбрасывает флаг ``in_sync``.

      bad_blocks
	Это указывает список всех известных плохих блоков (bad blocks) в виде
	начального адреса и длины (в секторах соответственно). Если вывод
	слишком велик, чтобы поместиться на странице, он будет усечён. Запись
	``sector length`` в этот файл добавляет новые подтверждённые (т.е.
	безопасно записанные на диск) плохие блоки.

      unacknowledged_bad_blocks
	Это указывает список известных, но ещё не сохранённых на диск плохих
	блоков в той же форме, что и ``bad_blocks``. Если вывод слишком велик,
	чтобы поместиться на странице, он будет усечён. Запись в этот файл
	добавляет плохие блоки без их подтверждения. Это в основном
	для тестирования.

      ppl_sector, ppl_size
        Расположение и размер (в секторах) пространства, используемого под журнал частичной чётности (Partial Parity Log)
        на этом устройстве.


Активное md-устройство также будет содержать запись для каждого активного устройства
в массиве.  Они называются::

    rdNN

где ``NN`` — позиция в массиве, начиная с 0.
Так, для массива из 3 дисков будут rd0, rd1, rd2.
Это символические ссылки на соответствующую запись ``dev-XXX``.
Так, например::

       cat /sys/block/md*/md/rd*/state

покажет ``in_sync`` в каждой строке.



Активные md-устройства для уровней, поддерживающих избыточность данных (1,4,5,6,10),
также имеют

   sync_action
     текстовый файл, который можно использовать для мониторинга и управления процессом
     восстановления.  Он содержит одно слово, которое может быть одним из:

       resync
			избыточность пересчитывается после нечистого
                        выключения или создания

       recover
			строится горячий резерв (hot spare) для замены
			отказавшего/отсутствующего устройства

       idle
			ничего не происходит
       check
			была запрошена и выполняется полная проверка
                        избыточности.  Она читает все блоки и проверяет
                        их. Для некоторых уровней raid также может произойти
                        исправление (repair).

       repair
			выполняется полная проверка и исправление.  Это
			аналогично ``resync``, но было запрошено
                        пользователем, и битовая карта намерения записи (write-intent bitmap)
			НЕ используется для оптимизации процесса.

      Этот файл доступен для записи, и каждая из строк, которые можно
      прочитать, имеет смысл для записи.

	``idle`` остановит активный resync/recovery и т.д.  Нет
	гарантии, что другой resync/recovery не будет автоматически
	запущен снова, хотя для этого потребуется некоторое событие, чтобы
	запустить его.

	``resync`` или ``recovery`` можно использовать для перезапуска
        соответствующей операции, если она была остановлена с помощью ``idle``.

	``check`` и ``repair`` запустят соответствующий процесс,
	при условии, что текущее состояние — ``idle``.

      Этот файл реагирует на select/poll.  Любое важное изменение значения
      запускает событие poll.  Иногда значение будет ненадолго
      ``recover``, если кажется, что восстановление необходимо, но не может быть
      достигнуто. В этом случае переход в ``recover`` не
      уведомляется, но переход из него — уведомляется.

   degraded
      Это содержит счётчик количества устройств, на которое
      массив деградировал.  Так, оптимальный массив покажет ``0``.
      Один отказавший/отсутствующий диск покажет ``1`` и т.д.

      Этот файл реагирует на select/poll, любое увеличение или уменьшение
      счётчика отсутствующих устройств запустит событие.

   mismatch_count
      При выполнении ``check`` и ``repair``, а возможно и при
      выполнении ``resync``, md будет считать количество найденных
      ошибок.  Счётчик в ``mismatch_cnt`` — это количество секторов,
      которые были перезаписаны, или (для ``check``) были бы
      перезаписаны.  Поскольку большинство уровней raid работают в единицах страниц, а не
      секторов, оно может быть больше количества фактических ошибок
      в число раз, равное количеству секторов на странице.

   bitmap_set_bits
      Если у массива есть битовая карта намерения записи (write-intent bitmap), то запись в этот
      атрибут может устанавливать биты в битовой карте, указывая, что resync
      должен будет проверить соответствующие блоки. Можно записывать либо отдельные
      числа, либо пары start-end.  Несколько чисел
      можно разделять пробелом.

      Обратите внимание, что числа — это номера ``bit`` (битов), а не номера ``block`` (блоков).
      Их следует масштабировать на bitmap_chunksize.

   sync_speed_min, sync_speed_max
     Они аналогичны ``/proc/sys/dev/raid/speed_limit_{min,max}``,
     однако применяются только к конкретному массиву.

     Если в них не было записано значение или если записано слово ``system``,
     то используется общесистемное значение.  Если записано значение
     в кибибайтах в секунду, то используется оно.

     При чтении файлов они показывают текущее активное значение,
     за которым следует ``(local)`` или ``(system)`` в зависимости от того, является ли это
     локально установленным или общесистемным значением.

   sync_completed
     Это показывает количество секторов, которые были завершены из
     того, что является текущим sync_action, за которым следует общее количество
     секторов, которые могло бы потребоваться обработать.  Эти два
     числа разделены символом ``/``, тем самым фактически показывая одно
     значение — долю завершённого процесса.

     ``select`` на этом атрибуте вернётся, когда resync завершится,
     когда он достигнет текущего sync_max (см. ниже) и, возможно, в
     другие моменты.

   sync_speed
     Это показывает текущую фактическую скорость, в К/сек, текущего
     sync_action.  Она усредняется за последние 30 секунд.

   suspend_lo, suspend_hi
     Эти два значения, заданные как количество секторов, указывают диапазон
     в пределах массива, где ввод-вывод будет блокироваться.  В настоящее время это
     поддерживается только для raid4/5/6.

   sync_min, sync_max
     Эти два значения, заданные как количество секторов, указывают диапазон
     в пределах массива, в котором будут работать ``check``/``repair``. Должны быть
     кратны chunk_size. Когда достигается ``sync_max``, процесс
     приостановится, а не завершится.
     Можно использовать ``select`` или ``poll`` на ``sync_completed``, чтобы дождаться,
     пока это число достигнет sync_max.  Затем можно либо увеличить
     ``sync_max``, либо записать ``idle`` в ``sync_action``.

     Значение ``max`` для ``sync_max`` фактически отключает ограничение.
     Когда resync активен, значение можно только увеличивать,
     но никогда не уменьшать.
     Значение ``0`` является минимумом для ``sync_min``.



Каждое активное md-устройство может также иметь атрибуты, специфичные для
модуля personality, который им управляет.
Они специфичны для реализации модуля и могут существенно
измениться, если изменится реализация.

В настоящее время к ним относятся:

  stripe_cache_size  (в настоящее время только raid5)
      количество записей в кэше полос (stripe cache).  Доступно для записи, но
      есть верхний и нижний пределы (32768, 17).  По умолчанию 256.

  strip_cache_active (в настоящее время только raid5)
      количество активных записей в кэше полос (stripe cache)

  preread_bypass_threshold (в настоящее время только raid5)
      количество раз, когда полоса (stripe), требующая предварительного чтения (preread), будет обойдена
      полосой, которая не требует предварительного чтения.  В целях справедливости по умолчанию
      равно 1.  Установка этого в 0 отключает учёт обходов и
      требует, чтобы полосы с предварительным чтением ждали, пока все полосы записи
      полной ширины не будут завершены.  Допустимые значения от 0 до stripe_cache_size.

  journal_mode (в настоящее время только raid5)
      Режим кэша для raid5. raid5 может включать дополнительный диск для
      кэширования. Режим может быть "write-through" или "write-back".
      По умолчанию "write-through".

  ppl_write_hint
      NVMe stream ID, который будет установлен для каждого запроса на запись PPL.
