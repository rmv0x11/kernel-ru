======================================================================
Syscall User Dispatch (диспетчеризация syscall на уровне пользователя)
======================================================================

Предпосылки
-----------

Слоям совместимости, таким как Wine, нужен способ эффективно эмулировать
системные вызовы лишь для части процесса — той части, что содержит
несовместимый код, — сохраняя при этом возможность выполнять нативные syscall
без существенных потерь производительности в нативной части процесса. Seccomp
не справляется с этой задачей, поскольку он лишь ограниченно поддерживает
эффективную фильтрацию syscall на основе областей памяти и не поддерживает
удаление фильтров. Поэтому необходим новый механизм.

Syscall User Dispatch возвращает фильтрацию адреса диспетчера syscall обратно
в пространство пользователя. Приложение управляет переключателем (flip switch),
указывающим текущую личину (personality) процесса. Многоличинное приложение
может затем переключать его без обращения к ядру, при пересечении границ API
слоя совместимости, чтобы включать/выключать перенаправление syscall и выполнять
syscall напрямую (при выключенном) либо отправлять их на эмуляцию в пространстве
пользователя через SIGSYS.

Цель этого решения — обеспечить очень быстрое пересечение границы слоя
совместимости, что достигается за счёт того, что для смены личины не требуется
выполнять syscall при каждом проходе через слой совместимости. Вместо этого
область памяти пространства пользователя, открытая ядру, указывает текущую
личину, и приложение просто изменяет эту переменную для настройки механизма.

На большинстве архитектур, например x86, обработка сигналов сопряжена с
относительно высокими затратами, однако по крайней мере для Wine syscall,
выдаваемые нативным кодом Windows, в настоящее время не известны как проблема
производительности, поскольку они довольно редки, по крайней мере для современных
игровых приложений.

Поскольку этот механизм предназначен для перехвата syscall, выдаваемых
ненативными приложениями, он должен работать с syscall, чей ABI вызова
совершенно неожиданен для Linux. Поэтому Syscall User Dispatch не опирается ни
на какую часть ABI syscall при выполнении фильтрации. Он использует только адрес
диспетчера syscall и ключ из пространства пользователя.

Так как ABI этих перехватываемых syscall неизвестен Linux, такие syscall не
поддаются инструментированию через ptrace или точки трассировки syscall.

Интерфейс
---------

Поток может настроить этот механизм на поддерживающих его ядрах, выполнив
следующий prctl:

  prctl(PR_SET_SYSCALL_USER_DISPATCH, <op>, <offset>, <length>, [selector])

<op> — это либо PR_SYS_DISPATCH_EXCLUSIVE_ON/PR_SYS_DISPATCH_INCLUSIVE_ON,
либо PR_SYS_DISPATCH_OFF, для глобального включения и выключения механизма для
данного потока. При использовании PR_SYS_DISPATCH_OFF остальные поля должны быть нулевыми.

Для PR_SYS_DISPATCH_EXCLUSIVE_ON интервал [<offset>, <offset>+<length>) задаёт
область памяти, из которой syscall всегда выполняются напрямую, независимо от
селектора пространства пользователя. Это обеспечивает быстрый путь для библиотеки
C, которая содержит наиболее распространённые диспетчеры syscall в приложениях с
нативным кодом, а также даёт обработчику сигнала способ вернуться без запуска
вложенного SIGSYS при (rt\_)sigreturn. Пользователям этого интерфейса следует
убедиться, что в эту область включён хотя бы код трамплина сигналов (signal
trampoline). Кроме того, для syscall, реализующих код трамплина в vDSO, этот
трамплин никогда не перехватывается.

Для PR_SYS_DISPATCH_INCLUSIVE_ON интервал [<offset>, <offset>+<length>) задаёт
область памяти, из которой syscall диспетчеризуются на основе селектора
пространства пользователя. Syscall за пределами этого диапазона всегда
выполняются напрямую.

[selector] — это указатель на область размера char в памяти процесса, которая
даёт быстрый способ включать и выключать перенаправление syscall в пределах всего
потока, без необходимости напрямую обращаться к ядру. selector может быть
установлен в SYSCALL_DISPATCH_FILTER_ALLOW или SYSCALL_DISPATCH_FILTER_BLOCK.
Любое другое значение должно завершить программу через SIGSYS.

Дополнительно конфигурацию syscall user dispatch задачи можно прочитать и
изменить через ptrace-запросы PTRACE_(GET|SET)_SYSCALL_USER_DISPATCH_CONFIG.
Это полезно для программ контрольных точек/восстановления (checkpoint/restart).

Замечания по безопасности
-------------------------

Syscall User Dispatch предоставляет слоям совместимости функциональность для
быстрого перехвата системных вызовов, выдаваемых ненативной частью приложения,
не затрагивая при этом нативные для Linux области процесса. Это не механизм
для песочницы (sandboxing) системных вызовов, и его не следует рассматривать как
механизм безопасности, поскольку вредоносному приложению тривиально обойти этот
механизм, перейдя в разрешённую область диспетчера перед выполнением syscall,
либо обнаружить адрес и изменить значение селектора. Если сценарий использования
требует какой-либо изоляции в песочнице с целью безопасности, вместо этого
следует использовать Seccomp.

Любой fork или exec существующего процесса сбрасывает механизм в
PR_SYS_DISPATCH_OFF.
