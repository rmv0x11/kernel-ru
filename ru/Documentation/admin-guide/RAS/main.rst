==================================================
Reliability, Availability and Serviceability (RAS)
==================================================

В этом документе описаны различные аспекты функциональности RAS, присутствующей
в ядре.

Концепции RAS
*************

Reliability, Availability and Serviceability (RAS) — это концепция, применяемая
на серверах и предназначенная для измерения их устойчивости.

Reliability (надёжность)
  — это вероятность того, что система выдаст корректные результаты.

  * Обычно измеряется как Mean Time Between Failures (MTBF, среднее время между отказами)
  * Усиливается за счёт функциональности, помогающей избегать, обнаруживать и устранять аппаратные неисправности

Availability (доступность)
  — это вероятность того, что система работоспособна в заданный момент времени

  * Обычно измеряется как процент простоя за период времени
  * Часто использует механизмы обнаружения и исправления аппаратных неисправностей
    в процессе работы;

Serviceability (или maintainability, удобство обслуживания)
  — это простота и скорость, с которой систему можно отремонтировать или
  обслужить

  * Обычно измеряется как Mean Time Between Repair (MTBR, среднее время между ремонтами)

Улучшение RAS
-------------

Чтобы сократить время простоя систем, система должна быть способна обнаруживать
аппаратные ошибки и, по возможности, исправлять их в процессе работы. Она также
должна предоставлять механизмы обнаружения деградации аппаратуры, чтобы
предупредить системного администратора о необходимости заменить компонент,
прежде чем он приведёт к потере данных или простою системы.

Среди мер мониторинга наиболее распространены следующие:

* CPU — обнаружение ошибок при выполнении инструкций и в кэшах L1/L2/L3;
* Память — добавление логики коррекции ошибок (ECC) для обнаружения и исправления ошибок;
* I/O — добавление контрольных сумм CRC для передаваемых данных;
* Накопители — RAID, журналируемые файловые системы, контрольные суммы,
  Self-Monitoring, Analysis and Reporting Technology (SMART).

Отслеживая количество случаев обнаружения ошибок, можно определить, растёт ли
вероятность аппаратных ошибок, и в таком случае провести профилактическое
обслуживание для замены деградировавшего компонента, пока эти ошибки ещё
исправимы.

Типы ошибок
-----------

Большинство механизмов, применяемых в современных системах, используют такие
технологии, как коды Хэмминга, которые позволяют исправлять ошибки, когда число
ошибок в пакете битов ниже порогового значения. Если число ошибок выше, эти
механизмы могут с высокой степенью уверенности указать, что ошибка произошла, но
не могут её исправить.

Кроме того, иногда ошибка возникает в компоненте, который не используется.
Например, в части памяти, которая в данный момент не выделена.

Это определяет некоторые категории ошибок:

* **Correctable Error (CE, исправимая ошибка)** — механизм обнаружения ошибок
  обнаружил и исправил ошибку. Такие ошибки обычно не являются фатальными, хотя
  некоторые механизмы ядра позволяют системному администратору считать их фатальными.

* **Uncorrected Error (UE, неисправленная ошибка)** — количество произошедших
  ошибок превысило порог коррекции ошибок, и система не смогла исправить их
  автоматически.

* **Fatal Error (фатальная ошибка)** — когда ошибка UE происходит в критически
  важном компоненте системы (например, часть ядра оказалась повреждена из-за UE),
  единственный надёжный способ избежать повреждения данных — это зависнуть или
  перезагрузить машину.

* **Non-fatal Error (нефатальная ошибка)** — когда ошибка UE происходит в
  неиспользуемом компоненте, например в CPU в состоянии пониженного
  энергопотребления или в неиспользуемом банке памяти, система может продолжать
  работать, при необходимости заменяя пострадавшую аппаратуру на горячий резерв,
  если он доступен.

  Кроме того, когда ошибка происходит в процессе пространства пользователя
  (userspace), такой процесс можно также завершить и позволить userspace
  перезапустить его.

Механизм обработки нефатальных ошибок обычно сложен и может требовать помощи
некоторого приложения пространства пользователя, чтобы применить политику,
желаемую системным администратором.

Идентификация неисправного аппаратного компонента
--------------------------------------------------

Просто обнаружить аппаратный дефект обычно недостаточно, так как системе
необходимо точно указать минимальную заменяемую единицу (MRU, minimal replaceable
unit), которую следует заменить, чтобы аппаратура снова стала надёжной.

Поэтому требуются не только средства журналирования ошибок, но и механизмы,
которые транслируют сообщение об ошибке в шелкографию или метку компонента для
MRU.

Как правило, это очень сложно для памяти, так как современные CPU чередуют память
из разных модулей памяти, чтобы обеспечить более высокую производительность. DMI
BIOS обычно содержит список меток модулей памяти, который можно получить с помощью
утилиты ``dmidecode``. Например, на настольной машине она показывает::

	Memory Device
		Total Width: 64 bits
		Data Width: 64 bits
		Size: 16384 MB
		Form Factor: SODIMM
		Set: None
		Locator: ChannelA-DIMM0
		Bank Locator: BANK 0
		Type: DDR4
		Type Detail: Synchronous
		Speed: 2133 MHz
		Rank: 2
		Configured Clock Speed: 2133 MHz

В приведённом выше примере модуль памяти DDR4 SO-DIMM находится в памяти системы,
помеченной как «BANK 0», что задаётся полем *bank locator*. Обратите внимание,
что в такой системе *total width* равно *data width*. Это означает, что данный
модуль памяти не имеет механизмов обнаружения/исправления ошибок.

К сожалению, не все системы используют одно и то же поле для указания банка
памяти. В этом примере, со старого сервера, ``dmidecode`` показывает::

	Memory Device
		Array Handle: 0x1000
		Error Information Handle: Not Provided
		Total Width: 72 bits
		Data Width: 64 bits
		Size: 8192 MB
		Form Factor: DIMM
		Set: 1
		Locator: DIMM_A1
		Bank Locator: Not Specified
		Type: DDR3
		Type Detail: Synchronous Registered (Buffered)
		Speed: 1600 MHz
		Rank: 2
		Configured Clock Speed: 1600 MHz

Здесь модуль памяти DDR3 RDIMM находится в памяти системы, помеченной как
«DIMM_A1», что задаётся полем *locator*. Обратите внимание, что этот модуль
памяти имеет 64 бита *data width* и 72 бита *total width*. Таким образом, у него
есть 8 дополнительных битов, которые используются механизмами обнаружения и
исправления ошибок. Такой вид памяти называется памятью с кодом коррекции ошибок
(Error-correcting code memory, ECC memory).

Чтобы ещё больше усложнить ситуацию, нередко системы с разными метками на их
системной плате используют ровно один и тот же BIOS, а это означает, что метки,
предоставляемые BIOS, не будут совпадать с реальными.

Память ECC
----------

Как упоминалось в предыдущем разделе, память ECC имеет дополнительные биты,
используемые для коррекции ошибок. В приведённом выше примере модуль памяти имеет
64 бита *data width* и 72 бита *total width*.  Дополнительные 8 битов, которые
используются механизмами обнаружения и исправления ошибок, называются *syndrome*
(синдром)\ [#f1]_\ [#f2]_.

Итак, когда CPU запрашивает у контроллера памяти запись слова с шириной *data
width*, контроллер памяти вычисляет *syndrome* в реальном времени, используя код
Хэмминга или какой-либо другой код коррекции ошибок, например SECDED+, формируя
код размером *total width*. Этот код затем записывается в модули памяти.

При чтении код шириной *total width* преобразуется обратно с использованием того
же ECC-кода, что и при записи, формируя слово с шириной *data width* и
*syndrome*. Слово с шириной *data width* отправляется в CPU, даже когда возникают
ошибки.

Контроллер памяти также проверяет *syndrome*, чтобы определить, была ли ошибка и
смог ли ECC-код её исправить. Если ошибка была исправлена, произошла Corrected
Error (CE). Если нет — произошла Uncorrected Error (UE).

Информация об ошибках CE/UE хранится в некоторых специальных регистрах контроллера
памяти и может быть получена путём чтения этих регистров — либо BIOS, либо
некоторыми специальными CPU, либо драйвером Linux EDAC. На 64-битных CPU x86 такие
ошибки также можно получить через Machine Check Architecture (MCA)\ [#f3]_.

.. [#f1] Обратите внимание, что некоторые контроллеры памяти допускают работу в
  режиме под названием «Lock-Step», в котором два модуля памяти группируются
  вместе, выполняя 128-битные чтения/записи. Это даёт 16 битов для коррекции
  ошибок, что значительно улучшает механизм коррекции ошибок, ценой того, что при
  возникновении ошибки невозможно узнать, какой модуль памяти виноват. Поэтому
  приходится винить оба модуля памяти.

.. [#f2] Некоторые контроллеры памяти также допускают использование памяти в
  зеркальном режиме (mirror mode). В этом режиме одни и те же данные записываются
  в два модуля памяти. При чтении система проверяет оба модуля памяти, чтобы
  убедиться, что оба предоставляют идентичные данные. При такой конфигурации, при
  возникновении ошибки, невозможно узнать, какой модуль памяти виноват. Поэтому
  приходится винить оба модуля памяти (или 4 модуля памяти, если система также
  работает в режиме Lock-step).

.. [#f3] Подробнее о Machine Check Architecture (MCA) см.
  Documentation/arch/x86/x86_64/machinecheck.rst в дереве ядра.

EDAC - Error Detection And Correction
*************************************

.. note::

   «bluesmoke» было названием этой подсистемы драйверов устройств, когда она была
   «out-of-tree» (вне дерева) и сопровождалась по адресу
   http://bluesmoke.sourceforge.net. Сейчас этот сайт большей частью устарел и
   может использоваться только в исторических целях.

   Когда подсистема была впервые отправлена в апстрим, в ядре 2.6.16, она была
   переименована в ``EDAC``.

Назначение
----------

Цель модуля ядра ``edac`` — обнаруживать и сообщать об аппаратных ошибках,
которые происходят в компьютерной системе, работающей под управлением linux.

Память
------

Memory Correctable Errors (CE) и Uncorrectable Errors (UE) — это основные
собираемые ошибки. Эти типы ошибок собираются устройством ``edac_mc``.

Обнаружение событий CE, последующий сбор этих событий и сообщение о них **могут**,
но не обязательно должны быть предиктором будущих событий UE. При наличии только
событий CE система может и будет продолжать работать, так как данные ещё не
повреждены.

Тем не менее профилактическое обслуживание и упреждающая замена модулей памяти, у
которых проявляются CE, могут снизить вероятность ужасных событий UE и паник
системы.

Другие аппаратные элементы
--------------------------

Новая функциональность EDAC, класс устройства ``edac_device``, была добавлена в
версии ядра 2.6.23.

Этот новый тип устройства позволяет собирать состояния немемористичных (не
относящихся к памяти) аппаратных детекторов ECC и представлять их пространству
пользователя через интерфейс sysfs.

Некоторые архитектуры имеют детекторы ECC для кэшей L1, L2 и L3, а также для
движков DMA, fabric-коммутаторов, коммутаторов основного тракта данных,
межсоединений и различных других аппаратных трактов данных. Если аппаратура
сообщает об этом, то, вероятно, можно сконструировать устройство edac_device для
сбора этих данных и их представления пространству пользователя.


Сканирование шины PCI
---------------------

Кроме того, устройства PCI сканируются на наличие ошибок PCI Bus Parity и SERR,
чтобы определить, происходят ли ошибки во время передачи данных.

Наличие ошибок PCI Parity следует воспринимать с долей скепсиса. Существует
несколько дополнительных адаптеров, которые **не** следуют спецификации PCI в
отношении генерации и сообщения чётности. Спецификация гласит, что производитель
должен привязать биты состояния чётности к 0, если он не намерен генерировать
чётность.  Некоторые производители этого не делают, и поэтому бит чётности может
«плавать», давая ложные срабатывания.

В sysfs есть атрибут устройства PCI, который проверяется кодом сканирования EDAC
PCI. Если этот атрибут установлен, сканирование чётности/ошибок PCI для данного
устройства пропускается. Этот атрибут::

	broken_parity_status

и находится в каталогах ``/sys/devices/pci<XXX>/0000:XX:YY.Z`` для устройств PCI.


Версионирование
---------------

EDAC состоит из «ядрового» модуля (``edac_core.ko``) и нескольких модулей
драйверов контроллеров памяти (Memory Controller, MC). На данной системе
загружается CORE и будет загружен один драйвер MC. И CORE, и драйвер MC (или
драйвер ``edac_device``) имеют индивидуальные версии, которые отражают текущий
уровень релиза соответствующих модулей.

Таким образом, чтобы «сообщить» о том, на какой версии работает система, нужно
сообщить версии как CORE, так и драйвера MC.


Загрузка
--------

Если ``edac`` был статически слинкован с ядром, то никакая загрузка не нужна.
Если ``edac`` был собран как модули, то просто выполните modprobe для тех частей
``edac``, которые вам нужны. Вы должны иметь возможность выполнить modprobe для
аппаратно-специфичных модулей, и зависимости загрузят необходимые ядровые модули.

Пример::

	$ modprobe amd76x_edac

загружает как модуль контроллера памяти ``amd76x_edac.ko``, так и ядровый модуль
``edac_mc.ko``.


Интерфейс sysfs
---------------

EDAC предоставляет интерфейс ``sysfs`` для целей управления и отчётности. Он
располагается в каталоге /sys/devices/system/edac.

В этом каталоге в настоящее время находятся 2 компонента:

	======= ==============================
	mc	система контроллеров памяти
	pci	система управления и состояния PCI
	======= ==============================



Модель контроллера памяти (mc)
------------------------------

Каждое устройство ``mc`` управляет набором модулей памяти [#f4]_. Эти модули
размещены в таблице Chip-Select Row (``csrowX``) и Channel (``chX``). Может быть
несколько csrow и несколько каналов.

.. [#f4] В наши дни термин DIMM (Dual In-line Memory Module) широко используется
  для обозначения модуля памяти, хотя существуют и другие варианты компоновки
  памяти, такие как SO-DIMM, SIMM и т. д. Спецификация UEFI (версия 2.7)
  определяет модуль памяти в секции Common Platform Error Record (CPER) как
  SMBIOS Memory Device (Type 17). На протяжении этого документа и внутри
  подсистемы EDAC термин «dimm» используется для всех модулей памяти, даже когда
  они используют другой вид компоновки.

Контроллеры памяти допускают несколько csrow, при этом типичным значением является
8 csrow. Однако фактическое число csrow зависит от компоновки конкретной
материнской платы, контроллера памяти и характеристик модулей памяти.

Двойные каналы позволяют передавать данные двойной длины (например, 128 бит на
64-битных системах) в/из CPU из/в память. Некоторые более новые наборы микросхем
допускают более 2 каналов, как, например, контроллеры памяти Fully Buffered DIMMs
(FB-DIMMs). В следующем примере предполагается 2 канала:

	+------------+-----------------------+
	| CS Rows    |       Channels        |
	+------------+-----------+-----------+
	|            |  ``ch0``  |  ``ch1``  |
	+============+===========+===========+
	|            |**DIMM_A0**|**DIMM_B0**|
	+------------+-----------+-----------+
	| ``csrow0`` |   rank0   |   rank0   |
	+------------+-----------+-----------+
	| ``csrow1`` |   rank1   |   rank1   |
	+------------+-----------+-----------+
	|            |**DIMM_A1**|**DIMM_B1**|
	+------------+-----------+-----------+
	| ``csrow2`` |    rank0  |  rank0    |
	+------------+-----------+-----------+
	| ``csrow3`` |    rank1  |  rank1    |
	+------------+-----------+-----------+

В приведённом выше примере на материнской плате есть 4 физических слота для
модулей памяти DIMM:

	+---------+---------+
	| DIMM_A0 | DIMM_B0 |
	+---------+---------+
	| DIMM_A1 | DIMM_B1 |
	+---------+---------+

Метки для этих слотов обычно наносятся шелкографией на материнскую плату. Слоты с
меткой ``A`` в этом примере являются каналом 0. Слоты с меткой ``B`` являются
каналом 1. Обратите внимание, что на физическом DIMM возможны два csrow. Этим
csrow назначается их принадлежность к csrow на основе слота, в который установлен
модуль памяти DIMM. Таким образом, когда в каждый канал установлен 1 DIMM, csrow
пересекают оба DIMM.

Модули памяти DIMM бывают одно- или двухранговыми («single» или «dual» ranked).
Ранг — это заполненный csrow. В приведённом выше примере 2 двухранговых DIMM
размещены одинаково. Таким образом, заполнены и csrow0, и csrow1. С другой
стороны, когда 2 одноранговых DIMM установлены в слоты DIMM_A0 и DIMM_B0, тогда у
них будет только один csrow (csrow0), а csrow1 будет пустым. Этот шаблон
повторяется для csrow2 и csrow3. Также обратите внимание, что некоторые
контроллеры памяти не имеют никакой логики для идентификации модуля памяти, см.
каталоги ``rankX`` ниже.

Представление вышеописанного отражено в дереве каталогов интерфейса sysfs EDAC.
Начиная с каталога ``/sys/devices/system/edac/mc``, каждый контроллер памяти будет
представлен своим собственным каталогом ``mcX``, где ``X`` — это индекс MC::

	..../edac/mc/
		   |
		   |->mc0
		   |->mc1
		   |->mc2
		   ....

Внутри каждого каталога ``mcX`` находятся несколько файлов управления и атрибутов
EDAC.

Каталоги ``mcX``
----------------

В каталогах ``mcX`` находятся файлы управления и атрибутов EDAC для этого
экземпляра ``X`` контроллеров памяти.

Описание sysfs API см.:

	Documentation/ABI/testing/sysfs-devices-edac


Каталоги ``dimmX`` или ``rankX``
--------------------------------

Рекомендуемый способ использования подсистемы EDAC — смотреть на информацию,
предоставляемую каталогами ``dimmX`` или ``rankX`` [#f5]_.

	│   │   │   ├── dimm_location
	│   │   │   ├── dimm_mem_type
	│   │   │   ├── dimm_ue_count
	│   │   │   ├── size
	│   │   │   └── uevent
	│   │   ├── max_location
	│   │   ├── mc_name
	│   │   ├── reset_counters
	│   │   ├── seconds_since_reset
	│   │   ├── size_mb
	│   │   ├── ue_count
	│   │   ├── ue_noinfo_count
	│   │   └── uevent
	│   ├── mc1
	│   │   ├── ce_count
	│   │   ├── ce_noinfo_count
	│   │   ├── dimm0
	│   │   │   ├── dimm_ce_count
	│   │   │   ├── dimm_dev_type
	│   │   │   ├── dimm_edac_mode
	│   │   │   ├── dimm_label
	│   │   │   ├── dimm_location
	│   │   │   ├── dimm_mem_type
	│   │   │   ├── dimm_ue_count
	│   │   │   ├── size
	│   │   │   └── uevent
	│   │   ├── max_location
	│   │   ├── mc_name
	│   │   ├── reset_counters
	│   │   ├── seconds_since_reset
	│   │   ├── size_mb
	│   │   ├── ue_count
	│   │   ├── ue_noinfo_count
	│   │   └── uevent
	│   └── uevent
	└── uevent

В каталогах ``dimmX`` находятся файлы управления и атрибутов EDAC для этого
модуля памяти ``X``:

- ``size`` - файл атрибута общего объёма памяти, управляемой этим csrow

	Этот файл атрибута отображает, в количестве мегабайт, память, которую
	содержит этот csrow.

- ``dimm_ue_count`` - файл атрибута счётчика Uncorrectable Errors

	Этот файл атрибута отображает общее количество неисправимых ошибок,
	произошедших на этом DIMM. Если установлен panic_on_ue, этот счётчик не
	будет иметь возможности увеличиться, так как EDAC вызовет панику системы.

- ``dimm_ce_count`` - файл атрибута счётчика Correctable Errors

	Этот файл атрибута отображает общее количество исправимых ошибок,
	произошедших на этом DIMM. Этот счётчик очень важно отслеживать. CE дают
	раннее указание на то, что DIMM начинает выходить из строя. Это поле
	счётчика следует отслеживать на ненулевые значения и сообщать такую
	информацию системному администратору.

- ``dimm_dev_type``  - файл атрибута типа устройства

	Этот файл атрибута будет отображать, какой тип устройства DRAM
	используется на этом DIMM.
	Примеры:

		- x1
		- x2
		- x4
		- x8

- ``dimm_edac_mode`` - файл атрибута режима работы EDAC

	Этот файл атрибута будет отображать, какой тип обнаружения и исправления
	ошибок используется.

- ``dimm_label`` - управляющий файл метки модуля памяти

	Этот управляющий файл позволяет назначить этому DIMM метку. Когда у модуля
	есть эта метка, при возникновении ошибок вывод может предоставлять метку
	DIMM в системном журнале. Это становится жизненно важным для событий паники,
	чтобы изолировать причину события UE.

	Метки DIMM должны назначаться после загрузки, с информацией, которая
	корректно идентифицирует физический слот по его шелкографической метке. Эта
	информация в настоящее время сильно зависит от материнской платы, и
	определение этой информации в настоящее время должно происходить в
	пространстве пользователя.

- ``dimm_location`` - расположение модуля памяти

	Расположение может иметь до 3 уровней и описывает, как контроллер памяти
	идентифицирует расположение модуля памяти. В зависимости от типа памяти и
	контроллера памяти это может быть:

		- *csrow* и *channel* - используется, когда контроллер памяти не
		  идентифицирует отдельный DIMM — например, в каталоге ``rankX``;
		- *branch*, *channel*, *slot* - обычно используется на контроллерах
		  памяти FB-DIMM;
		- *channel*, *slot* - используется на драйверах Nehalem и более новых
		  драйверах Intel.

- ``dimm_mem_type`` - файл атрибута типа памяти

	Этот файл атрибута будет отображать, какой тип памяти в настоящее время
	находится на этом csrow. Обычно это буферизованная или небуферизованная
	память. Примеры:

		- Registered-DDR
		- Unbuffered-DDR

.. [#f5] На некоторых системах контроллер памяти не имеет никакой логики для
  идентификации модуля памяти. На таких системах каталог называется ``rankX``.
  На современных контроллерах памяти Intel контроллер памяти идентифицирует
  модули памяти напрямую. На таких системах каталог называется ``dimmX``.

.. [#f6] Внутри отображения sysfs также есть некоторые каталоги ``power`` и
  символические ссылки ``subsystem``, которые автоматически создаются подсистемой
  sysfs. В настоящее время они не служат никакой цели.


Системное журналирование
-------------------------

Если журналирование для UE и CE включено, то системные журналы будут содержать
информацию, указывающую, что были обнаружены ошибки::

  EDAC MC0: CE page 0x283, offset 0xce0, grain 8, syndrome 0x6ec3, row 0, channel 1 "DIMM_B1": amd76x_edac
  EDAC MC0: CE page 0x1e5, offset 0xfb0, grain 8, syndrome 0xb741, row 0, channel 1 "DIMM_B1": amd76x_edac


Структура сообщения такова:

	+---------------------------------------+-------------+
	| Содержимое                            | Пример      |
	+=======================================+=============+
	| Контроллер памяти                     | MC0         |
	+---------------------------------------+-------------+
	| Тип ошибки                            | CE          |
	+---------------------------------------+-------------+
	| Страница памяти                       | 0x283       |
	+---------------------------------------+-------------+
	| Смещение в странице                   | 0xce0       |
	+---------------------------------------+-------------+
	| Байтовая гранулярность                | grain 8     |
	| или разрешение ошибки                 |             |
	+---------------------------------------+-------------+
	| Синдром ошибки                        | 0xb741      |
	+---------------------------------------+-------------+
	| Строка памяти                         | row 0       |
	+---------------------------------------+-------------+
	| Канал памяти                          | channel 1   |
	+---------------------------------------+-------------+
	| Метка DIMM, если задана ранее         | DIMM B1     |
	+---------------------------------------+-------------+
	| И затем необязательное, специфичное   |             |
	| для драйвера сообщение, которое может |             |
	| содержать дополнительную информацию.  |             |
	+---------------------------------------+-------------+

Как UE, так и CE без информации будут содержать всё, кроме контроллера памяти,
типа ошибки, уведомления «no info» и затем необязательного, специфичного для
драйвера сообщения об ошибке.


Обнаружение чётности шины PCI
-----------------------------

На устройствах с заголовком Header Type 00 для любой ошибки чётности
рассматривается первичный статус (primary status) независимо от того, включена ли
чётность на устройстве. (Спецификация указывает, что в некоторых случаях чётность
генерируется). На мостах с заголовком Header Type 01 также рассматривается регистр
вторичного статуса (secondary status), чтобы определить, произошла ли чётность на
шине по другую сторону моста.


Конфигурация sysfs
------------------

Под ``/sys/devices/system/edac/pci`` находятся файлы управления и атрибутов
следующим образом:


- ``check_pci_parity`` - управляющий файл включения/отключения проверки PCI Parity

	Этот управляющий файл включает или отключает операцию сканирования PCI Bus
	Parity. Запись 1 в этот файл включает сканирование. Запись 0 в этот файл
	отключает сканирование.

	Включить::

		echo "1" >/sys/devices/system/edac/pci/check_pci_parity

	Отключить::

		echo "0" >/sys/devices/system/edac/pci/check_pci_parity


- ``pci_parity_count`` - счётчик чётности

	Этот файл атрибута будет отображать количество ошибок чётности, которые
	были обнаружены.


Параметры модуля
----------------

- ``edac_mc_panic_on_ue`` - управляющий файл паники по UE

	Неисправимая ошибка вызовет панику машины.  Обычно это желательно.  Плохая
	идея продолжать работу при возникновении неисправимой ошибки — неопределённо,
	что именно осталось неисправленным, и контекст операционной системы может
	быть настолько искажён, что продолжение приведёт к дальнейшему повреждению.
	Если в ядре сконфигурирован MCE, то EDAC никогда не заметит UE.

	LOAD TIME::

		module/kernel parameter: edac_mc_panic_on_ue=[0|1]

	RUN TIME::

		echo "1" > /sys/module/edac_core/parameters/edac_mc_panic_on_ue


- ``edac_mc_log_ue`` - управляющий файл журналирования UE


	Генерировать сообщения ядра, описывающие неисправимые ошибки.  Об этих
	ошибках сообщается через систему системного журнала сообщений.  Статистика UE
	будет накапливаться, даже когда журналирование UE отключено.

	LOAD TIME::

		module/kernel parameter: edac_mc_log_ue=[0|1]

	RUN TIME::

		echo "1" > /sys/module/edac_core/parameters/edac_mc_log_ue


- ``edac_mc_log_ce`` - управляющий файл журналирования CE


	Генерировать сообщения ядра, описывающие исправимые ошибки.  Об этих ошибках
	сообщается через систему системного журнала сообщений.  Статистика CE будет
	накапливаться, даже когда журналирование CE отключено.

	LOAD TIME::

		module/kernel parameter: edac_mc_log_ce=[0|1]

	RUN TIME::

		echo "1" > /sys/module/edac_core/parameters/edac_mc_log_ce


- ``edac_mc_poll_msec`` - управляющий файл периода опроса


	Период времени, в миллисекундах, для опроса информации об ошибках.  Слишком
	малое значение расходует ресурсы впустую.  Слишком большое значение может
	задержать необходимую обработку ошибок и может потерять ценную информацию для
	локализации ошибки.  1000 миллисекунд (раз в секунду) — текущее значение по
	умолчанию. Системы, которым требуется вся пропускная способность, какую они
	могут получить, могут увеличить его.

	LOAD TIME::

		module/kernel parameter: edac_mc_poll_msec=[0|1]

	RUN TIME::

		echo "1000" > /sys/module/edac_core/parameters/edac_mc_poll_msec


- ``panic_on_pci_parity`` - паника по ошибке PCI PARITY


	Этот управляющий файл включает или отключает панику при обнаружении ошибки
	чётности.


	module/kernel parameter::

			edac_panic_on_pci_pe=[0|1]

	Включить::

		echo "1" > /sys/module/edac_core/parameters/edac_panic_on_pci_pe

	Отключить::

		echo "0" > /sys/module/edac_core/parameters/edac_panic_on_pci_pe



Тип устройства EDAC
-------------------

В заголовочном файле edac_pci.h находится серия структур edac_device и API для
EDAC_DEVICE.

Доступ из пространства пользователя к edac_device осуществляется через интерфейс
sysfs.

В местоположении ``/sys/devices/system/edac`` (sysfs) будут появляться новые
устройства edac_device.

Под указанным выше каталогом ``edac`` есть трёхуровневое дерево. Например,
устройство ``test_device_edac`` (которое можно найти на веб-сайте
http://bluesmoke.sourceforget.net) устанавливает себя как::

	/sys/devices/system/edac/test-instance

в этом каталоге находятся различные элементы управления, символическая ссылка и
один или более каталогов ``instance``.

Стандартные элементы управления по умолчанию:

	==============	=======================================================
	log_ce		булево значение для журналирования событий CE
	log_ue		булево значение для журналирования событий UE
	panic_on_ue	булево значение для ``panic`` системы при возникновении UE
			(по умолчанию выключено, можно установить в true через
			стартовый скрипт)
	poll_msec	период времени между циклами POLL для событий
	==============	=======================================================

Устройство test_device_edac добавляет как минимум один свой собственный
пользовательский элемент управления:

	==============	==================================================
	test_bits	который в текущем тестовом драйвере не делает ничего,
			кроме демонстрации того, как он установлен. Портированный
			драйвер может добавить один или более таких элементов
			управления и/или атрибутов для конкретных целей.
			Один out-of-tree драйвер использует здесь элементы
			управления, чтобы разрешить операции ERROR INJECTION
			(внедрения ошибок) в аппаратные регистры внедрения
	==============	==================================================

Символическая ссылка указывает на «struct dev», который зарегистрирован для этого
edac_device.

Экземпляры
----------

Присутствуют один или более каталогов экземпляров (instance). Для случая
``test_device_edac``:

	+----------------+
	| test-instance0 |
	+----------------+


В этом каталоге есть два атрибута-счётчика по умолчанию, которые являются итогами
счётчиков в более глубоких подкаталогах.

	==============	====================================
	ce_count	итог событий CE подкаталогов
	ue_count	итог событий UE подкаталогов
	==============	====================================

Блоки
-----

На самом нижнем уровне каталогов находится каталог ``block``. В каждом экземпляре
может быть указано 0, 1 или более блоков:

	+-------------+
	| test-block0 |
	+-------------+

В этом каталоге атрибуты по умолчанию таковы:

	==============	================================================
	ce_count	который является счётчиком событий CE для этого
			``block`` отслеживаемой аппаратуры
	ue_count	который является счётчиком событий UE для этого
			``block`` отслеживаемой аппаратуры
	==============	================================================


Устройство ``test_device_edac`` добавляет 4 атрибута и 1 элемент управления:

	================== ====================================================
	test-block-bits-0	за каждый цикл POLL этот счётчик
				инкрементируется
	test-block-bits-1	каждые 10 циклов этот счётчик увеличивается один раз,
				а test-block-bits-0 устанавливается в 0
	test-block-bits-2	каждые 100 циклов этот счётчик увеличивается один раз,
				а test-block-bits-1 устанавливается в 0
	test-block-bits-3	каждые 1000 циклов этот счётчик увеличивается один раз,
				а test-block-bits-2 устанавливается в 0
	================== ====================================================


	================== ====================================================
	reset-counters		запись ЧЕГО УГОДНО в этот элемент управления
				сбросит все вышеперечисленные счётчики.
	================== ====================================================


Использование драйвера ``test_device_edac`` должно позволить любым другим
создавать свои собственные уникальные драйверы для своих аппаратных систем.

Образцовый драйвер ``test_device_edac`` находится на сайте проекта
http://bluesmoke.sourceforge.net для EDAC.


Использование EDAC API на Nehalem и более новых CPU Intel
---------------------------------------------------------

На старых архитектурах Intel контроллер памяти был частью набора микросхем North
Bridge. Nehalem, Sandy Bridge, Ivy Bridge, Haswell, Sky Lake и более новые
архитектуры Intel интегрировали улучшенную версию контроллера памяти (MC) внутрь
CPU.

В этой главе будут рассмотрены отличия улучшенных контроллеров памяти, имеющихся на
более новых CPU Intel, таких как драйверы ``i7core_edac``, ``sb_edac`` и
``sbx_edac``.

.. note::

   Семейства процессоров Xeon E7 используют отдельную микросхему для контроллера
   памяти, называемую Intel Scalable Memory Buffer. Этот раздел не применим к
   таким семействам.

1) Существует один контроллер памяти на каждый Quick Patch Interconnect
   (QPI). В драйвере термин «socket» означает один QPI. Это связано с
   физическим сокетом CPU.

   Каждый MC имеет 3 физических канала чтения, 3 физических канала записи и
   3 логических канала. Драйвер в настоящее время видит это просто как 3 канала.
   Каждый канал может иметь до 3 DIMM.

   Минимальная известная единица — это DIMM. Информации о csrow нет.
   Так как EDAC API отображает минимальную единицу как csrow, драйвер
   последовательно отображает channel/DIMM в разные csrow.

   Например, предположим следующую компоновку::

	Ch0 phy rd0, wr0 (0x063f4031): 2 ranks, UDIMMs
	  dimm 0 1024 Mb offset: 0, bank: 8, rank: 1, row: 0x4000, col: 0x400
	  dimm 1 1024 Mb offset: 4, bank: 8, rank: 1, row: 0x4000, col: 0x400
        Ch1 phy rd1, wr1 (0x063f4031): 2 ranks, UDIMMs
	  dimm 0 1024 Mb offset: 0, bank: 8, rank: 1, row: 0x4000, col: 0x400
	Ch2 phy rd3, wr3 (0x063f4031): 2 ranks, UDIMMs
	  dimm 0 1024 Mb offset: 0, bank: 8, rank: 1, row: 0x4000, col: 0x400

   Драйвер отобразит это как::

	csrow0: channel 0, dimm0
	csrow1: channel 0, dimm1
	csrow2: channel 1, dimm0
	csrow3: channel 2, dimm0

   экспортирует один DIMM на csrow.

   Каждый QPI экспортируется как отдельный контроллер памяти.

2) MC обладает способностью внедрять ошибки для тестирования драйверов. Драйверы
   реализуют эту функциональность через некоторые узлы внедрения ошибок:

   Для внедрения ошибки памяти есть несколько узлов sysfs под
   ``/sys/devices/system/edac/mc/mc?/``:

   - ``inject_addrmatch/*``:
      Управляет регистром маски внедрения ошибок. Можно задать несколько
      характеристик адреса для сопоставления с кодом ошибки::

         dimm = пострадавший dimm. Номера задаются относительно канала;
         rank = ранг памяти;
         channel = канал, который будет генерировать ошибку;
         bank = пострадавший банк;
         page = адрес страницы;
         column (или col) = столбец адреса.

      каждое из приведённых выше значений может быть установлено в «any» для
      сопоставления с любым допустимым значением.

      При инициализации драйвера все значения устанавливаются в any.

      Например, чтобы сгенерировать ошибку на rank 1 dimm 2, для любого канала,
      любого банка, любой страницы, любого столбца::

		echo 2 >/sys/devices/system/edac/mc/mc0/inject_addrmatch/dimm
		echo 1 >/sys/devices/system/edac/mc/mc0/inject_addrmatch/rank

	Чтобы вернуться к поведению по умолчанию сопоставления с любым, можно
	сделать::

		echo any >/sys/devices/system/edac/mc/mc0/inject_addrmatch/dimm
		echo any >/sys/devices/system/edac/mc/mc0/inject_addrmatch/rank

   - ``inject_eccmask``:
          задаёт, какие биты будут иметь проблемы,

   - ``inject_section``:
       задаёт, какая секция кэша ECC получит ошибку::

		3 для обеих
		2 для самой старшей
		1 для самой младшей

   - ``inject_type``:
       задаёт тип ошибки, являясь комбинацией следующих битов::

		bit 0 - repeat
		bit 1 - ecc
		bit 2 - parity

   - ``inject_enable``:
       запускает генерацию ошибки, когда записывается что-то отличное от 0.

   Все переменные inject можно читать. Для записи требуются права root.

   В datasheet указано, что ошибка будет генерироваться только после записи по
   адресу, который соответствует inject_addrmatch. Однако, по-видимому, чтение
   также приведёт к ошибке.

   Например, следующий код сгенерирует ошибку при любом доступе на запись по
   сокету 0, на любом DIMM/адресе на канале 2::

	echo 2 >/sys/devices/system/edac/mc/mc0/inject_addrmatch/channel
	echo 2 >/sys/devices/system/edac/mc/mc0/inject_type
	echo 64 >/sys/devices/system/edac/mc/mc0/inject_eccmask
	echo 3 >/sys/devices/system/edac/mc/mc0/inject_section
	echo 1 >/sys/devices/system/edac/mc/mc0/inject_enable
	dd if=/dev/mem of=/dev/null seek=16k bs=4k count=1 >& /dev/null

   Для сокета 1 необходимо заменить «mc0» на «mc1» в приведённых выше командах.

   Сгенерированное сообщение об ошибке будет выглядеть так::

	EDAC MC0: UE row 0, channel-a= 0 channel-b= 0 labels "-": NON_FATAL (addr = 0x0075b980, socket=0, Dimm=0, Channel=2, syndrome=0x00000040, count=1, Err=8c0000400001009f:4000080482 (read error: read ECC error))

3) Счётчики регистров памяти Corrected Error

   У этих более новых MC есть некоторые регистры для подсчёта ошибок памяти.
   Драйвер использует эти регистры для сообщения о Corrected Errors на устройствах
   с Registered DIMM.

   Однако эти счётчики не работают с Unregistered DIMM. Так как набор микросхем
   предлагает некоторые счётчики, которые также работают с UDIMM (но с худшим
   уровнем гранулярности, чем у счётчиков по умолчанию), драйвер предоставляет эти
   регистры для памяти UDIMM.

   Их можно прочитать, посмотрев на содержимое ``all_channel_counts/``::

     $ for i in /sys/devices/system/edac/mc/mc0/all_channel_counts/*; do echo $i; cat $i; done
	/sys/devices/system/edac/mc/mc0/all_channel_counts/udimm0
	0
	/sys/devices/system/edac/mc/mc0/all_channel_counts/udimm1
	0
	/sys/devices/system/edac/mc/mc0/all_channel_counts/udimm2
	0

   Здесь происходит то, что ошибки на разных csrow, но с одним и тем же номером
   dimm будут инкрементировать один и тот же счётчик.
   Итак, в этом отображении памяти::

	csrow0: channel 0, dimm0
	csrow1: channel 0, dimm1
	csrow2: channel 1, dimm0
	csrow3: channel 2, dimm0

   Аппаратура инкрементирует udimm0 при ошибке на первом dimm в любом из
   csrow0, csrow2  или csrow3;

   Аппаратура инкрементирует udimm1 при ошибке на втором dimm в любом из
   csrow0, csrow2  или csrow3;

   Аппаратура инкрементирует udimm2 при ошибке на третьем dimm в любом из
   csrow0, csrow2  или csrow3;

4) Стандартные счётчики ошибок

   Стандартные счётчики ошибок генерируются, когда драйвер получает ошибку
   mcelog. Поскольку с UDIMM это подсчитывается программно, возможно, что
   некоторые ошибки могут быть потеряны. С RDIMM они отображают
   содержимое регистров
