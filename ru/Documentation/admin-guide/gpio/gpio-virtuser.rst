Виртуальный потребитель GPIO
============================

Модуль виртуального потребителя GPIO (Virtual GPIO Consumer) позволяет
пользователям создавать виртуальные устройства, которые запрашивают линии GPIO,
а затем управлять их поведением через debugfs. Виртуальные устройства-потребители
могут создаваться из device-tree или через configfs.

Виртуальный потребитель использует GPIO API, ориентированный на драйверы, и
позволяет покрыть его автоматизированными тестами, управляемыми из пространства
пользователя. Линии GPIO запрашиваются с помощью ``gpiod_get_array()``, поэтому
поддерживается несколько линий GPIO на один идентификатор соединения.

Создание потребителей GPIO
--------------------------

Модуль gpio-consumer регистрирует подсистему configfs с именем
``'gpio-virtuser'``. Подробности о файловой системе configfs см. в документации
по configfs.

Пользователь может создавать иерархию групп и элементов configfs, а также
изменять значения доступных атрибутов. После создания потребителя эта иерархия
будет преобразована в соответствующие свойства устройства. Общая структура
такова:

**Group:** ``/config/gpio-virtuser``

Это верхний каталог дерева configfs модуля gpio-consumer.

**Group:** ``/config/gpio-consumer/example-name``

**Attribute:** ``/config/gpio-consumer/example-name/live``

**Attribute:** ``/config/gpio-consumer/example-name/dev_name``

Это каталог, представляющий устройство-потребитель GPIO.

Атрибут ``dev_name``, доступный только для чтения, отображает имя устройства в
том виде, в каком оно будет фигурировать в системе на шине платформы. Это
полезно для нахождения соответствующего каталога debugfs по пути
``/sys/kernel/debug/gpio-virtuser/$dev_name``.

Атрибут ``'live'`` позволяет инициировать фактическое создание устройства после
того, как оно полностью настроено. Принимаемые значения: ``'1'`` — включить
виртуальное устройство и ``'0'`` — выключить и демонтировать его.

Создание таблиц поиска GPIO
---------------------------

Пользователи могут создавать ряд групп configfs внутри группы устройства:

**Group:** ``/config/gpio-consumer/example-name/con_id``

Каталог ``'con_id'`` представляет один поиск GPIO, и его значение соответствует
аргументу ``'con_id'`` функции ``gpiod_get()``. Например: ``con_id`` ==
``'reset'`` соответствует свойству устройства ``reset-gpios``.

Пользователи могут назначить каждому поиску несколько линий GPIO. Каждая линия
GPIO — это подкаталог с заданным пользователем именем внутри группы ``'con_id'``.

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/key``

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/offset``

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/drive``

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/pull``

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/active_low``

**Attribute:** ``/config/gpio-consumer/example-name/con_id/0/transitory``

Это группа, описывающая одну линию GPIO в свойстве ``con_id-gpios``.

Для виртуальных потребителей, созданных через configfs, используются машинные
таблицы поиска, поэтому эту группу можно рассматривать как сопоставление между
файловой системой и полями одной записи в ``'struct gpiod_lookup'``.

Атрибут ``'key'`` представляет либо имя чипа, которому принадлежит данная линия
GPIO, либо имя линии GPIO. Это зависит от значения атрибута ``'offset'``: если
его значение >= 0, то ``'key'`` представляет метку искомого чипа, а ``'offset'``
представляет смещение линии в этом чипе. Если ``'offset'`` < 0, то ``'key'``
представляет имя линии.

Остальные атрибуты соответствуют полю ``'flags'`` структуры поиска GPIO. Первые
два принимают в качестве аргументов строковые значения:

**``'drive'``:** ``'push-pull'``, ``'open-drain'``, ``'open-source'``
**``'pull'``:** ``'pull-up'``, ``'pull-down'``, ``'pull-disabled'``, ``'as-is'``

``'active_low'`` и ``'transitory'`` — это булевы атрибуты.

Активация потребителей GPIO
---------------------------

После завершения настройки атрибуту ``'live'`` необходимо присвоить значение 1,
чтобы создать потребитель. Ему можно снова присвоить значение 0, чтобы уничтожить
виртуальное устройство. Модуль будет синхронно ждать успешного зондирования
(probe) нового симулируемого устройства, и если этого не произойдёт, запись в
``'live'`` приведёт к ошибке.

Device-tree
-----------

Виртуальные потребители GPIO могут также определяться в device-tree. Строка
compatible должна быть: ``"gpio-virtuser"`` с как минимум одним свойством,
следующим стандартизированному шаблону GPIO.

Пример кода device-tree, определяющего виртуальный потребитель GPIO:

.. code-block :: none

    gpio-virt-consumer {
        compatible = "gpio-virtuser";

        foo-gpios = <&gpio0 5 GPIO_ACTIVE_LOW>, <&gpio1 2 0>;
        bar-gpios = <&gpio0 6 0>;
    };

Управление виртуальными потребителями GPIO
------------------------------------------

После активации устройство экспортирует атрибуты debugfs для управления массивами
GPIO, а также каждой запрошенной линией GPIO по отдельности. Рассмотрим следующее
свойство устройства: ``foo-gpios = <&gpio0 0 0>, <&gpio0 4 0>;``.

Будут созданы следующие группы атрибутов debugfs:

**Group:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo/``

Это группа, которая будет содержать атрибуты для всего массива GPIO.

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo/values``

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo/values_atomic``

Оба атрибута позволяют читать и устанавливать массивы значений GPIO. Пользователь
должен передать ровно столько значений, сколько содержит массив, в виде строки из
нулей и единиц, представляющих соответственно неактивное и активное состояния
GPIO. В этом примере: ``echo 11 > values``.

Атрибут ``values_atomic`` работает так же, как ``values``, но ядро будет выполнять
функции обратного вызова драйвера GPIO в контексте прерывания.

**Group:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/``

Это группа, которая представляет одну линию GPIO, где ``$index`` — её смещение в
массиве.

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/consumer``

Позволяет устанавливать и читать метку потребителя линии GPIO.

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/debounce``

Позволяет устанавливать и читать период подавления дребезга (debounce) линии GPIO.

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/direction``

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/direction_atomic``

Эти два атрибута позволяют устанавливать направление линии GPIO. Они принимают в
качестве значений "input" и "output". Атомарный вариант выполняет функцию
обратного вызова драйвера в контексте прерывания.

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/interrupts``

Если линия запрошена в режиме ввода, запись ``1`` в этот атрибут заставит модуль
прослушивать прерывания по фронту на линии GPIO. Запись ``0`` отключает
мониторинг. Чтение этого атрибута возвращает текущее число зарегистрированных
прерываний (по обоим фронтам).

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/value``

**Attribute:** ``/sys/kernel/debug/gpio-virtuser/$dev_name/gpiod:foo:$index/value_atomic``

Оба атрибута позволяют читать и устанавливать значения отдельных запрошенных
линий GPIO. Они принимают следующие значения: ``1`` и ``0``.
