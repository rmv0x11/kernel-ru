.. _perf_security:

Безопасность perf events и инструментов
=======================================

Обзор
-----

Использование Performance Counters for Linux (perf_events) [1]_ , [2]_ , [3]_
может создавать значительный риск утечки конфиденциальных данных, к которым
обращаются отслеживаемые процессы. Утечка данных возможна как в сценариях
прямого использования API системного вызова perf_events [2]_ , так и через
файлы данных, генерируемые пользовательской утилитой Perf tool (Perf) [3]_ , [4]_ .
Риск зависит от характера данных, которые блоки мониторинга производительности
(PMU) [2]_ perf_events и Perf собирают и предоставляют для анализа
производительности. Собранные системные данные и данные производительности
можно разделить на несколько категорий:

1. Данные о конфигурации аппаратного и программного обеспечения системы,
   например: модель CPU и конфигурация её кэша, объём доступной памяти и её
   топология, используемые версии ядра и Perf, настройка мониторинга
   производительности, включая время эксперимента, конфигурацию событий,
   параметры командной строки Perf и т. д.

2. Пути модулей пользователя и ядра и их адреса загрузки с размерами, имена
   процессов и потоков с их PID и TID, метки времени для захваченных аппаратных
   и программных событий.

3. Содержимое программных счётчиков ядра (например, для переключений контекста,
   страничных ошибок, миграций CPU), архитектурных аппаратных счётчиков
   производительности (PMC) [8]_ и machine specific registers (MSR) [9]_ ,
   которые предоставляют метрики выполнения для различных отслеживаемых частей
   системы (например, uncore-счётчиков контроллера памяти (IMC),
   межсоединения (QPI/UPI) или периферии (PCIe)) без прямой привязки к какому-либо
   состоянию контекста выполнения.

4. Содержимое архитектурных регистров контекста выполнения (например, RIP, RSP,
   RBP на x86_64), адреса памяти процесса в пространстве пользователя и ядра и
   данные, содержимое различных архитектурных MSR, которые захватывают данные из
   этой категории.

Данные, относящиеся к четвёртой категории, потенциально могут содержать
конфиденциальные данные процесса. Если PMU в некоторых режимах мониторинга
захватывают значения регистров контекста выполнения или данные из памяти
процесса, то доступ к таким режимам мониторинга требует надлежащего упорядочения
и защиты. Поэтому операции мониторинга производительности и наблюдаемости
perf_events являются предметом управления контролем доступа на уровне
безопасности [5]_ .

Контроль доступа perf_events
----------------------------

Для выполнения проверок безопасности реализация Linux разделяет процессы на две
категории [6]_ : а) привилегированные процессы (чей эффективный идентификатор
пользователя равен 0, называемые суперпользователем или root) и
б) непривилегированные процессы (чей эффективный UID отличен от нуля).
Привилегированные процессы обходят все проверки прав безопасности ядра, поэтому
мониторинг производительности perf_events полностью доступен привилегированным
процессам без ограничений доступа, области действия и ресурсов.

Непривилегированные процессы подлежат полной проверке прав безопасности на
основе учётных данных процесса [5]_ (обычно: эффективный UID, эффективный GID и
список дополнительных групп).

Linux разделяет привилегии, традиционно связанные с суперпользователем, на
отдельные единицы, известные как возможности (capabilities) [6]_ , которые можно
независимо включать и отключать на уровне отдельных потоков для процессов и
файлов непривилегированных пользователей.

Непривилегированные процессы с включённой возможностью CAP_PERFMON
рассматриваются как привилегированные процессы в отношении операций мониторинга
производительности и наблюдаемости perf_events, и, таким образом, обходят
проверки прав на *область действия* (*scope*) в ядре. CAP_PERFMON реализует
принцип наименьших привилегий [13]_ (POSIX 1003.1e: 2.2.2.39) для операций
мониторинга производительности и наблюдаемости в ядре и предоставляет
безопасный подход к мониторингу производительности и наблюдаемости в системе.

По причинам обратной совместимости доступ к операциям мониторинга
производительности и наблюдаемости perf_events также открыт для процессов с
привилегией CAP_SYS_ADMIN, но использование CAP_SYS_ADMIN для безопасных
сценариев мониторинга и наблюдаемости не рекомендуется в пользу возможности
CAP_PERFMON. Если системные записи аудита [14]_ для процесса, использующего API
системного вызова perf_events, содержат записи об отказе в получении обеих
возможностей CAP_PERFMON и CAP_SYS_ADMIN, то предоставление процессу только
возможности CAP_PERFMON рекомендуется как предпочтительный безопасный подход для
устранения двойного журналирования отказов в доступе, связанного с использованием
мониторинга производительности и наблюдаемости.

До Linux v5.9 непривилегированные процессы, использующие системный вызов
perf_events, также подлежали проверке режима ptrace-доступа
PTRACE_MODE_READ_REALCREDS [7]_ , результат которой определяет, разрешён ли
мониторинг. Поэтому непривилегированным процессам, которым предоставлена
возможность CAP_SYS_PTRACE, эффективно разрешено проходить эту проверку. Начиная
с Linux v5.9 возможность CAP_SYS_PTRACE не требуется, и для выполнения процессами
операций мониторинга производительности и наблюдаемости достаточно предоставить
CAP_PERFMON.

Другие возможности, предоставляемые непривилегированным процессам, могут
эффективно включать захват дополнительных данных, необходимых для последующего
анализа производительности отслеживаемых процессов или системы. Например,
возможность CAP_SYSLOG разрешает чтение адресов памяти пространства ядра из файла
/proc/kallsyms.

Группы привилегированных пользователей Perf
-------------------------------------------

Механизмы возможностей, привилегированных capability-dumb-файлов [6]_,
ACL файловой системы [10]_ и утилиты sudo [15]_ можно использовать для создания
выделенных групп привилегированных пользователей Perf, которым разрешено
выполнять мониторинг производительности и наблюдаемость без ограничений. Для
создания таких групп привилегированных пользователей Perf можно выполнить
следующие шаги.

1. Создайте группу perf_users привилегированных пользователей Perf, назначьте
   группу perf_users исполняемому файлу инструмента Perf и ограничьте доступ к
   исполняемому файлу для других пользователей в системе, которые не входят в
   группу perf_users:

::

   # groupadd perf_users
   # ls -alhF
   -rwxr-xr-x  2 root root  11M Oct 19 15:12 perf
   # chgrp perf_users perf
   # ls -alhF
   -rwxr-xr-x  2 root perf_users  11M Oct 19 15:12 perf
   # chmod o-rwx perf
   # ls -alhF
   -rwxr-x---  2 root perf_users  11M Oct 19 15:12 perf

2. Назначьте требуемые возможности исполняемому файлу инструмента Perf и наделите
   членов группы perf_users привилегиями мониторинга и наблюдаемости [6]_ :

::

   # setcap "cap_perfmon,cap_sys_ptrace,cap_syslog=ep" perf
   # setcap -v "cap_perfmon,cap_sys_ptrace,cap_syslog=ep" perf
   perf: OK
   # getcap perf
   perf = cap_sys_ptrace,cap_syslog,cap_perfmon+ep

Если установленная libcap [16]_ ещё не поддерживает "cap_perfmon", используйте
вместо этого "38", т. е.:

::

   # setcap "38,cap_ipc_lock,cap_sys_ptrace,cap_syslog=ep" perf

Обратите внимание, что вам может понадобиться добавить 'cap_ipc_lock' для
инструментов вроде 'perf top'; в качестве альтернативы используйте 'perf top -m N',
чтобы уменьшить объём памяти, используемой для кольцевого буфера perf, см. раздел
о выделении памяти ниже.

Использование libcap без поддержки CAP_PERFMON приведёт к сбою cap_get_flag(caps, 38,
CAP_EFFECTIVE, &val), что приведёт к тому, что событием по умолчанию станет 'cycles:u',
поэтому в качестве обходного решения явно запросите событие 'cycles', т. е.:

::

  # perf top -e cycles

чтобы получить выборки ядра и пользователя с бинарным файлом perf, имеющим только
CAP_PERFMON.

В результате члены группы perf_users могут проводить мониторинг производительности
и наблюдаемость, используя функциональность настроенного исполняемого файла
инструмента Perf, который при выполнении проходит проверки области действия
подсистемы perf_events.

В случае, если исполняемому файлу инструмента Perf не могут быть назначены
требуемые возможности (например, файловая система смонтирована с опцией nosuid
или расширенные атрибуты не поддерживаются файловой системой), то возможно
создание привилегированной по возможностям среды, естественно — оболочки. Оболочка
предоставляет наследующим процессам CAP_PERFMON и другие требуемые возможности,
так что операции мониторинга производительности и наблюдаемости доступны в среде
без ограничений. Доступ к среде можно открыть через утилиту sudo только для членов
группы perf_users. Чтобы создать такую среду:

1. Создайте сценарий оболочки, использующий утилиту capsh [16]_ для назначения
   CAP_PERFMON и других требуемых возможностей в ambient-набор возможностей
   процесса оболочки, блокирующий security-биты процесса после включения битов
   SECBIT_NO_SETUID_FIXUP, SECBIT_NOROOT и SECBIT_NO_CAP_AMBIENT_RAISE, а затем
   меняющий идентичность процесса на вызвавшего сценарий через sudo пользователя,
   который по сути должен быть членом группы perf_users:

::

   # ls -alh /usr/local/bin/perf.shell
   -rwxr-xr-x. 1 root root 83 Oct 13 23:57 /usr/local/bin/perf.shell
   # cat /usr/local/bin/perf.shell
   exec /usr/sbin/capsh --iab=^cap_perfmon --secbits=239 --user=$SUDO_USER -- -l

2. Дополните политику sudo в файле /etc/sudoers правилом для группы perf_users:

::

   # grep perf_users /etc/sudoers
   %perf_users    ALL=/usr/local/bin/perf.shell

3. Убедитесь, что члены группы perf_users имеют доступ к привилегированной
   оболочке и что CAP_PERFMON и другие требуемые возможности включены в наборах
   permitted, effective и ambient наследующего процесса:

::

  $ id
  uid=1003(capsh_test) gid=1004(capsh_test) groups=1004(capsh_test),1000(perf_users) context=unconfined_u:unconfined_r:unconfined_t:s0-s0:c0.c1023
  $ sudo perf.shell
  [sudo] password for capsh_test:
  $ grep Cap /proc/self/status
  CapInh:        0000004000000000
  CapPrm:        0000004000000000
  CapEff:        0000004000000000
  CapBnd:        000000ffffffffff
  CapAmb:        0000004000000000
  $ capsh --decode=0000004000000000
  0x0000004000000000=cap_perfmon

В результате члены группы perf_users получают доступ к привилегированной среде,
где они могут использовать инструменты, применяющие API мониторинга
производительности, управляемые возможностью Linux CAP_PERFMON.

Это конкретное управление контролем доступа доступно только суперпользователю
или процессам, выполняемым от имени root с возможностями CAP_SETPCAP, CAP_SETFCAP [6]_ .

Непривилегированные пользователи
--------------------------------

Контроль *области действия* (*scope*) и *доступа* (*access*) perf_events для
непривилегированных процессов управляется настройкой perf_event_paranoid [2]_ :

-1:
     Не накладывать ограничений на *область действия* и *доступ* при
     использовании мониторинга производительности perf_events. Ограничение
     блокировки памяти perf_event_mlock_kb [2]_ на уровне пользователя и
     процессора (per-user per-cpu) игнорируется при выделении буферов памяти для
     хранения данных производительности. Это наименее безопасный режим, поскольку
     разрешённая отслеживаемая *область действия* максимальна и не накладываются
     специфичные для perf_events ограничения на *ресурсы*, выделяемые для
     мониторинга производительности.

>=0:
     *область действия* включает мониторинг производительности на уровне
     процессов и в масштабах всей системы, но исключает мониторинг сырых
     трейспоинтов (raw tracepoints) и трейспоинтов функций ftrace. События CPU и
     системы, происходящие при выполнении как в пространстве пользователя, так и в
     пространстве ядра, могут отслеживаться и захватываться для последующего
     анализа. Ограничение блокировки perf_event_mlock_kb на уровне пользователя и
     процессора накладывается, но игнорируется для непривилегированных процессов с
     возможностью CAP_IPC_LOCK [6]_ .

>=1:
     *область действия* включает только мониторинг производительности на уровне
     процессов и исключает мониторинг в масштабах всей системы. События CPU и
     системы, происходящие при выполнении как в пространстве пользователя, так и в
     пространстве ядра, могут отслеживаться и захватываться для последующего
     анализа. Ограничение блокировки perf_event_mlock_kb на уровне пользователя и
     процессора накладывается, но игнорируется для непривилегированных процессов с
     возможностью CAP_IPC_LOCK.

>=2:
     *область действия* включает только мониторинг производительности на уровне
     процессов. События CPU и системы, происходящие при выполнении только в
     пространстве пользователя, могут отслеживаться и захватываться для
     последующего анализа. Ограничение блокировки perf_event_mlock_kb на уровне
     пользователя и процессора накладывается, но игнорируется для
     непривилегированных процессов с возможностью CAP_IPC_LOCK.

Контроль ресурсов
-----------------

Открытые файловые дескрипторы
+++++++++++++++++++++++++++++

API системного вызова perf_events [2]_ выделяет файловые дескрипторы для каждого
настроенного события PMU. Открытые файловые дескрипторы являются учитываемым
ресурсом на уровне процесса, управляемым ограничением RLIMIT_NOFILE [11]_
(ulimit -n), которое обычно наследуется от процесса оболочки входа в систему. При
настройке сбора Perf для длинного списка событий на большой серверной системе это
ограничение легко достичь, что препятствует требуемой конфигурации мониторинга.
Ограничение RLIMIT_NOFILE можно увеличить на уровне пользователя, изменив
содержимое файла limits.conf [12]_ . Обычно сеанс выборки Perf (perf record)
требует количества открытых файловых дескрипторов perf_event, не меньшего, чем
число отслеживаемых событий, умноженное на число отслеживаемых CPU.

Выделение памяти
++++++++++++++++

Объём памяти, доступный пользовательским процессам для захвата данных мониторинга
производительности, управляется настройкой perf_event_mlock_kb [2]_ . Эта
специфичная для perf_event настройка ресурсов определяет общие ограничения на
уровне процессора (per-cpu) для памяти, разрешённой для отображения пользовательскими
процессами для выполнения мониторинга производительности. По сути, настройка
расширяет ограничение RLIMIT_MEMLOCK [11]_ , но только для областей памяти,
отображаемых специально для захвата отслеживаемых событий производительности и
связанных данных.

Например, если машина имеет восемь ядер, а ограничение perf_event_mlock_kb
установлено в 516 КиБ, то пользовательскому процессу предоставляется 516 КиБ * 8 =
4128 КиБ памяти сверх ограничения RLIMIT_MEMLOCK (ulimit -l) для mmap-буферов
perf_event. В частности, это означает, что если пользователь хочет запустить два
или более процессов мониторинга производительности, ему необходимо вручную
распределить доступные 4128 КиБ между процессами мониторинга, например, с помощью
опции --mmap-pages режима perf record. В противном случае первый запущенный
процесс мониторинга производительности выделит все доступные 4128 КиБ, и
остальные процессы не смогут продолжить работу из-за нехватки памяти.

Ограничения ресурсов RLIMIT_MEMLOCK и perf_event_mlock_kb игнорируются для
процессов с возможностью CAP_IPC_LOCK. Таким образом, привилегированным
пользователям perf_events/Perf может быть предоставлена память сверх ограничений
для целей мониторинга производительности perf_events/Perf путём наделения
исполняемого файла Perf возможностью CAP_IPC_LOCK.
