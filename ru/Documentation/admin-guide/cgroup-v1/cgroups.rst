1. Control Groups
=================

1.1 Что такое cgroups?
----------------------

Control Groups предоставляют механизм агрегирования/разбиения наборов
задач и всех их будущих потомков на иерархические группы со
специализированным поведением.

Определения:

*cgroup* связывает набор задач с набором параметров для одной или
нескольких подсистем.

*Подсистема* (subsystem) — это модуль, использующий предоставляемые
cgroups средства группировки задач, чтобы обрабатывать группы задач
особым образом. Обычно подсистема представляет собой «контроллер
ресурсов» (resource controller), который планирует ресурс или применяет
ограничения на уровне cgroup, но это может быть что угодно, что желает
воздействовать на группу процессов, например подсистема виртуализации.

*Иерархия* (hierarchy) — это набор cgroups, организованных в дерево, так
что каждая задача в системе находится ровно в одной из cgroups этой
иерархии, а также набор подсистем; каждая подсистема имеет специфичное
для системы состояние, прикреплённое к каждой cgroup в иерархии. С каждой
иерархией связан экземпляр виртуальной файловой системы cgroup.

В любой момент времени может существовать несколько активных иерархий
cgroups задач. Каждая иерархия является разбиением всех задач в системе.

Код пользовательского уровня может создавать и уничтожать cgroups по
имени в экземпляре виртуальной файловой системы cgroup, указывать и
запрашивать, к какой cgroup назначена задача, и перечислять PID задач,
назначенных в cgroup. Эти создания и назначения затрагивают только
иерархию, связанную с данным экземпляром файловой системы cgroup.

Сами по себе cgroups годятся только для простого отслеживания заданий.
Замысел в том, чтобы другие подсистемы подключались к общей поддержке
cgroup и предоставляли cgroups новые атрибуты, такие как
учёт/ограничение ресурсов, к которым могут обращаться процессы в cgroup.
Например, cpusets (см. Documentation/admin-guide/cgroup-v1/cpusets.rst) позволяют
вам связать набор CPU и набор узлов памяти с задачами в каждой cgroup.

1.2 Зачем нужны cgroups?
------------------------

В ядре Linux существует несколько разработок, обеспечивающих агрегацию
процессов, главным образом для целей отслеживания ресурсов. К таким
разработкам относятся cpusets, CKRM/ResGroups, UserBeanCounters и
пространства имён виртуальных серверов. Все они требуют базового понятия
группировки/разбиения процессов, при котором вновь порождённые процессы
оказываются в той же группе (cgroup), что и их родительский процесс.

Патч cgroup ядра предоставляет минимально необходимые механизмы ядра,
требуемые для эффективной реализации таких групп. Он оказывает
минимальное влияние на быстрые пути системы и предоставляет хуки для
конкретных подсистем, таких как cpusets, чтобы те по желанию
обеспечивали дополнительное поведение.

Поддержка нескольких иерархий предоставлена для ситуаций, когда деление
задач на cgroups существенно различается для разных подсистем — наличие
параллельных иерархий позволяет каждой иерархии быть естественным
делением задач, не приходясь обрабатывать сложные комбинации задач,
которые возникли бы, если бы несколько несвязанных подсистем пришлось бы
принудительно втиснуть в одно и то же дерево cgroups.

В одной крайности каждый контроллер ресурсов или подсистема могли бы
находиться в отдельной иерархии; в другой крайности все подсистемы были
бы прикреплены к одной и той же иерархии.

В качестве примера сценария (изначально предложенного vatsa@in.ibm.com),
который может выиграть от нескольких иерархий, рассмотрим крупный
университетский сервер с различными пользователями — студентами,
преподавателями, системными задачами и т.д. Планирование ресурсов для
этого сервера могло бы выглядеть примерно так::

       CPU :          "Top cpuset"
                       /       \
               CPUSet1         CPUSet2
                  |               |
               (Professors)    (Students)

               In addition (system tasks) are attached to topcpuset (so
               that they can run anywhere) with a limit of 20%

       Memory : Professors (50%), Students (30%), system (20%)

       Disk : Professors (50%), Students (30%), system (20%)

       Network : WWW browsing (20%), Network File System (60%), others (20%)
                               / \
               Professors (15%)  students (5%)

Браузеры вроде Firefox/Lynx попадают в сетевой класс WWW, а (k)nfsd
попадает в сетевой класс NFS.

В то же время Firefox/Lynx будут разделять соответствующий класс
CPU/Memory в зависимости от того, кто его запустил (преподаватель/студент).

Имея возможность классифицировать задачи по-разному для разных ресурсов
(помещая эти подсистемы ресурсов в разные иерархии), администратор может
легко настроить скрипт, который получает уведомления об exec, и в
зависимости от того, кто запускает браузер, он может::

    # echo browser_pid > /sys/fs/cgroup/<restype>/<userclass>/tasks

При наличии всего одной иерархии ему теперь потенциально пришлось бы
создавать отдельную cgroup для каждого запущенного браузера и связывать
её с соответствующим сетевым и другим классом ресурсов. Это может
привести к разрастанию таких cgroups.

Допустим также, что администратор хотел бы временно предоставить
расширенный сетевой доступ браузеру студента (поскольку сейчас ночь, и
пользователь хочет поиграть в онлайн-игры :)) ИЛИ предоставить одному из
симуляционных приложений студента увеличенную вычислительную мощность CPU.

Имея возможность записывать PID непосредственно в классы ресурсов, это
сводится лишь к::

       # echo pid > /sys/fs/cgroup/network/<new_class>/tasks
       (after some time)
       # echo pid > /sys/fs/cgroup/network/<orig_class>/tasks

Без этой возможности администратору пришлось бы разбивать cgroup на
несколько отдельных, а затем связывать новые cgroups с новыми классами
ресурсов.



1.3 Как реализованы cgroups?
----------------------------

Control Groups расширяет ядро следующим образом:

 - Каждая задача в системе имеет указатель со счётчиком ссылок на
   css_set.

 - css_set содержит набор указателей со счётчиками ссылок на объекты
   cgroup_subsys_state, по одному на каждую подсистему cgroup,
   зарегистрированную в системе. Прямой связи от задачи к cgroup, членом
   которой она является в каждой иерархии, нет, но это можно определить,
   проследовав по указателям через объекты cgroup_subsys_state. Так
   сделано потому, что доступ к состоянию подсистемы — это то, что, как
   ожидается, происходит часто и в критичном к производительности коде,
   тогда как операции, требующие фактических назначений cgroup задачи (в
   частности, перемещение между cgroups), менее распространены. Через
   поле cg_list каждой task_struct проходит связный список,
   использующий css_set и закреплённый за css_set->tasks.

 - Файловая система иерархии cgroup может быть смонтирована для
   просмотра и манипуляций из пространства пользователя.

 - Вы можете перечислить все задачи (по PID), прикреплённые к любой cgroup.

Реализация cgroups требует нескольких простых хуков в остальной части
ядра, ни одного в критичных к производительности путях:

 - в init/main.c, чтобы инициализировать корневые cgroups и начальный
   css_set при загрузке системы.

 - в fork и exit, чтобы прикреплять задачу к её css_set и откреплять
   от него.

Кроме того, может быть смонтирована новая файловая система типа
"cgroup", чтобы включить просмотр и изменение cgroups, известных ядру в
данный момент. При монтировании иерархии cgroup вы можете указать
разделённый запятыми список подсистем для монтирования в качестве опций
монтирования файловой системы. По умолчанию монтирование файловой
системы cgroup пытается смонтировать иерархию, содержащую все
зарегистрированные подсистемы.

Если уже существует активная иерархия с точно таким же набором
подсистем, она будет переиспользована для нового монтирования. Если ни
одна существующая иерархия не подходит, и какая-либо из запрошенных
подсистем используется в существующей иерархии, монтирование завершится
неудачей с -EBUSY. В противном случае активируется новая иерархия,
связанная с запрошенными подсистемами.

В настоящее время невозможно привязать новую подсистему к активной
иерархии cgroup или отвязать подсистему от активной иерархии cgroup. Это
может стать возможным в будущем, но сопряжено с неприятными проблемами
восстановления после ошибок.

Когда файловая система cgroup размонтируется, если ниже cgroup верхнего
уровня были созданы какие-либо дочерние cgroups, эта иерархия останется
активной, несмотря на размонтирование; если дочерних cgroups нет, то
иерархия будет деактивирована.

Для cgroups не добавлено никаких новых системных вызовов — вся поддержка
запроса и изменения cgroups осуществляется через эту файловую систему
cgroup.

Каждая задача в /proc имеет добавленный файл с именем 'cgroup',
отображающий для каждой активной иерархии имена подсистем и имя cgroup в
виде пути относительно корня файловой системы cgroup.

Каждая cgroup представлена каталогом в файловой системе cgroup,
содержащим следующие файлы, описывающие эту cgroup:

 - tasks: список задач (по PID), прикреплённых к этой cgroup. Не
   гарантируется, что этот список отсортирован. Запись идентификатора
   потока в этот файл перемещает поток в эту cgroup.
 - cgroup.procs: список идентификаторов групп потоков в cgroup. Не
   гарантируется, что этот список отсортирован или свободен от
   дублирующихся TGID, и пространство пользователя должно
   сортировать/удалять дубликаты в списке, если это свойство требуется.
   Запись идентификатора группы потоков в этот файл перемещает все потоки
   этой группы в данную cgroup.
 - флаг notify_on_release: запускать ли release agent при выходе?
 - release_agent: путь, используемый для уведомлений об освобождении
   (этот файл существует только в верхней cgroup)

Другие подсистемы, такие как cpusets, могут добавлять дополнительные
файлы в каталог каждой cgroup.

Новые cgroups создаются с помощью системного вызова mkdir или команды
оболочки. Свойства cgroup, такие как её флаги, изменяются записью в
соответствующий файл в каталоге этой cgroup, как перечислено выше.

Именованная иерархическая структура вложенных cgroups позволяет
разбивать большую систему на вложенные, динамически изменяемые «мягкие
разделы» (soft-partitions).

Прикрепление каждой задачи к cgroup, автоматически наследуемое при fork
любыми потомками этой задачи, позволяет организовать рабочую нагрузку в
системе в связанные наборы задач. Задача может быть повторно прикреплена
к любой другой cgroup, если это разрешено правами на необходимые
каталоги файловой системы cgroup.

Когда задача перемещается из одной cgroup в другую, она получает новый
указатель css_set — если уже существует css_set с желаемой коллекцией
cgroups, то эта группа переиспользуется, иначе выделяется новый css_set.
Подходящий существующий css_set находится поиском в хеш-таблице.

Чтобы обеспечить доступ из cgroup к css_sets (и, следовательно, к
задачам), которые её составляют, набор объектов cg_cgroup_link образует
решётку; каждый cg_cgroup_link связан в список cg_cgroup_links для
отдельной cgroup в своём поле cgrp_link_list, а также в список
cg_cgroup_links для отдельного css_set в своём поле cg_link_list.

Таким образом, набор задач в cgroup можно перечислить, итерируя по
каждому css_set, который ссылается на cgroup, и под-итерируя по набору
задач каждого css_set.

Использование виртуальной файловой системы Linux (vfs) для представления
иерархии cgroup обеспечивает привычное пространство прав и имён для
cgroups при минимуме дополнительного кода ядра.

1.4 Что делает notify_on_release?
---------------------------------

Если флаг notify_on_release включён (1) в cgroup, то всякий раз, когда
последняя задача в cgroup покидает её (завершается или прикрепляется к
какой-либо другой cgroup) и удаляется последняя дочерняя cgroup этой
cgroup, ядро запускает команду, заданную содержимым файла
"release_agent" в корневом каталоге этой иерархии, передавая ей путь
(относительно точки монтирования файловой системы cgroup) покинутой
cgroup. Это позволяет автоматически удалять покинутые cgroups. Значение
notify_on_release по умолчанию в корневой cgroup при загрузке системы —
отключено (0). Значение по умолчанию для других cgroups при создании —
это текущее значение настройки notify_on_release их родителей. Значение
по умолчанию пути release_agent иерархии cgroup — пустое.

1.5 Что делает clone_children?
------------------------------

Этот флаг влияет только на контроллер cpuset. Если флаг clone_children
включён (1) в cgroup, новая cgroup cpuset скопирует свою конфигурацию от
родителя во время инициализации.

1.6 Как использовать cgroups?
-----------------------------

Чтобы запустить новое задание, которое должно содержаться в cgroup, с
использованием подсистемы cgroup "cpuset", шаги примерно таковы::

 1) mount -t tmpfs cgroup_root /sys/fs/cgroup
 2) mkdir /sys/fs/cgroup/cpuset
 3) mount -t cgroup -ocpuset cpuset /sys/fs/cgroup/cpuset
 4) Create the new cgroup by doing mkdir's and write's (or echo's) in
    the /sys/fs/cgroup/cpuset virtual file system.
 5) Start a task that will be the "founding father" of the new job.
 6) Attach that task to the new cgroup by writing its PID to the
    /sys/fs/cgroup/cpuset tasks file for that cgroup.
 7) fork, exec or clone the job tasks from this founding father task.

Например, следующая последовательность команд настроит cgroup с именем
"Charlie", содержащую только CPU 2 и 3 и узел памяти 1, а затем запустит
в этой cgroup дочернюю оболочку 'sh'::

  mount -t tmpfs cgroup_root /sys/fs/cgroup
  mkdir /sys/fs/cgroup/cpuset
  mount -t cgroup cpuset -ocpuset /sys/fs/cgroup/cpuset
  cd /sys/fs/cgroup/cpuset
  mkdir Charlie
  cd Charlie
  /bin/echo 2-3 > cpuset.cpus
  /bin/echo 1 > cpuset.mems
  /bin/echo $$ > tasks
  sh
  # The subshell 'sh' is now running in cgroup Charlie
  # The next line should display '/Charlie'
  cat /proc/self/cgroup

2. Usage Examples and Syntax
============================

2.1 Базовое использование
--------------------------

Создание, изменение и использование cgroups можно выполнять через
виртуальную файловую систему cgroup.

Чтобы смонтировать иерархию cgroup со всеми доступными подсистемами,
введите::

  # mount -t cgroup xxx /sys/fs/cgroup

Строка "xxx" не интерпретируется кодом cgroup, но появится в
/proc/mounts, так что это может быть любая полезная вам идентифицирующая
строка.

Замечание: некоторые подсистемы не работают без некоторого
предварительного ввода от пользователя. Например, если включены cpusets,
пользователю придётся заполнить файлы cpus и mems для каждой новой
создаваемой cgroup, прежде чем эту группу можно будет использовать.

Как объяснено в разделе `1.2 Зачем нужны cgroups?` вам следует создавать
разные иерархии cgroups для каждого отдельного ресурса или группы
ресурсов, которыми вы хотите управлять. Поэтому вам следует смонтировать
tmpfs на /sys/fs/cgroup и создать каталоги для каждого ресурса cgroup
или группы ресурсов::

  # mount -t tmpfs cgroup_root /sys/fs/cgroup
  # mkdir /sys/fs/cgroup/rg1

Чтобы смонтировать иерархию cgroup только с подсистемами cpuset и memory,
введите::

  # mount -t cgroup -o cpuset,memory hier1 /sys/fs/cgroup/rg1

Хотя перемонтирование cgroups в настоящее время поддерживается, его
использование не рекомендуется. Перемонтирование позволяет изменять
привязанные подсистемы и release_agent. Перепривязка едва ли полезна,
поскольку она работает только когда иерархия пуста, а сам release_agent
следует заменить обычным fsnotify. Поддержка перемонтирования будет
удалена в будущем.

Чтобы указать release_agent иерархии::

  # mount -t cgroup -o cpuset,release_agent="/sbin/cpuset_release_agent" \
    xxx /sys/fs/cgroup/rg1

Обратите внимание, что указание 'release_agent' более одного раза вернёт
ошибку.

Обратите внимание, что изменение набора подсистем в настоящее время
поддерживается только когда иерархия состоит из единственной (корневой)
cgroup. Поддержку возможности произвольно привязывать/отвязывать
подсистемы от существующей иерархии cgroup планируется реализовать в
будущем.

Затем под /sys/fs/cgroup/rg1 вы можете найти дерево, соответствующее
дереву cgroups в системе. Например, /sys/fs/cgroup/rg1 — это cgroup,
которая содержит всю систему.

Если вы хотите изменить значение release_agent::

  # echo "/sbin/new_release_agent" > /sys/fs/cgroup/rg1/release_agent

Его также можно изменить через перемонтирование.

Если вы хотите создать новую cgroup под /sys/fs/cgroup/rg1::

  # cd /sys/fs/cgroup/rg1
  # mkdir my_cgroup

Теперь вы хотите что-то сделать с этой cgroup:

  # cd my_cgroup

В этом каталоге вы можете найти несколько файлов::

  # ls
  cgroup.procs notify_on_release tasks
  (plus whatever files added by the attached subsystems)

Теперь прикрепите вашу оболочку к этой cgroup::

  # /bin/echo $$ > tasks

Вы также можете создавать cgroups внутри вашей cgroup, используя mkdir в
этом каталоге::

  # mkdir my_sub_cs

Чтобы удалить cgroup, просто используйте rmdir::

  # rmdir my_sub_cs

Это завершится неудачей, если cgroup используется (содержит cgroups
внутри, или к ней прикреплены процессы, или она удерживается живой другой
специфичной для подсистемы ссылкой).

2.2 Прикрепление процессов
--------------------------

::

  # /bin/echo PID > tasks

Обратите внимание, что это PID, а не PIDs. За один раз можно прикрепить
только ОДНУ задачу. Если у вас есть несколько задач для прикрепления, вы
должны делать это одну за другой::

  # /bin/echo PID1 > tasks
  # /bin/echo PID2 > tasks
	  ...
  # /bin/echo PIDn > tasks

Вы можете прикрепить текущую задачу оболочки, записав (echo) 0::

  # echo 0 > tasks

Вы можете использовать файл cgroup.procs вместо файла tasks, чтобы
переместить все потоки группы потоков сразу. Запись (echo) PID любой
задачи из группы потоков в cgroup.procs приводит к прикреплению всех
задач этой группы потоков к cgroup. Запись 0 в cgroup.procs перемещает
все задачи из группы потоков записывающей задачи.

Замечание: поскольку каждая задача всегда является членом ровно одной
cgroup в каждой смонтированной иерархии, чтобы удалить задачу из её
текущей cgroup, вы должны переместить её в новую cgroup (возможно,
корневую cgroup), записав в файл tasks новой cgroup.

Замечание: из-за некоторых ограничений, навязываемых некоторыми
подсистемами cgroup, перемещение процесса в другую cgroup может
завершиться неудачей.

2.3 Монтирование иерархий по имени
----------------------------------

Передача опции name=<x> при монтировании иерархии cgroups связывает
заданное имя с иерархией. Это можно использовать при монтировании уже
существующей иерархии, чтобы ссылаться на неё по имени, а не по её набору
активных подсистем. Каждая иерархия либо безымянна, либо имеет уникальное
имя.

Имя должно соответствовать [\w.-]+

При передаче опции name=<x> для новой иерархии вам нужно указать
подсистемы вручную; устаревшее поведение монтирования всех подсистем,
когда ни одна не указана явно, не поддерживается, когда вы даёте
подсистеме имя.

Имя подсистемы появляется как часть описания иерархии в /proc/mounts и
/proc/<pid>/cgroups.


3. Kernel API
=============

3.1 Обзор
---------

Каждой подсистеме ядра, которая хочет подключиться к общей системе
cgroup, нужно создать объект cgroup_subsys. Он содержит различные методы,
являющиеся обратными вызовами из системы cgroup, а также идентификатор
подсистемы, который будет назначен системой cgroup.

Другие поля объекта cgroup_subsys включают:

- subsys_id: уникальный индекс массива для подсистемы, указывающий, какой
  записью в cgroup->subsys[] должна управлять эта подсистема.

- name: должно быть инициализировано уникальным именем подсистемы. Не
  должно быть длиннее MAX_CGROUP_TYPE_NAMELEN.

- early_init: указывает, нуждается ли подсистема в ранней инициализации
  при загрузке системы.

Каждый объект cgroup, созданный системой, имеет массив указателей,
индексируемый по идентификатору подсистемы; этот указатель полностью
управляется подсистемой; общий код cgroup никогда не трогает этот
указатель.

3.2 Синхронизация
-----------------

Существует глобальный mutex, cgroup_mutex, используемый системой cgroup.
Его следует захватывать всему, что хочет изменить cgroup. Его также можно
захватывать, чтобы предотвратить изменение cgroups, но в такой ситуации
могут быть более уместны более специфичные блокировки.

См. kernel/cgroup.c для дополнительных подробностей.

Подсистемы могут захватывать/освобождать cgroup_mutex с помощью функций
cgroup_lock()/cgroup_unlock().

Доступ к указателю cgroup задачи может выполняться следующими способами:
- при удержании cgroup_mutex
- при удержании alloc_lock задачи (через task_lock())
- внутри секции rcu_read_lock() через rcu_dereference()

3.3 Subsystem API
-----------------

Каждая подсистема должна:

- добавить запись в linux/cgroup_subsys.h
- определить объект cgroup_subsys с именем <name>_cgrp_subsys

Каждая подсистема может экспортировать следующие методы. Единственные
обязательные методы — css_alloc/free. Любые другие, которые равны null,
считаются успешными no-op-ами.

``struct cgroup_subsys_state *css_alloc(struct cgroup *cgrp)``
(cgroup_mutex held by caller)

Вызывается для выделения объекта состояния подсистемы для cgroup.
Подсистема должна выделить свой объект состояния подсистемы для
переданной cgroup, возвращая указатель на новый объект при успехе или
значение ERR_PTR(). При успехе указатель подсистемы должен указывать на
структуру типа cgroup_subsys_state (обычно встроенную в более крупный
специфичный для подсистемы объект), которая будет инициализирована
системой cgroup. Обратите внимание, что это будет вызвано при
инициализации для создания корневого состояния подсистемы для этой
подсистемы; этот случай можно опознать по тому, что переданный объект
cgroup имеет NULL-родителя (поскольку это корень иерархии), и он может
быть подходящим местом для кода инициализации.

``int css_online(struct cgroup *cgrp)``
(cgroup_mutex held by caller)

Вызывается после того, как @cgrp успешно завершила все выделения и стала
видимой для итераторов cgroup_for_each_child/descendant_*(). Подсистема
может выбрать провалить создание, вернув -errno. Этот обратный вызов
можно использовать для реализации надёжного совместного использования и
распространения состояния вдоль иерархии. Подробности см. в комментарии к
cgroup_for_each_live_descendant_pre().

``void css_offline(struct cgroup *cgrp);``
(cgroup_mutex held by caller)

Это противоположность css_online() и вызывается тогда и только тогда,
когда css_online() завершился успехом на @cgrp. Это означает начало
конца @cgrp. @cgrp удаляется, и подсистема должна начать сбрасывать все
ссылки, которые она удерживает на @cgrp. Когда все ссылки сброшены,
удаление cgroup перейдёт к следующему шагу — css_free(). После этого
обратного вызова @cgrp следует считать мёртвой для подсистемы.

``void css_free(struct cgroup *cgrp)``
(cgroup_mutex held by caller)

Система cgroup собирается освободить @cgrp; подсистема должна освободить
свой объект состояния подсистемы. К моменту вызова этого метода @cgrp
полностью не используется; @cgrp->parent всё ещё действителен.
(Замечание — может также вызываться для вновь созданной cgroup, если
ошибка происходит после того, как для новой cgroup был вызван метод
create() этой подсистемы).

``int can_attach(struct cgroup *cgrp, struct cgroup_taskset *tset)``
(cgroup_mutex held by caller)

Вызывается перед перемещением одной или нескольких задач в cgroup; если
подсистема возвращает ошибку, это прервёт операцию прикрепления. @tset
содержит задачи, подлежащие прикреплению, и гарантированно содержит как
минимум одну задачу.

Если в taskset несколько задач, то:
  - гарантируется, что все они из одной группы потоков
  - @tset содержит все задачи из группы потоков, независимо от того,
    переключают они cgroups или нет
  - первая задача — лидер

Каждая запись @tset также содержит старую cgroup задачи, и задачи,
которые не переключают cgroup, можно легко пропустить с помощью итератора
cgroup_taskset_for_each(). Обратите внимание, что это не вызывается при
fork. Если этот метод возвращает 0 (успех), то это должно оставаться
действительным, пока вызывающий удерживает cgroup_mutex, и гарантируется,
что в будущем будет вызван либо attach(), либо cancel_attach().

``void css_reset(struct cgroup_subsys_state *css)``
(cgroup_mutex held by caller)

Необязательная операция, которая должна восстановить конфигурацию @css в
исходное состояние. В настоящее время используется только в единой
иерархии, когда подсистема отключается на cgroup через
"cgroup.subtree_control", но должна оставаться включённой, потому что от
неё зависят другие подсистемы. Ядро cgroup делает такой css невидимым,
удаляя связанные интерфейсные файлы, и вызывает этот обратный вызов,
чтобы скрытая подсистема могла вернуться в исходное нейтральное
состояние. Это предотвращает неожиданное управление ресурсами со стороны
скрытого css и гарантирует, что конфигурация находится в исходном
состоянии, когда она снова становится видимой позже.

``void cancel_attach(struct cgroup *cgrp, struct cgroup_taskset *tset)``
(cgroup_mutex held by caller)

Вызывается, когда операция прикрепления задачи провалилась после того,
как can_attach() завершился успехом. Подсистема, чей can_attach() имеет
некоторые побочные эффекты, должна предоставить эту функцию, чтобы
подсистема могла реализовать откат. Если нет, то она не нужна. Это будет
вызвано только для подсистем, чья операция can_attach() завершилась
успехом. Параметры идентичны can_attach().

``void attach(struct cgroup *cgrp, struct cgroup_taskset *tset)``
(cgroup_mutex held by caller)

Вызывается после того, как задача была прикреплена к cgroup, чтобы
позволить любую пост-прикрепительную деятельность, требующую выделений
памяти или блокировки. Параметры идентичны can_attach().

``void fork(struct task_struct *task)``

Вызывается, когда задача порождается (fork) в cgroup.

``void exit(struct task_struct *task)``

Вызывается во время выхода задачи.

``void free(struct task_struct *task)``

Вызывается, когда task_struct освобождается.

``void bind(struct cgroup *root)``
(cgroup_mutex held by caller)

Вызывается, когда подсистема cgroup перепривязывается к другой иерархии и
корневой cgroup. В настоящее время это будет затрагивать только
перемещение между иерархией по умолчанию (которая никогда не имеет
суб-cgroups) и иерархией, которая создаётся/уничтожается (и, следовательно,
не имеет суб-cgroups).

4. Extended attribute usage
===========================

Файловая система cgroup поддерживает определённые типы расширенных
атрибутов в своих каталогах и файлах. В настоящее время поддерживаемые
типы:

	- Trusted (XATTR_TRUSTED)
	- Security (XATTR_SECURITY)

Оба требуют для установки полномочия CAP_SYS_ADMIN.

Как и в tmpfs, расширенные атрибуты в файловой системе cgroup хранятся с
использованием памяти ядра, и рекомендуется держать их использование на
минимуме. Это причина, по которой пользовательские расширенные атрибуты
не поддерживаются, поскольку их может задать любой пользователь, и нет
ограничения на размер значения.

Текущие известные пользователи этой возможности — SELinux, чтобы
ограничивать использование cgroup в контейнерах, и systemd для
разнообразных метаданных, таких как основной PID в cgroup (systemd
создаёт по одной cgroup на службу).

5. Questions
============

::

  Q: what's up with this '/bin/echo' ?
  A: bash's builtin 'echo' command does not check calls to write() against
     errors. If you use it in the cgroup file system, you won't be
     able to tell whether a command succeeded or failed.

  Q: When I attach processes, only the first of the line gets really attached !
  A: We can only return one error code per call to write(). So you should also
     put only ONE PID.
