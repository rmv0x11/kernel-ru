==========================
Контроллер Block IO
==========================

Обзор
=====
Подсистема cgroup «blkio» реализует контроллер блочного ввода-вывода (block io).
Судя по всему, существует потребность в различных видах политик управления вводом-выводом
(таких как пропорциональная полоса пропускания (BW), максимальная BW) как на конечных
узлах, так и на промежуточных узлах иерархии хранилища. План состоит в том, чтобы
использовать один и тот же интерфейс управления на основе cgroup для контроллера blkio
и, в зависимости от выбранных пользователем параметров, переключать политики
ввода-вывода в фоновом режиме.

Одна из политик управления вводом-выводом — это политика троттлинга (throttling),
которую можно использовать для задания верхних пределов скорости ввода-вывода на
устройствах. Эта политика реализована в обобщённом блочном слое и может применяться как
на конечных узлах, так и на логических устройствах более высокого уровня, таких как
device mapper.

HOWTO
=====

Политика троттлинга / верхнего предела
--------------------------------------
Включите контроллер Block IO::

	CONFIG_BLK_CGROUP=y

Включите троттлинг в блочном слое::

	CONFIG_BLK_DEV_THROTTLING=y

Смонтируйте контроллер blkio (см. cgroups.txt, Why are cgroups needed?)::

        mount -t cgroup -o blkio none /sys/fs/cgroup/blkio

Задайте ограничение скорости полосы пропускания на конкретном устройстве для корневой
группы. Формат политики — "<major>:<minor>  <bytes_per_second>"::

        echo "8:16  1048576" > /sys/fs/cgroup/blkio/blkio.throttle.read_bps_device

Это установит ограничение в 1 МБ/секунду на чтения, происходящие для корневой группы
на устройстве с номерами major/minor 8:16.

Запустите dd для чтения файла и проверьте, ограничивается ли скорость до 1 МБ/с::

        # dd iflag=direct if=/mnt/common/zerofile of=/dev/null bs=4K count=1024
        1024+0 records in
        1024+0 records out
        4194304 bytes (4.2 MB) copied, 4.0001 s, 1.0 MB/s

Ограничения для записей можно задать с помощью файла blkio.throttle.write_bps_device.

Иерархические Cgroups
=====================

Троттлинг реализует поддержку иерархии; однако поддержка иерархии для троттлинга
включается только в том случае, если со стороны cgroup включён режим «sane_behavior»,
который в настоящее время является опцией для разработчиков и публично недоступен.

Если кто-либо создал иерархию вроде следующей::

			root
			/  \
		     test1 test2
			|
		     test3

Троттлинг с «sane_behavior» будет обрабатывать иерархию корректно. Для троттлинга все
пределы применяются ко всему поддереву, тогда как вся статистика локальна для операций
ввода-вывода, непосредственно сгенерированных задачами в данной cgroup.

Троттлинг без включённого со стороны cgroup «sane_behavior» практически будет
рассматривать все группы на одном уровне так, как если бы это выглядело следующим
образом::

				pivot
			     /  /   \  \
			root  test1 test2  test3

Различные видимые пользователю параметры конфигурации
=====================================================

  CONFIG_BLK_CGROUP
	  Контроллер Block IO.

  CONFIG_BFQ_CGROUP_DEBUG
	  Помощь в отладке. На данный момент при включении этой опции в cgroup
	  появляются некоторые дополнительные файлы статистики.

  CONFIG_BLK_DEV_THROTTLING
	  Включает в блочном слое поддержку троттлинга блочных устройств.

Подробности о файлах cgroup
===========================

Файлы политики пропорционального веса
-------------------------------------

  blkio.bfq.weight
	  Задаёт вес для каждой cgroup. Это вес группы по умолчанию на всех устройствах,
	  пока он не будет переопределён правилом для конкретного устройства
	  (см. `blkio.bfq.weight_device` ниже).

	  В настоящее время допустимый диапазон весов — от 1 до 1000. Подробнее
          см. Documentation/block/bfq-iosched.rst.

  blkio.bfq.weight_device
          Задаёт веса для каждой cgroup на каждом устройстве, переопределяя вес группы
          по умолчанию. Подробнее см. Documentation/block/bfq-iosched.rst.

	  Формат следующий::

	    # echo dev_maj:dev_minor weight > blkio.bfq.weight_device

	  Настройка weight=300 на /dev/sdb (8:16) в этой cgroup::

	    # echo 8:16 300 > blkio.bfq.weight_device
	    # cat blkio.bfq.weight_device
	    dev     weight
	    8:16    300

	  Настройка weight=500 на /dev/sda (8:0) в этой cgroup::

	    # echo 8:0 500 > blkio.bfq.weight_device
	    # cat blkio.bfq.weight_device
	    dev     weight
	    8:0     500
	    8:16    300

	  Удаление конкретного веса для /dev/sda в этой cgroup::

	    # echo 8:0 0 > blkio.bfq.weight_device
	    # cat blkio.bfq.weight_device
	    dev     weight
	    8:16    300

  blkio.time
	  Дисковое время, выделенное cgroup для каждого устройства, в миллисекундах.
	  Первые два поля задают major- и minor-номер устройства, а третье поле задаёт
	  дисковое время, выделенное группе, в миллисекундах.

  blkio.sectors
	  Количество секторов, переданных группой на диск/с диска. Первые два поля задают
	  major- и minor-номер устройства, а третье поле задаёт количество секторов,
	  переданных группой на устройство/с устройства.

  blkio.io_service_bytes
	  Количество байт, переданных группой на диск/с диска. Они дополнительно
	  разделяются по типу операции — чтение или запись, синхронная или асинхронная.
	  Первые два поля задают major- и minor-номер устройства, третье поле задаёт тип
	  операции, а четвёртое поле задаёт количество байт.

  blkio.io_serviced
	  Количество операций ввода-вывода (bio), выданных группой на диск. Они
	  дополнительно разделяются по типу операции — чтение или запись, синхронная или
	  асинхронная. Первые два поля задают major- и minor-номер устройства, третье поле
	  задаёт тип операции, а четвёртое поле задаёт количество операций ввода-вывода.

  blkio.io_service_time
	  Суммарное время между диспетчеризацией запроса и завершением запроса для
	  операций ввода-вывода, выполненных этой cgroup. Оно указано в наносекундах, чтобы
	  быть осмысленным и для флеш-устройств. Для устройств с глубиной очереди 1 это
	  время представляет фактическое время обслуживания. При queue_depth > 1 это уже
	  не так, поскольку запросы могут обслуживаться не по порядку. Это может привести к
	  тому, что время обслуживания данной операции ввода-вывода будет включать время
	  обслуживания нескольких операций ввода-вывода при обслуживании не по порядку, что
	  может приводить к суммарному io_service_time > фактически прошедшего времени. Это
	  время дополнительно разделяется по типу операции — чтение или запись, синхронная
	  или асинхронная. Первые два поля задают major- и minor-номер устройства, третье
	  поле задаёт тип операции, а четвёртое поле задаёт io_service_time в нс.

  blkio.io_wait_time
	  Суммарное время, которое операции ввода-вывода этой cgroup провели в ожидании
	  обслуживания в очередях планировщика. Оно может быть больше общего прошедшего
	  времени, поскольку это совокупное io_wait_time для всех операций ввода-вывода.
	  Это не мера общего времени ожидания cgroup, а мера wait_time для её отдельных
	  операций ввода-вывода. Для устройств с queue_depth > 1 эта метрика не включает
	  время, проведённое в ожидании обслуживания после того, как операция ввода-вывода
	  была направлена на устройство, но до того, как она фактически будет обслужена
	  (здесь может быть временна́я задержка из-за переупорядочивания запросов
	  устройством). Оно указано в наносекундах, чтобы быть осмысленным и для
	  флеш-устройств. Это время дополнительно разделяется по типу операции — чтение или
	  запись, синхронная или асинхронная. Первые два поля задают major- и minor-номер
	  устройства, третье поле задаёт тип операции, а четвёртое поле задаёт io_wait_time
	  в нс.

  blkio.io_merged
	  Общее количество bio/запросов, слитых с запросами, принадлежащими этой cgroup.
	  Оно дополнительно разделяется по типу операции — чтение или запись, синхронная
	  или асинхронная.

  blkio.io_queued
	  Общее количество запросов, поставленных в очередь в любой данный момент для этой
	  cgroup. Оно дополнительно разделяется по типу операции — чтение или запись,
	  синхронная или асинхронная.

  blkio.avg_queue_size
	  Средство отладки, включается только при CONFIG_BFQ_CGROUP_DEBUG=y.
	  Средний размер очереди для этой cgroup за всё время существования этой cgroup.
	  Выборки размера очереди берутся каждый раз, когда одна из очередей этой cgroup
	  получает временной квант.

  blkio.group_wait_time
	  Средство отладки, включается только при CONFIG_BFQ_CGROUP_DEBUG=y.
	  Это время, которое cgroup пришлось ждать с момента, как она стала занятой
	  (т. е. перешла от 0 к 1 запросу в очереди), до получения временного кванта для
	  одной из своих очередей. Это отличается от io_wait_time, которое представляет
	  совокупную сумму времени, проведённого каждой операцией ввода-вывода этой cgroup в
	  ожидании в очереди планировщика. Оно указано в наносекундах. Если это значение
	  читается, когда cgroup находится в состоянии ожидания (временного кванта),
	  статистика сообщит только group_wait_time, накопленное до последнего момента
	  получения временного кванта, и не будет включать текущую дельту.

  blkio.empty_time
	  Средство отладки, включается только при CONFIG_BFQ_CGROUP_DEBUG=y.
	  Это время, которое cgroup проводит без каких-либо ожидающих запросов, не будучи
	  обслуживаемой, т. е. оно не включает время, проведённое в простое (idling) для
	  одной из очередей cgroup. Оно указано в наносекундах. Если это значение читается,
	  когда cgroup находится в пустом состоянии, статистика сообщит только empty_time,
	  накопленное до последнего момента, когда у неё был ожидающий запрос, и не будет
	  включать текущую дельту.

  blkio.idle_time
	  Средство отладки, включается только при CONFIG_BFQ_CGROUP_DEBUG=y.
	  Это время, проведённое планировщиком ввода-вывода в простое (idling) для данной
	  cgroup в ожидании лучшего запроса, чем существующие из других очередей/cgroups.
	  Оно указано в наносекундах. Если это значение читается, когда cgroup находится в
	  состоянии простоя, статистика сообщит только idle_time, накопленное до последнего
	  периода простоя, и не будет включать текущую дельту.

  blkio.dequeue
	  Средство отладки, включается только при CONFIG_BFQ_CGROUP_DEBUG=y. Оно
	  даёт статистику о том, сколько раз группа была удалена из очереди дерева
	  обслуживания устройства. Первые два поля задают major- и minor-номер устройства,
	  а третье поле задаёт количество раз, когда группа была удалена из очереди
	  конкретного устройства.

  blkio.*_recursive
	  Рекурсивная версия различных статистик. Эти файлы показывают
          ту же информацию, что и их нерекурсивные аналоги, но
          включают статистику из всех cgroups-потомков.

Файлы политики троттлинга / верхнего предела
--------------------------------------------
  blkio.throttle.read_bps_device
	  Задаёт верхний предел скорости чтения (READ) с устройства. Скорость
	  ввода-вывода задаётся в байтах в секунду. Правила задаются для каждого
	  устройства. Формат следующий::

	    echo "<major>:<minor>  <rate_bytes_per_second>" > /cgrp/blkio.throttle.read_bps_device

  blkio.throttle.write_bps_device
	  Задаёт верхний предел скорости записи (WRITE) на устройство. Скорость
	  ввода-вывода задаётся в байтах в секунду. Правила задаются для каждого
	  устройства. Формат следующий::

	    echo "<major>:<minor>  <rate_bytes_per_second>" > /cgrp/blkio.throttle.write_bps_device

  blkio.throttle.read_iops_device
	  Задаёт верхний предел скорости чтения (READ) с устройства. Скорость
	  ввода-вывода задаётся в операциях ввода-вывода в секунду. Правила задаются для
	  каждого устройства. Формат следующий::

	   echo "<major>:<minor>  <rate_io_per_second>" > /cgrp/blkio.throttle.read_iops_device

  blkio.throttle.write_iops_device
	  Задаёт верхний предел скорости записи (WRITE) на устройство. Скорость
	  ввода-вывода задаётся в операциях ввода-вывода в секунду. Правила задаются для
	  каждого устройства. Формат следующий::

	    echo "<major>:<minor>  <rate_io_per_second>" > /cgrp/blkio.throttle.write_iops_device

          Примечание: если для устройства заданы правила и по BW, и по IOPS, то
          ввод-вывод подчиняется обоим ограничениям.

  blkio.throttle.io_serviced
	  Количество операций ввода-вывода (bio), выданных группой на диск. Они
	  дополнительно разделяются по типу операции — чтение или запись, синхронная или
	  асинхронная. Первые два поля задают major- и minor-номер устройства, третье поле
	  задаёт тип операции, а четвёртое поле задаёт количество операций ввода-вывода.

  blkio.throttle.io_service_bytes
	  Количество байт, переданных группой на диск/с диска. Они дополнительно
	  разделяются по типу операции — чтение или запись, синхронная или асинхронная.
	  Первые два поля задают major- и minor-номер устройства, третье поле задаёт тип
	  операции, а четвёртое поле задаёт количество байт.

Общие файлы для различных политик
---------------------------------
  blkio.reset_stats
	  Запись целого числа (int) в этот файл приведёт к сбросу всей статистики
	  для данной cgroup.
