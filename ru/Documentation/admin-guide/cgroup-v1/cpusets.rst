1. Cpusets
==========

1.1 Что такое cpusets?
----------------------

Cpusets предоставляют механизм для назначения набора CPU и узлов памяти
(Memory Nodes) набору задач. В этом документе под «узлом памяти» (Memory
Node) понимается включённый (on-line) узел, содержащий память.

Cpusets ограничивают размещение CPU и памяти для задач только теми
ресурсами, которые входят в текущий cpuset задачи. Они образуют вложенную
иерархию, видимую в виртуальной файловой системе. Это те необходимые
точки подключения (hooks), помимо уже имеющихся, которые требуются для
управления динамическим размещением заданий на больших системах.

Cpusets используют общую подсистему cgroup, описанную в
Documentation/admin-guide/cgroup-v1/cgroups.rst.

Запросы задачи на включение CPU в свою маску привязки к CPU (CPU affinity
mask) с помощью системного вызова sched_setaffinity(2), а также на
включение узлов памяти в свою политику памяти с помощью системных вызовов
mbind(2) и set_mempolicy(2), оба фильтруются через cpuset этой задачи,
отсекая любые CPU или узлы памяти, не входящие в этот cpuset. Планировщик
не будет планировать задачу на CPU, не разрешённом в её векторе
cpus_allowed, а страничный аллокатор ядра не будет выделять страницу на
узле, не разрешённом в векторе mems_allowed запрашивающей задачи.

Код пространства пользователя может создавать и уничтожать cpusets по
имени в виртуальной файловой системе cgroup, управлять атрибутами и
правами доступа этих cpusets, а также тем, какие CPU и узлы памяти
назначены каждому cpuset, задавать и запрашивать, какому cpuset назначена
задача, и перечислять pid задач, назначенных cpuset.


1.2 Зачем нужны cpusets?
------------------------

Управление большими вычислительными системами, имеющими множество
процессоров (CPU), сложные иерархии кэшей памяти и несколько узлов памяти
с неоднородным временем доступа (NUMA), создаёт дополнительные сложности
для эффективного планирования и размещения памяти процессов.

Зачастую системы более скромного размера могут работать с приемлемой
эффективностью просто за счёт того, что операционная система автоматически
распределяет доступные ресурсы CPU и памяти между запрашивающими их
задачами.

Но более крупные системы, которые в большей степени выигрывают от
тщательного размещения процессоров и памяти для сокращения времени
доступа к памяти и конкуренции за неё, и которые обычно представляют
собой более крупные вложения для заказчика, могут выиграть от явного
размещения заданий на подмножествах системы подходящего размера.

Это может быть особенно ценно в следующих случаях:

    * веб-серверы, выполняющие несколько экземпляров одного и того же
      веб-приложения;
    * серверы, выполняющие разные приложения (например, веб-сервер и
      базу данных); или
    * NUMA-системы, выполняющие крупные HPC-приложения с высокими
      требованиями к производительности.

Эти подмножества, или «мягкие разделы» (soft partitions), должны иметь
возможность динамически корректироваться при изменении набора заданий,
не затрагивая другие одновременно выполняющиеся задания. Расположение
страниц выполняющихся заданий также может перемещаться при изменении
расположения памяти.

Патч cpuset для ядра предоставляет минимально необходимые механизмы ядра,
требуемые для эффективной реализации таких подмножеств. Он опирается на
существующие в ядре Linux средства размещения CPU и памяти, чтобы избежать
дополнительного влияния на критичный код планировщика или аллокатора
памяти.


1.3 Как реализованы cpusets?
----------------------------

Cpusets предоставляют механизм ядра Linux для ограничения того, какие CPU
и узлы памяти используются процессом или набором процессов.

В ядре Linux уже есть пара механизмов для указания того, на каких CPU
может планироваться задача (sched_setaffinity) и на каких узлах памяти
она может получать память (mbind, set_mempolicy).

Cpusets расширяют эти два механизма следующим образом:

 - Cpusets — это наборы разрешённых CPU и узлов памяти, известные ядру.
 - Каждая задача в системе привязана к cpuset через указатель в структуре
   задачи на структуру cgroup со счётчиком ссылок.
 - Вызовы sched_setaffinity фильтруются только до тех CPU, которые
   разрешены в cpuset этой задачи.
 - Вызовы mbind и set_mempolicy фильтруются только до тех узлов памяти,
   которые разрешены в cpuset этой задачи.
 - Корневой cpuset содержит все CPU и узлы памяти системы.
 - Для любого cpuset можно определить дочерние cpusets, содержащие
   подмножество ресурсов CPU и узлов памяти родителя.
 - Иерархию cpusets можно смонтировать в /dev/cpuset для просмотра и
   изменения из пространства пользователя.
 - cpuset может быть помечен как исключительный (exclusive), что
   гарантирует, что никакой другой cpuset (кроме прямых предков и
   потомков) не может содержать пересекающихся CPU или узлов памяти.
 - Можно перечислить все задачи (по pid), привязанные к любому cpuset.

Реализация cpusets требует нескольких простых точек подключения (hooks)
в остальной части ядра, ни одной из них — в критичных по
производительности путях:

 - в init/main.c — для инициализации корневого cpuset при загрузке
   системы.
 - в fork и exit — для привязки и отвязки задачи от её cpuset.
 - в sched_setaffinity — для маскирования запрошенных CPU тем, что
   разрешено в cpuset этой задачи.
 - в sched.c migrate_live_tasks() — для того, чтобы по возможности
   мигрировать задачи в пределах CPU, разрешённых их cpuset.
 - в системных вызовах mbind и set_mempolicy — для маскирования
   запрошенных узлов памяти тем, что разрешено в cpuset этой задачи.
 - в page_alloc.c — для ограничения памяти разрешёнными узлами.
 - в vmscan.c — для ограничения восстановления страниц текущим cpuset.

Чтобы включить просмотр и изменение cpusets, известных ядру в данный
момент, следует смонтировать файловую систему типа "cgroup". Для cpusets
не добавляется никаких новых системных вызовов — вся поддержка запроса и
изменения cpusets осуществляется через эту файловую систему cpuset.

В файл /proc/<pid>/status для каждой задачи добавлены четыре строки,
отображающие cpus_allowed задачи (на каких CPU она может планироваться)
и mems_allowed (на каких узлах памяти она может получать память) в двух
форматах, показанных в следующем примере::

  Cpus_allowed:   ffffffff,ffffffff,ffffffff,ffffffff
  Cpus_allowed_list:      0-127
  Mems_allowed:   ffffffff,ffffffff
  Mems_allowed_list:      0-63

Каждый cpuset представлен каталогом в файловой системе cgroup, содержащим
(помимо стандартных файлов cgroup) следующие файлы, описывающие этот
cpuset:

 - cpuset.cpus: список CPU в этом cpuset
 - cpuset.mems: список узлов памяти в этом cpuset
 - флаг cpuset.memory_migrate: если установлен, перемещать страницы на
   узлы cpuset
 - флаг cpuset.cpu_exclusive: является ли размещение CPU исключительным?
 - флаг cpuset.mem_exclusive: является ли размещение памяти
   исключительным?
 - флаг cpuset.mem_hardwall: является ли выделение памяти изолированным
   «жёсткой стеной» (hardwalled)
 - cpuset.memory_pressure: мера того, насколько велико давление пейджинга
   в cpuset
 - флаг cpuset.memory_spread_page: если установлен, равномерно
   распределять страничный кэш по разрешённым узлам
 - флаг cpuset.memory_spread_slab: УСТАРЕЛ. Не выполняет никакой функции.
 - флаг cpuset.sched_load_balance: если установлен, выполнять
   балансировку нагрузки в пределах CPU этого cpuset
 - cpuset.sched_relax_domain_level: диапазон поиска при миграции задач

Кроме того, только корневой cpuset имеет следующий файл:

 - флаг cpuset.memory_pressure_enabled: вычислять memory_pressure?

Новые cpusets создаются с помощью системного вызова mkdir или команды
оболочки. Свойства cpuset, такие как его флаги, разрешённые CPU и узлы
памяти, а также привязанные задачи, изменяются записью в соответствующий
файл в каталоге этого cpuset, как перечислено выше.

Именованная иерархическая структура вложенных cpusets позволяет разбивать
большую систему на вложенные, динамически изменяемые «мягкие разделы»
(soft-partitions).

Привязка каждой задачи к cpuset, автоматически наследуемая при fork
любыми потомками этой задачи, позволяет организовать рабочую нагрузку
системы в связанные наборы задач так, чтобы каждый набор был ограничен
использованием CPU и узлов памяти определённого cpuset. Задача может быть
повторно привязана к любому другому cpuset, если это разрешено правами
доступа к необходимым каталогам файловой системы cpuset.

Такое управление системой «в целом» гладко интегрируется с детальным
размещением, выполняемым для отдельных задач и областей памяти с помощью
системных вызовов sched_setaffinity, mbind и set_mempolicy.

К каждому cpuset применяются следующие правила:

 - Его CPU и узлы памяти должны быть подмножеством CPU и узлов памяти его
   родителя.
 - Он не может быть помечен исключительным, если только им не помечен его
   родитель.
 - Если его CPU или память исключительны, они не могут пересекаться ни с
   одним соседним (sibling) cpuset.

Эти правила и естественная иерархия cpusets обеспечивают эффективное
соблюдение гарантии исключительности без необходимости сканировать все
cpusets при каждом изменении любого из них, чтобы убедиться, что ничто не
пересекается с исключительным cpuset. Кроме того, использование
виртуальной файловой системы Linux (vfs) для представления иерархии cpuset
обеспечивает привычное пространство прав доступа и имён для cpusets при
минимуме дополнительного кода ядра.

Файлы cpus и mems в корневом (top_cpuset) cpuset доступны только для
чтения. Файл cpus автоматически отслеживает значение cpu_online_mask с
помощью уведомителя горячего подключения CPU (CPU hotplug notifier), а
файл mems автоматически отслеживает значение node_states[N_MEMORY] —
т. е. узлы с памятью — с помощью точки подключения
cpuset_track_online_nodes().

Файлы cpuset.effective_cpus и cpuset.effective_mems обычно являются
доступными только для чтения копиями файлов cpuset.cpus и cpuset.mems
соответственно. Если файловая система cgroup для cpuset смонтирована со
специальной опцией "cpuset_v2_mode", поведение этих файлов становится
похожим на поведение соответствующих файлов в cpuset v2. Иначе говоря,
события горячего подключения (hotplug) не будут изменять cpuset.cpus и
cpuset.mems. Эти события будут затрагивать только cpuset.effective_cpus и
cpuset.effective_mems, которые показывают фактические CPU и узлы памяти,
используемые этим cpuset в данный момент. Подробнее о поведении cpuset v2
см. Documentation/admin-guide/cgroup-v2.rst.


1.4 Что такое исключительные (exclusive) cpusets?
-------------------------------------------------

Если cpuset является исключительным по CPU или по памяти, то никакой
другой cpuset, кроме прямого предка или потомка, не может разделять с ним
ни одного из тех же CPU или узлов памяти.

cpuset, для которого установлен cpuset.mem_exclusive *или*
cpuset.mem_hardwall, является «изолированным жёсткой стеной» (hardwalled),
т. е. он ограничивает выделения памяти ядром для страниц, буферов и других
данных, обычно разделяемых ядром между несколькими пользователями. Все
cpusets, независимо от того, изолированы они жёсткой стеной или нет,
ограничивают выделения памяти для пространства пользователя. Это позволяет
настроить систему так, чтобы несколько независимых заданий могли разделять
общие данные ядра, такие как страницы файловой системы, изолируя при этом
пользовательские выделения памяти каждого задания в его собственном
cpuset. Чтобы сделать это, постройте большой mem_exclusive cpuset для
размещения всех заданий и постройте дочерние, не-mem_exclusive cpusets для
каждого отдельного задания. Лишь небольшому объёму типичной памяти ядра,
такому как запросы от обработчиков прерываний, разрешено выходить за
пределы даже mem_exclusive cpuset.


1.5 Что такое memory_pressure?
------------------------------
memory_pressure для cpuset предоставляет простую метрику в расчёте на один
cpuset для скорости, с которой задачи в cpuset пытаются освободить
используемую память на узлах cpuset, чтобы удовлетворить дополнительные
запросы памяти.

Это позволяет пакетным менеджерам (batch managers), отслеживающим задания,
выполняющиеся в выделенных cpusets, эффективно определять, какой уровень
давления на память создаёт это задание.

Это полезно как на жёстко управляемых системах, выполняющих широкий набор
поданных заданий, которые могут принять решение завершить или
перерасставить приоритеты заданиям, пытающимся использовать больше памяти,
чем разрешено на назначенных им узлах, так и при тесно связанных,
долго работающих, массово параллельных научных вычислительных заданиях,
которые радикально не достигнут требуемых целей производительности, если
начнут использовать больше памяти, чем им разрешено.

Этот механизм предоставляет очень экономичный способ для пакетного
менеджера отслеживать cpuset на предмет признаков давления на память.
Решать, что с этим делать, и предпринимать действия — это уже задача
пакетного менеджера или другого пользовательского кода.

==>
    Если эта функциональность не включена записью "1" в специальный файл
    /dev/cpuset/memory_pressure_enabled, то точка подключения в коде
    перебалансировки __alloc_pages() для этой метрики сводится просто к
    тому, чтобы заметить, что флаг cpuset_memory_pressure_enabled равен
    нулю. Поэтому метрику будут вычислять только системы, в которых эта
    функциональность включена.

Почему скользящее среднее в расчёте на один cpuset:

    Поскольку этот измеритель ведётся в расчёте на один cpuset, а не на
    задачу или mm, нагрузка на систему, создаваемая пакетным планировщиком
    при отслеживании этой метрики, резко снижается на больших системах,
    так как при каждом наборе запросов можно избежать сканирования списка
    задач.

    Поскольку этот измеритель является скользящим средним, а не
    накапливающим счётчиком, пакетный планировщик может обнаружить
    давление на память одним чтением, вместо того чтобы считывать и
    накапливать результаты в течение некоторого периода времени.

    Поскольку этот измеритель ведётся в расчёте на один cpuset, а не на
    задачу или mm, пакетный планировщик может получить ключевую информацию,
    давление на память в cpuset, одним чтением, вместо того чтобы
    запрашивать и накапливать результаты по всему (динамически
    изменяющемуся) набору задач в cpuset.

Для каждого cpuset ведётся простой цифровой фильтр (требующий спинлока
(spinlock) и 3 слов данных на один cpuset), который обновляется любой
задачей, привязанной к этому cpuset, если она входит в код синхронного
(прямого) высвобождения страниц (page reclaim).

Файл, ведущийся в расчёте на один cpuset, предоставляет целое число,
представляющее недавнюю (с периодом полураспада в 10 секунд) скорость
прямых высвобождений страниц, вызванных задачами в cpuset, в единицах
попыток высвобождения в секунду, умноженных на 1000.


1.6 Что такое распределение памяти (memory spread)?
---------------------------------------------------
Для каждого cpuset существует два файла с булевыми флагами, которые
управляют тем, где ядро выделяет страницы для буферов файловой системы и
связанных с ними структур данных ядра. Они называются
'cpuset.memory_spread_page' и 'cpuset.memory_spread_slab'.

Если установлен файл булева флага 'cpuset.memory_spread_page' для cpuset,
то ядро будет равномерно распределять буферы файловой системы (страничный
кэш) по всем узлам, которые разрешено использовать задаче, вызвавшей
страничный сбой (faulting task), вместо того чтобы предпочтительно
размещать эти страницы на узле, где выполняется задача.

Если установлен файл булева флага 'cpuset.memory_spread_slab' для cpuset,
то ядро будет равномерно распределять некоторые slab-кэши, связанные с
файловой системой, такие как кэши для inode и dentry, по всем узлам,
которые разрешено использовать задаче, вызвавшей страничный сбой, вместо
того чтобы предпочтительно размещать эти страницы на узле, где выполняется
задача.

Установка этих флагов не влияет на страницы сегмента анонимных данных или
сегмента стека задачи.

По умолчанию оба вида распределения памяти выключены, и страницы памяти
выделяются на узле, локальном для того места, где выполняется задача, за
исключением, возможно, изменений, вносимых NUMA-политикой памяти (NUMA
mempolicy) задачи или конфигурацией cpuset, при условии что доступно
достаточно свободных страниц памяти.

При создании новых cpusets они наследуют настройки распределения памяти
своего родителя.

Установка распределения памяти приводит к тому, что выделения для
затрагиваемого страничного или slab-кэша игнорируют NUMA-политику памяти
задачи и вместо этого распределяются. Задачи, использующие вызовы mbind()
или set_mempolicy() для установки NUMA-политик памяти, не заметят никаких
изменений в этих вызовах в результате настроек распределения памяти
содержащей их задачи. Если распределение памяти выключено, то текущая
заданная NUMA-политика памяти снова применяется к выделениям страниц
памяти.

Оба файла, 'cpuset.memory_spread_page' и 'cpuset.memory_spread_slab',
являются файлами булевых флагов. По умолчанию они содержат "0", что
означает, что для этого cpuset функциональность выключена. Если в такой
файл записать "1", то это включает соответствующую функциональность.

Реализация проста.

Установка флага 'cpuset.memory_spread_page' включает флаг
PFA_SPREAD_PAGE на уровне процесса для каждой задачи, которая находится в
этом cpuset или впоследствии присоединяется к этому cpuset. Вызовы
выделения страниц для страничного кэша изменены так, чтобы выполнять
встроенную (inline) проверку этого флага задачи PFA_SPREAD_PAGE, и если он
установлен, вызов новой подпрограммы cpuset_mem_spread_node() возвращает
узел, предпочтительный для выделения.

Аналогично, установка 'cpuset.memory_spread_slab' включает флаг
PFA_SPREAD_SLAB, и соответствующим образом помеченные slab-кэши будут
выделять страницы с узла, возвращённого cpuset_mem_spread_node().

Подпрограмма cpuset_mem_spread_node() также проста. Она использует
значение ротора cpuset_mem_spread_rotor на уровне задачи, чтобы выбрать
следующий узел в mems_allowed текущей задачи, предпочтительный для
выделения.

Эта политика размещения памяти также известна (в других контекстах) как
round-robin или interleave (циклическое или чередующееся размещение).

Эта политика может дать существенные улучшения для заданий, которым нужно
размещать локальные данные потока (thread local data) на соответствующем
узле, но которым нужно обращаться к большим наборам данных файловой
системы, которые требуется распределить по нескольким узлам в cpuset
задания, чтобы они поместились. Без этой политики, особенно для заданий, в
которых один поток мог бы считывать набор данных, выделение памяти по
узлам в cpuset задания может стать очень неравномерным.

1.7 Что такое sched_load_balance?
---------------------------------

Планировщик ядра (kernel/sched/core.c) автоматически выполняет
балансировку нагрузки задач. Если один CPU недогружен, выполняющийся на
этом CPU код ядра будет искать задачи на других, более перегруженных CPU
и перемещать эти задачи к себе, в рамках ограничений таких механизмов
размещения, как cpusets и sched_setaffinity.

Алгоритмическая стоимость балансировки нагрузки и её влияние на ключевые
разделяемые структуры данных ядра, такие как список задач, растёт более
чем линейно с числом балансируемых CPU. Поэтому планировщик имеет
поддержку разбиения CPU системы на несколько доменов планирования (sched
domains) так, чтобы он выполнял балансировку нагрузки только в пределах
каждого домена планирования. Каждый домен планирования покрывает некоторое
подмножество CPU в системе; никакие два домена планирования не
пересекаются; некоторые CPU могут не входить ни в один домен планирования
и, следовательно, не будут балансироваться по нагрузке.

Проще говоря, балансировка между двумя меньшими доменами планирования
обходится дешевле, чем между одним большим, но это означает, что перегрузки
в одном из двух доменов не будут балансироваться по нагрузке в другой.

По умолчанию существует один домен планирования, покрывающий все CPU,
включая те, что помечены как изолированные с помощью аргумента
"isolcpus=" во время загрузки ядра. Однако изолированные CPU не будут
участвовать в балансировке нагрузки и не будут иметь выполняющихся на них
задач, если только они не назначены явно.

Эта используемая по умолчанию балансировка нагрузки по всем CPU плохо
подходит для следующих двух ситуаций:

 1) На больших системах балансировка нагрузки по многим CPU обходится
    дорого. Если система управляется с помощью cpusets для размещения
    независимых заданий на отдельных наборах CPU, полная балансировка
    нагрузки не нужна.
 2) Системам, поддерживающим реальное время (realtime) на некоторых CPU,
    необходимо минимизировать системные накладные расходы на этих CPU,
    включая избегание балансировки нагрузки задач, если она не нужна.

Когда флаг "cpuset.sched_load_balance" на уровне cpuset включён (настройка
по умолчанию), он запрашивает, чтобы все CPU, разрешённые в 'cpuset.cpus'
этого cpuset, содержались в одном домене планирования, обеспечивая, что
балансировка нагрузки может переместить задачу (не закреплённую иным
образом, как, например, с помощью sched_setaffinity) с любого CPU в этом
cpuset на любой другой.

Когда флаг "cpuset.sched_load_balance" на уровне cpuset выключен,
планировщик будет избегать балансировки нагрузки между CPU в этом cpuset,
--за исключением-- случаев, когда это необходимо из-за того, что у
некоторого пересекающегося cpuset включён "sched_load_balance".

Так, например, если у верхнего cpuset включён флаг
"cpuset.sched_load_balance", то у планировщика будет один домен
планирования, покрывающий все CPU, и установка флага
"cpuset.sched_load_balance" в любых других cpusets не будет иметь
значения, поскольку мы уже выполняем полную балансировку нагрузки.

Поэтому в двух вышеуказанных ситуациях флаг
"cpuset.sched_load_balance" верхнего cpuset следует выключить, и лишь у
некоторых из меньших, дочерних cpusets этот флаг должен быть включён.

При этом обычно нежелательно оставлять в верхнем cpuset какие-либо
незакреплённые задачи, которые могли бы использовать нетривиальные объёмы
CPU, так как такие задачи могут оказаться искусственно ограничены неким
подмножеством CPU в зависимости от особенностей установки этого флага в
дочерних cpusets. Даже если такая задача могла бы использовать свободные
циклы CPU на каких-то других CPU, планировщик ядра может не рассмотреть
возможность балансировки этой задачи на тот недоиспользуемый CPU.

Разумеется, задачи, закреплённые за определённым CPU, можно оставить в
cpuset, который выключает "cpuset.sched_load_balance", так как эти задачи
всё равно никуда не денутся.

Здесь есть несоответствие импедансов (impedance mismatch) между cpusets и
доменами планирования. Cpusets иерархичны и вкладываются. Домены
планирования плоские; они не пересекаются, и каждый CPU входит не более
чем в один домен планирования.

Домены планирования должны быть плоскими, потому что балансировка нагрузки
по частично пересекающимся наборам CPU рисковала бы породить нестабильную
динамику, которая была бы за пределами нашего понимания. Поэтому, если
каждый из двух частично пересекающихся cpusets включает флаг
'cpuset.sched_load_balance', то мы формируем один домен планирования,
являющийся надмножеством обоих. Мы не будем перемещать задачу на CPU вне
её cpuset, но код балансировки нагрузки планировщика может зря потратить
некоторые вычислительные циклы, рассматривая такую возможность.

Из-за этого несоответствия нет простого взаимно однозначного соответствия
между тем, у каких cpusets включён флаг "cpuset.sched_load_balance", и
конфигурацией доменов планирования. Если cpuset включает флаг, он получит
балансировку по всем своим CPU, но если он выключает флаг, отсутствие
балансировки нагрузки будет гарантировано только в том случае, если флаг
не включён ни у одного другого пересекающегося cpuset.

Если у двух cpusets разрешённые 'cpuset.cpus' частично пересекаются, и
лишь у одного из них включён этот флаг, то у другого его задачи могут
оказаться лишь частично сбалансированными по нагрузке, только на
пересекающихся CPU. Это просто общий случай примера с top_cpuset,
приведённого несколькими абзацами выше. В общем случае, как и в случае с
верхним cpuset, не оставляйте в таких частично сбалансированных по
нагрузке cpusets задачи, которые могли бы использовать нетривиальные
объёмы CPU, так как они могут оказаться искусственно ограничены неким
подмножеством разрешённых им CPU из-за отсутствия балансировки нагрузки на
другие CPU.

CPU в "cpuset.isolcpus" были исключены из балансировки нагрузки опцией
загрузки ядра isolcpus= и никогда не будут балансироваться по нагрузке
независимо от значения "cpuset.sched_load_balance" в любом cpuset.

1.7.1 Детали реализации sched_load_balance.
-------------------------------------------

Флаг 'cpuset.sched_load_balance' на уровне cpuset по умолчанию включён (в
отличие от большинства флагов cpuset). Когда он включён для cpuset, ядро
обеспечит, что оно может выполнять балансировку нагрузки по всем CPU в
этом cpuset (убеждается, что все CPU в cpus_allowed этого cpuset находятся
в одном домене планирования).

Если у двух пересекающихся cpusets обоих включён 'cpuset.sched_load_balance',
то они будут (должны быть) оба в одном домене планирования.

Если, как это происходит по умолчанию, у верхнего cpuset включён
'cpuset.sched_load_balance', то из вышесказанного следует, что существует
единственный домен планирования, покрывающий всю систему, независимо от
любых других настроек cpuset.

Ядро обязуется перед пространством пользователя избегать балансировки
нагрузки там, где может. Оно выберет настолько мелкозернистое разбиение
доменов планирования, насколько сможет, при этом по-прежнему обеспечивая
балансировку нагрузки для любого набора CPU, разрешённых cpuset, у
которого включён 'cpuset.sched_load_balance'.

Внутренний интерфейс ядра между cpuset и планировщиком передаёт из кода
cpuset в код планировщика разбиение балансируемых по нагрузке CPU в
системе. Это разбиение представляет собой набор подмножеств
(представленных в виде массива struct cpumask) CPU, попарно
непересекающихся, которые покрывают все CPU, которые должны
балансироваться по нагрузке.

Код cpuset строит новое такое разбиение и передаёт его коду настройки
доменов планирования планировщика, чтобы домены планирования были
перестроены по мере необходимости, всякий раз когда:

 - изменяется флаг 'cpuset.sched_load_balance' у cpuset с непустым набором
   CPU,
 - или CPU добавляются или удаляются из cpuset с включённым этим флагом,
 - или изменяется значение 'cpuset.sched_relax_domain_level' у cpuset с
   непустым набором CPU и с включённым этим флагом,
 - или удаляется cpuset с непустым набором CPU и с включённым этим флагом,
 - или CPU переводится в offline/online.

Это разбиение в точности определяет, какие домены планирования должен
настроить планировщик — по одному домену планирования для каждого элемента
(struct cpumask) в разбиении.

Планировщик запоминает текущие активные разбиения доменов планирования.
Когда подпрограмма планировщика partition_sched_domains() вызывается из
кода cpuset для обновления этих доменов планирования, она сравнивает новое
запрошенное разбиение с текущим и обновляет свои домены планирования,
удаляя старые и добавляя новые, для каждого изменения.


1.8 Что такое sched_relax_domain_level?
---------------------------------------

В домене планирования планировщик мигрирует задачи 2 способами:
периодическая балансировка нагрузки по тику (tick) и во время некоторых
событий планирования.

Когда задача пробуждается, планировщик пытается переместить задачу на
простаивающий CPU. Например, если задача A, выполняющаяся на CPU X,
активирует другую задачу B на том же CPU X, и если CPU Y является
соседним (sibling) для X и простаивает, то планировщик мигрирует задачу B
на CPU Y, чтобы задача B могла стартовать на CPU Y, не дожидаясь задачи A
на CPU X.

И если у CPU в его очереди выполнения (runqueue) заканчиваются задачи,
этот CPU пытается перетянуть лишние задачи с других занятых CPU, чтобы
помочь им, прежде чем сам перейдёт в простой.

Разумеется, поиск перемещаемых задач и/или простаивающих CPU стоит
некоторых затрат на поиск, и планировщик может не каждый раз искать по
всем CPU в домене. Фактически на некоторых архитектурах диапазоны поиска
при событиях ограничены тем же сокетом или узлом, где расположен CPU, в то
время как балансировка нагрузки по тику ищет по всем.

Например, предположим, что CPU Z находится относительно далеко от CPU X.
Даже если CPU Z простаивает, в то время как CPU X и соседние заняты,
планировщик не может мигрировать пробуждённую задачу B с X на Z, поскольку
он выходит за пределы своего диапазона поиска. В результате задаче B на
CPU X нужно ждать задачу A или ждать балансировки нагрузки на следующем
тике. Для некоторых приложений в особых ситуациях ожидание в 1 тик может
быть слишком долгим.

Файл 'cpuset.sched_relax_domain_level' позволяет вам запрашивать изменение
этого диапазона поиска так, как вам нужно. Этот файл принимает значение
типа int, которое указывает размер диапазона поиска в уровнях примерно
следующим образом, в противном случае начальное значение -1 указывает,
что у cpuset нет запроса.

====== ===========================================================
  -1   нет запроса. использовать системное значение по умолчанию или
       следовать запросу других.
   0   не искать.
   1   искать среди соседних (hyperthreads в ядре).
   2   искать ядра в пакете (package).
   3   искать cpu в узле [= по всей системе на не-NUMA системе]
   4   искать узлы в группе узлов [на NUMA-системе]
   5   искать по всей системе [на NUMA-системе]
====== ===========================================================

Не все уровни могут присутствовать, и значения могут меняться в
зависимости от архитектуры системы и конфигурации ядра. Подробности,
зависящие от системы, см. в
/sys/kernel/debug/sched/domains/cpu*/domain*/.

Системное значение по умолчанию зависит от архитектуры. Системное значение
по умолчанию можно изменить с помощью параметра загрузки
relax_domain_level=.

Этот файл задаётся для каждого cpuset и влияет на домен планирования,
которому принадлежит cpuset. Поэтому, если флаг
'cpuset.sched_load_balance' у cpuset выключен, то
'cpuset.sched_relax_domain_level' не имеет эффекта, поскольку нет домена
планирования, которому принадлежит cpuset.

Если несколько cpusets пересекаются и, следовательно, образуют один домен
планирования, используется наибольшее значение среди них. Будьте
внимательны: если один запрашивает 0, а другие — -1, то используется 0.

Обратите внимание, что изменение этого файла будет иметь как хорошие, так
и плохие последствия, и приемлемо это или нет — зависит от вашей ситуации.
Не изменяйте этот файл, если не уверены.

Если ваша ситуация такова:

 - Стоимость миграции между каждым CPU можно считать достаточно малой (для
   вас) благодаря поведению вашего особого приложения или особой
   аппаратной поддержке кэша CPU и т. п.
 - Стоимость поиска не оказывает влияния (для вас), или вы можете сделать
   стоимость поиска достаточно малой, управляя cpuset так, чтобы он был
   компактным, и т. п.
 - Требуется малая задержка (latency), даже если это жертвует частотой
   попаданий в кэш и т. п.,
   тогда увеличение 'sched_relax_domain_level' принесёт вам пользу.


1.9 Как использовать cpusets?
-----------------------------

Чтобы минимизировать влияние cpusets на критичный код ядра, такой как
планировщик, и в силу того факта, что ядро не поддерживает напрямую
обновление одной задачей размещения памяти другой задачи, влияние на
задачу изменения размещения её CPU или узлов памяти в cpuset, либо
изменения того, к какому cpuset привязана задача, является тонким.

Если у cpuset изменены его узлы памяти, то для каждой задачи, привязанной
к этому cpuset, в следующий раз, когда ядро попытается выделить страницу
памяти для этой задачи, ядро заметит изменение в cpuset задачи и обновит
её размещение памяти на уровне задачи так, чтобы оно оставалось в пределах
размещения памяти нового cpuset. Если задача использовала политику памяти
MPOL_BIND, и узлы, к которым она была привязана, пересекаются с её новым
cpuset, то задача продолжит использовать то подмножество узлов MPOL_BIND,
которое по-прежнему разрешено в новом cpuset. Если задача использовала
MPOL_BIND и теперь ни один из её узлов MPOL_BIND не разрешён в новом
cpuset, то задача будет по существу рассматриваться так, как если бы она
была привязана через MPOL_BIND к новому cpuset (несмотря на то, что её
NUMA-размещение, запрашиваемое через get_mempolicy(), не меняется). Если
задача перемещается из одного cpuset в другой, то ядро скорректирует
размещение памяти задачи, как описано выше, в следующий раз, когда ядро
попытается выделить страницу памяти для этой задачи.

Если у cpuset изменён его 'cpuset.cpus', то у каждой задачи в этом cpuset
немедленно изменится её разрешённое размещение CPU. Аналогично, если pid
задачи записан в файл 'tasks' другого cpuset, то её разрешённое размещение
CPU изменяется немедленно. Если такая задача была привязана к некоторому
подмножеству своего cpuset с помощью вызова sched_setaffinity(), задаче
будет разрешено выполняться на любом CPU, разрешённом в её новом cpuset,
сводя на нет эффект предшествующего вызова sched_setaffinity().

В итоге размещение памяти задачи, чей cpuset изменён, обновляется ядром
при следующем выделении страницы для этой задачи, а размещение процессора
обновляется немедленно.

Обычно после того, как страница выделена (ей дана физическая страница
основной памяти), эта страница остаётся на том узле, на котором была
выделена, до тех пор, пока остаётся выделенной, даже если впоследствии
изменяется политика размещения памяти cpuset 'cpuset.mems'. Если файл
флага cpuset 'cpuset.memory_migrate' установлен в true, то при привязке
задач к этому cpuset любые страницы, которые задача имела выделенными для
себя на узлах в её предыдущем cpuset, мигрируются в новый cpuset задачи.
Относительное размещение страницы в пределах cpuset по возможности
сохраняется во время этих операций миграции. Например, если страница была
на втором допустимом узле предыдущего cpuset, то страница будет размещена
на втором допустимом узле нового cpuset.

Также, если 'cpuset.memory_migrate' установлен в true, то при изменении
файла 'cpuset.mems' этого cpuset страницы, выделенные задачам в этом
cpuset, которые были на узлах в предыдущей настройке 'cpuset.mems', будут
перемещены на узлы в новой настройке 'mems'. Страницы, которые не были в
предыдущем cpuset задачи или в предыдущей настройке 'cpuset.mems' cpuset,
перемещены не будут.

Из вышесказанного есть исключение. Если функциональность горячего
подключения (hotplug) используется для удаления всех CPU, в данный момент
назначенных cpuset, то все задачи в этом cpuset будут перемещены к
ближайшему предку с непустым набором CPU. Но перемещение некоторых (или
всех) задач может завершиться неудачей, если cpuset связан с другой
подсистемой cgroup, имеющей некоторые ограничения на привязку задач. В
этом случае неудачи такие задачи останутся в исходном cpuset, и ядро
автоматически обновит их cpus_allowed, чтобы разрешить все включённые
(online) CPU. Когда станет доступна функциональность горячего подключения
памяти для удаления узлов памяти, ожидается, что аналогичное исключение
будет применяться и там. В целом ядро предпочитает нарушить размещение
cpuset, нежели обречь на голодание задачу, у которой все её разрешённые
CPU или узлы памяти были отключены (taken offline).

Из вышесказанного есть второе исключение. Запросы GFP_ATOMIC — это
внутренние выделения ядра, которые должны быть удовлетворены немедленно.
Ядро может отбросить какой-то запрос, в редких случаях даже вызвать
panic, если выделение GFP_ATOMIC не удаётся. Если запрос не может быть
удовлетворён в пределах cpuset текущей задачи, то мы ослабляем cpuset и
ищем память везде, где можем её найти. Лучше нарушить cpuset, чем
создавать стресс для ядра.

Чтобы запустить новое задание, которое должно содержаться в cpuset,
выполняются следующие шаги:

 1) mkdir /sys/fs/cgroup/cpuset
 2) mount -t cgroup -ocpuset cpuset /sys/fs/cgroup/cpuset
 3) Создайте новый cpuset, выполняя mkdir и write (или echo) в
    виртуальной файловой системе /sys/fs/cgroup/cpuset.
 4) Запустите задачу, которая будет «отцом-основателем» нового задания.
 5) Привяжите эту задачу к новому cpuset, записав её pid в файл tasks
    этого cpuset в /sys/fs/cgroup/cpuset.
 6) Выполните fork, exec или clone задач задания из этой задачи
    отца-основателя.

Например, следующая последовательность команд настроит cpuset с именем
"Charlie", содержащий только CPU 2 и 3 и узел памяти 1, а затем запустит
вложенную оболочку 'sh' в этом cpuset::

  mount -t cgroup -ocpuset cpuset /sys/fs/cgroup/cpuset
  cd /sys/fs/cgroup/cpuset
  mkdir Charlie
  cd Charlie
  /bin/echo 2-3 > cpuset.cpus
  /bin/echo 1 > cpuset.mems
  /bin/echo $$ > tasks
  sh
  # The subshell 'sh' is now running in cpuset Charlie
  # The next line should display '/Charlie'
  cat /proc/self/cpuset

Существуют способы запрашивать или изменять cpusets:

 - напрямую через файловую систему cpuset, используя различные команды cd,
   mkdir, echo, cat, rmdir из оболочки или их эквиваленты из C.
 - через библиотеку C libcpuset.
 - через библиотеку C libcgroup.
   (https://github.com/libcgroup/libcgroup/)
 - через python-приложение cset.
   (http://code.google.com/p/cpuset/)

Вызовы sched_setaffinity также можно выполнять из командной строки оболочки
с помощью runon от SGI или taskset Роберта Лава (Robert Love). Вызовы
mbind и set_mempolicy можно выполнять из командной строки оболочки с
помощью команды numactl (часть numa-пакета Анди Клина (Andi Kleen)).

2. Примеры использования и синтаксис
====================================

2.1 Базовое использование
--------------------------

Создание, изменение и использование cpusets можно выполнять через
виртуальную файловую систему cpuset.

Чтобы смонтировать её, введите:
# mount -t cgroup -o cpuset cpuset /sys/fs/cgroup/cpuset

Затем в /sys/fs/cgroup/cpuset вы можете найти дерево, которое соответствует
дереву cpusets в системе. Например, /sys/fs/cgroup/cpuset — это cpuset,
который содержит всю систему.

Если вы хотите создать новый cpuset под /sys/fs/cgroup/cpuset::

  # cd /sys/fs/cgroup/cpuset
  # mkdir my_cpuset

Теперь вы хотите что-то сделать с этим cpuset::

  # cd my_cpuset

В этом каталоге вы можете найти несколько файлов::

  # ls
  cgroup.clone_children  cpuset.memory_pressure
  cgroup.event_control   cpuset.memory_spread_page
  cgroup.procs           cpuset.memory_spread_slab
  cpuset.cpu_exclusive   cpuset.mems
  cpuset.cpus            cpuset.sched_load_balance
  cpuset.mem_exclusive   cpuset.sched_relax_domain_level
  cpuset.mem_hardwall    notify_on_release
  cpuset.memory_migrate  tasks

Чтение их даст вам информацию о состоянии этого cpuset: о CPU и узлах
памяти, которые он может использовать, о процессах, которые его
используют, о его свойствах. Записывая в эти файлы, вы можете
манипулировать cpuset.

Установите некоторые флаги::

  # /bin/echo 1 > cpuset.cpu_exclusive

Добавьте некоторые cpus::

  # /bin/echo 0-7 > cpuset.cpus

Добавьте некоторые mems::

  # /bin/echo 0-7 > cpuset.mems

Теперь привяжите вашу оболочку к этому cpuset::

  # /bin/echo $$ > tasks

Вы также можете создавать cpusets внутри вашего cpuset, используя mkdir в
этом каталоге::

  # mkdir my_sub_cs

Чтобы удалить cpuset, просто используйте rmdir::

  # rmdir my_sub_cs

Это завершится неудачей, если cpuset используется (содержит cpusets
внутри или имеет привязанные процессы).

Обратите внимание, что по историческим причинам файловая система "cpuset"
существует как обёртка вокруг файловой системы cgroup.

Команда::

  mount -t cpuset X /sys/fs/cgroup/cpuset

эквивалентна::

  mount -t cgroup -ocpuset,noprefix X /sys/fs/cgroup/cpuset
  echo "/sbin/cpuset_release_agent" > /sys/fs/cgroup/cpuset/release_agent

2.2 Добавление/удаление cpus
----------------------------

Это синтаксис для использования при записи в файлы cpus или mems в
каталогах cpuset::

  # /bin/echo 1-4 > cpuset.cpus		-> set cpus list to cpus 1,2,3,4
  # /bin/echo 1,2,3,4 > cpuset.cpus	-> set cpus list to cpus 1,2,3,4

Чтобы добавить CPU в cpuset, запишите новый список CPU, включающий
добавляемый CPU. Чтобы добавить 6 к указанному выше cpuset::

  # /bin/echo 1-4,6 > cpuset.cpus	-> set cpus list to cpus 1,2,3,4,6

Аналогично, чтобы удалить CPU из cpuset, запишите новый список CPU без
удаляемого CPU.

Чтобы удалить все CPU::

  # /bin/echo "" > cpuset.cpus		-> clear cpus list

2.3 Установка флагов
--------------------

Синтаксис очень прост::

  # /bin/echo 1 > cpuset.cpu_exclusive 	-> set flag 'cpuset.cpu_exclusive'
  # /bin/echo 0 > cpuset.cpu_exclusive 	-> unset flag 'cpuset.cpu_exclusive'

2.4 Привязка процессов
----------------------

::

  # /bin/echo PID > tasks

Обратите внимание, что это PID, а не PIDs. Вы можете привязать только ОДНУ
задачу за раз. Если у вас есть несколько задач для привязки, вам нужно
делать это одну за другой::

  # /bin/echo PID1 > tasks
  # /bin/echo PID2 > tasks
	...
  # /bin/echo PIDn > tasks


3. Вопросы
==========

Q:
   что это за '/bin/echo'?

A:
   встроенная команда 'echo' в bash не проверяет вызовы write() на наличие
   ошибок. Если вы используете её в файловой системе cpuset, вы не сможете
   определить, завершилась команда успешно или неудачно.

Q:
   Когда я привязываю процессы, реально привязывается только первый в
   строке!

A:
   Мы можем вернуть только один код ошибки на вызов write(). Поэтому вам
   также следует указывать только ОДИН pid.
