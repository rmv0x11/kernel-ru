======================
Freezer для cgroup
======================

Freezer для cgroup полезен системам пакетного управления заданиями, которые
запускают и останавливают наборы задач, чтобы распределять ресурсы машины
в соответствии с пожеланиями системного администратора. Программы такого рода
часто применяются на HPC-кластерах для планирования доступа к кластеру в
целом. Freezer для cgroup использует cgroup'ы для описания набора задач,
которые система пакетного управления заданиями должна запускать/останавливать.
Он также предоставляет средство для запуска и остановки задач, составляющих
задание.

Freezer для cgroup также будет полезен для создания контрольных точек
(checkpointing) работающих групп задач. Freezer позволяет коду создания
контрольных точек получить согласованный образ задач, пытаясь принудительно
перевести задачи cgroup'а в состояние покоя. Как только задачи переходят в
состояние покоя, другая задача может обойти /proc или вызвать интерфейс ядра,
чтобы собрать информацию о приведённых в покой задачах. Задачи, для которых
была создана контрольная точка, могут быть позже перезапущены при возникновении
исправимой ошибки. Это также позволяет переносить такие задачи между узлами
кластера, копируя собранную информацию на другой узел и перезапуская задачи там.

Последовательностей SIGSTOP и SIGCONT не всегда достаточно для остановки и
возобновления задач в пространстве пользователя (userspace). Оба этих сигнала
наблюдаемы изнутри задач, которые мы хотим заморозить. Хотя SIGSTOP нельзя
перехватить, заблокировать или проигнорировать, его можно увидеть в ожидающих
или отслеживающих через ptrace родительских задачах. SIGCONT особенно
непригоден, так как может быть перехвачен самой задачей. Любые программы,
рассчитанные на отслеживание SIGSTOP и SIGCONT, могут быть сломаны при попытке
использовать SIGSTOP и SIGCONT для остановки и возобновления задач. Мы можем
продемонстрировать эту проблему с помощью вложенных оболочек bash::

	$ echo $$
	16644
	$ bash
	$ echo $$
	16690

	From a second, unrelated bash shell:
	$ kill -SIGSTOP 16690
	$ kill -SIGCONT 16690

	<at this point 16690 exits and causes 16644 to exit too>

Это происходит потому, что bash может наблюдать оба сигнала и выбирать, как на
них реагировать.

Ещё один пример программы, которая перехватывает эти сигналы и реагирует на
них, — gdb. На самом деле любая программа, рассчитанная на использование
ptrace, скорее всего, будет иметь проблему с этим методом остановки и
возобновления задач.

В отличие от этого, freezer для cgroup использует код ядра freezer, чтобы
предотвратить наблюдаемость цикла заморозки/разморозки замораживаемыми
задачами. Это позволяет приведённому выше примеру с bash и gdb работать так,
как ожидается.

Freezer для cgroup иерархичен. Заморозка cgroup'а замораживает все задачи,
принадлежащие этому cgroup'у и всем его потомкам. У каждого cgroup'а есть своё
собственное состояние (self-state) и состояние, унаследованное от родителя
(parent-state). Cgroup находится в состоянии THAWED тогда и только тогда, когда
оба состояния равны THAWED.

Freezer для cgroup создаёт следующие файлы cgroupfs.

* freezer.state: чтение-запись.

  При чтении возвращает эффективное состояние cgroup'а — "THAWED",
  "FREEZING" или "FROZEN". Это комбинация собственного и родительского
  состояний. Если любое из них заморожено (freezing), cgroup замораживается
  (FREEZING или FROZEN).

  Cgroup в состоянии FREEZING переходит в состояние FROZEN, когда все задачи,
  принадлежащие этому cgroup'у и его потомкам, становятся замороженными.
  Обратите внимание, что cgroup возвращается из FROZEN в FREEZING после
  добавления новой задачи в cgroup или в один из его cgroup'ов-потомков, и
  остаётся в нём до тех пор, пока новая задача не будет заморожена.

  При записи задаёт собственное состояние (self-state) cgroup'а. Разрешены два
  значения — "FROZEN" и "THAWED". Если записано FROZEN, cgroup, если он ещё не
  замораживается, переходит в состояние FREEZING вместе со всеми своими
  cgroup'ами-потомками.

  Если записано THAWED, собственное состояние cgroup'а меняется на THAWED.
  Обратите внимание, что эффективное состояние может не измениться на THAWED,
  если родительское состояние всё ещё заморожено. Если эффективное состояние
  cgroup'а становится THAWED, все его потомки, которые замораживаются из-за
  этого cgroup'а, также выходят из состояния заморозки.

* freezer.self_freezing: только для чтения.

  Показывает собственное состояние. 0, если собственное состояние равно THAWED;
  иначе 1. Это значение равно 1 тогда и только тогда, когда последней записью в
  freezer.state было "FROZEN".

* freezer.parent_freezing: только для чтения.

  Показывает родительское состояние. 0, если ни один из предков cgroup'а не
  заморожен; иначе 1.

Корневой cgroup не может быть заморожен, и описанные выше файлы интерфейса для
него не существуют.

* Примеры использования::

   # mkdir /sys/fs/cgroup/freezer
   # mount -t cgroup -ofreezer freezer /sys/fs/cgroup/freezer
   # mkdir /sys/fs/cgroup/freezer/0
   # echo $some_pid > /sys/fs/cgroup/freezer/0/tasks

чтобы получить статус подсистемы freezer::

   # cat /sys/fs/cgroup/freezer/0/freezer.state
   THAWED

чтобы заморозить все задачи в контейнере::

   # echo FROZEN > /sys/fs/cgroup/freezer/0/freezer.state
   # cat /sys/fs/cgroup/freezer/0/freezer.state
   FREEZING
   # cat /sys/fs/cgroup/freezer/0/freezer.state
   FROZEN

чтобы разморозить все задачи в контейнере::

   # echo THAWED > /sys/fs/cgroup/freezer/0/freezer.state
   # cat /sys/fs/cgroup/freezer/0/freezer.state
   THAWED

Это базовый механизм, который должен делать правильную вещь для задач
пространства пользователя в простом сценарии.

Эта реализация freezer'а имеет недостатки (см. коммит
76f969e8948d8 ("cgroup: cgroup v2 freezer")), и рекомендуется использовать
freezer cgroup v2.
