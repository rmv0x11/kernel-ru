================================================
Контроллер белого списка устройств (Whitelist)
================================================

1. Описание
===========

Реализует cgroup для отслеживания и применения ограничений на open и mknod
для файлов устройств.  cgroup устройств связывает с каждой cgroup белый
список доступа к устройствам.  Запись белого списка состоит из 4 полей.
'type' — это a (все), c (символьное) или b (блочное).  'all' означает, что
запись применяется ко всем типам и ко всем старшим и младшим номерам.
Старший и младший номера задаются либо целым числом, либо символом * для
всех значений.  Доступ — это комбинация r (чтение), w (запись) и m (mknod).

Корневая cgroup устройств начинается с rwm для 'all'.  Дочерняя cgroup
устройств получает копию родительской.  После этого администраторы могут
удалять устройства из белого списка или добавлять новые записи.  Дочерняя
cgroup никогда не может получить доступ к устройству, который запрещён её
родителем.

2. Пользовательский интерфейс
=============================

Запись добавляется с помощью devices.allow и удаляется с помощью
devices.deny.  Например::

	echo 'c 1:3 mr' > /sys/fs/cgroup/1/devices.allow

позволяет cgroup 1 читать и выполнять mknod для устройства, обычно
известного как /dev/null.  Выполнение::

	echo a > /sys/fs/cgroup/1/devices.deny

удалит запись по умолчанию 'a *:* rwm'. Выполнение::

	echo a > /sys/fs/cgroup/1/devices.allow

добавит запись 'a *:* rwm' в белый список.

3. Безопасность
===============

Любая задача может перемещать себя между cgroups.  Очевидно, этого
недостаточно, но мы можем определить наилучший способ адекватно ограничить
перемещение по мере накопления людьми опыта работы с этим.  Возможно, мы
просто захотим требовать CAP_SYS_ADMIN, который, по крайней мере, является
отдельным битом от CAP_MKNOD.  Возможно, мы захотим просто отказывать в
перемещении в cgroup, не являющуюся потомком текущей.  Или мы можем захотеть
использовать CAP_MAC_ADMIN, поскольку на самом деле мы пытаемся ограничить
root.

CAP_SYS_ADMIN необходим для изменения белого списка или перемещения другой
задачи в новую cgroup.  (Опять же, мы, вероятно, захотим это изменить).

cgroup не может получить больше прав, чем имеет её родительская cgroup.

4. Иерархия
===========

cgroups устройств поддерживают иерархию, гарантируя, что cgroup никогда не
имеет больше прав доступа, чем её родитель.  Каждый раз, когда в файл
devices.deny некоторой cgroup записывается запись, эта запись будет удалена
из белого списка всех её потомков, а все локально заданные записи белого
списка будут пересчитаны.  В случае, если одна из локально заданных записей
белого списка предоставляла бы больше доступа, чем родитель cgroup, она будет
удалена из белого списка.

Пример::

      A
     / \
        B

    group        behavior	exceptions
    A            allow		"b 8:* rwm", "c 116:1 rw"
    B            deny		"c 1:3 rwm", "c 116:2 rwm", "b 3:* rwm"

Если устройство запрещается в группе A::

	# echo "c 116:* r" > A/devices.deny

это распространится вниз, и после повторной проверки записей B запись белого
списка "c 116:2 rwm" будет удалена::

    group        whitelist entries                        denied devices
    A            all                                      "b 8:* rwm", "c 116:* rw"
    B            "c 1:3 rwm", "b 3:* rwm"                 all the rest

В случае, если исключения родителя изменяются и локальные исключения больше
не разрешены, они будут удалены.

Обратите внимание, что новые записи белого списка не будут распространяться::

      A
     / \
        B

    group        whitelist entries                        denied devices
    A            "c 1:3 rwm", "c 1:5 r"                   all the rest
    B            "c 1:3 rwm", "c 1:5 r"                   all the rest

при добавлении ``c *:3 rwm``::

	# echo "c *:3 rwm" >A/devices.allow

результат::

    group        whitelist entries                        denied devices
    A            "c *:3 rwm", "c 1:5 r"                   all the rest
    B            "c 1:3 rwm", "c 1:5 r"                   all the rest

но теперь станет возможным добавлять новые записи в B::

	# echo "c 2:3 rwm" >B/devices.allow
	# echo "c 50:3 r" >B/devices.allow

или даже::

	# echo "c *:3 rwm" >B/devices.allow

Разрешение или запрет всего путём записи 'a' в devices.allow или
devices.deny станет невозможным, как только у cgroup устройств появятся
потомки.

4.1 Иерархия (внутренняя реализация)
------------------------------------

cgroups устройств реализованы внутренне с использованием поведения (ALLOW,
DENY) и списка исключений.  Внутреннее состояние управляется через тот же
пользовательский интерфейс для сохранения совместимости с предыдущей
реализацией, основанной только на белом списке.  Удаление или добавление
исключений, которые уменьшат доступ к устройствам, будет распространяться
вниз по иерархии.
Для каждого распространённого исключения действующие правила будут
пересчитаны на основе текущих правил доступа родителя.
