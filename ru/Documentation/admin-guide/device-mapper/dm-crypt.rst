========
dm-crypt
========

Цель «crypt» в Device-Mapper обеспечивает прозрачное шифрование блочных устройств
с использованием crypto API ядра.

Более подробное описание поддерживаемых параметров см.:
https://gitlab.com/cryptsetup/cryptsetup/wikis/DMCrypt

Параметры::

	      <cipher> <key> <iv_offset> <device path> \
	      <offset> [<#opt_params> <opt_params>]

<cipher>
    Шифр, режим шифрования и генератор начального вектора (Initial Vector, IV).

    Формат спецификации шифра::

       cipher[:keycount]-chainmode-ivmode[:ivopts]

    Примеры::

       aes-cbc-essiv:sha256
       aes-xts-plain64
       serpent-xts-plain64

    Формат шифра также поддерживает прямое указание в формате crypto API ядра
    (выбирается префиксом capi:). Спецификация IV такая же,
    как и для первого типа формата.
    Этот формат используется в основном для указания режимов с аутентификацией.

    Формат спецификации шифра crypto API::

        capi:cipher_api_spec-ivmode[:ivopts]

    Примеры::

        capi:cbc(aes)-essiv:sha256
        capi:xts(aes)-plain64

    Примеры режимов с аутентификацией::

        capi:gcm(aes)-random
        capi:authenc(hmac(sha256),xts(aes))-random
        capi:rfc7539(chacha20,poly1305)-random

    Файл /proc/crypto содержит список загруженных в данный момент режимов шифрования.

<key>
    Ключ, используемый для шифрования. Кодируется либо как шестнадцатеричное число,
    либо может быть передан в виде <key_string> с префиксом из одного символа
    двоеточия (':') для ключей, находящихся в службе keyring ядра.
    Можно использовать только те размеры ключей, которые допустимы для выбранного шифра
    в сочетании с выбранным режимом iv.
    Обратите внимание, что для некоторых режимов iv строка ключа может содержать
    дополнительные ключи (например, IV seed), поэтому ключ состоит из нескольких частей,
    объединённых в одну строку.

<key_string>
    Ключ keyring ядра идентифицируется строкой в следующем формате:
    <key_size>:<key_type>:<key_description>.

<key_size>
    Размер ключа шифрования в байтах. Размер полезной нагрузки ключа ядра должен
    совпадать со значением, переданным в <key_size>.

<key_type>
    Тип ключа ядра: 'logon', 'user', 'encrypted' или 'trusted'.

<key_description>
    Описание ключа keyring ядра, которое цель crypt должна искать
    при загрузке ключа типа <key_type>.

<keycount>
    Режим совместимости с несколькими ключами. Можно определить <keycount> ключей,
    и тогда сектора шифруются в соответствии со своими смещениями (сектор 0 использует key0;
    сектор 1 использует key1 и т. д.).  <keycount> должно быть степенью двойки.

<iv_offset>
    Смещение IV — это число секторов, которое добавляется к номеру сектора
    перед созданием IV.

<device path>
    Это устройство, которое будет использоваться в качестве бэкенда и содержит
    зашифрованные данные.  Его можно указать как путь вида /dev/xxx или как номер
    устройства <major>:<minor>.

<offset>
    Начальный сектор внутри устройства, с которого начинаются зашифрованные данные.

<#opt_params>
    Число необязательных параметров. Если необязательных параметров нет,
    секцию необязательных параметров можно пропустить или указать #opt_params равным нулю.
    В противном случае #opt_params — это число следующих далее аргументов.

    Пример секции необязательных параметров:
        3 allow_discards same_cpu_crypt submit_from_crypt_cpus

allow_discards
    Запросы block discard (он же TRIM) пропускаются через устройство crypt.
    По умолчанию запросы discard игнорируются.

    ПРЕДУПРЕЖДЕНИЕ: тщательно оцените конкретные риски безопасности перед включением
    этой опции.  Например, разрешение discard на зашифрованных устройствах может привести
    к утечке информации об устройстве с шифротекстом (тип файловой системы,
    использованное пространство и т. п.), если отброшенные блоки впоследствии можно
    будет легко обнаружить на устройстве.

same_cpu_crypt
    Выполнять шифрование на том же процессоре, на котором был отправлен ввод-вывод.
    По умолчанию используется несвязанная (unbound) рабочая очередь, так что работа
    по шифрованию автоматически балансируется между доступными CPU.

high_priority
    Установить высокий приоритет для рабочих очередей dm-crypt и потока записи. Это
    улучшает пропускную способность и задержку (latency) dm-crypt за счёт ухудшения
    общей отзывчивости системы.

submit_from_crypt_cpus
    Отключить вынос (offloading) записей в отдельный поток после шифрования.
    В некоторых ситуациях вынос bio записи из потоков шифрования
    в единственный поток значительно ухудшает производительность.
    По умолчанию bio записи выносятся в тот же
    поток, потому что CFQ выгодно, чтобы записи отправлялись в одном
    и том же контексте.

no_read_workqueue
    Обходить внутреннюю рабочую очередь dm-crypt и обрабатывать запросы чтения синхронно.

no_write_workqueue
    Обходить внутреннюю рабочую очередь dm-crypt и обрабатывать запросы записи синхронно.
    Эта опция автоматически включается для зонированных блочных устройств с управлением
    со стороны хоста (host-managed) (например, жёстких дисков host-managed SMR).

integrity:<bytes>:<type>
    Устройству требуется дополнительно <bytes> метаданных на сектор, хранимых
    в структуре целостности (integrity) на каждый bio. Эти метаданные должны
    предоставляться нижележащей целью dm-integrity.

    <type> может быть "none", если метаданные используются только для постоянного IV.

    Для аутентифицированного шифрования с дополнительными данными (Authenticated
    Encryption with Additional Data, AEAD) <type> равен "aead". Режим AEAD
    дополнительно вычисляет и проверяет целостность зашифрованного устройства.
    Дополнительное пространство затем используется для хранения тега аутентификации
    (и постоянного IV, если необходимо).

integrity_key_size:<bytes>
    Опционально задать размер ключа целостности, если он отличается от размера дайджеста.
    Это позволяет использовать алгоритмы с обёрнутым (wrapped) ключом, где размер ключа
    не зависит от размера криптографического ключа.

sector_size:<bytes>
    Использовать <bytes> в качестве единицы шифрования вместо секторов по 512 байт.
    Эта опция может находиться в диапазоне 512 - 4096 байт и должна быть степенью двойки.
    Виртуальное устройство будет объявлять этот размер как минимальный размер ввода-вывода
    и логического сектора.

iv_large_sectors
   Генераторы IV будут использовать номер сектора, отсчитываемый в единицах <sector_size>,
   вместо секторов по умолчанию в 512 байт.

   Например, если <sector_size> равен 4096 байт, plain64 IV для второго
   сектора будет равен 8 (без флага) и 1, если присутствует iv_large_sectors.
   <iv_offset> должно быть кратно <sector_size> (в единицах по 512 байт),
   если указан этот флаг.

integrity_key_size:<bytes>
   Использовать ключ целостности размером <bytes> вместо использования ключа целостности
   размером с дайджест применяемого алгоритма HMAC.


Параметры модуля::
   max_read_size
      Максимальный размер запросов чтения. Когда получен запрос больше этого размера,
      dm-crypt разделит его. Разделение улучшает
      параллелизм (разделённые запросы могут шифроваться параллельно несколькими
      ядрами), но также вызывает накладные расходы. Пользователю следует подстроить этот
      параметр под фактическую нагрузку.

   max_write_size
      Максимальный размер запросов записи. Когда получен запрос больше этого размера,
      dm-crypt разделит его. Разделение улучшает
      параллелизм (разделённые запросы могут шифроваться параллельно несколькими
      ядрами), но также вызывает накладные расходы. Пользователю следует подстроить этот
      параметр под фактическую нагрузку.


Примеры скриптов
================
LUKS (Linux Unified Key Setup) теперь является предпочтительным способом настройки
шифрования диска с помощью dm-crypt с использованием утилиты 'cryptsetup', см.
https://gitlab.com/cryptsetup/cryptsetup

::

	#!/bin/sh
	# Create a crypt device using dmsetup
	dmsetup create crypt1 --table "0 `blockdev --getsz $1` crypt aes-cbc-essiv:sha256 babebabebabebabebabebabebabebabe 0 $1 0"

::

	#!/bin/sh
	# Create a crypt device using dmsetup when encryption key is stored in keyring service
	dmsetup create crypt2 --table "0 `blockdev --getsize $1` crypt aes-cbc-essiv:sha256 :32:logon:my_prefix:my_key 0 $1 0"

::

	#!/bin/sh
	# Create a crypt device using cryptsetup and LUKS header with default cipher
	cryptsetup luksFormat $1
	cryptsetup luksOpen $1 crypt1
