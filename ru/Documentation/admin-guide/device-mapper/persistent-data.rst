==================
Постоянные данные
==================

Введение
========

Более сложные цели device-mapper требуют сложных метаданных, которые
управляются в ядре. В конце 2010 года мы наблюдали, что различные цели
изобретали собственные структуры данных, например:

- реализация multisnap Микулаша Патоцки (Mikulas Patocka)
- цель тонкого выделения (thin provisioning) Хайнца Мауэльсхагена (Heinz Mauelshagen)
- ещё одна цель кэширования на основе btree, отправленная в dm-devel
- ещё одна цель множественных снимков на основе проекта Дэниела Филлипса (Daniel Phillips)

Сопровождение этих структур данных требует большого объёма работы, поэтому,
если возможно, мы хотели бы сократить их число.

Библиотека persistent-data — это попытка предоставить переиспользуемый
каркас для тех, кто хочет хранить метаданные в целях device-mapper. В
настоящее время она используется целью тонкого выделения (thin-provisioning)
и предстоящей целью иерархического хранения.

Обзор
=====

Основная документация находится в заголовочных файлах, которые все можно
найти в drivers/md/persistent-data.

Менеджер блоков
---------------

dm-block-manager.[hc]

Предоставляет доступ к данным на диске в блоках фиксированного размера.
Имеется интерфейс блокировки на чтение/запись для предотвращения
конкурентных обращений и сохранения используемых данных в кэше.

Клиенты persistent-data вряд ли будут использовать его напрямую.

Менеджер транзакций
-------------------

dm-transaction-manager.[hc]

Ограничивает доступ к блокам и обеспечивает соблюдение семантики
copy-on-write. Единственный способ получить доступ к блоку, доступному для
записи, через менеджер транзакций — это затенить (shadow) существующий блок
(то есть выполнить copy-on-write) или выделить новый. Затенение опускается в
рамках одной транзакции, поэтому производительность остаётся приемлемой.
Метод commit гарантирует, что все данные сброшены до записи суперблока.
При сбое питания ваши метаданные будут такими, какими они были на момент
последнего коммита.

Карты пространства (Space Maps)
-------------------------------

dm-space-map.h
dm-space-map-metadata.[hc]
dm-space-map-disk.[hc]

Структуры данных на диске, которые отслеживают счётчики ссылок блоков.
Также выступают в роли распределителя новых блоков. В настоящее время есть
две реализации: более простая для управления блоками на другом устройстве
(например, для тонко выделяемых блоков данных) и одна для управления
пространством метаданных. Вторая усложнена необходимостью хранить собственные
данные в пределах пространства, которым она управляет.

Структуры данных
----------------

dm-btree.[hc]
dm-btree-remove.c
dm-btree-spine.c
dm-btree-internal.h

В настоящее время есть только одна структура данных — иерархическое btree.
Есть планы добавить больше. Например, что-то с интерфейсом, подобным массиву,
нашло бы широкое применение.

btree является «иерархическим» в том смысле, что его можно определить как
составленное из вложенных btree и принимающее несколько ключей. Например,
цель тонкого выделения (thin-provisioning) использует btree с двумя уровнями
вложенности. Первый отображает идентификатор устройства в дерево отображений,
а оно, в свою очередь, отображает виртуальный блок в физический блок.

Значения, хранящиеся в btree, могут иметь произвольный размер. Ключи всегда
64-битные, хотя вложенность позволяет использовать несколько ключей.
