=============
dm-log-writes
=============

Эта цель принимает 2 устройства: одно — для пропускания всего ввода-вывода в
обычном режиме, и другое — для журналирования всех операций записи. Это
предназначено для разработчиков файловых систем, желающих проверить целостность
метаданных или данных по мере записи в файловую систему. Для каждого запроса
WRITE записывается log_write_entry, и цель способна принимать из пространства
пользователя (userspace) произвольные данные для вставки в журнал. Данные,
содержащиеся в запросах WRITE, копируются в журнал, чтобы воспроизведение
происходило в точности так, как оно происходило изначально.

Порядок журналирования
======================

Мы журналируем события в порядке их завершения, как только убеждаемся, что
запись больше не находится в кэше. Это означает, что обычные запросы WRITE
фактически не журналируются до следующего запроса REQ_PREFLUSH. Сделано это для
того, чтобы упростить для пространства пользователя воспроизведение журнала так,
чтобы оно соответствовало тому, что находится на диске, а не тому, что находится
в кэше, и упростить обнаружение некорректного ожидания/сброса.

Это работает за счёт присоединения всех запросов WRITE к списку после завершения
записи. Как только мы видим запрос REQ_PREFLUSH, мы соединяем этот список с
запросом, и после завершения запроса FLUSH мы журналируем все WRITE, а затем
FLUSH. Добавляются только те WRITE, которые завершились на момент выдачи
REQ_PREFLUSH, чтобы смоделировать наихудший сценарий в отношении сбоев питания.
Рассмотрим следующий пример (W означает запись, C означает завершение):

	W1,W2,W3,C3,C2,Wflush,C1,Cflush

Журнал покажет следующее:

	W3,W2,flush,W1....

Опять же, это сделано для моделирования того, что в действительности находится на
диске; это позволяет нам обнаруживать случаи, когда сбой питания в определённый
момент времени привёл бы к несогласованной файловой системе.

Любые запросы REQ_FUA обходят этот механизм сброса и журналируются сразу же после
их завершения, поскольку такие запросы очевидным образом будут обходить кэш
устройства.

Любые запросы REQ_OP_DISCARD обрабатываются как запросы WRITE. В противном случае
у нас сначала были бы все запросы DISCARD, затем запросы WRITE, а затем запрос
FLUSH. Рассмотрим следующий пример:

	WRITE блок 1, DISCARD блок 1, FLUSH

Если бы мы журналировали DISCARD по его завершении, воспроизведение выглядело бы
так:

	DISCARD 1, WRITE 1, FLUSH

что не вполне соответствует тому, что произошло, и не было бы обнаружено при
воспроизведении журнала.

Интерфейс цели
==============

i) Конструктор

   log-writes <dev_path> <log_dev_path>

   ============= ==============================================
   dev_path	 Устройство, на которое в обычном режиме будет идти весь ввод-вывод.
   log_dev_path  Устройство, на которое записываются записи журнала.
   ============= ==============================================

ii) Состояние

    <#logged entries> <highest allocated sector>

    =========================== ========================
    #logged entries	        Количество записанных записей
    highest allocated sector    Наибольший выделенный сектор
    =========================== ========================

iii) Сообщения

    mark <description>

	Вы можете использовать сообщение dmsetup, чтобы установить произвольную
	метку в журнале. Например, предположим, что вы хотите выполнять fsck
	файловой системы после каждой записи, но сначала вам нужно воспроизвести
	журнал вплоть до mkfs, чтобы убедиться, что мы проверяем посредством fsck
	что-то разумное; вы бы сделали что-то вроде такого::

	  mkfs.btrfs -f /dev/mapper/log
	  dmsetup message log 0 mark mkfs
	  <run test>

	Это позволило бы вам воспроизвести журнал вплоть до метки mkfs, а затем
	воспроизводить с этой точки далее, выполняя проверку fsck в том интервале,
	который вам нужен.

	В каждом журнале в конце есть метка с надписью "dm-log-writes-end".

Компонент в пространстве пользователя
=====================================

Существует инструмент в пространстве пользователя, который воспроизведёт журнал
для вас различными способами. Его можно найти здесь:
https://github.com/josefbacik/log-writes

Пример использования
====================

Предположим, вы хотите протестировать fsync в вашей файловой системе. Вы бы
сделали что-то вроде такого::

  TABLE="0 $(blockdev --getsz /dev/sdb) log-writes /dev/sdb /dev/sdc"
  dmsetup create log --table "$TABLE"
  mkfs.btrfs -f /dev/mapper/log
  dmsetup message log 0 mark mkfs

  mount /dev/mapper/log /mnt/btrfs-test
  <some test that does fsync at the end>
  dmsetup message log 0 mark fsync
  md5sum /mnt/btrfs-test/foo
  umount /mnt/btrfs-test

  dmsetup remove log
  replay-log --log /dev/sdc --replay /dev/sdb --end-mark fsync
  mount /dev/sdb /mnt/btrfs-test
  md5sum /mnt/btrfs-test/foo
  <verify md5sum's are correct>

  Ещё один вариант — выполнить сложную операцию над файловой системой и проверить,
  что файловая система остаётся согласованной в течение всей операции. Вы могли бы
  сделать это так:

  TABLE="0 $(blockdev --getsz /dev/sdb) log-writes /dev/sdb /dev/sdc"
  dmsetup create log --table "$TABLE"
  mkfs.btrfs -f /dev/mapper/log
  dmsetup message log 0 mark mkfs

  mount /dev/mapper/log /mnt/btrfs-test
  <fsstress to dirty the fs>
  btrfs filesystem balance /mnt/btrfs-test
  umount /mnt/btrfs-test
  dmsetup remove log

  replay-log --log /dev/sdc --replay /dev/sdb --end-mark mkfs
  btrfsck /dev/sdb
  replay-log --log /dev/sdc --replay /dev/sdb --start-mark mkfs \
	--fsck "btrfsck /dev/sdb" --check fua

И это будет воспроизводить журнал, пока не встретит запрос FUA, выполнит команду
fsck, и если fsck проходит успешно, воспроизведение продолжится до следующего
FUA, пока оно не завершится или команда fsck не завершится аварийно.
