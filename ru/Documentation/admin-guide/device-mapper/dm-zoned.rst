========
dm-zoned
========

Цель device mapper dm-zoned представляет зонированное блочное устройство
(устройства, совместимые с ZBC и ZAC) как обычное блочное устройство без
каких-либо ограничений на схему записи. По сути, она реализует зонированное
блочное устройство с управлением со стороны накопителя (drive-managed),
которое скрывает от пользователя (файловой системы или приложения,
выполняющего прямой доступ к блочному устройству) ограничения на
последовательную запись зонированных блочных устройств с управлением со
стороны хоста (host-managed) и может смягчать потенциальную деградацию
производительности на стороне устройства, вызванную чрезмерным количеством
случайных записей на зонированных блочных устройствах с осведомлённостью со
стороны хоста (host-aware).

Более подробное описание моделей зонированных блочных устройств и их
ограничений см. (для устройств SCSI):

https://www.t10.org/drafts.htm#ZBC_Family

и (для устройств ATA):

http://www.t13.org/Documents/UploadedDocuments/docs2015/di537r05-Zoned_Device_ATA_Command_Set_ZAC.pdf

Реализация dm-zoned проста и минимизирует накладные расходы системы
(использование CPU и памяти, а также потери ёмкости хранилища). Для диска
ёмкостью 10 ТБ с управлением со стороны хоста и зонами по 256 МБ
использование памяти dm-zoned на один экземпляр диска составляет не более
4,5 МБ, и внутренне будет задействовано всего лишь 5 зон для хранения
метаданных и выполнения операций освобождения (reclaim).

Целевые устройства dm-zoned форматируются и проверяются с помощью утилиты
dmzadm, доступной по адресу:

https://github.com/hgst/dm-zoned-tools

Алгоритм
========

dm-zoned реализует схему буферизации на диске для обработки
непоследовательных операций записи в последовательные зоны зонированного
блочного устройства. Обычные (conventional) зоны используются как для
кэширования, так и для хранения внутренних метаданных. Также может
использоваться обычное блочное устройство совместно с зонированным блочным
устройством; в этом случае обычное блочное устройство будет логически
разбито на зоны такого же размера, как у зонированного блочного устройства.
Эти зоны будут размещены перед зонами зонированного блочного устройства и
будут обрабатываться так же, как обычные зоны.

Зоны устройства (или устройств) разделяются на 2 типа:

1) Зоны метаданных: это обычные зоны, используемые для хранения метаданных.
Зоны метаданных не отображаются пользователю как полезная ёмкость.

2) Зоны данных: все остальные зоны, подавляющее большинство которых будет
последовательными зонами, используемыми исключительно для хранения данных
пользователя. Обычные зоны устройства могут также использоваться для
буферизации случайных записей пользователя. Данные в этих зонах могут
напрямую отображаться на обычную зону, но позже перемещаться в
последовательную зону, чтобы обычную зону можно было повторно использовать
для буферизации входящих случайных записей.

dm-zoned представляет логическое устройство с размером сектора 4096 байт
независимо от физического размера сектора используемого внутреннего
зонированного блочного устройства. Это позволяет сократить объём метаданных,
необходимых для управления валидными блоками (записанными блоками).

Формат метаданных на диске следующий:

1) Первый блок первой найденной обычной зоны содержит суперблок, который
описывает объём и положение блоков метаданных на диске.

2) Следом за суперблоком набор блоков используется для описания отображения
блоков логического устройства. Отображение выполняется поблочно по чанкам
(chunk), при этом размер чанка равен размеру зонированного блочного
устройства. Таблица отображения индексируется по номеру чанка, и каждая
запись отображения указывает номер зоны устройства, хранящей этот чанк
данных. Каждая запись отображения может также указывать, используется ли
номер зоны обычной зоны для буферизации случайных изменений зоны данных.

3) За таблицей отображения следует набор блоков, используемых для хранения
битовых карт, указывающих валидность блоков в зонах данных. Валидным блоком
считается блок, который был записан и не был отброшен (discard). Для
буферизованного чанка данных блок всегда валиден только в зоне данных,
отображающей чанк, либо в буферной зоне этого чанка.

Для логического чанка, отображённого на обычную зону, все операции записи
обрабатываются путём прямой записи в зону. Если отображающая зона является
последовательной, операция записи обрабатывается напрямую только в том
случае, если смещение записи в пределах логического чанка равно смещению
указателя записи в пределах последовательной зоны данных (то есть операция
записи выровнена по указателю записи зоны). В противном случае операции
записи обрабатываются косвенно с использованием буферной зоны. В этом случае
неиспользуемая обычная зона выделяется и назначается чанку, к которому
осуществляется доступ. Запись блока в буферную зону чанка автоматически
делает невалидным тот же блок в последовательной зоне, отображающей чанк.
Если все блоки последовательной зоны становятся невалидными, зона
освобождается, а буферная зона чанка становится основной зоной, отображающей
чанк, что обеспечивает естественную производительность случайной записи,
аналогичную обычному блочному устройству.

Операции чтения обрабатываются в соответствии с информацией о валидности
блоков, предоставляемой битовыми картами. Валидные блоки читаются либо из
последовательной зоны, отображающей чанк, либо, если чанк буферизован, из
назначенной буферной зоны. Если у запрашиваемого чанка нет отображения или
запрашиваемые блоки невалидны, буфер чтения заполняется нулями, и операция
чтения завершается.

Спустя некоторое время ограниченное число доступных обычных зон может быть
исчерпано (все используются для отображения чанков или буферизации
последовательных зон), и невыровненные записи в небуферизованные чанки
становятся невозможными. Чтобы избежать этой ситуации, процесс освобождения
(reclaim) регулярно сканирует используемые обычные зоны и пытается
освободить наименее недавно использовавшиеся зоны путём копирования валидных
блоков буферной зоны в свободную последовательную зону. После завершения
копирования отображение чанка обновляется так, чтобы указывать на
последовательную зону, и буферная зона освобождается для повторного
использования.

Защита метаданных
=================

Для защиты метаданных от повреждения в случае внезапной потери питания или
сбоя системы используются 2 набора зон метаданных. Один набор, основной
(primary), используется как главная область метаданных, в то время как
вторичный (secondary) набор используется как промежуточная (staging)
область. Изменённые метаданные сначала записываются во вторичный набор и
подтверждаются обновлением суперблока во вторичном наборе; счётчик поколений
(generation counter) используется для указания того, что этот набор содержит
самые новые метаданные. После завершения этой операции на месте могут быть
выполнены обновления блоков метаданных в основном наборе метаданных. Это
гарантирует, что один из наборов всегда согласован (все изменения
зафиксированы либо не зафиксировано ни одно). Операции сброса (flush)
используются как точка фиксации. По получении запроса на сброс активность по
изменению метаданных временно блокируется (как обработка входящих BIO, так и
процесс освобождения), и все «грязные» блоки метаданных подготавливаются и
обновляются. Затем возобновляется нормальная работа. Таким образом, сброс
метаданных лишь временно задерживает запросы на запись и отбрасывание
(discard). Запросы на чтение могут обрабатываться параллельно во время
выполнения сброса метаданных.

Если обычное устройство используется совместно с зонированным блочным
устройством, в начало зонированного блочного устройства записывается третий
набор метаданных (без битовых карт зон). У этих метаданных счётчик поколений
равен «0», и они никогда не обновляются во время нормальной работы; они
служат лишь для целей идентификации. Первая и вторая копии метаданных
располагаются в начале обычного блочного устройства.

Использование
=============

Зонированное блочное устройство сначала должно быть отформатировано с
помощью инструмента dmzadm. При этом будет проанализирована конфигурация зон
устройства, определено, где разместить наборы метаданных на устройстве, и
инициализированы наборы метаданных.

Пример::

	dmzadm --format /dev/sdxx


Если предполагается использовать два накопителя, должны быть указаны оба
устройства, при этом обычное блочное устройство указывается первым.

Пример::

	dmzadm --format /dev/sdxx /dev/sdyy


Отформатированные устройства также можно запустить с помощью утилиты dmzadm.:

Пример::

	dmzadm --start /dev/sdxx /dev/sdyy


Информацию о внутренней структуре и текущем использовании зон можно получить
с помощью обратного вызова «status» из dmsetup:

Пример::

	dmsetup status /dev/dm-X

вернёт строку

	0 <size> zoned <nr_zones> zones <nr_unmap_rnd>/<nr_rnd> random <nr_unmap_seq>/<nr_seq> sequential

где <nr_zones> — это общее число зон, <nr_unmap_rnd> — число неотображённых
(то есть свободных) случайных зон, <nr_rnd> — общее число зон,
<nr_unmap_seq> — число неотображённых последовательных зон, а <nr_seq> —
общее число последовательных зон.

Обычно процесс освобождения запускается, когда остаётся менее 50 процентов
свободных случайных зон. Чтобы запустить процесс освобождения вручную ещё до
достижения этого порога, можно использовать функцию «dmsetup message»:

Пример::

	dmsetup message /dev/dm-X 0 reclaim

запустит процесс освобождения, и случайные зоны будут перемещены в
последовательные зоны.
