============
dm-integrity
============

Цель dm-integrity эмулирует блочное устройство, имеющее дополнительные
теги для каждого сектора, которые могут использоваться для хранения
информации о целостности.

Общая проблема при хранении тегов целостности вместе с каждым сектором
заключается в том, что запись сектора и тега целостности должна быть
атомарной — то есть в случае сбоя должны быть записаны либо одновременно
и сектор, и тег целостности, либо ни один из них.

Чтобы гарантировать атомарность записи, цель dm-integrity использует журнал:
она записывает данные сектора и теги целостности в журнал, фиксирует журнал,
а затем копирует данные и теги целостности в их соответствующие места.

Цель dm-integrity может использоваться вместе с целью dm-crypt — в этом
случае цель dm-crypt создаёт данные целостности и передаёт их цели
dm-integrity через bio_integrity_payload, прикреплённый к bio.
В этом режиме цели dm-crypt и dm-integrity обеспечивают аутентифицированное
шифрование диска: если злоумышленник изменит зашифрованное устройство,
вместо случайных данных будет возвращена ошибка I/O.

Цель dm-integrity также может использоваться как самостоятельная цель,
в этом режиме она вычисляет и проверяет тег целостности внутренне. В этом
режиме цель dm-integrity может использоваться для обнаружения скрытого
повреждения данных на диске или в пути I/O.

Существует альтернативный режим работы, в котором dm-integrity использует
битовую карту вместо журнала. Если бит в битовой карте равен 1, данные
и теги целостности соответствующей области не синхронизированы — если
машина даёт сбой, несинхронизированные области будут пересчитаны. Режим
битовой карты быстрее режима журнала, так как нам не нужно записывать
данные дважды, но он также менее надёжен, потому что если повреждение
данных происходит при сбое машины, оно может быть не обнаружено.

При первой загрузке цели драйвер ядра отформатирует устройство. Но он
отформатирует устройство только в том случае, если суперблок содержит нули.
Если суперблок ни корректен, ни обнулён, цель dm-integrity не может быть
загружена.

Доступы к области метаданных на диске, содержащей контрольные суммы
(они же теги), буферизуются с помощью dm-bufio. Когда происходит доступ
к любой заданной области метаданных, каждая уникальная область метаданных
получает свой собственный буфер (или буферы). Размер буфера ограничен
размером области метаданных, но может быть меньше, что приводит к
необходимости использовать несколько буферов для представления полной
области метаданных. Меньший размер буфера приведёт к меньшей результирующей
операции чтения/записи в область метаданных для небольших операций
чтения/записи. Метаданные по-прежнему читаются даже при полной записи
данных, покрываемых одним буфером.

Чтобы использовать цель в первый раз:

1. перезапишите суперблок нулями
2. загрузите цель dm-integrity с размером в один сектор, драйвер ядра
   отформатирует устройство
3. выгрузите цель dm-integrity
4. прочитайте значение "provided_data_sectors" из суперблока
5. загрузите цель dm-integrity с размером цели
   "provided_data_sectors"
6. если вы хотите использовать dm-integrity вместе с dm-crypt, загрузите
   цель dm-crypt с размером "provided_data_sectors"


Аргументы цели:

1. базовое блочное устройство

2. количество зарезервированных секторов в начале устройства —
   dm-integrity не будет читать или записывать эти секторы

3. размер тега целостности (если используется "-", размер берётся из
   алгоритма internal-hash)

4. режим:

	D - прямые записи (без журнала)
		в этом режиме журналирование
		не используется, а секторы данных и теги целостности записываются
		раздельно. В случае сбоя возможно, что данные
		и тег целостности не совпадают.
	J - журналируемые записи
		данные и теги целостности записываются в
		журнал, и атомарность гарантируется. В случае сбоя
		записываются либо одновременно данные и тег, либо ни один из них.
		Журналируемый режим вдвое снижает пропускную способность записи,
		так как данные приходится записывать дважды.
	B - режим битовой карты — данные и метаданные записываются без какой-либо
		синхронизации, драйвер ведёт битовую карту «грязных»
		областей, где данные и метаданные не совпадают. Этот режим может
		использоваться только с внутренним хешем.
	R - режим восстановления — в этом режиме журнал не воспроизводится,
		контрольные суммы не проверяются, а записи на устройство не
		разрешены. Этот режим полезен для восстановления данных, если
		устройство не может быть активировано ни в одном из других
		стандартных режимов.
	I - inline-режим — в этом режиме dm-integrity будет хранить данные
		целостности непосредственно в секторах базового устройства.
		Базовое устройство должно иметь профиль целостности, который
		позволяет хранить пользовательские данные целостности и
		предоставляет достаточно места для выбранного тега целостности.

5. количество дополнительных аргументов

Дополнительные аргументы:

journal_sectors:number
	Размер журнала, этот аргумент используется только при форматировании
	устройства. Если устройство уже отформатировано, используется значение
	из суперблока.

interleave_sectors:number (по умолчанию 32768)
	Количество чередуемых секторов. Это значение округляется вниз до
	степени двойки. Если устройство уже отформатировано, используется
	значение из суперблока.

meta_device:device
	Не чередовать данные и метаданные на устройстве. Использовать
	отдельное устройство для метаданных.

buffer_sectors:number (по умолчанию 128)
	Количество секторов в одном буфере метаданных. Значение округляется
	вниз до степени двойки.

journal_watermark:number (по умолчанию 50)
	Ватерлиния журнала в процентах. Когда размер журнала
	превышает эту ватерлинию, запускается поток, сбрасывающий журнал.

commit_time:number (по умолчанию 10000)
	Время фиксации в миллисекундах. Когда это время истекает, журнал
	записывается. Журнал также записывается немедленно при получении
	запроса FLUSH.

internal_hash:algorithm(:key)	(ключ необязателен)
	Использовать внутренний хеш или crc.
	Когда используется этот аргумент, цель dm-integrity не будет принимать
	теги целостности от вышестоящей цели, а будет автоматически
	генерировать и проверять теги целостности.

	Вы можете использовать алгоритм crc (например, crc32), тогда цель
	целостности будет защищать данные от случайного повреждения.
	Вы также можете использовать алгоритм hmac (например,
	"hmac(sha256):0123456789abcdef"), в этом режиме он будет обеспечивать
	криптографическую аутентификацию данных без шифрования.

	Когда этот аргумент не используется, теги целостности принимаются
	от вышестоящей по уровню цели, такой как dm-crypt. Вышестоящая по
	уровню цель должна проверять корректность тегов целостности.

recalculate
	Пересчитывать теги целостности автоматически. Это допустимо только
	при использовании внутреннего хеша.

journal_crypt:algorithm(:key)	(ключ необязателен)
	Шифровать журнал с использованием заданного алгоритма, чтобы убедиться,
	что злоумышленник не сможет прочитать журнал. Здесь можно использовать
	блочный шифр (например, "cbc(aes)") или потоковый шифр (например, "chacha20"
	или "ctr(aes)").

	Журнал содержит историю последних записей на блочное устройство,
	злоумышленник, читающий журнал, мог бы увидеть номера последних
	секторов, которые были записаны. По номерам секторов злоумышленник
	может вывести размер записанных файлов. Чтобы защититься от такой
	ситуации, вы можете зашифровать журнал.

journal_mac:algorithm(:key)	(ключ необязателен)
	Защитить номера секторов в журнале от случайного или злонамеренного
	изменения. Чтобы защититься от случайного изменения, используйте
	алгоритм crc, чтобы защититься от злонамеренного изменения, используйте
	алгоритм hmac с ключом.

	Этот параметр не нужен при использовании internal-hash, потому что в этом
	режиме целостность записей журнала проверяется при воспроизведении
	журнала. Таким образом, изменённый номер сектора будет обнаружен на
	этом этапе.

block_size:number (по умолчанию 512)
	Размер блока данных в байтах. Чем больше размер блока, тем
	меньше накладных расходов на метаданные целостности для каждого блока.
	Поддерживаемые значения: 512, 1024, 2048 и 4096 байт.

sectors_per_bit:number
	В режиме битовой карты этот параметр задаёт количество
	512-байтовых секторов, соответствующих одному биту битовой карты.

bitmap_flush_interval:number
	Интервал сброса битовой карты в миллисекундах. Буферы метаданных
	синхронизируются по истечении этого интервала.

allow_discards
	Разрешить запросы block discard (он же TRIM) для устройства целостности.
	Discard разрешены только для устройств, использующих внутренний хеш.

fix_padding
	Использовать меньшее выравнивание области тегов, которое более
	эффективно по месту. Если этот параметр отсутствует, используется
	большое выравнивание — это для совместимости с более старыми ядрами.

fix_hmac
	Улучшить безопасность internal_hash и journal_mac:

	- номер секции примешивается к mac, так что злоумышленник не может
	  скопировать секторы из одной секции журнала в другую секцию журнала
	- суперблок защищён посредством journal_mac
	- 16-байтовая соль, хранящаяся в суперблоке, примешивается к mac, так
	  что злоумышленник не может обнаружить, что два диска имеют один и тот
	  же ключ hmac, а также чтобы запретить злоумышленнику перемещать
	  секторы с одного диска на другой

legacy_recalculate
	Разрешить пересчёт томов с ключами HMAC. По умолчанию это отключено
	из соображений безопасности — злоумышленник мог бы изменить том,
	установить recalc_sector в ноль, и ядро не обнаружило бы это
	изменение.

Режим журнала (D/J), buffer_sectors, journal_watermark, commit_time и
allow_discards могут быть изменены при перезагрузке цели (загрузите неактивную
таблицу и поменяйте таблицы местами через suspend и resume). Остальные аргументы
не следует менять при перезагрузке цели, так как от них зависит компоновка
данных на диске, и перезагруженная цель окажется нерабочей.

Например, на устройстве, использующем interleave_sectors по умолчанию равное 32768,
block_size равный 512 и internal_hash crc32c с размером тега 4
байта, потребуется 128 КиБ тегов для отслеживания полной области данных, что
требует 256 секторов метаданных на область данных. При buffer_sectors по умолчанию равном
128 это означает, что на одну область метаданных будет 2 буфера, или 2 буфера
на 16 МиБ данных.

Строка статуса:

1. количество несоответствий целостности
2. provided data sectors — это количество секторов, которые пользователь
   мог бы использовать
3. текущая позиция пересчёта (или '-', если мы не пересчитывали)


Компоновка отформатированного блочного устройства:

* зарезервированные секторы
    (они не используются этой целью, они могут использоваться для
    хранения метаданных LUKS или для других целей), размер зарезервированной
    области указывается в аргументах цели

* суперблок (4киБ)
	* magic string — определяет, что устройство было отформатировано
	* версия
	* log2(interleave sectors)
	* размер тега целостности
	* количество секций журнала
	* provided data sectors — количество секторов, которые предоставляет
	  эта цель (то есть размер устройства за вычетом размера всех
	  метаданных и выравнивания). Пользователь этой цели не должен посылать
	  bio, которые обращаются к данным за пределом "provided data sectors".
	* флаги
	    SB_FLAG_HAVE_JOURNAL_MAC
		- флаг устанавливается, если используется journal_mac
	    SB_FLAG_RECALCULATING
		- идёт пересчёт
	    SB_FLAG_DIRTY_BITMAP
		- область журнала содержит битовую карту «грязных»
		  блоков
	* log2(sectors per block)
	* позиция, на которой завершился пересчёт
* журнал
	Журнал делится на секции, каждая секция содержит:

	* область метаданных (4киБ), она содержит записи журнала

	  - каждая запись журнала содержит:

		* логический сектор (задаёт, куда должны быть записаны данные
		  и тег)
		* последние 8 байт данных
		* тег целостности (размер задаётся в суперблоке)

	  - каждый сектор метаданных заканчивается:

		* mac (8 байт), все mac в 8 секторах метаданных образуют
		  64-байтовое значение. Оно используется для хранения hmac номеров
		  секторов в секции журнала, чтобы защититься от
		  возможности того, что злоумышленник подделает номера
		  секторов в журнале.
		* commit id

	* область данных (размер переменный; он зависит от того, сколько записей
	  журнала помещается в область метаданных)

	    - каждый сектор в области данных содержит:

		* данные (504 байта данных, последние 8 байт хранятся в
		  записи журнала)
		* commit id

	Чтобы проверить, была ли вся секция журнала записана корректно, каждый
	512-байтовый сектор журнала заканчивается 8-байтовым commit id. Если
	commit id совпадает на всех секторах в секции журнала, то
	предполагается, что секция была записана корректно. Если commit id
	не совпадает, секция была записана частично, и её не следует
	воспроизводить.

* один или несколько прогонов чередующихся тегов и данных.
    Каждый прогон содержит:

	* область тегов — она содержит теги целостности. Есть один тег на каждый
	  сектор в области данных. Размер этой области всегда 4КиБ или
	  больше.
	* область данных — она содержит секторы данных. Количество секторов данных
	  в одном прогоне должно быть степенью двойки. log2 этого значения хранится
	  в суперблоке.
