===========================================
Руководство по написанию политик (policies)
===========================================

Старайтесь не вовлекать в них транзакционность. Ядро тщательно
избегает запросов о чём-либо, что находится в процессе миграции. Это
неудобно, но облегчает написание политик.

Отображения (mappings) загружаются в политику в момент её создания.

Каждый bio, отображаемый целевым устройством (target), передаётся
политике. Политика может вернуть простое попадание (HIT) или промах
(MISS) либо инициировать миграцию.

В настоящее время у политики нет способа инициировать фоновую работу,
например начать обратную запись «грязных» блоков, которые вскоре будут
вытеснены.

Поскольку мы отображаем bio, а не запросы, политику легко ввести в
заблуждение множеством мелких bio. По этой причине целевое ядро
выдаёт политике периодические тики (ticks). Рекомендуется, чтобы
политика не обновляла состояния (например, счётчики попаданий) для
блока более одного раза за каждый тик. Ядро формирует тики, наблюдая
за завершением bio, и тем самым пытаясь определить, когда планировщик
ввода-вывода дал ходу операциям ввода-вывода.


Обзор поставляемых политик замещения кэша
=========================================

multiqueue (mq)
---------------

Эта политика теперь является псевдонимом для smq (см. ниже).

Принимаются следующие настраиваемые параметры, но они не оказывают
эффекта::

	'sequential_threshold <#nr_sequential_ios>'
	'random_threshold <#nr_random_ios>'
	'read_promote_adjustment <value>'
	'write_promote_adjustment <value>'
	'discard_promote_adjustment <value>'

Stochastic multiqueue (smq)
---------------------------

Эта политика используется по умолчанию.

Политика stochastic multi-queue (smq) решает некоторые проблемы
политики multiqueue (mq).

Политика smq (в сравнении с mq) обещает меньшее потребление памяти,
повышенную производительность и большую адаптивность при изменении
нагрузок. Кроме того, у smq нет каких-либо громоздких регулировочных
параметров.

Пользователи могут переключиться с «mq» на «smq», просто
соответствующим образом перезагрузив DM-таблицу, которая использует
целевое устройство cache. Это приведёт к отбрасыванию всех подсказок
(hints) политики mq. Кроме того, производительность кэша может
несколько снизиться, пока smq не пересчитает «горячие» области
(hotspots) исходного устройства, которые следует кэшировать.

Использование памяти
^^^^^^^^^^^^^^^^^^^^^

Политика mq использовала много памяти: 88 байт на блок кэша на
64-битной машине.

smq использует для реализации своих структур данных 28-битные
индексы, а не указатели. Она избегает хранения явного счётчика
попаданий для каждого блока. У неё есть очередь «горячих» областей
(hotspot), а не предварительный кэш (pre-cache), которая использует
четверть записей (каждый блок hotspot покрывает большую область, чем
один блок кэша).

Всё это означает, что smq использует ~25 байт на блок кэша. Всё ещё
много памяти, но тем не менее существенное улучшение.

Балансировка уровней
^^^^^^^^^^^^^^^^^^^^^

mq размещала записи на разных уровнях структур multiqueue в
зависимости от их счётчика попаданий (~ln(счётчик попаданий)). Это
означало, что нижние уровни обычно содержали наибольшее число
записей, а верхние — очень мало. Такие несбалансированные уровни
снижали эффективность multiqueue.

smq не поддерживает счётчик попаданий, вместо этого она меняет местами
записи с попаданиями с наименее давно использованной (least recently
used) записью с уровня выше. Общий порядок при этом является побочным
эффектом данного стохастического процесса. При такой схеме мы можем
решать, сколько записей занимает каждый уровень multiqueue, что
приводит к лучшим решениям о повышении/понижении (promotion/demotion).

Адаптивность:
Политика mq поддерживала счётчик попаданий для каждого блока кэша.
Чтобы другой блок был повышен (promoted) в кэш, его счётчик попаданий
должен был превысить наименьший из имеющихся в данный момент в кэше.
Это означало, что кэшу могло потребоваться много времени для адаптации
к различным шаблонам ввода-вывода.

smq не поддерживает счётчики попаданий, поэтому значительная часть
этой проблемы попросту исчезает. Кроме того, она отслеживает
производительность очереди hotspot, которая используется для принятия
решения о том, какие блоки повышать. Если очередь hotspot работает
плохо, то она начинает быстрее перемещать записи между уровнями. Это
позволяет ей очень быстро адаптироваться к новым шаблонам ввода-вывода.

Производительность
^^^^^^^^^^^^^^^^^^

Тестирование smq показывает существенно более высокую
производительность, чем у mq.

cleaner
-------

cleaner выполняет обратную запись всех «грязных» блоков в кэше, чтобы
вывести его из эксплуатации.

Примеры
=======

Синтаксис таблицы следующий::

	cache <metadata dev> <cache dev> <origin dev> <block size>
	<#feature_args> [<feature arg>]*
	<policy> <#policy_args> [<policy arg>]*

Синтаксис для отправки сообщения с помощью команды dmsetup
следующий::

	dmsetup message <mapped device> 0 sequential_threshold 1024
	dmsetup message <mapped device> 0 random_threshold 8

Использование dmsetup::

	dmsetup create blah --table "0 268435456 cache /dev/sdb /dev/sdc \
	    /dev/sdd 512 0 mq 4 sequential_threshold 1024 random_threshold 8"
	creates a 128GB large mapped device named 'blah' with the
	sequential threshold set to 1024 and the random_threshold set to 8.
