dm-dust
=======

Этот target эмулирует поведение сбойных секторов в произвольных
местах, а также возможность включать эмуляцию сбоев в произвольный
момент времени.

Данный target ведёт себя аналогично linear target. В заданный момент
времени пользователь может отправить target сообщение, чтобы начать
завершать с ошибкой запросы на чтение определённых блоков (для эмуляции
поведения жёсткого диска со сбойными секторами).

Когда поведение сбоев включено (то есть когда вывод
"dmsetup status" отображает "fail_read_on_bad_block"), чтения блоков
из "bad block list" будут завершаться с ошибкой EIO ("Input/output error").

Запись блоков из "bad block list" приведёт к следующему:

1. Удаление блока из "bad block list".
2. Успешное завершение записи.

Это эмулирует поведение "remapped sector" накопителя со сбойными
секторами.

Как правило, накопитель, на котором появляются сбойные секторы, с
наибольшей вероятностью столкнётся с дополнительными сбойными секторами
в неизвестное время или в неизвестном месте.
С помощью dm-dust пользователь может использовать сообщения "addbadblock"
и "removebadblock" для добавления произвольных сбойных блоков в новых
местах, а также сообщения "enable" и "disable" для управления состоянием
того, будут ли настроенные "bad blocks" рассматриваться как сбойные или
обходиться.
Это позволяет заранее записать тестовые данные и метаданные до
моделирования события "failure", когда начинают появляться сбойные
секторы.

Параметры таблицы
-----------------
<device_path> <offset> <blksz>

Обязательные параметры:
    <device_path>:
        Путь к блочному устройству.

    <offset>:
        Смещение области данных от начала device_path

    <blksz>:
        Размер блока в байтах

	     (минимум 512, максимум 1073741824, должен быть степенью 2)

Инструкции по использованию
---------------------------

Сначала найдите размер (в 512-байтных секторах) используемого устройства::

        $ sudo blockdev --getsz /dev/vdb1
        33552384

Создайте устройство dm-dust:
(Для устройства с размером блока 512 байт)

::

        $ sudo dmsetup create dust1 --table '0 33552384 dust /dev/vdb1 0 512'

(Для устройства с размером блока 4096 байт)

::

        $ sudo dmsetup create dust1 --table '0 33552384 dust /dev/vdb1 0 4096'

Проверьте состояние поведения чтения ("bypass" означает, что весь ввод-вывод
будет передаваться на нижележащее устройство; "verbose" означает, что
добавления, удаления и переотображения сбойных блоков будут подробно
протоколироваться)::

        $ sudo dmsetup status dust1
        0 33552384 dust 252:17 bypass verbose

        $ sudo dd if=/dev/mapper/dust1 of=/dev/null bs=512 count=128 iflag=direct
        128+0 records in
        128+0 records out

        $ sudo dd if=/dev/zero of=/dev/mapper/dust1 bs=512 count=128 oflag=direct
        128+0 records in
        128+0 records out

Добавление и удаление сбойных блоков
------------------------------------

В любой момент (то есть независимо от того, включена или отключена на
устройстве эмуляция "bad block"), сбойные блоки могут быть добавлены или
удалены с устройства с помощью сообщений "addbadblock" и "removebadblock"::

        $ sudo dmsetup message dust1 0 addbadblock 60
        kernel: device-mapper: dust: badblock added at block 60

        $ sudo dmsetup message dust1 0 addbadblock 67
        kernel: device-mapper: dust: badblock added at block 67

        $ sudo dmsetup message dust1 0 addbadblock 72
        kernel: device-mapper: dust: badblock added at block 72

Эти сбойные блоки будут сохранены в "bad block list".
Пока устройство находится в режиме "bypass", чтения и записи будут успешными::

        $ sudo dmsetup status dust1
        0 33552384 dust 252:17 bypass

Включение сбоев чтения блоков
-----------------------------

Чтобы включить поведение "fail read on bad block", отправьте сообщение "enable"::

        $ sudo dmsetup message dust1 0 enable
        kernel: device-mapper: dust: enabling read failures on bad sectors

        $ sudo dmsetup status dust1
        0 33552384 dust 252:17 fail_read_on_bad_block

Когда устройство находится в режиме "fail read on bad block", попытка чтения
блока приведёт к "Input/output error"::

        $ sudo dd if=/dev/mapper/dust1 of=/dev/null bs=512 count=1 skip=67 iflag=direct
        dd: error reading '/dev/mapper/dust1': Input/output error
        0+0 records in
        0+0 records out
        0 bytes copied, 0.00040651 s, 0.0 kB/s

...а запись в сбойные блоки удалит блоки из списка,
тем самым эмулируя поведение "remap" жёстких дисков::

        $ sudo dd if=/dev/zero of=/dev/mapper/dust1 bs=512 count=128 oflag=direct
        128+0 records in
        128+0 records out

        kernel: device-mapper: dust: block 60 removed from badblocklist by write
        kernel: device-mapper: dust: block 67 removed from badblocklist by write
        kernel: device-mapper: dust: block 72 removed from badblocklist by write
        kernel: device-mapper: dust: block 87 removed from badblocklist by write

Обработка ошибок добавления/удаления сбойных блоков
---------------------------------------------------

Попытка добавить сбойный блок, который уже существует в списке, приведёт
к ошибке "Invalid argument", а также к информативному сообщению::

        $ sudo dmsetup message dust1 0 addbadblock 88
        device-mapper: message ioctl on dust1  failed: Invalid argument
        kernel: device-mapper: dust: block 88 already in badblocklist

Попытка удалить сбойный блок, которого нет в списке, приведёт
к ошибке "Invalid argument", а также к информативному сообщению::

        $ sudo dmsetup message dust1 0 removebadblock 87
        device-mapper: message ioctl on dust1  failed: Invalid argument
        kernel: device-mapper: dust: block 87 not found in badblocklist

Подсчёт количества сбойных блоков в списке сбойных блоков
---------------------------------------------------------

Чтобы подсчитать количество сбойных блоков, настроенных на устройстве,
выполните следующую команду-сообщение::

        $ sudo dmsetup message dust1 0 countbadblocks

Будет напечатано сообщение с количеством сбойных блоков, настроенных
на устройстве в данный момент::

        countbadblocks: 895 badblock(s) found

Запрос конкретных сбойных блоков
--------------------------------

Чтобы узнать, находится ли конкретный блок в списке сбойных блоков,
выполните следующую команду-сообщение::

        $ sudo dmsetup message dust1 0 queryblock 72

Следующее сообщение будет напечатано, если блок находится в списке::

        dust_query_block: block 72 found in badblocklist

Следующее сообщение будет напечатано, если блок не находится в списке::

        dust_query_block: block 72 not found in badblocklist

Команда-сообщение "queryblock" будет работать как в режиме "enabled",
так и в режиме "disabled", позволяя проверить, будет ли блок
рассматриваться как "bad", без необходимости выполнять ввод-вывод на
устройство или "enable" эмуляцию сбойных блоков.

Очистка списка сбойных блоков
-----------------------------

Чтобы очистить список сбойных блоков (без необходимости по отдельности
выполнять команду-сообщение "removebadblock" для каждого блока), выполните
следующую команду-сообщение::

        $ sudo dmsetup message dust1 0 clearbadblocks

После очистки списка сбойных блоков появится следующее сообщение::

        dust_clear_badblocks: badblocks cleared

Если сбойных блоков для очистки не было, появится следующее
сообщение::

        dust_clear_badblocks: no badblocks found

Вывод списка сбойных блоков
---------------------------

Чтобы вывести все сбойные блоки из списка сбойных блоков (на примере
устройства с блоками 1 и 2 в списке сбойных блоков), выполните следующую
команду-сообщение::

        $ sudo dmsetup message dust1 0 listbadblocks
        1
        2

Если в списке сбойных блоков нет сбойных блоков, команда
выполнится без вывода::

        $ sudo dmsetup message dust1 0 listbadblocks

Список команд-сообщений
-----------------------

Ниже приведён список сообщений, которые можно отправить устройству dust:

Операции над блоками (требуют аргумента <blknum>)::

        addbadblock <blknum>
        queryblock <blknum>
        removebadblock <blknum>

...где <blknum> — это номер блока в пределах диапазона устройства
(соответствующий размеру блока устройства.)

Команды-сообщения с одним аргументом::

        countbadblocks
        clearbadblocks
        listbadblocks
        disable
        enable
        quiet

Удаление устройства
-------------------

По завершении удалите устройство с помощью команды "dmsetup remove"::

        $ sudo dmsetup remove dust1

Тихий режим
-----------

При тестовых запусках с большим количеством сбойных блоков может быть
желательно избежать чрезмерного протоколирования (от добавленных, удалённых
или "remapped" сбойных блоков).
Это можно сделать, включив "quiet mode" с помощью следующего сообщения::

        $ sudo dmsetup message dust1 0 quiet

Это подавит сообщения журнала от операций добавления / удаления / удаления
записью. Сообщения журнала от команд-сообщений "countbadblocks" или
"queryblock" по-прежнему будут печататься в тихом режиме.

Состояние тихого режима можно увидеть, выполнив "dmsetup status"::

        $ sudo dmsetup status dust1
        0 33552384 dust 252:17 fail_read_on_bad_block quiet

Чтобы отключить тихий режим, отправьте сообщение "quiet" повторно::

        $ sudo dmsetup message dust1 0 quiet

        $ sudo dmsetup status dust1
        0 33552384 dust 252:17 fail_read_on_bad_block verbose

(Наличие "verbose" означает обычное протоколирование.)

"Почему не...?"
---------------

scsi_debug имеет режим "medium error", который может завершать с ошибкой
чтения одного указанного сектора (сектор 0x1234, жёстко прописанный в
исходном коде), но он использует RAM для постоянного хранилища, что
радикально уменьшает потенциальный размер устройства.

dm-flakey завершает с ошибкой весь ввод-вывод из всех мест блочного
устройства с указанной частотой по времени, а не в заданный момент времени.

Когда на жёстком диске возникает сбойный сектор, чтения этого сектора
завершаются устройством с ошибкой, обычно приводя к коду ошибки EIO
("I/O error") или ENODATA ("No data available"). Однако запись в
сектор может завершиться успешно и привести к тому, что сектор станет
читаемым после того, как контроллер устройства перестанет испытывать
ошибки при чтении сектора (или после переразмещения сектора). Однако
в будущем на устройстве могут возникнуть сбойные секторы в другом,
непредсказуемом месте.

Данный target стремится предоставить устройство, способное демонстрировать
поведение сбойного сектора в известном месте сектора, в известное время,
на основе большого устройства хранения (по меньшей мере десятки гигабайт,
не занимающего системную память).
