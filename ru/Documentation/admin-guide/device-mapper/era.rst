======
dm-era
======

Введение
========

dm-era — это target, поведение которого похоже на linear target. Вдобавок
он отслеживает, какие блоки были записаны в течение заданного пользователем
периода времени, называемого «эрой» (era). Каждый экземпляр era target
поддерживает текущую эру в виде монотонно возрастающего 32-битного счётчика.

Сценарии использования включают отслеживание изменённых блоков для ПО
резервного копирования, а также частичную инвалидацию содержимого кэша для
восстановления когерентности кэша после отката снимка, сделанного средствами
производителя.

Конструктор
===========

era <metadata dev> <origin dev> <block size>

 ================ ======================================================
 metadata dev     быстрое устройство, хранящее постоянные метаданные
 origin dev	  устройство, хранящее блоки данных, которые могут меняться
 block size       размер блока устройства данных origin, гранулярность,
		  которая отслеживается target
 ================ ======================================================

Сообщения
=========

Ни одно из сообщений dm не принимает аргументов.

checkpoint
----------

Возможно, перейти в новую эру. Не следует предполагать, что эра увеличилась.
После отправки этого сообщения следует проверить текущую эру через строку
состояния.

take_metadata_snap
------------------

Создать клон метаданных, чтобы процесс в пространстве пользователя мог их
прочитать.

drop_metadata_snap
------------------

Удалить снимок метаданных.

Состояние
=========

<metadata block size> <#used metadata blocks>/<#total metadata blocks>
<current era> <held metadata root | '-'>

========================= ==============================================
metadata block size	  Фиксированный размер каждого блока метаданных
			  в секторах
#used metadata blocks	  Число использованных блоков метаданных
#total metadata blocks	  Общее число блоков метаданных
current era		  Текущая эра
held metadata root	  Расположение, в блоках, корня метаданных,
			  который был «удержан» (held) для доступа на
			  чтение из пространства пользователя. '-'
			  означает, что удержанного корня нет
========================= ==============================================

Подробный сценарий использования
================================

Сценарий инвалидации кэша при откате снимка, сделанного средствами
производителя, был основным сценарием использования при разработке этого
target:

Создание снимка средствами производителя
----------------------------------------

- Отправить сообщение checkpoint в era target
- Записать текущую эру из его строки состояния
- Сделать снимок средствами производителя (теперь эра и снимок должны быть
  навсегда связаны).

Откат к снимку, сделанному средствами производителя
---------------------------------------------------

- Кэш переходит в режим passthrough (см.: документацию по dm-cache в cache.txt)
- Откатить хранилище производителя
- Сделать снимок метаданных
- Установить, какие блоки были записаны с момента создания снимка, проверив
  эру каждого блока
- Инвалидировать эти блоки в кэширующем ПО
- Кэш возвращается в режим writeback/writethrough

Использование памяти
====================

Target использует битовое множество (bitset) для записи операций записи в
текущей эре. Также у него наготове есть запасное битовое множество для
переключения на новую эру. Помимо этого, он использует несколько блоков по 4k
для обновления метаданных::

   (4 * nr_blocks) bytes + buffers

Устойчивость
============

Метаданные обновляются на диске до того, как выполняется запись в ранее не
записанный блок. Благодаря этому dm-era не должен страдать от внезапного сбоя,
такого как отключение питания.
