=====
Cache
=====

Введение
========

dm-cache — это device-mapper target, написанный Joe Thornber, Heinz
Mauelshagen и Mike Snitzer.

Его цель — повысить производительность блочного устройства (например,
шпиндельного диска) за счёт динамической миграции части его данных на
более быстрое устройство меньшего объёма (например, SSD).

Это решение на базе device-mapper позволяет нам встраивать такое
кэширование на разных уровнях стека dm, например, над устройством данных
для пула с thin-provisioning.  Решения для кэширования, более тесно
интегрированные с системой виртуальной памяти, должны давать лучшую
производительность.

Этот target повторно использует библиотеку метаданных, применяемую в
библиотеке thin-provisioning.

Решение о том, какие данные мигрировать и когда, оставлено на усмотрение
подключаемого модуля политики (plug-in policy).  Несколько таких модулей
было написано в ходе наших экспериментов, и мы надеемся, что другие люди
напишут собственные для конкретных сценариев io (например, для сервера
образов vm).

Глоссарий
=========

  Migration
	       Перемещение основной копии логического блока с одного
	       устройства на другое.
  Promotion
	       Миграция с медленного устройства на быстрое.
  Demotion
	       Миграция с быстрого устройства на медленное.

Устройство origin всегда содержит копию логического блока, которая может
быть устаревшей либо поддерживаться в синхронизации с копией на устройстве
cache (в зависимости от политики).

Устройство
==========

Подустройства
-------------

Этот target создаётся путём передачи ему трёх устройств (вместе с другими
параметрами, описанными далее):

1. Устройство origin — большое и медленное.

2. Устройство cache — маленькое и быстрое.

3. Небольшое устройство метаданных — фиксирует, какие блоки находятся в
   кэше, какие из них «грязные» (dirty), а также дополнительные подсказки
   для использования объектом политики.
   Эту информацию можно было бы разместить на устройстве cache, но
   хранение её отдельно позволяет менеджеру томов настраивать её иначе,
   например, как зеркало для дополнительной надёжности.  Это устройство
   метаданных может использоваться только одним устройством cache.

Фиксированный размер блока
--------------------------

Устройство origin разбивается на блоки фиксированного размера.  Этот
размер блока настраивается при первом создании кэша.  Обычно мы
использовали размеры блока от 256KB до 1024KB.  Размер блока должен быть
между 64 секторами (32KB) и 2097152 секторами (1GB) и кратным 64 секторам
(32KB).

Фиксированный размер блока значительно упрощает target.  Но это
определённый компромисс.  Например, небольшая часть блока может
обращаться очень часто, и при этом в кэш будет повышен (promoted) весь
блок целиком.  Поэтому большие размеры блока плохи тем, что они
расходуют пространство кэша впустую.  А маленькие размеры блока плохи
тем, что увеличивают объём метаданных (как в оперативной памяти, так и на
диске).

Режимы работы кэша
------------------

Кэш имеет три режима работы: writeback, writethrough и passthrough.

Если выбран writeback (используется по умолчанию), то запись в
кэшированный блок пойдёт только в кэш, а блок будет помечен как «грязный»
(dirty) в метаданных.

Если выбран writethrough, то запись в кэшированный блок не завершится до
тех пор, пока она не попадёт как на устройство origin, так и на устройство
cache.  «Чистые» (clean) блоки должны оставаться чистыми.

Если выбран passthrough — полезно, когда неизвестно, согласовано ли
содержимое кэша с устройством origin, — то все чтения обслуживаются с
устройства origin (все чтения промахиваются мимо кэша), а все записи
перенаправляются на устройство origin; кроме того, попадания при записи
(write hits) вызывают инвалидацию блоков кэша.  Чтобы включить режим
passthrough, кэш должен быть чистым (clean).  Режим passthrough позволяет
активировать устройство cache без необходимости беспокоиться о
согласованности.  Существующая согласованность поддерживается, хотя по
мере выполнения записей кэш будет постепенно остывать.  Если позднее
согласованность кэша удастся проверить или установить с помощью сообщения
"invalidate_cblocks", устройство cache может быть переведено в режим
writethrough или writeback, оставаясь при этом ещё «тёплым».  В противном
случае содержимое кэша можно сбросить перед переходом в желаемый режим
работы.

Предусмотрена простая политика очистки (cleaner), которая очистит
(запишет обратно) все «грязные» блоки в кэше.  Полезно при выводе кэша из
эксплуатации или при уменьшении кэша.  Уменьшение быстрого устройства
кэша требует, чтобы все блоки кэша в той области кэша, которая удаляется,
были чистыми.  Если удаляемая из кэша область всё ещё содержит «грязные»
блоки, изменение размера завершится неудачей.  Необходимо позаботиться о
том, чтобы никогда не уменьшать том, используемый для быстрого устройства
кэша, до тех пор, пока кэш не станет чистым.  Это особенно важно при
использовании режима writeback.  Режимы writethrough и passthrough уже
поддерживают кэш чистым.  Будущая поддержка частичной очистки кэша, сверх
заданного порога, позволит сохранять кэш «тёплым» и в режиме writeback во
время изменения размера.

Регулирование миграции
-----------------------

Миграция данных между устройством origin и устройством cache потребляет
пропускную способность.  Пользователь может задать ограничение
(throttle), чтобы предотвратить выполнение более определённого объёма
миграции в любой момент времени.  В настоящее время мы никак не учитываем
обычный трафик io, идущий к устройствам.  Здесь требуется доработка, чтобы
избегать миграции в моменты пиковой нагрузки io.

Пока что для задания максимального числа мигрируемых секторов можно
использовать сообщение "migration_threshold <#sectors>"; значение по
умолчанию — 2048 секторов (1MB).

Обновление метаданных на диске
------------------------------

Метаданные на диске фиксируются (committed) каждый раз при записи bio с
флагом FLUSH или FUA.  Если таких запросов не поступает, то фиксация
происходит каждую секунду.  Это означает, что кэш ведёт себя как
физический диск с энергозависимым кэшем записи.  При отключении питания
вы можете потерять часть недавних записей.  Метаданные должны всегда
оставаться согласованными, несмотря на любой сбой.

Состояние «dirty» для блока кэша меняется слишком часто, чтобы мы могли
обновлять его на лету.  Поэтому мы трактуем его как подсказку.  При
нормальной работе оно будет записано при приостановке (suspend) dm-устройства.
Если система аварийно завершит работу, после перезапуска все блоки кэша
будут считаться «грязными».

Подсказки политики для каждого блока
------------------------------------

Подключаемые модули политики (plug-ins) могут хранить порцию данных для
каждого блока кэша.  Размер этой порции определяется политикой, но он
должен оставаться небольшим.  Подобно флагам dirty, эти данные теряются
при сбое, поэтому всегда должно быть возможно безопасное значение по
умолчанию.

Подсказки политики влияют на производительность, а не на корректность.

Обмен сообщениями с политикой
-----------------------------

У политик будут различные настраиваемые параметры (tunables), специфичные
для каждой из них, поэтому нам нужен универсальный способ их получения и
установки.  Для этого используются сообщения device-mapper.  См.
cache-policies.txt.

Разрешение битовой карты discard
--------------------------------

Мы можем избежать копирования данных во время миграции, если знаем, что
блок был отброшен (discarded).  Яркий пример этого — когда mkfs отбрасывает
всё блочное устройство.  Мы храним битовую карту (bitset), отслеживающую
состояние discard блоков.  Однако мы допускаем, чтобы у этой битовой карты
был размер блока, отличный от размера блоков кэша.  Это связано с тем, что
нам нужно отслеживать состояние discard для всего устройства origin
(сравните с битовой картой dirty, которая относится только к меньшему
устройству cache).

Интерфейс target
================

Конструктор
-----------

  ::

   cache <metadata dev> <cache dev> <origin dev> <block size>
         <#feature args> [<feature arg>]*
         <policy> <#policy args> [policy args]*

 ================ =======================================================
 metadata dev     быстрое устройство, хранящее постоянные метаданные
 cache dev        быстрое устройство, хранящее кэшированные блоки данных
 origin dev       медленное устройство, хранящее исходные блоки данных
 block size       размер единицы кэша в секторах

 #feature args    число передаваемых аргументов возможностей
 feature args     writethrough или passthrough (по умолчанию writeback)

 policy           используемая политика замещения
 #policy args     чётное число аргументов, соответствующих парам
                  ключ/значение, передаваемым политике
 policy args      пары ключ/значение, передаваемые политике
                  Например, 'sequential_threshold 1024'
                  Подробности см. в cache-policies.txt.
 ================ =======================================================

Необязательные аргументы возможностей (feature):


   ==================== ========================================================
   writethrough         кэширование со сквозной записью, при котором
                        содержимое блока кэша не может отличаться от
                        содержимого блока origin.  Без этого аргумента
                        поведение по умолчанию состоит в том, чтобы по
                        соображениям производительности записывать
                        содержимое блока кэша обратно позднее, поэтому оно
                        может отличаться от соответствующих блоков origin.

   passthrough          ухудшенный режим, полезный в различных ситуациях с
                        согласованностью кэша (например, при откате снимков
                        нижележащего хранилища).  Чтения и записи всегда идут
                        на origin.  Если запись попадает в кэшированный блок
                        origin, то блок кэша инвалидируется.
                        Чтобы включить режим passthrough, кэш должен быть
                        чистым.

   metadata2            использовать версию 2 метаданных.  При этом биты dirty
                        хранятся в отдельном btree, что повышает скорость
                        завершения работы кэша.

   no_discard_passdown  отключить передачу discard-запросов от кэша
                        на устройство данных origin.
   ==================== ========================================================

Политика с именем 'default' зарегистрирована всегда.  Это псевдоним
(alias) для той политики, которая, по нашему текущему мнению, даёт лучшую
всестороннюю производительность.

Поскольку политика по умолчанию может различаться между ядрами, если вы
полагаетесь на характеристики конкретной политики, всегда запрашивайте её
по имени.

                          В серьёзных случаях, когда даже режим только для

                          не будет разрешён, и статус будет содержать просто
                          строку 'Fail'.  Тогда следует использовать
                          инструменты восстановления из пространства
                          пользователя.
needs_check               'needs_check', если установлен, '-' если нет
                          Операция с метаданными завершилась неудачей, в

Сообщения
---------

У политик будут различные настраиваемые параметры (tunables), специфичные
для каждой из них, поэтому нам нужен универсальный способ их получения и
установки.  Для этого используются сообщения device-mapper.  (Интерфейс на
основе sysfs также был бы возможен.)

Формат сообщения::

   <key> <value>

Например::

   dmsetup message my_cache 0 sequential_threshold 1024


Инвалидация — это удаление записи из кэша без её записи обратно.  Блоки
кэша можно инвалидировать с помощью сообщения invalidate_cblocks, которое
принимает произвольное число диапазонов cblock.  Конечное значение каждого
диапазона cblock — это «один за последним», то есть 5-10 выражает диапазон
значений от 5 до 9.  Каждый cblock должен быть выражен десятичным
значением; в будущем может потребоваться вариант сообщения, принимающий
диапазоны cblock в шестнадцатеричном виде, чтобы лучше поддерживать
эффективную инвалидацию более крупных кэшей.  При использовании
invalidate_cblocks кэш должен находиться в режиме passthrough::

   invalidate_cblocks [<cblock>|<cblock begin>-<cblock end>]*

Например::

   dmsetup message my_cache 0 invalidate_cblocks 2345 3456-4567 5678-6789
