============================
Журналирование Device-Mapper
============================
Код журналирования device-mapper используется некоторыми RAID-целями
device-mapper для отслеживания областей диска, находящихся в несогласованном
состоянии. Область (или часть адресного пространства) диска может быть
несогласованной из-за того, что в данный момент выполняется операция над
RAID-страйпом, или из-за того, что машина вышла из строя во время изменения
этой области. В случае зеркал область считается «грязной»/несогласованной во
время записи в неё, поскольку записи должны быть реплицированы на все ветви
зеркала и могут достигать ветвей не одновременно. После завершения всех
записей область снова считается «чистой».

Существует обобщённый интерфейс журналирования, который реализации RAID
device-mapper используют для выполнения операций журналирования (см.
dm_dirty_log_type в include/linux/dm-dirty-log.h). Доступны различные
реализации журналирования, предоставляющие разные возможности. В список входят:

==============	==============================================================
Тип		Файлы
==============	==============================================================
disk		drivers/md/dm-log.c
core		drivers/md/dm-log.c
userspace	drivers/md/dm-log-userspace* include/linux/dm-log-userspace.h
==============	==============================================================

Тип журнала «disk»
------------------
Эта реализация журнала фиксирует состояние журнала на диске. Таким образом,
состояние журналирования сохраняется при перезагрузках/сбоях.

Тип журнала «core»
------------------
Эта реализация журнала хранит состояние журнала в памяти. Состояние журнала
не сохраняется при перезагрузке или сбое, но возможно небольшое повышение
производительности. Этот метод также можно использовать, если для хранения
состояния журнала недоступно ни одно устройство хранения.

Тип журнала «userspace»
-----------------------
Этот тип журнала просто предоставляет способ экспортировать API журналирования
в пространство пользователя (userspace), благодаря чему реализации
журналирования могут выполняться там. Это достигается путём перенаправления
большинства запросов журналирования в пространство пользователя, где запрос
принимает и обрабатывает демон.

Структуры, используемые для обмена данными между ядром и пространством
пользователя, расположены в include/linux/dm-log-userspace.h. Из-за частоты,
разнообразия и двунаправленного характера обменов между ядром и пространством
пользователя в качестве интерфейса для обмена данными используется «connector».

В настоящее время существуют две реализации журнала пространства пользователя,
которые используют этот фреймворк — «clustered-disk» и «clustered-core». Эти
реализации предоставляют когерентный в рамках кластера журнал для разделяемого
хранилища. Зеркалирование device-mapper может использоваться в среде с
разделяемым хранилищем, когда применяются кластерные реализации журнала.
