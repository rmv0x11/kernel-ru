=========================
Цель (target) writecache
=========================

Цель writecache кэширует операции записи на persistent memory или на SSD.
Она не кэширует операции чтения, поскольку предполагается, что чтения
кэшируются в page cache в обычной RAM.

При создании устройства первый сектор должен быть обнулён, либо первый
сектор должен содержать корректный суперблок от предыдущего запуска.

Параметры конструктора:

1. тип кэширующего устройства - "p" или "s"
	- p - persistent memory
	- s - SSD
2. нижележащее устройство, которое будет кэшироваться
3. кэширующее устройство
4. размер блока (рекомендуется 4096; максимальный размер блока равен
   размеру страницы)
5. количество необязательных параметров (параметры с аргументом
   считаются за два)

	start_sector n		(по умолчанию: 0)
		смещение от начала кэширующего устройства в 512-байтных секторах
	high_watermark n	(по умолчанию: 50)
		начинать обратную запись (writeback), когда число используемых
		блоков достигает этого порога
	low_watermark x		(по умолчанию: 45)
		останавливать обратную запись (writeback), когда число
		используемых блоков опускается ниже этого порога
	writeback_jobs n	(по умолчанию: без ограничений)
		ограничить число блоков, находящихся в обработке во время
		обратной записи. Установка этого значения снижает пропускную
		способность обратной записи, но может улучшить задержку (latency)
		запросов чтения
	autocommit_blocks n	(по умолчанию: 64 для pmem, 65536 для ssd)
		когда приложение записывает такое количество блоков, не выдавая
		запрос FLUSH, блоки фиксируются (commit) автоматически
	autocommit_time ms	(по умолчанию: 1000)
		время автоматической фиксации в миллисекундах. Данные фиксируются
		автоматически, если истекает это время и не получен запрос FLUSH
	fua			(по умолчанию включено)
		применимо только к persistent memory - использовать флаг FUA при
		записи данных из persistent memory обратно на нижележащее
		устройство
	nofua
		применимо только к persistent memory - не использовать флаг FUA
		при обратной записи данных, а после этого отправлять запрос FLUSH

		- некоторые нижележащие устройства работают лучше с fua, некоторые
		  с nofua. Пользователю следует протестировать это
	cleaner
		когда эта опция активирована (либо в аргументах конструктора,
		либо сообщением), кэш не будет продвигать (promote) новые записи
		(однако записи в уже кэшированные блоки продвигаются, чтобы
		избежать повреждения данных из-за нарушения порядка записей) и
		будет постепенно выполнять обратную запись всех кэшированных
		данных. После этого пространство пользователя может отслеживать
		процесс очистки с помощью "dmsetup status". Когда число
		кэшированных блоков опускается до нуля, пространство пользователя
		может выгрузить цель dm-writecache и заменить её на dm-linear или
		другие цели.
	max_age n
		задаёт максимальный возраст блока в миллисекундах. Если блок
		хранится в кэше слишком долго, он будет записан на нижележащее
		устройство и очищен.
	metadata_only
		в кэш продвигаются только метаданные. Эта опция улучшает
		производительность для более нагруженных рабочих нагрузок
		REQ_META.
	pause_writeback n	(по умолчанию: 3000)
		приостанавливать обратную запись, если за последние n
		миллисекунд был какой-либо ввод-вывод записи, перенаправленный на
		исходный (origin) том

Состояние (Status):

1. индикатор ошибки - 0, если ошибки не было, иначе номер ошибки
2. число блоков
3. число свободных блоков
4. число блоков, находящихся в обратной записи
5. число прочитанных блоков
6. число прочитанных блоков, попавших в кэш
7. число записанных блоков
8. число записанных блоков, попавших в незафиксированный блок
9. число записанных блоков, попавших в зафиксированный блок
10. число записанных блоков, минующих кэш
11. число записанных блоков, выделенных в кэше
12. число запросов записи, заблокированных на списке свободных блоков (freelist)
13. число запросов сброса (flush)
14. число отброшенных (discarded) блоков

Сообщения (Messages):
	flush
		Сбросить (flush) кэширующее устройство. Сообщение возвращает
		успех, если кэширующее устройство было сброшено без ошибки
	flush_on_suspend
		Сбросить кэширующее устройство при следующей приостановке
		(suspend). Используйте это сообщение, когда собираетесь удалить
		кэширующее устройство. Правильная последовательность удаления
		кэширующего устройства такова:

		1. отправить сообщение "flush_on_suspend"
		2. загрузить неактивную таблицу с целью linear, которая
		   отображается на нижележащее устройство
		3. приостановить устройство
		4. запросить состояние и убедиться, что ошибок нет
		5. возобновить устройство, чтобы оно использовало цель linear
		6. кэширующее устройство теперь неактивно и может быть удалено
	cleaner
		См. выше документацию по конструктору "cleaner".
	clear_stats
		Очистить статистику, которая выводится в строке состояния
