==================================
dm-pcache — Persistent Cache
==================================

*Автор: Dongsheng Yang <dongsheng.yang@linux.dev>*

Этот документ описывает *dm-pcache* — target Device-Mapper, позволяющий
байт-адресуемому региону *DAX* (persistent-memory, «pmem») выступать в роли
высокопроизводительного, устойчивого к сбоям кэша перед более медленным
блочным устройством.  Код располагается в `drivers/md/dm-pcache/`.

Краткая сводка возможностей
===========================

* Кэширование в режиме *Write-back* (единственный поддерживаемый сейчас режим).
* *Сегменты по 16 MiB*, выделяемые на устройстве pmem.
* Проверка *CRC32 данных* (необязательная, для каждого кэша).
* Устойчивость к сбоям: каждая структура метаданных дублируется (`PCACHE_META_INDEX_MAX
  == 2`) и защищена с помощью CRC и порядковых номеров.
* *Многодеревная индексация* (деревья индексов шардируются по логическому адресу) для высокого параллелизма PMem
* Чистый *путь DAX* для I/O — без дополнительных проходов BIO
* *Log-structured write-back*, сохраняющий устойчивость backend-а к сбоям


Конструктор
===========

::

    pcache <cache_dev> <backing_dev> [<number_of_optional_arguments> <cache_mode writeback> <data_crc true|false>]

=========================  ====================================================
``cache_dev``               Любое DAX-совместимое блочное устройство (``/dev/pmem0``…).
                            Здесь хранятся все метаданные *и* кэшированные блоки.

``backing_dev``             Медленное блочное устройство, которое кэшируется.

``cache_mode``              Необязательный, на данный момент принимается только
                            ``writeback``.

``data_crc``                Необязательный, по умолчанию ``false``

                            * ``true``  – хранить CRC32 для каждой кэшированной записи
			      и проверять при чтении
                            * ``false`` – не вычислять CRC (быстрее)
=========================  ====================================================

Пример
------

.. code-block:: shell

   dmsetup create pcache_sdb --table \
     "0 $(blockdev --getsz /dev/sdb) pcache /dev/pmem0 /dev/sdb 4 cache_mode writeback data_crc true"

При первом использовании устройства pmem dm-pcache форматирует его автоматически
(super-block, cache_info и т.д.).


Строка состояния
================

``dmsetup status <device>`` (``STATUSTYPE_INFO``) выводит:

::

   <sb_flags> <seg_total> <cache_segs> <segs_used> \
   <gc_percent> <cache_flags> \
   <key_head_seg>:<key_head_off> \
   <dirty_tail_seg>:<dirty_tail_off> \
   <key_tail_seg>:<key_tail_off>

Значения полей
--------------

===============================  =============================================
``sb_flags``                     Флаги super-block (например, маркер порядка байт).

``seg_total``                    Число физических сегментов *pmem*.

``cache_segs``                   Число сегментов, используемых под кэш.

``segs_used``                    Сегменты, выделенные в данный момент (вес битовой маски).

``gc_percent``                   Текущий порог срабатывания GC (0-90).

``cache_flags``                  Бит 0 – включён DATA_CRC
                                 Бит 1 – INIT_DONE (кэш инициализирован)
                                 Биты 2-5 – режим кэша (0 == WB).

``key_head``                     Куда записываются новые key-set-ы.

``dirty_tail``                   Первый «грязный» key-set, которому ещё требуется
                                 write-back на backing-устройство.

``key_tail``                     Первый key-set, который может быть освобождён GC.
===============================  =============================================


Сообщения
=========

*Изменение порога срабатывания GC*

::

   dmsetup message <dev> 0 gc_percent <0-90>


Принцип работы
==============

Под-устройства
--------------

====================  =========================================================
backing_dev             Любое блочное устройство (SSD/HDD/loop/LVM и т.д.).
cache_dev               DAX-устройство; должно предоставлять память прямого доступа.
====================  =========================================================

Сегменты и key-set-ы
--------------------

* Пространство pmem делится на *сегменты по 16 MiB*.
* Каждая запись выделяет место из *data_head*, своего для каждого CPU, внутри сегмента.
* *cache-key* фиксирует логический диапазон на origin и место его размещения
  внутри pmem (сегмент + смещение + поколение).
* 128 ключей образуют *key-set* (kset); kset-ы записываются в pmem последовательно
  и сами по себе устойчивы к сбоям (CRC).
* Пара *(key_tail, dirty_tail)* разграничивает чистые/грязные и живые/мёртвые kset-ы.

Write-back
----------

Грязные ключи помещаются в очередь в дереве; фоновый рабочий поток копирует данные
обратно на backing_dev и продвигает *dirty_tail*.  Bio с флагом FLUSH/FUA от
вышележащих слоёв принудительно вызывает немедленную фиксацию метаданных.

Сборка мусора
-------------

GC запускается, когда ``segs_used >= seg_total * gc_percent / 100``.  Он проходит
от *key_tail*, освобождает сегменты, все ключи которых были инвалидированы, и
продвигает *key_tail*.

Проверка CRC
------------

Если ``data_crc is enabled``, dm-pcache вычисляет CRC32 для каждого кэшируемого
диапазона данных при его вставке и сохраняет его в ключе на носителе.  Чтения
проверяют CRC перед копированием вызывающей стороне.


Обработка сбоев
===============

* *Ошибки носителя pmem* – все копии метаданных читаются с помощью
  ``copy_mc_to_kernel``; неисправимая ошибка логируется и прерывает инициализацию.
* *Кэш заполнен* – если не удаётся найти свободный сегмент, записи возвращают ``-EBUSY``;
  dm-pcache повторяет попытку внутренне (отложенная обработка запроса).
* *Сбой системы* – при подключении драйвер воспроизводит kset-ы начиная с *key_tail*,
  чтобы перестроить деревья в памяти; поколение каждого сегмента защищает от
  ключей с обращением после освобождения (use-after-free).


Ограничения и TODO
==================

* Только режим *write-back*; другие режимы запланированы.
* Только FIFO-инвалидация кэша; другие (LRU, ARC...) запланированы.
* Перезагрузка таблицы в настоящее время не поддерживается.
* Discard запланирован.


Пример рабочего процесса
========================

.. code-block:: shell

   # 1.  Create devices
   dmsetup create pcache_sdb --table \
     "0 $(blockdev --getsz /dev/sdb) pcache /dev/pmem0 /dev/sdb 4 cache_mode writeback data_crc true"

   # 2.  Put a filesystem on top
   mkfs.ext4 /dev/mapper/pcache_sdb
   mount /dev/mapper/pcache_sdb /mnt

   # 3.  Tune GC threshold to 80 %
   dmsetup message pcache_sdb 0 gc_percent 80

   # 4.  Observe status
   watch -n1 'dmsetup status pcache_sdb'

   # 5.  Shutdown
   umount /mnt
   dmsetup remove pcache_sdb


``dm-pcache`` находится в активной разработке; отзывы, сообщения об ошибках и
патчи очень приветствуются!
