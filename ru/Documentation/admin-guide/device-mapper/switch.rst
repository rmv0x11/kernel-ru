=========
dm-switch
=========

Target switch подсистемы device-mapper создаёт устройство, которое поддерживает
произвольное отображение регионов ввода-вывода (I/O) фиксированного размера на
фиксированный набор путей. Путь, используемый для любого конкретного региона,
может динамически переключаться отправкой target'у сообщения.

Он эффективно отображает I/O на нижележащие блочные устройства в ситуации, когда
имеется большое число адресных регионов фиксированного размера, но отсутствует
простой шаблон, который позволил бы компактно представить отображение, как это
делает dm-stripe.

Предпосылки
-----------

Системы хранения Dell EqualLogic и некоторые другие iSCSI-массивы хранения
используют распределённую безфреймовую архитектуру. В этой архитектуре группа
хранения состоит из нескольких отдельных массивов хранения («членов» — members),
каждый из которых имеет независимые контроллеры, дисковые накопители и сетевые
адаптеры. При создании LUN он распределяется по нескольким членам. Детали
распределения скрыты от инициаторов, подключённых к этой системе хранения.
Группа хранения предоставляет единый портал обнаружения target'ов, независимо от
того, сколько членов используется. При создании iSCSI-сессий каждая сессия
подключается к eth-порту на отдельном члене. Данные для LUN могут быть отправлены
по любой iSCSI-сессии, и если запрашиваемые блоки хранятся на другом члене, I/O
будет перенаправлен по необходимости. Это перенаправление невидимо для инициатора.
Раскладка хранения также является динамической, и блоки, хранящиеся на диске,
могут перемещаться от члена к члену по мере необходимости для балансировки нагрузки.

Эта архитектура упрощает управление и настройку как группы хранения, так и
инициаторов. В конфигурации с multipathing можно настроить несколько iSCSI-сессий
для использования нескольких сетевых интерфейсов как на хосте, так и на target'е,
чтобы воспользоваться увеличенной пропускной способностью сети. Инициатор мог бы
применять простой алгоритм round robin для отправки I/O по всем путям и позволять
членам массива хранения перенаправлять его по необходимости, но при отправке данных
напрямую правильному члену есть выигрыш в производительности.

Таблица device-mapper уже позволяет отображать различные регионы устройства на
различные target'ы. Однако в этой архитектуре LUN распределяется с размером
адресного региона порядка десятков МБ, что означает, что результирующая таблица
могла бы иметь более миллиона записей и потреблять слишком много памяти.

Используя этот target switch подсистемы device-mapper, мы теперь можем построить
двухуровневую иерархию устройств:

    Верхний уровень (Upper Tier) — определяет, на какой член массива должен быть
    отправлен I/O.
    Нижний уровень (Lower Tier) — балансирует нагрузку среди путей к конкретному
    члену.

Нижний уровень состоит из одного устройства dm multipath для каждого члена.
Каждое из этих multipath-устройств содержит набор путей напрямую к члену массива
в одной priority group и использует существующие селекторы путей для балансировки
нагрузки среди этих путей. Мы также строим неприоритетную priority group,
содержащую пути к другим членам массива, на случай failover.

Верхний уровень состоит из одного устройства dm-switch. Это устройство использует
битовую карту (bitmap) для поиска расположения I/O и выбора подходящего устройства
нижнего уровня для маршрутизации I/O. Используя битовую карту, мы можем задействовать
4 бита для каждого адресного диапазона в группе из 16 членов (что для нас весьма
много). Это гораздо более плотное представление, чем то, которого может достичь
b-дерево таблицы dm.

Параметры конструктора
======================

    <num_paths> <region_size> <num_optional_args> [<optional_args>...] [<dev_path> <offset>]+
	<num_paths>
	    Число путей, по которым распределяется I/O.

	<region_size>
	    Число 512-байтовых секторов в регионе. Каждый регион может быть перенаправлен
	    на любой из доступных путей.

	<num_optional_args>
	    Число необязательных аргументов. В настоящее время никакие необязательные
	    аргументы не поддерживаются, поэтому это значение должно быть равно нулю.

	<dev_path>
	    Блочное устройство, представляющее конкретный путь к устройству.

	<offset>
	    Смещение начала данных на конкретном <dev_path> (в единицах
	    512-байтовых секторов). Это число добавляется к номеру сектора при
	    перенаправлении запроса на конкретный путь. Обычно оно равно нулю.

Сообщения
=========

set_region_mappings <index>:<path_nr> [<index>]:<path_nr> [<index>]:<path_nr>...

Изменяет таблицу регионов, указывая, какие регионы перенаправляются на какие пути.

<index>
    Номер региона (размер региона задавался в параметрах конструктора).
    Если index опущен, используется следующий регион (предыдущий index + 1).
    Выражается в шестнадцатеричном виде (БЕЗ какого-либо префикса вроде 0x).

<path_nr>
    Номер пути в диапазоне 0 ... (<num_paths> - 1).
    Выражается в шестнадцатеричном виде (БЕЗ какого-либо префикса вроде 0x).

R<n>,<m>
    Этот параметр позволяет быстро загружать повторяющиеся шаблоны. <n> и <m>
    являются шестнадцатеричными числами. Последние <n> отображений повторяются
    в следующих <m> слотах.

Состояние
=========

Строка состояния не сообщается.

Пример
======

Предположим, что у вас есть тома vg1/switch0 vg1/switch1 vg1/switch2 одинакового
размера.

Создайте устройство switch с размером региона 64 КБ::

    dmsetup create switch --table "0 `blockdev --getsz /dev/vg1/switch0`
	switch 3 128 0 /dev/vg1/switch0 0 /dev/vg1/switch1 0 /dev/vg1/switch2 0"

Задайте отображения для первых 7 записей так, чтобы они указывали на устройства
switch0, switch1, switch2, switch0, switch1, switch2, switch1::

    dmsetup message switch 0 set_region_mappings 0:0 :1 :2 :0 :1 :2 :1

Задайте повторяющееся отображение. Эта команда::

    dmsetup message switch 0 set_region_mappings 1000:1 :2 R2,10

эквивалентна::

    dmsetup message switch 0 set_region_mappings 1000:1 :2 :1 :2 :1 :2 :1 :2 \
	:1 :2 :1 :2 :1 :2 :1 :2 :1 :2
