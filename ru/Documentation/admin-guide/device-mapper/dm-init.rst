===============================================
Раннее создание отображаемых устройств (mapped)
===============================================

Существует два способа настроить устройство device-mapper так, чтобы оно
выступало в роли корневого устройства вашей системы.

Первый — собрать начальный ramdisk, который загружается в минимальное
пространство пользователя (userspace), настраивающее устройство, после чего
выполнить в него pivot_root(8).

Второй — создать одно или несколько устройств device-mapper с помощью параметра
модуля «dm-mod.create=», передаваемого через аргумент командной строки загрузки
ядра.

Формат задаётся как строка данных, разделённых запятыми и, при необходимости,
точками с запятой, где:

 - запятая используется для разделения полей, таких как name, uuid, flags и table
   (задают одно устройство)
 - точка с запятой используется для разделения устройств.

Таким образом, формат будет выглядеть так::

 dm-mod.create=<name>,<uuid>,<minor>,<flags>,<table>[,<table>+][;<name>,<uuid>,<minor>,<flags>,<table>[,<table>+]+]

Где::

	<name>		::= The device name.
	<uuid>		::= xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx | ""
	<minor>		::= The device minor number | ""
	<flags>		::= "ro" | "rw"
	<table>		::= <start_sector> <num_sectors> <target_type> <target_args>
	<target_type>	::= "verity" | "linear" | ... (see list below)

Строка dm должна быть эквивалентна той, которую использует утилита dmsetup с
аргументом `--concise`.

Типы целей (target)
===================

Доступны не все типы целей, поскольку существуют серьёзные риски при разрешении
активации некоторых целей DM без предварительной проверки корректности
связанных с ними метаданных средствами пространства пользователя.

======================= =======================================================
`cache`			с ограничениями, userspace должен проверить устройство кэша
`crypt`			разрешено
`delay`			разрешено
`era`			с ограничениями, userspace должен проверить устройство метаданных
`flakey`		с ограничениями, предназначено для тестирования
`linear`		разрешено
`log-writes`		с ограничениями, userspace должен проверить устройство метаданных
`mirror`		с ограничениями, userspace должен проверить main/mirror-устройство
`raid`			с ограничениями, userspace должен проверить устройство метаданных
`snapshot`		с ограничениями, userspace должен проверить src/dst-устройство
`snapshot-origin`	разрешено
`snapshot-merge`	с ограничениями, userspace должен проверить src/dst-устройство
`striped`		разрешено
`switch`		с ограничениями, userspace должен проверить путь к устройству
`thin`			с ограничениями, требует dm-сообщения цели из userspace
`thin-pool`		с ограничениями, требует dm-сообщения цели из userspace
`verity`		разрешено
`writecache`		с ограничениями, userspace должен проверить устройство кэша
`zero`			с ограничениями, не предназначено для rootfs
======================= =======================================================

Если цель не указана в списке выше, она по умолчанию считается ограниченной
(не протестированной).

Примеры
=======
Пример загрузки в массив linear, состоящий из блочных устройств user-mode linux::

  dm-mod.create="lroot,,,rw, 0 4096 linear 98:16 0, 4096 4096 linear 98:32 0" root=/dev/dm-0

Это приведёт к загрузке в rw-цель dm-linear размером 8192 секторов, распределённых
по двум блочным устройствам, идентифицированным по их номерам major:minor. После
загрузки udev переименует эту цель в /dev/mapper/lroot (в зависимости от правил).
Никакой uuid назначен не был.

Пример нескольких устройств device-mapper, где содержимое dm-mod.create="..."
показано здесь разбитым на несколько строк для удобства чтения::

  dm-linear,,1,rw,
    0 32768 linear 8:1 0,
    32768 1024000 linear 8:2 0;
  dm-verity,,3,ro,
    0 1638400 verity 1 /dev/sdc1 /dev/sdc2 4096 4096 204800 1 sha256
    ac87db56303c9c1da433d7209b5a6ef3e4779df141200cbd7c157dcb8dd89c42
    5ebfe87f7df3235b80a117ebc4078e44f55045487ad4a96581d1adb564615b51

Другие примеры (по целям):

"crypt"::

  dm-crypt,,8,ro,
    0 1048576 crypt aes-xts-plain64
    babebabebabebabebabebabebabebabebabebabebabebabebabebabebabebabe 0
    /dev/sda 0 1 allow_discards

"delay"::

  dm-delay,,4,ro,0 409600 delay /dev/sda1 0 500

"linear"::

  dm-linear,,,rw,
    0 32768 linear /dev/sda1 0,
    32768 1024000 linear /dev/sda2 0,
    1056768 204800 linear /dev/sda3 0,
    1261568 512000 linear /dev/sda4 0

"snapshot-origin"::

  dm-snap-orig,,4,ro,0 409600 snapshot-origin 8:2

"striped"::

  dm-striped,,4,ro,0 1638400 striped 4 4096
  /dev/sda1 0 /dev/sda2 0 /dev/sda3 0 /dev/sda4 0

"verity"::

  dm-verity,,4,ro,
    0 1638400 verity 1 8:1 8:2 4096 4096 204800 1 sha256
    fb1a5a0f00deb908d8b53cb270858975e76cf64105d412ce764225d53b8f3cfd
    51934789604d1b92399c52e7cb149d1b3a1b74bbbcb103b2a0aaacbed5c08584

Для конфигураций, использующих device-mapper поверх асинхронно опрашиваемых
блочных устройств (MMC, USB, ..), может потребоваться явно указать dm-init
дождаться их появления, прежде чем настраивать таблицы device-mapper. Это можно
сделать с помощью параметра модуля "dm-mod.waitfor=", который принимает список
устройств для ожидания::

  dm-mod.waitfor=<device1>[,..,<deviceN>]
