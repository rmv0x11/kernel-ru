======
dm-vdo
======

Цель device mapper'а dm-vdo (virtual data optimizer) обеспечивает дедупликацию,
сжатие и тонкое выделение пространства (thin provisioning) на уровне блоков.
Будучи целью device mapper'а, она может добавить эти возможности в стек хранения,
совместимый с любой файловой системой. Цель vdo не защищает от повреждения данных,
полагаясь вместо этого на защиту целостности нижележащего хранилища. Настоятельно
рекомендуется использовать lvm для управления томами vdo. См. lvmvdo(7).

Компонент пространства пользователя
===================================

Форматирование тома vdo требует использования инструмента «vdoformat», доступного
по адресу:

https://github.com/dm-vdo/vdo/

В большинстве случаев цель vdo автоматически восстановится после сбоя при следующем
запуске. В случаях, когда она столкнулась с невосстановимой ошибкой (либо в ходе
обычной работы, либо при восстановлении после сбоя), цель перейдёт или поднимется
в режиме «только чтение». Поскольку режим «только чтение» свидетельствует о потере
данных, для вывода vdo из режима «только чтение» необходимо предпринять явное
действие. Инструмент «vdoforcerebuild», доступный из того же репозитория,
используется для подготовки vdo в режиме «только чтение» к выходу из этого режима.
После запуска этого инструмента цель vdo перестроит свои метаданные при следующем
запуске. Хотя часть данных может быть потеряна, перестроенные метаданные vdo будут
внутренне согласованными, и цель снова станет доступной для записи.

Репозиторий также содержит дополнительные инструменты пространства пользователя,
которые можно использовать для исследования метаданных цели vdo на диске. К счастью,
эти инструменты редко нужны кому-либо, кроме разработчиков dm-vdo.

Требования к метаданным
=======================

Каждый том vdo резервирует под метаданные 3 ГБ пространства или больше в зависимости
от своей конфигурации. Полезно проверить, что пространство, сэкономленное за счёт
дедупликации и сжатия, не сводится на нет требованиями к метаданным. Оценку
пространства, сэкономленного для конкретного набора данных, можно вычислить с помощью
инструмента оценки vdo (vdo estimator), который доступен по адресу:

https://github.com/dm-vdo/vdoestimator/

Интерфейс цели
==============

Строка таблицы
--------------

::

	<offset> <logical device size> vdo V4 <storage device>
	<storage device size> <minimum I/O size> <block map cache size>
	<block map era length> [optional arguments]


Обязательные параметры:

	offset:
		Смещение в секторах, с которого начинается логическое
		пространство тома vdo.

	logical device size:
		Размер устройства, которое будет обслуживать том vdo,
		в секторах. Должен совпадать с текущим логическим размером тома
		vdo.

	storage device:
		Устройство, на котором хранятся данные и метаданные тома vdo.

	storage device size:
		Размер устройства, на котором хранится том vdo, в виде числа
		блоков по 4096 байт. Должен совпадать с текущим размером тома
		vdo.

	minimum I/O size:
		Минимальный размер операции I/O, который этот том vdo будет
		принимать, в байтах. Допустимые значения — 512 или 4096.
		Рекомендуемое значение — 4096.

	block map cache size:
		Размер кэша карты блоков в виде числа блоков по 4096 байт.
		Минимальное и рекомендуемое значение — 32768 блоков.
		Если число логических потоков не равно нулю, размер кэша
		должен составлять не менее 4096 блоков на каждый логический поток.

	block map era length:
		Скорость, с которой кэш карты блоков выгружает изменённые
		страницы карты блоков. Меньшая длина эры (era length) скорее
		всего уменьшит время, затрачиваемое на перестроение, ценой
		увеличения числа записей карты блоков в ходе обычной работы.
		Максимальное и рекомендуемое значение — 16380; минимальное
		значение — 1.

Необязательные параметры:
-------------------------
Некоторые или все эти параметры могут указываться в виде пар <key> <value>.

Параметры, связанные с потоками:

Разные категории работы назначаются отдельным группам потоков, и число потоков
в каждой группе можно настраивать независимо.

Если <hash>, <logical> и <physical> все установлены в 0, работа, выполняемая всеми
тремя типами потоков, будет выполняться одним потоком. Если любое из этих значений
не равно нулю, все они должны быть не равны нулю.

	ack:
		Число потоков, используемых для завершения bio. Поскольку
		завершение bio вызывает произвольную функцию завершения вне
		тома vdo, потоки этого типа позволяют тому vdo продолжать
		обработку запросов даже тогда, когда завершение bio происходит
		медленно. Значение по умолчанию — 1.

	bio:
		Число потоков, используемых для выдачи bio нижележащему
		хранилищу. Потоки этого типа позволяют тому vdo продолжать
		обработку запросов даже тогда, когда подача bio происходит
		медленно. Значение по умолчанию — 4.

	bioRotationInterval:
		Число bio, помещаемых в очередь на каждом потоке bio перед
		переключением на следующий поток. Значение должно быть больше
		0 и не больше 1024; значение по умолчанию — 64.

	cpu:
		Число потоков, используемых для выполнения работы,
		интенсивно нагружающей CPU, такой как хеширование и сжатие.
		Значение по умолчанию — 1.

	hash:
		Число потоков, используемых для управления сравнениями данных
		при дедупликации на основе значения хеша блоков данных.
		Значение по умолчанию — 0.

	logical:
		Число потоков, используемых для управления кэшированием и
		блокировкой на основе логического адреса входящих bio.
		Значение по умолчанию — 0; максимум — 60.

	physical:
		Число потоков, используемых для управления администрированием
		нижележащего устройства хранения. Во время форматирования для
		vdo выбирается размер слаба (slab); устройство хранения vdo
		должно быть достаточно большим, чтобы иметь как минимум 1 слаб
		на каждый физический поток. Значение по умолчанию — 0;
		максимум — 16.

Прочие параметры:

	maxDiscard:
		Максимальный размер принимаемого discard-bio, в блоках по
		4096 байт. Запросы I/O к тому vdo обычно разбиваются на блоки
		по 4096 байт и обрабатываются до 2048 за раз. Однако
		discard-запросы к тому vdo могут автоматически разбиваться на
		больший размер — до <maxDiscard> блоков по 4096 байт в одном
		bio — и ограничены до 1500 за раз. Увеличение этого значения
		может обеспечить лучшую общую производительность ценой
		увеличения задержки (latency) для отдельных discard-запросов.
		Значение по умолчанию и минимальное — 1; максимум —
		UINT_MAX / 4096.

	deduplication:
		Включена ли дедупликация. Значение по умолчанию — «on»;
		допустимые значения — «on» и «off».

	compression:
		Включено ли сжатие. Значение по умолчанию — «off»;
		допустимые значения — «on» и «off».

Изменение устройства
--------------------

В работающий, не приостановленный том vdo может быть загружена изменённая таблица.
Изменения вступят в силу при следующем возобновлении работы устройства. Изменяемые
параметры — <logical device size>, <physical device size>, <maxDiscard>,
<compression> и <deduplication>.

Если логический размер устройства или физический размер устройства изменены, при
успешном возобновлении vdo сохранит новые значения и будет требовать их при будущих
запусках. Эти два параметра не могут быть уменьшены. Логический размер устройства
не может превышать 4 ПБ. Физический размер устройства, если он вообще увеличивается,
должен увеличиваться не менее чем на 32832 блока по 4096 байт и не должен превышать
размер нижележащего устройства хранения. Кроме того, при форматировании устройства
vdo выбирается размер слаба: физический размер устройства никогда не может вырасти
выше размера, который обеспечивает 8192 слаба, и каждое увеличение должно быть
достаточно большим, чтобы добавить как минимум один новый слаб.

Примеры:

Запуск ранее отформатированного тома vdo с 1 ГБ логического пространства и 1 ГБ
физического пространства, с хранением на /dev/dm-1, где имеется более 1 ГБ
пространства.

::

	dmsetup create vdo0 --table \
	"0 2097152 vdo V4 /dev/dm-1 262144 4096 32768 16380"

Увеличение логического размера до 4 ГБ.

::

	dmsetup reload vdo0 --table \
	"0 8388608 vdo V4 /dev/dm-1 262144 4096 32768 16380"
	dmsetup resume vdo0

Увеличение физического размера до 2 ГБ.

::

	dmsetup reload vdo0 --table \
	"0 8388608 vdo V4 /dev/dm-1 524288 4096 32768 16380"
	dmsetup resume vdo0

Увеличение физического размера ещё на 1 ГБ и увеличение максимального числа
discard-секторов.

::

	dmsetup reload vdo0 --table \
	"0 10485760 vdo V4 /dev/dm-1 786432 4096 32768 16380 maxDiscard 8"
	dmsetup resume vdo0

Остановка тома vdo.

::

	dmsetup remove vdo0

Повторный запуск тома vdo. Обратите внимание, что логический и физический размеры
устройства по-прежнему должны совпадать, но другие параметры могут меняться.

::

	dmsetup create vdo1 --table \
	"0 10485760 vdo V4 /dev/dm-1 786432 512 65550 5000 hash 1 logical 3 physical 2"

Сообщения
---------
Все устройства vdo принимают сообщения в форме:

::

        dmsetup message <target-name> 0 <message-name> <message-parameters>

Сообщения таковы:

        stats:
		Выводит текущее представление статистики vdo. В основном
		используется программой vdostats пространства пользователя
		для интерпретации выходного буфера.

	config:
		Выводит полезную информацию о конфигурации vdo. В основном
		используется пользователями, которые хотят воссоздать
		похожий том VDO и хотят узнать использованную при создании
		конфигурацию.

	dump:
		Сбрасывает множество внутренних структур в системный журнал.
		Это не всегда безопасно запускать, поэтому это следует
		использовать только для отладки зависшего vdo. Необязательные
		параметры для указания структур, которые нужно сбросить:

				viopool: Пул запросов I/O — входящих bio
				pools: Синоним «viopool»
				vdo: Большинство структур, управляющих данными на диске
				queues: Базовая информация о каждом потоке vdo
				threads: Синоним «queues»
				default: Эквивалентно «queues vdo»
				all: Всё вышеперечисленное.

        dump-on-shutdown:
		Выполнить дамп по умолчанию при следующем завершении работы vdo.


Статус
------

::

    <device> <operating mode> <in recovery> <index state>
    <compression state> <physical blocks used> <total physical blocks>

	device:
		Имя тома vdo.

	operating mode:
		Текущий режим работы тома vdo; значениями могут быть
		«normal», «recovering» (том обнаружил проблему со своими
		метаданными и пытается восстановиться) и «read-only»
		(произошла ошибка, которая вынуждает том vdo поддерживать
		только операции чтения, но не записи).

	in recovery:
		Находится ли том vdo в данный момент в режиме восстановления;
		значениями могут быть «recovering» или «-», что указывает на
		отсутствие восстановления.

	index state:
		Текущее состояние индекса дедупликации в томе vdo; значениями
		могут быть «closed», «closing», «error», «offline», «online»,
		«opening» и «unknown».

	compression state:
		Текущее состояние сжатия в томе vdo; значениями могут быть
		«offline» и «online».

	used physical blocks:
		Число физических блоков, используемых томом vdo.

	total physical blocks:
		Общее число физических блоков, которые том vdo может
		использовать; разница между этим значением и
		<used physical blocks> — это число блоков, которые остаются
		у тома vdo до заполнения.

Требования к памяти
===================

Цель vdo требует фиксированных 38 МБ ОЗУ вместе со следующими объёмами, которые
масштабируются вместе с целью:

- 1,15 МБ ОЗУ на каждый 1 МБ настроенного размера кэша карты блоков. Кэш карты
  блоков требует минимум 150 МБ.
- 1,6 МБ ОЗУ на каждый 1 ТБ логического пространства.
- 268 МБ ОЗУ на каждый 1 ТБ физического хранилища, управляемого томом.

Индекс дедупликации требует дополнительной памяти, которая масштабируется вместе
с размером окна дедупликации. Для плотных (dense) индексов индекс требует 1 ГБ ОЗУ
на 1 ТБ окна. Для разреженных (sparse) индексов индекс требует 1 ГБ ОЗУ на 10 ТБ
окна. Конфигурация индекса задаётся при форматировании цели и не может быть изменена.

Параметры модуля
================

Драйвер vdo имеет числовой параметр «log_level», который управляет
детализацией логирования драйвера. Значение по умолчанию — 6
(LOGLEVEL_INFO и более серьёзные сообщения).

Использование во время выполнения
=================================

При использовании dm-vdo важно знать о том, чем его поведение отличается от других
целей хранения.

- Нет гарантии, что перезаписи существующих блоков завершатся успешно. Поскольку
  нижележащее хранилище может иметь множественные ссылки, перезапись существующего
  блока обычно требует, чтобы у vdo был доступен свободный блок.

- Когда блоки больше не используются, отправка discard-запроса для этих блоков
  позволяет vdo освободить ссылки на эти блоки. Если vdo использует тонкое
  выделение пространства, отбрасывание (discard) неиспользуемых блоков существенно
  для предотвращения исчерпания пространства целью. Однако из-за совместного
  использования дублирующихся блоков ни один discard-запрос для какого-либо данного
  логического блока не гарантирует освобождения пространства.

- Если предположить, что нижележащее хранилище правильно реализует запросы сброса
  (flush), vdo устойчив к сбоям; тем не менее, несброшенные записи могут как
  сохраниться, так и не сохраниться после сбоя.

- Каждая запись в цель vdo влечёт за собой значительный объём обработки. Однако
  большая часть работы поддаётся распараллеливанию. Поэтому цели vdo достигают
  лучшей пропускной способности при больших глубинах I/O и могут поддерживать до
  2048 запросов параллельно.

Тюнинг
======

Устройство vdo имеет множество опций, и может быть сложно сделать оптимальный выбор
без идеального знания рабочей нагрузки. Кроме того, большинство опций конфигурации
должны быть установлены при запуске цели vdo и не могут быть изменены без её полного
завершения; конфигурацию нельзя изменить, пока цель активна. В идеале тюнинг с
имитируемыми рабочими нагрузками следует выполнять перед развёртыванием vdo в
производственных средах.

Самое важное значение для регулировки — размер кэша карты блоков. Чтобы обслужить
запрос для любого логического адреса, vdo должен загрузить ту часть карты блоков,
которая содержит соответствующее отображение. Эти отображения кэшируются.
Производительность будет страдать, когда рабочий набор не помещается в кэш. По
умолчанию vdo выделяет 128 МБ кэша метаданных в ОЗУ для поддержки эффективного
доступа к 100 ГБ логического пространства за раз. Это следует масштабировать
пропорционально для бóльших рабочих наборов.

Число логических и физических потоков также следует регулировать. Логический поток
управляет непересекающейся секцией карты блоков, поэтому дополнительные логические
потоки увеличивают параллелизм и могут увеличить пропускную способность. Физические
потоки управляют непересекающейся секцией блоков данных, поэтому дополнительные
физические потоки также могут увеличить пропускную способность. Однако избыток
потоков может тратить впустую ресурсы и увеличивать конкуренцию (contention).

Потоки подачи bio управляют параллелизмом при отправке I/O нижележащему хранилищу;
меньшее число потоков означает, что есть больше возможностей переупорядочить запросы
I/O для повышения производительности, но также и то, что каждый запрос I/O должен
дольше ждать перед подачей.

Потоки подтверждения bio используются для завершения запросов I/O. Это делается на
выделенных потоках, поскольку объём работы, необходимый для выполнения callback'а
bio, не может контролироваться самим vdo. Обычно достаточно одного потока, но
дополнительные потоки могут быть полезны, особенно когда у bio есть callback'и,
интенсивно нагружающие CPU.

Потоки CPU используются для хеширования и для сжатия; в рабочих нагрузках с
включённым сжатием большее число потоков может привести к более высокой пропускной
способности.

Потоки хеширования используются для сортировки активных запросов по хешу и для
определения того, следует ли их дедуплицировать; самые интенсивно нагружающие CPU
действия, выполняемые этими потоками, — это сравнение блоков данных по 4096 байт.
В большинстве случаев достаточно одного потока хеширования.
