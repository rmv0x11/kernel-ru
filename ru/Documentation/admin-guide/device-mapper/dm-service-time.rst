===============
dm-service-time
===============

dm-service-time — это модуль выбора пути (path selector) для целей device-mapper,
который выбирает путь с наименьшим расчётным временем обслуживания (service time)
для входящего ввода-вывода.

Время обслуживания каждого пути оценивается путём деления суммарного размера
незавершённых операций ввода-вывода (in-flight I/Os) на пути на значение
производительности этого пути. Значение производительности — это относительная
пропускная способность среди всех путей в группе путей (path-group), и его можно
задать как аргумент таблицы.

Имя данного модуля выбора пути — «service-time».

Параметры таблицы для каждого пути:

    [<repeat_count> [<relative_throughput>]]
	<repeat_count>:
			Число операций ввода-вывода, отправляемых по выбранному
			пути перед переключением на следующий путь.
			Если не задано, используется внутреннее значение по умолчанию.
			Чтобы узнать значение по умолчанию, посмотрите активированную таблицу.
	<relative_throughput>:
			Значение относительной пропускной способности пути
			среди всех путей в группе путей.
			Допустимый диапазон — 0-100.
			Если не задано, используется минимальное значение «1».
			Если задано «0», путь не выбирается, пока доступны
			другие пути с положительным значением.

Состояние для каждого пути:

    <status> <fail-count> <in-flight-size> <relative_throughput>
	<status>:
		«A», если путь активен, «F», если путь отказал.
	<fail-count>:
		Число отказов пути.
	<in-flight-size>:
		Размер незавершённых операций ввода-вывода на пути.
	<relative_throughput>:
		Значение относительной пропускной способности пути
		среди всех путей в группе путей.


Алгоритм
========

dm-service-time добавляет размер операции ввода-вывода к «in-flight-size» при
отправке операции и вычитает по её завершении.
По сути, dm-service-time выбирает путь с минимальным временем обслуживания,
которое вычисляется как::

	('in-flight-size' + 'size-of-incoming-io') / 'relative_throughput'

Однако ниже приведены некоторые оптимизации, используемые для максимально
возможного сокращения вычислений.

	1. Если у путей одинаковое значение «relative_throughput», деление
	   пропускается и сравниваются только значения «in-flight-size».

	2. Если у путей одинаковое значение «in-flight-size», деление пропускается
	   и сравниваются только значения «relative_throughput».

	3. Если у части путей значение «relative_throughput» ненулевое, а у других
	   нулевое, то пути с нулевым «relative_throughput» игнорируются.

Если такие оптимизации применить нельзя, вычисляется время обслуживания, и
время обслуживания сравнивается.
Если вычисленное время обслуживания одинаково, путь с максимальным значением
«relative_throughput» может оказаться лучше. Поэтому в этом случае сравниваются
значения «relative_throughput».


Примеры
=======
В случае, когда используются 2 пути (sda и sdb) с repeat_count == 128,
причём sda имеет среднюю пропускную способность 1ГБ/с, а sdb — 4ГБ/с,
значение «relative_throughput» может быть «1» для sda и «4» для sdb::

  # echo "0 10 multipath 0 0 1 1 service-time 0 2 2 8:0 128 1 8:16 128 4" \
    dmsetup create test
  #
  # dmsetup table
  test: 0 10 multipath 0 0 1 1 service-time 0 2 2 8:0 128 1 8:16 128 4
  #
  # dmsetup status
  test: 0 10 multipath 2 0 0 0 1 1 E 0 2 2 8:0 A 0 0 1 8:16 A 0 0 4


Или «2» для sda и «8» для sdb тоже будет верно::

  # echo "0 10 multipath 0 0 1 1 service-time 0 2 2 8:0 128 2 8:16 128 8" \
    dmsetup create test
  #
  # dmsetup table
  test: 0 10 multipath 0 0 1 1 service-time 0 2 2 8:0 128 2 8:16 128 8
  #
  # dmsetup status
  test: 0 10 multipath 2 0 0 0 1 1 E 0 2 2 8:0 A 0 0 2 8:16 A 0 0 8
