========
dm-clone
========

Введение
========

dm-clone — это цель device mapper, которая создаёт точную копию (one-to-one)
существующего, доступного только для чтения исходного устройства на доступном
для записи целевом устройстве. Она представляет виртуальное блочное устройство,
благодаря которому все данные становятся доступны немедленно, и соответствующим
образом перенаправляет операции чтения и записи.

Основной сценарий использования dm-clone — клонирование потенциально удалённого
блочного устройства архивного типа с высокой задержкой, доступного только для
чтения, на доступное для записи быстрое устройство основного типа для быстрого
ввода-вывода с малой задержкой. Клонированное устройство сразу же видимо и может
быть смонтировано, а копирование с исходного устройства на целевое происходит в
фоновом режиме, параллельно с пользовательским вводом-выводом.

Например, можно восстановить резервную копию приложения из доступной только для
чтения копии, доступ к которой осуществляется через сетевой протокол хранения
(NBD, Fibre Channel, iSCSI, AoE и т. п.), на локальное устройство SSD или NVMe и
сразу начать использовать устройство, не дожидаясь завершения восстановления.

После завершения клонирования таблицу dm-clone можно полностью удалить и
заменить, например, на linear-таблицу, отображающую напрямую на целевое
устройство.

Цель dm-clone повторно использует библиотеку метаданных, применяемую целью
thin-provisioning.

Глоссарий
=========

   Hydration (гидратация)
     Процесс заполнения области целевого устройства данными из той же области
     исходного устройства, то есть копирование области с исходного устройства на
     целевое.

После того как область гидратирована, весь относящийся к ней ввод-вывод
перенаправляется на целевое устройство.

Устройство
==========

Подустройства
-------------

Цель создаётся передачей ей трёх устройств (наряду с другими параметрами,
описанными далее):

1. Исходное устройство — доступное только для чтения устройство, которое
   клонируется и является источником гидратации.

2. Целевое устройство — назначение гидратации, которое станет клоном исходного
   устройства.

3. Небольшое устройство метаданных — оно фиксирует, какие области уже
   действительны на целевом устройстве, то есть какие области уже были
   гидратированы или были записаны напрямую посредством пользовательского
   ввода-вывода.

Размер целевого устройства должен быть как минимум равен размеру исходного
устройства.

Области
-------

dm-clone делит исходное и целевое устройства на области фиксированного размера.
Области являются единицей гидратации, то есть минимальным объёмом данных,
копируемым с исходного устройства на целевое.

Размер области настраивается при первом создании устройства dm-clone.
Рекомендуемый размер области совпадает с размером блока файловой системы, который
обычно составляет 4 КБ. Размер области должен находиться в пределах от 8 секторов
(4 КБ) до 2097152 секторов (1 ГБ) и быть степенью двойки.

Операции чтения и записи из/в гидратированные области обслуживаются целевым
устройством.

Чтение из ещё не гидратированной области обслуживается напрямую с исходного
устройства.

Запись в ещё не гидратированную область будет задержана до тех пор, пока
соответствующая область не будет гидратирована, и гидратация этой области
начинается немедленно.

Обратите внимание, что запрос на запись с размером, равным размеру области,
пропустит копирование соответствующей области с исходного устройства и перезапишет
область целевого устройства напрямую.

Discard-запросы
---------------

dm-clone интерпретирует discard-запрос к диапазону, который ещё не был
гидратирован, как указание пропустить гидратацию областей, охватываемых
запросом, то есть пропускает копирование данных области с исходного устройства на
целевое и лишь обновляет свои метаданные.

Если целевое устройство поддерживает discard-запросы, то по умолчанию dm-clone
будет передавать discard-запросы на него.

Фоновая гидратация
------------------

dm-clone непрерывно копирует данные с исходного устройства на целевое до тех пор,
пока не будет скопировано всё устройство.

Копирование данных с исходного устройства на целевое использует полосу
пропускания. Пользователь может установить ограничение, чтобы не допустить
выполнения более определённого объёма копирования в любой момент времени. Более
того, dm-clone учитывает пользовательский трафик ввода-вывода, идущий к
устройствам, и приостанавливает фоновую гидратацию при наличии незавершённого
ввода-вывода.

Сообщение `hydration_threshold <#regions>` может использоваться для установки
максимального числа копируемых областей; по умолчанию оно равно 1 области.

dm-clone применяет dm-kcopyd для копирования частей исходного устройства на
целевое. По умолчанию мы выдаём запросы на копирование с размером, равным размеру
области. Сообщение `hydration_batch_size <#regions>` может использоваться для
настройки размера этих запросов на копирование. Увеличение размера пакета
гидратации приводит к тому, что dm-clone пытается объединять в пакеты смежные
области, так что мы копируем данные пакетами по столько областей.

Когда гидратация целевого устройства завершается, в пространство пользователя
будет отправлено dm-событие.

Обновление метаданных на диске
------------------------------

Метаданные на диске фиксируются (commit) каждый раз при записи bio с флагом FLUSH
или FUA. Если таких запросов не поступает, то фиксация будет происходить каждую
секунду. Это означает, что устройство dm-clone ведёт себя как физический диск с
энергозависимым кэшем записи. При потере питания вы можете потерять часть
недавних записей. Метаданные всегда должны оставаться согласованными, несмотря на
любой сбой.

Интерфейс цели
==============

Конструктор
-----------

  ::

   clone <metadata dev> <destination dev> <source dev> <region size>
         [<#feature args> [<feature arg>]* [<#core args> [<core arg>]*]]

 ================ ==============================================================
 metadata dev     Быстрое устройство, хранящее постоянные метаданные
 destination dev  Целевое устройство, на котором будет клонирован источник
 source dev       Доступное только для чтения устройство с клонируемыми данными
 region size      Размер области в секторах

 #feature args    Число передаваемых аргументов функциональности
 feature args     no_hydration или no_discard_passdown

 #core args       Чётное число аргументов, соответствующих парам ключ/значение,
                  передаваемым dm-clone
 core args        Пары ключ/значение, передаваемые dm-clone, например `hydration_threshold
                  256`
 ================ ==============================================================

Необязательные аргументы функциональности:

 ==================== =========================================================
 no_hydration         Создать экземпляр dm-clone с отключённой фоновой
                      гидратацией
 no_discard_passdown  Отключить передачу discard-запросов на целевое устройство
 ==================== =========================================================

Необязательные основные (core) аргументы:

 ================================ ==============================================
 hydration_threshold <#regions>   Максимальное число областей, копируемых с
                                  исходного устройства на целевое в любой момент
                                  времени, во время фоновой гидратации.
 hydration_batch_size <#regions>  Во время фоновой гидратации пытаться
                                  объединять в пакеты смежные области, так чтобы
                                  данные копировались с исходного устройства на
                                  целевое пакетами по столько областей.
 ================================ ==============================================

Статус
------

  ::

   <metadata block size> <#used metadata blocks>/<#total metadata blocks>
   <region size> <#hydrated regions>/<#total regions> <#hydrating regions>
   <#feature args> <feature args>* <#core args> <core args>*
   <clone metadata mode>

 ======================= =======================================================
 metadata block size     Фиксированный размер каждого блока метаданных в секторах
 #used metadata blocks   Число используемых блоков метаданных
 #total metadata blocks  Общее число блоков метаданных
 region size             Настраиваемый размер области для устройства в секторах
 #hydrated regions       Число областей, завершивших гидратацию
 #total regions          Общее число областей для гидратации
 #hydrating regions      Число областей, гидратируемых в данный момент
 #feature args           Число следующих далее аргументов функциональности
 feature args            Аргументы функциональности, например `no_hydration`
 #core args              Чётное число следующих далее основных аргументов
 core args               Пары ключ/значение для настройки ядра, например
                         `hydration_threshold 256`
 clone metadata mode     ro, если только для чтения, rw, если для чтения и записи

                         В серьёзных случаях, когда даже режим только для чтения
                         признан небезопасным, дальнейший ввод-вывод не будет
                         разрешён, а статус будет содержать лишь строку 'Fail'.
                         При изменении режима метаданных в пространство
                         пользователя будет отправлено dm-событие.
 ======================= =======================================================

Сообщения
---------

  `disable_hydration`
      Отключить фоновую гидратацию целевого устройства.

  `enable_hydration`
      Включить фоновую гидратацию целевого устройства.

  `hydration_threshold <#regions>`
      Установить порог фоновой гидратации.

  `hydration_batch_size <#regions>`
      Установить размер пакета фоновой гидратации.

Примеры
=======

Клонирование устройства, содержащего файловую систему
-----------------------------------------------------

1. Создайте устройство dm-clone.

   ::

    dmsetup create clone --table "0 1048576000 clone $metadata_dev $dest_dev \
      $source_dev 8 1 no_hydration"

2. Смонтируйте устройство и выполните trim файловой системы. dm-clone
   интерпретирует discard-запросы, отправленные файловой системой, и не будет
   гидратировать неиспользуемое пространство.

   ::

    mount /dev/mapper/clone /mnt/cloned-fs
    fstrim /mnt/cloned-fs

3. Включите фоновую гидратацию целевого устройства.

   ::

    dmsetup message clone 0 enable_hydration

4. После завершения гидратации мы можем заменить таблицу dm-clone на
   linear-таблицу.

   ::

    dmsetup suspend clone
    dmsetup load clone --table "0 1048576000 linear $dest_dev 0"
    dmsetup resume clone

   Устройство метаданных больше не нужно и может быть безопасно отброшено
   (discard) или повторно использовано для других целей.

Известные проблемы
==================

1. Мы перенаправляем чтения из ещё не гидратированных областей на исходное
   устройство. Если чтение с исходного устройства имеет высокую задержку, а
   пользователь многократно читает из одних и тех же областей, такое поведение
   может ухудшить производительность. Нам следует использовать эти чтения как
   подсказки для более ранней гидратации соответствующих областей. В настоящее
   время мы полагаемся на кэш страниц для кэширования этих областей, так что,
   надеемся, мы не будем читать их с исходного устройства многократно.

2. Освобождать ресурсы в памяти (in-core), то есть битовые карты, отслеживающие,
   какие области гидратированы, после завершения гидратации.

3. Во время фоновой гидратации, если нам не удаётся прочитать с исходного
   устройства или записать на целевое, мы выводим сообщение об ошибке, но процесс
   гидратации продолжается бесконечно, пока не завершится успешно. Нам следует
   останавливать фоновую гидратацию после некоторого числа сбоев и генерировать
   dm-событие, чтобы пространство пользователя могло его заметить.

Почему не...?
=============

Перед реализацией dm-clone мы рассмотрели следующие альтернативы:

1. Использовать dm-cache с размером кэша, равным исходному устройству, и
   реализовать новую политику клонирования:

   * Результирующее кэш-устройство не является точной копией (one-to-one)
     исходного устройства, и поэтому мы не можем удалить кэш-устройство после
     завершения клонирования.

   * dm-cache производит запись на исходное устройство, что нарушает наше
     требование о том, что исходное устройство должно рассматриваться как
     доступное только для чтения.

   * Кэширование семантически отличается от клонирования.

2. Использовать dm-snapshot с COW-устройством, равным исходному устройству:

   * dm-snapshot хранит свои метаданные в COW-устройстве, поэтому результирующее
     устройство не является точной копией (one-to-one) исходного устройства.

   * Отсутствует механизм фонового копирования.

   * dm-snapshot необходимо фиксировать свои метаданные всякий раз при завершении
     ожидающего исключения, чтобы обеспечить согласованность снимка. В случае
     клонирования нам не нужно быть настолько строгими, и мы можем полагаться на
     фиксацию метаданных каждый раз при записи bio с флагом FLUSH или FUA либо
     периодически, как это делают dm-thin и dm-cache. Это значительно повышает
     производительность.

3. Использовать dm-mirror: цель mirror имеет механизм фонового
   копирования/зеркалирования, но она производит запись на все зеркала, тем самым
   нарушая наше требование о том, что исходное устройство должно рассматриваться
   как доступное только для чтения.

4. Использовать функциональность внешнего снимка (external snapshot) dm-thin.
   Этот подход является наиболее многообещающим среди всех альтернатив,
   поскольку том с тонким выделением (thinly-provisioned) является точной копией
   (one-to-one) исходного устройства и обрабатывает чтения и записи в
   невыделенные/ещё не клонированные области так же, как это делает dm-clone.

   И всё же:

   * Отсутствует механизм фонового копирования, хотя его можно было бы
     реализовать.

   * Самое главное, мы хотим поддерживать произвольные блочные устройства в
     качестве назначения процесса клонирования и не ограничивать себя томами с
     тонким выделением. Тонкое выделение (thin-provisioning) влечёт за собой
     неотъемлемые накладные расходы на метаданные для поддержания отображений
     тонкого тома, что значительно ухудшает производительность.

   Более того, клонирование устройства не должно вынуждать использовать тонкое
   выделение. С другой стороны, если мы хотим использовать тонкое выделение, мы
   можем просто использовать тонкий LV в качестве целевого устройства dm-clone.
