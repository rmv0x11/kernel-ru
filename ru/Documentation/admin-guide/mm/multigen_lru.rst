=============
Multi-Gen LRU
=============
Multi-gen LRU — это альтернативная реализация LRU, которая оптимизирует
вытеснение страниц (page reclaim) и повышает производительность при
нехватке памяти. Вытеснение страниц определяет политику кэширования ядра
и его способность к перерасходу памяти (overcommit). Оно напрямую влияет
на потребление CPU процессом kswapd и эффективность использования RAM.

Быстрый старт
=============
Соберите ядро со следующими настройками конфигурации.

* ``CONFIG_LRU_GEN=y``
* ``CONFIG_LRU_GEN_ENABLED=y``

Готово!

Параметры времени выполнения
============================
``/sys/kernel/mm/lru_gen/`` содержит стабильные ABI, описанные в
следующих подразделах.

Аварийный выключатель
---------------------
``enabled`` принимает различные значения для включения или отключения
перечисленных ниже компонентов. Его значение по умолчанию зависит от
``CONFIG_LRU_GEN_ENABLED``. Все компоненты следует включить, если только
у некоторых из них не проявляются непредвиденные побочные эффекты. Запись
в ``enabled`` не оказывает эффекта, когда компонент не поддерживается
оборудованием, при этом допустимые значения будут приняты, даже когда
главный выключатель выключен.

====== ===============================================================
Значе- Компоненты
ния
====== ===============================================================
0x0001 Главный выключатель multi-gen LRU.
0x0002 Сброс бита accessed в листовых записях таблицы страниц
       большими пакетами, когда MMU устанавливает его (например, на
       x86). Теоретически такое поведение может усугубить
       конкуренцию за блокировку (mmap_lock). Если оно отключено,
       multi-gen LRU будет испытывать незначительную деградацию
       производительности для нагрузок, которые непрерывно отображают
       горячие страницы, биты accessed которых иначе могли бы
       сбрасываться меньшим числом более крупных пакетов.
0x0004 Сброс бита accessed также и в нелистовых записях таблицы
       страниц, когда MMU устанавливает его (например, на x86). Это
       поведение не проверялось на разновидностях x86, отличных от
       Intel и AMD. Если оно отключено, multi-gen LRU будет
       испытывать пренебрежимо малую деградацию производительности.
[yYnN] Применяется ко всем перечисленным выше компонентам.
====== ===============================================================

Например,
::

    echo y >/sys/kernel/mm/lru_gen/enabled
    cat /sys/kernel/mm/lru_gen/enabled
    0x0007
    echo 5 >/sys/kernel/mm/lru_gen/enabled
    cat /sys/kernel/mm/lru_gen/enabled
    0x0005

Предотвращение пробуксовки
--------------------------
Персональные компьютеры более чувствительны к пробуксовке (thrashing),
поскольку она может вызывать рывки (задержки при отрисовке UI) и
негативно влиять на пользовательский опыт. Multi-gen LRU предлагает
предотвращение пробуксовки большинству пользователей ноутбуков и
настольных систем, у которых нет ``oomd``.

Пользователи могут записать ``N`` в ``min_ttl_ms``, чтобы предотвратить
вытеснение рабочего набора за ``N`` миллисекунд. OOM killer
запускается, если этот рабочий набор не удаётся удержать в памяти.
Иными словами, этот параметр работает как регулируемый предохранительный
клапан сброса давления, и, когда он открыт, он завершает приложения,
которые, как можно надеяться, не используются.

Исходя из средней различимой человеком задержки (~100 мс), ``N=1000``
обычно устраняет нетерпимые рывки, вызванные пробуксовкой. Бо́льшие
значения, такие как ``N=3000``, делают рывки менее заметными ценой риска
преждевременных OOM-завершений.

Значение по умолчанию ``0`` означает «отключено».

Экспериментальные возможности
=============================
``/sys/kernel/debug/lru_gen`` принимает команды, описанные в следующих
подразделах. Поддерживается несколько командных строк, а также их
конкатенация с разделителями ``,`` и ``;``.

``/sys/kernel/debug/lru_gen_full`` предоставляет дополнительную
статистику для отладки. ``CONFIG_LRU_GEN_STATS=y`` сохраняет в этом файле
историческую статистику по вытесненным поколениям.

Оценка рабочего набора
----------------------
Оценка рабочего набора (working set estimation) измеряет, сколько памяти
требуется приложению в заданном интервале времени, и обычно выполняется с
небольшим влиянием на производительность приложения. Например, центры
обработки данных хотят оптимизировать планирование задач (bin packing),
чтобы повысить эффективность использования памяти. Когда поступает новая
задача, планировщику задач нужно выяснить, может ли каждый управляемый им
сервер выделить определённый объём памяти под эту новую задачу, прежде чем
он сможет выбрать кандидата. Для этого планировщику задач нужно оценить
рабочие наборы существующих задач.

При чтении ``lru_gen`` возвращает гистограмму количеств страниц, к которым
был доступ за различные интервалы времени, для каждой memcg и узла.
``MAX_NR_GENS`` определяет число корзин (bins) в каждой гистограмме.
Гистограммы некумулятивны.
::

    memcg  memcg_id  memcg_path
       node  node_id
           min_gen_nr  age_in_ms  nr_anon_pages  nr_file_pages
           ...
           max_gen_nr  age_in_ms  nr_anon_pages  nr_file_pages

Каждая корзина содержит оценочное число страниц, к которым был доступ в
пределах ``age_in_ms``. Например, ``min_gen_nr`` содержит самые холодные
страницы, а ``max_gen_nr`` содержит самые горячие страницы, поскольку
``age_in_ms`` у первого наибольшее, а у последнего наименьшее.

Пользователи могут записать в ``lru_gen`` следующую команду, чтобы создать
новое поколение ``max_gen_nr+1``:

    ``+ memcg_id node_id max_gen_nr [can_swap [force_scan]]``

``can_swap`` по умолчанию принимает значение из настройки swap, и, если
оно установлено в ``1``, оно принудительно сканирует анонимные страницы,
когда swap выключен, и наоборот. ``force_scan`` по умолчанию равно ``1``,
и, если оно установлено в ``0``, применяются эвристики для снижения
накладных расходов, что, вероятно, также снизит охват.

Типичный сценарий использования: планировщик задач выполняет эту команду
через определённый интервал времени, чтобы создавать новые поколения, и
ранжирует управляемые им серверы на основе размеров их холодных страниц,
определяемых этим интервалом времени.

Упреждающее вытеснение
----------------------
Упреждающее вытеснение (proactive reclaim) инициирует вытеснение страниц,
когда нехватки памяти нет. Обычно оно нацелено только на холодные
страницы. Например, когда поступает новая задача, планировщик задач хочет
упреждающе вытеснить холодные страницы на выбранном им сервере, чтобы
повысить шансы успешного размещения этой новой задачи.

Пользователи могут записать в ``lru_gen`` следующую команду, чтобы
вытеснить поколения, меньшие или равные ``min_gen_nr``.

    ``- memcg_id node_id min_gen_nr [swappiness [nr_to_reclaim]]``

``min_gen_nr`` должно быть меньше, чем ``max_gen_nr-1``, поскольку
``max_gen_nr`` и ``max_gen_nr-1`` не полностью состарены (эквивалентно
активному списку) и поэтому не могут быть вытеснены. ``swappiness``
переопределяет значение по умолчанию в ``/proc/sys/vm/swappiness``, и
допустимый диапазон — [0-200, max], где max используется исключительно для
вытеснения анонимной памяти. ``nr_to_reclaim`` ограничивает число
вытесняемых страниц.

Типичный сценарий использования: планировщик задач выполняет эту команду
перед тем, как попытаться разместить новую задачу на сервере. Если ему не
удаётся высвободить достаточно холодных страниц из-за переоценки, он
повторяет попытку на следующем сервере в соответствии с результатом
ранжирования, полученным на этапе оценки рабочего набора. Этот менее
форсированный подход ограничивает влияние на существующие задачи.
