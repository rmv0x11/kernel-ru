================================
Изучение таблиц страниц процесса
================================

pagemap — это новый (начиная с 2.6.25) набор интерфейсов в ядре, который
позволяет программам пространства пользователя (userspace) изучать таблицы
страниц и сопутствующую информацию путём чтения файлов в ``/proc``.

У pagemap есть четыре компонента:

 * ``/proc/pid/pagemap``.  Этот файл позволяет процессу пространства
   пользователя выяснить, в какой физический фрейм отображена каждая
   виртуальная страница.  Он содержит одно 64-битное значение для каждой
   виртуальной страницы со следующими данными (из
   ``fs/proc/task_mmu.c``, выше pagemap_read):

    * Биты 0-54  номер фрейма страницы (PFN), если присутствует
    * Биты 0-4   тип swap, если выгружена в swap
    * Биты 5-54  смещение в swap, если выгружена в swap
    * Бит  55    pte помечен как soft-dirty (см.
      Documentation/admin-guide/mm/soft-dirty.rst)
    * Бит  56    страница отображена эксклюзивно (начиная с 4.2)
    * Бит  57    pte защищён от записи через uffd-wp (начиная с 5.13) (см.
      Documentation/admin-guide/mm/userfaultfd.rst)
    * Бит  58    pte является защитной областью (guard region) (начиная с 6.15) (см. man-страницу madvise (2))
    * Биты 59-60 ноль
    * Бит  61    страница является file-page или shared-anon (начиная с 3.5)
    * Бит  62    страница выгружена в swap
    * Бит  63    страница присутствует

   Начиная с Linux 4.0 только пользователи с возможностью CAP_SYS_ADMIN могут
   получать PFN.  В 4.0 и 4.1 попытки открытия непривилегированными
   пользователями завершаются с ошибкой -EPERM.  Начиная с 4.2 поле PFN
   обнуляется, если у пользователя нет CAP_SYS_ADMIN.
   Причина: информация о PFN помогает в эксплуатации уязвимости Rowhammer.

   Если страница отсутствует, но находится в swap, то PFN содержит
   закодированный номер swap-файла и смещение страницы в swap. Неотображённые
   страницы возвращают нулевой PFN. Это позволяет точно определить, какие
   страницы отображены (или находятся в swap), и сравнивать отображённые
   страницы между процессами.

   Традиционно бит 56 указывает, что страница отображена ровно один раз, и
   бит 56 сброшен, когда страница отображена несколько раз, даже если она
   отображена несколько раз в одном и том же процессе. В некоторых
   конфигурациях ядра семантика для страниц, входящих в более крупное
   выделение (например, THP), может отличаться: бит 56 устанавливается, если
   все страницы, входящие в соответствующее крупное выделение, *определённо*
   отображены в одном и том же процессе, даже если страница отображена в этом
   процессе несколько раз. Бит 56 сброшен, когда любая страница более крупного
   выделения *возможно* отображена в другом процессе. В некоторых случаях
   крупное выделение может рассматриваться как «возможно отображённое
   несколькими процессами», даже если это уже не так.

   Эффективные потребители этого интерфейса будут использовать
   ``/proc/pid/maps``, чтобы определить, какие области памяти действительно
   отображены, и llseek, чтобы пропускать неотображённые области.

 * ``/proc/kpagecount``.  Этот файл содержит 64-битный счётчик того, сколько
   раз отображена каждая страница, индексируемый по PFN. Некоторые
   конфигурации ядра не отслеживают точное число раз, которое отображена
   страница, входящая в более крупное выделение (например, THP). В таких
   конфигурациях вместо этого возвращается среднее число отображений на
   страницу в этом более крупном выделении. Однако, если отображена хотя бы
   одна страница крупного выделения, возвращаемое значение будет не меньше 1.

Инструмент page-types в каталоге tools/mm можно использовать для запроса
числа раз, которое отображена страница.

 * ``/proc/kpageflags``.  Этот файл содержит 64-битный набор флагов для каждой
   страницы, индексируемый по PFN.

   Флаги таковы (из ``fs/proc/page.c``, выше kpageflags_read):

    0. LOCKED
    1. ERROR
    2. REFERENCED
    3. UPTODATE
    4. DIRTY
    5. LRU
    6. ACTIVE
    7. SLAB
    8. WRITEBACK
    9. RECLAIM
    10. BUDDY
    11. MMAP
    12. ANON
    13. SWAPCACHE
    14. SWAPBACKED
    15. COMPOUND_HEAD
    16. COMPOUND_TAIL
    17. HUGE
    18. UNEVICTABLE
    19. HWPOISON
    20. NOPAGE
    21. KSM
    22. THP
    23. OFFLINE
    24. ZERO_PAGE
    25. IDLE
    26. PGTABLE

 * ``/proc/kpagecgroup``.  Этот файл содержит 64-битный номер inode группы
   memory cgroup, на которую отнесена каждая страница, индексируемый по PFN.
   Доступен только при установленном CONFIG_MEMCG.

Краткие описания флагов страниц
===============================

0 - LOCKED
   Страница блокируется для эксклюзивного доступа, например при прохождении
   операции ввода-вывода чтения/записи.
7 - SLAB
   Страница управляется аллокатором памяти ядра SLAB/SLUB.
   При использовании составной (compound) страницы любой из них устанавливает
   этот флаг только на головной (head) странице.
10 - BUDDY
    Свободный блок памяти, управляемый аллокатором buddy system.
    Buddy system организует свободную память в блоки различных порядков
    (orders).
    Блок порядка N содержит 2^N физически непрерывных страниц, при этом флаг
    BUDDY установлен для всех страниц.
    До 4.6 флаг был установлен только на первой странице блока.
15 - COMPOUND_HEAD
    Составная страница порядка N состоит из 2^N физически непрерывных страниц.
    Составная страница порядка 2 принимает вид «HTTT», где H обозначает её
    головную (head) страницу, а T обозначает её хвостовую (tail) страницу(ы).
    Основные потребители составных страниц — это страницы hugeTLB
    (Documentation/admin-guide/mm/hugetlbpage.rst), аллокаторы памяти SLUB и
    т. п., а также различные драйверы устройств.
    Однако в этом интерфейсе конечным пользователям видимы только
    huge/giga-страницы.
16 - COMPOUND_TAIL
    Хвост (tail) составной страницы (см. описание выше).
17 - HUGE
    Это неотъемлемая часть страницы HugeTLB.
19 - HWPOISON
    Аппаратно обнаружено повреждение памяти на этой странице: не трогайте
    данные!
20 - NOPAGE
    По запрошенному адресу нет фрейма страницы.
21 - KSM
    Идентичные страницы памяти, динамически разделяемые между одним или
    несколькими процессами.
22 - THP
    Непрерывные страницы, которые образуют THP любого размера и отображены с
    любой гранулярностью.
23 - OFFLINE
    Страница логически отключена (offline).
24 - ZERO_PAGE
    Нулевая страница для pfn_zero или huge_zero.
25 - IDLE
    К странице не было обращений с тех пор, как она была помечена как idle
    (см. Documentation/admin-guide/mm/idle_page_tracking.rst).
    Обратите внимание, что этот флаг может быть устаревшим в случае, если к
    странице обращались через PTE. Чтобы убедиться, что флаг актуален, нужно
    сначала прочитать ``/sys/kernel/mm/page_idle/bitmap``.
26 - PGTABLE
    Страница используется как таблица страниц.

Флаги страниц, связанные с вводом-выводом
-----------------------------------------

1 - ERROR
   Произошла ошибка ввода-вывода.
3 - UPTODATE
   Страница содержит актуальные данные.
   То есть для страницы, отображённой на файл: (ревизия данных в памяти >=
   ревизии на диске)
4 - DIRTY
   В страницу была произведена запись, следовательно, она содержит новые
   данные.
   То есть для страницы, отображённой на файл: (ревизия данных в памяти >
   ревизии на диске)
8 - WRITEBACK
   Страница синхронизируется на диск.

Флаги страниц, связанные с LRU
------------------------------

5 - LRU
   Страница находится в одном из списков LRU.
6 - ACTIVE
   Страница находится в активном списке LRU.
18 - UNEVICTABLE
   Страница находится в (не-)LRU-списке невытесняемых страниц. Она каким-то
   образом закреплена (pinned) и не является кандидатом на освобождение страниц
   LRU, например, страницы ramfs, сегменты памяти shmctl(SHM_LOCK) и mlock().
2 - REFERENCED
   К странице обращались с момента её последнего добавления/переноса в список
   LRU.
9 - RECLAIM
   Страница будет освобождена вскоре после завершения операции ввода-вывода
   её вытеснения (pageout).
11 - MMAP
   Отображённая в память страница.
12 - ANON
   Отображённая в память страница, не являющаяся частью файла.
13 - SWAPCACHE
   Страница отображена на пространство swap, то есть имеет связанную запись в
   swap.
14 - SWAPBACKED
   Страница обеспечена swap/RAM.

Инструмент page-types в каталоге tools/mm можно использовать для запроса
указанных выше флагов.

Исключения для разделяемой памяти
=================================

Записи таблицы страниц для разделяемых страниц очищаются, когда страницы
сбрасываются (zapped) или выгружаются в swap. Из-за этого выгруженные в swap
страницы становятся неотличимы от никогда не выделявшихся.

В пространстве ядра расположение в swap всё ещё можно получить из page cache.
Однако значения, хранящиеся только в обычном PTE, безвозвратно теряются, когда
страница выгружается в swap (то есть SOFT_DIRTY).

В пространстве пользователя то, присутствует ли страница, выгружена ли она в
swap или отсутствует, можно вывести с помощью системных вызовов lseek и/или
mincore.

lseek() может различать страницы, к которым обращались (присутствующие или
выгруженные в swap), и дыры (отсутствующие/невыделенные), указав флаг
SEEK_DATA для файла, которым обеспечены страницы. Для анонимных разделяемых
страниц файл можно найти в ``/proc/pid/map_files/``.

mincore() может различать страницы в памяти (присутствующие, включая swap
cache) и вне памяти (выгруженные в swap или отсутствующие/невыделенные).

Прочие замечания
================

Чтение из любого из этих файлов вернёт -EINVAL, если вы начинаете чтение не с
границы в 8 байт (например, если вы выполнили seek на нечётное число байт в
файле) или если размер чтения не кратен 8 байтам.

До Linux 3.11 биты 55-60 pagemap использовались для «page-shift» (который на
большинстве архитектур всегда равен 12). Начиная с Linux 3.11 их значение
изменяется после первой очистки битов soft-dirty. Начиная с Linux 4.2 они
используются для флагов безусловно.

IOCTL для сканирования pagemap
==============================

IOCTL ``PAGEMAP_SCAN`` на файле pagemap можно использовать для получения или,
опционально, очистки информации о записях таблицы страниц. В этом IOCTL
поддерживаются следующие операции:

- Сканирование диапазона адресов и получение диапазонов памяти, соответствующих
  заданным критериям. Это выполняется, когда указан выходной буфер.
- Защита страниц от записи. ``PM_SCAN_WP_MATCHING`` используется для защиты от
  записи интересующих страниц. ``PM_SCAN_CHECK_WPASYNC`` прерывает операцию,
  если обнаружены страницы, не защищённые от записи в режиме Async.
  ``PM_SCAN_WP_MATCHING`` можно использовать с ``PM_SCAN_CHECK_WPASYNC`` или
  без него.
- Обе эти операции можно объединить в одну атомарную операцию, в которой мы
  можем как получить страницы, так и защитить их от записи.

В настоящее время поддерживаются следующие флаги о страницах:

- ``PAGE_IS_WPALLOWED`` - У страницы включена асинхронная защита от записи
- ``PAGE_IS_WRITTEN`` - В страницу производилась запись с момента, когда она
  была защищена от записи
- ``PAGE_IS_FILE`` - Страница обеспечена файлом
- ``PAGE_IS_PRESENT`` - Страница присутствует в памяти
- ``PAGE_IS_SWAPPED`` - Страница выгружена в swap
- ``PAGE_IS_PFNZERO`` - У страницы нулевой PFN
- ``PAGE_IS_HUGE`` - Страница является PMD-отображённой THP или обеспечена
  Hugetlb
- ``PAGE_IS_SOFT_DIRTY`` - Страница помечена как soft-dirty
- ``PAGE_IS_GUARD`` - Страница является частью защитной области (guard region)

В качестве аргумента IOCTL используется ``struct pm_scan_arg``.

 1. Размер ``struct pm_scan_arg`` должен быть указан в поле ``size``. Это поле
    будет полезно для распознавания структуры, если впоследствии будут
    выполнены расширения.
 2. Флаги можно указать в поле ``flags``. На данный момент добавлены только
    флаги ``PM_SCAN_WP_MATCHING`` и ``PM_SCAN_CHECK_WPASYNC``. Операция
    получения опционально выполняется в зависимости от того, предоставлен
    выходной буфер или нет.
 3. Диапазон задаётся через ``start`` и ``end``.
 4. Обход может прерваться до посещения полного диапазона, например, когда
    пользовательский буфер может заполниться и т. п. Адрес окончания обхода
    указывается в``end_walk``.
 5. Выходной буфер в виде массива ``struct page_region`` и его размер задаются
    в ``vec`` и ``vec_len``.
 6. Опциональное максимальное запрашиваемое число страниц задаётся в
    ``max_pages``.
 7. Маски задаются в ``category_mask``, ``category_anyof_mask``,
    ``category_inverted`` и ``return_mask``.

Найти страницы, в которые производилась запись, и также защитить их от записи::

   struct pm_scan_arg arg = {
   .size = sizeof(arg),
   .flags = PM_SCAN_CHECK_WPASYNC | PM_SCAN_CHECK_WPASYNC,
   ..
   .category_mask = PAGE_IS_WRITTEN,
   .return_mask = PAGE_IS_WRITTEN,
   };

Найти страницы, в которые производилась запись, которые обеспечены файлом, не
выгружены в swap и либо присутствуют, либо являются huge::

   struct pm_scan_arg arg = {
   .size = sizeof(arg),
   .flags = 0,
   ..
   .category_mask = PAGE_IS_WRITTEN | PAGE_IS_SWAPPED,
   .category_inverted = PAGE_IS_SWAPPED,
   .category_anyof_mask = PAGE_IS_PRESENT | PAGE_IS_HUGE,
   .return_mask = PAGE_IS_WRITTEN | PAGE_IS_SWAPPED |
                  PAGE_IS_PRESENT | PAGE_IS_HUGE,
   };

Флаг ``PAGE_IS_WRITTEN`` можно рассматривать как более производительную
альтернативу флагу soft-dirty. На него не влияет слияние VMA, выполняемое
ядром, и поэтому пользователь может найти настоящие soft-dirty-страницы в
случае обычных страниц. (Для страниц THP или Hugetlb всё ещё могут сообщаться
дополнительные dirty-страницы.)

Категория "PAGE_IS_WRITTEN" используется с диапазонами, для которых включена
защита от записи uffd, чтобы реализовать отслеживание изменений памяти (dirty
tracking) в пространстве пользователя:

 1. Файловый дескриптор userfaultfd создаётся системным вызовом
    ``userfaultfd``.
 2. Функциональности ``UFFD_FEATURE_WP_UNPOPULATED`` и
    ``UFFD_FEATURE_WP_ASYNC`` устанавливаются с помощью IOCTL ``UFFDIO_API``.
 3. Диапазон памяти регистрируется в режиме ``UFFDIO_REGISTER_MODE_WP`` через
    IOCTL ``UFFDIO_REGISTER``.
 4. Затем любая часть зарегистрированной памяти или вся область памяти должна
    быть защищена от записи с помощью IOCTL ``PAGEMAP_SCAN`` с флагом
    ``PM_SCAN_WP_MATCHING``, либо можно использовать IOCTL
    ``UFFDIO_WRITEPROTECT``. Оба этих варианта выполняют одну и ту же операцию.
    Первый из них лучше с точки зрения производительности.
 5. Теперь IOCTL ``PAGEMAP_SCAN`` можно использовать либо для того, чтобы
    просто найти страницы, в которые производилась запись с момента их
    последней пометки, и/или, опционально, также защитить страницы от записи.
