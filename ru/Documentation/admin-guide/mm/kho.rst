=====================================
Использование Kexec Handover (KHO)
=====================================

Kexec HandOver (KHO) — это механизм, который позволяет Linux сохранять области
памяти, способные содержать сериализованное состояние системы, при выполнении
kexec.

Этот документ предполагает, что вы знакомы с базовыми
:ref:`концепциями <kho-concepts>` KHO. Если вы их ещё не читали,
сделайте это сейчас.

Предварительные требования
==========================

KHO доступен, когда ядро собрано с параметром ``CONFIG_KEXEC_HANDOVER``,
установленным в y. У каждого производителя (producer) KHO может быть свой
собственный параметр конфигурации, который необходимо включить, если вы хотите
сохранить соответствующее состояние при выполнении kexec.

Чтобы использовать KHO, загрузите ядро с параметром командной строки
``kho=on``. Вы можете использовать параметр ``kho_scratch`` для задания размера
scratch-областей. Например, ``kho_scratch=16M,512M,256M`` зарезервирует при
загрузке scratch-область в нижней памяти размером 16 МиБ, глобальную
scratch-область размером 512 МиБ и scratch-области размером 256 МиБ на каждый
узел NUMA.

Выполнение KHO kexec
====================

Сначала, прежде чем выполнять KHO kexec, необходимо перевести систему в
:ref:`фазу финализации KHO <kho-finalization-phase>` ::

  $ echo 1 > /sys/kernel/debug/kho/out/finalize

После этой команды KHO FDT становится доступным в
``/sys/kernel/debug/kho/out/fdt``. Другие подсистемы также могут зарегистрировать
свои собственные сохранённые суб-FDT в каталоге
``/sys/kernel/debug/kho/out/sub_fdts/``.

Далее загрузите целевую полезную нагрузку (payload) и выполните в неё kexec.
Важно использовать параметр ``-s``, чтобы задействовать встроенный в ядро
загрузчик файлов kexec, поскольку пользовательский инструментарий kexec в
настоящее время не поддерживает KHO при использовании файлового загрузчика на
стороне пространства пользователя ::

  # kexec -l /path/to/bzImage --initrd /path/to/initrd -s
  # kexec -e

Новое ядро загрузится и будет содержать часть состояния предыдущего ядра.

Например, если вы использовали параметр командной строки ``reserve_mem`` для
создания раннего резервирования памяти, новое ядро будет иметь эту память по
тому же физическому адресу, что и старое ядро.

Прерывание KHO exec
===================

Вы можете снова вывести систему из фазы финализации KHO, выполнив ::

  $ echo 0 > /sys/kernel/debug/kho/out/active

После этой команды KHO FDT больше не доступен в
``/sys/kernel/debug/kho/out/fdt``.

Интерфейсы debugfs
==================

В настоящее время KHO создаёт следующие интерфейсы debugfs. Обратите внимание,
что эти интерфейсы могут измениться в будущем. Они будут перенесены в sysfs,
как только KHO стабилизируется.

``/sys/kernel/debug/kho/out/finalize``
    Kexec HandOver (KHO) позволяет Linux передавать состояние
    совместимых драйверов в следующее ядро, запущенное через kexec.
    Для этого драйверы устройств указывают KHO сохранить области
    памяти, которые могут содержать сериализованное состояние ядра.
    Пока состояние сериализовано, драйверы не могут вносить
    какие-либо изменения в сериализованное состояние, например в
    переданные выделения памяти.

    Когда этот файл содержит "1", система находится в переходном
    состоянии. Когда содержит "0" — нет. Чтобы переключаться между
    двумя состояниями, запишите (echo) соответствующее число в этот файл.

``/sys/kernel/debug/kho/out/fdt``
    Когда дерево состояния KHO финализировано, ядро предоставляет
    в этом файле flattened device tree blob, который несёт текущее
    состояние KHO. Пользовательский инструментарий kexec может
    использовать его как входной файл для образа полезной нагрузки KHO.

``/sys/kernel/debug/kho/out/scratch_len``
    Длины scratch-областей KHO — физически непрерывных областей
    памяти, которые всегда будут оставаться доступными для будущих
    выделений kexec. Пользовательские инструменты kexec могут
    использовать этот файл, чтобы определить, куда следует размещать
    образы полезной нагрузки.

``/sys/kernel/debug/kho/out/scratch_phys``
    Физические расположения scratch-областей KHO. Пользовательские
    инструменты kexec могут использовать этот файл совместно с
    scratch_phys, чтобы определить, куда следует размещать образы
    полезной нагрузки.

``/sys/kernel/debug/kho/out/sub_fdts/``
    В фазе финализации KHO производители (producers) KHO регистрируют
    в этом каталоге свой собственный FDT blob.

``/sys/kernel/debug/kho/in/fdt``
    Когда ядро было загружено через Kexec HandOver (KHO),
    дерево состояния, несущее метаданные о состоянии предыдущего
    ядра, находится в этом файле в формате flattened device tree.
    Этот файл может исчезнуть, когда все его потребители (consumers)
    завершат интерпретацию своих метаданных.

``/sys/kernel/debug/kho/in/sub_fdts/``
    Аналогично ``kho/out/sub_fdts/``, но содержит суб-FDT blob'ы
    производителей (producers) KHO, переданные от старого ядра.
