==========================
Политика памяти NUMA
==========================

Что такое политика памяти NUMA?
===============================

В ядре Linux «политика памяти» (memory policy) определяет, с какого узла ядро
будет выделять память в системе NUMA или в эмулированной системе NUMA.  Linux
поддерживает платформы с архитектурой неоднородного доступа к памяти
(Non-Uniform Memory Access) начиная с версии 2.4.?.  Текущая поддержка политики
памяти была добавлена в Linux 2.6 примерно в мае 2004 года.  Этот документ
ставит целью описать концепции и API поддержки политики памяти в 2.6.

Политику памяти не следует путать с cpusets
(``Documentation/admin-guide/cgroup-v1/cpusets.rst``),
которые представляют собой административный механизм ограничения набора узлов,
с которых набор процессов может выделять память. Политика памяти — это
программный интерфейс, которым может воспользоваться NUMA-осведомлённое
приложение.  Когда к задаче применяются и cpusets, и политики, ограничения
cpuset имеют приоритет.  Подробнее см. ниже в разделе
:ref:`Политика памяти и cpusets <mem_pol_and_cpusets>`.

Концепции политики памяти
=========================

Область действия политики памяти
--------------------------------

Ядро Linux поддерживает _области_ действия политики памяти, описанные здесь от
наиболее общей к наиболее специфической:

Системная политика по умолчанию (System Default Policy)
	эта политика «жёстко зашита» в ядро.  Это политика, которая
	управляет всеми выделениями страниц, не контролируемыми одной из
	более специфических областей политики, рассматриваемых ниже.  Когда
	система «запущена и работает», системная политика по умолчанию
	использует «локальное выделение», описанное ниже.  Однако во время
	загрузки системная политика по умолчанию устанавливается на
	чередование (interleave) выделений по всем узлам с «достаточным»
	объёмом памяти, чтобы не перегружать начальный узел загрузки
	выделениями во время загрузки.

Политика задачи/процесса (Task/Process Policy)
	это необязательная политика, задаваемая для каждой задачи.  Когда
	она определена для конкретной задачи, эта политика управляет всеми
	выделениями страниц, выполняемыми задачей или от её имени, которые
	не контролируются более специфической областью. Если задача не
	определяет политику задачи, то все выделения страниц, которые
	контролировались бы политикой задачи, «откатываются» (fall back) к
	системной политике по умолчанию.

	Политика задачи применяется ко всему адресному пространству задачи.
	Поэтому она наследуема и действительно наследуется как через fork()
	[clone() без флага CLONE_VM], так и через exec*().  Это позволяет
	родительской задаче установить политику задачи для дочерней задачи,
	запущенной через exec() из исполняемого образа, не осведомлённого о
	политике памяти.  Обзор системного вызова, который задача может
	использовать для установки/изменения своей политики задачи/процесса,
	см. ниже в разделе :ref:`API политики памяти <memory_policy_apis>`.

	В многопоточной задаче политики задачи применяются только к потоку
	[задаче ядра Linux], который устанавливает политику, и к любым
	потокам, впоследствии созданным этим потоком.  Любые соседние потоки,
	существующие на момент установки новой политики задачи, сохраняют
	свою текущую политику.

	Политика задачи применяется только к страницам, выделенным после
	установки политики.  Любые страницы, уже обработанные при отказе
	страницы задачей на момент изменения её политики задачи, остаются
	там, где они были выделены, в соответствии с политикой, действовавшей
	на момент их выделения.

.. _vma_policy:

Политика VMA (VMA Policy)
	«VMA», или «Virtual Memory Area» (область виртуальной памяти),
	обозначает диапазон виртуального адресного пространства задачи.  Задача
	может определить конкретную политику для диапазона своего виртуального
	адресного пространства.   Обзор системного вызова mbind(), используемого
	для установки политики VMA, см. ниже в разделе
	:ref:`API политики памяти <memory_policy_apis>`.

	Политика VMA будет управлять выделением страниц, обеспечивающих этот
	регион адресного пространства.  Любые регионы адресного пространства
	задачи, не имеющие явной политики VMA, откатятся к политике задачи,
	которая, в свою очередь, может откатиться к системной политике по
	умолчанию.

	У политик VMA есть несколько усложняющих деталей:

	* Политика VMA применяется ТОЛЬКО к анонимным страницам.  К ним
	  относятся страницы, выделенные для анонимных сегментов, таких как
	  стек и куча задачи, а также любые регионы адресного пространства,
	  отображённые через mmap() с флагом MAP_ANONYMOUS.  Если политика VMA
	  применяется к файловому отображению, она будет проигнорирована, если
	  отображение использовало флаг MAP_SHARED.  Если файловое отображение
	  использовало флаг MAP_PRIVATE, политика VMA будет применена только
	  тогда, когда анонимная страница выделяется при попытке записи в
	  отображение, то есть при копировании при записи (Copy-On-Write).

	* Политики VMA являются общими для всех задач, разделяющих виртуальное
	  адресное пространство, — иначе говоря, для потоков, — независимо от
	  того, когда политика была установлена; и они наследуются через
	  fork().  Однако, поскольку политики VMA относятся к конкретному
	  региону адресного пространства задачи и поскольку адресное
	  пространство отбрасывается и пересоздаётся при exec*(), политики VMA
	  НЕ наследуются через exec().  Таким образом, использовать политики
	  VMA могут только NUMA-осведомлённые приложения.

	* Задача может установить новую политику VMA для поддиапазона ранее
	  отображённого через mmap() региона.  Когда это происходит, Linux
	  разбивает существующую область виртуальной памяти на 2 или 3 VMA,
	  каждая со своей собственной политикой.

	* По умолчанию политика VMA применяется только к страницам, выделенным
	  после установки политики.  Любые страницы, уже обработанные по отказу
	  в диапазон VMA, остаются там, где они были выделены, в соответствии с
	  политикой, действовавшей на момент их выделения.  Однако начиная с
	  2.6.16 Linux поддерживает миграцию страниц через системный вызов
	  mbind(), так что содержимое страниц может быть перемещено в
	  соответствии с вновь установленной политикой.

Разделяемая политика (Shared Policy)
	Концептуально разделяемые политики применяются к «объектам памяти»,
	отображённым в разделяемом режиме в одно или несколько различных
	адресных пространств задач.  Приложение устанавливает разделяемые
	политики тем же способом, что и политики VMA, — используя системный
	вызов mbind() с указанием диапазона виртуальных адресов, отображающих
	разделяемый объект.  Однако, в отличие от политик VMA, которые можно
	считать атрибутом диапазона адресного пространства задачи, разделяемые
	политики применяются непосредственно к разделяемому объекту.  Таким
	образом, все задачи, подключающиеся к объекту, разделяют политику, и
	все страницы, выделенные для разделяемого объекта любой задачей, будут
	подчиняться разделяемой политике.

	По состоянию на 2.6.22 разделяемую политику поддерживают только
	сегменты разделяемой памяти, созданные через shmget() или
	mmap(MAP_ANONYMOUS|MAP_SHARED).  Когда поддержка разделяемой политики
	была добавлена в Linux, соответствующие структуры данных были добавлены
	к сегментам hugetlbfs shmem.  В то время hugetlbfs не поддерживала
	выделение во время отказа страницы — иначе говоря, ленивое выделение
	(lazy allocation), — поэтому сегменты hugetlbfs shmem так и не были
	«подключены» к поддержке разделяемой политики.  Хотя сегменты hugetlbfs
	теперь поддерживают ленивое выделение, их поддержка разделяемой политики
	не была завершена.

	Как упомянуто выше в разделе :ref:`политик VMA <vma_policy>`,
	выделения страниц страничного кэша для обычных файлов, отображённых
	через mmap() с MAP_SHARED, игнорируют любую политику VMA, установленную
	на диапазоне виртуальных адресов, обеспечиваемом разделяемым файловым
	отображением.  Вместо этого разделяемые страницы страничного кэша,
	включая страницы, обеспечивающие приватные отображения, ещё не
	записанные задачей, следуют политике задачи, если она есть, иначе —
	системной политике по умолчанию.

	Инфраструктура разделяемой политики поддерживает разные политики на
	поддиапазонах разделяемого объекта.  Однако Linux всё же разбивает VMA
	задачи, устанавливающей политику, для каждого диапазона с отдельной
	политикой.  Таким образом, разные задачи, подключающиеся к сегменту
	разделяемой памяти, могут иметь разные конфигурации VMA, отображающие
	этот один разделяемый объект.  Это можно увидеть, изучая
	/proc/<pid>/numa_maps задач, разделяющих регион разделяемой памяти, когда
	одна задача установила разделяемую политику на один или несколько
	диапазонов этого региона.

Компоненты политики памяти
--------------------------

Политика памяти NUMA состоит из «режима» (mode), необязательных флагов режима
и необязательного набора узлов.  Режим определяет поведение политики,
необязательные флаги режима определяют поведение режима, а необязательный
набор узлов можно рассматривать как аргументы поведения политики.

Внутренне политики памяти реализуются структурой со счётчиком ссылок —
struct mempolicy.  Детали этой структуры будут обсуждаться в контексте ниже по
мере необходимости для объяснения поведения.

Политика памяти NUMA поддерживает следующие 4 поведенческих режима:

Режим по умолчанию — MPOL_DEFAULT
	Этот режим используется только в API политики памяти.  Внутренне
	MPOL_DEFAULT преобразуется в NULL-политику памяти во всех областях
	политики.  Любая существующая не-умолчательная политика просто
	удаляется при указании MPOL_DEFAULT.  В результате MPOL_DEFAULT
	означает «откатиться к следующей наиболее специфической области
	политики».

	Например, NULL-политика, или политика задачи по умолчанию, откатится
	к системной политике по умолчанию.  NULL-политика, или политика vma по
	умолчанию, откатится к политике задачи.

	При указании в одном из API политики памяти режим по умолчанию не
	использует необязательный набор узлов.

	Указание непустого набора узлов для этой политики является ошибкой.

MPOL_BIND
	Этот режим указывает, что память должна поступать из набора узлов,
	заданного политикой.  Память будет выделяться из узла в наборе с
	достаточным объёмом свободной памяти, ближайшего к узлу, на котором
	происходит выделение.

MPOL_PREFERRED
	Этот режим указывает, что выделение следует пытаться выполнить из
	единственного узла, заданного в политике.  Если это выделение
	завершается неудачей, ядро будет искать другие узлы в порядке
	возрастания расстояния от предпочтительного узла на основе информации,
	предоставленной прошивкой (firmware) платформы.

	Внутренне политика Preferred использует единственный узел — член
	preferred_node структуры struct mempolicy.  Когда установлен внутренний
	флаг режима MPOL_F_LOCAL, preferred_node игнорируется, и политика
	интерпретируется как локальное выделение.  Политику «локального»
	выделения можно рассматривать как политику Preferred, начинающуюся с
	узла, содержащего CPU, на котором происходит выделение.

	Пользователь может указать, что локальное выделение всегда
	предпочтительно, передав с этим режимом пустую nodemask.  Если
	передана пустая nodemask, политика не может использовать флаги
	MPOL_F_STATIC_NODES или MPOL_F_RELATIVE_NODES, описанные ниже.

MPOL_INTERLEAVED
	Этот режим указывает, что выделения страниц чередуются с
	гранулярностью страницы по узлам, заданным в политике.  Этот режим
	также ведёт себя немного по-разному в зависимости от контекста, в
	котором он используется:

	Для выделения анонимных страниц и страниц разделяемой памяти режим
	Interleave индексирует набор узлов, заданный политикой, используя
	смещение страницы адреса отказа в сегменте [VMA], содержащем адрес,
	по модулю числа узлов, заданных политикой.  Затем он пытается выделить
	страницу, начиная с выбранного узла, как если бы узел был задан
	политикой Preferred или был выбран локальным выделением.  То есть
	выделение будет следовать зонному списку (zonelist) для каждого узла.

	Для выделения страниц страничного кэша режим Interleave индексирует
	набор узлов, заданный политикой, используя счётчик узлов, поддерживаемый
	для каждой задачи.  Этот счётчик возвращается к наименьшему заданному
	узлу после того, как достигает наибольшего заданного узла.  Это будет
	способствовать распределению страниц по узлам, заданным политикой, на
	основе порядка их выделения, а не на основе какого-либо смещения
	страницы в адресном диапазоне или файле.  Во время загрузки системы в
	этом режиме работает временная чередующаяся системная политика по
	умолчанию.

MPOL_PREFERRED_MANY
	Этот режим указывает, что выделение следует предпочтительно
	удовлетворять из nodemask, заданной в политике. Если на всех узлах в
	nodemask наблюдается давление на память, выделение может откатиться ко
	всем существующим узлам numa. По сути это MPOL_PREFERRED, разрешённый
	для маски, а не для единственного узла.

MPOL_WEIGHTED_INTERLEAVE
	Этот режим работает так же, как MPOL_INTERLEAVE, за тем исключением,
	что поведение чередования выполняется на основе весов, заданных в
	/sys/kernel/mm/mempolicy/weighted_interleave/

	Взвешенное чередование выделяет страницы на узлах в соответствии с
	весом.  Например, если узлы [0,1] имеют веса [5,2], то на каждые 2
	страницы, выделяемые на node1, будет выделяться 5 страниц на node0.

Политика памяти NUMA поддерживает следующие необязательные флаги режима:

MPOL_F_STATIC_NODES
	Этот флаг указывает, что nodemask, переданная пользователем, не должна
	переотображаться, если набор разрешённых узлов задачи или VMA меняется
	после того, как политика памяти была определена.

	Без этого флага каждый раз, когда mempolicy переустанавливается
	(rebound) из-за изменения набора разрешённых узлов, предпочтительная
        nodemask (Preferred Many), предпочтительный узел (Preferred) или
        nodemask (Bind, Interleave) переотображается на новый набор
        разрешённых узлов.  Это может привести к использованию узлов, которые
        ранее были нежелательны.

	С этим флагом, если заданные пользователем узлы пересекаются с узлами,
	разрешёнными cpuset задачи, то политика памяти применяется к их
	пересечению.  Если два набора узлов не пересекаются, используется
	политика по умолчанию (Default).

	Например, рассмотрим задачу, подключённую к cpuset с mems 1-3, которая
	устанавливает политику Interleave по тому же набору.  Если mems cpuset
	меняются на 3-5, чередование теперь будет происходить по узлам 3, 4 и 5.
	Однако с этим флагом, поскольку из пользовательской nodemask разрешён
	только узел 3, «чередование» происходит только по этому узлу.  Если ни
	один из узлов пользовательской nodemask теперь не разрешён, используется
	поведение по умолчанию (Default).

	MPOL_F_STATIC_NODES нельзя сочетать с флагом MPOL_F_RELATIVE_NODES.
	Его также нельзя использовать для политик MPOL_PREFERRED, которые были
	созданы с пустой nodemask (локальное выделение).

MPOL_F_RELATIVE_NODES
	Этот флаг указывает, что nodemask, переданная пользователем, будет
	отображена относительно набора разрешённых узлов задачи или VMA.  Ядро
	сохраняет переданную пользователем nodemask, и если набор разрешённых
	узлов меняется, то эта исходная nodemask будет переотображена
	относительно нового набора разрешённых узлов.

	Без этого флага (и без MPOL_F_STATIC_NODES) каждый раз, когда mempolicy
	переустанавливается из-за изменения набора разрешённых узлов, узел
	(Preferred) или nodemask (Bind, Interleave) переотображается на новый
	набор разрешённых узлов.  Это переотображение может не сохранять
	относительную природу переданной пользователем nodemask по отношению к
	её набору разрешённых узлов при последовательных переустановках:
	nodemask 1,3,5 может быть переотображена на 7-9, а затем на 1-3, если
	набор разрешённых узлов восстанавливается в своё исходное состояние.

	С этим флагом переотображение выполняется так, что номера узлов из
	переданной пользователем nodemask относительны набору разрешённых
	узлов.  Иными словами, если в пользовательской nodemask установлены
	узлы 0, 2 и 4, политика будет действовать на первый (а в случае Bind
	или Interleave — на третий и пятый) узлы из набора разрешённых узлов.
	Переданная пользователем nodemask представляет узлы относительно набора
	разрешённых узлов задачи или VMA.

	Если пользовательская nodemask включает узлы за пределами диапазона
	нового набора разрешённых узлов (например, узел 5 установлен в
	пользовательской nodemask, когда набор разрешённых узлов составляет лишь
	0-3), то переотображение оборачивается к началу nodemask и, если узел
	ещё не установлен, устанавливает его в nodemask политики mempolicy.

	Например, рассмотрим задачу, подключённую к cpuset с mems 2-5, которая
	устанавливает политику Interleave по тому же набору с
	MPOL_F_RELATIVE_NODES.  Если mems cpuset меняются на 3-7, чередование
	теперь происходит по узлам 3,5-7.  Если затем mems cpuset меняются на
	0,2-3,5, то чередование происходит по узлам 0,2-3,5.

	Благодаря согласованному переотображению приложения, готовящие
	nodemask для задания политик памяти с помощью этого флага, должны
	игнорировать своё текущее, фактическое размещение памяти, навязанное
	cpuset, и готовить nodemask так, как если бы они всегда располагались
	на узлах памяти от 0 до N-1, где N — число узлов памяти, которыми
	политика призвана управлять.  Пусть ядро затем переотобразит на набор
	узлов памяти, разрешённых cpuset задачи, так как он может меняться со
	временем.

	MPOL_F_RELATIVE_NODES нельзя сочетать с флагом MPOL_F_STATIC_NODES.
	Его также нельзя использовать для политик MPOL_PREFERRED, которые были
	созданы с пустой nodemask (локальное выделение).

Подсчёт ссылок политики памяти
==============================

Для разрешения гонок использования/освобождения struct mempolicy содержит поле
атомарного счётчика ссылок.  Внутренние интерфейсы mpol_get()/mpol_put()
соответственно увеличивают и уменьшают этот счётчик ссылок.  mpol_put()
освободит структуру обратно в kmem-кэш mempolicy, только когда счётчик ссылок
достигнет нуля.

Когда выделяется новая политика памяти, её счётчик ссылок инициализируется
значением «1», представляющим ссылку, удерживаемую задачей, которая
устанавливает новую политику.  Когда указатель на структуру политики памяти
сохраняется в другой структуре, добавляется ещё одна ссылка, поскольку ссылка
задачи будет сброшена по завершении установки политики.

Во время «использования» политики в run-time мы стараемся минимизировать
атомарные операции над счётчиком ссылок, поскольку это может привести к
перебрасыванию строк кэша между CPU и узлами NUMA.  «Использование» здесь
означает одно из следующего:

1) запрос политики либо самой задачей [с помощью API get_mempolicy(),
   обсуждаемого ниже], либо другой задачей через интерфейс
   /proc/<pid>/numa_maps.

2) изучение политики для определения режима политики и связанного узла или
   списков узлов, если они есть, для выделения страниц.  Это считается
   «горячим путём» (hot path).  Обратите внимание, что для MPOL_BIND
   «использование» простирается на весь процесс выделения, который может
   уходить в сон во время вытеснения страниц (page reclamation), поскольку
   nodemask политики BIND используется, по ссылке, для фильтрации непригодных
   узлов.

Мы можем избежать получения дополнительной ссылки во время перечисленных выше
использований следующим образом:

1) нам никогда не нужно получать/освобождать системную политику по умолчанию,
   поскольку она никогда не изменяется и не освобождается, как только система
   запущена и работает.

2) для запроса политики нам не нужно получать дополнительную ссылку на политику
   задачи целевой задачи или на политики vma, поскольку мы всегда захватываем
   mmap_lock структуры mm задачи на чтение во время запроса.  API
   set_mempolicy() и mbind() [см. ниже] всегда захватывают mmap_lock на запись
   при установке или замене политик задачи или vma.  Таким образом, нет
   возможности, чтобы задача или поток освободили политику, пока другая задача
   или поток её запрашивает.

3) Использование политики задачи или vma для выделения страниц происходит в
   пути отказа страницы, где мы удерживаем mmap_lock на чтение.  Опять же,
   поскольку замена политики задачи или vma требует, чтобы mmap_lock
   удерживался на запись, политика не может быть освобождена из-под нас, пока мы
   используем её для выделения страниц.

4) Разделяемые политики требуют особого рассмотрения.  Одна задача может
   заменить политику памяти разделяемой памяти, пока другая задача, с отдельным
   mmap_lock, запрашивает или выделяет страницу на основе политики.  Чтобы
   разрешить эту потенциальную гонку, инфраструктура разделяемой политики
   добавляет дополнительную ссылку на разделяемую политику во время поиска,
   удерживая спинлок на структуре управления разделяемой политикой.  Это
   требует, чтобы мы сбрасывали эту дополнительную ссылку, когда заканчиваем
   «использовать» политику.  Мы должны сбрасывать дополнительную ссылку на
   разделяемые политики в тех же путях запроса/выделения, что используются для
   неразделяемых политик.  По этой причине разделяемые политики помечаются как
   таковые, и дополнительная ссылка сбрасывается «условно» — то есть только для
   разделяемых политик.

   Из-за этого дополнительного подсчёта ссылок и из-за того, что мы должны
   искать разделяемые политики в древовидной структуре под спинлоком,
   разделяемые политики обходятся дороже при использовании в пути выделения
   страниц.  Это особенно верно для разделяемых политик на регионах разделяемой
   памяти, разделяемых задачами, выполняющимися на разных узлах NUMA.  Этих
   дополнительных накладных расходов можно избежать, всегда откатываясь к
   политике задачи или системной политике по умолчанию для регионов разделяемой
   памяти, либо предварительно обрабатывая отказы (prefaulting) всего региона
   разделяемой памяти в память и блокируя его в памяти.  Однако это может
   подходить не для всех приложений.

.. _memory_policy_apis:

API политики памяти
===================

Linux поддерживает 4 системных вызова для управления политикой памяти.  Эти
API всегда затрагивают только вызывающую задачу, адресное пространство
вызывающей задачи или некоторый разделяемый объект, отображённый в адресное
пространство вызывающей задачи.

.. note::
   заголовочные файлы, определяющие эти API, и типы данных параметров для
   приложений пространства пользователя находятся в пакете, не являющемся
   частью ядра Linux.  Интерфейсы системных вызовов ядра, с префиксом 'sys\_',
   определены в <linux/syscalls.h>; определения режимов и флагов определены в
   <linux/mempolicy.h>.

Установка политики памяти [задачи]::

	long set_mempolicy(int mode, const unsigned long *nmask,
					unsigned long maxnode);

Устанавливает «политику памяти задачи/процесса» вызывающей задачи в режим,
заданный аргументом 'mode', и набор узлов, определённый 'nmask'.  'nmask'
указывает на битовую маску идентификаторов узлов, содержащую как минимум
'maxnode' идентификаторов.  Необязательные флаги режима могут быть переданы
путём объединения аргумента 'mode' с флагом (например: MPOL_INTERLEAVE |
MPOL_F_STATIC_NODES).

Подробнее см. man-страницу set_mempolicy(2)


Получение политики памяти [задачи] или связанной информации::

	long get_mempolicy(int *mode,
			   const unsigned long *nmask, unsigned long maxnode,
			   void *addr, int flags);

Запрашивает «политику памяти задачи/процесса» вызывающей задачи либо политику
или местоположение указанного виртуального адреса, в зависимости от аргумента
'flags'.

Подробнее см. man-страницу get_mempolicy(2)


Установка политики VMA/разделяемой для диапазона адресного пространства задачи::

	long mbind(void *start, unsigned long len, int mode,
		   const unsigned long *nmask, unsigned long maxnode,
		   unsigned flags);

mbind() устанавливает политику, заданную (mode, nmask, maxnodes), как политику
VMA для диапазона адресного пространства вызывающей задачи, заданного
аргументами 'start' и 'len'.  Дополнительные действия могут быть запрошены
через аргумент 'flags'.

Подробнее см. man-страницу mbind(2).

Установка домашнего узла для диапазона адресного пространства задачи::

	long sys_set_mempolicy_home_node(unsigned long start, unsigned long len,
					 unsigned long home_node,
					 unsigned long flags);

sys_set_mempolicy_home_node устанавливает домашний узел для политики VMA,
присутствующей в адресном диапазоне задачи. Системный вызов обновляет домашний
узел только для существующего диапазона mempolicy. Другие адресные диапазоны
игнорируются. Домашний узел — это узел NUMA, ближе всего к которому будет
происходить выделение страниц. Указание домашнего узла переопределяет политику
выделения по умолчанию, чтобы выделять память близко к локальному узлу для
выполняющегося CPU.


Интерфейс командной строки политики памяти
==========================================

Хотя инструмент командной строки numactl(8) и не является строго частью
реализации политики памяти в Linux, он существует и позволяет:

+ установить политику задачи для указанной программы через set_mempolicy(2),
  fork(2) и exec(2)

+ установить разделяемую политику для сегмента разделяемой памяти через
  mbind(2)

Инструмент numactl(8) поставляется вместе с run-time-версией библиотеки,
содержащей обёртки системных вызовов политики памяти.  Некоторые дистрибутивы
поставляют заголовочные файлы и библиотеки времени компиляции в отдельном пакете
разработки.

.. _mem_pol_and_cpusets:

Политика памяти и cpusets
=========================

Политики памяти работают в рамках cpusets, как описано выше.  Для политик
памяти, требующих узел или набор узлов, узлы ограничены набором узлов, память
которых разрешена ограничениями cpuset.  Если nodemask, заданная для политики,
содержит узлы, не разрешённые cpuset, и MPOL_F_RELATIVE_NODES не используется,
используется пересечение набора узлов, заданного для политики, и набора узлов с
памятью.  Если результатом является пустой набор, политика считается
недопустимой и не может быть установлена.  Если используется
MPOL_F_RELATIVE_NODES, узлы политики отображаются на набор разрешённых узлов
задачи и сворачиваются в него, как описано ранее.

Взаимодействие политик памяти и cpusets может быть проблематичным, когда задачи
в двух cpusets разделяют доступ к региону памяти, такому как сегменты
разделяемой памяти, созданные через shmget() или mmap() с флагами MAP_ANONYMOUS
и MAP_SHARED, и любая из задач устанавливает разделяемую политику на регион:
тогда в политиках могут использоваться только узлы, память которых разрешена в
обоих cpusets.  Получение этой информации требует «выхода за пределы» API
политики памяти для использования информации cpuset и требует, чтобы было
известно, к каким cpusets могут подключаться другие задачи, обращающиеся к
разделяемому региону.  Кроме того, если разрешённые наборы памяти cpusets не
пересекаются, единственной допустимой политикой является «локальное» выделение.
