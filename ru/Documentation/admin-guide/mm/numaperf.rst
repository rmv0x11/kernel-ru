==============================
Производительность памяти NUMA
==============================

Локальность NUMA
================

На некоторых платформах к вычислительному узлу может быть подключено несколько
типов памяти. Эти разнородные диапазоны памяти могут иметь некоторые общие
характеристики, например когерентность кэша CPU, но при этом обладать различной
производительностью. Например, разные типы носителей и шины влияют на пропускную
способность и задержку.

Система поддерживает такую гетерогенную память, группируя каждый тип памяти под
разными доменами, или «узлами» (nodes), на основе характеристик локальности и
производительности. Часть памяти может располагаться в том же узле, что и CPU, а
другая предоставляется в виде узлов, содержащих только память. Хотя узлы, содержащие
только память, не предоставляют CPU, они всё же могут быть локальными для одного или
нескольких вычислительных узлов относительно других узлов. На следующей диаграмме
показан один такой пример: два вычислительных узла с локальной памятью и по одному
узлу, содержащему только память, для каждого из вычислительных узлов::

 +------------------+     +------------------+
 | Compute Node 0   +-----+ Compute Node 1   |
 | Local Node0 Mem  |     | Local Node1 Mem  |
 +--------+---------+     +--------+---------+
          |                        |
 +--------+---------+     +--------+---------+
 | Slower Node2 Mem |     | Slower Node3 Mem |
 +------------------+     +--------+---------+

«Инициатор памяти» (memory initiator) — это узел, содержащий одно или несколько
устройств, таких как CPU или отдельные устройства ввода-вывода памяти, которые
могут инициировать запросы к памяти. «Цель памяти» (memory target) — это узел,
содержащий один или несколько диапазонов физических адресов, доступных одному или
нескольким инициаторам памяти.

Когда существует несколько инициаторов памяти, они могут не все иметь одинаковую
производительность при доступе к данной цели памяти. Каждая пара инициатор-цель
может быть организована в различные ранжированные классы доступа для представления
этой связи. Инициатор с наивысшей производительностью по отношению к данной цели
считается одним из локальных инициаторов этой цели и получает наивысший класс
доступа, 0. Любая данная цель может иметь один или несколько локальных инициаторов,
а любой данный инициатор может иметь несколько локальных целей памяти.

Чтобы помочь приложениям сопоставлять цели памяти с их инициаторами, ядро
предоставляет символические ссылки друг на друга. В следующем примере перечислена
связь для инициаторов и целей памяти класса доступа «0»::

	# symlinks -v /sys/devices/system/node/nodeX/access0/targets/
	relative: /sys/devices/system/node/nodeX/access0/targets/nodeY -> ../../nodeY

	# symlinks -v /sys/devices/system/node/nodeY/access0/initiators/
	relative: /sys/devices/system/node/nodeY/access0/initiators/nodeX -> ../../nodeX

Инициатор памяти может иметь несколько целей памяти в одном и том же классе доступа.
Инициаторы целевой памяти в данном классе указывают на то, что характеристики
доступа узлов имеют одинаковую производительность относительно других связанных
узлов-инициаторов. Однако каждая цель в пределах класса доступа инициатора не
обязательно работает так же, как остальные.

Класс доступа «1» используется для того, чтобы можно было различать инициаторы,
являющиеся CPU и, следовательно, подходящие для общего планирования задач, и
инициаторы ввода-вывода, такие как GPU и NIC. В отличие от класса доступа 0,
рассматриваются только узлы, содержащие CPU.

Производительность NUMA
=======================

Приложения могут пожелать учитывать, из какого узла они хотят выделять память, на
основе характеристик производительности узла. Если система предоставляет эти
атрибуты, ядро экспортирует их в иерархии sysfs узлов, добавляя каталог атрибутов
под инициаторами класса доступа 0 узла памяти следующим образом::

	/sys/devices/system/node/nodeY/access0/initiators/

Эти атрибуты применяются только при доступе из узлов, которые связаны под
инициаторами этого класса доступа.

Характеристики производительности, которые ядро предоставляет для локальных
инициаторов и экспортирует, следующие::

	# tree -P "read*|write*" /sys/devices/system/node/nodeY/access0/initiators/
	/sys/devices/system/node/nodeY/access0/initiators/
	|-- read_bandwidth
	|-- read_latency
	|-- write_bandwidth
	`-- write_latency

Атрибуты пропускной способности указываются в МиБ/секунду.

Атрибуты задержки указываются в наносекундах.

Указанные здесь значения соответствуют номинальным задержке и пропускной
способности для платформы.

Класс доступа 1 имеет ту же форму, но включает значения только для активности
обращения CPU к памяти.

Кэш NUMA
========

Системная память может быть построена в виде иерархии элементов с различными
характеристиками производительности, чтобы предоставить большое адресное
пространство более медленной памяти, кэшируемой меньшей по объёму, но более
производительной памятью. Физические адреса системной памяти, о которых
осведомлены инициаторы памяти, предоставляются последним уровнем памяти в
иерархии. При этом система использует более производительную память для прозрачного
кэширования доступа к последовательно более медленным уровням.

Термин «дальняя память» (far memory) используется для обозначения памяти
последнего уровня в иерархии. Каждый возрастающий уровень кэша обеспечивает более
производительный доступ инициатора, а термин «ближняя память» (near memory)
представляет самый быстрый кэш, предоставляемый системой.

Эта нумерация отличается от кэшей CPU, где уровень кэша (напр.: L1, L2, L3)
использует представление со стороны CPU, в котором каждый возрастающий уровень
менее производителен. В отличие от этого, уровень кэша памяти отсчитывается
относительно памяти последнего уровня, поэтому более высокий номер уровня кэша
соответствует памяти, расположенной ближе к CPU и дальше от дальней памяти.

Кэши на стороне памяти не адресуются программно напрямую. Когда программное
обеспечение обращается к системному адресу, система возвращает его из кэша ближней
памяти, если он там присутствует. Если он отсутствует, система обращается к
следующему уровню памяти, пока не произойдёт попадание на этом уровне кэша или пока
не будет достигнута дальняя память.

Приложению не нужно знать об атрибутах кэширования, чтобы использовать систему.
Программное обеспечение может по желанию запросить атрибуты кэша памяти, чтобы
максимизировать производительность такой конфигурации. Если система предоставляет
ядру способ обнаружить эту информацию, например с помощью ACPI HMAT (Heterogeneous
Memory Attribute Table), ядро добавит эти атрибуты к цели памяти узла NUMA.

Когда ядро впервые регистрирует кэш памяти для узла, ядро создаёт следующий
каталог::

	/sys/devices/system/node/nodeX/memory_side_cache/

Если этот каталог отсутствует, то система либо не предоставляет кэш на стороне
памяти, либо эта информация недоступна ядру.

Атрибуты для каждого уровня кэша предоставляются под индексом его уровня кэша::

	/sys/devices/system/node/nodeX/memory_side_cache/indexA/
	/sys/devices/system/node/nodeX/memory_side_cache/indexB/
	/sys/devices/system/node/nodeX/memory_side_cache/indexC/

Каталог каждого уровня кэша предоставляет его атрибуты. Например, ниже показан один
уровень кэша и атрибуты, доступные для запроса программным обеспечением::

	# tree /sys/devices/system/node/node0/memory_side_cache/
	/sys/devices/system/node/node0/memory_side_cache/
	|-- index1
	|   |-- indexing
	|   |-- line_size
	|   |-- size
	|   `-- write_policy

Значение «indexing» будет равно 0, если это кэш с прямым отображением
(direct-mapped), и ненулевым для любой другой многоканальной ассоциативности на
основе индексирования.

Значение «line_size» — это количество байт, считываемых со следующего уровня кэша
при промахе.

Значение «size» — это количество байт, предоставляемых данным уровнем кэша.

Значение «write_policy» будет равно 0 для кэширования с обратной записью
(write-back) и ненулевым для кэширования со сквозной записью (write-through).
