===================================================
Краткое руководство пользователя по slab-аллокатору
===================================================

Slab-аллокатор включает полную поддержку отладки (при сборке с
CONFIG_SLUB_DEBUG=y), но по умолчанию она выключена (если только сборка не
выполнена с CONFIG_SLUB_DEBUG_ON=y). Отладку можно включить только для
выбранных slab'ов, чтобы избежать влияния на общую производительность системы,
которое может затруднить поиск ошибки.

Чтобы включить отладку, можно добавить опцию ``slab_debug`` в командную строку
ядра. Это включит полную отладку для всех slab'ов.

Обычно затем используется команда ``slabinfo`` для получения статистических
данных и выполнения операций над slab'ами. По умолчанию ``slabinfo`` выводит
только те slab'ы, в которых есть данные. См. "slabinfo -h" для других опций при
запуске команды. ``slabinfo`` можно скомпилировать с помощью
::

	gcc -o slabinfo tools/mm/slabinfo.c

Некоторые режимы работы ``slabinfo`` требуют, чтобы отладка slub была включена в
командной строке. Например, без включённой отладки не будет доступна информация
отслеживания, а проверка (validation) может быть выполнена лишь частично, если
отладка не была включена.

Некоторые более сложные применения slab_debug:
-----------------------------------------------

Параметры могут быть переданы ``slab_debug``. Если ничего не указано, то
включается полная отладка. Формат:

slab_debug=<Debug-Options>
	Включить опции для всех slab'ов

slab_debug=<Debug-Options>,<slab name1>,<slab name2>,...
	Включить опции только для выбранных slab'ов (без пробелов
	после запятой)

Может быть задано несколько блоков опций для всех slab'ов или выбранных slab'ов,
причём блоки опций разделяются символом ';'. Последний из блоков «всех slab'ов»
применяется ко всем slab'ам, кроме тех, что соответствуют одному из блоков
«выбранных slab'ов». Применяются опции первого блока «выбранных slab'ов»,
который соответствует имени slab'а.

Возможные опции отладки::

	F		Sanity checks on (enables SLAB_DEBUG_CONSISTENCY_CHECKS
			Sorry SLAB legacy issues)
	Z		Red zoning
	P		Poisoning (object and padding)
	U		User tracking (free and alloc)
	T		Trace (please only use on single slabs)
	A		Enable failslab filter mark for the cache
	O		Switch debugging off for caches that would have
			caused higher minimum slab orders
	-		Switch all debugging off (useful if the kernel is
			configured with CONFIG_SLUB_DEBUG_ON)

Например, чтобы загрузиться только с проверками целостности и красными зонами,
следует указать::

	slab_debug=FZ

Пытаетесь найти проблему в кэше dentry? Попробуйте::

	slab_debug=,dentry

чтобы включить отладку только для кэша dentry. Можно использовать звёздочку в
конце имени slab'а, чтобы охватить все slab'ы с одинаковым префиксом. Например,
вот как можно отравить кэш dentry, а также все slab'ы kmalloc::

	slab_debug=P,kmalloc-*,dentry

Красные зоны и отслеживание могут изменить выравнивание slab'а. Можно применить к
кэшу dentry только проверки целостности с помощью::

	slab_debug=F,dentry

Опции отладки могут потребовать увеличения минимально возможного порядка slab'а в
результате хранения метаданных (например, кэши с размерами объектов PAGE_SIZE).
Это с большей вероятностью приведёт к ошибкам выделения slab'ов в ситуациях
нехватки памяти или при высокой фрагментации памяти. Чтобы по умолчанию выключить
отладку для таких кэшей, используйте::

	slab_debug=O

Можно применять различные опции к различным спискам имён slab'ов, используя блоки
опций. Это включит красные зоны для dentry и отслеживание пользователя для
kmalloc. Для всех остальных slab'ов отладка не будет включена::

	slab_debug=Z,dentry;U,kmalloc-*

Можно также включить опции (например, проверки целостности и отравление) для всех
кэшей, кроме некоторых, которые считаются слишком критичными для
производительности и не нуждаются в отладке, указав глобальные опции отладки, за
которыми следует список имён slab'ов с "-" в качестве опций::

	slab_debug=FZ;-,zs_handle,zspage

Состояние каждой опции отладки для slab'а можно найти в соответствующих файлах в
каталоге::

	/sys/kernel/slab/<slab name>/

Если файл содержит 1, опция включена, 0 означает выключено. Опции отладки из
параметра ``slab_debug`` соответствуют следующим файлам::

	F	sanity_checks
	Z	red_zone
	P	poison
	U	store_user
	T	trace
	A	failslab

Файл failslab доступен для записи, поэтому запись 1 или 0 включит или выключит
опцию во время выполнения. Запись возвращает -EINVAL, если кэш является
псевдонимом (alias). Будьте осторожны с трассировкой: она может выдавать большой
объём информации и никогда не останавливаться, если использовать её на
неправильном slab'е.

Объединение slab'ов
===================

Если опции отладки не указаны, то SLUB может объединять схожие slab'ы вместе,
чтобы снизить накладные расходы и повысить «горячесть» кэша (cache hotness)
объектов. ``slabinfo -a`` показывает, какие slab'ы были объединены вместе.

Проверка (validation) slab'ов
=============================

SLUB может проверять все объекты, если ядро было загружено с slab_debug. Чтобы
сделать это, у вас должен быть инструмент ``slabinfo``. Затем можно выполнить
::

	slabinfo -v

что протестирует все объекты. Вывод будет сгенерирован в syslog.

Это также работает более ограниченным образом, если загрузка была без отладки
slab. В этом случае ``slabinfo -v`` просто тестирует все достижимые объекты.
Обычно это объекты в slab'ах CPU и в частичных (partial) slab'ах. Полные (full)
slab'ы не отслеживаются SLUB в неотладочной ситуации.

Повышение производительности
============================

В некоторой степени производительность SLUB ограничена необходимостью время от
времени захватывать list_lock для работы с частичными slab'ами. Эти накладные
расходы определяются порядком выделения для каждого slab'а. На выделения можно
влиять с помощью параметров ядра:

.. slab_min_objects=x		(default: automatically scaled by number of cpus)
.. slab_min_order=x		(default 0)
.. slab_max_order=x		(default 3 (PAGE_ALLOC_COSTLY_ORDER))

``slab_min_objects``
	позволяет указать, сколько объектов должно как минимум помещаться в один
	slab, чтобы порядок выделения был приемлемым. В общем случае slub сможет
	выполнить это количество выделений на slab'е без обращения к
	централизованным ресурсам (list_lock), где может возникать конкуренция
	(contention).

``slab_min_order``
	задаёт минимальный порядок slab'ов. Эффект аналогичен
	``slab_min_objects``.

``slab_max_order``
	задаёт порядок, при котором ``slab_min_objects`` больше не должен
	проверяться. Это полезно, чтобы избежать попыток SLUB генерировать
	сверхбольшие страницы высокого порядка для размещения ``slab_min_objects``
	slab-кэша с большими размерами объектов в одной странице высокого порядка.
	Установка параметра командной строки ``debug_guardpage_minorder=N``
	(N > 0) принудительно устанавливает ``slab_max_order`` в 0, что вызывает
	минимально возможный порядок выделения slab'ов.

``slab_strict_numa``
        Включает применение политик памяти при каждом
        выделении. Это приводит к более точному размещению
        объектов, что может привести к снижению обращений
        к удалённым узлам. По умолчанию политики памяти
        применяются только на уровне folio, когда новая folio приобретается
        или folio извлекается из списков. Включение
        этой опции снижает производительность быстрого пути (fastpath) slab-аллокатора.

Если SLUB обнаруживает повреждённый объект (полное обнаружение требует, чтобы
ядро было загружено с slab_debug), то в syslog будет сброшен следующий вывод:

1. Описание обнаруженной проблемы

   Это будет сообщение в системном журнале, начинающееся с::

     ===============================================
     BUG <slab cache affected>: <What went wrong>
     -----------------------------------------------

     INFO: <corruption start>-<corruption_end> <more info>
     INFO: Slab <address> <slab information>
     INFO: Object <address> <object information>
     INFO: Allocated in <kernel function> age=<jiffies since alloc> cpu=<allocated by
	cpu> pid=<pid of the process>
     INFO: Freed in <kernel function> age=<jiffies since free> cpu=<freed by cpu>
	pid=<pid of the process>

   (Информация о выделении / освобождении объекта доступна только если для
   slab'а установлен SLAB_STORE_USER. slab_debug устанавливает эту опцию)

2. Содержимое объекта, если был задействован объект.

   За строкой BUG SLUB могут следовать различные типы строк:

   Bytes b4 <address> : <bytes>
	Показывает несколько байт перед объектом, где была обнаружена проблема.
	Может быть полезно, если повреждение не заканчивается на начале
	объекта.

   Object <address> : <bytes>
	Байты объекта. Если объект неактивен, то байты
	обычно содержат значения отравления (poison values). Любое не-poison
	значение указывает на повреждение из-за записи после освобождения.

   Redzone <address> : <bytes>
	Redzone, следующая за объектом. Redzone используется для обнаружения
	записей после объекта. Все байты всегда должны иметь одинаковое
	значение. Любое отклонение вызвано записью за
	границей объекта.

	(Информация о Redzone доступна только если установлен SLAB_RED_ZONE.
	slab_debug устанавливает эту опцию)

   Padding <address> : <bytes>
	Неиспользуемые данные, заполняющие пространство для правильного
	выравнивания следующего объекта. В отладочном случае мы гарантируем, что
	имеется как минимум 4 байта заполнения. Это позволяет обнаруживать записи
	перед объектом.

3. Дамп стека (stackdump)

   Дамп стека описывает место, где была обнаружена ошибка. Причину
   повреждения, вероятно, легче найти, посмотрев на функцию, которая
   выделила или освободила объект.

4. Отчёт о том, как проблема была обработана для обеспечения дальнейшей
   работы системы.

   Это сообщения в системном журнале, начинающиеся с::

	FIX <slab cache affected>: <corrective action taken>

   В приведённом выше примере SLUB обнаружил, что Redzone активного объекта была
   перезаписана. Здесь строка из 8 символов была записана в slab, который имеет
   длину 8 символов. Однако строке из 8 символов нужен завершающий 0. Этот ноль
   перезаписал первый байт поля Redzone. После сообщения подробностей возникшей
   проблемы сообщение FIX SLUB говорит нам, что SLUB восстановил Redzone до её
   правильного значения, после чего работа системы продолжается.

Аварийные операции
==================

Минимальную отладку (только проверки целостности) можно включить, загрузившись с::

	slab_debug=F

Этого, как правило, будет достаточно для включения функций устойчивости slub,
которые позволят системе продолжать работу даже если неисправный компонент ядра
продолжит повреждать объекты. Это может быть важно для производственных систем.
На производительность повлияют проверки целостности, и будет непрерывный поток
сообщений об ошибках в syslog, но дополнительная память использоваться не будет
(в отличие от полной отладки).

Никаких гарантий. Компонент ядра всё равно нужно исправить. Производительность
можно оптимизировать ещё больше, найдя slab, в котором происходит повреждение, и
включив отладку только для этого кэша

То есть::

	slab_debug=F,dentry

Если повреждение происходит из-за записи после конца объекта, то может быть
целесообразно включить Redzone, чтобы избежать повреждения начала других
объектов::

	slab_debug=FZ,dentry

Расширенный режим slabinfo и построение графиков
================================================

Инструмент ``slabinfo`` имеет специальный «расширенный» ('-X') режим, который
включает:

 - Итоги slab-кэша (Slabcache Totals)
 - Slab'ы, отсортированные по размеру (до -N <num> slab'ов, по умолчанию 1)
 - Slab'ы, отсортированные по потерям (loss) (до -N <num> slab'ов, по умолчанию 1)

Кроме того, в этом режиме ``slabinfo`` не масштабирует размеры динамически
(G/M/K) и сообщает всё в байтах (эта функциональность также доступна в других
режимах slabinfo через опцию '-B'), что делает отчёты более точными и аккуратными.
Более того, в некотором смысле режим `-X' также упрощает анализ поведения
slab'ов, потому что его вывод можно построить в виде графика с помощью скрипта
``slabinfo-gnuplot.sh``. Таким образом, он переводит анализ от просматривания
чисел (массы чисел) к чему-то более простому — визуальному анализу.

Чтобы сгенерировать графики:

a) соберите расширенные записи slabinfo, например::

	while [ 1 ]; do slabinfo -X >> FOO_STATS; sleep 1; done

b) передайте файл(ы) статистики скрипту ``slabinfo-gnuplot.sh``::

	slabinfo-gnuplot.sh FOO_STATS [FOO_STATS2 .. FOO_STATSN]

   Скрипт ``slabinfo-gnuplot.sh`` предварительно обработает собранные записи и
   сгенерирует 3 png-файла (и 3 файла кэша предварительной обработки) на каждый
   файл STATS:
   - Итоги slab-кэша: FOO_STATS-totals.png
   - Slab'ы, отсортированные по размеру: FOO_STATS-slabs-by-size.png
   - Slab'ы, отсортированные по потерям: FOO_STATS-slabs-by-loss.png

Другой случай использования, когда ``slabinfo-gnuplot.sh`` может быть полезен, —
это когда нужно сравнить поведение slab'ов «до» и «после» некоторого изменения
кода. Чтобы помочь в этом, скрипт ``slabinfo-gnuplot.sh`` может «объединять»
секции `Slabcache Totals` из разных измерений. Чтобы визуально сравнить N
графиков:

a) Соберите столько файлов STATS1, STATS2, .. STATSN, сколько нужно::

	while [ 1 ]; do slabinfo -X >> STATS<X>; sleep 1; done

b) Предварительно обработайте эти файлы STATS::

	slabinfo-gnuplot.sh STATS1 STATS2 .. STATSN

c) Выполните ``slabinfo-gnuplot.sh`` в режиме '-t', передав все
   сгенерированные предварительно обработанные \*-totals::

	slabinfo-gnuplot.sh -t STATS1-totals STATS2-totals .. STATSN-totals

   Это создаст единый график (png-файл).

   Графики, как и ожидается, могут быть большими, поэтому некоторые колебания
   или небольшие всплески могут остаться незамеченными. Чтобы справиться с этим,
   ``slabinfo-gnuplot.sh`` имеет две опции для «приближения»/«отдаления»:

   a) ``-s %d,%d`` -- переопределяет ширину и высоту изображения по умолчанию
   b) ``-r %d,%d`` -- задаёт диапазон используемых выборок (например,
      в случае ``slabinfo -X >> FOO_STATS; sleep 1;`` использование диапазона
      ``-r 40,60`` построит график только по выборкам, собранным между 40-й и
      60-й секундами).


Файлы DebugFS для SLUB
======================

Для получения дополнительной информации о текущем состоянии кэшей SLUB с
включённой опцией отладки отслеживания пользователя доступны файлы debugfs,
обычно в каталоге /sys/kernel/debug/slab/<cache>/ (создаются только для кэшей с
включённым отслеживанием пользователя). Существует 2 типа этих файлов со
следующей отладочной информацией:

1. alloc_traces::

    Выводит информацию об уникальных трассировках выделения текущих
    выделенных объектов. Вывод отсортирован по частоте каждой трассировки.

    Информация в выводе:
    Количество объектов, выделяющая функция, возможные потери памяти
    объектов kmalloc (всего/на объект), минимальное/среднее/максимальное число
    jiffies с момента выделения, диапазон pid выделяющих процессов, маска cpu
    выделяющих cpu, маска узлов numa — источников памяти и трассировка стека.

    Пример:::

    338 pci_alloc_dev+0x2c/0xa0 waste=521872/1544 age=290837/291891/293509 pid=1 cpus=106 nodes=0-1
        __kmem_cache_alloc_node+0x11f/0x4e0
        kmalloc_trace+0x26/0xa0
        pci_alloc_dev+0x2c/0xa0

2. free_traces::

    Выводит информацию об уникальных трассировках освобождения текущих
    выделенных объектов. Трассировки освобождения, таким образом, относятся к
    предыдущему жизненному циклу объектов и сообщаются как недоступные для
    объектов, выделенных впервые. Вывод отсортирован по частоте каждой трассировки.

    Информация в выводе:
    Количество объектов, освобождающая функция, минимальное/среднее/максимальное число jiffies с момента освобождения,
    диапазон pid освобождающих процессов, маска cpu освобождающих cpu и трассировка стека.
