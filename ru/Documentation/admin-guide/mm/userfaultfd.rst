===========
Userfaultfd
===========

Назначение
==========

Userfaults (пользовательские сбои страниц) позволяют реализовать подкачку
страниц по требованию из пространства пользователя, а в более общем смысле
дают пространству пользователя возможность взять под контроль различные
сбои страниц памяти — то, что иначе мог бы делать только код ядра.

Например, userfaults позволяют корректно и более оптимально реализовать
приём ``PROT_NONE+SIGSEGV``.

Устройство
==========

Пространство пользователя создаёт новый userfaultfd, инициализирует его и
регистрирует в нём одну или несколько областей виртуальной памяти. После
этого любые сбои страниц, происходящие внутри такой области (областей),
приводят к доставке сообщения в userfaultfd, уведомляющего пространство
пользователя о сбое.

``userfaultfd`` (помимо регистрации и снятия регистрации диапазонов
виртуальной памяти) предоставляет две основные функциональности:

1) протокол ``read/POLLIN`` для уведомления потока пространства пользователя
   о происходящих сбоях

2) различные ioctl-вызовы ``UFFDIO_*``, которые могут управлять областями
   виртуальной памяти, зарегистрированными в ``userfaultfd``, что позволяет
   пространству пользователя эффективно разрешать получаемые им userfaults
   через пункт 1) или управлять виртуальной памятью в фоновом режиме

Настоящее преимущество userfaults по сравнению с обычным управлением
виртуальной памятью посредством mremap/mprotect состоит в том, что
userfaults во всех своих операциях никогда не задействуют тяжеловесные
структуры вроде vma (по сути нагрузка времени выполнения ``userfaultfd``
никогда не берёт mmap_lock на запись). Vma не подходят для отслеживания
сбоев с гранулярностью страницы (или огромной страницы) при работе с
виртуальными адресными пространствами, которые могут охватывать терабайты.
Для этого потребовалось бы слишком много vma.

``userfaultfd``, будучи созданным, может также передаваться через
unix-сокеты домена процессу-менеджеру, так что один и тот же
процесс-менеджер может обрабатывать userfaults множества различных
процессов без того, чтобы они знали о происходящем (если, конечно, они
позже сами не попытаются использовать ``userfaultfd`` на той же области,
которую менеджер уже отслеживает, — это краевой случай, который в настоящее
время вернул бы ``-EBUSY``).

API
===

Создание userfaultfd
--------------------

Существует два способа создать новый userfaultfd, каждый из которых
предоставляет средства ограничения доступа к этой функциональности
(поскольку исторически userfaultfd, обрабатывающие сбои страниц ядра, были
полезным инструментом для эксплуатации ядра).

Первый способ, поддерживаемый с момента появления userfaultfd, — это
системный вызов userfaultfd(2). Доступ к нему контролируется несколькими
способами:

- Любой пользователь всегда может создать userfaultfd, который перехватывает
  только сбои страниц пространства пользователя. Такой userfaultfd можно
  создать с помощью системного вызова userfaultfd(2) с флагом
  UFFD_USER_MODE_ONLY.

- Чтобы также перехватывать сбои страниц ядра для адресного пространства,
  либо процесс должен обладать привилегией CAP_SYS_PTRACE, либо в системе
  должно быть установлено vm.unprivileged_userfaultfd в 1. По умолчанию
  vm.unprivileged_userfaultfd установлено в 0.

Второй способ, добавленный в ядро позднее, — это открытие
/dev/userfaultfd и выполнение для него ioctl USERFAULTFD_IOC_NEW. Этот метод
даёт userfaultfd, эквивалентные тем, что создаёт системный вызов
userfaultfd(2).

В отличие от userfaultfd(2), доступ к /dev/userfaultfd контролируется через
обычные права файловой системы (пользователь/группа/режим), что даёт
тонко гранулированный доступ именно к userfaultfd, не предоставляя при этом
других не связанных с ним привилегий (как сделало бы, например,
предоставление CAP_SYS_PTRACE). Пользователи, имеющие доступ к
/dev/userfaultfd, всегда могут создавать userfaultfd, перехватывающие сбои
страниц ядра; vm.unprivileged_userfaultfd при этом не учитывается.

Инициализация userfaultfd
-------------------------

При первом открытии ``userfaultfd`` должен быть включён вызовом ioctl
``UFFDIO_API`` с указанием значения ``uffdio_api.api``, установленного в
``UFFD_API`` (или более поздней версии API), которое задаёт протокол
``read/POLLIN``, на котором пространство пользователя намеревается общаться
по ``UFFD``, а также требуемые им ``uffdio_api.features``. В случае успеха
ioctl ``UFFDIO_API`` (то есть если запрошенный ``uffdio_api.api``
поддерживается также и работающим ядром, а запрошенные возможности будут
включены) вернёт в ``uffdio_api.features`` и ``uffdio_api.ioctls`` две
64-битные битовые маски: соответственно всех доступных возможностей протокола
read(2) и доступных универсальных ioctl.

Битовая маска ``uffdio_api.features``, возвращаемая ioctl ``UFFDIO_API``,
определяет, какие типы памяти поддерживаются ``userfaultfd`` и какие события,
кроме уведомлений о сбоях страниц, могут генерироваться:

- Флаги ``UFFD_FEATURE_EVENT_*`` указывают, что поддерживаются различные
  другие события помимо сбоев страниц. Эти события более подробно описаны
  ниже в разделе `Non-cooperative userfaultfd`_.

- ``UFFD_FEATURE_MISSING_HUGETLBFS`` и ``UFFD_FEATURE_MISSING_SHMEM``
  указывают, что ядро поддерживает регистрации
  ``UFFDIO_REGISTER_MODE_MISSING`` соответственно для областей виртуальной
  памяти hugetlbfs и разделяемой памяти (охватывая все API shmem, то есть
  tmpfs, ``IPCSHM``, ``/dev/zero``, ``MAP_SHARED``, ``memfd_create`` и т.д.).

- ``UFFD_FEATURE_MINOR_HUGETLBFS`` указывает, что ядро поддерживает
  регистрацию ``UFFDIO_REGISTER_MODE_MINOR`` для областей виртуальной памяти
  hugetlbfs. ``UFFD_FEATURE_MINOR_SHMEM`` — аналогичная возможность,
  указывающая на поддержку для областей виртуальной памяти shmem.

- ``UFFD_FEATURE_MOVE`` указывает, что ядро поддерживает перемещение
  содержимого существующей страницы из пространства пользователя.

Приложение пространства пользователя должно установить флаги возможностей,
которые оно намеревается использовать, при вызове ioctl ``UFFDIO_API``,
чтобы запросить включение этих возможностей, если они поддерживаются.

После того как API ``userfaultfd`` включён, следует вызвать ioctl
``UFFDIO_REGISTER`` (если он присутствует в возвращённой битовой маске
``uffdio_api.ioctls``), чтобы зарегистрировать диапазон памяти в
``userfaultfd``, соответствующим образом задав структуру uffdio_register.
Битовая маска ``uffdio_register.mode`` укажет ядру, какие виды сбоев
отслеживать для данного диапазона. Ioctl ``UFFDIO_REGISTER`` вернёт битовую
маску ``uffdio_register.ioctls`` тех ioctl, которые пригодны для разрешения
userfaults в зарегистрированном диапазоне. Не все ioctl обязательно будут
поддерживаться для всех типов памяти (например, анонимная память против
shmem против hugetlbfs) или для всех типов перехватываемых сбоев.

Пространство пользователя может использовать ``uffdio_register.ioctls`` для
управления виртуальным адресным пространством в фоновом режиме (для
добавления, а потенциально и для удаления памяти из зарегистрированного
диапазона ``userfaultfd``). Это означает, что userfault может срабатывать
прямо перед тем, как пространство пользователя в фоновом режиме отобразит
страницу, вызвавшую сбой.

Разрешение userfaults
---------------------

Существует три основных способа разрешения userfaults:

- ``UFFDIO_COPY`` атомарно копирует содержимое некоторой существующей
  страницы из пространства пользователя.

- ``UFFDIO_ZEROPAGE`` атомарно обнуляет новую страницу.

- ``UFFDIO_CONTINUE`` отображает существующую, ранее заполненную страницу.

Эти операции атомарны в том смысле, что они гарантируют невозможность увидеть
наполовину заполненную страницу, поскольку читатели будут продолжать вызывать
userfault, пока операция не завершится.

По умолчанию они пробуждают userfaults, заблокированные на рассматриваемом
диапазоне. Они поддерживают флаг ``mode`` ``UFFDIO_*_MODE_DONTWAKE``, который
указывает, что пробуждение будет выполнено отдельно в некоторый более поздний
момент.

Какой ioctl выбрать, зависит от вида сбоя страницы и от того, что мы хотели
бы сделать для его разрешения:

- Для сбоев ``UFFDIO_REGISTER_MODE_MISSING`` сбой нужно разрешить либо
  предоставлением новой страницы (``UFFDIO_COPY``), либо отображением нулевой
  страницы (``UFFDIO_ZEROPAGE``). По умолчанию ядро отобразило бы нулевую
  страницу для сбоя по отсутствию. С userfaultfd пространство пользователя
  может решить, какое содержимое предоставить, прежде чем поток, вызвавший
  сбой, продолжит работу.

- Для сбоев ``UFFDIO_REGISTER_MODE_MINOR`` существует уже имеющаяся страница
  (в кэше страниц). У пространства пользователя есть возможность изменить
  содержимое страницы перед разрешением сбоя. Как только содержимое корректно
  (изменено или нет), пространство пользователя просит ядро отобразить
  страницу и позволить вызвавшему сбой потоку продолжить работу с помощью
  ``UFFDIO_CONTINUE``.

Примечания:

- Вы можете определить, какой вид сбоя произошёл, проанализировав
  ``pagefault.flags`` внутри ``uffd_msg``, проверив флаги
  ``UFFD_PAGEFAULT_FLAG_*``.

- Ни один из ioctl, доставляющих страницы, не использует по умолчанию
  диапазон, который вы зарегистрировали.  Вы должны заполнить все поля
  соответствующей структуры ioctl, включая диапазон.

- Адрес обращения, вызвавшего событие отсутствующей страницы, вы получаете
  из структуры uffd_msg, которую читаете в потоке из uffd.  С помощью этих
  IOCTL вы можете предоставить столько страниц, сколько хотите.
  Имейте в виду, что, если только вы не использовали DONTWAKE, первый из любых
  из этих IOCTL пробуждает поток, вызвавший сбой.

- Обязательно проверяйте все ошибки, включая
  (``pollfd[0].revents & POLLERR``).  Это может произойти, например, когда
  предоставленные диапазоны были некорректны.

Уведомления о защите от записи
------------------------------

Это эквивалентно использованию mprotect и обработчика сигнала SIGSEGV (но
быстрее).

Сначала вам нужно зарегистрировать диапазон с ``UFFDIO_REGISTER_MODE_WP``.
Вместо использования mprotect(2) вы используете
``ioctl(uffd, UFFDIO_WRITEPROTECT, struct *uffdio_writeprotect)``
при ``mode = UFFDIO_WRITEPROTECT_MODE_WP``
в передаваемой структуре.  Диапазон не используется по умолчанию и не обязан
совпадать с диапазоном, который вы зарегистрировали.  Вы можете защитить от
записи столько диапазонов, сколько пожелаете (внутри зарегистрированного
диапазона). Затем в потоке, читающем из uffd, в структуре будет установлен
``msg.arg.pagefault.flags & UFFD_PAGEFAULT_FLAG_WP``. Теперь вы посылаете
``ioctl(uffd, UFFDIO_WRITEPROTECT, struct *uffdio_writeprotect)``
снова, при этом в ``pagefault.mode`` не установлен
``UFFDIO_WRITEPROTECT_MODE_WP``. Это пробуждает поток, который продолжит
выполнение с записями. Это позволяет вам вести учёт записи в потоке,
читающем uffd, до выполнения ioctl.

Если вы зарегистрировались одновременно с ``UFFDIO_REGISTER_MODE_MISSING`` и
``UFFDIO_REGISTER_MODE_WP``, то вам нужно подумать о последовательности, в
которой вы предоставляете страницу и снимаете защиту от записи.  Обратите
внимание, что есть разница между записью в область WP и в область !WP.
В первом случае будет установлен ``UFFD_PAGEFAULT_FLAG_WP``, во втором —
``UFFD_PAGEFAULT_FLAG_WRITE``.  Во втором случае сбой произошёл не из-за
защиты, но вам всё равно нужно предоставить страницу, если был использован
``UFFDIO_REGISTER_MODE_MISSING``.

Режим защиты от записи userfaultfd в настоящее время ведёт себя по-разному в
отношении none-pte (когда, например, страница отсутствует) для разных типов
памяти.

Для анонимной памяти ``ioctl(UFFDIO_WRITEPROTECT)`` будет игнорировать
none-pte (например, когда страницы отсутствуют и не заполнены).  Для памяти,
подкреплённой файлами, такой как shmem и hugetlbfs, none-pte будут защищены
от записи точно так же, как присутствующая pte.  Иными словами, при записи в
отсутствующую страницу для памяти файлового типа будет генерироваться
сообщение о сбое записи userfaultfd, при условии что диапазон страниц был
защищён от записи до этого.  Такое сообщение по умолчанию не будет
генерироваться для анонимной памяти.

Если приложение хочет иметь возможность защищать от записи none-pte на
анонимной памяти, можно предварительно заполнить память, например с помощью
MADV_POPULATE_READ.  На более новых ядрах можно также обнаружить возможность
UFFD_FEATURE_WP_UNPOPULATED и заранее установить её бит, чтобы убедиться, что
none-pte будут защищены от записи даже на анонимной памяти.

При использовании ``UFFDIO_REGISTER_MODE_WP`` в сочетании с либо
``UFFDIO_REGISTER_MODE_MISSING``, либо ``UFFDIO_REGISTER_MODE_MINOR``, при
разрешении сбоев missing / minor с помощью ``UFFDIO_COPY`` или
``UFFDIO_CONTINUE`` соответственно, может быть желательно, чтобы новая
страница / отображение были защищены от записи (так чтобы будущие записи
также приводили к сбою WP). Эти ioctl поддерживают флаг режима
(``UFFDIO_COPY_MODE_WP`` или ``UFFDIO_CONTINUE_MODE_WP`` соответственно),
чтобы настроить отображение таким образом.

Если в контексте userfaultfd установлен бит возможности
``UFFD_FEATURE_WP_ASYNC``, любой vma, зарегистрированный с защитой от записи,
будет работать в асинхронном режиме, а не в синхронном режиме по умолчанию.

В асинхронном режиме при выполнении операции записи не будет генерироваться
никакого сообщения, при этом защита от записи будет автоматически разрешаться
ядром.  Это можно рассматривать как более точную версию отслеживания
soft-dirty, и оно может отличаться в нескольких аспектах:

  - Результат dirty не будет зависеть от изменений vma (например, слияния
    vma), поскольку dirty отслеживается только в pte.

  - Он по умолчанию поддерживает операции над диапазонами, так что можно
    включить отслеживание для любого диапазона памяти, выровненного по
    границе страницы.

  - Информация о dirty не будет потеряна, если pte была обнулена по различным
    причинам (например, при разделении прозрачной огромной страницы shmem).

  - Из-за обратного смысла soft-dirty (страница чистая, когда бит uffd-wp
    установлен; грязная, когда бит uffd-wp снят) он имеет иную семантику для
    некоторых операций над памятью.  Например: ``MADV_DONTNEED`` для
    анонимной (или ``MADV_REMOVE`` для файлового отображения) будет
    трактоваться как загрязнение памяти посредством сброса бита uffd-wp в
    ходе процедуры.

Приложение пользователя может собирать статус «записана/грязная»
("written/dirty"), просматривая бит uffd-wp для интересующих его страниц в
/proc/pagemap.

Страница не будет находиться под отслеживанием асинхронного режима uffd-wp,
пока она явно не защищена от записи вызовом ``ioctl(UFFDIO_WRITEPROTECT)`` с
установленным флагом режима ``UFFDIO_WRITEPROTECT_MODE_WP``.  Попытка
разрешить сбой страницы, которая отслеживалась асинхронным режимом
userfaultfd-wp, недопустима.

Когда асинхронный режим userfaultfd-wp используется отдельно, он может
применяться ко всем видам памяти.

Эмуляция отравления памяти
--------------------------

В ответ на сбой (либо missing, либо minor) одним из действий, которые
пространство пользователя может предпринять для его «разрешения», является
выполнение ``UFFDIO_POISON``. Это приведёт к тому, что любые будущие
обращения, вызывающие сбой, либо получат SIGBUS, либо, в случае KVM, гость
получит MCE, как если бы имело место аппаратное отравление памяти.

Это используется для эмуляции аппаратного отравления памяти. Представьте ВМ,
работающую на машине, в которой возникает реальная аппаратная ошибка памяти.
Позднее мы живьём мигрируем ВМ на другую физическую машину. Поскольку мы
хотим, чтобы миграция была прозрачной для гостя, мы хотим, чтобы тот же
диапазон адресов вёл себя так, как будто он по-прежнему отравлен, хотя теперь
он находится на новом физическом хосте, у которого, по идее, нет ошибки
памяти в точно том же месте.

QEMU/KVM
========

QEMU/KVM использует системный вызов ``userfaultfd`` для реализации
посткопирующей живой миграции (postcopy live migration). Посткопирующая живая
миграция — это одна из форм экстернализации памяти, состоящая в том, что
виртуальная машина работает с частью или всей своей памятью, находящейся на
другом узле в облаке. Абстракция ``userfaultfd`` достаточно универсальна,
чтобы не пришлось изменять ни единой строки кода ядра KVM ради добавления
посткопирующей живой миграции в QEMU.

Асинхронные сбои страниц гостя, ``FOLL_NOWAIT`` и все другие возможности
``GUP*`` прекрасно работают в сочетании с userfaults. Userfaults вызывают
асинхронные сбои страниц в планировщике гостя, так что те гостевые процессы,
которые не ожидают userfaults (то есть привязанные к сети), могут продолжать
выполняться в гостевых vcpu.

Как правило, полезно выполнить один проход прекопирующей живой миграции
непосредственно перед запуском посткопирующей живой миграции, чтобы избежать
генерации userfaults для областей гостя, доступных только для чтения.

Реализация посткопирующей живой миграции в настоящее время использует один
двунаправленный сокет, но в будущем будут использоваться два разных сокета
(чтобы свести задержку userfaults к минимально возможной без необходимости
уменьшать ``/proc/sys/net/ipv4/tcp_wmem``).

QEMU на узле-источнике записывает в сокет все страницы, которые, как ему
известно, отсутствуют на узле-приёмнике, а поток миграции QEMU, работающего
на узле-приёмнике, выполняет ioctl ``UFFDIO_COPY|ZEROPAGE`` на
``userfaultfd``, чтобы отобразить полученные страницы в гостя
(``UFFDIO_ZEROCOPY`` используется, если исходная страница была нулевой
страницей).

Отдельный посткопирующий поток на узле-приёмнике параллельно слушает
``userfaultfd`` с помощью poll(). Когда после срабатывания userfault
генерируется событие ``POLLIN``, посткопирующий поток выполняет read() из
``userfaultfd`` и получает адрес сбоя (или ``-EAGAIN`` в случае, если
userfault уже был разрешён и пробуждён выполнением ``UFFDIO_COPY|ZEROPAGE``
параллельным потоком миграции QEMU).

После того как посткопирующий поток QEMU (работающий на узле-приёмнике)
получает адрес userfault, он записывает информацию об отсутствующей странице
в сокет. Узел-источник QEMU получает эту информацию и приблизительно
«перематывает» к этому адресу страницы и продолжает отправлять все оставшиеся
отсутствующие страницы с этого нового смещения страницы. Вскоре после этого
(ровно столько времени, сколько нужно, чтобы протолкнуть очередь tcp_wmem по
сети) поток миграции в QEMU, работающем на узле-приёмнике, получит страницу,
вызвавшую userfault, и отобразит её обычным образом с помощью
``UFFDIO_COPY|ZEROPAGE`` (фактически не зная, была ли она спонтанно отправлена
источником или это была срочная страница, запрошенная через userfault).

К моменту начала userfaults QEMU на узле-приёмнике не нужно хранить никакой
постраничной битовой карты состояния, относящейся к живой миграции, а в QEMU,
работающем на узле-источнике, должна поддерживаться единственная постраничная
битовая карта, чтобы знать, какие страницы всё ещё отсутствуют на
узле-приёмнике. Битовая карта на узле-источнике проверяется, чтобы выяснить,
какие отсутствующие страницы отправлять по принципу round robin, и мы
перематываем по ней при получении входящих userfaults. После отправки каждой
страницы битовая карта, разумеется, обновляется соответствующим образом. Это
также полезно, чтобы избежать отправки одной и той же страницы дважды (на
случай, если userfault прочитан посткопирующим потоком прямо перед тем, как в
потоке миграции выполнится ``UFFDIO_COPY|ZEROPAGE``).

Non-cooperative userfaultfd
===========================

Когда ``userfaultfd`` отслеживается внешним менеджером, менеджер должен иметь
возможность отслеживать изменения в раскладке виртуальной памяти процесса.
Userfaultfd может уведомлять менеджер о таких изменениях, используя тот же
протокол read(2), что и для уведомлений о сбоях страниц. Менеджер должен явно
включить эти события, установив соответствующие биты в
``uffdio_api.features``, передаваемых в ioctl ``UFFDIO_API``:

``UFFD_FEATURE_EVENT_FORK``
	включает перехваты (hooks) ``userfaultfd`` для fork(). Когда эта
	возможность включена, контекст ``userfaultfd`` родительского процесса
	дублируется во вновь созданный процесс. Менеджер получает
	``UFFD_EVENT_FORK`` с файловым дескриптором нового контекста
	``userfaultfd`` в ``uffd_msg.fork``.

``UFFD_FEATURE_EVENT_REMAP``
	включает уведомления о вызовах mremap(). Когда некооперативный процесс
	перемещает область виртуальной памяти в другое место, менеджер получит
	``UFFD_EVENT_REMAP``. ``uffd_msg.remap`` будет содержать старый и новый
	адреса области и её исходную длину.

``UFFD_FEATURE_EVENT_REMOVE``
	включает уведомления о вызовах madvise(MADV_REMOVE) и
	madvise(MADV_DONTNEED). Событие ``UFFD_EVENT_REMOVE`` будет
	генерироваться при этих вызовах madvise(). ``uffd_msg.remove``
	будет содержать начальный и конечный адреса удалённой области.

``UFFD_FEATURE_EVENT_UNMAP``
	включает уведомления о снятии отображения памяти. Менеджер получит
	``UFFD_EVENT_UNMAP`` с ``uffd_msg.remove``, содержащим начальный и
	конечный адреса области, отображение которой было снято.

Хотя ``UFFD_FEATURE_EVENT_REMOVE`` и ``UFFD_FEATURE_EVENT_UNMAP`` довольно
похожи, они существенно различаются по действию, ожидаемому от менеджера
``userfaultfd``. В первом случае удаляется виртуальная память, но не область:
область остаётся отслеживаемой ``userfaultfd``, и если в ней произойдёт сбой
страницы, он будет доставлен менеджеру. Правильное разрешение такого сбоя
страницы — обнулить отображение по адресу сбоя. Однако во втором случае, когда
отображение области снимается, либо явно (системным вызовом munmap()), либо
неявно (например, в ходе mremap()), область удаляется, и в свою очередь
контекст ``userfaultfd`` для такой области тоже исчезает, и менеджер больше не
будет получать сбои страниц пространства пользователя из удалённой области.
Тем не менее уведомление необходимо, чтобы не дать менеджеру использовать
``UFFDIO_COPY`` на области, отображение которой снято.

В отличие от сбоев страниц пространства пользователя, которые должны быть
синхронными и требуют явного или неявного пробуждения, все события доставляются
асинхронно, и некооперативный процесс возобновляет выполнение, как только
менеджер выполнит read(). Менеджер ``userfaultfd`` должен тщательно
синхронизировать вызовы ``UFFDIO_COPY`` с обработкой событий. Чтобы облегчить
синхронизацию, ioctl ``UFFDIO_COPY`` вернёт ``-ENOSPC``, когда отслеживаемый
процесс завершается во время ``UFFDIO_COPY``, и ``-ENOENT``, когда
некооперативный процесс изменил свою раскладку виртуальной памяти одновременно
с незавершённой операцией ``UFFDIO_COPY``.

Текущая асинхронная модель доставки событий оптимальна для однопоточных
реализаций некооперативного менеджера ``userfaultfd``. Синхронная модель
доставки событий может быть добавлена позднее как новая возможность
``userfaultfd``, чтобы способствовать усовершенствованиям многопоточности
некооперативного менеджера, например чтобы позволить ioctl ``UFFDIO_COPY``
выполняться параллельно с приёмом событий. Однопоточным реализациям следует
по-прежнему использовать текущую асинхронную модель доставки событий.
