=====================
Подробное руководство
=====================

DAMON предоставляет следующие интерфейсы для разных категорий пользователей.

- *Специализированные модули DAMON.*
  :ref:`Этот вариант <damon_modules_special_purpose>` предназначен для тех, кто
  собирает, распространяет и/или администрирует ядро со специализированными
  способами применения DAMON.  С его помощью пользователи могут простым образом
  применять основные возможности DAMON для заданных целей на этапе сборки,
  загрузки или во время выполнения.
- *Инструмент DAMON для пространства пользователя.*
  `Этот вариант <https://github.com/damonitor/damo>`_ предназначен для
  привилегированных пользователей, таких как системные администраторы, которым
  нужен готовый к работе и удобный для человека интерфейс.  С его помощью
  пользователи могут применять основные возможности DAMON в удобной для человека
  форме.  Однако он может быть не очень хорошо настроен для особых случаев.
  Дополнительные сведения см. в его `документе об использовании
  <https://github.com/damonitor/damo/blob/next/USAGE.md>`_.
- *Интерфейс sysfs.*
  :ref:`Этот вариант <sysfs_interface>` предназначен для привилегированных
  программистов пространства пользователя, которым нужно более оптимизированное
  использование DAMON.  С его помощью пользователи могут применять основные
  возможности DAMON, читая из специальных файлов sysfs и записывая в них.
  Таким образом, вы можете написать и использовать свои персональные
  программы-обёртки DAMON для sysfs, которые читают/записывают файлы sysfs
  вместо вас.  `Инструмент DAMON для пространства пользователя
  <https://github.com/damonitor/damo>`_ — один из примеров таких программ.
- *Интерфейс программирования для пространства ядра.*
  :doc:`Этот вариант </mm/damon/api>` предназначен для программистов пространства
  ядра.  С его помощью пользователи могут наиболее гибко и эффективно использовать
  все возможности DAMON, написав для себя программы DAMON для пространства ядра.
  Вы можете даже расширить DAMON для различных адресных пространств.  Подробности
  см. в :doc:`документе </mm/damon/api>` об интерфейсе.

.. _sysfs_interface:

Интерфейс sysfs
===============

Интерфейс sysfs DAMON собирается, когда определён ``CONFIG_DAMON_SYSFS``.  Он
создаёт несколько каталогов и файлов в своём каталоге sysfs,
``<sysfs>/kernel/mm/damon/``.  Вы можете управлять DAMON, записывая в файлы
этого каталога и читая из них.

В качестве короткого примера: пользователи могут отслеживать виртуальное
адресное пространство заданной рабочей нагрузки, как показано ниже. ::

    # cd /sys/kernel/mm/damon/admin/
    # echo 1 > kdamonds/nr_kdamonds && echo 1 > kdamonds/0/contexts/nr_contexts
    # echo vaddr > kdamonds/0/contexts/0/operations
    # echo 1 > kdamonds/0/contexts/0/targets/nr_targets
    # echo $(pidof <workload>) > kdamonds/0/contexts/0/targets/0/pid_target
    # echo on > kdamonds/0/state

.. _sysfs_root:

Root
----

Корень интерфейса sysfs DAMON — это ``<sysfs>/kernel/mm/damon/``, и в нём есть
один каталог с именем ``admin``.  Этот каталог содержит файлы для управления
DAMON со стороны привилегированных программ пространства пользователя.  Этим
каталогом могут пользоваться инструменты или демоны пространства пользователя,
имеющие права root.

.. _sysfs_kdamonds:

kdamonds/
---------

В каталоге ``admin`` находится один каталог, ``kdamonds``, содержащий файлы для
управления экземплярами kdamond (подробнее см.
:ref:`design <damon_design_execution_model_and_data_structures>`).  Изначально
в этом каталоге есть только один файл, ``nr_kdamonds``.  Запись числа (``N``) в
этот файл создаёт указанное число дочерних каталогов с именами от ``0`` до
``N-1``.  Каждый каталог представляет отдельный kdamond.

.. _sysfs_kdamond:

kdamonds/<N>/
-------------

В каждом каталоге kdamond есть три файла (``state``, ``pid`` и ``refresh_ms``) и
один каталог (``contexts``).

Чтение ``state`` возвращает ``on``, если kdamond сейчас выполняется, или
``off``, если он не выполняется.

Пользователи могут записывать в файл ``state`` приведённые ниже команды для
kdamond.

- ``on``: Запустить выполнение.
- ``off``: Остановить выполнение.
- ``commit``: Снова прочитать пользовательский ввод из файлов sysfs, кроме файла
  ``state``.  Входные данные :ref:`целевого региона <sysfs_regions>`
  мониторинга также игнорируются, если ни один целевой регион не задан.
- ``update_tuned_intervals``: Обновить содержимое файлов ``sample_us`` и
  ``aggr_us`` соответствующего kdamond значениями ``sampling interval`` и
  ``aggregation interval``, полученными в результате автоматической настройки.
  Подробнее см. в разделе
  :ref:`intervals_goal <damon_usage_sysfs_monitoring_intervals_goal>`.
- ``commit_schemes_quota_goals``: Прочитать
  :ref:`цели по квотам <sysfs_schemes_quota_goals>` для схем операций на основе
  DAMON.
- ``update_schemes_stats``: Обновить содержимое файлов статистики для каждой
  схемы операций на основе DAMON у данного kdamond.  Подробнее о статистике
  см. в разделе :ref:`stats <sysfs_schemes_stats>`.
- ``update_schemes_tried_regions``: Обновить каталог регионов, к которым было
  предпринято применение действия схемы операций на основе DAMON, для каждой
  схемы операций на основе DAMON у данного kdamond.  Подробнее о каталоге
  регионов, к которым было предпринято применение действия схемы операций на
  основе DAMON, см. в разделе
  :ref:`tried_regions <sysfs_schemes_tried_regions>`.
- ``update_schemes_tried_bytes``: Обновить только файлы
  ``.../tried_regions/total_bytes``.
- ``clear_schemes_tried_regions``: Очистить каталог регионов, к которым было
  предпринято применение действия схемы операций на основе DAMON, для каждой
  схемы операций на основе DAMON у данного kdamond.
- ``update_schemes_effective_quotas``: Обновить содержимое файлов
  ``effective_bytes`` для каждой схемы операций на основе DAMON у данного
  kdamond.  Подробнее см. в разделе :ref:`каталог quotas <sysfs_quotas>`.

Если состояние — ``on``, чтение ``pid`` показывает pid потока kdamond.

Пользователи могут попросить ядро периодически обновлять файлы, показывающие
автоматически настроенные параметры и статистику DAMOS, вместо того чтобы
вручную записывать в файл ``state`` ключевые слова вроде
``update_tuned_intervals``.  Для этого пользователям следует записать желаемый
интервал обновления в миллисекундах в файл ``refresh_ms``.  Если интервал равен
нулю, периодическое обновление отключается.  Чтение этого файла показывает
текущий заданный интервал времени.

Каталог ``contexts`` содержит файлы для управления контекстами мониторинга,
которые будет выполнять данный kdamond.

.. _sysfs_contexts:

kdamonds/<N>/contexts/
----------------------

Изначально в этом каталоге есть только один файл, ``nr_contexts``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельный контекст мониторинга
(подробнее см. :ref:`design <damon_design_execution_model_and_data_structures>`).
На данный момент поддерживается только один контекст на kdamond, поэтому в файл
можно записать только ``0`` или ``1``.

.. _sysfs_context:

contexts/<N>/
-------------

В каждом каталоге контекста есть три файла (``avail_operations``, ``operations``
и ``addr_unit``) и три каталога (``monitoring_attrs``, ``targets`` и
``schemes``).

DAMON поддерживает несколько типов :ref:`операций мониторинга
<damon_design_configurable_operations_set>`, включая операции для виртуального
адресного пространства и для физического адресного пространства.  Вы можете
получить список доступных наборов операций мониторинга на текущем работающем
ядре, прочитав файл ``avail_operations``.  В зависимости от конфигурации ядра в
этом файле будут перечислены разные доступные наборы операций.  Список всех
доступных наборов операций и их краткие описания см. в
:ref:`design <damon_operations_set>`.

Вы можете задать и узнать, какой тип операций мониторинга DAMON будет
использовать для контекста, записав одно из ключевых слов, перечисленных в файле
``avail_operations``, и прочитав файл ``operations``.

Файл ``addr_unit`` предназначен для задания и получения параметра
:ref:`единицы адреса <damon_design_addr_unit>` набора операций.

.. _sysfs_monitoring_attrs:

contexts/<N>/monitoring_attrs/
------------------------------

Файлы для задания атрибутов мониторинга, включая требуемые качество и
эффективность мониторинга, находятся в каталоге ``monitoring_attrs``.  В
частности, в этом каталоге есть два каталога, ``intervals`` и ``nr_regions``.

В каталоге ``intervals`` есть три файла для интервала выборки DAMON
(``sample_us``), интервала агрегации (``aggr_us``) и интервала обновления
(``update_us``).  Вы можете задавать и получать значения в микросекундах,
записывая в эти файлы и читая из них.

В каталоге ``nr_regions`` есть два файла для нижней и верхней границ числа
регионов мониторинга DAMON (соответственно ``min`` и ``max``), которые управляют
накладными расходами на мониторинг.  Вы можете задавать и получать эти значения,
записывая в эти файлы и читая из них.

Подробнее об интервалах и диапазоне числа регионов мониторинга см. в документе
Design (:doc:`/mm/damon/design`).

.. _damon_usage_sysfs_monitoring_intervals_goal:

contexts/<N>/monitoring_attrs/intervals/intervals_goal/
-------------------------------------------------------

В каталоге ``intervals`` также есть один каталог для автоматической настройки
``sample_us`` и ``aggr_us``, а именно каталог ``intervals_goal``.  В этом
каталоге есть четыре файла для управления автоматической настройкой, а именно
``access_bp``, ``aggrs``, ``min_sample_us`` и ``max_sample_us``.  Описание
внутреннего устройства механизма настройки см. в :ref:`документе Design об этой
возможности <damon_design_monitoring_intervals_autotuning>`.  Чтение и запись
четырёх файлов в каталоге ``intervals_goal`` показывает и обновляет параметры
настройки, описанные в :ref:`документе Design
<damon_design_monitoring_intervals_autotuning>` под теми же именами.  Настройка
начинается с заданных пользователем значений ``sample_us`` и ``aggr_us``.
Текущие значения этих двух интервалов после применения настройки можно прочитать
из файлов ``sample_us`` и ``aggr_us`` после записи ``update_tuned_intervals`` в
файл ``state``.

.. _sysfs_targets:

contexts/<N>/targets/
---------------------

Изначально в этом каталоге есть только один файл, ``nr_targets``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельную цель мониторинга.

.. _sysfs_target:

targets/<N>/
------------

В каждом каталоге цели есть два файла (``pid_target`` и ``obsolete_target``) и
один каталог (``regions``).

Если вы записали ``vaddr`` в ``contexts/<N>/operations``, каждая цель должна быть
процессом.  Вы можете указать процесс для DAMON, записав pid процесса в файл
``pid_target``.

Пользователи могут выборочно удалять цели из середины массива целей, записав
ненулевое значение в файл ``obsolete_target`` и зафиксировав это (записав
``commit`` в файл ``state``).  DAMON удалит соответствующие цели из своего
внутреннего массива целей.  Пользователи обязаны заново сформировать каталоги
целей так, чтобы они корректно представляли изменившийся внутренний массив
целей.


.. _sysfs_regions:

targets/<N>/regions
-------------------

В случае наборов операций мониторинга ``fvaddr`` или ``paddr`` пользователи
обязаны задать целевые диапазоны адресов мониторинга.  В случае набора операций
``vaddr`` это не обязательно, но пользователи могут по желанию задать начальный
регион мониторинга для конкретных диапазонов адресов.  Подробнее см. в
:ref:`design <damon_design_vaddr_target_regions_construction>`.

В таких случаях пользователи могут явно задать начальные целевые регионы
мониторинга по своему усмотрению, записав нужные значения в файлы этого каталога.

Изначально в этом каталоге есть только один файл, ``nr_regions``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельный начальный целевой регион
мониторинга.

Если при онлайн-фиксации новых параметров DAMON (записи ``commit`` в файл
``state`` :ref:`kdamond <sysfs_kdamond>`) ``nr_regions`` равно нулю, логика
фиксации игнорирует целевые регионы.  Иными словами, текущие результаты
мониторинга для данной цели сохраняются.

.. _sysfs_region:

regions/<N>/
------------

В каждом каталоге региона вы найдёте два файла (``start`` и ``end``).  Вы можете
задавать и получать начальный и конечный адреса начального целевого региона
мониторинга, записывая в эти файлы и читая из них соответственно.

Регионы не должны перекрываться друг с другом.  ``end`` каталога ``N`` должен
быть равен или меньше, чем ``start`` каталога ``N+1``.

.. _sysfs_schemes:

contexts/<N>/schemes/
---------------------

Каталог для схем операций на основе DAMON (:ref:`DAMOS
<damon_design_damos>`).  Пользователи могут получать и задавать схемы, читая из
файлов в этом каталоге и записывая в них.

Изначально в этом каталоге есть только один файл, ``nr_schemes``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельную схему операций на основе
DAMON.

.. _sysfs_scheme:

schemes/<N>/
------------

В каждом каталоге схемы есть восемь каталогов (``access_pattern``, ``quotas``,
``watermarks``, ``core_filters``, ``ops_filters``, ``filters``, ``dests``,
``stats`` и ``tried_regions``) и три файла (``action``, ``target_nid`` и
``apply_interval``).

Файл ``action`` предназначен для задания и получения :ref:`действия
<damon_design_damos_action>` схемы.  Ключевые слова, которые можно записывать в
этот файл и читать из него, и их значение те же, что и в списке в
:ref:`документе Design <damon_design_damos_action>`.

Файл ``target_nid`` предназначен для задания целевого узла миграции, что имеет
смысл только когда ``action`` равно ``migrate_hot`` или ``migrate_cold``.

Файл ``apply_interval_us`` предназначен для задания и получения
:ref:`apply_interval <damon_design_damos>` схемы в микросекундах.

.. _sysfs_access_pattern:

schemes/<N>/access_pattern/
---------------------------

Каталог для целевого :ref:`паттерна <damon_design_damos_access_pattern>` доступа
данной схемы операций на основе DAMON.

В каталоге ``access_pattern`` есть три каталога (``sz``, ``nr_accesses`` и
``age``), каждый из которых содержит два файла (``min`` и ``max``).  Вы можете
задавать и получать паттерн доступа для данной схемы, записывая в файлы ``min`` и
``max`` в каталогах ``sz``, ``nr_accesses`` и ``age`` и читая из них
соответственно.  Обратите внимание, что ``min`` и ``max`` образуют замкнутый
интервал.

.. _sysfs_quotas:

schemes/<N>/quotas/
-------------------

Каталог для :ref:`квот <damon_design_damos_quotas>` данной схемы операций на
основе DAMON.

В каталоге ``quotas`` есть четыре файла (``ms``, ``bytes``,
``reset_interval_ms``, ``effective_bytes``) и два каталога (``weights`` и
``goals``).

Вы можете задать ``time quota`` в миллисекундах, ``size quota`` в байтах и
``reset interval`` в миллисекундах, записав значения в эти три файла
соответственно.  Тогда DAMON будет стараться использовать не более чем ``time
quota`` миллисекунд для применения ``action`` к регионам памяти, удовлетворяющим
``access_pattern``, и применять действие не более чем к ``bytes`` байтам
регионов памяти в пределах ``reset_interval_ms``.  Установка одновременно ``ms``
и ``bytes`` в ноль отключает ограничения квоты, если только не задана хотя бы
одна :ref:`цель <sysfs_schemes_quota_goals>`.

Временная квота внутренне преобразуется в квоту по размеру.  Из преобразованной
квоты по размеру и заданной пользователем квоты по размеру применяется меньшая.
На основе заданной пользователем :ref:`цели <sysfs_schemes_quota_goals>`
эффективная квота по размеру дополнительно корректируется.  Чтение
``effective_bytes`` возвращает текущую эффективную квоту по размеру.  Этот файл
не обновляется в реальном времени, поэтому пользователям следует попросить
интерфейс sysfs DAMON обновить содержимое этого файла для статистики, записав
специальное ключевое слово ``update_schemes_effective_quotas`` в
соответствующий файл ``kdamonds/<N>/state``.

В каталоге ``weights`` есть три файла (``sz_permil``, ``nr_accesses_permil`` и
``age_permil``).
Вы можете задать :ref:`веса приоритизации
<damon_design_damos_quotas_prioritization>` для размера, частоты доступа и
возраста в единицах на тысячу, записав значения в эти три файла в каталоге
``weights``.

.. _sysfs_schemes_quota_goals:

schemes/<N>/quotas/goals/
-------------------------

Каталог для :ref:`целей автоматической настройки квот
<damon_design_damos_quotas_auto_tuning>` данной схемы операций на основе DAMON.

Изначально в этом каталоге есть только один файл, ``nr_goals``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельную цель и текущее её
достижение.  Из нескольких видов обратной связи используется наилучший.

Каждый каталог цели содержит пять файлов, а именно ``target_metric``,
``target_value``, ``current_value`` ``nid`` и ``path``.  Пользователи могут
задавать и получать пять параметров целей автоматической настройки квот,
указанных в :ref:`документе Design <damon_design_damos_quotas_auto_tuning>`,
записывая в каждый из файлов и читая из них.  Обратите внимание, что для передачи
обратной связи в DAMON пользователям следует дополнительно записать
``commit_schemes_quota_goals`` в файл ``state`` :ref:`каталога kdamond
<sysfs_kdamond>`.

.. _sysfs_watermarks:

schemes/<N>/watermarks/
-----------------------

Каталог для :ref:`пороговых значений <damon_design_damos_watermarks>` данной
схемы операций на основе DAMON.

В каталоге watermarks есть пять файлов (``metric``, ``interval_us``, ``high``,
``mid`` и ``low``) для задания метрики, интервала времени между проверками
метрики и трёх пороговых значений.  Вы можете задавать и получать эти пять
значений, записывая в файлы соответственно.

Ключевые слова и значения тех из них, которые можно записать в файл ``metric``,
приведены ниже.

 - none: Игнорировать пороговые значения
 - free_mem_rate: Доля свободной памяти системы (на тысячу)

``interval`` следует записывать в единицах микросекунд.

.. _sysfs_filters:

schemes/<N>/{core\_,ops\_,}filters/
-----------------------------------

Каталоги для :ref:`фильтров <damon_design_damos_filters>` данной схемы операций
на основе DAMON.

Каталоги ``core_filters`` и ``ops_filters`` предназначены для фильтров,
обрабатываемых соответственно уровнем ядра DAMON и уровнем набора операций.
Каталог ``filters`` можно использовать для установки фильтров независимо от
уровней их обработки.  Фильтры, запрошенные через ``core_filters`` и
``ops_filters``, будут установлены раньше фильтров из ``filters``.  Все три
каталога содержат одни и те же файлы.

Использование каталога ``filters`` может несколько запутать ожидаемые порядки
вычисления заданных фильтров по файлам в каталоге.  Поэтому пользователям
рекомендуется использовать каталоги ``core_filters`` и ``ops_filters``.  Каталог
``filters`` может стать устаревшим в будущем.

Изначально в этом каталоге есть только один файл, ``nr_filters``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельный фильтр.  Фильтры
вычисляются в числовом порядке.

Каждый каталог фильтра содержит девять файлов, а именно ``type``, ``matching``,
``allow``, ``memcg_path``, ``addr_start``, ``addr_end``, ``min``, ``max`` и
``target_idx``.  В файл ``type`` вы можете записать тип фильтра.  Доступные имена
типов, их значение и то, на каком уровне они обрабатываются, см. в
:ref:`документе Design <damon_design_damos_filters>`.

Для типа ``memcg`` вы можете указать интересующую контрольную группу памяти,
записав путь к контрольной группе памяти от точки монтирования cgroups в файл
``memcg_path``.  Для типа ``addr`` вы можете указать начальный и конечный адреса
диапазона (полуоткрытый интервал) в файлы ``addr_start`` и ``addr_end``
соответственно.  Для типа ``hugepage_size`` вы можете указать минимальный и
максимальный размер диапазона (замкнутый интервал) в файлы ``min`` и ``max``
соответственно.  Для типа ``target`` вы можете указать индекс цели в списке
целей мониторинга контекста DAMON в файл ``target_idx``.

Вы можете записать ``Y`` или ``N`` в файл ``matching``, чтобы указать,
предназначен ли фильтр для памяти, которая соответствует ``type``.  Вы можете
записать ``Y`` или ``N`` в файл ``allow``, чтобы указать, следует ли разрешать
применение действия к памяти, которая удовлетворяет ``type`` и ``matching``.

Например, приведённое ниже ограничивает применение действия DAMOS только к
неанонимным страницам всех контрольных групп памяти, кроме
``/having_care_already``.::

    # cd ops_filters/0/
    # echo 2 > nr_filters
    # # disallow anonymous pages
    echo anon > 0/type
    echo Y > 0/matching
    echo N > 0/allow
    # # further filter out all cgroups except one at '/having_care_already'
    echo memcg > 1/type
    echo /having_care_already > 1/memcg_path
    echo Y > 1/matching
    echo N > 1/allow

Подробнее, в том числе о том, как работают несколько фильтров с разными
значениями ``allow``, когда поддерживается каждый из фильтров и в чём различия в
статистике, см. в :ref:`документации Design о фильтрах DAMOS
<damon_design_damos_filters>`.

.. _damon_sysfs_dests:

schemes/<N>/dests/
------------------

Каталог для задания назначений действия данной схемы операций на основе DAMON.
Этот каталог игнорируется, если действие данной схемы не поддерживает несколько
назначений.  Несколько назначений поддерживают только действия
``DAMOS_MIGRATE_{HOT,COLD}``.

Изначально в этом каталоге есть только один файл, ``nr_dests``.  Запись числа
(``N``) в этот файл создаёт указанное число дочерних каталогов с именами от
``0`` до ``N-1``.  Каждый каталог представляет отдельное назначение действия.

Каждый каталог назначения содержит два файла, а именно ``id`` и ``weight``.
Пользователи могут записывать и читать идентификатор назначения в файл ``id``.
Для действий ``DAMOS_MIGRATE_{HOT,COLD}`` в файл ``id`` следует записать
идентификатор узла назначения миграции.  Пользователи могут записывать и читать
вес назначения среди заданных назначений в файл ``weight``.  Вес может быть
произвольным целым числом.  Когда DAMOS применяет действие к каждой сущности
региона памяти, он будет выбирать назначение действия на основе относительных
весов назначений.

.. _sysfs_schemes_stats:

schemes/<N>/stats/
------------------

DAMON ведёт подсчёт статистики для каждой схемы.  Эту статистику можно
использовать для онлайн-анализа или настройки схем.  Подробнее о статистике см.
в :ref:`документе Design <damon_design_damos_stat>`.

Статистику можно получить, прочитав файлы в каталоге ``stats`` (``nr_tried``,
``sz_tried``, ``nr_applied``, ``sz_applied``, ``sz_ops_filter_passed``,
``qt_exceeds``, ``nr_snapshots`` и ``max_nr_snapshots``) соответственно.

По умолчанию эти файлы не обновляются в реальном времени.  Пользователям следует
попросить интерфейс sysfs DAMON периодически обновлять их с помощью
``refresh_ms`` или выполнить однократное обновление, записав специальное
ключевое слово ``update_schemes_stats`` в соответствующий файл
``kdamonds/<N>/state``.  Подробнее см. в :ref:`каталоге kdamond
<sysfs_kdamond>`.

.. _sysfs_schemes_tried_regions:

schemes/<N>/tried_regions/
--------------------------

Изначально в этом каталоге есть один файл, ``total_bytes``.

Когда в соответствующий файл ``kdamonds/<N>/state`` записывается специальное
ключевое слово ``update_schemes_tried_regions``, DAMON обновляет файл
``total_bytes`` так, чтобы при его чтении возвращался общий размер регионов, к
которым было предпринято применение схемы, и создаёт в этом каталоге каталоги с
целочисленными именами, начиная с ``0``.  Каждый каталог содержит файлы,
раскрывающие подробную информацию о каждом из регионов памяти, к которым
действие ``action`` соответствующей схемы пыталось быть применено в этом
каталоге, в течение следующего :ref:`интервала применения
<damon_design_damos>` соответствующей схемы.  Эта информация включает диапазон
адресов, ``nr_accesses`` и ``age`` региона.

Запись ``update_schemes_tried_bytes`` в соответствующий файл
``kdamonds/<N>/state`` обновит только файл ``total_bytes`` и не будет создавать
подкаталоги.

Эти каталоги будут удалены, когда в соответствующий файл
``kdamonds/<N>/state`` будет записано другое специальное ключевое слово,
``clear_schemes_tried_regions``.

Предполагаемое использование этого каталога — исследование поведения схем и
запросоподобное эффективное получение результатов мониторинга доступа к данным.
В частности, для последнего сценария использования пользователи могут задать
``action`` как ``stat`` и задать ``access pattern`` как интересующий их паттерн,
по которому они хотят выполнять запрос.

.. _sysfs_schemes_tried_region:

tried_regions/<N>/
------------------

В каждом каталоге региона вы найдёте пять файлов (``start``, ``end``,
``nr_accesses``, ``age`` и ``sz_filter_passed``).  Чтение этих файлов покажет
свойства региона, к которому пыталось быть применено действие ``action``
соответствующей схемы операций на основе DAMON.

Пример
~~~~~~

Приведённые ниже команды применяют схему, которая говорит: «Если регион памяти
размером в [4KiB, 8KiB] показывает число обращений за интервал агрегации в
[0, 5] для интервала агрегации в [10, 20], выгрузить регион в страничный файл.
Для выгрузки использовать не более 10 мс в секунду, а также не выгружать более
1GiB в секунду.  В рамках этого ограничения сначала выгружать регионы памяти с
большим возрастом.  Кроме того, проверять долю свободной памяти системы каждые 5
секунд, запускать мониторинг и выгрузку, когда доля свободной памяти становится
ниже 50%, но останавливать её, если доля свободной памяти становится больше 60%
или ниже 30%». ::

    # cd <sysfs>/kernel/mm/damon/admin
    # # populate directories
    # echo 1 > kdamonds/nr_kdamonds; echo 1 > kdamonds/0/contexts/nr_contexts;
    # echo 1 > kdamonds/0/contexts/0/schemes/nr_schemes
    # cd kdamonds/0/contexts/0/schemes/0
    # # set the basic access pattern and the action
    # echo 4096 > access_pattern/sz/min
    # echo 8192 > access_pattern/sz/max
    # echo 0 > access_pattern/nr_accesses/min
    # echo 5 > access_pattern/nr_accesses/max
    # echo 10 > access_pattern/age/min
    # echo 20 > access_pattern/age/max
    # echo pageout > action
    # # set quotas
    # echo 10 > quotas/ms
    # echo $((1024*1024*1024)) > quotas/bytes
    # echo 1000 > quotas/reset_interval_ms
    # # set watermark
    # echo free_mem_rate > watermarks/metric
    # echo 5000000 > watermarks/interval_us
    # echo 600 > watermarks/high
    # echo 500 > watermarks/mid
    # echo 300 > watermarks/low

Обратите внимание, что настоятельно рекомендуется использовать инструменты
пространства пользователя, такие как `damo
<https://github.com/damonitor/damo>`_, а не вручную читать и записывать файлы,
как показано выше.  Приведённое выше — только пример.

.. _tracepoint:

Точки трассировки для результатов мониторинга
=============================================

Пользователи могут получать результаты мониторинга через :ref:`tried_regions
<sysfs_schemes_tried_regions>`.  Этот интерфейс полезен для получения снимка, но
он может быть неэффективен для полной записи всех результатов мониторинга.  Для
этой цели предоставляются две точки трассировки, а именно
``damon:damon_aggregated`` и ``damon:damos_before_apply``.
``damon:damon_aggregated`` предоставляет все результаты мониторинга, а
``damon:damos_before_apply`` предоставляет результаты мониторинга для регионов,
к которым будет применена каждая схема операций на основе DAMON (:ref:`DAMOS
<damon_design_damos>`).  Следовательно, ``damon:damos_before_apply`` более
полезна для записи внутреннего поведения DAMOS или для запросоподобной
эффективной записи результатов мониторинга на основе целевого
:ref:`паттерна <damon_design_damos_access_pattern>` доступа DAMOS.

Пока мониторинг включён, вы можете записывать события точек трассировки и
показывать результаты с помощью инструментов, поддерживающих точки трассировки,
таких как ``perf``.  Например::

    # echo on > kdamonds/0/state
    # perf record -e damon:damon_aggregated &
    # sleep 5
    # kill 9 $(pidof perf)
    # echo off > kdamonds/0/state
    # perf script
    kdamond.0 46568 [027] 79357.842179: damon:damon_aggregated: target_id=0 nr_regions=11 122509119488-135708762112: 0 864
    [...]

Каждая строка вывода perf script представляет отдельный регион мониторинга.
Первые пять полей такие же, как и в выводе других точек трассировки.  Шестое поле
(``target_id=X``) показывает идентификатор цели мониторинга региона.  Седьмое
поле (``nr_regions=X``) показывает общее число регионов мониторинга для данной
цели.  Восьмое поле (``X-Y:``) показывает начальный (``X``) и конечный (``Y``)
адреса региона в байтах.  Девятое поле (``X``) показывает ``nr_accesses``
региона (подробнее об этом счётчике см.
:ref:`design <damon_design_region_based_sampling>`).  Наконец, десятое поле
(``X``) показывает ``age`` региона (подробнее об этом счётчике см.
:ref:`design <damon_design_age_tracking>`).

Если событием было ``damon:damos_beofre_apply``, вывод ``perf script`` был бы
примерно таким, как показано ниже::

    kdamond.0 47293 [000] 80801.060214: damon:damos_before_apply: ctx_idx=0 scheme_idx=0 target_idx=0 nr_regions=11 121932607488-135128711168: 0 136
    [...]

Каждая строка вывода представляет отдельный регион мониторинга, к которому каждая
схема операций на основе DAMON собиралась быть применена в трассируемый момент.
Первые пять полей обычные.  Она показывает индекс контекста DAMON (``ctx_idx=X``)
данной схемы в списке контекстов kdamond этого контекста, индекс схемы
(``scheme_idx=X``) в списке схем данного контекста — в дополнение к выводу точки
трассировки ``damon_aggregated``.
