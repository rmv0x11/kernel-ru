=================================================
Поддержка отображения памяти в режиме No-MMU
=================================================

Ядро имеет ограниченную поддержку отображения памяти в условиях отсутствия MMU
(no-MMU), например в окружениях uClinux. С точки зрения пространства пользователя
(userspace) отображение памяти используется совместно с системным вызовом mmap(),
вызовом shmat() и системным вызовом execve(). С точки зрения ядра отображение для
execve() фактически выполняется драйверами binfmt, которые обращаются обратно к
процедурам mmap() для выполнения собственно работы.

Поведение отображения памяти также затрагивает то, как работают fork(), vfork(),
clone() и ptrace(). В uClinux нет fork(), а вызову clone() необходимо передавать
флаг CLONE_VM.

Поведение в случаях с MMU и без MMU (no-MMU) схоже, но не идентично; кроме того, в
последнем случае оно гораздо более ограничено:

 (#) Анонимное отображение, MAP_PRIVATE

	В случае с MMU: VM-области, обеспечиваемые произвольными страницами;
	копирование при записи (copy-on-write) при fork.

	В случае без MMU (no-MMU): VM-области, обеспечиваемые произвольными
	непрерывными последовательностями страниц.

 (#) Анонимное отображение, MAP_SHARED

	Они ведут себя во многом как приватные отображения, за исключением того, что
	в случае с MMU они являются общими при fork() или clone() без CLONE_VM.
	Поскольку случай без MMU (no-MMU) их не поддерживает, поведение там
	идентично MAP_PRIVATE.

 (#) Файл, MAP_PRIVATE, PROT_READ / PROT_EXEC, !PROT_WRITE

	В случае с MMU: VM-области, обеспечиваемые страницами, прочитанными из файла;
	изменения нижележащего файла отражаются в отображении; копируется при fork.

	В случае без MMU (no-MMU):

         - Если оно существует, ядро повторно использует существующее отображение
           на тот же сегмент того же файла, если у него совместимые права доступа,
           даже если оно было создано другим процессом.

         - Если это возможно, файловое отображение будет выполнено напрямую на
           обеспечивающем устройстве, если обеспечивающее устройство обладает
           возможностью NOMMU_MAP_DIRECT и соответствующими возможностями защиты
           отображения. Ramfs, romfs, cramfs и mtd могут все это разрешать.

	 - Если обеспечивающее устройство не может или не будет разрешать прямое
           совместное использование, но обладает возможностью NOMMU_MAP_COPY,
           то копия соответствующей части файла будет прочитана в непрерывную
           область памяти, а любое лишнее пространство за пределами EOF будет
           очищено

	 - Записи в файл не влияют на отображение; записи в отображение видны в
	   других процессах (нет защиты MMU), но их быть не должно.

 (#) Файл, MAP_PRIVATE, PROT_READ / PROT_EXEC, PROT_WRITE

	В случае с MMU: как в случае без PROT_WRITE, за исключением того, что
	соответствующие страницы копируются прежде, чем фактически произойдёт запись.
	С этого момента записи в файл, лежащий под этой страницей, больше не
	отражаются в обеспечивающих страницах отображения. Затем страница
	обеспечивается из области подкачки (swap).

	В случае без MMU (no-MMU): работает во многом как в случае без PROT_WRITE,
	за исключением того, что копия всегда создаётся и никогда не является общей.

 (#) Обычный файл / blockdev, MAP_SHARED, PROT_READ / PROT_EXEC / PROT_WRITE

	В случае с MMU: VM-области, обеспечиваемые страницами, прочитанными из файла;
	изменения страниц записываются обратно в файл; записи в файл отражаются в
	страницах, обеспечивающих отображение; являются общими при fork.

	В случае без MMU (no-MMU): не поддерживается.

 (#) Обычный файл с обеспечением из памяти, MAP_SHARED, PROT_READ / PROT_EXEC / PROT_WRITE

	В случае с MMU: как для обычных файлов.

	В случае без MMU (no-MMU): файловая система, предоставляющая файл с
	обеспечением из памяти (такая как ramfs или tmpfs), может выбрать выполнение
	последовательности open, truncate, mmap, предоставив непрерывную
	последовательность страниц для отображения. В этом случае станет возможным
	совместно используемое отображение памяти с доступом на запись. Оно будет
	работать так же, как в случае с MMU. Если файловая система не предоставляет
	такой поддержки, то запрос на отображение будет отклонён.

 (#) Blockdev с обеспечением из памяти, MAP_SHARED, PROT_READ / PROT_EXEC / PROT_WRITE

	В случае с MMU: как для обычных файлов.

	В случае без MMU (no-MMU): как для обычных файлов с обеспечением из памяти,
	но blockdev должен быть способен предоставить непрерывную последовательность
	страниц без вызова truncate. Драйвер ramdisk мог бы делать это, если бы
	выделял всю свою память как непрерывный массив заранее.

 (#) Chardev с обеспечением из памяти, MAP_SHARED, PROT_READ / PROT_EXEC / PROT_WRITE

	В случае с MMU: как для обычных файлов.

	В случае без MMU (no-MMU): драйвер символьного устройства может выбрать
	выполнение mmap(), предоставив прямой доступ к нижележащему устройству, если
	оно предоставляет память или квазипамять, к которой можно обращаться напрямую.
	Примерами такого являются буферы кадров (frame buffers) и flash-устройства.
	Если драйвер не предоставляет такой поддержки, то запрос на отображение будет
	отклонён.


Дополнительные замечания о MMAP в режиме no-MMU
===============================================

 (#) Запрос на приватное отображение файла может вернуть буфер, который не
     выровнен по границе страницы. Это связано с тем, что может происходить XIP, а
     данные в обеспечивающем хранилище могут быть не выровнены по странице.

 (#) Запрос на анонимное отображение всегда будет выровнен по границе страницы.
     Если это возможно, размер запроса должен быть степенью двойки, иначе часть
     пространства может быть потрачена впустую, так как ядро должно выделить гранулу
     размером в степень двойки, но отбросит излишек только в том случае, если оно
     соответствующим образом настроено, поскольку это влияет на фрагментацию.

 (#) Память, выделенная по запросу на анонимное отображение, обычно очищается ядром
     перед возвратом в соответствии с man-страницами Linux (версии 2.22 или более
     поздней).

     В случае с MMU это может быть достигнуто с приемлемой производительностью, так
     как области обеспечиваются виртуальными страницами, причём содержимое
     отображается на очищенные физические страницы только при возникновении записи
     на конкретную страницу (до этого страницы фактически отображаются на глобальную
     нулевую страницу, из которой можно выполнять чтение). Это распределяет во
     времени затраты на инициализацию содержимого страницы — в зависимости от
     характера использования отображения на запись.

     В случае без MMU (no-MMU), однако, анонимные отображения обеспечиваются
     физическими страницами, и всё отображение очищается во время выделения. Это
     может вызвать значительные задержки при выполнении malloc() в пространстве
     пользователя, так как библиотека C делает анонимное отображение, а ядро затем
     выполняет memset для всего отображения.

     Однако для памяти, которую не требуется предварительно очищать — такой как та,
     что возвращается malloc() — mmap() может принимать флаг MAP_UNINITIALIZED,
     чтобы указать ядру, что ему не следует беспокоиться об очистке памяти перед её
     возвратом. Обратите внимание, что для разрешения этого должна быть включена
     CONFIG_MMAP_ALLOW_UNINITIALIZED, иначе флаг будет проигнорирован.

     uClibc использует это для ускорения malloc(), а binfmt ELF-FDPIC использует это
     для выделения областей brk и стека.

 (#) Список всех приватных копий и анонимных отображений в системе виден через
     /proc/maps в режиме no-MMU.

 (#) Список всех отображений, используемых процессом, виден через
     /proc/<pid>/maps в режиме no-MMU.

 (#) Передача MAP_FIXED или запрос конкретного адреса отображения приведут к ошибке.

 (#) Для файлов, отображаемых приватно, обычно драйвер или файловая система должны
     предоставлять метод read, чтобы содержимое можно было прочитать в выделенную
     память, если mmap() решает не отображать обеспечивающее устройство напрямую.
     Если они этого не делают, возникнет ошибка. Это наиболее вероятно встретить с
     файлами символьных устройств, каналами (pipes), fifo и сокетами.


Межпроцессная разделяемая память
================================

В режиме NOMMU поддерживается как разделяемая память SYSV IPC SHM, так и разделяемая
память POSIX. Первая — через обычный механизм, вторая — через файлы, создаваемые на
точках монтирования ramfs или tmpfs.


Futex'ы
=======

Futex'ы поддерживаются в режиме NOMMU, если их поддерживает архитектура. Будет выдана
ошибка, если адрес, переданный системному вызову futex, находится за пределами
отображений, сделанных процессом, или если отображение, в котором находится адрес, не
поддерживает futex'ы (такое как отображение символьного устройства I/O).


mremap в режиме No-MMU
======================

Функция mremap() поддерживается частично. Она может изменить размер отображения и
может переместить его [#]_, если указан MREMAP_MAYMOVE и если новый размер отображения
превышает размер slab-объекта, в настоящее время занятого памятью, на которую ссылается
отображение, или если можно было бы использовать меньший slab-объект.

MREMAP_FIXED не поддерживается, хотя он игнорируется, если нет изменения адреса и объект
не нужно перемещать.

Разделяемые отображения перемещать нельзя. Допускающие совместное использование
(shareable) отображения тоже перемещать нельзя, даже если в настоящее время они не
являются общими.

Функции mremap() необходимо передать точное соответствие базового адреса и размера
ранее отображённого объекта. Её нельзя использовать для создания дыр в существующих
отображениях, перемещения частей существующих отображений или изменения размера частей
отображений. Она должна действовать на отображение целиком.

.. [#] В настоящее время не поддерживается.


Предоставление поддержки разделяемых символьных устройств
=========================================================

Чтобы предоставить поддержку разделяемых символьных устройств, драйвер должен
предоставить операцию file->f_op->get_unmapped_area(). Процедуры mmap() будут вызывать
её для получения предлагаемого адреса для отображения. Она может вернуть ошибку, если не
желает выполнять отображение, потому что оно слишком длинное, находится по странному
смещению, при некоторой неподдерживаемой комбинации флагов и так далее.

Драйвер также должен предоставить информацию об обеспечивающем устройстве с
установленными возможностями, чтобы указать разрешённые типы отображения на таких
устройствах. По умолчанию предполагается, что устройство доступно для чтения и записи,
не исполняемо и допускает совместное использование только напрямую (не может быть
скопировано).

Операция file->f_op->mmap() будет вызвана для фактического создания отображения. На этом
этапе оно может быть отклонено. Возврат ошибки ENOSYS приведёт к тому, что вместо этого
отображение будет скопировано, если указан NOMMU_MAP_COPY.

Процедура vm_ops->close() будет вызвана при удалении последнего отображения на chardev.
Существующее отображение будет совместно использовано, частично или нет, если это
возможно без уведомления драйвера.

Операции file->f_op->get_unmapped_area() также разрешается возвращать -ENOSYS. Это будет
истолковано как то, что данная операция просто не хочет обрабатывать этот случай,
несмотря на то, что операция имеется. Например, она может попытаться перенаправить вызов
к вторичному драйверу, который, как оказывается, его не реализует. Так обстоит дело с
драйвером framebuffer, который пытается перенаправить вызов к драйверу, специфичному для
устройства. В таких обстоятельствах запрос на отображение будет отклонён, если не указан
NOMMU_MAP_COPY, и отображён через копирование в противном случае.

.. important::

	Некоторые типы устройств могут представлять разный вид тому, кто смотрит на них
	в определённых режимах. Flash-чипы могут быть такими; например, если они
	находятся в режиме программирования или стирания, вы можете увидеть в отображении
	отражённый статус вместо данных.

	В таком случае необходимо проявлять осторожность, чтобы пространство пользователя
	не увидело в разделяемом или приватном отображении такую информацию, когда драйвер
	занят управлением устройством. Помните особенно: приватные исполняемые отображения
	при некоторых обстоятельствах всё ещё могут отображаться напрямую с устройства!


Предоставление поддержки разделяемых файлов с обеспечением из памяти
====================================================================

Предоставление разделяемых отображений на файлах с обеспечением из памяти аналогично
предоставлению поддержки разделяемых отображаемых символьных устройств. Главное отличие
в том, что файловая система, предоставляющая этот сервис, вероятно, выделит непрерывный
набор страниц и разрешит выполнять отображения на нём.

Рекомендуется, чтобы операция truncate, применённая к такому файлу и увеличивающая его
размер, если этот файл пуст, воспринималась как запрос на сбор достаточного количества
страниц для выполнения отображения. Это требуется для поддержки разделяемой памяти POSIX.

Устройства с обеспечением из памяти обозначаются установленным флагом memory_backed в
информации об обеспечивающем устройстве отображения.


Предоставление поддержки разделяемых блочных устройств
======================================================

Предоставление разделяемых отображений на файлах блочных устройств в точности такое же,
как для символьных устройств. Если под ними нет реального устройства, то драйвер должен
выделить достаточно непрерывной памяти, чтобы выполнить любое поддерживаемое отображение.


Настройка поведения усечения страниц
====================================

NOMMU mmap при выполнении выделения автоматически округляет вверх до ближайшего числа
страниц, являющегося степенью двойки. Это может неблагоприятно сказаться на фрагментации
памяти, и поэтому оставлено настраиваемым. Поведение по умолчанию — агрессивно усекать
выделения и отбрасывать любые излишние страницы обратно в распределитель страниц. Чтобы
сохранить более тонкий контроль над фрагментацией, это поведение можно либо полностью
отключить, либо поднять до более высокого порога числа страниц, при котором начинается
усечение.

Поведение усечения страниц настраивается через sysctl ``vm.nr_trim_pages``.
