=======================
Kernel Samepage Merging
=======================

Обзор
=====

KSM — это функциональность дедупликации для экономии памяти, включаемая параметром
CONFIG_KSM=y и добавленная в ядро Linux в версии 2.6.32.  Её реализацию см. в ``mm/ksm.c``,
а также http://lwn.net/Articles/306704/ и https://lwn.net/Articles/330589/

Изначально KSM разрабатывалась для использования с KVM (где она была известна как
Kernel Shared Memory), чтобы разместить больше виртуальных машин в физической памяти
за счёт совместного использования общих для них данных.  Но она может быть полезна
любому приложению, которое создаёт много экземпляров одних и тех же данных.

Демон KSM ksmd периодически сканирует те области памяти пользователя,
которые были в нём зарегистрированы, отыскивая страницы с идентичным
содержимым, которые можно заменить одной защищённой от записи страницей
(автоматически копируемой, если впоследствии процесс захочет обновить её
содержимое). Количество страниц, которое демон KSM сканирует за один проход,
и время между проходами настраиваются через :ref:`интерфейс sysfs
<ksm_sysfs>`

KSM объединяет только анонимные (приватные) страницы и никогда — страницы pagecache (файловые).
Объединённые KSM страницы изначально блокировались в памяти ядра, но теперь могут
выгружаться в swap так же, как и другие страницы пользователя (но при их выгрузке совместное
использование нарушается: ksmd должен заново обнаружить их идентичность и объединить снова).

Управление KSM с помощью madvise
=================================

KSM работает только с теми областями адресного пространства, которые приложение
указало как вероятных кандидатов для объединения, используя системный вызов
madvise(2)::

	int madvise(addr, length, MADV_MERGEABLE)

Приложение может вызвать

::

	int madvise(addr, length, MADV_UNMERGEABLE)

чтобы отменить это указание и восстановить необъединённые страницы: после чего KSM
разъединяет всё, что он объединил в этом диапазоне.  Примечание: этот вызов разъединения
может внезапно потребовать больше памяти, чем доступно — возможно, завершившись неудачей
с EAGAIN, но более вероятно — пробудив Out-Of-Memory killer.

Если KSM не сконфигурирована в работающем ядре, madvise MADV_MERGEABLE
и MADV_UNMERGEABLE просто завершаются неудачей с EINVAL.  Если работающее ядро было
собрано с CONFIG_KSM=y, эти вызовы обычно завершаются успешно: даже если
демон KSM в данный момент не запущен, MADV_MERGEABLE всё равно регистрирует
диапазон на тот случай, когда демон KSM будет запущен; даже если диапазон
не может содержать ни одной страницы, которую KSM мог бы реально объединить; даже если
MADV_UNMERGEABLE применяется к диапазону, который никогда не был MADV_MERGEABLE.

Если область памяти должна быть разбита хотя бы на одну новую область MADV_MERGEABLE
или MADV_UNMERGEABLE, madvise может вернуть ENOMEM, если процесс
превысит ``vm.max_map_count`` (см. Documentation/admin-guide/sysctl/vm.rst).

Как и другие вызовы madvise, они предназначены для использования на отображённых областях
адресного пространства пользователя: они сообщат ENOMEM, если указанный диапазон
включает неотображённые промежутки (при этом обрабатывая промежуточные отображённые области),
и могут завершиться неудачей с EAGAIN, если памяти для внутренних структур недостаточно.

Приложениям следует разумно использовать MADV_MERGEABLE,
ограничивая его применение областями, которые, вероятно, выиграют от этого.  Сканирования KSM
могут потреблять много вычислительной мощности: некоторые инсталляции по этой причине отключают KSM.

Интерфейс sysfs демона KSM
==========================

Демон KSM управляется файлами sysfs в ``/sys/kernel/mm/ksm/``,
доступными для чтения всем, но для записи — только root:

pages_to_scan
        сколько страниц сканировать, прежде чем ksmd уйдёт в сон
        напр. ``echo 100 > /sys/kernel/mm/ksm/pages_to_scan``.

        Значение pages_to_scan нельзя изменить, если ``advisor_mode``
        установлен в scan-time.

        По умолчанию: 100 (выбрано в демонстрационных целях)

sleep_millisecs
        сколько миллисекунд ksmd должен спать перед следующим сканированием
        напр. ``echo 20 > /sys/kernel/mm/ksm/sleep_millisecs``

        По умолчанию: 20 (выбрано в демонстрационных целях)

merge_across_nodes
        указывает, можно ли объединять страницы из разных узлов NUMA.
        Когда установлено в 0, ksm объединяет только страницы, которые физически
        находятся в области памяти одного и того же узла NUMA. Это обеспечивает меньшую
        задержку (latency) доступа к разделяемым страницам. Системы с большим числом узлов,
        при значительных расстояниях NUMA, скорее всего выиграют от меньшей
        задержки при значении 0. Меньшие системы, которым нужно
        минимизировать использование памяти, скорее всего выиграют от большего
        совместного использования при значении 1 (по умолчанию). Возможно, вы захотите сравнить, как
        ваша система работает при каждом из значений, прежде чем решить,
        какое использовать. Настройку ``merge_across_nodes`` можно изменить только
        тогда, когда в системе нет разделяемых ksm страниц: установите run в 2, чтобы
        сначала разъединить страницы, затем в 1 после изменения
        ``merge_across_nodes``, чтобы заново объединить согласно новой настройке.

        По умолчанию: 1 (объединение между узлами, как в более ранних релизах)

run
        * установите в 0, чтобы остановить работу ksmd, но сохранить объединённые страницы,
        * установите в 1, чтобы запустить ksmd, напр. ``echo 1 > /sys/kernel/mm/ksm/run``,
        * установите в 2, чтобы остановить ksmd и разъединить все объединённые в данный момент страницы, но
	  оставить объединяемые области зарегистрированными для следующего запуска.

        По умолчанию: 0 (должно быть изменено на 1 для активации KSM, кроме случая,
        когда CONFIG_SYSFS отключён)

use_zero_pages
        указывает, следует ли обрабатывать пустые страницы (т.е. выделенные страницы,
        содержащие только нули) особым образом.  Когда установлено в 1,
        пустые страницы объединяются с нулевой страницей (страницами) ядра вместо
        объединения друг с другом, как это происходило бы обычно. Это может повысить
        производительность на архитектурах с раскрашенными (coloured) нулевыми страницами,
        в зависимости от рабочей нагрузки. Следует проявлять осторожность при включении
        этой настройки, так как она потенциально может ухудшить производительность
        KSM для некоторых рабочих нагрузок, например, если контрольные суммы страниц —
        кандидатов на объединение совпадают с контрольной суммой пустой
        страницы. Эту настройку можно изменить в любой момент, она действует только
        для страниц, объединённых после изменения.

        По умолчанию: 0 (обычное поведение KSM, как в более ранних релизах)

max_page_sharing
        Максимальное совместное использование, разрешённое для каждой страницы KSM. Это устанавливает
        предел дедупликации, чтобы избежать высокой задержки для операций виртуальной памяти,
        которые включают обход виртуальных отображений, которые
        разделяют страницу KSM. Минимальное значение — 2, так как вновь созданная
        страница KSM будет иметь как минимум двух участников совместного использования. Чем выше это значение,
        тем быстрее KSM будет объединять память и тем выше будет
        коэффициент дедупликации, но тем медленнее в худшем случае мог бы быть
        обход виртуальных отображений для любой заданной страницы
        KSM. Замедление этого обхода означает, что будет более высокая
        задержка для определённых операций виртуальной памяти, происходящих во время
        выгрузки в swap, уплотнения (compaction), балансировки NUMA и миграции страниц, что, в свою
        очередь, снижает отзывчивость для вызывающего эти операции виртуальной
        памяти. Задержка планировщика для других задач, не вовлечённых
        в операции виртуальной памяти, выполняющие обход виртуальных отображений,
        не зависит от этого параметра, так как эти
        обходы сами по себе всегда дружественны к планировщику.

stable_node_chains_prune_millisecs
        указывает, как часто KSM проверяет метаданные страниц,
        достигших предела дедупликации, на устаревшую информацию.
        Меньшие значения в миллисекундах будут освобождать метаданные KSM с
        меньшей задержкой, но при этом заставят ksmd использовать больше CPU во время
        сканирования. Это пустая операция (noop), если ни одна страница KSM ещё не
        достигла ``max_page_sharing``.

smart_scan
        Исторически KSM проверял каждую страницу-кандидата при каждом сканировании. Он не
        учитывал историческую информацию.  Когда smart scan включён,
        страницы, которые ранее не были дедуплицированы, пропускаются. Как часто
        эти страницы пропускаются, зависит от того, как часто
        дедупликация уже пробовалась и завершалась неудачей. По умолчанию эта
        оптимизация включена.  Метрика ``pages_skipped`` показывает, насколько
        эффективна эта настройка.

advisor_mode
        ``advisor_mode`` выбирает текущий advisor. Поддерживаются два режима:
        none и scan-time. По умолчанию — none. Установив
        ``advisor_mode`` в scan-time, вы включаете advisor по времени сканирования (scan time advisor).
        Раздел про ``advisor`` подробно объясняет, как работает advisor
        по времени сканирования.

adivsor_max_cpu
        указывает верхний предел использования CPU в процентах для фонового
        потока ksmd. По умолчанию — 70.

advisor_target_scan_time
        указывает целевое время сканирования в секундах для сканирования всех страниц-кандидатов.
        Значение по умолчанию — 200 секунд.

advisor_min_pages_to_scan
        указывает нижний предел параметра ``pages_to_scan`` для
        advisor по времени сканирования. По умолчанию — 500.

adivsor_max_pages_to_scan
        указывает верхний предел параметра ``pages_to_scan`` для
        advisor по времени сканирования. По умолчанию — 30000.

Эффективность KSM и MADV_MERGEABLE отображается в ``/sys/kernel/mm/ksm/``:

general_profit
        насколько эффективен KSM. Расчёт объясняется ниже.
pages_scanned
        сколько страниц сканируется для ksm
pages_shared
        сколько разделяемых страниц используется
pages_sharing
        сколько ещё мест разделяют их, т.е. сколько сэкономлено
pages_unshared
        сколько страниц уникальны, но многократно проверяются для объединения
pages_volatile
        сколько страниц меняются слишком быстро, чтобы их можно было поместить в дерево
pages_skipped
        сколько страниц пропустил «умный» алгоритм сканирования страниц
full_scans
        сколько раз все объединяемые области были просканированы
stable_node_chains
        число страниц KSM, достигших предела ``max_page_sharing``
stable_node_dups
        число продублированных страниц KSM
ksm_zero_pages
        сколько нулевых страниц, всё ещё отображённых в процессы, были отображены
        KSM при дедупликации.

Когда ``use_zero_pages`` включён/был включён, сумма ``pages_sharing`` +
``ksm_zero_pages`` представляет фактическое число страниц, сэкономленных KSM.
если ``use_zero_pages`` никогда не включался, ``ksm_zero_pages`` равен 0.

Высокое отношение ``pages_sharing`` к ``pages_shared`` указывает на хорошее
совместное использование, но высокое отношение ``pages_unshared`` к ``pages_sharing``
указывает на напрасные усилия.  ``pages_volatile`` охватывает несколько
различных видов активности, но высокая их доля также
указывала бы на плохое использование madvise MADV_MERGEABLE.

Максимально возможное отношение ``pages_sharing/pages_shared`` ограничено
настройкой ``max_page_sharing``. Чтобы увеличить это отношение, нужно соответственно
увеличить ``max_page_sharing``.

Мониторинг выгоды от KSM
========================

KSM может экономить память, объединяя идентичные страницы, но также может потреблять
дополнительную память, поскольку ему нужно создавать некоторое количество rmap_items для
сохранения краткой информации rmap каждой просканированной страницы. Некоторые из этих страниц могут
быть объединены, но некоторые могут оказаться необъединяемыми после нескольких
проверок, что является невыгодно потреблённой памятью.

1) Как определить, экономит ли KSM память или потребляет память в масштабе всей
   системы? Вот простой приблизительный расчёт для справки::

	general_profit =~ ksm_saved_pages * sizeof(page) - (all_rmap_items) *
			  sizeof(rmap_item);

   где ksm_saved_pages равно сумме ``pages_sharing`` +
   ``ksm_zero_pages`` системы, а all_rmap_items можно легко
   получить, сложив ``pages_sharing``, ``pages_shared``, ``pages_unshared``
   и ``pages_volatile``.

2) Выгоду от KSM внутри одного процесса можно аналогично получить с помощью
   следующего приблизительного расчёта::

	process_profit =~ ksm_saved_pages * sizeof(page) -
			  ksm_rmap_items * sizeof(rmap_item).

   где ksm_saved_pages равно сумме ``ksm_merging_pages`` и
   ``ksm_zero_pages``, оба из которых показаны в каталоге
   ``/proc/<pid>/ksm_stat``, а ksm_rmap_items также показан в
   ``/proc/<pid>/ksm_stat``. Выгода от процесса также показана в
   ``/proc/<pid>/ksm_stat`` как ksm_process_profit.

С точки зрения приложения, высокое отношение ``ksm_rmap_items`` к
``ksm_merging_pages`` означает плохую политику применения madvise, поэтому разработчикам или
администраторам приходится переосмысливать, как изменить политику madvise. Приведём пример
для справки: размер страницы обычно составляет 4K, а размер rmap_item составляет
соответственно 32B на 32-битной архитектуре CPU и 64B на 64-битной архитектуре CPU.
так что если отношение ``ksm_rmap_items/ksm_merging_pages`` превышает 64 на 64-битном CPU
или превышает 128 на 32-битном CPU, то политику madvise приложения следует отбросить,
поскольку выгода от ksm приблизительно нулевая или отрицательная.

Мониторинг событий KSM
======================

В /proc/vmstat есть некоторые счётчики, которые можно использовать для мониторинга событий KSM.
KSM может помочь сэкономить память, это компромисс ценой возможной задержки при KSM COW
или при копировании во время выгрузки из swap. Эти события могут помочь пользователям оценить, стоит ли и как
использовать KSM. Например, если cow_ksm растёт слишком быстро, пользователь может уменьшить
диапазон madvise(, , MADV_MERGEABLE).

cow_ksm
	инкрементируется каждый раз, когда страница KSM запускает copy on write (COW)
	когда пользователи пытаются записать в страницу KSM, нам приходится делать копию.

ksm_swpin_copy
	инкрементируется каждый раз, когда страница KSM копируется при выгрузке из swap
	обратите внимание, что страница KSM может копироваться при выгрузке из swap, потому что do_swap_page()
	не может выполнить всю блокировку, необходимую для восстановления KSM-страницы, охватывающей несколько anon_vma.

Advisor
=======

Число страниц-кандидатов для KSM динамично. Часто можно наблюдать,
что во время запуска приложения нужно обработать больше страниц-кандидатов.
Без advisor параметр ``pages_to_scan`` нужно
рассчитывать на максимальное число страниц-кандидатов. Advisor по времени сканирования может
изменять параметр ``pages_to_scan`` в зависимости от потребности.

Advisor можно включить, чтобы KSM мог автоматически адаптироваться к изменениям
числа сканируемых страниц-кандидатов. Реализованы два advisor: none и
scan-time. При none ни один advisor не включён. По умолчанию — none.

Advisor по времени сканирования изменяет параметр ``pages_to_scan`` на основе
наблюдаемого времени сканирования. Возможные значения для параметра ``pages_to_scan``
ограничены параметром ``advisor_max_cpu``. Кроме того, существует ещё и
параметр ``advisor_target_scan_time``. Этот параметр задаёт целевое время для
сканирования всех страниц-кандидатов KSM. Параметр ``advisor_target_scan_time``
определяет, насколько агрессивно advisor по времени сканирования сканирует страницы-кандидаты. Меньшие
значения заставляют advisor по времени сканирования сканировать более агрессивно. Это самый
важный параметр для настройки advisor по времени сканирования.

Начальное значение и максимальное значение можно изменить с помощью
``advisor_min_pages_to_scan`` и ``advisor_max_pages_to_scan``. Значения по умолчанию
достаточны для большинства рабочих нагрузок и сценариев использования.

Параметр ``pages_to_scan`` пересчитывается после завершения сканирования.
