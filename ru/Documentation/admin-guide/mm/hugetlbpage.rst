=============
HugeTLB Pages
=============

Обзор
=====

Назначение этого файла — дать краткую сводку о поддержке hugetlbpage в ядре
Linux.  Эта поддержка построена поверх поддержки нескольких размеров страниц,
которая предоставляется большинством современных архитектур.  Например, CPU
архитектуры x86 обычно поддерживают размеры страниц 4K и 2M (а также 1G, если
это поддерживается архитектурно), архитектура ia64 поддерживает несколько
размеров страниц: 4K, 8K, 64K, 256K, 1M, 4M, 16M, 256M, а ppc64 поддерживает
4K и 16M.  TLB — это кэш трансляций «виртуальный-в-физический».  Как правило,
это весьма дефицитный ресурс процессора.  Операционные системы стараются
наилучшим образом использовать ограниченное количество ресурсов TLB.  Эта
оптимизация сейчас тем более важна, что всё более и более крупные объёмы
физической памяти (несколько ГБ) становятся всё доступнее.

Пользователи могут задействовать поддержку огромных страниц (huge page) в ядре
Linux либо с помощью системного вызова mmap, либо с помощью стандартных
системных вызовов разделяемой памяти SYSV (shmget, shmat).

Сначала ядро Linux должно быть собрано с параметрами конфигурации
CONFIG_HUGETLBFS (находится в разделе "File systems") и CONFIG_HUGETLB_PAGE
(выбирается автоматически при выборе CONFIG_HUGETLBFS).

Файл ``/proc/meminfo`` предоставляет информацию об общем количестве постоянных
страниц hugetlb в пуле огромных страниц ядра.  Он также отображает размер
огромной страницы по умолчанию и информацию о количестве свободных,
зарезервированных и избыточных огромных страниц в пуле огромных страниц размера
по умолчанию.  Размер огромной страницы нужен для формирования корректного
выравнивания и размера аргументов системных вызовов, которые отображают области
огромных страниц.

Вывод команды ``cat /proc/meminfo`` будет включать строки вида::

	HugePages_Total: uuu
	HugePages_Free:  vvv
	HugePages_Rsvd:  www
	HugePages_Surp:  xxx
	Hugepagesize:    yyy kB
	Hugetlb:         zzz kB

где:

HugePages_Total
	— это размер пула огромных страниц.
HugePages_Free
	— это количество огромных страниц в пуле, которые ещё не
        выделены.
HugePages_Rsvd
	— это сокращение от "reserved" (зарезервированные), и это количество
        огромных страниц, для которых было дано обязательство выделить их из
        пула, но выделение ещё не было произведено.  Зарезервированные огромные
        страницы гарантируют, что приложение сможет выделить огромную страницу
        из пула огромных страниц во время отказа (fault).
HugePages_Surp
	— это сокращение от "surplus" (избыточные), и это количество огромных
        страниц в пуле сверх значения в ``/proc/sys/vm/nr_hugepages``.
        Максимальное количество избыточных огромных страниц управляется
        параметром ``/proc/sys/vm/nr_overcommit_hugepages``.
	Примечание: когда включена функция освобождения неиспользуемых страниц
	vmemmap, связанных с каждой страницей hugetlb, количество избыточных
	огромных страниц может временно превышать максимальное количество
	избыточных огромных страниц, когда система испытывает нехватку памяти.
Hugepagesize
	— это размер огромной страницы по умолчанию (в кБ).
Hugetlb
        — это общий объём памяти (в кБ), потребляемый огромными страницами всех
        размеров.
        Если используются огромные страницы разных размеров, это число будет
        превышать HugePages_Total \* Hugepagesize. Для получения более подробной
        информации обратитесь, пожалуйста, к ``/sys/kernel/mm/hugepages``
        (описано ниже).


``/proc/filesystems`` также должен показывать настроенную в ядре файловую
систему типа "hugetlbfs".

``/proc/sys/vm/nr_hugepages`` указывает текущее количество "постоянных"
огромных страниц в пуле огромных страниц ядра.  "Постоянные" огромные страницы
возвращаются в пул огромных страниц при их освобождении задачей.  Пользователь
с привилегиями root может динамически выделять дополнительные постоянные
огромные страницы или освобождать некоторые из них, увеличивая или уменьшая
значение ``nr_hugepages``.

Примечание: когда включена функция освобождения неиспользуемых страниц vmemmap,
связанных с каждой страницей hugetlb, освобождение огромных страниц,
инициированное пользователем, может завершиться неудачей, когда система
испытывает нехватку памяти.  Попробуйте, пожалуйста, ещё раз позже.

Страницы, используемые как огромные страницы, зарезервированы внутри ядра и не
могут использоваться для других целей.  Огромные страницы не могут быть
вытеснены в swap при нехватке памяти.

После того как некоторое количество огромных страниц предварительно выделено в
пул огромных страниц ядра, пользователь с соответствующими привилегиями может
использовать либо системный вызов mmap, либо системные вызовы разделяемой
памяти для использования огромных страниц.  См. обсуждение в разделе
:ref:`Использование огромных страниц <using_huge_pages>` ниже.

Администратор может выделить постоянные огромные страницы в командной строке
загрузки ядра, указав параметр "hugepages=N", где 'N' = запрашиваемое количество
огромных страниц.  Это наиболее надёжный метод выделения огромных страниц, так
как память ещё не успела фрагментироваться.

Некоторые платформы поддерживают несколько размеров огромных страниц.  Чтобы
выделить огромные страницы определённого размера, перед параметрами командной
строки загрузки для огромных страниц необходимо указать параметр выбора размера
огромной страницы "hugepagesz=<size>".  <size> должен быть указан в байтах с
необязательным масштабным суффиксом [kKmMgG].  Размер огромной страницы по
умолчанию может быть выбран с помощью параметра загрузки
"default_hugepagesz=<size>".

Семантика параметров командной строки загрузки hugetlb

hugepagesz
	Задаёт размер огромной страницы.  Используется совместно с параметром
	hugepages для предварительного выделения некоторого количества огромных
	страниц указанного размера.  Поэтому hugepagesz и hugepages обычно
	указываются парами, например::

		hugepagesz=2M hugepages=512

	hugepagesz может быть указан в командной строке только один раз для
	конкретного размера огромной страницы.  Допустимые размеры огромных
	страниц зависят от архитектуры.
hugepages
	Задаёт количество огромных страниц для предварительного выделения.  Этот
	параметр обычно следует за корректным параметром hugepagesz или
	default_hugepagesz.  Однако если hugepages является первым или единственным
	параметром командной строки hugetlb, он неявно задаёт количество огромных
	страниц размера по умолчанию для выделения.  Если количество огромных
	страниц размера по умолчанию задано неявно, оно не может быть
	перезаписано парой параметров hugepagesz,hugepages для размера по
	умолчанию.  Этот параметр также имеет узловой формат (node format).
	Узловой формат задаёт количество огромных страниц для выделения на
	конкретных узлах.

	Например, на архитектуре с размером огромной страницы по умолчанию 2M::

		hugepages=256 hugepagesz=2M hugepages=512

	приведёт к выделению 256 огромных страниц по 2M и к выводу
	предупреждающего сообщения о том, что параметр hugepages=512 игнорируется.
	Если параметру hugepages предшествует некорректный параметр hugepagesz, он
	будет проигнорирован.

	Пример узлового формата::

		hugepagesz=2M hugepages=0:1,1:2

	Это выделит 1 огромную страницу 2M на node0 и 2 огромные страницы 2M на
	node1.  Если номер узла некорректен, параметр будет проигнорирован.
hugepage_alloc_threads
	Задаёт количество потоков, которые должны использоваться для выделения
	огромных страниц во время загрузки. Этот параметр может использоваться для
	сокращения времени загрузки системы при выделении большого количества
	огромных страниц.

	Значение по умолчанию — 25% от доступных аппаратных потоков.
	Пример использования 8 потоков выделения::

		hugepage_alloc_threads=8

	Обратите внимание, что этот параметр применяется только к негигантским
	огромным страницам.
default_hugepagesz
	Задаёт размер огромной страницы по умолчанию.  Этот параметр может быть
	указан в командной строке только один раз.  За default_hugepagesz
	опционально может следовать параметр hugepages для предварительного
	выделения определённого количества огромных страниц размера по умолчанию.
	Количество огромных страниц размера по умолчанию для предварительного
	выделения также может быть задано неявно, как упомянуто в разделе
	hugepages выше.  Поэтому на архитектуре с размером огромной страницы по
	умолчанию 2M::

		hugepages=256
		default_hugepagesz=2M hugepages=256
		hugepages=256 default_hugepagesz=2M

	все приведут к выделению 256 огромных страниц по 2M.  Допустимый размер
	огромной страницы по умолчанию зависит от архитектуры.
hugetlb_free_vmemmap
	Когда задан CONFIG_HUGETLB_PAGE_OPTIMIZE_VMEMMAP, это включает оптимизацию
	HugeTLB Vmemmap Optimization (HVO).

Когда поддерживается несколько размеров огромных страниц,
``/proc/sys/vm/nr_hugepages`` указывает текущее количество предварительно
выделенных огромных страниц размера по умолчанию.  Таким образом, можно
использовать следующую команду для динамического выделения/освобождения
постоянных огромных страниц размера по умолчанию::

	echo 20 > /proc/sys/vm/nr_hugepages

Эта команда попытается скорректировать количество огромных страниц размера по
умолчанию в пуле огромных страниц до 20, выделяя или освобождая огромные
страницы по мере необходимости.

На платформе NUMA ядро попытается распределить пул огромных страниц по всему
набору разрешённых узлов, заданному политикой памяти NUMA задачи, которая
изменяет ``nr_hugepages``. По умолчанию набор разрешённых узлов — когда задача
имеет политику памяти по умолчанию — это все онлайн-узлы с памятью.  Разрешённые
узлы с недостаточным объёмом доступной непрерывной памяти для огромной страницы
будут молча пропускаться при выделении постоянных огромных страниц.  См. ниже
:ref:`обсуждение <mem_policy_and_hp_alloc>`
взаимодействия политики памяти задачи, cpusets и атрибутов отдельных узлов с
выделением и освобождением постоянных огромных страниц.

Успех или неудача выделения огромной страницы зависит от объёма физически
непрерывной памяти, присутствующей в системе на момент попытки выделения.  Если
ядро не может выделить огромные страницы на некоторых узлах системы NUMA, оно
попытается компенсировать разницу, выделив дополнительные страницы на других
узлах с достаточным объёмом доступной непрерывной памяти, если таковые имеются.

Системные администраторы могут захотеть поместить эту команду в один из локальных
init-файлов rc.  Это позволит ядру выделять огромные страницы на ранней стадии
процесса загрузки, когда вероятность получить физически непрерывные страницы
ещё очень высока.  Администраторы могут проверить фактически выделенное
количество огромных страниц, обратившись к sysctl или meminfo.  Чтобы проверить
распределение огромных страниц по узлам в системе NUMA, используйте::

	cat /sys/devices/system/node/node*/meminfo | fgrep Huge

``/proc/sys/vm/nr_overcommit_hugepages`` задаёт, насколько может вырасти пул
огромных страниц, если приложения запрашивают больше огромных страниц, чем
``/proc/sys/vm/nr_hugepages``.  Запись любого ненулевого значения в этот файл
означает, что подсистеме hugetlb разрешено попытаться получить это количество
"избыточных" (surplus) огромных страниц из обычного пула страниц ядра, когда пул
постоянных огромных страниц исчерпан. Когда эти избыточные огромные страницы
становятся неиспользуемыми, они освобождаются обратно в обычный пул страниц
ядра.

При увеличении размера пула огромных страниц через ``nr_hugepages`` любые
существующие избыточные страницы сначала будут повышены до постоянных огромных
страниц.  Затем, при необходимости и при возможности, будут выделены
дополнительные огромные страницы, чтобы достичь нового размера пула постоянных
огромных страниц.

Администратор может уменьшить пул постоянных огромных страниц размера по
умолчанию, установив sysctl ``nr_hugepages`` в меньшее значение.  Ядро
попытается сбалансировать освобождение огромных страниц по всем узлам в политике
памяти задачи, изменяющей ``nr_hugepages``.  Любые свободные огромные страницы на
выбранных узлах будут освобождены обратно в обычный пул страниц ядра.

Предостережение: уменьшение пула постоянных огромных страниц через
``nr_hugepages`` так, что он становится меньше числа используемых огромных
страниц, преобразует остаток используемых огромных страниц в избыточные огромные
страницы.  Это произойдёт, даже если количество избыточных страниц превысит
значение overcommit.  До тех пор, пока это условие сохраняется — то есть пока
``nr_hugepages+nr_overcommit_hugepages`` не будет увеличено в достаточной мере
или избыточные огромные страницы не выйдут из употребления и не будут
освобождены — выделение новых избыточных огромных страниц не будет разрешено.

С появлением поддержки нескольких пулов огромных страниц во время выполнения
значительная часть пользовательского интерфейса огромных страниц в
``/proc/sys/vm`` была продублирована в sysfs.
``/proc``-интерфейсы, обсуждавшиеся выше, были сохранены для обратной
совместимости. Корневой управляющий каталог огромных страниц в sysfs — это::

	/sys/kernel/mm/hugepages

Для каждого размера огромной страницы, поддерживаемого работающим ядром, будет
существовать подкаталог вида::

	hugepages-${size}kB

Внутри каждого из этих каталогов будет существовать набор файлов, содержащихся в
``/proc``.  Кроме того, могут существовать два дополнительных интерфейса для
понижения (demote) огромных страниц::

        demote
        demote_size
	nr_hugepages
	nr_hugepages_mempolicy
	nr_overcommit_hugepages
	free_hugepages
	resv_hugepages
	surplus_hugepages

Интерфейсы demote предоставляют возможность разбить огромную страницу на огромные
страницы меньшего размера.  Например, архитектура x86 поддерживает размеры
огромных страниц как 1GB, так и 2MB.  Огромную страницу 1GB можно разбить на 512
огромных страниц по 2MB.  Интерфейсы demote недоступны для наименьшего размера
огромной страницы.  Интерфейсы demote таковы:

demote_size
        — это размер понижаемых страниц.  Когда страница понижается, создаётся
        соответствующее количество огромных страниц размера demote_size.  По
        умолчанию demote_size установлен в следующий меньший размер огромной
        страницы.  Если существует несколько меньших размеров огромных страниц,
        demote_size может быть установлен в любой из этих меньших размеров.
        Допускаются только размеры огромных страниц, меньшие текущего размера
        огромной страницы.

demote
        — используется для понижения некоторого количества огромных страниц.
        Пользователь с привилегиями root может выполнять запись в этот файл.
        Может оказаться невозможным понизить запрошенное количество огромных
        страниц.  Чтобы определить, сколько страниц было фактически понижено,
        сравните значение nr_hugepages до и после записи в интерфейс demote.
        demote — это интерфейс только для записи.

Интерфейсы, которые совпадают с интерфейсами в ``/proc`` (все, кроме demote и
demote_size), функционируют так, как описано выше для случая огромной страницы
размера по умолчанию.

.. _mem_policy_and_hp_alloc:

Взаимодействие политики памяти задачи с выделением/освобождением огромных страниц
=================================================================================

Независимо от того, выделяются и освобождаются ли огромные страницы через
интерфейс ``/proc`` или через интерфейс ``/sysfs`` с использованием атрибута
``nr_hugepages_mempolicy``, узлы NUMA, из которых выделяются или освобождаются
огромные страницы, управляются политикой памяти NUMA задачи, которая изменяет
sysctl или атрибут ``nr_hugepages_mempolicy``.  Когда используется атрибут
``nr_hugepages``, mempolicy игнорируется.

Рекомендуемый метод выделения или освобождения огромных страниц в/из пула
огромных страниц ядра, на примере ``nr_hugepages`` выше, таков::

    numactl --interleave <node-list> echo 20 \
				>/proc/sys/vm/nr_hugepages_mempolicy

или, более кратко::

    numactl -m <node-list> echo 20 >/proc/sys/vm/nr_hugepages_mempolicy

Это выделит или освободит ``abs(20 - nr_hugepages)`` страниц на узлах, указанных
в <node-list>, или с них, в зависимости от того, меньше или больше 20 было
изначально количество постоянных огромных страниц соответственно.  Никакие
огромные страницы не будут выделены или освобождены ни на одном узле, не
входящем в указанный <node-list>.

При корректировке количества постоянных огромных страниц через
``nr_hugepages_mempolicy`` может использоваться любой режим политики памяти —
bind, preferred, local или interleave.  Результирующий эффект на выделение
постоянных огромных страниц таков:

#. Независимо от режима mempolicy [см.
   Documentation/admin-guide/mm/numa_memory_policy.rst]
   постоянные огромные страницы будут распределены по узлу или узлам, указанным
   в mempolicy, как если бы был указан режим "interleave".
   Однако если узел в политике не содержит достаточно непрерывной памяти для
   огромной страницы, выделение не будет «откатываться» (fallback) к ближайшему
   соседнему узлу с достаточным объёмом непрерывной памяти.  Это привело бы к
   нежелательному дисбалансу в распределении пула огромных страниц или, возможно,
   к выделению постоянных огромных страниц на узлах, не разрешённых политикой
   памяти задачи.

#. С политикой bind или interleave может быть указан один или несколько узлов.
   Если с политикой preferred указано более одного узла, будет использован
   только узел с наименьшим числовым идентификатором.  Политика local выберет
   узел, на котором выполняется задача в момент построения маски nodes_allowed.
   Чтобы политика local была детерминированной, задача должна быть привязана к
   одному CPU или нескольким CPU в пределах одного узла.  В противном случае
   задача может быть мигрирована на какой-либо другой узел в любой момент после
   запуска, и результирующий узел будет неопределённым.  Поэтому политика local
   не очень полезна для этой цели.  Любой из других режимов mempolicy может быть
   использован для указания одного узла.

#. Маска разрешённых узлов будет получена из любой неумолчательной политики
   mempolicy задачи, независимо от того, была ли эта политика установлена явно
   самой задачей или одним из её предков, таким как numactl.  Это означает, что
   если задача вызывается из оболочки с неумолчательной политикой, будет
   использована эта политика.  Можно указать список узлов "all" с numactl
   --interleave или --membind [-m], чтобы добиться чередования (interleaving) по
   всем узлам в системе или cpuset.

#. Любая указанная политика mempolicy задачи — например, с использованием
   numactl — будет ограничена пределами ресурсов любого cpuset, в котором
   выполняется задача.  Таким образом, у задачи с неумолчательной политикой,
   выполняющейся в cpuset с подмножеством узлов системы, не будет способа
   выделить огромные страницы вне cpuset без предварительного перехода в cpuset,
   содержащий все нужные узлы.

#. Выделение огромных страниц во время загрузки пытается распределить
   запрошенное количество огромных страниц по всем онлайн-узлам с памятью.

Атрибуты огромных страниц для отдельных узлов
=============================================

Подмножество содержимого корневого управляющего каталога огромных страниц в
sysfs, описанного выше, будет реплицировано под системным устройством каждого
узла NUMA с памятью в::

	/sys/devices/system/node/node[0-9]*/hugepages/

Под этим каталогом подкаталог для каждого поддерживаемого размера огромной
страницы содержит следующие файлы атрибутов::

	nr_hugepages
	free_hugepages
	surplus_hugepages

Файлы атрибутов free\_' и surplus\_' доступны только для чтения.  Они возвращают
количество свободных и избыточных [overcommitted] огромных страниц
соответственно на родительском узле.

Атрибут ``nr_hugepages`` возвращает общее количество огромных страниц на
указанном узле.  При записи в этот атрибут количество постоянных огромных
страниц на родительском узле будет скорректировано до указанного значения, если
имеются достаточные ресурсы, независимо от mempolicy задачи или ограничений
cpuset.

Обратите внимание, что количество страниц overcommit и reserve остаётся
глобальными величинами, так как мы не знаем до момента отказа (fault time), когда
применяется mempolicy задачи, вызвавшей отказ, с какого узла будет предпринята
попытка выделения огромной страницы.

Страницы hugetlb могут мигрировать между пулами огромных страниц отдельных узлов
в следующих сценариях: вывод памяти из эксплуатации (memory offline), сбой памяти
(memory failure), долгосрочное закрепление (longterm pinning), системные вызовы
(mbind, migrate_pages и move_pages), alloc_contig_range() и
alloc_contig_pages().  Сейчас только вывод памяти из эксплуатации, сбой памяти и
системные вызовы допускают откат (fallback) к выделению новой страницы hugetlb на
другом узле, если текущий узел не может выполнить выделение во время миграции
hugetlb; это означает, что эти 3 случая могут нарушить пул огромных страниц
отдельного узла.

.. _using_huge_pages:

Использование огромных страниц
==============================

Если пользовательские приложения собираются запрашивать огромные страницы с
помощью системного вызова mmap, то требуется, чтобы системный администратор
смонтировал файловую систему типа hugetlbfs::

  mount -t hugetlbfs \
	-o uid=<value>,gid=<value>,mode=<value>,pagesize=<value>,size=<value>,\
	min_size=<value>,nr_inodes=<value> none /mnt/huge

Эта команда монтирует (псевдо)файловую систему типа hugetlbfs в каталог
``/mnt/huge``.  Любой файл, создаваемый в ``/mnt/huge``, использует огромные
страницы.

Опции ``uid`` и ``gid`` задают владельца и группу корня файловой системы.  По
умолчанию берутся ``uid`` и ``gid`` текущего процесса.

Опция ``mode`` устанавливает режим корня файловой системы в значение & 01777.
Это значение задаётся в восьмеричном виде. По умолчанию выбирается значение 0755.

Если платформа поддерживает несколько размеров огромных страниц, опцию
``pagesize`` можно использовать для указания размера огромной страницы и
связанного пула. ``pagesize`` указывается в байтах. Если ``pagesize`` не указан,
будут использованы размер огромной страницы по умолчанию для платформы и
связанный пул.

Опция ``size`` задаёт максимальное значение памяти (огромных страниц),
допустимое для этой файловой системы (``/mnt/huge``). Опция ``size`` может быть
указана в байтах или как процент от заданного пула огромных страниц
(``nr_hugepages``).  Размер округляется вниз до границы HPAGE_SIZE.

Опция ``min_size`` задаёт минимальное значение памяти (огромных страниц),
допустимое для файловой системы. ``min_size`` может быть указан так же, как
``size``, — либо в байтах, либо как процент от пула огромных страниц.
Во время монтирования количество огромных страниц, заданное ``min_size``,
резервируется для использования файловой системой.
Если доступного свободного количества огромных страниц недостаточно,
монтирование завершится неудачей.
По мере того как огромные страницы выделяются файловой системе и освобождаются,
счётчик резерва корректируется так, чтобы сумма выделенных и зарезервированных
огромных страниц всегда была не менее ``min_size``.

Опция ``nr_inodes`` задаёт максимальное количество inode, которое может
использовать ``/mnt/huge``.

Если опция ``size``, ``min_size`` или ``nr_inodes`` не указана в командной
строке, то никакие ограничения не устанавливаются.

Для опций ``pagesize``, ``size``, ``min_size`` и ``nr_inodes`` вы можете
использовать [G|g]/[M|m]/[K|k] для обозначения гига/мега/кило.
Например, size=2K имеет то же значение, что и size=2048.

Хотя системные вызовы read поддерживаются для файлов, находящихся в файловых
системах hugetlb, системные вызовы write не поддерживаются.

Обычные команды chown, chgrp и chmod (с правильными разрешениями) могут
использоваться для изменения атрибутов файлов в hugetlbfs.

Также важно отметить, что такая команда mount не требуется, если приложения
собираются использовать только системные вызовы shmat/shmget или mmap с
MAP_HUGETLB.  Пример того, как использовать mmap с MAP_HUGETLB, см. в
:ref:`map_hugetlb <map_hugetlb>` ниже.

Пользователи, которые хотят использовать память hugetlb через сегмент разделяемой
памяти, должны быть членами дополнительной группы, и системному администратору
нужно настроить этот gid в ``/proc/sys/vm/hugetlb_shm_group``.  Одни и те же или
разные приложения могут использовать любую комбинацию вызовов mmap и shm*, хотя
монтирование файловой системы потребуется для использования вызовов mmap без
MAP_HUGETLB.

У системных вызовов, оперирующих памятью, обеспеченной страницами hugetlb, длины
выравниваются только по родному размеру страницы процессора; они обычно
завершаются неудачей с errno, установленным в EINVAL, или исключают страницы
hugetlb, выходящие за пределы длины, если она не выровнена по огромной странице.
Например, munmap(2) завершится неудачей, если память обеспечена страницей
hugetlb, а длина меньше размера огромной страницы.
