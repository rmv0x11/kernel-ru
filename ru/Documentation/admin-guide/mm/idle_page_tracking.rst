=======================================================
Отслеживание простаивающих страниц (idle page tracking)
=======================================================

Мотивация
=========

Функция отслеживания простаивающих страниц (idle page tracking) позволяет
определять, к каким страницам памяти обращается рабочая нагрузка, а какие
простаивают. Эта информация может быть полезна для оценки размера рабочего
множества (working set) нагрузки, что, в свою очередь, можно учитывать при
настройке параметров нагрузки, установке лимитов memory cgroup или при решении,
куда разместить нагрузку в вычислительном кластере.

Функция включается параметром CONFIG_IDLE_PAGE_TRACKING=y.

.. _user_api:

Пользовательский API
====================

API отслеживания простаивающих страниц расположен в ``/sys/kernel/mm/page_idle``.
В настоящее время он состоит из единственного файла, доступного для чтения и
записи, ``/sys/kernel/mm/page_idle/bitmap``.

Этот файл реализует битовую карту, в которой каждый бит соответствует странице
памяти. Битовая карта представлена в виде массива 8-байтовых целых чисел, и
страница с PFN #i отображается на бит #i%64 элемента массива #i/64, порядок байт
— платформенный (native). Когда бит установлен, соответствующая страница
простаивает.

Страница считается простаивающей, если к ней не было обращений с тех пор, как она
была помечена как простаивающая (более подробно о том, что в действительности
означает «обращение», см. в разделе :ref:`Детали реализации <impl_details>`).
Чтобы пометить страницу как простаивающую, нужно установить соответствующий ей бит,
выполнив запись в файл. Записанное в файл значение объединяется по OR с текущим
значением битовой карты.

Отслеживаются только обращения к страницам пользовательской памяти. Это страницы,
отображённые в адресное пространство процесса, страницы page cache и буферов,
страницы swap cache. Для других типов страниц (например, страниц SLAB) попытка
пометить страницу как простаивающую молча игнорируется, и поэтому такие страницы
никогда не сообщаются как простаивающие.

Для huge-страниц флаг простоя устанавливается только на головной странице (head
page), поэтому для корректного подсчёта простаивающих huge-страниц необходимо
читать ``/proc/kpageflags``.

Чтение из ``/sys/kernel/mm/page_idle/bitmap`` или запись в него вернёт -EINVAL,
если чтение/запись начинается не на 8-байтовой границе или если размер
чтения/записи не кратен 8 байтам. Запись в этот файл за пределами максимального
PFN вернёт -ENXIO.

С учётом сказанного, чтобы оценить количество страниц, не используемых рабочей
нагрузкой, следует:

 1. Пометить все страницы нагрузки как простаивающие, установив соответствующие
    биты в ``/sys/kernel/mm/page_idle/bitmap``. Страницы можно найти, читая
    ``/proc/pid/pagemap``, если нагрузка представлена процессом, либо отфильтровав
    чужие страницы с помощью ``/proc/kpagecgroup``, если нагрузка размещена в
    memory cgroup.

 2. Дождаться, пока нагрузка обратится к своему рабочему множеству.

 3. Прочитать ``/sys/kernel/mm/page_idle/bitmap`` и подсчитать число
    установленных битов. Если нужно проигнорировать определённые типы страниц,
    например mlocked-страницы, поскольку они не подлежат высвобождению, их можно
    отфильтровать с помощью ``/proc/kpageflags``.

Для содействия в этом можно использовать инструмент page-types из каталога
tools/mm. Если запустить инструмент с подходящей опцией в первый раз, он пометит
все опрашиваемые страницы как простаивающие. Последующие запуски инструмента
смогут показать, у каких страниц за это время был снят флаг простоя.

Дополнительную информацию о ``/proc/pid/pagemap``, ``/proc/kpageflags`` и
``/proc/kpagecgroup`` см. в Documentation/admin-guide/mm/pagemap.rst.

Детали реализации
=================

Ядро внутренне отслеживает обращения к страницам пользовательской памяти, чтобы
при нехватке памяти в первую очередь высвобождать страницы, на которые нет ссылок.
Страница считается имеющей ссылку (referenced), если к ней недавно обращались через
адресное пространство процесса — в этом случае у одного или нескольких PTE, на
которые она отображена, будет установлен бит Accessed, — либо если ядро явно
пометило её как accessed (см. mark_page_accessed()). Последнее происходит, когда:

 - процесс пространства пользователя читает или записывает страницу с помощью
   системного вызова (например, read(2) или write(2))

 - страница, используемая для хранения буферов файловой системы, читается или
   записывается, потому что процессу нужны хранящиеся в ней метаданные файловой
   системы (например, при выводе списка дерева каталогов)

 - к странице обращается драйвер устройства с помощью get_user_pages()

Когда «грязная» (dirty) страница записывается в swap или на диск в результате
высвобождения памяти или превышения лимита грязной памяти, она не помечается как
имеющая ссылку.

Функция отслеживания простаивающей памяти добавляет новый флаг страницы — флаг
Idle. Этот флаг устанавливается вручную, путём записи в
``/sys/kernel/mm/page_idle/bitmap`` (см. раздел :ref:`Пользовательский API
<user_api>`), и автоматически сбрасывается всякий раз, когда на страницу появляется
ссылка в определённом выше смысле.

Когда страница помечается как простаивающая, бит Accessed должен быть сброшен во
всех PTE, на которые она отображена, иначе мы не сможем обнаружить обращения к
странице, поступающие из адресного пространства процесса. Чтобы избежать
интерференции с механизмом высвобождения (reclaimer), который, как отмечено выше,
использует бит Accessed для повышения приоритета активно используемых страниц,
вводится ещё один флаг страницы — флаг Young. Когда бит PTE Accessed сбрасывается в
результате установки или обновления флага Idle страницы, на странице устанавливается
флаг Young. Механизм высвобождения трактует флаг Young как дополнительный бит PTE
Accessed и поэтому будет считать такую страницу имеющей ссылку.

Поскольку функция отслеживания простаивающей памяти основана на логике механизма
высвобождения памяти, она работает только со страницами, находящимися в списке LRU,
а другие страницы молча игнорируются. Это означает, что она проигнорирует страницу
пользовательской памяти, если та изолирована, но поскольку таких страниц обычно
немного, это не должно заметно влиять на общий результат. Чтобы не задерживать
сканирование битовой карты простаивающих страниц, заблокированные (locked) страницы
также могут пропускаться.
