==========================================
Мониторинг файловой системы через fanotify
==========================================

Сообщения об ошибках файловой системы
=====================================

Fanotify поддерживает тип событий FAN_FS_ERROR для общесистемного
оповещения об ошибках файловой системы. Он предназначен для использования
демонами мониторинга состояния файловой системы, которые прослушивают эти
события и предпринимают действия (уведомить системного администратора,
запустить восстановление) при обнаружении проблемы с файловой системой.

По своему замыслу уведомление FAN_FS_ERROR предоставляет инструменту
мониторинга достаточно информации, чтобы узнать о возникшей в файловой
системе проблеме. Оно не обязательно предоставляет приложению в пространстве
пользователя семантику, позволяющую проверить, что операция ввода-вывода была
выполнена успешно. Это выходит за рамки данной функциональности. Напротив,
она задумана лишь как каркас для раннего обнаружения проблем файловой системы
и оповещения средств восстановления.

При сбое операции файловой системы часто бывает так, что вслед за исходным
сбоем каскадом следуют десятки ошибок ядра, скрывающих журнал исходного
сбоя, который обычно является самыми полезными отладочными данными для
устранения проблемы. По этой причине FAN_FS_ERROR старается сообщать только
о первой ошибке, произошедшей с файловой системой с момента последнего
уведомления, а дополнительные ошибки просто подсчитывает. Это гарантирует,
что наиболее важные сведения никогда не теряются.

FAN_FS_ERROR требует, чтобы группа fanotify была настроена с флагом
FAN_REPORT_FID.

На момент написания данного текста единственной файловой системой, которая
порождает уведомления FAN_FS_ERROR, является Ext4.

Уведомление FAN_FS_ERROR имеет следующий формат::

  ::

     [ Notification Metadata (Mandatory) ]
     [ Generic Error Record  (Mandatory) ]
     [ FID record            (Mandatory) ]

Порядок записей не гарантируется, и в будущем могут добавляться новые записи.
Поэтому приложения не должны полагаться на порядок и должны быть готовы
пропускать неизвестные записи. Пример разборщика см. в
``samples/fanotify/fs-monitor.c``.

Обобщённая запись об ошибке
---------------------------

Обобщённая запись об ошибке предоставляет достаточно информации, чтобы
независимый от файловой системы инструмент узнал о проблеме в файловой
системе, не предоставляя при этом каких-либо дополнительных подробностей о
проблеме. Эта запись идентифицируется по тому, что
``struct fanotify_event_info_header.info_type`` установлено в
FAN_EVENT_INFO_TYPE_ERROR.

  ::

     struct fanotify_event_info_error {
          struct fanotify_event_info_header hdr;
         __s32 error;
         __u32 error_count;
     };

Поле `error` идентифицирует тип ошибки с помощью значений errno.
`error_count` отслеживает количество ошибок, которые произошли и были
подавлены ради сохранения информации об исходной ошибке, с момента
последнего уведомления.

Запись FID
----------

Запись FID может использоваться для однозначной идентификации inode,
вызвавшего ошибку, посредством сочетания fsid и дескриптора файла (file
handle). Приложение, специфичное для конкретной файловой системы, может
использовать эту информацию для попытки выполнить процедуру восстановления.
Об ошибках, не связанных с inode, сообщается с пустым дескриптором файла
типа FILEID_INVALID.
