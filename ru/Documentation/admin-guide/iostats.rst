=========================================
Поля статистики ввода-вывода (I/O)
=========================================

Ядро предоставляет статистику дисков через ``/proc/diskstats`` и
``/sys/block/<device>/stat``. Обычно к этой статистике обращаются с помощью
таких инструментов, как ``sar`` и ``iostat``.

Ниже приведены примеры для диска с двумя разделами::

   /proc/diskstats:
     259       0 nvme0n1 255999 814 12369153 47919 996852 81 36123024 425995 0 301795 580470 0 0 0 0 60602 106555
     259       1 nvme0n1p1 492 813 17572 96 848 81 108288 210 0 76 307 0 0 0 0 0 0
     259       2 nvme0n1p2 255401 1 12343477 47799 996004 0 36014736 425784 0 344336 473584 0 0 0 0 0 0

   /sys/block/nvme0n1/stat:
     255999 814 12369153 47919 996858 81 36123056 426009 0 301809 580491 0 0 0 0 60605 106562

   /sys/block/nvme0n1/nvme0n1p1/stat:
     492 813 17572 96 848 81 108288 210 0 76 307 0 0 0 0 0 0

Оба файла содержат одни и те же 17 показателей статистики. ``/sys/block/<device>/stat``
содержит поля для ``<device>``. В ``/proc/diskstats`` полям предшествуют старший и
младший номера устройства, а также имя устройства. В приведённом выше примере первое
значение статистики для ``nvme0n1`` равно 255999 в обоих файлах.

Файл ``stat`` в sysfs эффективен для мониторинга небольшого, заранее известного
набора дисков. Если вы отслеживаете большое количество устройств,
``/proc/diskstats`` часто оказывается лучшим выбором, поскольку он позволяет избежать
накладных расходов на открытие и закрытие множества файлов для каждого снимка.

Все поля являются накопительными, монотонно возрастающими счётчиками, за исключением
поля 9, которое сбрасывается в ноль по мере завершения операций ввода-вывода. Остальные
поля сбрасываются при загрузке системы, при повторном подключении или повторной
инициализации устройства, либо при переполнении лежащего в основе счётчика. Приложения,
читающие эти счётчики, должны обнаруживать и обрабатывать сбросы при сравнении снимков
статистики.

Каждый набор статистики относится только к указанному устройству; если вам нужна
статистика по всей системе, придётся найти все устройства и просуммировать их все.

Поле  1 -- число завершённых операций чтения (unsigned long)
    Это общее число успешно завершённых операций чтения.

Поле  2 -- число объединённых операций чтения, поле 6 -- число объединённых операций записи (unsigned long)
    Операции чтения и записи, расположенные смежно друг с другом, могут быть объединены
    в целях эффективности.  Так, две операции чтения по 4K могут стать одной операцией
    чтения 8K перед тем, как она будет в конечном итоге передана на диск, и поэтому она
    будет учтена (и поставлена в очередь) как одна операция ввода-вывода.  Это поле
    позволяет узнать, как часто это происходило.

Поле  3 -- число прочитанных секторов (unsigned long)
    Это общее число успешно прочитанных секторов.

Поле  4 -- число миллисекунд, затраченных на чтение (unsigned int)
    Это общее число миллисекунд, затраченных всеми операциями чтения (измеряется
    от blk_mq_alloc_request() до __blk_mq_end_request()).

Поле  5 -- число завершённых операций записи (unsigned long)
    Это общее число успешно завершённых операций записи.

Поле  6 -- число объединённых операций записи  (unsigned long)
    См. описание поля 2.

Поле  7 -- число записанных секторов (unsigned long)
    Это общее число успешно записанных секторов.

Поле  8 -- число миллисекунд, затраченных на запись (unsigned int)
    Это общее число миллисекунд, затраченных всеми операциями записи (измеряется
    от blk_mq_alloc_request() до __blk_mq_end_request()).

Поле  9 -- число операций ввода-вывода, выполняемых в данный момент (unsigned int)
    Единственное поле, которое должно обнуляться. Увеличивается по мере передачи запросов
    в соответствующую struct request_queue и уменьшается по мере их завершения.

Поле 10 -- число миллисекунд, затраченных на выполнение операций ввода-вывода (unsigned int)
    Это поле увеличивается, пока поле 9 не равно нулю.

    Начиная с версии 5.0 это поле подсчитывает jiffies, когда была начата или завершена
    хотя бы одна операция. Если запрос выполняется дольше 2 jiffies, то часть времени
    ввода-вывода может не учитываться в случае параллельных запросов.

Поле 11 -- взвешенное число миллисекунд, затраченных на выполнение операций ввода-вывода (unsigned int)
    Это поле увеличивается при каждом начале операции ввода-вывода, завершении операции
    ввода-вывода, объединении операций ввода-вывода или чтении этой статистики на число
    выполняемых операций ввода-вывода (поле 9), умноженное на число миллисекунд,
    затраченных на ввод-вывод с момента последнего обновления этого поля.  Это может дать
    простую оценку как времени завершения ввода-вывода, так и накапливающегося отставания
    (backlog).

Поле 12 -- число завершённых операций отбрасывания (discard) (unsigned long)
    Это общее число успешно завершённых операций отбрасывания.

Поле 13 -- число объединённых операций отбрасывания (unsigned long)
    См. описание поля 2

Поле 14 -- число отброшенных секторов (unsigned long)
    Это общее число успешно отброшенных секторов.

Поле 15 -- число миллисекунд, затраченных на отбрасывание (unsigned int)
    Это общее число миллисекунд, затраченных всеми операциями отбрасывания (измеряется
    от blk_mq_alloc_request() до __blk_mq_end_request()).

Поле 16 -- число завершённых запросов сброса (flush)
    Это общее число успешно завершённых запросов сброса.

    Блочный уровень объединяет запросы сброса и выполняет не более одного за раз.
    Здесь подсчитываются запросы сброса, выполненные диском. Для разделов не отслеживается.

Поле 17 -- число миллисекунд, затраченных на сброс
    Это общее число миллисекунд, затраченных всеми запросами сброса.

Чтобы не вносить узкие места в производительность, при изменении этих счётчиков никакие
блокировки не удерживаются.  Это означает, что при совпадении изменений могут возникать
незначительные неточности, поэтому (например) сумма всех операций чтения, выполненных
по каждому разделу, должна быть равна операциям, выполненным с дисками... но из-за
отсутствия блокировок она может лишь быть очень близкой к этому.

В 2.6+ существуют счётчики для каждого CPU, что делает отсутствие блокировок почти
несущественным.  При чтении статистики счётчики по каждому CPU суммируются (возможно,
с переполнением переменной типа unsigned long, в которую они суммируются), и результат
выдаётся пользователю.  Удобного пользовательского интерфейса для доступа к самим
счётчикам по каждому CPU не существует.

Начиная с версии 4.19 времена запросов измеряются с точностью до наносекунд и
усекаются до миллисекунд перед отображением в этом интерфейсе.

Диски и разделы
-------------------

Между версиями 2.4 и 2.6+ в подсистеме ввода-вывода произошли значительные изменения.
В результате часть статистической информации исчезла. Преобразование адреса с диска,
относящегося к разделу, в адрес с диска, относящийся к хост-диску, теперь происходит
гораздо раньше.  Все объединения и измерения времени теперь происходят на уровне диска,
а не на уровне как диска, так и раздела, как это было в 2.4.  Следовательно, в 2.6+ вы
увидите для разделов иной вывод статистики, чем для дисков.  Для разделов на машинах
2.6+ доступно лишь *четыре* поля.  Это отражено в приведённых выше примерах.

Поле  1 -- число выданных операций чтения
    Это общее число операций чтения, выданных этому разделу.

Поле  2 -- число прочитанных секторов
    Это общее число секторов, запрошенных для чтения с этого раздела.

Поле  3 -- число выданных операций записи
    Это общее число операций записи, выданных этому разделу.

Поле  4 -- число записанных секторов
    Это общее число секторов, запрошенных для записи на этот раздел.

Обратите внимание, что поскольку адрес преобразуется в адрес, относящийся к диску, и
запись об адресе, относящемся к разделу, не сохраняется, последующий успех или неудача
чтения не может быть отнесён к разделу.  Иными словами, число операций чтения для
разделов подсчитывается немного раньше момента постановки в очередь для разделов, а для
дисков в целом — в момент завершения.  Это тонкое различие, которое, вероятно,
неинтересно в большинстве случаев.

Более значимой является погрешность, вызванная подсчётом числа операций чтения/записи
до объединений для разделов и после — для дисков. Поскольку типичная рабочая нагрузка
обычно содержит множество последовательных и смежных запросов, число выданных операций
чтения/записи может быть в несколько раз выше числа завершённых операций чтения/записи.

В 2.6.25 полный набор статистики снова доступен для разделов, и статистика дисков и
разделов снова согласована. Поскольку мы по-прежнему не сохраняем запись об адресе,
относящемся к разделу, операция относится к тому разделу, который содержит первый сектор
запроса после возможных объединений. Так как запросы могут быть объединены через границы
разделов, это может привести к некоторой (вероятно, незначительной) неточности.

Дополнительные замечания
------------------------

В 2.6+ sysfs не монтируется по умолчанию.  Если ваш дистрибутив Linux ещё не добавил
её, вот строка, которую вам нужно добавить в ``/etc/fstab``::

	none /sys sysfs defaults 0 0


В 2.6+ вся дисковая статистика была удалена из ``/proc/stat``.  В 2.4 она присутствует
как в ``/proc/partitions``, так и в ``/proc/stat``, хотя данные в ``/proc/stat`` имеют
формат, сильно отличающийся от данных в ``/proc/partitions``
(см. proc(5), если он есть в вашей системе.)
