==================================================
zram: Сжатые блочные устройства на основе RAM
==================================================

Введение
========

Модуль zram создаёт блочные устройства на основе RAM с именами /dev/zram<id>
(<id> = 0, 1, ...). Страницы, записываемые на эти диски, сжимаются и хранятся
непосредственно в памяти. Эти диски обеспечивают очень быстрый I/O, а сжатие
даёт хорошую экономию памяти. Среди вариантов использования — хранилище /tmp,
применение в качестве swap-дисков, различные кэши в /var и, возможно, многое
другое. :)

Статистика по отдельным устройствам zram экспортируется через узлы sysfs в
/sys/block/zram<id>/

Использование
=============

Существует несколько способов настройки устройств(а) zram и управления ими:

a) с помощью атрибутов sysfs zram и zram_control
b) с помощью утилиты zramctl, предоставляемой util-linux (util-linux@vger.kernel.org).

В этом документе мы опишем только «ручную» настройку zram, то есть атрибуты
sysfs zram и zram_control.

Чтобы получить лучшее представление о zramctl, обратитесь к документации
util-linux, man-странице zramctl или выполните `zramctl --help`. Обратите
внимание, что сопровождающие (maintainers) zram не разрабатывают/не сопровождают
util-linux или zramctl; если у вас возникнут вопросы, обращайтесь по адресу
util-linux@vger.kernel.org

Ниже показана типичная последовательность шагов для использования zram.

ПРЕДУПРЕЖДЕНИЕ
==============

Ради простоты в большинстве приведённых ниже примеров мы опускаем части,
отвечающие за проверку ошибок. Однако обработка ошибок целиком на вашей
ответственности.

Атрибуты sysfs zram в случае ошибок всегда возвращают отрицательные значения.
Список возможных кодов возврата:

========  =============================================================
-EBUSY	  попытка изменить атрибут, который нельзя изменить после того,
	  как устройство было инициализировано. Сначала сбросьте устройство.
-ENOMEM	  zram не смог выделить достаточно памяти для удовлетворения ваших
	  потребностей.
-EINVAL	  предоставлены некорректные входные данные.
-EAGAIN	  повторите операцию позже (например, при попытке одновременно
	  выполнить recompress и writeback).
========  =============================================================

Если вы используете 'echo', возвращаемое значение устанавливается утилитой
'echo', и в общем случае чего-то вроде::

	echo foo > /sys/block/zram0/comp_algorithm
	if [ $? -ne 0 ]; then
		handle_error
	fi

должно быть достаточно.

1) Загрузка модуля
==================

::

	modprobe zram num_devices=4

Это создаёт 4 устройства: /dev/zram{0,1,2,3}

Параметр num_devices необязателен и сообщает zram, сколько устройств должно быть
создано заранее. По умолчанию: 1.

2) Выбор алгоритма сжатия
=========================

С помощью атрибута устройства comp_algorithm можно увидеть доступные и
выбранные в данный момент (показаны в квадратных скобках) алгоритмы сжатия
или изменить выбранный алгоритм сжатия (после инициализации устройства изменить
алгоритм сжатия невозможно).

Примеры::

	#show supported compression algorithms
	cat /sys/block/zram0/comp_algorithm
	lzo [lz4]

	#select lzo compression algorithm
	echo lzo > /sys/block/zram0/comp_algorithm

На данный момент содержимое `comp_algorithm` показывает только те алгоритмы
сжатия, которые поддерживаются zram.

3) Установка параметров алгоритма сжатия: необязательно
=======================================================

Алгоритмы сжатия могут поддерживать специфические параметры, которые можно
настраивать под конкретный набор данных. У ZRAM есть атрибут устройства
`algorithm_params`, который обеспечивает конфигурацию параметров для каждого
алгоритма.

Например, несколько алгоритмов сжатия поддерживают параметр `level`. Кроме того,
некоторые алгоритмы сжатия поддерживают предобученные словари, которые
существенно меняют характеристики алгоритмов. Чтобы настроить алгоритм сжатия на
использование внешнего предобученного словаря, передайте полный путь к `dict`
вместе с другими параметрами::

	#pass path to pre-trained zstd dictionary
	echo "algo=zstd dict=/etc/dictionary" > /sys/block/zram0/algorithm_params

	#same, but using algorithm priority
	echo "priority=1 dict=/etc/dictionary" > \
		/sys/block/zram0/algorithm_params

	#pass path to pre-trained zstd dictionary and compression level
	echo "algo=zstd level=8 dict=/etc/dictionary" > \
		/sys/block/zram0/algorithm_params

Параметры специфичны для алгоритма: не все алгоритмы поддерживают предобученные
словари, не все алгоритмы поддерживают `level`. Более того, для некоторых
алгоритмов `level` управляет уровнем сжатия (чем выше значение, тем лучше
коэффициент сжатия; для некоторых алгоритмов он даже может принимать
отрицательные значения), а для других алгоритмов `level` — это уровень ускорения
(чем выше значение, тем ниже коэффициент сжатия).

4) Установка disksize
=====================

Задайте размер диска, записав значение в узел sysfs 'disksize'.
Значение может быть указано в байтах, либо можно использовать суффиксы памяти.
Примеры::

	# Initialize /dev/zram0 with 50MB disksize
	echo $((50*1024*1024)) > /sys/block/zram0/disksize

	# Using mem suffixes
	echo 256K > /sys/block/zram0/disksize
	echo 512M > /sys/block/zram0/disksize
	echo 1G > /sys/block/zram0/disksize

Примечание:
Нет особого смысла создавать zram размером более чем вдвое больше объёма памяти,
поскольку мы ожидаем коэффициент сжатия 2:1. Обратите внимание, что zram
использует около 0,1% размера диска, когда не используется, поэтому огромный
zram расточителен.

5) Установка лимита памяти: необязательно
=========================================

Задайте лимит памяти, записав значение в узел sysfs 'mem_limit'.
Значение может быть указано в байтах, либо можно использовать суффиксы памяти.
Кроме того, вы можете изменить значение во время выполнения.
Примеры::

	# limit /dev/zram0 with 50MB memory
	echo $((50*1024*1024)) > /sys/block/zram0/mem_limit

	# Using mem suffixes
	echo 256K > /sys/block/zram0/mem_limit
	echo 512M > /sys/block/zram0/mem_limit
	echo 1G > /sys/block/zram0/mem_limit

	# To disable memory limit
	echo 0 > /sys/block/zram0/mem_limit

6) Активация
============

::

	mkswap /dev/zram0
	swapon /dev/zram0

	mkfs.ext4 /dev/zram1
	mount /dev/zram1 /tmp

7) Добавление/удаление устройств zram
=====================================

zram предоставляет управляющий интерфейс, который позволяет динамически (по
требованию) добавлять и удалять устройства.

Чтобы добавить новое устройство /dev/zramX, выполните операцию чтения атрибута
hot_add. Это вернёт либо идентификатор нового устройства (это означает, что вы
можете использовать /dev/zram<id>), либо код ошибки.

Пример::

	cat /sys/class/zram-control/hot_add
	1

Чтобы удалить существующее устройство /dev/zramX (где X — идентификатор
устройства), выполните::

	echo X > /sys/class/zram-control/hot_remove

8) Статистика
=============

Статистика по каждому устройству экспортируется в виде различных узлов в
/sys/block/zram<id>/

Ниже приводится краткое описание экспортируемых атрибутов устройства. Более
подробную информацию см. в Documentation/ABI/testing/sysfs-block-zram.

======================  ======  ===============================================
Имя             	доступ            описание
======================  ======  ===============================================
disksize          	RW	показать и задать размер диска устройства
initstate         	RO	показывает состояние инициализации устройства
reset             	WO	запустить сброс устройства
mem_used_max      	WO	сбросить счётчик `mem_used_max` (см. далее)
mem_limit         	WO	задаёт максимальный объём памяти, который ZRAM может
				использовать для хранения сжатых данных
writeback_limit   	WO	задаёт максимальный объём записи (write IO), которую
				zram может выполнить на backing device, в единицах 4 КБ
writeback_limit_enable  RW	показать и задать функцию writeback_limit
writeback_batch_size	RW	показать и задать максимальное число одновременных
				(in-flight) операций writeback
compressed_writeback	RW	показать и задать функцию сжатого writeback
comp_algorithm    	RW	показать и изменить алгоритм сжатия
algorithm_params	WO	настроить параметры алгоритма сжатия
compact           	WO	запустить уплотнение памяти (compaction)
debug_stat        	RO	этот файл используется в целях отладки zram
backing_dev	  	RW	настроить backend-хранилище, в которое zram будет писать
idle		  	WO	пометить выделенный слот как простаивающий (idle)
======================  ======  ===============================================

Пространству пользователя рекомендуется использовать следующие файлы для чтения
статистики устройства.

Файл /sys/block/zram<id>/stat

Представляет статистику блочного уровня. Подробности см. в
Documentation/block/stat.rst.

Файл /sys/block/zram<id>/io_stat

Файл stat представляет статистику I/O устройства, не учитываемую блочным
уровнем и, следовательно, недоступную в файле zram<id>/stat. Он состоит из
одной строки текста и содержит следующую статистику, разделённую пробелами:

 =============    =============================================================
 failed_reads     число неудачных чтений
 failed_writes    число неудачных записей
 invalid_io       число запросов I/O, не выровненных по размеру страницы
 notify_free      в зависимости от сценария использования устройства может
                  учитывать

                  a) число страниц, освобождённых из-за уведомлений об
                     освобождении swap-слота
                  b) число страниц, освобождённых из-за запросов
                     REQ_OP_DISCARD, отправленных bio. Первые отправляются
                     на swap-устройство при освобождении swap-слота, что
                     подразумевает, что этот диск используется как swap-диск.

                  Вторые отправляются файловой системой, смонтированной с
                  опцией discard, всякий раз, когда отбрасываются какие-либо
                  блоки данных.
 =============    =============================================================

Файл /sys/block/zram<id>/mm_stat

Файл mm_stat представляет mm-статистику устройства. Он состоит из одной строки
текста и содержит следующую статистику, разделённую пробелами:

 ================ =============================================================
 orig_data_size   несжатый размер данных, хранящихся на этом диске.
                  Единица измерения: байты
 compr_data_size  сжатый размер данных, хранящихся на этом диске
 mem_used_total   объём памяти, выделенный для этого диска. Сюда входят
                  фрагментация аллокатора и накладные расходы на метаданные,
                  выделенные для этого диска. Таким образом, эффективность
                  использования пространства аллокатором можно вычислить с
                  помощью compr_data_size и этой статистики.
                  Единица измерения: байты
 mem_limit        максимальный объём памяти, который ZRAM может использовать
                  для хранения сжатых данных
 mem_used_max     максимальный объём памяти, который zram потребил для
                  хранения данных
 same_pages       число записанных на этот диск страниц, заполненных
                  одинаковым элементом. Для таких страниц память не выделяется.
 pages_compacted  число страниц, освобождённых во время уплотнения (compaction)
 huge_pages	  число несжимаемых страниц
 huge_pages_since число несжимаемых страниц с момента настройки zram
 ================ =============================================================

Файл /sys/block/zram<id>/bd_stat

Файл bd_stat представляет статистику backing device устройства. Он состоит из
одной строки текста и содержит следующую статистику, разделённую пробелами:

 ============== =============================================================
 bd_count	объём данных, записанных на backing device.
		Единица измерения: 4 КБ
 bd_reads	число чтений с backing device
		Единица измерения: 4 КБ
 bd_writes	число записей на backing device
		Единица измерения: 4 КБ
 ============== =============================================================

9) Деактивация
==============

::

	swapoff /dev/zram0
	umount /dev/zram1

10) Сброс
=========

	Запишите любое положительное значение в узел sysfs 'reset'::

		echo 1 > /sys/block/zram0/reset
		echo 1 > /sys/block/zram1/reset

	Это освобождает всю память, выделенную для данного устройства, и
	сбрасывает disksize в ноль. Перед повторным использованием устройства
	вы должны снова задать disksize.

Необязательная функциональность
================================

Отслеживание простаивающих (IDLE) страниц
------------------------------------------

zram имеет встроенную поддержку отслеживания простаивающих страниц (то есть
выделенных, но не используемых страниц). Эта функциональность полезна,
например, для writeback и recompression в zram. Чтобы пометить страницы как
простаивающие, выполните следующую команду::

	echo all > /sys/block/zramX/idle

Это пометит все выделенные страницы zram как простаивающие. Отметка о простое
будет снята только при обращении к странице (блоку) (например, при перезаписи
или освобождении). Кроме того, когда включён CONFIG_ZRAM_TRACK_ENTRY_ACTIME,
страницы могут помечаться как простаивающие в зависимости от того, сколько
секунд прошло с момента последнего обращения к конкретной странице zram::

	echo 86400 > /sys/block/zramX/idle

В этом примере все страницы, к которым не обращались более 86400 секунд (одни
сутки), будут помечены как простаивающие.

writeback
---------

При CONFIG_ZRAM_WRITEBACK zram может записывать простаивающую/несжимаемую
страницу в backing-хранилище вместо того, чтобы держать её в памяти.
Чтобы использовать эту функциональность, администратор должен настроить
backing device через::

	echo /dev/sda5 > /sys/block/zramX/backing_dev

перед установкой disksize. На данный момент поддерживаются только разделы.
Если администратор хочет использовать writeback несжимаемых страниц, он может
сделать это через::

	echo huge > /sys/block/zramX/writeback

Администратор может запросить writeback простаивающих страниц в нужный момент
через::

	echo idle > /sys/block/zramX/writeback

С помощью этой команды zram выполнит writeback простаивающих страниц из памяти
в хранилище.

Кроме того, если пользователь решает выполнять writeback только для huge- и
простаивающих страниц, этого можно добиться с помощью::

        echo huge_idle > /sys/block/zramX/writeback

Если пользователь решает выполнять writeback только несжимаемых страниц
(страниц, которые не может сжать ни один из алгоритмов), этого можно добиться
с помощью::

	echo incompressible > /sys/block/zramX/writeback

Если администратор хочет записать конкретную страницу устройства zram на
backing device, он может записать индекс страницы в интерфейс::

	echo "page_index=1251" > /sys/block/zramX/writeback

В Linux 6.16 этот интерфейс был частично переработан. Во-первых, теперь
интерфейс поддерживает формат `key=value` для всех своих параметров
(`type=huge_idle` и т.д.). Во-вторых, была введена поддержка `page_indexes`,
которая задаёт диапазон (или диапазоны) страниц `LOW-HIGH` для writeback. Это
сокращает число системных вызовов, но, что важнее, позволяет применять
оптимальную стратегию выбора целей постобработки. Пример использования::

	echo "type=idle" > /sys/block/zramX/writeback
	echo "page_indexes=1-100 page_indexes=200-300" > \
		/sys/block/zramX/writeback

Теперь мы также разрешаем несколько параметров page_index за один вызов и
смешивание отдельных страниц с диапазонами страниц::

	echo page_index=42 page_index=99 page_indexes=100-200 \
		page_indexes=500-700 > /sys/block/zramX/writeback

Если на flash-устройстве выполняется большой объём операций записи (write IO),
оно потенциально может столкнуться с проблемой износа flash-памяти, поэтому
администратору необходимо предусмотреть ограничение записи, чтобы гарантировать
исправность хранилища на протяжении всего срока службы продукта.

Чтобы решить эту проблему, zram поддерживает функциональность "writeback_limit".
Значение "writeback_limit_enable" по умолчанию равно 0, поэтому оно не
ограничивает никакой writeback. Иными словами, если администратор хочет применить
бюджет writeback, он должен включить writeback_limit_enable через::

	$ echo 1 > /sys/block/zramX/writeback_limit_enable

После того как writeback_limit_enable установлен, zram не разрешает никакой
writeback, пока администратор не задаст бюджет через
/sys/block/zramX/writeback_limit.

(Если администратор не включает writeback_limit_enable, значение
writeback_limit, заданное через /sys/block/zramX/writeback_limit, не имеет
смысла.)

Если администратор хочет ограничить writeback значением 400M в день, он может
сделать это, как показано ниже::

	$ MB_SHIFT=20
	$ 4K_SHIFT=12
	$ echo $((400<<MB_SHIFT>>4K_SHIFT)) > \
		/sys/block/zram0/writeback_limit.
	$ echo 1 > /sys/block/zram0/writeback_limit_enable

Если администратор хочет снова разрешить дальнейшую запись после исчерпания
бюджета, он может сделать это, как показано ниже::

	$ echo $((400<<MB_SHIFT>>4K_SHIFT)) > \
		/sys/block/zram0/writeback_limit

Если администратор хочет увидеть оставшийся бюджет writeback с момента последней
установки::

	$ cat /sys/block/zramX/writeback_limit

Если администратор хочет отключить лимит writeback, он может выполнить::

	$ echo 0 > /sys/block/zramX/writeback_limit_enable

Счётчик writeback_limit сбрасывается всякий раз, когда вы сбрасываете zram
(например, при перезагрузке системы, echo 1 > /sys/block/zramX/reset), поэтому
отслеживание того, сколько writeback произошло до сброса zram, чтобы выделить
дополнительный бюджет writeback при следующей установке, — задача пользователя.

По умолчанию zram хранит записанные обратно (written back) страницы в
распакованной (raw) форме, что означает, что операция writeback включает
распаковку страницы перед её записью на backing device. Это поведение можно
изменить, включив функциональность `compressed_writeback`, которая заставляет
zram записывать на backing device сжатые страницы, тем самым избегая накладных
расходов на распаковку. Чтобы включить эту функциональность, выполните::

	$ echo yes > /sys/block/zramX/compressed_writeback

Обратите внимание, что эту функциональность следует настраивать до
инициализации устройства `zramX`.

В зависимости от типа хранилища backing device операция writeback может выиграть
от большего числа одновременных (in-flight) запросов записи (пакетные записи).
Максимальное число одновременных операций writeback можно настроить через
атрибут `writeback_batch_size`. Чтобы изменить значение по умолчанию (которое
равно 32), выполните::

	$ echo 64 > /sys/block/zramX/writeback_batch_size

Если администратор хочет измерить количество writeback за определённый период,
он может узнать это через 3-й столбец /sys/block/zram0/bd_stat.

recompression
-------------

При CONFIG_ZRAM_MULTI_COMP zram может повторно сжимать страницы, используя
альтернативные (вторичные) алгоритмы сжатия. Основная идея в том, что
альтернативный алгоритм сжатия может обеспечить лучший коэффициент сжатия ценой
(потенциально) более медленных скоростей сжатия/распаковки. Альтернативный
алгоритм сжатия может, например, успешнее сжимать huge-страницы (те, которые не
удалось сжать алгоритму по умолчанию). Другое применение — recompression
простаивающих страниц: страницы, которые «остыли» и лежат в памяти, можно
повторно сжать более эффективным алгоритмом и тем самым снизить использование
памяти zsmalloc.

При CONFIG_ZRAM_MULTI_COMP zram поддерживает до 4 алгоритмов сжатия: один
первичный и до 3 вторичных. Первичный компрессор zram объясняется в разделе
«3) Выбор алгоритма сжатия», вторичные алгоритмы настраиваются с помощью
атрибута устройства recomp_algorithm.

Пример:::

	#show supported recompression algorithms
	cat /sys/block/zramX/recomp_algorithm
	#1: lzo lzo-rle lz4 lz4hc [zstd]
	#2: lzo lzo-rle lz4 [lz4hc] zstd

Альтернативные алгоритмы сжатия упорядочены по приоритету. В примере выше zstd
используется как первый альтернативный алгоритм, имеющий приоритет 1, тогда как
lz4hc настроен как алгоритм сжатия с приоритетом 2. Приоритет альтернативного
алгоритма сжатия задаётся во время настройки алгоритмов:::

	#select zstd recompression algorithm, priority 1
	echo "algo=zstd priority=1" > /sys/block/zramX/recomp_algorithm

	#select deflate recompression algorithm, priority 2
	echo "algo=deflate priority=2" > /sys/block/zramX/recomp_algorithm

Ещё один атрибут устройства, который включает CONFIG_ZRAM_MULTI_COMP, — это
recompress, который управляет recompression.

Примеры:::

	#IDLE pages recompression is activated by `idle` mode
	echo "type=idle" > /sys/block/zramX/recompress

	#HUGE pages recompression is activated by `huge` mode
	echo "type=huge" > /sys/block/zram0/recompress

	#HUGE_IDLE pages recompression is activated by `huge_idle` mode
	echo "type=huge_idle" > /sys/block/zramX/recompress

Число простаивающих страниц может быть значительным, поэтому пространство
пользователя может передать пороговое значение размера (в байтах) в управляющий
элемент recompress: zram будет повторно сжимать только страницы равного или
большего размера:::

	#recompress all pages larger than 3000 bytes
	echo "threshold=3000" > /sys/block/zramX/recompress

	#recompress idle pages larger than 2000 bytes
	echo "type=idle threshold=2000" > /sys/block/zramX/recompress

Также можно ограничить число страниц, которые re-compression zram попытается
повторно сжать:::

	echo "type=huge_idle max_pages=42" > /sys/block/zramX/recompress

Во время re-compression для каждой страницы, удовлетворяющей критериям
re-compression, ZRAM перебирает список зарегистрированных альтернативных
алгоритмов сжатия в порядке их приоритетов. ZRAM останавливается либо когда
re-compression прошла успешно (повторно сжатый объект меньше по размеру, чем
исходный) и удовлетворяет критериям re-compression (например, порогу размера),
либо когда не осталось вторичных алгоритмов для попытки. Если ни один из
вторичных алгоритмов не смог успешно повторно сжать страницу, такая страница
помечается как несжимаемая, поэтому ZRAM не будет пытаться повторно сжать её в
будущем.

Это поведение re-compression, когда выполняется перебор списка
зарегистрированных алгоритмов сжатия, повышает наши шансы найти алгоритм,
который успешно сожмёт конкретную страницу. Иногда, однако, удобно (а иногда
даже необходимо) ограничить recompression только одним конкретным алгоритмом,
чтобы она не пробовала никакие другие алгоритмы. Этого можно добиться, указав
параметр `algo` или `priority`:::

	#use zstd algorithm only (if registered)
	echo "type=huge algo=zstd" > /sys/block/zramX/recompress

	#use zstd algorithm only (if zstd was registered under priority 1)
	echo "type=huge priority=1" > /sys/block/zramX/recompress

memory tracking
===============

При CONFIG_ZRAM_MEMORY_TRACKING пользователь может узнать информацию о блоке
zram. Это может быть полезно для отлова «холодных» или несжимаемых страниц
процесса с помощью pagemap.

Если вы включите эту функциональность, вы сможете увидеть состояние блока через
/sys/kernel/debug/zram/zram0/block_state". Вывод выглядит следующим образом::

	  300    75.033841 .wh...
	  301    63.806904 s.....
	  302    63.806919 ..hi..
	  303    62.801919 ....r.
	  304   146.781902 ..hi.n

Первый столбец
	индекс блока zram.
Второй столбец
	время доступа с момента загрузки системы
Третий столбец
	состояние блока:

	s:
		одинаковая (same) страница
	w:
		страница, записанная в backing-хранилище
	h:
		huge-страница
	i:
		простаивающая (idle) страница
	r:
		повторно сжатая (recompressed) страница (вторичный алгоритм сжатия)
	n:
		ни один (включая вторичные) из алгоритмов не смог её сжать

Первая строка в примере выше говорит, что к 300-му блоку обращались в момент
75.033841 сек и состояние блока — huge, поэтому он записан обратно в
backing-хранилище. Это отладочная функциональность, поэтому никто не должен
полагаться на её корректную работу.
