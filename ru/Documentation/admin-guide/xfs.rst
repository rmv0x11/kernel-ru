============================
Файловая система SGI XFS
============================

XFS — это высокопроизводительная журналируемая файловая система, которая
возникла на платформе SGI IRIX. Она полностью многопоточная, может
поддерживать большие файлы и большие файловые системы, расширенные атрибуты,
переменные размеры блоков, основана на экстентах и интенсивно использует
B-деревья (каталоги, экстенты, свободное пространство) для повышения как
производительности, так и масштабируемости.

За подробностями обращайтесь к документации по адресу
https://xfs.wiki.kernel.org/. Эта реализация совместима на уровне диска
с версией XFS для IRIX.


Параметры монтирования
======================

При монтировании файловой системы XFS принимаются следующие параметры.

  allocsize=size
	Задаёт размер предварительного выделения в конце файла для буферизованного
	ввода-вывода при отложенной записи (default size is 64KiB).
	Допустимые значения этого параметра — от размера страницы (обычно 4KiB)
	до 1GiB включительно, с шагом в степени двойки.

	По умолчанию используется динамический размер предварительного выделения
	в конце файла, который применяет набор эвристик для оптимизации размера
	предварительного выделения на основе текущих шаблонов выделения внутри файла
	и шаблонов доступа к файлу. Задание фиксированного значения ``allocsize``
	отключает динамическое поведение.

  discard or nodiscard (default)
	Включает/отключает выдачу команд, позволяющих блочному устройству
	освобождать пространство, высвобожденное файловой системой. Это
	полезно для SSD-устройств, тонко выделенных LUN'ов и образов
	виртуальных машин, но может влиять на производительность.

	Примечание: в настоящее время рекомендуется использовать приложение
	``fstrim`` для ``discard`` неиспользуемых блоков, а не параметр монтирования
	``discard``, поскольку влияние этого параметра на производительность
	довольно серьёзное.

  grpid/bsdgroups or nogrpid/sysvgroups (default)
	Эти параметры определяют, какой идентификатор группы получает
	вновь созданный файл. Когда установлен ``grpid``, файл получает
	идентификатор группы каталога, в котором он создан; в противном случае он
	получает ``fsgid`` текущего процесса, если только у каталога не установлен
	бит ``setgid``, — в этом случае он получает ``gid`` из родительского
	каталога, а также получает установленный бит ``setgid``, если сам является
	каталогом.

  filestreams
	Заставляет распределитель данных использовать режим выделения filestreams
	по всей файловой системе, а не только в каталогах, настроенных на его
	использование.

  inode32 or inode64 (default)
	Когда указан ``inode32``, это означает, что XFS ограничивает создание
	inode местами, которые не приведут к номерам inode со значимостью более
	32 бит.

	Когда указан ``inode64``, это означает, что XFS разрешено создавать
	inode в любом месте файловой системы, включая те, которые приведут
	к номерам inode, занимающим более 32 бит значимости.

	``inode32`` предоставляется для обратной совместимости со старыми
	системами и приложениями, поскольку 64-битные номера inode могут
	вызывать проблемы у некоторых приложений, которые не способны работать
	с большими номерами inode. Если используются приложения, которые
	не работают с номерами inode больше 32 бит, следует указать параметр
	``inode32``.

  largeio or nolargeio (default)
	Если указан ``nolargeio``, оптимальный ввод-вывод, сообщаемый в
	``st_blksize`` через **stat(2)**, будет настолько малым, насколько возможно,
	чтобы пользовательские приложения могли избежать неэффективного
	ввода-вывода типа read/modify/write. Обычно это размер страницы машины,
	так как это гранулярность страничного кэша.

	Если указан ``largeio``, файловая система, созданная с заданным
	``swidth``, будет возвращать значение ``swidth`` (в байтах) в ``st_blksize``.
	Если в файловой системе не задан ``swidth``, но задан ``allocsize``, то
	вместо него будет возвращаться ``allocsize`` (в байтах). В противном случае
	поведение такое же, как если бы был указан ``nolargeio``.

  logbufs=value
	Задаёт количество буферов лога в памяти. Допустимые числа —
	в диапазоне от 2 до 8 включительно.

	Значение по умолчанию — 8 буферов.

	Если затраты памяти на 8 буферов лога слишком велики для малых
	систем, их можно уменьшить ценой некоторого снижения производительности
	на нагрузках, интенсивно работающих с метаданными. Параметр ``logbsize``
	ниже управляет размером каждого буфера и поэтому также имеет отношение к
	этому случаю.

  lifetime (default) or nolifetime
	Включает размещение данных на основе подсказок о времени жизни записи,
	предоставляемых пользователем. Это включает совместное размещение данных
	со схожим временем жизни, когда это статистически выгодно, для снижения
	стоимости сборки мусора.

	Эти параметры доступны только для зонированных rt-файловых систем.

  logbsize=value
	Задаёт размер каждого буфера лога в памяти. Размер может быть
	указан в байтах или в килобайтах с суффиксом «k».
	Допустимые размеры для логов версии 1 и версии 2 — 16384 (16k)
	и 32768 (32k). Допустимые размеры для логов версии 2 также
	включают 65536 (64k), 131072 (128k) и 262144 (256k).
	logbsize должен быть целым кратным единицы чередования лога,
	настроенной во время **mkfs(8)**.

	Значение по умолчанию для логов версии 1 — 32768, в то время как
	значение по умолчанию для логов версии 2 — MAX(32768, log_sunit).

  logdev=device and rtdev=device
	Использовать внешний лог (журнал метаданных) и/или устройство реального
	времени. Файловая система XFS состоит максимум из трёх частей: секции
	данных, секции лога и секции реального времени. Секция реального времени
	необязательна, а секция лога может быть отделена от секции данных или
	содержаться внутри неё.

  max_atomic_write=value
	Задаёт максимальный размер атомарной записи. Размер может быть
	указан в байтах, в килобайтах с суффиксом «k», в мегабайтах
	с суффиксом «m» или в гигабайтах с суффиксом «g». Размер
	не может быть больше максимального размера записи, больше
	размера любой группы выделения или больше размера операции
	переотображения, которую лог может завершить атомарно.

	По умолчанию максимальный размер завершения ввода-вывода устанавливается
	так, чтобы каждый CPU мог обрабатывать по одному за раз.

  max_open_zones=value
	Задаёт максимальное число зон, которые остаются открытыми для записи на
	зонированном rt-устройстве. Большое количество открытых зон помогает
	разделению данных файлов, но может влиять на производительность на HDD.

	Если ``max_open_zones`` не указан, значение определяется
	возможностями и размером зонированного rt-устройства.

  noalign
	Выделения данных не будут выравниваться по границам единицы
	чередования. Это актуально только для файловых систем, созданных
	с ненулевыми параметрами выравнивания данных (``sunit``, ``swidth``) с помощью
	**mkfs(8)**.

  norecovery
	Файловая система будет смонтирована без выполнения восстановления лога.
	Если файловая система не была чисто размонтирована, она, скорее всего,
	будет несогласованной при монтировании в режиме ``norecovery``.
	Некоторые файлы или каталоги могут быть недоступны из-за этого.
	Файловые системы, смонтированные с ``norecovery``, должны монтироваться
	только для чтения, иначе монтирование завершится неудачей.

  nouuid
	Не проверять дважды смонтированные файловые системы по ``uuid``
	файловой системы. Это полезно для монтирования томов LVM-снимков
	и часто используется в сочетании с ``norecovery`` для монтирования
	снимков только для чтения.

  noquota
	Принудительно отключает весь учёт квот и контроль их соблюдения
	внутри файловой системы.

  uquota/usrquota/uqnoenforce/quota
	Включён учёт пользовательских дисковых квот, а лимиты (опционально)
	контролируются. За подробностями обращайтесь к **xfs_quota(8)**.

  gquota/grpquota/gqnoenforce
	Включён учёт групповых дисковых квот, а лимиты (опционально)
	контролируются. За подробностями обращайтесь к **xfs_quota(8)**.

  pquota/prjquota/pqnoenforce
	Включён учёт проектных дисковых квот, а лимиты (опционально)
	контролируются. За подробностями обращайтесь к **xfs_quota(8)**.

  sunit=value and swidth=value
	Используются для задания единицы и ширины чередования для RAID-устройства
	или тома с чередованием. "value" должно быть указано в единицах
	512-байтовых блоков. Эти параметры актуальны только для файловых систем,
	которые были созданы с ненулевыми параметрами выравнивания данных.

	Указанные параметры ``sunit`` и ``swidth`` должны быть совместимы
	с существующими характеристиками выравнивания файловой системы. В
	общем случае это означает, что единственными допустимыми изменениями ``sunit``
	являются его увеличение в кратное степени двойки число раз. Допустимые
	значения ``swidth`` — любое целое кратное допустимого значения ``sunit``.

	Обычно эти параметры монтирования необходимы только в том случае,
	если у нижележащего RAID-устройства была изменена геометрия,
	например при добавлении нового диска в RAID5 lun и его
	перестроении.

  swalloc
	Выделения данных будут округляться вверх до границ ширины чередования,
	когда текущий конец файла расширяется и размер файла
	больше размера ширины чередования.

  wsync
	Когда указан, все операции пространства имён файловой системы
	выполняются синхронно. Это гарантирует, что когда операция с пространством
	имён (создание, удаление и т. д.) завершается, изменение в
	пространстве имён находится на стабильном носителе. Это полезно в
	HA-конфигурациях, где аварийное переключение не должно приводить к тому,
	что клиенты видят несогласованное представление пространства имён во время
	или после события аварийного переключения.

  errortag=tagname
	Когда указан, включает тег инъекции ошибок с именем "tagname" с
	частотой по умолчанию. Может быть указан несколько раз для включения
	нескольких errortag. Указание этого параметра при перемонтировании сбросит
	тег ошибки к значению по умолчанию, если ранее ему было присвоено любое
	другое значение.
	Этот параметр поддерживается только при включённом CONFIG_XFS_DEBUG и
	не будет отражён в /proc/self/mounts.

Прекращение поддержки формата V4
================================

В формате файловой системы V4 отсутствуют некоторые возможности, которые
поддерживаются форматом V5, такие как контрольное суммирование метаданных,
усиленная проверка метаданных и возможность хранить метки времени после
2038 года. Из-за этого формат V4 считается устаревшим. Всем пользователям
следует выполнить обновление, сделав резервную копию своих файлов,
переформатировав и восстановив из резервной копии.

Администраторы и пользователи могут обнаружить файловую систему V4, запустив
xfs_info на точке монтирования файловой системы и проверив наличие строки,
содержащей "crc=". Если такая строка не найдена, обновите xfsprogs до
последней версии и попробуйте снова.

Прекращение поддержки будет происходить в две части. Поддержку монтирования
файловых систем V4 теперь можно отключить во время сборки ядра через параметр
Kconfig. Эти параметры были изменены на значение по умолчанию «no» в сентябре
2025 года. В сентябре 2030 года поддержка будет полностью удалена из кодовой
базы.

Примечание: дистрибьюторы могут принять решение отозвать поддержку формата V4
раньше указанных выше дат.

sysctls
=======

Для файловой системы XFS доступны следующие sysctl:

  fs.xfs.stats_clear		(Min: 0  Default: 0  Max: 1)
	Установка этого параметра в "1" очищает накопленную статистику XFS
	в /proc/fs/xfs/stat. Затем он сразу же сбрасывается в "0".

  fs.xfs.xfssyncd_centisecs	(Min: 100  Default: 3000  Max: 720000)
	Интервал, с которым файловая система сбрасывает метаданные
	на диск и запускает внутренние процедуры очистки кэша.

  fs.xfs.filestream_centisecs	(Min: 1  Default: 3000  Max: 360000)
	Интервал, с которым файловая система устаревает ссылки кэша filestreams
	и возвращает истёкшие по времени AG обратно в пул свободных потоков.

  fs.xfs.speculative_prealloc_lifetime
	(Units: seconds   Min: 1  Default: 300  Max: 86400)
	Интервал, с которым выполняется фоновое сканирование inode
	с неиспользуемым спекулятивным предварительным выделением. Сканирование
	удаляет неиспользуемое предварительное выделение из чистых inode и
	освобождает неиспользуемое пространство обратно в пул свободного.

  fs.xfs.error_level		(Min: 0  Default: 3  Max: 11)
	Регулятор громкости для отчётов об ошибках при возникновении внутренних
	ошибок. Это, например, генерирует подробные сообщения и трассировки
	для остановок файловой системы. Текущие пороговые значения:

		XFS_ERRLEVEL_OFF:       0
		XFS_ERRLEVEL_LOW:       1
		XFS_ERRLEVEL_HIGH:      5

  fs.xfs.panic_mask		(Min: 0  Default: 0  Max: 511)
	Заставляет определённые состояния ошибок вызывать BUG(). Значение — это
	битовая маска; объедините через OR теги, представляющие ошибки, которые
	должны вызывать panic:

		XFS_NO_PTAG                     0
		XFS_PTAG_IFLUSH                 0x00000001
		XFS_PTAG_LOGRES                 0x00000002
		XFS_PTAG_AILDELETE              0x00000004
		XFS_PTAG_ERROR_REPORT           0x00000008
		XFS_PTAG_SHUTDOWN_CORRUPT       0x00000010
		XFS_PTAG_SHUTDOWN_IOERROR       0x00000020
		XFS_PTAG_SHUTDOWN_LOGERROR      0x00000040
		XFS_PTAG_FSBLOCK_ZERO           0x00000080
		XFS_PTAG_VERIFIER_ERROR         0x00000100

	Этот параметр предназначен только для отладки.

  fs.xfs.inherit_sync		(Min: 0  Default: 1  Max: 1)
	Установка этого параметра в "1" приведёт к тому, что флаг "sync",
	установленный командой chattr из **xfs_io(8)** на каталоге, будет
	наследоваться файлами в этом каталоге.

  fs.xfs.inherit_nodump		(Min: 0  Default: 1  Max: 1)
	Установка этого параметра в "1" приведёт к тому, что флаг "nodump",
	установленный командой chattr из **xfs_io(8)** на каталоге, будет
	наследоваться файлами в этом каталоге.

  fs.xfs.inherit_noatime	(Min: 0  Default: 1  Max: 1)
	Установка этого параметра в "1" приведёт к тому, что флаг "noatime",
	установленный командой chattr из **xfs_io(8)** на каталоге, будет
	наследоваться файлами в этом каталоге.

  fs.xfs.inherit_nosymlinks	(Min: 0  Default: 1  Max: 1)
	Установка этого параметра в "1" приведёт к тому, что флаг "nosymlinks",
	установленный командой chattr из **xfs_io(8)** на каталоге, будет
	наследоваться файлами в этом каталоге.

  fs.xfs.inherit_nodefrag	(Min: 0  Default: 1  Max: 1)
	Установка этого параметра в "1" приведёт к тому, что флаг "nodefrag",
	установленный командой chattr из **xfs_io(8)** на каталоге, будет
	наследоваться файлами в этом каталоге.

  fs.xfs.rotorstep		(Min: 1  Default: 1  Max: 256)
	В режиме выделения "inode32" этот параметр определяет, сколько
	файлов распределитель пытается выделить в одной группе выделения,
	прежде чем перейти к следующей группе выделения. Цель состоит
	в том, чтобы управлять скоростью, с которой распределитель перемещается
	между группами выделения при выделении экстентов для новых файлов.

Обработка ошибок
================

XFS может вести себя по-разному в зависимости от типа ошибки, обнаруженной
во время её работы. Реализация вводит следующие понятия для обработчика
ошибок:

 -скорость отказа:
	Определяет, насколько быстро XFS должна распространять ошибку вверх, когда
	конкретная ошибка обнаружена во время работы файловой системы. Она может
	распространять её немедленно, после заданного числа повторных попыток,
	после установленного периода времени или просто повторять вечно.

 -классы ошибок:
	Указывает подсистему, к которой будет применяться конфигурация ошибок,
	например ввод-вывод метаданных или выделение памяти. Разные подсистемы
	будут иметь разные обработчики ошибок, поведение которых можно настроить.

 -обработчики ошибок:
	Определяет поведение для конкретной ошибки.

Поведение файловой системы во время ошибки можно задать через файлы ``sysfs``.
Каждый обработчик ошибок работает независимо — первое условие, выполненное
обработчиком ошибок для конкретного класса, приведёт к распространению ошибки,
а не к её сбросу и повторной попытке.

Действие, предпринимаемое файловой системой при распространении ошибки,
зависит от контекста — оно может вызвать остановку в случае неустранимой
ошибки, может быть передано обратно в пространство пользователя или даже может
быть проигнорировано, потому что с ошибкой нельзя сделать ничего полезного и
некому о ней сообщить (например, во время размонтирования).

Конфигурационные файлы организованы в следующую иерархию для каждой
смонтированной файловой системы:

  /sys/fs/xfs/<dev>/error/<class>/<error>/

Где:
  <dev>
	Короткое имя устройства смонтированной файловой системы. Это то же имя
	устройства, которое появляется в сообщениях об ошибках ядра XFS как
	"XFS(<dev>): ..."

  <class>
	Подсистема, к которой принадлежит конфигурация ошибки. По состоянию на 4.9
	определены следующие классы:

		- "metadata": применяется к вводу-выводу записи буфера метаданных

  <error>
	Конфигурации отдельных обработчиков ошибок.


Каждая файловая система имеет "глобальные" параметры конфигурации ошибок,
определённые в её каталоге верхнего уровня:

  /sys/fs/xfs/<dev>/error/

  fail_at_unmount		(Min:  0  Default:  1  Max: 1)
	Определяет поведение файловой системы при ошибке во время размонтирования.

	Если установлено в значение 1, XFS переопределит все прочие конфигурации
	ошибок во время размонтирования и заменит их характеристиками "немедленный
	отказ". То есть никаких повторных попыток, никакого таймаута повторных
	попыток. Это всегда позволит размонтированию завершиться успешно при
	наличии постоянных ошибок.

	Если установлено в 0, настроенное поведение повторных попыток будет
	продолжаться до тех пор, пока не будут исчерпаны все повторные попытки
	и/или таймауты. Это задержит завершение размонтирования при наличии
	постоянных ошибок и может вообще помешать файловой системе полностью
	размонтироваться в случае конфигураций обработчиков "повторять вечно".

	Примечание: нет гарантии, что fail_at_unmount можно установить, пока
	размонтирование уже выполняется. Возможно, что записи ``sysfs`` будут
	удалены размонтируемой файловой системой до того, как конфигурация
	обработчика ошибок "повторять вечно" приведёт к зависанию размонтирования,
	и поэтому файловая система должна быть настроена соответствующим образом
	до начала размонтирования, чтобы предотвратить зависания размонтирования.

Каждая файловая система имеет специфические обработчики класса ошибок, которые
определяют поведение распространения ошибок для конкретных ошибок. Также
определён обработчик "по умолчанию", который определяет поведение для всех
ошибок, не имеющих определённых для них обработчиков. Когда для одной ошибки
настроено несколько ограничений на повторные попытки, первая конфигурация
повторных попыток, срок действия которой истечёт, приведёт к распространению
ошибки. Конфигурации обработчиков находятся в каталоге:

  /sys/fs/xfs/<dev>/error/<class>/<error>/

  max_retries			(Min: -1  Default: Varies  Max: INTMAX)
	Определяет допустимое число повторных попыток конкретной ошибки, прежде
	чем файловая система распространит ошибку. Счётчик повторных попыток для
	данного контекста ошибки (например, конкретного буфера метаданных)
	сбрасывается каждый раз, когда происходит успешное завершение операции.

	Установка значения в "-1" заставит XFS повторять вечно для этой
	конкретной ошибки.

	Установка значения в "0" заставит XFS немедленно завершиться отказом, когда
	сообщается о конкретной ошибке.

	Установка значения в "N" (где 0 < N < Max) заставит XFS повторить
	операцию "N" раз перед распространением ошибки.

  retry_timeout_seconds		(Min:  -1  Default:  Varies  Max: 1 day)
	Определяет промежуток времени (в секундах), в течение которого файловой
	системе разрешено повторять свои операции при обнаружении конкретной
	ошибки.

	Установка значения в "-1" позволит XFS повторять вечно для этой
	конкретной ошибки.

	Установка значения в "0" заставит XFS немедленно завершиться отказом, когда
	сообщается о конкретной ошибке.

	Установка значения в "N" (где 0 < N < Max) позволит XFS повторять
	операцию в течение до "N" секунд перед распространением ошибки.

**Примечание:** поведение по умолчанию для конкретного обработчика ошибок
зависит как от класса, так и от контекста ошибки. Например, значения по
умолчанию для "metadata/ENODEV" равны "0", а не "-1", так что этот обработчик
ошибок по умолчанию имеет поведение "немедленный отказ". Это сделано потому,
что ENODEV — фатальная, неустранимая ошибка, сколько бы раз ни повторялся
ввод-вывод метаданных.

Параллелизм рабочих очередей
============================

XFS использует рабочие очереди ядра для распараллеливания процессов обновления
метаданных. Это позволяет ей воспользоваться преимуществами оборудования
хранения, способного обслуживать множество операций ввода-вывода одновременно.
Этот интерфейс раскрывает внутренние детали реализации XFS и поэтому явно не
является частью какой-либо гарантии API/ABI пространства пользователя, которую
ядро может давать пространству пользователя. Это недокументированные
особенности обобщённой реализации рабочих очередей, которую XFS использует для
параллелизма, и они предоставлены здесь исключительно в целях диагностики и
настройки и могут измениться в любой момент в будущем.

Регуляторы управления рабочими очередями файловой системы организованы по
выполняемой задаче и короткому имени устройства данных. Все они находятся в:

  /sys/bus/workqueue/devices/${task}!${device}

================  ===========
  Task            Description
================  ===========
  xfs_iwalk-$pid  Сканирования inode всей файловой системы. В настоящее время
                  ограничены quotacheck во время монтирования.
  xfs-gc          Фоновая сборка мусора дискового пространства, которое было
                  спекулятивно выделено за пределами EOF или для промежуточных
                  операций copy on write.
================  ===========

Например, регуляторы для рабочей очереди quotacheck для /dev/nvme0n1 будут
находиться в /sys/bus/workqueue/devices/xfs_iwalk-1111!nvme0n1/.

Интересные регуляторы для рабочих очередей XFS следующие:

============     ===========
  Knob           Description
============     ===========
  max_active     Максимальное число фоновых потоков, которые могут быть
                 запущены для выполнения работы.
  cpumask        CPU, на которых потокам разрешено выполняться.
  nice           Относительный приоритет планирования потоков. Это те же
                 уровни nice, которые могут применяться к процессам
                 пространства пользователя.
============     ===========

Зонированные файловые системы
=============================

Для зонированных файловых систем следующие атрибуты раскрываются в:

  /sys/fs/xfs/<dev>/zoned/

  max_open_zones		(Min:  1  Default:  Varies  Max:  UINTMAX)
	Этот атрибут только для чтения раскрывает максимальное число открытых зон,
	доступных для размещения данных. Значение определяется во время
	монтирования и ограничено возможностями нижележащего зонированного
	устройства, размером файловой системы и параметром монтирования
	max_open_zones.

  zonegc_low_space		(Min:  0  Default:  0  Max:  100)
	Задаёт процент того, какую часть неиспользуемого пространства GC должен
	держать доступной для записи. Высокое значение освободит больше
	пространства, занятого неиспользуемыми блоками, создавая больший буфер
	против всплесков записи ценой увеличения write amplification.
	Независимо от этого значения, сборка мусора всегда будет стремиться
	освободить минимальное количество блоков, чтобы держать max_open_zones
	открытыми в целях размещения данных.
