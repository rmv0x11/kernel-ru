================================================================
Как проверять ошибки и выполнять бисекцию (bisect) регрессий
================================================================

Этот документ описывает, как проверить, проявляется ли некоторая проблема ядра
Linux в коде, который в настоящее время поддерживается разработчиками, а затем
поясняет, как локализовать изменение, вызвавшее проблему, если это регрессия
(например, она не возникала в более ранних версиях).

Текст ориентирован на людей, запускающих ядра из распространённых дистрибутивов
Linux на обычном оборудовании, которые хотят сообщить об ошибке ядра разработчикам
апстрима (upstream) Linux. Несмотря на это, инструкции одинаково хорошо подходят и
пользователям, уже знакомым со сборкой собственных ядер: они помогают избежать
ошибок, которые иногда допускают даже опытные разработчики.

..
   Примечание: если вы видите это примечание, значит вы читаете исходный файл
   текста. Возможно, вам стоит переключиться на отрендеренную версию: с ней
   гораздо проще читать этот документ и перемещаться по нему — особенно когда вы
   хотите что-то найти в справочном разделе, а затем вернуться туда, где остановились.
..
   Самую свежую отрендеренную версию этого текста можно найти здесь:
   https://docs.kernel.org/admin-guide/verify-bugs-and-bisect-regressions.html

Суть процесса (он же «TL;DR»)
=============================

*[Если вы только начинаете осваивать сборку или бисекцию Linux, пропустите этот
раздел и перейдите к* ':ref:`пошаговому руководству <introguide_bissbs>`' *ниже.
В нём используются те же команды, что и в этом разделе, но описываются они кратко.
Тем не менее шаги легко выполнять, а вместе с сопроводительными записями в
справочном разделе упоминается множество альтернатив, подводных камней и
дополнительных аспектов, любой из которых может оказаться существенным именно в
вашем случае.]*

**Если вы хотите проверить, присутствует ли ошибка в коде, поддерживаемом в
настоящее время разработчиками**, выполните только *подготовку* и *сегмент 1*; при
этом считайте «рабочим» ядром самое новое ядро Linux, которым вы регулярно
пользуетесь. В следующем примере предполагается, что это 6.0, поэтому его исходники
будут использоваться для подготовки файла .config.

**Если вы столкнулись с регрессией**, выполняйте шаги как минимум до конца
*сегмента 2*. После этого вы можете отправить предварительный отчёт — или
продолжить с *сегмента 3*, в котором описывается, как выполнить бисекцию,
необходимую для полноценного отчёта о регрессии. В следующем примере
предполагается, что 6.0.13 — «рабочее» ядро, а 6.1.5 — первое «сломанное», поэтому
6.0 будет считаться «хорошим» выпуском и использоваться для подготовки файла .config.

* **Подготовка**: настройте всё необходимое для сборки собственных ядер::

    # * Удалите всё ПО, которое зависит от внешне сопровождаемых модулей ядра
    #   или собирает какие-либо из них автоматически при загрузке.
    # * Убедитесь, что Secure Boot разрешает загрузку самостоятельно собранных ядер Linux.
    # * Если вы ещё не работаете в «рабочем» ядре, перезагрузитесь в него.
    # * Установите компиляторы и всё прочее, что нужно для сборки Linux.
    # * Обеспечьте 15 гигабайт свободного места в своём домашнем каталоге.
    git clone -o mainline --no-checkout \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git ~/linux/
    cd ~/linux/
    git remote add -t master stable \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git
    git switch --detach v6.0
    # * Подсказка: если вы использовали существующий клон, убедитесь, что рядом нет устаревшего .config.
    make olddefconfig
    # * Убедитесь, что предыдущая команда выбрала .config «рабочего» ядра.
    # * Подключите внешнее оборудование (USB-накопители, токены, ...), запустите ВМ, поднимите
    #   VPN, смонтируйте сетевые шары и кратко попробуйте сломанную функциональность.
    yes '' | make localmodconfig
    ./scripts/config --set-str CONFIG_LOCALVERSION '-local'
    ./scripts/config -e CONFIG_LOCALVERSION_AUTO
    # * Примечание: при нехватке места на диске см. в руководстве альтернативный вариант:
    ./scripts/config -d DEBUG_INFO_NONE -e KALLSYMS_ALL -e DEBUG_KERNEL \
      -e DEBUG_INFO -e DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT -e KALLSYMS
    # * Подсказка: на этом этапе вы можете захотеть скорректировать конфигурацию сборки;
    #   это придётся сделать, если вы работаете в Debian.
    make olddefconfig
    cp .config ~/kernel-config-working

* **Сегмент 1**: соберите ядро из последней кодовой базы mainline.

  Это, среди прочего, проверяет, не была ли проблема уже исправлена и каким
  разработчикам впоследствии нужно сообщить о проблеме; в случае регрессии это
  исключает изменение .config как первопричину проблемы.

  a) Получение последнего кода mainline::

       cd ~/linux/
       git switch --discard-changes --detach mainline/master

  b) Соберите, установите и загрузите ядро::

       cp ~/kernel-config-working .config
       make olddefconfig
       make -j $(nproc --all)
       # * Убедитесь, что достаточно места на диске для размещения ещё одного ядра:
       df -h /boot/ /lib/modules/
       # * Примечание: на Arch Linux, его производных и некоторых других дистрибутивах
       #   следующие команды не сделают ничего или выполнят лишь часть
       #   работы. Подробности см. в пошаговом руководстве.
       sudo make modules_install
       command -v installkernel && sudo make install
       # * Проверьте, сколько места реально требует ваше самостоятельно собранное ядро, что
       #   позволит вам позже делать более точные оценки:
       du -ch /boot/*$(make -s kernelrelease)* | tail -n 1
       du -sh /lib/modules/$(make -s kernelrelease)/
       # * Подсказка: вывод следующей команды поможет вам выбрать
       #   правильное ядро в загрузочном меню:
       make -s kernelrelease | tee -a ~/kernels-built
       reboot
       # * После загрузки убедитесь, что работаете именно в только что собранном ядре,
       #   проверив, совпадает ли вывод следующих двух команд:
       tail -n 1 ~/kernels-built
       uname -r
       cat /proc/sys/kernel/tainted

  c) Проверьте, возникает ли проблема и с этим ядром.

* **Сегмент 2**: убедитесь, что «хорошее» ядро также является «рабочим».

  Это, среди прочего, проверяет, что урезанный файл .config действительно
  работает нормально, поскольку в противном случае бисекция с ним была бы пустой
  тратой времени:

  a) Начните с получения исходников «хорошей» версии::

       cd ~/linux/
       git switch --discard-changes --detach v6.0

  b) Соберите, установите и загрузите ядро, как описано ранее в *сегменте 1,
     раздел b* — только можете пропустить команды «du», поскольку у вас уже есть
     приблизительная оценка.

  c) Убедитесь, что функциональность, которая дала регрессию в «сломанном» ядре,
     действительно работает в этом ядре.

* **Сегмент 3**: выполните и проверьте бисекцию.

  a) Получите исходники для своей «плохой» версии::

       git remote set-branches --add stable linux-6.1.y
       git fetch stable

  b) Инициализируйте бисекцию::

       cd ~/linux/
       git bisect start
       git bisect good v6.0
       git bisect bad v6.1.5

  c) Соберите, установите и загрузите ядро, как описано ранее в *сегменте 1,
     раздел b*.

     Если сборка или загрузка ядра завершилась неудачей по не связанным с проблемой
     причинам, выполните ``git bisect skip``. Во всех остальных случаях проверьте,
     работает ли регрессировавшая функциональность в только что собранном ядре. Если
     работает — сообщите об этом Git, выполнив ``git bisect good``; если нет —
     выполните ``git bisect bad``.

     Все три команды заставят Git переключиться (check out) на другой коммит; затем
     заново выполните этот шаг (то есть соберите, установите, загрузите и протестируйте
     ядро, после чего сообщите Git результат). Повторяйте это снова и снова, пока Git
     не покажет, какой коммит всё сломал. Если в ходе этого процесса у вас закончится
     место на диске, обратитесь к разделу «Дополнительные задачи: очистка во время и
     после процесса» ниже.

  d) Завершив бисекцию, отложите кое-что в сторону::

       cd ~/linux/
       git bisect log > ~/bisect-log
       cp .config ~/bisection-config-culprit
       git bisect reset

  e) Попробуйте проверить результат бисекции::

       git switch --discard-changes --detach mainline/master
       git revert --no-edit cafec0cacaca0
       cp ~/kernel-config-working .config
       ./scripts/config --set-str CONFIG_LOCALVERSION '-local-cafec0cacaca0-reverted'

    Это необязательно, так как некоторые коммиты откатить невозможно. Но если вторая
    команда отработала безупречно, соберите, установите и загрузите ещё одно ядро;
    только на этот раз пропустите первую команду, копирующую базовый файл .config,
    поскольку это уже сделано.

* **Дополнительные задачи**: очистка во время и после процесса.

  a) Чтобы не остаться без места на диске во время бисекции, вам может потребоваться
     удалить некоторые из собранных ранее ядер. Скорее всего, те ядра, которые вы
     собрали в сегментах 1 и 2, вы захотите сохранить на какое-то время, но ядра,
     протестированные в ходе самой бисекции (сегмент 3 c), вам, скорее всего, больше
     не понадобятся. Перечислить их в порядке сборки можно так::

       ls -ltr /lib/modules/*-local*

    Чтобы, например, удалить ядро, идентифицирующее себя как
    «6.0-rc1-local-gcafec0cacaca0», используйте следующее::

       sudo rm -rf /lib/modules/6.0-rc1-local-gcafec0cacaca0
       sudo kernel-install -v remove 6.0-rc1-local-gcafec0cacaca0
       # * Примечание: в некоторых дистрибутивах kernel-install отсутствует
       #   или выполняет лишь часть работы.

  b) Если вы выполнили бисекцию и успешно проверили её результат, можете спокойно
     удалить все ядра, собранные в ходе самой бисекции (сегмент 3 c); ядра, собранные
     ранее и позднее, возможно, стоит сохранить на неделю-другую.

* **Необязательная задача**: протестировать отладочный патч или предложенное
  исправление позже::

    git fetch mainline
    git switch --discard-changes --detach mainline/master
    git apply /tmp/foobars-proposed-fix-v1.patch
    cp ~/kernel-config-working .config
    ./scripts/config --set-str CONFIG_LOCALVERSION '-local-foobars-fix-v1'

  Соберите, установите и загрузите ядро, как описано в *сегменте 1, раздел b* — но
  на этот раз пропустите первую команду, копирующую конфигурацию сборки, поскольку
  это уже сделано.

Пошаговое руководство по проверке ошибок и бисекции регрессий
=============================================================

Это руководство описывает, как настроить собственные ядра Linux для исследования
ошибок или регрессий, о которых вы намерены сообщить. То, насколько далеко вам
следовать инструкциям, зависит от вашей проблемы:

Выполните все шаги до конца *сегмента 1*, чтобы **проверить, присутствует ли ваша
проблема ядра в коде, поддерживаемом разработчиками ядра Linux**. Если присутствует,
вы готовы сообщить об ошибке — если только она не отсутствовала в более ранних
версиях ядра, поскольку тогда вам стоит как минимум продолжить с *сегмента 2*, чтобы
**проверить, квалифицируется ли проблема как регрессия**, которая получает
приоритетную обработку. В зависимости от результата вы затем будете готовы сообщить
об ошибке или отправить предварительный отчёт о регрессии; вместо последнего вы
также могли бы сразу перейти к *сегменту 3* и **выполнить бисекцию** для полноценного
отчёта о регрессии, по которому разработчики обязаны действовать.

 :ref:`Подготовка: настройте всё необходимое для сборки собственных ядер <introprep_bissbs>`.

 :ref:`Сегмент 1: попробуйте воспроизвести проблему на последней кодовой базе <introlatestcheck_bissbs>`.

 :ref:`Сегмент 2: проверьте, нормально ли работают собираемые вами ядра <introworkingcheck_bissbs>`.

 :ref:`Сегмент 3: выполните бисекцию и проверьте результат <introbisect_bissbs>`.

 :ref:`Дополнительные задачи: очистка во время и после следования этому руководству <introclosure_bissbs>`.

 :ref:`Необязательные задачи: тестирование откатов, патчей или более поздних версий <introoptional_bissbs>`.

Шаги в каждом сегменте иллюстрируют важные аспекты процесса, а исчерпывающий
справочный раздел содержит дополнительные подробности почти для каждого из шагов.
Справочный раздел иногда также описывает альтернативные подходы, подводные камни, а
также проблемы, которые могут возникнуть на конкретном шаге — и как снова сдвинуть
дело с мёртвой точки.

Дополнительные сведения о том, как сообщать о проблемах или регрессиях ядра Linux,
см. в Documentation/admin-guide/reporting-issues.rst, который работает совместно с
этим документом. Среди прочего там объясняется, почему ошибки нужно проверять на
последнем ядре «mainline» (например, версиях вроде 6.0, 6.1-rc1 или 6.1-rc6), даже
если вы столкнулись с проблемой в ядре из серии «stable/longterm» (скажем, 6.0.13).

Для пользователей, столкнувшихся с регрессией, тот документ также объясняет, почему
отправка предварительного отчёта после сегмента 2 может быть разумной, поскольку
регрессия и её виновник могут быть уже известны. Дополнительные сведения о том, что
именно квалифицируется как регрессия, см. в
Documentation/admin-guide/reporting-regressions.rst.

Если при следовании этому руководству у вас возникнут какие-либо проблемы или
появятся идеи о том, как его улучшить,
:ref:`пожалуйста, сообщите об этом разработчикам ядра <submit_improvements_vbbr>`.

Подготовка: настройте всё необходимое для сборки собственных ядер
-----------------------------------------------------------------

Следующие шаги закладывают основу для всех дальнейших задач.

Примечание: инструкции предполагают, что вы выполняете сборку и тестирование на
одной и той же машине; если вы хотите компилировать ядро на другой системе,
обратитесь к :ref:`Сборка ядер на другой машине <buildhost_bis>` ниже.

.. _backup_bissbs:

* Создайте свежую резервную копию и держите под рукой средства восстановления и
  ремонта системы — просто на случай маловероятной ситуации, когда что-то пойдёт
  не так.

  [:ref:`подробности <backup_bisref>`]

.. _vanilla_bissbs:

* Удалите всё ПО, которое зависит от разрабатываемых сторонними сторонами драйверов
  ядра или собирает их автоматически. Сюда входят, помимо прочего, DKMS, openZFS,
  VirtualBox и графические драйверы Nvidia (включая GPL-модуль ядра).

  [:ref:`подробности <vanilla_bisref>`]

.. _secureboot_bissbs:

* На платформах с «Secure Boot» или подобными решениями подготовьте всё необходимое,
  чтобы гарантировать, что система разрешит загрузку вашего самостоятельно
  скомпилированного ядра. Самый быстрый и простой способ добиться этого на обычных
  системах x86 — отключить такие технологии в утилите настройки BIOS; в качестве
  альтернативы снимите их ограничения с помощью процесса, инициируемого командой
  ``mokutil --disable-validation``.

  [:ref:`подробности <secureboot_bisref>`]

.. _rangecheck_bissbs:

* Определите версии ядра, которые на протяжении всего этого руководства считаются
  «хорошей» и «плохой»:

  * Вы следуете этому руководству, чтобы проверить, присутствует ли ошибка в коде,
    которым занимаются основные разработчики? Тогда считайте «хорошей» версию самого
    нового ядра, которым вы в настоящее время регулярно пользуетесь (например, 6.0,
    6.0.13 или 6.1-rc2).

  * Вы столкнулись с регрессией, например что-то сломалось или стало работать хуже
    после перехода на более новую версию ядра? В этом случае всё зависит от диапазона
    версий, в котором появилась проблема:

    * Регрессия возникла при обновлении с выпуска stable/longterm (скажем, 6.0.13) на
      более новую серию mainline (например, 6.1-rc7 или 6.1) или версию
      stable/longterm, основанную на ней (скажем, 6.1.5)? Тогда считайте «хорошей»
      версией выпуск mainline, на котором основано ваше рабочее ядро (например, 6.0),
      а первую сломанную версию — «плохой» (например, 6.1-rc7, 6.1 или 6.1.5).
      Обратите внимание, на данном этапе лишь предполагается, что 6.0 в порядке; эта
      гипотеза будет проверена в сегменте 2.

    * Регрессия возникла при переходе с одной версии mainline (скажем, 6.0) на более
      позднюю (например, 6.1-rc1) или выпуск stable/longterm, основанный на ней
      (скажем, 6.1.5)? Тогда считайте «хорошей» последнюю рабочую версию (например,
      6.0), а «плохой» — первую сломанную (например, 6.1-rc1 или 6.1.5).

    * Регрессия возникла при обновлении внутри серии stable/longterm (скажем, с
      6.0.13 на 6.0.15)? Тогда считайте эти версии «хорошей» и «плохой» (например,
      6.0.13 и 6.0.15), так как вам нужно выполнять бисекцию внутри этой серии.

  *Обратите внимание, не путайте «хорошую» версию с «рабочим» ядром; последний термин
  на протяжении этого руководства будет относиться к последнему ядру, которое
  работало нормально.*

  [:ref:`подробности <rangecheck_bisref>`]

.. _bootworking_bissbs:

* Загрузитесь в «рабочее» ядро и кратко воспользуйтесь предположительно сломанной
  функциональностью.

  [:ref:`подробности <bootworking_bisref>`]

.. _diskspace_bissbs:

* Обеспечьте достаточно свободного места для сборки Linux. Обычно достаточно 15
  гигабайт в вашем домашнем каталоге. Если у вас доступно меньше, обязательно
  обратите внимание на дальнейшие шаги о получении исходников Linux и обращении с
  отладочными символами: оба описывают подходы, уменьшающие объём занимаемого места,
  что должно позволить вам выполнить эти задачи примерно при 4 гигабайтах свободного
  места.

  [:ref:`подробности <diskspace_bisref>`]

.. _buildrequires_bissbs:

* Установите всё ПО, необходимое для сборки ядра Linux. Часто вам понадобятся:
  'bc', 'binutils' ('ld' и др.), 'bison', 'flex', 'gcc', 'git', 'openssl',
  'pahole', 'perl', а также заголовочные файлы разработки для 'libelf' и 'openssl'.
  В справочном разделе показано, как быстро установить их на различных популярных
  дистрибутивах Linux.

  [:ref:`подробности <buildrequires_bisref>`]

.. _sources_bissbs:

* Получите исходники mainline Linux; затем перейдите в каталог, содержащий их, так
  как все дальнейшие команды в этом руководстве предполагается выполнять оттуда.

  *Обратите внимание, ниже описывается, как получить исходники с помощью полного
  клона mainline, который загружает около 2,75 ГБайт по состоянию на начало 2024
  года. В* :ref:`справочном разделе описаны две альтернативы <sources_bisref>` *:
  одна загружает менее 500 МБайт, другая лучше работает при ненадёжных
  интернет-соединениях.*

  Выполните следующую команду, чтобы получить свежую кодовую базу mainline,
  одновременно подготовив всё для добавления ветвей серий stable/longterm позже::

    git clone -o mainline --no-checkout \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git ~/linux/
    cd ~/linux/
    git remote add -t master stable \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git

  [:ref:`подробности <sources_bisref>`]

.. _stablesources_bissbs:

* Является ли одна из версий, которые вы ранее определили как «хорошую» или «плохую»,
  выпуском stable или longterm (скажем, 6.1.5)? Тогда загрузите код для серии, к
  которой она относится (в этом примере — 'linux-6.1.y')::

    git remote set-branches --add stable linux-6.1.y
    git fetch stable

.. _oldconfig_bissbs:

* Начните подготовку конфигурации сборки ядра (файла '.config').

  Прежде чем сделать это, убедитесь, что вы по-прежнему работаете в «рабочем» ядре, в
  которое вам велел загрузиться один из предыдущих шагов; если вы не уверены,
  проверьте текущий идентификатор kernelrelease с помощью ``uname -r``.

  Затем переключитесь (check out) на исходный код версии, ранее определённой как
  «хорошая». В следующем примере команды предполагается, что это 6.0; обратите
  внимание, что номеру версии в этой и всех последующих командах Git должна
  предшествовать буква 'v'::

    git switch --discard-changes --detach v6.0

  Теперь создайте файл конфигурации сборки::

    make olddefconfig

  Затем сценарии сборки ядра попытаются найти файл конфигурации сборки для
  работающего ядра, а затем скорректировать его под нужды исходников ядра, на которые
  вы переключились. При этом будет выведено несколько строк, которые вам нужно
  проверить.

  Найдите строку, начинающуюся с '# using defaults found in'. За ней должен следовать
  путь к файлу в '/boot/', содержащему идентификатор выпуска вашего текущего рабочего
  ядра. Если же строка продолжается чем-то вроде
  'arch/x86/configs/x86_64_defconfig', значит инфраструктура сборки не смогла найти
  файл .config для вашего работающего ядра — в этом случае вам придётся вручную
  поместить его туда, как объясняется в справочном разделе.

  Если такой строки найти не удаётся, поищите строку, содержащую '# configuration
  written to .config'. Если так и есть, значит у вас остался устаревший файл
  конфигурации сборки. Если вы не намерены использовать его, удалите его; затем снова
  выполните 'make olddefconfig' и проверьте, выбрал ли он теперь правильный файл
  конфигурации в качестве основы.

  [:ref:`подробности <oldconfig_bisref>`]

.. _localmodconfig_bissbs:

* Отключите любые модули ядра, явно лишние для вашей конфигурации. Это необязательно,
  но особенно разумно для бисекций, так как это колоссально ускоряет процесс сборки —
  по крайней мере если файл .config, выбранный на предыдущем шаге, не был уже подогнан
  под ваши нужды и нужды вашего оборудования, и в этом случае этот шаг следует
  пропустить.

  Чтобы подготовиться к урезанию, подключите внешнее оборудование, которым вы
  периодически пользуетесь (USB-накопители, токены, ...), быстро запустите ВМ и
  поднимите VPN. И если вы перезагружались с тех пор, как начали следовать этому
  руководству, убедитесь, что после запуска системы вы попробовали воспользоваться
  функциональностью, вызывающей проблему. Только после этого урежьте свой .config::

     yes '' | make localmodconfig

  В этом есть подвох, на который уже намекали слова «явно» в начальном предложении
  этого шага и инструкции по подготовке:

  Цель 'localmodconfig' легко отключает модули ядра для функциональности, используемой
  лишь время от времени, — например, модули для внешней периферии, ещё не подключённой
  с момента загрузки, ещё не задействованного ПО виртуализации, VPN-туннелей и
  некоторых других вещей. Это потому, что некоторые задачи опираются на модули ядра,
  которые Linux загружает только тогда, когда вы впервые выполняете задачи вроде
  упомянутых выше.

  Этот недостаток localmodconfig — не то, из-за чего стоит терять сон, но то, о чём
  стоит помнить: если что-то ведёт себя неправильно в ядрах, собранных в ходе этого
  руководства, скорее всего, причина именно в этом. Вы можете снизить или почти
  устранить этот риск с помощью приёмов, изложенных в справочном разделе; но когда вы
  собираете ядро лишь для целей быстрого тестирования, на это обычно не стоит тратить
  много усилий, пока оно загружается и позволяет нормально протестировать
  функциональность, вызывающую проблему.

  [:ref:`подробности <localmodconfig_bisref>`]

.. _tagging_bissbs:

* Обеспечьте, чтобы все собираемые вами ядра были чётко идентифицируемы с помощью
  специального тега и уникального номера версии::

    ./scripts/config --set-str CONFIG_LOCALVERSION '-local'
    ./scripts/config -e CONFIG_LOCALVERSION_AUTO

  [:ref:`подробности <tagging_bisref>`]

.. _debugsymbols_bissbs:

* Решите, как обращаться с отладочными символами.

  В контексте этого документа часто разумно включить их, так как есть приличная
  вероятность, что вам понадобится декодировать трассировку стека из 'panic', 'Oops',
  'warning' или 'BUG'::

    ./scripts/config -d DEBUG_INFO_NONE -e KALLSYMS_ALL -e DEBUG_KERNEL \
      -e DEBUG_INFO -e DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT -e KALLSYMS

  Но если вы крайне ограничены в месте на накопителе, возможно, вы захотите вместо
  этого отключить отладочные символы::

    ./scripts/config -d DEBUG_INFO -d DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT \
      -d DEBUG_INFO_DWARF4 -d DEBUG_INFO_DWARF5 -e CONFIG_DEBUG_INFO_NONE

  [:ref:`подробности <debugsymbols_bisref>`]

.. _configmods_bissbs:

* Проверьте, не хотите ли вы или не нужно ли вам скорректировать некоторые другие
  параметры конфигурации ядра:

  * Вы работаете в Debian? Тогда вам стоит избежать известных проблем, выполнив
    дополнительные корректировки, описанные в справочном разделе.

    [:ref:`подробности <configmods_distros_bisref>`].

  * Если вы хотите повлиять на другие аспекты конфигурации, сделайте это сейчас с
    помощью предпочитаемого вами инструмента. Обратите внимание, чтобы использовать
    цели make вроде 'menuconfig' или 'nconfig', вам нужно установить файлы разработки
    ncurses; для 'xconfig' вам аналогично нужны заголовки Qt5 или Qt6.

    [:ref:`подробности <configmods_individual_bisref>`].

.. _saveconfig_bissbs:

* Повторно обработайте .config после последних корректировок и сохраните его в
  безопасном месте::

     make olddefconfig
     cp .config ~/kernel-config-working

  [:ref:`подробности <saveconfig_bisref>`]

.. _introlatestcheck_bissbs:

Сегмент 1: попробуйте воспроизвести проблему на последней кодовой базе
----------------------------------------------------------------------

Следующие шаги проверяют, возникает ли проблема в коде, поддерживаемом в настоящее
время разработчиками. Если вы столкнулись с регрессией, это также проверяет, что
проблема не вызвана каким-либо изменением .config, так как сообщение о проблеме в
этом случае было бы пустой тратой времени. [:ref:`подробности <introlatestcheck_bisref>`]

.. _checkoutmaster_bissbs:

* Переключитесь (check out) на последнюю кодовую базу Linux.

  * Ваши «хорошая» и «плохая» версии относятся к одной и той же серии stable или
    longterm? Тогда проверьте `главную страницу kernel.org <https://kernel.org/>`_:
    если она перечисляет выпуск из этой серии без тега '[EOL]', переключитесь на
    последнюю версию серии (в следующем примере — 'linux-6.1.y')::

      cd ~/linux/
      git switch --discard-changes --detach stable/linux-6.1.y

    Ваша серия не поддерживается, если она не указана в списке или несёт тег 'end of
    life'. В этом случае вы можете захотеть проверить, исправляет ли ошибку серия-
    преемник (скажем, linux-6.2.y) или mainline (см. следующий пункт).

  * Во всех остальных случаях выполните::

      cd ~/linux/
      git switch --discard-changes --detach mainline/master

  [:ref:`подробности <checkoutmaster_bisref>`]

.. _build_bissbs:

* Соберите образ и модули вашего первого ядра, используя подготовленный вами файл
  конфигурации::

    cp ~/kernel-config-working .config
    make olddefconfig
    make -j $(nproc --all)

  Если вы хотите упаковать своё ядро в виде файла deb, rpm или tar, см. в справочном
  разделе альтернативы, которые, очевидно, потребуют и других шагов для установки.

  [:ref:`подробности <build_bisref>`]

.. _install_bissbs:

* Установите только что собранное ядро.

  Прежде чем сделать это, подумайте о том, чтобы проверить, достаточно ли ещё места
  для него::

    df -h /boot/ /lib/modules/

  Пока что предположим, что хватит 150 МБайт в /boot/ и 200 в /lib/modules/; сколько
  места реально требуют ваши ядра, будет определено позже в ходе этого руководства.

  Теперь установите модули ядра и его образ, которые будут размещены параллельно
  ядрам вашего дистрибутива Linux::

    sudo make modules_install
    command -v installkernel && sudo make install

  Вторая команда в идеале позаботится о трёх шагах, необходимых на этом этапе:
  копировании образа ядра в /boot/, генерации initramfs и добавлении записи для обоих
  в конфигурацию загрузчика.

  К сожалению, некоторые дистрибутивы (среди них Arch Linux, его производные и многие
  неизменяемые дистрибутивы Linux) не выполнят ни одной из этих задач или выполнят
  лишь часть. Поэтому вам стоит проверить, обо всех ли из них позаботились, и вручную
  выполнить те, о которых не позаботились. В справочном разделе приводятся
  дополнительные подробности об этом; документация вашего дистрибутива тоже может
  помочь.

  Как только вы выясните, какие шаги нужны на этом этапе, подумайте о том, чтобы
  записать их: если вы будете собирать дополнительные ядра, как описано в сегментах 2
  и 3, вам придётся выполнять их снова после выполнения
  ``command -v installkernel [...]``.

  [:ref:`подробности <install_bisref>`]

.. _storagespace_bissbs:

* Если вы планируете следовать этому руководству дальше, проверьте, сколько места на
  накопителе занимают ядро, его модули и другие связанные файлы вроде initramfs::

    du -ch /boot/*$(make -s kernelrelease)* | tail -n 1
    du -sh /lib/modules/$(make -s kernelrelease)/

  Запишите или запомните эти два значения на потом: они позволят вам предотвратить
  случайную нехватку места на диске во время бисекции.

  [:ref:`подробности <storagespace_bisref>`]

.. _kernelrelease_bissbs:

* Покажите и сохраните идентификатор kernelrelease только что собранного ядра::

    make -s kernelrelease | tee -a ~/kernels-built

  Запомните этот идентификатор ненадолго, так как он поможет вам выбрать правильное
  ядро в загрузочном меню при перезапуске.

* Перезагрузитесь в только что собранное ядро. Чтобы убедиться, что вы действительно
  запустили именно то ядро, которое только что собрали, вы можете захотеть проверить,
  совпадает ли вывод этих команд::

    tail -n 1 ~/kernels-built
    uname -r

.. _tainted_bissbs:

* Проверьте, не пометило ли ядро себя как «tainted» (заражённое)::

    cat /proc/sys/kernel/tainted

  Если эта команда возвращает не '0', обратитесь к справочному разделу, так как
  причина этого может помешать вашему тестированию.

  [:ref:`подробности <tainted_bisref>`]

.. _recheckbroken_bissbs:

* Проверьте, возникает ли ваша ошибка в только что собранном ядре. Если нет, см.
  инструкции в справочном разделе, чтобы убедиться, что в ходе ваших тестов ничего
  не пошло не так.

  [:ref:`подробности <recheckbroken_bisref>`]

.. _recheckstablebroken_bissbs:

* Вы только что собрали ядро stable или longterm? И вам удалось воспроизвести
  регрессию с ним? Тогда вам следует протестировать и последнюю кодовую базу mainline,
  поскольку результат определяет, каким разработчикам нужно отправить отчёт об ошибке.

  Чтобы подготовиться к этому тесту, переключитесь (check out) на текущий mainline::

    cd ~/linux/
    git switch --discard-changes --detach mainline/master

  Теперь используйте переключённый код, чтобы собрать и установить ещё одно ядро с
  помощью команд, которые предыдущие шаги уже описали более подробно::

    cp ~/kernel-config-working .config
    make olddefconfig
    make -j $(nproc --all)
    # * Проверьте, хватает ли свободного места для размещения ещё одного ядра:
    df -h /boot/ /lib/modules/
    sudo make modules_install
    command -v installkernel && sudo make install
    make -s kernelrelease | tee -a ~/kernels-built
    reboot

  Подтвердите, что вы загрузили именно то ядро, которое намеревались запустить, и
  проверьте его статус tainted::

    tail -n 1 ~/kernels-built
    uname -r
    cat /proc/sys/kernel/tainted

  Теперь проверьте, проявляется ли проблема в этом ядре. Если да, то вам нужно
  сообщить об ошибке основным разработчикам; если нет, сообщите о ней команде stable.
  Подробности см. в Documentation/admin-guide/reporting-issues.rst.

  [:ref:`подробности <recheckstablebroken_bisref>`]

Вы следуете этому руководству, чтобы проверить, присутствует ли проблема в коде,
поддерживаемом в настоящее время разработчиками ядра Linux? Тогда на этом вы
закончили. Если позже вы захотите удалить только что собранное ядро, обратитесь к
:ref:`Дополнительные задачи: очистка во время и после следования этому руководству <introclosure_bissbs>`.

Если вы столкнулись с регрессией, двигайтесь дальше и выполните как минимум также
следующий сегмент.

.. _introworkingcheck_bissbs:

Сегмент 2: проверьте, нормально ли работают собираемые вами ядра
----------------------------------------------------------------

В случае регрессии вы теперь хотите убедиться, что подготовленный ранее урезанный
файл конфигурации работает как ожидается; в противном случае бисекция с этим файлом
.config была бы пустой тратой времени. [:ref:`подробности <introworkingcheck_bisref>`]

.. _recheckworking_bissbs:

* Соберите свой собственный вариант «рабочего» ядра и проверьте, работает ли с ним
  функциональность, давшая регрессию, как ожидается.

  Начните с переключения (check out) на исходники версии, ранее определённой как
  «хорошая» (здесь снова предполагается 6.0)::

    cd ~/linux/
    git switch --discard-changes --detach v6.0

  Теперь используйте переключённый код, чтобы сконфигурировать, собрать и установить
  ещё одно ядро с помощью команд, которые предыдущий подраздел объяснил более
  подробно::

    cp ~/kernel-config-working .config
    make olddefconfig
    make -j $(nproc --all)
    # * Проверьте, хватает ли свободного места для размещения ещё одного ядра:
    df -h /boot/ /lib/modules/
    sudo make modules_install
    command -v installkernel && sudo make install
    make -s kernelrelease | tee -a ~/kernels-built
    reboot

  Когда система загрузится, вы можете захотеть ещё раз убедиться, что запущенное вами
  ядро — это то, которое вы только что собрали::

    tail -n 1 ~/kernels-built
    uname -r

  Теперь проверьте, работает ли это ядро как ожидается; если нет, обратитесь к
  справочному разделу за дальнейшими инструкциями.

  [:ref:`подробности <recheckworking_bisref>`]

.. _introbisect_bissbs:

Сегмент 3: выполните бисекцию и проверьте результат
---------------------------------------------------

После того как обо всей подготовке и предосторожностях позаботились, вы теперь
готовы приступить к бисекции. Это заставит вас собрать довольно много ядер — обычно
около 15 в случае, когда вы столкнулись с регрессией при обновлении на более новую
серию (скажем, с 6.0.13 на 6.1.5). Но не беспокойтесь: благодаря созданной ранее
урезанной конфигурации сборки это работает гораздо быстрее, чем многие предполагают:
в целом в среднем компиляция каждого ядра на обычных машинах x86 часто будет занимать
всего около 10–15 минут.

.. _bisectstart_bissbs:

* Запустите бисекцию и сообщите Git о версиях, ранее определённых как «хорошая» (6.0
  в следующем примере команды) и «плохая» (6.1.5)::

    cd ~/linux/
    git bisect start
    git bisect good v6.0
    git bisect bad v6.1.5

  [:ref:`подробности <bisectstart_bisref>`]

.. _bisectbuild_bissbs:

* Теперь используйте код, который Git переключил (checked out), чтобы собрать,
  установить и загрузить ядро с помощью введённых ранее команд::

    cp ~/kernel-config-working .config
    make olddefconfig
    make -j $(nproc --all)
    # * Проверьте, хватает ли свободного места для размещения ещё одного ядра:
    df -h /boot/ /lib/modules/
    sudo make modules_install
    command -v installkernel && sudo make install
    make -s kernelrelease | tee -a ~/kernels-built
    reboot

  Если компиляция по какой-то причине завершится неудачей, выполните ``git bisect
  skip`` и заново запустите выполнение стека команд с начала.

  Если вы пропустили шаг «проверка последней кодовой базы» в руководстве, обратитесь
  к его описанию, чтобы понять, зачем здесь команды 'df [...]' и 'make -s
  kernelrelease [...]'.

  Важное примечание: последняя команда с этого момента будет печатать идентификаторы
  выпуска, которые могут показаться вам странными или ошибочными — но это не так,
  поскольку совершенно нормально видеть идентификаторы выпуска вроде
  '6.0-rc1-local-gcafec0cacaca0', если вы выполняете бисекцию, например, между
  версиями 6.1 и 6.2.

  [:ref:`подробности <bisectbuild_bisref>`]

.. _bisecttest_bissbs:

* Теперь проверьте, работает ли функциональность, давшая регрессию, в только что
  собранном ядре.

  Снова, возможно, вы захотите начать с того, чтобы убедиться, что загруженное вами
  ядро — это то, которое вы только что собрали::

    cd ~/linux/
    tail -n 1 ~/kernels-built
    uname -r

  Теперь проверьте, работает ли функциональность, давшая регрессию, в этой точке
  бисекции ядра. Если работает, выполните это::

    git bisect good

  Если не работает, выполните это::

    git bisect bad

  Будьте уверены в том, что вы сообщаете Git, поскольку даже одна ошибка здесь уведёт
  всю остальную бисекцию совершенно не в ту сторону.

  Пока бисекция продолжается, Git использует предоставленную вами информацию, чтобы
  найти и переключиться (check out) на другую точку бисекции для вашего тестирования.
  При этом он напечатает что-то вроде 'Bisecting: 675 revisions left to test after
  this (roughly 10 steps)', чтобы указать, сколько ещё изменений он ожидает
  протестировать. Теперь соберите и установите ещё одно ядро с помощью инструкций из
  предыдущего шага; затем снова следуйте инструкциям этого шага.

  Повторяйте это снова и снова, пока не завершите бисекцию — это произойдёт, когда
  Git после пометки изменения как 'good' или 'bad' напечатает что-то вроде
  'cafecaca0c0dacafecaca0c0dacafecaca0c0da is the first bad commit'; сразу после
  этого он покажет некоторые подробности о виновнике, включая описание патча этого
  изменения. Последнее может заполнить экран вашего терминала, поэтому вам, возможно,
  потребуется прокрутить вверх, чтобы увидеть сообщение, упоминающее виновника; в
  качестве альтернативы выполните ``git bisect log > ~/bisection-log``.

  [:ref:`подробности <bisecttest_bisref>`]

.. _bisectlog_bissbs:

* Сохраните журнал бисекции Git и текущий файл .config в безопасном месте, прежде чем
  велеть Git сбросить исходники в состояние до бисекции::

    cd ~/linux/
    git bisect log > ~/bisection-log
    cp .config ~/bisection-config-culprit
    git bisect reset

  [:ref:`подробности <bisectlog_bisref>`]

.. _revert_bissbs:

* Попробуйте откатить (revert) виновника поверх последнего mainline, чтобы
  посмотреть, исправит ли это вашу регрессию.

  Это необязательно, так как может оказаться невозможным или трудноосуществимым.
  Первое имеет место, если бисекция определила виновником коммит слияния (merge
  commit); второе случается, если другие изменения зависят от виновника. Но если
  откат удался, стоит собрать ещё одно ядро, так как это проверяет результат бисекции,
  который легко может оказаться сбитым с пути; кроме того, это даст разработчикам ядра
  знать, могут ли они устранить регрессию быстрым откатом.

  Начните с переключения (check out) на последнюю кодовую базу в зависимости от
  диапазона, в котором вы выполняли бисекцию:

  * Вы столкнулись с регрессией внутри серии stable/longterm (скажем, между 6.0.13 и
    6.0.15), которой нет в mainline? Тогда переключитесь на последнюю кодовую базу для
    затронутой серии так::

      git fetch stable
      git switch --discard-changes --detach linux-6.0.y

  * Во всех остальных случаях переключитесь на последний mainline::

      git fetch mainline
      git switch --discard-changes --detach mainline/master

    Если вы выполняли бисекцию регрессии внутри серии stable/longterm, которая также
    возникает в mainline, нужно сделать ещё одну вещь: найти идентификатор коммита
    (commit-id) в mainline. Для этого используйте команду вроде ``git show
    abcdcafecabcd``, чтобы просмотреть описание патча виновника. Ближе к началу будет
    строка, которая выглядит как 'commit cafec0cacaca0 upstream.' или 'Upstream
    commit cafec0cacaca0'; используйте этот commit-id в следующей команде, а не тот,
    который обвинила бисекция.

  Теперь попробуйте откатить виновника, указав его commit id::

    git revert --no-edit cafec0cacaca0

  Если это не удастся, прекратите попытки и переходите к следующему шагу; если же
  получилось, скорректируйте тег, чтобы облегчить идентификацию и предотвратить
  случайную перезапись другого ядра::

    cp ~/kernel-config-working .config
    ./scripts/config --set-str CONFIG_LOCALVERSION '-local-cafec0cacaca0-reverted'

  Соберите ядро с помощью знакомой последовательности команд, только без копирования
  базового .config поверх::

    make olddefconfig &&
    make -j $(nproc --all)
    # * Проверьте, хватает ли свободного места для размещения ещё одного ядра:
    df -h /boot/ /lib/modules/
    sudo make modules_install
    command -v installkernel && sudo make install
    make -s kernelrelease | tee -a ~/kernels-built
    reboot

  Теперь в последний раз проверьте, работает ли с этим ядром функциональность, из-за
  которой вы выполняли бисекцию: если всё прошло хорошо, она не должна показывать
  регрессию.

  [:ref:`подробности <revert_bisref>`]

Дополнительные задачи: очистка во время и после бисекции
--------------------------------------------------------

Во время и после следования этому руководству вы можете захотеть или вам может
потребоваться удалить некоторые из установленных ядер: иначе загрузочное меню станет
запутанным или может закончиться место.

.. _makeroom_bissbs:

* Чтобы удалить одно из установленных ядер, найдите его идентификатор 'kernelrelease'.
  Это руководство хранит их в '~/kernels-built', но следующая команда также напечатает
  их::

    ls -ltr /lib/modules/*-local*

  В большинстве ситуаций вы захотите удалить самые старые ядра, собранные в ходе самой
  бисекции (например, сегмента 3 этого руководства). Два ядра, которые вы создали до
  этого (например, для тестирования последней кодовой базы и версии, считающейся
  «хорошей»), могут пригодиться, чтобы что-то проверить позже, — поэтому лучше
  сохраните их, если только вы действительно не ограничены в месте на накопителе.

  Чтобы удалить модули ядра с идентификатором kernelrelease
  '*6.0-rc1-local-gcafec0cacaca0*', начните с удаления каталога, содержащего его
  модули::

    sudo rm -rf /lib/modules/6.0-rc1-local-gcafec0cacaca0

  Затем попробуйте следующую команду::

    sudo kernel-install -v remove 6.0-rc1-local-gcafec0cacaca0

  На довольно многих дистрибутивах это удалит все остальные установленные файлы ядра,
  а также уберёт запись ядра из загрузочного меню. Но на некоторых дистрибутивах
  kernel-install не существует или оставляет позади записи загрузчика либо образ ядра
  и связанные файлы; в этом случае удалите их, как описано в справочном разделе.

  [:ref:`подробности <makeroom_bisref>`]

.. _finishingtouch_bissbs:

* После того как вы завершили бисекцию, не удаляйте сразу всё, что настроили,
  поскольку кое-что может вам снова понадобиться. Что безопасно удалять, зависит от
  результата бисекции:

  * Вам изначально удалось воспроизвести регрессию на последней кодовой базе, а после
    бисекции вы смогли исправить проблему, откатив виновника поверх последней кодовой
    базы? Тогда вам стоит сохранить эти два ядра на какое-то время, но безопасно
    удалить все остальные с '-local' в идентификаторе выпуска.

  * Бисекция завершилась на коммите слияния или кажется сомнительной по другим
    причинам? Тогда вам стоит сохранить как можно больше ядер на несколько дней:
    весьма вероятно, что вас попросят что-то перепроверить.

  * В остальных случаях, скорее всего, хорошей идеей будет сохранить на некоторое
    время следующие ядра: собранное из последней кодовой базы, созданное из версии,
    считающейся «хорошей», и последние три-четыре, которые вы скомпилировали в ходе
    самого процесса бисекции.

  [:ref:`подробности <finishingtouch_bisref>`]

Необязательно: тестирование откатов, патчей или более поздних версий
--------------------------------------------------------------------

Во время или после сообщения об ошибке вы можете захотеть или вас потенциально
попросят протестировать откаты, отладочные патчи, предложенные исправления или другие
версии. В этом случае следуйте этим инструкциям.

* Обновите свой клон Git и переключитесь (check out) на последний код.

  * Если вы хотите протестировать mainline, получите его последние изменения, прежде
    чем переключаться на его код::

      git fetch mainline
      git switch --discard-changes --detach mainline/master

  * Если вы хотите протестировать ядро stable или longterm, сначала добавьте ветвь,
    содержащую интересующую вас серию (6.2 в примере), если только вы уже не сделали
    этого ранее::

      git remote set-branches --add stable linux-6.2.y

    Затем получите последние изменения и переключитесь на последнюю версию из серии::

      git fetch stable
      git switch --discard-changes --detach stable/linux-6.2.y

* Скопируйте свою конфигурацию сборки ядра поверх::

    cp ~/kernel-config-working .config

* Ваш следующий шаг зависит от того, что вы хотите сделать:

  * Если вы просто хотите протестировать последнюю кодовую базу, переходите к
    следующему шагу, вы уже всё подготовили.

  * Если вы хотите проверить, исправляет ли откат проблему, откатите одно или
    несколько изменений, указав их commit id::

      git revert --no-edit cafec0cacaca0

    Теперь дайте этому ядру специальный тег, чтобы облегчить его идентификацию и
    предотвратить случайную перезапись другого ядра::

      ./scripts/config --set-str CONFIG_LOCALVERSION '-local-cafec0cacaca0-reverted'

  * Если вы хотите протестировать патч, сохраните патч в файл вроде
    '/tmp/foobars-proposed-fix-v1.patch' и примените его так::

      git apply /tmp/foobars-proposed-fix-v1.patch

    В случае нескольких патчей повторите этот шаг с остальными.

    Теперь дайте этому ядру специальный тег, чтобы облегчить его идентификацию и
    предотвратить случайную перезапись другого ядра::

    ./scripts/config --set-str CONFIG_LOCALVERSION '-local-foobars-fix-v1'

* Соберите ядро с помощью знакомых команд, только без копирования конфигурации сборки
  ядра поверх, так как об этом уже позаботились::

    make olddefconfig &&
    make -j $(nproc --all)
    # * Проверьте, хватает ли свободного места для размещения ещё одного ядра:
    df -h /boot/ /lib/modules/
    sudo make modules_install
    command -v installkernel && sudo make install
    make -s kernelrelease | tee -a ~/kernels-built
    reboot

* Теперь убедитесь, что вы загрузили только что собранное ядро, и проверьте его.

[:ref:`подробности <introoptional_bisref>`]

Подготовьтесь к непредвиденным ситуациям
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Создайте свежую резервную копию и держите под рукой средства восстановления и
  ремонта системы.* [:ref:`... <backup_bissbs>`]

Помните, вы имеете дело с компьютерами, которые иногда делают неожиданные вещи —
особенно если вы возитесь с такими важнейшими частями, как ядро операционной

подготовиться к тому, что что-то пойдёт не так, даже если этого произойти не должно.

.. _vanilla_bisref:

Удалите всё, что связано с внешне сопровождаемыми модулями ядра
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Удалите всё ПО, которое зависит от разрабатываемых сторонними сторонами драйверов
  ядра или собирает их автоматически.* [:ref:`...<vanilla_bissbs>`]

Разрабатываемые сторонними сторонами модули ядра легко могут вызвать проблемы во
время бисекции.

Но есть более важная причина, по которой это руководство содержит этот шаг:
большинство разработчиков ядра не станут обращать внимание на отчёты о регрессиях,
возникающих с ядрами, которые используют такие модули. Это потому, что такие ядра
больше не считаются «vanilla», как более подробно объясняет
Documentation/admin-guide/reporting-issues.rst.

[:ref:`назад к пошаговому руководству <vanilla_bissbs>`]

.. _secureboot_bisref:

Разберитесь с технологиями вроде Secure Boot
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *На платформах с «Secure Boot» или подобными технологиями подготовьте всё
  необходимое, чтобы гарантировать, что система позже разрешит загрузку вашего
  самостоятельно скомпилированного ядра.* [:ref:`... <secureboot_bissbs>`]

Многие современные системы позволяют запускаться только определённым операционным
системам; вот почему по умолчанию они отказываются загружать самостоятельно
скомпилированные ядра.

В идеале вы разбираетесь с этим, заставляя свою платформу доверять вашим
самостоятельно собранным ядрам с помощью сертификата. Как это сделать, здесь не
описывается, так как это требует различных шагов, которые увели бы текст слишком
далеко от его цели; 'Documentation/admin-guide/module-signing.rst' и различные
веб-сайты уже объясняют всё необходимое более подробно.

Временное отключение решений вроде Secure Boot — другой способ заставить ваш
собственный Linux загружаться. На обычных системах x86 это можно сделать в утилите
настройки BIOS; необходимые шаги сильно различаются между машинами и поэтому не могут
быть описаны здесь.

В основных дистрибутивах Linux для x86 есть третий и универсальный вариант: отключить
все ограничения Secure Boot для вашего окружения Linux. Вы можете инициировать этот
процесс, выполнив ``mokutil --disable-validation``; это велит вам создать одноразовый
пароль, который безопасно записать. Теперь перезапустите систему; сразу после того,
как ваш BIOS выполнит все самотесты, загрузчик Shim покажет синее окно с сообщением
'Press any key to perform MOK management'. Нажмите какую-нибудь клавишу до того, как
истечёт обратный отсчёт, что откроет меню. Выберите 'Change Secure Boot state'.
'MokManager' от Shim теперь попросит вас ввести три случайно выбранных символа из
ранее заданного одноразового пароля. После того как вы их предоставите,
подтвердите, что вы действительно хотите отключить проверку. Затем разрешите
MokManager перезагрузить машину.

[:ref:`назад к пошаговому руководству <secureboot_bissbs>`]

.. _diskspace_bisref:

Требования к месту
~~~~~~~~~~~~~~~~~~~

  *Обеспечьте достаточно свободного места для сборки Linux.*
  [:ref:`... <diskspace_bissbs>`]

Упомянутые числа — это грубые оценки с большой надбавкой для подстраховки, так что
часто вам потребуется меньше.

Если у вас есть ограничения по месту, обязательно обратите внимание на :ref:`шаг об
отладочных символах <debugsymbols_bissbs>` и :ref:`сопровождающий его справочный
раздел <debugsymbols_bisref>`, так как их отключение уменьшит занимаемое место на
диске на несколько гигабайт.

[:ref:`назад к пошаговому руководству <diskspace_bissbs>`]

.. _rangecheck_bisref:

Диапазон бисекции
~~~~~~~~~~~~~~~~~~

  *Определите версии ядра, которые на протяжении всего этого руководства считаются
  «хорошей» и «плохой».* [:ref:`...<rangecheck_bissbs>`]

Установление диапазона коммитов для проверки в основном прямолинейно, за исключением
случая, когда регрессия возникла при переходе с выпуска одной серии stable на выпуск
более поздней серии (например, с 6.0.13 на 6.1.5). В этом случае Git понадобится
небольшая помощь, поскольку прямой линии родства нет.

Это потому, что с выпуском 6.0 mainline двинулась дальше к 6.1, в то время как серия
stable 6.0.y ответвилась в сторону. Поэтому теоретически возможно, что проблема, с
которой вы столкнулись в 6.1.5, работала только в 6.0.13, поскольку была исправлена
коммитом, который попал в один из выпусков 6.0.y, но так и не попал в mainline или
серию 6.1.y. К счастью, в норме этого происходить не должно из-за того, как
сопровождающие stable/longterm ведут код. Поэтому довольно безопасно считать 6.0
«хорошим» ядром. Это предположение в любом случае будет проверено, так как это ядро
будет собрано и протестировано в сегменте «2» этого руководства; Git заставил бы вас
сделать это и так, если бы вы попытались выполнить бисекцию между 6.0.13 и 6.1.15.

[:ref:`назад к пошаговому руководству <rangecheck_bissbs>`]

.. _buildrequires_bisref:

Установка требований для сборки
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Установите всё ПО, необходимое для сборки ядра Linux.*
  [:ref:`...<buildrequires_bissbs>`]

Ядро довольно самодостаточно, но помимо инструментов вроде компилятора вам иногда
понадобится несколько библиотек, чтобы собрать его. Как установить всё необходимое,
зависит от вашего дистрибутива Linux и конфигурации ядра, которое вы собираетесь
собрать.

Вот несколько примеров того, что вам обычно нужно на некоторых основных
дистрибутивах:

* Arch Linux и производные::

    sudo pacman --needed -S bc binutils bison flex gcc git kmod libelf openssl \
      pahole perl zlib ncurses qt6-base

* Debian, Ubuntu и производные::

    sudo apt install bc binutils bison dwarves flex gcc git kmod libelf-dev \
      libssl-dev make openssl pahole perl-base pkg-config zlib1g-dev \
      libncurses-dev qt6-base-dev g++

* Fedora и производные::

    sudo dnf install binutils \
      /usr/bin/{bc,bison,flex,gcc,git,openssl,make,perl,pahole,rpmbuild} \
      /usr/include/{libelf.h,openssl/pkcs7.h,zlib.h,ncurses.h,qt6/QtGui/QAction}

* openSUSE и производные::

    sudo zypper install bc binutils bison dwarves flex gcc git \
      kernel-install-tools libelf-devel make modutils openssl openssl-devel \
      perl-base zlib-devel rpm-build ncurses-devel qt6-base-devel

Эти команды устанавливают несколько пакетов, которые часто, но не всегда нужны. Вы,
например, можете захотеть пропустить установку заголовочных файлов разработки для
ncurses, которые понадобятся вам, только если вы позже захотите скорректировать
конфигурацию сборки ядра с помощью целей make 'menuconfig' или 'nconfig'; аналогично
опустите заголовки Qt6, если вы не планируете корректировать .config с помощью
'xconfig'.

Кроме того, вам могут понадобиться дополнительные библиотеки и их заголовочные файлы
разработки для задач, не охваченных этим руководством, — например, при сборке утилит
из каталога tools/ ядра.

[:ref:`назад к пошаговому руководству <buildrequires_bissbs>`]

Загрузка исходников с помощью Git
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Получите исходники mainline Linux.*
  [:ref:`...<sources_bissbs>`]

Пошаговое руководство описывает, как загрузить исходники Linux с помощью полного
клона Git репозитория mainline Линуса. Больше об этом сказать нечего — но есть два
альтернативных способа получения исходников, которые могут подойти вам лучше:

* Если у вас ненадёжное интернет-соединение, рассмотрите
  :ref:`использование «Git bundle» <sources_bundle_bisref>`.

* Если загрузка полного репозитория заняла бы слишком много времени или потребовала
  бы слишком много места на накопителе, рассмотрите :ref:`использование «поверхностного
  клона» (shallow clone) <sources_shallow_bisref>`.

.. _sources_bundle_bisref:

Загрузка исходников mainline Linux с помощью bundle
"""""""""""""""""""""""""""""""""""""""""""""""""""

Используйте следующие команды, чтобы получить исходники mainline Linux с помощью
bundle::

    wget -c \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/clone.bundle
    git clone --no-checkout clone.bundle ~/linux/
    cd ~/linux/
    git remote remove origin
    git remote add mainline \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git
    git fetch mainline
    git remote add -t master stable \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git

Если команда 'wget' завершится неудачей, просто повторите её, она продолжит с того
места, где остановилась.

.. _sources_shallow_bisref:

Загрузка исходников mainline Linux с помощью поверхностного клона
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Сначала выполните следующую команду, чтобы получить последнюю кодовую базу mainline::

    git clone -o mainline --no-checkout --depth 1 -b master \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git ~/linux/
    cd ~/linux/
    git remote add -t master stable \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git

Теперь углубите историю своего клона до второго предшественника mainline-выпуска
вашей «хорошей» версии. Если последняя — 6.0 или 6.0.13, то 5.19 будет первым
предшественником, а 5.18 — вторым, поэтому углубите историю вплоть до этой версии::

    git fetch --shallow-exclude=v5.18 mainline

Затем добавьте репозиторий Git stable как удалённый (remote) и все необходимые ветви
stable, как объясняется в пошаговом руководстве.

Обратите внимание, поверхностные клоны имеют несколько своеобразных особенностей:

* Для бисекций историю нужно углублять на несколько mainline-версий дальше, чем
  кажется необходимым, как уже объяснялось выше. Это потому, что иначе Git не сможет
  откатить или описать большинство коммитов в диапазоне (скажем, 6.1..6.2), так как
  внутренне они основаны на более ранних выпусках ядра (вроде 6.0-rc2 или 5.19-rc3).

* Этот документ в большинстве мест использует ``git fetch`` с ``--shallow-exclude=``,
  чтобы указать самую раннюю версию, которая вас интересует (или, точнее: её
  git-тег). В качестве альтернативы вы можете использовать параметр
  ``--shallow-since=``, чтобы указать абсолютную (скажем, ``'2023-07-15'``) или
  относительную (``'12 months'``) дату для определения глубины истории, которую вы
  хотите загрузить. При их использовании во время бисекции mainline обязательно
  углубите историю как минимум на 7 месяцев до выпуска mainline-выпуска, на котором
  основано ваше «хорошее» ядро.

* Будьте предупреждены: при углублении своего клона вы можете столкнуться с ошибкой
  вроде 'fatal: error in object: unshallow
  cafecaca0c0dacafecaca0c0dacafecaca0c0da'. В этом случае выполните ``git repack -d``
  и попробуйте снова.

.. _oldconfig_bisref:

Начните определять конфигурацию сборки для своего ядра
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Начните подготовку конфигурации сборки ядра (файла '.config').*
  [:ref:`... <oldconfig_bissbs>`]

*Обратите внимание, это первый из нескольких шагов в этом руководстве, которые
создают или изменяют артефакты сборки. Команды, используемые в этом руководстве,
хранят их прямо в дереве исходников, чтобы всё было проще. Если вы предпочитаете
хранить артефакты сборки отдельно, создайте каталог вроде '~/linux-builddir/' и
добавьте параметр* ``O=~/linux-builddir/`` *ко всем вызовам make, используемым на
протяжении этого руководства. Вам придётся указать на этот каталог и в других
командах — среди них команды* ``./scripts/config [...]``, *которым потребуется*
``--file ~/linux-builddir/.config``, *чтобы найти правильную конфигурацию сборки.*

Две вещи легко могут пойти не так при создании файла .config рекомендованным
способом:

* Цель oldconfig будет использовать файл .config из вашего каталога сборки, если он
  там уже присутствует (например, '~/linux/.config'). Это совершенно нормально, если
  вы именно этого и хотите (см. следующий шаг), но во всех остальных случаях вам стоит
  удалить его. Это, например, важно, если вы следовали этому руководству дальше, но
  из-за проблем вернулись сюда, чтобы переделать конфигурацию с нуля.

* Иногда olddefconfig не может найти файл .config для вашего работающего ядра и будет
  использовать значения по умолчанию, как кратко описано в руководстве. В этом случае
  проверьте, поставляет ли ваш дистрибутив конфигурацию где-нибудь, и, если да,
  вручную поместите её в правильное место (например, '~/linux/.config'). На
  дистрибутивах, где существует /proc/config.gz, этого можно добиться с помощью этой
  команды::

    zcat /proc/config.gz > .config

  После того как вы поместите его туда, снова выполните ``make olddefconfig``, чтобы
  скорректировать его под нужды собираемого ядра.

Обратите внимание, цель olddefconfig установит любые неопределённые параметры сборки
в их значение по умолчанию. Если вы предпочитаете устанавливать такие параметры
конфигурации вручную, используйте вместо этого ``make oldconfig``. Тогда для каждого
неопределённого параметра конфигурации вас спросят, как поступить; если вы не уверены,
что ответить, просто нажмите 'enter', чтобы применить значение по умолчанию. Однако
учтите, что для бисекций вы обычно хотите оставлять значения по умолчанию, так как
иначе вы можете включить новую функциональность, которая вызывает проблему, похожую
на регрессию (например, из-за ограничений безопасности).

Время от времени странные вещи происходят при попытке использовать файл конфигурации,
подготовленный для одного ядра (скажем, 6.1), на более старом mainline-выпуске —
особенно если он намного старше (скажем, 5.15). Это одна из причин, по которой
предыдущий шаг в руководстве велел вам загрузить ядро, где всё работает. Поэтому,
если вы вручную добавляете файл .config, вы хотите убедиться, что он из рабочего
ядра, а не из того, которое показывает регрессию.

Если вы хотите собрать ядра для другой машины, найдите её конфигурацию сборки ядра;
обычно ``ls /boot/config-$(uname -r)`` напечатает её имя. Скопируйте этот файл на
машину сборки и сохраните как ~/linux/.config; затем выполните ``make olddefconfig``,
чтобы скорректировать его.

[:ref:`назад к пошаговому руководству <oldconfig_bissbs>`]

.. _localmodconfig_bisref:

Урежьте конфигурацию сборки для своего ядра
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Отключите любые модули ядра, явно лишние для вашей конфигурации.*
  [:ref:`... <localmodconfig_bissbs>`]

Как уже кратко объяснялось в пошаговом руководстве: с localmodconfig легко может
случиться, что вашим самостоятельно собранным ядрам будут недоставать модули для
задач, которые вы не выполнили хотя бы раз перед использованием этой цели make. Это
происходит, когда задача требует модулей ядра, которые автоматически загружаются
только тогда, когда вы выполняете её впервые. Поэтому, если вы ни разу не выполняли
эту задачу с момента запуска ядра, модули не будут загружены — и с точки зрения
localmodconfig выглядят лишними, что приводит к их отключению, чтобы сократить объём
компилируемого кода.

Вы можете попытаться избежать этого, выполнив типичные задачи, которые часто
автоматически загружают дополнительные модули ядра: запустите ВМ, установите
VPN-соединения, смонтируйте ISO-образ CD/DVD через loop, смонтируйте сетевые шары
(CIFS, NFS, ...) и подключите все внешние устройства (ключи 2FA, гарнитуры,
веб-камеры, ...), а также накопители с файловыми системами, которые вы иначе не
используете (btrfs, ext4, FAT, NTFS, XFS, ...). Но трудно подумать обо всём, что
может понадобиться, — даже разработчики ядра часто забывают о том или ином на этом
этапе.

Не позволяйте этому риску беспокоить вас, особенно при компиляции ядра только для
целей тестирования: всё типично важное там будет. И если вы забудете что-то важное,
вы можете включить недостающую функциональность вручную позже и быстро снова
выполнить команды, чтобы скомпилировать и установить ядро, в котором есть всё, что вам
нужно.

Но если вы планируете регулярно собирать и использовать самостоятельно собранные
ядра, вы можете захотеть снизить риск, записывая, какие модули загружает ваша система
на протяжении нескольких недель. Это можно автоматизировать с помощью `modprobed-db
<https://github.com/graysky2/modprobed-db>`_. Затем используйте ``LSMOD=<path>``,
чтобы указать localmodconfig на список модулей, которые modprobed-db заметил
используемыми::

  yes '' | make LSMOD='${HOME}'/.config/modprobed.db localmodconfig

Этот параметр также позволяет вам собирать урезанные ядра для другой машины, если вы
скопировали подходящий .config для использования в качестве основы (см. предыдущий
шаг). Просто выполните ``lsmod > lsmod_foo-machine`` на той системе и скопируйте
сгенерированный файл в домашний каталог хоста сборки. Затем выполните эти команды
вместо той, что упоминает пошаговое руководство::

  yes '' | make LSMOD=~/lsmod_foo-machine localmodconfig

[:ref:`назад к пошаговому руководству <localmodconfig_bissbs>`]

Пометьте собираемые ядра тегом
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Обеспечьте, чтобы все собираемые вами ядра были чётко идентифицируемы с помощью
  специального тега и уникального идентификатора версии.* [:ref:`... <tagging_bissbs>`]

Это позволяет вам отличать ядра вашего дистрибутива от созданных в ходе этого

это также помогает выбирать правильную запись в загрузочном меню и не терять из виду
ваши ядра, так как их номера версий будут выглядеть немного запутанно во время
бисекции.

.. _debugsymbols_bisref:

Решите, включать или отключать отладочные символы
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Решите, как обращаться с отладочными символами.* [:ref:`... <debugsymbols_bissbs>`]

Наличие доступных отладочных символов может быть важно, когда ваше ядро позже при
работе выбросит 'panic', 'Oops', 'warning' или 'BUG', так как тогда вы сможете найти
точное место в коде, где возникла проблема. Но сбор и встраивание необходимой
отладочной информации занимает время и потребляет немало места: в конце 2022 года
артефакты сборки для типичного ядра x86, урезанного с помощью localmodconfig, занимали
около 5 гигабайт места с отладочными символами, но менее 1, когда они были отключены.
Получаемые образ ядра и модули также больше, что увеличивает требования к месту для
/boot/ и время загрузки.

Если вы хотите небольшое ядро и вряд ли будете позже декодировать трассировку стека,
вы, таким образом, можете захотеть отключить отладочные символы, чтобы избежать этих
недостатков. Если позже окажется, что они вам нужны, просто включите их, как показано,
и пересоберите ядро.

С другой стороны, вы определённо хотите включить их для этого процесса, если есть
приличная вероятность, что вам понадобится позже декодировать трассировку стека.
Раздел 'Decode failure messages' в Documentation/admin-guide/reporting-issues.rst
объясняет этот процесс более подробно.

[:ref:`назад к пошаговому руководству <debugsymbols_bissbs>`]

.. _configmods_distros_bisref:

Корректировки, специфичные для дистрибутива
"""""""""""""""""""""""""""""""""""""""""""

  *Вы работаете в* [:ref:`... <configmods_bissbs>`]

Следующие разделы помогают вам избежать проблем сборки, которые, как известно,

**Debian:**

* Удалите устаревшую ссылку на файл сертификата, которая привела бы к неудаче вашей

  В качестве альтернативы загрузите нужный сертификат и заставьте этот параметр

  Documentation/admin-guide/module-signing.rst.

Отложите файл .config в сторону
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Повторно обработайте .config после последних изменений и сохраните его в
  безопасном месте.* [:ref:`... <saveconfig_bissbs>`]

Отложите подготовленный вами .config в сторону, так как вы захотите копировать его

Попробуйте воспроизвести проблему на последней кодовой базе
-----------------------------------------------------------

  *Проверьте, что регрессия не вызвана каким-либо изменением .config, и проверьте,
  возникает ли она всё ещё на последней кодовой базе.*
  [:ref:`... <introlatestcheck_bissbs>`]

Некоторым читателям может показаться излишним проверять последнюю кодовую базу на этом
этапе, особенно если вы уже делали это с ядром, подготовленным вашим дистрибьютором,
или сталкиваетесь с регрессией внутри серии stable/longterm. Но это настоятельно
рекомендуется по следующим причинам:

* Вы столкнётесь с любыми проблемами, вызванными вашей конфигурацией, до того, как
  фактически начнёте бисекцию. Это значительно облегчит различение между «это, скорее
  всего, какая-то проблема в моей конфигурации» и «это изменение нужно пропустить во
  время бисекции, так как исходники ядра на этом этапе содержат не связанную с
  проблемой ошибку, которая приводит к неудаче сборки или загрузки».

* Эти шаги исключат, вызвана ли ваша проблема каким-либо изменением в конфигурации
  сборки между «рабочим» и «сломанным» ядром. Это, например, может произойти, когда
  ваш дистрибьютор включил дополнительную функцию безопасности в более новом ядре,
  которая была отключена или ещё не поддерживалась в более старом ядре. Эта функция
  безопасности может встать на пути чего-то, что вы делаете, — в этом случае ваша
  проблема с точки зрения разработчиков апстрима ядра Linux не является регрессией,
  как более подробно объясняет Documentation/admin-guide/reporting-regressions.rst.
  Таким образом, вы бы зря потратили своё время, если бы попытались выполнить бисекцию
  этого.

* Если причина вашей регрессии была уже исправлена в последней кодовой базе mainline,
  вы бы выполняли бисекцию впустую. Это справедливо и для регрессии, с которой вы
  столкнулись с выпуском stable/longterm, так как они часто вызваны проблемами в
  изменениях mainline, которые были бэкпортированы, — в этом случае проблему сначала
  придётся исправить в mainline. Возможно, она уже была там исправлена, и исправление
  уже находится в процессе бэкпортирования.

* Для регрессий внутри серии stable/longterm, кроме того, критически важно знать,
  специфична ли проблема для этой серии или также возникает в ядре mainline, так как
  отчёт нужно отправлять разным людям:

  * Регрессии, специфичные для серии stable/longterm, — это ответственность команды
    stable; разработчиков mainline Linux они могут касаться, а могут и не касаться.

  * Регрессии, возникающие также в mainline, — это то, чем должны заниматься обычные
    разработчики и сопровождающие Linux; команду stable они не касаются, и её не нужно
    вовлекать в отчёт, ей лишь следует сказать бэкпортировать исправление, как только
    оно будет готово.

  Ваш отчёт могут проигнорировать, если вы отправите его не той стороне — и даже когда
  вы получите ответ, есть приличная вероятность, что разработчики попросят вас
  определить, какой из двух случаев имеет место, прежде чем они присмотрятся
  внимательнее.

[:ref:`назад к пошаговому руководству <introlatestcheck_bissbs>`]

Соберите своё ядро
~~~~~~~~~~~~~~~~~~~

  *Соберите образ и модули вашего первого ядра, используя подготовленный вами файл

Многое может пойти не так на этом этапе, но приведённые ниже инструкции помогут вам
помочь себе. Другой подраздел объясняет, как напрямую упаковать ваше ядро в виде

Борьба с ошибками сборки
""""""""""""""""""""""""

Когда возникает ошибка сборки, она может быть вызвана каким-либо аспектом настройки
вашей машины, который часто можно быстро исправить; в других же случаях проблема
кроется в коде и может быть исправлена только разработчиком. Внимательное изучение
сообщений об ошибке вместе с некоторым исследованием в интернете часто подскажет вам,
какой из двух случаев имеет место. Чтобы провести такое расследование, перезапустите
процесс сборки так::

  make V=1

``V=1`` активирует подробный вывод, который может понадобиться, чтобы увидеть
фактическую ошибку. Чтобы её было легче заметить, эта команда также опускает ``-j
$(nproc --all)``, использовавшийся ранее для задействования каждого ядра CPU в системе
для этой работы, — но этот параллелизм также приводит к некоторому беспорядку при
возникновении сбоев.

Через несколько секунд процесс сборки должен снова наткнуться на ошибку. Теперь
попробуйте найти самую важную строку, описывающую проблему. Затем поищите в интернете
самый важный и не-общий фрагмент этой строки (скажем, 4–8 слов); избегайте или
удаляйте всё, что выглядит хоть сколько-нибудь специфичным для системы, вроде вашего
имени пользователя или локальных путей вроде ``/home/username/linux/``. Сначала
попробуйте свою обычную поисковую систему в интернете с этой строкой, затем поищите по
спискам рассылки ядра Linux через `lore.kernel.org/all/
<https://lore.kernel.org/all/>`_.

В большинстве случаев это найдёт что-то, что объяснит, что не так; довольно часто один
из результатов предоставит и решение вашей проблемы. Если вы не найдёте ничего, что
соответствует вашей проблеме, попробуйте снова под другим углом, изменив поисковые
запросы или используя другую строку из сообщений об ошибке.

В конечном счёте большинство проблем, с которыми вы столкнётесь, скорее всего, уже
встречались и были описаны другими. Сюда относятся и проблемы, причина которых не в
вашей системе, а кроется в коде. Если вы столкнётесь с одной из таких, вы, таким
образом, можете найти и решение (например, патч) или обходной путь для вашей проблемы.

Упакуйте своё ядро
""""""""""""""""""

Пошаговое руководство использует цели make по умолчанию (например, 'bzImage' и
'modules' на x86), чтобы собрать образ и модули вашего ядра, которые затем
устанавливают более поздние шаги руководства. Вместо этого вы также можете напрямую
собрать всё и сразу упаковать с помощью одной из следующих целей:

* ``make -j $(nproc --all) bindeb-pkg``, чтобы сгенерировать пакет deb

* ``make -j $(nproc --all) binrpm-pkg``, чтобы сгенерировать пакет rpm

* ``make -j $(nproc --all) tarbz2-pkg``, чтобы сгенерировать сжатый bz2 тарбол

Это лишь выборка доступных для этой цели целей make, остальные см. в ``make help``. Вы
также можете использовать эти цели после выполнения ``make -j $(nproc --all)``, так
как они подхватят всё, что уже собрано.

Если вы задействуете цели для генерации пакетов deb или rpm, проигнорируйте инструкции
пошагового руководства по установке и удалению вашего ядра; вместо этого устанавливайте
и удаляйте пакеты с помощью пакетной утилиты для этого формата (например, dpkg и rpm)
или утилиты управления пакетами, построенной поверх них (apt, aptitude, dnf/yum,
zypper, ...). Учтите, что пакеты, сгенерированные с помощью этих двух целей make,
спроектированы так, чтобы работать на различных дистрибутивах, использующих эти
форматы, поэтому они иногда будут вести себя иначе, чем пакеты ядра вашего
дистрибутива.

[:ref:`назад к пошаговому руководству <build_bissbs>`]

.. _install_bisref:

Поместите ядро на место
~~~~~~~~~~~~~~~~~~~~~~~~

  *Установите только что собранное ядро.* [:ref:`... <install_bissbs>`]

То, что вам нужно сделать после выполнения команды из пошагового руководства, зависит
от существования и реализации исполняемого файла ``/sbin/installkernel`` в вашем
дистрибутиве.

Если installkernel найден, система сборки ядра делегирует фактическую установку образа
вашего ядра этому исполняемому файлу, который затем выполняет некоторые или все из
этих задач:

* Почти на всех дистрибутивах Linux installkernel сохранит образ вашего ядра в /boot/,
  обычно как '/boot/vmlinuz-<kernelrelease_id>'; часто он положит рядом с ним
  'System.map-<kernelrelease_id>'.

* На большинстве дистрибутивов installkernel затем сгенерирует 'initramfs' (иногда
  также называемый 'initrd'), который обычно сохраняется как
  '/boot/initramfs-<kernelrelease_id>.img' или '/boot/initrd-<kernelrelease_id>'.
  Распространённые дистрибутивы опираются на этот файл для загрузки, поэтому обязательно
  выполните сначала цель make 'modules_install', так как иначе генератор initramfs
  вашего дистрибутива не сможет найти модули, которые входят в образ.

* На некоторых дистрибутивах installkernel затем добавит запись для вашего ядра в
  конфигурацию вашего загрузчика.

Вам придётся позаботиться о некоторых или всех задачах самостоятельно, если в вашем
дистрибутиве нет сценария installkernel или он обрабатывает лишь часть из них.
Обратитесь к документации дистрибутива за подробностями. Если сомневаетесь, установите
ядро вручную::

   sudo install -m 0600 $(make -s image_name) /boot/vmlinuz-$(make -s kernelrelease)
   sudo install -m 0600 System.map /boot/System.map-$(make -s kernelrelease)

Теперь сгенерируйте свой initramfs с помощью инструментов, которые ваш дистрибутив
предоставляет для этого процесса. Затем добавьте своё ядро в конфигурацию загрузчика и
перезагрузитесь.

[:ref:`назад к пошаговому руководству <install_bissbs>`]

.. _storagespace_bisref:

Требования к месту на накопителе на одно ядро
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Проверьте, сколько места на накопителе занимают ядро, его модули и другие связанные
  файлы вроде initramfs.* [:ref:`... <storagespace_bissbs>`]

Ядра, собранные во время бисекции, занимают немало места в /boot/ и /lib/modules/,
особенно если вы включили отладочные символы. Из-за этого легко заполнить тома во
время бисекции — и из-за этого даже ядра, которые раньше работали, могут не
загрузиться. Чтобы предотвратить это, вам нужно знать, сколько места обычно требует
каждое установленное ядро.

Обратите внимание, в большинстве случаев шаблон '/boot/*$(make -s kernelrelease)*',
используемый в руководстве, будет соответствовать всем файлам, необходимым для
загрузки вашего ядра, — но ни путь, ни схема именования не являются обязательными. На
некоторых дистрибутивах вам поэтому придётся искать в других местах.

[:ref:`назад к пошаговому руководству <storagespace_bissbs>`]

.. _tainted_bisref:

Проверьте, не считает ли только что собранное ядро себя «tainted»
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Проверьте, не пометило ли ядро себя как «tainted».*
  [:ref:`... <tainted_bissbs>`]

Linux помечает себя как tainted, когда происходит что-то, что потенциально приводит к
последующим ошибкам, выглядящим совершенно не связанными. Вот почему разработчики
могут проигнорировать отчёты от tainted-ядер или отреагировать на них скудно — если
только, конечно, ядро не установило флаг прямо тогда, когда возникла сообщаемая
ошибка.

Вот почему вы хотите проверить, почему ядро является tainted, как объясняется в
Documentation/admin-guide/tainted-kernels.rst; сделать это также в ваших собственных
интересах, так как иначе ваше тестирование может быть с изъяном.

[:ref:`назад к пошаговому руководству <tainted_bissbs>`]

.. _recheckbroken_bisref:

Проверьте ядро, собранное из свежей кодовой базы mainline
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Проверьте, возникает ли ваша ошибка в только что собранном ядре.*
  [:ref:`... <recheckbroken_bissbs>`]

Есть несколько причин, по которым ваша ошибка или регрессия может не проявиться с
ядром, собранным вами из последней кодовой базы. Вот самые частые:

* Ошибка тем временем была исправлена.

* То, что вы подозревали как регрессию, было вызвано изменением в конфигурации сборки,
  которое произвёл поставщик вашего ядра.

* Ваша проблема может быть состоянием гонки (race condition), которое не проявляется с
  вашим ядром; это могут вызывать урезанная конфигурация сборки, другая настройка для
  отладочных символов, используемый компилятор и различные другие вещи.

* Если вы столкнулись с регрессией с ядром stable/longterm, это может быть проблема,
  специфичная для этой серии; следующий шаг в этом руководстве проверит это.

[:ref:`назад к пошаговому руководству <recheckbroken_bissbs>`]

.. _recheckstablebroken_bisref:

Проверьте ядро, собранное из последней кодовой базы stable/longterm
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Вы сталкиваетесь с регрессией внутри выпуска stable/longterm, но вам не удалось
  воспроизвести её с ядром, которое вы только что собрали из последних исходников
  mainline? Тогда проверьте, может ли последняя кодовая база для конкретной серии уже
  исправить проблему.* [:ref:`... <recheckstablebroken_bissbs>`]

Если это ядро тоже не показывает регрессию, то, скорее всего, в бисекции нет
необходимости.

[:ref:`назад к пошаговому руководству <recheckstablebroken_bissbs>`]

.. _introworkingcheck_bisref:

Убедитесь, что «хорошая» версия действительно работает нормально
----------------------------------------------------------------

  *Проверьте, нормально ли работают собираемые вами ядра.*
  [:ref:`... <introworkingcheck_bissbs>`]

Этот раздел заново установит известную рабочую основу. Пропустить его может быть
заманчиво, но обычно это плохая идея, так как он делает кое-что важное:

Он гарантирует, что подготовленный вами ранее файл .config действительно работает как
ожидается. Это в ваших собственных интересах, так как урезание конфигурации не
защищено от ошибок — и вы можете собрать и протестировать десять или более ядер
впустую, прежде чем начать подозревать, что с конфигурацией сборки что-то не так.

Одного этого достаточно, чтобы потратить на это время, но это не единственная причина.

Многие читатели этого руководства обычно запускают ядра, которые пропатчены,
используют дополнительные модули или и то, и другое. Поэтому такие ядра не считаются
«vanilla» — следовательно, возможно, что то, что дало регрессию, могло вообще никогда
не работать в vanilla-сборках «хорошей» версии.

Есть и третья причина для тех, кто заметил регрессию между ядрами stable/longterm
разных серий (например, 6.0.13..6.1.5): это гарантирует, что версия ядра, которую вы
ранее в процессе предположили «хорошей» (например, 6.0), действительно работает.

[:ref:`назад к пошаговому руководству <introworkingcheck_bissbs>`]

.. _recheckworking_bisref:

Соберите свою собственную версию «хорошего» ядра
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Соберите свой собственный вариант рабочего ядра и проверьте, работает ли с ним
  функциональность, давшая регрессию, как ожидается.*
  [:ref:`... <recheckworking_bissbs>`]

Если функциональность, которая сломалась с более новыми ядрами, не работает с вашим
первым самостоятельно собранным ядром, найдите и устраните причину, прежде чем
двигаться дальше. Есть множество причин, по которым это может произойти. Несколько
идей, где искать:

* Проверьте статус taint и вывод ``dmesg``, может быть, что-то не связанное пошло не
  так.

* Может быть, localmodconfig сделал что-то странное и отключил модуль, необходимый для
  тестирования функциональности? Тогда вы можете захотеть пересоздать файл .config на
  основе того, что от последнего рабочего ядра, и пропустить его урезание; ручное
  отключение некоторых функций в .config тоже может сработать для сокращения времени
  сборки.

* Может быть, это не регрессия ядра, а что-то, вызванное случайностью, сломанным
  initramfs (также известным как initrd), новыми файлами прошивки (firmware) или
  обновлённым ПО пространства пользователя (userland)?

* Может быть, это была функция, добавленная в ядро вашего дистрибьютора, которую
  vanilla Linux на тот момент никогда не поддерживал?

Обратите внимание, если вы нашли и исправили проблемы с файлом .config, вы захотите
использовать его, чтобы собрать ещё одно ядро из последней кодовой базы, так как ваши
более ранние тесты с mainline и последней версией из затронутой серии stable/longterm,
скорее всего, были с изъяном.

[:ref:`назад к пошаговому руководству <recheckworking_bissbs>`]

Выполните бисекцию и проверьте результат
----------------------------------------

  *После того как обо всей подготовке и предосторожностях позаботились, вы теперь
  готовы приступить к бисекции.* [:ref:`... <introbisect_bissbs>`]

Шаги в этом сегменте выполняют и проверяют бисекцию.

[:ref:`назад к пошаговому руководству <introbisect_bissbs>`].

.. _bisectstart_bisref:

Запустите бисекцию
~~~~~~~~~~~~~~~~~~~

  *Запустите бисекцию и сообщите Git о версиях, ранее определённых как «хорошая» и
  «плохая».* [:ref:`... <bisectstart_bissbs>`]

Это запустит процесс бисекции; последняя из команд заставит Git переключиться (check
out) на коммит примерно посередине между «хорошим» и «плохим» изменениями для вашего
тестирования.

[:ref:`назад к пошаговому руководству <bisectstart_bissbs>`]

.. _bisectbuild_bisref:

Соберите ядро из точки бисекции
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Соберите, установите и загрузите ядро из кода, который Git переключил (checked
  out), с помощью тех же команд, что вы использовали ранее.*
  [:ref:`... <bisectbuild_bissbs>`]

Здесь стоит отметить две вещи:

* Время от времени сборка ядра будет завершаться неудачей или оно может не
  загрузиться из-за какой-либо проблемы в коде в точке бисекции. В этом случае
  выполните эту команду::

    git bisect skip

  Git затем переключится на другой коммит поблизости, который с небольшой долей удачи
  должен сработать лучше. Затем заново запустите выполнение этого шага.

* Те немного странно выглядящие идентификаторы версий могут возникать во время
  бисекций, потому что подсистемы ядра Linux готовят свои изменения для нового
  mainline-выпуска (скажем, 6.2) до того, как закончится его предшественник
  (например, 6.1). Поэтому они основывают их на несколько более ранней точке вроде
  6.1-rc1 или даже 6.0 — а затем они вливаются (merge) в 6.2 без ребейза или сжатия,
  как только выйдет 6.1. Это приводит к появлению тех немного странно выглядящих
  идентификаторов версий во время бисекций.

[:ref:`назад к пошаговому руководству <bisectbuild_bissbs>`]

.. _bisecttest_bisref:

Контрольная точка бисекции
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Проверьте, работает ли функциональность, давшая регрессию, в только что собранном
  ядре.* [:ref:`... <bisecttest_bissbs>`]

Обеспечьте, чтобы то, что вы сообщаете Git, было точным: одна ошибка уведёт всю
остальную бисекцию совершенно не в ту сторону, поэтому всё тестирование после этой
точки будет впустую.

[:ref:`назад к пошаговому руководству <bisecttest_bissbs>`]

.. _bisectlog_bisref:

Отложите журнал бисекции в сторону
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Сохраните журнал бисекции Git и текущий файл .config в безопасном месте.*
  [:ref:`... <bisectlog_bissbs>`]

Как указано выше: ошибочное объявление всего одного ядра «хорошим» или «плохим»
сделает конечный результат бисекции бесполезным. В этом случае вам обычно пришлось бы
перезапускать бисекцию с нуля. Журнал может предотвратить это, так как он может
позволить кому-нибудь указать, где бисекция, вероятно, пошла не так, — и тогда вместо
тестирования десяти или более ядер вам, возможно, придётся собрать всего несколько,
чтобы всё уладить.

Файл .config откладывается в сторону, так как есть приличная вероятность, что
разработчики могут попросить его после того, как вы сообщите о регрессии.

[:ref:`назад к пошаговому руководству <bisectlog_bissbs>`]

.. _revert_bisref:

Попробуйте откатить виновника
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Попробуйте откатить (revert) виновника поверх последней кодовой базы, чтобы
  посмотреть, исправит ли это вашу регрессию.* [:ref:`... <revert_bissbs>`]

Это необязательный шаг, но всякий раз, когда это возможно, вам стоит попробовать его:
есть приличная вероятность, что разработчики попросят вас выполнить этот шаг, когда вы
поднимете вопрос о результате бисекции. Так что попробуйте, вы уже в процессе, сборка
ещё одного ядра не должна быть большой проблемой на данном этапе.

Пошаговое руководство уже охватывает всё существенное, кроме одной немного редкой
вещи: вы выполняли бисекцию регрессии, которая также возникала с mainline, используя
серию stable/longterm, но Git не смог откатить коммит в mainline? Тогда попробуйте
откатить виновника в затронутой серии stable/longterm — и если это удастся,
протестируйте вместо этого ту версию ядра.

[:ref:`назад к пошаговому руководству <revert_bissbs>`]

Шаги по очистке во время и после следования этому руководству
-------------------------------------------------------------

  *Во время и после следования этому руководству вы можете захотеть или вам может
  потребоваться удалить некоторые из установленных ядер.*
  [:ref:`... <introclosure_bissbs>`]

Шаги в этом разделе описывают процедуры очистки.

[:ref:`назад к пошаговому руководству <introclosure_bissbs>`].

.. _makeroom_bisref:

Очистка во время бисекции
~~~~~~~~~~~~~~~~~~~~~~~~~~~

  *Чтобы удалить одно из установленных ядер, найдите его идентификатор
  'kernelrelease'.* [:ref:`... <makeroom_bissbs>`]

Ядра, которые вы устанавливаете в ходе этого процесса, легко удалить позже, так как их
части хранятся только в двух местах и чётко идентифицируемы. Поэтому вам не нужно
беспокоиться о том, что вы испортите свою машину, когда устанавливаете ядро вручную (и
тем самым обходите систему пакетов вашего дистрибутива): все части ваших ядер
относительно легко удалить позже.

Одно из двух мест — это каталог в /lib/modules/, который содержит модули для каждого
установленного ядра. Этот каталог назван по идентификатору выпуска ядра; поэтому,
чтобы удалить все модули для одного из собранных вами ядер, просто удалите его каталог
модулей в /lib/modules/.

Другое место — это /boot/, где обычно во время установки ядра размещается от двух до
пяти файлов. Все они обычно содержат имя выпуска в имени файла, но сколько файлов и их
точные имена несколько зависят от исполняемого файла installkernel вашего дистрибутива
и его генератора initramfs. На некоторых дистрибутивах упомянутая в пошаговом
руководстве команда ``kernel-install remove...`` удалит за вас все эти файлы, а также
уберёт запись меню для ядра из конфигурации вашего загрузчика. На других вам придётся
позаботиться об этих двух задачах самостоятельно. Следующая команда должна
интерактивно удалить три основных файла ядра с именем выпуска
'6.0-rc1-local-gcafec0cacaca0'::

  rm -i /boot/{System.map,vmlinuz,initr}-6.0-rc1-local-gcafec0cacaca0

Затем проверьте, нет ли в /boot/ других файлов, содержащих
'6.0-rc1-local-gcafec0cacaca0' в имени, и подумайте об их удалении тоже. Теперь
удалите загрузочную запись для ядра из конфигурации вашего загрузчика; шаги для этого
довольно сильно различаются между дистрибутивами Linux.

Обратите внимание, будьте осторожны с шаблонами вроде '*' при ручном удалении файлов
или каталогов для ядер: вы можете случайно удалить файлы ядра 6.0.13, когда всё, что
вы хотите, — удалить 6.0 или 6.0.1.

[:ref:`назад к пошаговому руководству <makeroom_bissbs>`]

Очистка после бисекции
~~~~~~~~~~~~~~~~~~~~~~~~

.. _finishingtouch_bisref:

  *После того как вы завершили бисекцию, не удаляйте сразу всё, что настроили,
  поскольку кое-что может вам снова понадобиться.*
  [:ref:`... <finishingtouch_bissbs>`]

Когда вы действительно ограничены в месте на накопителе, удаление ядер, как описано в
пошаговом руководстве, может освободить не так много места, как вам хотелось бы. В
этом случае подумайте о том, чтобы теперь также выполнить ``rm -rf ~/linux/*``. Это
удалит артефакты сборки и исходники Linux, но оставит позади репозиторий Git
(~/linux/.git/) — поэтому простая ``git reset --hard`` вернёт исходники обратно.

Удаление и репозитория на данном этапе, скорее всего, было бы неразумным: есть
приличная вероятность, что разработчики попросят вас собрать ещё одно ядро для
выполнения дополнительных тестов — вроде тестирования отладочного патча или
предложенного исправления. Подробности о том, как их выполнять, можно найти в разделе
:ref:`Необязательные задачи: тестирование откатов, патчей или более поздних версий <introoptional_bissbs>`.

Дополнительные тесты — это также причина, по которой вы хотите сохранить файл
~/kernel-config-working на несколько недель.

[:ref:`назад к пошаговому руководству <finishingtouch_bissbs>`]

.. _introoptional_bisref:

Тестирование откатов, патчей или более поздних версий
-----------------------------------------------------

  *Во время или после сообщения об ошибке вы можете захотеть или вас потенциально
  попросят протестировать откаты, патчи, предложенные исправления или другие версии.*
  [:ref:`... <introoptional_bissbs>`]

Все команды, используемые в этом разделе, должны быть довольно прямолинейны, так что
добавить особо нечего, кроме одного: при установке тега ядра, как указано, убедитесь,
что он не намного длиннее того, что использовался в примере, так как возникнут
проблемы, если идентификатор kernelrelease превысит 63 символа.

[:ref:`назад к пошаговому руководству <introoptional_bissbs>`].

.. _buildhost_bis:

Сборка ядер на другой машине
----------------------------

Чтобы компилировать ядра на другой системе, немного измените инструкции пошагового
руководства:

* Начните следовать руководству на машине, где вы хотите позже устанавливать и
  тестировать ядра.

* После выполнения ':ref:`Загрузитесь в рабочее ядро и кратко воспользуйтесь
  предположительно сломанной функциональностью <bootworking_bissbs>`' сохраните
  список загруженных модулей в файл с помощью ``lsmod > ~/test-machine-lsmod``. Затем
  найдите конфигурацию сборки для работающего ядра (подсказки о том, где её найти, см.
  в ':ref:`Начните определять конфигурацию сборки для своего ядра <oldconfig_bisref>`')
  и сохраните её как '~/test-machine-config-working'. Перенесите оба файла в домашний
  каталог вашего хоста сборки.

* Продолжите руководство на хосте сборки (например, с ':ref:`Обеспечьте достаточно
  свободного места для сборки [...] <diskspace_bissbs>`').

* Когда вы дойдёте до ':ref:`Начните подготовку конфигурации сборки ядра [...]
  <oldconfig_bissbs>`': прежде чем впервые выполнить ``make olddefconfig``, выполните
  следующую команду, чтобы основать свою конфигурацию на конфигурации «рабочего» ядра
  тестовой машины::

    cp ~/test-machine-config-working ~/linux/.config

* Во время следующего шага по ':ref:`отключению любых явно лишних модулей ядра
  <localmodconfig_bissbs>`' используйте вместо этого следующую команду::

    yes '' | make localmodconfig LSMOD=~/lsmod_foo-machine localmodconfig

* Продолжайте руководство, но игнорируйте инструкции, описывающие, как компилировать,
  устанавливать и перезагружаться в ядро каждый раз, когда они встречаются. Вместо
  этого собирайте так::

    cp ~/kernel-config-working .config
    make olddefconfig &&
    make -j $(nproc --all) targz-pkg

  Это сгенерирует gzip-сжатый tar-файл, имя которого выводится в показанной последней
  строке; например, ядро с идентификатором kernelrelease
  '6.0.0-rc1-local-g928a87efa423', собранное для машин x86, обычно будет сохранено как
  '~/linux/linux-6.0.0-rc1-local-g928a87efa423-x86.tar.gz'.

  Скопируйте этот файл в домашний каталог вашей тестовой машины.

* Переключитесь на тестовую машину, чтобы проверить, достаточно ли у вас места для
  размещения ещё одного ядра. Затем распакуйте перенесённый вами файл::

    sudo tar -xvzf ~/linux-6.0.0-rc1-local-g928a87efa423-x86.tar.gz -C /

  Затем :ref:`сгенерируйте initramfs и добавьте ядро в конфигурацию вашего загрузчика
  <install_bisref>`; на некоторых дистрибутивах следующая команда позаботится об обеих
  этих задачах::

    sudo /sbin/installkernel 6.0.0-rc1-local-g928a87efa423 /boot/vmlinuz-6.0.0-rc1-local-g928a87efa423

  Теперь перезагрузитесь и убедитесь, что вы запустили предполагаемое ядро.

Этот подход работает даже при сборке для другой архитектуры: просто установите
кросс-компиляторы и добавьте соответствующие параметры к каждому вызову make
(например, ``make ARCH=arm64 CROSS_COMPILE=aarch64-linux-gnu- [...]``).
