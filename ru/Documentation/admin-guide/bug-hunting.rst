Такие трассировки стека предоставляют достаточно информации, чтобы определить
строку в исходном коде ядра, где произошёл баг. В зависимости от серьёзности
проблемы они также могут содержать слово **Oops**, как в этом примере::

	BUG: unable to handle kernel NULL pointer dereference at   (null)
	IP: [<c06969d4>] iret_exc+0x7d0/0xa59
	*pdpt = 000000002258a001 *pde = 0000000000000000
	Oops: 0002 [#1] PREEMPT SMP
	...

Независимо от того, является ли это **Oops** или каким-то другим видом
трассировки стека, для выявления и устранения бага обычно требуется проблемная
строка. На протяжении этой главы мы будем называть «Oops» все виды трассировок
стека, которые нужно анализировать.

Если ядро скомпилировано с ``CONFIG_DEBUG_INFO``, вы можете повысить качество
трассировки стека, используя ``scripts/decode_stacktrace.sh``.

Modules linked in
-----------------

Модули, помеченные как tainted, либо находящиеся в процессе загрузки или
выгрузки, отмечаются символами «(...)», где флаги tainted описаны в
Documentation/admin-guide/tainted-kernels.rst, «находится в процессе загрузки»
обозначается символом «+», а «находится в процессе выгрузки» обозначается
символом «-».


Где находится сообщение Oops?
-----------------------------

Обычно текст Oops считывается из буферов ядра с помощью klogd и передаётся
``syslogd``, который записывает его в файл syslog, как правило
``/var/log/messages`` (зависит от ``/etc/syslog.conf``). В системах с
systemd он также может сохраняться демоном ``journald`` и быть доступен
при запуске команды ``journalctl``.

Иногда ``klogd`` завершается аварийно; в этом случае вы можете выполнить
``dmesg > file``, чтобы считать данные из буферов ядра и сохранить их. Либо
можно выполнить ``cat /proc/kmsg > file``, однако для остановки передачи вам
придётся прервать процесс, поскольку ``kmsg`` — это «бесконечный файл».

Если машина дала сбой настолько серьёзный, что вы не можете вводить команды
или диск недоступен, то у вас есть три варианта:

(1) Перепишите текст с экрана от руки и введите его после перезапуска машины.
    Это неудобно, но это единственный вариант, если вы не подготовились к сбою.
    В качестве альтернативы можно сфотографировать экран цифровой камерой —
    не лучший вариант, но лучше, чем ничего. Если сообщения прокручиваются за
    верхний край консоли, может оказаться, что загрузка с более высоким
    разрешением (например, ``vga=791``) позволит вам прочитать больше текста.
    (Оговорка: для этого требуется ``vesafb``, поэтому такой способ не поможет
    с «ранними» oops.)

(2) Загрузитесь с последовательной консолью (см.
    :ref:`Documentation/admin-guide/serial-console.rst <serial_console>`),
    подключите нуль-модем ко второй машине и захватите вывод там с помощью
    вашей любимой программы для связи. Minicom хорошо подходит для этого.

(3) Используйте Kdump (см. Documentation/admin-guide/kdump/kdump.rst),
    извлеките кольцевой буфер ядра из старой памяти с помощью gdb-макроса dmesg
    из Documentation/admin-guide/kdump/gdbmacros.txt.

Поиск местоположения бага
-------------------------

Сообщение об ошибке работает лучше всего, если вы указываете местоположение
бага в исходном файле ядра. Для этого есть два метода. Обычно использование
``gdb`` проще, но ядро должно быть предварительно скомпилировано с отладочной
информацией.

gdb
^^^

Отладчик GNU (``gdb``) — это лучший способ выяснить точный файл и номер строки
для OOPS из файла ``vmlinux``.

Использование gdb лучше всего работает на ядре, скомпилированном с
``CONFIG_DEBUG_INFO``. Это можно задать, выполнив::

  $ ./scripts/config -d COMPILE_TEST -e DEBUG_KERNEL -e DEBUG_INFO

На ядре, скомпилированном с ``CONFIG_DEBUG_INFO``, вы можете просто скопировать
значение EIP из OOPS::

 EIP:    0060:[<c021e50e>]    Not tainted VLI

И воспользоваться GDB, чтобы перевести его в понятную человеку форму::

  $ gdb vmlinux
  (gdb) l *0xc021e50e

Если у вас не включён ``CONFIG_DEBUG_INFO``, используйте смещение функции
из OOPS::

 EIP is at vt_ioctl+0xda8/0x1482

И перекомпилируйте ядро с включённым ``CONFIG_DEBUG_INFO``::

  $ ./scripts/config -d COMPILE_TEST -e DEBUG_KERNEL -e DEBUG_INFO
  $ make vmlinux
  $ gdb vmlinux
  (gdb) l *vt_ioctl+0xda8
  0x1888 is in vt_ioctl (drivers/tty/vt/vt_ioctl.c:293).
  288	{
  289		struct vc_data *vc = NULL;
  290		int ret = 0;
  291
  292		console_lock();
  293		if (VT_BUSY(vc_num))
  294			ret = -EBUSY;
  295		else if (vc_num)
  296			vc = vc_deallocate(vc_num);
  297		console_unlock();

или, если вы хотите больше подробностей::

  (gdb) p vt_ioctl
  $1 = {int (struct tty_struct *, unsigned int, unsigned long)} 0xae0 <vt_ioctl>
  (gdb) l *0xae0+0xda8

Вместо этого вы могли бы использовать объектный файл::

  $ make drivers/tty/
  $ gdb drivers/tty/vt/vt_ioctl.o
  (gdb) l *vt_ioctl+0xda8

Если у вас есть трассировка вызовов, такая как::

     Call Trace:
      [<ffffffff8802c8e9>] :jbd:log_wait_commit+0xa3/0xf5
      [<ffffffff810482d9>] autoremove_wake_function+0x0/0x2e
      [<ffffffff8802770b>] :jbd:journal_stop+0x1be/0x1ee
      ...

она показывает, что проблема, вероятно, в модуле :jbd:. Вы можете загрузить
этот модуль в gdb и вывести соответствующий код::

  $ gdb fs/jbd/jbd.ko
  (gdb) l *log_wait_commit+0xa3

.. note::

     То же самое можно проделать для любого вызова функции в трассировке
     стека, например для такого::

	 [<f80bc9ca>] ? dvb_usb_adapter_frontend_exit+0x3a/0x70 [dvb_usb]

     Позицию, где произошёл указанный выше вызов, можно увидеть с помощью::

	$ gdb drivers/media/usb/dvb-usb/dvb-usb.o
	(gdb) l *dvb_usb_adapter_frontend_exit+0x3a

objdump
^^^^^^^

Для отладки ядра используйте objdump и найдите шестнадцатеричное смещение из
вывода сбоя, чтобы определить нужную строку кода/ассемблера. Без отладочных
символов вы увидите ассемблерный код показанной процедуры, но если в вашем ядре
есть отладочные символы, то будет доступен и код на C. (Отладочные символы можно
включить в меню kernel hacking при конфигурировании через меню.) Например::

    $ objdump -r -S -l --disassemble net/ipv4/tcp.o

.. note::

   Чтобы это подхватило ваши файлы на C, вам нужно находиться на верхнем уровне
   дерева ядра.

Если у вас нет доступа к исходному коду, вы всё равно можете отладить некоторые
дампы сбоев следующим методом (пример вывода дампа сбоя, показанный
Dave Miller)::

     EIP is at 	+0x14/0x4c0
      ...
     Code: 44 24 04 e8 6f 05 00 00 e9 e8 fe ff ff 8d 76 00 8d bc 27 00 00
     00 00 55 57  56 53 81 ec bc 00 00 00 8b ac 24 d0 00 00 00 8b 5d 08
     <8b> 83 3c 01 00 00 89 44  24 14 8b 45 28 85 c0 89 44 24 18 0f 85

     Поместите байты в файл «foo.s» следующим образом:

            .text
            .globl foo
     foo:
            .byte  .... /* bytes from Code: part of OOPS dump */

     Скомпилируйте его командой «gcc -c -o foo.o foo.s», затем посмотрите
     на вывод команды «objdump --disassemble foo.o».

     Вывод:

     ip_queue_xmit:
         push       %ebp
         push       %edi
         push       %esi
         push       %ebx
         sub        $0xbc, %esp
         mov        0xd0(%esp), %ebp        ! %ebp = arg0 (skb)
         mov        0x8(%ebp), %ebx         ! %ebx = skb->sk
         mov        0x13c(%ebx), %eax       ! %eax = inet_sk(sk)->opt

Для автоматизации большей части этого процесса можно использовать
``scripts/decodecode``, в зависимости от того, какая архитектура CPU
отлаживается.

Сообщение об ошибке
-------------------

Как только вы найдёте, где произошёл баг, исследовав его местоположение,
вы можете либо попытаться исправить его самостоятельно, либо сообщить о нём
в апстрим.

Чтобы сообщить о нём в апстрим, вам следует определить баг-трекер, если он есть,
или список рассылки, используемый для разработки затронутого кода. Это можно
сделать с помощью скрипта ``get_maintainer.pl``.

Например, если вы обнаружили баг в файле sonixj.c из gspca, вы можете получить
его сопровождающих с помощью::

	$ ./scripts/get_maintainer.pl --bug -f drivers/media/usb/gspca/sonixj.c
	Hans Verkuil <hverkuil@kernel.org> (odd fixer:GSPCA USB WEBCAM DRIVER,commit_signer:1/1=100%)
	Mauro Carvalho Chehab <mchehab@kernel.org> (maintainer:MEDIA INPUT INFRASTRUCTURE (V4L/DVB),commit_signer:1/1=100%)
	Tejun Heo <tj@kernel.org> (commit_signer:1/1=100%)
	Bhaktipriya Shridhar <bhaktipriya96@gmail.com> (commit_signer:1/1=100%,authored:1/1=100%,added_lines:4/4=100%,removed_lines:9/9=100%)
	linux-media@vger.kernel.org (open list:GSPCA USB WEBCAM DRIVER)
	linux-kernel@vger.kernel.org (open list)

Обратите внимание, что он укажет на:

- Последних разработчиков, которые трогали исходный код (если это делается
  внутри git-дерева). В примере выше — Tejun и Bhaktipriya (в данном конкретном
  случае никто из них на самом деле не участвовал в разработке этого файла);
- Сопровождающего драйвера (Hans Verkuil);
- Сопровождающего подсистемы (Mauro Carvalho Chehab);
- Список рассылки драйвера и/или подсистемы (linux-media@vger.kernel.org);
- Список рассылки ядра Linux (linux-kernel@vger.kernel.org);
- URI для сообщения об ошибках драйвера/подсистемы (в примере выше отсутствуют).

Если в перечне в конце содержатся URI для сообщения об ошибках, предпочтите их
электронной почте. В противном случае сообщайте об ошибках в список рассылки,
используемый для разработки кода (список рассылки linux-media), с копией
сопровождающему драйвера (Hans).

Если вы совершенно не представляете, кому отправить сообщение, и
``get_maintainer.pl`` не дал вам ничего полезного, отправьте его в
linux-kernel@vger.kernel.org.

Спасибо за вашу помощь в том, чтобы сделать Linux настолько стабильным,
насколько это по-человечески возможно.

Исправление бага
----------------

Если вы умеете программировать, вы могли бы помочь нам, не только сообщив о баге,
но и предоставив нам решение. В конце концов, open source — это про то, чтобы
делиться тем, что вы делаете, и разве вам не хочется, чтобы вашу гениальность
признали?

Если вы решите пойти этим путём, то, проработав исправление, отправьте его,
пожалуйста, в апстрим.

Однако, пожалуйста, прочтите
:ref:`Documentation/process/submitting-patches.rst <submittingpatches>`,
чтобы помочь принятию вашего кода.

Замечания об отслеживании Oops с помощью ``klogd``
--------------------------------------------------

Чтобы помочь Linus и другим разработчикам ядра, в ``klogd`` была встроена
значительная поддержка обработки protection fault. Чтобы иметь полную поддержку
разрешения адресов, следует использовать как минимум версию 1.3-pl3 пакета
``sysklogd``.

Когда происходит protection fault, демон ``klogd`` автоматически переводит
важные адреса в журнальных сообщениях ядра в их символьные эквиваленты. Это
переведённое сообщение ядра затем пересылается через любой механизм
информирования, который использует ``klogd``. Сообщение о protection fault можно
просто вырезать из файлов сообщений и переслать разработчикам ядра.

``klogd`` выполняет два типа разрешения адресов. Первый — это статический
перевод, а второй — динамический перевод. Статический перевод использует файл
System.map. Чтобы выполнять статический перевод, демон ``klogd`` должен иметь
возможность найти файл system map во время инициализации демона. Информацию о
том, как ``klogd`` ищет файлы map, см. на man-странице klogd.

Динамический перевод адресов важен, когда используются загружаемые модули ядра.
Поскольку память для модулей ядра выделяется из динамических пулов памяти ядра,
не существует фиксированных местоположений ни для начала модуля, ни для функций
и символов в модуле.

Ядро поддерживает системные вызовы, которые позволяют программе определить,
какие модули загружены и где они расположены в памяти. Используя эти системные
вызовы, демон klogd строит таблицу символов, которую можно использовать для
отладки protection fault, происходящего в загружаемом модуле ядра.

Как минимум klogd предоставит имя модуля, который сгенерировал protection fault.
Может быть доступна дополнительная символьная информация, если разработчик
загружаемого модуля решил экспортировать символьную информацию из модуля.

Поскольку окружение модулей ядра может быть динамическим, должен существовать
механизм для уведомления демона ``klogd``, когда происходит изменение в окружении
модулей. Доступны параметры командной строки, которые позволяют klogd подать
сигнал выполняющемуся в данный момент демону о том, что символьную информацию
следует обновить. Дополнительную информацию см. на man-странице ``klogd``.

В состав дистрибутива sysklogd включён патч, который модифицирует пакет
``modules-2.0.0`` так, чтобы он автоматически подавал сигнал klogd при загрузке
или выгрузке любого модуля. Применение этого патча обеспечивает по существу
бесшовную поддержку отладки protection fault, происходящих в загружаемых модулях
ядра.
