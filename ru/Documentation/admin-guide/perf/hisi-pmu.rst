==========================================================================
Блок мониторинга производительности (PMU) uncore для SoC HiSilicon
==========================================================================

Чип SoC HiSilicon включает различные независимые системные устройства PMU,
такие как L3 cache (L3C), Hydra Home Agent (HHA) и DDRC. Эти PMU
независимы и имеют аппаратную логику для сбора статистики и информации о
производительности.

SoC HiSilicon объединяет несколько CPU- и IO-кристаллов (die). Каждый
кластер CPU (CCL) состоит из 4 ядер CPU, разделяющих один L3 cache; каждый
CPU-кристалл называется Super CPU cluster (SCCL) и состоит из 6 CCL. Каждый
SCCL имеет соответственно два HHA (0 - 1) и четыре DDRC (0 - 3).

Драйвер PMU uncore для SoC HiSilicon
------------------------------------

Каждое устройство PMU имеет отдельные регистры для подсчёта событий,
управления и прерываний, и драйвер PMU должен регистрировать perf-драйверы
PMU, такие как L3C, HHA и DDRC и т.п. Доступные события и параметры
конфигурации должны быть описаны в sysfs, см.::

/sys/bus/event_source/devices/hisi_sccl{X}_<l3c{Y}/hha{Y}/ddrc{Y}>

Команда "perf list" должна выводить список доступных событий из sysfs.

Каждый L3C, HHA и DDRC регистрируется в perf как отдельный PMU. Имя PMU
будет появляться в списке событий как hisi_sccl<sccl-id>_module<index-id>,
где "sccl-id" — это идентификатор SCCL, а "index-id" — индекс модуля.

Например, hisi_sccl3_l3c0/rd_hit_cpipe — это событие READ_HIT_CPIPE для L3C
с индексом #0 в SCCL ID #3.

Например, hisi_sccl1_hha0/rx_operations — это событие RX_OPERATIONS для HHA
с индексом #0 в SCCL ID #1.

Драйвер также предоставляет атрибут sysfs "cpumask", который показывает
идентификатор ядра CPU, используемого для подсчёта события PMU uncore. Также
предоставляется атрибут sysfs "associated_cpus", показывающий CPU,
связанные с этим PMU. "cpumask" указывает CPU, на которых нужно открывать
события, обычно в качестве подсказки для инструментов пространства
пользователя, таких как perf. Он содержит только один связанный CPU из
"associated_cpus".

Пример использования perf::

  $# perf list
  hisi_sccl3_l3c0/rd_hit_cpipe/ [kernel PMU event]
  ------------------------------------------
  hisi_sccl3_l3c0/wr_hit_cpipe/ [kernel PMU event]
  ------------------------------------------
  hisi_sccl1_l3c0/rd_hit_cpipe/ [kernel PMU event]
  ------------------------------------------
  hisi_sccl1_l3c0/wr_hit_cpipe/ [kernel PMU event]
  ------------------------------------------

  $# perf stat -a -e hisi_sccl3_l3c0/rd_hit_cpipe/ sleep 5
  $# perf stat -a -e hisi_sccl3_l3c0/config=0x02/ sleep 5

Для PMU uncore HiSilicon v2, чей идентификатор равен 0x30, топология та же,
что и у PMU v1, но в оборудование добавлены некоторые новые функции.

1. L3C PMU поддерживает фильтрацию по core/thread в пределах кластера,
которая может быть задана в виде битовой маски::

  $# perf stat -a -e hisi_sccl3_l3c0/config=0x02,tt_core=0x3/ sleep 5

Это будет подсчитывать только операции от core/thread 0 и 1 в данном
кластере.

Пользователю не следует использовать tt_core_deprecated для задания
фильтрации по core/thread. Этот параметр предоставлен для обратной
совместимости и поддерживает только 8 бит, что может не охватывать все
core/thread, разделяющие L3C.

2. Tracetag позволяет пользователю выбрать подсчёт только операций чтения,
записи или атомарных операций через параметр tt_req в perf. Значение по
умолчанию подсчитывает все операции. tt_req имеет 3 бита: 3'b100 обозначает
операции чтения, 3'b101 обозначает операции записи, 3'b110 обозначает
атомарные операции store, а 3'b111 обозначает атомарные операции не-store,
остальные значения зарезервированы::

  $# perf stat -a -e hisi_sccl3_l3c0/config=0x02,tt_req=0x4/ sleep 5

Это будет подсчитывать только операции чтения в данном кластере.

3. Datasrc позволяет пользователю проверить, откуда поступают данные. Он
имеет 5 бит. Некоторые важные коды приведены ниже:

- 5'b00001: поступают из L3C в этом кристалле (die);
- 5'b01000: поступают из L3C в другом кристалле (cross-die);
- 5'b01001: поступают из L3C, который находится в другом сокете;
- 5'b01110: поступают из локального DDR;
- 5'b01111: поступают из DDR в другом кристалле (cross-die);
- 5'b10000: поступают из DDR в другом сокете (cross-socket);

и т.д., в основном это помогает определить, что источник данных ближе всего
к ядрам CPU. Если datasrc_cfg используется на нескольких чипах, в команде
perf необходимо настроить datasrc_skt::

  $# perf stat -a -e hisi_sccl3_l3c0/config=0xb9,datasrc_cfg=0xE/,
  hisi_sccl3_l3c0/config=0xb9,datasrc_cfg=0xF/ sleep 5

4. Некоторые SoC HiSilicon объединяют несколько CPU- и IO-кристаллов (die).
Каждый CPU-кристалл содержит несколько Compute Clusters (CCL). IO-кристаллы
называются Super I/O clusters (SICL) и содержат несколько I/O clusters
(ICL). Каждый CCL/ICL в SoC имеет уникальный ID. Каждый ID имеет 11 бит и
включает 6-битный SCCL-ID и 5-битный CCL/ICL-ID. Для IO-кристалла за ICL-ID
следует:

- 5'b00000: I/O_MGMT_ICL;
- 5'b00001: Network_ICL;
- 5'b00011: HAC_ICL;
- 5'b10000: PCIe_ICL;

5. uring_channel: события UC PMU 0x47~0x59 поддерживают фильтрацию по
каналу uring запросов tx. Он имеет 2 бита. Некоторые важные коды приведены
ниже:

- 2'b11: подсчитывать события, отправленные в канал uring_ext (MATA);
- 2'b01: то же самое, что и 2'b11;
- 2'b10: подсчитывать события, отправленные в канал uring (не-MATA);
- 2'b00: значение по умолчанию, подсчитывать события, отправленные как в
  канал uring, так и в канал uring_ext;

6. ch: NoC PMU поддерживает фильтрацию подсчёта событий определённого
канала транзакций с помощью этого параметра. Текущие поддерживаемые каналы
приведены ниже:

- 3'b010: Request channel
- 3'b100: Snoop channel
- 3'b110: Response channel
- 3'b111: Data channel

7. tt_en: NoC PMU поддерживает подсчёт только тех транзакций, у которых
установлен tracetag, если установлен этот параметр. См. 2-й список для
получения дополнительной информации о tracetag.

Для PMU uncore HiSilicon v3, чей идентификатор равен 0x40, некоторые PMU
uncore дополнительно делятся на части для более тонкой гранулярности
трассировки, каждая часть имеет свой собственный выделенный PMU, и все
такие PMU вместе охватывают задачу мониторинга событий на конкретном
устройстве uncore. Такие PMU описываются в sysfs с немного изменённым
форматом имени::

/sys/bus/event_source/devices/hisi_sccl{X}_<l3c{Y}_{Z}/ddrc{Y}_{Z}/noc{Y}_{Z}>

Z — это под-идентификатор (sub-id), указывающий на разные PMU для части
аппаратного устройства.

Использование большинства PMU с разными sub-id идентично. В частности, L3C
PMU предоставляет параметр ``ext``, чтобы разрешить исследование ещё более
тонкой гранулярности статистики L3C PMU.  Драйвер L3C PMU использует его в
качестве подсказки о завершении при доставке команды perf оборудованию:

- ext=0: по умолчанию, может использоваться с именами событий.
- ext=1 и ext=2: должны использоваться с кодами событий, имена событий не
  поддерживаются.

Пример команды perf может выглядеть так::

  $# perf stat -a -e hisi_sccl0_l3c1_0/rd_spipe/ sleep 5

или::

  $# perf stat -a -e hisi_sccl0_l3c1_0/event=0x1,ext=1/ sleep 5

Как показано выше, ``hisi_sccl0_l3c1_0`` указывает на PMU Super CPU CLuster
0, L3 cache 1, pipe0.

Первая команда указывает на первую часть L3C, поскольку ``ext=0``
подразумевается по умолчанию. Вторая команда запускает подсчёт на другой
части L3C с событием ``0x1``.

Пользователи могут настроить ID для подсчёта данных, поступающих из
определённого CCL/ICL, установив srcid_cmd и srcid_msk, и данных,
предназначенных для определённого CCL/ICL, установив tgtid_cmd и tgtid_msk.
Установленный бит в srcid_msk/tgtid_msk означает, что PMU не будет
проверять этот бит при сопоставлении с srcid_cmd/tgtid_cmd.

Если все эти параметры отключены, он может работать со значением по
умолчанию, которое не различает условие фильтрации и информацию об ID и
будет возвращать суммарные значения счётчиков в счётчиках PMU.

Текущий драйвер не поддерживает сэмплирование. Поэтому "perf record" не
поддерживается. Также не поддерживается присоединение к задаче, так как все
события относятся к uncore.

Примечание: при необходимости обратитесь к сопровождающему (maintainer) за
полным списком событий, поддерживаемых для устройств PMU в SoC, и
информацией о них.
