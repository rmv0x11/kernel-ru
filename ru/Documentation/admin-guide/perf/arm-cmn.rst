=============================
Arm Coherent Mesh Network PMU
=============================

CMN-600 — это настраиваемое сетчатое межсоединение (mesh interconnect),
состоящее из прямоугольной решётки точек пересечения (crosspoints, XP),
где каждая точка пересечения поддерживает до двух портов устройств, к
которым подключаются различные агенты AMBA CHI.

CMN реализует распределённую конструкцию PMU как часть своей функциональности
отладки и трассировки. Она состоит из локального монитора (DTM) в каждом XP,
который подсчитывает до 4 сигналов событий от подключённых узлов устройств
и/или от самого XP. Переполнение этих локальных счётчиков накапливается в
не более чем 8 глобальных счётчиках, реализованных в главном контроллере
(DTC), который обеспечивает общее управление PMU и прерывания при переполнении
глобальных счётчиков.

События PMU
-----------

Драйвер PMU регистрирует одно устройство PMU для всего межсоединения,
см. /sys/bus/event_source/devices/arm_cmn_0. В многочиповых системах
несколько CMN могут быть связаны вместе через внешние линии CCIX — в этой
ситуации каждая сеть подсчитывает свои события полностью независимо, а
дополнительные устройства PMU будут именоваться arm_cmn_{1..n}.

Большинство событий задаётся в формате, основанном непосредственно на
определениях TRM: "type" выбирает соответствующий тип узла, а "eventid" —
номер события. Некоторые события требуют дополнительного идентификатора
занятости (occupancy ID), который задаётся через "occupid".

* Поскольку узлы RN-D не имеют каких-либо событий, отличных от узлов RN-I,
  они рассматриваются как один и тот же тип (0xa), и общие шаблоны событий
  именуются "rnid_*".

* Счётчик циклов рассматривается как синтетическое событие, принадлежащее
  узлу DTC ("type" == 0x3, "eventid" игнорируется).

* События XP также кодируют порт и канал в поле "eventid", чтобы
  соответствовать лежащему в основе кодированию pmu_event0_id для регистра
  pmu_event_sel. Шаблоны событий именуются с префиксами, чтобы охватить все
  перестановки.

По умолчанию каждое событие предоставляет агрегированный подсчёт по всем
узлам заданного типа. Чтобы нацелиться на конкретный узел, "bynodeid"
должно быть установлено в 1, а "nodeid" — в соответствующее значение,
полученное из конфигурации CMN (как определено в разделе "Node ID Mapping"
в TRM).

Watchpoints
-----------

PMU также может подсчитывать события watchpoint для мониторинга
определённого трафика flit. Watchpoints рассматриваются как синтетический
тип событий и, как и события PMU, могут быть глобальными или нацеленными с
помощью значения "nodeid" конкретного XP. Поскольку направление watchpoint
в остальных случаях неявно задаётся лежащим в основе выбором регистра, для
выгрузок (uploads) и загрузок (downloads) flit предоставляются отдельные
события.

Значение совпадения flit и маска передаются в config1 и config2 ("val" и
"mask" соответственно). "wp_dev_sel", "wp_chn_sel", "wp_grp" и
"wp_exclusive" задаются согласно определениям TRM для dtm_wp_config0. Когда
watchpoint должен совпадать с полями из обеих групп совпадения на канале REQ
или SNP, он может быть задан как два события — по одному для каждой группы —
с одним и тем же ненулевым значением "combine". Подсчёт для такой пары
объединённых событий будет отнесён к первичному совпадению. События
watchpoint со значением "combine", равным 0, считаются независимыми и будут
подсчитываться по отдельности.
