==========================================================
Модуль мониторинга производительности (PMU) HiSilicon PCIe
==========================================================

В Hip09 модуль мониторинга производительности (PMU) HiSilicon PCIe может отслеживать
данные о пропускной способности, задержке (latency), загрузке шины и заполненности
буфера PCIe.

Каждое ядро PCIe Core имеет PMU для мониторинга нескольких портов Root Port этого
ядра PCIe Core и всех оконечных устройств Endpoint, расположенных ниже по дереву
от этих портов Root Port.


Драйвер HiSilicon PCIe PMU
==========================

Драйвер PCIe PMU регистрирует perf PMU с именем, состоящим из его sicl-id и
идентификатора PCIe Core.::

  /sys/bus/event_source/hisi_pcie<sicl>_core<core>

Драйвер PMU предоставляет описание доступных событий и параметров фильтрации в sysfs,
см. /sys/bus/event_source/devices/hisi_pcie<sicl>_core<core>.

Каталог "format" описывает все форматы полей config (события) и config1
(параметры фильтрации) структуры perf_event_attr. Каталог "events"
описывает все документированные события, отображаемые в perf list.

Файл sysfs "identifier" позволяет пользователям определить версию
аппаратного устройства PMU.

Файл sysfs "bus" позволяет пользователям получить номер шины портов Root Port,
отслеживаемых PMU. Кроме того, пользователи могут получить диапазон портов Root Port
в виде [bdf_min, bdf_max] из атрибутов sysfs "bdf_min" и "bdf_max"
соответственно.

Пример использования perf::

  $# perf list
  hisi_pcie0_core0/rx_mwr_latency/ [kernel PMU event]
  hisi_pcie0_core0/rx_mwr_cnt/ [kernel PMU event]
  ------------------------------------------

  $# perf stat -e hisi_pcie0_core0/rx_mwr_latency,port=0xffff/
  $# perf stat -e hisi_pcie0_core0/rx_mwr_cnt,port=0xffff/

Связанные события обычно используются для расчёта пропускной способности, задержки
и прочего. Они должны начинать и заканчивать подсчёт в одно и то же время, поэтому
связанные события лучше всего использовать в одной группе событий, чтобы получить
ожидаемое значение. Есть два способа узнать, являются ли события связанными:

a) По имени события, например, события задержки "xxx_latency, xxx_cnt" или
   события пропускной способности "xxx_flux, xxx_time".
b) По типу события, например, "event=0xXXXX, event=0x1XXXX".

Пример использования группы perf::

  $# perf stat -e "{hisi_pcie0_core0/rx_mwr_latency,port=0xffff/,hisi_pcie0_core0/rx_mwr_cnt,port=0xffff/}"

Текущий драйвер не поддерживает выборку (sampling). Поэтому "perf record" не поддерживается.
Также для PCIe PMU не поддерживается привязка к задаче.

Параметры фильтрации
--------------------

1. Целевой фильтр (Target filter)

   PMU может отслеживать производительность только трафика, направленного вниз по дереву
   к целевым портам Root Port или к целевому оконечному устройству Endpoint. Драйвер
   PCIe PMU поддерживает интерфейсы "port" и "bdf" для пользователей.
   Обратите внимание, что один из этих двух интерфейсов должен быть задан, и эти два
   интерфейса не поддерживаются одновременно. Если заданы оба, действителен только
   фильтр "port".
   Если фильтр "port" не задан или явно установлен в ноль (по умолчанию), будет
   действовать фильтр "bdf", поскольку "bdf=0" означает 0000:000:00.0.

   - port

     Фильтр "port" может использоваться во всех событиях PCIe PMU, целевой Root Port
     можно выбрать, настроив 16-битную битовую карту "port". Для событий уровня AP
     можно выбрать несколько портов, а для событий уровня TL/DL можно выбрать только
     один порт.

     Например, если целевой Root Port — это 0000:00:00.0 (x8 линий), должен быть
     установлен bit0 битовой карты, port=0x1; если целевой Root Port — это 0000:00:04.0
     (x4 линии), устанавливается bit8, port=0x100; если отслеживаются оба этих порта
     Root Port, port=0x101.

     Пример использования perf::

       $# perf stat -e hisi_pcie0_core0/rx_mwr_latency,port=0x1/ sleep 5

   - bdf

     Фильтр "bdf" может использоваться только в событиях пропускной способности, целевое
     оконечное устройство Endpoint выбирается путём настройки BDF в "bdf". Счётчик считает
     только пропускную способность сообщений, запрошенных целевым Endpoint.

     Например, "bdf=0x3900" означает, что BDF целевого Endpoint — это 0000:39:00.0.

     Пример использования perf::

       $# perf stat -e hisi_pcie0_core0/rx_mrd_flux,bdf=0x3900/ sleep 5

2. Фильтр триггера (Trigger filter)

   Сбор статистики событий начинается в момент, когда длина TLP впервые становится
   больше/меньше условия триггера. Вы можете задать условие триггера, записав
   "trig_len", и задать режим триггера, записав "trig_mode". Этот фильтр может
   использоваться только в событиях пропускной способности.

   Например, "trig_len=4" означает, что условие триггера — 2^4 DW, "trig_mode=0"
   означает, что сбор статистики начинается, когда длина TLP > условия триггера, "trig_mode=1"
   означает начало, когда длина TLP < условия.

   Пример использования perf::

     $# perf stat -e hisi_pcie0_core0/rx_mrd_flux,port=0xffff,trig_len=0x4,trig_mode=1/ sleep 5

3. Пороговый фильтр (Threshold filter)

   Счётчик считает, когда длина TLP находится в заданном диапазоне. Вы можете задать
   порог, записав "thr_len", и задать режим порога, записав "thr_mode". Этот фильтр может
   использоваться только в событиях пропускной способности.

   Например, "thr_len=4" означает, что порог — 2^4 DW, "thr_mode=0" означает, что
   счётчик считает, когда длина TLP >= порога, а "thr_mode=1" означает подсчёт,
   когда длина TLP < порога.

   Пример использования perf::

     $# perf stat -e hisi_pcie0_core0/rx_mrd_flux,port=0xffff,thr_len=0x4,thr_mode=1/ sleep 5

4. Фильтр длины TLP (TLP Length filter)

   При подсчёте пропускной способности данные могут состоять из определённых частей
   пакетов TLP. Вы можете указать это через "len_mode":

   - 2'b00: Зарезервировано (не используйте это значение, поскольку поведение не определено)
   - 2'b01: Пропускная способность полезной нагрузки TLP
   - 2'b10: Пропускная способность заголовков TLP
   - 2'b11: Пропускная способность как полезной нагрузки TLP, так и заголовков

   Например, "len_mode=2" означает подсчёт только пропускной способности заголовков TLP,
   а "len_mode=3" означает, что итоговые данные пропускной способности состоят как из
   заголовков TLP, так и из полезной нагрузки. Значение по умолчанию, если не указано, — 2'b11.

   Пример использования perf::

     $# perf stat -e hisi_pcie0_core0/rx_mrd_flux,port=0xffff,len_mode=0x1/ sleep 5
