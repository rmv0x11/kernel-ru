==========================
Бисекция регрессии
==========================

В этом документе описано, как с помощью ``git bisect`` найти изменение в
исходном коде, которое что-то сломало, — например, когда некоторая
функциональность перестала работать после обновления с Linux 6.0 на 6.1.

Текст сосредоточен на сути процесса. Если вы впервые занимаетесь бисекцией
ядра, лучше следуйте вместо него Documentation/admin-guide/verify-bugs-and-bisect-regressions.rst:
там всё описано от начала до конца и охвачено множество
аспектов, о которых даже разработчики ядра иногда забывают. Сюда входит и
раннее обнаружение ситуаций, когда бисекция была бы пустой тратой времени,
потому что её результат никого не заинтересует, — например, потому что
проблема возникает после того, как ядро пометило себя как «tainted»,
проявляется в заброшенной версии, уже была исправлена или вызвана
изменением .config, которое выполнили вы или ваш дистрибутив Linux.

Поиск изменения, вызвавшего проблему в ядре, с помощью бисекции
===============================================================

*Примечание: дальнейший процесс предполагает, что вы подготовили всё
необходимое для бисекции. Сюда входит наличие Git-клона с соответствующими
исходниками, установка программного обеспечения, необходимого для сборки и
установки ядер, а также файл .config, сохранённый в надёжном месте
(в следующем примере предполагается '~/prepared_kernel_.config'), который
используется как нетронутая основа на каждом шаге бисекции; в идеале вы также
проработали полностью надёжный и прямолинейный способ воспроизведения
регрессии.*

* Подготовка: запустите бисекцию и сообщите Git'у точки в истории, которые вы
  считаете работающими и сломанными, что Git называет 'good' и 'bad'::

     git bisect start
     git bisect good v6.0
     git bisect bad v6.1

  Вместо тегов Git наподобие 'v6.0' и 'v6.1' можно также указать commit-id.

1. Скопируйте подготовленный .config в каталог сборки и приведите его в
   соответствие с потребностями кодовой базы, которую Git извлёк для
   тестирования::

     cp ~/prepared_kernel_.config .config
     make olddefconfig

2. Теперь соберите, установите и загрузите ядро. Это может завершиться
   неудачей по не связанным с регрессией причинам, например, когда на текущем
   этапе бисекции происходит ошибка компиляции, которую устраняет более
   позднее изменение. В таких случаях выполните ``git bisect skip`` и
   вернитесь к шагу 1.

3. Проверьте, работает ли в только что собранном ядре функциональность, в
   которой произошла регрессия.

   Если она работает, выполните::

     git bisect good

   Если она сломана, выполните::

     git bisect bad

   Учтите, что одна-единственная ошибка здесь полностью собьёт с курса всю
   оставшуюся бисекцию. Поэтому, чтобы не пришлось начинать заново, нужно
   убедиться, что вы сообщаете Git'у корректные сведения; так что зачастую
   разумно потратить несколько лишних минут на тестирование, если ваш способ
   воспроизведения ненадёжен.

   После выполнения одной из этих двух команд Git обычно извлекает другую
   точку бисекции и печатает что-то вроде 'Bisecting: 675 revisions left to
   test after this (roughly 10 steps)'. В этом случае вернитесь к шагу 1.

   Если же Git печатает что-то вроде 'cafecaca0c0dacafecaca0c0dacafecaca0c0da
   is the first bad commit', значит, бисекция завершена. В этом случае
   переходите к следующему пункту ниже. Учтите, что сразу после вывода этой
   строки Git покажет некоторые подробности о виновнике, включая описание его
   патча; это легко может заполнить весь терминал, поэтому вам, возможно,
   придётся прокрутить вверх, чтобы увидеть сообщение с commit-id виновника.

   Если вы пропустили вывод Git'а, вы всегда можете выполнить ``git bisect
   log``, чтобы напечатать статус: он покажет, сколько шагов осталось, или
   укажет результат бисекции.

* Рекомендуемая дополнительная задача: отложите в сторону журнал бисекции и
  текущий файл .config для отчёта об ошибке; кроме того, велите Git'у сбросить
  исходники в состояние, бывшее до бисекции::

     git bisect log > ~/bisection-log
     cp .config ~/bisection-config-culprit
     git bisect reset

* Рекомендуемая необязательная задача: попробуйте откатить виновника поверх
  самой свежей кодовой базы и проверить, исправляет ли это вашу ошибку; если
  это так, это подтверждает бисекцию и позволяет разработчикам устранить
  регрессию через откат.

  Чтобы попробовать это, обновите свой клон и извлеките свежий mainline. Затем
  велите Git'у откатить изменение, указав его commit-id::

     git revert --no-edit cafec0cacaca0

  Git может отклонить это, например, когда бисекция привела к merge-коммиту. В
  этом случае откажитесь от попытки. Сделайте то же самое, если Git не сможет
  самостоятельно откатить виновника, потому что от него зависят более поздние
  изменения, — по крайней мере, если только вы не бисектировали серию stable-
  или longterm-ядра, в случае которой вам стоит извлечь её самую свежую
  кодовую базу и попробовать откат там.

  Если откат удался, соберите и протестируйте ещё одно ядро, чтобы проверить,
  устранил ли откат вашу регрессию.

На этом процесс завершён. Теперь сообщите о регрессии так, как описано в
Documentation/admin-guide/reporting-issues.rst.

Бисекция linux-next
-------------------

Если вы сталкиваетесь с проблемой, возникающей только в linux-next,
бисектируйте между ветками linux-next 'stable' и 'master'. Следующие команды
запустят процесс для дерева linux-next, которое вы добавили как удалённый
репозиторий с именем 'next'::

  git bisect start
  git bisect good next/stable
  git bisect bad next/master

Ветка 'stable' указывает на состояние linux-mainline, на котором основан
текущий релиз linux-next (находящийся в ветке 'master'), — поэтому первая
должна быть свободна от любых проблем, которые проявляются в -next, но не в
дереве Линуса.

Это бисектирует по широкому диапазону изменений, некоторые из которых вы,
возможно, использовали в более ранних релизах linux-next без проблем. К
сожалению, простого способа избежать их проверки нет: бисекция от одного
релиза linux-next к более позднему (скажем, между 'next-20241020' и
'next-20241021') невозможна, так как у них нет общей истории.
