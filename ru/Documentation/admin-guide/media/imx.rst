Драйвер видеозахвата i.MX
=========================

Введение
--------

Freescale i.MX5/6 содержит Image Processing Unit (IPU), который
управляет потоком кадров изображения к устройствам захвата и от них,
а также к устройствам отображения и от них.

Для захвата изображения IPU содержит следующие внутренние подблоки:

- Image DMA Controller (IDMAC)
- Camera Serial Interface (CSI)
- Image Converter (IC)
- Sensor Multi-FIFO Controller (SMFC)
- Image Rotator (IRT)
- Video De-Interlacing or Combining Block (VDIC)

IDMAC — это DMA-контроллер для передачи кадров изображения в память и
из неё. Существуют различные выделенные DMA-каналы как для путей
видеозахвата, так и для путей отображения. Во время передачи IDMAC также
способен выполнять вертикальный поворот изображения, передачу блоков
8x8 (см. описание IRT), переупорядочивание компонентов пикселей
(например, UYVY в YUYV) в пределах того же цветового пространства, а
также преобразование packed <--> planar. IDMAC также может выполнять
простое деинтерлейсинг путём чередования чётных и нечётных строк во время
передачи (без компенсации движения, которая требует VDIC).

CSI — это бэкенд-блок захвата, который напрямую взаимодействует с
сенсорами камер по шинам Parallel, BT.656/1120 и MIPI CSI-2.

IC выполняет операции преобразования цветового пространства, изменения
размера (уменьшение и увеличение), горизонтального поворота и поворота на
90/270 градусов.

Внутри IC есть три независимые «задачи» (tasks), которые могут выполнять
преобразования одновременно: pre-process encoding, pre-process viewfinder
и post-processing. В каждой задаче преобразования разбиты на три секции:
секция уменьшения, основная секция (увеличение, поворот, преобразование
цветового пространства и совмещение графической плоскости) и секция
вращения.

IPU разделяет операции задач IC по времени. Гранулярность временного
кванта составляет один пакет (burst) из восьми пикселей в секции
уменьшения, одну строку изображения в основной секции обработки и один
кадр изображения в секции вращения.

SMFC состоит из четырёх независимых FIFO, каждый из которых может
одновременно передавать захваченные кадры от сенсоров напрямую в память
через четыре канала IDMAC.

IRT выполняет операции поворота изображения на 90 и 270 градусов.
Операция поворота выполняется над блоками 8x8 пикселей за раз. Эта
операция поддерживается IDMAC, который выполняет передачу блоков 8x8
вместе с переупорядочиванием блоков, в координации с вертикальным
поворотом.

VDIC выполняет преобразование чересстрочного видео в прогрессивное, с
поддержкой различных режимов компенсации движения (низкий, средний и
высокий уровень движения). Деинтерлейсированные выходные кадры из VDIC
могут отправляться в задачу IC pre-process viewfinder для дальнейших
преобразований. VDIC также содержит блок Combiner, который совмещает две
плоскости изображения с альфа-смешиванием и цветовым ключом.

Помимо внутренних подблоков IPU, есть также два блока за пределами IPU,
которые также участвуют в видеозахвате на i.MX:

- MIPI CSI-2 Receiver для сенсоров камер с интерфейсом шины MIPI CSI-2.
  Это ядро Synopsys DesignWare.
- Два видеомультиплексора для выбора одного из нескольких сенсорных
  входов для отправки в CSI.

Дополнительную информацию см. в последних версиях справочных руководств
по i.MX5/6 [#f1]_ и [#f2]_.


Возможности
-----------

К числу возможностей этого драйвера относятся:

- Множество различных конвейеров (pipelines) можно настроить через
  media controller API, что соответствует аппаратным конвейерам
  видеозахвата, поддерживаемым в i.MX.

- Поддержка интерфейсов parallel, BT.565 и MIPI CSI-2.

- Одновременные независимые потоки за счёт настройки конвейеров на
  несколько интерфейсов видеозахвата с использованием независимых
  сущностей (entities).

- Масштабирование, преобразование цветового пространства, горизонтальный
  и вертикальный поворот, а также вращение изображения через subdev-ы
  задач IC.

- Поддержка множества пиксельных форматов (RGB, packed и planar YUV,
  partial planar YUV).

- Subdev VDIC поддерживает деинтерлейсинг с компенсацией движения, с
  тремя режимами компенсации движения: низкий, средний и высокий уровень
  движения. Определены конвейеры, позволяющие отправлять кадры в subdev
  VDIC напрямую из CSI. В будущем также предусмотрена поддержка отправки
  кадров в VDIC из буферов памяти через устройства output/mem2mem.

- Включает Frame Interval Monitor (FIM), который может исправлять
  проблемы вертикальной синхронизации с видеодекодерами ADV718x.

Сущности
--------

imx6-mipi-csi2
--------------

Это сущность приёмника MIPI CSI-2. У неё есть один sink pad для приёма
потока MIPI CSI-2 (обычно от сенсора камеры MIPI CSI-2). У неё есть
четыре source pad, соответствующие четырём демультиплексированным
выходам виртуальных каналов MIPI CSI-2. Несколько source pad могут быть
включены для независимой потоковой передачи из нескольких виртуальных
каналов.

Эта сущность фактически состоит из двух подблоков. Один из них — ядро
MIPI CSI-2. Это ядро Synopsys Designware MIPI CSI-2. Другой подблок — это
«CSI-2 to IPU gasket». Этот gasket действует как демультиплексор потоков
четырёх виртуальных каналов, предоставляя четыре отдельные параллельные
шины, содержащие каждый виртуальный канал, которые маршрутизируются к CSI
или видеомультиплексорам, как описано ниже.

На i.MX6 solo/dual-lite все четыре шины виртуальных каналов
маршрутизируются к двум видеомультиплексорам. И CSI0, и CSI1 могут
принимать любой виртуальный канал, выбираемый видеомультиплексорами.

На i.MX6 Quad виртуальный канал 0 маршрутизируется к IPU1-CSI0 (после
выбора видеомультиплексором), виртуальные каналы 1 и 2 жёстко подключены
к IPU1-CSI1 и IPU2-CSI0 соответственно, а виртуальный канал 3
маршрутизируется к IPU2-CSI1 (снова выбираемый видеомультиплексором).

ipuX_csiY_mux
-------------

Это видеомультиплексоры. У них есть два или более sink pad для выбора
либо из сенсоров камер с параллельным интерфейсом, либо из виртуальных
каналов MIPI CSI-2 от сущности imx6-mipi-csi2. У них есть единственный
source pad, который маршрутизируется к CSI (сущности ipuX_csiY).

На i.MX6 solo/dual-lite есть две сущности video mux. Одна находится перед
IPU1-CSI0 для выбора между параллельным сенсором и любым из четырёх
виртуальных каналов MIPI CSI-2 (всего пять sink pad). Другой mux
находится перед IPU1-CSI1 и снова имеет пять sink pad для выбора между
параллельным сенсором и любым из четырёх виртуальных каналов MIPI CSI-2.

На i.MX6 Quad есть две сущности video mux. Одна находится перед IPU1-CSI0
для выбора между параллельным сенсором и виртуальным каналом 0 MIPI CSI-2
(два sink pad). Другой mux находится перед IPU2-CSI1 для выбора между
параллельным сенсором и виртуальным каналом 3 MIPI CSI-2 (два sink pad).

ipuX_csiY
---------

Это сущности CSI. У них есть единственный sink pad, принимающий либо от
video mux, либо от виртуального канала MIPI CSI-2, как описано выше.

У этой сущности есть два source pad. Первый source pad может напрямую
связываться с сущностью ipuX_vdic или сущностью ipuX_ic_prp с
использованием аппаратных связей, которые не требуют передачи буфера
памяти через IDMAC.

Когда прямой source pad маршрутизируется к сущности ipuX_ic_prp, кадры из
CSI могут обрабатываться одной или обеими задачами предварительной
обработки IC.

Когда прямой source pad маршрутизируется к сущности ipuX_vdic, VDIC будет
выполнять деинтерлейсинг с компенсацией движения в режиме «high motion»
(см. описание сущности ipuX_vdic).

Второй source pad отправляет видеокадры напрямую в буферы памяти через
SMFC и канал IDMAC, минуя предварительную обработку IC. Этот source pad
маршрутизируется к узлу устройства захвата с именем узла формата
«ipuX_csiY capture».

Обратите внимание, что поскольку source pad IDMAC использует канал IDMAC,
переупорядочивание пикселей в пределах того же цветового пространства
может выполняться каналом IDMAC. Например, если sink pad CSI принимает в
порядке UYVY, устройство захвата, связанное с source pad IDMAC, может
захватывать в порядке YUYV. Также, если sink pad CSI принимает packed
YUV-формат, устройство захвата может захватывать planar YUV-формат, такой
как YUV420.

Канал IDMAC на source pad IDMAC также поддерживает простое чередование
(interweave) без компенсации движения, которое активируется, если тип
поля source pad — последовательный top-bottom или bottom-top, а
запрошенный тип поля интерфейса захвата установлен в interlaced (t-b, b-t
или неуточнённый interlaced). Интерфейс захвата будет применять тот же
порядок полей, что и порядок полей source pad (interlaced-bt, если source
pad равен seq-bt; interlaced-tb, если source pad равен seq-tb).

О событиях, производимых ipuX_csiY, см. ref:`imx_api_ipuX_csiY`.

Обрезка (cropping) в ipuX_csiY
------------------------------

CSI поддерживает обрезку входящих сырых кадров сенсора. Это реализовано в
сущностях ipuX_csiY на sink pad с использованием crop selection subdev
API.

CSI также поддерживает фиксированное уменьшение в два раза независимо по
ширине и высоте. Это реализовано в сущностях ipuX_csiY на sink pad с
использованием compose selection subdev API.

Выходной прямоугольник на source pad ipuX_csiY совпадает с прямоугольником
compose на sink pad. Поэтому прямоугольник source pad не может быть
согласован, он должен быть установлен с помощью compose selection API на
sink pad (если требуется уменьшение /2, иначе прямоугольник source pad
равен входящему прямоугольнику).

Чтобы привести пример обрезки и уменьшения /2: этот пример обрежет
входной кадр 1280x960 до 640x480, а затем уменьшит /2 по обоим измерениям
до 320x240 (предполагается, что ipu1_csi0 связан с ipu1_csi0_mux):

.. code-block:: none

   media-ctl -V "'ipu1_csi0_mux':2[fmt:UYVY2X8/1280x960]"
   media-ctl -V "'ipu1_csi0':0[crop:(0,0)/640x480]"
   media-ctl -V "'ipu1_csi0':0[compose:(0,0)/320x240]"

Пропуск кадров (frame skipping) в ipuX_csiY
-------------------------------------------

CSI поддерживает прореживание частоты кадров за счёт пропуска кадров.
Прореживание частоты кадров задаётся установкой интервалов между кадрами
на sink и source pad. Затем сущность ipuX_csiY применяет наилучшую
настройку пропуска кадров к CSI для достижения желаемой частоты кадров на
source pad.

Следующий пример вдвое уменьшает предполагаемую входящую частоту кадров
60 Гц на выходном source pad IDMAC:

.. code-block:: none

   media-ctl -V "'ipu1_csi0':0[fmt:UYVY2X8/640x480@1/60]"
   media-ctl -V "'ipu1_csi0':2[fmt:UYVY2X8/640x480@1/30]"

----------------------------------

ipuX_vdic
---------

VDIC выполняет деинтерлейсинг с компенсацией движения, с тремя режимами
компенсации движения: низкий, средний и высокий уровень движения. Режим
задаётся с помощью menu control V4L2_CID_DEINTERLACING_MODE. У VDIC есть
два sink pad и единственный source pad.

Прямой sink pad принимает от прямого pad ipuX_csiY. С этой связью VDIC
может работать только в режиме high motion.

Когда активирован sink pad IDMAC, он принимает от узла устройства output
или mem2mem. С этим конвейером VDIC также может работать в режимах low и
medium, потому что эти режимы требуют приёма кадров из буферов памяти.
Обратите внимание, что устройство output или mem2mem ещё не реализовано,
поэтому у этого sink pad в настоящее время нет связей.

Source pad маршрутизируется к сущности предварительной обработки IC
ipuX_ic_prp.

ipuX_ic_prp
-----------

Это сущность предварительной обработки IC. Она действует как
маршрутизатор, направляя данные со своего sink pad на один или оба своих
source pad.

У этой сущности есть единственный sink pad. Sink pad может принимать от
прямого pad ipuX_csiY или от ipuX_vdic.

У этой сущности есть два source pad. Один source pad маршрутизируется к
сущности задачи pre-process encode (ipuX_ic_prpenc), другой — к сущности
задачи pre-process viewfinder (ipuX_ic_prpvf). Оба source pad могут быть
активированы одновременно, если sink pad принимает от ipuX_csiY. Только
source pad к сущности задачи pre-process viewfinder может быть
активирован, если sink pad принимает от ipuX_vdic (кадры из VDIC могут
обрабатываться только задачей pre-process viewfinder).

ipuX_ic_prpenc
--------------

Это сущность предварительной обработки IC encode. У неё есть единственный
sink pad от ipuX_ic_prp и единственный source pad. Source pad
маршрутизируется к узлу устройства захвата с именем узла формата
«ipuX_ic_prpenc capture».

Эта сущность выполняет операции задачи IC pre-process encode:
преобразование цветового пространства, изменение размера (уменьшение и
увеличение), горизонтальный и вертикальный поворот, а также вращение на
90/270 градусов. Поворот и вращение предоставляются через стандартные
элементы управления V4L2.

Как и source IDMAC ipuX_csiY, эта сущность также поддерживает простой
деинтерлейсинг без компенсации движения и переупорядочивание пикселей.

ipuX_ic_prpvf
-------------

Это сущность предварительной обработки IC viewfinder. У неё есть
единственный sink pad от ipuX_ic_prp и единственный source pad. Source
pad маршрутизируется к узлу устройства захвата с именем узла формата
«ipuX_ic_prpvf capture».

Эта сущность идентична по работе ipuX_ic_prpenc, с теми же операциями
изменения размера и CSC и элементами управления поворотом/вращением. Она
будет принимать и обрабатывать деинтерлейсированные кадры из ipuX_vdic,
если ipuX_ic_prp принимает от ipuX_vdic.

Как и source IDMAC ipuX_csiY, эта сущность поддерживает простое
чередование (interweaving) без компенсации движения. Однако обратите
внимание, что если ipuX_vdic включён в конвейер (ipuX_ic_prp принимает от
ipuX_vdic), то использовать чередование в ipuX_ic_prpvf невозможно,
поскольку ipuX_vdic уже выполнил деинтерлейсинг (с компенсацией движения)
и поэтому тип поля на выходе из ipuX_vdic может быть только none
(progressive).

Конвейеры захвата
-----------------

Ниже описаны различные сценарии использования, поддерживаемые
конвейерами.

Показанные связи не включают связи бэкенд-сенсора, video mux или
приёмника mipi csi-2. Это зависит от типа интерфейса сенсора (parallel
или mipi csi-2). Поэтому эти конвейеры начинаются с:

sensor -> ipuX_csiY_mux -> ...

для параллельных сенсоров, или:

sensor -> imx6-mipi-csi2 -> (ipuX_csiY_mux) -> ...

для сенсоров mipi csi-2. Приёмнику imx6-mipi-csi2 может потребоваться
маршрутизация к video mux (ipuX_csiY_mux) перед отправкой в CSI, в
зависимости от виртуального канала mipi csi-2, поэтому ipuX_csiY_mux
показан в скобках.

Необработанный видеозахват:
---------------------------

Отправка кадров напрямую от сенсора к узлу интерфейса устройства камеры,
без преобразований, через source pad IDMAC ipuX_csiY:

-> ipuX_csiY:2 -> ipuX_csiY capture

Прямые преобразования IC:
-------------------------

Этот конвейер использует сущность preprocess encode для маршрутизации
кадров напрямую от CSI к IC, чтобы выполнить масштабирование до
разрешения 1024x1024, CSC, повороты и вращение изображения:

-> ipuX_csiY:1 -> 0:ipuX_ic_prp:1 -> 0:ipuX_ic_prpenc:1 -> ipuX_ic_prpenc capture

Деинтерлейсинг с компенсацией движения:
---------------------------------------

Этот конвейер маршрутизирует кадры от прямого pad CSI к сущности VDIC для
поддержки деинтерлейсинга с компенсацией движения (только режим high
motion), масштабирования до 1024x1024, CSC, поворота и вращения:

-> ipuX_csiY:1 -> 0:ipuX_vdic:2 -> 0:ipuX_ic_prp:2 -> 0:ipuX_ic_prpvf:1 -> ipuX_ic_prpvf capture


Замечания по использованию
--------------------------

Для помощи в настройке и для обратной совместимости с приложениями V4L2,
которые обращаются к элементам управления только из узлов
видеоустройств, интерфейсы устройств захвата наследуют элементы
управления от активных сущностей в текущем конвейере, поэтому к элементам
управления можно обращаться либо напрямую из subdev, либо из активного
интерфейса устройства захвата. Например, элементы управления FIM доступны
либо из subdev-ов ipuX_csiY, либо из активного устройства захвата.

Ниже приведены конкретные замечания по использованию для эталонных плат
Sabre*:


i.MX6Q SabreLite с OV5642 и OV5640
----------------------------------

Эта платформа требует модуль OmniVision OV5642 с параллельным
интерфейсом камеры и модуль OV5640 с интерфейсом MIPI CSI-2. Оба модуля
доступны от Boundary Devices:

- https://boundarydevices.com/product/nit6x_5mp
- https://boundarydevices.com/product/nit6x_5mp_mipi

Обратите внимание, что если доступен только один модуль камеры, другой
узел сенсора может быть отключён в дереве устройств (device tree).

Модуль OV5642 подключён к параллельному входу шины на внутреннем video
mux i.MX к IPU1 CSI0. Его шина i2c подключается к шине i2c 2.

Модуль MIPI CSI-2 OV5640 подключён к внутреннему приёмнику MIPI CSI-2
i.MX, и четыре выхода виртуальных каналов от приёмника маршрутизируются
следующим образом: vc0 к mux IPU1 CSI0, vc1 напрямую к IPU1 CSI1, vc2
напрямую к IPU2 CSI0 и vc3 к mux IPU2 CSI1. OV5640 на SabreLite также
подключён к шине i2c 2, поэтому OV5642 и OV5640 не должны использовать
один и тот же slave-адрес i2c.

Следующий базовый пример настраивает конвейеры необработанного
видеозахвата для обоих сенсоров. OV5642 маршрутизируется к ipu1_csi0, а
OV5640, передающий по виртуальному каналу 1 MIPI CSI-2 (которым является
pad 2 imx6-mipi-csi2), маршрутизируется к ipu1_csi1. Оба сенсора
настроены на вывод 640x480, причём OV5642 выводит YUYV2X8, а OV5640 —
UYVY2X8:

.. code-block:: none

   # Setup links for OV5642
   media-ctl -l "'ov5642 1-0042':0 -> 'ipu1_csi0_mux':1[1]"
   media-ctl -l "'ipu1_csi0_mux':2 -> 'ipu1_csi0':0[1]"
   media-ctl -l "'ipu1_csi0':2 -> 'ipu1_csi0 capture':0[1]"
   # Setup links for OV5640
   media-ctl -l "'ov5640 1-0040':0 -> 'imx6-mipi-csi2':0[1]"
   media-ctl -l "'imx6-mipi-csi2':2 -> 'ipu1_csi1':0[1]"
   media-ctl -l "'ipu1_csi1':2 -> 'ipu1_csi1 capture':0[1]"
   # Configure pads for OV5642 pipeline
   media-ctl -V "'ov5642 1-0042':0 [fmt:YUYV2X8/640x480 field:none]"
   media-ctl -V "'ipu1_csi0_mux':2 [fmt:YUYV2X8/640x480 field:none]"
   media-ctl -V "'ipu1_csi0':2 [fmt:AYUV32/640x480 field:none]"
   # Configure pads for OV5640 pipeline
   media-ctl -V "'ov5640 1-0040':0 [fmt:UYVY2X8/640x480 field:none]"
   media-ctl -V "'imx6-mipi-csi2':2 [fmt:UYVY2X8/640x480 field:none]"
   media-ctl -V "'ipu1_csi1':2 [fmt:AYUV32/640x480 field:none]"

Потоковую передачу затем можно начать независимо на узлах устройств
захвата «ipu1_csi0 capture» и «ipu1_csi1 capture». Утилита v4l2-ctl может
использоваться для выбора любого поддерживаемого пиксельного формата YUV
на узлах устройств захвата, включая planar.

i.MX6Q SabreAuto с декодером ADV7180
------------------------------------

На i.MX6Q SabreAuto встроенный SD-декодер ADV7180 подключён к
параллельному входу шины на внутреннем video mux к IPU1 CSI0.

Следующий пример настраивает конвейер для захвата с видеодекодера
ADV7180, предполагая входные сигналы NTSC 720x480, с использованием
простого чередования (interweave) (без преобразования и без компенсации
движения). adv7180 должен выводить последовательные или чередующиеся поля
(тип поля 'seq-bt' для NTSC или 'alternate'):

.. code-block:: none

   # Setup links
   media-ctl -l "'adv7180 3-0021':0 -> 'ipu1_csi0_mux':1[1]"
   media-ctl -l "'ipu1_csi0_mux':2 -> 'ipu1_csi0':0[1]"
   media-ctl -l "'ipu1_csi0':2 -> 'ipu1_csi0 capture':0[1]"
   # Configure pads
   media-ctl -V "'adv7180 3-0021':0 [fmt:UYVY2X8/720x480 field:seq-bt]"
   media-ctl -V "'ipu1_csi0_mux':2 [fmt:UYVY2X8/720x480]"
   media-ctl -V "'ipu1_csi0':2 [fmt:AYUV32/720x480]"
   # Configure "ipu1_csi0 capture" interface (assumed at /dev/video4)
   v4l2-ctl -d4 --set-fmt-video=field=interlaced_bt

Потоковую передачу затем можно начать на /dev/video4. Утилита v4l2-ctl
также может использоваться для выбора любого поддерживаемого пиксельного
формата YUV на /dev/video4.

Этот пример настраивает конвейер для захвата с видеодекодера ADV7180,
предполагая входные сигналы PAL 720x576, с деинтерлейсингом с компенсацией
движения. adv7180 должен выводить последовательные или чередующиеся поля
(тип поля 'seq-tb' для PAL или 'alternate').

.. code-block:: none

   # Setup links
   media-ctl -l "'adv7180 3-0021':0 -> 'ipu1_csi0_mux':1[1]"
   media-ctl -l "'ipu1_csi0_mux':2 -> 'ipu1_csi0':0[1]"
   media-ctl -l "'ipu1_csi0':1 -> 'ipu1_vdic':0[1]"
   media-ctl -l "'ipu1_vdic':2 -> 'ipu1_ic_prp':0[1]"
   media-ctl -l "'ipu1_ic_prp':2 -> 'ipu1_ic_prpvf':0[1]"
   media-ctl -l "'ipu1_ic_prpvf':1 -> 'ipu1_ic_prpvf capture':0[1]"
   # Configure pads
   media-ctl -V "'adv7180 3-0021':0 [fmt:UYVY2X8/720x576 field:seq-tb]"
   media-ctl -V "'ipu1_csi0_mux':2 [fmt:UYVY2X8/720x576]"
   media-ctl -V "'ipu1_csi0':1 [fmt:AYUV32/720x576]"
   media-ctl -V "'ipu1_vdic':2 [fmt:AYUV32/720x576 field:none]"
   media-ctl -V "'ipu1_ic_prp':2 [fmt:AYUV32/720x576 field:none]"
   media-ctl -V "'ipu1_ic_prpvf':1 [fmt:AYUV32/720x576 field:none]"
   # Configure "ipu1_ic_prpvf capture" interface (assumed at /dev/video2)
   v4l2-ctl -d2 --set-fmt-video=field=none

Потоковую передачу затем можно начать на /dev/video2. Утилита v4l2-ctl
также может использоваться для выбора любого поддерживаемого пиксельного
формата YUV на /dev/video2.

Эта платформа принимает аналоговые входы Composite Video на ADV7180 на
Ain1 (разъём J42).

i.MX6Q SabreSD с MIPI CSI-2 OV5640
----------------------------------

Аналогично i.MX6Q SabreLite, i.MX6Q SabreSD поддерживает модуль OV5642 с
параллельным интерфейсом на IPU1 CSI0 и модуль MIPI CSI-2 OV5640. OV5642
подключается к шине i2c 1, а OV5640 — к шине i2c 2.

Дерево устройств (device tree) для SabreSD включает графы OF как для
параллельного OV5642, так и для MIPI CSI-2 OV5640, но на момент написания
этого текста тестировался только MIPI CSI-2 OV5640, поэтому узел OV5642 в
настоящее время отключён. Модуль OV5640 подключается к MIPI-разъёму J5.
Номер детали NXP для модуля OV5640, который подключается к плате SabreSD,
— H120729.

Следующий пример настраивает конвейер необработанного видеозахвата для
захвата с OV5640, передающего по виртуальному каналу 0 MIPI CSI-2:

.. code-block:: none

   # Setup links
   media-ctl -l "'ov5640 1-003c':0 -> 'imx6-mipi-csi2':0[1]"
   media-ctl -l "'imx6-mipi-csi2':1 -> 'ipu1_csi0_mux':0[1]"
   media-ctl -l "'ipu1_csi0_mux':2 -> 'ipu1_csi0':0[1]"
   media-ctl -l "'ipu1_csi0':2 -> 'ipu1_csi0 capture':0[1]"
   # Configure pads
   media-ctl -V "'ov5640 1-003c':0 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'imx6-mipi-csi2':1 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'ipu1_csi0_mux':0 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'ipu1_csi0':0 [fmt:AYUV32/640x480]"

Потоковую передачу затем можно начать на узле «ipu1_csi0 capture».
Утилита v4l2-ctl может использоваться для выбора любого поддерживаемого
пиксельного формата на узле устройства захвата.

Чтобы определить, какой узел /dev/video соответствует
«ipu1_csi0 capture»:

.. code-block:: none

   media-ctl -e "ipu1_csi0 capture"
   /dev/video0

В этом случае потоковым элементом является /dev/video0.

Запуск потоковой передачи через v4l2-ctl:

.. code-block:: none

   v4l2-ctl --stream-mmap -d /dev/video0

Запуск потоковой передачи через Gstreamer и отправка содержимого на дисплей:

.. code-block:: none

   gst-launch-1.0 v4l2src device=/dev/video0 ! kmssink

Следующий пример настраивает конвейер прямого преобразования для захвата
с OV5640, передающего по виртуальному каналу 0 MIPI CSI-2. Он также
показывает преобразование цветового пространства и масштабирование на
выходе IC.

.. code-block:: none

   # Setup links
   media-ctl -l "'ov5640 1-003c':0 -> 'imx6-mipi-csi2':0[1]"
   media-ctl -l "'imx6-mipi-csi2':1 -> 'ipu1_csi0_mux':0[1]"
   media-ctl -l "'ipu1_csi0_mux':2 -> 'ipu1_csi0':0[1]"
   media-ctl -l "'ipu1_csi0':1 -> 'ipu1_ic_prp':0[1]"
   media-ctl -l "'ipu1_ic_prp':1 -> 'ipu1_ic_prpenc':0[1]"
   media-ctl -l "'ipu1_ic_prpenc':1 -> 'ipu1_ic_prpenc capture':0[1]"
   # Configure pads
   media-ctl -V "'ov5640 1-003c':0 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'imx6-mipi-csi2':1 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'ipu1_csi0_mux':2 [fmt:UYVY2X8/640x480]"
   media-ctl -V "'ipu1_csi0':1 [fmt:AYUV32/640x480]"
   media-ctl -V "'ipu1_ic_prp':1 [fmt:AYUV32/640x480]"
   media-ctl -V "'ipu1_ic_prpenc':1 [fmt:ARGB8888_1X32/800x600]"
   # Set a format at the capture interface
   v4l2-ctl -d /dev/video1 --set-fmt-video=pixelformat=RGB3

Потоковую передачу затем можно начать на узле «ipu1_ic_prpenc capture».

Чтобы определить, какой узел /dev/video соответствует
«ipu1_ic_prpenc capture»:

.. code-block:: none

   media-ctl -e "ipu1_ic_prpenc capture"
   /dev/video1


В этом случае потоковым элементом является /dev/video1.

Запуск потоковой передачи через v4l2-ctl:

.. code-block:: none

   v4l2-ctl --stream-mmap -d /dev/video1

Запуск потоковой передачи через Gstreamer и отправка содержимого на дисплей:

.. code-block:: none

   gst-launch-1.0 v4l2src device=/dev/video1 ! kmssink

Известные проблемы
------------------

1. При использовании элемента управления вращением на 90 или 270 градусов
   при разрешениях захвата вблизи предела ресайзера IC в 1024x1024 и в
   сочетании с planar пиксельными форматами (YUV420, YUV422p) захват
   кадров часто будет давать сбой без прерываний end-of-frame от канала
   IDMAC. Чтобы обойти это, используйте более низкое разрешение и/или
   packed форматы (YUYV, RGB3 и т. д.), когда требуется вращение на 90
   или 270 градусов.
