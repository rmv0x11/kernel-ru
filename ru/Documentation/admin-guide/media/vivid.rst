Виртуальный тестовый видеодрайвер (vivid)
==========================================

Этот драйвер эмулирует оборудование video4linux различных типов: захват видео, вывод
видео, захват и вывод vbi, захват и вывод метаданных, радиоприёмники и
радиопередатчики, захват касаний (touch) и приёмник программно-определяемого радио (SDR). Кроме того,
доступно простое устройство фреймбуфера для тестирования оверлеев захвата и вывода.

Можно создать до 64 экземпляров vivid, каждый из которых имеет до 16 входов и 16 выходов.

Каждый вход может быть веб-камерой, устройством TV-захвата, устройством захвата S-Video или
устройством захвата HDMI. Каждый выход может быть устройством вывода S-Video или устройством
вывода HDMI.

Эти входы и выходы ведут себя в точности так же, как вело бы себя реальное оборудование. Это
позволяет использовать данный драйвер как тестовый вход для разработки приложений, поскольку
вы можете тестировать различные возможности, не требуя специального оборудования.

Этот документ описывает возможности, реализованные данным драйвером:

- Поддержка потокового ввода-вывода read()/write(), MMAP, USERPTR и DMABUF.
- Большой список тестовых паттернов и их вариаций
- Работающие регуляторы яркости, контрастности, насыщенности и оттенка
- Поддержка альфа-компонента цвета
- Полная поддержка цветовых пространств, включая ограниченный/полный диапазон RGB
- Присутствуют все возможные типы регуляторов
- Поддержка различных соотношений сторон пикселя и соотношений сторон видео
- Инъекция ошибок для проверки того, что происходит при возникновении ошибок
- Поддержка обрезки/композиции/масштабирования в любой комбинации как для входа, так и для выхода
- Способность эмулировать разрешения вплоть до 4K
- Поддерживаются все настройки Field для тестирования чересстрочного захвата
- Поддерживаются все стандартные форматы YUV и RGB, включая два многоплоскостных формата YUV
- Поддержка захвата и вывода Raw и Sliced VBI
- Поддержка радиоприёмника и радиопередатчика, включая поддержку RDS
- Поддержка программно-определяемого радио (SDR)
- Поддержка оверлеев захвата и вывода
- Поддержка захвата и вывода метаданных
- Поддержка захвата касаний (touch)

Эти возможности будут описаны подробнее ниже.

Настройка драйвера
------------------

По умолчанию драйвер создаёт один экземпляр, имеющий устройство захвата видео
с входами веб-камеры, TV, S-Video и HDMI, устройство вывода видео с
выходами S-Video и HDMI, одно устройство захвата vbi, одно устройство вывода vbi, одно
устройство радиоприёмника, одно устройство радиопередатчика и одно устройство SDR.

Количество экземпляров, устройств, видеовходов и видеовыходов и их типы —
всё это настраивается с помощью следующих опций модуля:

- n_devs:

	количество создаваемых экземпляров драйвера. По умолчанию равно 1. Можно создать
	до 64 экземпляров.

- node_types:

	какие устройства должен создавать каждый экземпляр драйвера. Массив
	шестнадцатеричных значений, по одному для каждого экземпляра. По умолчанию 0xe1d3d.
	Каждое значение является битовой маской со следующим смыслом:

		- бит 0: узел Video Capture
		- бит 2-3: узел VBI Capture: 0 = нет, 1 = raw vbi, 2 = sliced vbi, 3 = оба
		- бит 4: узел Radio Receiver
		- бит 5: узел Software Defined Radio Receiver
		- бит 8: узел Video Output
		- бит 10-11: узел VBI Output: 0 = нет, 1 = raw vbi, 2 = sliced vbi, 3 = оба
		- бит 12: узел Radio Transmitter
		- бит 16: Framebuffer для тестирования оверлеев
		- бит 17: узел Metadata Capture
		- бит 18: узел Metadata Output
		- бит 19: узел Touch Capture

	Так, чтобы создать четыре экземпляра, первые два — с одним устройством захвата
	видео, вторые два — с одним устройством вывода видео, вы передали бы
	драйверу vivid следующие опции модуля:

	.. code-block:: none

		n_devs=4 node_types=0x1,0x1,0x100,0x100

- num_inputs:

	количество входов, по одному для каждого экземпляра. По умолчанию для каждого
	устройства захвата видео создаётся 4 входа. Можно создать не более 16 входов,
	и должен быть хотя бы один.

- input_types:

	типы входов для каждого экземпляра, по умолчанию 0xe4. Это определяет,
	какой тип имеет каждый вход при создании входов для каждого экземпляра
	драйвера. Это шестнадцатеричное значение, содержащее до 16 пар битов, каждая
	пара задаёт тип, и биты 0-1 соответствуют входу 0, биты 2-3 соответствуют входу 1,
	30-31 соответствуют входу 15. Каждая пара битов имеет следующий смысл:

		- 00: это вход веб-камеры
		- 01: это вход TV-тюнера
		- 10: это вход S-Video
		- 11: это вход HDMI

	Так, чтобы создать устройство захвата видео с 8 входами, где вход 0 — это
	TV-тюнер, входы 1-3 — входы S-Video, а входы 4-7 — входы HDMI, вы
	использовали бы следующие опции модуля:

	.. code-block:: none

		num_inputs=8 input_types=0xffa9

- num_outputs:

	количество выходов, по одному для каждого экземпляра. По умолчанию для каждого
	устройства вывода видео создаётся 2 выхода. Можно создать не более 16
	выходов, и должен быть хотя бы один.

- output_types:

	типы выходов для каждого экземпляра, по умолчанию 0x02. Это определяет,
	какой тип имеет каждый выход при создании выходов для каждого
	экземпляра драйвера. Это шестнадцатеричное значение, содержащее до 16 битов, каждый бит
	задаёт тип, и бит 0 соответствует выходу 0, бит 1 соответствует выходу 1, бит
	15 соответствует выходу 15. Смысл каждого бита следующий:

		- 0: это выход S-Video
		- 1: это выход HDMI

	Так, чтобы создать устройство вывода видео с 8 выходами, где выходы 0-3 —
	выходы S-Video, а выходы 4-7 — выходы HDMI, вы использовали бы
	следующие опции модуля:

	.. code-block:: none

		num_outputs=8 output_types=0xf0

- vid_cap_nr:

	задаёт желаемый начальный номер videoX для каждого устройства захвата видео.
	По умолчанию -1, что означает использование первого свободного номера. Это позволяет
	вам отображать узлы захвата видео на конкретные узлы устройств videoX. Пример:

	.. code-block:: none

		n_devs=4 vid_cap_nr=2,4,6,8

	Это приведёт к попытке назначить /dev/video2 для устройства захвата видео
	первого экземпляра vivid, video4 для следующего и вплоть до video8 для последнего
	экземпляра. Если это не удастся, то будет взят следующий свободный
	номер.

- vid_out_nr:

	задаёт желаемый начальный номер videoX для каждого устройства вывода видео.
	По умолчанию -1, что означает использование первого свободного номера.

- vbi_cap_nr:

	задаёт желаемый начальный номер vbiX для каждого устройства захвата vbi.
	По умолчанию -1, что означает использование первого свободного номера.

- vbi_out_nr:

	задаёт желаемый начальный номер vbiX для каждого устройства вывода vbi.
	По умолчанию -1, что означает использование первого свободного номера.

- radio_rx_nr:

	задаёт желаемый начальный номер radioX для каждого устройства радиоприёмника.
	По умолчанию -1, что означает использование первого свободного номера.

- radio_tx_nr:

	задаёт желаемый начальный номер radioX для каждого устройства радиопередатчика.
	По умолчанию -1, что означает использование первого свободного номера.

- sdr_cap_nr:

	задаёт желаемый начальный номер swradioX для каждого устройства захвата SDR.
	По умолчанию -1, что означает использование первого свободного номера.

- meta_cap_nr:

        задаёт желаемый начальный номер videoX для каждого устройства захвата метаданных.
        По умолчанию -1, что означает использование первого свободного номера.

- meta_out_nr:

        задаёт желаемый начальный номер videoX для каждого устройства вывода метаданных.
        По умолчанию -1, что означает использование первого свободного номера.

- touch_cap_nr:

        задаёт желаемый начальный номер v4l-touchX для каждого устройства захвата касаний.
        По умолчанию -1, что означает использование первого свободного номера.

- ccs_cap_mode:

	задаёт разрешённую комбинацию обрезки/композиции/масштабирования захвата видео
	для каждого экземпляра драйвера. Устройства захвата видео могут иметь любую комбинацию
	возможностей обрезки, композиции и масштабирования, и это указывает
	драйверу vivid, какие из них он должен эмулировать. По умолчанию пользователь может
	выбрать это через регуляторы.

	Значение — это либо -1 (управляется пользователем), либо набор из трёх битов,
	каждый из которых включает (1) или выключает (0) одну из возможностей:

	- бит 0:

		Включить поддержку обрезки. Обрезка будет брать только часть
		входящего изображения.
	- бит 1:

		Включить поддержку композиции. Композиция будет копировать входящее
		изображение в больший буфер.

	- бит 2:

		Включить поддержку масштабирования. Масштабирование может масштабировать
		входящее изображение. Масштабатор драйвера vivid может увеличивать
		или уменьшать вплоть до четырёхкратного исходного размера. Масштабатор
		очень простой и низкокачественный. Ключевыми были простота и скорость,
		а не качество.

	Обратите внимание, что это значение игнорируется входами веб-камеры: они перечисляют
	дискретные размеры кадров, а это несовместимо с обрезкой, композицией
	или масштабированием.

- ccs_out_mode:

	задаёт разрешённую комбинацию обрезки/композиции/масштабирования вывода видео
	для каждого экземпляра драйвера. Устройства вывода видео могут иметь любую комбинацию
	возможностей обрезки, композиции и масштабирования, и это указывает
	драйверу vivid, какие из них он должен эмулировать. По умолчанию пользователь может
	выбрать это через регуляторы.

	Значение — это либо -1 (управляется пользователем), либо набор из трёх битов,
	каждый из которых включает (1) или выключает (0) одну из возможностей:

	- бит 0:

		Включить поддержку обрезки. Обрезка будет брать только часть
		исходящего буфера.

	- бит 1:

		Включить поддержку композиции. Композиция будет копировать входящий
		буфер в больший кадр изображения.

	- бит 2:

		Включить поддержку масштабирования. Масштабирование может масштабировать
		входящий буфер. Масштабатор драйвера vivid может увеличивать
		или уменьшать вплоть до четырёхкратного исходного размера. Масштабатор
		очень простой и низкокачественный. Ключевыми были простота и скорость,
		а не качество.

- multiplanar:

	выбирает, поддерживает ли каждый экземпляр устройства многоплоскостные форматы,
	и, следовательно, многоплоскостной API V4L2. По умолчанию экземпляры устройств
	являются одноплоскостными.

	Эта опция модуля может переопределить это для каждого экземпляра. Значения:

		- 1: это одноплоскостной экземпляр.
		- 2: это многоплоскостной экземпляр.

- vivid_debug:

	включить отладочную информацию драйвера

- no_error_inj:

	если установлено, отключает регуляторы инъекции ошибок. Эта опция
	необходима для запуска инструментов вроде v4l2-compliance. Такие инструменты
	задействуют все регуляторы, включая регулятор вроде 'Disconnect', который
	эмулирует отключение USB, делая устройство недоступным, так что
	все тесты, которые выполняет v4l2-compliance, после этого завершатся неудачей.

	Могут быть и другие ситуации, когда вы захотите отключить
	поддержку инъекции ошибок vivid. Когда эта опция установлена, то
	регуляторы, выбирающие поведение обрезки, композиции и масштабирования, также
	удаляются. Если не переопределено через ccs_cap_mode и/или ccs_out_mode, то они
	по умолчанию будут включать обрезку, композицию и масштабирование.

- allocators:

	выбор аллокатора памяти, по умолчанию 0. Задаёт способ выделения
	буферов.

		- 0: vmalloc
		- 1: dma-contig

- cache_hints:

	задаёт, должно ли устройство устанавливать у очередей способность подсказок
	согласованности кэша пространства пользователя и памяти (V4L2_BUF_CAP_SUPPORTS_MMAP_CACHE_HINTS).
	Подсказки действительны только при использовании потокового ввода-вывода MMAP. По умолчанию 0.

		- 0: запретить подсказки
		- 1: разрешить подсказки

- supports_requests:

	задаёт, должно ли устройство поддерживать Request API. Есть
	три возможных значения, по умолчанию 1:

		- 0: нет запросов
		- 1: поддерживает запросы
		- 2: требует запросы

В совокупности все эти опции модуля позволяют вам точно настроить
поведение драйвера и протестировать ваше приложение со всевозможными перестановками.
Это также очень подходит для эмуляции оборудования, которое ещё недоступно, например
при разработке программного обеспечения для нового предстоящего устройства.


Захват видео
------------

Это, вероятно, наиболее часто используемая возможность. Устройство захвата видео
можно настроить с помощью опций модуля num_inputs, input_types и
ccs_cap_mode (см. "Настройка драйвера" для более подробной информации),
но по умолчанию настраиваются четыре входа: веб-камера, TV-тюнер, вход S-Video
и вход HDMI, по одному входу для каждого типа входа. Они описаны подробнее
ниже.

Особое внимание было уделено скорости, с которой становятся доступны новые кадры.
Джиттер составит около 1 jiffy (это зависит от конфигурации HZ
вашего ядра, так что обычно 1/100, 1/250 или 1/1000 секунды),
но долгосрочное поведение в точности следует частоте кадров. Так что
частота кадров 59.94 Гц действительно отличается от 60 Гц. Если частота кадров
превышает значение HZ вашего ядра, то вы получите потерянные кадры, но
подсчёт последовательности кадров/полей будет отслеживать это, так что счётчик
последовательности будет пропускать значения всякий раз, когда кадры теряются.


Вход веб-камеры
~~~~~~~~~~~~~~~

Вход веб-камеры поддерживает три размера кадров: 320x180, 640x360 и 1280x720. Он
поддерживает настройки кадров в секунду 10, 15, 25, 30, 50 и 60 fps. Какие из них
доступны, зависит от выбранного размера кадра: чем больше размер кадра, тем
ниже максимальное число кадров в секунду.

Изначально выбранным цветовым пространством при переключении на вход веб-камеры будет
sRGB.


Входы TV и S-Video
~~~~~~~~~~~~~~~~~~

Единственное различие между входом TV и S-Video в том, что у TV есть
тюнер. В остальном они ведут себя идентично.

Эти входы поддерживают также аудиовходы: один TV и один Line-In. Они
оба поддерживают все TV-стандарты. Если запросить стандарт, то Vivid-регуляторы
'Standard Signal Mode' и 'Standard' определяют, каким будет
результат.

Эти входы поддерживают все комбинации настройки field. Была проявлена особая
забота о точном воспроизведении того, как обрабатываются поля для различных
TV-стандартов. Это особенно заметно при генерации горизонтально
движущегося изображения, так что временной эффект использования чересстрочных форматов становится отчётливо
видимым. Для стандартов 50 Гц верхнее поле — самое старое, а нижнее поле —
самое новое по времени. Для стандартов 60 Гц всё наоборот: нижнее поле —
самое старое, а верхнее поле — самое новое по времени.

Когда вы начинаете захват в режиме V4L2_FIELD_ALTERNATE, первый буфер будет
содержать верхнее поле для стандартов 50 Гц и нижнее поле для стандартов 60 Гц.
Это то, что делает и оборудование захвата.

Наконец, для стандартов PAL/SECAM первая половина верхней строки содержит шум.
Это симулирует Wide Screen Signal, который обычно размещается там.

Изначально выбранным цветовым пространством при переключении на вход TV или S-Video
будет SMPTE-170M.

Соотношение сторон пикселя будет зависеть от TV-стандарта. Соотношение сторон видео
можно выбрать через Vivid-регулятор 'Standard Aspect Ratio'.
Варианты — '4x3', '16x9', что даст широкоэкранное видео с леттербоксингом, и
'16x9 Anamorphic', что даст полноэкранное сжатое анаморфное широкоэкранное
видео, которое нужно будет соответственно масштабировать.

TV-'тюнер' поддерживает диапазон частот 44-958 МГц. Каналы доступны
каждые 6 МГц, начиная с 49.25 МГц. Для каждого канала генерируемое изображение
будет цветным в пределах +/- 0.25 МГц вокруг него и в оттенках серого в пределах
+/- 1 МГц вокруг канала. За этими пределами — просто шум. ioctl VIDIOC_G_TUNER
вернёт 100% силу сигнала для +/- 0.25 МГц и 50% для +/- 1 МГц.
Он также вернёт корректные значения afc, чтобы показать, слишком ли низкая или
слишком высокая частота.

Возвращаемые аудиоподканалы — MONO для диапазона +/- 1 МГц вокруг
действительной частоты канала. Когда частота находится в пределах +/- 0.25 МГц от
канала, он вернёт либо MONO, STEREO, либо MONO | SAP (для NTSC) или
LANG1 | LANG2 (для остальных), либо STEREO | SAP.

Какое из них возвращается, зависит от выбранного канала, каждый следующий действительный канал
будет циклически перебирать возможные комбинации аудиоподканалов. Это позволяет
вам тестировать различные комбинации, просто переключая каналы..

Наконец, для этих входов структура v4l2_timecode заполняется в
извлекаемой из очереди структуре v4l2_buffer.


Вход HDMI
~~~~~~~~~

Входы HDMI поддерживают все тайминги CEA-861 и DMT, как прогрессивные, так и
чересстрочные, для частот пиксельного тактирования между 25 и 600 МГц. Режим
field для чересстрочных форматов всегда V4L2_FIELD_ALTERNATE. Для HDMI
порядок полей всегда сначала верхнее поле, и когда вы начинаете захват
чересстрочного формата, вы получите сначала верхнее поле.

Изначально выбранное цветовое пространство при переключении на вход HDMI или
выборе тайминга HDMI основано на разрешении формата: для разрешений
меньше или равных 720x576 цветовое пространство устанавливается в SMPTE-170M, для
остальных оно устанавливается в REC-709 (тайминги CEA-861) или sRGB (тайминги VESA DMT).

Соотношение сторон пикселя будет зависеть от тайминга HDMI: для 720x480 оно
устанавливается как для TV-стандарта NTSC, для 720x576 — как для TV-стандарта
PAL, а для всех остальных возвращается соотношение сторон пикселя 1:1.

Соотношение сторон видео можно выбрать через Vivid-регулятор 'DV Timings Aspect Ratio'.
Варианты — 'Source Width x Height' (просто использовать
то же соотношение, что и у выбранного формата), '4x3' или '16x9', любой из которых может
дать видео с пилларбоксингом или леттербоксингом.

Для входов HDMI можно установить EDID. По умолчанию предоставляется простой EDID.
Вы можете устанавливать EDID только для входов HDMI. Внутри, однако,
EDID является общим для всех входов HDMI.

Никакой интерпретации данных EDID не производится, за исключением
физического адреса. См. раздел CEC для более подробной информации.

Имеется максимум 15 входов HDMI (если их больше, то они будут
сокращены до 15), поскольку это ограничение физического адреса EDID.


Вывод видео
-----------

Устройство вывода видео можно настроить с помощью опций модуля
num_outputs, output_types и ccs_out_mode (см. "Настройка драйвера"
для более подробной информации), но по умолчанию настраиваются два выхода:
вход S-Video и вход HDMI, по одному выходу для каждого типа выхода. Они
описаны подробнее ниже.

Как и при захвате видео, частота кадров также точна в долгосрочной перспективе.


Вывод S-Video
~~~~~~~~~~~~~

Этот выход поддерживает также аудиовыходы: "Line-Out 1" и "Line-Out 2".
Выход S-Video поддерживает все TV-стандарты.

Этот выход поддерживает все комбинации настройки field.

Изначально выбранным цветовым пространством при переключении на вход TV или S-Video
будет SMPTE-170M.


Вывод HDMI
~~~~~~~~~~

Выход HDMI поддерживает все тайминги CEA-861 и DMT, как прогрессивные, так и
чересстрочные, для частот пиксельного тактирования между 25 и 600 МГц. Режим
field для чересстрочных форматов всегда V4L2_FIELD_ALTERNATE.

Изначально выбранное цветовое пространство при переключении на выход HDMI или
выборе тайминга HDMI основано на разрешении формата: для разрешений
меньше или равных 720x576 цветовое пространство устанавливается в SMPTE-170M, для
остальных оно устанавливается в REC-709 (тайминги CEA-861) или sRGB (тайминги VESA DMT).

Соотношение сторон пикселя будет зависеть от тайминга HDMI: для 720x480 оно
устанавливается как для TV-стандарта NTSC, для 720x576 — как для TV-стандарта
PAL, а для всех остальных возвращается соотношение сторон пикселя 1:1.

Выход HDMI имеет действительный EDID, который можно получить через VIDIOC_G_EDID.

Имеется максимум 15 выходов HDMI (если их больше, то они будут
сокращены до 15), поскольку это ограничение физического адреса EDID. См.
также раздел CEC для более подробной информации.

Захват VBI
----------

Существует три типа устройств захвата VBI: те, что поддерживают только raw
(недекодированное) VBI, те, что поддерживают только sliced (декодированное) VBI, и те, что
поддерживают оба. Это определяется опцией модуля node_types. Во всех
случаях драйвер будет генерировать действительные данные VBI: для стандартов 60 Гц он будет
генерировать данные Closed Caption и XDS. Поток closed caption будет
чередоваться между "Hello world!" и "Closed captions test" каждую секунду.
Поток XDS будет выдавать текущее время раз в минуту. Для стандартов 50 Гц
он будет генерировать Wide Screen Signal, который основан на текущей настройке
регулятора Video Aspect Ratio, и страницы телетекста 100-159, по одной странице на кадр.

Устройство VBI будет работать только для входов S-Video и TV, для входа
веб-камеры или HDMI оно вернёт ошибку.


Вывод VBI
---------

Существует три типа устройств вывода VBI: те, что поддерживают только raw
(недекодированное) VBI, те, что поддерживают только sliced (декодированное) VBI, и те, что
поддерживают оба. Это определяется опцией модуля node_types.

Вывод sliced VBI поддерживает Wide Screen Signal и сигнал телетекста
для стандартов 50 Гц и Closed Captioning + XDS для стандартов 60 Гц.

Устройство VBI будет работать только для выхода S-Video, для выхода HDMI оно
вернёт ошибку.


Радиоприёмник
-------------

Радиоприёмник эмулирует приёмник FM/AM/SW. Диапазон FM также поддерживает RDS.
Диапазоны частот:

	- FM: 64 МГц - 108 МГц
	- AM: 520 кГц - 1710 кГц
	- SW: 2300 кГц - 26.1 МГц

Действительные каналы эмулируются каждый 1 МГц для FM и каждые 100 кГц для AM и SW.
Сила сигнала уменьшается тем сильнее, чем дальше частота от действительной
частоты, пока она не станет 0% на +/- 50 кГц (FM) или 5 кГц (AM/SW) от
идеальной частоты. Начальная частота при загрузке драйвера устанавливается в
95 МГц.

FM-приёмник также поддерживает RDS, как в режиме 'Block I/O', так и в режиме 'Controls'.
В режиме 'Controls' информация RDS хранится в регуляторах только для чтения. Эти
регуляторы обновляются каждый раз при изменении частоты,
либо при запросе статуса тюнера. Метод Block I/O использует интерфейс read()
для передачи блоков RDS приложению для декодирования.

Сигнал RDS 'обнаруживается' в пределах +/- 12.5 кГц вокруг частоты канала,
и чем дальше частота от действительной частоты, тем больше ошибок RDS
случайным образом вносится в поток block I/O, вплоть до 50% всех
блоков, если вы находитесь в пределах +/- 12.5 кГц от частоты канала. Все четыре ошибки
могут возникать в равных пропорциях: блоки, помеченные 'CORRECTED', блоки, помеченные
'ERROR', блоки, помеченные 'INVALID', и потерянные блоки.

Генерируемый поток RDS содержит все стандартные поля, содержащиеся в
группе 0B, а также радиотекст и текущее время.

Приёмник поддерживает аппаратный поиск частоты (HW frequency seek), либо в режиме Bounded, либо в режиме Wrap Around,
либо в обоих, что настраивается регулятором "Radio HW Seek Mode".


Радиопередатчик
---------------

Радиопередатчик эмулирует передатчик FM/AM/SW. Диапазон FM также поддерживает RDS.
Диапазоны частот:

	- FM: 64 МГц - 108 МГц
	- AM: 520 кГц - 1710 кГц
	- SW: 2300 кГц - 26.1 МГц

Начальная частота при загрузке драйвера — 95.5 МГц.

FM-передатчик также поддерживает RDS, как в режиме 'Block I/O', так и в режиме 'Controls'.
В режиме 'Controls' передаваемая информация RDS настраивается
с помощью регуляторов, а в режиме 'Block I/O' блоки передаются драйверу
с помощью write().


Приёмник программно-определяемого радио
---------------------------------------

SDR-приёмник имеет три частотных диапазона для ADC-тюнера:

	- 300 кГц
	- 900 кГц - 2800 кГц
	- 3200 кГц

RF-тюнер поддерживает 50 МГц - 2000 МГц.

Генерируемые данные содержат синфазную (In-phase) и квадратурную (Quadrature) составляющие
тона 1 кГц, имеющего амплитуду sqrt(2).

Регуляторы
----------

Различные устройства поддерживают различные регуляторы. Разделы ниже описывают
каждый регулятор и то, какие устройства его поддерживают.


User Controls — Test Controls
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Button, Boolean, Integer 32 Bits, Integer 64 Bits, Menu, String, Bitmask и
Integer Menu — это регуляторы, представляющие все возможные типы регуляторов. Регулятор Menu
и регулятор Integer Menu оба имеют 'дыры' в своих списках меню,
означающие, что один или несколько пунктов меню возвращают EINVAL при вызове VIDIOC_QUERYMENU.
Оба регулятора меню также имеют ненулевое минимальное значение регулятора. Эти возможности
позволяют вам проверить, может ли ваше приложение корректно обрабатывать такие вещи.
Эти регуляторы поддерживаются для каждого типа устройства.


User Controls — Video Capture
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Следующие регуляторы специфичны для захвата видео.

Регуляторы Brightness, Contrast, Saturation и Hue действительно работают и являются
стандартными. У регулятора Brightness есть одна особая возможность: каждый
видеовход имеет своё собственное значение яркости, так что смена входа восстановит
яркость для этого входа. Кроме того, каждый видеовход использует другой
диапазон яркости (минимальное и максимальное значения регулятора). Переключение входов
вызовет отправку события регулятора с установленным флагом V4L2_EVENT_CTRL_CH_RANGE.
Это позволяет вам тестировать регуляторы, которые могут менять свой диапазон.

Регуляторы 'Gain, Automatic' и Gain можно использовать для тестирования волатильных регуляторов:
если установлен 'Gain, Automatic', то регулятор Gain является волатильным и постоянно
меняется. Если 'Gain, Automatic' сброшен, то регулятор Gain является обычным
регулятором.

Регуляторы 'Horizontal Flip' и 'Vertical Flip' можно использовать для отражения
изображения. Они комбинируются с Vivid-регуляторами 'Sensor Flipped Horizontally/Vertically'.

Регулятор 'Alpha Component' можно использовать для установки альфа-компонента для
форматов, содержащих альфа-канал.


User Controls — Audio
~~~~~~~~~~~~~~~~~~~~~~

Следующие регуляторы специфичны для захвата и вывода видео и радиоприёмников
и радиопередатчиков.

Аудиорегуляторы 'Volume' и 'Mute' типичны для таких устройств для
управления громкостью и отключения звука. На самом деле в драйвере vivid они
ничего не делают.


Vivid-регуляторы
~~~~~~~~~~~~~~~~

Эти пользовательские регуляторы vivid управляют генерацией изображения, инъекцией ошибок и т.д.


Регуляторы тестового паттерна
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Регуляторы тестового паттерна все специфичны для захвата видео.

- Test Pattern:

	выбирает, какой тестовый паттерн использовать. Используйте CSC Colorbar для
	тестирования преобразований цветовых пространств: цвета, используемые в этом тестовом паттерне,
	отображаются в действительные цвета во всех цветовых пространствах. Преобразование цветового пространства
	отключено для остальных тестовых паттернов.

- OSD Text Mode:

	выбирает, должен ли отображаться текст, наложенный на
	тестовый паттерн, и если да, то должны ли отображаться только счётчики
	или полный текст.

- Horizontal Movement:

	выбирает, должен ли тестовый паттерн
	двигаться влево или вправо и с какой скоростью.

- Vertical Movement:

	делает то же самое для вертикального направления.

- Show Border:

	показать двухпиксельную рамку по краю фактического изображения,
	исключая леттербоксинг или пилларбоксинг.

- Show Square:

	показать квадрат в середине изображения. Если изображение
	отображается с правильными коррекциями соотношения сторон пикселя и изображения,
	то ширина и высота квадрата на мониторе должны быть
	одинаковыми.

- Insert SAV Code in Image:

	добавляет код SAV (Start of Active Video) в изображение.
	Это можно использовать для проверки того, не интерпретируются ли такие коды в изображении
	непреднамеренно вместо того, чтобы игнорироваться.

- Insert EAV Code in Image:

	делает то же самое для кода EAV (End of Active Video).

- Insert Video Guard Band

	добавляет 4 столбца пикселей с кодом HDMI Video Guard Band с
	левой стороны изображения. Это работает только с 3- или 4-байтовыми форматами RGB-пикселей.
	Значение RGB-пикселя 0xab/0x55/0xab оказывается эквивалентным
	коду HDMI Video Guard Band, который предшествует каждой активной видеостроке
	(см. раздел 5.2.2.1 в спецификации HDMI 1.3). Чтобы проверить, имеет ли видео-
	приёмник корректную обработку HDMI Video Guard Band, включите этот
	регулятор и затем переместите изображение к левой стороне экрана.
	Это приведёт к видеострокам, которые начинаются с нескольких пикселей,
	имеющих то же значение, что и Video Guard Band, который им предшествует.
	Приёмники, которые просто продолжат пропускать значения Video Guard Band,
	теперь дадут сбой и либо потеряют синхронизацию, либо эти видеостроки сдвинутся.


Регуляторы выбора возможностей захвата
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Эти регуляторы все специфичны для захвата видео.

- Sensor Flipped Horizontally:

	изображение отражается горизонтально, и устанавливается
	флаг статуса входа V4L2_IN_ST_HFLIP. Это эмулирует случай, когда
	сенсор, например, установлен вверх ногами.

- Sensor Flipped Vertically:

	изображение отражается вертикально, и устанавливается
	флаг статуса входа V4L2_IN_ST_VFLIP. Это эмулирует случай, когда
	сенсор, например, установлен вверх ногами.

- Standard Aspect Ratio:

	выбирает, должно ли соотношение сторон изображения, используемое для входа TV или
	S-Video, быть 4x3, 16x9 или анаморфным широкоэкранным. Это может
	привести к леттербоксингу.

- DV Timings Aspect Ratio:

	выбирает, должно ли соотношение сторон изображения, используемое для входа HDMI,
	быть таким же, как соотношение ширины и высоты источника, или
	оно должно быть 4x3 или 16x9. Это может привести к леттербоксингу или пилларбоксингу.

- Timestamp Source:

	выбирает, когда берётся временная метка для каждого буфера.

- Colorspace:

	выбирает, какое цветовое пространство должно использоваться при генерации изображения.
	Это применяется только если выбран тестовый паттерн CSC Colorbar,
	в противном случае тестовый паттерн пройдёт без преобразования.
	Это поведение также является тем, чего вы хотите, поскольку 75% Colorbar
	действительно должен иметь 75% интенсивности сигнала и не должен затрагиваться
	преобразованиями цветовых пространств.

	Изменение цветового пространства приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует обнаруженное изменение цветового пространства.

- Transfer Function:

	выбирает, какая передаточная функция цветового пространства должна использоваться при
	генерации изображения. Это применяется только если выбран тестовый паттерн CSC Colorbar,
	в противном случае тестовый паттерн пройдёт без преобразования.
	Это поведение также является тем, чего вы хотите, поскольку 75% Colorbar
	действительно должен иметь 75% интенсивности сигнала и не должен затрагиваться
	преобразованиями цветовых пространств.

	Изменение передаточной функции приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует обнаруженное изменение цветового пространства.

- Y'CbCr Encoding:

	выбирает, какое кодирование Y'CbCr должно использоваться при генерации
	изображения Y'CbCr. Это применяется только если формат установлен в формат Y'CbCr,
	в отличие от формата RGB.

	Изменение кодирования Y'CbCr приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует обнаруженное изменение цветового пространства.

- Quantization:

	выбирает, какое квантование должно использоваться для кодирования RGB или Y'CbCr
	при генерации тестового паттерна.

	Изменение квантования приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует обнаруженное изменение цветового пространства.

- Limited RGB Range (16-235):

	выбирает, должен ли диапазон RGB источника HDMI быть
	ограниченным или полным. Это комбинируется с регулятором Digital Video 'Rx RGB
	Quantization Range' и может использоваться для тестирования того, что произойдёт, если
	источник предоставит вам неверную информацию о диапазоне квантования.
	См. описание этого регулятора для более подробной информации.

- Apply Alpha To Red Only:

	применить альфа-канал, установленный пользовательским регулятором 'Alpha Component',
	только к красному цвету тестового паттерна.

- Enable Capture Cropping:

	включает поддержку обрезки. Этот регулятор присутствует только если
	опция модуля ccs_cap_mode установлена в значение по умолчанию -1 и если
	опция модуля no_error_inj установлена в 0 (по умолчанию).

- Enable Capture Composing:

	включает поддержку композиции. Этот регулятор присутствует
	только если опция модуля ccs_cap_mode установлена в значение по умолчанию
	-1 и если опция модуля no_error_inj установлена в 0 (по умолчанию).

- Enable Capture Scaler:

	включает поддержку масштабатора (максимум четырёхкратное увеличение
	и уменьшение). Этот регулятор присутствует только если опция модуля ccs_cap_mode
	установлена в значение по умолчанию -1 и если опция модуля no_error_inj
	установлена в 0 (по умолчанию).

- Maximum EDID Blocks:

	определяет, сколько блоков EDID поддерживает драйвер.
	Обратите внимание, что драйвер vivid на самом деле не интерпретирует новые данные
	EDID, он просто хранит их. Он допускает до 256 блоков EDID,
	что является максимумом, поддерживаемым стандартом.

- Fill Percentage of Frame:

	можно использовать для отрисовки только верхних X процентов
	изображения. Поскольку каждый кадр должен отрисовываться драйвером, это
	сильно нагружает CPU. Для больших разрешений это становится
	проблематичным. Отрисовывая только часть изображения, эту нагрузку на CPU можно
	уменьшить.

Регуляторы инъекции ошибок

Следующие два регулятора действительны только для захвата видео и vbi.

- Standard Signal Mode:

	выбирает поведение VIDIOC_QUERYSTD: что он должен возвращать?

	Изменение этого регулятора приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,

- Standard:

	выбирает стандарт, который VIDIOC_QUERYSTD должен возвращать, если
	предыдущий регулятор установлен в "Selected Standard".

	Изменение этого регулятора приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует изменённый стандарт входа.


Следующие два регулятора действительны только для захвата видео.

- DV Timings Signal Mode:

	выбирает поведение VIDIOC_QUERY_DV_TIMINGS: что
	он должен возвращать?

	Изменение этого регулятора приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует изменённое условие входа (например, кабель
	был подключён или отключён).

- DV Timings:

	выбирает тайминги, которые VIDIOC_QUERY_DV_TIMINGS должен возвращать,
	если предыдущий регулятор установлен в "Selected DV Timings".

	Изменение этого регулятора приведёт к отправке V4L2_EVENT_SOURCE_CHANGE,
	поскольку оно эмулирует изменённые тайминги входа.


Следующие регуляторы присутствуют только если опция модуля no_error_inj
установлена в 0 (по умолчанию). Эти регуляторы действительны для потоков захвата
и вывода видео и vbi и для устройства захвата SDR, за исключением
регулятора Disconnect, который действителен для всех устройств.

- Wrap Sequence Number:

	проверить, что происходит при переполнении (wrap) номера последовательности в
	структуре v4l2_buffer.

- Wrap Timestamp:

	проверить, что происходит при переполнении (wrap) временной метки в структуре
	v4l2_buffer.

- Percentage of Dropped Buffers:

	устанавливает процент буферов, которые
	никогда не возвращаются драйвером (то есть, они теряются).

- Disconnect:

	эмулирует отключение USB. Устройство будет вести себя так, как если бы оно было
	отключено. Только после того, как будут закрыты все открытые дескрипторы файлов узла
	устройства, устройство снова станет 'подключённым'.

- Inject V4L2_BUF_FLAG_ERROR:

	при нажатии следующий кадр, возвращаемый
	драйвером, будет иметь установленный флаг ошибки (то есть кадр помечается
	как повреждённый).

- Inject VIDIOC_REQBUFS Error:

	при нажатии следующий вызов ioctl REQBUFS или CREATE_BUFS
	завершится с ошибкой. Точнее: операция videobuf2
	queue_setup() вернёт -EINVAL.

- Inject VIDIOC_QBUF Error:

	при нажатии следующий вызов ioctl VIDIOC_QBUF или
	VIDIOC_PREPARE_BUFFER завершится с ошибкой. Точнее:
	операция videobuf2 buf_prepare() вернёт -EINVAL.

- Inject VIDIOC_STREAMON Error:

	при нажатии следующий вызов ioctl VIDIOC_STREAMON
	завершится с ошибкой. Точнее: операция videobuf2
	start_streaming() вернёт -EINVAL.

- Inject Fatal Streaming Error:

	при нажатии ядро потоковой передачи будет
	помечено как пострадавшее от фатальной ошибки, единственный способ восстановиться
	после этого — остановить потоковую передачу. Точнее: вызывается функция videobuf2
	vb2_queue_error().

Зацикливание (Looping) видео, sliced VBI и HDMI CEC
---------------------------------------------------

Функциональность зацикливания видео поддерживается для устройств, созданных одним и тем же
экземпляром драйвера vivid, а также между несколькими экземплярами драйвера vivid.
Драйвер vivid поддерживает зацикливание данных видео и sliced VBI между выходом S-Video
и входом S-Video. Он также поддерживает зацикливание данных видео и HDMI CEC между
выходом HDMI и входом HDMI.

Чтобы включить зацикливание, установите регулятор(ы) 'HDMI/S-Video XXX-N Is Connected To', чтобы выбрать,
использует ли вход генератор тестового паттерна, или отключён, или подключён
к выходу. Вход может быть подключён к выходу любого экземпляра vivid.
Входы и выходы нумеруются XXX-N, где XXX — это номер экземпляра vivid
(см. опцию модуля n_devs). Если есть только один экземпляр vivid (по умолчанию), то
XXX будет 000. А N — это N-й вход или выход S-Video/HDMI этого экземпляра.
Если vivid загружен без опций модуля, то вы можете подключить вход S-Video 000-0
к выходу S-Video 000-0 или вход HDMI 000-0 к выходу HDMI 000-0.
Это эквивалентно подключению или отключению кабеля между входом и
выходом в физическом устройстве.

Если регулятор 'HDMI/S-Video XXX-N Is Connected To' выбрал выход, то видеовыход
будет зациклен на видеовход при условии, что:

- текущий выбранный вход совпадает со входом, указанным именем регулятора.

- в экземпляре vivid выходного коннектора текущий выбранный выход совпадает
  с выходом, указанным значением регулятора.

- разрешение видео видеовхода должно совпадать с разрешением видеовыхода.
  Так что невозможно зациклить выход S-Video 50 Гц (720x576) на вход S-Video
  60 Гц (720x480), либо выход HDMI 720p60 на вход 1080p30.

- форматы пикселей должны быть идентичны с обеих сторон. Иначе драйверу пришлось бы
  выполнять также преобразование форматов пикселей, а это уже заходит слишком далеко.

- настройки field должны быть идентичны с обеих сторон. По той же причине, что и выше:
  требование к драйверу преобразовывать из одного формата поля в другой слишком
  усложнило бы дело. Это также запрещает захват с 'Field Top' или 'Field
  Bottom', когда выходное видео установлено в 'Field Alternate'. Эта комбинация,
  хотя и легальна, стала слишком сложной для поддержки. Обе стороны должны быть 'Field
  Alternate', чтобы это работало. Также обратите внимание, что для этого конкретного случая
  подсчёт последовательности и полей в структуре v4l2_buffer на стороне захвата может быть
  не на 100% точным.

- настройки field V4L2_FIELD_SEQ_TB/BT не поддерживаются. Хотя это возможно
  реализовать, это означало бы большую работу, чтобы сделать это правильно. Поскольку эти
  значения поля используются редко, было принято решение пока не реализовывать это.

- на стороне входа "Standard Signal Mode" для входа S-Video или
  "DV Timings Signal Mode" для входа HDMI должен быть настроен так, чтобы на видеовход
  передавался действительный сигнал.

Если какое-либо условие недействительно, то показывается тестовый паттерн 'Noise'.

Частоты кадров не обязаны совпадать, хотя это может измениться в будущем.

По умолчанию вы увидите текст OSD, наложенный поверх зацикленного видео.
Это можно отключить, изменив регулятор "OSD Text Mode" устройства захвата
видео.

Чтобы зацикливание VBI работало, всё вышеперечисленное должно быть действительным, и в дополнение вывод
vbi должен быть настроен на sliced VBI. Сторона захвата VBI может быть настроена
либо на raw, либо на sliced VBI. Обратите внимание, что в настоящий момент зацикливаются только данные VBI CC/XDS (форматы 60 Гц)
и WSS (форматы 50 Гц). Данные VBI телетекста не зацикливаются.


Зацикливание (Looping) радио и RDS
----------------------------------

Драйвер vivid поддерживает зацикливание вывода RDS на вход RDS.

Поскольку радио беспроводное, это зацикливание всегда происходит, если частота
радиоприёмника близка к частоте радиопередатчика. В этом случае радио-
передатчик 'переопределит' эмулируемые радиостанции.

Зацикливание RDS в настоящее время поддерживается только между устройствами, созданными одним и тем же
экземпляром драйвера vivid.

Как упомянуто в разделе "Радиоприёмник", радиоприёмник эмулирует
станции через регулярные частотные интервалы. В зависимости от частоты
радиоприёмника вычисляется значение силы сигнала (это возвращается
VIDIOC_G_TUNER). Однако он также будет смотреть на частоту, установленную радио-
передатчиком, и если это приводит к более высокой силе сигнала, то будут использованы настройки
радиопередатчика, как если бы это была действительная станция. Это также
включает данные RDS (если есть), которые передатчик 'передаёт'. Это
точно принимается на стороне приёмника. Обратите внимание, что при загрузке драйвера
частоты радиоприёмника и радиопередатчика не идентичны, так что
изначально зацикливание не происходит.


Обрезка, композиция, масштабирование
------------------------------------

Этот драйвер поддерживает обрезку, композицию и масштабирование в любой комбинации. Обычно
поддерживаемые возможности можно выбрать через Vivid-регуляторы,
но также возможно жёстко задать это при загрузке модуля через
опции модуля ccs_cap_mode и ccs_out_mode. См. "Настройка драйвера" о
деталях этих опций модуля.

Это позволяет вам тестировать ваше приложение для всех этих вариаций.

Обратите внимание, что вход веб-камеры никогда не поддерживает обрезку, композицию или масштабирование. Это
применяется только к входам и выходам TV/S-Video/HDMI. Причина в том, что
веб-камеры, включая эту виртуальную реализацию, обычно используют
VIDIOC_ENUM_FRAMESIZES для перечисления набора дискретных размеров кадров, которые они поддерживают.
А это не сочетается с обрезкой, композицией или масштабированием. Это
прежде всего ограничение API V4L2, которое здесь тщательно воспроизводится.

Минимальное и максимальное разрешения, которых может достичь масштабатор, — это 16x16 и
(4096 * 4) x (2160 x 4), но он может масштабировать только в большую или меньшую сторону с коэффициентом 4 или
менее. Так что для исходного разрешения 1280x720 минимум, который может масштабатор, — это
320x180, а максимум — 5120x2880. Вы можете поэкспериментировать с этим, используя
тестовый инструмент qv4l2, и вы увидите эти зависимости.

Этот драйвер также поддерживает большие настройки 'bytesperline', нечто, что
VIDIOC_S_FMT допускает, но что немногие драйверы реализуют.

Масштабатор — это простой масштабатор, использующий алгоритм Coarse Bresenham. Он
разработан для скорости и простоты, а не для качества.

Если комбинация обрезки, композиции и масштабирования это позволяет, то возможно
менять прямоугольники обрезки и композиции на лету.


Форматы
-------

Драйвер поддерживает все обычные packed- и planar-форматы YUYV 4:4:4, 4:2:2 и 4:2:0,
packed-форматы RGB 8, 16, 24 и 32 и различные многоплоскостные
форматы.

Альфа-компонент можно установить через пользовательский регулятор 'Alpha Component'
для тех форматов, которые его поддерживают. Если установлен регулятор 'Apply Alpha To Red Only',
то альфа-компонент используется только для красного цвета и в остальных случаях устанавливается в
0.

Драйвер должен быть настроен для поддержки многоплоскостных форматов. По умолчанию
экземпляры драйвера являются одноплоскостными. Это можно изменить, установив
опцию модуля multiplanar, см. "Настройка драйвера" для более подробной информации об этой
опции.

Если экземпляр драйвера использует многоплоскостные форматы/API, то первый
одноплоскостной формат (YUYV) и многоплоскостные форматы NV16M и NV61M
будут иметь плоскость, у которой ненулевой data_offset в 128 байт. Редко
когда data_offset является ненулевым, так что это полезная возможность для тестирования приложений.

Вывод видео также будет учитывать любой data_offset, установленный приложением.


Оверлей вывода
--------------

Примечание: оверлеи вывода реализованы прежде всего для того, чтобы протестировать существующий
API оверлея вывода V4L2. Стоит ли использовать этот API для новых драйверов —
вопрос спорный.

Этот драйвер имеет поддержку оверлея вывода и способен на:

	- битмап-клиппинг (bitmap clipping),
	- клиппинг по списку (list clipping) (до 16 прямоугольников)
	- chromakey
	- source chromakey
	- глобальная альфа (global alpha)
	- локальная альфа (local alpha)
	- локальная инверсная альфа (local inverse alpha)

Оверлеи вывода не поддерживаются для многоплоскостных форматов. Кроме того,
формат пикселя формата захвата и формат пикселя фреймбуфера должны быть
одинаковыми, чтобы оверлей работал. Иначе VIDIOC_OVERLAY вернёт ошибку.

Оверлеи вывода работают только если драйвер был настроен на создание
фреймбуфера установкой флага 0x10000 в опции модуля node_types.
Созданный фреймбуфер имеет размер 720x576 и поддерживает ARGB 1:5:5:5 и
RGB 5:6:5.

Чтобы увидеть эффекты различных возможностей клиппинга, chromakey или
обработки альфа, вам нужно включить зацикливание видео и увидеть результаты
на стороне захвата. Использование возможностей клиппинга, chromakey или обработки альфа
значительно замедлит видеоцикл, так как нужно выполнять много проверок
для каждого пикселя.


CEC (Consumer Electronics Control)
----------------------------------

Если есть входы HDMI, то будет создан CEC-адаптер, имеющий
то же количество входных портов. Это эквивалентно, например, телевизору, который
имеет это количество входов. Каждый выход HDMI также создаст
CEC-адаптер, который подключён к соответствующему входному порту, или (если
выходов больше, чем входов) не подключён вообще. Другими словами,
это эквивалентно подключению каждого устройства вывода к входному порту
телевизора. Любые оставшиеся устройства вывода остаются неподключёнными.

EDID, который читает каждый выход, сообщает уникальный физический адрес CEC, который
основан на физическом адресе EDID входа. Так что если EDID
приёмника имеет физический адрес A.B.0.0, то каждый выход будет видеть EDID,
содержащий физический адрес A.B.C.0, где C — это от 1 до количества входов. Если
выходов больше, чем входов, то оставшиеся выходы имеют CEC-адаптер,
который отключён и сообщает недействительный физический адрес.
