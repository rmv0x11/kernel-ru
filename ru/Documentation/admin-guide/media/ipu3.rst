Драйвер блока обработки изображений Intel Image Processing Unit 3 (IPU3) (ImgU)
==================================================================================

Copyright |copy| 2018 Intel Corporation

Введение
========

Этот файл документирует драйверы блока обработки изображений Intel IPU3 (Image
Processing Unit третьего поколения), расположенные в drivers/media/pci/intel/ipu3
(CIO2), а также в drivers/staging/media/ipu3 (ImgU).

Блок Intel IPU3, присутствующий в некоторых платформах Kaby Lake (а также в
некоторых платформах Sky Lake) (линейки процессоров U/Y), состоит из двух частей,
а именно блока обработки изображений (ImgU) и устройства CIO2 (приёмник MIPI CSI2).

Устройство CIO2 принимает необработанные данные Байера от сенсоров и выдаёт кадры
в формате, специфичном для IPU3 (для последующей обработки блоком IPU3 ImgU).
Драйвер CIO2 доступен как drivers/media/pci/intel/ipu3/ipu3-cio2* и включается
через параметр конфигурации CONFIG_VIDEO_IPU3_CIO2.

Блок обработки изображений (ImgU) отвечает за обработку изображений, захваченных
устройством IPU3 CIO2. Исходные тексты драйвера ImgU можно найти в каталоге
drivers/staging/media/ipu3. Драйвер включается через параметр конфигурации
CONFIG_VIDEO_IPU3_IMGU.

Эти два модуля драйверов называются ipu3_csi2 и ipu3_imgu соответственно.

Драйверы были протестированы на платформах Kaby Lake (линейки процессоров U/Y).

Оба драйвера реализуют интерфейсы V4L2, Media Controller и V4L2 sub-device.
Драйвер IPU3 CIO2 поддерживает сенсоры камер, подключённые к интерфейсам CIO2
MIPI CSI-2, через драйверы сенсоров V4L2 sub-device.

CIO2
====

CIO2 представлен как единый V4L2 subdev, который предоставляет интерфейс V4L2
subdev пространству пользователя. Для каждого приёмника CSI-2 имеется
видео-узел, при этом для всего устройства существует единый интерфейс media
controller.

CIO2 содержит четыре независимых канала захвата, каждый со своим приёмником
MIPI CSI-2 и движком DMA. Каждый канал моделируется как V4L2 sub-device,
представленный пространству пользователя в виде узла V4L2 sub-device, и имеет два
pad'а:

.. tabularcolumns:: |p{0.8cm}|p{4.0cm}|p{4.0cm}|

.. flat-table::
    :header-rows: 1

    * - Pad
      - Направление
      - Назначение

    * - 0
      - sink
      - Вход MIPI CSI-2, подключённый к subdev сенсора

    * - 1
      - source
      - Захват необработанного видео, подключённый к видеоинтерфейсу V4L2

Видеоинтерфейсы V4L2 моделируют движки DMA. Они представлены пространству
пользователя в виде узлов видеоустройств V4L2.

Захват кадров в необработанном формате Байера
---------------------------------------------

Приёмник CIO2 MIPI CSI2 используется для захвата кадров (в упакованном
необработанном формате Байера) от необработанных сенсоров, подключённых к портам
CSI2. Захваченные кадры используются как вход для драйвера ImgU.

Обработка изображений с помощью IPU3 ImgU требует таких инструментов, как
raw2pnm [#f1]_ и yavta [#f2]_, вследствие следующих уникальных требований и/или
особенностей, специфичных для IPU3.

-- Приёмник IPU3 CSI2 выдаёт захваченные от сенсора кадры в упакованном
необработанном формате Байера, специфичном для IPU3.

-- Несколько видео-узлов должны работать одновременно.

Рассмотрим пример сенсора ov5670, подключённого к порту CSI2 0, для захвата
изображения 2592x1944.

С помощью API media controller сенсор ov5670 настраивается на отправку кадров в
упакованном необработанном формате Байера приёмнику IPU3 CSI2.

.. code-block:: none

    # This example assumes /dev/media0 as the CIO2 media device
    export MDEV=/dev/media0

    # and that ov5670 sensor is connected to i2c bus 10 with address 0x36
    export SDEV=$(media-ctl -d $MDEV -e "ov5670 10-0036")

    # Establish the link for the media devices using media-ctl
    media-ctl -d $MDEV -l "ov5670:0 -> ipu3-csi2 0:0[1]"

    # Set the format for the media devices
    media-ctl -d $MDEV -V "ov5670:0 [fmt:SGRBG10/2592x1944]"
    media-ctl -d $MDEV -V "ipu3-csi2 0:0 [fmt:SGRBG10/2592x1944]"
    media-ctl -d $MDEV -V "ipu3-csi2 0:1 [fmt:SGRBG10/2592x1944]"

После настройки медиаконвейера можно задать желаемые специфичные для сенсора
настройки (такие как настройки экспозиции и усиления) с помощью инструмента yavta.

например

.. code-block:: none

    yavta -w 0x009e0903 444 $SDEV
    yavta -w 0x009e0913 1024 $SDEV
    yavta -w 0x009e0911 2046 $SDEV

После задания желаемых настроек сенсора захват кадров можно выполнить, как показано
ниже.

например

.. code-block:: none

    yavta --data-prefix -u -c10 -n5 -I -s2592x1944 --file=/tmp/frame-#.bin \
          -f IPU3_SGRBG10 $(media-ctl -d $MDEV -e "ipu3-cio2 0")

С помощью приведённой выше команды захватывается 10 кадров с разрешением 2592x1944
в формате sGRBG10 и выдаётся в формате IPU3_SGRBG10.

Захваченные кадры доступны в виде файлов /tmp/frame-#.bin.

ImgU
====

ImgU представлен как два V4L2 subdev, каждый из которых предоставляет интерфейс
V4L2 subdev пространству пользователя.

Каждый V4L2 subdev представляет конвейер (pipe), способный поддерживать максимум 2
потока. Это помогает поддерживать продвинутые функции камеры, такие как Continuous
View Finder (CVF) и Snapshot During Video (SDV).

ImgU содержит два независимых конвейера, каждый из которых моделируется как V4L2
sub-device, представленный пространству пользователя в виде узла V4L2 sub-device.

Каждый конвейер имеет два pad'а sink и три pad'а source для следующих целей:

.. tabularcolumns:: |p{0.8cm}|p{4.0cm}|p{4.0cm}|

.. flat-table::
    :header-rows: 1

    * - Pad
      - Направление
      - Назначение

    * - 0
      - sink
      - Входной поток необработанного видео

    * - 1
      - sink
      - Параметры обработки

    * - 2
      - source
      - Выходной обработанный видеопоток

    * - 3
      - source
      - Выходной видеопоток видоискателя

    * - 4
      - source
      - Статистика 3A

Каждый pad подключён к соответствующему видеоинтерфейсу V4L2, представленному
пространству пользователя в виде узла видеоустройства V4L2.

Работа устройства
-----------------

В ImgU после того, как входному видео-узлу ("ipu3-imgu 0/1":0, в формате
<entity>:<pad-number>) поставлен в очередь буфер (в упакованном необработанном
формате Байера), ImgU начинает обработку буфера и выдаёт видеовыход в формате YUV
и выход статистики на соответствующие выходные узлы. Ожидается, что драйвер будет
иметь готовые буферы для всех узлов параметров, выхода и статистики в момент, когда
входному видео-узлу поставлен в очередь буфер.

Как минимум, все узлы — входной, основной выходной, статистики 3A и видоискателя —
должны быть включены, чтобы IPU3 начал обработку изображения.

Каждый V4L2 subdev ImgU имеет следующий набор видео-узлов.

Видео-узлы входа, выхода и видоискателя
---------------------------------------

Кадры (в упакованном необработанном формате Байера, специфичном для IPU3),
принятые входным видео-узлом, обрабатываются блоком обработки изображений IPU3 и
выдаются на 2 видео-узла, каждый из которых предназначен для своей цели (основной
выход и выход видоискателя).

Подробности о формате Байера, специфичном для IPU3, можно найти в
:ref:`v4l2-pix-fmt-ipu3-sbggr10`.

Драйвер поддерживает V4L2 Video Capture Interface, как определено в :ref:`devices`.

Поддерживается только многоплоскостной (multi-planar) API. Более подробную
информацию можно найти в :ref:`planar-apis`.

Видео-узел параметров
---------------------

Видео-узел параметров принимает параметры алгоритмов ImgU, которые используются
для настройки того, как алгоритмы ImgU обрабатывают изображение.

Подробности о параметрах обработки, специфичных для IPU3, можно найти в
:ref:`v4l2-meta-fmt-params`.

Видео-узел статистики 3A
------------------------

Видео-узел статистики 3A используется драйвером ImgU для выдачи статистики 3A
(автофокус, автоэкспозиция и автоматический баланс белого) для кадров,
обрабатываемых ImgU, приложениям пространства пользователя. Приложения
пространства пользователя могут использовать эти статистические данные для расчёта
желаемых параметров алгоритмов для ImgU.

Настройка Intel IPU3
====================

Конвейеры IPU3 ImgU можно настраивать с помощью Media Controller, определённого в
:ref:`media_controller`.

Режим работы и выбор двоичного файла прошивки
---------------------------------------------

ImgU работает на основе прошивки; в настоящее время прошивка ImgU поддерживает
запуск 2 конвейеров в режиме разделения времени с единым набором данных входного
кадра. Каждый конвейер может работать в определённом режиме — "VIDEO" или "STILL".
Режим "VIDEO" обычно используется для захвата видеокадров, а "STILL" используется
для захвата неподвижных кадров. Однако вы также можете выбрать "VIDEO" для захвата
неподвижных кадров, если хотите захватывать изображения с меньшей нагрузкой на
систему и меньшим энергопотреблением. Для режима "STILL" ImgU будет пытаться
использовать меньший коэффициент BDS и выдавать больший кадр Байера для дальнейшей
обработки YUV, чем в режиме "VIDEO", чтобы получить изображения высокого качества.
Кроме того, режиму "STILL" требуется XNR3 для подавления шума, поэтому режиму
"STILL" потребуется больше энергии и пропускной способности памяти, чем режиму
"VIDEO". TNR будет включён в режиме "VIDEO" и обойдён в режиме "STILL". По
умолчанию ImgU работает в режиме "VIDEO"; пользователь может использовать
управляющий элемент v4l2 V4L2_CID_INTEL_IPU3_MODE (в настоящее время определён в
drivers/staging/media/ipu3/include/uapi/intel-ipu3.h) для запроса и установки
режима работы. Для пользователя нет разницы в постановке буферов в очередь между
режимами "VIDEO" и "STILL"; обязательные входной и основной выходной узлы должны
быть включены и для них должны быть поставлены в очередь буферы, а очереди
статистики и видоискателя необязательны.

Двоичный файл прошивки будет выбран в соответствии с текущим режимом работы; такие
сообщения в журнале, как "using binary if_to_osys_striped " или "using binary
if_to_osys_primary_striped", можно наблюдать, если включить динамическую отладку
(dynamic debug) ImgU; двоичный файл if_to_osys_striped выбирается для "VIDEO", а
двоичный файл "if_to_osys_primary_striped" выбирается для "STILL".


Обработка изображения в необработанном формате Байера
-----------------------------------------------------

Настройка V4L2 subdev ImgU для обработки изображений
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

V4L2 subdev ImgU должны быть настроены с помощью API media controller, чтобы все
видео-узлы были корректно настроены.

Рассмотрим в качестве примера subdev "ipu3-imgu 0".

.. code-block:: none

    media-ctl -d $MDEV -r
    media-ctl -d $MDEV -l "ipu3-imgu 0 input":0 -> "ipu3-imgu 0":0[1]
    media-ctl -d $MDEV -l "ipu3-imgu 0":2 -> "ipu3-imgu 0 output":0[1]
    media-ctl -d $MDEV -l "ipu3-imgu 0":3 -> "ipu3-imgu 0 viewfinder":0[1]
    media-ctl -d $MDEV -l "ipu3-imgu 0":4 -> "ipu3-imgu 0 3a stat":0[1]

Также режим конвейера соответствующего V4L2 subdev должен быть установлен как
требуется (например, 0 для режима видео или 1 для режима неподвижного изображения)
через управляющий идентификатор 0x009819a1, как показано ниже.

.. code-block:: none

    yavta -w "0x009819A1 1" /dev/v4l-subdev7

Некоторые аппаратные блоки в конвейере ImgU могут изменять разрешение кадра путём
обрезки (cropping) или масштабирования (scaling); к этим аппаратным блокам
относятся Input Feeder (IF), Bayer Down Scaler (BDS) и Geometric Distortion
Correction (GDC).
Имеется также блок, который может изменять разрешение кадра, — YUV Scaler; он
применим только к вторичному выходу.

Кадры RAW Bayer проходят через эти аппаратные блоки конвейера ImgU, и итоговое
обработанное изображение выдаётся в память DDR.

.. kernel-figure::  ipu3_rcb.svg
   :alt: ipu3 resolution blocks image

   Аппаратные блоки изменения разрешения IPU3

**Input Feeder**

Input Feeder получает данные кадра Байера от сенсора; он может включать обрезку
строк и столбцов из кадра, а затем сохранять пиксели во внутренний пиксельный
буфер устройства, готовые к считыванию следующими блоками.

**Bayer Down Scaler**

Bayer Down Scaler способен выполнять масштабирование изображения в домене Байера;
коэффициент уменьшения может быть настроен от 1X до 1/4X по каждой оси с шагом
настройки 0.03125 (1/32).

**Geometric Distortion Correction**

Geometric Distortion Correction используется для выполнения коррекции искажений и
фильтрации изображения. Для работы ему нужны некоторые дополнительные
фильтрационные и обрамляющие (envelope padding) пиксели, поэтому входное разрешение
GDC должно быть больше выходного разрешения.

**YUV Scaler**

YUV Scaler подобен BDS, но в основном выполняет уменьшение изображения в домене
YUV; он может поддерживать уменьшение до 1/12X, но не может применяться к основному
выходу.

V4L2 subdev ImgU должен быть настроен с поддерживаемыми разрешениями во всех
вышеперечисленных аппаратных блоках для заданного входного разрешения.
Для заданного поддерживаемого разрешения входного кадра блоки Input Feeder, Bayer
Down Scaler и GDC должны быть настроены с поддерживаемыми разрешениями, поскольку
каждый аппаратный блок имеет собственное требование к выравниванию.

Вы должны настроить выходное разрешение аппаратных блоков разумно, чтобы
удовлетворить требованию оборудования при сохранении максимального поля зрения.
Промежуточные разрешения могут быть сгенерированы специальным инструментом —

https://github.com/intel/intel-ipu3-pipecfg

Этот инструмент можно использовать для генерации промежуточных разрешений. Больше
информации можно получить, ознакомившись со следующей таблицей конфигурации IPU3
ImgU.

https://chromium.googlesource.com/chromiumos/overlays/board-overlays/+/master

В каталоге baseboard-poppy/media-libs/cros-camera-hal-configs-poppy/files/gcss
файл graph_settings_ov5670.xml может быть использован в качестве примера.

Следующие шаги подготавливают конвейер ImgU к обработке изображения.

1. Формат данных V4L2 subdev ImgU должен быть установлен с помощью
VIDIOC_SUBDEV_S_FMT на pad'е 0, используя полученные выше ширину и высоту GDC.

2. Обрезка (cropping) V4L2 subdev ImgU должна быть установлена с помощью
VIDIOC_SUBDEV_S_SELECTION на pad'е 0, с V4L2_SEL_TGT_CROP в качестве цели,
используя высоту и ширину input feeder.

3. Композиция (composing) V4L2 subdev ImgU должна быть установлена с помощью
VIDIOC_SUBDEV_S_SELECTION на pad'е 0, с V4L2_SEL_TGT_COMPOSE в качестве цели,
используя высоту и ширину BDS.

Для примера с ov5670, для входного кадра с разрешением 2592x1944 (который подаётся
на pad 0 subdev ImgU), соответствующие разрешения для input feeder, BDS и GDC
равны 2592x1944, 2592x1944 и 2560x1920 соответственно.

После того как это сделано, принятые необработанные кадры Байера могут быть поданы
на V4L2 subdev ImgU, как показано ниже, с помощью приложения с открытым исходным
кодом v4l2n [#f1]_.

Для изображения, захваченного с разрешением 2592x1944 [#f4]_, с желаемым выходным
разрешением 2560x1920 и разрешением видоискателя 2560x1920, может быть использована
следующая команда v4l2n. Это помогает обработать необработанные кадры Байера и
выдаёт желаемые результаты для основного выходного изображения и выхода
видоискателя в формате NV12.

.. code-block:: none

    v4l2n --pipe=4 --load=/tmp/frame-#.bin --open=/dev/video4
          --fmt=type:VIDEO_OUTPUT_MPLANE,width=2592,height=1944,pixelformat=0X47337069 \
          --reqbufs=type:VIDEO_OUTPUT_MPLANE,count:1 --pipe=1 \
          --output=/tmp/frames.out --open=/dev/video5 \
          --fmt=type:VIDEO_CAPTURE_MPLANE,width=2560,height=1920,pixelformat=NV12 \
          --reqbufs=type:VIDEO_CAPTURE_MPLANE,count:1 --pipe=2 \
          --output=/tmp/frames.vf --open=/dev/video6 \
          --fmt=type:VIDEO_CAPTURE_MPLANE,width=2560,height=1920,pixelformat=NV12 \
          --reqbufs=type:VIDEO_CAPTURE_MPLANE,count:1 --pipe=3 --open=/dev/video7 \
          --output=/tmp/frames.3A --fmt=type:META_CAPTURE,? \
          --reqbufs=count:1,type:META_CAPTURE --pipe=1,2,3,4 --stream=5

Вы также можете использовать команду yavta [#f2]_ для выполнения того же самого,
что и выше:

.. code-block:: none

    yavta --data-prefix -Bcapture-mplane -c10 -n5 -I -s2592x1944 \
          --file=frame-#.out-f NV12 /dev/video5 & \
    yavta --data-prefix -Bcapture-mplane -c10 -n5 -I -s2592x1944 \
          --file=frame-#.vf -f NV12 /dev/video6 & \
    yavta --data-prefix -Bmeta-capture -c10 -n5 -I \
          --file=frame-#.3a /dev/video7 & \
    yavta --data-prefix -Boutput-mplane -c10 -n5 -I -s2592x1944 \
          --file=/tmp/frame-in.cio2 -f IPU3_SGRBG10 /dev/video4

где устройства /dev/video4, /dev/video5, /dev/video6 и /dev/video7 указывают на
видео-узлы входа, выхода, видоискателя и статистики 3A соответственно.

Преобразование необработанного изображения Байера в домен YUV
-------------------------------------------------------------

Обработанные изображения после вышеуказанного шага могут быть преобразованы в
домен YUV, как показано ниже.

Кадры основного выхода
~~~~~~~~~~~~~~~~~~~~~~~

.. code-block:: none

    raw2pnm -x2560 -y1920 -fNV12 /tmp/frames.out /tmp/frames.out.ppm

где 2560x1920 — выходное разрешение, NV12 — формат видео, за которыми следуют
входной кадр и выходной PNM-файл.

Обзор конвейера IPU3
====================

Конвейер IPU3 имеет ряд стадий обработки изображений, каждая из которых принимает
набор параметров в качестве входных данных. Основные стадии конвейеров показаны
здесь:

В таблице ниже представлено описание вышеуказанных алгоритмов.

======================== =======================================================
Название		 Описание
======================== =======================================================
Optical Black Correction Блок Optical Black Correction вычитает заранее
			 заданное значение из соответствующих значений
			 пикселей для получения лучшего качества изображения.
			 Определён в struct ipu3_uapi_obgrid_param.
Linearization		 Этот блок алгоритма использует параметры линеаризации
			 для устранения нелинейных эффектов сенсора. Таблица
			 поиска (Lookup table) определена в
			 struct ipu3_uapi_isp_lin_vmem_params.
SHD			 Lens shading correction используется для коррекции
			 пространственной неоднородности отклика пикселей,
			 вызванной оптическим затенением объектива. Это
			 делается путём применения разного усиления для
			 каждого пикселя. Усиление, уровень чёрного и т.д.
			 настраиваются в struct ipu3_uapi_shd_config_static.
BNR			 Блок Bayer noise reduction удаляет шум изображения
			 путём применения билатерального фильтра.
			 Подробности см. в struct ipu3_uapi_bnr_static_config.
ANR			 Advanced Noise Reduction — это блочный алгоритм,
			 который выполняет подавление шума в домене Байера.
			 Матрицу свёртки и т.д. можно найти в
			 struct ipu3_uapi_anr_config.
DM			 Demosaicing преобразует необработанные данные сенсора в
			 формате Байера в представление RGB (Red, Green,
			 Blue). Затем добавляет выходные данные оценки канала Y
			 для последующей потоковой обработки прошивкой.
			 Структура определена как
			 struct ipu3_uapi_dm_config.
Color Correction	 Алгоритм Color Correction преобразует специфичное для
			 сенсора цветовое пространство в стандартное цветовое
			 пространство "sRGB". Это делается путём применения
			 матрицы 3x3, определённой в
			 struct ipu3_uapi_ccm_mat_config.
Gamma correction	 Gamma correction struct ipu3_uapi_gamma_config — это
			 базовая нелинейная тоновая коррекция (tone mapping),
			 применяемая попиксельно к каждой компоненте пикселя.
CSC			 Color space conversion преобразует каждый пиксель из
			 представления первичных цветов RGB в представление YUV
			 (Y: яркость, UV: цветность). Это делается путём
			 применения матрицы 3x3, определённой в
			 struct ipu3_uapi_csc_mat_config
CDS			 Chroma down sampling
			 После выполнения CSC применяется Chroma Down Sampling
			 для уменьшения плоскости UV в 2 раза в каждом
			 направлении для YUV 4:2:0 с использованием
			 настраиваемого фильтра 4x2
			 struct ipu3_uapi_cds_params.
CHNR			 Chroma noise reduction
			 Этот блок обрабатывает только пиксели цветности и
			 выполняет подавление шума путём очистки
			 высокочастотного шума.
			 См. struct struct ipu3_uapi_yuvp1_chnr_config.
TCC			 Total color correction, как определено в struct
			 struct ipu3_uapi_yuvp2_tcc_static_config.
XNR3			 eXtreme Noise Reduction V3 — это третья редакция
			 алгоритма подавления шума, используемого для улучшения
			 качества изображения. Он удаляет низкочастотный шум в
			 захваченном изображении. Определены две связанные
			 структуры:
			 struct ipu3_uapi_isp_xnr3_params для памяти данных ISP
			 и struct ipu3_uapi_isp_xnr3_vmem_params для векторной
			 памяти.
TNR			 Блок Temporal Noise Reduction сравнивает
			 последовательные кадры во времени для удаления
			 аномалий/шума в значениях пикселей.
			 struct ipu3_uapi_isp_tnr3_vmem_params и
			 struct ipu3_uapi_isp_tnr3_params определены для
			 векторной памяти и памяти данных ISP соответственно.
======================== =======================================================

Прочие часто встречающиеся аббревиатуры, не перечисленные в таблице выше:

	ACC
		Accelerator cluster
	AWB_FR
		Auto white balance filter response statistics
	BDS
		Bayer downscaler parameters
	CCM
		Color correction matrix coefficients
	IEFd
		Image enhancement filter directed
	Obgrid
		Optical black level compensation
	OSYS
		Output system configuration
	ROI
		Region of interest
	YDS
		Y down sampling
	YTM
		Y-tone mapping

Несколько стадий конвейера будут выполняться прошивкой, работающей на процессоре
ISP, в то время как многие другие будут использовать набор фиксированных
аппаратных блоков, также называемых accelerator cluster (ACC), для обработки
данных пикселей и формирования статистики.

Параметры ACC отдельных алгоритмов, определённые в struct ipu3_uapi_acc_param,
могут быть выбраны для применения пространством пользователя через struct struct
ipu3_uapi_flags, встроенную в структуру struct ipu3_uapi_params. Для параметров,
которые настроены пространством пользователя как не включённые, соответствующие
структуры игнорируются драйвером; в этом случае существующая конфигурация
алгоритма будет сохранена.
