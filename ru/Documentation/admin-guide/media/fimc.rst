Драйвер FIMC для Samsung S5P/Exynos4
====================================

Copyright |copy| 2012 - 2013 Samsung Electronics Co., Ltd.

Устройство FIMC (Fully Interactive Mobile Camera), доступное в прикладных
процессорах Samsung SoC, представляет собой интегрированный интерфейс хоста
камеры, преобразователь цветового пространства, средство изменения размера
изображения и его поворота. Оно также способно захватывать данные с контроллера
LCD (FIMD) через внутренний для SoC тракт данных обратной записи (writeback). В
SoC присутствует несколько экземпляров FIMC (до 4), имеющих немного различающиеся
возможности, например ограничения на выравнивание пикселей, наличие средства
поворота, поддержку обратной записи на LCD и т. д. Драйвер расположен в каталоге
drivers/media/platform/samsung/exynos4-is.

Поддерживаемые SoC
------------------

S5PC100 (только mem-to-mem), S5PV210, Exynos4210

Поддерживаемые возможности
--------------------------

- захват через параллельный интерфейс камеры (ITU-R.BT601/565);
- захват через последовательный интерфейс камеры (MIPI-CSI2);
- обработка memory-to-memory (преобразование цветового пространства,
  масштабирование, зеркальное отражение и поворот);
- динамическая перенастройка конвейера во время работы (повторное подключение
  любого экземпляра FIMC к любому параллельному видеовходу или любому фронтенду
  MIPI-CSI);
- runtime PM и общесистемный suspend/resume

В настоящее время не поддерживается
-----------------------------------

- вход обратной записи с LCD
- покадровое стробирование тактового сигнала (mem-to-mem)

Интерфейсы пространства пользователя
------------------------------------

Интерфейс медиаустройства
~~~~~~~~~~~~~~~~~~~~~~~~~~

Драйвер поддерживает Media Controller API, как определено в :ref:`media_controller`.
Имя драйвера медиаустройства — "Samsung S5P FIMC".

Назначение этого интерфейса — позволить менять привязку экземпляров FIMC к
периферийному входу камеры SoC во время работы и опционально управлять
внутренними соединениями устройства(устройств) MIPI-CSIS с сущностями FIMC.

Интерфейс медиаустройства позволяет настроить SoC для захвата данных изображения
с датчика через более чем один экземпляр FIMC (например, для одновременной
настройки видоискателя и захвата неподвижного изображения).

Перенастройка выполняется путём включения/отключения медиасвязей (links),
создаваемых драйвером во время инициализации. Внутреннюю топологию устройства
можно легко обнаружить через перечисление медиасущностей и связей.

Видеоузел memory-to-memory
~~~~~~~~~~~~~~~~~~~~~~~~~~~

Интерфейс V4L2 memory-to-memory на узле устройства /dev/video?. Это автономное
видеоустройство, у него нет медиаконтактов (pads). Однако обратите внимание, что
работа узлов видео mem-to-mem и захвата на одном и том же экземпляре FIMC не
допускается. Драйвер обнаруживает такие случаи, но приложения должны
предотвращать их во избежание неопределённого поведения.

Видеоузел захвата
~~~~~~~~~~~~~~~~~

Драйвер поддерживает V4L2 Video Capture Interface, как определено в
:ref:`devices`.

На видеоузлах захвата и mem-to-mem поддерживается только многоплоскостной
(multi-planar) API. Подробнее см.: :ref:`planar-apis`.

Subdev-устройства захвата камеры
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Каждый экземпляр FIMC экспортирует узел субустройства (/dev/v4l-subdev?); узел
субустройства также создаётся для каждого доступного и включённого на уровне
платформы устройства приёмника MIPI-CSI (в настоящее время до двух).

sysfs
~~~~~

Для обеспечения более точного управления конвейером камеры через subdev API
драйвер создаёт запись sysfs, связанную с платформенным устройством
"s5p-fimc-md". Путь к записи: /sys/platform/devices/s5p-fimc-md/subdev_conf_mode.

В типичном случае использования возможна следующая конфигурация конвейера
захвата:
sensor subdev -> mipi-csi subdev -> fimc subdev -> video node

Когда мы настраиваем эти устройства через subdev API в пространстве пользователя,
поток настройки должен идти слева направо, и видеоузел настраивается последним.

Когда мы не используем subdev API пространства пользователя, вся настройка всех
устройств, принадлежащих конвейеру, выполняется в драйвере видеоузла.
Запись sysfs позволяет указать драйверу узла захвата не настраивать
субустройства (формат, обрезка), чтобы избежать сброса конфигурации субустройств
при выполнении последних шагов настройки на видеоузле.

Для полной поддержки управления субустройствами (субустройства настраиваются в
пространстве пользователя до начала потоковой передачи):

.. code-block:: none

	# echo "sub-dev" > /sys/platform/devices/s5p-fimc-md/subdev_conf_mode

Только для управления через видеоузел V4L2 (субустройства настраиваются
внутренне хост-драйвером):

.. code-block:: none

	# echo "vid-dev" > /sys/platform/devices/s5p-fimc-md/subdev_conf_mode

Это вариант по умолчанию.

5. Сопоставление устройств узлам устройств video и subdev
---------------------------------------------------------

С каждым экземпляром устройства в аппаратуре связаны два узла видеоустройства —
video capture и mem-to-mem, а дополнительно — узел subdev для более точного
управления подсистемой захвата FIMC. Кроме того, отдельный узел субустройства
v4l2 создаётся для каждого устройства MIPI-CSIS.

Как узнать, какой /dev/video? или /dev/v4l-subdev? назначен какому устройству?

Можно либо найти нужную информацию через grep в журнале ядра, т. е.

.. code-block:: none

	# dmesg | grep -i fimc

(обратите внимание, что udev, если он присутствует, всё ещё может переставить
видеоузлы),

либо получить информацию из /dev/media? с помощью инструмента media-ctl:

.. code-block:: none

	# media-ctl -p

7. Сборка
---------

Если драйвер собран как загружаемый модуль ядра
(CONFIG_VIDEO_SAMSUNG_S5P_FIMC=m), создаются два модуля (в дополнение к основным
модулям v4l2): s5p-fimc.ko и опциональный s5p-csis.ko (subdev приёмника
MIPI-CSI).
