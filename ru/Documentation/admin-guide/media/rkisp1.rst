=========================================
Rockchip Image Signal Processor (rkisp1)
=========================================

Введение
========

Этот файл документирует драйвер для Rockchip ISP1, входящего в состав SoC RK3288
и RK3399. Драйвер расположен в drivers/media/platform/rockchip/
rkisp1 и использует Media-Controller API.

Ревизии
=======

Существует несколько менее значительных ревизий этого ISP, появившихся в
более поздних SoC. Ревизии можно найти в enum :c:type:`rkisp1_cif_isp_version`
в UAPI, а ревизию ISP внутри работающей SoC можно прочитать
в поле hw_revision структуры media_device_info, возвращаемой
ioctl MEDIA_IOC_DEVICE_INFO.

Используемые версии:

- RKISP1_V10: используется как минимум в rk3288 и rk3399
- RKISP1_V11: объявлена в исходном вендорском коде, но не используется
- RKISP1_V12: используется как минимум в rk3326 и px30
- RKISP1_V13: используется как минимум в rk1808

Топология
=========
.. _rkisp1_topology_graph:

.. kernel-figure:: rkisp1.dot
    :alt:   Diagram of the default media pipeline topology
    :align: center


У драйвера есть 4 видеоустройства:

- rkisp1_mainpath: устройство захвата для получения изображений, обычно в более
  высоком разрешении.
- rkisp1_selfpath: устройство захвата для получения изображений.
- rkisp1_stats: устройство захвата метаданных, отправляющее статистику.
- rkisp1_params: устройство вывода метаданных, принимающее конфигурации
  параметров из пространства пользователя (userspace).

У драйвера есть 3 поддустройства (subdevice):

- rkisp1_resizer_mainpath: используется для изменения размера и понижающей
  дискретизации кадров для устройства захвата mainpath.
- rkisp1_resizer_selfpath: используется для изменения размера и понижающей
  дискретизации кадров для устройства захвата selfpath.
- rkisp1_isp: подключено к сенсору и отвечает за все операции isp.


rkisp1_mainpath, rkisp1_selfpath — видеоузлы захвата кадров
-----------------------------------------------------------
Это устройства захвата `mainpath` и `selfpath` для захвата кадров.
Эти сущности являются движками DMA, которые записывают кадры в память.
Видеоустройство selfpath может захватывать форматы YUV/RGB. Его входом является
поток в кодировке YUV, и оно способно конвертировать его в RGB. Selfpath не
способно захватывать форматы bayer.
Mainpath может захватывать как форматы bayer, так и YUV, но не способно
захватывать форматы RGB.
Оба видеоустройства захвата поддерживают
:ref:`возможность <device-capabilities>` ``V4L2_CAP_IO_MC``.


rkisp1_resizer_mainpath, rkisp1_resizer_selfpath — узлы поддустройств resizer
-----------------------------------------------------------------------------
Это сущности resizer для mainpath и selfpath. Эти сущности
могут масштабировать кадры в сторону увеличения и уменьшения, а также изменять
дискретизацию YUV (например, YUV4:2:2 -> YUV4:2:0). Они также имеют возможность
кадрирования на sink pad.
Сущности resizer могут работать только с форматом YUV:4:2:2
(MEDIA_BUS_FMT_YUYV8_2X8).
Устройство захвата mainpath поддерживает захват видео в форматах bayer. В этом
случае resizer для mainpath переводится в режим «обхода» (bypass) — он просто
передаёт кадр без операций над ним.

rkisp1_isp — узел поддустройства обработки сигнала изображения
--------------------------------------------------------------
Это сущность isp. Она подключена к сенсору на sink pad 0 и
получает кадры по протоколу CSI-2. Она отвечает за настройку
протокола CSI-2. Она имеет возможность кадрирования на sink pad 0, который
подключён к сенсору, и на source pad 2, подключённом к сущностям resizer.
Кадрирование на sink pad 0 определяет область изображения от сенсора.
Кадрирование на source pad 2 определяет область для стабилизатора изображения (IS).

.. _rkisp1_stats:

rkisp1_stats — видеоузел статистики
-----------------------------------
Видеоузел статистики выводит статистику 3A (автофокус, автоэкспозиция и
автоматический баланс белого), а также статистику гистограммы для кадров,
обрабатываемых rkisp1, в приложения пространства пользователя (userspace).
Используя эти данные, приложения могут реализовывать алгоритмы и заново
параметризовать драйвер через узел rkisp_params для улучшения качества
изображения во время видеопотока.
Формат буфера определяется структурой :c:type:`rkisp1_stat_buffer`, а
пространство пользователя должно установить
:ref:`V4L2_META_FMT_RK_ISP1_STAT_3A <v4l2-meta-fmt-rk-isp1-stat-3a>` в качестве
формата данных.

.. _rkisp1_params:

rkisp1_params — видеоузел параметров
------------------------------------
Видеоузел rkisp1_params получает из пространства пользователя набор параметров
для применения к оборудованию во время видеопотока, что позволяет пространству
пользователя динамически изменять такие значения, как уровень чёрного, коррекции
перекрёстных помех (cross talk) и другие.

Драйвер ISP поддерживает два различных метода конфигурации параметров —
`fixed parameters format` или `extensible parameters format`.

При использовании метода `fixed parameters` формат буфера определяется структурой
:c:type:`rkisp1_params_cfg`, а пространство пользователя должно установить
:ref:`V4L2_META_FMT_RK_ISP1_PARAMS <v4l2-meta-fmt-rk-isp1-params>` в качестве
формата данных.

При использовании метода `extensible parameters` формат буфера определяется
структурой :c:type:`rkisp1_ext_params_cfg`, а пространство пользователя должно
установить
:ref:`V4L2_META_FMT_RK_ISP1_EXT_PARAMS <v4l2-meta-fmt-rk-isp1-ext-params>` в
качестве формата данных.

Пример захвата видеокадров
==========================

В следующем примере сенсор, подключённый к pad 0 «rkisp1_isp», — это
imx219.

Следующие команды можно использовать для захвата видео с видеоузла selfpath
с размером 900x800 в планарном формате YUV 4:2:2. Используются все возможные
возможности кадрирования (см. пояснение прямо ниже)

.. code-block:: bash

	# set the links
	"media-ctl" "-d" "platform:rkisp1" "-r"
	"media-ctl" "-d" "platform:rkisp1" "-l" "'imx219 4-0010':0 -> 'rkisp1_isp':0 [1]"
	"media-ctl" "-d" "platform:rkisp1" "-l" "'rkisp1_isp':2 -> 'rkisp1_resizer_selfpath':0 [1]"
	"media-ctl" "-d" "platform:rkisp1" "-l" "'rkisp1_isp':2 -> 'rkisp1_resizer_mainpath':0 [0]"

	# set format for imx219 4-0010:0
	"media-ctl" "-d" "platform:rkisp1" "--set-v4l2" '"imx219 4-0010":0 [fmt:SRGGB10_1X10/1640x1232]'

	# set format for rkisp1_isp pads:
	"media-ctl" "-d" "platform:rkisp1" "--set-v4l2" '"rkisp1_isp":0 [fmt:SRGGB10_1X10/1640x1232 crop: (0,0)/1600x1200]'
	"media-ctl" "-d" "platform:rkisp1" "--set-v4l2" '"rkisp1_isp":2 [fmt:YUYV8_2X8/1600x1200 crop: (0,0)/1500x1100]'

	# set format for rkisp1_resizer_selfpath pads:
	"media-ctl" "-d" "platform:rkisp1" "--set-v4l2" '"rkisp1_resizer_selfpath":0 [fmt:YUYV8_2X8/1500x1100 crop: (300,400)/1400x1000]'
	"media-ctl" "-d" "platform:rkisp1" "--set-v4l2" '"rkisp1_resizer_selfpath":1 [fmt:YUYV8_2X8/900x800]'

	# set format for rkisp1_selfpath:
	"v4l2-ctl" "-z" "platform:rkisp1" "-d" "rkisp1_selfpath" "-v" "width=900,height=800,"
	"v4l2-ctl" "-z" "platform:rkisp1" "-d" "rkisp1_selfpath" "-v" "pixelformat=422P"

	# start streaming:
	v4l2-ctl "-z" "platform:rkisp1" "-d" "rkisp1_selfpath" "--stream-mmap" "--stream-count" "10"


В приведённом выше примере сенсор сконфигурирован в формат bayer:
`SRGGB10_1X10/1640x1232`. Pad rkisp1_isp:0 должен быть сконфигурирован в тот
же формат mbus и размеры, что и сенсор, иначе потоковая передача завершится
с ошибкой «EPIPE». Поэтому он также сконфигурирован в `SRGGB10_1X10/1640x1232`.
Кроме того, pad rkisp1_isp:0 сконфигурирован на кадрирование `(0,0)/1600x1200`.

Размеры кадрирования автоматически распространяются и становятся форматом
source pad isp `rkisp1_isp:2`. На source pad isp конфигурируется ещё одна
операция кадрирования: `(0,0)/1500x1100`.

Sink pad resizer `rkisp1_resizer_selfpath` должен быть сконфигурирован в формат
`YUYV8_2X8/1500x1100`, чтобы соответствовать формату на другой стороне
канала связи. Кроме того, на нём конфигурируется кадрирование `(300,400)/1400x1000`.

Source pad resizer, `rkisp1_resizer_selfpath:1`, сконфигурирован в
формат `YUYV8_2X8/900x800`. Это означает, что resizer сначала кадрирует окно
`(300,400)/1400x100` из полученного кадра, а затем масштабирует это окно
до размера `900x800`.

Обратите внимание, что в приведённом выше примере не используется управляющий
цикл stats-params. Поэтому захватываемые кадры не будут проходить через
алгоритмы 3A и, вероятно, не будут иметь хорошего качества, а могут даже
выглядеть тёмными и зеленоватыми.

Настройка квантования
=====================

Драйвер поддерживает ограниченный (limited) и полный (full) диапазон квантования
для форматов YUV, где по умолчанию используется ограниченный.
Для переключения между одним и другим пространство пользователя должно
использовать API преобразования цветовых пространств (Colorspace
Conversion API, CSC) для поддустройств на source pad 2 isp
(`rkisp1_isp:2`). Квантование, сконфигурированное на этом pad, является
квантованием захватываемых видеокадров на видеоузлах mainpath и selfpath.
Обратите внимание, что сущности resizer и захвата всегда будут сообщать
``V4L2_QUANTIZATION_DEFAULT``, даже если квантование сконфигурировано на полный
диапазон на `rkisp1_isp:2`. Поэтому, чтобы получить сконфигурированное
квантование, приложение должно получать его от pad `rkisp1_isp:2`.
