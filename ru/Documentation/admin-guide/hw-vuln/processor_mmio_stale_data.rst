===============================================================
Уязвимости устаревших данных MMIO процессора (Stale Data MMIO)
===============================================================

Уязвимости устаревших данных MMIO процессора (Processor MMIO Stale Data
Vulnerabilities) — это класс уязвимостей ввода-вывода с отображением в
память (MMIO), которые могут приводить к раскрытию данных. Последовательности
операций для раскрытия данных варьируются от простых до очень сложных.
Поскольку большинство этих уязвимостей требует, чтобы атакующий имел доступ
к MMIO, многие среды не затронуты. Системные среды, использующие виртуализацию,
в которых доступ к MMIO предоставляется недоверенным гостям, могут нуждаться в
смягчении. Эти уязвимости не являются атаками спекулятивного исполнения
(transient execution attacks). Однако они могут распространять устаревшие
данные в буферы заполнения ядра (core fill buffers), где эти данные впоследствии
могут быть выведены неустранённой атакой спекулятивного исполнения. Смягчение
этих уязвимостей включает комбинацию обновления микрокода и программных
изменений в зависимости от платформы и модели использования. Некоторые из этих
смягчений похожи на те, что применяются для смягчения Microarchitectural Data
Sampling (MDS), либо на те, что применяются для смягчения Special Register
Buffer Data Sampling (SRBDS).

Распространители данных (Data Propagators)
==========================================
Распространители (propagators) — это операции, в результате которых устаревшие
данные копируются или перемещаются из одного микроархитектурного буфера или
регистра в другой. Уязвимости устаревших данных MMIO процессора — это операции,
в результате которых устаревшие данные могут быть прочитаны напрямую в
архитектурное, видимое программам состояние, либо считаны (sampled) из буфера
или регистра.

Fill Buffer Stale Data Propagator (FBSDP)
-----------------------------------------
Устаревшие данные могут распространяться из буферов заполнения (fill buffers, FB)
в некогерентную часть uncore при некоторых некогерентных записях. Само по себе
распространение из буфера заполнения не делает устаревшие данные архитектурно
видимыми. Устаревшие данные должны быть распространены в место, где они подлежат
чтению или считыванию (sampling).

Sideband Stale Data Propagator (SSDP)
-------------------------------------
Распространитель устаревших данных по боковой шине (sideband stale data
propagator, SSDP) ограничен клиентской реализацией uncore (включая серверы Intel
Xeon E3). Буфер ответов боковой шины (sideband response buffer) совместно
используется всеми клиентскими ядрами. Для некогерентных чтений, идущих к
назначениям по боковой шине, логика uncore возвращает ядру 64 байта данных,
включающих как запрошенные данные, так и незапрошенные устаревшие данные, из
буфера транзакций и буфера ответов боковой шины. В результате устаревшие данные
из буфера ответов боковой шины и буфера транзакций могут теперь находиться в
буфере заполнения ядра.

Primary Stale Data Propagator (PSDP)
------------------------------------
Распространитель устаревших данных по основному каналу (primary stale data
propagator, PSDP) ограничен клиентской реализацией uncore (включая серверы Intel
Xeon E3). Подобно буферу ответов боковой шины, буфер основных ответов (primary
response buffer) совместно используется всеми клиентскими ядрами. Для некоторых
процессоров основные чтения MMIO будут возвращать в буфер заполнения ядра 64
байта данных, включающих как запрошенные данные, так и незапрошенные устаревшие
данные. Это похоже на распространитель устаревших данных по боковой шине.

Уязвимости
==========
Device Register Partial Write (DRPW) (CVE-2022-21166)
-----------------------------------------------------
Некоторые конечные (endpoint) регистры MMIO некорректно обрабатывают записи,
размер которых меньше размера регистра. Вместо прерывания записи или копирования
только корректного подмножества байтов (например, 2 байтов для 2-байтовой
записи) в регистр может быть записано больше байтов, чем указано в транзакции
записи. На процессорах, затронутых FBSDP, это может привести к раскрытию
устаревших данных из буферов заполнения ядра, создавшего транзакцию записи.

Shared Buffers Data Sampling (SBDS) (CVE-2022-21125)
----------------------------------------------------
После того как распространители могли переместить данные внутри uncore и
скопировать устаревшие данные в буферы заполнения клиентских ядер, процессоры,
затронутые MFBDS, могут допустить утечку данных из буфера заполнения. Это
ограничено клиентской реализацией uncore (включая серверы Intel Xeon E3).

Shared Buffers Data Read (SBDR) (CVE-2022-21123)
------------------------------------------------
Это похоже на Shared Buffer Data Sampling (SBDS), за исключением того, что данные
читаются напрямую в архитектурное, видимое программам состояние. Это ограничено
клиентской реализацией uncore (включая серверы Intel Xeon E3).

Затронутые процессоры
=====================
Не все процессоры затронуты всеми вариантами. Например, большинство процессоров
для серверного рынка (исключая процессоры Intel Xeon E3) подвержены только
Device Register Partial Write (DRPW).

   ===================  ============  =========
   HASWELL_X            06_3FH        2,4
   SKYLAKE_L            06_4EH        3
   BROADWELL_X          06_4FH        All
   SKYLAKE_X            06_55H        3,4,6,7,11
   BROADWELL_D          06_56H        3,4,5
   SKYLAKE              06_5EH        3
   ICELAKE_X            06_6AH        4,5,6
   ICELAKE_D            06_6CH        1
   ICELAKE_L            06_7EH        5
   ATOM_TREMONT_D       06_86H        All
   LAKEFIELD            06_8AH        1
   KABYLAKE_L           06_8EH        9 to 12
   ATOM_TREMONT         06_96H        1
   ATOM_TREMONT_L       06_9CH        0
   KABYLAKE             06_9EH        9 to 13
   COMETLAKE            06_A5H        2,3,5
   COMETLAKE_L          06_A6H        0,1
   ROCKETLAKE           06_A7H        1

Если процессор есть в списке затронутых, но не затронут каким-либо вариантом, это
обозначается новыми битами в MSR IA32_ARCH_CAPABILITIES. Как описано в одном из
последующих разделов, смягчение по большей части остаётся одинаковым для всех
вариантов, т. е. заключается в очистке буферов заполнения CPU с помощью
инструкции VERW.

Новые биты в MSR
================
В новых процессорах и в обновлениях микрокода для существующих затронутых
процессоров добавлены новые биты в MSR IA32_ARCH_CAPABILITIES. Эти биты могут
использоваться для перечисления конкретных вариантов уязвимостей устаревших
данных MMIO процессора и возможностей смягчения.

MSR IA32_ARCH_CAPABILITIES
--------------------------
Бит 13 - SBDR_SSDP_NO - Когда установлен, процессор не затронут ни уязвимостью
	 Shared Buffers Data Read (SBDR), ни распространителем устаревших данных
	 по боковой шине (SSDP).
Бит 14 - FBSDP_NO - Когда установлен, процессор не затронут Fill Buffer
	 Stale Data Propagator (FBSDP).
Бит 15 - PSDP_NO - Когда установлен, процессор не затронут Primary Stale Data
	 Propagator (PSDP).
Бит 17 - FB_CLEAR - Когда установлен, инструкция VERW будет перезаписывать
	 значения буфера заполнения CPU в рамках операций MD_CLEAR. Процессоры,
	 которые не перечисляют MDS_NO (то есть затронуты MDS), но при этом
	 перечисляют поддержку как L1D_FLUSH, так и MD_CLEAR, неявно перечисляют
	 FB_CLEAR в рамках своей поддержки MD_CLEAR.
Бит 18 - FB_CLEAR_CTRL - Процессор поддерживает чтение и запись MSR
	 IA32_MCU_OPT_CTRL[FB_CLEAR_DIS]. На таких процессорах бит FB_CLEAR_DIS
	 может быть установлен, чтобы инструкция VERW не выполняла действие
	 FB_CLEAR. Не все процессоры, поддерживающие FB_CLEAR, будут поддерживать
	 FB_CLEAR_CTRL.

MSR IA32_MCU_OPT_CTRL
---------------------
Бит 3 - FB_CLEAR_DIS - Когда установлен, инструкция VERW не выполняет действие
FB_CLEAR. Это может быть полезно для уменьшения влияния FB_CLEAR на
производительность в случаях, когда системное ПО сочтёт это оправданным
(например, когда производительность более критична, либо когда недоверенное ПО
не имеет доступа к MMIO). Обратите внимание, что FB_CLEAR_DIS не влияет на
перечисление (например, не изменяет перечисление FB_CLEAR или MD_CLEAR) и может
не поддерживаться на всех процессорах, перечисляющих FB_CLEAR.

Смягчение
=========
Как и MDS, все варианты уязвимостей устаревших данных MMIO процессора имеют одну
и ту же стратегию смягчения — заставить CPU очистить затронутые буферы прежде,
чем атакующий сможет извлечь секреты.

Это достигается использованием в иных случаях неиспользуемой и устаревшей
инструкции VERW в сочетании с обновлением микрокода. Микрокод очищает затронутые
буферы CPU при выполнении инструкции VERW.

Ядро выполняет очистку буферов с помощью x86_clear_cpu_buffers().

На CPU, затронутых MDS, ядро уже вызывает очистку буферов CPU при переходах
ядро/userspace, гипервизор/гость и при переходах C-состояний (простой, idle). На
таких CPU дополнительное смягчение не требуется.

Для CPU, не затронутых MDS или TAA, смягчение нужно только против атакующего с
возможностью доступа к MMIO. Поэтому VERW не требуется для перехода
ядро/userspace. В случае виртуализации VERW нужна только при VMENTER для гостя с
возможностью доступа к MMIO.

Точки смягчения
---------------
Возврат в пространство пользователя
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
То же смягчение, что и для MDS, при затронутости MDS/TAA; в противном случае
смягчение не требуется.

Переход C-состояния
^^^^^^^^^^^^^^^^^^^
Записи в управляющие регистры, выполняемые CPU при переходе C-состояния, могут
распространять данные из буфера заполнения в буферы uncore. Выполняйте VERW перед
переходом C-состояния, чтобы очистить буферы заполнения CPU.

Точка входа гостя
^^^^^^^^^^^^^^^^^
То же смягчение, что и для MDS, когда процессор также затронут MDS/TAA; в
противном случае выполняйте VERW при VMENTER только для гостей с возможностью
доступа к MMIO. На CPU, не затронутых MDS/TAA, гость без доступа к MMIO не может
извлекать секреты с помощью уязвимостей устаревших данных MMIO процессора,
поэтому для таких гостей выполнять VERW не нужно.

Управление смягчением через командную строку ядра
-------------------------------------------------
Командная строка ядра позволяет управлять смягчениями устаревших данных MMIO
процессора во время загрузки с помощью опции "mmio_stale_data=". Допустимые
аргументы для этой опции:

  ==========  =================================================================
  full        Если CPU уязвим, включить смягчение; очистка буферов CPU при
              выходе в userspace и при входе в виртуальную машину. Переходы в
              простой (idle) также защищены. Эта опция не отключает SMT
              автоматически.
  full,nosmt  То же, что full, но с отключённым SMT на уязвимых CPU. Это полное
              смягчение.
  off         Полностью отключает смягчение.
  ==========  =================================================================

Если CPU затронут и в командной строке ядра не указано mmio_stale_data=off, то
ядро выбирает подходящее смягчение.

Информация о статусе смягчения
------------------------------
Ядро Linux предоставляет интерфейс sysfs для перечисления текущего статуса
уязвимости системы: уязвима ли система и какие смягчения активны. Соответствующий
файл sysfs:

	/sys/devices/system/cpu/vulnerabilities/mmio_stale_data

Возможные значения в этом файле:

  .. list-table::

     * - 'Not affected'
       - Процессор не уязвим
     * - 'Vulnerable'
       - Процессор уязвим, но смягчение не включено
     * - 'Vulnerable: Clear CPU buffers attempted, no microcode'
       - Процессор уязвим, но микрокод не обновлён. Смягчение включено в режиме
         наилучших усилий (best effort).

         Если процессор уязвим, но доступность механизма смягчения на основе
         микрокода не объявлена через CPUID, ядро выбирает режим смягчения с
         наилучшими усилиями. В этом режиме вызываются инструкции смягчения без
         гарантии того, что они очищают буферы CPU.

         Это сделано для сценариев виртуализации, где на хосте применено
         обновление микрокода, но гипервизор ещё не обновлён, чтобы предоставить
         CPUID гостю. Если на хосте обновлён микрокод, защита вступает в силу; в
         противном случае несколько тактов CPU тратятся впустую.
     * - 'Mitigation: Clear CPU buffers'
       - Процессор уязвим, и смягчение очисткой буферов CPU включено.
     * - 'Unknown: No mitigations'
       - Статус уязвимости процессора неизвестен, поскольку он вне периода
	 обслуживания (Servicing period). Смягчение не выполняется.

Определения:
------------

Период обслуживания (Servicing period): процесс предоставления функциональных
обновлений и обновлений безопасности для процессоров или платформ Intel с
использованием процесса Intel Platform Update (IPU) или иных подобных механизмов.

Окончание обновлений обслуживания (End of Servicing Updates, ESU): ESU — это
дата, начиная с которой Intel больше не будет предоставлять обслуживание, в том
числе через IPU или иные подобные процессы обновления. Даты ESU обычно
приурочены к концу квартала.

Если процессор уязвим, то к приведённой выше информации добавляется следующая
информация:

  ========================  ===========================================
  'SMT vulnerable'          SMT включён
  'SMT disabled'            SMT отключён
  'SMT Host state unknown'  Ядро работает в VM, состояние SMT хоста неизвестно
  ========================  ===========================================
