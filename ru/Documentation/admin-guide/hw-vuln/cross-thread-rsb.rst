Межпоточные предсказания адресов возврата
==========================================

Некоторые процессоры AMD и Hygon подвержены уязвимости межпоточного
предсказания адресов возврата. При работе в режиме SMT, когда один из
парных потоков переходит из состояния C0, другой парный поток может
использовать предсказания адресов перехода возврата от парного потока,
который вышел из состояния C0.

Меры противодействия Spectre v2 защищают ядро Linux, так как оно заполняет
записи предсказаний адресов возврата безопасными целями при переключении
контекста на поток простоя. Однако KVM позволяет VMM предотвращать выход из
режима гостя при переходе из состояния C0. Это может привести к тому, что
управляемая гостем цель возврата будет использована парным потоком.

Затронутые процессоры
---------------------

Уязвимы следующие CPU:

    - процессоры AMD семейства 17h
    - процессоры Hygon семейства 18h

Связанные CVE
-------------

С этой проблемой связана следующая запись CVE:

   ==============  =======================================
   CVE-2022-27672  Cross-Thread Return Address Predictions
   ==============  =======================================

Проблема
--------

Затронутые процессоры с поддержкой SMT поддерживают режимы выполнения 1T и 2T,
когда SMT включён. В режиме 2T оба потока в ядре выполняют код. Чтобы ядро
процессора перешло в режим 1T, требуется, чтобы один из потоков запросил
переход из состояния C0. Это может быть сообщено с помощью инструкции HLT или
инструкции MWAIT, запрашивающей состояние не-C0.
Когда поток снова входит в состояние C0, процессор переходит обратно в режим
2T, при условии, что другой поток также по-прежнему находится в состоянии C0.

В затронутых процессорах предсказатель адресов возврата (RAP) разделён в
зависимости от режима SMT. Например, в режиме 2T каждый поток использует
приватный RAP на 16 записей, но в режиме 1T активный поток использует RAP на
32 записи. При переходе между режимами 1T/2T содержимое RAP не изменяется, но
указатели RAP (которые управляют тем, какая цель возврата используется
следующей для предсказаний) могут измениться. Такое поведение может привести к
тому, что цели возврата от одного потока SMT будут использованы предсказаниями
RET в парном потоке после переключения 1T/2T. В частности, инструкция RET,
выполненная сразу после перехода в 1T, может использовать цель возврата от
потока, который только что стал простаивающим. Теоретически это может привести
к раскрытию информации, если используемые цели возврата исходят не из
доверенного кода.

Сценарии атаки
--------------

Атака на затронутых процессорах может быть осуществлена путём выполнения серии
инструкций CALL с заданными адресами возврата и последующего перехода из
состояния C0.

Механизм противодействия
------------------------

Перед входом в состояние простоя ядро переключает контекст на поток простоя.
Переключение контекста заполняет записи RAP (называемые в Linux RSB)
безопасными целями путём выполнения последовательности инструкций CALL.

Предотвратить прямой перевод процессора в состояние простоя гостевой ВМ путём
перехвата инструкций HLT и MWAIT.

Для полного устранения этой проблемы требуются обе меры противодействия.

Управление мерами противодействия в командной строке ядра
---------------------------------------------------------

Используйте существующие меры противодействия Spectre v2, которые будут
заполнять RSB при переключении контекста.

Управление мерами противодействия для KVM — параметр модуля
-----------------------------------------------------------

По умолчанию гипервизор KVM смягчает эту проблему, перехватывая попытки гостя
выйти из состояния C0. VMM может использовать возможность
KVM_CAP_X86_DISABLE_EXITS, чтобы переопределить эти перехваты, но, поскольку
это нераспространённо, мера противодействия, охватывающая этот путь, по
умолчанию не включена.

Меру противодействия для возможности KVM_CAP_X86_DISABLE_EXITS можно включить
с помощью булева параметра модуля mitigate_smt_rsb, например
``kvm.mitigate_smt_rsb=1``.
