TAA - TSX Asynchronous Abort
======================================

TAA — это аппаратная уязвимость, которая позволяет осуществлять
непривилегированный спекулятивный доступ к данным, находящимся в различных
внутренних буферах CPU, путём использования асинхронных прерываний (aborts)
внутри транзакционной области Intel TSX.

Затронутые процессоры
---------------------

Эта уязвимость затрагивает только процессоры Intel, которые поддерживают Intel
Transactional Synchronization Extensions (TSX), когда бит TAA_NO (бит 8) равен 0
в MSR IA32_ARCH_CAPABILITIES.  На процессорах, где бит MDS_NO (бит 5) равен 0 в
MSR IA32_ARCH_CAPABILITIES, существующие меры защиты от MDS также защищают и от
TAA.

Затронут ли процессор, можно узнать из файла уязвимости TAA в sysfs.
См. :ref:`tsx_async_abort_sys_info`.

Связанные CVE
-------------

С данной проблемой TAA связана следующая запись CVE:

   ==============  =====  ===================================================
   CVE-2019-11135  TAA    TSX Asynchronous Abort (TAA) condition on some
                          microprocessors utilizing speculative execution may
                          allow an authenticated user to potentially enable
                          information disclosure via a side channel with
                          local access.
   ==============  =====  ===================================================

Проблема
--------

При выполнении операций сохранения (store), загрузки (load) или дозаполнения L1
(L1 refill) процессоры записывают данные во временные микроархитектурные
структуры (буферы). Данные из этих буферов могут перенаправляться операциям
загрузки в качестве оптимизации.

Intel TSX — это расширение архитектуры набора команд x86, которое добавляет
аппаратную поддержку транзакционной памяти для повышения производительности
многопоточного программного обеспечения. TSX позволяет процессору раскрывать и
использовать параллелизм, скрытый в приложении, за счёт динамического избегания
ненужной синхронизации.

TSX поддерживает атомарные транзакции памяти, которые либо фиксируются (успех),
либо прерываются (abort). Во время прерывания операции, произошедшие внутри
транзакционной области, откатываются. Асинхронное прерывание происходит, среди
прочих случаев, когда другой поток обращается к строке кэша, которая также
используется внутри транзакционной области, и этот доступ может привести к
состоянию гонки (data race).

Сразу после незавершённого асинхронного прерывания некоторые спекулятивно
выполненные загрузки могут считать данные из этих внутренних буферов и передать
их зависимым операциям. Затем это может быть использовано для вывода значения
посредством атаки по побочному каналу кэша.

Поскольку буферы потенциально являются общими для Hyper-Threads, возможны атаки
между Hyper-Thread'ами.

Жертве злоумышленника не требуется использовать TSX. Только атакующему нужно
начать транзакцию TSX и вызвать асинхронное прерывание, которое, в свою очередь,
потенциально приводит к утечке данных, хранящихся в буферах.

Более подробная техническая информация доступна в разделе архитектуры x86,
посвящённом TAA:
:ref:`Documentation/arch/x86/tsx_async_abort.rst <tsx_async_abort>`.


Сценарии атаки
--------------

Атаки на уязвимость TAA могут быть реализованы из непривилегированных приложений,
работающих на хостах или в гостевых системах.

Как и в случае с MDS, атакующий не контролирует адреса памяти, из которых могут
произойти утечки. Только жертва отвечает за доставку данных в CPU. В результате
злоумышленнику приходится считывать как можно больше данных, а затем выполнять их
постобработку, чтобы попытаться извлечь из них хоть какую-то полезную информацию.

Потенциальный атакующий имеет только доступ на чтение данных. Кроме того, прямого
повышения привилегий при использовании этой техники не происходит.


.. _tsx_async_abort_sys_info:

Системная информация TAA
------------------------

Ядро Linux предоставляет интерфейс sysfs для получения текущего статуса TAA на
системах с применёнными мерами защиты. Соответствующий файл sysfs:

/sys/devices/system/cpu/vulnerabilities/tsx_async_abort

В этом файле возможны следующие значения:

.. list-table::

   * - 'Vulnerable'
     - CPU затронут этой уязвимостью, а меры защиты на уровне микрокода и ядра не применены.
   * - 'Vulnerable: Clear CPU buffers attempted, no microcode'
     - Процессор уязвим, но микрокод не обновлён. Мера защиты
       включена по принципу наилучших усилий (best effort).

       Если процессор уязвим, но доступность механизма защиты на основе
       микрокода не объявлена через CPUID, ядро выбирает режим защиты по
       принципу наилучших усилий. Этот режим вызывает инструкции защиты без
       гарантии того, что они очищают буферы CPU.

       Это сделано для покрытия сценариев виртуализации, где на хосте применено
       обновление микрокода, но гипервизор ещё не обновлён, чтобы предоставлять
       CPUID гостевой системе. Если на хосте обновлён микрокод, защита вступает
       в силу; в противном случае несколько тактов CPU расходуются впустую.
   * - 'Mitigation: Clear CPU buffers'
     - Микрокод обновлён для очистки буферов. TSX по-прежнему включён.
   * - 'Mitigation: TSX disabled'
     - TSX отключён.
   * - 'Not affected'
     - CPU не затронут этой проблемой.

Механизм защиты
---------------

Ядро обнаруживает затронутые CPU и наличие требуемого микрокода. Если CPU
затронут и микрокод доступен, ядро включает меру защиты по умолчанию.


Мерой защиты можно управлять во время загрузки с помощью параметра командной
строки ядра. См. :ref:`taa_mitigation_control_command_line`.

Защита при виртуализации
^^^^^^^^^^^^^^^^^^^^^^^^^

Затронутые системы, на которых хост имеет микрокод TAA и TAA защищён за счёт
ранее отключённого TSX, не уязвимы независимо от статуса виртуальных машин.

Во всех остальных случаях, если на хосте либо отсутствует микрокод TAA, либо
ядро не защищено, система может быть уязвима.


.. _taa_mitigation_control_command_line:

Управление защитой через командную строку ядра
----------------------------------------------

Командная строка ядра позволяет управлять мерами защиты от TAA во время загрузки
с помощью параметра "tsx_async_abort=". Допустимые аргументы для этого параметра:

  ============  =============================================================
  off		This option disables the TAA mitigation on affected platforms.
                If the system has TSX enabled (see next parameter) and the CPU
                is affected, the system is vulnerable.

  full	        TAA mitigation is enabled. If TSX is enabled, on an affected
                system it will clear CPU buffers on ring transitions. On
                systems which are MDS-affected and deploy MDS mitigation,
                TAA is also mitigated. Specifying this option on those
                systems will have no effect.

  full,nosmt    The same as tsx_async_abort=full, with SMT disabled on
                vulnerable CPUs that have TSX enabled. This is the complete
                mitigation. When TSX is disabled, SMT is not disabled because
                CPU is not vulnerable to cross-thread TAA attacks.
  ============  =============================================================

Если этот параметр не указан, это эквивалентно "tsx_async_abort=full". Для
процессоров, затронутых одновременно TAA и MDS, указание только
"tsx_async_abort=off" без сопутствующего "mds=off" не возымеет эффекта, так как
для обеих уязвимостей используется одна и та же мера защиты.

Командная строка ядра также позволяет управлять функциональностью TSX с помощью
параметра "tsx=" на CPU, поддерживающих управление TSX. Для управления
функциональностью TSX и перечислением битов функциональности TSX (RTM и HLE) в
CPUID используется MSR_IA32_TSX_CTRL.

Допустимые варианты:

  ============  =============================================================
  off		Disables TSX on the system.

                Note that this option takes effect only on newer CPUs which are
                not vulnerable to MDS, i.e., have MSR_IA32_ARCH_CAPABILITIES.MDS_NO=1
                and which get the new IA32_TSX_CTRL MSR through a microcode
                update. This new MSR allows for the reliable deactivation of
                the TSX functionality.

  on		Enables TSX.

                Although there are mitigations for all known security
                vulnerabilities, TSX has been known to be an accelerator for
                several previous speculation-related CVEs, and so there may be
                unknown security risks associated with leaving it enabled.

  auto		Disables TSX if X86_BUG_TAA is present, otherwise enables TSX
                on the system.
  ============  =============================================================

Если этот параметр не указан, это эквивалентно "tsx=off".

Возможны следующие комбинации параметров "tsx_async_abort" и "tsx". Для
затронутых платформ tsx=auto эквивалентно tsx=off, и результат будет следующим:

  =========  ==========================   =========================================
  tsx=on     tsx_async_abort=full         The system will use VERW to clear CPU
                                          buffers. Cross-thread attacks are still
					  possible on SMT machines.
  tsx=on     tsx_async_abort=full,nosmt   As above, cross-thread attacks on SMT
                                          mitigated.
  tsx=on     tsx_async_abort=off          The system is vulnerable.
  tsx=off    tsx_async_abort=full         TSX might be disabled if microcode
                                          provides a TSX control MSR. If so,
					  system is not vulnerable.
  tsx=off    tsx_async_abort=full,nosmt   Ditto
  tsx=off    tsx_async_abort=off          ditto
  =========  ==========================   =========================================


Для незатронутых платформ "tsx=on" и "tsx_async_abort=full" не очищают буферы
CPU.  Для платформ без управления TSX (MSR_IA32_ARCH_CAPABILITIES.MDS_NO=0)
аргумент командной строки "tsx" не возымеет эффекта.

Для затронутых платформ приведённая ниже таблица указывает статус защиты для
комбинаций бита CPUID MD_CLEAR и битов MSR IA32_ARCH_CAPABILITIES MDS_NO и
TSX_CTRL_MSR.

  =======  =========  =============  ========================================
  MDS_NO   MD_CLEAR   TSX_CTRL_MSR   Status
  =======  =========  =============  ========================================
    0          0            0        Vulnerable (needs microcode)
    0          1            0        MDS and TAA mitigated via VERW
    1          1            0        MDS fixed, TAA vulnerable if TSX enabled
                                     because MD_CLEAR has no meaning and
                                     VERW is not guaranteed to clear buffers
    1          X            1        MDS fixed, TAA can be mitigated by
                                     VERW or TSX_CTRL_MSR
  =======  =========  =============  ========================================

Руководство по выбору защиты
----------------------------

1. Доверенное пространство пользователя и гостевые системы
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Если все приложения пространства пользователя получены из доверенного источника
и не выполняют недоверенный код, поставляемый извне, то меру защиты можно
отключить. То же относится к виртуализированным окружениям с доверенными
гостевыми системами.


2. Недоверенное пространство пользователя и гостевые системы
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Если в системе есть недоверенные приложения или гостевые системы, включение TSX
может позволить злоумышленнику получить утечку данных с хоста или из других
процессов, работающих на том же физическом ядре.

Если микрокод доступен и TSX отключён на хосте, атаки также предотвращаются и в
виртуализированном окружении, даже если виртуальные машины явно не включают меру
защиты.

Меры защиты по умолчанию
------------------------

Действие ядра по умолчанию для уязвимых процессоров:

  - Применить меру защиты с отключением TSX (tsx_async_abort=full tsx=off).
