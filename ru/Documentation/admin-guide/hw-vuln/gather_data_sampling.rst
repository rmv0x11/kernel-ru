GDS - Gather Data Sampling
==========================

Gather Data Sampling — это аппаратная уязвимость, которая позволяет
непривилегированный спекулятивный доступ к данным, ранее хранившимся в
векторных регистрах.

Проблема
--------
Когда инструкция gather выполняет загрузку из памяти, различные элементы данных
объединяются в целевой векторный регистр. Однако, когда инструкция gather,
выполняемая транзитно, сталкивается со сбоем (fault), вместо этого в целевой
векторный регистр могут транзитно перенаправляться устаревшие данные из
архитектурных или внутренних векторных регистров. Это позволит злоумышленнику
вывести устаревшие данные, используя типичные методы побочных каналов, такие как
атаки по времени доступа к кэшу. GDS — это атака, основанная исключительно на
сэмплировании.

Злоумышленник использует инструкции gather для вывода устаревших данных
векторных регистров. Жертве не нужно делать ничего особенного, кроме
использования векторных регистров. Жертве не нужно использовать инструкции
gather, чтобы быть уязвимой.

Поскольку буферы являются общими для Hyper-Threads, возможны атаки между
Hyper-Threads.

Сценарии атаки
--------------
Без мер защиты GDS может выводить устаревшие данные практически через все
границы разрешений:

	Не-анклавы могут выводить данные анклава SGX
	Userspace может выводить данные ядра
	Гости могут выводить данные с хостов
	Гость может выводить данные гостя от других гостей
	Пользователи могут выводить данные от других пользователей

В связи с этим важно обеспечить, чтобы меры защиты оставались включёнными в
контекстах с более низкими привилегиями, таких как гости, и при работе вне
анклавов SGX.

Для SGX меры защиты обеспечивает аппаратное обеспечение. Аналогично, VMM должны
гарантировать, что гостям не разрешено отключать меры защиты GDS. Если хост
ошибётся и разрешит это, гость теоретически сможет отключить меры защиты GDS,
осуществить атаку и снова их включить.

Механизм защиты
---------------
Эта проблема устраняется в микрокоде. Микрокод определяет следующие новые биты:

 ================================   ===   ============================
 IA32_ARCH_CAPABILITIES[GDS_CTRL]   R/O   Enumerates GDS vulnerability
                                          and mitigation support.
 IA32_ARCH_CAPABILITIES[GDS_NO]     R/O   Processor is not vulnerable.
 IA32_MCU_OPT_CTRL[GDS_MITG_DIS]    R/W   Disables the mitigation
                                          0 by default.
 IA32_MCU_OPT_CTRL[GDS_MITG_LOCK]   R/W   Locks GDS_MITG_DIS=0. Writes
                                          to GDS_MITG_DIS are ignored
                                          Can't be cleared once set.
 ================================   ===   ============================

GDS также может быть смягчён на системах без обновлённого микрокода путём
отключения AVX. Это можно сделать, задав gather_data_sampling="force" или
"clearcpuid=avx" в командной строке ядра.

В случае использования эти параметры отключат применение AVX путём выключения
поддержки XSAVE YMM. Однако процессор всё равно будет сообщать о поддержке AVX.
Userspace, который не следует корректному определению поддержки AVX с проверкой
как AVX, *так и* поддержки XSAVE YMM, перестанет работать.

Управление мерами защиты через командную строку ядра
----------------------------------------------------
Меры защиты можно отключить, задав "gather_data_sampling=off" или
"mitigations=off" в командной строке ядра. Если ни один из вариантов не указан,
по умолчанию меры защиты будут включены. Указание "gather_data_sampling=force"
приведёт к использованию мер защиты микрокода, когда они доступны, или к
отключению AVX на затронутых системах, где микрокод не был обновлён для
включения мер защиты.

Системная информация GDS
------------------------
Ядро предоставляет информацию о статусе уязвимости через sysfs. Для GDS доступ к
ней можно получить через следующий файл sysfs:

/sys/devices/system/cpu/vulnerabilities/gather_data_sampling

Возможные значения, содержащиеся в этом файле:

 ============================== =============================================
 Not affected                   Processor not vulnerable.
 Vulnerable                     Processor vulnerable and mitigation disabled.
 Vulnerable: No microcode       Processor vulnerable and microcode is missing
                                mitigation.
 Mitigation: AVX disabled,
 no microcode                   Processor is vulnerable and microcode is missing
                                mitigation. AVX disabled as mitigation.
 Mitigation: Microcode          Processor is vulnerable and mitigation is in
                                effect.
 Mitigation: Microcode (locked) Processor is vulnerable and mitigation is in
                                effect and cannot be disabled.
 Unknown: Dependent on
 hypervisor status              Running on a virtual guest processor that is
                                affected but with no way to know if host
                                processor is mitigated or vulnerable.
 ============================== =============================================

Меры защиты GDS по умолчанию
----------------------------
Обновлённый микрокод включит меры защиты по умолчанию. Действие ядра по умолчанию
— оставить меры защиты включёнными.
