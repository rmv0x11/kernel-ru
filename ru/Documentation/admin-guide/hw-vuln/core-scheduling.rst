===============
Core Scheduling
===============
Поддержка core scheduling позволяет userspace задавать группы задач, которые могут
совместно использовать ядро процессора (core). Эти группы могут указываться либо для
сценариев безопасности (одна группа задач не доверяет другой), либо для сценариев
производительности (некоторым нагрузкам может быть выгодно выполняться на одном и том
же ядре, поскольку им не нужны одни и те же аппаратные ресурсы общего ядра, либо они
могут предпочитать разные ядра, если у них действительно есть общие потребности в
аппаратных ресурсах). Этот документ описывает только сценарий безопасности.

Сценарий безопасности
---------------------
Атака cross-HT предполагает выполнение атакующего и жертвы на разных Hyper Threads одного
и того же ядра. MDS и L1TF — примеры таких атак. Единственное полное средство защиты от
атак cross-HT — отключение Hyper Threading (HT). Core scheduling — это возможность
планировщика, которая может смягчить некоторые (не все) атаки cross-HT. Она позволяет
безопасно включить HT, гарантируя, что только задачи из назначенной пользователем
доверенной группы могут совместно использовать ядро. Это увеличение совместного
использования ядра также может улучшить производительность, однако не гарантируется, что
производительность всегда будет улучшаться, хотя это наблюдается на ряде реальных нагрузок.
В теории core scheduling стремится работать по меньшей мере так же хорошо, как при
отключённом Hyper Threading. На практике это в основном так, хотя и не всегда: поскольку
синхронизация решений планировщика между 2 или более CPU в ядре влечёт дополнительные
накладные расходы — особенно когда система слабо загружена. При ``total_threads <= N_CPUS/2``
дополнительные накладные расходы могут привести к тому, что core scheduling будет работать
хуже по сравнению с отключённым SMT, где N_CPUS — это общее число CPU. Всегда измеряйте
производительность своих нагрузок.

Использование
-------------
Поддержка core scheduling включается с помощью параметра конфигурации ``CONFIG_SCHED_CORE``.
Используя эту возможность, userspace задаёт группы задач, которые могут планироваться
совместно на одном ядре. Core scheduler использует эту информацию, чтобы гарантировать, что
задачи, не входящие в одну группу, никогда не выполняются одновременно на ядре, при этом
делая всё возможное для удовлетворения требований планирования системы.

Core scheduling можно включить через интерфейс prctl ``PR_SCHED_CORE``.
Этот интерфейс предоставляет поддержку для создания групп core scheduling, а также для
добавления задач в созданные группы и удаления их из групп::

    #include <sys/prctl.h>

    int prctl(int option, unsigned long arg2, unsigned long arg3,
            unsigned long arg4, unsigned long arg5);

option:
    ``PR_SCHED_CORE``

arg2:
    Команда операции, должна быть одной из:

    - ``PR_SCHED_CORE_GET`` -- получить cookie core_sched для ``pid``.
    - ``PR_SCHED_CORE_CREATE`` -- создать новый уникальный cookie для ``pid``.
    - ``PR_SCHED_CORE_SHARE_TO`` -- передать cookie core_sched в ``pid``.
    - ``PR_SCHED_CORE_SHARE_FROM`` -- получить cookie core_sched из ``pid``.

arg3:
    ``pid`` задачи, к которой применяется операция.

arg4:
    ``pid_type``, для которого применяется операция. Это одна из макроконстант с
    префиксом ``PR_SCHED_CORE_SCOPE_``. Например, если arg4 равно
    ``PR_SCHED_CORE_SCOPE_THREAD_GROUP``, то операция этой команды будет выполнена
    для всех задач в группе задач ``pid``.

arg5:
    указатель userspace на unsigned long long для хранения cookie, возвращаемого
    командой ``PR_SCHED_CORE_GET``. Должен быть 0 для всех остальных команд.

Чтобы процесс мог передать cookie в другой процесс или получить cookie из другого
процесса, требуется иметь режим доступа ptrace `PTRACE_MODE_READ_REALCREDS` к этому
процессу.

Построение иерархий задач
~~~~~~~~~~~~~~~~~~~~~~~~~~
Простейший способ построить иерархии потоков/процессов, которые совместно используют
cookie и, следовательно, ядро, — опереться на тот факт, что cookie core-sched наследуется
при fork/clone и exec, поэтому установка cookie для «начального» скрипта/исполняемого
файла/демона помещает каждого порождённого потомка в ту же группу core-sched.

Передача cookie
~~~~~~~~~~~~~~~~
Передача cookie между текущей и другими задачами возможна с помощью
PR_SCHED_CORE_SHARE_FROM и PR_SCHED_CORE_SHARE_TO, чтобы унаследовать cookie от
указанной задачи или поделиться cookie с задачей. В сочетании это позволяет простой
вспомогательной программе получить cookie от задачи в существующей группе core
scheduling и поделиться им с уже выполняющимися задачами.

Проектирование/реализация
-------------------------
Каждой задаче, которая помечена тегом, внутри ядра назначается cookie. Как упомянуто в
разделе `Использование`_, задачи с одинаковым значением cookie считаются доверяющими друг
другу и совместно используют ядро.

Основная идея в том, что каждое событие планирования пытается выбрать задачи для всех
siblings ядра так, чтобы все выбранные задачи, выполняющиеся на ядре, были доверенными
(имели одинаковый cookie) в любой момент времени. Потоки ядра считаются доверенными.
Задача idle считается особой, так как она доверяет всему, и всё доверяет ей.

Во время события schedule() на любом sibling ядра выбирается задача с наивысшим приоритетом
на ядре этого sibling и назначается sibling'у, вызвавшему schedule(), если у этого sibling
данная задача поставлена в очередь. Для остальных siblings в ядре выбирается задача с
наивысшим приоритетом и тем же cookie, если в их индивидуальных очередях выполнения есть
такая готовая к выполнению. Если задача с тем же cookie недоступна, выбирается задача idle.
Задача idle глобально доверенная.

После того как задача выбрана для всех siblings в ядре, тем siblings, для которых была
выбрана новая задача, отправляется IPI. Siblings при получении IPI немедленно переключаются
на новую задачу. Если для sibling выбрана задача idle, то sibling считается находящимся в
состоянии `forced idle`. То есть в его собственной очереди выполнения могут быть задачи для
выполнения, однако ему всё равно придётся выполнять idle. Подробнее об этом в следующем
разделе.

Принудительный простой (forced-idling) hyperthreads
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Планировщик делает всё возможное, чтобы найти задачи, которые доверяют друг другу, так,
чтобы все задачи, выбранные для планирования, имели наивысший приоритет в ядре. Однако
возможно, что в некоторых очередях выполнения были задачи, несовместимые с задачами
наивысшего приоритета в ядре. Отдавая предпочтение безопасности над справедливостью, один
или более siblings может быть вынужден выбрать задачу с более низким приоритетом, если
задача наивысшего приоритета не является доверенной по отношению к общеядерной задаче
наивысшего приоритета. Если у sibling нет доверенной задачи для выполнения, он будет
принудительно переведён в состояние idle планировщиком (на выполнение ставится поток idle).

Когда задача наивысшего приоритета выбрана для выполнения, sibling'у отправляется
reschedule-IPI, чтобы принудительно перевести его в idle. Это приводит к 4 случаям, которые
необходимо рассмотреть в зависимости от того, выполнялась ли на любом из HT виртуальная
машина (VM) или обычный процесс пользовательского режима::

          HT1 (attack)            HT2 (victim)
   A      idle -> user space      user space -> idle
   B      idle -> user space      guest -> idle
   C      idle -> guest           user space -> idle
   D      idle -> guest           guest -> idle

Обратите внимание, что для лучшей производительности мы не ждём, пока целевой CPU (жертва)
войдёт в режим idle. Это связано с тем, что отправка IPI немедленно переведёт целевой CPU из
пространства пользователя в режим ядра, либо вызовет VMEXIT в случае гостей. В лучшем случае
это привело бы лишь к утечке некоторых метаданных планировщика, которые, возможно, не стоит
защищать. Также возможно, что на некоторых архитектурах IPI принимается слишком поздно, но
это не наблюдалось в случае x86.

Модель доверия
~~~~~~~~~~~~~~
Core scheduling поддерживает отношения доверия между группами задач, назначая им тег,
представляющий собой одно и то же значение cookie.
Когда загружается система с core scheduling, все задачи считаются доверяющими друг другу.
Это связано с тем, что core scheduler не имеет информации об отношениях доверия до тех пор,
пока userspace не воспользуется упомянутыми выше интерфейсами, чтобы сообщить их. Другими
словами, все задачи имеют значение cookie по умолчанию, равное 0, и считаются доверенными в
масштабах всей системы. Принудительного простоя siblings, выполняющих задачи с cookie 0,
также удаётся избежать.

Как только userspace использует упомянутые выше интерфейсы для группировки наборов задач,
задачи в таких группах считаются доверяющими друг другу, но не доверяют тем, кто вне группы.
Задачи вне группы также не доверяют задачам внутри.

Ограничения core-scheduling
----------------------------
Core scheduling пытается гарантировать, что только доверенные задачи выполняются одновременно
на ядре. Но может существовать небольшое окно времени, в течение которого недоверенные задачи
выполняются одновременно, либо ядро может выполняться одновременно с задачей, не доверенной
ядром.

Задержки обработки IPI
~~~~~~~~~~~~~~~~~~~~~~~
Core scheduling выбирает для совместного выполнения только доверенные задачи. IPI используется
для уведомления siblings о необходимости переключиться на новую задачу. Но на некоторых
архитектурах могут возникать аппаратные задержки при получении IPI (на x86 это не
наблюдалось). Это может привести к тому, что задача атакующего начнёт выполняться на CPU
раньше, чем его siblings получат IPI. Хотя кэш сбрасывается при входе в пользовательский
режим, задачи-жертвы на siblings могут заполнить данными кэш и микроархитектурные буферы
после того, как атакующий начнёт выполняться, и это является возможностью для утечки данных.

Открытые проблемы cross-HT, которые core scheduling не решает
-------------------------------------------------------------
1. Для MDS
~~~~~~~~~~
Core scheduling не может защитить от атак MDS между siblings, выполняющимися в
пользовательском режиме, и остальными, выполняющимися в режиме ядра. Хотя все siblings
выполняют задачи, которые доверяют друг другу, когда ядро выполняет код от имени задачи, оно
не может доверять коду, выполняющемуся в sibling. Такие атаки возможны для любой комбинации
режимов sibling CPU (режим хоста или гостя).

2. Для L1TF
~~~~~~~~~~~
Core scheduling не может защитить от гостевого атакующего L1TF, эксплуатирующего гостевую или
хостовую жертву. Это связано с тем, что гостевой атакующий может сформировать некорректные
PTE, которые не инвертируются из-за уязвимого гостевого ядра. Единственное решение —
отключить EPT (Extended Page Tables).

Как для MDS, так и для L1TF, если гостевые vCPU настроены не доверять друг другу (путём
раздельной пометки тегами), то атаки гость-гость исчезнут. Либо это может быть политика
системного администратора, которая рассматривает атаки гость-гость как проблему гостя.

Другой подход к решению этих проблем — сделать так, чтобы каждая недоверенная задача в системе
не доверяла любой другой недоверенной задаче. Хотя это могло бы снизить параллелизм
недоверенных задач, это всё равно решило бы указанные выше проблемы, позволяя при этом
системным процессам (доверенным задачам) совместно использовать ядро.

3. Защита ядра (IRQ, syscall, VMEXIT)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
К сожалению, core scheduling не защищает контексты ядра, выполняющиеся на sibling
hyperthreads, друг от друга. Прототипы средств защиты были опубликованы в LKML для решения
этой проблемы, но спорно, эксплуатируемы ли такие окна на практике и стоят ли накладные
расходы прототипов на производительность того (не говоря уже о добавленной сложности кода).

Другие сценарии использования
-----------------------------
Основной сценарий использования core scheduling — смягчение уязвимостей cross-HT при
включённом SMT. Существуют и другие сценарии, в которых эту возможность можно было бы
использовать:

- Изоляция задач, которым требуется целое ядро: к примерам относятся задачи реального
  времени, задачи, использующие инструкции SIMD, и т. д.
- Gang scheduling: требования к группе задач, которые должны планироваться вместе, также
  могут быть реализованы с помощью core scheduling. Один из примеров — vCPU виртуальной
  машины (VM).
