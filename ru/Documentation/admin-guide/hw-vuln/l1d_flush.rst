Сброс L1D
=========

В связи с ростом числа сообщений об уязвимостях, связанных с утечками
данных из кэша данных первого уровня (L1D), ядро предоставляет
механизм с возможностью включения по запросу (opt-in) для сброса кэша
L1D при переключении контекста.

Этот механизм может использоваться, например, для устранения
CVE-2020-0550. Для приложений данный механизм защищает их от
уязвимостей, связанных с утечками (подглядыванием за данными) из кэша
L1D.


Связанные CVE
-------------
Этим механизмом можно устранить следующие CVE:

    =============       ========================     ==================
    CVE-2020-0550       Improper Data Forwarding     OS related aspects
    =============       ========================     ==================

Руководство по использованию
----------------------------

Подробности см. в документе: :ref:`Documentation/userspace-api/spec_ctrl.rst
<set_spec_ctrl>`.

**ПРИМЕЧАНИЕ**: Эта возможность по умолчанию отключена, приложениям
необходимо явно подключиться к ней (opt-in), чтобы её включить.

Меры противодействия
--------------------

Когда для задачи включён PR_SPEC_L1D_FLUSH, сброс кэша L1D выполняется
в момент, когда задача снимается с выполнения, а входящая задача
принадлежит другому процессу и, следовательно, другому адресному
пространству.

Если нижележащий CPU поддерживает сброс L1D на аппаратном уровне,
используется аппаратный механизм; программный резервный вариант для
данной меры противодействия не поддерживается.

Управление мерой противодействия через командную строку ядра
------------------------------------------------------------

Командная строка ядра позволяет управлять мерами противодействия по
сбросу L1D во время загрузки с помощью опции "l1d_flush=". Допустимые
аргументы этой опции:

  ============  =============================================================
  on            Включает интерфейс prctl; приложения, пытающиеся
                использовать prctl(), завершатся с ошибкой, если l1d_flush
                не включён
  ============  =============================================================

По умолчанию механизм отключён.

Ограничения
-----------

Этот механизм не противодействует утечкам данных L1D между задачами,
принадлежащими разным процессам, которые одновременно выполняются на
соседних (sibling) потоках одного физического ядра CPU, когда в системе
включён SMT.

Это можно устранить путём контролируемого размещения процессов на
физических ядрах CPU или отключением SMT. См. соответствующую главу в
документе о мерах противодействия L1TF:
:ref:`Documentation/admin-guide/hw-vuln/l1tf.rst <smt_control>`.

**ПРИМЕЧАНИЕ** : Подключение задачи (opt-in) к сбросу L1D работает
только тогда, когда привязка (affinity) задачи ограничена ядрами,
работающими в режиме без SMT. Если задача, запросившая сброс L1D,
попадает на выполнение на ядро с включённым SMT, ядро отправляет задаче
сигнал SIGBUS.
