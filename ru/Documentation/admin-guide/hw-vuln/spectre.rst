Побочные каналы Spectre
=======================

Spectre — это класс атак по побочным каналам, использующих предсказание
переходов и спекулятивное исполнение в современных CPU для чтения памяти,
возможно, в обход контроля доступа. Эксплойты побочных каналов спекулятивного
исполнения не изменяют память, но пытаются вывести привилегированные данные,
находящиеся в памяти.

Этот документ охватывает Spectre вариант 1 и Spectre вариант 2.

Затронутые процессоры
---------------------

Методы побочных каналов спекулятивного исполнения затрагивают широкий спектр
современных высокопроизводительных процессоров, поскольку большинство
современных высокоскоростных процессоров используют предсказание переходов и
спекулятивное исполнение.

Следующие CPU уязвимы:

    - Процессоры Intel Core, Atom, Pentium и Xeon

    - Процессоры AMD Phenom, EPYC и Zen

    - Процессоры IBM POWER и zSeries

    - Процессоры ARM верхнего уровня

    - CPU Apple

    - CPU MIPS верхнего уровня

    - Вероятно, большинство других высокопроизводительных CPU. За подробностями обращайтесь к производителю вашего CPU.

Уязвим процессор или нет, можно узнать из файлов уязвимости Spectre в sysfs.
См. :ref:`spectre_sys_info`.

Связанные CVE
-------------

Следующие записи CVE описывают варианты Spectre:

   =============   =======================  ==========================
   CVE-2017-5753   Bounds check bypass      Spectre variant 1
   CVE-2017-5715   Branch target injection  Spectre variant 2
   CVE-2019-1125   Spectre v1 swapgs        Spectre variant 1 (swapgs)
   =============   =======================  ==========================

Проблема
--------

CPU используют спекулятивные операции для повышения производительности. Это
может оставлять следы обращений к памяти или вычислений в кэшах, буферах и
предсказателях переходов процессора. Вредоносное программное обеспечение может
быть способно влиять на пути спекулятивного исполнения, а затем использовать
побочные эффекты спекулятивного исполнения в кэшах и буферах CPU для вывода
привилегированных данных, к которым обращались во время спекулятивного
исполнения.

Атаки Spectre вариант 1 используют спекулятивное исполнение условных
переходов, тогда как атаки Spectre вариант 2 используют спекулятивное
исполнение косвенных переходов для утечки привилегированной памяти.
См. :ref:`[1] <spec_ref1>` :ref:`[5] <spec_ref5>` :ref:`[6] <spec_ref6>`
:ref:`[7] <spec_ref7>` :ref:`[10] <spec_ref10>` :ref:`[11] <spec_ref11>`.

Spectre вариант 1 (Bounds Check Bypass)
---------------------------------------

Атака с обходом проверки границ :ref:`[2] <spec_ref2>` использует
спекулятивное исполнение, которое обходит инструкции условного перехода,
применяемые для проверки границ доступа к памяти (например, проверки того,
что индекс массива приводит к обращению к памяти в допустимом диапазоне). Это
приводит к обращениям к недопустимой памяти (с выходящим за границы индексом),
которые выполняются спекулятивно до того, как разрешатся проверки валидации.
Такие спекулятивные обращения к памяти могут оставлять побочные эффекты,
создавая побочные каналы, через которые информация утекает к атакующему.

Существуют некоторые расширения атак Spectre вариант 1 для чтения данных по
сети, см. :ref:`[12] <spec_ref12>`. Однако такие атаки сложны, обладают низкой
пропускной способностью, ненадёжны и считаются малорисковыми.

Обратите внимание, что, несмотря на название «Bounds Check Bypass», Spectre
вариант 1 связан не только с проверками границ массивов, контролируемых
пользователем. Он может затрагивать любые условные проверки. Код входа в ядро
— обработчики прерываний, исключений и NMI — все имеют условные проверки
swapgs. Они могут быть проблематичными в контексте Spectre v1, поскольку код
ядра может спекулятивно выполняться с пользовательским GS.

Spectre вариант 2 (Branch Target Injection)
-------------------------------------------

Атака с внедрением цели перехода использует спекулятивное исполнение косвенных
переходов :ref:`[3] <spec_ref3>`. На предсказатели косвенных переходов внутри
процессора, используемые для угадывания цели косвенных переходов, может влиять
атакующий, заставляя код-гаджет исполняться спекулятивно и тем самым раскрывая
конфиденциальные данные, к которым обращалась жертва. Побочные эффекты,
оставленные в кэшах CPU во время спекулятивного исполнения, можно измерить для
вывода значений данных.

.. _poison_btb:

В атаках Spectre вариант 2 атакующий может направлять спекулятивные косвенные
переходы в жертве к коду-гаджету путём отравления буфера целей переходов
(branch target buffer) CPU, используемого для предсказания адресов косвенных
переходов. Такое отравление может выполняться косвенным переходом в
существующий код, при этом смещение адреса косвенного перехода находится под
контролем атакующего. Поскольку предсказание переходов на затронутом
оборудовании не полностью различает адреса переходов и использует для
предсказания смещение, это может привести к тому, что косвенный переход
привилегированного кода перейдёт к коду-гаджету с тем же смещением.

Наиболее полезные гаджеты принимают входной параметр, контролируемый
атакующим (например, значение регистра), так что чтение памяти можно
контролировать. Гаджеты без входных параметров возможны, но атакующий имел бы
очень малый контроль над тем, какую память можно прочитать, что снижает риск
того, что атака раскроет полезные данные.

Ещё один вектор атаки варианта 2 заключается в том, что атакующий отравляет
буфер стека возврата (return stack buffer, RSB) :ref:`[13] <spec_ref13>`,
чтобы заставить спекулятивное исполнение инструкции возврата из подпрограммы
перейти к гаджету. Несбалансированные инструкции вызова подпрограмм
атакующего могут «отравить» записи в буфере стека возврата, которые позже
используются инструкциями возврата из подпрограмм жертвы. Эту атаку можно
смягчить, сбрасывая буфер стека возврата при переключении контекста или при
выходе из виртуальной машины (VM).

В системах с одновременной многопоточностью (SMT) атаки возможны со стороны
соседнего потока, поскольку кэш уровня 1 и буфер целей переходов (BTB) могут
совместно использоваться аппаратными потоками в ядре CPU. Вредоносная
программа, выполняющаяся в соседнем потоке, может влиять на BTB своего соседа,
чтобы направить его спекуляции косвенных переходов к коду-гаджету, и измерять
побочные эффекты спекулятивного исполнения, оставленные в кэше уровня 1, для
вывода данных жертвы.

Ещё один вектор атаки варианта 2 заключается в том, что атакующий отравляет
буфер истории переходов (Branch History Buffer, BHB), чтобы спекулятивно
направить косвенный переход к конкретной записи буфера целей переходов (BTB),
даже если эта запись не связана с адресом источника косвенного перехода. В
частности, BHB может совместно использоваться разными уровнями привилегий даже
при наличии Enhanced IBRS.

Ранее единственным известным реальным вектором атаки на BHB был
непривилегированный eBPF. Дальнейшие исследования обнаружили атаки, не
требующие непривилегированного eBPF. Для полного смягчения атак на BHB
рекомендуется установить BHI_DIS_S или использовать последовательность очистки
BHB.

Сценарии атак
-------------

Следующий список сценариев атак был предусмотрен, но может не охватывать все
возможные векторы атак.

1. Пользовательский процесс, атакующий ядро
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Spectre вариант 1
~~~~~~~~~~~~~~~~~

   Атакующий передаёт ядру параметр через регистр или через известный адрес
   в памяти во время системного вызова. Такой параметр может позже
   использоваться ядром как индекс массива или для получения указателя для
   атаки Spectre вариант 1. Индекс или указатель является недопустимым, но
   проверки границ обходятся в ветви кода, выбранной для спекулятивного
   исполнения. Это может привести к обращению к привилегированной памяти и её
   утечке.

   Для кода ядра, в котором были выявлены места, где указатели на данные
   потенциально могут подвергаться влиянию для атак Spectre, используются новые
   макросы доступа «nospec», предотвращающие спекулятивную загрузку данных.

Spectre вариант 1 (swapgs)
~~~~~~~~~~~~~~~~~~~~~~~~~~~

   Атакующий может натренировать предсказатель переходов так, чтобы он
   спекулятивно пропускал путь swapgs для прерывания или исключения. Если он
   инициализирует регистр GS значением из пространства пользователя, то при
   спекулятивном пропуске swapgs последующие связанные с GS обращения к percpu
   в окне спекуляции будут выполняться со значением GS, контролируемым
   атакующим. Это может привести к обращению к привилегированной памяти и её
   утечке.

   Например:

   ::

     if (coming from user space)
         swapgs
     mov %gs:<percpu_offset>, %reg
     mov (%reg), %reg1

   При приходе из пространства пользователя CPU может спекулятивно пропустить
   swapgs, а затем выполнить спекулятивную загрузку percpu с использованием
   пользовательского значения GS. Таким образом пользователь может
   спекулятивно вызвать чтение любого значения ядра. Если существует гаджет,
   который использует значение percpu как адрес в другой загрузке/сохранении,
   то содержимое значения ядра может стать видимым через атаку по побочному
   каналу L1.

   Аналогичная атака существует при приходе из пространства ядра. CPU может
   спекулятивно выполнить swapgs, что приведёт к использованию
   пользовательского GS на оставшуюся часть окна спекуляции.

Spectre вариант 2
~~~~~~~~~~~~~~~~~

   Атакующий Spectre вариант 2 может :ref:`отравить <poison_btb>` буфер целей
   переходов (BTB) перед выполнением системного вызова, чтобы запустить атаку.
   После входа в ядро ядро может использовать отравленный буфер целей
   переходов при косвенном переходе и перейти к коду-гаджету при спекулятивном
   исполнении.

   Если атакующий пытается контролировать адреса памяти, утекающие во время
   спекулятивного исполнения, ему также потребуется передать гаджету параметр —
   либо через регистр, либо через известный адрес в памяти. После исполнения
   гаджета он может измерить побочный эффект.

   Ядро может защитить себя от использования отравленных записей буфера целей
   переходов, применяя возвратные трамплины (также известные как «retpoline»)
   :ref:`[3] <spec_ref3>` :ref:`[9] <spec_ref9>` для всех косвенных переходов.
   Возвратные трамплины перехватывают пути спекулятивного исполнения, чтобы
   предотвратить переход к коду-гаджету во время спекулятивного исполнения.
   CPU x86 с Enhanced Indirect Branch Restricted Speculation (Enhanced IBRS),
   доступным в оборудовании, должны использовать эту возможность для смягчения
   Spectre вариант 2 вместо retpoline. Enhanced IBRS эффективнее retpoline.

   В прошивке может находиться код-гаджет, который мог бы быть использован
   атакой Spectre вариант 2 со стороны вредоносного пользовательского процесса.
   Для смягчения таких атак на x86 возможность Indirect Branch Restricted
   Speculation (IBRS) включается перед тем, как ядро вызовет любой код прошивки.

2. Пользовательский процесс, атакующий другой пользовательский процесс
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Вредоносный пользовательский процесс может попытаться атаковать другой
   пользовательский процесс — либо через переключение контекста на том же
   аппаратном потоке, либо со стороны соседнего гиперпотока, разделяющего
   физическое ядро процессора в системе с одновременной многопоточностью (SMT).

   Атаки Spectre вариант 1, как правило, требуют передачи параметров между
   процессами, что требует наличия отношения передачи данных, такого как
   удалённые вызовы процедур (RPC). Эти параметры используются в коде-гаджете
   для получения недопустимых указателей на данные, обращающихся к
   привилегированной памяти в атакуемом процессе.

   Атаки Spectre вариант 2 могут быть запущены из вредоносного процесса путём
   :ref:`отравления <poison_btb>` буфера целей переходов. Это может влиять на
   цели косвенных переходов для процесса-жертвы, который либо запускается позже
   на том же аппаратном потоке, либо выполняется одновременно на соседнем
   аппаратном потоке, разделяющем то же физическое ядро.

   Пользовательский процесс может защитить себя от атак Spectre вариант 2,
   используя системный вызов prctl() для отключения спекуляции косвенных
   переходов для себя. Администратор также может изолировать небезопасный
   процесс от загрязнения буфера целей переходов, отключив спекуляцию косвенных
   переходов этого процесса. Это сопряжено с потерей производительности из-за
   неиспользования спекуляции косвенных переходов и очистки буфера целей
   переходов. Когда на x86 включён SMT, для процесса, у которого отключена
   спекуляция косвенных переходов, включаются Single Threaded Indirect Branch
   Predictors (STIBP) :ref:`[4] <spec_ref4>`, чтобы соседний поток не мог
   контролировать буфер целей переходов. Кроме того, выдаётся Indirect Branch
   Prediction Barrier (IBPB) для очистки буфера целей переходов при
   переключении контекста к такому процессу и от него.

   На x86 буфер стека возврата заполняется при переключении контекста. Это
   предотвращает использование буфера целей переходов для предсказания
   переходов при опустошении буфера стека возврата во время переключения к
   более глубокому стеку вызовов. Любые отравленные записи в буфере стека
   возврата, оставленные предыдущим процессом, также будут очищены.

   Пользовательские программы должны использовать рандомизацию адресного
   пространства, чтобы затруднить атаки (установить /proc/sys/kernel/randomize_va_space = 1 или 2).

3. Виртуализированный гость, атакующий хост
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Механизм атаки похож на то, как пользовательские процессы атакуют ядро.
   Вход в ядро осуществляется через гипервызовы или другие пути выхода из
   виртуализации.

   Для атак Spectre вариант 1 вредоносные гости могут передавать параметры
   (например, в регистрах) через гипервызовы для получения недопустимых
   указателей, чтобы спекулятивно проникнуть в привилегированную память после
   входа в ядро. В тех местах, где такой код ядра был выявлен, для остановки
   спекулятивного доступа к памяти используются макросы доступа nospec.

   Для атак Spectre вариант 2 вредоносные гости могут :ref:`отравить
   <poison_btb>` буфер целей переходов или буфер стека возврата, заставляя
   ядро перейти к коду-гаджету на путях спекулятивного исполнения.

   Для смягчения варианта 2 ядро хоста может использовать возвратные трамплины
   для косвенных переходов, чтобы обойти отравленный буфер целей переходов, и
   сбрасывать буфер стека возврата при выходе из VM. Это предотвращает влияние
   вредоносных гостей на косвенные переходы в ядре хоста.

   Чтобы защитить процессы хоста от вредоносных гостей, процессы хоста могут
   иметь отключённую спекуляцию косвенных переходов через prctl(). Буфер целей
   переходов очищается перед переключением контекста к таким процессам.

4. Виртуализированный гость, атакующий другого гостя
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Вредоносный гость может атаковать другого гостя, чтобы получить данные,
   доступные другому гостю.

   Атаки Spectre вариант 1 возможны, если параметры можно передавать между
   гостями. Это может осуществляться через такие механизмы, как разделяемая
   память или передача сообщений. Такие параметры могут использоваться для
   получения указателей на привилегированные данные у гостя. К привилегированным
   данным может обратиться код-гаджет на путях спекуляции жертвы.

   Атаки Spectre вариант 2 могут быть запущены из вредоносного гостя путём
   :ref:`отравления <poison_btb>` буфера целей переходов или буфера стека
   возврата. Такие отравленные записи могут использоваться для влияния на пути
   спекулятивного исполнения в госте-жертве.

   Ядро Linux смягчает атаки на других гостей, выполняющихся в том же
   аппаратном потоке CPU, сбрасывая буфер стека возврата при выходе из VM и
   очищая буфер целей переходов перед переключением к новому гостю.

   Если используется SMT, атаки Spectre вариант 2 со стороны недоверенного
   гостя в соседнем гиперпотоке могут быть смягчены администратором путём
   отключения спекуляции косвенных переходов небезопасного гостя через
   prctl(). Гость также может защитить себя, включив основанные на микрокоде
   смягчения (такие как IBPB или STIBP на x86) внутри гостя.

.. _spectre_sys_info:

Системная информация о Spectre
------------------------------

Ядро Linux предоставляет интерфейс sysfs для перечисления текущего статуса
смягчения системы по отношению к Spectre: уязвима ли система и какие смягчения
активны.

Файл sysfs, показывающий статус смягчения Spectre вариант 1:

   /sys/devices/system/cpu/vulnerabilities/spectre_v1

Возможные значения в этом файле:

  .. list-table::

     * - 'Not affected'
       - Процессор не уязвим.
     * - 'Vulnerable: __user pointer sanitization and usercopy barriers only; no swapgs barriers'
       - Защиты swapgs отключены; в остальном защита в ядре обеспечивается в
         каждом конкретном случае с помощью явной санитизации указателей и
         барьеров usercopy LFENCE.
     * - 'Mitigation: usercopy/swapgs barriers and __user pointer sanitization'
       - Защита в ядре в каждом конкретном случае с помощью явной санитизации
         указателей, барьеров usercopy LFENCE и барьеров swapgs LFENCE.

Однако защиты применяются в каждом конкретном случае, и нет гарантии, что
охвачены все возможные векторы атак для Spectre вариант 1.

Файл ядра spectre_v2 сообщает, скомпилировано ли ядро со смягчением retpoline
или есть ли у CPU аппаратное смягчение, а также поддерживает ли CPU
дополнительное смягчение, специфичное для процесса.

Этот файл также сообщает о возможностях CPU, включённых микрокодом для
смягчения атак между пользовательскими процессами:

1. Indirect Branch Prediction Barrier (IBPB) для добавления дополнительной
   изоляции между процессами разных пользователей.
2. Single Thread Indirect Branch Predictors (STIBP) для добавления
   дополнительной изоляции между потоками CPU, выполняющимися на одном ядре.

Эти возможности CPU могут влиять на производительность при использовании и
могут включаться по отдельности для каждого процесса в каждом конкретном
случае.

Файл sysfs, показывающий статус смягчения Spectre вариант 2:

   /sys/devices/system/cpu/vulnerabilities/spectre_v2

Возможные значения в этом файле:

  - Статус ядра:

  ========================================  =================================
  'Not affected'                            The processor is not vulnerable
  'Mitigation: None'                        Vulnerable, no mitigation
  'Mitigation: Retpolines'                  Use Retpoline thunks
  'Mitigation: LFENCE'                      Use LFENCE instructions
  'Mitigation: Enhanced IBRS'               Hardware-focused mitigation
  'Mitigation: Enhanced IBRS + Retpolines'  Hardware-focused + Retpolines
  'Mitigation: Enhanced IBRS + LFENCE'      Hardware-focused + LFENCE
  ========================================  =================================

  - Статус прошивки: показывает, используется ли Indirect Branch Restricted
    Speculation (IBRS) для защиты от атак Spectre вариант 2 при вызове прошивки (только x86).

  ========== =============================================================
  'IBRS_FW'  Protection against user program attacks when calling firmware
  ========== =============================================================

  - Статус Indirect branch prediction barrier (IBPB) для защиты между
    процессами разных пользователей. Этой возможностью можно управлять через
    prctl() для каждого процесса или через параметры командной строки ядра. Это
    возможность только для x86. Подробнее см. ниже.

  ===================   ========================================================
  'IBPB: disabled'      IBPB unused
  'IBPB: always-on'     Use IBPB on all tasks
  'IBPB: conditional'   Use IBPB on SECCOMP or indirect branch restricted tasks
  ===================   ========================================================

  - Статус Single threaded indirect branch prediction (STIBP) для защиты между
    разными гиперпотоками. Этой возможностью можно управлять через prctl для
    каждого процесса или через параметры командной строки ядра. Это возможность
    только для x86. Подробнее см. ниже.

  ====================  ========================================================
  'STIBP: disabled'     STIBP unused
  'STIBP: forced'       Use STIBP on all tasks
  'STIBP: conditional'  Use STIBP on SECCOMP or indirect branch restricted tasks
  ====================  ========================================================

  - Статус защиты буфера стека возврата (RSB):

  =============   ===========================================
  'RSB filling'   Protection of RSB on context switch enabled
  =============   ===========================================

  - Статус защиты EIBRS Post-barrier Return Stack Buffer (PBRSB):

  ===========================  =======================================================
  'PBRSB-eIBRS: SW sequence'   CPU is affected and protection of RSB on VMEXIT enabled
  'PBRSB-eIBRS: Vulnerable'    CPU is vulnerable
  'PBRSB-eIBRS: Not affected'  CPU is not affected by PBRSB
  ===========================  =======================================================

  - Статус защиты Branch History Injection (BHI):

.. list-table::

 * - BHI: Not affected
   - System is not affected
 * - BHI: Retpoline
   - System is protected by retpoline
 * - BHI: BHI_DIS_S
   - System is protected by BHI_DIS_S
 * - BHI: SW loop, KVM SW loop
   - System is protected by software clearing sequence
 * - BHI: Vulnerable
   - System is vulnerable to BHI
 * - BHI: Vulnerable, KVM: SW loop
   - System is vulnerable; KVM is protected by software clearing sequence

Полное смягчение может потребовать обновления микрокода от производителя CPU.
Когда необходимый микрокод недоступен, ядро сообщит об уязвимости.

Включение смягчения для Spectre вариант 1 и Spectre вариант 2
-------------------------------------------------------------

1. Смягчение в ядре
^^^^^^^^^^^^^^^^^^^

Spectre вариант 1
~~~~~~~~~~~~~~~~~

   Для Spectre вариант 1 уязвимый код ядра (определённый аудитом кода или
   инструментами сканирования) аннотируется в каждом конкретном случае для
   использования макросов доступа nospec для ограничения границ :ref:`[2]
   <spec_ref2>`, чтобы избежать любых пригодных для использования
   раскрывающих гаджетов. Однако это может не охватывать все векторы атак для
   Spectre вариант 1.

   Код копирования из пространства пользователя имеет барьер LFENCE, чтобы
   предотвратить неправильную спекуляцию проверки access_ok(). Барьер
   выполняется макросом barrier_nospec().

   Для варианта swapgs у Spectre вариант 1 барьеры LFENCE добавлены при входе
   в прерывания, исключения и NMI там, где это необходимо. Эти барьеры
   выполняются макросами FENCE_SWAPGS_KERNEL_ENTRY и FENCE_SWAPGS_USER_ENTRY.

Spectre вариант 2
~~~~~~~~~~~~~~~~~

   Для смягчения Spectre вариант 2 компилятор превращает косвенные вызовы или
   переходы в ядре в эквивалентные возвратные трамплины (retpolines)
   :ref:`[3] <spec_ref3>` :ref:`[9] <spec_ref9>` для перехода к целевым
   адресам. Пути спекулятивного исполнения под retpolines перехватываются в
   бесконечный цикл, чтобы предотвратить любой спекулятивный переход к гаджету.

   Чтобы включить смягчение retpoline на уязвимом CPU, ядро должно быть
   скомпилировано компилятором gcc, поддерживающим опции
   -mindirect-branch=thunk-extern -mindirect-branch-register. Если ядро
   компилируется компилятором Clang, компилятор должен поддерживать опцию
   -mretpoline-external-thunk. Параметр конфигурации ядра
   CONFIG_MITIGATION_RETPOLINE должен быть включён, а CPU должен работать с
   последним обновлённым микрокодом.

   В системах эпохи Intel Skylake смягчение охватывает большинство, но не все,
   случаи. Подробнее см. :ref:`[3] <spec_ref3>`.

   На CPU с аппаратным смягчением для Spectre вариант 2 (например, IBRS
   или enhanced IBRS на x86) retpoline автоматически отключается во время работы.

   Системы, поддерживающие enhanced IBRS (eIBRS), включают защиту IBRS один раз
   при загрузке, устанавливая бит IBRS, и автоматически защищены от некоторых
   вариантов атак Spectre v2. BHB всё ещё может влиять на выбор записи
   предсказателя косвенных переходов, и хотя записи предсказателя переходов
   изолированы между режимами при включённом eIBRS, сам BHB не изолирован между
   режимами. Системы, поддерживающие BHI_DIS_S, установят его для защиты от
   атак BHI.

   В системах Intel с enhanced IBRS это включает межпотоковые внедрения цели
   перехода в системах SMT (STIBP). Иными словами, Intel eIBRS включает и
   STIBP.

   AMD Automatic IBRS не защищает пространство пользователя, а системы Legacy
   IBRS очищают бит IBRS при выходе в пространство пользователя, поэтому обе
   явно включают STIBP.

   Смягчение retpoline включено по умолчанию на уязвимых CPU. Оно может быть
   принудительно включено или отключено администратором через командную строку
   ядра и управляющие файлы sysfs. См.
   :ref:`spectre_mitigation_control_command_line`.

   На x86 indirect branch restricted speculation включается по умолчанию перед
   вызовом любого кода прошивки, чтобы предотвратить эксплойты Spectre вариант 2,
   использующие прошивку.

   Использование рандомизации адресного пространства ядра (CONFIG_RANDOMIZE_BASE=y
   и CONFIG_SLAB_FREELIST_RANDOM=y в конфигурации ядра) в целом усложняет
   атаки на ядро.

2. Смягчение в пользовательских программах
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Пользовательские программы могут смягчать Spectre вариант 1, используя
   LFENCE или «ограничение границ». Подробнее см. :ref:`[2] <spec_ref2>`.

   Для смягчения Spectre вариант 2 отдельные пользовательские программы могут
   компилироваться с возвратными трамплинами для косвенных переходов. Это
   защищает их от использования отравленных записей в буфере целей переходов,
   оставленных вредоносным программным обеспечением.

   В системах legacy IBRS при возврате в пространство пользователя неявный
   STIBP отключается, поскольку ядро очищает бит IBRS. В этом случае
   пользовательские программы могут отключить спекуляцию косвенных переходов
   через prctl() (см.
   :ref:`Documentation/userspace-api/spec_ctrl.rst <set_spec_ctrl>`).
   На x86 это включит STIBP для защиты от атак со стороны соседнего потока во
   время работы пользовательской программы и будет использовать IBPB для
   сброса буфера целей переходов при переключении к программе и от неё.

   Ограничение спекуляции косвенных переходов для пользовательской программы
   также предотвратит запуск этой программой атаки варианта 2 на x86.
   Администраторы могут изменить это поведение через командную строку ядра и
   управляющие файлы sysfs.
   См. :ref:`spectre_mitigation_control_command_line`.

   Программы, отключающие свою спекуляцию косвенных переходов, будут иметь
   больше накладных расходов и работать медленнее.

   Пользовательские программы должны использовать рандомизацию адресного
   пространства (/proc/sys/kernel/randomize_va_space = 1 или 2), чтобы
   затруднить атаки.

3. Смягчение в VM
^^^^^^^^^^^^^^^^^

   Внутри ядра атаки Spectre вариант 1 со стороны вредоносных гостей
   смягчаются в каждом конкретном случае на путях выхода из VM. Уязвимый код
   использует макросы доступа nospec для «ограничения границ», чтобы избежать
   любых пригодных для использования раскрывающих гаджетов. Однако это может не
   охватывать все векторы атак варианта 1.

   Для атак Spectre вариант 2 со стороны вредоносных гостей на ядро ядро Linux
   использует retpoline или Enhanced IBRS для предотвращения использования
   отравленных записей в буфере целей переходов, оставленных вредоносными
   гостями. Оно также сбрасывает буфер стека возврата при каждом выходе из VM,
   чтобы предотвратить опустошение буфера стека возврата, при котором мог бы
   использоваться отравленный буфер целей переходов, либо чтобы атакующие гости
   не оставляли отравленных записей в буфере стека возврата.

   Для смягчения атак гость-к-гостю в том же аппаратном потоке CPU буфер целей
   переходов санируется путём сброса перед переключением к новому гостю на CPU.

   Перечисленные выше смягчения включены по умолчанию на уязвимых CPU.

   Для смягчения атак гость-к-гостю со стороны соседнего потока при
   использовании SMT недоверенный гость, выполняющийся в соседнем потоке, может
   иметь отключённую спекуляцию косвенных переходов администратором через prctl().

   Ядро также позволяет гостям использовать любое основанное на микрокоде
   смягчение по их выбору (такое как IBPB или STIBP на x86) для защиты себя.

.. _spectre_mitigation_control_command_line:

Управление смягчением через командную строку ядра
-------------------------------------------------

В общем случае ядро выбирает разумные смягчения по умолчанию для текущего CPU.

Смягчения Spectre по умолчанию могут быть отключены или изменены в командной
строке ядра с помощью следующих опций:

	- nospectre_v1
	- nospectre_v2
	- spectre_v2={option}
	- spectre_v2_user={option}
	- spectre_bhi={option}

Подробнее о доступных опциях см. Documentation/admin-guide/kernel-parameters.txt

Руководство по выбору смягчения
-------------------------------

1. Доверенное пространство пользователя
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Если все приложения пространства пользователя получены из доверенных
   источников и не исполняют поставляемый извне недоверенный код, то смягчения
   могут быть отключены.

2. Защита чувствительных программ
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Для программ с повышенными требованиями к безопасности, имеющих секреты
   (например, криптографические ключи), защита от Spectre вариант 2 может быть
   обеспечена отключением спекуляции косвенных переходов во время работы
   программы (см. :ref:`Documentation/userspace-api/spec_ctrl.rst <set_spec_ctrl>`).

3. Песочница для недоверенных программ
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Недоверенные программы, которые могут быть источником атак, можно изолировать
   путём отключения их спекуляции косвенных переходов во время их работы
   (см. :ref:`Documentation/userspace-api/spec_ctrl.rst <set_spec_ctrl>`).
   Это предотвращает загрязнение буфера целей переходов недоверенными
   программами. Это поведение можно изменить через командную строку ядра и
   управляющие файлы sysfs. См.
   :ref:`spectre_mitigation_control_command_line`.

3. Режим высокой безопасности
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Все смягчения Spectre вариант 2 могут быть принудительно включены
   во время загрузки для всех программ (см. опцию «on» в
   :ref:`spectre_mitigation_control_command_line`). Это добавит накладные
   расходы, поскольку спекуляция косвенных переходов для всех программ будет
   ограничена.

   На x86 буфер целей переходов будет сбрасываться с помощью IBPB при
   переключении к новой программе. STIBP оставляется включённым всё время для
   защиты программ от атак варианта 2, исходящих от программ, выполняющихся на
   соседних потоках.

   Как вариант, STIBP можно использовать только при выполнении программ,
   спекуляция косвенных переходов которых явно отключена, при этом IBPB
   по-прежнему используется всё время при переключении к новой программе для
   очистки буфера целей переходов (см. опцию «ibpb» в
   :ref:`spectre_mitigation_control_command_line`). Эта опция «ibpb» имеет
   меньшую стоимость по производительности, чем опция «on», которая оставляет
   STIBP включённым всё время.
