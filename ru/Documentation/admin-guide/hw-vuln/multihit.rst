iTLB multihit
=============

iTLB multihit — это аппаратная ошибка (erratum), при которой некоторые процессоры
могут получить ошибку машинного контроля (machine check error), возможно
приводящую к невосстановимому зависанию CPU, когда выборка инструкции попадает в
несколько записей в TLB инструкций. Это может произойти при изменении размера
страницы вместе с изменением физического адреса или типа кэширования.
Вредоносный гость, работающий в виртуализированной системе, может использовать
эту ошибку для проведения атаки типа «отказ в обслуживании».


Затронутые процессоры
---------------------

Вариации этой ошибки присутствуют в большинстве моделей процессоров Intel Core и
Xeon. Эта ошибка отсутствует на:

   - процессорах не-Intel

   - некоторых процессорах Atom (Airmont, Bonnell, Goldmont, GoldmontPlus, Saltwell, Silvermont)

   - процессорах Intel, у которых установлен бит PSCHANGE_MC_NO в MSR
     IA32_ARCH_CAPABILITIES.

-------------

Следующая запись CVE относится к данной проблеме:

Проблема
--------

Привилегированное программное обеспечение, включая ОС и менеджеры виртуальных
машин (VMM), отвечает за управление памятью. Ключевым компонентом управления
памятью является контроль таблиц страниц. Современные процессоры используют
виртуальную память — технику, которая создаёт для процессоров иллюзию очень
большого объёма памяти. Это виртуальное пространство разбивается на страницы
заданного размера. Таблицы страниц транслируют виртуальные адреса в физические
адреса.

Чтобы снизить задержку (latency) при выполнении трансляции виртуального адреса
в физический, процессоры включают структуру, называемую TLB, которая кэширует
недавние трансляции. Существуют отдельные TLB для инструкций (iTLB) и для данных
(dTLB).

При данной ошибке инструкции выбираются из линейного адреса, транслированного с
использованием 4-КБ трансляции, закэшированной в iTLB. Привилегированное
программное обеспечение изменяет структуру страничной организации так, что тот
же линейный адрес использует большой размер страницы (2 МБ, 4 МБ, 1 ГБ) с другим
физическим адресом или типом памяти. После изменения структуры страниц, но до
того, как программное обеспечение аннулирует какие-либо записи iTLB для данного
линейного адреса, выборка кода, происходящая по тому же линейному адресу, может
вызвать ошибку машинного контроля (machine-check error), которая может привести
к зависанию или выключению системы.


Сценарии атаки
--------------

Атаки на ошибку iTLB multihit могут быть проведены из вредоносных гостей в
виртуализированной системе.


Системная информация об iTLB multihit
--------------------------------------

Ядро Linux предоставляет интерфейс sysfs для перечисления текущего статуса iTLB
multihit системы: уязвима ли система и какие меры противодействия активны.
Соответствующий файл sysfs:

/sys/devices/system/cpu/vulnerabilities/itlb_multihit

Возможные значения в этом файле:

.. list-table::

     * - Not affected
       - Процессор не уязвим.
     * - KVM: Mitigation: Split huge pages
       - Программные изменения смягчают эту проблему.
     * - KVM: Mitigation: VMX unsupported
       - KVM не уязвим, поскольку Virtual Machine Extensions (VMX) не поддерживаются.
     * - KVM: Mitigation: VMX disabled
       - KVM не уязвим, поскольку Virtual Machine Extensions (VMX) отключены.
     * - KVM: Vulnerable
       - Процессор уязвим, но никаких мер противодействия не включено


Перечисление ошибки
--------------------------------

В MSR IA32_ARCH_CAPABILITIES был выделен новый бит (PSCHANGE_MC_NO), который
будет установлен на CPU, защищённых от этой проблемы.

   =======================================   ===========   ===============================
   IA32_ARCH_CAPABILITIES MSR                Not present   Possibly vulnerable,check model
   IA32_ARCH_CAPABILITIES[PSCHANGE_MC_NO]    '0'           Likely vulnerable,check model
   IA32_ARCH_CAPABILITIES[PSCHANGE_MC_NO]    '1'           Not vulnerable
   =======================================   ===========   ===============================


Механизм противодействия
-------------------------

Эту ошибку можно смягчить, ограничив использование больших размеров страниц
только неисполняемыми страницами. Это вынуждает все записи iTLB быть 4-КБ и
устраняет возможность множественных попаданий.

Для смягчения этой уязвимости KVM изначально помечает все огромные страницы
(huge pages) как неисполняемые. Если гость пытается выполнить код в одной из
таких страниц, страница разбивается на 4-КБ страницы, которые затем помечаются
как исполняемые.

Если EPT отключён или недоступен на хосте, KVM контролирует сбросы TLB, и
проблемная ситуация не может возникнуть. Однако механизм теневой страничной
организации EPT (shadow EPT), используемый вложенной виртуализацией, уязвим,
поскольку вложенный гость может вызвать множественные попадания iTLB, изменяя
свои собственные (невложенные) таблицы страниц. Для простоты KVM делает большие
страницы неисполняемыми во всех режимах теневой страничной организации.

Управление противодействием через командную строку ядра и KVM — параметр модуля
-------------------------------------------------------------------------------

Механизм противодействия гипервизора KVM для пометки огромных страниц как
неисполняемых можно контролировать с помощью параметра модуля «nx_huge_pages=».
Командная строка ядра позволяет управлять мерами противодействия iTLB multihit
во время загрузки с помощью опции «kvm.nx_huge_pages=».

Допустимые аргументы для этих опций:

  ==========  ================================================================
  force       Противодействие включено. В этом случае противодействие
              реализует неисполняемые огромные страницы в модуле KVM ядра
              Linux. Все огромные страницы в EPT помечаются как неисполняемые.
              Если гость пытается выполнить код в одной из таких страниц,
              страница разбивается на 4-КБ страницы, которые затем помечаются
              как исполняемые.

  off	      Противодействие отключено.

  auto        Включить противодействие, только если платформа затронута и ядро
              не было загружено с параметром командной строки «mitigations=off».
	      Это опция по умолчанию.
  ==========  ================================================================


Руководство по выбору противодействия
-------------------------------------

1. Виртуализация не используется
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Система безусловно защищается ядром, и никаких дополнительных действий не
   требуется.

2. Виртуализация с доверенными гостями
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

   Если гость происходит из доверенного источника, можно предположить, что гость
   не будет пытаться злонамеренно использовать эти ошибки, и никаких
   дополнительных действий не требуется.

3. Виртуализация с недоверенными гостями
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
   Если гость происходит из недоверенного источника, ядру хоста гостя потребуется
   применить противодействие iTLB multihit через командную строку ядра или
   параметр модуля kvm.
