Управление векторами атак
=========================

Управление векторами атак предоставляет простой способ настроить только те меры
противодействия (mitigations) уязвимостям CPU, которые актуальны с учётом
предполагаемого назначения системы. Администраторам рекомендуется определить,
какие векторы атак актуальны, и отключить все остальные, чтобы вернуть
производительность системы.

При обнаружении новых актуальных уязвимостей CPU они будут добавлены в это
управление векторами атак, поэтому администраторам, скорее всего, не потребуется
перенастраивать параметры командной строки, так как меры противодействия будут
по-прежнему корректно применяться на основе выбранного управления векторами атак.

Векторы атак
------------

В настоящее время ядро поддерживает 5 наборов мер противодействия для векторов
атак:

#. :ref:`user_kernel`
#. :ref:`user_user`
#. :ref:`guest_host`
#. :ref:`guest_guest`
#. :ref:`smt`

Для управления включёнными векторами атак см. :ref:`cmdline`.

.. _user_kernel:

Пользователь-в-ядро
^^^^^^^^^^^^^^^^^^^^

Вектор атаки «пользователь-в-ядро» подразумевает, что вредоносная программа из
пространства пользователя пытается извлечь данные ядра в пространство
пользователя, эксплуатируя уязвимость CPU. Затрагиваемые данные ядра могут быть
ограничены определённой памятью ядра либо включать всю память системы, в
зависимости от эксплуатируемой уязвимости.

Если не запускаются недоверенные приложения пространства пользователя, например
в однопользовательских системах, рассмотрите возможность отключения мер
противодействия для вектора «пользователь-в-ядро».

Обратите внимание, что в общем случае не было показано, что уязвимости CPU, для
которых Linux применяет меры противодействия, могут эксплуатироваться из
песочниц на основе браузера. Поэтому меры противодействия для вектора
«пользователь-в-ядро» актуальны главным образом в том случае, если недоверенные
пользователи могут запускать неизвестные приложения пространства пользователя.

*меры противодействия для вектора «пользователь-в-ядро» включены по умолчанию*

.. _user_user:

Пользователь-в-пользователя
^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Вектор атаки «пользователь-в-пользователя» подразумевает, что вредоносная
программа пространства пользователя пытается повлиять на поведение другой
ничего не подозревающей программы пространства пользователя с целью извлечения
данных. Уязвимость программы пространства пользователя определяется самой
программой и предоставляемыми ею интерфейсами.

Если не запускаются недоверенные приложения пространства пользователя,
рассмотрите возможность отключения мер противодействия для вектора
«пользователь-в-пользователя».

Обратите внимание, что, поскольку ядро Linux содержит отображение всей
физической памяти, для полной защиты предотвращение извлечения вредоносной
программой пространства пользователя данных другой программы пространства
пользователя требует также применения мер противодействия атакам
«пользователь-в-ядро».

*меры противодействия для вектора «пользователь-в-пользователя» включены по умолчанию*

.. _guest_host:

Гость-в-хост
^^^^^^^^^^^^

Вектор атаки «гость-в-хост» подразумевает, что вредоносная виртуальная машина
(VM) пытается извлечь данные гипервизора в VM. Затрагиваемые данные могут быть
ограничены либо потенциально включать всю память системы, в зависимости от
эксплуатируемой уязвимости.

Если не запускаются недоверенные VM, рассмотрите возможность отключения мер
противодействия для вектора «гость-в-хост».

*меры противодействия для вектора «гость-в-хост» включены по умолчанию при наличии поддержки KVM*

.. _guest_guest:

Гость-в-гость
^^^^^^^^^^^^^

Вектор атаки «гость-в-гость» подразумевает, что вредоносная VM пытается повлиять
на поведение другой ничего не подозревающей VM с целью извлечения данных.
Уязвимость VM определяется кодом внутри самой VM и предоставляемыми ею
интерфейсами.

Если не запускаются недоверенные VM или запущена только одна VM, рассмотрите
возможность отключения мер противодействия для вектора «гость-в-гость».

Аналогично вектору атаки «пользователь-в-пользователя», предотвращение
извлечения вредоносной VM данных другой VM требует также применения мер
противодействия атакам «гость-в-хост» из-за отображения физической памяти
(phys map) в ядре Linux.

*меры противодействия для вектора «гость-в-гость» включены по умолчанию при наличии поддержки KVM*

.. _smt:

Межпоточный
^^^^^^^^^^^

Межпоточный вектор атаки подразумевает, что вредоносная программа пространства
пользователя или вредоносная VM наблюдает за поведением кода, выполняющегося на
соседнем SMT-потоке, либо пытается повлиять на него с целью извлечения данных.

Многие межпоточные атаки можно нейтрализовать только при отключённом SMT, что
приведёт к уменьшению числа ядер CPU и снижению производительности.

Если межпоточные меры противодействия полностью включены ('auto,nosmt'), будут
включены все меры противодействия для межпоточных атак. SMT может быть отключён
в зависимости от того, какие уязвимости присутствуют в CPU.

Если межпоточные меры противодействия включены частично ('auto'), меры
противодействия для межпоточных атак будут включены, но SMT не будет отключён.

Если межпоточные меры противодействия отключены, ни одна мера противодействия
для межпоточных атак включена не будет.

Межпоточные меры противодействия могут не требоваться, если для предотвращения
выполнения недоверенных нагрузок на соседних SMT-потоках используется
планирование на уровне ядра (core-scheduling) или аналогичные методы.

*межпоточные меры противодействия по умолчанию включены частично*

.. _cmdline:

Управление через командную строку
----------------------------------

Векторы атак управляются через параметр командной строки mitigations=.
Передаваемое значение начинается с глобального параметра, а затем может
опционально включать один или несколько параметров для отключения различных
векторов атак.

Формат:
	| ``mitigations=[global]``
	| ``mitigations=[global],[attack vectors]``

Глобальные параметры:

============ =============================================================
Параметр     Описание
============ =============================================================
'off'        Все векторы атак отключены.
'auto'       Все векторы атак включены, частичные межпоточные меры противодействия.
'auto,nosmt' Все векторы атак включены, полные межпоточные меры противодействия.
============ =============================================================

Параметры векторов атак:

================= =======================================
Параметр          Описание
================= =======================================
'no_user_kernel'  Отключает меры для вектора «пользователь-в-ядро».
'no_user_user'    Отключает меры для вектора «пользователь-в-пользователя».
'no_guest_host'   Отключает меры для вектора «гость-в-хост».
'no_guest_guest'  Отключает меры для вектора «гость-в-гость»
'no_cross_thread' Отключает все межпоточные меры противодействия.
================= =======================================

В виде списка с разделителем-запятой можно указать несколько параметров векторов
атак. Если глобальный параметр не указан, по умолчанию используется 'auto'.
Глобальный параметр 'off' эквивалентен отключению всех векторов атак.

Примеры:
	| ``mitigations=auto,no_user_kernel``

	Включить все векторы атак, кроме «пользователь-в-ядро». Частичные межпоточные
	меры противодействия.

	| ``mitigations=auto,nosmt,no_guest_host,no_guest_guest``

	Включить все векторы атак и межпоточные меры противодействия, кроме мер для
	векторов «гость-в-хост» и «гость-в-гость».

	| ``mitigations=,no_cross_thread``

	Включить все векторы атак, но не межпоточные меры противодействия.

Взаимодействие с параметрами командной строки
----------------------------------------------

Средства управления, специфичные для конкретной уязвимости (например,
"retbleed=off"), имеют приоритет над всеми средствами управления векторами атак.
Меры противодействия отдельным уязвимостям можно включать или отключать через их
параметры командной строки независимо от управления векторами атак.

Сводка мер противодействия для векторов атак
--------------------------------------------

Когда уязвимость нейтрализуется благодаря управлению векторами атак,
используется вариант меры противодействия по умолчанию для этой конкретной
уязвимости. Чтобы использовать другую меру противодействия, воспользуйтесь
параметром командной строки, специфичным для уязвимости.

В таблице ниже сведено, какие уязвимости нейтрализуются при включении различных
векторов атак, в предположении, что CPU уязвим.

=============== ============== ============ ============= ============== ============ ========
Уязвимость      User-to-Kernel User-to-User Guest-to-Host Guest-to-Guest Cross-Thread Прим.
=============== ============== ============ ============= ============== ============ ========
BHI                   X                           X
ITS                   X                           X
GDS                   X              X            X              X            *       (Прим. 1)
L1TF                  X                           X                           *       (Прим. 2)
MDS                   X              X            X              X            *       (Прим. 2)
MMIO                  X              X            X              X            *       (Прим. 2)
Meltdown              X
Retbleed              X                           X                           *       (Прим. 3)
RFDS                  X              X            X              X
Spectre_v1            X
Spectre_v2            X                           X
Spectre_v2_user                      X                           X            *       (Прим. 1)
SRBDS                 X              X            X              X
SRSO                  X              X            X              X
SSB                                  X
TAA                   X              X            X              X            *       (Прим. 2)
TSA                   X              X            X              X
VMSCAPE                                           X
=============== ============== ============ ============= ============== ============ ========

Примечания:
   1 --  Может быть нейтрализована без отключения SMT.

   2 --  Отключает SMT, если межпоточные меры противодействия полностью включены
   и CPU уязвим

   3 --  Отключает SMT, если межпоточные меры противодействия полностью включены,
   CPU уязвим и STIBP не поддерживается

Когда вектор атаки отключён, отключаются все меры противодействия для
перечисленных в приведённой выше таблице уязвимостей, если только мера
противодействия не требуется для другого включённого вектора атаки или мера
противодействия не выбрана явно через параметр командной строки, специфичный для
уязвимости.
