Этот файл содержит документацию по файлам и каталогам sysctl
в ``/proc/sys/fs/``.

Файлы в этом каталоге могут использоваться для настройки и наблюдения за
различными и общими аспектами работы ядра Linux. Поскольку некоторые из
этих файлов *могут* быть использованы для нарушения работы вашей
системы, перед тем как вносить изменения, рекомендуется ознакомиться как
с документацией, так и с исходным кодом.

1. /proc/sys/fs
===============

В настоящее время в ``/proc/sys/fs`` могут (в зависимости от вашей
конфигурации) присутствовать следующие файлы:

.. contents:: :local:


aio-nr & aio-max-nr
-------------------

``aio-nr`` показывает текущее общесистемное число запросов асинхронного
ввода-вывода.  ``aio-max-nr`` позволяет изменить максимальное значение,
до которого может вырасти ``aio-nr``.  Если ``aio-nr`` достигает
``aio-nr-max``, то ``io_setup`` завершится с ошибкой ``EAGAIN``.  Обратите
внимание, что увеличение ``aio-max-nr`` не приводит к
предварительному выделению или изменению размера каких-либо структур данных ядра.

dentry-negative
----------------------------

Политика для отрицательных dentry. Установите в 1, чтобы всегда удалять
dentry при удалении файла, и в 0, чтобы отключить это. По умолчанию это
поведение отключено.

dentry-state
------------

Этот файл показывает значения из ``struct dentry_stat_t``, как определено в
``fs/dcache.c``::

  struct dentry_stat_t dentry_stat {
        long nr_dentry;
        long nr_unused;
        long age_limit;         /* age in seconds */
        long want_pages;        /* pages requested by system */
        long nr_negative;       /* # of unused negative dentries */
        long dummy;             /* Reserved for future use */
  };

Dentry выделяются и освобождаются динамически.

``nr_dentry`` показывает общее число выделенных dentry (активных
+ неиспользуемых). ``nr_unused shows`` число dentry, которые не
используются активно, но сохраняются в списке LRU для повторного
использования в будущем.

``age_limit`` — это возраст в секундах, по истечении которого записи
dcache могут быть высвобождены при нехватке памяти, а ``want_pages``
отличен от нуля, когда была вызвана функция ``shrink_dcache_pages()`` и
dcache ещё не очищен.

``nr_negative`` показывает число неиспользуемых dentry, которые также
являются отрицательными dentry, не отображающимися ни на какие файлы.
Вместо этого они помогают ускорить отклонение несуществующих файлов,
запрашиваемых пользователями.


file-max & file-nr
------------------

Значение в ``file-max`` обозначает максимальное число дескрипторов
файлов (file handles), которое будет выделять ядро Linux. Когда вы
получаете множество сообщений об ошибках, связанных с исчерпанием
дескрипторов файлов, возможно, стоит увеличить этот предел.

Исторически ядро могло выделять дескрипторы файлов
динамически, но не могло их снова освобождать. Три значения в
``file-nr`` обозначают число выделенных дескрипторов файлов, число
выделенных, но неиспользуемых дескрипторов файлов, и максимальное число
дескрипторов файлов. Linux 2.6 и более поздние версии всегда сообщают 0
в качестве числа свободных дескрипторов файлов — это не ошибка, это
просто означает, что число выделенных дескрипторов файлов в точности
совпадает с числом используемых дескрипторов файлов.

Попытки выделить больше файловых дескрипторов, чем ``file-max``,
сообщаются через ``printk``, ищите::

  VFS: file-max limit <number> reached

в логах ядра.


inode-nr & inode-state
----------------------

Как и в случае с дескрипторами файлов, ядро выделяет структуры inode
динамически, но пока не может их освобождать.

Файл ``inode-nr`` содержит первые два элемента из
``inode-state``, поэтому мы перейдём к этому файлу...

``inode-state`` содержит три настоящих числа и четыре заглушки (dummies).
Настоящие числа — это, в порядке их появления, ``nr_inodes``,
``nr_free_inodes`` и ``preshrink``.

``nr_inodes`` обозначает число inode, которые выделила
система.

``nr_free_inodes`` представляет число свободных inode (?), а
preshrink отличен от нуля, когда
системе требуется обрезать список inode вместо выделения
новых.


mount-max
---------

Это обозначает максимальное число монтирований, которое может
существовать в пространстве имён монтирования (mount namespace).


nr_open
-------

Это обозначает максимальное число дескрипторов файлов, которое может
выделить процесс. Значение по умолчанию — 1024*1024 (1048576), чего
должно быть достаточно для большинства машин. Фактический предел зависит
от ограничения ресурса ``RLIMIT_NOFILE``.


overflowgid & overflowuid
-------------------------

Некоторые файловые системы поддерживают только 16-битные UID и GID, хотя
в Linux UID и GID являются 32-битными. Когда одна из таких файловых
систем монтируется с разрешённой записью, любой UID или GID, превышающий
65535, перед записью на диск транслируется в фиксированное значение.

Эти sysctl позволяют изменить значение фиксированного UID и GID.
По умолчанию используется 65534.


pipe-user-pages-hard
--------------------

Максимальное суммарное число страниц, которое непривилегированный
пользователь может выделить под каналы (pipes). Как только этот предел
достигнут, новые каналы не могут быть выделены, пока использование снова
не опустится ниже предела. При значении 0 предел не применяется, и это
настройка по умолчанию.


pipe-user-pages-soft
--------------------

Максимальное суммарное число страниц, которое непривилегированный
пользователь может выделить под каналы (pipes), прежде чем размер канала
будет ограничен двумя страницами. Как только этот предел достигнут,
новые каналы для этого пользователя будут ограничены размером в две
страницы, чтобы ограничить общее использование памяти, а попытки
увеличить их с помощью ``fcntl()`` будут отклоняться, пока использование
снова не опустится ниже предела. Значение по умолчанию позволяет
выделить до 1024 каналов их размера по умолчанию. При значении 0 предел
не применяется.


protected_fifos
---------------

Цель этой защиты — избежать непреднамеренной записи в
контролируемый злоумышленником FIFO в случае, когда программа
рассчитывала создать обычный файл.

При значении «0» запись в FIFO не ограничена.

При значении «1» не разрешается открытие с ``O_CREAT`` для FIFO, которые
нам не принадлежат, в общедоступных для записи sticky-каталогах, если
только они не принадлежат владельцу каталога.

При значении «2» это также применяется к доступным для записи группе
sticky-каталогам.

Эта защита основана на ограничениях из Openwall.


protected_hardlinks
--------------------

Давний класс проблем безопасности — это основанная на жёстких ссылках
гонка вида time-of-check-time-of-use, чаще всего наблюдаемая в
общедоступных для записи каталогах вроде ``/tmp``. Распространённый
метод эксплуатации этого изъяна — пересечение границ привилегий при
переходе по заданной жёсткой ссылке (т. е. процесс root переходит по
жёсткой ссылке, созданной другим пользователем). Кроме того, в системах
без раздельных разделов это не даёт неавторизованным пользователям
«закреплять» уязвимые файлы setuid/setgid от обновления
администратором или создавать ссылки на специальные файлы.

При значении «0» поведение при создании жёстких ссылок не ограничено.

При значении «1» жёсткие ссылки не могут создаваться пользователями,
если они ещё не владеют исходным файлом или не имеют доступа к нему на
чтение/запись.

Эта защита основана на ограничениях из Openwall и grsecurity.


protected_regular
-----------------

Эта защита аналогична `protected_fifos`_, но она
предотвращает запись в контролируемый злоумышленником обычный файл в
случае, когда программа рассчитывала его создать.

При значении «0» запись в обычные файлы не ограничена.

При значении «1» не разрешается открытие с ``O_CREAT`` для обычных
файлов, которые нам не принадлежат, в общедоступных для записи
sticky-каталогах, если только они не принадлежат владельцу каталога.

При значении «2» это также применяется к доступным для записи группе
sticky-каталогам.


protected_symlinks
------------------

Давний класс проблем безопасности — это основанная на символьных
ссылках гонка вида time-of-check-time-of-use, чаще всего наблюдаемая в
общедоступных для записи каталогах вроде ``/tmp``. Распространённый
метод эксплуатации этого изъяна — пересечение границ привилегий при
переходе по заданной символьной ссылке (т. е. процесс root переходит по
символьной ссылке, принадлежащей другому пользователю). С вероятно
неполным списком из сотен примеров за прошедшие годы можно ознакомиться
здесь:
https://cve.mitre.org/cgi-bin/cvekey.cgi?keyword=/tmp

При значении «0» поведение при переходе по символьным ссылкам не
ограничено.

При значении «1» переход по символьным ссылкам разрешён только тогда,
когда они находятся вне sticky-каталога, общедоступного для записи, или
когда uid символьной ссылки и переходящего по ней совпадают, или когда
владелец каталога совпадает с владельцем символьной ссылки.

Эта защита основана на ограничениях из Openwall и grsecurity.


suid_dumpable
-------------

Это значение может использоваться для запроса и установки режима дампа
ядра (core dump) для setuid- или иным образом защищённых/испорченных
(tainted) бинарных файлов. Доступны следующие режимы

=   ==========  ===============================================================
0   (default)	Традиционное поведение. Любой процесс, который сменил
		уровень привилегий или доступен только на исполнение, не
		будет сброшен в дамп.
1   (debug)	Все процессы создают дамп ядра, когда это возможно. Дамп ядра
		принадлежит текущему пользователю, и никакая защита не
		применяется. Это предназначено только для ситуаций отладки
		системы.
		Ptrace не проверяется.
		Это небезопасно, так как позволяет обычным пользователям
		изучать содержимое памяти привилегированных процессов.
2   (suidsafe)	Любой бинарный файл, который обычно не был бы сброшен в дамп,
		всё равно сбрасывается, но только если sysctl ядра
		``core_pattern`` (см.
		:ref:`Documentation/admin-guide/sysctl/kernel.rst <core_pattern>`)
		установлен в
		значение либо обработчика канала (pipe handler), либо полностью
		определённого пути. (Подробнее об этом ограничении см.
		CVE-2006-2451.) Этот режим
		уместен, когда администраторы пытаются отладить
		проблемы в обычной среде и либо имеют обработчик канала для
		дампов ядра, который умеет аккуратно обращаться с
		привилегированными дампами ядра, либо определённый каталог для
		сбора дампов ядра.
		Если дамп ядра происходит без обработчика канала или полностью
		определённого пути, в syslog будет выдано сообщение с
		предупреждением об отсутствии корректной настройки.
=   ==========  ===============================================================

3. /proc/sys/fs/mqueue - файловая система очередей сообщений POSIX
==================================================================


Файловая система «mqueue» предоставляет необходимые возможности ядра для
обеспечения создания библиотеки пространства пользователя, реализующей
API очередей сообщений POSIX (как отмечено тегом MSG в версии POSIX
1003.1-2001 спецификации System Interfaces.)

Файловая система «mqueue» содержит значения для определения/установки
объёма ресурсов, используемых файловой системой.

``/proc/sys/fs/mqueue/queues_max`` — это файл для чтения/записи для
установки/получения максимального числа очередей сообщений, разрешённых
в системе.

``/proc/sys/fs/mqueue/msg_max`` — это файл для чтения/записи для
установки/получения значения максимального числа сообщений в очереди.  По
сути это ограничивающее значение для другого (пользовательского) предела,
который задаётся при вызове ``mq_open``.  Этот атрибут очереди должен быть
меньше или равен ``msg_max``.

``/proc/sys/fs/mqueue/msgsize_max`` — это файл для чтения/записи для
установки/получения значения максимального размера сообщения (это атрибут
каждой очереди сообщений, задаваемый во время её создания).

``/proc/sys/fs/mqueue/msg_default`` — это файл для чтения/записи для
установки/получения значения числа сообщений в очереди по умолчанию, если
параметр ``attr`` функции ``mq_open(2)`` равен ``NULL``. Если оно
превышает ``msg_max``, значение по умолчанию инициализируется значением
``msg_max``.

``/proc/sys/fs/mqueue/msgsize_default`` — это файл для чтения/записи для
установки/получения значения размера сообщения по умолчанию, если
параметр ``attr`` функции ``mq_open(2)`` равен ``NULL``. Если оно
превышает ``msgsize_max``, значение по умолчанию инициализируется
значением ``msgsize_max``.

4. /proc/sys/fs/epoll - параметры конфигурации для интерфейса epoll
===================================================================

Этот каталог содержит параметры конфигурации для интерфейса epoll(7).

max_user_watches
----------------

Каждый файловый дескриптор epoll может хранить некоторое число файлов
для наблюдения за готовностью событий. Каждый из этих наблюдаемых файлов
составляет «наблюдение» (watch).
Этот параметр конфигурации задаёт максимальное число «наблюдений»,
разрешённых для каждого пользователя.
Каждое «наблюдение» стоит примерно 90 байт на 32-битном ядре и примерно
160 байт на 64-битном.
Текущее значение по умолчанию для ``max_user_watches`` — это 4% от
доступной нижней памяти (low memory), делённые на стоимость «наблюдения»
в байтах.

5. /proc/sys/fs/fuse - параметры конфигурации для файловых систем FUSE
======================================================================

Этот каталог содержит следующие параметры конфигурации для файловых
систем FUSE:

``/proc/sys/fs/fuse/max_pages_limit`` — это файл для чтения/записи для
установки/получения максимального числа страниц, которые могут
использоваться для обслуживания запросов в FUSE.

``/proc/sys/fs/fuse/default_request_timeout`` — это файл для
чтения/записи для установки/получения тайм-аута по умолчанию (в
секундах), в течение которого сервер fuse должен ответить на
выданный ядром запрос в случае, когда сервер не указал тайм-аут при
монтировании. Если сервер задал тайм-аут,
то default_request_timeout будет проигнорирован.  Значение
«default_request_timeout» по умолчанию равно 0. 0 означает отсутствие
тайм-аута по умолчанию.
Максимальное значение, которое можно установить, равно 65535.

``/proc/sys/fs/fuse/max_request_timeout`` — это файл для чтения/записи
для установки/получения максимального тайм-аута (в секундах), в течение
которого сервер fuse должен ответить на выданный ядром запрос. Значение
больше 0 автоматически включает для сервера тайм-аут, который будет
установлен не более чем в «max_request_timeout»,
даже если сервер не указал тайм-аут, а default_request_timeout
равен 0. Если max_request_timeout больше 0 и сервер задал тайм-аут
больше max_request_timeout, или default_request_timeout установлен в
значение больше max_request_timeout, система будет использовать
max_request_timeout в качестве
тайм-аута. 0 означает отсутствие максимального тайм-аута запроса.
Максимальное значение, которое можно установить, равно 65535.

Что касается тайм-аутов: если сервер не отвечает на запрос к моменту
истечения заданного тайм-аута, то соединение с сервером fuse будет
прервано.
Обратите внимание, что тайм-ауты не на 100% точны (например, вы можете
задать 60 секунд, но тайм-аут может сработать через 70 секунд). Верхняя
граница погрешности для тайм-аута составляет примерно
FUSE_TIMEOUT_TIMER_FREQ секунд.
