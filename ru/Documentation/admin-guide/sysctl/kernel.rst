Этот файл содержит документацию по sysctl-файлам в
``/proc/sys/kernel/``.

Файлы в этом каталоге можно использовать для настройки и мониторинга
различных общих аспектов работы ядра Linux. Поскольку некоторые из этих
файлов *могут* быть использованы для нарушения работы вашей системы,
рекомендуется ознакомиться как с документацией, так и с исходным кодом,
прежде чем вносить изменения.

В настоящее время эти файлы могут (в зависимости от вашей конфигурации)
присутствовать в ``/proc/sys/kernel``:

.. contents:: :local:


acct
====

::

    highwater lowwater frequency

Если включён учёт процессов в стиле BSD, эти значения управляют его
поведением. Если объём свободного места в файловой системе, где находится
журнал, опускается ниже ``lowwater``\ %, учёт приостанавливается. Если
объём свободного места поднимается выше ``highwater``\ %, учёт возобновляется.
``frequency`` определяет, как часто мы проверяем количество свободного места
(значение задаётся в секундах). По умолчанию:

::

    4 2 30

То есть приостановить учёт, если объём свободного места падает ниже 2%;
возобновить его, если он увеличивается как минимум до 4%; считать информацию
об объёме свободного места действительной в течение 30 секунд.


acpi_video_flags
================

См. Documentation/power/video.rst. Это позволяет задать режим возобновления
видео аналогично параметру ядра ``acpi_sleep``, комбинируя следующие значения:

= =======
1 s3_bios
2 s3_mode
4 s3_beep
= =======

arch
====

Имя аппаратного обеспечения машины, тот же вывод, что и у ``uname -m``
(например, ``x86_64`` или ``aarch64``).

auto_msgmni
===========

Эта переменная не имеет эффекта и может быть удалена в будущих выпусках
ядра. Её чтение всегда возвращает 0.
Вплоть до Linux 3.17 она включала/отключала автоматический пересчёт
`msgmni`_
при добавлении/удалении памяти или при создании/удалении пространства имён IPC.
Запись «1» в этот файл включала автоматический пересчёт msgmni.
Запись «0» отключала его. Значение по умолчанию было равно 1.


bootloader_type (только x86)
============================

Здесь указывается числовой код типа загрузчика, сообщаемый загрузчиком,
сдвинутый влево на 4 разряда и объединённый по OR с младшими четырьмя битами
версии загрузчика.  Причина такого кодирования в том, что оно раньше
соответствовало полю ``type_of_loader`` в заголовке ядра; кодирование
сохранено для обратной совместимости.  То есть если полный числовой код типа
загрузчика равен 0x15, а полный номер версии равен 0x234, то этот файл будет
содержать значение 340 = 0x154.

Дополнительную информацию см. в полях ``type_of_loader`` и ``ext_loader_type``
в Documentation/arch/x86/boot.rst.


bootloader_version (только x86)
===============================

Полный номер версии загрузчика.  В приведённом выше примере этот файл будет
содержать значение 564 = 0x234.

Дополнительную информацию см. в полях ``type_of_loader`` и ``ext_loader_ver``
в Documentation/arch/x86/boot.rst.


bpf_stats_enabled
=================

Управляет тем, должно ли ядро собирать статистику по BPF-программам
(общее время выполнения, количество запусков...). Включение
статистики приводит к незначительному снижению производительности при каждом
запуске программы. Статистику можно просмотреть с помощью ``bpftool``.

= ===================================
0 Не собирать статистику (по умолчанию).
1 Собирать статистику.
= ===================================


cad_pid
=======

Это pid, которому будет послан сигнал при перезагрузке (в частности, по
Ctrl-Alt-Delete). Запись в этот файл значения, которое не
соответствует выполняющемуся процессу, приведёт к ``-ESRCH``.

См. также `ctrl-alt-del`_.


cap_last_cap
============

Наивысшая допустимая capability выполняющегося ядра.  Экспортирует
``CAP_LAST_CAP`` из ядра.


.. _core_pattern:

core_pattern
============

``core_pattern`` используется для задания имени-шаблона файла core dump.

* максимальная длина 127 символов; значение по умолчанию — «core»
* ``core_pattern`` используется как шаблон-образец для имени выходного
  файла; некоторые строковые шаблоны (начинающиеся с «%») заменяются
  их фактическими значениями.
* обратная совместимость с ``core_uses_pid``:

	Если ``core_pattern`` не содержит «%p» (по умолчанию не содержит)
	и установлен ``core_uses_pid``, то к имени файла будет добавлено .PID.

* спецификаторы формата corename

	========	==========================================
	%<NUL>		«%» отбрасывается
	%%		вывести один «%»
	%p		pid
	%P		глобальный pid (начальное пространство имён PID)
	%i		tid
	%I		глобальный tid (начальное пространство имён PID)
	%u		uid (в начальном пространстве имён пользователей)
	%g		gid (в начальном пространстве имён пользователей)
	%d		режим дампа, соответствует ``PR_SET_DUMPABLE`` и
			``/proc/sys/fs/suid_dumpable``
	%s		номер сигнала
	%t		UNIX-время дампа
	%h		имя хоста
	%e		имя исполняемого файла (может быть укорочено, может быть изменено через prctl и т.п.)
	%f      	имя исполняемого файла
	%E		путь к исполняемому файлу
	%c		максимальный размер файла core согласно лимиту ресурсов RLIMIT_CORE
	%C		CPU, на котором выполнялась задача
	%F		номер pidfd
	%<OTHER>	оба отбрасываются
	========	==========================================

* Если первый символ шаблона — «|», ядро будет трактовать
  остаток шаблона как команду для запуска.  Дамп ядра будет
  записан на стандартный ввод этой программы, а не в файл.


core_pipe_limit
===============

Этот sysctl применим только тогда, когда `core_pattern`_ настроен на
передачу файлов core по конвейеру вспомогательной программе в пространстве
пользователя (когда первый символ ``core_pattern`` — «|», см. выше).
При сборе дампов через конвейер в приложение иногда
бывает полезно, чтобы собирающее приложение получало данные о
аварийно завершающемся процессе из его каталога ``/proc/pid``.
Чтобы делать это безопасно, ядро должно дождаться завершения собирающего
процесса, чтобы не удалить proc-файлы аварийно завершающихся процессов
преждевременно.
Это в свою очередь создаёт возможность того, что некорректно работающий
собирающий процесс в пространстве пользователя может заблокировать сбор
(reaping) аварийно завершившегося процесса просто никогда не завершаясь.
Этот sysctl защищает от такого.
Он определяет, сколько одновременно аварийно завершающихся процессов могут
быть параллельно переданы по конвейеру приложениям в пространстве пользователя.
Если это значение превышено, то аварийно завершающиеся процессы сверх этого
значения отмечаются в журнале ядра, а их дампы пропускаются.
0 — особое значение, указывающее, что параллельно может захватываться
неограниченное число процессов, но при этом ожидание не выполняется (т.е.
собирающему процессу не гарантируется доступ к ``/proc/<crashing
pid>/``).
По умолчанию это значение равно 0.


core_sort_vma
=============

По умолчанию coredump записывает VMA в порядке адресов. Установив
``core_sort_vma`` в 1, VMA будут записываться от наименьшего размера
к наибольшему. Известно, что это ломает как минимум elfutils, но
может быть удобно при работе с очень большими (и усечёнными)
дампами ядра, где более полезные детали для отладки содержатся
в меньших VMA.


core_uses_pid
=============

По умолчанию имя файла coredump — «core».  Установив
``core_uses_pid`` в 1, имя файла coredump становится core.PID.
Если `core_pattern`_ не содержит «%p» (по умолчанию не содержит)
и установлен ``core_uses_pid``, то к имени файла будет добавлено .PID.


ctrl-alt-del
============

Когда значение в этом файле равно 0, ctrl-alt-del перехватывается и
передаётся программе ``init(1)`` для выполнения корректного перезапуска.
Однако когда значение > 0, реакцией Linux на «вулканский нервный
зажим» (Vulcan Nerve Pinch (tm)) будет немедленная перезагрузка, даже без
синхронизации «грязных» буферов.

Примечание:
  когда программа (например, dosemu) держит клавиатуру в «сыром» (raw)
  режиме, ctrl-alt-del перехватывается программой до того, как он
  вообще достигнет слоя tty ядра, и программа сама решает,
  что с ним делать.


dmesg_restrict
==============

Этот переключатель указывает, запрещено ли непривилегированным пользователям
использовать ``dmesg(8)`` для просмотра сообщений из буфера журнала ядра.
Когда ``dmesg_restrict`` установлен в 0, ограничений нет.
Когда ``dmesg_restrict`` установлен в 1, пользователи должны иметь
``CAP_SYSLOG``, чтобы использовать ``dmesg(8)``.

Параметр конфигурации ядра ``CONFIG_SECURITY_DMESG_RESTRICT`` задаёт
значение ``dmesg_restrict`` по умолчанию.


domainname & hostname
=====================

Эти файлы можно использовать для задания доменного имени NIS/YP и
имени хоста вашей машины точно так же, как командами
domainname и hostname, т.е.::

	# echo "darkstar" > /proc/sys/kernel/hostname
	# echo "mydomain" > /proc/sys/kernel/domainname

имеет такой же эффект, как::

	# hostname "darkstar"
	# domainname "mydomain"

Учтите, однако, что классический darkstar.frop.org имеет
имя хоста «darkstar» и доменное имя DNS (Internet Domain Name Server)
«frop.org», что не следует путать с доменным именем NIS (Network
Information Service) или YP (Yellow Pages). Эти два
доменных имени в общем случае различны. Подробное обсуждение
см. в man-странице ``hostname(1)``.


firmware_config
===============

См. Documentation/driver-api/firmware/fallback-mechanisms.rst.

Записи в этом каталоге позволяют управлять резервным механизмом
вспомогательного загрузчика прошивок (firmware loader helper fallback):

* ``force_sysfs_fallback``, при установке в 1, принудительно использует
  резервный механизм;
* ``ignore_sysfs_fallback``, при установке в 1, игнорирует любой резервный механизм.


ftrace_dump_on_oops
===================

Определяет, должна ли вызываться ``ftrace_dump()`` при oops (или
панике ядра). Это выведет содержимое буферов ftrace на
консоль.  Это очень полезно для захвата трасс, ведущих к
сбоям, и вывода их на последовательную консоль.

======================= ===========================================
0                       Отключено (по умолчанию).
1                       Сбросить буферы всех CPU.
2(orig_cpu)             Сбросить буфер того CPU, который вызвал
                        oops.
<instance>              Сбросить буфер указанного экземпляра на всех CPU.
<instance>=2(orig_cpu)  Сбросить буфер указанного экземпляра на том CPU,
                        который вызвал oops.
======================= ===========================================

Поддерживается также сброс нескольких экземпляров, при этом экземпляры
разделяются запятыми. Если необходимо также сбросить глобальный буфер,
пожалуйста, укажите режим сброса (1/2/orig_cpu) для глобального буфера первым.

Так, например, чтобы сбросить буферы экземпляров «foo» и «bar» на всех CPU,
пользователь может выполнить::

  echo "foo,bar" > /proc/sys/kernel/ftrace_dump_on_oops

Чтобы сбросить глобальный буфер и буфер экземпляра «foo» на всех
CPU вместе с буфером экземпляра «bar» на том CPU, который вызвал
oops, пользователь может выполнить::

  echo "1,foo,bar=2" > /proc/sys/kernel/ftrace_dump_on_oops

hardlockup_all_cpu_backtrace
============================

Это значение управляет поведением детектора аппаратных зависаний (hard lockup)
при обнаружении состояния аппаратного зависания — собирать ли дополнительную
отладочную информацию. Если включено, будет инициирован специфичный для
архитектуры сброс стека по всем CPU.

= ============================================
0 Ничего не делать. Это поведение по умолчанию.
1 При обнаружении захватывать больше отладочной информации.
= ============================================


hardlockup_panic
================

Этот параметр можно использовать для управления тем, паникует ли ядро
при обнаружении аппаратного зависания (hard lockup).

= ===========================
0 Не паниковать при hard lockup.
1 Паниковать при hard lockup.
= ===========================

Дополнительную информацию см. в Documentation/admin-guide/lockup-watchdogs.rst.
Это также можно задать с помощью параметра ядра nmi_watchdog.


hotplug
=======

Путь к агенту политики горячего подключения (hotplug).
Значение по умолчанию — ``CONFIG_UEVENT_HELPER_PATH``, который, в свою очередь,
по умолчанию равен пустой строке.

Этот файл существует только тогда, когда включён ``CONFIG_UEVENT_HELPER``. Большинство
современных систем полагаются исключительно на основанный на netlink источник
uevent и не нуждаются в этом.


hung_task_all_cpu_backtrace
===========================

Если этот параметр установлен, ядро будет посылать NMI всем CPU, чтобы сбросить
их трассы стека при обнаружении зависшей задачи. Этот файл появляется, если
включены CONFIG_DETECT_HUNG_TASK и CONFIG_SMP.

0: Не показывать трассы стека всех CPU при обнаружении зависшей задачи.
Это поведение по умолчанию.

1: Немаскируемо прервать все CPU и сбросить их трассы стека при обнаружении
зависшей задачи.


hung_task_panic
===============

При установке в ненулевое значение будет вызвана паника ядра, если
число зависших задач, обнаруженных за один проход сканирования, достигает этого значения.
Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.

= =======================================================
0 Продолжить работу. Это поведение по умолчанию.
N Паниковать, когда за один проход сканирования обнаружено N зависших задач.
= =======================================================


hung_task_check_count
=====================

Верхняя граница числа задач, которые проверяются.
Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.


hung_task_detect_count
======================

Указывает общее число задач, которые были обнаружены как зависшие с момента
загрузки системы.

Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.

hung_task_sys_info
==================
Список дополнительной системной информации, разделённый запятыми, которая
сбрасывается при обнаружении зависшей задачи, например, «tasks,mem,timers,locks,...».
Подробнее см. раздел 'panic_sys_info' ниже.

hung_task_timeout_secs
======================

Когда задача в состоянии D не получала планирования
дольше этого значения, выдать предупреждение.
Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.

0 означает бесконечный тайм-аут, проверка не выполняется.

Возможные устанавливаемые значения находятся в диапазоне {0:``LONG_MAX``/``HZ``}.


hung_task_check_interval_secs
=============================

Интервал проверки зависших задач. Если проверка зависших задач включена
(см. `hung_task_timeout_secs`_), проверка выполняется каждые
``hung_task_check_interval_secs`` секунд.
Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.

0 (по умолчанию) означает использовать ``hung_task_timeout_secs`` как интервал
проверки.

Возможные устанавливаемые значения находятся в диапазоне {0:``LONG_MAX``/``HZ``}.


hung_task_warnings
==================

Максимальное число предупреждений для выдачи. В течение интервала проверки,
если обнаруживается зависшая задача, это значение уменьшается на 1.
Когда это значение достигает 0, предупреждения больше выдаваться не будут.
Этот файл появляется, если включён ``CONFIG_DETECT_HUNG_TASK``.

-1: выдавать бесконечное число предупреждений.

ignore-unaligned-usertrap
=========================

На архитектурах, где невыровненные обращения вызывают ловушки (traps), и где
эта возможность поддерживается (``CONFIG_SYSCTL_ARCH_UNALIGN_NO_WARN``;
в настоящее время — ``arc``, ``parisc`` и ``loongarch``), управляет тем,
журналируются ли все невыровненные ловушки.

= =============================================================
0 Журналировать все невыровненные обращения.
1 Предупреждать только при первой ловушке процесса. Это настройка
  по умолчанию.
= =============================================================

См. также `unaligned-trap`_.

io_uring_disabled
=================

Запрещает всем процессам создавать новые экземпляры io_uring. Включение этого
сокращает поверхность атаки ядра.

= ======================================================================
0 Все процессы могут создавать экземпляры io_uring как обычно. Это
  настройка по умолчанию.
1 Создание io_uring отключено (io_uring_setup() завершится с ошибкой
  -EPERM) для непривилегированных процессов, не входящих в группу io_uring_group.
  Существующие экземпляры io_uring по-прежнему могут использоваться.  См.
  документацию по io_uring_group для дополнительной информации.
2 Создание io_uring отключено для всех процессов. io_uring_setup()
  всегда завершается с ошибкой -EPERM. Существующие экземпляры io_uring по-прежнему
  могут использоваться.
= ======================================================================


io_uring_group
==============

Когда io_uring_disabled установлен в 1, процесс должен быть либо
привилегированным (CAP_SYS_ADMIN), либо входить в группу io_uring_group, чтобы
создать экземпляр io_uring.  Если io_uring_group установлен в -1 (по
умолчанию), только процессы с capability CAP_SYS_ADMIN могут создавать
экземпляры io_uring.


kernel_sys_info
===============
Список дополнительной системной информации, разделённый запятыми, которая
сбрасывается при обнаружении мягкого/аппаратного зависания (soft/hard lockup),
например, «tasks,mem,timers,locks,...».
Подробнее см. раздел 'panic_sys_info' ниже.

Он служит управляющим параметром ядра по умолчанию, который вступает в силу,
когда модуль ядра вызывает sys_info() с parameter==0.

kexec_load_disabled
===================

Переключатель, указывающий, отключены ли системные вызовы ``kexec_load`` и
``kexec_file_load``.
По умолчанию это значение равно 0 (ложь: ``kexec_*load`` включён), но может быть
установлено в 1 (истина: ``kexec_*load`` отключён).
После установки в истину kexec больше нельзя использовать, и переключатель
нельзя установить обратно в ложь.
Это позволяет загрузить образ kexec перед отключением системного вызова,
позволяя системе настроить (и позже использовать) образ без его
изменения.
Обычно используется вместе с sysctl `modules_disabled`_.

kexec_load_limit_panic
======================

Этот параметр задаёт лимит на число раз, которое системные вызовы
``kexec_load`` и ``kexec_file_load`` могут быть вызваны с образом для
аварийного дампа (crash image). Его можно установить только в более
ограничительное значение, чем текущее.

== ======================================================
-1 Неограниченное число вызовов kexec. Это настройка по умолчанию.
N  Число оставшихся вызовов.
== ======================================================

kexec_load_limit_reboot
=======================

Аналогичная функциональность, как у ``kexec_load_limit_panic``, но для обычного
образа.

kptr_restrict
=============

Этот переключатель указывает, накладываются ли ограничения на
раскрытие адресов ядра через ``/proc`` и другие интерфейсы.

Когда ``kptr_restrict`` установлен в 0 (по умолчанию), адрес хешируется
перед печатью.
(Это эквивалентно %p.)

Когда ``kptr_restrict`` установлен в 1, указатели ядра, печатаемые с
помощью спецификатора формата %pK, будут заменены нулями, если только
пользователь не имеет ``CAP_SYSLOG`` и эффективные идентификаторы пользователя
и группы не равны реальным идентификаторам.
Это потому, что проверки %pK выполняются во время read(), а не open(),
поэтому если права повышены между open() и read()
(например, через setuid-бинарник), то %pK не будет раскрывать указатели ядра
непривилегированным пользователям.
Учтите, это лишь временное решение.
Правильное долгосрочное решение — выполнять проверки прав во время
open().
Рассмотрите возможность убрать права чтения для всех (world read) у файлов,
использующих %pK, и использовать `dmesg_restrict`_ для защиты от использования %pK в ``dmesg(8)``,
если раскрытие значений указателей ядра непривилегированным пользователям вызывает беспокойство.

Когда ``kptr_restrict`` установлен в 2, указатели ядра, печатаемые с помощью
%pK, будут заменены нулями независимо от привилегий.

Для отключения этих ограничений безопасности на раннем этапе загрузки (и раз
и навсегда) используйте вместо этого параметр загрузки ``hash_pointers``.

сбрасывается при обнаружении мягкого/аппаратного зависания (soft/hard lockup),

modprobe
========

Полный путь к вспомогательной программе в режиме пользователя для
автозагрузки модулей ядра, по умолчанию ``CONFIG_MODPROBE_PATH``, который,
в свою очередь, по умолчанию равен
«/sbin/modprobe».  Этот бинарник выполняется, когда ядро запрашивает
модуль.  Например, если пространство пользователя передаёт неизвестный тип
файловой системы в mount(), то ядро автоматически запросит
соответствующий модуль файловой системы, выполнив эту вспомогательную программу.
Эта вспомогательная программа должна вставить нужный модуль в ядро.

Этот sysctl влияет только на автозагрузку модулей.  Он не влияет на
возможность явной вставки модулей.

Этот sysctl можно использовать для отладки запросов на загрузку модулей::

    echo '#! /bin/sh' > /tmp/modprobe
    echo 'echo "$@" >> /tmp/modprobe.log' >> /tmp/modprobe
    echo 'exec /sbin/modprobe "$@"' >> /tmp/modprobe
    chmod a+x /tmp/modprobe
    echo /tmp/modprobe > /proc/sys/kernel/modprobe

Как вариант, если этот sysctl установлен в пустую строку, то автозагрузка
модулей полностью отключается.  Ядро вообще не будет пытаться
выполнить вспомогательную программу в режиме пользователя, а также не будет
вызывать LSM-хук kernel_module_request.

Если в конфигурации ядра установлено CONFIG_STATIC_USERMODEHELPER=y,
то настроенная статическая вспомогательная программа в режиме пользователя
переопределяет этот sysctl,
за исключением того, что пустая строка всё же принимается для полного отключения
автозагрузки модулей, как описано выше.

modules_disabled
================

Переключаемое значение, указывающее, разрешена ли загрузка модулей
в ином случае модульном ядре.  По умолчанию этот переключатель выключен
(0), но может быть установлен в истину (1).  После установки в истину модули
нельзя ни загрузить, ни выгрузить, и переключатель нельзя установить обратно
в ложь.  Обычно используется вместе с переключателем `kexec_load_disabled`_.


.. _msgmni:

msgmax, msgmnb, and msgmni
==========================

``msgmax`` — это максимальный размер IPC-сообщения в байтах. 8192 по
умолчанию (``MSGMAX``).

``msgmnb`` — это максимальный размер IPC-очереди в байтах. 16384 по
умолчанию (``MSGMNB``).

``msgmni`` — это максимальное число IPC-очередей. 32000 по умолчанию
(``MSGMNI``).

Все эти параметры задаются для каждого пространства имён ipc. Максимальное число байт
в POSIX-очередях сообщений ограничено ``RLIMIT_MSGQUEUE``. Этот лимит
соблюдается иерархически в каждом пространстве имён пользователей.

msg_next_id, sem_next_id, and shm_next_id (System V IPC)
========================================================

Эти три переключателя позволяют задать желаемый id для следующего выделяемого
IPC-объекта: сообщения, семафора или разделяемой памяти соответственно.

По умолчанию они равны -1, что означает обобщённую логику выделения.
Возможные устанавливаемые значения находятся в диапазоне {0:``INT_MAX``}.

Примечания:
  1) ядро не гарантирует, что новый объект получит желаемый id. Так что
     пространство пользователя само решает, как обрабатывать объект с «неверным» id.
  2) Переключатель с не-умолчательным значением будет установлен ядром обратно в -1 после
     успешного выделения IPC-объекта. Если системный вызов выделения IPC-объекта
     завершается с ошибкой, не определено, остаётся ли значение неизменным или сбрасывается в -1.


ngroups_max
===========

Максимальное число дополнительных групп, _т.е._ максимальный размер, который
``setgroups`` будет принимать. Экспортирует ``NGROUPS_MAX`` из ядра.



nmi_watchdog
============

Этот параметр можно использовать для управления NMI-сторожем (NMI watchdog)
(т.е. детектором аппаратных зависаний) на системах x86.

= =================================
0 Отключить детектор аппаратных зависаний.
1 Включить детектор аппаратных зависаний.
= =================================

Детектор аппаратных зависаний контролирует каждый CPU на его способность отвечать на
прерывания таймера. Механизм использует регистры счётчиков производительности CPU,
которые программируются на периодическую генерацию немаскируемых прерываний (NMI)
пока CPU занят. Отсюда альтернативное название «NMI watchdog».

NMI watchdog по умолчанию отключён, если ядро выполняется в качестве гостя
в виртуальной машине KVM. Это умолчание можно переопределить, добавив::

   nmi_watchdog=1

в командную строку гостевого ядра (см.
Documentation/admin-guide/kernel-parameters.rst).


nmi_wd_lpm_factor (только PPC)
==============================

Коэффициент, применяемый к тайм-ауту NMI watchdog (только когда ``nmi_watchdog``
установлен в 1). Этот коэффициент представляет процент, добавляемый к
``watchdog_thresh`` при вычислении тайм-аута NMI watchdog во время
LPM. Тайм-аут мягких зависаний (soft lockup) не затрагивается.

Значение 0 означает отсутствие изменений. Значение по умолчанию — 200, что означает, что NMI
watchdog установлен в 30с (при ``watchdog_thresh`` равном 10).


numa_balancing
==============

Включает/отключает и настраивает автоматическую балансировку памяти NUMA на
основе page fault.  Память автоматически перемещается к узлам, которые часто к ней обращаются.
Устанавливаемое значение может быть результатом операции OR следующих:

= =================================
0 NUMA_BALANCING_DISABLED
1 NUMA_BALANCING_NORMAL
2 NUMA_BALANCING_MEMORY_TIERING
= =================================

Либо NUMA_BALANCING_NORMAL для оптимизации размещения страниц среди разных
NUMA-узлов с целью уменьшения удалённых обращений.  На NUMA-машинах есть
штраф к производительности, если CPU обращается к удалённой памяти. Когда эта
возможность включена, ядро выбирает (сэмплирует), какой поток задачи обращается к
памяти, периодически снимая отображение (unmapping) страниц и затем перехватывая page
fault. В момент page fault определяется, должны ли данные,
к которым происходит обращение, быть мигрированы на локальный узел памяти.

Снятие отображения страниц и перехват ошибок влекут дополнительные накладные расходы, которые
в идеале компенсируются улучшенной локальностью памяти, но универсальной
гарантии нет. Если целевая нагрузка уже привязана к NUMA-узлам, то эту
возможность следует отключить.

Либо NUMA_BALANCING_MEMORY_TIERING для оптимизации размещения страниц среди
разных типов памяти (представленных как разные NUMA-узлы), чтобы
разместить горячие страницы в быстрой памяти.  Это реализовано тоже на основе
снятия отображения и page fault.

numa_balancing_promote_rate_limit_MBps
======================================

Слишком высокая пропускная способность повышения/понижения (promotion/demotion) между разными типами памяти
может ухудшить задержку приложения.  Это можно использовать для ограничения скорости
пропускной способности повышения.  Максимальная пропускная способность повышения на узел в МБ/с
будет ограничена так, чтобы быть не более установленного значения.

Эмпирическое правило — установить это в значение, меньшее 1/10 от пропускной способности
записи узла PMEM.

oops_all_cpu_backtrace
======================

Если этот параметр установлен, ядро будет посылать NMI всем CPU, чтобы сбросить
их трассы стека при возникновении события oops. Это следует использовать как крайнюю
меру в случае, если панику нельзя вызвать (чтобы защитить, например, работающие
ВМ) или kdump не может быть собран. Этот файл появляется, если включён CONFIG_SMP.

0: Не показывать трассы стека всех CPU при обнаружении oops.
Это поведение по умолчанию.

1: Немаскируемо прервать все CPU и сбросить их трассы стека при обнаружении
события oops.


oops_limit
==========

Число oops ядра, после которого ядро должно паниковать, когда
``panic_on_oops`` не установлен. Установка этого в 0 отключает проверку
счётчика. Установка этого в 1 имеет тот же эффект, что и установка
``panic_on_oops=1``. Значение по умолчанию — 10000.


osrelease, ostype & version
===========================

::

  # cat osrelease
  2.1.88
  # cat ostype
  Linux
  # cat version
  #5 Wed Feb 25 21:49:24 MET 1998

Файлы ``osrelease`` и ``ostype`` должны быть достаточно понятны.
``version``
требует, однако, небольшого пояснения. «#5» означает, что
это пятое ядро, собранное из этой исходной базы, а
дата за ним указывает время сборки ядра.
Единственный способ настроить эти значения — пересобрать ядро :-)


overflowgid & overflowuid
=========================

если ваша архитектура не всегда поддерживала 32-битные UID (т.е. arm,
i386, m68k, sh и sparc32), фиксированные UID и GID будут возвращаться
приложениям, использующим старые 16-битные системные вызовы UID/GID, если
фактический UID или GID превысил бы 65535.

Эти sysctl позволяют изменить значение фиксированных UID и GID.
По умолчанию — 65534.


panic
=====

Значение в этом файле определяет поведение ядра при
панике:

* если ноль, ядро будет бесконечно зацикливаться;
* если отрицательное, ядро перезагрузится немедленно;
* если положительное, ядро перезагрузится через соответствующее число
  секунд.

Когда вы используете программный watchdog, рекомендуемое значение — 60.


panic_on_io_nmi
===============

Управляет поведением ядра, когда CPU получает NMI, вызванное
ошибкой ввода-вывода.

= ==================================================================
0 Попытаться продолжить работу (по умолчанию).
1 Паниковать немедленно. Ошибка ввода-вывода вызвала NMI. Это указывает на
  серьёзное состояние системы, которое может привести к повреждению данных ввода-вывода.
  Вместо продолжения работы паника может быть лучшим выбором. Некоторые
  серверы выдают такого рода NMI при нажатии кнопки дампа,
  и вы можете использовать этот параметр для снятия аварийного дампа.
= ==================================================================

Управляет поведением ядра при возникновении oops или BUG.

panic_on_stackoverflow

Управляет поведением ядра при обнаружении переполнений

Этот файл появляется, если включён ``CONFIG_DEBUG_STACKOVERFLOW``.

panic_on_unrecovered_nmi
========================

Поведение Linux по умолчанию при NMI, связанном с памятью или неизвестном,
заключается в продолжении работы. Для многих окружений, таких как научные
вычисления, предпочтительнее, чтобы машину вывели из эксплуатации и ошибку
устранили, чем чтобы неисправленная ошибка чётности/ECC распространилась дальше.

Небольшое число систем действительно генерирует NMI по странным случайным причинам,
таким как управление питанием, поэтому по умолчанию это выключено. Этот sysctl работает как
существующие в том каталоге средства управления паникой.


panic_on_warn
=============

Вызывает panic() в пути WARN() при установке в 1.  Это полезно для того, чтобы избежать
пересборки ядра при попытке снять kdump в месте WARN().

= ================================================
0 Только WARN(), поведение по умолчанию.
1 Вызвать panic() после печати местоположения WARN().
= ================================================


panic_print
===========

Битовая маска для печати системной информации при возникновении паники. Пользователь может выбрать
комбинацию следующих битов:

=====  ============================================
бит 0  печатать информацию о всех задачах
бит 1  печатать информацию о системной памяти
бит 2  печатать информацию о таймерах
бит 3  печатать информацию о блокировках, если включён ``CONFIG_LOCKDEP``
бит 4  печатать буфер ftrace
бит 5  воспроизвести все сообщения ядра на консолях в конце паники
бит 6  печатать трассы стека всех CPU (если доступно в архитектуре)
бит 7  печатать только задачи в непрерываемом (заблокированном) состоянии
=====  ============================================

Так, например, чтобы напечатать информацию о задачах и памяти при панике, пользователь может::

  echo 3 > /proc/sys/kernel/panic_print


panic_sys_info
==============

Список дополнительной информации, разделённый запятыми, которая сбрасывается при панике,
например, «tasks,mem,timers,...».  Это удобочитаемая для человека альтернатива
'panic_print'. Возможные значения:

=============   ===================================================
tasks           печатать информацию о всех задачах
mem             печатать информацию о системной памяти
timers          печатать информацию о таймерах
locks           печатать информацию о блокировках, если включён CONFIG_LOCKDEP
ftrace          печатать буфер ftrace
all_bt          печатать трассы стека всех CPU (если доступно в архитектуре)
blocked_tasks   печатать только задачи в непрерываемом (заблокированном) состоянии
=============   ===================================================


panic_on_rcu_stall
==================

При установке в 1 вызывает panic() после сообщений об обнаружении застоя RCU (RCU stall). Это
полезно для определения первопричины застоев RCU с помощью vmcore.

= ============================================================
0 Не вызывать panic() при возникновении застоя RCU, поведение по умолчанию.
1 panic() после печати сообщений о застое RCU.
= ============================================================

max_rcu_stall_to_panic
======================

Когда ``panic_on_rcu_stall`` установлен в 1, это значение определяет
число раз, которое RCU может застаиваться до вызова panic().

Когда ``panic_on_rcu_stall`` установлен в 0, это значение не имеет эффекта.

perf_cpu_time_max_percent
=========================

Подсказывает ядру, сколько процессорного времени ему разрешено
использовать на обработку событий сэмплирования perf.  Если подсистеме perf
сообщается, что её сэмплы превышают этот лимит, она
снизит частоту своего сэмплирования, чтобы попытаться сократить своё
потребление CPU.

Часть сэмплирования perf происходит в NMI.  Если выполнение этих сэмплов
неожиданно занимает слишком много времени, NMI могут наслаиваться
друг на друга настолько, что ничему другому не будет
позволено выполняться.

===== ========================================================
0     Отключить механизм.  Не контролировать и не корректировать
      частоту сэмплирования perf независимо от того, сколько процессорного времени это занимает.

1-100 Попытаться ограничить (throttle) частоту сэмплов perf до этого
      процента CPU.  Примечание: ядро вычисляет
      «ожидаемую» длину каждого события сэмпла.  100 здесь означает
      100% от этой ожидаемой длины.  Даже если это установлено в
      100, вы всё ещё можете наблюдать ограничение сэмплирования, если эта
      длина превышена.  Установите 0, если вам действительно не важно,
      сколько CPU потребляется.
===== ========================================================


perf_event_paranoid
===================

Управляет использованием системы событий производительности непривилегированными
пользователями (без CAP_PERFMON).  Значение по умолчанию — 2.

По соображениям обратной совместимости доступ к системному мониторингу
производительности и наблюдаемости остаётся открытым для процессов с привилегией
CAP_SYS_ADMIN, но использование CAP_SYS_ADMIN для безопасных операций мониторинга
производительности системы и наблюдаемости не рекомендуется
в пользу сценариев использования CAP_PERFMON.

===  ==================================================================
 -1  Разрешить использование (почти) всех событий всем пользователям.

     Игнорировать лимит mlock после perf_event_mlock_kb без
     ``CAP_IPC_LOCK``.

>=0  Запретить ftrace function tracepoint пользователям без
     ``CAP_PERFMON``.

     Запретить доступ к raw tracepoint пользователям без ``CAP_PERFMON``.

>=1  Запретить доступ к событиям CPU пользователям без ``CAP_PERFMON``.

>=2  Запретить профилирование ядра пользователям без ``CAP_PERFMON``.
===  ==================================================================

================

Частота некоторых предупреждающих сообщений ограничивается. ``printk_ratelimit`` задаёт

Значение 0 отключит ограничение частоты.


printk_ratelimit_burst
======================

Хотя в долгосрочной перспективе мы обеспечиваем одно сообщение в `printk_ratelimit`_

может быть отправлен другой всплеск сообщений.

Значение по умолчанию — 10 сообщений.

random
======

Это каталог со следующими записями:

* ``boot_id``: UUID, генерируемый при первом извлечении и
  неизменный после этого;

* ``uuid``: UUID, генерируемый при каждом извлечении (это, таким образом,

randomize_va_space
==================

Этот параметр можно использовать для выбора типа рандомизации адресного
пространства процесса, используемой в системе, для архитектур,
поддерживающих эту возможность.

==  ===========================================================================
0   Выключить рандомизацию адресного пространства процесса.  Это значение
    по умолчанию для архитектур, которые в любом случае не поддерживают эту возможность,
    и для ядер, загруженных с параметром «norandmaps».

1   Сделать адреса базы mmap, стека и страницы VDSO рандомизированными.
    Это, помимо прочего, означает, что разделяемые библиотеки будут
    загружаться по случайным адресам.  Также для PIE-скомпонованных бинарников
    местоположение начала кода рандомизируется.  Это значение по умолчанию, если
    включён параметр ``CONFIG_COMPAT_BRK``.

2   Дополнительно включить рандомизацию кучи (heap).  Это значение по умолчанию, если
    ``CONFIG_COMPAT_BRK`` отключён.

    Существует несколько унаследованных приложений (таких как некоторые древние
    версии libc.so.5 от 1996 года), которые предполагают, что область brk начинается
    сразу после конца code+bss.  Эти приложения ломаются, когда
    начало области brk рандомизируется.  Однако нет известных
    не-унаследованных приложений, которые были бы сломаны таким образом, поэтому для большинства
    систем безопасно выбрать полную рандомизацию.

    Системы с древними и/или сломанными бинарниками должны быть сконфигурированы
    с включённым ``CONFIG_COMPAT_BRK``, который исключает кучу из рандомизации адресного
    пространства процесса.
==  ===========================================================================

sched_energy_aware
==================

Включает/отключает энергоэффективное планирование (Energy Aware Scheduling, EAS). EAS запускается
автоматически на платформах, где он может работать (то есть,

требованиям для EAS, но вы не хотите его использовать, измените
это значение на 0. На не-EAS-платформах операция записи завершается с ошибкой, а
чтение ничего не возвращает.

Это означает, что любое запрошенное значение uclamp.max не может быть больше
sched_util_clamp_max, т.е. оно ограничено диапазоном
[0:sched_util_clamp_max].

sched_util_clamp_min_rt_default
===============================

По умолчанию Linux настроен на производительность. Что означает, что задачи RT всегда работают
на наивысшей частоте и наиболее производительном (наибольшей capacity) CPU (в
гетерогенных системах).

Uclamp достигает этого, устанавливая запрошенное uclamp.min всех задач RT в
1024 по умолчанию, что фактически ускоряет (boosts) задачи до работы на наивысшей
частоте и смещает их к работе на самом большом CPU.

Этот параметр (knob) позволяет администраторам изменить поведение по умолчанию, когда uclamp используется.
В частности, в устройствах с батарейным питанием работа на максимальной
capacity и частоте увеличит энергопотребление и сократит срок службы
батареи.

Этот параметр эффективен только для задач RT, для которых пользователь не модифицировал их
запрошенное значение uclamp.min через системный вызов sched_setattr().

Этот параметр не выходит за пределы ограничения диапазона, налагаемого sched_util_clamp_min,
определённым выше.

Например, если

	sched_util_clamp_min_rt_default = 800
	sched_util_clamp_min = 600

Тогда boost будет ограничен до 600, потому что 800 находится вне допустимого
диапазона [0:600]. Это может произойти, например, если режим энергосбережения
временно ограничит все boost-ы, модифицировав sched_util_clamp_min. Как только
это ограничение снимается, запрошенное sched_util_clamp_min_rt_default
вступит в силу.

shmall
======

Этот параметр задаёт общее количество страниц разделяемой памяти, которые можно использовать
внутри пространства имён ipc. Подсчёт страниц разделяемой памяти происходит для каждого пространства
имён ipc отдельно и не наследуется. Следовательно, ``shmall`` всегда должен быть как
минимум ``ceil(shmmax/PAGE_SIZE)``.

Если вы не уверены, каков ``PAGE_SIZE`` по умолчанию в вашей системе Linux,
вы можете выполнить следующую команду::

	# getconf PAGE_SIZE

Чтобы уменьшить или отключить возможность выделять разделяемую память, вы должны создать
новое пространство имён ipc, установить этот параметр в требуемое значение и запретить
создание нового пространства имён ipc в текущем пространстве имён пользователей; либо можно использовать
cgroups.

Сегменты разделяемой памяти размером до 1Гб теперь поддерживаются в

shm_rmid_forced
===============

Linux позволяет задавать лимиты ресурсов, включая то, сколько памяти один
процесс может потребить, через ``setrlimit(2)``.  К сожалению, сегменты разделяемой памяти
могут существовать без связи с каким-либо процессом, и
поэтому могут не учитываться ни в каких лимитах ресурсов.  Если включено,
сегменты разделяемой памяти автоматически уничтожаются, когда их счётчик подключений
(attach count) становится равным нулю после отсоединения или завершения процесса.  Это также
уничтожит сегменты, которые были созданы, но никогда не подключались, при выходе
из процесса.  Единственное оставшееся применение для ``IPC_RMID`` — немедленно
уничтожить неподключённый сегмент.  Разумеется, это ломает то, как всё
определено, поэтому некоторые приложения могут перестать работать.  Учтите, что эта
возможность не принесёт вам пользы, если вы также не настроите свои лимиты
ресурсов (в частности, ``RLIMIT_AS`` и ``RLIMIT_NPROC``).  Большинство систем в этом не
нуждаются.

Учтите, что если вы измените это с 0 на 1, уже созданные сегменты
без пользователей и с мёртвым процессом-создателем будут уничтожены.


sysctl_writes_strict
====================

Управляет тем, как позиция в файле влияет на поведение обновления значений sysctl
через интерфейс ``/proc/sys``:

  ==   ======================================================================
  -1   Унаследованная обработка значения sysctl на каждую запись, без предупреждений printk.
       Каждый системный вызов записи должен полностью содержать записываемое значение
       sysctl, и несколько записей в один файловый дескриптор sysctl
       будут перезаписывать значение sysctl независимо от позиции в файле.
   0   То же поведение, что и выше, но с предупреждением о процессах, которые выполняют записи
       в файловый дескриптор sysctl, когда позиция в файле не равна 0.
   1   (по умолчанию) Учитывать позицию в файле при записи строк sysctl. Несколько
       записей будут дописываться в буфер значения sysctl. Всё, что выходит за максимальную
       длину буфера значения sysctl, будет игнорироваться. Записи в числовые
       записи sysctl всегда должны быть на позиции 0 в файле, и значение должно
       полностью содержаться в буфере, отправленном в системном вызове записи.
  ==   ======================================================================

soft_watchdog
=============

Этот параметр можно использовать для управления детектором мягких зависаний (soft lockup).

= =================================
0 Отключить детектор мягких зависаний.
1 Включить детектор мягких зависаний.
= =================================

Детектор мягких зависаний контролирует CPU на предмет потоков, монополизирующих CPU
без добровольного перепланирования и тем самым мешающих потокам 'migration/N'
выполняться, что приводит к сбою выполнения работы watchdog. Механизм зависит
от способности CPU отвечать на прерывания таймера, которые нужны для того,
чтобы работа watchdog была поставлена в очередь функцией таймера watchdog, иначе NMI
watchdog — если включён — может обнаружить состояние аппаратного зависания.


split_lock_mitigate (только x86)
================================

На x86 каждая «split lock» налагает общесистемный штраф к производительности. На больших
системах большое число split lock от непривилегированных пользователей может привести к
отказам в обслуживании (denials of service) для добропорядочных и потенциально более важных пользователей.

Ядро смягчает действия этих недобросовестных пользователей, обнаруживая split lock и налагая
штрафы: заставляя их ждать и позволяя только одному ядру (core) выполнять split
lock одновременно.

Эти меры смягчения могут сделать эти недобросовестные приложения невыносимо медленными. Установка
split_lock_mitigate=0 может восстановить некоторую производительность приложений, но также
увеличит подверженность системы атакам типа «отказ в обслуживании» от пользователей split lock.

= ===================================================================
0 Отключить режим смягчения — просто предупреждает о split lock в журнале ядра
  и подвергает систему отказам в обслуживании от тех, кто использует split lock.
1 Включить режим смягчения (это значение по умолчанию) — наказывает тех, кто использует split
  lock, намеренной деградацией производительности.
= ===================================================================


stack_erasing
=============

Этот параметр можно использовать для управления очисткой стека ядра в конце
системных вызовов для ядер, собранных с ``CONFIG_KSTACK_ERASE``.

Эта очистка уменьшает информацию, которую могут раскрыть баги утечки стека
ядра, и блокирует некоторые атаки на неинициализированные переменные стека.
Компромисс — влияние на производительность: на однопроцессорной системе компиляция
ядра показывает замедление на 1%, на других системах и нагрузках может различаться.

= ====================================================================
0 Очистка стека ядра отключена, KSTACK_ERASE_METRICS не обновляются.
1 Очистка стека ядра включена (по умолчанию), она выполняется перед
  возвратом в пространство пользователя в конце системных вызовов.
= ====================================================================


stop-a (только SPARC)
=====================

Управляет Stop-A:

= ====================================
0 Stop-A не имеет эффекта.
1 Stop-A прерывает в PROM (по умолчанию).
= ====================================

Stop-A всегда включён при панике, чтобы пользователь мог вернуться к
загрузочному PROM.

tainted
=======

Ненулевое, если ядро было «испорчено» (tainted). Числовые значения, которые можно
объединять по OR. Буквы видны в строке «Tainted» отчётов Oops.

======  =====  ==============================================================
     1  `(P)`  был загружен проприетарный модуль
     2  `(F)`  модуль был загружен принудительно
     4  `(S)`  ядро выполняется на системе вне спецификации
     8  `(R)`  модуль был выгружен принудительно
    16  `(M)`  процессор сообщил об исключении Machine Check Exception (MCE)
    32  `(B)`  была обращение к плохой странице или некоторые неожиданные флаги страницы
    64  `(U)`  taint запрошен приложением пространства пользователя
   128  `(D)`  ядро недавно «умерло», т.е. был OOPS или BUG
   256  `(A)`  таблица ACPI была переопределена пользователем
   512  `(W)`  ядро выдало предупреждение
  1024  `(C)`  был загружен staging-драйвер
  2048  `(I)`  применён обходной путь (workaround) для бага в прошивке платформы
  4096  `(O)`  был загружен собранный внешне («out-of-tree») модуль
  8192  `(E)`  был загружен неподписанный модуль
 16384  `(L)`  произошёл soft lockup
 32768  `(K)`  ядро было пропатчено «вживую» (live patched)
 65536  `(X)`  Вспомогательный taint, определяемый и используемый дистрибутивами
131072  `(T)`  Ядро было собрано с плагином рандомизации структур (struct randomization)
======  =====  ==============================================================

Дополнительную информацию см. в Documentation/admin-guide/tainted-kernels.rst.

Примечание:
  записи в интерфейс этого sysctl завершатся с ошибкой ``EINVAL``, если ядро
  загружено с параметром командной строки ``panic_on_taint=<bitmask>,nousertaint``
  и любое из объединённых по OR значений, записываемых в ``tainted``, совпадает с
  битовой маской, объявленной в panic_on_taint.
  Подробнее об этом конкретном параметре командной строки ядра и его необязательном
  переключателе ``nousertaint`` см. в
  Documentation/admin-guide/kernel-parameters.rst.

threads-max
===========

Это значение управляет максимальным числом потоков, которые можно создать
с помощью ``fork()``.

Во время инициализации ядро устанавливает это значение так, чтобы даже если
будет создано максимальное число потоков, структуры потоков занимали лишь
часть (1/8) доступных страниц RAM.

Минимальное значение, которое можно записать в ``threads-max``, — 1.

Максимальное значение, которое можно записать в ``threads-max``, задаётся
константой ``FUTEX_TID_MASK`` (0x3fffffff).

Если в ``threads-max`` записывается значение вне этого диапазона, возникает
ошибка ``EINVAL``.

timer_migration
===============

При установке в ненулевое значение пытаться мигрировать таймеры прочь от простаивающих CPU, чтобы
позволить им дольше оставаться в состояниях низкого энергопотребления.

По умолчанию установлено (1).

traceoff_on_warning
===================

При установке отключает трассировку (см. Documentation/trace/ftrace.rst) при
срабатывании ``WARN()``.


tracepoint_printk
=================

Когда tracepoint-ы отправляются в printk() (включается параметром загрузки ``tp_printk``),
эта запись обеспечивает управление во время выполнения::

    echo 0 > /proc/sys/kernel/tracepoint_printk

остановит отправку tracepoint-ов в printk(), а::

    echo 1 > /proc/sys/kernel/tracepoint_printk

отправит их в printk() снова.

Это работает только если ядро было загружено с включённым ``tp_printk``.

См. Documentation/admin-guide/kernel-parameters.rst и
Documentation/trace/boottime-trace.rst.


unaligned-trap
==============

На архитектурах, где невыровненные обращения вызывают ловушки (traps), и где
эта возможность поддерживается (``CONFIG_SYSCTL_ARCH_UNALIGN_ALLOW``; в настоящее время —
``arc``, ``parisc`` и ``loongarch``), управляет тем, перехватываются ли невыровненные ловушки
и эмулируются (вместо того чтобы завершаться ошибкой).

= ========================================================
0 Не эмулировать невыровненные обращения.
1 Эмулировать невыровненные обращения. Это настройка по умолчанию.
= ========================================================

См. также `ignore-unaligned-usertrap`_.


unknown_nmi_panic
=================

Значение в этом файле влияет на поведение обработки NMI. Когда
значение ненулевое, неизвестное NMI перехватывается и затем происходит паника. В
этот момент на консоль выводится отладочная информация ядра.

Переключатель NMI, который есть у большинства серверов IA32, вызывает неизвестное NMI, к
примеру.  Если система зависает, попробуйте нажать переключатель NMI.


unprivileged_bpf_disabled
=========================

Запись 1 в эту запись отключит непривилегированные вызовы ``bpf()``;
после отключения вызов ``bpf()`` без ``CAP_SYS_ADMIN`` или ``CAP_BPF``
вернёт ``-EPERM``. После установки в 1 это уже нельзя сбросить из
работающего ядра.

Запись 2 в эту запись также отключит непривилегированные вызовы ``bpf()``,
однако администратор всё ещё может изменить эту настройку позже, при необходимости,
записав 0 или 1 в эту запись.

Если в конфигурации ядра включён ``BPF_UNPRIV_DEFAULT_OFF``, то эта
запись по умолчанию будет равна 2, а не 0.

= =============================================================
0 Непривилегированные вызовы ``bpf()`` включены
1 Непривилегированные вызовы ``bpf()`` отключены без возможности восстановления
2 Непривилегированные вызовы ``bpf()`` отключены
= =============================================================


warn_limit
==========

Число предупреждений ядра, после которого ядро должно паниковать, когда
``panic_on_warn`` не установлен. Установка этого в 0 отключает проверку
счётчика предупреждений. Установка этого в 1 имеет тот же эффект, что и установка
``panic_on_warn=1``. Значение по умолчанию — 0.


watchdog
========

Этот параметр можно использовать для отключения или включения детектора мягких зависаний (soft lockup)
*и* NMI watchdog (т.е. детектора аппаратных зависаний) одновременно.

= ==============================
0 Отключить оба детектора зависаний.
1 Включить оба детектора зависаний.
= ==============================

Детектор мягких зависаний и NMI watchdog также можно отключать или
включать по отдельности, используя параметры ``soft_watchdog`` и ``nmi_watchdog``.
Если параметр ``watchdog`` читается, например выполнением::

   cat /proc/sys/kernel/watchdog

вывод этой команды (0 или 1) показывает логический OR
``soft_watchdog`` и ``nmi_watchdog``.


watchdog_cpumask
================

Это значение можно использовать для управления тем, на каких CPU может работать watchdog.
По умолчанию cpumask — все возможные ядра (cores), но если в конфигурации ядра включён
``NO_HZ_FULL``, и ядра указаны
аргументом загрузки ``nohz_full=``, эти ядра исключаются по умолчанию.
Офлайн-ядра могут быть включены в эту маску, и если ядро позже
переводится в онлайн, watchdog будет запущен на основе значения маски.

Обычно это значение трогают только в случае ``nohz_full``, чтобы
повторно включить ядра, которые по умолчанию не выполняли watchdog,
если на этих ядрах было заподозрено зависание ядра.

Значение аргумента — стандартный формат cpulist для cpumask,
так, например, чтобы включить watchdog на ядрах 0, 2, 3 и 4, вы
можете написать::

  echo 0,2-4 > /proc/sys/kernel/watchdog_cpumask


watchdog_thresh
===============

Это значение можно использовать для управления частотой событий hrtimer и NMI
и порогами мягких и аппаратных зависаний. Порог по умолчанию —
10 секунд.

Порог soft lockup равен (``2 * watchdog_thresh``). Установка этого
настраиваемого параметра в ноль полностью отключит обнаружение зависаний.
