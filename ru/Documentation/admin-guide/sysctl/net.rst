1. /proc/sys/net/core - Параметры сетевого ядра
===============================================

bpf_jit_enable
--------------

Включает JIT-компилятор (Just in Time) для BPF. BPF — это гибкая
и эффективная инфраструктура, позволяющая выполнять байт-код в различных
точках перехвата (hook points). Она используется в ряде подсистем ядра Linux, таких
как сеть (например, XDP, tc), трассировка (например, kprobes, uprobes, tracepoints)
и безопасность (например, seccomp). В LLVM есть BPF-бэкенд, способный компилировать
ограниченное подмножество C в последовательность инструкций BPF. После загрузки программы
через bpf(2) и прохождения верификатора в ядре JIT транслирует
эти BPF-проглеты в нативные инструкции CPU. Существуют
два варианта JIT — более новый eBPF JIT, в настоящее время поддерживаемый на:

  - x86_64
  - x86_32
  - arm64
  - arm32
  - ppc64
  - ppc32
  - sparc64
  - mips64
  - s390x
  - riscv64
  - riscv32
  - loongarch64
  - arc

И более старый cBPF JIT, поддерживаемый на следующих архитектурах:

  - mips
  - sparc

eBPF JIT являются надмножеством cBPF JIT, то есть ядро будет
переносить инструкции cBPF в инструкции eBPF и затем прозрачно
выполнять их JIT-компиляцию. Более старые cBPF JIT способны транслировать только
фильтры tcpdump, правила seccomp и т. п., но не упомянутые eBPF-программы,
загружаемые через bpf(2).

Значения:

	- 0 - отключить JIT (значение по умолчанию)
	- 1 - включить JIT
	- 2 - включить JIT и попросить компилятор выводить трассировки в журнал ядра.

bpf_jit_harden
--------------

Включает усиление защиты (hardening) для JIT-компилятора BPF. Поддерживаются бэкенды
eBPF JIT. Включение усиления снижает производительность, но может
смягчить атаки типа JIT spraying.

Значения:

	- 0 - отключить усиление JIT (значение по умолчанию)
	- 1 - включить усиление JIT только для непривилегированных пользователей
	- 2 - включить усиление JIT для всех пользователей

где «привилегированный пользователь» в данном контексте означает процесс, имеющий
CAP_BPF или CAP_SYS_ADMIN в корневом пользовательском пространстве имён.

bpf_jit_kallsyms
----------------

Когда JIT-компилятор BPF включён, скомпилированные образы являются неизвестными
адресами для ядра, то есть они не отображаются ни в трассировках, ни
в /proc/kallsyms. Этот параметр включает экспорт этих адресов, который может
использоваться для отладки/трассировки. Если включён bpf_jit_harden, эта
возможность отключается.

Значения :

	- 0 - отключить экспорт JIT в kallsyms (значение по умолчанию)
	- 1 - включить экспорт JIT в kallsyms только для привилегированных пользователей

bpf_jit_limit
-------------

Задаёт глобальный лимит на выделение памяти для JIT-компилятора BPF
с целью отклонения непривилегированных запросов JIT после его
превышения. bpf_jit_limit содержит значение глобального лимита
в байтах.

dev_weight
----------

Максимальное число пакетов, которое ядро может обработать за одно прерывание NAPI;
это переменная типа Per-CPU. Для драйверов, поддерживающих LRO или GRO_HW, аппаратно
агрегированный пакет считается в данном контексте за один пакет.

По умолчанию: 64

dev_weight_rx_bias
------------------

Обработка RPS (например, RFS, aRFS) конкурирует с зарегистрированной функцией опроса NAPI
драйвера за netdev_budget в рамках одного цикла softirq. Этот параметр влияет
на долю настроенного netdev_budget, которая тратится на обработку пакетов на основе RPS
во время циклов RX softirq. Кроме того, он предназначен для того, чтобы сделать текущий
dev_weight адаптируемым к асимметричным потребностям CPU на стороне RX/TX сетевого стека.
(см. dev_weight_tx_bias) Действует на уровне каждого CPU. Определение основано
на dev_weight и вычисляется мультипликативно (dev_weight * dev_weight_rx_bias).

По умолчанию: 1

dev_weight_tx_bias
------------------

Масштабирует максимальное число пакетов, которое может быть обработано за один цикл TX softirq.
Действует на уровне каждого CPU. Позволяет масштабировать текущий dev_weight для асимметричных
потребностей обработки сетевого стека. Будьте осторожны, чтобы обработка TX softirq не стала «пожирателем» CPU.

Вычисление основано на dev_weight (dev_weight * dev_weight_tx_bias).

По умолчанию: 1

default_qdisc
-------------

Дисциплина обслуживания очереди (queuing discipline) по умолчанию, используемая для сетевых устройств. Это позволяет
переопределить дисциплину pfifo_fast по умолчанию альтернативной. Поскольку дисциплина обслуживания очереди
по умолчанию создаётся без дополнительных параметров, она лучше всего подходит
для дисциплин обслуживания очереди, которые хорошо работают без настройки, таких как stochastic
fair queue (sfq), CoDel (codel) или fair queue CoDel (fq_codel). Не используйте
дисциплины обслуживания очереди типа Hierarchical Token Bucket или Deficit Round Robin,
которые требуют настройки классов и пропускных способностей. Обратите внимание, что физические многоочередные
интерфейсы по-прежнему используют mq в качестве корневого qdisc, который в свою очередь использует это значение по умолчанию для своих
листьев. Виртуальные устройства (например, lo или veth) игнорируют этот параметр и вместо этого
по умолчанию используют noqueue.

По умолчанию: pfifo_fast

busy_read
---------

Тайм-аут busy poll с низкой задержкой для чтения из сокета. (требуется CONFIG_NET_RX_BUSY_POLL)
Приблизительное время в мкс активного цикла ожидания пакетов в очереди устройства.
Задаёт значение по умолчанию для опции сокета SO_BUSY_POLL.
Может быть задано или переопределено для каждого сокета путём установки опции сокета SO_BUSY_POLL,
что является предпочтительным методом включения. Если вам нужно включить эту возможность
глобально через sysctl, рекомендуется значение 50.

Увеличит энергопотребление.

По умолчанию: 0 (выключено)

busy_poll
----------------
Тайм-аут busy poll с низкой задержкой для poll и select. (требуется CONFIG_NET_RX_BUSY_POLL)
Приблизительное время в мкс активного цикла ожидания событий.
Рекомендуемое значение зависит от числа сокетов, которые вы опрашиваете.
Для нескольких сокетов — 50, для нескольких сотен — 100.
Для большего числа вам, вероятно, стоит использовать epoll.
Обратите внимание, что busy poll выполняется только для сокетов с установленным SO_BUSY_POLL,
поэтому вам нужно либо выборочно установить SO_BUSY_POLL на этих сокетах, либо задать
sysctl.net.busy_read глобально.

Увеличит энергопотребление.

По умолчанию: 0 (выключено)

mem_pcpu_rsv
------------

Размер per-cpu зарезервированного кэша forward alloc в единицах страниц. По умолчанию 1 МБ на CPU.

bypass_prot_mem
---------------

Пропускать учёт буферов сокетов в глобальном попротокольном учёте
памяти, управляемом net.ipv4.tcp_mem, net.ipv4.udp_mem и т. д.

По умолчанию: 0 (выключено)

rmem_default
------------

Значение по умолчанию приёмного буфера сокета в байтах.

rmem_max
--------

Максимальный размер приёмного буфера сокета в байтах.

По умолчанию: 4194304

rps_default_mask
----------------

Маска CPU для RPS по умолчанию, используемая на вновь создаваемых сетевых устройствах. Пустая
маска означает, что RPS по умолчанию отключён.

tstamp_allow_data
-----------------
Разрешает процессам получать tx-метки времени, возвращаемые вместе с исходным
содержимым пакета. Если отключено, запросы меток времени передачи от непривилегированных
процессов отбрасываются, если не установлена опция сокета SOF_TIMESTAMPING_OPT_TSONLY.

По умолчанию: 1 (включено)


wmem_default
------------

Значение по умолчанию (в байтах) буфера отправки сокета.

wmem_max
--------

Максимальный размер буфера отправки сокета в байтах.

По умолчанию: 4194304

message_burst and message_cost
------------------------------

Эти параметры используются для ограничения числа предупреждающих сообщений, записываемых в журнал
ядра сетевым кодом. Они обеспечивают ограничение частоты, чтобы сделать
атаку типа «отказ в обслуживании» невозможной. Чем выше коэффициент message_cost, тем
меньше сообщений будет записано. Message_burst управляет тем, когда сообщения будут
отбрасываться. Настройки по умолчанию ограничивают предупреждающие сообщения одним каждые пять
секунд.

warnings
--------

Этот sysctl теперь не используется.

Он использовался для управления сообщениями на консоли от сетевого стека, которые
возникают из-за проблем в сети, таких как дублирующийся адрес или неверные
контрольные суммы.

Эти сообщения теперь выводятся на уровне KERN_DEBUG и, как правило, могут быть включены
и управляться средством dynamic_debug.

netdev_budget
-------------

Максимальное число пакетов, забираемых со всех интерфейсов за один цикл опроса (NAPI
poll). За один цикл опроса интерфейсы, зарегистрированные на опрос,
проверяются по принципу round-robin. Кроме того, цикл опроса не может превышать
netdev_budget_usecs микросекунд, даже если netdev_budget не был
исчерпан.

netdev_budget_usecs
---------------------

Максимальное число микросекунд в одном цикле опроса NAPI. Опрос
завершится, когда либо истекут netdev_budget_usecs в течение
цикла опроса, либо число обработанных пакетов достигнет netdev_budget.

netdev_max_backlog
------------------

Максимальное число пакетов, поставленных в очередь на стороне INPUT, когда интерфейс
получает пакеты быстрее, чем ядро может их обработать.

qdisc_max_burst
------------------

Максимальное число пакетов, которое может быть временно сохранено перед
достижением qdisc.

По умолчанию: 1000

netdev_rss_key
--------------

Драйверы с поддержкой RSS (Receive Side Scaling) используют ключ хоста,
который генерируется случайным образом.
Некоторому ПО из пространства пользователя может потребоваться получить его содержимое, даже если драйверы пока
не обеспечивают поддержку ethtool -x.

::

  myhost:~# cat /proc/sys/net/core/netdev_rss_key
  84:50:f4:00:a8:15:d1:a7:e9:7f:1d:60:35:c7:47:25:42:97:74:ca:56:bb:b6:a1:d8: ... (256 bytes total)

Файл содержит все нулевые байты, если ни один драйвер так и не вызвал функцию
netdev_rss_key_fill().

Note:
  /proc/sys/net/core/netdev_rss_key содержит 256 байт ключа,
  но многие драйверы используют только 40 или 52 байта из них.

::

  myhost:~# ethtool -x eth0
  RX flow hash indirection table for eth0 with 8 RX ring(s):
      0:    0     1     2     3     4     5     6     7
  RSS hash key:
  84:50:f4:00:a8:15:d1:a7:e9:7f:1d:60:35:c7:47:25:42:97:74:ca:56:bb:b6:a1:d8:43:e3:c9:0c:fd:17:55:c2:3a:4d:69:ed:f1:42:89

netdev_tstamp_prequeue
----------------------

Если установлено в 0, метки времени RX-пакетов могут отбираться после обработки RPS, когда
целевой CPU обрабатывает пакеты. Это может вносить некоторую задержку в метки времени, но
позволяет распределить нагрузку по нескольким CPU.

Если установлено в 1 (по умолчанию), метки времени отбираются как можно раньше, до
постановки в очередь.

netdev_unregister_timeout_secs
------------------------------

Тайм-аут отмены регистрации сетевого устройства в секундах.
Эта опция управляет тайм-аутом (в секундах), используемым для выдачи предупреждения во время
ожидания падения счётчика ссылок сетевого устройства до 0 в ходе
отмены регистрации устройства. Меньшее значение может быть полезно при бисекции для более быстрого обнаружения
утечки ссылки. Большее значение может быть полезно для предотвращения ложных
предупреждений на медленных/нагруженных системах.
Значение по умолчанию — 10, минимум — 1, максимум — 3600.

skb_defer_max
-------------

Максимальный размер (в skb) per-cpu списка skb, освобождаемых
тем CPU, который их выделил.

По умолчанию: 128

optmem_max
----------

Максимальный размер вспомогательного буфера, разрешённый на сокет. Вспомогательные данные — это последовательность
структур struct cmsghdr с присоединёнными данными. TCP tx zerocopy также использует
optmem_max в качестве лимита для своих внутренних структур.

По умолчанию : 128 КБ

fb_tunnels_only_for_init_net
----------------------------

Управляет тем, создаются ли автоматически резервные туннели (fallback tunnels, такие как tunl0, gre0, gretap0, erspan0,
sit0, ip6tnl0, ip6gre0). Существуют 3 возможности
(a) значение = 0; соответствующие резервные туннели создаются при загрузке
модуля в каждом сетевом пространстве имён (обратно совместимое поведение).
(b) значение = 1; [значение kcmd: initns] соответствующие резервные туннели
создаются только в сетевом пространстве имён init, и любое другое сетевое пространство имён
не будет их иметь.
(c) значение = 2; [значение kcmd: none] резервные туннели не создаются
при загрузке модуля ни в одном из сетевых пространств имён. Установка значения
«2» бессмысленна после загрузки, если эти модули встроены в ядро, поэтому существует
параметр командной строки ядра, который может изменить это значение по умолчанию. Дополнительные сведения см.
в Documentation/admin-guide/kernel-parameters.txt.

Несоздание резервных туннелей передаёт управление пространству пользователя, позволяя создавать
только то, что необходимо, и избегать создания избыточных устройств.

По умолчанию : 0  (по соображениям совместимости)

devconf_inherit_init_net
------------------------

Управляет тем, должно ли новое сетевое пространство имён наследовать все текущие
настройки из /proc/sys/net/{ipv4,ipv6}/conf/{all,default}/. По
умолчанию мы сохраняем текущее поведение: для IPv4 мы наследуем все текущие
настройки из init_net, а для IPv6 сбрасываем все настройки в значения по умолчанию.

Если установлено в 1, и настройки IPv4, и настройки IPv6 принудительно наследуются от
текущих в init_net. Если установлено в 2, и настройки IPv4, и настройки IPv6
принудительно сбрасываются в значения по умолчанию. Если установлено в 3, и настройки IPv4, и настройки IPv6
принудительно наследуются от текущих в том netns, где было создано
это новое netns.

По умолчанию : 0  (по соображениям совместимости)

txrehash
--------

Управляет поведением переосмысления хэша по умолчанию для сокета, когда опция SO_TXREHASH установлена
в SOCK_TXREHASH_DEFAULT (т. е. не переопределена через setsockopt).

Если установлено в 1 (по умолчанию), переосмысление хэша выполняется на слушающем сокете.
Если установлено в 0, переосмысление хэша не выполняется.

txq_reselection_ms
------------------

Управляет тем, как часто (в мс) занятый подключённый поток может выбирать другую tx-очередь.

Повторный выбор желателен, когда/если пользовательский поток мигрировал и XPS
выбрал бы другую очередь. То же самое может произойти и без XPS,
если изменился хэш потока.

Но переключение txq может вносить переупорядочивания (reorders), особенно если
старая очередь находится под высокой нагрузкой. Современные TCP-стеки хорошо
справляются с переупорядочиваниями, если они происходят не слишком часто.

Чтобы отключить эту возможность, установите значение в 0.

По умолчанию : 1000

gro_normal_batch
----------------

Максимальное число сегментов для пакетной группировки на выходе GRO. Когда пакет
покидает GRO — либо как объединённый суперфрейм, либо как исходный пакет, который
GRO решил не объединять, — он помещается в per-NAPI список. Затем этот
список передаётся в стек, когда число сегментов достигает
лимита gro_normal_batch.

high_order_alloc_disable
------------------------

По умолчанию аллокатор для page frags пытается использовать страницы высокого порядка (order-3
на x86). Хотя поведение по умолчанию даёт хорошие результаты в большинстве случаев, некоторые пользователи
могли столкнуться с конкуренцией при выделении/освобождении страниц. Это было особенно
актуально на более старых ядрах (< 5.14), когда страницы высокого порядка не хранились в per-cpu
списках. Этот параметр позволяет включить вместо этого выделение order-0, но теперь имеет в основном
историческое значение.

По умолчанию: 0

4. Appletalk
------------

Каталог /proc/sys/net/appletalk содержит данные конфигурации Appletalk,
когда Appletalk загружен. Настраиваемые параметры:

aarp-expiry-time
----------------

Время, в течение которого мы храним запись ARP до её устаревания. Используется для устаревания
старых хостов.

aarp-resolve-time
-----------------

Время, которое мы потратим на попытку разрешить адрес Appletalk.

aarp-retransmit-limit
---------------------

Число раз, которое мы повторно отправим запрос, прежде чем сдаться.

aarp-tick-time
--------------

Управляет частотой, с которой проверяются устаревания.

Каталог /proc/net/appletalk содержит список активных сокетов Appletalk
на машине.

Поля показывают тип DDP, локальный адрес (в формате network:node),
удалённый адрес, размер очереди передачи в ожидании, размер
приёмной очереди (байты, ожидающие чтения приложениями), состояние и uid,
владеющий сокетом.

/proc/net/atalk_iface перечисляет все интерфейсы, настроенные для appletalk. Он
показывает имя интерфейса, его адрес Appletalk, диапазон сети на
этом адресе (или номер сети для сетей фазы 1) и состояние
интерфейса.

/proc/net/atalk_route перечисляет каждый известный сетевой маршрут. Он перечисляет цель
(сеть), к которой ведёт маршрут, маршрутизатор (может быть подключён напрямую),
флаги маршрута и устройство, используемое маршрутом.

5. TIPC
-------

tipc_rmem
---------

Протокол TIPC теперь имеет настраиваемый параметр для приёмной памяти, аналогичный
tcp_rmem — т. е. вектор из 3 значений INTEGER: (min, default, max)

::

    # cat /proc/sys/net/tipc/tipc_rmem
    4252725 34021800        68043600
    #

Значение max устанавливается в CONN_OVERLOAD_LIMIT, а значения default и min
являются масштабированными (сдвинутыми) версиями того же самого значения. Обратите внимание, что значение min
на данный момент не используется сколько-нибудь осмысленным образом, но тройка
сохраняется для согласованности с такими вещами, как tcp_rmem.

named_timeout
-------------

Обновления таблицы имён TIPC распространяются асинхронно в кластере, без
какой-либо формы обработки транзакций. Это означает, что возможны различные сценарии гонок.
Один из них состоит в том, что отзыв имени (name withdrawal), отправленный одним узлом и полученный
другим узлом, может прийти после того, как второе, пересекающееся объявление имени уже
было принято от третьего узла, хотя конфликтующие обновления
изначально могли быть выпущены в правильной последовательности.
Если named_timeout ненулевой, неудавшиеся обновления топологии будут помещаться в очередь отложенных (defer
queue) до тех пор, пока не придёт другое событие, устраняющее ошибку, или пока не истечёт тайм-аут.
Значение указывается в миллисекундах.

6. /proc/sys/net/vsock - Сокеты VSOCK
-------------------------------------

Сокеты VSOCK (AF_VSOCK) обеспечивают связь между виртуальными машинами и
их хостами. Поведение сокетов VSOCK в сетевом пространстве имён определяется
режимом пространства имён (``global`` или ``local``), который управляет тем, как выделяются CID
(Context ID) и как сокеты взаимодействуют между пространствами имён.

ns_mode
-------

Только для чтения. Сообщает режим текущего пространства имён, задаваемый при создании пространства имён
и неизменяемый в дальнейшем.

Значения:

	- ``global`` - пространство имён использует общесистемное выделение CID, и
	  его сокеты могут достигать любой ВМ или сокета в любом global-пространстве имён.
	  Сокеты в этом пространстве имён не могут достигать сокетов в local-пространствах
	  имён.
	- ``local`` - пространство имён имеет приватное выделение CID, и его
	  сокеты могут подключаться только к ВМ или сокетам в пределах того же
	  пространства имён.

Режим init_net всегда ``global``.

child_ns_mode
-------------

Управляет тем, какой режим будут наследовать вновь создаваемые дочерние пространства имён. При создании
пространства имён ``ns_mode`` наследуется из ``child_ns_mode`` родителя.
Начальное значение совпадает с собственным ``ns_mode`` пространства имён.

Значения:

	- ``global`` - дочерние пространства имён будут использовать общесистемное выделение CID,
	  и их сокеты смогут достигать любой ВМ или сокета в любом
	  global-пространстве имён.
	- ``local`` - дочерние пространства имён будут иметь приватное выделение CID, и
	  их сокеты смогут подключаться только в пределах своего собственного
	  пространства имён.

Первая запись в ``child_ns_mode`` фиксирует его значение. Последующие записи того
же значения завершаются успешно, но запись другого значения возвращает ``-EBUSY``.

Изменение ``child_ns_mode`` влияет только на пространства имён, создаваемые после изменения;
оно не изменяет текущее пространство имён или какие-либо существующие дочерние.

Пространство имён с ``ns_mode``, установленным в ``local``, не может изменить
``child_ns_mode`` на ``global`` (возвращает ``-EPERM``).
