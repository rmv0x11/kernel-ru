Этот файл содержит документацию по sysctl-файлам в каталоге
/proc/sys/vm и действителен для ядра Linux версии 2.6.29.

Файлы в этом каталоге можно использовать для настройки работы
подсистемы виртуальной памяти (VM) ядра Linux и записи
«грязных» данных на диск.

Значения по умолчанию и процедуры инициализации для большинства этих
файлов можно найти в mm/swap.c.

admin_reserve_kbytes
====================

Объём свободной памяти в системе, который должен резервироваться для пользователей
с правом cap_sys_admin.

Значение admin_reserve_kbytes по умолчанию равно min(3% свободных страниц, 8MB).

Этого должно быть достаточно, чтобы администратор мог войти в систему и при
необходимости завершить процесс в режиме overcommit «guess», используемом по умолчанию.

Системы, работающие в режиме overcommit «never», должны увеличить это значение, чтобы
учесть полный объём виртуальной памяти (Virtual Memory Size) программ, используемых для
восстановления. Иначе root может не суметь войти в систему для её восстановления.

Как рассчитать минимальный полезный резерв?

sshd или login + bash (или другая оболочка) + top (или ps, kill и т. п.)

Для режима overcommit «guess» можно суммировать размеры резидентных множеств (RSS).
На x86_64 это около 8MB.

Для режима overcommit «never» можно взять максимум их виртуальных размеров (VSZ)
и прибавить сумму их RSS.
На x86_64 это около 128MB.

Изменение этого значения вступает в силу при каждом запросе памяти приложением.


compact_memory
==============

Доступно только при установленном CONFIG_COMPACTION. Когда в файл записывается 1,
все зоны уплотняются так, чтобы свободная память была доступна в виде непрерывных
блоков там, где это возможно. Это может быть важно, например, при выделении
огромных страниц (huge pages), хотя процессы также напрямую уплотняют память по мере необходимости.

compaction_proactiveness
========================

Этот параметр принимает значение в диапазоне [0, 100] со значением по умолчанию,
равным 20. Этот параметр определяет, насколько агрессивно уплотнение выполняется в
фоновом режиме. Запись ненулевого значения в этот параметр немедленно
запускает упреждающее уплотнение. Установка его в 0 отключает упреждающее уплотнение.

Обратите внимание, что уплотнение оказывает нетривиальное общесистемное влияние, так как
страницы, принадлежащие разным процессам, перемещаются, что также может привести
к всплескам задержек (latency) в не подозревающих об этом приложениях. Ядро использует
различные эвристики, чтобы не тратить впустую такты CPU, если оно обнаруживает, что
упреждающее уплотнение неэффективно.

Установка значения выше 80, помимо снижения приемлемого уровня
фрагментации, делает код уплотнения более чувствительным к росту
фрагментации, то есть уплотнение будет запускаться чаще, но уменьшать
фрагментацию на меньшую величину.
Это делает уровень фрагментации более стабильным во времени.

Будьте осторожны при установке экстремальных значений вроде 100, поскольку это может
вызвать чрезмерную активность фонового уплотнения.

compact_unevictable_allowed
===========================

Доступно только при установленном CONFIG_COMPACTION. Когда установлено в 1, уплотнению
разрешается просматривать невытесняемый lru (страницы под mlock) в поисках страниц для уплотнения.
Это следует использовать на системах, где приостановки из-за минорных страничных отказов являются
приемлемой платой за большой объём непрерывной свободной памяти. Установка в 0 предотвращает
перемещение уплотнением страниц, которые невытесняемы. Значение по умолчанию — 1.
На CONFIG_PREEMPT_RT значение по умолчанию — 0, чтобы избежать страничного отказа, вызванного
уплотнением, который заблокировал бы задачу от перехода в активное состояние до тех пор, пока отказ
не будет разрешён.

defrag_mode
===========

Когда установлено в 1, аллокатор страниц прилагает больше усилий, чтобы избежать фрагментации
и сохранить способность создавать огромные страницы / страницы более высокого порядка.

Рекомендуется включать это сразу после загрузки, поскольку фрагментация,
однажды возникнув, может быть долгосрочной или даже постоянной.

dirty_background_bytes
======================

Содержит объём «грязной» памяти, при достижении которого фоновые потоки
сброса данных ядра начнут запись (writeback).

Примечание:
  dirty_background_bytes — это аналог dirty_background_ratio. Одновременно может
  быть задан только один из них. Когда записывается один sysctl, он
  немедленно учитывается при вычислении лимитов «грязной» памяти, а
  другой при чтении отображается как 0.


dirty_background_ratio
======================

Содержит — в виде процента от общего объёма доступной памяти, в которую входят свободные
страницы и страницы, пригодные для возврата (reclaimable), — количество страниц, при достижении которого фоновые потоки
сброса данных ядра начнут запись «грязных» данных.

Общий объём доступной памяти не равен общему объёму памяти системы.


dirty_bytes
===========

Содержит объём «грязной» памяти, при достижении которого процесс, порождающий записи на диск,
сам начнёт запись (writeback).

Примечание: dirty_bytes — это аналог dirty_ratio. Одновременно может быть
задан только один из них. Когда записывается один sysctl, он немедленно учитывается
при вычислении лимитов «грязной» памяти, а другой при чтении отображается как 0.

Примечание: минимально допустимое значение для dirty_bytes — две страницы (в байтах); любое
значение ниже этого предела будет проигнорировано, а старая конфигурация будет
сохранена.


dirty_expire_centisecs
======================

Этот параметр используется для определения того, когда «грязные» данные достаточно стары, чтобы быть
пригодными для записи (writeout) потоками сброса данных ядра. Он выражается в сотых
долях секунды. Данные, которые были «грязными» в памяти дольше этого
интервала, будут записаны при следующем пробуждении потока сброса.


dirty_ratio
===========

Содержит — в виде процента от общего объёма доступной памяти, в которую входят свободные
страницы и страницы, пригодные для возврата (reclaimable), — количество страниц, при достижении которого процесс,
порождающий записи на диск, сам начнёт запись «грязных» данных.

Общий объём доступной памяти не равен общему объёму памяти системы.


dirtytime_expire_seconds
========================

Когда у lazytime-inode постоянно «загрязняются» его страницы, inode с
обновлённой меткой времени никогда не получит шанса быть записанным. И, если
единственное, что произошло в файловой системе, — это dirtytime-inode, вызванный
обновлением atime, будет запланирован рабочий поток, чтобы убедиться, что этот inode
в конечном итоге будет вытолкнут на диск. Этот параметр используется для определения того, когда «грязный»
inode достаточно стар, чтобы быть пригодным для записи (writeback) потоками сброса данных ядра.
И он также используется как интервал пробуждения потока dirtytime_writeback.

Установка этого значения в ноль отключает периодическую запись dirtytime.


dirty_writeback_centisecs
=========================

Потоки сброса данных ядра периодически просыпаются и записывают `old` данные
на диск. Этот параметр выражает интервал между такими пробуждениями, в
сотых долях секунды.

Установка этого значения в ноль полностью отключает периодическую запись (writeback).


drop_caches
===========

Запись в этот файл заставляет ядро сбрасывать чистые кэши, а также
slab-объекты, пригодные для возврата, такие как dentry и inode. После сброса их
память освобождается.

Чтобы освободить pagecache::

	echo 1 > /proc/sys/vm/drop_caches

Чтобы освободить slab-объекты, пригодные для возврата (включая dentry и inode)::

	echo 2 > /proc/sys/vm/drop_caches

Чтобы освободить slab-объекты и pagecache::

	echo 3 > /proc/sys/vm/drop_caches

Это неразрушающая операция, и она не освобождает никаких «грязных» объектов.
Чтобы увеличить количество объектов, освобождаемых этой операцией, пользователь может выполнить
`sync` перед записью в /proc/sys/vm/drop_caches. Это минимизирует
количество «грязных» объектов в системе и создаст больше кандидатов на
сброс.

Этот файл не является средством управления ростом различных кэшей ядра
(inode, dentry, pagecache и т. д.). Эти объекты автоматически
возвращаются ядром, когда память нужна где-то ещё в системе.

Использование этого файла может вызвать проблемы с производительностью. Поскольку он отбрасывает кэшированные
объекты, повторное создание сброшенных объектов может стоить значительного объёма I/O и времени CPU,
особенно если они интенсивно использовались. Из-за этого
использование вне тестовой или отладочной среды не рекомендуется.

Вы можете увидеть информационные сообщения в логе ядра, когда этот файл
используется::

	cat (1234): drop_caches: 3

Они носят чисто информационный характер. Они не означают, что с вашей системой
что-то не так. Чтобы отключить их, запишите echo 4 (bit 2) в drop_caches.

enable_soft_offline
===================
Исправимые ошибки памяти очень распространены на серверах. Soft-offline — это решение ядра
для страниц памяти, имеющих (чрезмерное количество) исправленных ошибок памяти.

Для разных типов страниц soft-offline имеет разное поведение / стоимость.

- Для одиночной (raw) ошибочной страницы soft-offline мигрирует содержимое используемой страницы на
  новую одиночную страницу.

- Для страницы, входящей в transparent hugepage, soft-offline разбивает
  transparent hugepage на одиночные страницы, затем мигрирует только одиночную ошибочную страницу.
  В результате пользователь прозрачно обеспечивается на 1 hugepage меньше, что влияет на
  производительность доступа к памяти.

- Для страницы, входящей в HugeTLB hugepage, soft-offline сначала мигрирует
  весь HugeTLB hugepage, при этом в качестве цели миграции будет израсходован свободный hugepage.
  Затем исходный hugepage растворяется в одиночные
  страницы без компенсации, уменьшая ёмкость пула HugeTLB на 1.

Выбор между надёжностью (держаться подальше от ненадёжной
физической памяти) и последствиями для производительности / ёмкости в случаях transparent и
HugeTLB остаётся за пользователем.

Для всех архитектур enable_soft_offline управляет тем, выполнять ли soft-offline
страниц памяти. Когда установлено в 1, ядро пытается выполнить soft-offline страниц
всякий раз, когда считает это нужным. Когда установлено в 0, ядро возвращает EOPNOTSUPP на
запрос soft-offline страниц. Его значение по умолчанию — 1.

Стоит упомянуть, что после установки enable_soft_offline в 0
следующие запросы на soft-offline страниц не будут выполнены:

- Запрос на soft-offline страниц от RAS Correctable Errors Collector.

- На ARM — запрос на soft-offline страниц от драйвера GHES.

- На PARISC — запрос на soft-offline страниц от Page Deallocation Table.

extfrag_threshold
=================

Этот параметр влияет на то, будет ли ядро уплотнять память или выполнять прямой
возврат (direct reclaim), чтобы удовлетворить выделение высокого порядка (high-order). Файл extfrag/extfrag_index в
debugfs показывает, каков индекс фрагментации для каждого порядка в каждой зоне в
системе. Значения, стремящиеся к 0, означают, что выделения будут завершаться неудачей из-за нехватки
памяти, значения, стремящиеся к 1000, означают, что неудачи происходят из-за фрагментации, а -1
означает, что выделение будет успешным, пока соблюдаются водяные знаки (watermarks).

Ядро не будет уплотнять память в зоне, если
индекс фрагментации <= extfrag_threshold. Значение по умолчанию — 500.


highmem_is_dirtyable
====================

Доступно только для систем с включённым CONFIG_HIGHMEM (32-битные системы).

Этот параметр управляет тем, учитывается ли память highmem при троттлинге
«грязных» писателей. По умолчанию это не так, что означает, что
«загрязнять» можно только тот объём памяти, который непосредственно виден/используется ядром.
В результате на системах с большим объёмом памяти и
по сути исчерпанной lowmem писатели могут троттлиться слишком рано, и
потоковые записи могут стать очень медленными.

Изменение значения на ненулевое позволило бы «загрязнять» больше памяти
и, таким образом, позволить писателям записывать больше данных, которые можно более эффективно
сбрасывать в хранилище. Обратите внимание, что это также сопряжено с риском преждевременного
срабатывания OOM-киллера, поскольку некоторые писатели (например, прямые записи на блочное устройство) могут
использовать только память lowmem и могут заполнить её «грязными» данными без
какого-либо троттлинга.


hugetlb_shm_group
=================

hugetlb_shm_group содержит идентификатор группы, которой разрешено создавать сегмент
разделяемой памяти SysV с использованием страницы hugetlb.


legacy_va_layout
================

Если ненулевое, этот sysctl отключает новую 32-битную раскладку mmap — ядро
будет использовать устаревшую (2.4) раскладку для всех процессов.


lowmem_reserve_ratio
====================

Для некоторых специализированных нагрузок на машинах с highmem опасно
позволять ядру выделять память процесса из зоны «lowmem».
Это связано с тем, что такая память затем может быть закреплена (pinned) через системный вызов
mlock() или из-за недоступности пространства подкачки (swapspace).

А на больших машинах с highmem такая нехватка пригодной для возврата памяти lowmem
может быть фатальной.

Поэтому аллокатор страниц Linux имеет механизм, который предотвращает выделения,
которые *могли бы* использовать highmem, от чрезмерного использования lowmem. Это означает, что
определённый объём lowmem защищается от возможности
захвата в закреплённую пользовательскую память.

(Тот же аргумент применим к старой области ISA DMA размером 16 мегабайт. Этот
механизм также защитит эту область от выделений, которые могли бы использовать
highmem или lowmem).

Параметр `lowmem_reserve_ratio` определяет, насколько агрессивно ядро
защищает эти нижние зоны.

Если у вас есть машина, которая использует highmem или ISA DMA, и ваши
приложения используют mlock(), или если вы работаете без подкачки, то
вам, вероятно, следует изменить настройку lowmem_reserve_ratio.

lowmem_reserve_ratio — это массив. Вы можете увидеть его, прочитав этот файл::

	% cat /proc/sys/vm/lowmem_reserve_ratio
	256     256     32

Но эти значения не используются напрямую. Ядро вычисляет из них количество защитных
страниц для каждой зоны. Они показываются в виде массива защитных страниц
в /proc/zoneinfo, как показано ниже. (Это пример для машины x86-64).
У каждой зоны есть массив защитных страниц вроде этого::

  Node 0, zone      DMA
    pages free     1355
          min      3
          low      3
          high     4
	:
	:
      numa_other   0
          protection: (0, 2004, 2004, 2004)
	^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    pagesets
      cpu: 0 pcp: 0
          :

Эти защиты добавляются к оценке, чтобы решить, следует ли использовать эту зону
для выделения страниц или её следует подвергнуть возврату.

В этом примере, если для этой зоны DMA требуются обычные (normal) страницы (index=2) и
для водяного знака используется watermark[WMARK_HIGH], ядро решает, что эту зону
не следует использовать, поскольку pages_free(1355) меньше, чем watermark + protection[2]
(4 + 2004 = 2008). Если бы это значение protection было равно 0, эта зона использовалась бы для
обычного запроса страницы. Если же требуется зона DMA(index=0), используется protection[0]
(=0).

protection[j] зоны zone[i] вычисляется по следующему выражению::

  (i < j):
    zone[i]->protection[j]
    = (total sums of managed_pages from zone[i+1] to zone[j] on the node)
      / lowmem_reserve_ratio[i];
  (i = j):
     (should not be protected. = 0;
  (i > j):
     (not necessary, but looks 0)

Значения lowmem_reserve_ratio[i] по умолчанию равны

    === ====================================
    256 (if zone[i] means DMA or DMA32 zone)
    32  (others)
    === ====================================

Как видно из выражения выше, они являются обратными числами отношения.
256 означает 1/256. Количество защитных страниц составляет около «0.39%» от общего числа управляемых
страниц вышестоящих зон на узле.

Если вы хотите защитить больше страниц, эффективны меньшие значения.
Минимальное значение — 1 (1/1 -> 100%). Значение меньше 1 полностью
отключает защиту страниц.


max_map_count
=============

Этот файл содержит максимальное число областей карты памяти, которое может иметь
процесс. Области карты памяти используются как побочный эффект вызова
malloc, напрямую через mmap, mprotect и madvise, а также при загрузке
разделяемых библиотек.

Хотя большинству приложений нужно менее тысячи карт, определённые
программы, особенно отладчики malloc, могут потреблять их в большом количестве,
например, до одной-двух карт на каждое выделение.

Значение по умолчанию — 65530.


mem_profiling
==============

Включить профилирование памяти (когда CONFIG_MEM_ALLOC_PROFILING=y)

1: Включить профилирование памяти.

0: Отключить профилирование памяти.

Включение профилирования памяти вносит небольшие накладные расходы на производительность для всех
выделений памяти.

Значение по умолчанию зависит от CONFIG_MEM_ALLOC_PROFILING_ENABLED_BY_DEFAULT.

При CONFIG_MEM_ALLOC_PROFILING_DEBUG=y этот элемент управления доступен только для чтения, чтобы избежать
предупреждений, порождаемых выделениями, сделанными при отключённом профилировании и
освобождёнными при включённом.


memory_failure_early_kill
=========================

Управляет тем, как завершать процессы, когда неисправимая ошибка памяти (обычно
2-битная ошибка в модуле памяти) обнаруживается в фоновом режиме оборудованием
и не может быть обработана ядром. В некоторых случаях (например, когда страница
всё ещё имеет действительную копию на диске) ядро обработает сбой
прозрачно, не затрагивая никакие приложения. Но если нет
другой актуальной копии данных, оно завершит процесс, чтобы предотвратить распространение любых
повреждений данных.

1: Завершать все процессы, у которых отображена (mapped) повреждённая и не перезагружаемая страница,
как только повреждение обнаружено. Обратите внимание, что это не поддерживается
для нескольких типов страниц, таких как данные, выделенные ядром внутренне, или
swap cache, но работает для большинства пользовательских страниц.

0: Только отменить отображение (unmap) повреждённой страницы из всех процессов и завершать только процесс,
который пытается к ней обратиться.

Завершение выполняется с помощью перехватываемого SIGBUS с BUS_MCEERR_AO, так что процессы могут
обработать это, если захотят.

Это активно только на архитектурах/платформах с продвинутой обработкой machine
check и зависит от возможностей оборудования.

Приложения могут переопределить эту настройку индивидуально с помощью prctl PR_MCE_KILL


memory_failure_recovery
=======================

Включить восстановление после сбоя памяти (когда поддерживается платформой)

1: Попытаться восстановиться.

0: Всегда выполнять panic при сбое памяти.


min_free_kbytes
===============

Используется, чтобы заставить VM Linux держать свободным минимальное число
килобайт. VM использует это число для вычисления значения
watermark[WMARK_MIN] для каждой зоны lowmem в системе.
Каждая зона lowmem получает количество зарезервированных свободных страниц
пропорционально своему размеру.

Некоторый минимальный объём памяти нужен для удовлетворения выделений PF_MEMALLOC;
если вы установите это значение ниже 1024KB, ваша система станет
незаметно неисправной и склонной к взаимоблокировкам (deadlock) под высокой нагрузкой.

Установка слишком высокого значения мгновенно вызовет OOM на вашей машине.


min_slab_ratio
==============

Доступно только на NUMA-ядрах.

Процент от общего числа страниц в каждой зоне. При возврате зоны (Zone reclaim,
происходит при переходе от локальной зоны) slab-объекты будут возвращаться, если более
этого процента страниц в зоне являются slab-страницами, пригодными для возврата.
Это гарантирует, что рост slab остаётся под контролем даже в NUMA-
системах, которые редко выполняют глобальный возврат.

По умолчанию — 5 процентов.

Обратите внимание, что возврат slab запускается на уровне отдельной зоны / узла.
Процесс возврата slab-памяти в настоящее время не привязан к конкретному узлу
и может быть небыстрым.


min_unmapped_ratio
==================

Доступно только на NUMA-ядрах.

Это процент от общего числа страниц в каждой зоне. Возврат зоны (Zone reclaim) будет
происходить только если более этого процента страниц находятся в состоянии, которое
zone_reclaim_mode позволяет возвращать.

Если значение zone_reclaim_mode имеет установленный по OR бит 4, то этот процент сравнивается
со всеми неотображёнными (unmapped) страницами с файловой подложкой, включая страницы swapcache и файлы tmpfs.
Иначе учитываются только неотображённые страницы с подложкой из обычных файлов, но не из файлов tmpfs
и подобных.

По умолчанию — 1 процент.


mmap_min_addr
=============

Этот файл указывает объём адресного пространства, для которого пользовательскому процессу будет
запрещено выполнять mmap. Поскольку ошибки разыменования null в ядре могли бы
случайно опираться на информацию в первых нескольких страницах
памяти, процессам пространства пользователя не следует разрешать запись в них. По
умолчанию это значение установлено в 0, и модуль безопасности не будет применять никаких защит.
Установка этого значения во что-то вроде 64k позволит
подавляющему большинству приложений работать корректно и обеспечит эшелонированную защиту
от будущих потенциальных ошибок ядра.


mmap_rnd_bits
=============

Это значение можно использовать для выбора числа бит, используемых для
определения случайного смещения относительно базового адреса областей vma,
получающихся в результате выделений mmap, на архитектурах, которые поддерживают
настройку рандомизации адресного пространства. Это значение будет ограничено
минимальным и максимальным поддерживаемыми архитектурой значениями.

Это значение можно изменить после загрузки с помощью
параметра /proc/sys/vm/mmap_rnd_bits


mmap_rnd_compat_bits
====================

Это значение можно использовать для выбора числа бит, используемых для
определения случайного смещения относительно базового адреса областей vma,
получающихся в результате выделений mmap для приложений, запущенных в
режиме совместимости, на архитектурах, которые поддерживают настройку рандомизации

Это значение можно изменить после загрузки с помощью

movable_gigantic_pages
======================

Этот параметр управляет тем, можно ли выделять гигантские (gigantic) страницы из
ZONE_MOVABLE. Если установлено в ненулевое значение, гигантские страницы можно выделять
из ZONE_MOVABLE. Память ZONE_MOVABLE может быть создана через параметр загрузки
ядра `kernelcore` или через memory hotplug, как описано в
Documentation/admin-guide/mm/memory-hotplug.rst.

Поддержка может зависеть от конкретной архитектуры.

Обратите внимание, что использование гигантских страниц ZONE_MOVABLE делает memory hotremove ненадёжным.

Операции memory hot-remove будут блокироваться бесконечно, пока администратор не зарезервирует
достаточное количество гигантских страниц для обслуживания запросов миграции, связанных с
процессом отключения (offlining) памяти. Поскольку резервирование гигантских страниц HugeTLB — это ручной
процесс (через интерфейсы `nodeN/hugepages/.../nr_hugepages`), это может быть не
очевидно при простой попытке отключить блок памяти.

Кроме того, поскольку несколько гигантских страниц могут быть зарезервированы в одном блоке,
может показаться, что гигантские страницы доступны для миграции, тогда как в действительности
они находятся в процессе удаления. Например, если `memoryN` содержит
две гигантские страницы, одна зарезервирована, а другая выделена, и администратор пытается
отключить этот блок, эта операция может зависнуть бесконечно, если только другая
зарезервированная гигантская страница не доступна в другом блоке `memoryM`.

hugetlb_optimize_vmemmap
========================

Этот регулятор недоступен, когда размер 'struct page' (структуры, определённой
в include/linux/mm_types.h) не является степенью двойки (необычная конфигурация системы могла бы
привести к этому).

Включить (установить в 1) или отключить (установить в 0) HugeTLB Vmemmap Optimization (HVO).

После включения vmemmap-страницы последующего выделения страниц HugeTLB из
buddy-аллокатора будут оптимизированы (7 страниц на 2MB-страницу HugeTLB и 4095 страниц
на 1GB-страницу HugeTLB), тогда как уже выделенные страницы HugeTLB не будут
оптимизированы. Когда эти оптимизированные страницы HugeTLB освобождаются из пула HugeTLB
в buddy-аллокатор, vmemmap-страницы, представляющие этот диапазон, нужно
переотобразить заново, а vmemmap-страницы, отброшенные ранее, нужно перевыделить
заново. Если ваш сценарий использования таков, что страницы HugeTLB выделяются «на лету» (например,
никогда явно не выделяя страницы HugeTLB через 'nr_hugepages', а только задавая
'nr_overcommit_hugepages', и эти страницы HugeTLB сверх лимита выделяются «на
лету») вместо извлечения из пула HugeTLB, вам следует взвесить
выгоды от экономии памяти против больших накладных расходов (~2x медленнее, чем раньше)
на выделение или освобождение страниц HugeTLB между пулом HugeTLB и buddy-
аллокатором. Ещё одно поведение, на которое стоит обратить внимание: если система находится под сильным
давлением на память, это может помешать пользователю освобождать страницы HugeTLB из пула HugeTLB
в buddy-аллокатор, поскольку выделение vmemmap-страниц может
завершиться неудачей, и вам придётся повторить попытку позже, если ваша система столкнётся с этой ситуацией.

После отключения vmemmap-страницы последующего выделения страниц HugeTLB из
buddy-аллокатора не будут оптимизироваться, что означает, что дополнительные накладные расходы при выделении
из buddy-аллокатора исчезают, тогда как уже оптимизированные страницы HugeTLB
не будут затронуты. Если вы хотите убедиться, что нет оптимизированных страниц HugeTLB,
вы можете сначала установить "nr_hugepages" в 0, а затем отключить это. Обратите внимание, что
запись 0 в nr_hugepages превратит любые «используемые» (in use) страницы HugeTLB в избыточные (surplus)
страницы. Так что эти избыточные страницы остаются оптимизированными, пока они больше не
используются. Вам нужно будет дождаться освобождения этих избыточных страниц, прежде чем
в системе не останется оптимизированных страниц.

nr_trim_pages
=============

Доступно только на NOMMU-ядрах.

Это значение регулирует поведение обрезки (trimming) избыточных страниц для выровненных по степени 2

numa_zonelist_order
===================

Этот sysctl предназначен только для NUMA и является устаревшим. Любое значение, кроме
порядка Node, завершится неудачей!

То, «откуда выделяется память», управляется zonelist'ами.

(В этой документации для простоты объяснения игнорируется ZONE_HIGHMEM/ZONE_DMA32.
вы можете читать ZONE_DMA как ZONE_DMA32...)

В не-NUMA случае zonelist для GFP_KERNEL упорядочен следующим образом.
ZONE_NORMAL -> ZONE_DMA
Это означает, что запрос на выделение памяти для GFP_KERNEL
получит память из ZONE_DMA только когда ZONE_NORMAL недоступна.

В NUMA случае можно представить следующие 2 типа порядка.
Предположим NUMA с 2 узлами, а ниже — zonelist для GFP_KERNEL узла Node(0)::

  (A) Node(0) ZONE_NORMAL -> Node(0) ZONE_DMA -> Node(1) ZONE_NORMAL
  (B) Node(0) ZONE_NORMAL -> Node(1) ZONE_NORMAL -> Node(0) ZONE_DMA.

Тип(A) предлагает наилучшую локальность для процессов на Node(0), но ZONE_DMA
будет использоваться до исчерпания ZONE_NORMAL. Это увеличивает вероятность
нехватки памяти (out-of-memory, OOM) в ZONE_DMA, поскольку ZONE_DMA обычно мала.

Тип(B) не может предложить наилучшую локальность, но более устойчив к OOM
зоны DMA.

Тип(A) называется порядком "Node". Тип(B) — это порядок "Zone".

«Порядок Node» упорядочивает zonelist'ы по узлу, затем по зоне внутри каждого узла.
Укажите "[Nn]ode" для порядка node

«Порядок Zone» упорядочивает zonelist'ы по типу зоны, затем по узлу внутри каждой
зоны. Укажите "[Zz]one" для порядка zone.

Укажите "[Dd]efault", чтобы запросить автоматическую конфигурацию.

На 32-битных системах зону Normal нужно сохранять для выделений, доступных
ядру, поэтому будет выбран порядок "zone".

На 64-битных системах устройства, требующие DMA32/DMA, относительно редки, поэтому будет выбран порядок "node".

Порядок default рекомендуется, если только это не вызывает проблем для вашей
системы/приложения.


oom_dump_tasks
==============

Включает создание общесистемного дампа задач (исключая потоки ядра)
при выполнении ядром OOM-killing и включает такую информацию, как
pid, uid, tgid, размер vm, rss, pgtables_bytes, swapents, оценку oom_score_adj
и имя. Это помогает определить, почему был вызван OOM-киллер,
выявить вышедшую из-под контроля задачу, которая его вызвала, и определить, почему
OOM-киллер выбрал именно ту задачу, которую завершил.

Если это установлено в ноль, эта информация подавляется. На очень
больших системах с тысячами задач может быть нецелесообразно дампить
информацию о состоянии памяти для каждой из них. Такие системы не должны
вынужденно нести штраф производительности в условиях OOM, когда
эта информация может быть нежелательна.

Если это установлено в ненулевое значение, эта информация показывается всякий раз, когда
OOM-киллер действительно завершает прожорливую по памяти задачу.

Значение по умолчанию — 1 (включено).


oom_kill_allocating_task
========================

Это включает или отключает завершение задачи, вызвавшей OOM, в
ситуациях нехватки памяти (out-of-memory).

Если это установлено в ноль, OOM-киллер просканирует весь
список задач и выберет задачу для завершения на основе эвристик. Обычно это
выбирает вышедшую из-под контроля прожорливую по памяти задачу, которая при завершении освобождает большой объём
памяти.

Если это установлено в ненулевое значение, OOM-киллер просто завершает задачу, которая
вызвала состояние нехватки памяти. Это позволяет избежать дорогостоящего
сканирования списка задач.

Если выбран panic_on_oom, он имеет приоритет над любым значением,
используемым в oom_kill_allocating_task.

Значение по умолчанию — 0.


overcommit_kbytes
=================

Когда overcommit_memory установлено в 2, выделенное (committed) адресное пространство не
может превышать swap плюс этот объём физической RAM. См. ниже.

Примечание: overcommit_kbytes — это аналог overcommit_ratio. Одновременно может быть
задан только один из них. Установка одного отключает другой (который
затем при чтении отображается как 0).


overcommit_memory
=================

Это значение содержит флаг, который включает overcommit (перерасход) памяти.

Когда этот флаг равен 0, ядро сравнивает размер запроса памяти из пространства пользователя
с общим объёмом памяти плюс swap и отклоняет очевидные перерасходы (overcommit).

Когда этот флаг равен 1, ядро делает вид, что памяти всегда
достаточно, пока она реально не закончится.

Когда этот флаг равен 2, ядро использует политику «never overcommit»,
которая пытается предотвратить любой перерасход памяти.
Обратите внимание, что user_reserve_kbytes влияет на эту политику.

Эта возможность может быть очень полезной, поскольку существует множество
программ, которые выделяют через malloc() огромные объёмы памяти «на всякий случай»
и используют лишь малую её часть.

Значение по умолчанию — 0.

Дополнительную информацию см. в Documentation/mm/overcommit-accounting.rst и
mm/util.c::__vm_enough_memory().


overcommit_ratio
================

Когда overcommit_memory установлено в 2, выделенное (committed) адресное
пространство не может превышать swap плюс этот процент
физической RAM. См. выше.


page-cluster
============

page-cluster управляет числом страниц, до которого последовательные страницы
читаются из swap за одну попытку. Это аналог readahead страничного кэша
для swap.
Упомянутая последовательность — не в терминах виртуальных/физических адресов,
а последовательность в пространстве swap — то есть они были выгружены вместе.

Это логарифмическое значение: установка его в ноль означает «1 страница», установка
его в 1 означает «2 страницы», установка его в 2 означает «4 страницы» и т. д.
Ноль полностью отключает swap readahead.

Значение по умолчанию — три (восемь страниц за раз). Может быть некоторая
небольшая выгода от настройки этого значения на другое, если ваша нагрузка
интенсивно использует swap.

Меньшие значения означают меньшие задержки для начальных отказов, но в то же время
дополнительные отказы и задержки I/O для последующих отказов, если бы они были частью
тех последовательных страниц, которые принёс бы readahead.


page_lock_unfairness
====================

Это значение определяет, сколько раз блокировка страницы может быть
украдена из-под ожидающего. После того как блокировка украдена столько раз,
сколько указано в этом файле (по умолчанию 5), будут применяться семантики
«справедливой передачи блокировки» (fair lock handoff), и ожидающий будет пробуждён только если блокировка может быть взята.

panic_on_oom
============

Это включает или отключает возможность panic при нехватке памяти (out-of-memory).

Если это установлено в 0, ядро завершит какой-нибудь вышедший из-под контроля процесс
с помощью так называемого oom_killer. Обычно oom_killer может завершить вышедшие из-под контроля процессы, и
система выживет.

Если это установлено в 1, ядро выполняет panic при возникновении нехватки памяти.
Однако, если процесс ограничивает использование узлов через mempolicy/cpuset,
и эти узлы достигают состояния исчерпания памяти, один процесс
может быть завершён oom-киллером. В этом случае panic не происходит.
Потому что память других узлов может быть свободна. Это означает, что общее состояние системы
может быть ещё не фатальным.

Если это установлено в 2, ядро выполняет panic принудительно даже при
вышеупомянутом. Даже если oom происходит в memory cgroup, panic выполняет вся
система.

Значение по умолчанию — 0.

1 и 2 предназначены для отказоустойчивости кластеризации. Выберите одно из них
в соответствии с вашей политикой отказоустойчивости.

panic_on_oom=2+kdump даёт вам очень мощный инструмент для исследования
того, почему происходит oom. Вы можете получить снимок (snapshot).


percpu_pagelist_high_fraction
=============================

Это доля страниц в каждой зоне, которые могут храниться в
per-cpu списках страниц. Это верхняя граница, которая делится в зависимости
от числа подключённых (online) CPU. Минимальное значение для этого — 8, что означает,
что мы не позволяем хранить более 1/8 страниц каждой зоны
в per-cpu списках страниц. Этот элемент изменяет только значение «горячих» per-cpu
списков страниц. Пользователь может указать число вроде 100, чтобы выделить 1/100
каждой зоны между per-cpu списками.

Значение batch каждого per-cpu списка страниц остаётся неизменным независимо от
значения high fraction, поэтому задержки выделения не затрагиваются.

Начальное значение — ноль. Ядро использует это значение для установки отметки high pcp->high
на основе низкого водяного знака зоны и числа локальных
подключённых (online) CPU. Если пользователь записывает '0' в этот sysctl, оно вернётся к
этому поведению по умолчанию.


stat_interval
=============

Интервал времени, через который обновляется статистика vm. По умолчанию
— 1 секунда.


stat_refresh
============

Любое чтение или запись (только root'ом) сбрасывает всю per-cpu статистику vm
в её глобальные суммы для более точных отчётов при тестировании,
например, cat /proc/sys/vm/stat_refresh /proc/meminfo

В качестве побочного эффекта это также проверяет наличие отрицательных сумм (в других местах сообщаемых
как 0) и «завершается неудачей» с EINVAL, если такие найдены, с предупреждением в dmesg.
(На момент написания известно, что несколько статистик иногда оказываются отрицательными,
без вредных последствий: ошибки и предупреждения по этим статистикам подавляются.)


numa_stat
=========

Этот интерфейс позволяет конфигурировать статистику numa во время выполнения.

Когда производительность выделения страниц становится узким местом и вы можете допустить
некоторую возможную поломку инструментов и сниженную точность счётчиков numa, вы можете
сделать::

	echo 0 > /proc/sys/vm/numa_stat

Когда производительность выделения страниц не является узким местом и вы хотите, чтобы весь
инструментарий работал, вы можете сделать::

	echo 1 > /proc/sys/vm/numa_stat


swappiness
==========

Этот элемент управления используется для определения примерной относительной стоимости IO при подкачке (swapping)
и страничной подкачке файловой системы, в виде значения от 0 до 200. При 100 VM
предполагает равную стоимость IO и, таким образом, будет применять давление на память к страничному
кэшу и страницам с подложкой из swap одинаково; меньшие значения означают более
дорогой IO swap, большие значения указывают на более дешёвый.

Имейте в виду, что шаблоны IO файловой системы под давлением на память обычно
более эффективны, чем случайный IO swap. Оптимальное значение потребует
экспериментов и также будет зависеть от нагрузки.

Значение по умолчанию — 60.

Для swap в памяти, такого как zram или zswap, а также для гибридных конфигураций, у которых
swap находится на более быстрых устройствах, чем файловая система, можно рассмотреть значения
больше 100. Например, если случайный IO к устройству swap
в среднем в 2 раза быстрее, чем IO от файловой системы, swappiness должно
быть 133 (x + 2x = 200, 2x = 133.33).

При 0 ядро не будет инициировать swap, пока количество свободных страниц и
страниц с файловой подложкой не станет меньше высокого водяного знака в зоне.


unprivileged_userfaultfd
========================

Этот флаг управляет режимом, в котором непривилегированные пользователи могут использовать
системные вызовы userfaultfd. Установите это в 0, чтобы ограничить непривилегированных пользователей
обработкой страничных отказов только в пользовательском режиме. В этом случае пользователи без
SYS_CAP_PTRACE должны передать UFFD_USER_MODE_ONLY, чтобы userfaultfd
выполнился успешно. Запрет использования userfaultfd для обработки отказов из режима ядра
может сделать определённые уязвимости более трудными для эксплуатации.

Установите это в 1, чтобы разрешить непривилегированным пользователям использовать системные вызовы
userfaultfd без каких-либо ограничений.

Значение по умолчанию — 0.

Другой способ управления правами для userfaultfd — использовать
/dev/userfaultfd вместо userfaultfd(2). См.
Documentation/admin-guide/mm/userfaultfd.rst.

user_reserve_kbytes
===================

Когда overcommit_memory установлено в 2, режим «never overcommit», резервируется
min(3% от размера текущего процесса, user_reserve_kbytes) свободной памяти.
Это предназначено для предотвращения запуска пользователем единственного прожорливого по памяти
процесса так, что он не сможет восстановиться (завершить прожорливый процесс).

Значение user_reserve_kbytes по умолчанию равно min(3% от размера текущего процесса, 128MB).

Если это уменьшено до нуля, то пользователю будет разрешено выделить
всю свободную память одним процессом, минус admin_reserve_kbytes.
Любые последующие попытки выполнить команду приведут к
"fork: Cannot allocate memory".

Изменение этого значения вступает в силу при каждом запросе памяти приложением.


vfs_cache_pressure
==================

Это процентное значение управляет склонностью ядра возвращать
память, которая используется для кэширования объектов каталогов и inode.

При значении по умолчанию vfs_cache_pressure=vfs_cache_pressure_denom ядро
будет пытаться возвращать dentry и inode со «справедливой» скоростью по отношению к
возврату pagecache и swapcache. Уменьшение vfs_cache_pressure заставляет
ядро предпочитать удерживать кэши dentry и inode. Когда vfs_cache_pressure=0,
ядро никогда не будет возвращать dentry и inode из-за давления на память, и
это может легко привести к состояниям нехватки памяти. Увеличение vfs_cache_pressure
сверх vfs_cache_pressure_denom заставляет ядро предпочитать возврат dentry
и inode.

Увеличение vfs_cache_pressure значительно сверх vfs_cache_pressure_denom может
оказать негативное влияние на производительность. Коду возврата нужно брать различные блокировки, чтобы
найти освобождаемые объекты каталогов и inode. Когда vfs_cache_pressure равно
(10 * vfs_cache_pressure_denom), он будет искать в десять раз больше освобождаемых
объектов, чем существует.

Примечание: эта настройка всегда должна использоваться вместе с vfs_cache_pressure_denom.

vfs_cache_pressure_denom
========================

По умолчанию 100 (минимально допустимое значение). Требует соответствующей
настройки vfs_cache_pressure, чтобы вступить в силу.

watermark_boost_factor
======================

Этот коэффициент управляет уровнем возврата (reclaim), когда память фрагментируется.
Он определяет процент от высокого водяного знака зоны, который будет
возвращён, если страницы разной мобильности смешиваются внутри pageblock'ов.
Цель состоит в том, чтобы у уплотнения было меньше работы в будущем и чтобы
повысить вероятность успеха будущих выделений высокого порядка, таких как выделения SLUB,
THP и страницы hugetlbfs.

Чтобы это имело смысл по отношению к параметру watermark_scale_factor,
единица измерения — в долях от 10000. Значение по умолчанию,
равное 15000, означает, что до 150% высокого водяного знака будет возвращено в
случае смешивания pageblock'а из-за фрагментации. Уровень возврата
определяется числом событий фрагментации, произошедших в
недавнем прошлом. Если это значение меньше pageblock'а, то будет возвращено
страниц на величину одного pageblock'а (например, 2MB на 64-битном x86). Коэффициент усиления
(boost factor), равный 0, отключает эту возможность.


watermark_scale_factor
======================

Этот коэффициент управляет агрессивностью kswapd. Он определяет
объём памяти, остающийся в узле/системе перед тем, как kswapd будет разбужен, и
сколько памяти должно быть свободно, прежде чем kswapd снова уснёт.

Единица измерения — в долях от 10000. Значение по умолчанию, равное 10, означает, что
расстояния между водяными знаками составляют 0.1% от доступной памяти в
узле/системе. Максимальное значение — 3000, или 30% памяти.

Высокая частота входа потоков в прямой возврат (allocstall) или преждевременного засыпания
kswapd (kswapd_low_wmark_hit_quickly) может указывать на то,
что число свободных страниц, которое kswapd поддерживает по соображениям задержки,
слишком мало для всплесков выделений, происходящих в системе. Этот регулятор
тогда можно использовать для соответствующей настройки агрессивности kswapd.


zone_reclaim_mode
=================

Zone_reclaim_mode позволяет задать более или менее агрессивные подходы к
возврату памяти, когда в зоне заканчивается память. Если это установлено в ноль, то возврат
зоны не происходит. Выделения будут удовлетворяться из других зон / узлов
в системе.

Это значение, объединённое по OR из

=	===================================
1	Zone reclaim on
2	Zone reclaim writes dirty pages out
4	Zone reclaim swaps pages
=	===================================

zone_reclaim_mode по умолчанию отключён. Для файловых серверов или нагрузок,
которые выигрывают от кэширования их данных, zone_reclaim_mode следует
оставить отключённым, поскольку эффект кэширования, вероятно, важнее, чем
локальность данных.

Рассмотрите включение одного или нескольких битов режима zone_reclaim, если известно, что
нагрузка разбита так, что каждый раздел помещается в пределах узла NUMA
и что доступ к удалённой памяти вызвал бы измеримое снижение
производительности. Аллокатор страниц предпримет дополнительные действия перед
выделением страниц вне узла.

Разрешение возврату зоны записывать страницы останавливает процессы, которые
записывают большие объёмы данных, от «загрязнения» страниц на других узлах. Возврат
зоны будет записывать «грязные» страницы, если зона заполняется, и таким образом эффективно
троттлит процесс. Это может снизить производительность одного процесса,
поскольку он больше не может использовать всю память системы для буферизации исходящих записей,
но это сохраняет память на других узлах, так что производительность
других процессов, работающих на других узлах, не будет затронута.

Разрешение обычного swap эффективно ограничивает выделения локальным
узлом, если только это явно не переопределено политиками памяти или конфигурациями
cpuset.
