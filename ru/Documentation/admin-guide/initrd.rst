Использование начального RAM-диска (initrd)
===========================================

Написано в 1996, 2000 годах Werner Almesberger <werner.almesberger@epfl.ch> и
Hans Lermen <lermen@fgan.de>


initrd предоставляет возможность загрузки RAM-диска загрузчиком (boot loader).
Этот RAM-диск затем может быть смонтирован как корневая файловая система, и из
него могут запускаться программы. После этого новая корневая файловая система
может быть смонтирована с другого устройства. Прежний корень (из initrd) затем
перемещается в каталог и впоследствии может быть размонтирован.

initrd в основном предназначен для того, чтобы запуск системы происходил в две
фазы: на первой ядро поднимается с минимальным набором встроенных драйверов, а на
второй из initrd загружаются дополнительные модули.

Этот документ даёт краткий обзор использования initrd. Более подробное обсуждение
процесса загрузки можно найти в [#f1]_.


Принцип работы
--------------

При использовании initrd система обычно загружается следующим образом:

  1) загрузчик загружает ядро и начальный RAM-диск
  2) ядро преобразует initrd в «обычный» RAM-диск и
     освобождает память, использованную initrd
  3) если корневое устройство не ``/dev/ram0``, выполняется старая (устаревшая)
     процедура change_root. См. раздел «Устаревший механизм смены корня»
     ниже.
  4) монтируется корневое устройство. Если это ``/dev/ram0``, то образ initrd
     монтируется как корень
  5) выполняется /sbin/init (это может быть любой допустимый исполняемый файл,
     включая сценарии оболочки; он запускается с uid 0 и может делать практически
     всё, что может init).
  6) init монтирует «настоящую» корневую файловую систему
  7) init помещает корневую файловую систему в корневой каталог с помощью
     системного вызова pivot_root
  8) init запускает ``/sbin/init`` на новой корневой файловой системе, выполняя
     обычную последовательность загрузки
  9) файловая система initrd удаляется

Обратите внимание, что смена корневого каталога не подразумевает его
размонтирования. Поэтому в ходе этой процедуры можно оставлять процессы
работающими на initrd. Также обратите внимание, что файловые системы,
смонтированные под initrd, остаются доступными.


Параметры командной строки загрузки
-----------------------------------

initrd добавляет следующие новые параметры::

  initrd=<path>    (e.g. LOADLIN)

    Loads the specified file as the initial RAM disk. When using LILO, you
    have to specify the RAM disk image file in /etc/lilo.conf, using the
    INITRD configuration variable.

  noinitrd

    initrd data is preserved but it is not converted to a RAM disk and
    the "normal" root file system is mounted. initrd data can be read
    from /dev/initrd. Note that the data in initrd can have any structure
    in this case and doesn't necessarily have to be a file system image.
    This option is used mainly for debugging.

    Note: /dev/initrd is read-only and it can only be used once. As soon
    as the last process has closed it, all data is freed and /dev/initrd
    can't be opened anymore.

  root=/dev/ram0

    initrd is mounted as root, and the normal boot procedure is followed,
    with the RAM disk mounted as root.

Сжатые cpio-образы
------------------

Современные ядра поддерживают наполнение ramdisk из сжатого cpio-архива. На таких
системах создание образа ramdisk не требует использования специальных блочных
устройств или loopback'ов; достаточно создать на диске каталог с нужным
содержимым initrd, перейти (cd) в этот каталог и выполнить (в качестве примера)::

	find . | cpio --quiet -H newc -o | gzip -9 -n > /boot/imagefile.img

Просмотр содержимого существующего файла образа столь же прост::

	mkdir /tmp/imagefile
	cd /tmp/imagefile
	gzip -cd /boot/imagefile.img | cpio -imd --quiet

Установка
---------

Сначала на «обычной» корневой файловой системе необходимо создать каталог для
файловой системы initrd, например::

	# mkdir /initrd

Имя не имеет значения. Более подробные сведения можно найти на man-странице
:manpage:`pivot_root(2)`.

Если корневая файловая система создаётся в ходе процедуры загрузки (т. е. если
вы собираете установочную дискету), процедура создания корневой файловой системы
должна создавать каталог ``/initrd``.

Если в некоторых случаях initrd не будет монтироваться, его содержимое всё равно
остаётся доступным, если было создано следующее устройство::

	# mknod /dev/initrd b 1 250
	# chmod 400 /dev/initrd

Во-вторых, ядро должно быть скомпилировано с поддержкой RAM-диска и с включённой
поддержкой начального RAM-диска. Кроме того, в ядро должны быть встроены как
минимум все компоненты, необходимые для выполнения программ из initrd (например,
формат исполняемых файлов и файловая система).

В-третьих, необходимо создать образ RAM-диска. Это делается путём создания
файловой системы на блочном устройстве, копирования на неё нужных файлов и
последующего копирования содержимого блочного устройства в файл initrd. В
современных ядрах для этого подходят как минимум три типа устройств:

 - дискета (работает везде, но мучительно медленно)
 - RAM-диск (быстро, но выделяет физическую память)
 - loopback-устройство (самое элегантное решение)

Опишем метод с loopback-устройством:

 1) убедитесь, что loopback-устройства настроены в ядре
 2) создайте пустую файловую систему подходящего размера, например::

	# dd if=/dev/zero of=initrd bs=300k count=1
	# mke2fs -F -m0 initrd

    (если место критично, вы можете использовать Minix FS вместо Ext2)
 3) смонтируйте файловую систему, например::

	# mount -t ext2 -o loop initrd /mnt

 4) создайте устройство консоли::

    # mkdir /mnt/dev
    # mknod /mnt/dev/console c 5 1

 5) скопируйте все файлы, необходимые для корректного использования среды
    initrd. Не забудьте самый важный файл — ``/sbin/init``

    .. note:: Права доступа ``/sbin/init`` должны включать «x» (выполнение).

 6) корректную работу среды initrd зачастую можно проверить даже без
    перезагрузки с помощью команды::

	# chroot /mnt /sbin/init

    Это, конечно, ограничено такими initrd, которые не вмешиваются в общее
    состояние системы (например, переконфигурируя сетевые интерфейсы,
    перезаписывая смонтированные устройства, пытаясь запустить уже работающие
    демоны и т. п. Заметьте, однако, что в такой chroot-окружённой среде initrd
    обычно можно использовать pivot_root.)
 7) размонтируйте файловую систему::

	# umount /mnt

 8) теперь initrd находится в файле «initrd». При желании его можно сжать::

	# gzip -9 initrd

Для экспериментов с initrd вы можете взять спасательную дискету и просто добавить
символическую ссылку с ``/sbin/init`` на ``/bin/sh``. В качестве альтернативы вы
можете попробовать экспериментальное окружение newlib [#f2]_ для создания
небольшого initrd.

Наконец, вам нужно загрузить ядро и initrd. Почти все загрузчики Linux
поддерживают initrd. Поскольку процесс загрузки по-прежнему совместим со старым
механизмом, необходимо передать следующие параметры командной строки загрузки::

  root=/dev/ram0 rw

(rw необходимо только при записи в файловую систему initrd.)

С LOADLIN вы просто выполняете::

     LOADLIN <kernel> initrd=<disk_image>

например::

	LOADLIN C:\LINUX\BZIMAGE initrd=C:\LINUX\INITRD.GZ root=/dev/ram0 rw

С LILO вы добавляете параметр ``INITRD=<path>`` либо в глобальную секцию, либо в
секцию соответствующего ядра в ``/etc/lilo.conf`` и передаёте параметры с помощью
APPEND, например::

  image = /bzImage
    initrd = /boot/initrd.gz
    append = "root=/dev/ram0 rw"

и запускаете ``/sbin/lilo``

Для других загрузчиков обращайтесь к соответствующей документации.

Теперь вы можете загрузиться и наслаждаться использованием initrd.


Смена корневого устройства
--------------------------

Завершив свои обязанности, init обычно меняет корневое устройство и приступает к
запуску системы Linux на «настоящем» корневом устройстве.

Эта процедура включает следующие шаги:
 - монтирование новой корневой файловой системы
 - превращение её в корневую файловую систему
 - удаление всех обращений к старой (initrd) корневой файловой системе
 - размонтирование файловой системы initrd и освобождение RAM-диска

Смонтировать новую корневую файловую систему легко: её всего лишь нужно
смонтировать в каталог под текущим корнем. Пример::

	# mkdir /new-root
	# mount -o ro /dev/hda1 /new-root

Смена корня осуществляется с помощью системного вызова pivot_root, который также
доступен через утилиту ``pivot_root`` (см. man-страницу :manpage:`pivot_root(8)`;
``pivot_root`` распространяется в составе util-linux версии 2.10h или выше
[#f3]_). ``pivot_root`` перемещает текущий корень в каталог под новым корнем и
помещает на его место новый корень. Каталог для старого корня должен существовать
до вызова ``pivot_root``. Пример::

	# cd /new-root
	# mkdir initrd
	# pivot_root . initrd

Теперь процесс init всё ещё может обращаться к старому корню через свой
исполняемый файл, разделяемые библиотеки, стандартные ввод/вывод/ошибки и свой
текущий корневой каталог. Все эти ссылки сбрасываются следующей командой::

	# exec chroot . what-follows <dev/console >dev/console 2>&1

Здесь what-follows — это программа под новым корнем, например ``/sbin/init``.
Если новая корневая файловая система будет использоваться с udev и не имеет
действительного каталога ``/dev``, то перед вызовом chroot необходимо
инициализировать udev, чтобы предоставить ``/dev/console``.

Примечание: детали реализации pivot_root могут со временем меняться. Чтобы
обеспечить совместимость, следует соблюдать следующие моменты:

 - перед вызовом pivot_root текущий каталог вызывающего процесса должен указывать
   на новый корневой каталог
 - используйте . в качестве первого аргумента и *относительный* путь к каталогу
   для старого корня в качестве второго аргумента
 - программа chroot должна быть доступна под старым и под новым корнем
 - после этого выполните chroot в новый корень
 - используйте относительные пути для dev/console в команде exec

Теперь initrd можно размонтировать, а память, выделенную RAM-диском, можно
освободить::

	# umount /initrd
	# blockdev --flushbufs /dev/ram0

Также можно использовать initrd с корнем, смонтированным по NFS; подробности см.
на man-странице :manpage:`pivot_root(8)`.


Сценарии использования
----------------------

Основной мотивацией для реализации initrd было обеспечение модульной настройки
ядра при установке системы. Процедура работала бы следующим образом:

  1) система загружается с дискеты или другого носителя с минимальным ядром
     (например, с поддержкой RAM-дисков, initrd, a.out и Ext2 FS) и
     загружает initrd
  2) ``/sbin/init`` определяет, что необходимо для того, чтобы (1) смонтировать
     «настоящую» корневую FS (т. е. тип устройства, драйверы устройств, файловую
     систему) и (2) дистрибутивный носитель (например, CD-ROM, сеть, ленту, ...).
     Это можно сделать, спросив у пользователя, путём автоопределения или с
     помощью гибридного подхода.
  3) ``/sbin/init`` загружает необходимые модули ядра
  4) ``/sbin/init`` создаёт и наполняет корневую файловую систему (это пока не
     обязательно должна быть полноценно пригодная для использования система)
  5) ``/sbin/init`` вызывает ``pivot_root``, чтобы сменить корневую файловую
     систему, и запускает — через chroot — программу, которая продолжает
     установку
  6) устанавливается загрузчик
  7) загрузчик настраивается на загрузку initrd с тем набором модулей, который
     использовался для поднятия системы (например, ``/initrd`` можно изменить,
     затем размонтировать и, наконец, записать образ из ``/dev/ram0`` или
     ``/dev/rd/0`` в файл)
  8) теперь система загружаема, и можно выполнять дополнительные задачи установки

Ключевая роль initrd здесь — повторное использование конфигурационных данных во
время нормальной работы системы без необходимости применять раздутое «обобщённое»
ядро либо перекомпилировать или перекомпоновывать ядро.

Второй сценарий — установки, в которых Linux работает на системах с различными
аппаратными конфигурациями в рамках единого административного домена. В таких
случаях желательно генерировать лишь небольшой набор ядер (в идеале только одно) и
делать системно-зависимую часть конфигурационной информации как можно меньше. В
этом случае можно сгенерировать общий initrd со всеми необходимыми модулями. Тогда
различаться должны были бы только ``/sbin/init`` или читаемый им файл.

Третий сценарий — более удобные спасательные диски, поскольку такую информацию,
как расположение раздела с корневой FS, не нужно предоставлять во время загрузки;
загруженная из initrd система может вызвать дружественный к пользователю диалог, а
также может выполнить некоторые проверки корректности (или даже какую-либо форму
автоопределения).

Наконец, что не менее важно, распространители CD-ROM могут использовать его для
лучшей установки с CD, например, используя загрузочную дискету и подгружая больший
RAM-диск через initrd с CD; либо загружаясь через такой загрузчик, как
``LOADLIN``, или напрямую с CD-ROM, и загружая RAM-диск с CD без необходимости в
дискетах.


Устаревший механизм смены корня
-------------------------------

Следующий механизм использовался до появления pivot_root. Текущие ядра по-прежнему
поддерживают его, но вам *не* следует полагаться на его дальнейшую доступность.

Он работает путём монтирования «настоящего» корневого устройства (т. е. того,
которое задано с помощью rdev в образе ядра или с помощью root=... в командной
строке загрузки) в качестве корневой файловой системы при выходе linuxrc.
Файловая система initrd затем размонтируется или, если она всё ещё занята,
перемещается в каталог ``/initrd``, если такой каталог существует на новой
корневой файловой системе.

Чтобы использовать этот механизм, вам не нужно указывать параметры загрузки root,
init или rw. (Если они указаны, они повлияют на настоящую корневую файловую
систему, а не на среду initrd.)

Если /proc смонтирован, «настоящее» корневое устройство можно сменить изнутри
linuxrc, записав номер устройства новой корневой FS в специальный файл
/proc/sys/kernel/real-root-dev, например::

  # echo 0x301 >/proc/sys/kernel/real-root-dev

Обратите внимание, что этот механизм несовместим с NFS и подобными файловыми
системами.

Этот старый, устаревший механизм обычно называют ``change_root``, тогда как новый,
поддерживаемый механизм называют ``pivot_root``.


Смешанный механизм change_root и pivot_root
-------------------------------------------

Если вы не хотели использовать ``root=/dev/ram0`` для запуска механизма
pivot_root, вы можете создать в своём образе initrd оба файла — ``/linuxrc`` и
``/sbin/init``.

``/linuxrc`` содержал бы только следующее::

	#! /bin/sh
	mount -n -t proc proc /proc
	echo 0x0100 >/proc/sys/kernel/real-root-dev
	umount -n /proc

После выхода linuxrc ядро снова смонтирует ваш initrd как корень, на этот раз
выполняя ``/sbin/init``. И снова обязанностью этого init будет построить
правильное окружение (возможно, с использованием ``root= device``, переданного в
командной строке) перед окончательным запуском настоящего ``/sbin/init``.
