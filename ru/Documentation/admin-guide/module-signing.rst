========
Обзор
========

Механизм подписи модулей ядра криптографически подписывает модули во время их
установки и затем проверяет подпись при загрузке модуля. Это позволяет повысить
безопасность ядра, запрещая загрузку неподписанных модулей или модулей,
подписанных недействительным ключом. Подпись модулей повышает безопасность,
усложняя загрузку вредоносного модуля в ядро. Проверка подписи модулей
выполняется ядром, поэтому нет необходимости доверять каким-либо компонентам
пространства пользователя (userspace).

Этот механизм использует сертификаты стандарта X.509 ITU-T для кодирования
задействованных открытых ключей. Сами подписи не кодируются ни в каком
промышленном стандартном формате. Встроенный механизм в настоящее время
поддерживает только стандарты подписи открытым ключом RSA, NIST P-384 ECDSA и
NIST FIPS-204 ML-DSA (хотя он является подключаемым и позволяет использовать
другие). Для RSA и ECDSA возможными алгоритмами хеширования являются SHA-2 и
SHA-3 размеров 256, 384 и 512 (алгоритм выбирается на основе данных в подписи);
ML-DSA выполняет собственное хеширование, но допускается его использование с
хешем SHA512 для подписанных атрибутов.


==========================
Настройка подписи модулей
==========================

Механизм подписи модулей включается переходом в раздел
:menuselection:`Enable Loadable Module Support` конфигурации ядра и включением::

	CONFIG_MODULE_SIG	"Module signature verification"

Здесь доступен ряд параметров:

 (1) :menuselection:`Require modules to be validly signed`
     (``CONFIG_MODULE_SIG_FORCE``)

     Этот параметр определяет, как ядро должно обращаться с модулем, имеющим
     подпись, для которой ключ неизвестен, или с модулем, который не подписан.

     Если он выключен (то есть «разрешающий» режим), то модули, для которых ключ
     недоступен, и модули, которые не подписаны, разрешены, но ядро будет
     помечено как испорченное (tainted), а соответствующие модули будут помечены
     как испорченные и показаны с символом «E».

     Если он включён (то есть «ограничивающий» режим), будут загружаться только
     модули, имеющие действительную подпись, которая может быть проверена
     открытым ключом, находящимся во владении ядра. Все остальные модули будут
     приводить к ошибке.

     Независимо от значения этого параметра, если модуль имеет блок подписи,
     который не удаётся разобрать, он будет отвергнут сразу же.


 (2) :menuselection:`Automatically sign all modules`
     (``CONFIG_MODULE_SIG_ALL``)

     Если он включён, то модули будут автоматически подписываться на этапе
     modules_install сборки. Если он выключен, то модули должны быть подписаны
     вручную с помощью::

	scripts/sign-file


 (3) :menuselection:`Which hash algorithm should modules be signed with?`

     Этот параметр предлагает выбор, каким алгоритмом хеширования этап установки
     будет подписывать модули:

        =============================== ==========================================
	``CONFIG_MODULE_SIG_SHA256``	:menuselection:`Sign modules with SHA-256`
	``CONFIG_MODULE_SIG_SHA384``	:menuselection:`Sign modules with SHA-384`
	``CONFIG_MODULE_SIG_SHA512``	:menuselection:`Sign modules with SHA-512`
	``CONFIG_MODULE_SIG_SHA3_256``	:menuselection:`Sign modules with SHA3-256`
	``CONFIG_MODULE_SIG_SHA3_384``	:menuselection:`Sign modules with SHA3-384`
	``CONFIG_MODULE_SIG_SHA3_512``	:menuselection:`Sign modules with SHA3-512`
        =============================== ==========================================

     Выбранный здесь алгоритм также будет встроен в ядро (а не оформлен как
     модуль), чтобы подписи модулей, подписанных этим алгоритмом, можно было
     проверять, не вызывая зацикливания зависимостей.


 (4) :menuselection:`File name or PKCS#11 URI of module signing key`
     (``CONFIG_MODULE_SIG_KEY``)

     Установка этого параметра в значение, отличное от значения по умолчанию
     ``certs/signing_key.pem``, отключит автогенерацию ключей подписи и позволит
     подписывать модули ядра ключом по вашему выбору. Указанная строка должна
     идентифицировать файл, содержащий как закрытый ключ, так и соответствующий
     ему сертификат X.509 в формате PEM, или — на системах, где функционирует
     OpenSSL ENGINE_pkcs11 — PKCS#11 URI, как определено в RFC7512. В последнем
     случае PKCS#11 URI должен ссылаться как на сертификат, так и на закрытый
     ключ.

     Если PEM-файл, содержащий закрытый ключ, зашифрован, или если токен PKCS#11
     требует PIN-кода, его можно предоставить во время сборки с помощью
     переменной ``KBUILD_SIGN_PIN``.


 (5) :menuselection:`Additional X.509 keys for default system keyring`
     (``CONFIG_SYSTEM_TRUSTED_KEYS``)

     Этому параметру можно задать имя файла в формате PEM, содержащего
     дополнительные сертификаты, которые по умолчанию будут включены в системную
     связку ключей (keyring).

Обратите внимание, что включение подписи модулей добавляет в процессы сборки
ядра зависимость от пакетов разработки OpenSSL для инструмента, выполняющего
подпись.


========================
Генерация ключей подписи
========================

Для генерации и проверки подписей требуются криптографические пары ключей.
Закрытый ключ используется для генерации подписи, а соответствующий открытый
ключ — для её проверки. Закрытый ключ нужен только во время сборки, после чего
его можно удалить или сохранить в защищённом месте. Открытый ключ встраивается в
ядро, чтобы его можно было использовать для проверки подписей по мере загрузки
модулей.

В обычных условиях, когда ``CONFIG_MODULE_SIG_KEY`` не изменён относительно
значения по умолчанию, сборка ядра автоматически сгенерирует новую пару ключей с
помощью openssl, если она ещё не существует в файле::

	certs/signing_key.pem

во время сборки vmlinux (открытая часть ключа должна быть встроена в vmlinux) с
использованием параметров в файле::

	certs/x509.genkey

(который также генерируется, если ещё не существует).

Можно выбрать между RSA (``MODULE_SIG_KEY_TYPE_RSA``), ECDSA
(``MODULE_SIG_KEY_TYPE_ECDSA``) и ML-DSA (``MODULE_SIG_KEY_TYPE_MLDSA_*``) для
генерации пары ключей RSA 4k, NIST P-384 или пары ключей ML-DSA 44, 65 или 87.

Настоятельно рекомендуется предоставить собственный файл x509.genkey.

В частности, в файле x509.genkey раздел req_distinguished_name следует изменить
относительно значения по умолчанию::

	[ req_distinguished_name ]
	#O = Unspecified company
	CN = Build time autogenerated kernel key
	#emailAddress = unspecified.user@unspecified.company

Размер генерируемого ключа RSA также можно задать с помощью::

	[ req ]
	default_bits = 4096


Также можно вручную сгенерировать файлы закрытого/открытого ключей, используя
конфигурационный файл генерации ключей x509.genkey в корневом узле дерева
исходных кодов ядра Linux и команду openssl. Ниже приведён пример генерации
файлов открытого/закрытого ключей::

	openssl req -new -nodes -utf8 -sha256 -days 36500 -batch -x509 \
	   -config x509.genkey -outform PEM -out kernel_key.pem \
	   -keyout kernel_key.pem

Полное путевое имя полученного файла kernel_key.pem затем можно указать в
параметре ``CONFIG_MODULE_SIG_KEY``, и содержащиеся в нём сертификат и ключ будут
использованы вместо автоматически сгенерированной пары ключей.


=========================
Открытые ключи в ядре
=========================

Ядро содержит кольцо открытых ключей, которое может просмотреть root. Они
находятся в связке ключей с именем «.builtin_trusted_keys», которую можно
увидеть с помощью::

	[root@deneb ~]# cat /proc/keys
	...
	223c7853 I------     1 perm 1f030000     0     0 keyring   .builtin_trusted_keys: 1
	302d2d52 I------     1 perm 1f010000     0     0 asymmetri Fedora kernel signing key: d69a84e6bce3d216b979e9505b3e3ef9a7118079: X509.RSA a7118079 []
	...

Помимо открытого ключа, сгенерированного специально для подписи модулей,
дополнительные доверенные сертификаты можно предоставить в PEM-файле, на который
ссылается параметр конфигурации ``CONFIG_SYSTEM_TRUSTED_KEYS``.

Кроме того, код архитектуры может брать открытые ключи из аппаратного хранилища
и добавлять их тоже (например, из базы ключей UEFI).

Наконец, можно добавить дополнительные открытые ключи следующим образом::

	keyctl padd asymmetric "" [.builtin_trusted_keys-ID] <[key-file]

например::

	keyctl padd asymmetric "" 0x223c7853 <my_public_key.x509

Обратите внимание, однако, что ядро разрешит добавлять ключи в
``.builtin_trusted_keys`` **только если** X.509-обёртка нового ключа действительно
подписана ключом, который уже находится в ``.builtin_trusted_keys`` на момент
добавления ключа.


========================
Подпись модулей вручную
========================

Чтобы подписать модуль вручную, используйте инструмент scripts/sign-file,
доступный в дереве исходных кодов ядра Linux. Скрипт требует 4 аргумента:

	1.  Алгоритм хеширования (например, sha256)
	2.  Имя файла закрытого ключа или PKCS#11 URI
	3.  Имя файла открытого ключа
	4.  Подписываемый модуль ядра

Ниже приведён пример подписи модуля ядра::

	scripts/sign-file sha512 kernel-signkey.priv \
		kernel-signkey.x509 module.ko

Используемый алгоритм хеширования не обязан совпадать с настроенным, но если он
не совпадает, вы должны убедиться, что этот алгоритм хеширования либо встроен в
ядро, либо может быть загружен без необходимости в самом себе.

Если закрытый ключ требует парольную фразу или PIN-код, его можно предоставить в
переменной окружения $KBUILD_SIGN_PIN.


===================================================
Подписанные модули и удаление отладочной информации
===================================================

Подписанный модуль имеет цифровую подпись, просто добавленную в конец. Строка
``~Module signature appended~.`` в конце файла модуля подтверждает наличие
подписи, но не подтверждает её действительность!

Подписанные модули ХРУПКИ, так как подпись находится вне определённого
ELF-контейнера. Поэтому из них НЕЛЬЗЯ удалять данные (stripping) после того, как
подпись вычислена и присоединена. Обратите внимание, что подписанной полезной
нагрузкой является весь модуль целиком, включая любую отладочную информацию,
присутствующую на момент подписи.


============================
Загрузка подписанных модулей
============================

Модули загружаются с помощью insmod, modprobe, ``init_module()`` или
``finit_module()`` точно так же, как и неподписанные модули, поскольку в
пространстве пользователя никакой обработки не выполняется. Вся проверка подписи
выполняется внутри ядра.


===============================================
Недействительные подписи и неподписанные модули
===============================================

Если включён ``CONFIG_MODULE_SIG_FORCE`` или в командной строке ядра указан
module.sig_enforce=1, ядро будет загружать только действительно подписанные
модули, для которых у него есть открытый ключ. В противном случае оно также
будет загружать модули, которые не подписаны. Любой модуль, для которого у ядра
есть ключ, но который, как выясняется, имеет несовпадение подписи, не будет
допущен к загрузке.

Любой модуль с неразбираемой подписью будет отвергнут.


=========================================
Администрирование/защита закрытого ключа
=========================================

Поскольку закрытый ключ используется для подписи модулей, вирусы и вредоносное
ПО могли бы использовать закрытый ключ для подписи модулей и компрометации
операционной системы. Закрытый ключ должен быть либо уничтожен, либо перемещён в
защищённое место и не должен храниться в корневом узле дерева исходных кодов
ядра.

Если вы используете один и тот же закрытый ключ для подписи модулей нескольких
конфигураций ядра, вы должны обеспечить, чтобы информации о версии модуля было
достаточно для предотвращения загрузки модуля в другое ядро. Либо установите
``CONFIG_MODVERSIONS=y``, либо обеспечьте, чтобы каждая конфигурация имела
отличную строку релиза ядра, изменив ``EXTRAVERSION`` или
``CONFIG_LOCALVERSION``.
