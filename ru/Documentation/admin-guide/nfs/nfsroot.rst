Монтирование корневой файловой системы через NFS (nfsroot)
==========================================================

:Authors:
	Written 1996 by Gero Kuhlmann <gero@gkminix.han.de>

	Updated 1997 by Martin Mares <mj@atrey.karlin.mff.cuni.cz>

	Updated 2006 by Nico Schottelius <nico-kernel-nfsroot@schottelius.org>

	Updated 2006 by Horms <horms@verge.net.au>

	Updated 2018 by Chris Novakovic <chris@chrisn.me.uk>



Чтобы использовать бездисковую систему, такую как, например, X-терминал или сервер
печати, необходимо, чтобы корневая файловая система располагалась на устройстве,
не являющемся диском. Это может быть initramfs (см.
Documentation/filesystems/ramfs-rootfs-initramfs.rst), ramdisk (см.
Documentation/admin-guide/initrd.rst) или файловая система, смонтированная через NFS.
Следующий текст описывает, как использовать NFS для корневой файловой системы. В
остальной части этого текста «клиент» означает бездисковую систему, а «сервер» —
NFS-сервер.




Включение возможностей nfsroot
==============================

Чтобы использовать nfsroot, при конфигурации необходимо выбрать поддержку
NFS-клиента как встроенную (built-in). После того как это выбрано, станет доступен
параметр nfsroot, который также следует выбрать.

В сетевых параметрах можно выбрать автоконфигурацию на уровне ядра, а также типы
поддерживаемой автоконфигурации. Безопасно выбрать сразу все: DHCP, BOOTP и RARP.




Командная строка ядра
=====================

После того как ядро загружено загрузчиком (см. ниже), ему необходимо сообщить, какое
устройство использовать в качестве корневой файловой системы. А в случае nfsroot —
где найти сервер и имя каталога на сервере, который монтируется как корневой. Это
можно задать с помощью следующих параметров командной строки ядра:


root=/dev/nfs
  Это необходимо для включения псевдо-NFS-устройства. Обратите внимание, что это не
  настоящее устройство, а лишь синоним, сообщающий ядру использовать NFS вместо
  настоящего устройства.


nfsroot=[<server-ip>:]<root-dir>[,<nfs-options>]
  Если параметр \`nfsroot' НЕ задан в командной строке,
  будет использовано значение по умолчанию ``"/tftpboot/%s"``.

  <server-ip>	Задаёт IP-адрес NFS-сервера.
		Адрес по умолчанию определяется параметром ip
		(см. ниже). Этот параметр позволяет использовать разные
		серверы для автоконфигурации IP и для NFS.

  <root-dir>	Имя каталога на сервере, который монтируется как корневой.
		Если в строке присутствует токен "%s", он будет
		заменён ASCII-представлением IP-адреса
		клиента.

  <nfs-options>	Стандартные параметры NFS. Все параметры разделяются запятыми.
		Используются следующие значения по умолчанию::

				port		= as given by server portmap daemon
				rsize		= 4096
				wsize		= 4096
				timeo		= 7
				retrans		= 3
				acregmin	= 3
				acregmax	= 60
				acdirmin	= 30
				acdirmax	= 60
				flags		= hard, nointr, noposix, cto, ac


ip=<client-ip>:<server-ip>:<gw-ip>:<netmask>:<hostname>:<device>:<autoconf>:<dns0-ip>:<dns1-ip>:<ntp0-ip>
  Этот параметр сообщает ядру, как настроить IP-адреса устройств,
  а также как сформировать таблицу IP-маршрутизации. Изначально он назывался
  nfsaddrs, но теперь настройка IP во время загрузки работает независимо от
  NFS, поэтому он был переименован в ip, а старое имя осталось как псевдоним по
  соображениям совместимости.

  Если этот параметр отсутствует в командной строке ядра, все поля
  считаются пустыми, и применяются значения по умолчанию, описанные ниже. В общем
  случае это означает, что ядро пытается настроить всё с помощью
  автоконфигурации.

  Параметр <autoconf> может выступать в качестве значения параметра ip
  сам по себе (без всех предшествующих символов ':').  Если значение равно
  "ip=off" или "ip=none", автоконфигурация не выполняется, в противном случае
  автоконфигурация выполняется.  Наиболее распространённый способ его использования —
  "ip=dhcp".

  <client-ip>	IP-адрес клиента.
  		По умолчанию:  определяется с помощью автоконфигурации.

  <server-ip>	IP-адрес NFS-сервера.
		Если для определения адреса клиента используется RARP
		и этот параметр НЕ пуст, принимаются только
		ответы от указанного сервера.

		Требуется только для NFS root. То есть автоконфигурация
		не будет запущена, если он отсутствует и NFS root не
		используется.

		Значение экспортируется в /proc/net/pnp с префиксом "bootserver "
		(см. ниже).

		По умолчанию: определяется с помощью автоконфигурации.
		Используется адрес сервера автоконфигурации.

  <gw-ip>	IP-адрес шлюза, если сервер находится в другой подсети.
		По умолчанию: определяется с помощью автоконфигурации.

  <netmask>	Маска сети для локального сетевого интерфейса.
		Если не указана, маска сети выводится из IP-адреса клиента
		в предположении классовой адресации.

		По умолчанию:  определяется с помощью автоконфигурации.

  <hostname>	Имя клиента.
		Если присутствует символ '.', то всё, что
		находится перед первой '.', используется как имя хоста клиента, а всё,
		что после неё, используется как его имя домена NIS. Может быть предоставлено
		автоконфигурацией, но его отсутствие не запускает автоконфигурацию.
		Если оно указано и используется DHCP, заданное пользователем имя хоста (и имя
		домена NIS, если присутствует) будет передано в DHCP-запросе; это
		может привести к созданию или обновлению DNS-записи для клиента.

  		По умолчанию: используется IP-адрес клиента в нотации ASCII.

  <device>	Имя используемого сетевого устройства.
		По умолчанию: если у хоста только одно устройство, используется оно.
		В противном случае устройство определяется с помощью
		автоконфигурации. Это делается путём отправки
		запросов автоконфигурации со всех устройств
		и использования того устройства, которое получило первый ответ.

  <autoconf>	Метод, используемый для автоконфигурации.
		В случае параметров,
		которые задают несколько протоколов автоконфигурации,
		запросы отправляются с использованием всех протоколов, и используется первый
		ответивший.

		Будут использованы только те протоколы автоконфигурации, которые были
		скомпилированы в ядро, независимо от значения
		этого параметра::

                  off or none: don't use autoconfiguration
				(do static IP assignment instead)
		  on or any:   use any protocol available in the kernel
			       (default)
		  dhcp:        use DHCP
		  bootp:       use BOOTP
		  rarp:        use RARP
		  both:        use both BOOTP and RARP but not DHCP
		               (old option kept for backwards compatibility)

		если используется dhcp, идентификатор клиента может быть задан в следующем
		формате: "ip=dhcp,client-id-type,client-id-value"

                Default: any

  <dns0-ip>	IP-адрес первичного сервера имён.
		Значение экспортируется в /proc/net/pnp с префиксом "nameserver "
		(см. ниже).

		По умолчанию: отсутствует, если автоконфигурация не используется; определяется
		автоматически при использовании автоконфигурации.

  <dns1-ip>	IP-адрес вторичного сервера имён.
		См. <dns0-ip>.

  <ntp0-ip>	IP-адрес сервера Network Time Protocol (NTP).
		Значение экспортируется в /proc/net/ipconfig/ntp_servers, но в
		остальном не используется (см. ниже).

		По умолчанию: отсутствует, если автоконфигурация не используется; определяется
		автоматически при использовании автоконфигурации.

  После завершения конфигурации (как ручной, так и автоматической) создаются два файла
  в следующем формате; строки опускаются, если соответствующее им
  значение пусто после конфигурации:

  - /proc/net/pnp:

	#PROTO: <DHCP|BOOTP|RARP|MANUAL>	(depending on configuration method)
	domain <dns-domain>			(if autoconfigured, the DNS domain)
	nameserver <dns0-ip>			(primary name server IP)
	nameserver <dns1-ip>			(secondary name server IP)
	nameserver <dns2-ip>			(tertiary name server IP)
	bootserver <server-ip>			(NFS server IP)

  - /proc/net/ipconfig/ntp_servers:

	<ntp0-ip>				(NTP server IP)
	<ntp1-ip>				(NTP server IP)
	<ntp2-ip>				(NTP server IP)

  <dns-domain> и <dns2-ip> (в /proc/net/pnp) и <ntp1-ip> и <ntp2-ip>
  (в /proc/net/ipconfig/ntp_servers) запрашиваются во время автоконфигурации;
  их нельзя задать как часть параметра командной строки ядра "ip=".

  Поскольку параметры "domain" и "nameserver" распознаются DNS-резолверами,
  /etc/resolv.conf на системах, использующих корневую файловую систему NFS, часто
  делают ссылкой на /proc/net/pnp.

  Обратите внимание, что ядро не будет синхронизировать системное время ни с какими NTP-
  серверами, которые оно обнаружит; это входит в обязанности процесса в пространстве
  пользователя (например, скрипта initrd/initramfs, который передаёт IP-адреса,
  перечисленные в /proc/net/ipconfig/ntp_servers, NTP-клиенту перед монтированием
  настоящей корневой файловой системы, если она находится на NFS).


nfsrootdebug
  Этот параметр включает вывод отладочных сообщений в журнал ядра
  во время загрузки, чтобы администраторы могли убедиться, что NFS-клиенту
  передаются правильные параметры монтирования NFS, адрес сервера
  и корневой путь.


rdinit=<executable file>
  Чтобы указать, какой файл содержит программу, запускающую инициализацию
  системы, администраторы могут использовать этот параметр командной строки.
  Значение этого параметра по умолчанию — "/init".  Если указанный
  файл существует и ядро может его выполнить, параметры командной строки ядра,
  относящиеся к корневой файловой системе, включая 'nfsroot=', игнорируются.

  Описание процесса монтирования корневой файловой системы можно
  найти в Documentation/driver-api/early-userspace/early_userspace_support.rst


Загрузчик
=========

Чтобы загрузить ядро в память, можно использовать разные подходы.
Они зависят от доступности различных средств:


- Загрузка с дискеты с помощью syslinux

	При сборке ядер простой способ создать загрузочную дискету, использующую
	syslinux, — это применить make-цели zdisk или bzdisk, которые используют образы
      	zimage и bzimage соответственно. Обе цели принимают
     	параметр FDARGS, который можно использовать для задания командной строки ядра.

	например::

	   make bzdisk FDARGS="root=/dev/nfs"

   	Обратите внимание, что пользователю, выполняющему эту команду, потребуется
     	доступ к устройству дисковода, /dev/fd0

     	Дополнительные сведения о syslinux, включая то, как создавать загрузочные диски
     	для готовых ядер, см. на https://syslinux.zytor.com/

	.. note::
		Ранее можно было записать ядро непосредственно на
		дискету с помощью dd, настроить загрузочное устройство с помощью rdev и
		загрузиться с полученной дискеты. Linux больше не поддерживает этот
		способ загрузки.

- Загрузка с компакт-диска с помощью isolinux

     	При сборке ядер простой способ создать загрузочный компакт-диск,
     	использующий isolinux, — это применить цель isoimage, которая использует образ
     	bzimage. Подобно zdisk и bzdisk, эта цель принимает
     	параметр FDARGS, который можно использовать для задания командной строки ядра.

	например::

	  make isoimage FDARGS="root=/dev/nfs"

     	Полученный iso-образ будет находиться в arch/<ARCH>/boot/image.iso
     	Его можно записать на компакт-диск с помощью различных инструментов, включая
     	cdrecord.

	например::

	  cdrecord dev=ATAPI:1,0,0 arch/x86/boot/image.iso

     	Дополнительные сведения об isolinux, включая то, как создавать загрузочные диски
     	для готовых ядер, см. на https://syslinux.zytor.com/

- Использование LILO

	При использовании LILO все необходимые параметры командной строки можно
	указать с помощью директивы 'append=' в файле конфигурации
	LILO.

	Однако, чтобы использовать директиву 'root=', вам также нужно создать
	фиктивное корневое устройство, которое можно удалить после запуска LILO.

	например::

	  mknod /dev/boot255 c 0 255

	Сведения о настройке LILO см. в его документации.

- Использование GRUB

	При использовании GRUB параметры ядра просто добавляются после указания
	ядра: kernel <kernel> <parameters>

- Использование loadlin

	loadlin можно использовать для загрузки Linux из командной строки DOS без
	необходимости в локальном жёстком диске для монтирования в качестве корневого. Это не было
	тщательно протестировано авторами этого документа, но в целом
	должно быть возможно настроить командную строку ядра аналогично
	настройке LILO.

	Дополнительные сведения см. в документации loadlin.

- Использование загрузочного ПЗУ (boot ROM)

	Вероятно, это наиболее элегантный способ загрузки бездискового клиента.
	С загрузочным ПЗУ ядро загружается по протоколу TFTP. Авторам
	этого документа не известно о каких-либо некоммерческих загрузочных
	ПЗУ, поддерживающих загрузку Linux по сети. Однако
	существуют две свободные реализации загрузочного ПЗУ, netboot-nfs и
	etherboot, обе из которых доступны на sunsite.unc.edu и обе
	содержат всё необходимое для загрузки бездискового клиента Linux.

- Использование pxelinux

	Pxelinux можно использовать для загрузки linux с помощью загрузчика PXE,
	который присутствует на многих современных сетевых картах.

	При использовании pxelinux образ ядра указывается с помощью
	"kernel <relative-path-below /tftpboot>". Параметры nfsroot
	передаются ядру путём добавления их в строку "append".
	Распространено использование последовательной консоли совместно с pxelinux,
	дополнительные сведения см. в Documentation/admin-guide/serial-console.rst.

	Дополнительные сведения об isolinux, включая то, как создавать загрузочные диски
	для готовых ядер, см. на https://syslinux.zytor.com/
