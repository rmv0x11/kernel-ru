NFS-клиент
==========

Клиент NFS
==========

Протокол NFS версии 2 был впервые задокументирован в RFC1094 (март 1989).
С тех пор было опубликовано ещё два мажорных выпуска NFS: NFSv3 был
задокументирован в RFC1813 (июнь 1995), а NFSv4 — в RFC3530 (апрель
2003).

В настоящее время NFS-клиент Linux поддерживает все перечисленные выше
опубликованные версии, и ведётся работа по добавлению поддержки минорной
версии 1 протокола NFSv4.

Цель этого документа — предоставить сведения о некоторых особых
возможностях NFS-клиента, которые могут быть настроены системными
администраторами.


Параметр nfs4_unique_id
=======================

NFSv4 требует, чтобы клиенты идентифицировали себя перед серверами с помощью
уникальной строки.  Состояние открытия файлов и блокировок, разделяемое между
одним клиентом и одним сервером, связано с этой идентичностью.  Для поддержки
надёжного восстановления состояния NFSv4 и прозрачной миграции состояния эта
строка идентичности не должна изменяться при перезагрузках клиента.

Без какого-либо иного вмешательства клиент Linux использует строку, содержащую
имя узла (node name) локальной системы.  Однако системные администраторы часто
не заботятся о том, чтобы имена узлов были полностью квалифицированными и не
менялись на протяжении срока службы клиентской системы.  К именам узлов могут
предъявляться и другие административные требования, диктующие определённое
поведение, которое плохо работает в составе строки nfs_client_id4.

Параметр загрузки nfs.nfs4_unique_id задаёт уникальную строку, которая может
использоваться вместе с именем узла системы, когда NFS-клиент идентифицирует
себя перед сервером.  Таким образом, если имя узла системы не уникально, её
nfs.nfs4_unique_id может помочь предотвратить коллизии с другими клиентами.

Строка nfs.nfs4_unique_id обычно представляет собой UUID, хотя она может
содержать всё, что считается уникальным среди всех NFS-клиентов.  Строку
nfs4_unique_id следует выбирать при установке клиентской системы, точно так
же, как корневая файловая система системы получает свежий UUID в своей метке
во время установки.

Эта строка должна оставаться неизменной на протяжении срока службы клиента.
Её можно безопасно изменить, если позаботиться о том, чтобы клиент завершил
работу корректно и всё непогашенное состояние NFSv4 истекло, дабы
предотвратить потерю состояния NFSv4.

Эта строка может храниться в файле grub.conf NFS-клиента, либо может
предоставляться через средство сетевой загрузки, такое как PXE.  Она также
может быть указана как параметр модуля nfs.ko.

Эта строка-уникализатор будет одинаковой для всех NFS-клиентов, работающих
в контейнерах, если только она не будет переопределена значением, записанным
в /sys/fs/nfs/net/nfs_client/identifier, которое будет локальным для сетевого
пространства имён процесса, выполняющего запись.


DNS-резолвер
============

NFSv4 позволяет одному серверу перенаправить NFS-клиент к данным, которые были
перенесены на другой сервер, посредством специального атрибута "fs_locations".
См. `RFC3530 Section 6: Filesystem Migration and Replication`_ и
`Implementation Guide for Referrals in NFSv4`_.

.. _RFC3530 Section 6\: Filesystem Migration and Replication: https://tools.ietf.org/html/rfc3530#section-6
.. _Implementation Guide for Referrals in NFSv4: https://tools.ietf.org/html/draft-ietf-nfsv4-referrals-00

Информация fs_locations может иметь форму либо ip-адреса и пути, либо
DNS-имени узла и пути. Последнее требует, чтобы NFS-клиент выполнил
DNS-запрос для монтирования нового тома, отсюда и потребность в upcall'е,
позволяющем пространству пользователя предоставить эту услугу.

Предполагая, что у пользователя файловая система 'rpc_pipefs' смонтирована в
обычном /var/lib/nfs/rpc_pipefs, upcall состоит из следующих шагов:

   (1) Процесс проверяет кэш dns_resolve на наличие в нём корректной записи.
       Если таковая есть, он возвращает эту запись и завершается.

   (2) Если корректной записи нет, запускается вспомогательный скрипт
       '/sbin/nfs_cache_getent' (может быть изменён с помощью параметра
       загрузки ядра 'nfs.cache_getent') с двумя аргументами:
       - имя кэша, "dns_resolve"
       - имя узла для разрешения

   (3) После поиска соответствующего ip-адреса вспомогательный скрипт
       записывает результат в псевдофайл rpc_pipefs
       '/var/lib/nfs/rpc_pipefs/cache/dns_resolve/channel'
       в следующем (текстовом) формате:

		"<ip address> <hostname> <ttl>\n"

       Где <ip address> представлен в обычном формате IPv4 (123.456.78.90)
       или IPv6 (ffee:ddcc:bbaa:9988:7766:5544:3322:1100, ffee::1100, ...).
       <hostname> идентичен второму аргументу вспомогательного скрипта,
       а <ttl> — это 'время жизни' (time to live) данной записи кэша (в
       единицах секунд).

       .. note::
            Если <ip address> некорректен, скажем строка "0", то создаётся
            негативная запись, из-за которой ядро будет считать, что у имени
            узла нет корректного DNS-преобразования.




Базовый пример /sbin/nfs_cache_getent
=====================================
.. code-block:: sh

    #!/bin/bash
    #
    ttl=600
    #
    cut=/usr/bin/cut
    getent=/usr/bin/getent
    rpc_pipefs=/var/lib/nfs/rpc_pipefs
    #
    die()
    {
        echo "Usage: $0 cache_name entry_name"
        exit 1
    }

    [ $# -lt 2 ] && die
    cachename="$1"
    cache_path=${rpc_pipefs}/cache/${cachename}/channel

    case "${cachename}" in
        dns_resolve)
            name="$2"
            result="$(${getent} hosts ${name} | ${cut} -f1 -d\ )"
            [ -z "${result}" ] && result="0"
            ;;
        *)
            die
            ;;
    esac
    echo "${result} ${name} ${ttl}" >${cache_path}
