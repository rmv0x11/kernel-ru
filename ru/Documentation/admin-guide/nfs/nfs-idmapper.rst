=============
NFS ID Mapper
=============

Id mapper используется NFS для преобразования идентификаторов пользователей и
групп в имена, а также для преобразования имён пользователей и групп в
идентификаторы.  Часть этого преобразования включает выполнение upcall в
пространство пользователя (userspace) для запроса информации.  Существует два
способа, которыми NFS может получить эту информацию: выполнить вызов
/sbin/request-key или выполнить вызов к демону rpc.idmap.

Сначала NFS попытается вызвать /sbin/request-key.  В случае успеха результат
будет кэширован с помощью обобщённого кэша request-key.  Этот вызов должен
завершаться неудачей только в том случае, если /etc/request-key.conf не настроен
для типа ключа id_resolver; если вы хотите использовать метод request-key, см.
раздел «Настройка» ниже.

Если вызов /sbin/request-key завершается неудачей (если /etc/request-key.conf
не настроен с типом ключа id_resolver), то idmapper запросит сопоставление
идентификаторов у устаревшего демона rpc.idmap.  Этот результат будет сохранён
в специальном кэше idmap NFS.


Настройка
=========

Файл /etc/request-key.conf потребуется изменить, чтобы /sbin/request-key мог
направлять upcall.  Следует добавить такую строку:

``#OP	TYPE	DESCRIPTION	CALLOUT INFO	PROGRAM ARG1 ARG2 ARG3 ...``
``#======	=======	===============	===============	===============================``
``create	id_resolver	*	*		/usr/sbin/nfs.idmap %k %d 600``


Это направит все запросы id_resolver программе /usr/sbin/nfs.idmap.
Последний параметр, 600, определяет, через сколько секунд истечёт срок действия
ключа.  Этот параметр является необязательным для /usr/sbin/nfs.idmap.  Если
тайм-аут не задан, nfs.idmap по умолчанию использует 600 секунд.

id mapper использует следующие описания ключей::

	  uid:  Find the UID for the given user
	  gid:  Find the GID for the given group
	 user:  Find the user  name for the given UID
	group:  Find the group name for the given GID

Любой из этих случаев можно обработать по отдельности, вместо использования
обобщённой программы upcall.  Если вы хотите использовать собственную программу
для поиска uid, то вам следует отредактировать ваш request-key.conf так, чтобы
он выглядел примерно так:

``#OP	TYPE	DESCRIPTION	CALLOUT INFO	PROGRAM ARG1 ARG2 ARG3 ...``
``#======	=======	===============	===============	===============================``
``create	id_resolver	uid:*	*		/some/other/program %k %d 600``
``create	id_resolver	*	*		/usr/sbin/nfs.idmap %k %d 600``


Обратите внимание, что новая строка была добавлена над строкой для обобщённой
программы.  request-key найдёт первую совпадающую строку и соответствующую
программу.  В этом случае /some/other/program будет обрабатывать все запросы uid,
а /usr/sbin/nfs.idmap будет обрабатывать запросы gid, user и group.

См. Documentation/security/keys/request-key.rst для получения дополнительной
информации о функции request-key.


nfs.idmap
=========

nfs.idmap предназначена для вызова из request-key и не должна запускаться
«вручную».  Эта программа принимает два аргумента: сериализованный ключ и
описание ключа.  Сериализованный ключ сначала преобразуется в key_serial_t, а
затем передаётся в качестве аргумента в keyctl_instantiate (оба относятся к
keyutils.h).

Фактические поиски выполняются функциями, находящимися в nfsidmap.h.  nfs.idmap
определяет, какую функцию нужно вызвать, по первой части строки описания.
Например, описание поиска uid будет иметь вид «uid:user@domain».

nfs.idmap вернёт 0, если ключ был инстанцирован, и ненулевое значение в
противном случае.
