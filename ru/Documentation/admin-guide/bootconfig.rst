==========================
Конфигурация загрузки
==========================

:Author: Masami Hiramatsu <mhiramat@kernel.org>

Обзор
=====

Конфигурация загрузки (boot configuration) расширяет текущую командную строку
ядра, позволяя эффективно передавать дополнительные данные в формате «ключ-значение»
при загрузке ядра. Это даёт администраторам возможность передавать
структурированный конфигурационный файл с ключами.

Синтаксис конфигурационного файла
=================================

Синтаксис конфигурации загрузки — это простая структура «ключ-значение». Каждый
ключ состоит из слов, соединённых точкой, а ключ и значение соединяются символом
``=``. Строка значения должна завершаться одним из разделителей, описанных ниже.

Каждое слово ключа должно содержать только буквы, цифры, дефис (``-``) или знак
подчёркивания (``_``). А каждое значение содержит только печатные символы или
пробелы, за исключением таких разделителей, как точка с запятой (``;``), перевод
строки (``\n``), запятая (``,``), решётка (``#``) и закрывающая фигурная скобка
(``}``).

Если за ``=`` следуют пробельные символы вплоть до одного из этих разделителей,
ключу присваивается пустое значение.

Для массивов значения массива разделяются запятой (``,``), а для удобочитаемости
между значениями массива допускаются комментарии и переносы строк с помощью символа
новой строки (``\n``). Таким образом, первый элемент массива должен находиться в
той же строке, что и ключ.::

  KEY[.WORD[...]] = VALUE[, VALUE2[...]][;]

В отличие от синтаксиса командной строки ядра, пробельные символы (включая
табуляцию) вокруг запятой и ``=`` игнорируются.

Если вы хотите использовать эти разделители в значении, его можно заключить либо в
двойные кавычки (``"VALUE"``), либо в одинарные кавычки (``'VALUE'``). Обратите
внимание, что экранировать эти кавычки нельзя.

Может существовать ключ, у которого нет значения или есть пустое значение. Такие
ключи используются для проверки того, существует ключ или нет (как булево
значение).

Синтаксис «ключ-значение»
-------------------------

Синтаксис конфигурационного файла загрузки позволяет пользователю объединять
ключи с частично совпадающими словами с помощью фигурных скобок. Например::

 foo.bar.baz = value1
 foo.bar.qux.quux = value2

Это также можно записать как::

 foo.bar {
    baz = value1
    qux.quux = value2
 }

Или более коротко, следующим образом::

 foo.bar { baz = value1; qux.quux = value2 }

В обоих стилях одинаковые слова ключей автоматически объединяются при их разборе
во время загрузки. Поэтому вы можете дополнять похожие деревья или пары
«ключ-значение».

Значения с одинаковым ключом
----------------------------

Запрещено, чтобы два или более значений либо массивов имели общий одинаковый ключ.
Например,::

 foo = bar, baz
 foo = qux  # !ERROR! we can not re-define same key

Если вы хотите обновить значение, необходимо явно использовать оператор
переопределения ``:=``. Например::

 foo = bar, baz
 foo := qux

тогда ключу ``foo`` присваивается ``qux``. Это удобно для переопределения значения
по умолчанию путём добавления (частичных) пользовательских bootconfig без разбора
bootconfig по умолчанию.

Если вы хотите добавить значение к существующему ключу в качестве элемента массива,
можно использовать оператор ``+=``. Например::

 foo = bar, baz
 foo += qux

В этом случае ключ ``foo`` содержит ``bar``, ``baz`` и ``qux``.

Более того, под одним родительским ключом могут сосуществовать подключи и значение.
Например, следующая конфигурация допустима.::

 foo = value1
 foo.bar = value2
 foo := value3 # This will update foo's value.

Обратите внимание, что, поскольку нет синтаксиса для размещения «сырого» значения
непосредственно под структурированным ключом, его необходимо определить вне
фигурных скобок. Например::

 foo {
     bar = value1
     bar {
         baz = value2
         qux = value3
     }
 }

Кроме того, порядок узла значения под ключом фиксирован. Если присутствуют значение
и подключи, значение всегда является первым дочерним узлом ключа. Поэтому, если
пользователь сначала указывает подключи, например::

 foo.bar = value1
 foo = value2

в программе (и в /proc/bootconfig) это будет показано так, как ниже::

 foo = value2
 foo.bar = value1

Комментарии
-----------

Синтаксис конфигурации допускает комментарии в стиле shell-скриптов. Комментарии,
начинающиеся с решётки ("#") до конца строки ("\n"), будут проигнорированы.

::

 # comment line
 foo = value # value is set to foo.
 bar = 1, # 1st element
       2, # 2nd element
       3  # 3rd element

Это разбирается следующим образом::

 foo = value
 bar = 1, 2, 3

Обратите внимание, что НЕЛЬЗЯ помещать комментарий или перевод строки между
значением и разделителем (``,`` или ``;``). Это означает, что следующая
конфигурация содержит синтаксическую ошибку ::

 key = 1 # comment
       ,2


/proc/bootconfig
================

/proc/bootconfig — это интерфейс конфигурации загрузки для пространства
пользователя. В отличие от /proc/cmdline, этот файл показывает список в стиле
«ключ-значение». Каждая пара «ключ-значение» отображается в отдельной строке в
следующем стиле::

 KEY[.WORDS...] = "[VALUE]"[,"VALUE2"...]


Загрузка ядра с конфигурацией загрузки
======================================

Существует два варианта загрузки ядра с bootconfig: присоединение bootconfig к
образу initrd или встраивание его в само ядро.

Присоединение конфигурации загрузки к initrd
--------------------------------------------

Поскольку конфигурационный файл загрузки по умолчанию загружается вместе с initrd,
он будет добавлен в конец файла образа initrd (initramfs) вместе с дополнением
(padding), размером, контрольной суммой и 12-байтовым «магическим» словом, как
показано ниже.

[initrd][bootconfig][padding][size(le32)][checksum(le32)][#BOOTCONFIG\n]

Поля размера и контрольной суммы — это беззнаковые 32-битные значения с порядком
байтов little endian.

Когда конфигурация загрузки добавляется в образ initrd, общий размер файла
выравнивается по 4 байтам. Для заполнения промежутка добавляются нулевые символы
(``\0``). Таким образом, ``size`` — это длина файла bootconfig плюс байты
дополнения.

Ядро Linux декодирует последнюю часть образа initrd в памяти, чтобы получить данные
конфигурации загрузки.
Благодаря такому методу «piggyback» нет необходимости менять или обновлять
загрузчик и сам образ ядра, при условии что загрузчик передаёт правильный размер
файла initrd. Если по какой-либо причине загрузчик передаст больший размер, ядру не
удастся найти данные bootconfig.

Для выполнения этой операции ядро Linux предоставляет команду ``bootconfig`` в
каталоге tools/bootconfig, которая позволяет администратору применять
конфигурационный файл к образу initrd или удалять его оттуда. Собрать её можно
следующей командой::

 # make -C tools/bootconfig

Чтобы добавить ваш конфигурационный файл загрузки в образ initrd, запустите
bootconfig, как показано ниже (старые данные, если они есть, удаляются
автоматически)::

 # tools/bootconfig/bootconfig -a your-config /boot/initrd.img-X.Y.Z

Чтобы удалить конфигурацию из образа, можно использовать опцию -d, как показано
ниже::

 # tools/bootconfig/bootconfig -d /boot/initrd.img-X.Y.Z

Затем добавьте "bootconfig" в обычную командную строку ядра, чтобы сообщить ядру о
необходимости искать bootconfig в конце файла initrd. В качестве альтернативы
соберите ядро с выбранной опцией Kconfig ``CONFIG_BOOT_CONFIG_FORCE``.

Встраивание конфигурации загрузки в ядро
----------------------------------------

Если вы не можете использовать initrd, можно также встроить файл bootconfig в ядро
с помощью опций Kconfig. В этом случае ядро необходимо перекомпилировать со
следующими конфигурациями::

 CONFIG_BOOT_CONFIG_EMBED=y
 CONFIG_BOOT_CONFIG_EMBED_FILE="/PATH/TO/BOOTCONFIG/FILE"

``CONFIG_BOOT_CONFIG_EMBED_FILE`` требует абсолютного пути или относительного пути к
файлу bootconfig от дерева исходного кода или дерева объектных файлов.
Ядро встроит его в качестве bootconfig по умолчанию.

Так же, как и при присоединении bootconfig к initrd, для включения встроенного
bootconfig вам нужна опция ``bootconfig`` в командной строке ядра, либо, в качестве
альтернативы, соберите ядро с выбранной опцией Kconfig
``CONFIG_BOOT_CONFIG_FORCE``.

Обратите внимание, что даже если вы установите эту опцию, встроенный bootconfig
можно переопределить другим bootconfig, который присоединён к initrd.

Параметры ядра через конфигурацию загрузки
==========================================

В дополнение к командной строке ядра bootconfig можно использовать для передачи
параметров ядра. Все пары «ключ-значение» под ключом ``kernel`` будут переданы в
cmdline ядра напрямую. Более того, пары «ключ-значение» под ключом ``init`` будут
переданы процессу init через cmdline.
Параметры объединяются с заданной пользователем строкой cmdline ядра в следующем
порядке, чтобы параметр командной строки мог переопределять параметры bootconfig
(это зависит от того, как подсистема обрабатывает параметры, но в общем случае более
ранний параметр будет перезаписан более поздним).::

 [bootconfig params][cmdline params] -- [bootconfig init params][cmdline init params]

Вот пример файла bootconfig для параметров kernel/init.::

 kernel {
   root = 01234567-89ab-cdef-0123-456789abcd
 }
 init {
  splash
 }

Это будет скопировано в строку cmdline ядра следующим образом::

 root="01234567-89ab-cdef-0123-456789abcd" -- splash

Если пользователь задаёт какую-то другую командную строку, например,::

 ro bootconfig -- quiet

итоговая cmdline ядра будет следующей::

 root="01234567-89ab-cdef-0123-456789abcd" ro bootconfig -- splash quiet


Ограничение конфигурационного файла
===================================

В настоящее время максимальный размер конфигурации составляет 32 КБ, а общее число
слов-ключей (а не записей «ключ-значение») должно быть менее 1024 узлов.
Примечание: это не число записей, а число узлов; запись должна занимать более 2
узлов (слово-ключ и значение). Поэтому теоретически их может быть до 512 пар
«ключ-значение». Если ключи содержат в среднем 3 слова, конфигурация может
содержать 256 пар «ключ-значение». В большинстве случаев число элементов
конфигурации будет менее 100 записей и меньше 8 КБ, поэтому этого должно быть
достаточно.
Если число узлов превышает 1024, парсер возвращает ошибку, даже если размер файла
меньше 32 КБ. (Обратите внимание, что этот максимальный размер не включает нулевые
символы дополнения.)
В любом случае, поскольку команда bootconfig проверяет это при добавлении
конфигурации загрузки в образ initrd, пользователь может заметить это до загрузки.


API Bootconfig
==============

Пользователь может выполнять запрос или цикл по парам «ключ-значение»; также можно
найти корневой узел ключа (префикс) и найти пары «ключ-значение» под этим узлом.

Если у вас есть строка ключа, можно запросить значение напрямую по этому ключу с
помощью xbc_find_value(). Если вы хотите узнать, какие ключи существуют в
конфигурации загрузки, можно использовать xbc_for_each_key_value() для перебора пар
«ключ-значение». Обратите внимание, что для доступа к значению каждого массива
необходимо использовать xbc_array_for_each_value(), например::

 vnode = NULL;
 xbc_find_value("key.word", &vnode);
 if (vnode && xbc_node_is_array(vnode))
    xbc_array_for_each_value(vnode, value) {
      printk("%s ", value);
    }

Если вы хотите сосредоточиться на ключах, имеющих строку-префикс, можно использовать
xbc_find_node() для поиска узла по строке-префиксу и перебирать ключи под узлом
префикса с помощью xbc_node_for_each_key_value().

Но наиболее типичное использование — получить именованное значение под префиксом
или получить именованный массив под префиксом, как показано ниже::

 root = xbc_find_node("key.prefix");
 value = xbc_node_find_value(root, "option", &vnode);
 ...
 xbc_node_for_each_array_value(root, "array-option", value, anode) {
    ...
 }

Это обращается к значению "key.prefix.option" и к массиву
"key.prefix.array-option".

Блокировка не требуется, так как после инициализации конфигурация становится
доступной только для чтения. Все данные и ключи должны копироваться, если вам нужно
их изменить.
