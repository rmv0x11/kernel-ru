=========================
Загрузочный EFI-стаб
=========================

На платформах x86 и ARM ядро в виде zImage/bzImage может маскироваться
под образ PE/COFF, тем самым убеждая загрузчики EFI-прошивки загрузить
его как исполняемый файл EFI. Код, изменяющий заголовок bzImage, вместе с
EFI-специфичной точкой входа, на которую переходит загрузчик прошивки,
вместе называются «загрузочным EFI-стабом» (EFI boot stub) и находятся в
arch/x86/boot/header.S и drivers/firmware/efi/libstub/x86-stub.c
соответственно. Для ARM EFI-стаб реализован в
arch/arm/boot/compressed/efi-header.S и
drivers/firmware/efi/libstub/arm32-stub.c. Код EFI-стаба, общий для
разных архитектур, находится в drivers/firmware/efi/libstub.

Для arm64 поддержка сжатого ядра отсутствует, поэтому под образ PE/COFF
маскируется сам Image, а EFI-стаб компонуется внутрь ядра. EFI-стаб для
arm64 находится в drivers/firmware/efi/libstub/arm64.c
и drivers/firmware/efi/libstub/arm64-stub.c.

Используя загрузочный EFI-стаб, можно загрузить ядро Linux без
применения обычного загрузчика EFI, такого как grub или elilo. Поскольку
загрузочный EFI-стаб выполняет работу загрузчика, в определённом смысле
он *И ЕСТЬ* загрузчик.

Загрузочный EFI-стаб включается параметром ядра CONFIG_EFI_STUB.


Как установить bzImage.efi
--------------------------

Образ bzImage, расположенный в arch/x86/boot/bzImage, должен быть
скопирован в системный раздел EFI (EFI System Partition, ESP) и
переименован с расширением «.efi». Без этого расширения загрузчик
EFI-прошивки откажется его выполнять. Выполнить bzImage.efi из обычных
файловых систем Linux невозможно, потому что EFI-прошивка их не
поддерживает. Для ARM в системный раздел следует скопировать
arch/arm/boot/zImage, и его, возможно, переименовывать не нужно.
Аналогично для arm64 следует скопировать arch/arm64/boot/Image, но не
обязательно его переименовывать.


Передача параметров ядра из оболочки EFI
----------------------------------------

Аргументы ядру можно передавать после bzImage.efi, например::

	fs0:> bzImage.efi console=ttyS0 root=/dev/sda4


Параметр «initrd=»
------------------

Как и большинство загрузчиков, EFI-стаб позволяет пользователю указывать
несколько файлов initrd с помощью параметра «initrd=». Это единственный
параметр командной строки, специфичный для EFI-стаба, всё остальное
передаётся ядру при его загрузке.

Путь к файлу initrd должен быть абсолютным путём от начала ESP,
относительные имена путей не работают. Кроме того, этот путь — путь в
стиле EFI, и элементы каталогов должны разделяться обратными косыми
чертами (\). Например, при следующей структуре каталогов::

  fs0:>
	Kernels\
			bzImage.efi
			initrd-large.img

	Ramdisks\
			initrd-small.img
			initrd-medium.img

чтобы загрузиться с файлом initrd-large.img, если текущий рабочий
каталог — fs0:\Kernels, необходимо использовать следующую команду::

	fs0:\Kernels> bzImage.efi initrd=\Kernels\initrd-large.img

Обратите внимание, как bzImage.efi можно указать с относительным путём.
Это потому, что выполняемый нами образ интерпретируется оболочкой EFI,
которая понимает относительные пути, тогда как остальная часть командной
строки передаётся в bzImage.efi.

.. hint::
   Также можно предоставить initrd с помощью специфичного для Linux
   протокола UEFI во время загрузки. Подробности см. в
   :ref:`pe-coff-entry-point`.

Параметр «dtb=»
---------------

Для архитектур ARM и arm64 ядру необходимо предоставить дерево устройств
(device tree). Обычно прошивка должна предоставлять дерево устройств
через EFI CONFIGURATION TABLE. Однако параметр командной строки «dtb=»
можно использовать для переопределения предоставленного прошивкой дерева
устройств или для его предоставления, когда прошивка не может этого
сделать.

Обратите внимание: перед загрузкой ядра прошивка добавляет в дерево
устройств информацию о конфигурации времени выполнения. Если для
переопределения дерева устройств используется dtb=, то все данные
времени выполнения, предоставленные прошивкой, будут потеряны. Параметр
dtb= следует использовать только либо как инструмент отладки, либо как
крайнее средство, когда дерево устройств не предоставляется в EFI
CONFIGURATION TABLE.

«dtb=» обрабатывается так же, как и описанный выше параметр «initrd=».
