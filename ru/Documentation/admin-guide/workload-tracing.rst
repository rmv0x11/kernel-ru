Ключевые положения
==================

 * Важно понимать, какие системные ресурсы необходимы для сборки и запуска
   рабочей нагрузки.
 * Для выявления системных ресурсов, используемых рабочей нагрузкой, можно
   применять трассировку Linux и strace. Полнота информации об использовании
   системы зависит от полноты охвата рабочей нагрузки.
 * Производительность и безопасность операционной системы можно
   анализировать с помощью таких инструментов, как:
   `perf <https://man7.org/linux/man-pages/man1/perf.1.html>`_,
   `stress-ng <https://www.mankier.com/1/stress-ng>`_,
   `paxtest <https://github.com/opntr/paxtest-freebsd>`_.
 * После того как мы выявим и поймём потребности рабочей нагрузки, мы можем
   сосредоточиться на них, чтобы избежать регрессий, и использовать это для
   оценки соображений безопасности.

Методология
===========

`strace <https://man7.org/linux/man-pages/man1/strace.1.html>`_ — это
диагностический, обучающий и отладочный инструмент, который можно использовать
для выявления системных ресурсов, используемых рабочей нагрузкой. После того
как мы выявим и поймём потребности рабочей нагрузки, мы можем сосредоточиться
на них, чтобы избежать регрессий, и использовать это для оценки соображений
безопасности. Для трассировки рабочих нагрузок мы используем инструмент strace.

Этот метод трассировки с помощью strace сообщает нам о системных вызовах,
которые делает рабочая нагрузка, и не включает все системные вызовы, которые
она может сделать. Кроме того, этот метод трассировки сообщает нам только о
тех путях кода внутри этих системных вызовов, которые были выполнены.
Например, если рабочая нагрузка успешно открывает файл и читает из него, то
трассируется именно путь успешного выполнения. Любые пути ошибок в этом
системном вызове трассированы не будут. Если существует рабочая нагрузка,
обеспечивающая полный охват рабочей нагрузки, то изложенный здесь метод
трассирует и найдёт все возможные пути кода. Полнота информации об
использовании системы зависит от полноты охвата рабочей нагрузки.

Цель состоит в трассировке рабочей нагрузки в системе с ядром по умолчанию
без необходимости устанавливать собственные ядра.

Как собрать детальную информацию о системе?
===========================================

Инструмент strace можно использовать для трассировки системных вызовов,
выполняемых процессом, и получаемых им сигналов. Системные вызовы — это
фундаментальный интерфейс между приложением и ядром операционной системы. Они
позволяют программе запрашивать услуги у ядра. Например, системный вызов
open() в Linux используется для предоставления доступа к файлу в файловой
системе. strace позволяет нам отслеживать все системные вызовы, выполняемые
приложением. Он перечисляет все системные вызовы, выполненные процессом, и
их итоговый вывод.

Объединив инструменты strace и perf record, вы можете сгенерировать данные
профилирования для записи событий и информации, связанной с процессом. Это
даёт представление о процессе. Инструмент «perf annotate» генерирует
статистику по каждой инструкции программы. В этом документе подробно
рассматривается, как собрать детальную информацию об использовании системных
ресурсов рабочей нагрузкой.

Мы использовали strace для трассировки рабочих нагрузок perf, stress-ng,
paxtest, чтобы проиллюстрировать нашу методологию выявления ресурсов,
используемых рабочей нагрузкой. Этот процесс можно применять для трассировки
других рабочих нагрузок.

Подготовка системы к трассировке
================================

Прежде чем начать, мы покажем вам, как подготовить вашу систему. Мы
предполагаем, что у вас есть дистрибутив Linux, работающий на физической
системе или виртуальной машине. Большинство дистрибутивов включают команду
strace. Давайте установим остальные инструменты, которые обычно не входят в
комплект, для сборки ядра Linux. Обратите внимание, что приведённое ниже
работает в дистрибутивах на основе Debian. Возможно, вам придётся найти
эквивалентные пакеты в других дистрибутивах Linux.

Установите инструменты для сборки ядра Linux и инструменты в репозитории
ядра. scripts/ver_linux — хороший способ проверить, есть ли в вашей системе
уже необходимые инструменты::

  sudo apt-get install build-essential flex bison yacc
  sudo apt install libelf-dev systemtap-sdt-dev libslang2-dev libperl-dev libdw-dev

cscope — хороший инструмент для просмотра исходного кода ядра. Давайте
установим его сейчас::

  sudo apt-get install cscope

Установите stress-ng и paxtest::

  apt-get install stress-ng
  apt-get install paxtest

Обзор рабочих нагрузок
======================

Как упоминалось ранее, мы использовали strace для трассировки рабочих
нагрузок perf bench, stress-ng и paxtest, чтобы показать, как анализировать
рабочую нагрузку и определять подсистемы Linux, используемые этими рабочими
нагрузками. Начнём с обзора этих трёх рабочих нагрузок, чтобы лучше понять,
что они делают и как их использовать.

Рабочая нагрузка perf bench (all)
---------------------------------

Команда perf bench содержит несколько многопоточных микроядерных
бенчмарков для выполнения различных подсистем ядра Linux и системных
вызовов. Это позволяет нам легко измерять влияние изменений, что может помочь
смягчить регрессии производительности. Она также выступает как общий
фреймворк для бенчмаркинга, позволяя разработчикам легко создавать тестовые
случаи, прозрачно интегрироваться и использовать богатые на возможности
подсистемы инструментария производительности.

Рабочая нагрузка stress-ng netdev stressor
-------------------------------------------

stress-ng используется для проведения стресс-тестирования ядра. Он позволяет
вам нагружать различные физические подсистемы компьютера, а также интерфейсы
ядра ОС с помощью «стрессоров» (stressor-s). Они доступны для CPU, кэша CPU,
устройств, I/O, прерываний, файловой системы, памяти, сети, операционной
системы, конвейеров, планировщиков и виртуальных машин. Пожалуйста,
обратитесь к `man-странице stress-ng <https://www.mankier.com/1/stress-ng>`_,
чтобы найти описание всех доступных стрессоров. Стрессор netdev запускает
заданное число (N) рабочих процессов, которые нагружают различные ioctl-команды
netdevice для всех доступных сетевых устройств.

Рабочая нагрузка paxtest kiddie
-------------------------------

paxtest — это программа, тестирующая переполнения буферов в ядре. Она
проверяет, как ядро контролирует использование памяти. Как правило, выполнение
кода в некоторых сегментах памяти делает возможными переполнения буферов. Она
запускает набор программ, которые пытаются нарушить использование памяти. Она
используется как набор регрессионных тестов для PaX, но может быть полезна для
тестирования других патчей защиты памяти для ядра. Мы использовали режим paxtest
kiddie, который ищет простые уязвимости.

Что такое strace и как мы его используем?
=========================================

Как упоминалось ранее, strace — это полезный диагностический, обучающий и
отладочный инструмент, который можно использовать для выявления системных
ресурсов, используемых рабочей нагрузкой. Его можно использовать:

 * Чтобы увидеть, как процесс взаимодействует с ядром.
 * Чтобы увидеть, почему процесс завершается с ошибкой или зависает.
 * Для обратной разработки процесса.
 * Для поиска файлов, от которых зависит программа.
 * Для анализа производительности приложения.
 * Для устранения различных проблем, связанных с операционной системой.

Кроме того, strace может генерировать статистику времени выполнения по
временам, вызовам и ошибкам для каждого системного вызова и сообщать сводку
при завершении программы, подавляя обычный вывод. Это пытается показать
системное время (время CPU, проведённое за выполнением в ядре) независимо от
времени по настенным часам. Мы планируем использовать эти возможности для
получения информации об использовании системы рабочей нагрузкой.

Команда strace поддерживает базовый, подробный и статистический режимы.
Команда strace при запуске в подробном режиме даёт более детальную информацию
о системных вызовах, выполненных процессом.

Запуск strace -c генерирует отчёт о проценте времени, проведённого в каждом
системном вызове, общем времени в секундах, микросекундах на вызов, общем
числе вызовов, количестве каждого системного вызова, завершившегося с ошибкой,
и типе сделанного системного вызова.

 * Использование: strace <команда, которую мы хотим трассировать>
 * Использование в подробном режиме: strace -v <команда>
 * Сбор статистики: strace -c <команда>

Мы использовали опцию «-c» для сбора детальной статистики времени выполнения,
используемой тремя рабочими нагрузками, которые мы выбрали для этого анализа.

 * perf
 * stress-ng
 * paxtest

Что такое cscope и как мы его используем?
=========================================

Теперь давайте посмотрим на `cscope <https://cscope.sourceforge.net/>`_ —
инструмент командной строки для просмотра кодовых баз на C, C++ или Java. Мы
можем использовать его для поиска всех ссылок на символ, глобальных
определений, функций, вызываемых функцией, функций, вызывающих функцию,
текстовых строк, шаблонов регулярных выражений, файлов, включающих файл.

Мы можем использовать cscope, чтобы выяснить, какой системный вызов
принадлежит какой подсистеме. Таким образом мы можем найти подсистемы ядра,
используемые процессом во время его выполнения.

Давайте получим последний репозиторий Linux и соберём базу данных cscope::

  git clone git://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git linux
  cd linux
  cscope -R -p10  # builds cscope.out database before starting browse session
  cscope -d -p10  # starts browse session on cscope.out database

Примечание: запустите «cscope -R -p10», чтобы собрать базу данных, и
«cscope -d -p10», чтобы войти в сеанс просмотра. cscope по умолчанию
использует базу данных cscope.out. Чтобы выйти из этого режима, нажмите
ctrl+d. Опция -p используется для указания числа компонентов пути файла,
которые нужно отображать. -p10 оптимально для просмотра исходного кода ядра.

Что такое perf и как мы его используем?
=======================================

Perf — это инструмент анализа, основанный на системах Linux 2.6+, который
абстрагирует различия в оборудовании CPU при измерении производительности в
Linux и предоставляет простой интерфейс командной строки. Perf основан на
интерфейсе perf_events, экспортируемом ядром. Он очень полезен для
профилирования системы и поиска узких мест производительности в приложении.

Если вы ещё не получили репозиторий mainline-ветки Linux, вы можете сделать
это, а затем собрать ядро и инструмент perf::

  git clone git://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git linux
  cd linux
  make -j3 all
  cd tools/perf
  make

Примечание: команду perf можно собрать без сборки ядра в репозитории, и она
может работать на более старых ядрах. Однако совпадение ревизий ядра и perf
даёт более точную информацию об использовании подсистем.

Мы использовали опции «perf stat» и «perf bench». Для получения подробной
информации об инструменте perf запустите «perf -h».

perf stat
---------
Команда perf stat генерирует отчёт о различных аппаратных и программных
событиях. Она делает это с помощью регистров аппаратных счётчиков,
имеющихся в современных CPU, которые ведут счёт этих действий. «perf stat cal»
показывает статистику для команды cal.

Perf bench
----------
Команда perf bench содержит несколько многопоточных микроядерных бенчмарков
для выполнения различных подсистем ядра Linux и системных вызовов. Это
позволяет нам легко измерять влияние изменений, что может помочь смягчить
регрессии производительности. Она также выступает как общий фреймворк для
бенчмаркинга, позволяя разработчикам легко создавать тестовые случаи,
прозрачно интегрироваться и использовать богатый на возможности инструментарий
производительности.

Команда «perf bench all» запускает следующие бенчмарки:

 * sched/messaging
 * sched/pipe
 * syscall/basic
 * mem/memcpy
 * mem/memset

Что такое stress-ng и как мы его используем?
============================================

Как упоминалось ранее, stress-ng используется для проведения
стресс-тестирования ядра. Он позволяет вам нагружать различные физические
подсистемы компьютера, а также интерфейсы ядра ОС с помощью стрессоров. Они
доступны для CPU, кэша CPU, устройств, I/O, прерываний, файловой системы,
памяти, сети, операционной системы, конвейеров, планировщиков и виртуальных
машин.

Стрессор netdev запускает N рабочих процессов, которые нагружают различные
ioctl-команды netdevice для всех доступных сетевых устройств. Нагружаются
следующие ioctl:

 * SIOCGIFCONF, SIOCGIFINDEX, SIOCGIFNAME, SIOCGIFFLAGS
 * SIOCGIFADDR, SIOCGIFNETMASK, SIOCGIFMETRIC, SIOCGIFMTU
 * SIOCGIFHWADDR, SIOCGIFMAP, SIOCGIFTXQLEN

Следующая команда запускает стрессор::

  stress-ng --netdev 1 -t 60 --metrics command.

Мы можем использовать команду perf record для записи событий и информации,
связанной с процессом. Эта команда записывает данные профилирования в файл
perf.data в том же каталоге.

С помощью следующих команд вы можете записать события, связанные со стрессором
netdev, просмотреть сгенерированный отчёт perf.data и аннотировать его, чтобы
просмотреть статистику по каждой инструкции программы::

  perf record stress-ng --netdev 1 -t 60 --metrics command.
  perf report
  perf annotate

Что такое paxtest и как мы его используем?
==========================================

paxtest — это программа, тестирующая переполнения буферов в ядре. Она
проверяет, как ядро контролирует использование памяти. Как правило, выполнение
кода в некоторых сегментах памяти делает возможными переполнения буферов. Она
запускает набор программ, которые пытаются нарушить использование памяти. Она
используется как набор регрессионных тестов для PaX и будет полезна для
тестирования других патчей защиты памяти для ядра.

paxtest предоставляет режимы kiddie и blackhat. Режим paxtest kiddie работает
в обычном режиме, тогда как режим blackhat пытается обойти защиту ядра, тестируя
наличие уязвимостей. Здесь мы сосредоточимся на режиме kiddie и объединим запуск
«paxtest kiddie» с «perf record», чтобы собрать трассировки стека CPU для запуска
paxtest kiddie и увидеть, какая функция вызывает другие функции в профиле
производительности. Затем для разматывания стека можно использовать режим «dwarf»
(DWARF's Call Frame Information).

Следующую команду можно использовать для просмотра итогового отчёта в формате
графа вызовов::

  perf record --call-graph dwarf paxtest kiddie
  perf report --stdio
