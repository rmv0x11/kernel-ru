=========================================================================

:Author: Huang Rui <ray.huang@amd.com>


Введение
===================

``amd-pstate`` — это драйвер масштабирования производительности CPU от AMD,
который вводит новый механизм управления частотой CPU на современных сериях
APU и CPU от AMD в ядре Linux. Новый механизм основан на Collaborative
Processor Performance Control (CPPC), который обеспечивает более тонкое
управление частотой, чем устаревшие аппаратные P-состояния ACPI. Текущие
платформы CPU/APU от AMD используют драйвер P-состояний ACPI для управления
частотой и тактированием CPU с переключением лишь между 3 P-состояниями. CPPC
заменяет средства управления P-состояний ACPI и предоставляет ядру Linux
гибкий, низколатентный интерфейс для прямой передачи аппаратуре подсказок о
производительности.

``amd-pstate`` использует регуляторы (governors) ядра Linux, такие как
``schedutil``, ``ondemand`` и т. п., для управления подсказками о
производительности, которые предоставляются аппаратной функциональностью CPPC,
внутренне следующей аппаратной спецификации (подробности см. в AMD64
Architecture Programmer's Manual Volume 2: System Programming [1]_). В
настоящее время ``amd-pstate`` поддерживает базовую функцию управления частотой
в соответствии с регуляторами ядра на некоторых процессорах Zen2 и Zen3, и в
будущем мы реализуем больше специфичных для AMD функций после их проверки на
аппаратуре и SBIOS.


Обзор AMD CPPC
=======================

Интерфейс Collaborative Processor Performance Control (CPPC) перечисляет
непрерывное, абстрактное и безразмерное значение производительности в шкале,
не привязанной к конкретному состоянию производительности / частоте. Это
стандарт ACPI [2]_, с помощью которого программное обеспечение может задавать
цели и подсказки по производительности приложения как относительные значения
по отношению к ограничениям инфраструктуры. Процессоры AMD предоставляют
низколатентную регистровую модель (MSR) вместо интерпретатора AML-кода для
корректировки производительности. ``amd-pstate`` инициализирует экземпляр
``struct cpufreq_driver`` с именем ``amd_pstate_driver``, снабжённый
обратными вызовами для управления каждым видом обновления производительности. ::

.. _perf_cap:

Возможности производительности AMD CPPC
----------------------------------------

Highest Performance (RO)
.........................

Это абсолютная максимальная производительность, которой может достичь
отдельный процессор при идеальных условиях. Этот уровень производительности
может быть неподдерживаемым в течение длительного времени и может быть достижим
только в том случае, если другие компоненты платформы находятся в определённом
состоянии; например, он может требовать, чтобы другие процессоры находились в
состоянии простоя. Это эквивалентно наивысшим частотам, поддерживаемым
процессором.

Nominal (Guaranteed) Performance (RO)
......................................

Это максимальный устойчивый уровень производительности процессора при идеальных
рабочих условиях. При отсутствии внешнего ограничения (по питанию, тепловому
режиму и т. п.) это уровень производительности, который процессор, как
ожидается, способен поддерживать непрерывно. Ожидается, что все ядра/процессоры
способны одновременно поддерживать своё номинальное состояние
производительности.

Lowest non-linear Performance (RO)
...................................

Это наименьший уровень производительности, при котором достигается нелинейная
экономия энергии, например за счёт совокупного эффекта масштабирования
напряжения и частоты. Выше этого порога более низкие уровни производительности
должны быть в целом более энергоэффективными, чем более высокие уровни
производительности. Этот регистр фактически сообщает ``amd-pstate`` наиболее
эффективный уровень производительности.

Lowest Performance (RO)
........................

Это абсолютно наименьший уровень производительности процессора. Выбор уровня
производительности ниже наименьшего нелинейного уровня производительности может
вызвать снижение эффективности, но должен уменьшить мгновенное энергопотребление
процессора.

Управление производительностью AMD CPPC
----------------------------------------

``amd-pstate`` передаёт цели производительности через эти регистры. Регистр
управляет поведением желаемой целевой производительности.

Minimum requested performance (RW)
...................................

``amd-pstate`` задаёт минимально допустимый уровень производительности.

Maximum requested performance (RW)
...................................

``amd-pstate`` задаёт предел максимальной производительности, которая, как
ожидается, будет обеспечена аппаратурой.

Desired performance target (RW)
...................................

``amd-pstate`` задаёт желаемую цель в шкале производительности CPPC как
относительное число. Это может быть выражено в процентах от номинальной
производительности (максимума инфраструктуры). Ниже номинального устойчивого
уровня производительности желаемая производительность выражает средний уровень
производительности процессора, ограниченный аппаратурой. Выше номинального
уровня производительности процессор должен обеспечивать по меньшей мере
запрошенную номинальную производительность и подниматься выше, если текущие
рабочие условия это позволяют.

Energy Performance Preference (EPP) (RW)
.........................................

Этот атрибут предоставляет аппаратуре подсказку о том, хочет ли программное
обеспечение сместить акцент в сторону производительности (0x0) или
энергоэффективности (0xff).


Поддержка ключевых регуляторов
===============================

``amd-pstate`` может использоваться со всеми (универсальными) регуляторами
масштабирования, перечисленными в атрибуте политики ``scaling_available_governors``
в ``sysfs``. Затем он отвечает за настройку объектов политики, соответствующих
CPU, и предоставляет ядру ``CPUFreq`` (и регуляторам масштабирования,
привязанным к объектам политики) точную информацию о максимальной и минимальной
рабочих частотах, поддерживаемых аппаратурой. Пользователи могут проверить, что
информация ``scaling_cur_freq`` поступает из ядра ``CPUFreq``.

``amd-pstate`` в основном поддерживает ``schedutil`` и ``ondemand`` для
динамического управления частотой. Это позволяет точно настроить конфигурацию
процессора в ``amd-pstate`` под ``schedutil`` с планировщиком CPU CFS.
``amd-pstate`` регистрирует обратный вызов adjust_perf для реализации поведения
обновления производительности, подобного CPPC. Он инициализируется в
``sugov_start``, а затем заполняет указатель update_util_data процессора, чтобы
назначить ``sugov_update_single_perf`` в качестве функции обратного вызова
обновления загрузки в планировщике CPU. Планировщик CPU вызовет
``cpufreq_update_util`` и назначит целевую производительность согласно
``struct sugov_cpu``, которой принадлежит обновление загрузки. Затем
``amd-pstate`` обновляет желаемую производительность согласно назначению
планировщика CPU.

.. _processor_support:

Поддержка процессоров
=======================

Инициализация ``amd-pstate`` завершится неудачей, если запись ``_CPC`` в SBIOS
ACPI не существует в обнаруженном процессоре. Для проверки существования
``_CPC`` используется ``acpi_cpc_valid``. Все процессоры на основе Zen
поддерживают устаревшую функцию аппаратных P-состояний ACPI, поэтому когда
инициализация ``amd-pstate`` завершается неудачей, ядро откатится к
инициализации драйвера ``acpi-cpufreq``.

Существует два типа аппаратных реализаций ``amd-pstate``: один — это
`Full MSR Support <perf_cap_>`_, а другой — `Shared Memory Support
<perf_cap_>`_. Для указания различных типов может использоваться флаг
функциональности :c:macro:`X86_FEATURE_CPPC`. (Подробности см. в Processor
Programming Reference (PPR) for AMD Family 19h Model 51h, Revision A1
Processors [3]_.) ``amd-pstate`` регистрирует различные экземпляры
``static_call`` для различных аппаратных реализаций.

В настоящее время некоторые процессоры Zen2 и Zen3 поддерживают ``amd-pstate``.
В будущем поддержка появится на всё большем числе процессоров AMD.

Full MSR Support
-----------------

Некоторые новые процессоры Zen3, такие как Cezanne, предоставляют регистры MSR
напрямую, когда установлен флаг функциональности CPU :c:macro:`X86_FEATURE_CPPC`.
``amd-pstate`` может работать с регистром MSR для реализации функции быстрого
переключения в ``CPUFreq``, что позволяет уменьшить задержку (latency)
управления частотой в контексте прерывания. Функции с префиксом ``pstate_xxx``
представляют операции над регистрами MSR.

Shared Memory Support
----------------------

Если флаг функциональности CPU :c:macro:`X86_FEATURE_CPPC` не установлен,
процессор поддерживает решение с разделяемой памятью. В этом случае
``amd-pstate`` использует вспомогательные методы ``cppc_acpi`` для реализации
функций обратного вызова, определённых в ``static_call``. Функции с префиксом
``cppc_xxx`` представляют операции вспомогательных средств ACPI CPPC для решения
с разделяемой памятью.


AMD P-States и аппаратные P-состояния ACPI всегда могут поддерживаться в одном
процессоре. Но AMD P-States имеет более высокий приоритет, и если он включён
с помощью :c:macro:`MSR_AMD_CPPC_ENABLE` или ``cppc_set_enable``, он будет
отвечать на запрос от AMD P-States.


Интерфейс пространства пользователя в ``sysfs`` — управление на уровне политики
===============================================================================

``amd-pstate`` предоставляет несколько глобальных атрибутов (файлов) в ``sysfs``
для управления своей функциональностью на уровне системы. Они расположены в
каталоге ``/sys/devices/system/cpu/cpufreq/policyX/`` и затрагивают все CPU. ::

 root@hr-test1:/home/ray# ls /sys/devices/system/cpu/cpufreq/policy0/*amd*
 /sys/devices/system/cpu/cpufreq/policy0/amd_pstate_highest_perf
 /sys/devices/system/cpu/cpufreq/policy0/amd_pstate_lowest_nonlinear_freq
 /sys/devices/system/cpu/cpufreq/policy0/amd_pstate_max_freq


``amd_pstate_highest_perf / amd_pstate_max_freq``

Максимальная производительность CPPC и частота CPU, которые драйверу разрешено
устанавливать, в процентах от максимально поддерживаемого уровня
производительности CPPC (наивысшая производительность, поддерживаемая в
`AMD CPPC Performance Capability <perf_cap_>`_). В некоторых ASIC наивысшая
производительность CPPC не та, что указана в таблице ``_CPC``, поэтому нам нужно
предоставить её в sysfs. Если boost не активен, но всё ещё поддерживается, эта
максимальная частота будет больше, чем та, что указана в ``cpuinfo``.
Этот атрибут доступен только для чтения.

``amd_pstate_lowest_nonlinear_freq``

Наименьшая нелинейная частота CPU CPPC, которую драйверу разрешено
устанавливать, в процентах от максимально поддерживаемого уровня
производительности CPPC. (См. наименьшую нелинейную производительность в
`AMD CPPC Performance Capability <perf_cap_>`_.)
Этот атрибут доступен только для чтения.

``amd_pstate_hw_prefcore``

Поддерживает ли платформа функцию предпочтительного ядра (preferred core) и
включена ли она. Этот атрибут доступен только для чтения.

``amd_pstate_prefcore_ranking``

Рейтинг производительности ядра. Это число не имеет единицы измерения, но при
чтении большие числа являются предпочтительными. Оно может изменяться во время
выполнения в зависимости от условий платформы. Этот атрибут доступен только для
чтения.

``energy_performance_available_preferences``

Список всех поддерживаемых предпочтений EPP, которые могут использоваться для
``energy_performance_preference`` в этой системе.
Эти профили представляют различные подсказки, которые предоставляются
низкоуровневой прошивке (firmware) о желаемом пользователем компромиссе между
энергопотреблением и эффективностью. ``default`` означает, что значение epp
устанавливается прошивкой платформы. Этот атрибут доступен только для чтения.

``energy_performance_preference``

Текущее предпочтение энергопотребления и производительности можно прочитать из
этого атрибута. Пользователь может изменить текущее предпочтение в соответствии
с потребностями в энергопотреблении или производительности.
Список всех поддерживаемых профилей следует получать из атрибута
``energy_performance_available_preferences``; все профили являются целочисленными
значениями, определёнными в диапазоне от 0 до 255, когда функция EPP включена
прошивкой платформы; если функция EPP отключена, драйвер будет игнорировать
записываемое значение.
Этот атрибут доступен для чтения и записи.

``boost``
Атрибут sysfs `boost` обеспечивает управление повышением производительности ядер
CPU, позволяя пользователям управлять ограничением максимальной частоты CPU.
Этот атрибут может использоваться для включения или отключения функции boost на
отдельных CPU.

Когда функция boost включена, CPU может динамически увеличивать свою частоту
сверх базовой частоты, обеспечивая повышенную производительность для требовательных
рабочих нагрузок. С другой стороны, отключение функции boost ограничивает CPU
работой на базовой частоте, что может быть желательно в определённых сценариях
для приоритезации энергоэффективности или управления температурой.

Чтобы управлять атрибутом `boost`, пользователи могут записать значение `0` для
отключения boost или `1` для его включения для соответствующего CPU, используя
путь sysfs `/sys/devices/system/cpu/cpuX/cpufreq/boost`, где `X` представляет
номер CPU.

Другие значения производительности и частоты можно прочитать из
``/sys/devices/system/cpu/cpuX/acpi_cppc/``, см. :ref:`cppc_sysfs`.


``amd-pstate`` против ``acpi-cpufreq``
======================================

На большинстве платформ AMD, поддерживаемых ``acpi-cpufreq``, для масштабирования
производительности CPU используются таблицы ACPI, предоставляемые прошивкой
платформы, но они обеспечивают лишь 3 P-состояния на процессорах AMD.
Однако на современных сериях APU и CPU от AMD аппаратура предоставляет
Collaborative Processor Performance Control согласно протоколу ACPI и
адаптирует его для платформ AMD. То есть мелкозернистые и непрерывные диапазоны
частот вместо устаревших аппаратных P-состояний. ``amd-pstate`` — это модуль
ядра, который поддерживает новый механизм AMD P-States на большинстве будущих
платформ AMD. Механизм AMD P-States — это более производительный и
энергоэффективный метод управления частотой на процессорах AMD.


Режимы работы драйвера ``amd-pstate``
======================================

``amd_pstate`` CPPC имеет 3 режима работы: автономный (active) режим,
неавтономный (passive) режим и направляемый автономный (guided) режим.
Режим active/passive/guided может быть выбран с помощью различных параметров
ядра.

- В автономном режиме платформа игнорирует запрос на желаемый уровень
  производительности и учитывает только значения, установленные в регистрах
  минимальной, максимальной производительности и предпочтения энергопотребления.
- В неавтономном режиме платформа получает желаемый уровень производительности
  от ОС напрямую через регистр Desired Performance Register.
- В направляемом автономном режиме платформа автономно устанавливает рабочий
  уровень производительности в соответствии с текущей рабочей нагрузкой и в
  пределах, заданных ОС через регистры минимальной и максимальной
  производительности.

Active Mode
------------

``amd_pstate=active``

Это режим низкоуровневого управления прошивкой, который реализуется драйвером
``amd_pstate_epp`` при передаче ядру ``amd_pstate=active`` в командной строке.
В этом режиме драйвер ``amd_pstate_epp`` предоставляет аппаратуре подсказку о
том, хочет ли программное обеспечение сместить акцент в сторону
производительности (0x0) или энергоэффективности (0xff), прошивке CPPC. Затем
алгоритм питания CPPC рассчитывает текущую рабочую нагрузку и корректирует
частоту ядер в реальном времени в соответствии с электропитанием и тепловым
режимом, напряжением ядра и некоторыми другими аппаратными условиями.

Passive Mode
------------

``amd_pstate=passive``

Этот режим включается, если ядру в командной строке передаётся
``amd_pstate=passive``. В этом режиме программное обеспечение драйвера
``amd_pstate`` задаёт желаемую цель QoS в шкале производительности CPPC как
относительное число. Это может быть выражено в процентах от номинальной
производительности (максимума инфраструктуры). Ниже номинального устойчивого
уровня производительности желаемая производительность выражает средний уровень
производительности процессора, ограниченный регистром Performance Reduction
Tolerance. Выше номинального уровня производительности процессор должен
обеспечивать по меньшей мере запрошенную номинальную производительность и
подниматься выше, если текущие рабочие условия это позволяют.

Guided Mode
-----------

``amd_pstate=guided``

Если в командной строке ядра передаётся параметр ``amd_pstate=guided``, то
активируется этот режим. В этом режиме драйвер запрашивает минимальный и
максимальный уровень производительности, а платформа автономно выбирает уровень
производительности в этом диапазоне, соответствующий текущей рабочей нагрузке.

Предпочтительное ядро ``amd-pstate``
====================================

Частота ядра подвержена технологическому разбросу в полупроводниках. Не все
ядра способны достигать максимальной частоты в пределах ограничений
инфраструктуры. Следовательно, AMD переопределила понятие максимальной частоты
изделия. Это означает, что часть ядер может достигать максимальной частоты.
Чтобы найти наилучшую политику планирования процессов для данного сценария, ОС
необходимо знать порядок ядер, сообщаемый платформой через регистр возможности
наивысшей производительности интерфейса CPPC.

Предпочтительное ядро ``amd-pstate`` позволяет планировщику предпочитать
планирование на ядрах, которые могут достигать более высокой частоты при более
низком напряжении. Рейтинги предпочтительных ядер могут динамически изменяться в
зависимости от рабочей нагрузки, условий платформы, теплового режима и старения.

Метрика приоритета будет инициализирована драйвером ``amd-pstate``. Драйвер
``amd-pstate`` также определит, поддерживается ли предпочтительное ядро
``amd-pstate`` платформой.

Драйвер ``amd-pstate`` предоставит начальный порядок ядер при загрузке системы.
Платформа использует интерфейсы CPPC для передачи рейтинга ядер операционной
системе и планировщику, чтобы убедиться, что ОС в первую очередь выбирает для
планирования процессов ядра с наивысшей производительностью. Когда драйвер
``amd-pstate`` получает сообщение об изменении наивысшей производительности, он
обновляет рейтинг ядер и устанавливает приоритет CPU.

Переключение предпочтительного ядра ``amd-pstate``
==================================================
Параметры ядра
--------------

``amd-pstate`` peferred core`` имеет два состояния: включено и отключено.
Состояния включено/отключено можно выбрать с помощью различных параметров ядра.
По умолчанию предпочтительное ядро ``amd-pstate`` включено.

``amd_prefcore=disable``

Для систем, поддерживающих предпочтительное ядро ``amd-pstate``, рейтинги ядер
будут всегда сообщаться платформой. Но ОС может выбрать игнорирование этого с
помощью параметра ядра ``amd_prefcore=disable``.

Интерфейс пространства пользователя в ``sysfs`` — общий
=======================================================

Глобальные атрибуты
-------------------

``amd-pstate`` предоставляет несколько глобальных атрибутов (файлов) в ``sysfs``
для управления своей функциональностью на уровне системы. Они расположены в
каталоге ``/sys/devices/system/cpu/amd_pstate/`` и затрагивают все CPU.

``status``
	Режим работы драйвера: "active", "passive", "guided" или "disable".

	"active"
		Драйвер функционирует и находится в режиме ``active mode``

	"passive"
		Драйвер функционирует и находится в режиме ``passive mode``

	"guided"
		Драйвер функционирует и находится в режиме ``guided mode``

	"disable"
		Драйвер не зарегистрирован и в данный момент не функционирует.

        В этот атрибут можно записывать значения, чтобы изменить режим работы
        драйвера или отменить его регистрацию. Записываемая в него строка должна
        быть одним из его возможных значений, и в случае успеха запись одного из
        этих значений в файл sysfs приведёт к переключению драйвера в режим
        работы, представленный этой строкой, — или к отмене регистрации в случае
        "disable".

``prefcore``
	Состояние предпочтительного ядра драйвера: "enabled" или "disabled".

	"enabled"
		Включить предпочтительное ядро ``amd-pstate``.

	"disabled"
		Отключить предпочтительное ядро ``amd-pstate``


        Этот атрибут доступен только для чтения, чтобы проверить состояние
        предпочтительного ядра, заданное параметром ядра.

Tracer Tool
-------------

``amd_pstate_tracer.py`` может записывать и разбирать журнал трассировки
``amd-pstate``, а затем генерировать графики производительности. Эту утилиту
можно использовать для отладки и настройки производительности драйвера
``amd-pstate``. Инструмент трассировки должен импортировать трассировщик intel
pstate.

Инструмент трассировки расположен в ``linux/tools/power/x86/amd_pstate_tracer``.

разбирать файл напрямую командой ::

 ./amd_pstate_trace.py [-c cpus] -t <trace_file> -n <test_name>

Или сгенерировать файл трассировки с правами root, затем разобрать и построить
графики командой ::

 sudo ./amd_pstate_trace.py [-c cpus] -n <test_name> -i <interval> [-m kbytes]

Модульные тесты для amd-pstate
------------------------------

``amd-pstate-ut`` — это тестовый модуль для тестирования драйвера ``amd-pstate``.

 * Он может помочь всем пользователям проверить поддержку их процессора (SBIOS/Firmware или Hardware).

 * Ядро может выполнить базовый функциональный тест, чтобы избежать регрессии ядра во время обновления.

 * Мы можем добавить больше функциональных или нагрузочных тестов, чтобы согласовать результаты вместе, это принесёт пользу оптимизации масштаба питания и производительности.

#. Как выполнять тесты

   Для реализации этого мы используем тестовый модуль во фреймворках kselftest.
   Мы создаём модуль ``amd-pstate-ut`` и привязываем его к kselftest. (Подробности
   см. в Linux Kernel Selftests [4]_).

    1). Сборка

        + откройте опцию конфигурации :c:macro:`CONFIG_X86_AMD_PSTATE`.
        + установите опцию конфигурации :c:macro:`CONFIG_X86_AMD_PSTATE_UT` в M.
        + make project
        + make selftest ::

            $ cd linux
            $ make -C tools/testing/selftests

        + make perf ::

            $ cd tools/perf/
            $ make


    2). Установка и шаги ::

        $ make -C tools/testing/selftests install INSTALL_PATH=~/kselftest
        $ cp tools/perf/perf /usr/bin/perf
        $ sudo ./kselftest/run_kselftest.sh -c amd-pstate

    3). Указанный тестовый случай ::

        $ cd ~/kselftest/amd-pstate
        $ sudo ./run.sh -t basic
        $ sudo ./run.sh -t tbench
        $ sudo ./run.sh -t tbench -m acpi-cpufreq
        $ sudo ./run.sh -t gitsource
        $ sudo ./run.sh -t gitsource -m acpi-cpufreq
