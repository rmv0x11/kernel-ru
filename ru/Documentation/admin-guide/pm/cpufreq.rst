Концепция масштабирования производительности CPU
================================================

Большинство современных процессоров способны работать в нескольких различных
конфигурациях тактовой частоты и напряжения, которые часто называют рабочими
точками производительности (Operating Performance Points) или P-состояниями
(в терминологии ACPI).  Как правило, чем выше тактовая частота и чем выше
напряжение, тем больше инструкций CPU может выполнить за единицу времени, но
при этом чем выше тактовая частота и чем выше напряжение, тем больше энергии
потребляется за единицу времени (или тем больше мощность потребления) CPU в
данном P-состоянии.  Поэтому существует естественный компромисс между
вычислительной способностью CPU (числом инструкций, которые могут быть
выполнены за единицу времени) и потребляемой CPU мощностью.

В некоторых ситуациях желательно или даже необходимо выполнять программу как
можно быстрее, и тогда нет причин использовать какие-либо P-состояния, кроме
самого высокого (т.е. доступной конфигурации частоты/напряжения с наивысшей
производительностью).  Однако в некоторых других случаях может не быть
необходимости выполнять инструкции настолько быстро, и поддержание наивысшей
доступной вычислительной способности CPU в течение относительно
продолжительного времени без её полной загрузки может рассматриваться как
расточительство.  Также может быть физически невозможно поддерживать
максимальную вычислительную способность CPU слишком долго по причинам теплового
характера, ограничений системы питания или подобным.  Для покрытия этих случаев
существуют аппаратные интерфейсы, позволяющие переключать CPU между различными
конфигурациями частоты/напряжения или (в терминологии ACPI) переводить их в
различные P-состояния.

Обычно они используются вместе с алгоритмами для оценки требуемой
вычислительной способности CPU, чтобы решить, в какие P-состояния перевести
CPU.  Разумеется, поскольку загрузка системы в целом изменяется со временем,
это приходится делать повторно на регулярной основе.  Деятельность, в ходе
которой это происходит, называется масштабированием производительности CPU или
масштабированием частоты CPU (поскольку она включает регулировку тактовой
частоты CPU).


Масштабирование производительности CPU в Linux
===============================================

Ядро Linux поддерживает масштабирование производительности CPU посредством
подсистемы ``CPUFreq`` (CPU Frequency scaling — масштабирование частоты CPU),
которая состоит из трёх уровней кода: ядра (core), управляющих программ
масштабирования (scaling governors) и драйверов масштабирования (scaling
drivers).

Ядро (core) ``CPUFreq`` предоставляет общую инфраструктуру кода и интерфейсы
пространства пользователя для всех платформ, поддерживающих масштабирование
производительности CPU.  Оно определяет базовый каркас, в рамках которого
работают остальные компоненты.

Управляющие программы масштабирования реализуют алгоритмы для оценки требуемой
вычислительной способности CPU.  Как правило, каждая управляющая программа
реализует один, возможно параметризованный, алгоритм масштабирования.

Драйверы масштабирования взаимодействуют с аппаратурой.  Они предоставляют
управляющим программам масштабирования информацию о доступных P-состояниях (или,
в некоторых случаях, о диапазонах P-состояний) и обращаются к аппаратным
интерфейсам, специфичным для платформы, чтобы изменять P-состояния CPU по
запросу управляющих программ масштабирования.

В принципе, любая из доступных управляющих программ масштабирования может
использоваться с любым драйвером масштабирования.  Такая конструкция основана
на наблюдении, что информация, используемая алгоритмами масштабирования
производительности для выбора P-состояния, в большинстве случаев может быть
представлена в платформонезависимой форме, поэтому должна быть возможность
использовать один и тот же алгоритм масштабирования производительности,
реализованный совершенно одинаковым образом, независимо от того, какой драйвер
масштабирования применяется.  Следовательно, один и тот же набор управляющих
программ масштабирования должен подходить для любой поддерживаемой платформы.

Однако это наблюдение может не выполняться для алгоритмов масштабирования
производительности, основанных на информации, предоставляемой самой
аппаратурой, например через регистры обратной связи, поскольку такая информация
обычно специфична для аппаратного интерфейса, из которого она поступает, и может
быть нелегко представима в абстрактной, платформонезависимой форме.  По этой
причине ``CPUFreq`` позволяет драйверам масштабирования обходить уровень
управляющих программ и реализовывать собственные алгоритмы масштабирования
производительности.  Так поступает драйвер масштабирования |intel_pstate|.


Объекты политики ``CPUFreq``
============================

В некоторых случаях аппаратный интерфейс для управления P-состоянием совместно
используется несколькими CPU.  То есть, например, один и тот же регистр (или
набор регистров) применяется для управления P-состоянием нескольких CPU
одновременно, и запись в него влияет на все эти CPU сразу.

Наборы CPU, совместно использующих аппаратные интерфейсы управления
P-состоянием, представляются в ``CPUFreq`` объектами struct cpufreq_policy.
Для единообразия struct cpufreq_policy используется и тогда, когда в данном
наборе только один CPU.

Ядро ``CPUFreq`` хранит указатель на объект struct cpufreq_policy для каждого
CPU в системе, включая CPU, которые в текущий момент отключены (offline).  Если
несколько CPU совместно используют один и тот же аппаратный интерфейс управления
P-состоянием, все соответствующие им указатели указывают на один и тот же объект
struct cpufreq_policy.

``CPUFreq`` использует struct cpufreq_policy в качестве своего базового типа
данных, и конструкция его интерфейса пространства пользователя основана на
концепции политики.


Инициализация CPU
=================

Прежде всего, для работы ``CPUFreq`` должен быть зарегистрирован драйвер
масштабирования.  Возможно зарегистрировать только один драйвер масштабирования
в каждый момент времени, поэтому ожидается, что драйвер масштабирования способен
обслуживать все CPU в системе.

Драйвер масштабирования может быть зарегистрирован до или после регистрации
CPU.  Если CPU регистрируются раньше, то ядро драйверов (driver core) вызывает
ядро ``CPUFreq``, чтобы взять на учёт все уже зарегистрированные CPU во время
регистрации драйвера масштабирования.  В свою очередь, если какие-либо CPU
регистрируются после регистрации драйвера масштабирования, ядро ``CPUFreq``
будет вызвано, чтобы взять их на учёт во время их регистрации.

В любом случае ядро ``CPUFreq`` вызывается, чтобы взять на учёт любой логический
CPU, который оно до сих пор не видело, как только оно готово обслуживать этот
CPU.  [Заметим, что логический CPU может быть физическим одноядерным
процессором, или одним ядром в многоядерном процессоре, или аппаратным потоком в
физическом процессоре или ядре процессора.  В дальнейшем «CPU» всегда означает
«логический CPU», если явно не указано иное, а слово «процессор» используется
для обозначения физической части, возможно включающей несколько логических
CPU.]

Будучи вызванным, ядро ``CPUFreq`` проверяет, установлен ли уже указатель
политики для данного CPU, и если да, то пропускает создание объекта политики.
В противном случае создаётся и инициализируется новый объект политики, что
включает создание новой директории политики в ``sysfs``, а указатель политики,
соответствующий данному CPU, устанавливается на адрес нового объекта политики в
памяти.

Далее вызывается обратный вызов ``->init()`` драйвера масштабирования, которому
в качестве аргумента передаётся указатель политики нового CPU.  Ожидается, что
этот обратный вызов инициализирует аппаратный интерфейс масштабирования
производительности для данного CPU (или, точнее, для набора CPU, совместно
использующих аппаратный интерфейс, к которому он принадлежит, представленного
его объектом политики) и, если объект политики, для которого он был вызван,
является новым, установит параметры политики, такие как минимальная и
максимальная частоты, поддерживаемые аппаратурой, таблица доступных частот (если
набор поддерживаемых P-состояний не является непрерывным диапазоном) и маска
CPU, принадлежащих одной политике (включая как включённые online, так и
отключённые offline CPU).  Затем эта маска используется ядром для заполнения
указателей политики для всех входящих в неё CPU.

Следующий важный шаг инициализации нового объекта политики — присоединить к нему
управляющую программу масштабирования (для начала это управляющая программа
масштабирования по умолчанию, определяемая командной строкой ядра или
конфигурацией, но позже она может быть изменена через ``sysfs``).  Сначала
указатель на новый объект политики передаётся обратному вызову ``->init()``
управляющей программы, от которого ожидается инициализация всех структур данных,
необходимых для обслуживания данной политики, и, возможно, добавление к ней
интерфейса управляющей программы в ``sysfs``.  Затем управляющая программа
запускается путём вызова её обратного вызова ``->start()``.

Ожидается, что этот обратный вызов зарегистрирует у планировщика CPU
обратные вызовы обновления загрузки (utilization update callbacks) для каждого
CPU для всех включённых (online) CPU, принадлежащих данной политике.  Обратные
вызовы обновления загрузки будут вызываться планировщиком CPU при важных
событиях, таких как постановка задачи в очередь и снятие с неё, на каждой
итерации тика планировщика или в общем случае всякий раз, когда загрузка CPU
может измениться (с точки зрения планировщика).  Ожидается, что они выполнят
вычисления, необходимые для определения P-состояния, которое следует
использовать для данной политики в дальнейшем, и вызовут драйвер масштабирования
для внесения изменений в аппаратуру в соответствии с выбором P-состояния.
Драйвер масштабирования может быть вызван непосредственно из контекста
планировщика или асинхронно, через поток ядра или рабочую очередь (workqueue),
в зависимости от конфигурации и возможностей драйвера масштабирования и
управляющей программы.

Аналогичные шаги предпринимаются для объектов политики, которые не являются
новыми, но ранее были «неактивными», что означает, что все принадлежащие им CPU
были отключены (offline).  Единственное практическое отличие в этом случае
заключается в том, что ядро ``CPUFreq`` попытается использовать управляющую
программу масштабирования, ранее применявшуюся с политикой, которая стала
«неактивной» (и теперь переинициализируется), вместо управляющей программы по
умолчанию.

В свою очередь, если ранее отключённый CPU снова приводится во включённое
(online) состояние, но некоторые другие CPU, совместно использующие с ним объект
политики, уже включены, то нет необходимости переинициализировать объект
политики вовсе.  В этом случае необходимо лишь перезапустить управляющую
программу масштабирования, чтобы она могла учесть новый включённый CPU.  Это
достигается вызовом обратных вызовов ``->stop`` и ``->start()`` управляющей
программы, именно в таком порядке, для всей политики.

Как упоминалось ранее, драйвер масштабирования |intel_pstate| обходит уровень
управляющих программ масштабирования ``CPUFreq`` и предоставляет собственные
алгоритмы выбора P-состояния.  Следовательно, если используется |intel_pstate|,
управляющие программы масштабирования не присоединяются к новым объектам
политики.  Вместо этого вызывается обратный вызов ``->setpolicy()`` драйвера для
регистрации обратных вызовов обновления загрузки для каждого CPU для каждой
политики.  Эти обратные вызовы вызываются планировщиком CPU тем же образом, что
и для управляющих программ масштабирования, но в случае |intel_pstate| они
одновременно и определяют P-состояние для использования, и соответствующим
образом изменяют аппаратную конфигурацию за один проход из контекста
планировщика.

Объекты политики, создаваемые в ходе инициализации CPU, и другие связанные с
ними структуры данных уничтожаются, когда драйвер масштабирования отменяет
регистрацию (что происходит, например, при выгрузке содержащего его модуля
ядра) или когда отменяется регистрация последнего CPU, принадлежащего данной
политике.


Интерфейс политики в ``sysfs``
==============================

В ходе инициализации ядра ``CPUFreq`` создаёт директорию ``sysfs`` (kobject) с
именем ``cpufreq`` в :file:`/sys/devices/system/cpu/`.

Эта директория содержит поддиректорию ``policyX`` (где ``X`` представляет целое
число) для каждого объекта политики, обслуживаемого ядром ``CPUFreq``.  На
каждую директорию ``policyX`` указывают символические ссылки ``cpufreq`` в
:file:`/sys/devices/system/cpu/cpuY/` (где ``Y`` представляет целое число,
которое может отличаться от числа, представляемого ``X``) для всех CPU,
связанных с (или принадлежащих) данной политикой.  Каждая из директорий
``policyX`` в :file:`/sys/devices/system/cpu/cpufreq` содержит специфичные для
политики атрибуты (файлы) для управления поведением ``CPUFreq`` для
соответствующих объектов политики (то есть для всех связанных с ними CPU).

Некоторые из этих атрибутов являются обобщёнными.  Они создаются ядром
``CPUFreq``, и их поведение в общем случае не зависит от того, какой драйвер
масштабирования используется и какая управляющая программа масштабирования
присоединена к данной политике.  Некоторые драйверы масштабирования также
добавляют к директориям политики в ``sysfs`` специфичные для драйвера атрибуты
для управления специфичными для политики аспектами поведения драйвера.

Обобщённые атрибуты в :file:`/sys/devices/system/cpu/cpufreq/policyX/`
следующие:

``affected_cpus``
	Список включённых (online) CPU, принадлежащих этой политике (т.е.
	совместно использующих аппаратный интерфейс масштабирования
	производительности, представленный объектом политики ``policyX``).

``bios_limit``
	Если прошивка платформы (BIOS) указывает ОС применить верхний предел к
	частотам CPU, этот предел будет сообщаться через данный атрибут (если он
	присутствует).

	Существование предела может быть результатом некоторых (часто
	непреднамеренных) настроек BIOS, ограничений, исходящих от сервисного
	процессора, или других механизмов на основе BIOS/HW.

	Это не охватывает тепловые ограничения ACPI, которые могут быть обнаружены
	через обобщённый тепловой драйвер.

	Этот атрибут отсутствует, если используемый драйвер масштабирования его не
	поддерживает.

``cpuinfo_cur_freq``
	Текущая частота CPU, принадлежащих этой политике, полученная от аппаратуры
	(в КГц).

	Ожидается, что это частота, на которой аппаратура фактически работает.
	Если эту частоту невозможно определить, данный атрибут не должен
	присутствовать.

``cpuinfo_avg_freq``
        Средняя частота (в КГц) всех CPU, принадлежащих данной политике,
        полученная из предоставляемой аппаратурой обратной связи и сообщаемая за
        промежуток времени, охватывающий не более нескольких миллисекунд.

        Ожидается, что она основана на частоте, на которой аппаратура
        фактически работает, и, как таковая, может требовать специализированной
        аппаратной поддержки (такой как расширение AMU на ARM).  Если её
        невозможно определить, данный атрибут не должен присутствовать.

        Заметим, что неудачная попытка получить текущую частоту для данного
        CPU(ов) приведёт к соответствующей ошибке, т.е.: EAGAIN для CPU,
        который остаётся в простое (выдаётся на ARM).

``cpuinfo_max_freq``
	Максимальная возможная рабочая частота, на которой могут работать CPU,
	принадлежащие этой политике (в кГц).

``cpuinfo_min_freq``
	Минимальная возможная рабочая частота, на которой могут работать CPU,
	принадлежащие этой политике (в кГц).

``cpuinfo_transition_latency``
	Время, которое требуется для переключения CPU, принадлежащих этой
	политике, из одного P-состояния в другое, в наносекундах.

``related_cpus``
	Список всех (включённых online и отключённых offline) CPU, принадлежащих
	этой политике.

``scaling_available_frequencies``
	Список доступных частот CPU, принадлежащих этой политике (в кГц).

``scaling_available_governors``
	Список присутствующих в ядре управляющих программ масштабирования
	``CPUFreq``, которые могут быть присоединены к этой политике, или (если
	используется драйвер масштабирования |intel_pstate|) список алгоритмов
	масштабирования, предоставляемых драйвером, которые могут быть применены к
	этой политике.

	[Заметим, что некоторые управляющие программы являются модульными, и может
	потребоваться загрузить модуль ядра, чтобы содержащаяся в нём управляющая
	программа стала доступной и была перечислена в данном атрибуте.]

``scaling_cur_freq``
	Текущая частота всех CPU, принадлежащих этой политике (в кГц).

	В большинстве случаев это частота последнего P-состояния, запрошенного
	драйвером масштабирования у аппаратуры с использованием предоставляемого им
	интерфейса масштабирования, что может отражать или не отражать частоту, на
	которой CPU фактически работает (вследствие особенностей аппаратной
	конструкции и других ограничений).

	Некоторые архитектуры (например, ``x86``) могут пытаться предоставить через
	этот атрибут информацию, более точно отражающую текущую частоту CPU, но это
	всё равно может не быть точной текущей частотой CPU, как её видит аппаратура
	в данный момент.  Однако такое поведение доступно только через опцию
	c:macro:``CPUFREQ_ARCH_CUR_FREQ``.

``scaling_driver``
	Используемый в текущий момент драйвер масштабирования.

``scaling_governor``
	Управляющая программа масштабирования, присоединённая в текущий момент к
	этой политике, или (если используется драйвер масштабирования
	|intel_pstate|) алгоритм масштабирования, предоставляемый драйвером, который
	в текущий момент применяется к этой политике.

	Этот атрибут доступен для чтения и записи, и запись в него приведёт к тому,
	что к этой политике будет присоединена новая управляющая программа
	масштабирования или к ней будет применён новый алгоритм масштабирования,
	предоставляемый драйвером масштабирования (в случае |intel_pstate|), как
	указано строкой, записанной в этот атрибут (которая должна быть одним из
	имён, перечисленных в описанном выше атрибуте
	``scaling_available_governors``).

``scaling_max_freq``
	Максимальная частота, на которой CPU, принадлежащим этой политике,
	разрешено работать (в кГц).

	Этот атрибут доступен для чтения и записи, и запись в него строки,
	представляющей целое число, приведёт к установке нового предела (он не
	должен быть ниже значения атрибута ``scaling_min_freq``).

``scaling_min_freq``
	Минимальная частота, на которой CPU, принадлежащим этой политике,
	разрешено работать (в кГц).

	Этот атрибут доступен для чтения и записи, и запись в него строки,
	представляющей неотрицательное целое число, приведёт к установке нового
	предела (он не должен быть выше значения атрибута ``scaling_max_freq``).

``scaling_setspeed``
	Этот атрибут функционирует, только если к данной политике присоединена
	управляющая программа масштабирования `userspace`_.

	Он возвращает последнюю частоту, запрошенную управляющей программой (в кГц),
	или может быть записан для установки новой частоты для политики.


Обобщённые управляющие программы масштабирования
================================================

``CPUFreq`` предоставляет обобщённые управляющие программы масштабирования,
которые могут использоваться со всеми драйверами масштабирования.  Как
утверждалось ранее, каждая из них реализует единственный, возможно
параметризованный, алгоритм масштабирования производительности.

Управляющие программы масштабирования присоединяются к объектам политики, и
разные объекты политики могут обслуживаться разными управляющими программами
масштабирования одновременно (хотя в некоторых случаях это может приводить к
неоптимальным результатам).

Управляющую программу масштабирования для данного объекта политики можно
изменить в любой момент с помощью атрибута политики ``scaling_governor`` в
``sysfs``.

Некоторые управляющие программы предоставляют атрибуты ``sysfs`` для управления
или тонкой настройки реализуемых ими алгоритмов масштабирования.  Эти атрибуты,
называемые настройками управляющей программы (governor tunables), могут быть
либо глобальными (общесистемными), либо для каждой политики, в зависимости от
используемого драйвера масштабирования.  Если драйвер требует, чтобы настройки
управляющей программы были для каждой политики, они располагаются в
поддиректории каждой директории политики.  В противном случае они располагаются
в поддиректории в :file:`/sys/devices/system/cpu/cpufreq/`.  В любом случае имя
поддиректории, содержащей настройки управляющей программы, совпадает с именем
предоставляющей их управляющей программы.

``performance``
---------------

Будучи присоединённой к объекту политики, эта управляющая программа вызывает
запрос наивысшей частоты в пределах предела политики ``scaling_max_freq`` для
этой политики.

Запрос делается один раз в тот момент, когда управляющей программой для политики
устанавливается ``performance``, а также всякий раз, когда после этого
изменяются пределы политики ``scaling_max_freq`` или ``scaling_min_freq``.

``powersave``
-------------

Будучи присоединённой к объекту политики, эта управляющая программа вызывает
запрос наименьшей частоты в пределах предела политики ``scaling_min_freq`` для
этой политики.

Запрос делается один раз в тот момент, когда управляющей программой для политики
устанавливается ``powersave``, а также всякий раз, когда после этого изменяются
пределы политики ``scaling_max_freq`` или ``scaling_min_freq``.

``userspace``
-------------

Эта управляющая программа сама по себе ничего не делает.  Вместо этого она
позволяет пространству пользователя устанавливать частоту CPU для политики, к
которой она присоединена, путём записи в атрибут ``scaling_setspeed`` этой
политики.  Хотя намерением может быть установка точной частоты для политики,
фактическая частота может варьироваться в зависимости от аппаратной
координации, тепловых ограничений и ограничений мощности, а также других
факторов.

``schedutil``
-------------

Эта управляющая программа использует данные о загрузке CPU, доступные от
планировщика CPU.  Она обычно рассматривается как часть планировщика CPU,
поэтому может обращаться к внутренним структурам данных планировщика напрямую.

Она работает целиком в контексте планировщика, хотя в некоторых случаях ей может
потребоваться вызвать драйвер масштабирования асинхронно, когда она решает, что
частоту CPU следует изменить для данной политики (это зависит от того, способен
ли драйвер изменять частоту CPU из контекста планировщика).

Действия этой управляющей программы для конкретного CPU зависят от класса
планирования, вызывающего её обратный вызов обновления загрузки для этого CPU.
Если он вызывается классами планирования RT или deadline, управляющая программа
увеличит частоту до разрешённого максимума (то есть до предела политики
``scaling_max_freq``).  В свою очередь, если он вызывается классом планирования
CFS, управляющая программа будет использовать метрику Per-Entity Load Tracking
(PELT) для корневой контрольной группы данного CPU в качестве оценки загрузки
CPU (описание механизма PELT см. в статье LWN.net *Per-entity load tracking*
[1]_).  Затем новая частота CPU для применения вычисляется в соответствии с
формулой

	f = 1.25 * ``f_0`` * ``util`` / ``max``

где ``util`` — это число PELT, ``max`` — теоретический максимум ``util``, а
``f_0`` — это либо максимальная возможная частота CPU для данной политики (если
число PELT инвариантно к частоте), либо текущая частота CPU (в противном
случае).

Эта управляющая программа также применяет механизм, позволяющий ей временно
повышать частоту CPU для задач, которые недавно ожидали завершения ввода-вывода
(I/O), называемый «IO-wait boosting».  Это происходит, когда планировщик
передаёт обратному вызову управляющей программы флаг
:c:macro:`SCHED_CPUFREQ_IOWAIT`, что приводит к немедленному повышению частоты
до разрешённого максимума, а затем к её постепенному снижению со временем до
значения, возвращаемого приведённой выше формулой.

Эта управляющая программа предоставляет только одну настройку:

``rate_limit_us``
	Минимальное время (в микросекундах), которое должно пройти между двумя
	последовательными запусками вычислений управляющей программы (по умолчанию:
	1,5-кратная задержка перехода драйвера масштабирования или 1 мс, если
	драйвер не предоставляет значение задержки).

	Назначение этой настройки — сократить накладные расходы управляющей
	программы в контексте планировщика, которые без неё могли бы быть
	чрезмерными.

Эта управляющая программа в целом рассматривается как замена более старым
управляющим программам `ondemand`_ и `conservative`_ (описанным ниже),
поскольку она проще и теснее интегрирована с планировщиком CPU, её накладные
расходы в части переключений контекста CPU и тому подобного менее значительны, и
она использует собственную метрику загрузки CPU планировщика, так что в принципе
её решения не должны противоречить решениям, принимаемым другими частями
планировщика.

``ondemand``
------------

Эта управляющая программа использует загрузку CPU в качестве метрики для выбора
частоты CPU.

Чтобы оценить текущую загрузку CPU, она измеряет время, прошедшее между
последовательными вызовами своей рабочей процедуры, и вычисляет долю этого
времени, в течение которой данный CPU не находился в простое.  Отношение времени
не-простоя (активного) к общему времени CPU принимается за оценку загрузки.

Если эта управляющая программа присоединена к политике, совместно используемой
несколькими CPU, загрузка оценивается для каждого из них, и наибольший результат
принимается за оценку загрузки для всей политики.

Рабочая процедура этой управляющей программы должна выполняться в контексте
процесса, поэтому она вызывается асинхронно (через рабочую очередь workqueue), и
оттуда при необходимости обновляются P-состояния CPU.  В результате накладные
расходы в контексте планировщика от этой управляющей программы минимальны, но она
вызывает дополнительные переключения контекста CPU относительно часто, а
обновления P-состояний CPU, инициируемые ею, могут быть относительно
нерегулярными.  Также она влияет на собственную метрику загрузки CPU, выполняя
код, который сокращает время простоя CPU (хотя время простоя CPU при этом
сокращается лишь очень незначительно).

Она в целом выбирает частоты CPU пропорционально оценённой загрузке, так что
значение атрибута политики ``cpuinfo_max_freq`` соответствует загрузке 1 (или
100%), а значение атрибута политики ``cpuinfo_min_freq`` соответствует загрузке
0, за исключением случаев, когда загрузка превышает (настраиваемый) порог
ускорения, в каковом случае она сразу перейдёт к наивысшей частоте, которую ей
разрешено использовать (предел политики ``scaling_max_freq``).

Эта управляющая программа предоставляет следующие настройки:

``sampling_rate``
	Это то, как часто должна выполняться рабочая процедура управляющей
	программы, в микросекундах.

	Обычно она устанавливается в значения порядка 2000 (2 мс).  Её значение по
	умолчанию — добавить 50% запаса («дыхательного пространства») к
	``cpuinfo_transition_latency`` для каждой политики, к которой присоединена
	эта управляющая программа.  Минимум обычно равен длительности двух тиков
	планировщика.

	Если эта настройка задаётся для каждой политики, то следующая команда
	оболочки устанавливает представляемое ею время в значение, в 1,5 раза
	превышающее задержку перехода (по умолчанию)::

	# echo `$(($(cat cpuinfo_transition_latency) * 3 / 2))` > ondemand/sampling_rate

``up_threshold``
	Если оценённая загрузка CPU выше этого значения (в процентах), управляющая
	программа установит частоту в максимальное значение, разрешённое для
	политики.  В противном случае выбранная частота будет пропорциональна
	оценённой загрузке CPU.

``ignore_nice_load``
	Если установлено в 1 (по умолчанию 0), это заставит код оценки загрузки CPU
	рассматривать время CPU, потраченное на выполнение задач с уровнями «nice»
	больше 0, как время простоя CPU.

	Это может быть полезно, если в системе есть задачи, которые не следует
	учитывать при принятии решения о том, на какой частоте запускать CPU.
	Тогда, чтобы этого добиться, достаточно повысить уровень «nice» этих задач
	выше 0 и установить данный атрибут в 1.

``sampling_down_factor``
	Временный множитель, от 1 (по умолчанию) до 100 включительно, применяемый к
	значению ``sampling_rate``, если загрузка CPU превышает ``up_threshold``.

	Это приводит к задержке следующего выполнения рабочей процедуры управляющей
	программы (после установки частоты в разрешённый максимум), так что частота
	остаётся на максимальном уровне в течение более продолжительного времени.

	Таким образом можно избежать колебаний частоты в некоторых пульсирующих
	(bursty) рабочих нагрузках ценой дополнительной энергии, затрачиваемой на
	поддержание максимальной вычислительной способности CPU.

``powersave_bias``
	Коэффициент снижения, применяемый к исходной целевой частоте управляющей
	программы (включая максимальное значение, используемое при превышении
	значения ``up_threshold`` оценённой загрузкой CPU), или порог
	чувствительности для драйвера AMD frequency sensitivity powersave bias
	(:file:`drivers/cpufreq/amd_freq_sensitivity.c`), от 0 до 1000
	включительно.

	Если драйвер AMD frequency sensitivity powersave bias не загружен,
	эффективная частота для применения задаётся как

		f * (1 - ``powersave_bias`` / 1000)

	где f — исходная целевая частота управляющей программы.  Значение этого
	атрибута по умолчанию в этом случае равно 0.

	Если драйвер AMD frequency sensitivity powersave bias загружен, значение
	этого атрибута по умолчанию равно 400 и используется иным образом.

	На процессорах AMD семейства 16h (Family 16h) (и более поздних) существует
	механизм для получения измеренной чувствительности рабочей нагрузки, от 0 до
	100% включительно, от аппаратуры.  Это значение может использоваться для
	оценки того, как производительность рабочей нагрузки, выполняющейся на CPU,
	изменится в ответ на изменения частоты.

	Не ожидается, что производительность рабочей нагрузки с чувствительностью 0
	(ограниченной памятью или вводом-выводом) вообще возрастёт в результате
	повышения частоты CPU, тогда как рабочие нагрузки с чувствительностью 100%
	(ограниченные CPU), как ожидается, будут работать значительно лучше при
	повышении частоты CPU.

	Если чувствительность рабочей нагрузки меньше порога, представленного
	значением ``powersave_bias``, драйвер sensitivity powersave bias заставит
	управляющую программу выбрать частоту ниже её исходной цели, чтобы избежать
	избыточного выделения ресурсов рабочим нагрузкам, которые не выиграют от
	работы на более высоких частотах CPU.

``conservative``
----------------

Эта управляющая программа использует загрузку CPU в качестве метрики для выбора
частоты CPU.

Она оценивает загрузку CPU тем же образом, что и описанная выше управляющая
программа `ondemand`_, но реализуемый ею алгоритм выбора частоты CPU отличается.

А именно, она избегает значительного изменения частоты за короткие промежутки
времени, которое может быть непригодным для систем с ограниченной ёмкостью
системы питания (например, с питанием от батареи).  Чтобы достичь этого, она
изменяет частоту относительно малыми шагами, по одному шагу за раз, вверх или
вниз — в зависимости от того, был ли превышен (настраиваемый) порог оценённой
загрузкой CPU.

Эта управляющая программа предоставляет следующие настройки:

``freq_step``
	Шаг частоты в процентах от максимальной частоты, которую управляющей
	программе разрешено устанавливать (предел политики ``scaling_max_freq``), от
	0 до 100 (5 по умолчанию).

	Это то, насколько частоте разрешено изменяться за один раз.  Установка его в
	0 приведёт к использованию шага частоты по умолчанию (5 процентов), а
	установка его в 100 фактически заставляет управляющую программу периодически
	переключать частоту между пределами политики ``scaling_min_freq`` и
	``scaling_max_freq``.

``down_threshold``
	Пороговое значение (в процентах, 20 по умолчанию), используемое для
	определения направления изменения частоты.

	Если оценённая загрузка CPU больше этого значения, частота пойдёт вверх (на
	``freq_step``).  Если загрузка меньше этого значения (и механизм
	``sampling_down_factor`` не действует), частота пойдёт вниз.  В противном
	случае частота не будет изменена.

``sampling_down_factor``
	Коэффициент отсрочки снижения частоты, от 1 (по умолчанию) до 10
	включительно.

	Он фактически приводит к тому, что частота снижается в ``sampling_down_factor``
	раз медленнее, чем нарастает.


Поддержка повышения частоты (Frequency Boost)
=============================================

Общие сведения
--------------

Некоторые процессоры поддерживают механизм для временного повышения рабочей
частоты некоторых ядер в многоядерном корпусе (выше порога устойчивой частоты
для всего корпуса) при определённых условиях, например если весь чип не полностью
загружен и находится ниже своего предполагаемого теплового бюджета или бюджета
мощности.

Разные производители используют разные названия для этой функциональности.  Для
процессоров Intel она называется «Turbo Boost», AMD называет её «Turbo-Core» или
(в технической документации) «Core Performance Boost» и так далее.  Как правило,
она также реализуется по-разному разными производителями.  Простой термин
«frequency boost» (повышение частоты) используется здесь для краткости, чтобы
обозначить все эти реализации.

Механизм повышения частоты может быть либо аппаратным, либо программным.  Если он
аппаратный (например, на x86), решение о запуске повышения принимается
аппаратурой (хотя в общем случае это требует, чтобы аппаратура была переведена в
особое состояние, в котором она может управлять частотой CPU в определённых
пределах).  Если он программный (например, на ARM), драйвер масштабирования
решает, запускать ли повышение и когда это делать.

Файл ``boost`` в ``sysfs``
--------------------------

Этот файл располагается в :file:`/sys/devices/system/cpu/cpufreq/` и управляет
настройкой «boost» для всей системы.  Он отсутствует, если нижележащий драйвер
масштабирования не поддерживает механизм повышения частоты (или поддерживает его,
но предоставляет специфичный для драйвера интерфейс для управления им, как
|intel_pstate|).

Если значение в этом файле равно 1, механизм повышения частоты включён.  Это
означает, что либо аппаратура может быть переведена в состояния, в которых она
способна запускать повышение (в аппаратном случае), либо программному обеспечению
разрешено запускать повышение (в программном случае).  Это не означает, что
повышение фактически используется в данный момент на каких-либо CPU в системе.
Это означает лишь разрешение использовать механизм повышения частоты (который
всё равно может никогда не использоваться по другим причинам).

Если значение в этом файле равно 0, механизм повышения частоты отключён и не
может использоваться вовсе.

Единственные значения, которые можно записать в этот файл, — это 0 и 1.

Обоснование переключателя управления повышением
-----------------------------------------------

Механизм повышения частоты в целом предназначен для того, чтобы помочь достичь
оптимальной производительности CPU на временных масштабах ниже программного
разрешения (например, ниже интервала тика планировщика), и он наглядно подходит
для многих рабочих нагрузок, но в определённых ситуациях он может приводить к
проблемам.

По этой причине многие системы позволяют отключить механизм повышения частоты в
настройках прошивки платформы (BIOS), но это требует перезапуска системы, чтобы
настройка была изменена желаемым образом, что может быть непрактично по меньшей
мере в некоторых случаях.  Например:

  1. Повышение означает разгон (overclocking) процессора, хотя и в
     контролируемых условиях.  В общем случае энергопотребление процессора
     возрастает в результате повышения его частоты и напряжения, пусть даже
     временно.  Это может быть нежелательно на системах, которые переключаются
     на источники питания ограниченной ёмкости, такие как батареи, поэтому
     возможность отключить механизм повышения, пока система работает, может тут
     помочь (но это тоже зависит от рабочей нагрузки).

  2. В некоторых ситуациях детерминированное поведение важнее производительности
     или энергопотребления (или того и другого), и возможность отключить
     повышение, пока система работает, может тогда быть полезной.

  3. Чтобы изучить влияние самого механизма повышения частоты, полезно иметь
     возможность запускать тесты с повышением и без него, желательно без
     перезапуска системы в промежутке.

  4. Воспроизводимые результаты важны при запуске бенчмарков.  Поскольку
     функциональность повышения зависит от загрузки всего корпуса,
     производительность отдельного потока из-за неё может варьироваться, что
     порой может приводить к невоспроизводимым результатам.  Этого можно
     избежать, отключив механизм повышения частоты перед запуском бенчмарков,
     чувствительных к этой проблеме.

Унаследованный переключатель AMD ``cpb``
----------------------------------------

Драйвер масштабирования AMD powernow-k8 поддерживает переключатель ``sysfs``,
очень похожий на глобальный переключатель ``boost``.  Он используется для
отключения/включения функциональности «Core Performance Boost» некоторых
процессоров AMD.

Если он присутствует, то этот переключатель располагается в каждой директории
политики ``CPUFreq`` в ``sysfs``
(:file:`/sys/devices/system/cpu/cpufreq/policyX/`) и называется ``cpb``, что
указывает на более детализированный интерфейс управления.  Однако фактическая
реализация работает на общесистемной основе, и установка этого переключателя для
одной политики приводит к одновременной установке того же его значения для всех
остальных политик.

Этот переключатель по-прежнему поддерживается на процессорах AMD, поддерживающих
лежащую в его основе аппаратную функциональность, но он может быть исключён из
ядра конфигурацией (через опцию конфигурации
:c:macro:`CONFIG_X86_ACPI_CPUFREQ_CPB`), при этом глобальный переключатель
``boost`` присутствует независимо от этого.  Таким образом, всегда возможно
использовать переключатель ``boost`` вместо ``cpb``, что настоятельно
рекомендуется, поскольку это более согласуется с тем, как поступают все
остальные системы (и переключатель ``cpb`` может больше не поддерживаться в
будущем).

Переключатель ``cpb`` никогда не присутствует для каких-либо процессоров без
лежащей в основе аппаратной функциональности (например, для всех процессоров
Intel), даже если установлена опция конфигурации
:c:macro:`CONFIG_X86_ACPI_CPUFREQ_CPB`.
