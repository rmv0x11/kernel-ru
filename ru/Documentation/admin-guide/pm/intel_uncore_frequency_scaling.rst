Intel Uncore Frequency Scaling
==============================

:Copyright: |copy| 2022-2023 Intel Corporation

:Author: Srinivas Pandruvada <srinivas.pandruvada@linux.intel.com>

Введение
--------

Uncore может потреблять значительное количество энергии в серверах Intel Xeon
в зависимости от характеристик нагрузки. Чтобы оптимизировать суммарное
энергопотребление и улучшить общую производительность, SoC содержат внутренние
алгоритмы масштабирования частоты uncore. Эти алгоритмы отслеживают
использование uncore рабочей нагрузкой и устанавливают желаемую частоту.

Возможно, что пользователи имеют разные ожидания относительно
производительности uncore и хотят управлять ею. Цель аналогична возможности
задавать минимальную/максимальную частоты масштабирования через sysfs cpufreq
для улучшения производительности CPU. У пользователей могут быть
чувствительные к задержкам рабочие нагрузки, для которых нежелательны какие-либо
изменения частоты uncore. Также у пользователей могут быть рабочие нагрузки,
требующие разной производительности ядер и uncore на различных этапах, и они
могут захотеть использовать как cpufreq, так и интерфейс масштабирования uncore
для распределения энергии и улучшения общей производительности.

Интерфейс sysfs
---------------

Для управления частотой uncore предоставляется интерфейс sysfs в каталоге:
`/sys/devices/system/cpu/intel_uncore_frequency/`.

Существует один каталог для каждой комбинации package и die, так как область
действия управления масштабированием uncore — это die в SoC с несколькими
die/package или package для SoC с одним die на package. Имя представляет область
действия управления. Например: 'package_00_die_00' соответствует package с id 0
и die 0.

Каждый package_*_die_* содержит следующие атрибуты:

``initial_max_freq_khz``
	После сброса этот атрибут представляет максимально возможную частоту.
	Это атрибут только для чтения. Если пользователи изменяют max_freq_khz,
	они всегда могут вернуться к максимуму, используя значение из этого атрибута.

``initial_min_freq_khz``
	После сброса этот атрибут представляет минимально возможную частоту.
	Это атрибут только для чтения. Если пользователи изменяют min_freq_khz,
	они всегда могут вернуться к минимуму, используя значение из этого атрибута.

``max_freq_khz``
	Этот атрибут используется для задания максимальной частоты uncore.

``min_freq_khz``
	Этот атрибут используется для задания минимальной частоты uncore.

``current_freq_khz``
	Этот атрибут используется для получения текущей частоты uncore.

SoC с TPMI (Topology Aware Register and PM Capsule Interface)
-------------------------------------------------------------

SoC может содержать несколько энергодоменов (power domain) с отдельными
секциями mesh или их совокупностью. Такая секция называется fabric cluster.

Некоторым типам mesh потребуется работать на одной и той же частоте, и они будут
размещены в одном fabric cluster. Преимущество fabric cluster в том, что он
предоставляет масштабируемый механизм для работы с секционированными fabric в
SoC.

Текущий интерфейс sysfs поддерживает управление на уровне package и die. Этого
интерфейса недостаточно для поддержки более детального управления на уровне
fabric cluster.

SoC с поддержкой TPMI (Topology Aware Register and PM Capsule Interface) могут
иметь несколько энергодоменов. Каждый энергодомен может содержать один или
несколько fabric cluster.

Чтобы представить управление на уровне fabric cluster в дополнение к управлению
на уровне package и die (как в системах без поддержки TPMI), sysfs расширен.
Этот детальный интерфейс представлен в sysfs каталогами, имена которых имеют
префикс "uncore". Например: uncore00, uncore01 и т. д.

Область действия управления задаётся атрибутами "package_id", "domain_id" и
"fabric_cluster_id" в каталоге.

Атрибуты в каждом каталоге:

``domain_id``
	Этот атрибут используется для получения id энергодомена данного экземпляра.

``die_id``
	Этот атрибут используется для получения id die (в терминах Linux) данного
	экземпляра. Этот атрибут присутствует только для доменов с агентами ядер
	(core agents) и когда лист CPUID 0x1f предоставляет die ID.

``fabric_cluster_id``
	Этот атрибут используется для получения id fabric cluster данного экземпляра.

``package_id``
	Этот атрибут используется для получения id package данного экземпляра.

``agent_types``
	Этот атрибут отображает все аппаратные агенты, присутствующие внутри домена.
	Каждый агент способен управлять одной или несколькими аппаратными
	подсистемами, к которым относятся: core, cache, memory и I/O.

Остальные атрибуты совпадают с теми, что представлены на уровне
package_*_die_*.

В большинстве текущих сценариев использования "max_freq_khz" и "min_freq_khz"
обновляются на уровне "package_*_die_*". Эта модель по-прежнему будет
поддерживаться следующим образом:

Когда пользователь использует управление на уровне "package_*_die_*", тогда
затрагивается каждый fabric cluster в этом package и die. Например: пользователь
изменяет "max_freq_khz" в package_00_die_00, тогда "max_freq_khz" для каталога
uncore* с тем же id package будет обновлён. В этом случае пользователь
по-прежнему может обновлять "max_freq_khz" на уровне каждого uncore*, что более
ограничительно. Аналогично пользователь может обновить "min_freq_khz" на уровне
"package_*_die_*", чтобы применить его на уровне каждого uncore*.

Поддержка "current_freq_khz" доступна только на уровне каждого fabric cluster
(т. е. в каталоге uncore*).

Компромисс между эффективностью и задержкой
-------------------------------------------

Функция Efficiency Latency Control (ELC) улучшает производительность на ватт.
С помощью этой функции аппаратные алгоритмы управления энергопотреблением
оптимизируют компромисс между задержкой (latency) и энергопотреблением. Для
некоторых чувствительных к задержкам рабочих нагрузок дополнительную настройку
можно выполнить программно (SW), чтобы получить желаемую производительность.

Аппаратура отслеживает среднюю загрузку CPU по всем ядрам в энергодомене через
регулярные интервалы и принимает решение о частоте uncore. Хотя это может давать
наилучшую производительность на ватт, рабочая нагрузка может ожидать более
высокую производительность ценой энергопотребления. Рассмотрим приложение,
которое периодически пробуждается для выполнения чтений из памяти на в остальном
простаивающей системе. В таких случаях, если аппаратура снижает частоту uncore,
то может возникнуть задержка в наращивании частоты до достижения целевой
производительности.

Управление ELC определяет ряд параметров, которые можно изменять программно (SW).
Если средняя загрузка CPU ниже заданного пользователем порога (атрибут
elc_low_threshold_percent ниже), будет использована заданная пользователем
нижняя частота uncore (атрибут elc_floor_freq_khz ниже) вместо вычисленного
аппаратурой минимума.

Аналогично в сценарии высокой нагрузки, когда загрузка CPU превышает значение
верхнего порога (атрибут elc_high_threshold_percent ниже), вместо перехода к
максимальной частоте uncore частота увеличивается шагами по 100 МГц. Это
позволяет избежать немедленного потребления неоправданно высокой мощности при
всплесках загрузки CPU.

Атрибуты для управления эффективностью и задержкой:

``elc_floor_freq_khz``
	Этот атрибут используется для получения/задания нижней частоты управления
	эффективностью и задержкой. Если эта переменная меньше 'min_freq_khz', она
	игнорируется прошивкой (firmware).

``elc_low_threshold_percent``
	Этот атрибут используется для получения/задания нижнего порога управления
	эффективностью и задержкой. Этот атрибут задаётся в процентах загрузки CPU.

``elc_high_threshold_percent``
	Этот атрибут используется для получения/задания верхнего порога управления
	эффективностью и задержкой. Этот атрибут задаётся в процентах загрузки CPU.

``elc_high_threshold_enable``
	Этот атрибут используется для включения/отключения верхнего порога управления
	эффективностью и задержкой. Запишите '1' для включения, '0' для отключения.

Пример конфигурации системы ниже, которая делает следующее:
  * когда загрузка CPU меньше 10%: задаёт частоту uncore равной 800 МГц
  * когда загрузка CPU выше 95%: увеличивает частоту uncore шагами по 100 МГц,
    пока не будет достигнут предел мощности

  elc_floor_freq_khz:800000
  elc_high_threshold_percent:95
  elc_high_threshold_enable:1
  elc_low_threshold_percent:10
