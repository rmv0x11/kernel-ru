Драйвер масштабирования производительности CPU ``intel_pstate``
===============================================================

:Copyright: |copy| 2017 Intel Corporation

:Author: Rafael J. Wysocki <rafael.j.wysocki@intel.com>


Общие сведения
==============

``intel_pstate`` — это часть
:doc:`подсистемы масштабирования производительности CPU <cpufreq>` в ядре Linux
(``CPUFreq``).  Это драйвер масштабирования для процессоров Intel поколения
Sandy Bridge и последующих поколений.  Однако следует учитывать, что некоторые
из этих процессоров могут не поддерживаться.  [Чтобы разобраться в
``intel_pstate``, необходимо понимать, как ``CPUFreq`` работает в целом, поэтому
сейчас самое время прочитать Documentation/admin-guide/pm/cpufreq.rst, если вы
ещё этого не сделали.]

Для процессоров, поддерживаемых ``intel_pstate``, понятие P-state шире, чем
просто рабочая частота или рабочая точка производительности (см. презентацию
Kristen Accardi на LinuxCon Europe 2015 [1]_ для получения дополнительной
информации об этом).  По этой причине внутреннее представление P-states,
используемое ``intel_pstate``, соответствует аппаратной спецификации (подробности
см. в Intel Software Developer’s Manual [2]_).  Однако ядро ``CPUFreq``
использует частоты для идентификации рабочих точек производительности CPU, и
частоты задействованы в предоставляемом им интерфейсе пространства пользователя,
поэтому ``intel_pstate`` также отображает своё внутреннее представление P-states
на частоты (к счастью, это отображение однозначно).  В то же время для
``intel_pstate`` было бы непрактично снабжать ядро ``CPUFreq`` таблицей доступных
частот из-за её возможного размера, поэтому драйвер этого не делает.  Из-за этого
часть функциональности ядра ограничена.

Поскольку используемый ``intel_pstate`` аппаратный интерфейс выбора P-state
доступен на уровне логического CPU, драйвер всегда работает с отдельными CPU.
Следовательно, если используется ``intel_pstate``, каждый объект политики
``CPUFreq`` соответствует одному логическому CPU, и политики ``CPUFreq``
фактически эквивалентны CPU.  В частности, это означает, что они становятся
«неактивными» каждый раз, когда соответствующий CPU переводится в offline, и
должны быть повторно инициализированы, когда он снова переходит в online.

``intel_pstate`` не является модульным, поэтому его нельзя выгрузить, а это
означает, что единственный способ передать ему параметры на этапе ранней
конфигурации — это командная строка ядра.  Однако его конфигурацию можно в
значительной степени корректировать через ``sysfs``.  В некоторых конфигурациях
даже возможно отменить его регистрацию через ``sysfs``, что позволяет загрузить и
зарегистрировать другой драйвер масштабирования ``CPUFreq`` (см.
:ref:`ниже <status_attr>`).

.. _operation_modes:

Режимы работы
=============

``intel_pstate`` может работать в двух разных режимах — активном или пассивном.  В
активном режиме он использует собственный внутренний алгоритм управляющего модуля
(governor) масштабирования производительности либо позволяет аппаратуре выполнять
масштабирование производительности самостоятельно, тогда как в пассивном режиме
он откликается на запросы, поступающие от обобщённого управляющего модуля
``CPUFreq``, реализующего определённый алгоритм масштабирования
производительности.  Какой из них будет действовать, зависит от того, какие
параметры командной строки ядра используются, и от возможностей процессора.

.. _active_mode:

Активный режим
--------------

Это режим работы ``intel_pstate`` по умолчанию для процессоров с поддержкой
аппаратно управляемых P-states (HWP).  Если он работает в этом режиме, атрибут
политики ``scaling_driver`` в ``sysfs`` для всех политик ``CPUFreq`` содержит
строку "intel_pstate".

В этом режиме драйвер обходит слой управляющих модулей масштабирования
``CPUFreq`` и предоставляет собственные алгоритмы масштабирования для выбора
P-state.  Эти алгоритмы можно применять к политикам ``CPUFreq`` так же, как и
обобщённые управляющие модули масштабирования (то есть через атрибут политики
``scaling_governor`` в ``sysfs``).  [Обратите внимание, что для разных политик
могут быть выбраны разные алгоритмы выбора P-state, но это не рекомендуется.]

Они не являются обобщёнными управляющими модулями масштабирования, но их имена
совпадают с именами некоторых из этих управляющих модулей.  Более того, что
вносит дополнительную путаницу, в общем случае они работают не так, как
обобщённые управляющие модули, с которыми они делят имена.  Например, алгоритм
выбора P-state ``powersave``, предоставляемый ``intel_pstate``, не является
аналогом обобщённого управляющего модуля ``powersave`` (грубо говоря, он
соответствует управляющим модулям ``schedutil`` и ``ondemand``).

В активном режиме ``intel_pstate`` предоставляет два алгоритма выбора P-state:
``powersave`` и ``performance``.  То, как они оба работают, зависит от того,
включена ли в процессоре функция аппаратно управляемых P-states (HWP), и,
возможно, от модели процессора.

Какой из алгоритмов выбора P-state используется по умолчанию, зависит от опции
конфигурации ядра :c:macro:`CONFIG_CPU_FREQ_DEFAULT_GOV_PERFORMANCE`.  А именно,
если эта опция установлена, по умолчанию будет использоваться алгоритм
``performance``, а другой будет использоваться по умолчанию, если она не
установлена.

.. _active_mode_hwp:

Активный режим с HWP
~~~~~~~~~~~~~~~~~~~~~

Если процессор поддерживает функцию HWP, она будет включена во время
инициализации процессора и не может быть отключена после этого.  Можно избежать
её включения, передав ядру в командной строке аргумент ``intel_pstate=no_hwp``.

Если функция HWP включена, ``intel_pstate`` полагается на то, что процессор сам
выбирает P-states, но всё же может давать подсказки внутренней логике выбора
P-state процессора.  Каковы эти подсказки, зависит от того, какой алгоритм
выбора P-state применён к данной политике (или к соответствующему ей CPU).

Несмотря на то что выбор P-state выполняется процессором автоматически, в этом
режиме ``intel_pstate`` регистрирует у планировщика CPU колбэки обновления
загрузки (utilization update callbacks).  Однако они используются не для запуска
алгоритма выбора P-state, а для периодического обновления информации о текущей
частоте CPU, которая становится доступной через атрибут политики
``scaling_cur_freq`` в ``sysfs``.

HWP + ``performance``
.....................

В этой конфигурации ``intel_pstate`` запишет 0 в регулятор предпочтения
энергопотребления и производительности процессора Energy-Performance Preference
(EPP) (если поддерживается) или в его регулятор смещения энергопотребления и
производительности Energy-Performance Bias (EPB) (в противном случае), что
означает, что внутренняя логика выбора P-state процессора, как ожидается, будет
полностью сосредоточена на производительности.

Это переопределит настройку EPP/EPB, поступающую из интерфейса ``sysfs`` (см.
:ref:`energy_performance_hints` ниже).  Более того, любые попытки изменить EPP/EPB
на значение, отличное от 0 ("performance"), через ``sysfs`` в этой конфигурации
будут отклонены.

Кроме того, в этой конфигурации диапазон P-states, доступных внутренней логике
выбора P-state процессора, всегда ограничен верхней границей (то есть
максимальным P-state, который драйверу разрешено использовать).

HWP + ``powersave``
...................

В этой конфигурации ``intel_pstate`` установит регулятор предпочтения
энергопотребления и производительности процессора Energy-Performance Preference
(EPP) (если поддерживается) или его регулятор смещения энергопотребления и
производительности Energy-Performance Bias (EPB) (в противном случае) в то
значение, в которое он был установлен ранее через ``sysfs`` (или в то значение
по умолчанию, в которое он был установлен прошивкой платформы).  Обычно это
приводит к тому, что внутренняя логика выбора P-state процессора в меньшей
степени ориентируется на производительность.

Активный режим без HWP
~~~~~~~~~~~~~~~~~~~~~~~

Этот режим работы необязателен для процессоров, не поддерживающих функцию HWP,
или когда ядру в командной строке передан аргумент ``intel_pstate=no_hwp``.
Активный режим используется в этих случаях, если ядру в командной строке передан
аргумент ``intel_pstate=active``.  В этом режиме ``intel_pstate`` может
отказаться работать с процессорами, которые им не распознаны.  [Обратите
внимание, что ``intel_pstate`` никогда не откажется работать с каким-либо
процессором с включённой функцией HWP.]

В этом режиме ``intel_pstate`` регистрирует у планировщика CPU колбэки обновления
загрузки, чтобы запускать алгоритм выбора P-state — либо ``powersave``, либо
``performance``, в зависимости от настройки политики ``scaling_governor`` в
``sysfs``.  Информация о текущей частоте CPU, которая становится доступной через
атрибут политики ``scaling_cur_freq`` в ``sysfs``, также периодически обновляется
этими колбэками обновления загрузки.

``performance``
...............

Без HWP этот алгоритм выбора P-state всегда одинаков независимо от модели
процессора и конфигурации платформы.

Он выбирает максимальный P-state, который ему разрешено использовать, с учётом
лимитов, установленных через ``sysfs``, каждый раз, когда обновляется
конфигурация драйвера для данного CPU (например, через ``sysfs``).

Это алгоритм выбора P-state по умолчанию, если установлена опция конфигурации
ядра :c:macro:`CONFIG_CPU_FREQ_DEFAULT_GOV_PERFORMANCE`.

``powersave``
.............

Без HWP этот алгоритм выбора P-state похож на алгоритм, реализованный обобщённым
управляющим модулем масштабирования ``schedutil``, за исключением того, что
используемая им метрика загрузки основана на числах, поступающих из регистров
обратной связи (feedback registers) CPU.  Как правило, он выбирает P-states
пропорционально текущей загрузке CPU.

Этот алгоритм запускается колбэком обновления загрузки драйвера для данного CPU,
когда его вызывает планировщик CPU, но не чаще одного раза в 10 мс.  Как и в
случае ``performance``, аппаратная конфигурация не затрагивается, если новый
P-state оказывается таким же, как текущий.

Это алгоритм выбора P-state по умолчанию, если опция конфигурации ядра
:c:macro:`CONFIG_CPU_FREQ_DEFAULT_GOV_PERFORMANCE` не установлена.

.. _passive_mode:

Пассивный режим
---------------

Это режим работы ``intel_pstate`` по умолчанию для процессоров без поддержки
аппаратно управляемых P-states (HWP).  Он всегда используется, если ядру в
командной строке передан аргумент ``intel_pstate=passive``, независимо от того,
поддерживает ли данный процессор HWP.  [Обратите внимание, что настройка
``intel_pstate=no_hwp`` приводит к запуску драйвера в пассивном режиме, если она
не сочетается с ``intel_pstate=active``.]  Как и в активном режиме без поддержки
HWP, в этом режиме ``intel_pstate`` может отказаться работать с процессорами,
которые им не распознаны, если включение HWP заблокировано через командную
строку ядра.

Если драйвер работает в этом режиме, атрибут политики ``scaling_driver`` в
``sysfs`` для всех политик ``CPUFreq`` содержит строку "intel_cpufreq".  В этом
случае драйвер ведёт себя как обычный драйвер масштабирования ``CPUFreq``.  То
есть он вызывается обобщёнными управляющими модулями масштабирования, когда
необходимо обратиться к аппаратуре, чтобы изменить P-state CPU (в частности,
управляющий модуль ``schedutil`` может вызывать его напрямую из контекста
планировщика).

Находясь в этом режиме, ``intel_pstate`` можно использовать со всеми (обобщёнными)
управляющими модулями масштабирования, перечисленными в атрибуте политики
``scaling_available_governors`` в ``sysfs`` (при этом описанные выше алгоритмы
выбора P-state не используются).  В этом случае он отвечает за конфигурацию
объектов политики, соответствующих CPU, и снабжает ядро ``CPUFreq`` (и
управляющие модули масштабирования, прикреплённые к объектам политики) точной
информацией о максимальной и минимальной рабочих частотах, поддерживаемых
аппаратурой (включая так называемые "turbo"-диапазоны частот).  Другими словами,
в пассивном режиме весь диапазон доступных P-states предоставляется
``intel_pstate`` ядру ``CPUFreq``.  Однако в этом режиме драйвер не регистрирует
у планировщика CPU колбэки обновления загрузки, и информация ``scaling_cur_freq``
поступает от ядра ``CPUFreq`` (и является последней частотой, выбранной текущим
управляющим модулем масштабирования для данной политики).


.. _turbo:

Поддержка turbo P-states
========================

В большинстве случаев весь диапазон P-states, доступных ``intel_pstate``, можно
разделить на два поддиапазона, соответствующих разным типам поведения процессора,
выше и ниже границы, которая в дальнейшем будет называться «turbo-порогом»
(turbo threshold).

P-states выше turbo-порога называются «turbo P-states», а весь поддиапазон
P-states, к которому они принадлежат, называется «turbo-диапазоном» (turbo
range).  Эти названия связаны с технологией Turbo Boost, позволяющей
многоядерному процессору оппортунистически повышать P-state одного или нескольких
ядер, если для этого достаточно мощности и если это не приведёт к превышению
теплового бюджета (thermal envelope) корпуса процессора.

В частности, если программное обеспечение устанавливает P-state ядра CPU в
пределах turbo-диапазона (то есть выше turbo-порога), процессору разрешается
взять на себя управление масштабированием производительности для этого ядра и в
дальнейшем переводить его в turbo P-states по своему выбору.  Однако это
разрешение интерпретируется по-разному разными поколениями процессоров.  А
именно, процессоры поколения Sandy Bridge никогда не будут использовать какие-либо
P-states выше последнего установленного программным обеспечением для данного ядра,
даже если он находится в пределах turbo-диапазона, тогда как все последующие
поколения процессоров воспримут это как разрешение использовать любые P-states из
turbo-диапазона, даже выше установленного программным обеспечением.  Другими
словами, на этих процессорах установка любого P-state из turbo-диапазона позволит
процессору по своему усмотрению переводить данное ядро во все turbo P-states
вплоть до максимального поддерживаемого включительно.

Одно важное свойство turbo P-states состоит в том, что они не являются устойчивыми
(sustainable).  Точнее, нет гарантии, что какие-либо CPU смогут оставаться в любом
из этих состояний неограниченно долго, поскольку распределение мощности внутри
корпуса процессора может меняться со временем, или может быть превышен тепловой
бюджет, на который он был рассчитан, если turbo P-state используется слишком
долго.

В свою очередь, P-states ниже turbo-порога в общем случае являются устойчивыми.
Фактически, если один из них установлен программным обеспечением, не ожидается,
что процессор сменит его на более низкий, если только не возникнет ситуация
теплового стресса или нарушения лимита мощности (более высокий P-state всё же
может использоваться, если он установлен для другого CPU в том же корпусе в то же
время, например).

Некоторые процессоры позволяют нескольким ядрам одновременно находиться в turbo
P-states, но максимальный P-state, который для них можно установить, в общем
случае зависит от числа одновременно работающих ядер.  Максимальный turbo
P-state, который можно установить для 3 ядер одновременно, обычно ниже
аналогичного максимального P-state для 2 ядер, который, в свою очередь, обычно
ниже максимального turbo P-state, который можно установить для 1 ядра.  Таким
образом, одноядерный максимальный turbo P-state является максимальным
поддерживаемым в целом.

Максимальный поддерживаемый turbo P-state, turbo-порог (максимальный
поддерживаемый non-turbo P-state) и минимальный поддерживаемый P-state специфичны
для модели процессора и могут быть определены чтением модельно-специфичных
регистров (MSR) процессора.  Более того, некоторые процессоры поддерживают функцию
Configurable TDP (Thermal Design Power), и когда эта функция включена, turbo-порог
фактически становится конфигурируемым значением, которое может быть установлено
прошивкой платформы.

В отличие от объектов ``_PSS`` в таблицах ACPI, ``intel_pstate`` всегда
предоставляет весь диапазон доступных P-states, включая весь turbo-диапазон, ядру
``CPUFreq`` и (в пассивном режиме) обобщённым управляющим модулям масштабирования.
Это в общем случае приводит к тому, что turbo P-states устанавливаются чаще при
использовании ``intel_pstate`` относительно масштабирования производительности CPU
на основе ACPI (см. :ref:`ниже <acpi-cpufreq>` для получения дополнительной
информации).

Более того, поскольку ``intel_pstate`` всегда знает, каков реальный turbo-порог
(даже если в процессоре включена функция Configurable TDP), его атрибут
``no_turbo`` в ``sysfs`` (описанный :ref:`ниже <no_turbo_attr>`) должен работать
ожидаемым образом во всех случаях (то есть, если он установлен для отключения
turbo P-states, он всегда должен предотвращать использование их ``intel_pstate``).


Поддержка процессоров
=====================

Для обработки данного процессора ``intel_pstate`` требует, чтобы был известен ряд
различных сведений о нём, в том числе:

 * Минимальный поддерживаемый P-state.

 * Максимальный поддерживаемый :ref:`non-turbo P-state <turbo>`.

 * Поддерживаются ли turbo P-states вообще.

 * Максимальный поддерживаемый :ref:`одноядерный turbo P-state <turbo>` (если
   turbo P-states поддерживаются).

 * Формула масштабирования для перевода внутреннего представления P-states
   драйвера в частоты и обратно.

В общем случае способы получения этой информации специфичны для модели или
семейства процессора.  Хотя часто всю её можно получить из самого процессора
(используя модельно-специфичные регистры), бывают случаи, когда для её получения
также необходимо обращаться к аппаратным руководствам.

По этой причине в ``intel_pstate`` есть список поддерживаемых процессоров, и
инициализация драйвера завершится неудачей, если обнаруженный процессор
отсутствует в этом списке, если только он не поддерживает функцию HWP.
[Интерфейс для получения всей перечисленной выше информации одинаков для всех
процессоров, поддерживающих функцию HWP, поэтому ``intel_pstate`` работает со
всеми ними.]


Поддержка гибридных процессоров
===============================

Некоторые процессоры, поддерживаемые ``intel_pstate``, содержат два или более
типа ядер CPU, различающихся максимальным turbo P-state, характеристиками
производительности и энергопотребления, размерами кэша и, возможно, другими
свойствами.  Их обычно называют гибридными процессорами.  Для их поддержки
``intel_pstate`` требует, чтобы был включён HWP, и предполагает, что единицы
производительности HWP одинаковы для всех CPU в системе, так что данный уровень
производительности HWP всегда представляет приблизительно одну и ту же
физическую производительность независимо от типа ядра (CPU).

Гибридные процессоры с SMT
--------------------------

В системах, где SMT (Simultaneous Multithreading), также называемая
HyperThreading (HT) в контексте процессоров Intel, включена хотя бы на одном
ядре, ``intel_pstate`` назначает CPU приоритеты на основе производительности.  А
именно, приоритет данного CPU отражает его наивысший уровень производительности
HWP, что заставляет планировщик CPU в общем случае отдавать предпочтение более
производительным CPU, так что менее производительные CPU используются, когда
остальные полностью загружены.  Однако SMT-сиблинги (то есть логические CPU,
разделяющие одно физическое ядро) обрабатываются особым образом, так что если
один из них используется, эффективный приоритет остальных понижается ниже
приоритетов CPU, расположенных в других физических ядрах.

Этот подход максимизирует производительность в большинстве случаев, но, к
сожалению, он также приводит к чрезмерному энергопотреблению в некоторых важных
сценариях, таких как воспроизведение видео, что в общем случае нежелательно.
Хотя при включённом SMT иного жизнеспособного выбора нет, потому что эффективную
ёмкость и загрузку SMT-сиблингов сложно определить, гибридные процессоры без SMT
можно обрабатывать более энергоэффективными способами.

.. _CAS:

Поддержка планирования с учётом ёмкости (Capacity-Aware Scheduling)
-------------------------------------------------------------------

Поддержка планирования с учётом ёмкости (capacity-aware scheduling, CAS) в
планировщике CPU включается ``intel_pstate`` по умолчанию на гибридных
процессорах без SMT.  CAS в общем случае заставляет планировщик размещать задачи
на CPU до тех пор, пока на нём достаточно свободной ёмкости, и если загрузка
данной задачи слишком высока для него, задаче нужно будет перейти куда-то ещё.

Поскольку CAS учитывает ёмкости CPU, он не требует приоритизации CPU и позволяет
распределять задачи более симметрично между более производительными и менее
производительными CPU.  Будучи однажды размещённой на CPU с достаточной ёмкостью
для её размещения, задача может просто продолжать выполняться там независимо от
того, полностью ли загружены другие CPU, поэтому в среднем CAS снижает загрузку
более производительных CPU, что приводит к более сбалансированному
энергопотреблению, поскольку более производительные CPU, как правило, менее
энергоэффективны, чем менее производительные.

Чтобы использовать CAS, планировщику нужно знать ёмкость каждого CPU в системе, и
ему нужно уметь вычислять масштабно-инвариантную загрузку (scale-invariant
utilization) CPU, поэтому ``intel_pstate`` снабжает его необходимой информацией.

Прежде всего, ёмкость каждого CPU представлена отношением его наивысшего уровня
производительности HWP, умноженного на 1024, к наивысшему уровню
производительности HWP самого производительного CPU в системе, что работает,
потому что единицы производительности HWP одинаковы для всех CPU.  Во-вторых,
вычисления частотной инвариантности (frequency-invariance), выполняемые
планировщиком, чтобы всегда выражать загрузку CPU в одних и тех же единицах
независимо от частоты, на которой он в данный момент работает, скорректированы с
учётом ёмкости CPU.  Всё это происходит, когда ``intel_pstate``
зарегистрировался в ядре ``CPUFreq`` и определил, что работает на гибридном
процессоре без SMT.

Поддержка планирования с учётом энергопотребления (Energy-Aware Scheduling)
---------------------------------------------------------------------------

Если во время конфигурации ядра был установлен ``CONFIG_ENERGY_MODEL`` и
``intel_pstate`` работает на гибридном процессоре без SMT, то в дополнение к
включению :ref:`CAS` он регистрирует модель энергопотребления (Energy Model) для
процессора.  Это позволяет включить в планировщике CPU поддержку планирования с
учётом энергопотребления (Energy-Aware Scheduling, EAS), если в качестве
управляющего модуля ``CPUFreq`` используется ``schedutil``, что требует, чтобы
``intel_pstate`` работал в :ref:`пассивном режиме <passive_mode>`.

Модель энергопотребления, регистрируемая ``intel_pstate``, является искусственной
(то есть основана на абстрактных значениях стоимости и не включает какие-либо
реальные значения мощности) и относительно проста, чтобы избежать ненужных
вычислений в планировщике.  В ней есть домен производительности (performance
domain) для каждого CPU в системе, и значения стоимости для этих доменов
производительности выбраны так, что выполнение задачи на менее производительном
(малом) CPU всегда выглядит дешевле, чем выполнение этой задачи на более
производительном (большом) CPU.  Однако для двух CPU одного типа разница в
стоимости зависит от их текущей загрузки, и CPU, чья текущая загрузка выше, в
общем случае выглядит более дорогим местом назначения для данной задачи.  Это
помогает балансировать нагрузку между CPU одного типа.

Поскольку EAS работает поверх CAS, задачи с высокой загрузкой всегда мигрируют на
CPU с достаточной ёмкостью для их размещения, но благодаря EAS задачи с низкой
загрузкой стремятся быть размещёнными на тех CPU, которые выглядят менее дорогими
для планировщика.  Фактически это приводит к тому, что менее производительные и
менее загруженные CPU предпочитаются, пока у них достаточно свободной ёмкости для
выполнения данной задачи, что в общем случае ведёт к снижению энергопотребления.

Модель энергопотребления, создаваемую ``intel_pstate``, можно изучить, просмотрев
каталог ``energy_model`` в ``debugfs`` (обычно смонтированном на
``/sys/kernel/debug/``).


Интерфейс пространства пользователя в ``sysfs``
===============================================

.. _global_attributes:

Глобальные атрибуты
-------------------

``intel_pstate`` предоставляет несколько глобальных атрибутов (файлов) в
``sysfs`` для управления своей функциональностью на уровне системы.  Они
расположены в каталоге ``/sys/devices/system/cpu/intel_pstate/`` и влияют на все
CPU.

Некоторые из них отсутствуют, если ядру в командной строке передан аргумент
``intel_pstate=per_cpu_perf_limits``.

``max_perf_pct``
	Максимальный P-state, который драйверу разрешено устанавливать, в
	процентах от максимального поддерживаемого уровня производительности
	(наивысшего поддерживаемого :ref:`turbo P-state <turbo>`).

	Этот атрибут не будет предоставлен, если в командной строке ядра
	присутствует аргумент ``intel_pstate=per_cpu_perf_limits``.

``min_perf_pct``
	Минимальный P-state, который драйверу разрешено устанавливать, в
	процентах от максимального поддерживаемого уровня производительности
	(наивысшего поддерживаемого :ref:`turbo P-state <turbo>`).

	Этот атрибут не будет предоставлен, если в командной строке ядра
	присутствует аргумент ``intel_pstate=per_cpu_perf_limits``.

``num_pstates``
	Число P-states, поддерживаемых процессором (от 0 до 255 включительно),
	включая как turbo, так и non-turbo P-states (см. :ref:`turbo`).

	Этот атрибут присутствует только если предоставляемое им значение
	одинаково для всех CPU в системе.

	Значение этого атрибута не зависит от настройки ``no_turbo``, описанной
	:ref:`ниже <no_turbo_attr>`.

	Этот атрибут доступен только для чтения.

``turbo_pct``
	Отношение размера :ref:`turbo-диапазона <turbo>` к размеру всего диапазона
	поддерживаемых P-states, в процентах.

	Этот атрибут присутствует только если предоставляемое им значение
	одинаково для всех CPU в системе.

	Этот атрибут доступен только для чтения.

.. _no_turbo_attr:

``no_turbo``
	Если установлен (равен 1), драйверу не разрешается устанавливать
	какие-либо turbo P-states (см. :ref:`turbo`).  Если сброшен (равен 0, что
	является значением по умолчанию), turbo P-states могут устанавливаться
	драйвером.
	[Обратите внимание, что ``intel_pstate`` не поддерживает общий атрибут
	``boost`` (поддерживаемый некоторыми другими драйверами масштабирования),
	который заменяется этим.]

	Этот атрибут не влияет на значение максимальной поддерживаемой частоты,
	предоставляемое ядру ``CPUFreq`` и доступное через интерфейс политики, но
	он влияет на максимально возможное значение лимитов P-state на уровне
	политики (см. :ref:`policy_attributes_interpretation` ниже для
	подробностей).

``hwp_dynamic_boost``
	Этот атрибут присутствует только если ``intel_pstate`` работает в
	:ref:`активном режиме с включённой функцией HWP <active_mode_hwp>` в
	процессоре.  Если установлен (равен 1), он приводит к тому, что лимит
	минимального P-state динамически повышается на короткое время всякий раз,
	когда задача, ранее ожидавшая ввода-вывода (I/O), выбирается для
	выполнения на данном логическом CPU (цель этого механизма — улучшить
	производительность).

	Эта настройка не влияет на логические CPU, чей лимит минимального P-state
	непосредственно установлен на наивысший non-turbo P-state или выше него.

.. _status_attr:

``status``
	Режим работы драйвера: "active", "passive" или "off".

	"active"
		Драйвер функционален и находится в :ref:`активном режиме
		<active_mode>`.

	"passive"
		Драйвер функционален и находится в :ref:`пассивном режиме
		<passive_mode>`.

	"off"
		Драйвер нефункционален (он не зарегистрирован как драйвер
		масштабирования в ядре ``CPUFreq``).

	В этот атрибут можно записывать, чтобы изменить режим работы драйвера или
	отменить его регистрацию.  Записываемая в него строка должна быть одним из
	его возможных значений, и в случае успеха запись приведёт к тому, что
	драйвер переключится в режим работы, представленный этой строкой, — или к
	отмене его регистрации в случае "off".  [На самом деле переключение из
	активного режима в пассивный или наоборот приводит к отмене регистрации
	драйвера и его повторной регистрации с другим набором колбэков, поэтому все
	его настройки (как глобальные, так и на уровне политики) затем сбрасываются
	к их значениям по умолчанию, возможно, в зависимости от целевого режима
	работы.]

``energy_efficiency``
	Этот атрибут присутствует только на платформах с CPU, соответствующими
	модели десктопного CPU Kaby Lake или Coffee Lake. По умолчанию
	оптимизации энергоэффективности отключены на этих моделях CPU, если
	включён HWP.  Включение оптимизаций энергоэффективности может ограничить
	максимальную рабочую частоту с функцией HWP или без неё.  С включённым HWP
	оптимизации выполняются только в turbo-диапазоне частот.  Без него они
	выполняются во всём доступном диапазоне частот.  Установка этого атрибута в
	"1" включает оптимизации энергоэффективности, а установка в "0" отключает
	их.

.. _policy_attributes_interpretation:

Интерпретация атрибутов политики
--------------------------------

Интерпретация некоторых атрибутов политики ``CPUFreq``, описанных в
Documentation/admin-guide/pm/cpufreq.rst, является особой, когда ``intel_pstate``
является текущим драйвером масштабирования, и в общем случае она зависит от
:ref:`режима работы <operation_modes>` драйвера.

Прежде всего, значения атрибутов ``cpuinfo_max_freq``, ``cpuinfo_min_freq`` и
``scaling_cur_freq`` получаются путём применения процессорно-специфичного
множителя к внутреннему представлению P-state, используемому ``intel_pstate``.
Также значения атрибутов ``scaling_max_freq`` и ``scaling_min_freq`` ограничены
сверху частотой, соответствующей максимальному P-state, который драйверу
разрешено устанавливать.

Если установлен :ref:`глобальный атрибут <no_turbo_attr>` ``no_turbo``, драйверу
не разрешается использовать turbo P-states, поэтому максимальное значение
``scaling_max_freq`` и ``scaling_min_freq`` ограничено частотой максимального
non-turbo P-state.
Соответственно, установка ``no_turbo`` приводит к тому, что ``scaling_max_freq`` и
``scaling_min_freq`` снижаются до этого значения, если ранее они были выше него.
Однако старые значения ``scaling_max_freq`` и ``scaling_min_freq`` будут
восстановлены после сброса ``no_turbo``, если только эти атрибуты не были записаны
после установки ``no_turbo``.

Если ``no_turbo`` не установлен, максимально возможное значение
``scaling_max_freq`` и ``scaling_min_freq`` соответствует максимальному
поддерживаемому turbo P-state, который также является значением
``cpuinfo_max_freq`` в любом случае.

Далее, следующие атрибуты политики имеют особое значение, если ``intel_pstate``
работает в :ref:`активном режиме <active_mode>`:

``scaling_available_governors``
	Список алгоритмов выбора P-state, предоставляемых ``intel_pstate``.

``scaling_governor``
	Алгоритм выбора P-state, предоставляемый ``intel_pstate`` и используемый в
	данный момент с данной политикой.

``scaling_cur_freq``
	Частота среднего P-state CPU, представленного данной политикой, за
	временной интервал между двумя последними вызовами колбэка обновления
	загрузки драйвера планировщиком CPU для этого CPU.

Ещё один атрибут политики присутствует, если в процессоре включена функция HWP:

``base_frequency``
	Показывает базовую частоту CPU. Любая частота выше этой будет находиться в
	turbo-диапазоне частот.

Значение этих атрибутов в :ref:`пассивном режиме <passive_mode>` такое же, как и
для других драйверов масштабирования.

Кроме того, значение атрибута ``scaling_driver`` для ``intel_pstate`` зависит от
режима работы драйвера.  А именно, это либо "intel_pstate" (в :ref:`активном
режиме <active_mode>`), либо "intel_cpufreq" (в :ref:`пассивном режиме
<passive_mode>`).

.. _pstate_limits_coordination:

Координация лимитов P-State
---------------------------

``intel_pstate`` позволяет устанавливать лимиты P-state двумя способами: с помощью
:ref:`глобальных атрибутов <global_attributes>` ``max_perf_pct`` и
``min_perf_pct`` или через атрибуты политики ``CPUFreq`` ``scaling_max_freq`` и
``scaling_min_freq``.  Координация между этими лимитами основана на следующих
правилах, независимо от текущего режима работы драйвера:

 1. Все CPU подвержены глобальным лимитам (то есть ни от одного из них нельзя
    потребовать работать быстрее глобального максимума, и ни от одного из них
    нельзя потребовать работать медленнее глобального минимума).

 2. Каждый отдельный CPU подвержен своим собственным лимитам на уровне политики
    (то есть от него нельзя потребовать работать быстрее его собственного
    максимума на уровне политики, и от него нельзя потребовать работать медленнее
    его собственного минимума на уровне политики). Эффективная производительность
    зависит от того, поддерживает ли платформа P-states на уровне ядра, включён
    ли hyper-threading, и от текущих запросов производительности от других CPU.
    Когда платформа не поддерживает P-states на уровне ядра, эффективная
    производительность может быть больше лимитов политики, установленных на CPU,
    если другие CPU в этот момент запрашивают более высокую производительность.
    Даже при поддержке P-states на уровне ядра, когда включён hyper-threading,
    если CPU-сиблинг запрашивает более высокую производительность, остальные
    сиблинги получат более высокую производительность, чем их лимиты политики.

 3. Глобальные лимиты и лимиты на уровне политики можно устанавливать независимо.

В :ref:`активном режиме с включённой функцией HWP <active_mode_hwp>`
результирующие эффективные значения записываются в аппаратные регистры всякий
раз, когда меняются лимиты, чтобы потребовать от его внутренней логики выбора
P-state всегда устанавливать P-states в пределах этих лимитов.  В противном
случае лимиты учитываются управляющими модулями масштабирования (в
:ref:`пассивном режиме <passive_mode>`) и драйвером каждый раз перед установкой
нового P-state для CPU.

Кроме того, если ядру передан аргумент командной строки
``intel_pstate=per_cpu_perf_limits``, ``max_perf_pct`` и ``min_perf_pct`` не
предоставляются вообще, и единственный способ установить лимиты — это
использование атрибутов политики.

.. _energy_performance_hints:

Подсказки «энергопотребление против производительности»
-------------------------------------------------------

Если в процессоре включены аппаратно управляемые P-states (HWP), в каждом каталоге
политики ``CPUFreq`` в ``sysfs`` присутствуют дополнительные атрибуты,
предназначенные для того, чтобы позволить пространству пользователя помочь
``intel_pstate`` скорректировать внутреннюю логику выбора P-state процессора,
сосредоточив её на производительности или на энергоэффективности, либо где-то
между этими двумя крайностями.  Это:

``energy_performance_preference``
	Текущее значение подсказки «энергопотребление против производительности»
	для данной политики (или CPU, представленного ею).

	Подсказку можно изменить записью в этот атрибут.

``energy_performance_available_preferences``
	Список строк, которые можно записать в атрибут
	``energy_performance_preference``.

	Они представляют различные подсказки «энергопотребление против
	производительности» и должны быть самоочевидны, за исключением того, что
	``default`` представляет то значение подсказки, которое было установлено
	прошивкой платформы.

Строки, записываемые в атрибут ``energy_performance_preference``, внутренне
переводятся в целочисленные значения, записываемые в регулятор предпочтения
энергопотребления и производительности процессора Energy-Performance Preference
(EPP) (если поддерживается) или в его регулятор смещения энергопотребления и
производительности Energy-Performance Bias (EPB). Также возможно записать
положительное целочисленное значение от 0 до 255, если присутствует функция EPP.
Если функция EPP отсутствует, запись целочисленного значения в этот атрибут не
поддерживается. В этом случае пользователь может использовать интерфейс
"/sys/devices/system/cpu/cpu*/power/energy_perf_bias".

[Обратите внимание, что задачи могут мигрировать с одного CPU на другой
алгоритмом балансировки нагрузки планировщика, и если для этих CPU установлены
разные подсказки «энергопотребление против производительности», это может
привести к нежелательным результатам.  Чтобы избежать таких проблем, лучше
установить одну и ту же подсказку «энергопотребление против производительности»
для всех CPU или закрепить каждую задачу, потенциально чувствительную к ним, за
конкретным CPU.]

.. _acpi-cpufreq:

``intel_pstate`` против ``acpi-cpufreq``
========================================

На большинстве систем, поддерживаемых ``intel_pstate``, таблицы ACPI,
предоставляемые прошивкой платформы, содержат объекты ``_PSS``, возвращающие
информацию, которую можно использовать для масштабирования производительности CPU
(подробности об объектах ``_PSS`` и формате возвращаемой ими информации см. в
спецификации ACPI [3]_).

Информация, возвращаемая объектами ACPI ``_PSS``, используется драйвером
масштабирования ``acpi-cpufreq``.  На системах, поддерживаемых ``intel_pstate``,
драйвер ``acpi-cpufreq`` использует тот же аппаратный интерфейс масштабирования
производительности CPU, но набор P-states, который он может использовать,
ограничен выводом ``_PSS``.

На этих системах каждый объект ``_PSS`` возвращает список P-states,
поддерживаемых соответствующим CPU, который, по сути, является подмножеством
диапазона P-states, который может использоваться ``intel_pstate`` на той же
системе, с одним исключением: весь :ref:`turbo-диапазон <turbo>` представлен в нём
одним элементом (самым верхним).  По соглашению, частота, возвращаемая ``_PSS``
для этого элемента, на 1 МГц больше частоты наивысшего non-turbo P-state,
указанного в нём, но соответствующее представление P-state (следующее аппаратной
спецификации), возвращаемое для него, соответствует максимальному поддерживаемому
turbo P-state (или является специальным значением 255, означающим, по сути,
«забирайся так высоко, как только сможешь»).

Список P-states, возвращаемый ``_PSS``, отражается таблицей доступных частот,
предоставляемой ``acpi-cpufreq`` ядру ``CPUFreq`` и управляющим модулям
масштабирования, и минимальная и максимальная поддерживаемые частоты, сообщаемые
им, также берутся из этого списка.  В частности, с учётом описанного выше
специального представления turbo-диапазона, это означает, что максимальная
поддерживаемая частота, сообщаемая ``acpi-cpufreq``, на 1 МГц выше частоты
наивысшего поддерживаемого non-turbo P-state, указанного ``_PSS``, что, разумеется,
влияет на решения, принимаемые управляющими модулями масштабирования, за
исключением ``powersave`` и ``performance``.

Например, если данный управляющий модуль пытается выбрать частоту,
пропорциональную оценочной нагрузке CPU, и отображает нагрузку 100% на
максимальную поддерживаемую частоту (возможно, умноженную на константу), то он
будет стремиться выбирать P-states ниже turbo-порога, если в качестве драйвера
масштабирования используется ``acpi-cpufreq``, потому что в этом случае
turbo-диапазон соответствует малой доле полосы частот, которую он может
использовать (1 МГц против 1 ГГц или более).  Как следствие, он будет переходить в
turbo-диапазон только для наивысших нагрузок, а другим нагрузкам выше 50%, которые
могли бы выиграть от работы на turbo-частотах, вместо этого будут даны non-turbo
P-states.

Ещё одна связанная с этим проблема может появиться на системах, поддерживающих
:ref:`функцию Configurable TDP <turbo>`, позволяющую прошивке платформы
устанавливать turbo-порог.  А именно, если это не согласовано должным образом со
списками P-states, возвращаемыми ``_PSS``, в этих списках может быть более одного
элемента, соответствующего turbo P-state, и может возникнуть проблема с избеганием
turbo-диапазона (если это желательно или необходимо).  Обычно, чтобы вообще
избежать использования turbo P-states, ``acpi-cpufreq`` просто избегает
использования самого верхнего состояния, указанного ``_PSS``, но этого
недостаточно, когда в возвращаемом им списке есть другие turbo P-states.

Помимо вышеизложенного, ``acpi-cpufreq`` работает как ``intel_pstate`` в
:ref:`пассивном режиме <passive_mode>`, за исключением того, что число P-states,
которое он может устанавливать, ограничено перечисленными объектами ACPI ``_PSS``.


Параметры командной строки ядра для ``intel_pstate``
====================================================

Несколько параметров командной строки ядра можно использовать для передачи
``intel_pstate`` параметров на этапе ранней конфигурации, чтобы навязать ему
определённое поведение.  Все они должны предваряться префиксом
``intel_pstate=``.

``disable``
	Не регистрировать ``intel_pstate`` как драйвер масштабирования, даже если
	процессор им поддерживается.

``active``
	Зарегистрировать ``intel_pstate`` в :ref:`активном режиме <active_mode>`
	для начала.

``passive``
	Зарегистрировать ``intel_pstate`` в :ref:`пассивном режиме <passive_mode>`
	для начала.

``force``
	Зарегистрировать ``intel_pstate`` как драйвер масштабирования вместо
	``acpi-cpufreq``, даже если последний предпочтительнее на данной системе.

	Это может помешать некоторым функциям платформы (таким как термоконтроль и
	ограничение мощности (power capping)), которые полагаются на доступность
	информации о P-states ACPI, работать ожидаемым образом, поэтому это
	следует использовать с осторожностью.

	Эта опция не работает с процессорами, которые не поддерживаются
	``intel_pstate``, и на платформах, где вместо ``acpi-cpufreq``
	используется драйвер масштабирования ``pcc-cpufreq``.

``no_hwp``
	Не включать функцию аппаратно управляемых P-states (HWP), даже если она
	поддерживается процессором.

``hwp_only``
	Регистрировать ``intel_pstate`` как драйвер масштабирования, только если
	функция аппаратно управляемых P-states (HWP) поддерживается процессором.

``support_acpi_ppc``
	Учитывать лимиты производительности ACPI ``_PPC``.

	Если предпочтительный профиль управления питанием в FADT (Fixed ACPI
	Description Table) установлен в "Enterprise Server" или "Performance
	Server", лимиты ACPI ``_PPC`` учитываются по умолчанию, и эта опция не
	имеет эффекта.

``per_cpu_perf_limits``
	Использовать лимиты P-State на уровне отдельного логического CPU (см.
	:ref:`pstate_limits_coordination` для подробностей).

``no_cas``
	Не включать :ref:`планирование с учётом ёмкости <CAS>`, которое включено
	по умолчанию на гибридных системах без SMT.

Диагностика и настройка
=======================

События трассировки
-------------------

Существует два статических события трассировки, которые можно использовать для
диагностики ``intel_pstate``.  Одно из них — событие трассировки
``cpu_frequency``, обычно используемое ``CPUFreq``, а другое — событие
трассировки ``pstate_sample``, специфичное для ``intel_pstate``.  Оба они
запускаются ``intel_pstate`` только если он работает в :ref:`активном режиме
<active_mode>`.

Следующую последовательность команд оболочки можно использовать для их включения
и просмотра их вывода (если ядро в целом сконфигурировано для поддержки
трассировки событий)::

 # cd /sys/kernel/tracing/
 # echo 1 > events/power/pstate_sample/enable
 # echo 1 > events/power/cpu_frequency/enable
 # cat trace
 gnome-terminal--4510  [001] ..s.  1177.680733: pstate_sample: core_busy=107 scaled=94 from=26 to=26 mperf=1143818 aperf=1230607 tsc=29838618 freq=2474476
 cat-5235  [002] ..s.  1177.681723: cpu_frequency: state=2900000 cpu_id=2

Если ``intel_pstate`` работает в :ref:`пассивном режиме <passive_mode>`, событие
трассировки ``cpu_frequency`` будет запускаться либо управляющим модулем
масштабирования ``schedutil`` (для политик, к которым он прикреплён), либо ядром
``CPUFreq`` (для политик с другими управляющими модулями масштабирования).

``ftrace``
----------

Интерфейс ``ftrace`` можно использовать для низкоуровневой диагностики
``intel_pstate``.  Например, чтобы проверить, как часто вызывается функция
установки P-state, фильтр ``ftrace`` можно установить на
:c:func:`intel_pstate_set_pstate`::

 # cd /sys/kernel/tracing/
 # cat available_filter_functions | grep -i pstate
 intel_pstate_set_pstate
 intel_pstate_cpu_init
 ...
 # echo intel_pstate_set_pstate > set_ftrace_filter
 # echo function > current_tracer
 # cat trace | head -15
 # tracer: function
 #
 # entries-in-buffer/entries-written: 80/80   #P:4
 #
 #                              _-----=> irqs-off
 #                             / _----=> need-resched
 #                            | / _---=> hardirq/softirq
 #                            || / _--=> preempt-depth
 #                            ||| /     delay
 #           TASK-PID   CPU#  ||||    TIMESTAMP  FUNCTION
 #              | |       |   ||||       |         |
             Xorg-3129  [000] ..s.  2537.644844: intel_pstate_set_pstate <-intel_pstate_timer_func
  gnome-terminal--4510  [002] ..s.  2537.649844: intel_pstate_set_pstate <-intel_pstate_timer_func
      gnome-shell-3409  [001] ..s.  2537.650850: intel_pstate_set_pstate <-intel_pstate_timer_func
           <idle>-0     [000] ..s.  2537.654843: intel_pstate_set_pstate <-intel_pstate_timer_func
