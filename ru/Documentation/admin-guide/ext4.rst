==========================
Общие сведения об ext4
==========================

Ext4 — это усовершенствованная версия файловой системы ext3, в которую включены
улучшения масштабируемости и надёжности для поддержки больших файловых систем
(64-битных) в соответствии с растущими ёмкостями дисков и современными
требованиями к функциональности.

Список рассылки:	linux-ext4@vger.kernel.org
Веб-сайт:	http://ext4.wiki.kernel.org


Краткая инструкция по использованию
===================================

Примечание. Более подробную информацию о начале работы с ext4 можно найти
на wiki-сайте ext4 по адресу:
http://ext4.wiki.kernel.org/index.php/Ext4_Howto

  - Последнюю версию e2fsprogs можно найти по адресу:

    https://www.kernel.org/pub/linux/kernel/people/tytso/e2fsprogs/

	или

    http://sourceforge.net/project/showfiles.php?group_id=2406

	либо взять последний git-репозиторий по адресу:

   https://git.kernel.org/pub/scm/fs/ext2/e2fsprogs.git

  - Создайте новую файловую систему, используя тип файловой системы ext4:

        # mke2fs -t ext4 /dev/hda1

    Либо, чтобы настроить существующую файловую систему ext3 для поддержки экстентов:

	# tune2fs -O extents /dev/hda1

    Если файловая система была создана со 128-байтовыми inode, её можно
    преобразовать к использованию 256 байт для большей эффективности с помощью:

        # tune2fs -I 256 /dev/hda1

  - Монтирование:

	# mount -t ext4 /dev/hda1 /wherever

  - При сравнении производительности с другими файловыми системами всегда
    важно опробовать несколько типов нагрузки; очень часто незаметное изменение
    параметра нагрузки может полностью изменить расстановку того, какие
    файловые системы хорошо себя показывают по сравнению с другими.  При
    сравнении с ext3 учитывайте, что ext4 по умолчанию включает барьеры записи,
    тогда как ext3 по умолчанию барьеры записи не включает.  Поэтому для
    честного сравнения полезно явно указывать, включены барьеры или нет, через
    опцию монтирования '-o barriers=[0|1]' как для файловых систем ext3, так и
    для ext4.  При настройке ext3 для получения наилучших результатов
    бенчмарков часто имеет смысл попробовать изменить режим журналирования
    данных; '-o data=writeback' может быть быстрее для некоторых типов
    нагрузки.  (Однако учтите, что работа в смонтированном состоянии с
    data=writeback потенциально может оставить устаревшие данные доступными в
    недавно записанных файлах в случае нечистого завершения работы, что в
    некоторых ситуациях может представлять собой проблему безопасности.)
    Настройка файловой системы с большим журналом также может быть полезна для
    нагрузок, интенсивно работающих с метаданными.

Возможности
===========

Доступные в настоящее время
---------------------------

* возможность использования файловых систем > 16ТБ (поддержка в e2fsprogs пока недоступна)
* формат экстентов снижает накладные расходы на метаданные (RAM, IO для доступа, транзакции)
* формат экстентов более устойчив к повреждениям на диске благодаря magic-значениям,
* внутренняя избыточность в дереве
* улучшенное выделение файлов (многоблочное выделение)
* снятие ограничения в 32000 подкаталогов, налагаемого i_links_count[1]
* временные метки с наносекундной точностью для mtime, atime, ctime, времени создания
* поле версии inode на диске (NFSv4, Lustre)
* сокращение времени e2fsck благодаря возможности uninit_bg
* контрольное суммирование журнала ради надёжности и производительности
* постоянное предварительное выделение файлов (например, для потоковых медиа, баз данных)
* возможность упаковывать битовые карты и таблицы inode в более крупные виртуальные
  группы с помощью возможности flex_bg
* поддержка больших файлов
* выделение inode с использованием больших виртуальных групп блоков через flex_bg
* отложенное выделение
* поддержка больших блоков (вплоть до размера страницы)
* эффективный новый режим ordered в JBD2 и ext4 (без использования buffer head для
  принудительного упорядочивания)
* поиск имён файлов без учёта регистра
* поддержка пофайлового шифрования (fscrypt)
* поддержка пофайловой проверки целостности (fsverity)

[1] Файловые системы с размером блока 1k могут столкнуться с ограничением,
налагаемым тем, что хеш-дерево каталогов имеет максимальную глубину два.

поиск имён файлов без учёта регистра
======================================================

Возможность поиска имён файлов без учёта регистра поддерживается на
покаталожной основе, что позволяет пользователю смешивать каталоги без учёта
регистра и каталоги с учётом регистра в одной файловой системе.  Она
включается переключением атрибута inode +F пустого каталога.  Операция
сравнения строк без учёта регистра определена только тогда, когда мы знаем, как
текст закодирован в последовательности байтов.  По этой причине для включения
каталогов без учёта регистра файловая система должна обладать возможностью
casefold, которая хранит общефайловую модель кодирования.  По умолчанию
принятая кодировка — это последняя версия Unicode (12.1.0 на момент написания),
закодированная в форме UTF-8.  Алгоритм сравнения реализован путём нормализации
строк к канонической форме декомпозиции (Canonical decomposition), как
определено Unicode, с последующим побайтовым сравнением.

Учёт регистра сохраняет имена на диске, что означает, что имя файла,
предоставленное пространством пользователя, побайтово совпадает с тем, что
фактически записано на диск.  Таким образом, формат нормализации Unicode,
используемый ядром, является внутренним представлением и не раскрывается ни
пространству пользователя, ни диску, за важным исключением дисковых хешей,
используемых в больших каталогах без учёта регистра с возможностью DX.  В
каталогах DX хеш должен вычисляться с использованием casefold-версии имени
файла, что означает, что используемый формат нормализации фактически влияет на
то, где хранится запись каталога.

Когда мы переходим от рассмотрения имён файлов как непрозрачных
последовательностей байтов к рассмотрению их как закодированных строк, нам нужно
решить, что происходит, когда программа пытается создать файл с недопустимым
именем.  Подсистема Unicode внутри ядра оставляет решение о том, что делать в
этом случае, на усмотрение файловой системы, которая выбирает предпочтительное
поведение, включая/отключая строгий режим.  Когда Ext4 встречает одну из таких
строк, а файловая система не требовала строгого режима, она прибегает к
рассмотрению всей строки как непрозрачной последовательности байтов, что
по-прежнему позволяет пользователю работать с этим файлом, но поиск без учёта
регистра работать не будет.

Опции
=====

При монтировании файловой системы ext4 принимаются следующие опции:
(*) == по умолчанию

  ro
        Монтировать файловую систему только для чтения. Учтите, что ext4
        воспроизведёт журнал (и, следовательно, выполнит запись в раздел) даже
        при монтировании "только для чтения". Опции монтирования "ro,noload"
        можно использовать для предотвращения записи в файловую систему.

  journal_checksum
        Включить контрольное суммирование транзакций журнала.  Это позволит коду
        восстановления в e2fsck и в ядре обнаруживать повреждения в ядре.  Это
        совместимое изменение, и оно будет игнорироваться более старыми ядрами.

  journal_async_commit
        Блок фиксации (commit) может быть записан на диск без ожидания блоков
        дескрипторов. Если эта опция включена, более старые ядра не смогут
        смонтировать устройство. Это внутренне включит 'journal_checksum'.

  journal_path=path, journal_dev=devnum
        Когда старший/младший номера внешнего журнального устройства изменились,
        эти опции позволяют пользователю указать новое расположение журнала.
        Журнальное устройство идентифицируется либо по его новым старшим/младшим
        номерам, закодированным в devnum, либо по пути к устройству.

  norecovery, noload
        Не загружать журнал при монтировании.  Учтите, что если файловая система
        была размонтирована нечисто, пропуск воспроизведения журнала приведёт к
        тому, что файловая система будет содержать несогласованности, которые
        могут привести к любому числу проблем.

  data=journal
        Все данные фиксируются в журнал перед записью в основную файловую
        систему.  Включение этого режима отключит отложенное выделение и
        поддержку O_DIRECT.

  data=ordered	(*)
        Все данные принудительно выводятся непосредственно в основную файловую
        систему перед тем, как их метаданные будут зафиксированы в журнал.

  data=writeback
        Упорядочивание данных не сохраняется, данные могут быть записаны в
        основную файловую систему после того, как их метаданные были
        зафиксированы в журнал.

  commit=nrsec	(*)
        Эта настройка ограничивает максимальный возраст выполняющейся транзакции
        значением 'nrsec' секунд.  Значение по умолчанию — 5 секунд.  Это
        означает, что если вы потеряете питание, вы потеряете не более чем
        последние 5 секунд изменений метаданных (хотя ваша файловая система не
        будет повреждена благодаря журналированию). Это значение по умолчанию
        (или любое низкое значение) повредит производительности, но оно хорошо
        для безопасности данных.  Установка его в 0 будет иметь тот же эффект,
        что и оставление его в значении по умолчанию (5 секунд).  Установка его в
        очень большие значения улучшит производительность.  Учтите, что из-за
        отложенного выделения даже более старые данные могут быть потеряны при
        сбое питания, поскольку обратная запись (writeback) этих данных
        начинается только по истечении времени, заданного в
        /proc/sys/vm/dirty_expire_centisecs.

  barrier=<0|1(*)>, barrier(*), nobarrier
        Это включает/отключает использование барьеров записи в коде jbd.
        barrier=0 отключает, barrier=1 включает.  Это также требует стека IO,
        который может поддерживать барьеры, и если jbd получит ошибку при записи
        барьера, он снова отключится с предупреждением.  Барьеры записи
        обеспечивают правильное упорядочивание фиксаций журнала на диске, делая
        безопасным использование энергозависимых кэшей записи диска, ценой
        некоторого снижения производительности.  Если ваши диски тем или иным
        способом снабжены резервным питанием от батареи, отключение барьеров
        может безопасно улучшить производительность.  Опции монтирования
        "barrier" и "nobarrier" также можно использовать для включения или
        отключения барьеров, для согласованности с другими опциями монтирования
        ext4.

  inode_readahead_blks=n
        Этот настроечный параметр управляет максимальным числом блоков таблицы
        inode, которые алгоритм упреждающего чтения таблицы inode в ext4 будет
        предварительно считывать в буферный кэш.  Значение по умолчанию — 32
        блока.

  bsddf	(*)
        Заставить 'df' вести себя как в BSD.

  minixdf
        Заставить 'df' вести себя как в Minix.

  debug
        Дополнительная отладочная информация отправляется в syslog.

  abort
        Симулировать эффект вызова ext4_abort() для целей отладки.  Обычно это
        используется при перемонтировании файловой системы, которая уже
        смонтирована.

  errors=remount-ro
        Перемонтировать файловую систему только для чтения при ошибке.

  errors=continue
        Продолжать работу при ошибке файловой системы.

  errors=panic
        Вызвать панику и остановить машину при возникновении ошибки.  (Эти опции
        монтирования переопределяют поведение при ошибках, заданное в
        суперблоке, которое можно настроить с помощью tune2fs)

  data_err=ignore(*)
        Просто вывести сообщение об ошибке, если ошибка возникает в буфере данных
        файла.

  data_err=abort
        Прервать журнал, если ошибка возникает в буфере данных файла.

  grpid | bsdgroups
        Новые объекты получают идентификатор группы своего родителя.

  nogrpid (*) | sysvgroups
        Новые объекты получают идентификатор группы своего создателя.

  resgid=n
        Идентификатор группы, которая может использовать зарезервированные блоки.

  resuid=n
        Идентификатор пользователя, который может использовать зарезервированные
        блоки.

  sb=
        Использовать альтернативный суперблок в этом расположении.

  quota, noquota, grpquota, usrquota
        Эти опции игнорируются файловой системой. Они используются только
        инструментами квотирования для распознавания томов, на которых должны
        быть включены квоты. См. документацию в пакете quota-tools для
        дополнительных сведений
        (http://sourceforge.net/projects/linuxquota).

  jqfmt=<quota type>, usrjquota=<file>, grpjquota=<file>
        Эти опции сообщают файловой системе подробности о квотах, чтобы
        информация о квотах могла правильно обновляться во время воспроизведения
        журнала. Они заменяют приведённые выше опции квот. См. документацию в
        пакете quota-tools для дополнительных сведений
        (http://sourceforge.net/projects/linuxquota).

  stripe=n
        Число блоков файловой системы, которое mballoc будет пытаться
        использовать в качестве размера и выравнивания выделения. Для систем
        RAID5/6 это должно быть число дисков с данными *  размер RAID-чанка в
        блоках файловой системы.

  delalloc	(*)
        Откладывать выделение блоков до момента непосредственно перед тем, как
        ext4 запишет соответствующий блок(и).  Это позволяет ext4 более
        эффективно принимать лучшие решения о выделении.

  nodelalloc
        Отключить отложенное выделение.  Блоки выделяются, когда данные
        копируются из пространства пользователя в страничный кэш, либо через
        системный вызов write(2), либо когда mmap'нутая страница, которая ранее
        не была выделена, записывается в первый раз.

  max_batch_time=usec
        Максимальное время, которое ext4 должна ждать дополнительных операций
        файловой системы, чтобы объединить их в пакет с синхронной операцией
        записи.  Поскольку синхронная операция записи всё равно вызовет фиксацию,
        а затем ожидание завершения I/O, это стоит немного и может дать огромный
        выигрыш в пропускной способности, поэтому мы ждём небольшое время, чтобы
        увидеть, могут ли какие-либо другие транзакции присоединиться к синхронной
        записи.   Используемый алгоритм спроектирован для автоматической настройки
        под скорость диска путём измерения количества времени (в среднем), которое
        требуется для завершения фиксации транзакции.  Назовём это время "временем
        фиксации".  Если время, в течение которого выполнялась транзакция, меньше
        времени фиксации, ext4 попытается поспать в течение времени фиксации,
        чтобы увидеть, присоединятся ли другие операции к транзакции.   Время
        фиксации ограничивается сверху значением max_batch_time, которое по
        умолчанию равно 15000us (15мс).   Эту оптимизацию можно полностью
        отключить, установив max_batch_time в 0.

  min_batch_time=usec
        Этот параметр задаёт время фиксации (как описано выше) равным как минимум
        min_batch_time.  По умолчанию оно равно нулю микросекунд.  Увеличение
        этого параметра может улучшить пропускную способность многопоточных,
        синхронных нагрузок на очень быстрых дисках ценой увеличения задержки
        (latency).

  journal_ioprio=prio
        Приоритет I/O (от 0 до 7, где 0 — наивысший приоритет), который должен
        использоваться для операций I/O, отправляемых kjournald2 во время
        операции фиксации.  По умолчанию он равен 3, что немного выше приоритета
        I/O по умолчанию.

  auto_da_alloc(*), noauto_da_alloc
        Многие сломанные приложения не используют fsync() при замене
        существующих файлов с помощью шаблонов вроде fd = open("foo.new")/write(fd,..)/close(fd)/
        rename("foo.new", "foo") или, что ещё хуже, fd = open("foo",
        O_TRUNC)/write(fd,..)/close(fd).  Если auto_da_alloc включена, ext4
        обнаружит шаблоны «замена-через-rename» и «замена-через-truncate» и
        принудительно сделает так, чтобы любые отложенно выделяемые блоки были
        выделены таким образом, что при следующей фиксации журнала, в режиме по
        умолчанию data=ordered, блоки данных нового файла принудительно
        записываются на диск до того, как будет зафиксирована операция rename().
        Это обеспечивает примерно тот же уровень гарантий, что и ext3, и
        позволяет избежать проблемы "нулевой длины", которая может возникнуть,
        когда система аварийно завершается до того, как отложенно выделяемые блоки
        будут принудительно записаны на диск.

  noinit_itable
        Не инициализировать в фоновом режиме какие-либо неинициализированные
        блоки таблицы inode.  Эта возможность может использоваться
        установочными CD, чтобы процесс установки мог завершиться как можно
        быстрее; инициализация таблицы inode тогда будет отложена до следующего
        размонтирования файловой системы.

  init_itable=n
        Код ленивой инициализации itable будет ждать в n раз дольше числа
        миллисекунд, потребовавшегося на обнуление таблицы inode предыдущей
        группы блоков.  Это минимизирует влияние на производительность системы во
        время инициализации таблицы inode файловой системы.

  discard, nodiscard(*)
        Управляет тем, должна ли ext4 выдавать команды discard/TRIM
        нижележащему блочному устройству при освобождении блоков.  Это полезно
        для устройств SSD и разреженных/thin-provisioned LUN, но по умолчанию это
        отключено до тех пор, пока не будет проведено достаточное тестирование.

  nouid32
        Отключает 32-битные UID и GID.  Это для совместимости с более старыми
        ядрами, которые хранят и ожидают только 16-битные значения.

  block_validity(*), noblock_validity
        Эти опции включают или отключают внутриядерный механизм отслеживания
        блоков метаданных файловой системы во внутренних структурах данных.  Это
        позволяет многоблочному аллокатору и другим процедурам замечать ошибки
        или повреждённые битовые карты выделения, из-за которых выделяются блоки,
        перекрывающиеся с блоками метаданных файловой системы.

  dioread_lock, dioread_nolock
        Управляет тем, должна ли ext4 использовать блокировку чтения DIO. Если
        указана опция dioread_nolock, ext4 будет выделять неинициализированный
        экстент перед записью буфера и преобразовывать экстент в
        инициализированный после завершения IO. Этот подход позволяет коду ext4
        избегать использования мьютекса (mutex) inode, что улучшает
        масштабируемость на высокоскоростных хранилищах. Однако это не работает с
        журналированием данных, и опция dioread_nolock будет проигнорирована с
        предупреждением ядра. Учтите, что путь кода dioread_nolock используется
        только для файлов на основе экстентов.  Из-за ограничений, которые
        включает в себя эта опция, по умолчанию она отключена (т.е.
        dioread_lock).

  max_dir_size_kb=n
        Это ограничивает размер каталогов, так что любая попытка расширить их за
        пределы указанного предела в килобайтах вызовет ошибку ENOSPC.  Это
        полезно в средах с ограниченной памятью, где очень большой каталог может
        вызвать серьёзные проблемы с производительностью или даже спровоцировать
        Out Of Memory killer.  (Например, если доступно всего 512мб памяти,
        каталог в 176мб может серьёзно стеснить систему.)

  i_version
        Включить поддержку 64-битной версии inode. Эта опция по умолчанию
        отключена.

  dax
        Использовать прямой доступ (без страничного кэша).  См.
        Documentation/filesystems/dax.rst.  Учтите, что эта опция несовместима с
        data=journal.

  inlinecrypt
        Когда это возможно, шифровать/расшифровывать содержимое зашифрованных
        файлов с использованием фреймворка blk-crypto, а не шифрования на уровне
        файловой системы. Это позволяет использовать аппаратное встроенное
        (inline) шифрование. Формат на диске не затрагивается. Дополнительные
        сведения см. в
        Documentation/block/inline-encryption.rst.

Режим данных
============
Существует 3 различных режима данных:

* режим writeback

  В режиме data=writeback ext4 вообще не журналирует данные.  Этот режим
  обеспечивает уровень журналирования, аналогичный тому, что предоставляют XFS и
  JFS в своём режиме по умолчанию — журналирование метаданных.  Сбой и
  восстановление могут привести к появлению некорректных данных в файлах,
  которые были записаны незадолго до сбоя.  Этот режим обычно обеспечивает
  наилучшую производительность ext4.

* режим ordered

  В режиме data=ordered ext4 официально журналирует только метаданные, но
  логически группирует информацию метаданных, связанную с изменениями данных,
  вместе с блоками данных в единую сущность, называемую транзакцией.  Когда
  наступает время записать новые метаданные на диск, сначала записываются
  связанные блоки данных.  В общем случае этот режим работает чуть медленнее, чем
  writeback, но значительно быстрее, чем режим journal.

* режим journal

  Режим data=journal обеспечивает полное журналирование данных и метаданных.  Все
  новые данные сначала записываются в журнал, а затем в их окончательное
  расположение.  В случае сбоя журнал можно воспроизвести, приведя как данные,
  так и метаданные в согласованное состояние.  Этот режим самый медленный, за
  исключением случаев, когда данные нужно читать с диска и записывать на него
  одновременно — тогда он превосходит все остальные режимы.  Включение этого
  режима отключит отложенное выделение и поддержку O_DIRECT.

Записи /proc
============

Информацию о смонтированных файловых системах ext4 можно найти в
/proc/fs/ext4.  Каждая смонтированная файловая система будет иметь каталог в
/proc/fs/ext4 на основе имени своего устройства (т.е. /proc/fs/ext4/hdc или
/proc/fs/ext4/dm-0).   Файлы в каждом подкаталоге устройства показаны в таблице
ниже.

Файлы в /proc/fs/ext4/<devname>

  mb_groups
        подробности buddy-кэша свободных блоков многоблочного аллокатора

Записи /sys
===========

Информацию о смонтированных файловых системах ext4 можно найти в
/sys/fs/ext4.  Каждая смонтированная файловая система будет иметь каталог в
/sys/fs/ext4 на основе имени своего устройства (т.е. /sys/fs/ext4/hdc или
/sys/fs/ext4/dm-0).   Файлы в каждом подкаталоге устройства показаны в таблице
ниже.

Файлы в /sys/fs/ext4/<devname>:

(см. также Documentation/ABI/testing/sysfs-fs-ext4)

  delayed_allocation_blocks
        Этот файл доступен только для чтения и показывает число блоков, которые
        являются грязными (dirty) в страничном кэше, но для которых ещё не
        выделено расположение в файловой системе.

  inode_goal
        Настроечный параметр, который (если он ненулевой) управляет целевым
        inode, используемым аллокатором inode в предпочтение всем остальным
        эвристикам выделения.  Он предназначен только для использования при
        отладке и должен быть равен 0 в промышленных системах.

  inode_readahead_blks
        Настроечный параметр, который управляет максимальным числом блоков
        таблицы inode, которые алгоритм упреждающего чтения таблицы inode в ext4
        будет предварительно считывать в буферный кэш.

  lifetime_write_kbytes
        Этот файл доступен только для чтения и показывает число килобайт данных,
        которые были записаны в эту файловую систему с момента её создания.

  max_writeback_mb_bump
        Максимальное число мегабайт, которое код обратной записи (writeback)
        будет пытаться записать, прежде чем перейти к другому inode.

  mb_group_prealloc
        Многоблочный аллокатор будет округлять запросы на выделение вверх до
        кратного этому настроечному параметру, если размер stripe не задан в
        суперблоке ext4

  mb_max_to_scan
        Максимальное число экстентов, которое многоблочный аллокатор будет
        просматривать, чтобы найти наилучший экстент.

  mb_min_to_scan
        Минимальное число экстентов, которое многоблочный аллокатор будет
        просматривать, чтобы найти наилучший экстент.

  mb_order2_req
        Настроечный параметр, который управляет минимальным размером для запросов
        (как степень 2), при котором используется buddy-кэш.

  mb_stats
        Управляет тем, должен ли многоблочный аллокатор собирать статистику,
        которая показывается во время размонтирования. 1 означает собирать
        статистику, 0 означает не собирать статистику.

  mb_stream_req
        Файлы, у которых блоков меньше, чем этот настраиваемый параметр, будут
        иметь свои блоки выделенными из специфичного для группы блоков пула
        предварительного выделения, так что небольшие файлы упаковываются близко
        друг к другу.  Каждый большой файл будет иметь свои блоки выделенными из
        своего собственного уникального пула предварительного выделения.

  session_write_kbytes
        Этот файл доступен только для чтения и показывает число килобайт данных,
        которые были записаны в эту файловую систему с момента её монтирования.

  reserved_clusters
        Это файл RW (для чтения и записи), и он содержит число зарезервированных
        кластеров в файловой системе, которые будут использоваться в
        специфических ситуациях для избежания затратного обнуления (zeroout),
        неожиданного ENOSPC или возможной потери данных. Значение по умолчанию —
        2% или 4096 кластеров, в зависимости от того, что меньше, и это можно
        изменить, однако оно никогда не может превышать число кластеров в
        файловой системе. Если при монтировании недостаточно места для
        зарезервированного пространства, монтирование _не_ завершится с ошибкой.

Ioctls
======

Ext4 реализует различные ioctls, которые могут использоваться приложениями для
доступа к специфичной для ext4 функциональности. Неполный список этих ioctls
показан в таблице ниже. Этот список включает действительно специфичные для ext4
ioctls (``EXT4_IOC_*``), а также ioctls, которые, возможно, изначально были
специфичны для ext4, но теперь поддерживаются и некоторыми другими файловыми
системами (``FS_IOC_*``).

Таблица ioctls ext4

  FS_IOC_GETFLAGS
        Получить дополнительные атрибуты, ассоциированные с inode.  Аргумент
        ioctl — это целочисленное битовое поле, значения битов которого описаны
        в ext4.h.

  FS_IOC_SETFLAGS
        Установить дополнительные атрибуты, ассоциированные с inode.  Аргумент
        ioctl — это целочисленное битовое поле, значения битов которого описаны
        в ext4.h.

  EXT4_IOC_GETVERSION, EXT4_IOC_GETVERSION_OLD
        Получить номер i_generation inode, хранимый для каждого inode. Номер
        i_generation обычно изменяется только при создании нового inode, и он
        особенно полезен для сетевых файловых систем. Версия '_OLD' этого ioctl
        является псевдонимом для FS_IOC_GETVERSION.

  EXT4_IOC_SETVERSION, EXT4_IOC_SETVERSION_OLD
        Установить номер i_generation inode, хранимый для каждого inode. Версия
        '_OLD' этого ioctl является псевдонимом для FS_IOC_SETVERSION.

  EXT4_IOC_GROUP_EXTEND
        Этот ioctl имеет ту же цель, что и опция монтирования resize. Он
        позволяет изменить размер файловой системы до конца последней
        существующей группы блоков, дальнейшее изменение размера должно
        выполняться с помощью resize2fs, либо онлайн, либо офлайн. Аргумент
        указывает на беззнаковое длинное число, представляющее новое число блоков
        файловой системы.

  EXT4_IOC_MOVE_EXT
        Переместить блочные экстенты из orig_fd (тот, на который указывает этот
        ioctl) в donor_fd (тот, что указан в структуре move_extent, переданной в
        качестве аргумента этому ioctl). Затем обменять метаданные inode между
        orig_fd и donor_fd.  Это особенно полезно для онлайн-дефрагментации,
        поскольку у аллокатора появляется возможность лучше выделить
        перемещаемые блоки, в идеале в один непрерывный экстент.

  EXT4_IOC_GROUP_ADD
        Добавить новый дескриптор группы в существующий или новый блок
        дескрипторов групп. Новый дескриптор группы описывается структурой
        ext4_new_group_input, которая передаётся в качестве аргумента этому
        ioctl. Это особенно полезно в сочетании с EXT4_IOC_GROUP_EXTEND, который
        позволяет онлайн изменять размер файловой системы до конца последней
        существующей группы блоков.  Эти два ioctls, объединённые вместе,
        используются в пользовательском инструменте онлайн-изменения размера
        (например, resize2fs).

  EXT4_IOC_MIGRATE
        Этот ioctl работает с самой файловой системой.  Он преобразует
        (мигрирует) inode ext3 с косвенным отображением блоков в inode ext4 с
        отображением на основе экстентов, проходя по косвенному отображению
        блоков исходного inode и преобразуя непрерывные диапазоны блоков в
        экстенты ext4 временного inode. Затем inode меняются местами. Этот ioctl
        может помочь при миграции с файловой системы ext3 на ext4, однако
        рекомендация состоит в том, чтобы создать новую файловую систему ext4 и
        скопировать данные из резервной копии. Учтите, что для работы этого ioctl
        файловая система должна поддерживать экстенты.

  EXT4_IOC_ALLOC_DA_BLKS
        Принудительно выделить все отложенно выделяемые блоки, чтобы сохранить
        ожидаемое приложением поведение ext3. Учтите, что это также начнёт
        запускать запись блоков данных, но это поведение может измениться в
        будущем, поскольку оно не является необходимым и было сделано таким
        образом лишь ради простоты.

  EXT4_IOC_RESIZE_FS
        Изменить размер файловой системы до нового размера.  Число блоков
        файловой системы после изменения размера передаётся через 64-битный
        целочисленный аргумент.  Ядро выделяет битовые карты и таблицу inode,
        таким образом, пользовательский инструмент просто передаёт новое число
        блоков.

  EXT4_IOC_SWAP_BOOT
        Обменять i_blocks и ассоциированные атрибуты (такие как i_blocks,
        i_size, i_flags, ...) указанного inode с inode EXT4_BOOT_LOADER_INO
        (#5). Это обычно используется для хранения загрузчика в защищённой части
        файловой системы, где он не может быть случайно изменён обычным
        пользователем. Блоки данных предыдущего загрузчика будут
        ассоциированы с данным inode.
