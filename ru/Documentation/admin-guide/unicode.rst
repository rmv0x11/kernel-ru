Поддержка Unicode
=================

		 Последнее обновление: 2005-01-17, версия 1.4

Примечание: исходная версия этого документа, которая поддерживалась на
lanana.org в рамках проекта Linux Assigned Names And Numbers Authority
(LANANA), более не существует.  Поэтому данная версия в основной ветке
(mainline) ядра Linux теперь является поддерживаемым основным документом.

Введение
--------

Код ядра Linux был переписан так, чтобы использовать Unicode для
сопоставления символов со шрифтами.  Загрузив единую таблицу
Unicode-в-шрифт, можно изменить как восьмибитные наборы символов, так и
режим UTF-8, чтобы они использовали указанный шрифт.

Это немного меняет семантику восьмибитных таблиц символов.
Четыре таблицы символов теперь следующие:

=============== =============================== ================
Символ карты	Имя карты			Escape-код (G0)
=============== =============================== ================
LAT1_MAP	Latin-1 (ISO 8859-1)		ESC ( B
GRAF_MAP	Псевдографика DEC VT100		ESC ( 0
IBMPC_MAP	Кодовая страница IBM 437	ESC ( U
USER_MAP	Определяемая пользователем	ESC ( K
=============== =============================== ================

В частности, ESC ( U больше не означает "напрямую в шрифт", поскольку шрифт
может полностью отличаться от набора символов IBM.  Это позволяет, например,
использовать блочную графику даже при загруженном шрифте Latin-1.

Обратите внимание, что хотя эти коды похожи на ISO 2022, ни сами коды, ни их
использование не совпадают с ISO 2022; в Linux есть два 8-битных кода (G0 и
G1), тогда как в ISO 2022 четыре 7-битных кода (G0-G3).

В соответствии со стандартом Unicode/ISO 10646 диапазон от U+F000 до
U+F8FF зарезервирован для выделения в масштабе всей ОС (стандарт Unicode
называет это "Corporate Zone"; поскольку для Linux это неточно, мы называем
это "Linux Zone").  U+F000 был выбран в качестве начальной точки, так как это
позволяет области прямого отображения начинаться на большой степени двойки
(на случай, если когда-либо понадобятся шрифты на 1024 или 2048 символов).
В результате диапазон от U+E000 до U+EFFF остаётся как End User Zone.

[v1.2]: Юникод-коды в диапазоне от U+F000 и вплоть до U+F7FF жёстко заданы
для прямого отображения в загруженный шрифт в обход таблицы трансляции.
Определяемая пользователем карта теперь по умолчанию использует диапазон
от U+F000 до U+F0FF, эмулируя прежнее поведение.  На практике этот диапазон
может быть короче; например, vgacon может работать только со шрифтами на 256
символов (U+F000..U+F0FF) или 512 символов (U+F000..U+F1FF).


Реальные символы, назначенные в Linux Zone
------------------------------------------

Кроме того, были определены следующие символы, отсутствующие в Unicode 1.1.4;
они используются картой графики DEC VT.  [v1.2]
ЭТО ИСПОЛЬЗОВАНИЕ УСТАРЕЛО И БОЛЕЕ НЕ ДОЛЖНО ПРИМЕНЯТЬСЯ; СМ. НИЖЕ.

====== ======================================
U+F800 DEC VT GRAPHICS HORIZONTAL LINE SCAN 1
U+F801 DEC VT GRAPHICS HORIZONTAL LINE SCAN 3
U+F803 DEC VT GRAPHICS HORIZONTAL LINE SCAN 7
U+F804 DEC VT GRAPHICS HORIZONTAL LINE SCAN 9
====== ======================================

В DEC VT220 используется матрица символов 6x10, и эти символы образуют
плавную последовательность в наборе графических символов DEC VT.  Я опустил
линию scan 5, поскольку она также используется как символ блочной графики и
поэтому закодирована как U+2500 FORMS LIGHT HORIZONTAL.

[v1.3]: Эти символы были официально добавлены в Unicode 3.2.0;
они добавлены по адресам U+23BA, U+23BB, U+23BC, U+23BD.  Linux теперь
использует новые значения.

[v1.2]: Следующие символы были добавлены для представления распространённых
клавиатурных символов, которые вряд ли когда-либо будут добавлены в сам
Unicode, поскольку они ужасающе специфичны для конкретных производителей.
Это, разумеется, прекрасный пример ужасного дизайна.

====== ======================================
U+F810 KEYBOARD SYMBOL FLYING FLAG
U+F811 KEYBOARD SYMBOL PULLDOWN MENU
U+F812 KEYBOARD SYMBOL OPEN APPLE
U+F813 KEYBOARD SYMBOL SOLID APPLE
====== ======================================

Поддержка клингонского языка
----------------------------

В 1996 году Linux стал первой в мире операционной системой, добавившей
поддержку искусственного языка клингон, созданного Марком Окрандом
для телесериала "Star Trek".	Эта кодировка впоследствии была
принята реестром ConScript Unicode Registry и предлагалась (но в итоге была
отклонена) для включения в Unicode Plane 1.  Таким образом, она остаётся
частным присвоением Linux/CSUR в пределах Linux Zone.

Эта кодировка была одобрена Институтом клингонского языка (Klingon Language
Institute).  За дополнительной информацией обращайтесь к ним по адресу:

	http://www.kli.org/

Поскольку символы в начале Linux CZ относились скорее к типу
dingbats/символов/форм, а это язык, я разместил его в конце, на границе из
16 ячеек, в соответствии со стандартной практикой Unicode.

.. note::

  Этот диапазон теперь официально управляется реестром ConScript Unicode
  Registry.  Нормативная ссылка находится здесь:

	https://www.evertype.com/standards/csur/klingon.html

Клингонский имеет алфавит из 26 символов, позиционную числовую систему записи
с 10 цифрами и пишется слева направо, сверху вниз.

Для клингонского алфавита было предложено несколько форм глифов.
Однако, поскольку набор символов, по-видимому, остаётся неизменным во всех
вариантах и различаются лишь фактические начертания, в соответствии со
стандартной практикой Unicode эти различия считаются вариантами шрифта.

Другие вымышленные и искусственные письменности
-----------------------------------------------

После присвоения клингонского блока Unicode для Linux Джон Коуэн
<jcowan@reutershealth.com> и Майкл Эверсон <everson@evertype.com> создали
реестр вымышленных и искусственных письменностей.
Реестр ConScript Unicode Registry доступен по адресу:

	  https://www.evertype.com/standards/csur/

Используемые диапазоны попадают в нижнюю часть End User Zone и поэтому
не могут быть присвоены нормативно, однако рекомендуется, чтобы те, кто
желает кодировать вымышленные письменности, использовали эти коды в интересах
совместимости.  Для клингонского CSUR принял кодировку Linux.
Участники CSUR продвигают добавление Tengwar и Cirth в Unicode
Plane 1; добавление клингонского в Unicode Plane 1 было отклонено,
и поэтому приведённая выше кодировка остаётся официальной.
