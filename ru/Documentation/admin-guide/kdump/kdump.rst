=============================================================================
Документация по Kdump — решению для создания дампов при сбоях на основе kexec
=============================================================================

Этот документ содержит обзор, информацию о настройке, установке и анализе.

Обзор
=====

Kdump использует kexec для быстрой загрузки в ядро захвата дампа (dump-capture
kernel) всякий раз, когда требуется снять дамп памяти системного ядра (например,
при панике системы). Образ памяти системного ядра сохраняется при перезагрузке и
доступен ядру захвата дампа.

Для копирования образа памяти в файл дампа на локальном диске или по сети на
удалённую систему можно использовать обычные команды, такие как cp, scp или
makedumpfile.

В настоящее время Kdump и kexec поддерживаются на архитектурах x86, x86_64,
ppc64, s390x, arm и arm64.

При загрузке системного ядра оно резервирует небольшой участок памяти для ядра
захвата дампа. Это гарантирует, что выполняющийся прямой доступ к памяти (Direct
Memory Access, DMA) со стороны системного ядра не повредит ядро захвата дампа.
Команда kexec -p загружает ядро захвата дампа в эту зарезервированную память.

На машинах x86 первые 640 КБ физической памяти необходимы для загрузки,
независимо от того, куда загружается ядро. Для упрощения обработки резервируется
весь нижний 1 МБ, чтобы избежать записи данных в эту область со стороны более
позднего ядра или драйвера устройства. Таким образом, нижний 1 МБ может быть
повторно использован ядром kdump как системная оперативная память без
дополнительной обработки.

На машинах PPC64 первые 32 КБ физической памяти необходимы для загрузки
независимо от того, куда загружается ядро, и для поддержки размера страницы 64K
kexec создаёт резервную копию первых 64 КБ памяти.

Для s390x при срабатывании kdump область crashkernel меняется местами с областью
[0, размер области crashkernel], после чего ядро kdump выполняется в [0, размер
области crashkernel]. Поэтому для s390x перемещаемое ядро не требуется.

Вся необходимая информация об образе ядра системы (core image) кодируется в
формате ELF и сохраняется в зарезервированной области памяти перед сбоем.
Физический адрес начала заголовка ELF передаётся ядру захвата дампа через
параметр загрузки elfcorehdr=. Дополнительно можно также передать размер
заголовка ELF, используя синтаксис elfcorehdr=[size[KMG]@]offset[KMG].

С помощью ядра захвата дампа можно получить доступ к образу памяти через
/proc/vmcore. Это экспортирует дамп в виде файла формата ELF, который можно
записать с помощью команд копирования файлов, таких как cp или scp. Также можно
использовать утилиту makedumpfile для анализа и записи отфильтрованного
содержимого с помощью опций, например с '-d 31' она запишет только данные ядра.
Кроме того, для отладки файла дампа можно использовать инструменты анализа,
такие как GNU Debugger (GDB) и инструмент Crash. Этот метод гарантирует, что
страницы дампа упорядочены правильно.

Настройка и установка
======================

Установка kexec-tools
---------------------

1) Войдите в систему как пользователь root.

2) Загрузите пакет пользовательского пространства kexec-tools со следующего URL:

http://kernel.org/pub/linux/utils/kernel/kexec/kexec-tools.tar.gz

Это символическая ссылка на последнюю версию.

Последнее git-дерево kexec-tools доступно по адресу:

- git://git.kernel.org/pub/scm/utils/kernel/kexec/kexec-tools.git
- http://www.kernel.org/pub/scm/utils/kernel/kexec/kexec-tools.git

Также доступен интерфейс gitweb по адресу
http://www.kernel.org/git/?p=utils/kernel/kexec/kexec-tools.git

Дополнительную информацию о kexec-tools можно найти по адресу
http://horms.net/projects/kexec/

3) Распакуйте тарбол командой tar, как показано ниже::

	tar xvpzf kexec-tools.tar.gz

4) Перейдите в каталог kexec-tools, как показано ниже::

	cd kexec-tools-VERSION

5) Сконфигурируйте пакет, как показано ниже::

	./configure

6) Скомпилируйте пакет, как показано ниже::

	make

7) Установите пакет, как показано ниже::

	make install


Сборка системного ядра и ядра захвата дампа
-------------------------------------------
Существует два возможных метода использования Kdump.

1) Собрать отдельное специальное ядро захвата дампа для захвата
   core-дампа ядра.

2) Либо использовать в качестве ядра захвата дампа сам бинарный файл системного
   ядра, и тогда нет необходимости собирать отдельное ядро захвата дампа. Это
   возможно только на архитектурах, поддерживающих перемещаемое ядро. На
   сегодняшний день перемещаемое ядро поддерживают архитектуры i386, x86_64,
   ppc64, arm и arm64.

Сборка перемещаемого ядра выгодна с той точки зрения, что не нужно собирать
второе ядро для захвата дампа. Но в то же время может потребоваться собрать
специальное ядро захвата дампа, подходящее под собственные нужды.

Ниже приведены настройки конфигурации, необходимые для системного ядра и ядра
захвата дампа для включения поддержки kdump.

Опции конфигурации системного ядра
----------------------------------

1) Включите "kexec system call" или "kexec file based system call" в разделе
   "Processor type and features."::

	CONFIG_KEXEC=y or CONFIG_KEXEC_FILE=y

   И обе из них выберут KEXEC_CORE::

	CONFIG_KEXEC_CORE=y

2) Включите "sysfs file system support" в "Filesystem" -> "Pseudo
   filesystems." Обычно она включена по умолчанию::

	CONFIG_SYSFS=y

   Обратите внимание, что "sysfs file system support" может не появиться в меню
   "Pseudo filesystems", если "Configure standard kernel features (expert users)"
   не включено в "General Setup." В этом случае проверьте сам файл .config,
   чтобы убедиться, что sysfs включена, как показано ниже::

	grep 'CONFIG_SYSFS' .config

3) Включите "Compile the kernel with debug info" в "Kernel hacking."::

	CONFIG_DEBUG_INFO=Y

   Это приводит к тому, что ядро собирается с отладочными символами. Инструменты
   анализа дампа требуют vmlinux с отладочными символами для чтения и анализа
   файла дампа.

Опции конфигурации ядра захвата дампа (независимые от архитектуры)
------------------------------------------------------------------

1) Включите поддержку "kernel crash dumps" в разделе "Processor type and
   features"::

	CONFIG_CRASH_DUMP=y

   И это выберет VMCORE_INFO и CRASH_RESERVE::
	CONFIG_VMCORE_INFO=y
	CONFIG_CRASH_RESERVE=y

2) Включите "/proc/vmcore support" в "Filesystems" -> "Pseudo filesystems"::

	CONFIG_PROC_VMCORE=y

   (CONFIG_PROC_VMCORE устанавливается по умолчанию при выборе CONFIG_CRASH_DUMP.)

Опции конфигурации ядра захвата дампа (зависимые от архитектуры, i386 и x86_64)
-------------------------------------------------------------------------------

1) На i386 включите поддержку high memory в разделе "Processor type and
   features"::

	CONFIG_HIGHMEM4G

2) При CONFIG_SMP=y обычно требуется указать nr_cpus=1 в командной строке ядра
   при загрузке ядра захвата дампа, поскольку на большинстве систем одного CPU
   достаточно для того, чтобы ядро kdump сняло дамп vmcore.

   Однако можно также указать nr_cpus=X, чтобы включить несколько процессоров
   в ядре kdump.

   При CONFIG_SMP=n вышесказанное неактуально.

3) Рекомендуется по умолчанию собирать перемещаемое ядро. Если это ещё не так,
   включите поддержку "Build a relocatable kernel" в разделе "Processor type and
   features"::

	CONFIG_RELOCATABLE=y

4) Используйте подходящее значение для "Physical address where the kernel is
   loaded" (в разделе "Processor type and features"). Это появляется только тогда,
   когда включено "kernel crash dumps". Подходящее значение зависит от того,
   является ли ядро перемещаемым или нет.

   Если вы используете перемещаемое ядро, задайте CONFIG_PHYSICAL_START=0x100000.
   Это скомпилирует ядро для физического адреса 1 МБ, но поскольку ядро
   перемещаемое, оно может выполняться с любого физического адреса, и поэтому
   загрузчик kexec загрузит его в область памяти, зарезервированную для ядра
   захвата дампа.

   В противном случае это должно быть начало области памяти, зарезервированной
   для второго ядра с помощью параметра загрузки "crashkernel=Y@X". Здесь X —
   начало области памяти, зарезервированной для ядра захвата дампа.
   Обычно X равен 16 МБ (0x1000000). Так что можно задать
   CONFIG_PHYSICAL_START=0x1000000

5) Соберите и установите ядро и его модули. НЕ добавляйте это ядро
   в конфигурационные файлы загрузчика.

Опции конфигурации ядра захвата дампа (зависимые от архитектуры, ppc64)
-----------------------------------------------------------------------

1) Включите поддержку "Build a kdump crash kernel" в разделе опций "Kernel"::

	CONFIG_CRASH_DUMP=y

2)   Включите поддержку "Build a relocatable kernel"::

	CONFIG_RELOCATABLE=y

   Соберите и установите ядро и его модули.

Опции конфигурации ядра захвата дампа (зависимые от архитектуры, arm)
---------------------------------------------------------------------

-   Чтобы использовать перемещаемое ядро,
    включите поддержку "AUTO_ZRELADDR" в разделе опций "Boot"::

	AUTO_ZRELADDR=y

Опции конфигурации ядра захвата дампа (зависимые от архитектуры, arm64)
-----------------------------------------------------------------------

- Обратите внимание, что kvm ядра захвата дампа не будет включён
  на системах без VHE, даже если он сконфигурирован. Это связано с тем, что CPU
  не будет сброшен в EL2 при панике.

Синтаксис crashkernel
=====================
1) crashkernel=size@offset

   Здесь 'size' указывает, сколько памяти зарезервировать для ядра захвата дампа,
   а 'offset' указывает начало этой зарезервированной памяти. Например,
   "crashkernel=64M@16M" сообщает системному ядру зарезервировать 64 МБ памяти,
   начиная с физического адреса 0x01000000 (16 МБ), для ядра захвата дампа.

   Область crashkernel может быть автоматически размещена системным
   ядром во время выполнения. Это делается путём указания базового адреса как 0
   или путём его полного опускания::

         crashkernel=256M@0

   или::

         crashkernel=256M

   Если начальный адрес указан, обратите внимание, что начальный адрес ядра будет
   выровнен по некоторому значению (зависящему от архитектуры), поэтому, если
   начальный адрес не выровнен, всё пространство ниже точки выравнивания будет
   потрачено впустую.

2) range1:size1[,range2:size2,...][@offset]

   Хотя синтаксиса "crashkernel=size[@offset]" достаточно для большинства
   конфигураций, иногда удобно сделать так, чтобы зарезервированная память
   зависела от значения системной оперативной памяти — это в основном для
   дистрибьюторов, которые заранее настраивают командную строку ядра, чтобы
   избежать незагружаемой системы после того, как часть памяти была удалена из
   машины.

   Синтаксис следующий::

       crashkernel=<range1>:<size1>[,<range2>:<size2>,...][@offset]
       range=start-[end]

   Например::

       crashkernel=512M-2G:64M,2G-:128M

   Это означало бы:

       1) если оперативная память меньше 512M, то ничего не резервировать
          (это случай "rescue")
       2) если размер оперативной памяти между 512M и 2G (не включая), то
          зарезервировать 64M
       3) если размер оперативной памяти больше 2G, то зарезервировать 128M

3) crashkernel=size,high и crashkernel=size,low

   Если предпочтительна память выше 4G, для этого можно использовать
   crashkernel=size,high. С ней физическая память может выделяться сверху,
   так что она может находиться выше 4G, если в системе установлено более 4G
   оперативной памяти. В противном случае область памяти будет выделена ниже 4G,
   если она доступна.

   Когда передаётся crashkernel=X,high, ядро может выделить область физической
   памяти выше 4G, при этом необходима low-память ниже 4G. Есть
   три способа получить low-память:

      1) Ядро автоматически выделит не менее 256M памяти ниже 4G,
         если crashkernel=Y,low не указан.
      2) Позволить пользователю вместо этого указать размер low-памяти.
      3) Указанное значение 0 отключит выделение low-памяти::

            crashkernel=0,low

4) crashkernel=size,cma

	Резервирует дополнительную память crash kernel из CMA. Это резервирование
	пригодно для использования памятью пространства пользователя первой системы и
	перемещаемыми (movable) выделениями ядра (memory balloon, zswap). Страницы,
	выделенные из этого диапазона памяти, не будут включены в vmcore, поэтому это
	не следует использовать, если предполагается дамп памяти пространства
	пользователя, и нужно ожидать, что некоторые перемещаемые страницы ядра могут
	отсутствовать в дампе.

	Стандартное резервирование crashkernel, как описано выше, по-прежнему
	необходимо для размещения crash kernel и initrd.

	Эта опция увеличивает риск сбоя kdump: передачи DMA, сконфигурированные первым
	ядром, могут в итоге повредить память второго ядра.

	Этот метод резервирования предназначен для систем, которые не могут позволить
	себе пожертвовать достаточным объёмом памяти для стандартного резервирования
	crashkernel и в которых менее надёжный и, возможно, неполный kdump
	предпочтительнее полного отсутствия kdump.

Загрузка в системное ядро
-------------------------
1) Обновите при необходимости конфигурационные файлы загрузчика (такого как
   grub, yaboot или lilo).

2) Загрузите системное ядро с параметром загрузки "crashkernel=Y@X".

   На x86 и x86_64 используйте "crashkernel=Y[@X]". В большинстве случаев
   начальный адрес 'X' не нужен, ядро будет искать подходящую
   область. Если только не ожидается явный начальный адрес.

   На ppc64 используйте "crashkernel=128M@32M".

   На s390x обычно используется "crashkernel=xxM". Значение xx зависит
   от потребления памяти системой kdump. В общем случае это не
   зависит от размера памяти продакшн-системы.

   На arm использование "crashkernel=Y@X" больше не нужно; ядро
   автоматически найдёт образ crash kernel в пределах
   первых 512MB оперативной памяти, если X не задан.

   На arm64 используйте "crashkernel=Y[@X]".  Обратите внимание, что начальный
   адрес ядра, X, если он указан явно, должен быть выровнен по 2MiB (0x200000).

Загрузка ядра захвата дампа
===========================

После загрузки в системное ядро необходимо загрузить ядро захвата дампа.

В зависимости от архитектуры и типа образа (перемещаемый или нет) можно
выбрать загрузку несжатого vmlinux или сжатого bzImage/vmlinuz
ядра захвата дампа. Ниже приведено резюме.

Для i386 и x86_64:

	- Используйте bzImage/vmlinuz, если ядро перемещаемое.
	- Используйте vmlinux, если ядро не перемещаемое.

Для ppc64:

	- Используйте vmlinux

Для s390x:

	- Используйте image или bzImage

Для arm:

	- Используйте zImage

Для arm64:

	- Используйте vmlinux или Image

Если вы используете несжатый образ vmlinux, то используйте следующую команду
для загрузки ядра захвата дампа::

   kexec -p <dump-capture-kernel-vmlinux-image> \
   --initrd=<initrd-for-dump-capture-kernel> --args-linux \
   --append="root=<root-dev> <arch-specific-options>"

Если вы используете сжатый bzImage/vmlinuz, то используйте следующую команду
для загрузки ядра захвата дампа::

   kexec -p <dump-capture-kernel-bzImage> \
   --initrd=<initrd-for-dump-capture-kernel> \
   --append="root=<root-dev> <arch-specific-options>"

Если вы используете сжатый zImage, то используйте следующую команду
для загрузки ядра захвата дампа::

   kexec --type zImage -p <dump-capture-kernel-bzImage> \
   --initrd=<initrd-for-dump-capture-kernel> \
   --dtb=<dtb-for-dump-capture-kernel> \
   --append="root=<root-dev> <arch-specific-options>"

Если вы используете несжатый Image, то используйте следующую команду
для загрузки ядра захвата дампа::

   kexec -p <dump-capture-kernel-Image> \
   --initrd=<initrd-for-dump-capture-kernel> \
   --append="root=<root-dev> <arch-specific-options>"

Ниже приведены специфичные для архитектуры опции командной строки, которые
следует использовать при загрузке ядра захвата дампа.

Для i386 и x86_64:

	"1 irqpoll nr_cpus=1 reset_devices"

Для ppc64:

	"1 maxcpus=1 noirqdistrib reset_devices"

Для s390x:

	"1 nr_cpus=1 cgroup_disable=memory"

Для arm:

	"1 maxcpus=1 reset_devices"

Для arm64:

	"1 nr_cpus=1 reset_devices"

Замечания по загрузке ядра захвата дампа:

* По умолчанию заголовки ELF хранятся в формате ELF64 для поддержки
  систем с объёмом памяти более 4GB. На i386 kexec автоматически проверяет, не
  превышает ли размер физической оперативной памяти предел в 4 GB, и если нет,
  использует ELF32.
  Так что на системах без PAE всегда используется ELF32.

  Опцию --elf32-core-headers можно использовать для принудительной генерации
  заголовков ELF32. Это необходимо, поскольку GDB в настоящее время не может
  открывать файлы vmcore с заголовками ELF64 на 32-битных системах.

* Параметр загрузки "irqpoll" снижает количество сбоев инициализации драйверов
  из-за разделяемых прерываний в ядре захвата дампа.

* Вы должны указать <root-dev> в формате, соответствующем имени корневого
  устройства в выводе команды mount.

* Параметр загрузки "1" загружает ядро захвата дампа в однопользовательском
  режиме без сети. Если требуется сеть, используйте "3".

* Обычно нам не нужно поднимать SMP-ядро только для того, чтобы захватить
  дамп. Поэтому обычно полезно либо собрать однопроцессорное (UP) ядро захвата
  дампа, либо указать опцию maxcpus=1 при загрузке ядра захвата дампа.
  Обратите внимание, что хотя maxcpus всегда работает, лучше заменить её на
  nr_cpus для экономии памяти, если это поддерживается текущей архитектурой,
  например x86.

* Следует включить поддержку нескольких CPU в ядре захвата дампа, если вы
  намерены использовать с ним многопоточные программы, такие как функция
  параллельного дампа makedumpfile. В противном случае многопоточная программа
  может испытывать значительную деградацию производительности. Чтобы включить
  поддержку нескольких CPU, следует поднять SMP-ядро захвата дампа и указать
  опции maxcpus/nr_cpus при его загрузке.

* Для s390x существует два режима kdump: если заголовок ELF указан с помощью
  параметра ядра elfcorehdr=, он используется ядром kdump так же,
  как это делается на всех остальных архитектурах. Если параметр ядра elfcorehdr=
  не указан, ядро kdump для s390x динамически создаёт заголовок.
  Второй режим имеет преимущество в том, что при горячем подключении CPU и памяти
  kdump не нужно перезагружать с помощью kexec_load().

* Для систем s390x с большим количеством подключённых устройств для ядра kdump
  следует использовать параметр ядра "cio_ignore", чтобы предотвратить выделение
  памяти ядра для устройств, не имеющих отношения к kdump. То же
  относится к системам, использующим устройства SCSI/FCP. В этом случае
  параметр модуля zfcp "allow_lun_scan" должен быть установлен в ноль перед
  переводом FCP-устройств в онлайн.

Паника ядра
===========

После успешной загрузки ядра захвата дампа, как описано ранее,
система перезагрузится в ядро захвата дампа, если будет
вызван сбой системы.  Точки срабатывания находятся в panic(),
die(), die_nmi() и в обработчике sysrq (ALT-SysRq-c).

Следующие условия выполнят точку срабатывания сбоя:

Если обнаружен hard lockup и сконфигурирован "NMI watchdog", система
загрузится в ядро захвата дампа ( die_nmi() ).

Если вызывается die(), и это оказывается поток с pid 0 или 1, или die()
вызывается внутри контекста прерывания, или die() вызывается и установлен
panic_on_oops, система загрузится в ядро захвата дампа.

На системах powerpc, когда генерируется soft-reset, die() вызывается всеми cpu,
и система загрузится в ядро захвата дампа.

В целях тестирования можно вызвать сбой с помощью "ALT-SysRq-c",
"echo c > /proc/sysrq-trigger" или написать модуль для принудительной паники.

Запись файла дампа
==================

После загрузки ядра захвата дампа запишите файл дампа с помощью
следующей команды::

   cp /proc/vmcore <dump-file>

или используйте scp для записи файла дампа между хостами в сети, например::

   scp /proc/vmcore remote_username@remote_ip:<dump-file>

Также можно использовать утилиту makedumpfile для записи файла дампа
с указанными опциями для отфильтровывания ненужного содержимого, например::

   makedumpfile -l --message-level 1 -d 31 /proc/vmcore <dump-file>

Анализ
======

Перед анализом образа дампа следует перезагрузиться в стабильное ядро.

Можно выполнить ограниченный анализ с помощью GDB на файле дампа, скопированном
из /proc/vmcore. Используйте отладочный vmlinux, собранный с -g, и выполните
следующую команду::

   gdb vmlinux <dump-file>

Трассировка стека для задачи на процессоре 0, отображение регистров и отображение
памяти работают нормально.

Замечание: GDB не может анализировать core-файлы, сгенерированные в формате ELF64
для x86. На системах с максимум 4GB памяти можно сгенерировать заголовки формата
ELF32 с помощью опции ядра --elf32-core-headers на ядре дампа.

Также можно использовать утилиту Crash для анализа файлов дампа в формате Kdump.
Crash доступна по следующему URL:

   https://github.com/crash-utility/crash

Документацию по Crash можно найти по адресу:
   https://crash-utility.github.io/

Срабатывание Kdump при WARN()
=============================

Параметр ядра panic_on_warn вызывает panic() во всех путях WARN().  Это
вызовет возникновение kdump при вызове panic().  В случаях, когда пользователь
хочет указать это во время выполнения, /proc/sys/kernel/panic_on_warn можно
установить в 1, чтобы добиться того же поведения.

Срабатывание Kdump при add_taint()
==================================

Параметр ядра panic_on_taint обеспечивает условный вызов panic()
изнутри add_taint() всякий раз, когда значение, установленное в этой битовой
маске, совпадает с битовым флагом, устанавливаемым add_taint().
Это вызовет возникновение kdump при вызове add_taint()->panic().

Запись файла дампа на зашифрованный дисковый том
================================================

CONFIG_CRASH_DM_CRYPT можно включить для поддержки сохранения файла дампа на
зашифрованный дисковый том (пока поддерживается только x86_64). Пространство
пользователя может взаимодействовать с /sys/kernel/config/crash_dm_crypt_keys
для настройки,

1. Сообщите первому ядру, какие logon-ключи нужны для разблокировки дисковых томов,
    # Add key #1
    mkdir /sys/kernel/config/crash_dm_crypt_keys/7d26b7b4-e342-4d2d-b660-7426b0996720
    # Add key #1's description
    echo cryptsetup:7d26b7b4-e342-4d2d-b660-7426b0996720 > /sys/kernel/config/crash_dm_crypt_keys/description

    # how many keys do we have now?
    cat /sys/kernel/config/crash_dm_crypt_keys/count
    1

    # Add key #2 in the same way

    # how many keys do we have now?
    cat /sys/kernel/config/crash_dm_crypt_keys/count
    2

    # To support CPU/memory hot-plugging, reuse keys already saved to reserved
    # memory
    echo true > /sys/kernel/config/crash_dm_crypt_key/reuse

2. Загрузите ядро захвата дампа

3. После загрузки ядра захвата дампа восстановите ключи в keyring пользователя
   echo yes > /sys/kernel/crash_dm_crypt_keys/restore
