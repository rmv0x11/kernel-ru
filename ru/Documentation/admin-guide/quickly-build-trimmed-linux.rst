==========================================================
Как быстро собрать урезанное ядро Linux
==========================================================

Это руководство объясняет, как быстро собирать ядра Linux, идеально подходящие
для целей тестирования, но также вполне пригодные и для повседневного использования.

Суть процесса (он же «TL;DR»)
=============================

*[Если вы только начинаете компилировать Linux, пропустите этот раздел TLDR и
перейдите к следующему разделу ниже: он содержит пошаговое руководство, которое
подробнее, но при этом по-прежнему краткое и простое для восприятия; это
руководство и сопровождающий его справочный раздел также упоминают альтернативы,
подводные камни и дополнительные аспекты, которые все могут оказаться для вас
актуальными.]*

Если ваша система использует такие технологии, как Secure Boot, подготовьте её к
тому, чтобы разрешить запуск самостоятельно скомпилированных ядер Linux;
установите компиляторы и всё остальное, что нужно для сборки Linux; убедитесь,
что в вашем домашнем каталоге есть 12 гигабайт свободного места. Теперь выполните
следующие команды, чтобы загрузить свежие исходники Linux mainline, которые затем
вы используете для настройки, сборки и установки собственного ядра::

    git clone --depth 1 -b master \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git ~/linux/
    cd ~/linux/
    # Hint: if you want to apply patches, do it at this point. See below for details.
    # Hint: it's recommended to tag your build at this point. See below for details.
    yes "" | make localmodconfig
    # Hint: at this point you might want to adjust the build configuration; you'll
    #   have to, if you are running Debian. See below for details.
    make -j $(nproc --all)
    # Note: on many commodity distributions the next command suffices, but on Arch
    #   Linux, its derivatives, and some others it does not. See below for details.
    command -v installkernel && sudo make modules_install install
    reboot

Если позже вы захотите собрать более новый снимок mainline, используйте эти
команды::

    cd ~/linux/
    git fetch --depth 1 origin
    # Note: the next command will discard any changes you did to the code:
    git checkout --force --detach origin/master
    # Reminder: if you want to (re)apply patches, do it at this point.
    # Reminder: you might want to add or modify a build tag at this point.
    make olddefconfig
    make -j $(nproc --all)
    # Reminder: the next command on some distributions does not suffice.
    command -v installkernel && sudo make modules_install install
    reboot

Пошаговое руководство
=====================

В принципе скомпилировать собственное ядро Linux несложно. Сделать это можно
различными способами. Какой из них действительно работает и является наилучшим,
зависит от обстоятельств.

Это руководство описывает способ, идеально подходящий для тех, кто хочет быстро
установить Linux из исходников, не утруждая себя сложными деталями; цель —
охватить всё, что обычно нужно в распространённых дистрибутивах Linux,
работающих на типовом железе ПК или серверов.

Описанный подход прекрасно подходит для целей тестирования, например чтобы
опробовать предлагаемое исправление или проверить, не было ли уже исправлено
какое-то проблема в свежей кодовой базе. Тем не менее ядра, собранные таким
способом, также вполне годятся для повседневного использования и при этом легко
поддерживаются в актуальном состоянии.

Следующие шаги описывают важные аспекты процесса; обстоятельный справочный раздел
далее объясняет каждый из них подробнее. Иногда он также описывает альтернативные
подходы, подводные камни, а также ошибки, которые могут возникнуть в конкретный
момент, — и как затем снова всё наладить.

.. _backup_sbs:

 * Создайте свежую резервную копию и держите под рукой инструменты восстановления
   и ремонта системы — просто на тот маловероятный случай, если что-то пойдёт не
   так.

   [:ref:`подробнее<backup>`]

.. _secureboot_sbs:

 * На платформах с «Secure Boot» или аналогичными технологиями подготовьте всё
   так, чтобы система позже позволила загрузить ваше самостоятельно
   скомпилированное ядро. Самый быстрый и простой способ добиться этого на типовых
   системах x86 — отключить такие технологии в утилите настройки BIOS; в качестве
   альтернативы снимите их ограничения с помощью процесса, инициируемого командой
   ``mokutil --disable-validation``.

   [:ref:`подробнее<secureboot>`]

.. _buildrequires_sbs:

 * Установите всё ПО, необходимое для сборки ядра Linux. Часто вам понадобятся:
   'bc', 'binutils' ('ld' и др.), 'bison', 'flex', 'gcc', 'git', 'openssl',
   'pahole', 'perl', а также заголовочные файлы разработки для 'libelf' и
   'openssl'. В справочном разделе показано, как быстро установить их в различных
   популярных дистрибутивах Linux.

   [:ref:`подробнее<buildrequires>`]

.. _diskspace_sbs:

 * Убедитесь, что у вас достаточно свободного места для сборки и установки Linux.
   Для последней безопасной оценкой будут 150 мегабайт в /lib/ и 100 в /boot/.
   Для хранения исходников и артефактов сборки обычно достаточно 12 гигабайт в
   вашем домашнем каталоге. Если у вас доступно меньше, обязательно изучите
   справочный раздел для шага, который объясняет настройку конфигурации сборки
   ядра: там упоминается приём, уменьшающий объём требуемого места в /home/
   примерно до 4 гигабайт.

   [:ref:`подробнее<diskspace>`]

.. _sources_sbs:

 * Получите исходники той версии Linux, которую вы собираетесь собирать; затем
   перейдите в каталог, в котором они находятся, поскольку все дальнейшие команды
   в этом руководстве предполагается выполнять оттуда.

   *[Примечание: следующие абзацы описывают, как получить исходники путём
   частичного клонирования git-репозитория Linux stable. Это называется
   поверхностным клоном (shallow clone). В справочном разделе объясняются две
   альтернативы:* :ref:`упакованные архивы<sources_archive>` *и* :ref:`полный
   git-клон<sources_full>` *; предпочтительнее последний, если загрузка большого
   объёма данных вас не смущает, так как это позволит избежать некоторых*
   :ref:`особенностей поверхностных клонов, которые объясняет справочный
   раздел<sources_shallow>` *.]*

   Сначала выполните следующую команду, чтобы получить свежую кодовую базу
   mainline::

     git clone --no-checkout --depth 1 -b master \
       https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git ~/linux/
     cd ~/linux/

   Если вы хотите получить доступ к недавним релизам и пре-релизам mainline,
   углубите историю своего клона до самой старой версии mainline, которая вас
   интересует::

     git fetch --shallow-exclude=v6.0 origin

   Если вы хотите получить доступ к stable/longterm-релизу (скажем, v6.1.5),
   просто добавьте ветку, в которой находится эта серия; после этого получите
   историю как минимум вплоть до версии mainline, с которой началась серия
   (v6.1)::

     git remote set-branches --add origin linux-6.1.y
     git fetch --shallow-exclude=v6.0 origin

   Теперь сделайте checkout кода, который вас интересует. Если вы только что
   выполнили начальное клонирование, вы сможете сделать checkout свежей кодовой
   базы mainline, что идеально подходит для проверки того, исправили ли
   разработчики уже какую-то проблему::

      git checkout --detach origin/master

   Если вы углубили свой клон, то вместо ``origin/master`` вы можете указать
   версию, до которой вы углубились (``v6.0`` выше); более поздние релизы вроде
   ``v6.1`` и пре-релизы вроде ``v6.2-rc1`` тоже подойдут. Stable- или
   longterm-версии вроде ``v6.1.5`` работают точно так же, если вы добавили
   соответствующую stable/longterm-ветку, как описано.

   [:ref:`подробнее<sources>`]

.. _patching_sbs:

 * Если вы хотите применить патч к ядру, сделайте это сейчас. Часто помогает
   команда вроде этой::

     patch -p1 < ../proposed-fix.patch

   Действительно ли нужен ``-p1``, зависит от того, как был создан патч; поэтому,
   если он не применяется, попробуйте без него.

   Если вы клонировали исходники с помощью git и что-то пошло не так, выполните
   ``git reset --hard``, чтобы отменить любые изменения в исходниках.

   [:ref:`подробнее<patching>`]

.. _tagging_sbs:

 * Если вы пропатчили своё ядро или у вас уже установлено ядро той же версии,
   лучше добавьте уникальный тег к тому, которое вы собираетесь собрать::

     echo "-proposed_fix" > localversion

   После этого выполнение ``uname -r`` под вашим ядром будет выводить нечто вроде
   '6.1-rc4-proposed_fix'.

   [:ref:`подробнее<tagging>`]

 .. _configuration_sbs:

 * Создайте конфигурацию сборки для вашего ядра на основе существующей
   конфигурации.

   Если вы уже сами подготовили такой файл '.config', скопируйте его в
   ~/linux/ и выполните ``make olddefconfig``.

   Используйте ту же команду, если ваш дистрибутив или кто-то другой уже подогнал
   ваше работающее ядро под ваши нужды или нужды вашего железа: make-цель
   'olddefconfig' тогда попытается использовать .config этого ядра в качестве
   основы.

   Использование этой make-цели подойдёт и всем остальным, — но зачастую вы
   можете сэкономить много времени, используя вместо этого команду::

     yes "" | make localmodconfig

   Она попытается взять за основу ядро вашего дистрибутива, но затем отключит
   модули для любых функций, очевидно избыточных для вашей конфигурации. Это
   колоссально сократит время компиляции, особенно если вы используете
   универсальное ядро из типового дистрибутива Linux.

   Есть подвох: 'localmodconfig', скорее всего, отключит функции ядра, которые вы
   не использовали с момента загрузки вашего Linux, — например, драйверы для
   отключённых в данный момент периферийных устройств или ПО для виртуализации,
   которое вы пока не использовали. Вы можете снизить или почти устранить этот
   риск с помощью приёмов, описанных в справочном разделе; но при сборке ядра
   только для целей быстрого тестирования отсутствие таких функций зачастую
   несущественно. Тем не менее этот аспект следует держать в уме при
   использовании ядра, собранного с этой make-целью, поскольку это может оказаться
   причиной того, почему что-то, чем вы пользуетесь лишь изредка, перестало
   работать.

   [:ref:`подробнее<configuration>`]

.. _configmods_sbs:

 * Проверьте, не хотите ли вы или не должны ли вы скорректировать некоторые опции
   конфигурации ядра:

  * Оцените, как вы хотите поступить с отладочными символами. Включите их, если
    вам впоследствии может понадобиться декодировать трассировку стека,
    обнаруженную, например, в 'panic', 'Oops', 'warning' или 'BUG'; с другой
    стороны, отключите их, если вам не хватает места для хранения или вы
    предпочитаете меньший бинарный файл ядра. Подробности о том, как сделать то
    или другое, смотрите в справочном разделе. Если не подходит ни то, ни другое,
    то, скорее всего, можно просто не заморачиваться этим.
    [:ref:`подробнее<configmods_debugsymbols>`]

  * Вы используете Debian? Тогда во избежание известных проблем выполните
    дополнительные корректировки, описанные в справочном разделе.
    [:ref:`подробнее<configmods_distros>`].

  * Если вы хотите повлиять на другие аспекты конфигурации, сделайте это сейчас,
    используя make-цели вроде 'menuconfig' или 'xconfig'.
    [:ref:`подробнее<configmods_individual>`].

.. _build_sbs:

 * Соберите образ и модули вашего ядра::

     make -j $(nproc --all)

   Если вы хотите, чтобы ваше ядро было упаковано в файл deb, rpm или tar,
   смотрите альтернативы в справочном разделе.

   [:ref:`подробнее<build>`]

.. _install_sbs:

 * Теперь установите ваше ядро::

     command -v installkernel && sudo make modules_install install

   Часто всё, что вам остаётся сделать после этого, — это ``reboot``, поскольку
   многие типовые дистрибутивы Linux затем создадут initramfs (также известный
   как initrd) и запись для вашего ядра в конфигурации загрузчика; но в некоторых
   дистрибутивах вам придётся позаботиться об этих двух шагах вручную по причинам,
   которые объясняет справочный раздел.

   В нескольких дистрибутивах вроде Arch Linux и его производных приведённая выше
   команда не делает вообще ничего; в этом случае вам придётся установить ваше
   ядро вручную, как описано в справочном разделе.

   Если вы используете неизменяемый (immutable) дистрибутив Linux, изучите его
   документацию и интернет, чтобы выяснить, как установить там собственное ядро.

   [:ref:`подробнее<install>`]

.. _another_sbs:

 * Чтобы позже собрать ещё одно ядро, вам понадобятся похожие шаги, но иногда
   слегка отличающиеся команды.

   Сначала вернитесь обратно в дерево исходников::

      cd ~/linux/

   Если вы хотите собрать версию из stable/longterm-серии, которую вы ещё не
   использовали (скажем, 6.2.y), укажите git'у отслеживать её::

      git remote set-branches --add origin linux-6.2.y

   Теперь получите последние изменения из апстрима; вам снова нужно указать самую
   раннюю версию, которая вас интересует, иначе git может получить всю историю
   коммитов::

     git fetch --shallow-exclude=v6.0 origin

   Теперь переключитесь на интересующую вас версию, — но имейте в виду, что
   используемая здесь команда отбросит любые сделанные вами модификации, так как
   они конфликтовали бы с исходниками, которые вы хотите получить через checkout::

     git checkout --force --detach origin/master

   На этом этапе вы можете снова пропатчить исходники или задать/изменить тег
   сборки, как объяснялось ранее. После этого скорректируйте конфигурацию сборки
   под новую кодовую базу с помощью olddefconfig, который теперь подгонит файл
   конфигурации, подготовленный вами ранее с помощью localmodconfig
   (~/linux/.config), для вашего следующего ядра::

     # reminder: if you want to apply patches, do it at this point
     # reminder: you might want to update your build tag at this point
     make olddefconfig

   Теперь соберите ваше ядро::

     make -j $(nproc --all)

   После этого установите ядро, как описано выше::

     command -v installkernel && sudo make modules_install install

   [:ref:`подробнее<another>`]

.. _uninstall_sbs:

 * Ваше ядро легко удалить впоследствии, так как его части хранятся лишь в двух
   местах и однозначно идентифицируются по имени релиза ядра. Только обязательно
   не удаляйте ядро, под которым вы сейчас работаете, так как это может сделать
   вашу систему незагружаемой.

   Начните с удаления каталога, в котором находятся модули вашего ядра и который
   назван по имени его релиза — '6.0.1-foobar' в следующем примере::

     sudo rm -rf /lib/modules/6.0.1-foobar

   Теперь попробуйте следующую команду, которая в некоторых дистрибутивах удалит
   все остальные установленные файлы ядра, а также уберёт запись ядра из
   конфигурации загрузчика::

     command -v kernel-install && sudo kernel-install -v remove 6.0.1-foobar

   Если эта команда ничего не выводит или завершается с ошибкой, смотрите
   справочный раздел; сделайте то же самое, если в /boot/ остаются какие-либо
   файлы с именами вида '*6.0.1-foobar*'.

   [:ref:`подробнее<uninstall>`]

Справочный раздел к пошаговому руководству
==========================================

Этот раздел содержит дополнительную информацию для каждого из шагов
вышеприведённого руководства.

.. _backup:

Подготовьтесь к нештатным ситуациям
-----------------------------------

   *Создайте свежую резервную копию и держите под рукой инструменты
   восстановления и ремонта системы*
   [:ref:`... <backup_sbs>`]

Помните, что вы имеете дело с компьютерами, которые иногда делают неожиданные
вещи, — особенно если вы возитесь с такими критически важными частями, как ядро
операционной системы. Именно это вы и собираетесь делать в данном процессе.
Поэтому лучше подготовиться к тому, что что-то пойдёт не так, даже если этого не
должно произойти.

[:ref:`назад к пошаговому руководству <backup_sbs>`]

.. _secureboot:

Работа с такими технологиями, как Secure Boot
---------------------------------------------

   *На платформах с «Secure Boot» или аналогичными технологиями подготовьте всё
   так, чтобы система позже позволила загрузить ваше самостоятельно
   скомпилированное ядро.*
   [:ref:`... <secureboot_sbs>`]

Многие современные системы разрешают запуск лишь определённых операционных систем;
поэтому по умолчанию они отказываются загружать самостоятельно скомпилированные
ядра.

В идеале вы решаете это, заставляя вашу платформу доверять вашим самостоятельно
собранным ядрам с помощью сертификата и подписи. Как это сделать, здесь не
описывается, так как это требует различных шагов, которые увели бы текст слишком
далеко от его цели; 'Documentation/admin-guide/module-signing.rst' и различные
веб-сайты уже объясняют это подробнее.

Временное отключение решений вроде Secure Boot — ещё один способ заставить ваш
собственный Linux загружаться. На типовых системах x86 это можно сделать в
утилите настройки BIOS; шаги для этого здесь не описаны, так как они сильно
различаются от машины к машине.

В распространённых дистрибутивах Linux для x86 есть третий, универсальный вариант:
отключить все ограничения Secure Boot для вашей среды Linux. Вы можете
инициировать этот процесс, выполнив ``mokutil --disable-validation``; он
предложит вам создать одноразовый пароль, который можно безопасно записать.
Теперь перезагрузитесь; сразу после того, как ваш BIOS выполнит все
самопроверки, загрузчик Shim покажет синее окно с сообщением 'Press any key to
perform MOK management'. Нажмите какую-нибудь клавишу до того, как обратный
отсчёт истечёт. Это откроет меню, выберите там 'Change Secure Boot state'.
'MokManager' из Shim теперь попросит вас ввести три случайно выбранных символа из
одноразового пароля, заданного ранее. Как только вы их предоставите,
подтвердите, что вы действительно хотите отключить проверку. После этого
разрешите MokManager перезагрузить машину.

[:ref:`назад к пошаговому руководству <secureboot_sbs>`]

.. _buildrequires:

Установка зависимостей для сборки
---------------------------------

   *Установите всё ПО, необходимое для сборки ядра Linux.*
   [:ref:`...<buildrequires_sbs>`]

Ядро довольно самодостаточно, но помимо инструментов вроде компилятора вам иногда
понадобится несколько библиотек для его сборки. Как установить всё необходимое,
зависит от вашего дистрибутива Linux и конфигурации ядра, которое вы собираетесь
собирать.

Вот несколько примеров того, что обычно требуется в некоторых распространённых
дистрибутивах:

 * Debian, Ubuntu и производные::

     sudo apt install bc binutils bison dwarves flex gcc git make openssl \
       pahole perl-base libssl-dev libelf-dev

 * Fedora и производные::

     sudo dnf install binutils /usr/include/{libelf.h,openssl/pkcs7.h} \
       /usr/bin/{bc,bison,flex,gcc,git,openssl,make,perl,pahole}

 * openSUSE и производные::

     sudo zypper install bc binutils bison dwarves flex gcc git make perl-base \
       openssl openssl-devel libelf-dev

Если вам интересно, почему в эти списки включены openssl и его заголовочные файлы
разработки: они нужны для поддержки Secure Boot, которую многие дистрибутивы
включают в конфигурации своего ядра для машин x86.

Иногда вам также понадобятся инструменты для форматов сжатия вроде bzip2, gzip,
lz4, lzma, lzo, xz или zstd.

Вам могут понадобиться дополнительные библиотеки и их заголовочные файлы
разработки, если вы выполняете задачи, не охваченные этим руководством. Например,
zlib понадобится при сборке инструментов ядра из каталога tools/; для
корректировки конфигурации сборки с помощью make-целей вроде 'menuconfig' или
'xconfig' потребуются заголовочные файлы разработки для ncurses или Qt5.

[:ref:`назад к пошаговому руководству <buildrequires_sbs>`]

Загрузка исходников
-------------------

  *Получите исходники той версии Linux, которую вы собираетесь собирать.*
  [:ref:`...<sources_sbs>`]

В пошаговом руководстве описано, как получить исходники Linux с помощью
поверхностного git-клона. Об этом методе :ref:`есть что рассказать
ещё<sources_shallow>`, и существуют два альтернативных способа, заслуживающих
описания: :ref:`упакованные архивы<sources_archive>` и :ref:`полный
git-клон<sources_full>`. Также требуют пояснения аспекты ':ref:`не разумнее ли
использовать настоящий пре-релиз, чем свежий код mainline <sources_snapshot>`' и
':ref:`как получить ещё более свежую кодовую базу mainline <sources_fresher>`'.

Обратите внимание: для простоты используемые в этом руководстве команды хранят
артефакты сборки в дереве исходников. Если вы предпочитаете разделять их, просто
добавьте нечто вроде ``O=~/linux-builddir/`` ко всем вызовам make; также
скорректируйте путь во всех командах, которые добавляют файлы или изменяют любые
сгенерированные (например, ваш '.config').

[:ref:`назад к пошаговому руководству <sources_sbs>`]

.. _sources_shallow:

Заслуживающие внимания особенности поверхностных клонов
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

В пошаговом руководстве используется поверхностный клон, так как это наилучшее
решение для большей части целевой аудитории этого документа. Есть несколько
аспектов этого подхода, которые стоит упомянуть:

 * Этот документ в большинстве мест использует ``git fetch`` с
   ``--shallow-exclude=``, чтобы указать самую раннюю версию, которая вас
   интересует (или, точнее: её git-тег). В качестве альтернативы вы можете
   использовать параметр ``--shallow-since=``, чтобы указать абсолютную (скажем,
   ``'2023-07-15'``) или относительную (``'12 months'``) дату для определения
   глубины истории, которую вы хотите загрузить. В качестве второй альтернативы
   вы также можете явно указать определённую глубину параметром вроде
   ``--depth=1``, если только вы не добавляете ветки для stable/longterm-ядер.

 * При выполнении ``git fetch`` помните, что всегда нужно указывать самую старую
   версию, интересующий вас момент времени или явную глубину, как показано в
   пошаговом руководстве. Иначе вы рискуете загрузить почти всю git-историю, что
   займёт немало времени и пропускной способности, а также создаст нагрузку на
   серверы.

   Обратите внимание: вам не обязательно всё время использовать одну и ту же
   версию или дату. Но когда вы меняете её со временем, git будет углублять или
   уплощать историю до указанной точки. Это позволяет вам получать версии,
   которые поначалу вы считали ненужными, — или же отбрасывать исходники более
   старых версий, например, если вы хотите освободить немного места на диске.
   Последнее будет происходить автоматически при использовании
   ``--shallow-since=`` или ``--depth=``.

 * Будьте предупреждены: при углублении вашего клона вы можете столкнуться с
   ошибкой вроде 'fatal: error in object: unshallow
   cafecaca0c0dacafecaca0c0dacafecaca0c0da'. В этом случае выполните
   ``git repack -d`` и попробуйте снова``

 * Если вы хотите откатить изменения определённой версии (скажем, Linux 6.3) или
   выполнить бисекцию (v6.2..v6.3), лучше укажите ``git fetch`` получать объекты
   вплоть до трёх версий ранее (например, 6.0): тогда ``git describe`` сможет
   описать большинство коммитов так же, как и в полном git-клоне.

[:ref:`назад к пошаговому руководству <sources_sbs>`] [:ref:`назад к введению раздела <sources>`]

.. _sources_archive:

Загрузка исходников с помощью упакованного архива
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Люди, впервые компилирующие Linux, часто полагают, что загрузка архива через
главную страницу https://kernel.org — это наилучший подход к получению исходников
Linux. Это действительно может быть так, если вы уверены, что будете собирать
только одну конкретную версию ядра, не меняя никакого кода. Дело в том: вы можете
быть уверены, что так и будет, но на практике это часто оказывается ошибочным
предположением.

Это потому, что при сообщении о проблеме или её отладке разработчики часто просят
попробовать другую версию. Они также могут предложить временно отменить коммит с
помощью ``git revert`` или предоставить различные патчи для пробы. Иногда
сообщающих о проблеме также просят использовать ``git bisect`` для поиска
изменения, вызывающего проблему. Эти вещи опираются на git или гораздо проще и
быстрее обрабатываются с его помощью.

Поверхностный клон также не добавляет сколь-нибудь значимых накладных расходов.
Например, когда вы используете ``git clone --depth=1`` для создания поверхностного
клона свежей кодовой базы mainline, git получит лишь немногим больше данных, чем
при загрузке свежего пре-релиза mainline (он же 'rc') через главную страницу
kernel.org.

Поэтому поверхностный клон зачастую является лучшим выбором. Если вы тем не менее
хотите использовать упакованный архив исходников, загрузите его через
kernel.org; после этого извлеките его содержимое в какой-нибудь каталог и
перейдите в подкаталог, созданный при извлечении. Остальная часть пошагового
руководства будет работать прекрасно, за исключением вещей, опирающихся на git,
— но это в основном касается раздела о последовательных сборках других версий.

[:ref:`назад к пошаговому руководству <sources_sbs>`] [:ref:`назад к введению раздела <sources>`]

.. _sources_full:

Загрузка исходников с помощью полного git-клона
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Если загрузка и хранение большого объёма данных (~4,4 гигабайта по состоянию на
начало 2023 года) вас не смущает, то вместо поверхностного клона выполните вместо
него полный git-клон. Тогда вы избежите упомянутых выше особенностей, и у вас в
любой момент будут под рукой все версии и отдельные коммиты::

    curl -L \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git/clone.bundle \
      -o linux-stable.git.bundle
    git clone linux-stable.git.bundle ~/linux/
    rm linux-stable.git.bundle
    cd ~/linux/
    git remote set-url origin \
      https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git
    git fetch origin
    git checkout --detach origin/master

[:ref:`назад к пошаговому руководству <sources_sbs>`] [:ref:`назад к введению раздела <sources>`]

.. _sources_snapshot:

Настоящие пре-релизы (RC) против свежего mainline
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

При клонировании исходников с помощью git и checkout'е origin/master вы зачастую
получите кодовую базу, которая находится где-то между последним и следующим
релизом или пре-релизом. Это почти всегда тот код, который вам нужен, когда вы
пробуете mainline: пре-релизы вроде v6.1-rc5 ничем не примечательны, так как
перед публикацией не проходят какого-либо значимого дополнительного тестирования.

Есть одно исключение: вы можете захотеть остаться на последнем релизе mainline
(скажем, v6.1) до того, как выйдет первый пре-релиз его преемника (v6.2-rc1). Это
потому, что в это время более вероятны ошибки компиляции и другие проблемы, так
как mainline тогда находится в своём 'merge window': обычно двухнедельной фазе, в
течение которой объединяется основная масса изменений для следующего релиза.

[:ref:`назад к пошаговому руководству <sources_sbs>`] [:ref:`назад к введению раздела <sources>`]

.. _sources_fresher:

Как избежать запаздывания mainline
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Пояснения как для поверхностного клона, так и для полного клона оба получают код
из git-репозитория Linux stable. Это упрощает дело для аудитории этого документа,
так как обеспечивает лёгкий доступ как к mainline, так и к stable/longterm-релизам.
У этого подхода есть лишь один недостаток:

Изменения, влитые в репозиторий mainline, синхронизируются с веткой master
репозитория Linux stable лишь каждые несколько часов. Об этом запаздывании
большую часть времени беспокоиться не стоит; но если вам действительно нужен
самый свежий код, просто добавьте репозиторий mainline как дополнительный remote
и сделайте checkout кода оттуда::

    git remote add mainline \
      https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git
    git fetch mainline
    git checkout --detach mainline/master

Делая это с поверхностным клоном, не забудьте вызвать ``git fetch`` с одним из
параметров, описанных ранее, чтобы ограничить глубину.

[:ref:`назад к пошаговому руководству <sources_sbs>`] [:ref:`назад к введению раздела <sources>`]

Тегирование этой сборки ядра (необязательно, часто разумно)

.. _configuration:

Определение конфигурации сборки для вашего ядра
-----------------------------------------------

  *Создайте конфигурацию сборки для вашего ядра на основе существующей
  конфигурации.* [:ref:`... <configuration_sbs>`]

У этого шага есть различные аспекты, требующие более внимательного пояснения:

Подводные камни при использовании другого файла конфигурации в качестве основы
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

У make-целей вроде localmodconfig и olddefconfig есть несколько общих ловушек, о
которых стоит знать:

 * Эти цели повторно используют конфигурацию сборки ядра в вашем каталоге сборки
   (например, '~/linux/.config'), если таковая существует. Если вы хотите начать
   с чистого листа, вам, следовательно, нужно удалить её.

 * Make-цели пытаются найти конфигурацию для вашего работающего ядра
   автоматически, но могут выбрать неудачно. Строка вроде '# using defaults found
   in /boot/config-6.0.7-250.fc36.x86_64' или 'using config:
   '/boot/config-6.0.7-250.fc36.x86_64' сообщает вам, какой файл они выбрали.
   Если это не тот, который предполагался, просто сохраните его как
   '~/linux/.config' перед использованием этих make-целей.

 * Могут произойти неожиданные вещи, если вы попытаетесь использовать файл
   конфигурации, подготовленный для одного ядра (скажем, v6.0), на более старом
   поколении (скажем, v5.15). В этом случае вы можете захотеть использовать в
   качестве основы конфигурацию, которую ваш дистрибутив применял, когда
   использовал ту или несколько более старую версию ядра.

Влияние на конфигурацию
~~~~~~~~~~~~~~~~~~~~~~~~

Make-цель olddefconfig и ``yes "" |``, используемый при работе с localmodconfig,
устанавливают любые неопределённые опции сборки в их значение по умолчанию. Это,
среди прочего, отключит многие функции ядра, которые были введены после выпуска
вашего базового ядра.

Если вы хотите задать эти опции конфигурации вручную, используйте ``oldconfig``
вместо ``olddefconfig`` или опустите ``yes "" |`` при работе с localmodconfig.
Тогда для каждой неопределённой опции конфигурации вас спросят, как поступить.
Если вы не уверены, что ответить, просто нажмите 'enter', чтобы применить значение
по умолчанию.

Большой подводный камень при использовании localmodconfig
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Как уже было кратко объяснено в пошаговом руководстве: с localmodconfig легко
может случиться так, что в вашем самостоятельно собранном ядре будут отсутствовать
модули для задач, которые вы не выполняли до использования этой make-цели. Это
потому, что такие задачи требуют модулей ядра, которые обычно автоматически
загружаются, когда вы выполняете задачу впервые; если вы не выполнили эту задачу
хотя бы раз перед использованием localmodconfig, то последний, следовательно,
сочтёт эти модули избыточными и отключит их.

Вы можете попытаться избежать этого, выполнив типичные задачи, которые часто
автоматически загружают дополнительные модули ядра: запустить ВМ, установить
VPN-соединения, смонтировать через loop ISO-образ CD/DVD, смонтировать сетевые
ресурсы (CIFS, NFS, ...) и подключить все внешние устройства (ключи 2FA,
гарнитуры, веб-камеры, ...), а также накопители с файловыми системами, которые вы
иначе не используете (btrfs, ext4, FAT, NTFS, XFS, ...). Но трудно предусмотреть
всё, что может понадобиться, — даже разработчики ядра часто на этом этапе
забывают что-то одно или другое.

Не позволяйте этому риску вас беспокоить, особенно при компиляции ядра только для
целей тестирования: всё, что обычно критически важно, будет на месте. А если вы
забудете что-то важное, вы сможете позже включить недостающую функцию и быстро
выполнить команды, чтобы скомпилировать и установить лучшее ядро.

Но если вы планируете регулярно собирать и использовать самостоятельно собранные
ядра, вы можете захотеть снизить риск, записывая, какие модули загружает ваша
система на протяжении нескольких недель. Вы можете автоматизировать это с помощью
`modprobed-db <https://github.com/graysky2/modprobed-db>`_. После этого
используйте ``LSMOD=<path>``, чтобы указать localmodconfig на список модулей,
которые modprobed-db заметил используемыми::

    yes "" | make LSMOD="${HOME}"/.config/modprobed.db localmodconfig

Удалённая сборка с localmodconfig
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Если вы хотите использовать localmodconfig для сборки ядра для другой машины,
выполните на ней ``lsmod > lsmod_foo-machine`` и перенесите этот файл на ваш хост
сборки. Теперь укажите скриптам сборки на этот файл так: ``yes "" | make
LSMOD=~/lsmod_foo-machine localmodconfig``. Обратите внимание: в этом случае вам,
скорее всего, захочется скопировать также и базовую конфигурацию ядра с другой
машины и поместить её как .config в ваш каталог сборки.

[:ref:`назад к пошаговому руководству <configuration_sbs>`]

Отладочные символы
~~~~~~~~~~~~~~~~~~~

   *Оцените, как вы хотите поступить с отладочными символами.*
   [:ref:`...<configmods_sbs>`]

Большинству пользователей не нужно об этом беспокоиться, зачастую можно оставить
всё как есть; но вам следует присмотреться к этому, если вам может понадобиться
декодировать трассировку стека или вы хотите снизить потребление места.

Наличие доступных отладочных символов может быть важным, когда ваше ядро
впоследствии при работе выбрасывает 'panic', 'Oops', 'warning' или 'BUG', так как
тогда вы сможете найти точное место в коде, где произошла проблема. Но сбор и
встраивание необходимой отладочной информации требует времени и потребляет
немало места: в конце 2022 года артефакты сборки для типичного ядра x86,
сконфигурированного с localmodconfig, потребляли около 5 гигабайт места с
отладочными символами, но менее 1, когда они были отключены. Результирующий образ
ядра и модули тоже больше, что увеличивает время загрузки.

Поэтому, если вы хотите небольшое ядро и вряд ли будете впоследствии декодировать
трассировку стека, вы можете захотеть отключить отладочные символы, чтобы избежать
указанных выше недостатков::

    ./scripts/config --file .config -d DEBUG_INFO \
      -d DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT -d DEBUG_INFO_DWARF4 \
      -d DEBUG_INFO_DWARF5 -e CONFIG_DEBUG_INFO_NONE
    make olddefconfig

С другой стороны, вы определённо захотите включить их, если есть приличная
вероятность того, что вам впоследствии понадобится декодировать трассировку стека
(как подробнее объяснено в 'Decode failure messages' в
Documentation/admin-guide/tainted-kernels.rst)::

    ./scripts/config --file .config -d DEBUG_INFO_NONE -e DEBUG_KERNEL
      -e DEBUG_INFO -e DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT -e KALLSYMS -e KALLSYMS_ALL
    make olddefconfig

Обратите внимание: многие распространённые дистрибутивы включают отладочные
символы в конфигурациях своих ядер, — поэтому make-цели вроде localmodconfig и
olddefconfig зачастую подхватят эту настройку.

[:ref:`назад к пошаговому руководству <configmods_sbs>`]

.. _build:

Сборка вашего ядра
------------------

  *Соберите образ и модули вашего ядра* [:ref:`... <build_sbs>`]

На этом этапе многое может пойти не так, но приведённые ниже инструкции помогут
вам помочь себе самим. Другой подраздел объясняет, как напрямую упаковать ваше
ядро в файл deb, rpm или tar.

Работа с ошибками сборки
~~~~~~~~~~~~~~~~~~~~~~~~~

Когда возникает ошибка сборки, она может быть вызвана каким-то аспектом настройки
вашей машины, который зачастую можно быстро исправить; в других же случаях
проблема кроется в коде и может быть исправлена только разработчиком. Внимательное
изучение сообщений о сбое в сочетании с некоторым поиском в интернете часто
подскажет вам, какой из двух вариантов имеет место. Чтобы провести такое
расследование, перезапустите процесс сборки так::

    make V=1

``V=1`` активирует подробный вывод, который может понадобиться, чтобы увидеть
фактическую ошибку. Чтобы её было легче заметить, эта команда также опускает
``-j $(nproc --all)``, использовавшийся ранее для задействования каждого ядра CPU
в системе под эту задачу, — но этот параллелизм также приводит к некоторой
загромождённости вывода при возникновении сбоев.

Через несколько секунд процесс сборки должен снова наткнуться на ошибку. Теперь
попробуйте найти самую важную строку, описывающую проблему. Затем поищите в
интернете самую важную и неспецифичную (неуниверсальную) часть этой строки
(скажем, от 4 до 8 слов); избегайте или удаляйте всё, что хоть отдалённо выглядит
специфичным для системы, вроде вашего имени пользователя или локальных имён путей
вроде ``/home/username/linux/``. Сначала попробуйте обычную интернет-поисковую
систему с этой строкой, после этого поищите в списках рассылки ядра Linux через
`lore.kernel.org/all/ <https://lore.kernel.org/all/>`_.

Это в большинстве случаев найдёт что-то, что объяснит, в чём дело; довольно часто
один из результатов также предоставит и решение вашей проблемы. Если вы не
находите ничего, что соответствует вашей проблеме, попробуйте снова под другим
углом, изменив поисковые термины или используя другую строку из сообщений об
ошибках.

В конечном счёте большинство неприятностей, с которыми вы столкнётесь, скорее
всего, уже встречались и были описаны другими. Это включает и проблемы, причина
которых не в вашей системе, а кроется в коде. Если вы столкнётесь с одной из них,
вы, следовательно, тоже сможете найти решение (например, патч) или обходной путь
для вашей проблемы.

Упаковка вашего ядра
~~~~~~~~~~~~~~~~~~~~~

В пошаговом руководстве используются make-цели по умолчанию (например, 'bzImage' и
'modules' на x86) для сборки образа и модулей вашего ядра, которые затем
устанавливаются на более поздних шагах руководства. Вместо этого вы также можете
напрямую собрать всё и напрямую упаковать это, используя одну из следующих целей:

 * ``make -j $(nproc --all) bindeb-pkg`` для генерации пакета deb

 * ``make -j $(nproc --all) binrpm-pkg`` для генерации пакета rpm

 * ``make -j $(nproc --all) tarbz2-pkg`` для генерации сжатого bz2-тарбола

Это лишь подборка доступных для этой цели make-целей, остальные смотрите в
``make help``. Вы также можете использовать эти цели после выполнения
``make -j $(nproc --all)``, так как они подхватят всё уже собранное.

Если вы задействуете цели для генерации пакетов deb или rpm, игнорируйте
инструкции пошагового руководства по установке и удалению вашего ядра; вместо
этого устанавливайте и удаляйте пакеты с помощью пакетной утилиты для данного
формата (например, dpkg и rpm) или утилиты управления пакетами, построенной поверх
них (apt, aptitude, dnf/yum, zypper, ...). Имейте в виду, что пакеты,
сгенерированные с помощью этих двух make-целей, спроектированы так, чтобы работать
на различных дистрибутивах, использующих эти форматы, поэтому они иногда будут
вести себя иначе, чем пакеты ядра вашего дистрибутива.

[:ref:`назад к пошаговому руководству <build_sbs>`]

.. _install:

Установка вашего ядра
---------------------

  *Теперь установите ваше ядро* [:ref:`... <install_sbs>`]

То, что вам нужно сделать после выполнения команды из пошагового руководства,
зависит от существования и реализации исполняемого файла ``installkernel``. Многие
типовые дистрибутивы Linux поставляют такой установщик ядра в ``/sbin/``, который
делает всё необходимое, поэтому вам не остаётся ничего, кроме перезагрузки. Но
некоторые дистрибутивы содержат installkernel, который делает лишь часть работы,
— а в нескольких он полностью отсутствует, и вся работа оставляется на вас.

Если ``installkernel`` найден, система сборки ядра делегирует фактическую
установку образа вашего ядра и связанных файлов этому исполняемому файлу. Почти на
всех дистрибутивах Linux он сохранит образ как '/boot/vmlinuz-<имя релиза вашего
ядра>' и поместит рядом с ним 'System.map-<имя релиза вашего ядра>'. Таким
образом, ваше ядро будет установлено параллельно любым существующим, если только
у вас уже нет ядра с точно таким же именем релиза.

Installkernel на многих дистрибутивах после этого сгенерирует 'initramfs' (часто
также называемый 'initrd'), на который типовые дистрибутивы опираются при
загрузке; поэтому обязательно сохраняйте порядок двух make-целей, используемых в
пошаговом руководстве, так как всё пойдёт не так, если вы установите образ ядра
до его модулей. Часто installkernel затем также добавит ваше ядро в конфигурацию
загрузчика. Вам придётся позаботиться об одной или обеих этих задачах самим, если
installkernel вашего дистрибутива их не обрабатывает.

В нескольких дистрибутивах вроде Arch Linux и его производных исполняемый файл
installkernel полностью отсутствует. На них просто установите модули с помощью
системы сборки ядра, а затем установите образ и файл System.map вручную::

     sudo make modules_install
     sudo install -m 0600 $(make -s image_name) /boot/vmlinuz-$(make -s kernelrelease)
     sudo install -m 0600 System.map /boot/System.map-$(make -s kernelrelease)

Если ваш дистрибутив загружается с помощью initramfs, теперь сгенерируйте его для
вашего ядра, используя инструменты, которые ваш дистрибутив предоставляет для
этого процесса. После этого добавьте ваше ядро в конфигурацию загрузчика и
перезагрузитесь.

[:ref:`назад к пошаговому руководству <install_sbs>`]

.. _another:

Ещё один заход позже
--------------------

  *Чтобы позже собрать ещё одно ядро, вам понадобятся похожие, но иногда слегка
  отличающиеся команды* [:ref:`... <another_sbs>`]

Процесс сборки последующих ядер похож, но в некоторых местах слегка отличается.
Например, вы не захотите использовать 'localmodconfig' для последующих сборок
ядра, так как вы уже создали урезанную конфигурацию, которую хотите использовать
отныне. Поэтому вместо этого просто используйте ``oldconfig`` или ``olddefconfig``
для корректировки конфигураций сборки под нужды той версии ядра, которую вы
собираетесь собирать.

Если вы создали поверхностный клон с помощью git, вспомните, что говорил
:ref:`раздел, который подробнее объяснял описанную настройку <sources>`: вам нужно
использовать слегка иную команду ``git fetch``, а при переключении на другую серию
нужно добавить дополнительную удалённую ветку.

[:ref:`назад к пошаговому руководству <another_sbs>`]

.. _uninstall:

Удаление ядра впоследствии
--------------------------

  *Все части вашего установленного ядра идентифицируются по имени его релиза и
  поэтому легко удаляются впоследствии.* [:ref:`... <uninstall_sbs>`]

Не беспокойтесь, что установка вашего ядра вручную и, как следствие, обход системы
пакетов вашего дистрибутива полностью испортит вашу машину: все части вашего ядра
легко удалить впоследствии, так как файлы хранятся лишь в двух местах и обычно
идентифицируются по имени релиза ядра.

Одно из этих двух мест — каталог в /lib/modules/, в котором находятся модули для
каждого установленного ядра. Этот каталог назван по имени релиза ядра; поэтому,
чтобы удалить все модули для одного из ваших ядер, просто удалите его каталог
модулей в /lib/modules/.

Другое место — /boot/, куда при установке ядра обычно помещается от одного до пяти
файлов. Все они обычно содержат имя релиза в имени файла, но то, сколько файлов и
каковы их имена, в некоторой степени зависит от исполняемого файла installkernel
вашего дистрибутива (:ref:`см. выше <install>`) и его генератора initramfs. В
некоторых дистрибутивах команда ``kernel-install``, упомянутая в пошаговом
руководстве, удалит все эти файлы за вас — и в то же время также запись для
вашего ядра в конфигурации загрузчика. В других вам придётся позаботиться об этих
шагах самим. Следующая команда должна интерактивно удалить два основных файла ядра
с именем релиза '6.0.1-foobar'::

    rm -i /boot/{System.map,vmlinuz}-6.0.1-foobar

Теперь удалите принадлежащий ему initramfs, который часто будет называться нечто
вроде ``/boot/initramfs-6.0.1-foobar.img`` или ``/boot/initrd.img-6.0.1-foobar``.
После этого проверьте наличие других файлов в /boot/, у которых в имени есть
'6.0.1-foobar', и удалите их тоже. Теперь удалите ядро из конфигурации вашего
загрузчика.

Обратите внимание: будьте очень осторожны с подстановочными символами вроде '*'
при удалении файлов или каталогов ядер вручную: вы можете случайно удалить файлы
ядра 6.0.11, когда всё, чего вы хотите, — это удалить 6.0 или 6.0.1.

[:ref:`назад к пошаговому руководству <uninstall_sbs>`]

.. _faq:

ЧАВО
====

Почему это руководство (how-to) не работает на моей системе?
------------------------------------------------------------

Как было изначально заявлено, это руководство 'разработано так, чтобы охватить
всё, что обычно нужно [для сборки ядра] в распространённых дистрибутивах Linux,
работающих на типовом железе ПК или серверов'. Описанный подход, несмотря на это,
должен работать и на многих других конфигурациях. Но попытка охватить в одном
руководстве каждый возможный сценарий использования свела бы на нет его цель, так
как без такого фокуса вам понадобились бы десятки или сотни конструкций в духе 'в
случае, если у вас <вставьте машину или дистрибутив>, вам на этом этапе нужно
сделать <то и сё> <вместо|дополнительно>'. Каждая из них делала бы текст длиннее,
сложнее и труднее для восприятия.

При этом следует сказать: это, конечно, балансирование. Поэтому, если вы считаете,
что дополнительный сценарий использования стоит описать, предложите его
сопровождающим (maintainers) этого документа, как :ref:`описано выше
<submit_improvements_qbtl>`.
