=================
Использование
=================

Этот модуль поддерживает семейство современных сетевых протоколов SMB3 (а также
более старые диалекты, изначально называвшиеся «CIFS» или SMB1).

Модуль CIFS VFS для Linux поддерживает множество продвинутых возможностей сетевых
файловых систем, таких как иерархическое пространство имён вроде DFS, жёсткие ссылки,
блокировки и многое другое. Он был разработан в соответствии с SNIA CIFS Technical
Reference (которая заменяет стандарт X/Open SMB 1992 года), а также для обеспечения
практической совместимости по лучшим практикам с Windows 2000, Windows XP, Samba и
эквивалентными серверами. Этот код разрабатывался при участии Protocol Freedom
Information Foundation. CIFS, а теперь и SMB3, стал де-факто стандартом для
взаимодействия между Mac и Windows, а также основными NAS-устройствами.

Дополнительные подробности см. в
MS-SMB2 (подробная спецификация протоколов SMB2/SMB3/SMB3.1.1)
или на https://samba.org/samba/PFIF/.

Инструкции по сборке
====================

Для Linux:

1) Загрузите ядро (например, с https://www.kernel.org)
   и перейдите в каталог верхнего уровня дерева исходного кода ядра
   (например, /usr/src/linux-2.5.73)
2) make menuconfig (или make xconfig)
3) выберите cifs среди вариантов сетевых файловых систем
4) сохраните и выйдите
5) make


Инструкции по установке
=======================

Если вы (успешно) собрали CIFS vfs как модуль, просто
выполните ``make modules_install`` (или, если предпочитаете, вручную скопируйте файл в
каталог модулей, например /lib/modules/6.3.0-060300-generic/kernel/fs/smb/client/cifs.ko).

Если вы встроили CIFS vfs непосредственно в ядро, следуйте инструкциям
вашего дистрибутива по установке нового ядра (обычно вы
просто выполняете ``make install``).

Если у вас нет утилиты mount.cifs (она есть в дереве исходного кода Samba 4.x и на
веб-сайте CIFS VFS), скопируйте её в тот же каталог, в котором находятся вспомогательные
программы монтирования (обычно /sbin). Хотя вспомогательное ПО не
обязательно, mount.cifs рекомендуется. В большинстве дистрибутивов есть пакет
``cifs-utils``, включающий эту утилиту, поэтому рекомендуется его установить.

Обратите внимание, что запуск модуля Winbind pam/nss (службы входа в систему) на всех ваших
Linux-клиентах полезен для согласованного сопоставления Uid и Gid по всему
домену соответствующему сетевому пользователю. Вспомогательную программу монтирования
mount.cifs можно найти в cifs-utils.git на git.samba.org

Если cifs собран как модуль, то размер и количество сетевых буферов,
а также максимальное число одновременных запросов к одному серверу можно настроить.
Изменять эти значения относительно значений по умолчанию не рекомендуется. Выполнив modinfo::

	modinfo <path to cifs.ko>

для kernel/fs/smb/client/cifs.ko, можно увидеть список изменений конфигурации, которые можно
внести во время инициализации модуля (при запуске insmod cifs.ko).

Рекомендации
============

Для повышения безопасности диалект SMB2.1 или более поздний (обычно будет выбран SMB3.1.1) теперь
является новым значением по умолчанию. Чтобы использовать старые диалекты (например, для монтирования
Windows XP), укажите «vers=1.0» при монтировании (или vers=2.0 для Windows Vista). Обратите внимание,
что CIFS (vers=1.0) намного старее и менее безопасен, чем диалект по умолчанию SMB3, который включает
множество продвинутых функций безопасности, таких как обнаружение атак понижения версии,
шифрование общих ресурсов, а также более стойкие алгоритмы подписи и аутентификации.
Существуют дополнительные параметры монтирования, которые могут быть полезны для SMB3, чтобы получить
улучшенное POSIX-поведение (примечание: можно использовать vers=3, чтобы принудительно задать SMB3 или более поздний, но никогда 2.1):

   ``mfsymlinks`` и либо ``cifsacl``, либо ``modefromsid`` (обычно вместе с ``idsfromsid``)

Разрешение пользовательских монтирований
========================================

Разрешить пользователям монтировать и размонтировать каталоги, которыми они владеют, возможно
с помощью cifs vfs. Один из способов включить такое монтирование — пометить утилиту mount.cifs
как suid (например, ``chmod +s /sbin/mount.cifs``). Чтобы разрешить пользователям
размонтировать смонтированные ими общие ресурсы, требуется

1) mount.cifs версии 1.4 или более поздней
2) запись для общего ресурса в /etc/fstab, указывающая, что пользователь может
   его размонтировать, например::

     //server/usersharename  /mnt/username cifs user 0 0

Обратите внимание, что когда утилита mount.cifs запускается с suid (разрешая пользовательские монтирования),
для снижения рисков при монтировании передаётся флаг монтирования ``nosuid``, чтобы
запретить выполнение suid-программы, смонтированной с удалённой цели.
Когда mount выполняется от имени root, nosuid по умолчанию не передаётся,
и выполнение suid-программ с удалённой цели по умолчанию будет разрешено.
Это можно изменить, как и в случае с nfs и другими файловыми системами,
просто указав ``nosuid`` среди параметров монтирования. Однако для пользовательских монтирований
возможность передавать флаг suid в mount требует пересборки
mount.cifs со следующим флагом: CIFS_ALLOW_USR_SUID

Соответствующая страница руководства по монтированию cifs есть в дереве исходного кода Samba 3.0 и
более поздних версий в docs/manpages/mount.cifs.8

Разрешение пользовательских размонтирований
===========================================

Чтобы разрешить пользователям размонтировать каталоги, которые они смонтировали как пользователи (см. выше),
можно использовать утилиту umount.cifs. Её можно вызвать напрямую, либо, если
umount.cifs помещена в /sbin, umount может вызвать вспомогательную программу размонтирования cifs
(по крайней мере для большинства версий утилиты umount) для размонтирования cifs-монтирований,
если только umount не вызван с -i (что позволит избежать вызова вспомогательной
программы размонтирования). Как и в случае с mount.cifs, чтобы включить пользовательские размонтирования, umount.cifs
должна быть помечена как suid (например, ``chmod +s /sbin/umount.cifs``) или аналогично (некоторые дистрибутивы
позволяют добавлять записи в файл /etc/permissions для достижения
эквивалентного эффекта suid). Чтобы эта утилита сработала, целевой путь
должен быть cifs-монтированием, а uid текущего пользователя должен совпадать с uid
пользователя, смонтировавшего ресурс.

Также обратите внимание, что обычный способ разрешения пользовательских монтирований и размонтирований
(вместо использования mount.cifs и unmount.cifs как suid) состоит в добавлении строки
в файл /etc/fstab для каждого //server/share, который вы хотите смонтировать, но
это может стать громоздким, когда потенциальные цели монтирования включают много
или непредсказуемые UNC-имена.

Особенности Samba
=================

Большинство современных серверов поддерживают SMB2.1 и SMB3, которые более безопасны,
но для более старого, менее безопасного диалекта CIFS существуют полезные расширения протокола,
поэтому, чтобы получить максимальную выгоду при монтировании с использованием старого диалекта
(CIFS/SMB1), мы рекомендуем использовать сервер, поддерживающий стандарт SNIA CIFS
Unix Extensions (например, практически любую версию Samba, то есть версию
2.2.5 или более позднюю), однако CIFS vfs прекрасно работает с самыми разными CIFS-серверами.
Обратите внимание, что uid, gid и права доступа к файлам будут отображаться со значениями по умолчанию, если у вас
нет сервера, поддерживающего Unix-расширения для CIFS (такого как Samba
2.2.5 или более поздней). Чтобы включить Unix CIFS Extensions на сервере Samba, добавьте
строку::

	unix extensions = yes

в файл smb.conf на сервере. Обратите внимание, что следующие настройки smb.conf
также полезны (на сервере Samba), когда большинство клиентов работают под Unix или
Linux::

	case sensitive = yes
	delete readonly = yes
	ea support = yes

Обратите внимание, что поддержка ea на сервере необходима для поддержки xattrs со стороны Linux
cifs-клиента, и что поддержка EA присутствует в более поздних версиях Samba (например,
3.0.6 и более поздних) (также поддержка EA работает во всех версиях Windows, по крайней мере для
общих ресурсов на файловых системах NTFS). Поддержка расширенных атрибутов (xattr) является необязательной
возможностью большинства файловых систем Linux, которая может потребовать включения через
make menuconfig. Поддержку расширенных атрибутов со стороны клиента (user xattr) можно
отключить для отдельного монтирования, указав ``nouser_xattr`` при монтировании.

CIFS-клиент может получать и устанавливать POSIX ACL (getfacl, setfacl) для серверов Samba
версии 3.10 и более поздних. Установка POSIX ACL требует включения как XATTR, так и
POSIX-поддержки в параметрах конфигурации CIFS при сборке cifs-модуля.
Поддержку POSIX ACL можно отключить для отдельного монтирования, указав
``noacl`` при монтировании.

Некоторые администраторы могут захотеть изменить параметры smb.conf ``map archive`` и
``create mask`` для Samba относительно значений по умолчанию. Если create mask не изменена,
вновь создаваемые файлы могут получить излишне ограничительный режим по умолчанию,
который может оказаться не тем, что вам нужно, хотя, если CIFS Unix extensions
включены на сервере и клиенте, последующие вызовы setattr (например, chmod) могут
исправить режим. Обратите внимание, что создание специальных устройств (mknod)
удалённо может потребовать указания функции mkdev для Samba, если вы не используете
Samba 3.0.6 или более позднюю. Подробнее об этом см. страницы руководства
(``man smb.conf``) на системе сервера Samba. Обратите внимание, что cifs vfs,
в отличие от smbfs vfs, не читает smb.conf на клиентской системе
(вместо этого немногочисленные необязательные настройки передаются при монтировании через параметры -o).
Обратите внимание, что Samba 2.2.7 или более поздняя включает исправление, позволяющее CIFS VFS удалять
открытые файлы (необходимо для строгого соответствия POSIX). Серверы Windows уже
поддерживали эту возможность. Сервер Samba не разрешает символические ссылки, ссылающиеся на файлы
вне общего ресурса, поэтому в версиях Samba до 3.0.6 большинство символических ссылок на
файлы с абсолютными путями (то есть начинающимися со слеша), такие как::

	 ln -s /mnt/foo bar

были бы запрещены. Сервер Samba 3.0.6 или более поздний обладает возможностью безопасно создавать
такие символические ссылки, преобразуя небезопасные символические ссылки (то есть символические ссылки на
файлы сервера, находящиеся вне общего ресурса) в специфический для samba формат на сервере,
который игнорируется локальными серверными приложениями и не-cifs-клиентами и который
не будет обходиться сервером Samba). Это прозрачно для приложения Linux-клиента,
использующего cifs vfs. Абсолютные символические ссылки будут работать с Samba 3.0.5 или
более поздней, но только для удалённых клиентов, использующих CIFS Unix extensions, и будут
невидимы для Windows-клиентов и обычно не повлияют на локальные
приложения, работающие на том же сервере, что и Samba.

Инструкции по использованию
===========================

Как только поддержка CIFS VFS встроена в ядро или установлена как модуль
(cifs.ko), вы можете использовать синтаксис монтирования вроде следующего для доступа к серверам Samba,
Mac или Windows::

  mount -t cifs //9.53.216.11/e$ /mnt -o username=myname,password=mypassword

Перед -o можно указать параметр -v, чтобы вспомогательная программа монтирования mount.cifs
более подробно отображала шаги монтирования.
После -o поддерживаются следующие часто используемые параметры,
специфичные для cifs vfs::

  username=<username>
  password=<password>
  domain=<domain name>

Другие параметры монтирования cifs описаны ниже. Использование TCP-имён (в дополнение к
IP-адресам) доступно, если установлена вспомогательная программа монтирования (mount.cifs). Если
вы не доверяете серверу, к которому подключены, или если у вас не включена
подпись cifs (а физическая сеть небезопасна), рассмотрите использование
стандартных параметров монтирования ``noexec`` и ``nosuid``, чтобы снизить риск
запуска изменённого исполняемого файла в вашей локальной системе (загруженного с враждебного сервера
или изменённого враждебным маршрутизатором).

Хотя монтирование в формате, соответствующем спецификации CIFS URL,
в mount.cifs пока невозможно, можно использовать альтернативный формат
для сервера и имени общего ресурса (несколько похожий на синтаксис монтирования в стиле NFS)
вместо более широко используемого формата UNC (то есть \\server\share)::

  mount -t cifs tcp_name_of_server:share_name /mnt -o user=myname,pass=mypasswd

При использовании вспомогательной программы монтирования mount.cifs пароли можно указывать с помощью альтернативных
механизмов, вместо указания его после -o с помощью обычного синтаксиса ``pass=``
в командной строке:
1) Включив его в файл учётных данных. Укажите credentials=filename в качестве одного
из параметров монтирования. Файлы учётных данных содержат две строки::

	username=someuser
	password=your_password

2) Указав пароль в переменной окружения PASSWD (аналогично
   имя пользователя может быть взято из переменной окружения USER).
3) Указав пароль в файле по имени через PASSWD_FILE
4) Указав пароль в файле по файловому дескриптору через PASSWD_FD

Если пароль не предоставлен, mount.cifs запросит ввод пароля

Ограничения
===========

Серверы должны поддерживать либо «чистый TCP» (CIFS-соединения по TCP/IP через порт 445), либо
поддержку RFC 1001/1002 для «Netbios-Over-TCP/IP». Это вряд ли станет
проблемой, поскольку большинство серверов это поддерживают.

Допустимые имена файлов различаются между Windows и Linux. Windows обычно ограничивает
имена файлов, содержащие определённые зарезервированные символы (например, символ :,
который используется в Windows для обозначения начала имени потока), тогда как
Linux допускает несколько более широкий набор допустимых символов в именах файлов. Серверы Windows
могут переотображать такие символы, когда явное сопоставление задано
в реестре сервера. Samba, начиная с версии 3.10, будет разрешать такие
имена файлов (то есть содержащие допустимые символы Linux, которые обычно
были бы запрещены семантикой Windows/CIFS), при условии, что сервер
настроен на Unix Extensions (и клиент не отключил
/proc/fs/cifs/LinuxExtensionsEnabled). Кроме того, параметр монтирования
``mapposix`` можно использовать с CIFS (vers=1.0), чтобы принудительно сопоставить
недопустимые символы Windows/NTFS/SMB в диапазон переотображения (этот параметр монтирования
является значением по умолчанию для SMB3). Этот диапазон переотображения (``mapposix``) также
совместим с Mac (и «Services for Mac» на некоторых более старых версиях Windows).
Когда согласуются POSIX Extensions для SMB 3.1.1, переотображение автоматически
отключается.

Параметры монтирования CIFS VFS
===============================
Ниже приведён частичный список поддерживаемых параметров монтирования:

  username
		Имя пользователя, используемое при попытке установить
		сеанс CIFS.
  password
		Пароль пользователя. Если установлена вспомогательная программа монтирования,
		у пользователя будет запрошен пароль,
		если он не указан.
  ip
		IP-адрес целевого сервера
  unc
		Universal Network Name (экспорт) целевого сервера для
		монтирования.
  domain
		Задаёт имя рабочей группы SMB/CIFS, добавляемое к имени
		пользователя при установлении сеанса CIFS
  forceuid
		Задаёт uid по умолчанию для inode равным uid,
		переданному при монтировании. Для монтирований к серверам,
		которые поддерживают CIFS Unix extensions, например к
		правильно настроенному серверу Samba, сервер предоставляет
		uid, gid и режим, поэтому этот параметр не следует
		указывать, если только нумерация uid и gid на сервере и клиенте
		не различается. Если сервер и клиент находятся в
		одном домене (например, работают winbind или nss_ldap) и
		сервер поддерживает Unix Extensions, то uid
		и gid могут быть получены с сервера (и uid
		и gid не нужно указывать при монтировании).
		Для серверов, которые не поддерживают CIFS Unix
		extensions, uid (и gid) по умолчанию, возвращаемый при поиске
		существующих файлов, будет uid (gid) человека,
		выполнившего монтирование (root, за исключением случаев, когда mount.cifs
		настроена с setuid для пользовательских монтирований), если только не указан параметр монтирования ``uid=``
		(gid). Также обратите внимание, что проверки прав доступа
		(проверки авторизации) при доступе к файлу происходят
		на сервере, но есть случаи, когда администратор
		может захотеть ограничить доступ и на клиенте. Для тех
		серверов, которые не сообщают владельца uid/gid
		(таких как Windows), права доступа также могут проверяться на
		клиенте, и грубую форму проверки прав доступа на стороне клиента
		можно включить, указав file_mode и dir_mode на
		клиенте. (по умолчанию)
  forcegid
		(аналогично указанному выше, но для groupid вместо uid) (по умолчанию)
  noforceuid
		Заполнять информацию о владельце файла (uid), запрашивая её с
		сервера, если возможно. С этим параметром значение, заданное в
		параметре uid= (при монтировании), будет использоваться, только если сервер
		не может поддерживать возврат uid для inode.
  noforcegid
		(аналогично указанному выше, но для владельца-группы, gid, вместо uid)
  uid
		Задаёт uid по умолчанию для inode и указывает
		cifs-драйверу ядра, какой локальный пользователь выполнил монтирование. Если сервер
		поддерживает Unix extensions, uid по умолчанию
		не используется для заполнения полей владельца inode (файлов),
		если не указан параметр ``forceuid``.
  gid
		Задаёт gid по умолчанию для inode (аналогично указанному выше).
  file_mode
		Если CIFS Unix extensions не поддерживаются сервером,
		это переопределяет режим по умолчанию для inode файлов.
  fsc
		Включить локальное кэширование на диске с помощью FS-Cache (по умолчанию отключено). Этот
		параметр может быть полезен для повышения производительности на медленном канале,
		сильно нагруженном сервере и/или сети, где чтение с
		диска быстрее, чем чтение с сервера (по сети).
		Это также может положительно повлиять на масштабируемость, поскольку
		число вызовов к серверу сокращается. Однако локальное
		кэширование подходит не для всех рабочих нагрузок, например для рабочих нагрузок
		типа «прочитать один раз». Поэтому необходимо тщательно обдумать вашу
		рабочую нагрузку/сценарий перед использованием этого параметра. В настоящее время локальное
		кэширование на диске функционирует для CIFS-файлов, открытых только на чтение.
  dir_mode
		Если CIFS Unix extensions не поддерживаются сервером,
		это переопределяет режим по умолчанию для inode каталогов.
  port
		пытаться связаться с сервером по этому tcp-порту, прежде чем
		пробовать обычные порты (порт 445, затем 139).
  iocharset
		Кодовая страница, используемая для преобразования локальных имён путей в Unicode и
		из него. Unicode используется по умолчанию для сетевых имён
		путей, если сервер его поддерживает. Если iocharset не
		указан, то будет использоваться nls_default, заданный
		при сборке ядра локального клиента.
		Если сервер не поддерживает Unicode, этот параметр
		не используется.
  rsize
		размер чтения по умолчанию (обычно 16K). Клиент в настоящее время
		не может использовать rsize больше, чем CIFSMaxBufSize. CIFSMaxBufSize
		по умолчанию равен 16K и может быть изменён (от 8K до максимального
		размера kmalloc, разрешённого вашим ядром) во время установки модуля
		для cifs.ko. Установка CIFSMaxBufSize в очень большое значение
		приведёт к тому, что cifs будет использовать больше памяти, и может снизить производительность
		в некоторых случаях. Для использования rsize больше 127K (исходный
		максимум протокола cifs) также требуется, чтобы сервер поддерживал
		новый флаг Unix Capability (для очень больших чтений), что поддерживают некоторые
		более новые серверы (например, Samba 3.0.26 или более поздняя). rsize можно
		задать от минимума 2048 до максимума 130048 (127K или
		CIFSMaxBufSize, в зависимости от того, что меньше)
  wsize
		размер записи по умолчанию (по умолчанию 57344)
		максимальный wsize, разрешённый в настоящее время в CIFS, равен 57344 (четырнадцать
		страниц по 4096 байт)
  actimeo=n
		тайм-аут кэша атрибутов в секундах (по умолчанию 1 секунда).
		По истечении этого тайм-аута cifs-клиент запрашивает свежую информацию
		об атрибутах с сервера. Этот параметр позволяет настроить
		тайм-аут кэша атрибутов под нужды рабочей нагрузки. Более короткие
		тайм-ауты означают лучшую когерентность кэша, но увеличенное число
		вызовов к серверу. Более длинные тайм-ауты означают сокращённое число
		вызовов к серверу ценой менее строгих проверок когерентности кэша
		(то есть некорректный кэш атрибутов в течение короткого
		периода времени).
  rw
		монтировать сетевой общий ресурс на чтение-запись (обратите внимание, что
		сервер всё равно может считать общий ресурс доступным только для чтения)
  ro
		монтировать сетевой общий ресурс только для чтения
  version
		используется для различения разных версий
		вспомогательной утилиты монтирования (обычно не требуется)
  sep
		если это первый параметр монтирования (после -o), переопределяет
		запятую как разделитель между параметрами
		монтирования. например::

			-o user=myname,password=mypassword,domain=mydom

		вместо этого можно было бы передать с точкой в качестве разделителя так::

			-o sep=.user=myname.password=mypassword.domain=mydom

		это может быть полезно, когда запятая содержится внутри имени пользователя,
		пароля или домена. Этот параметр менее важен,
		когда используется вспомогательная программа монтирования cifs cifs.mount (версии 1.1 или более поздней).
  nosuid
		Не разрешать выполнение удалённых исполняемых файлов с
		установленным suid-битом. Это имеет смысл только для монтирований
		к серверам, таким как Samba, поддерживающим CIFS Unix Extensions.
		Если вы не доверяете серверам в своей сети (вашим целям
		монтирования), рекомендуется указывать этот параметр для
		большей безопасности.
  exec
		Разрешить выполнение двоичных файлов на монтировании.
  noexec
		Не разрешать выполнение двоичных файлов на монтировании.
  dev
		Распознавать блочные устройства на удалённом монтировании.
  nodev
		Не распознавать устройства на удалённом монтировании.
  suid
		Разрешить выполнение удалённых файлов на этой точке монтирования с включённым
		suid (по умолчанию для монтирований, выполняемых от имени root,
		nosuid является значением по умолчанию для пользовательских монтирований).
  credentials
		Хотя cifs-компонент ядра его игнорирует, он используется
		вспомогательной программой монтирования mount.cifs. Когда mount.cifs установлена, она
		открывает и читает указанный файл учётных данных, чтобы
		получить аргументы userid и password, которые передаются в
		cifs vfs.
  guest
		Хотя компонент ядра его игнорирует, вспомогательная программа монтирования
		mount.cifs не будет запрашивать у пользователя пароль,
		если в параметрах монтирования указан guest. Если пароль
		не указан, будет использован пустой пароль.
  perm
		Клиент выполняет проверки прав доступа (проверка vfs_permission для uid
		и gid файла относительно режима и желаемой операции).
		Обратите внимание, что это дополнение к обычной проверке ACL на
		целевой машине, выполняемой серверным ПО.
		Проверка прав доступа на клиенте включена по умолчанию.
  noperm
		Клиент не выполняет проверки прав доступа. Это может открыть
		файлы на этом монтировании для доступа другими пользователями локальной
		клиентской системы. Обычно это требуется только тогда, когда сервер
		поддерживает CIFS Unix Extensions, но UID/GID на
		клиенте и сервере не совпадают достаточно близко, чтобы разрешить
		доступ пользователю, выполняющему монтирование, но это также может быть полезно при
		монтированиях без CIFS Unix Extension в случаях, когда режим по умолчанию
		указан при монтировании, но не должен принудительно применяться на
		клиенте (например, возможно, когда включён MultiUserMount).
		Обратите внимание, что это не влияет на обычную проверку ACL на
		целевой машине, выполняемую серверным ПО (ACL сервера
		относительно имени пользователя, предоставленного во время монтирования).
  serverino
		Использовать номера inode сервера вместо генерации автоматически
		инкрементируемых номеров inode на клиенте. Хотя это облегчит
		обнаружение файлов с жёсткими ссылками (так как они будут иметь
		одинаковые номера inode), и номера inode могут быть постоянными,
		обратите внимание, что сервер не гарантирует, что номера inode
		уникальны, если несколько серверных монтирований экспортируются под
		одним общим ресурсом (поскольку номера inode на серверах могут не
		быть уникальными, если несколько файловых систем смонтированы под одним
		и тем же общим каталогом более высокого уровня). Обратите внимание, что некоторые более старые
		(например, до Windows 2000) не поддерживают возврат UniqueIDs
		или эквивалента CIFS Unix Extensions, и для таких
		этот параметр монтирования не будет иметь эффекта. Экспорт cifs-монтирований
		под nfsd требует этого параметра монтирования на cifs-монтировании.
		Теперь это значение по умолчанию, если сервер поддерживает
		требуемую сетевую операцию.
  noserverino
		Клиент генерирует номера inode (вместо использования фактического
		с сервера). Эти номера inode будут изменяться после
		размонтирования или перезагрузки, что может сбить с толку некоторые приложения,
		но не все серверные файловые системы поддерживают уникальные номера
		inode.
  setuids
		Если CIFS Unix extensions согласованы с сервером,
		клиент попытается установить эффективные uid и gid
		локального процесса для вновь создаваемых файлов, каталогов и
		устройств (create, mkdir, mknod). Если CIFS Unix Extensions
		не согласованы, для вновь создаваемых файлов и каталогов
		вместо использования uid и gid по умолчанию, указанных при
		монтировании, кэшировать uid и gid нового файла локально, что означает,
		что uid для файла может измениться при перезагрузке inode
		(или при повторном монтировании пользователем общего ресурса).
  nosetuids
		Клиент не будет пытаться устанавливать uid и gid
		для вновь создаваемых файлов, каталогов и устройств (create,
		mkdir, mknod), что приведёт к тому, что сервер установит
		uid и gid в значения по умолчанию (обычно серверный uid
		пользователя, смонтировавшего общий ресурс). Позволять серверу (а не
		клиенту) устанавливать uid и gid — это значение по умолчанию. Если CIFS
		Unix Extensions не согласованы, то uid и gid для
		новых файлов будут отображаться как uid (gid) того, кто выполнил монтирование, или
		uid (gid), указанный в параметре при монтировании.
  netbiosname
		При монтировании к серверам через порт 139 указывает исходное имя RFC1001,
		используемое для представления netbios-имени клиентской машины
		при выполнении инициализации netbios-сеанса RFC1001.
  direct
		Не выполнять кэширование данных inode для файлов, открытых на этом монтировании.
		Это исключает mmap-отображение файлов на этом монтировании. В некоторых случаях
		с быстрыми сетями и малой выгодой от кэширования (или её отсутствием) на
		клиенте (например, когда приложение выполняет большие последовательные
		чтения, превышающие размер страницы, без повторного чтения тех же данных)
		это может обеспечить лучшую производительность, чем поведение по умолчанию,
		которое кэширует чтения (упреждающее чтение, readahead) и записи
		(отложенная запись, writebehind) через локальный pagecache Linux-клиента,
		если предоставлен и удерживается oplock (токен кэширования). Обратите внимание, что
		direct позволяет отправлять на сервер операции записи, превышающие
		размер страницы.
  strictcache
		Использовать для включения строгого режима кэша. В этом режиме
		клиент читает из кэша всё время, пока имеет Oplock Level II,
		в противном случае — читает с сервера. Все записанные данные хранятся
		в кэше, но если у клиента нет Exclusive Oplock,
		он записывает данные на сервер.
  rwpidforward
		Перенаправлять pid процесса, открывшего файл, в любую операцию чтения или записи
		этого файла. Это предотвращает сбои приложений, таких как WINE,
		при чтении и записи, если мы используем стиль обязательных brlock.
  acl
		Разрешить setfacl и getfacl управлять posix ACL, если сервер
		их поддерживает. (по умолчанию)
  noacl
		Не разрешать вызовы setfacl и getfacl на этом монтировании
  user_xattr
		Разрешить получение и установку пользовательских xattrs (тех атрибутов, имя
		которых начинается с ``user.`` или ``os2.``) как OS/2 EA (расширенных
		атрибутов) на сервере. Это позволяет поддерживать
		утилиты setfattr и getfattr. (по умолчанию)
  nouser_xattr
		Не разрешать getfattr/setfattr получать/устанавливать/перечислять xattrs
  mapchars
		Транслировать шесть из семи зарезервированных символов (не обратный слеш)::

			*?<>|:

		в диапазон переотображения (выше 0xF000), что также
		позволяет CIFS-клиенту распознавать файлы, созданные с
		такими символами POSIX-эмуляцией Windows. Это может
		также быть полезно при монтировании к большинству версий Samba
		(которая также запрещает создание и открытие файлов,
		имена которых содержат любой из этих семи символов).
		Это не имеет эффекта, если сервер не поддерживает
		Unicode при передаче.
  nomapchars
		Не транслировать ни один из этих семи символов (по умолчанию).
  nocase
		Запросить нечувствительное к регистру сопоставление имён путей (чувствительное
		к регистру является значением по умолчанию, если сервер его поддерживает).
		(параметр монтирования ``ignorecase`` идентичен ``nocase``)
  posixpaths
		Если CIFS Unix extensions поддерживаются, попытаться
		согласовать поддержку posix-имён путей, которая разрешает определённые
		символы, запрещённые в типичных именах файлов CIFS, без
		необходимости переотображения. (по умолчанию)
  noposixpaths
		Если CIFS Unix extensions поддерживаются, не запрашивать
		поддержку posix-имён путей (это может привести к тому, что серверы
		будут отклонять создание файла с определёнными зарезервированными символами).
  nounix
		Отключить CIFS Unix Extensions для этого монтирования (древовидного
		соединения). Это требуется редко, но может быть полезно
		для отключения нескольких настроек сразу (то есть
		posix acls, posix locks, posix paths, поддержки символических ссылок
		и получения uids/gids/mode с сервера) или для
		обхода ошибки в сервере, реализующем Unix
		Extensions.
  nobrl
		Не отправлять серверу запросы блокировки диапазона байтов.
		Это необходимо для некоторых приложений, которые ломаются
		на обязательных блокировках диапазона байтов в стиле cifs (а большинство
		cifs-серверов пока не поддерживают запрос рекомендательных
		блокировок диапазона байтов).
  forcemandatorylock
		Даже если сервер поддерживает posix (рекомендательную) блокировку
		диапазона байтов, отправлять только обязательные запросы блокировки. Для некоторых
		(предположительно редких) приложений, изначально написанных для
		DOS/Windows, которым требуется обязательная блокировка диапазона байтов в стиле Windows,
		они могут воспользоваться этим параметром,
		заставляя cifs-клиента отправлять только обязательные блокировки,
		даже если cifs-сервер поддерживал бы рекомендательные posix-блокировки.
		``forcemand`` принимается как более короткая форма этого параметра
		монтирования.
  nostrictsync
		Если этот параметр монтирования установлен, то когда приложение выполняет
		вызов fsync, cifs-клиент не отправляет SMB Flush
		серверу (чтобы заставить сервер немедленно записать на диск все «грязные» данные
		для этого файла), хотя cifs всё равно отправляет
		все «грязные» (кэшированные) данные файла на сервер и ждёт,
		пока сервер ответит на запись. Поскольку SMB Flush может быть
		очень медленным, а некоторые серверы могут быть достаточно надёжными (чтобы рискнуть
		немного задержать сброс данных на диск на сервере),
		включение этого параметра может быть полезно для повышения производительности
		приложений, которые слишком часто вызывают fsync, при небольшом риске сбоя сервера.
		Если этот параметр монтирования не установлен, по умолчанию cifs будет
		отправлять запрос SMB flush (и ждать ответа) при каждом
		вызове fsync.
  nodfs
		Отключить DFS (поддержку глобального пространства имён), даже если
		сервер заявляет, что её поддерживает. Это может помочь обойти
		проблему с разбором путей DFS на серверах Samba
		версий 3.0.24 и 3.0.25.
  remount
		перемонтировать общий ресурс (часто используется для смены с ro на rw монтирование
		или наоборот)
  cifsacl
		Сообщать биты режима (например, при stat) на основе Windows ACL для
		файла. (ЭКСПЕРИМЕНТАЛЬНО)
  servern
		Указать netbios-имя сервера (имя RFC1001) для использования
		при попытке установить сеанс с сервером.
		Это необходимо для монтирования к некоторым более старым серверам (таким
		как OS/2 или Windows 98 и Windows ME), поскольку они не
		поддерживают имя сервера по умолчанию. Имя сервера может быть
		длиной до 15 символов и обычно записывается заглавными буквами.
  sfu
		Когда CIFS Unix Extensions не согласованы, попытаться
		создавать файлы устройств и fifo в формате, совместимом со
		Services for Unix (SFU). Кроме того, извлекать биты 10-12
		режима через расширенный атрибут SETFILEBITS (как
		это делает SFU). В будущем нижние 9 битов
		режима также будут эмулироваться с помощью запросов дескриптора
		безопасности (ACL).
  mfsymlinks
		Включить поддержку символических ссылок Minshall+French
		(см. http://wiki.samba.org/index.php/UNIX_Extensions#Minshall.2BFrench_symlinks)
		Этот параметр игнорируется, если указан вместе с
		параметром 'sfu'. Символические ссылки Minshall+French используются, даже если
		сервер поддерживает CIFS Unix Extensions.
  sign
		Обязательно использовать подпись пакетов (помогает избежать нежелательного изменения данных
		промежуточными системами в маршруте). Обратите внимание, что подпись
		не работает с аутентификацией lanman или открытым текстом.
  seal
		Обязательно запечатывать (шифровать) все данные на этом смонтированном общем ресурсе перед
		отправкой по сети. Требует поддержки Unix Extensions.
		Обратите внимание, что это отличается от параметра монтирования sign тем, что
		вызывает шифрование данных, отправляемых через этот смонтированный общий ресурс, но другие
		общие ресурсы, смонтированные к тому же серверу, не затрагиваются.
  locallease
		Этот параметр требуется редко. Fcntl F_SETLEASE
		используется некоторыми приложениями, такими как Samba и сервер NFSv4, для
		проверки того, является ли файл кэшируемым. У CIFS нет способа
		явно запросить лизинг (lease), но он может проверить, является ли файл
		кэшируемым (oplocked). К сожалению, даже если файл
		не oplocked, он всё равно может быть кэшируемым (то есть cifs-клиент
		мог бы предоставить fcntl-лизинги, если никакие другие локальные процессы не используют
		файл) для случаев, например, таких как когда сервер не
		поддерживает oplocks, а пользователь уверен, что единственные обновления
		файла будут от этого клиента. Указание этого параметра монтирования
		позволит cifs-клиенту проверять лизинги (только) локально
		для файлов, которые не oplocked, вместо отклонения лизингов
		в этом случае. (ЭКСПЕРИМЕНТАЛЬНО)
  sec
		Режим безопасности. Допустимые значения:

			none
				пытаться подключиться как нулевой пользователь (без имени)
			krb5
				Использовать аутентификацию Kerberos версии 5
			krb5i
				Использовать аутентификацию Kerberos и подпись пакетов
			ntlm
				Использовать хеширование паролей NTLM (по умолчанию)
			ntlmi
				Использовать хеширование паролей NTLM с подписью (если
				/proc/fs/cifs/PacketSigningEnabled включено или если
				сервер требует подписи, также может быть значением по умолчанию)
			ntlmv2
				Использовать хеширование паролей NTLMv2
			ntlmv2i
				Использовать хеширование паролей NTLMv2 с подписью пакетов
			lanman
				(если настроено в конфигурации ядра) использовать более старый
				хеш lanman
  hard
		Повторять файловые операции, если сервер не отвечает
  soft
		Ограничивать количество повторов к неотвечающим серверам (обычно только
		одним повтором) перед возвратом ошибки. (по умолчанию)

Прочие флаги /proc/fs/cifs и отладочная информация
==================================================

Информационные псевдофайлы:

======================= =======================================================
DebugData		Отображает информацию об активных сеансах и общих ресурсах
			CIFS, включённых возможностях, а также версию
			cifs.ko.
Stats			Выводит сводную информацию об использовании ресурсов, а также статистику
			по каждому общему ресурсу.
open_files		Перечисляет все открытые файловые дескрипторы во всех активных SMB-сеансах.
mount_params            Список всех параметров монтирования, доступных для модуля
======================= =======================================================

Конфигурационные псевдофайлы:

======================= =======================================================
SecurityFlags		Флаги, управляющие согласованием безопасности, а
			также подписью пакетов. Флаги аутентификации (may/must)
			(например, для NTLMv2) могут комбинироваться с
			флагами подписи. С другой стороны, указание двух разных механизмов
			хеширования паролей (как «must use»)
			не имеет большого смысла. Флаги по умолчанию::

				0x00C5

			(разрешены NTLMv2 и подпись пакетов). Некоторые SecurityFlags
			могут требовать включения соответствующего параметра menuconfig.

			  may use packet signing			0x00001
			  must use packet signing			0x01001
			  may use NTLMv2				0x00004
			  must use NTLMv2				0x04004
			  may use Kerberos security (krb5)		0x00008
			  must use Kerberos                             0x08008
			  may use NTLMSSP               		0x00080
			  must use NTLMSSP           			0x80080
			  seal (packet encryption)			0x00040
			  must seal                                     0x40040

cifsFYI			Если установлено ненулевое значение, дополнительная отладочная информация
			будет записываться в системный журнал ошибок. Это поле
			содержит три флага, управляющих разными классами
			отладочных записей. Максимальное значение, в которое его можно
			установить, равно 7, что включает все точки отладки (по умолчанию 0).
			Некоторые отладочные операторы не компилируются в
			cifs-ядро, если в конфигурации ядра не включён
			CONFIG_CIFS_DEBUG2. cifsFYI может быть установлен в один или
			несколько из следующих флагов (7 устанавливает их все)::

			  +-----------------------------------------------+------+
			  | log cifs informational messages		  | 0x01 |
			  +-----------------------------------------------+------+
			  | log return codes from cifs entry points	  | 0x02 |
			  +-----------------------------------------------+------+
			  | log slow responses				  | 0x04 |
			  | (ie which take longer than 1 second)	  |      |
			  |                                               |      |
			  | CONFIG_CIFS_STATS2 must be enabled in .config |      |
			  +-----------------------------------------------+------+

traceSMB		Если установлено в единицу, отладочная информация записывается в
			системный журнал ошибок с началом smb-запросов
			и ответов (по умолчанию 0)
LookupCacheEnable	Если установлено в единицу, информация об inode кэшируется
			в течение одной секунды, повышая производительность поиска
			(по умолчанию 1)
LinuxExtensionsEnabled	Если установлено в единицу, то клиент попытается
			использовать «UNIX»-расширения CIFS, которые являются необязательными
			улучшениями протокола, позволяющими CIFS-серверам
			возвращать точную информацию UID/GID, а также
			поддерживать символические ссылки. Если вы используете серверы
			вроде Samba, которые поддерживают CIFS Unix
			extensions, но не хотите использовать поддержку символических ссылок
			и хотите сопоставить поля uid и gid
			значениям, предоставленным при монтировании (а не
			фактическим значениям), то установите это в ноль. (по умолчанию 1)
dfscache		Перечисляет содержимое кэша DFS.
			Если установлено в 0, клиент очистит кэш.
======================= =======================================================

Эти экспериментальные возможности и трассировку можно включить, изменив флаги в
/proc/fs/cifs (после того, как cifs-модуль установлен или встроен в
ядро, например insmod cifs). Чтобы включить возможность, установите её в 1, например, чтобы включить
трассировку в журнал сообщений ядра, введите::

	echo 7 > /proc/fs/cifs/cifsFYI

cifsFYI функционирует как битовая маска. Установка его в 1 включает дополнительную
запись в журнал ядра различных информационных сообщений. 2 включает запись ненулевых
кодов возврата SMB, а 4 включает запись запросов, выполнение которых занимает дольше
одной секунды (за исключением запросов блокировки диапазона байтов).
Установка его в 4 требует, чтобы в конфигурации ядра был установлен CONFIG_CIFS_STATS2
(.config). Установка его в семь включает все три. Наконец, трассировку
начала smb-запросов и ответов можно включить с помощью::

	echo 1 > /proc/fs/cifs/traceSMB

Статистика по каждому общему ресурсу (по каждому клиентскому монтированию) доступна в /proc/fs/cifs/Stats.
Дополнительная информация доступна, если в конфигурации ядра (.config) включён CONFIG_CIFS_STATS2.
Возвращаемая статистика включает счётчики, которые
представляют число выполненных и неудачных (то есть с ненулевым кодом возврата от
сервера) запросов SMB3 (или cifs), сгруппированных по типу запроса (read, write, close и т. д.).
Также записывается общее число байтов, прочитанных и записанных на сервер для
этого общего ресурса. Обратите внимание, что из-за эффектов клиентского кэширования это может быть меньше, чем
число байтов, прочитанных и записанных приложением, работающим на клиенте.
Статистику можно сбросить в ноль командой ``echo 0 > /proc/fs/cifs/Stats``, что может быть
полезно при сравнении производительности двух разных сценариев.

Также обратите внимание, что ``cat /proc/fs/cifs/DebugData`` отобразит информацию об
активных сеансах и смонтированных общих ресурсах.

Включение Kerberos (расширенной безопасности) работает, но требует наличия версии 1.2 или более поздней
вспомогательной программы cifs.upcall и её настройки в
файле /etc/request-key.conf. Вспомогательная программа cifs.upcall происходит из проекта Samba
(https://www.samba.org). Поддержка NTLM, NTLMv2 и LANMAN не
требует этой вспомогательной программы. Обратите внимание, что безопасности NTLMv2 (которая не требует
вспомогательной программы cifs.upcall) вместо использования Kerberos достаточно для
некоторых сценариев использования.

Поддержка DFS позволяет прозрачно перенаправлять на общие ресурсы в пространстве имён MS-DFS.
Кроме того, поддержка DFS для целевых общих ресурсов, заданных как UNC-имена,
начинающиеся с имён хостов (а не IP-адресов), требует наличия
вспомогательной программы из пространства пользователя (такой как cifs.upcall) для
преобразования имён хостов в IP-адрес, и эта вспомогательная программа из пространства пользователя также должна
быть настроена в файле /etc/request-key.conf. Samba, серверы Windows и
многие NAS-устройства поддерживают DFS как способ построения глобального пространства
имён для облегчения настройки сети и повышения надёжности.

Чтобы использовать поддержку Kerberos и DFS в cifs, должен быть установлен пакет keyutils для Linux,
и в файл /etc/request-key.conf должны быть добавлены строки вроде
следующих::

  create cifs.spnego * * /usr/local/sbin/cifs.upcall %k
  create dns_resolver * * /usr/local/sbin/cifs.upcall %k

Параметры cifs-модуля ядра
==========================
Эти параметры модуля можно указать или изменить либо во время
загрузки модуля, либо во время выполнения с помощью интерфейса::

	/sys/module/cifs/parameters/<param>

то есть::

    echo "value" > /sys/module/cifs/parameters/<param>

Более подробные описания доступных параметров модуля и их значений
можно увидеть, выполнив:

    modinfo cifs (или modinfo smb3)

================= ==========================================================
1. enable_oplocks Включить или отключить oplocks. Oplocks включены по умолчанию.
		  [Y/y/1]. Для отключения используйте любое из [N/n/0].
