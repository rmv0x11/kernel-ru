Поддержка ядром разнообразных бинарных форматов (binfmt_misc)
=============================================================

Эта возможность ядра позволяет вам запускать почти (об ограничениях см. ниже)
любую программу, просто набрав её имя в командной оболочке.
Сюда входят, например, скомпилированные программы Java(TM), Python или Emacs.

Чтобы добиться этого, вы должны сообщить binfmt_misc, какой интерпретатор должен
запускаться для какого бинарного файла. Binfmt_misc распознаёт тип бинарного файла,
сопоставляя некоторые байты в начале файла с предоставленной вами магической
последовательностью байтов (с маскированием указанных битов). Binfmt_misc также
может распознавать расширение имени файла, например ``.com`` или ``.exe``.

Сначала вы должны смонтировать binfmt_misc::

	mount binfmt_misc -t binfmt_misc /proc/sys/fs/binfmt_misc

Чтобы фактически зарегистрировать новый бинарный тип, вы должны составить строку вида
``:name:type:offset:magic:mask:interpreter:flags`` (где вы можете выбирать символ
``:`` по своему усмотрению) и направить её через echo в ``/proc/sys/fs/binfmt_misc/register``.

Вот что означают поля:

- ``name``
   это идентификационная строка. С этим именем будет создан новый файл /proc
   ниже ``/proc/sys/fs/binfmt_misc``; не может содержать слешей ``/`` по
   очевидным причинам.
- ``type``
   это тип распознавания. Укажите ``M`` для магической последовательности и ``E`` для расширения.
- ``offset``
   это смещение магической последовательности/маски в файле, отсчитываемое в байтах. По
   умолчанию оно равно 0, если вы его опускаете (то есть пишете ``:name:type::magic...``).
   Игнорируется при сопоставлении по расширению имени файла.
- ``magic``
   это последовательность байтов, которую сопоставляет binfmt_misc. Магическая строка
   может содержать символы в шестнадцатеричном виде, например ``\x0a`` или ``\xA4``. Учтите, что вы
   должны экранировать любые байты NUL; разбор останавливается на первом из них. В среде
   командной оболочки вам, возможно, придётся писать ``\\x0a``, чтобы оболочка не
   съела ваш ``\``.
   Если вы выбрали сопоставление по расширению имени файла, это расширение, которое нужно
   распознать (без ``.``, специальные последовательности ``\x0a`` не допускаются).
   Сопоставление    по расширению учитывает регистр, и слеши ``/`` не допускаются!
- ``mask``
   это (необязательная, по умолчанию все 0xff) маска. Вы можете замаскировать некоторые
   биты при сопоставлении, предоставив строку, подобную magic и той же длины, что и magic.
   Маска применяется по операции И к последовательности байтов файла. Учтите, что вы должны
   экранировать любые байты NUL; разбор останавливается на первом из них. Игнорируется при
   сопоставлении по расширению имени файла.
- ``interpreter``
   это программа, которая должна быть вызвана с бинарным файлом в качестве первого
   аргумента (укажите полный путь)
- ``flags``
   это необязательное поле, которое управляет несколькими аспектами вызова
   интерпретатора. Это строка из заглавных букв, каждая из которых управляет
   определённым аспектом. Поддерживаются следующие флаги:

      ``P`` - preserve-argv[0]
            Унаследованное поведение binfmt_misc состоит в том, чтобы перезаписать
            исходный argv[0] полным путём к бинарному файлу. Когда этот
            флаг включён, binfmt_misc добавит для этой цели аргумент в вектор
            аргументов, тем самым сохраняя исходный ``argv[0]``.
            Например, если ваш interp задан как ``/bin/foo`` и вы запускаете ``blah``
            (который находится в ``/usr/local/bin``), то ядро выполнит
            ``/bin/foo`` с ``argv[]``, заданным как ``["/bin/foo", "/usr/local/bin/blah", "blah"]``.  Interp должен знать об этом, чтобы он мог
            выполнить ``/usr/local/bin/blah``
            с ``argv[]``, заданным как ``["blah"]``.
      ``O`` - open-binary
	    Унаследованное поведение binfmt_misc состоит в том, чтобы передавать полный путь
            к бинарному файлу интерпретатору в качестве аргумента. Когда этот флаг
            включён, binfmt_misc откроет файл на чтение и передаст его
            дескриптор в качестве аргумента вместо полного пути, тем самым позволяя
            интерпретатору выполнять нечитаемые бинарные файлы. Эту возможность
            следует использовать с осторожностью — интерпретатор должен быть доверенным, чтобы он не
            выдавал содержимое нечитаемого бинарного файла.
      ``C`` - credentials
            В настоящее время поведение binfmt_misc состоит в том, чтобы вычислять
            учётные данные и токен безопасности нового процесса в соответствии с
            интерпретатором. Когда этот флаг включён, эти атрибуты
            вычисляются в соответствии с бинарным файлом. Он также подразумевает флаг ``O``.
            Эту возможность следует использовать с осторожностью, поскольку интерпретатор
            будет работать с правами root, когда бинарный файл setuid, принадлежащий root,
            запускается через binfmt_misc.
      ``F`` - fix binary
            Обычное поведение binfmt_misc состоит в том, чтобы порождать
	    бинарный файл лениво, когда вызывается файл формата misc.  Однако
	    это не очень хорошо работает в условиях пространств имён монтирования и
	    смены корня (changeroot), поэтому режим ``F`` открывает бинарный файл сразу же, как только
	    устанавливается эмуляция, и использует открытый образ для порождения
	    эмулятора, что означает, что он всегда доступен после установки,
	    независимо от того, как меняется окружение.


Есть некоторые ограничения:

 - вся строка регистрации не может превышать 1920 символов
 - magic должна находиться в первых 128 байтах файла, то есть
   offset+size(magic) должно быть меньше 128
 - строка интерпретатора не может превышать 127 символов

Чтобы использовать binfmt_misc, вы должны сначала смонтировать его. Вы можете смонтировать его
командой ``mount -t binfmt_misc none /proc/sys/fs/binfmt_misc`` или можете добавить
строку ``none  /proc/sys/fs/binfmt_misc binfmt_misc defaults 0 0`` в ваш
``/etc/fstab``, чтобы он монтировался автоматически при загрузке.

Возможно, вы захотите добавлять бинарные форматы в одном из ваших скриптов ``/etc/rc`` во время
загрузки. Прочитайте руководство к вашей программе init, чтобы выяснить, как сделать это
правильно.

Подумайте о порядке добавления записей! Записи, добавленные позже, сопоставляются первыми!


Несколько примеров (предполагается, что вы находитесь в ``/proc/sys/fs/binfmt_misc``):

- включить поддержку em86 (как binfmt_em86, только для Alpha AXP)::

    echo ':i386:M::\x7fELF\x01\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x02\x00\x03:\xff\xff\xff\xff\xff\xfe\xfe\xff\xff\xff\xff\xff\xff\xff\xff\xff\xfb\xff\xff:/bin/em86:' > register
    echo ':i486:M::\x7fELF\x01\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x00\x02\x00\x06:\xff\xff\xff\xff\xff\xfe\xfe\xff\xff\xff\xff\xff\xff\xff\xff\xff\xfb\xff\xff:/bin/em86:' > register

- включить поддержку упакованных приложений DOS (предварительно настроенные hdimages dosemu)::

    echo ':DEXE:M::\x0eDEX::/usr/bin/dosexec:' > register

- включить поддержку исполняемых файлов Windows с помощью wine::

    echo ':DOSWin:M::MZ::/usr/local/bin/wine:' > register

О поддержке java см. Documentation/admin-guide/java.rst


Вы можете включать/выключать binfmt_misc или отдельный бинарный тип, направляя через echo 0 (чтобы выключить)
или 1 (чтобы включить) в ``/proc/sys/fs/binfmt_misc/status`` или
``/proc/.../the_name``.
Вывод файла командой cat сообщает вам текущий статус ``binfmt_misc/the_entry``.

Вы можете удалить одну запись или все записи, направив через echo -1 в ``/proc/.../the_name``
или ``/proc/sys/fs/binfmt_misc/status``.


Подсказки
---------

Если вы хотите передать особые аргументы вашему интерпретатору, вы можете
написать для него скрипт-обёртку.
Пример см. в :doc:`Documentation/admin-guide/java.rst <./java>`.

Ваш интерпретатор НЕ должен искать имя файла в PATH; ядро
передаёт ему полное имя файла (или файловый дескриптор) для использования.  Использование ``$PATH`` может
вызвать неожиданное поведение и может представлять угрозу безопасности.
