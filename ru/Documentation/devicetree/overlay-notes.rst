==================================
Заметки об оверлеях Devicetree
==================================

Этот документ описывает реализацию внутриядерной функциональности оверлеев
дерева устройств (device tree overlay), которая находится в drivers/of/overlay.c,
и является сопроводительным документом к Documentation/devicetree/dynamic-resolution-notes.rst[1]

Как работают оверлеи
--------------------

Назначение оверлея Devicetree — модифицировать действующее (live) дерево ядра
так, чтобы эта модификация влияла на состояние ядра в соответствии с внесёнными
изменениями.
Поскольку ядро в основном имеет дело с устройствами, любой новый узел устройства,
порождающий активное устройство, должен приводить к созданию этого устройства;
а если узел устройства отключён или удалён целиком, то затронутое устройство
должно быть дерегистрировано.

Рассмотрим пример, где у нас есть плата foo со следующим базовым деревом::

    ---- foo.dts ---------------------------------------------------------------
	/* FOO platform */
	/dts-v1/;
	/ {
		compatible = "corp,foo";

		/* shared resources */
		res: res {
		};

		/* On chip peripherals */
		ocp: ocp {
			/* peripherals that are always instantiated */
			peripheral1 { ... };
		};
	};
    ---- foo.dts ---------------------------------------------------------------

Оверлей bar.dtso,
::

    ---- bar.dtso - overlay target location by label ---------------------------
	/dts-v1/;
	/plugin/;
	&ocp {
		/* bar peripheral */
		bar {
			compatible = "corp,bar";
			... /* various properties and child nodes */
		};
	};
    ---- bar.dtso --------------------------------------------------------------

будучи загружен (и разрешён, как описано в [1]), должен дать foo+bar.dts::

    ---- foo+bar.dts -----------------------------------------------------------
	/* FOO platform + bar peripheral */
	/ {
		compatible = "corp,foo";

		/* shared resources */
		res: res {
		};

		/* On chip peripherals */
		ocp: ocp {
			/* peripherals that are always instantiated */
			peripheral1 { ... };

			/* bar peripheral */
			bar {
				compatible = "corp,bar";
				... /* various properties and child nodes */
			};
		};
	};
    ---- foo+bar.dts -----------------------------------------------------------

В результате применения оверлея был создан новый узел устройства (bar), поэтому
будет зарегистрировано платформенное устройство bar, и, если загружен подходящий
драйвер устройства, устройство будет создано ожидаемым образом.

Если базовое DT не было скомпилировано с опцией -@, то метка "&ocp" будет
недоступна для разрешения узла(ов) оверлея в нужное место базового DT. В этом
случае можно указать целевой путь. Синтаксис задания целевого расположения по
метке предпочтительнее, поскольку оверлей можно применить к любому базовому DT,
содержащему данную метку, независимо от того, где в DT эта метка встречается.

Приведённый выше пример bar.dtso, изменённый для использования синтаксиса
целевого пути::

    ---- bar.dtso - overlay target location by explicit path -------------------
	/dts-v1/;
	/plugin/;
	&{/ocp} {
		/* bar peripheral */
		bar {
			compatible = "corp,bar";
			... /* various properties and child nodes */
		}
	};
    ---- bar.dtso --------------------------------------------------------------


Внутриядерный API оверлеев
--------------------------

API весьма прост в использовании.

1) Вызовите of_overlay_fdt_apply() для создания и применения набора изменений
   (changeset) оверлея. Возвращаемое значение — это либо ошибка, либо cookie,
   идентифицирующий данный оверлей.

2) Вызовите of_overlay_remove() для удаления и очистки набора изменений оверлея,
   ранее созданного вызовом of_overlay_fdt_apply(). Удаление набора изменений
   оверлея, поверх которого наложен (stacked) другой, не будет разрешено.

Наконец, если вам нужно удалить все оверлеи разом, просто вызовите
of_overlay_remove_all(), который удалит каждый из них в правильном порядке.

Имеется возможность зарегистрировать notifier'ы, которые вызываются при операциях
с оверлеями. Подробности см. в of_overlay_notifier_register/unregister и
enum of_overlay_notify_action.

Callback notifier'а для OF_OVERLAY_PRE_APPLY, OF_OVERLAY_POST_APPLY или
OF_OVERLAY_PRE_REMOVE может сохранять указатели на узел дерева устройств в
оверлее или его содержимом, но эти указатели не должны сохраняться после
callback'а notifier'а для OF_OVERLAY_POST_REMOVE. Память, содержащая оверлей,
будет освобождена через kfree() после вызова notifier'ов OF_OVERLAY_POST_REMOVE.
Обратите внимание, что память будет освобождена через kfree(), даже если notifier
для OF_OVERLAY_POST_REMOVE вернёт ошибку.

Notifier'ы наборов изменений (changeset) в drivers/of/dynamic.c — это второй тип
notifier'ов, который может срабатывать при применении или удалении оверлея. Этим
notifier'ам не разрешается сохранять указатели на узел дерева устройств в оверлее
или его содержимом. Код оверлеев не защищает от ситуации, когда такие указатели
остаются активными в момент освобождения памяти, содержащей оверлей, в результате
удаления оверлея.

Любой другой код, удерживающий указатель на узлы или данные оверлея, считается
содержащим ошибку, поскольку после удаления оверлея этот указатель будет
ссылаться на освобождённую память.

Пользователи оверлеев должны особенно внимательно относиться к общему набору
операций, происходящих в системе, чтобы гарантировать, что другой код ядра не
удерживает никаких указателей на узлы или данные оверлея. Примером
непреднамеренного использования таких указателей является случай, когда драйвер
или модуль подсистемы загружается после применения оверлея, и этот драйвер или
подсистема сканирует всё дерево устройств или его значительную часть, включая
узлы оверлея.
