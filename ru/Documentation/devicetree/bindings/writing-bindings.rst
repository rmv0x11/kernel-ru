========================================================================================
Правила (что делать и чего не делать) при проектировании и написании привязок Devicetree
========================================================================================

Это список типовых замечаний при ревью, посвящённых проектированию привязок. У
каждого правила есть исключения, и в привязках много «серых зон».

Рекомендации, касающиеся патчей, см. в
Documentation/devicetree/bindings/submitting-patches.rst


Общее проектирование
====================

- ДЕЛАЙТЕ привязки полными, даже если драйвер не поддерживает некоторые
  возможности. Например, если у устройства есть прерывание, включайте свойство
  'interrupts', даже если драйвер работает только в режиме опроса.

- НЕ ссылайтесь на Linux или «драйвер устройства» в привязках. Привязки должны
  опираться на то, что есть в аппаратуре, а не на то, что в данный момент
  поддерживают ОС и драйвер.

- ДЕЛАЙТЕ имена узлов соответствующими классу устройства. Многие стандартные
  имена определены в DT Spec. Если подходящего имени нет, подумайте о его
  добавлении.

- ДЕЛАЙТЕ проверку того, что пример соответствует документации, особенно после
  внесения правок по итогам ревью.

- НЕ создавайте узлы только ради создания экземпляров драйверов. Многофункциональным
  устройствам нужны дочерние узлы только тогда, когда у дочерних узлов есть свои
  собственные DT-ресурсы. Один узел может быть несколькими провайдерами (например,
  тактовых сигналов (clocks) и сбросов (resets)).

- НЕ рассматривайте имена узлов устройств как стабильный ABI; вместо этого
  используйте phandle или compatible для поиска соседних устройств. Исключение:
  подузлы данного устройства могут рассматриваться как ABI, если это явно
  задокументировано в привязках.

- НЕ используйте 'syscon' в одиночку без конкретной строки compatible. Аппаратный
  блок 'syscon' должен иметь строку compatible, достаточно уникальную, чтобы
  вывести (как минимум) разметку регистров всего блока.

- НЕ используйте compatible 'simple-mfd' для нетривиальных устройств, у которых
  дочерние элементы зависят от некоторых ресурсов родителя. Аналогично, 'simple-bus'
  не следует использовать для сложных шин, и даже наличие свойства 'regs' означает,
  что устройство не является простой шиной.


Свойства
========

- ДЕЛАЙТЕ свойства 'compatible' конкретными.

   - НЕ используйте подстановочные символы или имена семейств устройств в строках
     compatible.

   - ДЕЛАЙТЕ резервные (fallback) compatible, когда устройства совпадают с
     предыдущими реализациями или являются их надмножеством.

   - ДЕЛАЙТЕ добавление новых compatible в случае появления новых возможностей или
     ошибок.

   - ДЕЛАЙТЕ использование специфичного для SoC compatible для всех устройств SoC,
     за которым при необходимости следует резервный. Специфичные для SoC compatible
     также предпочтительны в качестве резервных.

   - НЕ используйте суффиксы шин для кодирования типа интерфейса, который использует
     устройство. Родительский узел шины уже подразумевает этот интерфейс.  НЕ
     добавляйте тип устройства, если устройство не может быть ничем иным.

- ДЕЛАЙТЕ использование префикса производителя в именах свойств, специфичных для
  устройства. Подумайте, не могут ли свойства быть общими для устройств одного
  класса. Проверьте другие существующие привязки для похожих устройств.

- НЕ переопределяйте общие свойства. Просто сошлитесь на определение и задайте
  ограничения, специфичные для устройства.

- НЕ добавляйте свойства, чтобы избежать конкретного compatible. НЕ добавляйте
  свойства, если они подразумеваются (выводятся) из compatible.

- ДЕЛАЙТЕ использование общих суффиксов единиц измерения для свойств с научными
  единицами. Рекомендуемые суффиксы перечислены по адресу
  https://github.com/devicetree-org/dt-schema/blob/main/dtschema/schemas/property-units.yaml

- ДЕЛАЙТЕ определение свойств через ограничения. Сколько элементов? Какие
  возможные значения? Каков порядок? Все эти ограничения также представляют собой
  ABI.

- НЕ вносите изменения, нарушающие ABI, без явного и подробного обоснования того,
  почему эти изменения должны быть сделаны, и их последствий. Влияние на ABI выходит
  за пределы ядра Linux, поскольку оно охватывает также другие апстрим-проекты с
  открытым исходным кодом.


Типичные случаи и оговорки
==========================

- Элементы phandle, такие как clocks/dmas/interrupts/resets, всегда должны быть
  явно упорядочены. Включайте {clock,dma,interrupt,reset}-names, если phandle
  больше одного. При использовании оба этих поля нуждаются в одинаковых
  ограничениях (например, список элементов).

- Для имён, используемых в {clock,dma,interrupt,reset}-names, не добавляйте никаких
  суффиксов, например: "tx" вместо "txirq" (для прерывания).

- Свойства без типов схемы (например, без стандартного суффикса или не определённые
  схемой) нуждаются в типе, даже если это enum.

- Если схема включает другую схему (например, /schemas/i2c/i2c-controller.yaml),
  используйте "unevaluatedProperties:false". В остальных случаях обычно используйте
  "additionalProperties:false".

- Для подблоков/компонентов более крупного устройства (например, блоков SoC)
  используйте лучше compatible на основе устройства (например, compatible на основе
  SoC), а не собственное версионирование этого компонента.
  Например, используйте "vendor,soc1234-i2c" вместо "vendor,i2c-v2".

- "syscon" не является обобщённым свойством. Используйте производителя и тип,
  например "vendor,power-manager-syscon".

- Не добавляйте свойства индекса экземпляра (ID) или собственные псевдонимы OF.
  Если устройства имеют разную модель программирования, им могут понадобиться разные
  compatible.  Если такие устройства используют какое-то другое устройство иначе,
  например по-разному программируют phy, используйте аргументы cell/phandle.

- Файлы привязок должны именоваться как compatible: vendor,device.yaml. В случае
  нескольких compatible в привязке используйте один из резервных или более общее
  имя, при этом всё равно соответствующее стилю compatible.

Файлы .dts для плат/SoC
=======================

- ДЕЛАЙТЕ размещение всех устройств MMIO под узлом шины, а не на верхнем уровне.

- ДЕЛАЙТЕ использование непустого 'ranges' для ограничения размера дочерних
  шин/устройств. 64-битным платформам не требуется, чтобы все устройства имели
  64-битные адрес и размер.
