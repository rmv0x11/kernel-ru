Написание привязок devicetree на json-schema
=============================================

Привязки devicetree пишутся с использованием словаря json-schema. Файлы схем
пишутся на JSON-совместимом подмножестве YAML. YAML используется вместо JSON,
поскольку он считается более удобным для чтения человеком и имеет ряд
преимуществ, например, позволяет добавлять комментарии (с префиксом '#').

Смотрите также :ref:`example-schema`.

Содержимое схемы
----------------

Каждый документ схемы — это структурированная json-schema, которая определяется
набором свойств верхнего уровня. Как правило, в одном файле определяется одна
привязка. Используются следующие свойства json-schema верхнего уровня:

$id
  Строка-уникальный идентификатор json-schema. Строка должна быть корректным
  URI, обычно содержащим имя файла привязки и путь к нему. Для DT-схемы он должен
  начинаться с "http://devicetree.org/schemas/". Этот URL используется при
  построении ссылок на другие файлы, указанные в свойствах "$ref" схемы. К
  значению $ref, начинающемуся с '/', будет добавлено имя хоста. К значению $ref,
  содержащему только относительный путь или имя файла, будут добавлены
  компоненты имени хоста и пути из значения '$id' текущего файла схемы. URL
  используется даже для локальных файлов, но фактически файлов по этим адресам
  может и не быть.

$schema
  Указывает мета-схему, которой соответствует файл схемы.

title
  Однострочное описание аппаратного обеспечения, описываемого в схеме привязки.

maintainers
  DT-специфичное свойство. Содержит список email-адресов сопровождающих
  (maintainers) данной привязки.

description
  Необязательное. Многострочный текстовый блок, содержащий любую подробную
  информацию об этом аппаратном обеспечении. Он должен содержать такие сведения,
  как что делает данный блок или устройство, каким стандартам соответствует
  устройство, а также ссылки на даташиты для получения дополнительной информации.

  Формат YAML предоставляет несколько вариантов задания форматирования
  текстового блока. Эти варианты управляются символами-индикаторами, следующими
  за ключом (например, "description: \|"). Следует использовать минимальное
  форматирование, необходимое для блока. Управляющие символы форматирования могут
  влиять не только на то, удастся ли корректно разобрать YAML, но и важны при
  отображении текстовых блоков в другую форму. Варианты следующие.

  По умолчанию, без каких-либо индикаторов, используется flowed-стиль, plain
  scalar, в котором одиночные разрывы строк и начальные пробелы удаляются. Абзацы
  разделяются пустыми строками (то есть двойным разрывом строки). Этот стиль не
  может содержать ": ", так как это будет интерпретировано как ключ. Любая
  последовательность " #" будет интерпретирована как комментарий. Существуют и
  другие ограничения на символы. Большинство ограничений касается того, каким
  может быть первый символ.

  Второй стиль — folded, который обозначается символом ">". Помимо сохранения
  разрывов строк при двойных разрывах, folded-стиль также сохраняет начальные
  пробелы, выходящие за пределы отступа первой строки. Разрывы строк на строках с
  отступом также сохраняются.

  Третий стиль — literal, который обозначается символом "\|". Literal-стиль
  сохраняет все разрывы строк и пробелы (за пределами отступа первой строки).

  Сказанное выше не является полным описанием текстовых блоков YAML. Более
  подробную информацию о многострочных текстовых блоках YAML можно найти в сети:

  https://yaml-multiline.info/

  https://www.yaml.info/learn/quote.html

select
  Необязательное. json-schema, используемая для сопоставления узлов с целью
  применения схемы. По умолчанию, без 'select', узлы сопоставляются по их
  возможным значениям compatible-строки или по имени узла. Большинству привязок
  select не нужен.

allOf
  Необязательное. Список других схем для включения. Используется для включения
  других схем, которым соответствует привязка. Это могут быть схемы для
  определённого класса устройств, например контроллеров I2C или SPI.

properties
  Набор под-схем, определяющих все DT-свойства привязки. Точный синтаксис схемы
  зависит от того, являются ли свойства известными, общими (например,
  'interrupts') или это свойства, специфичные для привязки/вендора.

Свойство также может определять дочерний DT-узел с определёнными под ним
дочерними свойствами.

Подробнее о разделах properties см. раздел 'Property Schema'.

patternProperties
  Необязательное. Аналогично 'properties', но имена задаются регулярными
  выражениями.

required
  Список DT-свойств из раздела 'properties', которые должны всегда присутствовать.

additionalProperties / unevaluatedProperties
  Ключевые слова, управляющие тем, как схема будет валидировать свойства, не
  сопоставленные разделами 'properties' или 'patternProperties' данной схемы.
  Предполагается, что каждая схема имеет ровно одно из этих ключевых слов в части
  верхнего уровня, то есть либо additionalProperties, либо
  unevaluatedProperties. Вложенные узлы, то есть свойства, являющиеся объектами,
  также должны иметь одно из них.

  * additionalProperties: false
      Наиболее распространённый случай, когда не ссылаются ни на одну
      дополнительную схему или когда данная привязка разрешает подмножество
      свойств из других схем, на которые ссылаются.

  * unevaluatedProperties: false
      Используется, когда данная привязка ссылается на другую схему, все свойства
      которой должны быть разрешены.

  * additionalProperties: true
      - Часть верхнего уровня:
        Редкий случай, используется для схем, реализующих общий набор свойств. На
        такие схемы должны ссылаться другие схемы, которые затем используют
        'unevaluatedProperties: false'. Обычно это схемы шин или общих частей.
      - Вложенный узел:
        Когда указывается только ожидаемый compatible вложенного узла и
        существует другая схема, сопоставляющаяся с этим compatible, которая
        заканчивается одним из двух приведённых выше случаев ('false').

examples
  Необязательное. Список из одного или более фрагментов DTS, реализующих только
  данную привязку. Пример не должен содержать несвязанных узлов устройств,
  например узлов-потребителей в привязке поставщика, других узлов, на которые
  ссылаются по phandle.
  Примечание: YAML не допускает начальных табуляций, поэтому вместо них следует
  использовать пробелы.

Если не указано иное, все свойства являются обязательными.

Property Schema
---------------

Раздел 'properties' схемы содержит все DT-свойства привязки. Каждое свойство
содержит набор ограничений для этого свойства, заданных с использованием словаря
json-schema. Именно схемы свойств используются для валидации DT-файлов.

Для общих свойств необходимо определять только дополнительные ограничения, не
покрытые общей схемой привязки, например сколько значений допустимо или какие
значения возможны.

Для специфичных для вендора свойств обычно требуется более детальная схема. За
исключением булевых свойств, они должны содержать ссылку на тип в
schemas/types.yaml. Свойство "description" требуется всегда.

Схемы Devicetree не точно соответствуют YAML-кодированным DT-данным,
производимым dtc. Они упрощены, чтобы сделать их более компактными и избежать
множества шаблонного кода. Инструменты обрабатывают файлы схем, чтобы получить
окончательную схему для валидации. В настоящее время инструменты выполняют 2
преобразования.

По умолчанию массивы в json-schema имеют переменный размер и допускают больше
элементов, чем явно определено. Это можно ограничить, определив 'minItems',
'maxItems' и 'additionalItems'. Однако для схем DeviceTree в большинстве случаев
желателен фиксированный размер, поэтому эти свойства добавляются на основе
количества элементов в списке 'items'.

Формат YAML Devicetree также делает все строковые значения массивом, а скалярные
значения — матрицей (для того чтобы определять группировки), даже когда
присутствует только одно значение. Одиночные записи в схемах приводятся в
соответствие этому кодированию.

Когда привязки охватывают несколько схожих устройств, различающихся некоторыми
свойствами, эти свойства должны быть ограничены для каждого устройства. Обычно
это означает следующее:

 * В 'properties' верхнего уровня определить свойство с самыми широкими
   ограничениями.
 * В блоках 'if:then:' дополнительно сузить ограничения для этих свойств.
 * Не определять свойства внутри блока 'if:then:' (обратите внимание, что
   'additionalItems' также этого не допустит).

Coding style
------------

Используйте стиль кодирования YAML (отступ в два пробела). Для примеров DTS в
схеме предпочтителен отступ в четыре пробела.

Размещайте записи в разделах 'properties' и 'required' в одном и том же порядке,
используя стиль из Documentation/devicetree/bindings/dts-coding-style.rst.

Testing
-------

Dependencies
~~~~~~~~~~~~

Для валидации документов привязок DT-схемы и валидации DTS-файлов с помощью
DT-схемы должен быть установлен проект DT schema. Проект DT schema можно
установить с помощью pip::

    pip3 install dtschema

Обратите внимание, что установка 'dtschema' требует, чтобы сначала были
установлены 'swig' и файлы для разработки Python. В системах Debian/Ubuntu::

    apt install swig python3-dev

Будут установлены несколько исполняемых файлов (dt-doc-validate, dt-mk-schema,
dt-validate). Убедитесь, что они находятся в вашем PATH (по умолчанию
~/.local/bin).

Также рекомендуется установить yamllint (используется dtschema при его наличии).

Running checks
~~~~~~~~~~~~~~

Документы привязок DT-схемы должны быть провалидированы с помощью мета-схемы
(схемы для схемы), чтобы убедиться, что они являются как корректной json-schema,
так и корректной схемой привязки. Все документы привязок DT можно
провалидировать с помощью цели ``dt_binding_check``::

    make dt_binding_check

Или для валидации одной схемы и её примера::

    make sram/sram.yaml

Чтобы выполнить валидацию исходных DT-файлов, используйте цель ``dtbs_check``::

    make dtbs_check

Обратите внимание, что ``dtbs_check`` пропустит любые файлы схем привязок с
ошибками. Чтобы получить все ошибки валидации в файлах схем привязок, необходимо
использовать ``dt_binding_check``.

Можно запустить обе проверки одной командой::

    make dt_binding_check dtbs_check

Также можно сочетать запуск приведённых выше команд с подмножеством
соответствующих файлов схем, установив переменную ``DT_SCHEMA_FILES`` равной
одному или более конкретным файлам схем или шаблонам (частичное совпадение
фиксированной строки). Каждый файл или шаблон следует разделять символом ':'.

::

    make dt_binding_check DT_SCHEMA_FILES=trivial-devices.yaml
    make dt_binding_check DT_SCHEMA_FILES=trivial-devices.yaml:rtc.yaml
    make dt_binding_check DT_SCHEMA_FILES=/gpio/
    make dtbs_check DT_SCHEMA_FILES=trivial-devices.yaml
