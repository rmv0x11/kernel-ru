==============================================================
Стиль оформления исходных файлов Devicetree (DTS)
==============================================================

При написании исходных файлов Devicetree (DTS) соблюдайте приведённые ниже
рекомендации. Их следует рассматривать как дополнение к любым правилам, уже
изложенным в Devicetree Specification и реализованным в компиляторе dtc
(включая сборки с W=1 и W=2).

Отдельные архитектуры и субархитектуры могут определять дополнительные правила,
делая стиль оформления более строгим.

Именование и допустимые символы
--------------------------------

Devicetree Specification допускает широкий диапазон символов в именах узлов и
свойств, однако этот стиль оформления сужает диапазон ради лучшей читаемости
кода.

1. В именах узлов и свойств можно использовать только следующие символы:

   * Строчные буквы: [a-z]
   * Цифры: [0-9]
   * Дефис: -

2. В метках можно использовать только следующие символы:

   * Строчные буквы: [a-z]
   * Цифры: [0-9]
   * Подчёркивание: _

3. Если шина не определяет иное, адреса узлов (unit addresses) должны
   использовать строчные шестнадцатеричные цифры без ведущих нулей (без
   дополнения).

4. Шестнадцатеричные значения в свойствах, например "reg", должны использовать
   строчный шестнадцатеричный формат. Адресная часть может дополняться ведущими
   нулями.

Пример::

	gpi_dma2: dma-controller@a00000 {
		compatible = "qcom,sm8550-gpi-dma", "qcom,sm6350-gpi-dma";
		reg = <0x0 0x00a00000 0x0 0x60000>;
	}

Порядок узлов
-------------

1. Узлы на любой шине, то есть использующие адреса узлов для потомков, должны
   быть упорядочены по адресу узла в порядке возрастания.
   В качестве альтернативы для некоторых субархитектур узлы одного типа могут
   группироваться вместе, например все контроллеры I2C один за другим, даже
   если это нарушает упорядочивание по адресу узла.

2. Узлы без адресов узлов должны быть упорядочены буквенно-цифровым образом по
   имени узла. Для нескольких типов узлов их можно упорядочить по основному
   свойству, например состояния конфигурации выводов упорядочиваются по значению
   свойства "pins".

3. При расширении узлов в DTS платы через &label записи должны быть упорядочены
   либо буквенно-цифровым образом, либо с сохранением порядка из DTSI, причём
   выбор зависит от субархитектуры.

Описанные выше правила упорядочивания легко контролировать во время рецензии,
они уменьшают вероятность конфликтов при одновременном добавлении новых узлов в
файл и помогают ориентироваться в исходном коде DTS.

Пример::

	/* SoC DTSI */

	/ {
		cpus {
			/* ... */
		};

		psci {
			/* ... */
		};

		soc@0 {
			dma: dma-controller@10000 {
				/* ... */
			};

			clk: clock-controller@80000 {
				/* ... */
			};
		};
	};

	/* Board DTS - alphabetical order */

	&clk {
		/* ... */
	};

	&dma {
		/* ... */
	};

	/* Board DTS - alternative order, keep as DTSI */

	&dma {
		/* ... */
	};

	&clk {
		/* ... */
	};

Порядок свойств в узле устройства
---------------------------------

Предпочтителен следующий порядок свойств в узлах устройств:

1. "compatible"
2. "reg"
3. "ranges"
4. Стандартные/общие свойства (определённые общими привязками, например без
   префиксов производителя)
5. Свойства, специфичные для производителя
6. "status" (если применимо), которому предшествует пустая строка, если перед
   этим свойством есть содержимое
7. Дочерние узлы, каждому из которых предшествует пустая строка

Свойство "status" по умолчанию равно "okay", поэтому его можно опускать.

Описанное выше упорядочивание следует такому подходу:

1. Узел начинается с наиболее важных свойств: compatible, затем адресация на
   шине, чтобы соответствовать адресу узла.
2. У каждого узла общие свойства будут располагаться в схожем месте.
3. Status — это последняя информация, отмечающая, завершён узел устройства или
   нет (требуются ресурсы платы).

Отдельные свойства внутри каждой группы должны использовать естественный порядок
сортировки по имени свойства.

Пример::

	/* SoC DTSI */

	device_node: device-class@6789abc {
		compatible = "vendor,device";
		reg = <0x0 0x06789abc 0x0 0xa123>;
		ranges = <0x0 0x0 0x06789abc 0x1000>;
		#dma-cells = <1>;
		clocks = <&clock_controller 0>, <&clock_controller 1>;
		clock-names = "bus", "host";
		#address-cells = <1>;
		#size-cells = <1>;
		vendor,custom-property = <2>;

		status = "disabled";

		child_node: child-class@100 {
			reg = <0x100 0x200>;
			/* ... */
		};
	};

	/* Board DTS */

	&device_node {
		vdd-0v9-supply = <&board_vreg1>;
		vdd-1v8-supply = <&board_vreg4>;
		vdd-3v3-supply = <&board_vreg2>;
		vdd-12v-supply = <&board_vreg3>;

		status = "okay";
	}

Отступы и перенос строк
-----------------------

1. Используйте отступы и переносите строки в соответствии с
   Documentation/process/coding-style.rst.
2. Каждая запись в массивах с несколькими ячейками, например "reg" с двумя
   адресами ввода-вывода, должна заключаться в <>.
3. Для массивов, занимающих несколько строк, предпочтительно разбивать по
   границе элемента и выравнивать продолжаемые записи по открывающему < из
   первой строки.
   Как правило, избегайте разбиения отдельных элементов, если только они
   значительно не превышают предельную длину строки при переносе.

Пример::

	thermal-sensor@c271000 {
		compatible = "qcom,sm8550-tsens", "qcom,tsens-v2";
		reg = <0x0 0x0c271000 0x0 0x1000>,
		      <0x0 0x0c222000 0x0 0x1000>;
		/* Lines exceeding coding style line wrap limit: */
		interconnects = <&aggre1_noc MASTER_USB3_0 0 &mc_virt SLAVE_EBI1 0>,
				<&gem_noc MASTER_APPSS_PROC 0 &config_noc SLAVE_USB3_0 0>;
	};

Организация DTSI и DTS
----------------------

Файлы DTSI и DTS должны быть организованы так, чтобы отражать общие,
переиспользуемые части оборудования. Обычно это означает организацию файлов
DTSI и DTS в несколько файлов:

1. DTSI с содержимым всего SoC, без узлов для оборудования, отсутствующего на
   SoC.
2. Если применимо: DTSI с общими или переиспользуемыми частями оборудования,
   например всем System-on-Module.
3. DTS, представляющий плату.

Компоненты оборудования, присутствующие на плате, должны размещаться в DTS
платы, а не в DTSI для SoC или SoM. Частичное исключение — общий внешний
опорный входной тактовый сигнал SoC, который может быть закодирован как
fixed-clock в DTSI для SoC, а его частота предоставляется каждым DTS платы.
