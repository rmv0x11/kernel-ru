==================================================
Модульное тестирование Open Firmware Devicetree
==================================================

Автор: Gaurav Minocha <gaurav.minocha.os@gmail.com>

1. Введение
===========

Этот документ объясняет, как тестовые данные, необходимые для запуска модульных
тестов OF, динамически присоединяются к «живому» дереву (live tree) независимо
от архитектуры машины.

Перед тем как двигаться дальше, рекомендуется прочитать следующие документы.

(1) Documentation/devicetree/usage-model.rst
(2) http://www.devicetree.org/Device_Tree_Usage

OF Selftest спроектирован для тестирования интерфейса (include/linux/of.h),
предоставляемого разработчикам драйверов устройств для получения информации об
устройствах и т.д. из развёрнутой (unflattened) структуры данных дерева устройств.
Этот интерфейс используется большинством драйверов устройств в различных сценариях.


2. Подробный вывод (EXPECT)
===========================

Если модульный тест обнаруживает проблему, он печатает в консоль предупреждение
или сообщение об ошибке.  Модульный тест также вызывает предупреждения и
сообщения об ошибках из другого кода ядра в результате намеренно некорректных
тестовых данных.  Это приводило к путанице относительно того, являются ли
вызванные сообщения ожидаемым результатом теста или это реальная проблема,
не зависящая от модульного теста.

В модульный тест были добавлены сообщения 'EXPECT \ : text' (начало) и
'EXPECT / : text' (конец), чтобы сообщать о том, что предупреждение или ошибка
ожидаемы.  Начало печатается перед тем, как вызвать предупреждение или ошибку,
а конец печатается после того, как предупреждение или ошибка вызваны.

Сообщения EXPECT приводят к очень шумным консольным сообщениям, которые трудно
читать.  Сценарий scripts/dtc/of_unittest_expect был создан, чтобы отфильтровать
эту многословность и подсветить расхождения между вызванными предупреждениями и
ошибками и ожидаемыми предупреждениями и ошибками.  Дополнительную информацию
можно получить с помощью 'scripts/dtc/of_unittest_expect --help'.


3. Тестовые данные
==================

Файл исходного кода дерева устройств (drivers/of/unittest-data/testcases.dts)
содержит тестовые данные, необходимые для выполнения модульных тестов,
автоматизированных в drivers/of/unittest.c. См. содержимое каталога::

    drivers/of/unittest-data/tests-*.dtsi

для включаемых файлов исходного кода дерева устройств (.dtsi), включённых в
testcases.dts.

Когда ядро собирается с включённым CONFIG_OF_UNITTEST, для компиляции исходного
файла DT (testcases.dts) в двоичный блок (testcases.dtb), также называемый
плоским (flattened) DT, используется следующее правило make::

    $(obj)/%.dtb: $(src)/%.dts FORCE
	    $(call if_changed_dep, dtc)

После этого, с помощью следующего правила приведённый выше двоичный блок
оборачивается в ассемблерный файл (testcases.dtb.S)::

    $(obj)/%.dtb.S: $(obj)/%.dtb
	    $(call cmd, dt_S_dtb)

Ассемблерный файл компилируется в объектный файл (testcases.dtb.o) и
компонуется в образ ядра.


3.1. Добавление тестовых данных
-------------------------------

Развёрнутая структура дерева устройств:

Развёрнутое дерево устройств состоит из связанных device_node(s) в виде древовидной
структуры, описанной ниже::

    // following struct members are used to construct the tree
    struct device_node {
	...
	struct  device_node *parent;
	struct  device_node *child;
	struct  device_node *sibling;
	...
    };

Рисунок 1 описывает обобщённую структуру развёрнутого дерева устройств машины,
учитывая только указатели child и sibling. Существует ещё один указатель,
``*parent``, который используется для обхода дерева в обратном направлении.
Таким образом, на определённом уровне дочерний узел (child) и все соседние узлы
(sibling) будут иметь указатель parent, указывающий на общий узел (например,
parent узлов child1, sibling2, sibling3, sibling4 указывает на корневой узел)::

    root ('/')
    |
    child1 -> sibling2 -> sibling3 -> sibling4 -> null
    |         |           |           |
    |         |           |          null
    |         |           |
    |         |        child31 -> sibling32 -> null
    |         |           |          |
    |         |          null       null
    |         |
    |      child21 -> sibling22 -> sibling23 -> null
    |         |          |            |
    |        null       null         null
    |
    child11 -> sibling12 -> sibling13 -> sibling14 -> null
    |           |           |            |
    |           |           |           null
    |           |           |
    null        null       child131 -> null
			    |
			    null

Рисунок 1: Обобщённая структура развёрнутого дерева устройств


Перед запуском модульного теста OF необходимо присоединить тестовые данные к
дереву устройств машины (если оно присутствует). Поэтому, когда вызывается
selftest_data_add(), она сначала читает плоские данные дерева устройств,
скомпонованные в образ ядра, через следующие символы ядра::

    __dtb_testcases_begin - address marking the start of test data blob
    __dtb_testcases_end   - address marking the end of test data blob

Во-вторых, она вызывает of_fdt_unflatten_tree(), чтобы развернуть плоский блок.
И, наконец, если дерево устройств машины (т.е. live tree) присутствует, то она
присоединяет развёрнутое дерево тестовых данных к live tree, иначе оно
присоединяется само как live-дерево устройств.

attach_node_and_children() использует of_attach_node() для присоединения узлов в
live tree, как объясняется ниже. Для пояснения этого дерево тестовых данных,
описанное на рисунке 2, присоединяется к live tree, описанному на рисунке 1::

    root ('/')
	|
    testcase-data
	|
    test-child0 -> test-sibling1 -> test-sibling2 -> test-sibling3 -> null
	|               |                |                |
    test-child01      null             null             null


Рисунок 2: Пример дерева тестовых данных для присоединения к live tree.

Согласно приведённому выше сценарию, live tree уже присутствует, поэтому не
требуется присоединять корневой узел root('/'). Все остальные узлы присоединяются
вызовом of_attach_node() для каждого узла.

В функции of_attach_node() новый узел присоединяется как дочерний к заданному
родителю в live tree. Но, если у родителя уже есть дочерний узел, то новый узел
заменяет текущий дочерний и превращает его в свой соседний (sibling). Поэтому,
когда узел данных testcase присоединяется к приведённому выше live tree
(рисунок 1), итоговая структура выглядит так, как показано на рисунке 3::

    root ('/')
    |
    testcase-data -> child1 -> sibling2 -> sibling3 -> sibling4 -> null
    |               |          |           |           |
    (...)             |          |           |          null
		    |          |         child31 -> sibling32 -> null
		    |          |           |           |
		    |          |          null        null
		    |          |
		    |        child21 -> sibling22 -> sibling23 -> null
		    |          |           |            |
		    |         null        null         null
		    |
		    child11 -> sibling12 -> sibling13 -> sibling14 -> null
		    |          |            |            |
		    null       null          |           null
					    |
					    child131 -> null
					    |
					    null
    -----------------------------------------------------------------------

    root ('/')
    |
    testcase-data -> child1 -> sibling2 -> sibling3 -> sibling4 -> null
    |               |          |           |           |
    |             (...)      (...)       (...)        null
    |
    test-sibling3 -> test-sibling2 -> test-sibling1 -> test-child0 -> null
    |                |                   |                |
    null             null                null         test-child01


Рисунок 3: Структура live-дерева устройств после присоединения testcase-data.


Проницательные читатели заметили бы, что узел test-child0 становится последним
соседним узлом (sibling) по сравнению с прежней структурой (рисунок 2). После
присоединения сначала test-child0 присоединяется test-sibling1, который вытесняет
дочерний узел (т.е. test-child0), делая его соседним, а себя делает дочерним
узлом, как упомянуто выше.

Если обнаруживается дублирующийся узел (т.е. если узел с тем же свойством
full_name уже присутствует в live tree), то узел не присоединяется, а его свойства
обновляются в узле live tree вызовом функции update_node_properties().


3.2. Удаление тестовых данных
-----------------------------

После завершения выполнения тестового набора вызывается selftest_data_remove,
чтобы удалить узлы устройств, присоединённые изначально (сначала отсоединяются
конечные узлы — листья, затем, поднимаясь вверх, удаляются родительские узлы и
в конечном итоге всё дерево). selftest_data_remove() вызывает
detach_node_and_children(), которая использует of_detach_node() для отсоединения
узлов от live-дерева устройств.

Чтобы отсоединить узел, of_detach_node() либо обновляет указатель child родителя
заданного узла на его соседний узел (sibling), либо присоединяет предыдущий
соседний узел к соседнему узлу заданного узла, в зависимости от ситуации. Вот и
всё :)
