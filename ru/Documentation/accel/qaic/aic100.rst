===============================
 Qualcomm Cloud AI 100 (AIC100)
===============================

Обзор
=====

Семейство продуктов Qualcomm Cloud AI 100/AIC100 (включая SA9000P — часть
Snapdragon Ride) представляет собой PCIe-платы расширения, содержащие выделенный
SoC-ASIC, предназначенный для эффективного выполнения задач инференса в области
глубокого обучения искусственного интеллекта (AI). Это AI-ускорители.

Интерфейс PCIe устройства AIC100 поддерживает скорости PCIe Gen4 по восьми линиям
(x8). Отдельный SoC на плате может содержать до 16 NSP для выполнения рабочих
нагрузок. Каждый SoC имеет управляющий CPU A53. На плате может быть до 32 ГБ DDR.

В одной системе можно разместить несколько плат AIC100 для масштабирования общей
производительности. Платы AIC100 поддерживают работу в многопользовательском
режиме и способны параллельно выполнять рабочие нагрузки нескольких
пользователей.

Описание аппаратной части
=========================

Плата AIC100 состоит из SoC AIC100, расположенной на плате DDR и набора
вспомогательной периферии (PMIC и т. п.).

Плата AIC100 может иметь форм-фактор PCIe HHHL (традиционная PCIe-плата) либо быть
двойной картой M.2. Оба варианта используют PCIe для подключения к хост-системе.

Будучи PCIe-оконечным устройством/адаптером, AIC100 использует стандартную
комбинацию VendorID(VID)/DeviceID(DID) для уникальной идентификации себя перед
хостом. AIC100 использует стандартный VID Qualcomm (0x17cb). Все SKU AIC100
используют один и тот же DID AIC100 (0xa100).

AIC100 не реализует FLR (function level reset).

AIC100 реализует MSI, но не реализует MSI-X. AIC100 предпочитает для работы 17 MSI
(1 для MHI, 16 для DMA Bridge). Откат к 1 MSI возможен в сценариях, где
резервирование 32 MSI невозможно.

Будучи PCIe-устройством, AIC100 использует BAR для предоставления хосту
интерфейсов к аппаратной части устройства. AIC100 предоставляет 3 64-битных BAR.

* Первый BAR имеет размер 4K и предоставляет хосту интерфейс MHI.

* Второй BAR имеет размер 2M и предоставляет хосту интерфейс DMA Bridge.

* Третий BAR имеет переменный размер, зависящий от конфигурации конкретного
  AIC100, но по умолчанию составляет 64K. На данный момент этот BAR не имеет
  назначения.

С точки зрения хоста, AIC100 имеет несколько ключевых аппаратных компонентов —

* MHI (Modem Host Interface)
* QSM (QAIC Service Manager)
* NSP (Neural Signal Processor)
* DMA Bridge
* DDR

MHI
---

AIC100 имеет один интерфейс MHI поверх PCIe. Сам MHI документирован в
Documentation/mhi/index.rst MHI — это механизм, который хост использует для связи
с QSM. За исключением данных рабочих нагрузок, передаваемых через DMA Bridge, всё
взаимодействие с устройством происходит через MHI.

QSM
---

QAIC Service Manager. Это CPU ARM A53, который выполняет основную прошивку платы и
осуществляет задачи управления на плате. Он также взаимодействует с хостом через
MHI. Каждый AIC100 имеет один такой.

NSP
---

Neural Signal Processor. Каждый AIC100 имеет до 16 таких. Это процессоры, которые
выполняют рабочие нагрузки на AIC100. Каждый NSP — это DSP Qualcomm Hexagon (Q6) с
HVX и HMX. Каждый NSP может выполнять только одну рабочую нагрузку одновременно, но
одной рабочей нагрузке может быть назначено несколько NSP. Поскольку каждый NSP
может выполнять только одну рабочую нагрузку, AIC100 ограничен 16 параллельными
рабочими нагрузками. «Планирование» рабочих нагрузок находится в ведении хоста.
AIC100 не выполняет автоматическое разделение по времени.

DMA Bridge
----------

DMA Bridge — это специализированный механизм DMA, который управляет потоком данных
в рабочие нагрузки и из них. AIC100 имеет один такой. DMA Bridge имеет 16 каналов,
каждый из которых состоит из набора FIFO запросов/ответов. Каждой активной рабочей
нагрузке назначается один канал DMA Bridge. DMA Bridge предоставляет аппаратные
регистры для управления FIFO (указатели головы/хвоста), но требует память хоста
для хранения FIFO.

DDR
---

AIC100 имеет DDR на плате. В сумме AIC100 может иметь до 32 ГБ DDR. Эта DDR
используется для хранения рабочих нагрузок, данных для рабочих нагрузок и
используется QSM для управления устройством. NSP получают доступ к участкам DDR от
QSM. Хост не имеет прямого доступа к DDR и должен направлять запросы к QSM для
передачи данных в DDR.

Высокоуровневый порядок использования
=====================================

AIC100 — это многопользовательский программируемый ускоритель, обычно
используемый для выполнения нейронных сетей в режиме инференса с целью
эффективного выполнения AI-операций. AIC100 не предназначен для обучения нейронных
сетей. AIC100 может использоваться для рабочих нагрузок общего назначения
(вычислений).

Предположим, пользователь хочет задействовать AIC100; в этом случае он выполнил бы
следующие шаги:

1. Скомпилировать рабочую нагрузку в ELF, нацеленный на NSP
2. Направить запросы к QSM для загрузки рабочей нагрузки и связанных артефактов в
   DDR устройства
3. Направить запрос к QSM для активации рабочей нагрузки на наборе простаивающих
   NSP
4. Направить запросы к DMA Bridge для отправки входных данных рабочей нагрузке для
   обработки, а также другие запросы для получения обработанных выходных данных от
   рабочей нагрузки.
5. Как только рабочая нагрузка больше не требуется, направить запрос к QSM для
   деактивации рабочей нагрузки, тем самым возвращая NSP в простаивающее состояние.
6. Как только рабочая нагрузка и связанные артефакты больше не нужны для будущих
   сессий, направить запросы к QSM для выгрузки данных из DDR. Это освобождает DDR
   для использования другими пользователями.


Порядок загрузки
================

AIC100 использует беспрошивочный (flashless) порядок загрузки, унаследованный от
Qualcomm MSM.

При первом включении AIC100 начинает выполнять PBL (Primary Bootloader) из ROM.
PBL перечисляет PCIe-линк и инициализирует компонент BHI (Boot Host Interface)
интерфейса MHI.

Используя BHI, хост указывает PBL местоположение образа SBL (Secondary
Bootloader). PBL извлекает образ с хоста, проверяет его и начинает выполнение SBL.

SBL инициализирует MHI и использует MHI для уведомления хоста о том, что
устройство вошло в стадию SBL. SBL выполняет ряд операций:

* SBL инициализирует большую часть аппаратной части (всё, что PBL оставил
  неинициализированным), включая DDR.
* SBL выгружает журнал загрузки (bootlog) на хост.
* SBL синхронизирует временные метки с хостом для будущего журналирования.
* SBL использует протокол Sahara для получения образов runtime-прошивки с хоста.

Как только SBL получил и проверил runtime-прошивку, он выводит NSP из сброса и
переходит в QSM.

QSM использует MHI для уведомления хоста о том, что устройство вошло в стадию QSM
(AMSS в терминах MHI). На этом этапе устройство AIC100 полностью функционально и
готово к обработке рабочих нагрузок.

DMA Bridge
==========

Обзор
-----

DMA Bridge — это один из основных интерфейсов к хосту со стороны устройства
(другим является MHI). В рамках активации рабочей нагрузки для выполнения на NSP
QSM назначает этой сети канал DMA Bridge. Канал DMA Bridge рабочей нагрузки (далее
DBC) предназначен исключительно для использования этой рабочей нагрузкой и не
разделяется с другими рабочими нагрузками.

Каждый DBC — это пара FIFO, управляющих данными, поступающими в рабочую нагрузку и
из неё. Одна FIFO — это FIFO запросов. Другая FIFO — это FIFO ответов.

Каждый DBC содержит 4 регистра в аппаратной части:

* Указатель головы FIFO запросов (смещение 0x0). Доступен хосту только для чтения.
  Указывает на последний элемент FIFO, который устройство потребило.
* Указатель хвоста FIFO запросов (смещение 0x4). Доступен хосту для чтения/записи.
  Хост инкрементирует этот регистр для добавления новых элементов в FIFO.
* Указатель головы FIFO ответов (смещение 0x8). Доступен хосту для чтения/записи.
  Указывает на последний элемент FIFO, который потребил хост.
* Указатель хвоста FIFO ответов (смещение 0xc). Доступен хосту только для чтения.
  Устройство инкрементирует этот регистр для добавления новых элементов в FIFO.

Значения в каждом регистре — это индексы в FIFO. Чтобы получить местоположение
элемента FIFO, на который указывает регистр: базовый адрес FIFO + регистр * размер
элемента.

Регистры DBC предоставляются хосту через второй BAR. Каждый DBC занимает 4KB
пространства в BAR.

Сами FIFO размещаются в памяти хоста. При отправке запроса к QSM на активацию сети
хост должен предоставить (донорно) память для использования под FIFO. Из-за
внутренних ограничений отображения устройства на каждый DBC должен предоставляться
единый непрерывный блок памяти, в котором размещаются обе FIFO. FIFO запросов
займёт начало блока памяти, а FIFO ответов займёт конец блока памяти.

Request FIFO
------------

Элемент FIFO запросов имеет следующую структуру:

.. code-block:: c

  struct request_elem {
	u16 req_id;
	u8  seq_id;
	u8  pcie_dma_cmd;
	u32 reserved;
	u64 pcie_dma_source_addr;
	u64 pcie_dma_dest_addr;
	u32 pcie_dma_len;
	u32 reserved;
	u64 doorbell_addr;
	u8  doorbell_attr;
	u8  reserved;
	u16 reserved;
	u32 doorbell_data;
	u32 sem_cmd0;
	u32 sem_cmd1;
	u32 sem_cmd2;
	u32 sem_cmd3;
  };

Описания полей запроса:

req_id
	идентификатор запроса. Элемент FIFO запросов и элемент FIFO ответов с
	одинаковым идентификатором запроса относятся к одной и той же команде.

seq_id
	порядковый идентификатор (sequence ID) в рамках запроса. Игнорируется DMA
	Bridge.

pcie_dma_cmd
	описывает DMA-элемент данного запроса.

	* Бит(7) — флаг force msi, который переопределяет логику MSI DMA Bridge и
	  генерирует MSI по завершении этого запроса, и QSM настраивает DMA Bridge
	  на учёт этого бита.
	* Биты(6:5) зарезервированы.
	* Бит(4) — флаг кода завершения, указывающий, что DMA Bridge должен
	  сгенерировать элемент FIFO ответов по завершении этого запроса.
	* Бит(3) указывает, является ли данный запрос передачей по связному списку(0)
	  или массовой передачей(1).
	* Бит(2) зарезервирован.
	* Биты(1:0) указывают тип передачи. Нет передачи(0), на устройство(1), с
	  устройства(2). Значение 3 недопустимо.

pcie_dma_source_addr
	адрес источника для массовой передачи или адрес связного списка.

pcie_dma_dest_addr
	адрес назначения для массовой передачи.

pcie_dma_len
	длина массовой передачи. Обратите внимание, что размер этого поля ограничивает
	передачи размером 4G.

doorbell_addr
	адрес doorbell, в который нужно записать по завершении этого запроса.

doorbell_attr
	атрибуты doorbell.

	* Бит(7) указывает, должна ли произойти запись в doorbell.
	* Биты(6:2) зарезервированы.
	* Биты(1:0) содержат кодировку длины doorbell. 0 — 32 бита, 1 — 16 бит, 2 —
	  8 бит, 3 зарезервировано. Адрес doorbell должен быть естественно выровнен по
	  указанной длине.

doorbell_data
	данные для записи в doorbell. Действительны только биты, соответствующие длине
	doorbell.

sem_cmdN
	команда семафора.

	* Бит(31) указывает, что данная команда семафора включена.
	* Бит(30) — DMA-барьер на устройство (to-device DMA fence). Блокирует данный
	  запрос до завершения всех DMA-передач на устройство.
	* Бит(29) — DMA-барьер с устройства (from-device DMA fence). Блокирует данный
	  запрос до завершения всех DMA-передач с устройства.
	* Биты(28:27) зарезервированы.
	* Биты(26:24) — команда семафора. 0 — NOP. 1 — инициализировать указанным
	  значением. 2 — инкремент. 3 — декремент. 4 — ждать, пока семафор не станет
	  равен указанному значению. 5 — ждать, пока семафор не станет больше или равен
	  указанному значению. 6 — «P», ждать, пока семафор не станет больше 0, затем
	  декрементировать на 1. 7 зарезервировано.
	* Бит(23) зарезервирован.
	* Бит(22) — синхронизация семафора. 0 — постсинхронизация (post sync), что
	  означает, что операция семафора выполняется после DMA-передачи. 1 —
	  пресинхронизация (presync), которая управляет (гейтирует) DMA-передачей. На
	  один запрос допускается только одна presync.
	* Бит(21) зарезервирован.
	* Биты(20:16) — индекс семафора, над которым выполняется операция.
	* Биты(15:12) зарезервированы.
	* Биты(11:0) — значение семафора, используемое в операциях.

В целом запрос обрабатывается в 4 шага:

1. Если указано, условие presync-семафора должно быть истинным
2. Если включено, происходит DMA-передача
3. Если указано, условия postsync-семафора должны быть истинными
4. Если включено, выполняется запись в doorbell

Используя семафоры в сочетании с рабочей нагрузкой, выполняющейся на NSP, конвейер
данных можно синхронизировать так, чтобы хост мог поставить в очередь несколько
запросов данных для обработки рабочей нагрузкой, но DMA Bridge будет копировать
данные в память рабочей нагрузки только тогда, когда рабочая нагрузка готова
обработать следующий вход.

Response FIFO
-------------

Как только запрос полностью обработан, генерируется элемент FIFO ответов, если это
указано в pcie_dma_cmd. Структура элемента FIFO ответов:

.. code-block:: c

  struct response_elem {
	u16 req_id;
	u16 completion_code;
  };

req_id
	совпадает с req_id запроса, который сгенерировал этот элемент.

completion_code
	статус данного запроса. 0 — успех. Ненулевое значение — ошибка.

DMA Bridge будет генерировать MSI хосту в качестве реакции на активность в FIFO
ответов какого-либо DBC. Аппаратная часть DMA Bridge имеет алгоритм смягчения
шторма IRQ, при котором она генерирует MSI только тогда, когда FIFO ответов
переходит из пустого в непустое состояние (если только не включён и не сработал
force MSI). В ответ на этот MSI хост должен опустошить FIFO ответов и обязан
позаботиться об обработке любых состояний гонки (race condition) между
опустошением FIFO и вставкой устройством элементов в FIFO.

Протокол Neural Network Control (NNC)
=====================================

Протокол NNC — это способ, которым хост направляет запросы к QSM для управления
рабочими нагрузками. Он использует MHI-канал QAIC_CONTROL.

Каждый запрос NNC упаковывается в сообщение. Каждое сообщение представляет собой
последовательность транзакций. Транзакция типа passthrough может содержать
элементы, известные как команды.

QSM требует, чтобы NNC-сообщения были закодированы в little endian, а поля были
естественно выровнены. Поскольку в некоторых NNC-сообщениях есть 64-битные
элементы, необходимо поддерживать 64-битное выравнивание.

Сообщение содержит заголовок, а затем последовательность транзакций. Сообщение от
QSM к хосту может иметь размер не более 4K. От хоста к QSM сообщение может иметь
размер не более 64K (максимальный размер одного MHI-пакета), но существует функция
продолжения, при которой сообщение N+1 может быть помечено как продолжение
сообщения N. Это используется для чрезвычайно больших транзакций DMA-передачи.

Описания транзакций
-------------------

passthrough
	Позволяет пространству пользователя отправлять непрозрачную полезную нагрузку
	напрямую к QSM. Это используется для NNC-команд. Пространство пользователя
	отвечает за соблюдение требований QSM к сообщению в полезной нагрузке.

dma_xfer
	DMA-передача. Описывает объект, который QSM должен передать по DMA в устройство
	с помощью кортежей адреса и размера.

activate
	Активировать рабочую нагрузку на NSP. Хост должен предоставить память для
	использования DBC.

deactivate
	Деактивировать активную рабочую нагрузку и вернуть NSP в простаивающее
	состояние.

status
	Запросить у QSM сведения о его реализации NNC. Возвращает версию NNC и сведения
	о том, используется ли CRC.

terminate
	Освободить ресурсы пользователя.

dma_xfer_cont
	Продолжение предыдущей DMA-передачи. Если DMA-передачу нельзя указать в одном
	сообщении (сильно фрагментирована), эту транзакцию можно использовать для
	указания дополнительных диапазонов.

validate_partition
	Запрос к QSM для определения того, является ли идентификатор раздела
	действительным.

Каждое сообщение помечается идентификатором пользователя и идентификатором
раздела. Идентификатор пользователя позволяет QSM отслеживать ресурсы и
освобождать их, когда пользователь исчезает (например, процесс аварийно
завершается). Идентификатор раздела определяет ресурсный раздел, которым управляет
QSM и к которому относится данное сообщение.

Сообщения могут иметь CRC. К сообщениям следует применять CRC до тех пор, пока QSM
не сообщит через транзакцию status, что CRC не нужны. QSM на SA9000P требует CRC
для обеспечения безопасности «чёрного канала» (black channel safing).

Перезапуск подсистемы (SSR)
===========================

SSR — это концепция ограничения влияния ошибки. Устройство AIC100 может иметь
несколько пользователей, у каждого из которых выполняется своя рабочая нагрузка.
Если рабочая нагрузка одного пользователя аварийно завершается, последствия этого
должны ограничиваться этой рабочей нагрузкой и не затрагивать другие рабочие
нагрузки. SSR обеспечивает это.

Если конкретная рабочая нагрузка аварийно завершается, QSM уведомляет хост через
MHI-канал QAIC_SSR. Это уведомление идентифицирует рабочую нагрузку по назначенному
ей DBC. Затем используется многоэтапный процесс восстановления для очистки обеих
сторон, который приводит DBC/NSP в рабочее состояние.

Когда происходит SSR, любое состояние рабочей нагрузки теряется. Любые входные
данные, которые находились в процессе обработки или были поставлены в очередь, но
ещё не обслужены, теряются. Загруженные артефакты останутся в DDR на плате, но
хосту потребуется повторно активировать рабочую нагрузку, если он желает её
восстановить.

Когда SSR происходит для конкретного NSP, назначенный DBC проходит следующие
переходы состояний по порядку:

DBC_STATE_BEFORE_SHUTDOWN
	Указывает, что затронутый NSP обнаружен в невосстановимом состоянии ошибки.
DBC_STATE_AFTER_SHUTDOWN
	Указывает, что NSP находится под сбросом.
DBC_STATE_BEFORE_POWER_UP
	Указывает, что отладочная информация NSP собрана и готова к сбору хостом (при
	желании). На этой стадии NSP перезапускается QSM.
DBC_STATE_AFTER_POWER_UP
	Указывает, что NSP перезапущен, полностью работоспособен и находится в
	простаивающем состоянии.

SSR также имеет необязательную функцию сбора crashdump. Если она включена, хост
может собрать дамп памяти аварийно завершившегося NSP и выгрузить его в
пространство пользователя через подсистему dev_coredump. Хост также может
отклонить запрос на сбор crashdump от устройства.

Reliability, Accessibility, Serviceability (RAS)
================================================

Предполагается, что AIC100 будет развёрнут в серверных системах, где применяется
идеология RAS. Проще говоря, RAS — это концепция обнаружения, классификации и
сообщения об ошибках. Хотя PCIe имеет AER (Advanced Error Reporting), что является
частью RAS, AER не позволяет устройству сообщать детали о внутренних ошибках.
Поэтому AIC100 реализует собственный механизм RAS. Когда происходит событие RAS,
QSM сообщает о событии с соответствующими деталями через MHI-канал QAIC_STATUS.
Системный администратор может определить, что конкретному устройству требуется
дополнительное обслуживание, на основе отчётов RAS.

Телеметрия
==========

QSM имеет возможность сообщать различные физические атрибуты устройства, а в
некоторых случаях — позволять хосту управлять ими. Примеры включают температурные
пределы, температурные показания и показания мощности. Эти элементы передаются
через MHI-канал QAIC_TELEMETRY.
