=============
 Драйвер QAIC
=============

Драйвер QAIC — это драйвер режима ядра (Kernel Mode Driver, KMD) для семейства
продуктов-ускорителей ИИ AIC100.

Прерывания
==========

Подавление шторма прерываний
----------------------------

Хотя аппаратное обеспечение DMA Bridge AIC100 реализует механизм подавления
шторма прерываний (IRQ storm), шторм прерываний всё же может возникнуть. Шторм
может произойти, если рабочая нагрузка особенно быстрая, а хост отзывчив. Если
хост способен опустошать FIFO ответов так же быстро, как устройство вставляет в
него элементы, то устройство будет часто переводить FIFO ответов из пустого
состояния в непустое и генерировать MSI с частотой, эквивалентной скорости
обработки рабочей нагрузкой своих входных данных. Известно, что рабочая нагрузка
lprnet (license plate reader network — сеть распознавания автомобильных номеров)
вызывает это состояние и может генерировать свыше 100 тыс. MSI в секунду.
Замечено, что большинство систем не могут долго это выдерживать и аварийно
завершают работу из-за той или иной формы watchdog по причине накладных расходов
на прерывание хост-CPU контроллером прерываний.

Чтобы смягчить эту проблему, драйвер QAIC реализует специальную обработку
прерываний. Когда QAIC получает IRQ, он отключает эту линию. Это не даёт
контроллеру прерываний прерывать CPU. Затем AIC опустошает FIFO. После того как
FIFO опустошён, QAIC реализует алгоритм опроса «последнего шанса», в котором QAIC
спит некоторое время, чтобы посмотреть, сгенерирует ли рабочая нагрузка ещё
активность. В течение этого времени линия IRQ остаётся отключённой. Если
активность не обнаружена, QAIC выходит из режима опроса и снова включает линию
IRQ.

Это подавление в QAIC очень эффективно. Тот же сценарий использования lprnet,
который генерирует 100 тыс. IRQ в секунду (по данным /proc/interrupts),
сокращается примерно до 64 IRQ за 5 минут при сохранении стабильности хост-системы
и той же производительности по пропускной способности рабочей нагрузки (в пределах
вариации шума от запуска к запуску).

Режим одиночного MSI
--------------------

MultiMSI поддерживается не на всех системах хорошо; на виртуализированных — ещё
хуже (примерно по состоянию на 2023 год). Из-за маскировки гипервизорами структуры
возможностей PCIe MSI и больших требований к памяти у vIOMMU (необходимых для
поддержки MultiMSI) полезно иметь возможность откатываться к одиночному MSI, когда
это требуется.

Для поддержки этого отката мы допускаем случай, когда удаётся выделить только один
MSI, и разделяем этот один MSI между MHI и DBC. Устройство обнаруживает, что
сконфигурирован только один MSI, и направляет прерывания для DBC на прерывание,
обычно используемое для MHI. К сожалению, это означает, что обработчики прерываний
для каждого DBC и MHI пробуждаются на каждое поступающее прерывание; однако
поточные обработчики прерываний DBC запускаются только тогда, когда обнаружена
работа, которую нужно выполнить (MHI всегда запускает свой поточный обработчик).

Если DBC сконфигурирован на принудительные прерывания MSI, это может обходить
программное подавление шторма прерываний, упомянутое выше. Поскольку MSI является
разделяемым, он никогда не отключается, что позволяет каждой новой записи в FIFO
вызывать новое прерывание.


Протокол Neural Network Control (NNC)
=====================================

Реализация NNC разделена между KMD (QAIC) и UMD. В общем случае QAIC понимает, как
кодировать/декодировать сетевой протокол NNC, и те элементы протокола, обработка
которых требует знаний пространства ядра (например, отображение памяти хоста в
IOVA устройства). QAIC понимает структуру сообщения и все транзакции. QAIC не
понимает команд (полезной нагрузки транзакции passthrough).

QAIC обрабатывает и обеспечивает соблюдение требуемого порядка байтов little endian
и 64-битного выравнивания в той степени, в какой это возможно. Поскольку QAIC не
знает содержимое транзакции passthrough, он полагается на UMD в удовлетворении этих
требований.

Транзакция terminate особенно полезна для QAIC. QAIC не осведомлён о ресурсах,
загруженных на устройство, поскольку большая часть этой активности происходит
внутри команд NNC. Как следствие, у QAIC нет средств для отката активности
пространства пользователя. Чтобы гарантировать, что ресурсы клиента пространства
пользователя полностью освобождаются в случае аварийного завершения процесса или
ошибки, QAIC использует команду terminate, чтобы сообщить QSM, когда пользователь
ушёл и ресурсы могут быть освобождены.

QSM может сообщать номер версии протокола NNC, который он поддерживает. Это делается
в форме старшего (Major) и младшего (Minor) номеров.

Обновления старшего номера указывают на изменения в протоколе NNC, которые
затрагивают формат сообщений или транзакции (затрагивает QAIC).

Обновления младшего номера указывают на изменения в протоколе NNC, которые
затрагивают команды (не затрагивает QAIC).

uAPI
====

QAIC создаёт accel-устройство для каждого физического PCIe-устройства. Это
accel-устройство существует до тех пор, пока PCIe-устройство известно Linux.

PCIe-устройство может не всегда находиться в состоянии, готовом принимать запросы
из пространства пользователя. QAIC будет генерировать uevent-события
KOBJ_ONLINE/OFFLINE, чтобы оповещать, когда устройство может принимать запросы
(ONLINE) и когда устройство больше не принимает запросы (OFFLINE) из-за сброса или
другого перехода состояния.

QAIC определяет ряд специфичных для драйвера IOCTL как часть API пространства
пользователя.

DRM_IOCTL_QAIC_MANAGE
  Этот IOCTL позволяет пространству пользователя отправить запрос NNC в QSM. Вызов
  будет блокироваться до получения ответа или истечения тайм-аута запроса.

DRM_IOCTL_QAIC_CREATE_BO
  Этот IOCTL позволяет пространству пользователя выделить объект буфера (buffer
  object, BO), который может отправлять или принимать данные от рабочей нагрузки.
  Вызов вернёт дескриптор GEM, представляющий выделенный буфер. BO непригоден к
  использованию, пока он не нарезан (см. DRM_IOCTL_QAIC_ATTACH_SLICE_BO).

DRM_IOCTL_QAIC_MMAP_BO
  Этот IOCTL позволяет пространству пользователя подготовить выделенный BO к
  отображению через mmap в процесс пространства пользователя.

DRM_IOCTL_QAIC_ATTACH_SLICE_BO
  Этот IOCTL позволяет пространству пользователя нарезать BO в рамках подготовки к
  отправке BO на устройство. Нарезка — это операция описания того, какие части BO
  отправляются и куда в рабочую нагрузку. Это требует набора DMA-передач для DMA
  Bridge, и поэтому привязывает BO к конкретному DBC.

DRM_IOCTL_QAIC_EXECUTE_BO
  Этот IOCTL позволяет пространству пользователя отправить набор нарезанных BO на
  устройство. Вызов неблокирующий. Успех указывает лишь на то, что BO поставлены в
  очередь к устройству, но не гарантирует, что они были выполнены.

DRM_IOCTL_QAIC_PARTIAL_EXECUTE_BO
  Этот IOCTL работает подобно DRM_IOCTL_QAIC_EXECUTE_BO, но позволяет пространству
  пользователя уменьшить BO, отправляемые на устройство для данного конкретного
  вызова. Если BO обычно имеет N входов, но доступно только их подмножество, этот
  IOCTL позволяет пространству пользователя указать, что на устройство следует
  отправить только первые M байт BO, чтобы минимизировать накладные расходы на
  передачу данных. Этот IOCTL динамически пересчитывает нарезку и поэтому несёт
  некоторые накладные расходы на обработку, прежде чем BO смогут быть поставлены в
  очередь к устройству.

DRM_IOCTL_QAIC_WAIT_BO
  Этот IOCTL позволяет пространству пользователя определить, когда конкретный BO был
  обработан устройством. Вызов будет блокироваться до тех пор, пока BO не будет
  обработан и не сможет быть снова поставлен в очередь к устройству, либо пока не
  произойдёт тайм-аут.

DRM_IOCTL_QAIC_PERF_STATS_BO
  Этот IOCTL позволяет пространству пользователя собрать статистику производительности
  по самому последнему выполнению BO. Это позволяет пространству пользователя
  построить сквозную временную диаграмму обработки BO для анализа производительности.

DRM_IOCTL_QAIC_DETACH_SLICE_BO
  Этот IOCTL позволяет пространству пользователя удалить информацию о нарезке из BO,
  которая изначально была предоставлена вызовом DRM_IOCTL_QAIC_ATTACH_SLICE_BO. Это
  обратная операция к DRM_IOCTL_QAIC_ATTACH_SLICE_BO. Для вызова
  DRM_IOCTL_QAIC_DETACH_SLICE_BO BO должен быть бездействующим. После успешной
  операции отсоединения нарезки BO может получить новую информацию о нарезке,
  присоединённую новым вызовом DRM_IOCTL_QAIC_ATTACH_SLICE_BO. После отсоединения
  нарезки BO не может быть выполнен до новой операции присоединения нарезки.
  Сочетание вызовов присоединения и отсоединения нарезки позволяет пространству
  пользователя использовать один BO с несколькими рабочими нагрузками.

Изоляция клиентов пространства пользователя
===========================================

AIC100 поддерживает несколько клиентов. Несколько DBC могут потребляться одним
клиентом, и несколько клиентов могут каждый потреблять один или более DBC. Рабочие
нагрузки могут содержать конфиденциальную информацию, поэтому только клиенту,
владеющему рабочей нагрузкой, должно быть разрешено взаимодействовать с DBC.

Клиенты идентифицируются по экземпляру, связанному с их вызовом open(). Клиент может
использовать только память, которую он выделяет, и DBC, назначенные его рабочим
нагрузкам. Попытки получить доступ к ресурсам, назначенным другим клиентам, будут
отклонены.

Параметры модуля
================

QAIC поддерживает следующие параметры модуля:

**datapath_polling (bool)**

Конфигурирует QAIC на использование потока опроса для событий тракта данных вместо
полагания на прерывания устройства. Полезно для платформ с неработающим multiMSI.
Должен быть установлен при инициализации драйвера QAIC. По умолчанию 0 (выключено).

**mhi_timeout_ms (unsigned int)**

Устанавливает значение тайм-аута для операций MHI в миллисекундах (ms). Должен быть
установлен в момент, когда драйвер обнаруживает устройство. По умолчанию 2000 (2
секунды).

**control_resp_timeout_s (unsigned int)**

Устанавливает значение тайм-аута для ответов QSM на сообщения NNC в секундах (s).
Должен быть установлен в момент, когда драйвер отправляет запрос в QSM. По умолчанию
60 (одна минута).

**wait_exec_default_timeout_ms (unsigned int)**

Устанавливает тайм-аут по умолчанию для ioctl wait_exec в миллисекундах (ms). Должен
быть установлен до вызова ioctl waic_exec. Значение, указанное в вызове ioctl,
переопределяет это для данного вызова. По умолчанию 5000 (5 секунд).

**datapath_poll_interval_us (unsigned int)**

Устанавливает интервал опроса в микросекундах (us), когда опрос тракта данных
активен. Вступает в силу на следующем интервале опроса. По умолчанию 100 (100 us).

**timesync_delay_ms (unsigned int)**

Устанавливает интервал времени в миллисекундах (ms) между двумя последовательными
операциями timesync. По умолчанию 1000 (1000 ms).
