Обзор
=====

AMD NPU (Neural Processing Unit) — это многопользовательский ускоритель
AI-инференса, интегрированный в клиентский APU от AMD. NPU обеспечивает
эффективное выполнение приложений машинного обучения, таких как CNN, LLM и т. п.
NPU основан на `AMD XDNA Architecture`_. NPU управляется драйвером **amdxdna**.


Описание оборудования
=====================

AMD NPU состоит из следующих аппаратных компонентов:

AMD XDNA Array
--------------

AMD XDNA Array представляет собой двумерный массив вычислительных тайлов и тайлов
памяти, построенный на основе `AMD AI Engine Technology`_. Каждый столбец
содержит 4 ряда вычислительных тайлов и 1 ряд тайла памяти. Каждый
вычислительный тайл содержит VLIW-процессор с собственной выделенной памятью
программ и данных. Тайл памяти выступает в роли памяти L2. Двумерный массив
может быть разбит по границе столбца, что создаёт пространственно изолированный
раздел, который может быть привязан к контексту рабочей нагрузки.

Каждый столбец также имеет выделенные DMA-движки для перемещения данных между
хостовой DDR и тайлом памяти.

Клиентские NPU AMD Phoenix и AMD Hawk Point имеют топологию 4x5, то есть 4 ряда
вычислительных тайлов, организованных в 5 столбцов. Клиентский APU AMD Strix
Point имеет топологию 4x8, то есть 4 ряда вычислительных тайлов, организованных
в 8 столбцов.

Общая память L2
---------------

Единственный ряд тайлов памяти образует пул управляемой программно встроенной
памяти L2. DMA-движки используются для перемещения данных между хостовой DDR и
тайлами памяти. NPU AMD Phoenix и AMD Hawk Point имеют в общей сложности 2560 КБ
памяти L2. NPU AMD Strix Point имеет в общей сложности 4096 КБ памяти L2.

Микроконтроллер
---------------

Микроконтроллер исполняет прошивку NPU (NPU Firmware), которая отвечает за
обработку команд, настройку разделов XDNA Array, конфигурацию XDNA Array,
управление контекстами рабочих нагрузок и оркестрацию рабочих нагрузок.

Прошивка NPU использует выделенный экземпляр изолированного непривилегированного
контекста под названием ERT для обслуживания каждого контекста рабочей нагрузки.
ERT также используется для выполнения предоставленного пользователем
``ctrlcode``, связанного с контекстом рабочей нагрузки.

Прошивка NPU использует единственный изолированный привилегированный контекст
под названием MERT для обслуживания команд управления от драйвера amdxdna.

Mailboxes
---------

Микроконтроллер и драйвер amdxdna используют привилегированный канал для задач
управления, таких как настройка контекстов, телеметрия, запросы, обработка
ошибок, настройка пользовательского канала и т. п. Как уже упоминалось ранее,
запросы по привилегированному каналу обслуживаются MERT. Привилегированный канал
привязан к единственному почтовому ящику (mailbox).

Микроконтроллер и драйвер amdxdna используют выделенный пользовательский канал
для каждого контекста рабочей нагрузки. Пользовательский канал в основном
используется для отправки работы в NPU. Как уже упоминалось ранее, запросы по
пользовательскому каналу обслуживаются экземпляром ERT. Каждый пользовательский
канал привязан к собственному выделенному почтовому ящику.

PCIe EP
-------

NPU виден для хостового CPU архитектуры x86 как PCIe-устройство с несколькими BAR
и некоторым числом векторов прерываний MSI-X. NPU использует выделенную
высокопропускную фабрику уровня SoC для чтения из хостовой памяти и записи в неё.
Каждый экземпляр ERT получает собственное выделенное прерывание MSI-X. MERT
получает единственный экземпляр прерывания MSI-X.

Количество PCIe BAR варьируется в зависимости от конкретного устройства. По своим
функциям PCIe BAR в целом можно разделить на следующие типы.

* PSP BAR: предоставляет функцию AMD PSP (Platform Security Processor)
* SMU BAR: предоставляет функцию AMD SMU (System Management Unit)
* SRAM BAR: предоставляет кольцевые буферы для почтового ящика
* Mailbox BAR: предоставляет управляющие регистры почтового ящика (регистры head,
  tail, ISR и т. п.)
* Public Register BAR: предоставляет публичные регистры

На определённых устройствах упомянутые выше типы BAR могут быть объединены в
один физический PCIe BAR. Либо модулю может потребоваться два физических PCIe BAR
для полноценной работы. Например,

* На устройстве AMD Phoenix PSP, SMU и Public Register BAR находятся на PCIe BAR
  с индексом 0.
* На устройстве AMD Strix Point Mailbox и Public Register BAR находятся на PCIe
  BAR с индексом 0. У PSP некоторые регистры находятся на PCIe BAR с индексом 0
  (Public Register BAR) и на PCIe BAR с индексом 4 (PSP BAR).

Аппаратная изоляция процессов
-----------------------------

Как пояснялось ранее, XDNA Array может динамически разделяться на изолированные
пространственные разделы, каждый из которых может иметь один или несколько
столбцов. Пространственный раздел настраивается путём программирования регистров
изоляции столбцов микроконтроллером. Каждый пространственный раздел связан с
PASID, который также программируется микроконтроллером. Благодаря этому несколько
пространственных разделов в NPU могут осуществлять параллельный доступ к хосту,
защищённый посредством PASID.

Сама прошивка NPU использует изолированные контексты, ограничиваемые MMU
микроконтроллера, для обслуживания запросов по пользовательскому и
привилегированному каналам.


Смешанное пространственное и временное планирование
===================================================

Архитектура AMD XDNA поддерживает смешанное пространственное и временное (с
разделением по времени) планирование двумерного массива. Это означает, что
пространственные разделы могут динамически создаваться и разрушаться для
размещения различных рабочих нагрузок. Пространственный (*spatial*) раздел может
быть *монопольно* (exclusively) привязан к одному контексту рабочей нагрузки, в
то время как другой раздел может быть *временно* (temporarily) привязан более чем
к одному контексту рабочей нагрузки. Микроконтроллер обновляет PASID для временно
разделяемого раздела так, чтобы он соответствовал контексту, который привязан к
разделу в данный момент.

Resource Solver
---------------

Компонент Resource Solver драйвера amdxdna управляет распределением двумерного
массива между различными рабочими нагрузками. Каждая рабочая нагрузка описывает в
своих метаданных количество столбцов, необходимое для выполнения двоичного файла
NPU. Компонент Resource Solver использует подсказки, передаваемые рабочей
нагрузкой, и собственную эвристику, чтобы решить, какую стратегию
(пере)разбиения двумерного массива выбрать и как отображать рабочие нагрузки для
пространственного и временного разделения столбцов. Прошивка обеспечивает
соблюдение принятых Resource Solver решений по привязке ресурсов
«контекст — столбец(столбцы)».

Клиентские NPU AMD Phoenix и AMD Hawk Point могут поддерживать 6 одновременных
контекстов рабочих нагрузок. AMD Strix Point может поддерживать 16 одновременных
контекстов рабочих нагрузок.


Двоичные файлы приложений
=========================

Рабочая нагрузка приложения NPU состоит из двух отдельных двоичных файлов,
которые генерируются компилятором NPU.

1. Оверлей AMD XDNA Array, который используется для конфигурации
   пространственного раздела NPU. Оверлей содержит инструкции для настройки
   конфигурации stream switch и ELF для вычислительных тайлов. Оверлей
   загружается в пространственный раздел, привязанный к рабочей нагрузке,
   соответствующим экземпляром ERT. Дополнительные сведения см. в
   `Versal Adaptive SoC AIE-ML Architecture Manual (AM020)`_.

2. ``ctrlcode`` — используется для оркестрации оверлея, загруженного в
   пространственный раздел. ``ctrlcode`` выполняется ERT, работающим в защищённом
   режиме на микроконтроллере в контексте рабочей нагрузки. ``ctrlcode`` состоит
   из последовательности опкодов под названием ``XAie_TxnOpcode``. Дополнительные
   сведения см. в `AI Engine Run Time`_.


Специальные хостовые буферы
===========================

Буфер инструкций для каждого контекста
--------------------------------------

Каждый контекст рабочей нагрузки использует размещённый на хосте буфер размером
64 МБ, который отображается в память экземпляра ERT, созданного для обслуживания
рабочей нагрузки. Используемый рабочей нагрузкой ``ctrlcode`` копируется в эту
специальную память. Этот буфер защищён посредством PASID, как и все другие
входные/выходные буферы, используемые данной рабочей нагрузкой. Буфер инструкций
также отображается в пространство пользователя рабочей нагрузки.

Глобальный привилегированный буфер
----------------------------------

Кроме того, драйвер также выделяет единственный буфер для задач обслуживания,
таких как запись ошибок от MERT. Этот глобальный буфер использует глобальный
домен IOMMU и доступен только для MERT.


Высокоуровневый поток использования
===================================

Ниже приведены шаги для запуска рабочей нагрузки на AMD NPU:

1.  Скомпилировать рабочую нагрузку в оверлей и двоичный файл ``ctrlcode``.
2.  Пространство пользователя открывает контекст в драйвере и предоставляет
    оверлей.
3.  Драйвер уточняет у Resource Solver выделение набора столбцов для рабочей
    нагрузки.
4.  Затем драйвер просит MERT создать контекст на устройстве с желаемыми
    столбцами.
5.  Затем MERT создаёт экземпляр ERT. MERT также отображает буфер инструкций
    (Instruction Buffer) в память ERT.
6.  Затем пространство пользователя копирует ``ctrlcode`` в буфер инструкций.
7.  Затем пространство пользователя создаёт буфер команд с указателями на
    входной, выходной буферы и буфер инструкций; после чего отправляет буфер
    команд драйверу и засыпает в ожидании завершения.
8.  Драйвер отправляет команду по почтовому ящику в ERT.
9.  ERT *выполняет* (executes) ``ctrlcode`` в буфере инструкций.
10. Выполнение ``ctrlcode`` запускает DMA-передачи в хостовую DDR и из неё, пока
    работает AMD XDNA Array.
11. Когда ERT достигает конца ``ctrlcode``, он генерирует MSI-X, чтобы отправить
    сигнал завершения драйверу, который затем пробуждает ожидающую рабочую
    нагрузку.


Поток загрузки
==============

Драйвер amdxdna использует PSP для безопасной загрузки подписанной прошивки NPU
и запуска загрузки микроконтроллера NPU. Затем драйвер amdxdna ожидает сигнала
жизнеспособности в специальной области на BAR 0. NPU отключается во время
приостановки SoC и включается после возобновления, когда прошивка NPU
загружается повторно и рукопожатие выполняется снова.


Компоненты пространства пользователя
====================================

Компилятор
----------

Peano — это одноядерный компилятор с открытым исходным кодом на основе LLVM для
вычислительного тайла AMD XDNA Array. Peano доступен по адресу:
https://github.com/Xilinx/llvm-aie

IRON — это компилятор массивов с открытым исходным кодом для NPU на основе AMD
XDNA Array, который использует Peano в своей основе. IRON доступен по адресу:
https://github.com/Xilinx/mlir-aie

Драйвер пользовательского режима (UMD)
--------------------------------------

Стек среды выполнения XRT с открытым исходным кодом взаимодействует с драйвером
ядра amdxdna. XRT можно найти по адресу:
https://github.com/Xilinx/XRT

Прослойку (shim) XRT с открытым исходным кодом для NPU можно найти по адресу:
https://github.com/amd/xdna-driver


Операции DMA
============

Инструкции операций DMA закодированы в ``ctrlcode`` как опкод
``XAIE_IO_BLOCKWRITE``. Когда ERT выполняет ``XAIE_IO_BLOCKWRITE``,
осуществляются операции DMA между хостовой DDR и памятью L2.


Обработка ошибок
================

Когда MERT обнаруживает ошибку в AMD XDNA Array, он приостанавливает выполнение
для данного контекста рабочей нагрузки и отправляет асинхронное сообщение
драйверу по привилегированному каналу. Затем драйвер отправляет MERT указатель
на буфер для захвата состояний регистров раздела, привязанного к сбойному
контексту рабочей нагрузки. Затем драйвер декодирует ошибку, считывая содержимое
по указателю на буфер.


Телеметрия
==========

MERT может сообщать различные виды телеметрической информации, например
следующие:

* Счётчик прерываний L1
* Счётчик DMA
* Счётчик глубокого сна (Deep Sleep)
* и т. п.
