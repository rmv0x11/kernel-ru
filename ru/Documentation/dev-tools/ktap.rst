==============================================================
Kernel Test Anything Protocol (KTAP), версия 1
==============================================================

TAP, или Test Anything Protocol, — это формат описания результатов тестов,
используемый рядом проектов. Его сайт и спецификация находятся по этой `ссылке
<https://testanything.org/>`_. Ядро Linux в основном использует вывод в формате
TAP для результатов тестов. Однако фреймворки тестирования ядра имеют особые
потребности в части результатов тестов, которые не согласуются с исходной
спецификацией TAP. Поэтому был задан формат «Kernel TAP» (KTAP), расширяющий и
изменяющий TAP для поддержки этих сценариев использования. Эта спецификация
описывает общепринятый формат KTAP в том виде, в каком он применяется в ядре в
настоящее время.

Результаты тестов KTAP описывают серию тестов (которые могут быть вложенными,
т. е. тест может иметь подтесты), каждый из которых может содержать как
диагностические данные — например, строки журнала, — так и итоговый результат.
Структура и результаты тестов машиночитаемы, тогда как диагностические данные
неструктурированы и предназначены для облегчения отладки человеком.

Вывод KTAP состоит из четырёх различных типов строк:

- Строки версии
- Строки плана
- Строки результата тестового случая
- Диагностические строки

В общем случае корректный вывод KTAP должен также образовывать корректный вывод
TAP, но часть информации, в частности результаты вложенных тестов, может быть
потеряна. Также обратите внимание, что существует приостановленный черновик
спецификации TAP14; KTAP расходится с ним в нескольких местах (в частности, в
заголовке «Subtest»), которые описаны далее в этом документе там, где это
уместно.

Строки версии
-------------

Все результаты в формате KTAP начинаются со «строки версии», которая указывает,
какой версии стандарта (K)TAP соответствует результат.

Например:

- "KTAP version 1"
- "TAP version 13"
- "TAP version 14"

Обратите внимание, что в KTAP подтесты также начинаются со строки версии,
которая обозначает начало результатов вложенного теста. Этим он отличается от
TAP14, в котором используется отдельная строка «Subtest».

Хотя в дальнейшем совместимым тестам следует использовать "KTAP version 1",
ожидается, что большинство парсеров и других инструментов будут принимать и
другие перечисленные здесь версии для совместимости с существующими тестами и
фреймворками.

Строки плана
------------

План теста задаёт число тестов (или подтестов) в выводе KTAP.

Строки плана должны соответствовать формату "1..N", где N — число тестов или
подтестов. Строки плана следуют за строками версии, чтобы указать число
вложенных тестов.

Хотя существуют случаи, когда число тестов заранее неизвестно — в этом случае
план теста может быть опущен, — настоятельно рекомендуется указывать его там,
где это возможно.

Строки результата тестового случая
----------------------------------

Строки результата тестового случая указывают итоговый статус теста.
Они обязательны и должны иметь формат:

.. code-block:: none

	<result> <number> [<description>][ # [<directive>] [<diagnostic data>]]

Результат может быть либо "ok", что означает, что тестовый случай пройден, либо
"not ok", что означает, что тестовый случай не пройден.

<number> представляет номер выполняемого теста. Первый тест должен иметь номер
1, и затем номер должен увеличиваться на 1 для каждого дополнительного подтеста
внутри того же теста на том же уровне вложенности.

Описание — это описание теста, как правило имя теста, и может быть любой строкой
символов, кроме # или символа новой строки. Описание необязательно, но
рекомендуется.

Директива и любые диагностические данные необязательны. Если присутствует
что-либо из них, оно должно следовать за знаком решётки, "#".

Директива — это ключевое слово, указывающее на иной исход теста, нежели пройден
или не пройден. Директива необязательна и состоит из одного ключевого слова,
предшествующего диагностическим данным. В случае, когда парсер встречает
директиву, которую он не поддерживает, он должен вернуться к результату "ok" /
"not ok".

В настоящее время принимаются следующие директивы:

- "SKIP", которая указывает, что тест был пропущен (обратите внимание, что
  результат в строке результата тестового случая может быть как "ok", так и
  "not ok", если используется директива SKIP)
- "TODO", которая указывает, что тест в данный момент не ожидается пройденным,
  например потому что функциональность, которую он тестирует, заведомо
  неисправна. Хотя эта директива унаследована из TAP, её использование в ядре не
  поощряется.
- "XFAIL", которая указывает, что ожидается, что тест не пройдёт. Это аналогично
  "TODO" выше и используется некоторыми тестами kselftest.
- “TIMEOUT”, которая указывает, что у теста истекло время ожидания (обратите
  внимание, что результат в строке результата тестового случая должен быть “not
  ok”, если используется директива TIMEOUT)
- “ERROR”, которая указывает, что выполнение теста завершилось неудачей из-за
  конкретной ошибки, которая включена в диагностические данные. (обратите
  внимание, что результат в строке результата тестового случая должен быть “not
  ok”, если используется директива ERROR)

Диагностические данные — это поле в виде простого текста, содержащее любые
дополнительные подробности о том, почему был получен этот результат. Обычно это
сообщение об ошибке для ERROR или непройденных тестов либо описание
отсутствующих зависимостей для результата SKIP.

Поле диагностических данных необязательно, и результатам, не имеющим ни
директивы, ни каких-либо диагностических данных, не требуется включать
разделитель полей "#".

Примеры строк результата::

	ok 1 test_case_name

Тест "test_case_name" пройден.

::

	not ok 1 test_case_name

Тест "test_case_name" не пройден.

::

	ok 1 test # SKIP necessary dependency unavailable

Тест "test" был ПРОПУЩЕН с диагностическим сообщением "necessary dependency
unavailable".

::

	not ok 1 test # TIMEOUT 30 seconds

У теста "test" истекло время ожидания, с диагностическими данными "30 seconds".

::

	ok 5 check return code # rcode=0

Тест "check return code" пройден, с дополнительными диагностическими данными “rcode=0”


Диагностические строки
-----------------------

Если тесты хотят вывести любую дополнительную информацию, им следует делать это с
помощью «диагностических строк». Диагностические строки необязательны,
представляют собой текст в свободной форме и часто используются для описания
того, что тестируется, и любых промежуточных результатов более подробно, чем это
обеспечивают итоговый результат и строка диагностических данных.

Диагностические строки оформляются как "# <diagnostic_description>", где описание
может быть любой строкой. Диагностические строки могут находиться в любом месте
вывода теста. Как правило, диагностические строки, относящиеся к тесту,
располагаются непосредственно перед строкой результата для этого теста.

Обратите внимание, что большинство инструментов будут трактовать неизвестные
строки (см. ниже) как диагностические, даже если они не начинаются с "#": это
сделано для того, чтобы захватить любой другой полезный вывод ядра, который может
помочь в отладке теста. Тем не менее рекомендуется, чтобы тесты всегда
предваряли любой свой диагностический вывод символом "#".

Неизвестные строки
------------------

В выводе KTAP могут встречаться строки, которые не соответствуют ни одному из
четырёх описанных выше форматов строк. Это допускается, однако они не будут
влиять на статус тестов.

Это важное отличие от TAP. Тесты ядра могут выводить сообщения на системную
консоль или в файл журнала. Оба этих места назначения могут содержать сообщения
либо от не связанной с тестом активности ядра или пространства пользователя,
либо сообщения ядра от не относящегося к тестам кода, вызываемого тестом. Код
ядра, вызываемый тестом, скорее всего, не знает, что выполняется тест, и потому
не может вывести сообщение как диагностическое.

Вложенные тесты
---------------

В KTAP тесты могут быть вложенными. Это достигается за счёт того, что тест
включает в свой вывод целый набор результатов в формате KTAP. Это можно
использовать для категоризации и группировки связанных тестов или для разделения
различных результатов одного и того же теста.

Результат «родительского» теста должен состоять из всех результатов его
подтестов, начинаться с очередной строки версии KTAP и плана теста и
заканчиваться общим результатом. Если, например, один из подтестов не проходит,
родительский тест также должен быть непройденным.

Кроме того, все строки в подтесте должны иметь отступ. Один уровень отступа — это
два пробела: "  ". Отступ должен начинаться со строки версии и заканчиваться
перед строкой результата родительского теста.

«Неизвестные строки» не считаются строками подтеста и потому могут иметь как
отступ, так и не иметь его.

Пример теста с двумя вложенными подтестами:

::

	KTAP version 1
	1..1
	  KTAP version 1
	  1..2
	  ok 1 test_1
	  not ok 2 test_2
	# example failed
	not ok 1 example

Пример формата с несколькими уровнями вложенного тестирования:

::

	KTAP version 1
	1..2
	  KTAP version 1
	  1..2
	    KTAP version 1
	    1..2
	    not ok 1 test_1
	    ok 2 test_2
	  not ok 1 test_3
	  ok 2 test_4 # SKIP
	not ok 1 example_test_1
	ok 2 example_test_2


Основные различия между TAP и KTAP
----------------------------------

==================================================   =========  ===============
Возможность                                          TAP        KTAP
==================================================   =========  ===============
yaml и json в диагностическом сообщении              ok         not recommended
директива TODO                                       ok         not recognized
допускает произвольное число вложенных тестов        no         yes
«Неизвестные строки» в категории «Anything else»     yes        no
«Неизвестные строки»                                 incorrect  allowed
==================================================   =========  ===============

Спецификация TAP14 действительно разрешает вложенные тесты, но вместо
использования очередной вложенной строки версии использует строку вида
"Subtest: <name>", где <name> — это имя родительского теста.

Пример вывода KTAP
------------------
::

	KTAP version 1
	1..1
	  KTAP version 1
	  1..3
	    KTAP version 1
	    1..1
	    # test_1: initializing test_1
	    ok 1 test_1
	  ok 1 example_test_1
	    KTAP version 1
	    1..2
	    ok 1 test_1 # SKIP test_1 skipped
	    ok 2 test_2
	  ok 2 example_test_2
	    KTAP version 1
	    1..3
	    ok 1 test_1
	    # test_2: FAIL
	    not ok 2 test_2
	    ok 3 test_3 # SKIP test_3 skipped
	  not ok 3 example_test_3
	not ok 1 main_test

Этот вывод определяет следующую иерархию:

Единственный тест с именем "main_test", который не проходит и имеет три подтеста:

- "example_test_1", который проходит и имеет один подтест:

   - "test_1", который проходит и выводит диагностическое сообщение "test_1: initializing test_1"

- "example_test_2", который проходит и имеет два подтеста:

   - "test_1", который пропускается, с пояснением "test_1 skipped"
   - "test_2", который проходит

- "example_test_3", который не проходит и имеет три подтеста

   - "test_1", который проходит
   - "test_2", который выводит диагностическую строку "test_2: FAIL" и не проходит.
   - "test_3", который пропускается с пояснением "test_3 skipped"

Обратите внимание, что отдельные подтесты с одинаковыми именами не конфликтуют,
так как они находятся в разных родительских тестах. Этот вывод также
демонстрирует некоторые разумные правила «всплытия» результатов тестов: тест не
проходит, если не проходит любой из его подтестов. Пропущенные тесты не влияют на
результат родительского теста (хотя часто имеет смысл помечать тест как
пропущенный, если пропущены _все_ его подтесты).
