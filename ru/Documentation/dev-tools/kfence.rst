Kernel Electric-Fence (KFENCE)
==============================

Kernel Electric-Fence (KFENCE) — это детектор ошибок безопасности работы с
памятью на основе выборочной проверки с низкими накладными расходами. KFENCE
обнаруживает выход за границы кучи (out-of-bounds), использование после
освобождения (use-after-free) и ошибки некорректного освобождения
(invalid-free).

KFENCE разработан для включения в production-ядрах и имеет почти нулевые
накладные расходы на производительность. По сравнению с KASAN, KFENCE
жертвует точностью в пользу производительности. Основная идея, лежащая в
основе устройства KFENCE, состоит в том, что при достаточно большом суммарном
времени работы KFENCE будет обнаруживать ошибки в путях исполнения кода,
которые обычно не задействуются непроизводственными тестовыми нагрузками. Один
из способов быстро достичь достаточно большого суммарного времени работы —
развернуть инструмент на большом парке машин.

Использование
-------------

Чтобы включить KFENCE, сконфигурируйте ядро с::

    CONFIG_KFENCE=y

Чтобы собрать ядро с поддержкой KFENCE, но выключенным по умолчанию (для
включения задайте ``kfence.sample_interval`` ненулевым значением),
сконфигурируйте ядро с::

    CONFIG_KFENCE=y
    CONFIG_KFENCE_SAMPLE_INTERVAL=0

KFENCE предоставляет ряд других параметров конфигурации для настройки поведения
(подробнее см. соответствующий текст подсказки в ``lib/Kconfig.kfence``).

Настройка производительности
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Самый важный параметр — это интервал выборки KFENCE, который можно задать через
параметр загрузки ядра ``kfence.sample_interval`` в миллисекундах. Интервал
выборки определяет частоту, с которой выделения в куче будут защищаться
средствами KFENCE. Значение по умолчанию настраивается через опцию Kconfig
``CONFIG_KFENCE_SAMPLE_INTERVAL``. Установка ``kfence.sample_interval=0``
отключает KFENCE.

Интервал выборки управляет таймером, который настраивает выделения KFENCE. По
умолчанию, чтобы реальный интервал выборки оставался предсказуемым, обычный
таймер также вызывает пробуждения CPU, когда система полностью простаивает. Это
может быть нежелательно на системах с ограниченным энергопотреблением. Параметр
загрузки ``kfence.deferrable=1`` вместо этого переключает на «отложенный»
(deferrable) таймер, который не принуждает к пробуждениям CPU на простаивающих
системах, ценой непредсказуемости интервалов выборки. Значение по умолчанию
настраивается через опцию Kconfig ``CONFIG_KFENCE_DEFERRABLE``.

.. warning::
   Набор тестов KUnit с очень высокой вероятностью завершится неудачей при
   использовании отложенного таймера, поскольку в настоящее время он приводит
   к очень непредсказуемым интервалам выборки.

По умолчанию KFENCE выбирает только 1 выделение в куче в пределах каждого
интервала выборки. *Пакетный режим* (burst mode) позволяет выбирать
последовательные выделения в куче; параметр загрузки ядра ``kfence.burst``
можно задать ненулевым значением, обозначающим *дополнительные*
последовательные выделения в пределах интервала выборки; установка
``kfence.burst=N`` означает, что для каждого интервала выборки через KFENCE
предпринимается попытка обработать ``1 + N`` последовательных выделений.

Пул памяти KFENCE имеет фиксированный размер, и если пул исчерпан, дальнейшие
выделения KFENCE не происходят. С помощью ``CONFIG_KFENCE_NUM_OBJECTS`` (по
умолчанию 255) можно управлять количеством доступных защищённых объектов.
Каждый объект требует 2 страницы: одну для самого объекта, а другую — в
качестве сторожевой страницы (guard page); страницы объектов чередуются со
сторожевыми страницами, и поэтому каждая страница объекта окружена двумя
сторожевыми страницами.

Общий объём памяти, выделенной под пул памяти KFENCE, можно вычислить как::

    ( #objects + 1 ) * 2 * PAGE_SIZE

При использовании конфигурации по умолчанию и в предположении размера страницы
4 КиБ это приводит к выделению 2 МиБ под пул памяти KFENCE.

Примечание: на архитектурах, поддерживающих huge pages, KFENCE гарантирует, что
пул использует страницы размером ``PAGE_SIZE``. Это приведёт к выделению
дополнительных таблиц страниц.

некорректно записанные байты (со смещением от этого адреса); в этом

внимание, что реальные значения показываются только если ядро было загружено с
``no_hash_pointers``; в противном случае, во избежание раскрытия информации,
вместо них используется «!» для обозначения некорректно записанных байтов.

Интерфейс DebugFS
~~~~~~~~~~~~~~~~~

Часть отладочной информации доступна через debugfs:

* Файл ``/sys/kernel/debug/kfence/stats`` предоставляет статистику времени
  выполнения.

* Файл ``/sys/kernel/debug/kfence/objects`` предоставляет список объектов,
  выделенных средствами KFENCE, включая уже освобождённые, но защищённые.

Детали реализации
-----------------

Защищённые выделения настраиваются на основе интервала выборки. После
истечения интервала выборки следующее выделение через основной аллокатор (SLAB
или SLUB) возвращает защищённое выделение из пула объектов KFENCE
(поддерживаются размеры выделений вплоть до PAGE_SIZE). В этот момент таймер
сбрасывается, и следующее выделение настраивается после истечения интервала.

При использовании ``CONFIG_KFENCE_STATIC_KEYS=y`` выделения KFENCE
«пропускаются» через быстрый путь основного аллокатора, опираясь на статические
ветвления (static branches) посредством инфраструктуры статических ключей
(static keys). Статическое ветвление переключается, чтобы перенаправить
выделение в KFENCE. В зависимости от интервала выборки, целевых нагрузок и
архитектуры системы это может работать лучше, чем простое динамическое
ветвление. Рекомендуется тщательное тестирование производительности.

Каждый объект KFENCE располагается на выделенной странице, у левой или правой
границы страницы, выбираемой случайным образом. Страницы слева и справа от
страницы объекта являются «сторожевыми страницами» (guard pages), атрибуты
которых изменены на защищённое состояние, и которые вызывают страничные
исключения (page faults) при любой попытке доступа. Такие страничные исключения
затем перехватываются KFENCE, который аккуратно обрабатывает исключение,
сообщая о доступе с выходом за границы и помечая страницу как доступную, так
что код, вызвавший исключение, может (ошибочно) продолжить выполнение
(установите ``panic_on_warn``, чтобы вместо этого вызывать панику).

Чтобы обнаруживать записи за границы в памяти внутри самой страницы объекта,
KFENCE также использует основанные на шаблоне красные зоны (redzones). Для
каждой страницы объекта красная зона настраивается для всей памяти, не
относящейся к объекту. При типичных выравниваниях красная зона требуется только
на незащищённой стороне объекта. Поскольку KFENCE обязан соблюдать запрошенное
кэшем выравнивание, особые выравнивания могут приводить к незащищённым
промежуткам с любой стороны объекта, и все они покрываются красными зонами.

Следующий рисунок иллюстрирует размещение страниц::

    ---+-----------+-----------+-----------+-----------+-----------+---
       | xxxxxxxxx | O :       | xxxxxxxxx |       : O | xxxxxxxxx |
       | xxxxxxxxx | B :       | xxxxxxxxx |       : B | xxxxxxxxx |
       | x GUARD x | J : RED-  | x GUARD x | RED-  : J | x GUARD x |
       | xxxxxxxxx | E :  ZONE | xxxxxxxxx |  ZONE : E | xxxxxxxxx |
       | xxxxxxxxx | C :       | xxxxxxxxx |       : C | xxxxxxxxx |
       | xxxxxxxxx | T :       | xxxxxxxxx |       : T | xxxxxxxxx |
    ---+-----------+-----------+-----------+-----------+-----------+---

При освобождении объекта KFENCE страница объекта снова защищается, а объект
помечается как освобождённый. Любой дальнейший доступ к объекту вызывает
исключение, и KFENCE сообщает о доступе с использованием после освобождения.
Освобождённые объекты вставляются в хвост списка свободных объектов KFENCE
(freelist), так что наименее недавно освобождённые объекты используются
повторно первыми, и шансы обнаружить использование после освобождения недавно
освобождённых объектов повышаются.

Если использование пула достигает 75 % (по умолчанию) или выше, то, чтобы
снизить риск того, что пул в конце концов окажется полностью занят выделенными
объектами, и одновременно обеспечить разнообразное покрытие выделений, KFENCE
ограничивает текущие покрытые выделения из одного и того же источника от
дальнейшего заполнения пула. «Источник» выделения определяется на основе
частичной трассировки стека выделения. Побочный эффект состоит в том, что это
также ограничивает частые долгоживущие выделения (например, pagecache) из
одного и того же источника от постоянного заполнения пула, что является
наиболее распространённым риском того, что пул заполнится, а частота выборочных
выделений упадёт до нуля. Порог, при котором начинается ограничение текущих
покрытых выделений, можно настроить через параметр загрузки
``kfence.skip_covered_thresh`` (использование пула в %).

Сопутствующие инструменты
-------------------------

В пространстве пользователя похожий подход применяется в `GWP-ASan
<http://llvm.org/docs/GwpAsan.html>`_. GWP-ASan также опирается на сторожевые
страницы и стратегию выборки для обнаружения ошибок небезопасной работы с
памятью в больших масштабах. Устройство KFENCE напрямую заимствует идеи
GWP-ASan, и его можно рассматривать как его ядерного собрата. Другой похожий, но
невыборочный подход, который также вдохновил название «KFENCE», можно найти в
пространстве пользователя — `Electric Fence Malloc Debugger
<https://linux.die.net/man/3/efence>`_.

В ядре существует несколько инструментов для отладки ошибок доступа к памяти, и,
в частности, KASAN может обнаруживать все классы ошибок, которые может
обнаруживать KFENCE. Хотя KASAN более точен, опираясь на инструментирование
компилятором, это даётся ценой производительности.

Стоит подчеркнуть, что KASAN и KFENCE дополняют друг друга, ориентируясь на
разные целевые окружения. Например, KASAN — лучшее средство отладки там, где
существуют тестовые случаи или воспроизводящие сценарии: из-за более низкой
вероятности обнаружить ошибку использование KFENCE для отладки потребовало бы
больших усилий. Однако развёртывания в больших масштабах, которые не могут
позволить себе включить KASAN, выиграли бы от использования KFENCE для
обнаружения ошибок в путях исполнения кода, не задействованных тестовыми
случаями или фаззерами.
