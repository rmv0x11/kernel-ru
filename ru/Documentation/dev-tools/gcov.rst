Использование gcov с ядром Linux
================================

Поддержка профилирования ядра с помощью gcov позволяет использовать
инструмент тестирования покрытия gcov_ из состава GCC с ядром Linux.
Данные о покрытии работающего ядра экспортируются в формате, совместимом
с gcov, через каталог debugfs «gcov». Чтобы получить данные о покрытии для
конкретного файла, перейдите в каталог сборки ядра и используйте gcov с
опцией ``-o`` следующим образом (требуются права root)::

    # cd /tmp/linux-out
    # gcov -o /sys/kernel/debug/gcov/tmp/linux-out/kernel spinlock.c

При этом в текущем каталоге будут созданы файлы исходного кода, снабжённые
аннотациями с числом исполнений. Кроме того, графические front-end'ы для
gcov, такие как lcov_, можно использовать для автоматизации процесса сбора
данных по всему ядру и получения обзоров покрытия в формате HTML.

Возможные применения:

* отладка (была ли эта строка вообще достигнута?)
* улучшение тестов (как мне изменить тест, чтобы покрыть эти строки?)
* минимизация конфигураций ядра (нужна ли мне эта опция, если связанный
  с ней код никогда не выполняется?)

.. _gcov: https://gcc.gnu.org/onlinedocs/gcc/Gcov.html
.. _lcov: https://github.com/linux-test-project/lcov


Подготовка
----------

Сконфигурируйте ядро с параметрами::

        CONFIG_DEBUG_FS=y
        CONFIG_GCOV_KERNEL=y

а чтобы получить данные о покрытии для всего ядра::

        CONFIG_GCOV_PROFILE_ALL=y

Обратите внимание, что ядра, скомпилированные с флагами профилирования,
будут значительно крупнее и работать медленнее. Кроме того,
CONFIG_GCOV_PROFILE_ALL может поддерживаться не на всех архитектурах.

Данные профилирования станут доступны только после монтирования debugfs::

        mount -t debugfs none /sys/kernel/debug


Настройка
---------

Чтобы включить профилирование для конкретных файлов или каталогов,
добавьте в соответствующий Makefile ядра строку, подобную следующей:

- Для отдельного файла (например, main.o)::

	GCOV_PROFILE_main.o := y

- Для всех файлов в одном каталоге::

	GCOV_PROFILE := y

Чтобы исключить файлы из профилирования даже при заданном
CONFIG_GCOV_PROFILE_ALL, используйте::

	GCOV_PROFILE_main.o := n

и::

	GCOV_PROFILE := n

Этот механизм поддерживает только файлы, которые компонуются в основной
образ ядра или собираются как модули ядра.


Конфиги, специфичные для модулей
--------------------------------

Конфиги gcov ядра для конкретных модулей описаны ниже:

CONFIG_GCOV_PROFILE_RDS:
        Включает профилирование GCOV для RDS, чтобы проверять, какие
        функции или строки выполняются. Этот конфиг используется
        самотестом rds для генерации отчётов о покрытии. Если он не задан,
        отчёт пропускается.


Файлы
-----

Поддержка gcov ядра создаёт в debugfs следующие файлы:

``/sys/kernel/debug/gcov``
	Родительский каталог для всех связанных с gcov файлов.

``/sys/kernel/debug/gcov/reset``
	Глобальный файл сброса: обнуляет все данные о покрытии при
        записи в него.

``/sys/kernel/debug/gcov/path/to/compile/dir/file.gcda``
	Собственно файл данных gcov в формате, понятном инструменту
        gcov. Обнуляет данные о покрытии файла при записи в него.

``/sys/kernel/debug/gcov/path/to/compile/dir/file.gcno``
	Символьная ссылка на файл статических данных, необходимый
        инструменту gcov. Этот файл генерируется gcc при компиляции с
        опцией ``-ftest-coverage``.


Модули
------

Модули ядра могут содержать код очистки, который выполняется только во
время выгрузки модуля. Механизм gcov предоставляет средство для сбора
данных о покрытии такого кода, сохраняя копию данных, связанных с
выгруженным модулем. Эти данные остаются доступными через debugfs.
После повторной загрузки модуля связанные с ним счётчики покрытия
инициализируются данными от его предыдущего экземпляра.

Это поведение можно отключить, задав параметр ядра gcov_persist::

        gcov_persist=0

Во время выполнения пользователь также может сбросить данные для
выгруженного модуля, записав в его файл данных или в глобальный файл
сброса.


Разделённые машины сборки и тестирования
----------------------------------------

Инфраструктура профилирования ядра gcov спроектирована так, чтобы работать
«из коробки» в конфигурациях, где ядра собираются и запускаются на одной и
той же машине. В случаях, когда ядро работает на отдельной машине,
необходимо выполнить специальную подготовку, в зависимости от того, где
используется инструмент gcov:

.. _gcov-test:

a) gcov запускается на ТЕСТОВОЙ машине

    Версия инструмента gcov на тестовой машине должна быть совместима с
    версией gcc, использованной для сборки ядра. Кроме того, со сборочной
    машины на тестовую нужно скопировать следующие файлы:

    из дерева исходных кодов:
      - все исходные файлы на C + заголовки

    из дерева сборки:
      - все исходные файлы на C + заголовки
      - все файлы .gcda и .gcno
      - все ссылки на каталоги

    Важно отметить, что эти файлы необходимо разместить на тестовой машине
    точно в том же расположении файловой системы, что и на сборочной
    машине. Если какой-либо из компонентов пути является символьной
    ссылкой, вместо неё нужно использовать фактический каталог (из-за
    обработки CURDIR в make).

.. _gcov-build:

b) gcov запускается на СБОРОЧНОЙ машине

    После каждого тестового случая с тестовой машины на сборочную нужно
    скопировать следующие файлы:

    из каталога gcov в sysfs:
      - все файлы .gcda
      - все ссылки на файлы .gcno

    Эти файлы можно скопировать в любое расположение на сборочной машине.
    Затем gcov нужно вызвать с опцией -o, указывающей на этот каталог.

    Пример организации каталогов на сборочной машине::

      /tmp/linux:    дерево исходных кодов ядра
      /tmp/out:      каталог сборки ядра, заданный через make O=
      /tmp/coverage: расположение файлов, скопированных с тестовой машины

      [user@build] cd /tmp/out
      [user@build] gcov -o /tmp/coverage/tmp/out/init main.c


Замечание о компиляторах
------------------------

Инструменты gcov от GCC и LLVM не обязательно совместимы. Используйте gcov_
для работы с файлами .gcno и .gcda, сгенерированными GCC, и llvm-cov_ для
Clang.

.. _gcov: https://gcc.gnu.org/onlinedocs/gcc/Gcov.html
.. _llvm-cov: https://llvm.org/docs/CommandGuide/llvm-cov.html

Различия в сборке между gcov от GCC и Clang обрабатываются Kconfig. Он
автоматически выбирает подходящий формат gcov в зависимости от
обнаруженного тулчейна.


Устранение неполадок
--------------------

Проблема
    Компиляция прерывается на этапе компоновки.

Причина
    Флаги профилирования указаны для исходных файлов, которые не
    компонуются в основное ядро или компонуются специальной
    процедурой компоновщика.

Решение
    Исключите затронутые исходные файлы из профилирования, указав
    ``GCOV_PROFILE := n`` или ``GCOV_PROFILE_basename.o := n`` в
    соответствующем Makefile.

Проблема
    Файлы, скопированные из sysfs, оказываются пустыми или неполными.

Причина
    Из-за особенностей работы seq_file некоторые инструменты, такие
    как cp или tar, могут некорректно копировать файлы из sysfs.

Решение
    Используйте ``cat`` для чтения файлов ``.gcda`` и ``cp -d`` для
    копирования ссылок. Либо используйте механизм, показанный в
    Приложении B.


Приложение A: gather_on_build.sh
--------------------------------

Пример скрипта для сбора метафайлов покрытия на сборочной машине
(см. :ref:`Разделённые машины сборки и тестирования, п. a. <gcov-test>`):

.. code-block:: sh

    #!/bin/bash

    KSRC=$1
    KOBJ=$2
    DEST=$3

    if [ -z "$KSRC" ] || [ -z "$KOBJ" ] || [ -z "$DEST" ]; then
      echo "Usage: $0 <ksrc directory> <kobj directory> <output.tar.gz>" >&2
      exit 1
    fi

    KSRC=$(cd $KSRC; printf "all:\n\t@echo \${CURDIR}\n" | make -f -)
    KOBJ=$(cd $KOBJ; printf "all:\n\t@echo \${CURDIR}\n" | make -f -)

    find $KSRC $KOBJ \( -name '*.gcno' -o -name '*.[ch]' -o -type l \) -a \
                     -perm /u+r,g+r | tar cfz $DEST -P -T -

    if [ $? -eq 0 ] ; then
      echo "$DEST successfully created, copy to test system and unpack with:"
      echo "  tar xfz $DEST -P"
    else
      echo "Could not create file $DEST"
    fi


Приложение B: gather_on_test.sh
-------------------------------

Пример скрипта для сбора файлов данных покрытия на тестовой машине
(см. :ref:`Разделённые машины сборки и тестирования, п. b. <gcov-build>`):

.. code-block:: sh

    #!/bin/bash -e

    DEST=$1
    GCDA=/sys/kernel/debug/gcov

    if [ -z "$DEST" ] ; then
      echo "Usage: $0 <output.tar.gz>" >&2
      exit 1
    fi

    TEMPDIR=$(mktemp -d)
    echo Collecting data..
    find $GCDA -type d -exec mkdir -p $TEMPDIR/\{\} \;
    find $GCDA -name '*.gcda' -exec sh -c 'cat < $0 > '$TEMPDIR'/$0' {} \;
    find $GCDA -name '*.gcno' -exec sh -c 'cp -d $0 '$TEMPDIR'/$0' {} \;
    tar czf $DEST -C $TEMPDIR sys
    rm -rf $TEMPDIR

    echo "$DEST successfully created, copy to build system and unpack with:"
    echo "  tar xfz $DEST"
