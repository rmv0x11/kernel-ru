==================================
Тестирование устройств с kselftest
==================================


Существует несколько разных kselftest-ов для обобщённого тестирования устройств,
с некоторым перекрытием по охвату и с разными требованиями. Этот документ призван
дать обзор каждого из них.

Примечание: пути в этом документе указаны относительно каталога kselftest
(``tools/testing/selftests``).

Ориентированные на устройства kselftest-ы:

* Devicetree (``dt``)

  * **Охват**: статус probe для устройств, описанных в Devicetree
  * **Требования**: нет

* Журналы ошибок (``devices/error_logs``)

  * **Охват**: наличие сообщений в журнале уровня ошибки (или более критичных),
    исходящих от любого устройства
  * **Требования**: нет

* Обнаруживаемая шина (``devices/probe``)

  * **Охват**: наличие и статус probe устройств USB или PCI, которые были
    описаны в файле-эталоне
  * **Требования**: вручную описать устройства, которые должны быть протестированы,
    в YAML-файле эталона (пример см. в ``devices/probe/boards/google,spherion.yaml``)

* Exist (``devices/exist``)

  * **Охват**: наличие всех устройств
  * **Требования**: сгенерировать эталон (подробности см. в ``devices/exist/README.rst``)
    на заведомо исправном ядре

Поэтому рекомендуется включать тесты журналов ошибок и devicetree на всех
(основанных на DT) платформах, так как у них нет никаких требований. Затем, чтобы
значительно увеличить охват, сгенерировать эталон для каждой платформы и включить
тест exist. Тест обнаруживаемой шины можно использовать для проверки статуса probe
конкретных устройств USB или PCI, но в большинстве случаев это, вероятно, не стоит того.
