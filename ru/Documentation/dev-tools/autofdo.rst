============================================
Использование AutoFDO с ядром Linux
============================================

Это включает поддержку сборки ядра с AutoFDO при использовании
компилятора Clang. AutoFDO (Auto-Feedback-Directed Optimization) —
это разновидность profile-guided optimization (PGO), применяемая для
повышения производительности бинарных исполняемых файлов. Она собирает
информацию о частоте выполнения различных путей кода внутри бинарного
файла с помощью аппаратного сэмплирования. Затем эти данные используются
для управления решениями компилятора по оптимизации, что приводит к
более эффективному бинарному файлу. AutoFDO — это мощная техника
оптимизации, и данные показывают, что она может существенно улучшить
производительность ядра. Она особенно полезна для нагрузок, страдающих
от простоев во front-end.

Для сборок с AutoFDO, в отличие от сборок без FDO, пользователь должен
предоставить профиль. Получить профиль AutoFDO можно несколькими
способами. Профили AutoFDO создаются путём преобразования данных
аппаратного сэмплирования с помощью утилиты «perf». Крайне важно, чтобы
нагрузка, используемая для создания этих файлов perf, была
репрезентативной; она должна демонстрировать характеристики времени
выполнения, схожие с теми нагрузками, которые предполагается
оптимизировать. Несоблюдение этого приведёт к тому, что компилятор
будет оптимизировать под неверную цель.

Профиль AutoFDO зачастую инкапсулирует поведение программы. Если
критичный для производительности код не зависит от архитектуры, профиль
можно применять на разных платформах для достижения прироста
производительности. Например, использование профиля, сгенерированного на
архитектуре Intel, для сборки ядра под архитектуру AMD также может дать
улучшение производительности.

Существует два метода получения репрезентативного профиля:
(1) сэмплирование реальных нагрузок в продакшен-окружении.
(2) генерация профиля с помощью репрезентативного нагрузочного теста.
При включении конфигурации сборки AutoFDO без предоставления профиля
AutoFDO компилятор лишь модифицирует информацию dwarf в ядре, не влияя на
производительность во время выполнения. Рекомендуется использовать
бинарный файл ядра, собранный с той же конфигурацией AutoFDO, для сбора
профиля perf. Хотя можно использовать ядро, собранное с другими опциями,
это может привести к худшей производительности.

Профили можно собирать, используя сборку с AutoFDO для предыдущего ядра.
AutoFDO применяет относительные номера строк для сопоставления профилей,
обеспечивая некоторую устойчивость к изменениям исходного кода. Этот
режим обычно используется в продакшен-окружении для сбора профилей.

При сборе профиля на основе нагрузочного теста процесс сбора AutoFDO
состоит из следующих шагов:

#. Первоначальная сборка: ядро собирается с опциями AutoFDO
   без профиля.

#. Профилирование: указанное выше ядро затем запускается с
   репрезентативной нагрузкой для сбора данных о частоте выполнения. Эти
   данные собираются с помощью аппаратного сэмплирования, через perf.
   AutoFDO наиболее эффективен на платформах, поддерживающих продвинутые
   возможности PMU, такие как LBR на машинах Intel.

#. Генерация профиля AutoFDO: выходной файл perf преобразуется в
   профиль AutoFDO с помощью офлайн-инструментов.

Поддержка требует компилятора Clang LLVM 17 или более позднего.

Подготовка
==========

Сконфигурируйте ядро со следующим::

   CONFIG_AUTOFDO_CLANG=y

Настройка
=========

Настройка CONFIG_AUTOFDO_CLANG по умолчанию охватывает объекты
пространства ядра для сборок AutoFDO. Однако можно включить или отключить
сборку AutoFDO для отдельных файлов и каталогов, добавив строку,
аналогичную следующей, в соответствующий Makefile ядра:

- Для включения одного файла (например, foo.o) ::

   AUTOFDO_PROFILE_foo.o := y

- Для включения всех файлов в одном каталоге ::

   AUTOFDO_PROFILE := y

- Для отключения одного файла ::

   AUTOFDO_PROFILE_foo.o := n

- Для отключения всех файлов в одном каталоге ::

   AUTOFDO_PROFILE := n

Рабочий процесс
===============

Вот пример рабочего процесса для ядра с AutoFDO:

1)  Соберите ядро на хост-машине с включённым LLVM,
    например, ::

      $ make menuconfig LLVM=1

    Включите конфигурацию сборки AutoFDO::

      CONFIG_AUTOFDO_CLANG=y

    С конфигурацией, в которой включён LLVM, используйте следующую команду::

      $ scripts/config -e AUTOFDO_CLANG

    После получения конфигурации соберите с помощью ::

      $ make LLVM=1

2) Установите ядро на тестовую машину.

3) Запустите нагрузочные тесты. Опция '-c' в perf задаёт период события
   сэмплирования. Мы рекомендуем использовать подходящее простое число,
   например 500009, для этой цели.

   - Для платформ Intel::

      $ perf record -e BR_INST_RETIRED.NEAR_TAKEN:k -a -N -b -c <count> -o <perf_file> -- <loadtest>

   - Для платформ AMD:

     Поддерживаются следующие системы: Zen3 с BRS или Zen4 с amd_lbr_v2.
     Чтобы проверить,

     Для Zen3::

      $ cat /proc/cpuinfo | grep " brs"

     Для Zen4::

      $ cat /proc/cpuinfo | grep amd_lbr_v2

     Следующая команда сгенерировала файл данных perf::

      $ perf record --pfm-events RETIRED_TAKEN_BRANCH_INSTRUCTIONS:k -a -N -b -c <count> -o <perf_file> -- <loadtest>

4) (Необязательно) Загрузите сырой файл perf на хост-машину.

5) Для генерации профиля AutoFDO доступны два офлайн-инструмента:
   create_llvm_prof и llvm_profgen. Инструмент create_llvm_prof является
   частью проекта AutoFDO и его можно найти на GitHub
   (https://github.com/google/autofdo), версии v0.30.1 или более поздней.
   Инструмент llvm_profgen включён в сам компилятор LLVM. Важно отметить,
   что версия llvm_profgen не обязана совпадать с версией Clang. Он должен
   быть из релиза LLVM 19 для Clang или более позднего, либо просто из
   ствола (trunk) LLVM. ::

      $ llvm-profgen --kernel --binary=<vmlinux> --perfdata=<perf_file> -o <profile_file>

   или ::

      $ create_llvm_prof --binary=<vmlinux> --profile=<perf_file> --format=extbinary --out=<profile_file>

   Обратите внимание, что несколько файлов профиля AutoFDO можно
   объединить в один с помощью::

      $ llvm-profdata merge -o <profile_file> <profile_1> <profile_2> ... <profile_n>

6) Пересоберите ядро, используя файл профиля AutoFDO, с той же
   конфигурацией, что и на шаге 1
   (обратите внимание, что CONFIG_AUTOFDO_CLANG должен быть включён)::

      $ make LLVM=1 CLANG_AUTOFDO_PROFILE=<profile_file>
