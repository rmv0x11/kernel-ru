Kernel Address Sanitizer (KASAN)
================================

Обзор
-----

Kernel Address Sanitizer (KASAN) — это динамический детектор ошибок безопасности
памяти, предназначенный для обнаружения ошибок выхода за границы (out-of-bounds)
и использования после освобождения (use-after-free).

KASAN имеет три режима:

1. Generic KASAN
2. Software Tag-Based KASAN
3. Hardware Tag-Based KASAN

Generic KASAN, включаемый параметром CONFIG_KASAN_GENERIC, — это режим,
предназначенный для отладки, аналогичный ASan из пространства пользователя.
Этот режим поддерживается на многих архитектурах CPU, но имеет значительные
накладные расходы по производительности и памяти.

Software Tag-Based KASAN, или SW_TAGS KASAN, включаемый параметром
CONFIG_KASAN_SW_TAGS, может использоваться как для отладки, так и для
dogfood-тестирования, аналогично HWASan из пространства пользователя.
Этот режим поддерживается только для arm64, но его умеренные накладные расходы
по памяти позволяют использовать его для тестирования на устройствах с
ограниченной памятью под реальными нагрузками.

Hardware Tag-Based KASAN, или HW_TAGS KASAN, включаемый параметром
CONFIG_KASAN_HW_TAGS, — это режим, предназначенный для использования в качестве
полевого детектора ошибок памяти или меры по снижению рисков безопасности.
Этот режим работает только на CPU arm64, поддерживающих MTE (Memory Tagging
Extension), но имеет низкие накладные расходы по памяти и производительности и
поэтому может использоваться в продакшене.

Подробности о влиянии каждого режима KASAN на память и производительность см. в
описаниях соответствующих опций Kconfig.

Режимы Generic и Software Tag-Based обычно называют программными режимами.
Режимы Software Tag-Based и Hardware Tag-Based называют tag-based-режимами
(основанными на тегах).

Поддержка
---------

Архитектуры
~~~~~~~~~~~

Generic KASAN поддерживается на x86_64, arm, arm64, powerpc, riscv, s390, xtensa
и loongarch, а tag-based-режимы KASAN поддерживаются только на arm64.

Компиляторы
~~~~~~~~~~~

Программные режимы KASAN используют инструментирование на этапе компиляции для
вставки проверок корректности перед каждым обращением к памяти и поэтому требуют
версии компилятора, которая обеспечивает поддержку этого. Режим Hardware
Tag-Based полагается на оборудование для выполнения этих проверок, но всё равно
требует версии компилятора, поддерживающей инструкции тегирования памяти.

Generic KASAN требует GCC версии 8.3.0 или новее
либо любой версии Clang, поддерживаемой ядром.

Software Tag-Based KASAN требует GCC 11+
либо любой версии Clang, поддерживаемой ядром.

Hardware Tag-Based KASAN требует GCC 10+ или Clang 12+.

Типы памяти
~~~~~~~~~~~

Generic KASAN поддерживает обнаружение ошибок во всех видах памяти: slab,
page_alloc, vmap, vmalloc, стек и глобальная память.

Software Tag-Based KASAN поддерживает память slab, page_alloc, vmalloc и стек.

Hardware Tag-Based KASAN поддерживает память slab, page_alloc и
неисполняемую vmalloc-память.

Для slab оба программных режима KASAN поддерживают аллокаторы SLUB и SLAB, тогда
как Hardware Tag-Based KASAN поддерживает только SLUB.

Использование
-------------

Чтобы включить KASAN, сконфигурируйте ядро с::

	  CONFIG_KASAN=y

и выберите между ``CONFIG_KASAN_GENERIC`` (чтобы включить Generic KASAN),
``CONFIG_KASAN_SW_TAGS`` (чтобы включить Software Tag-Based KASAN) и
``CONFIG_KASAN_HW_TAGS`` (чтобы включить Hardware Tag-Based KASAN).

Для программных режимов также выберите между ``CONFIG_KASAN_OUTLINE`` и
``CONFIG_KASAN_INLINE``. Outline и inline — это типы инструментирования
компилятором. Первый создаёт меньший бинарный файл, тогда как второй до 2 раз
быстрее.

Чтобы включать в отчёты трассировки стека выделения (alloc) и освобождения (free)
затронутых объектов slab, включите ``CONFIG_STACKTRACE``. Чтобы включать
трассировки стека выделения и освобождения затронутых физических страниц,
включите ``CONFIG_PAGE_OWNER`` и загрузитесь с ``page_owner=on``.

Параметры загрузки
~~~~~~~~~~~~~~~~~~

На KASAN влияет общий параметр командной строки ``panic_on_warn``. Когда он
включён, KASAN вызывает панику ядра после печати отчёта об ошибке.

По умолчанию KASAN печатает отчёт об ошибке только для первого некорректного
обращения к памяти. С ``kasan_multi_shot`` KASAN печатает отчёт при каждом
некорректном обращении. Это фактически отключает ``panic_on_warn`` для отчётов
KASAN.

Альтернативно, независимо от ``panic_on_warn``, для управления поведением паники
и формирования отчётов можно использовать параметр загрузки ``kasan.fault=``:

- ``kasan.fault=report``, ``=panic`` или ``=panic_on_write`` управляет тем,
  печатать ли только отчёт KASAN, вызывать панику ядра или вызывать панику ядра
  только при некорректных записях (по умолчанию: ``report``). Паника происходит,
  даже если включён ``kasan_multi_shot``. Обратите внимание, что при
  использовании асинхронного режима Hardware Tag-Based KASAN
  ``kasan.fault=panic_on_write`` всегда вызывает панику для асинхронно
  проверяемых обращений (включая чтения).

Режимы Software и Hardware Tag-Based KASAN (см. раздел о различных режимах ниже)
поддерживают изменение поведения сбора трассировок стека:

- ``kasan.stacktrace=off`` или ``=on`` отключает или включает сбор трассировок
  стека выделения и освобождения (по умолчанию: ``on``).
- ``kasan.stack_ring_size=<число записей>`` задаёт число записей в
  кольце стека (stack ring) (по умолчанию: ``32768``).

Режим Hardware Tag-Based KASAN предназначен для использования в продакшене в
качестве меры по снижению рисков безопасности. Поэтому он поддерживает
дополнительные параметры загрузки, позволяющие полностью отключить KASAN или
управлять его возможностями:

- ``kasan=off`` или ``=on`` управляет тем, включён ли KASAN (по умолчанию:
  ``on``).

- ``kasan.mode=sync``, ``=async`` или ``=asymm`` управляет тем, настроен ли
  KASAN на синхронный, асинхронный или асимметричный режим выполнения
  (по умолчанию: ``sync``).
  Синхронный режим: некорректное обращение обнаруживается немедленно при
  возникновении сбоя проверки тега (tag check fault).
  Асинхронный режим: обнаружение некорректного обращения откладывается. При
  возникновении сбоя проверки тега информация сохраняется в оборудовании
  (в регистре TFSR_EL1 для arm64). Ядро периодически проверяет оборудование и
  сообщает о сбоях тегов только во время этих проверок.
  Асимметричный режим: некорректное обращение обнаруживается синхронно при
  чтениях и асинхронно при записях.

- ``kasan.write_only=off`` или ``kasan.write_only=on`` управляет тем, проверяет
  ли KASAN только обращения на запись (store) или все обращения (по умолчанию:
  ``off``).

- ``kasan.vmalloc=off`` или ``=on`` отключает или включает тегирование
  vmalloc-выделений (по умолчанию: ``on``).

- ``kasan.page_alloc.sample=<интервал выборки>`` заставляет KASAN тегировать
  только каждое N-е page_alloc-выделение с порядком (order), равным или большим
  ``kasan.page_alloc.sample.order``, где N — значение параметра ``sample``
  (по умолчанию: ``1``, или тегировать каждое такое выделение).
  Этот параметр предназначен для снижения накладных расходов на
  производительность, вносимых KASAN.
  Обратите внимание, что включение этого параметра заставляет Hardware Tag-Based
  KASAN пропускать проверки выделений, выбранных выборкой, и тем самым пропускать
  некорректные обращения к этим выделениям. Используйте значение по умолчанию для
  точного обнаружения ошибок.

- ``kasan.page_alloc.sample.order=<минимальный порядок страницы>`` задаёт
  минимальный порядок (order) выделений, на которые влияет выборка (по умолчанию:
  ``3``).
  Применяется только когда ``kasan.page_alloc.sample`` установлен в значение
  больше ``1``.
  Этот параметр предназначен для того, чтобы выборка применялась только к крупным
  page_alloc-выделениям, которые являются крупнейшим источником накладных
  расходов на производительность.

Заголовок отчёта суммирует, какого рода ошибка произошла и какого рода обращение
её вызвало. За ним следуют трассировка стека некорректного обращения, трассировка
стека места, где была выделена память, к которой произошло обращение (в случае
обращения к объекту slab), и трассировка стека места, где объект был освобождён
(в случае отчёта об ошибке use-after-free). Далее идёт описание объекта slab, к
которому произошло обращение, и информация о странице памяти, к которой
произошло обращение.

В конце отчёт показывает состояние памяти вокруг адреса, к которому произошло
обращение. Внутренне KASAN отслеживает состояние памяти отдельно для каждой
гранулы памяти, которая составляет либо 8, либо 16 выровненных байтов в
зависимости от режима KASAN. Каждое число в разделе состояния памяти отчёта
показывает состояние одной из гранул памяти, окружающих адрес, к которому
произошло обращение.

Для Generic KASAN размер каждой гранулы памяти равен 8. Состояние каждой гранулы
кодируется в одном теневом байте (shadow byte). Эти 8 байтов могут быть
доступными, частично доступными, освобождёнными или частью redzone. KASAN
использует следующее кодирование для каждого теневого байта: 00 означает, что
все 8 байтов соответствующей области памяти доступны; число N (1 <= N <= 7)
означает, что первые N байтов доступны, а остальные (8 - N) байтов — нет; любое
отрицательное значение указывает, что всё 8-байтовое слово недоступно. KASAN
использует разные отрицательные значения, чтобы различать разные виды
недоступной памяти, такие как redzone или освобождённая память (см. mm/kasan/kasan.h).

В отчёте выше стрелка указывает на теневой байт ``03``, что означает, что адрес,
к которому произошло обращение, частично доступен.

Для tag-based-режимов KASAN этот последний раздел отчёта показывает теги памяти
вокруг адреса, к которому произошло обращение (см. раздел `Детали реализации`_).

Обратите внимание, что заголовки ошибок KASAN (такие как ``slab-out-of-bounds``
или ``use-after-free``) формируются по принципу наилучшего предположения: KASAN
печатает наиболее вероятный тип ошибки на основе той ограниченной информации,
которой он располагает. Фактический тип ошибки может отличаться.

Generic KASAN также сообщает до двух вспомогательных трассировок стека вызовов.
Эти трассировки стека указывают на места в коде, которые взаимодействовали с
объектом, но которые напрямую не присутствуют в трассировке стека некорректного
обращения. В настоящее время сюда входят call_rcu() и постановка в очередь
workqueue.

CONFIG_KASAN_EXTRA_INFO
~~~~~~~~~~~~~~~~~~~~~~~

Включение CONFIG_KASAN_EXTRA_INFO позволяет KASAN записывать и сообщать больше
информации. В настоящее время поддерживается такая дополнительная информация, как
номер CPU и временна́я метка на момент выделения и освобождения. Больше информации
может помочь найти причину ошибки и соотнести ошибку с другими событиями системы
ценой использования дополнительной памяти для записи большего объёма информации
(подробнее о цене см. в тексте справки CONFIG_KASAN_EXTRA_INFO).

Вот отчёт с включённым CONFIG_KASAN_EXTRA_INFO (показаны только
различающиеся части)::

    ...
    Allocated by task 134 on cpu 5 at 229.133855s:
    ...
    Freed by task 136 on cpu 3 at 230.199335s:
    ...
    ==================================================================

Детали реализации
-----------------

Generic KASAN
~~~~~~~~~~~~~

Программные режимы KASAN используют теневую память (shadow memory), чтобы
записывать, безопасен ли каждый байт памяти для обращения, и используют
инструментирование на этапе компиляции для вставки проверок теневой памяти перед
каждым обращением к памяти.

Generic KASAN выделяет 1/8 памяти ядра под свою теневую память (16 ТБ для
покрытия 128 ТБ на x86_64) и использует прямое отображение с масштабом (scale) и
смещением (offset) для трансляции адреса памяти в соответствующий ему теневой
адрес.

Вот функция, которая транслирует адрес в соответствующий ему теневой адрес::

    static inline void *kasan_mem_to_shadow(const void *addr)
    {
	return (void *)((unsigned long)addr >> KASAN_SHADOW_SCALE_SHIFT)
		+ KASAN_SHADOW_OFFSET;
    }

где ``KASAN_SHADOW_SCALE_SHIFT = 3``.

Инструментирование на этапе компиляции используется для вставки проверок
обращений к памяти. Компилятор вставляет вызовы функций (``__asan_load*(addr)``,
``__asan_store*(addr)``) перед каждым обращением к памяти размером 1, 2, 4, 8 или
16. Эти функции проверяют, корректны ли обращения к памяти, проверяя
соответствующую теневую память.

При inline-инструментировании компилятор вместо вызовов функций напрямую
вставляет код для проверки теневой памяти. Эта опция значительно увеличивает
ядро, но даёт прирост производительности в x1.1—x2 по сравнению с
outline-инструментированным ядром.

Generic KASAN — единственный режим, который задерживает повторное использование
освобождённых объектов через карантин (quarantine) (реализацию см. в
mm/kasan/quarantine.c).

Software Tag-Based KASAN
~~~~~~~~~~~~~~~~~~~~~~~~

Software Tag-Based KASAN использует подход программного тегирования памяти для
проверки корректности обращений. В настоящее время он реализован только для
архитектуры arm64.

Software Tag-Based KASAN использует возможность Top Byte Ignore (TBI) CPU arm64
для хранения тега указателя в старшем байте указателей ядра. Он использует
теневую память для хранения тегов памяти, связанных с каждой 16-байтовой ячейкой
памяти (поэтому он выделяет 1/16 памяти ядра под теневую память).

При каждом выделении памяти Software Tag-Based KASAN генерирует случайный тег,
тегирует выделенную память этим тегом и встраивает тот же тег в возвращаемый
указатель.

Software Tag-Based KASAN использует инструментирование на этапе компиляции для
вставки проверок перед каждым обращением к памяти. Эти проверки убеждаются, что
тег памяти, к которой происходит обращение, равен тегу указателя, используемого
для обращения к этой памяти. В случае несовпадения тегов Software Tag-Based KASAN
печатает отчёт об ошибке.

Software Tag-Based KASAN также имеет два режима инструментирования (outline,
который испускает callback-вызовы для проверки обращений к памяти; и inline,
который выполняет проверки теневой памяти inline). В режиме
outline-инструментирования отчёт об ошибке печатается из функции, выполняющей
проверку обращения. При inline-инструментировании компилятор испускает инструкцию
``brk``, и для печати отчётов об ошибках используется выделенный обработчик
``brk``.

Software Tag-Based KASAN использует 0xFF в качестве тега указателя «match-all»
(обращения через указатели с тегом указателя 0xFF не проверяются). Значение 0xFE
в настоящее время зарезервировано для тегирования освобождённых областей памяти.

Hardware Tag-Based KASAN
~~~~~~~~~~~~~~~~~~~~~~~~

Hardware Tag-Based KASAN по концепции аналогичен программному режиму, но
использует аппаратную поддержку тегирования памяти вместо инструментирования
компилятором и теневой памяти.

Hardware Tag-Based KASAN в настоящее время реализован только для архитектуры
arm64 и основан как на arm64 Memory Tagging Extension (MTE), представленном в
архитектуре набора команд ARMv8.5, так и на Top Byte Ignore (TBI).

Для назначения тегов памяти каждому выделению используются специальные инструкции
arm64. Те же теги назначаются указателям на эти выделения. При каждом обращении к
памяти оборудование убеждается, что тег памяти, к которой происходит обращение,
равен тегу указателя, используемого для обращения к этой памяти. В случае
несовпадения тегов генерируется сбой (fault), и печатается отчёт.

Hardware Tag-Based KASAN использует 0xFF в качестве тега указателя «match-all»
(обращения через указатели с тегом указателя 0xFF не проверяются). Значение 0xFE
в настоящее время зарезервировано для тегирования освобождённых областей памяти.

Если оборудование не поддерживает MTE (до ARMv8.5), Hardware Tag-Based KASAN не
будет включён. В этом случае все параметры загрузки KASAN игнорируются.

Обратите внимание, что включение CONFIG_KASAN_HW_TAGS всегда приводит к включению
TBI внутри ядра. Даже когда передан ``kasan.mode=off`` или когда оборудование не
поддерживает MTE (но поддерживает TBI).

Hardware Tag-Based KASAN сообщает только о первой найденной ошибке. После этого
проверка тегов MTE отключается.

Теневая память
--------------

Содержимое этого раздела применимо только к программным режимам KASAN.

Ядро отображает память в нескольких разных частях адресного пространства.
Диапазон виртуальных адресов ядра велик: реальной памяти недостаточно для
поддержки реальной теневой области для каждого адреса, к которому могло бы
обратиться ядро. Поэтому KASAN отображает реальную теневую память только для
определённых частей адресного пространства.

Поведение по умолчанию
~~~~~~~~~~~~~~~~~~~~~~~

По умолчанию архитектуры отображают реальную память поверх теневой области только
для линейного отображения (и, возможно, других небольших областей). Для всех
остальных областей — таких как пространство vmalloc и vmemmap — поверх теневой
области отображается единственная страница только для чтения. Эта теневая
страница только для чтения объявляет все обращения к памяти разрешёнными.

Это создаёт проблему для модулей: они располагаются не в линейном отображении, а
в выделенном пространстве модулей. Перехватывая вызовы аллокатора модулей, KASAN
временно отображает реальную теневую память для их покрытия. Это позволяет,
например, обнаруживать некорректные обращения к глобальным переменным модулей.

Это также создаёт несовместимость с ``VMAP_STACK``: если стек располагается в
пространстве vmalloc, он будет затенён страницей только для чтения, и ядро вызовет
сбой при попытке настроить теневые данные для переменных стека.

CONFIG_KASAN_VMALLOC
~~~~~~~~~~~~~~~~~~~~

С ``CONFIG_KASAN_VMALLOC`` KASAN может покрывать пространство vmalloc ценой
большего использования памяти. В настоящее время это поддерживается на x86,
arm64, riscv, s390 и powerpc.

Это работает за счёт перехвата вызовов vmalloc и vmap и динамического выделения
реальной теневой памяти для обеспечения этих отображений.

Большинство отображений в пространстве vmalloc невелики и требуют менее полной
страницы теневого пространства. Поэтому выделение полной теневой страницы на
каждое отображение было бы расточительным. Кроме того, чтобы гарантировать, что
разные отображения используют разные теневые страницы, отображения пришлось бы
выравнивать по ``KASAN_GRANULE_SIZE * PAGE_SIZE``.

Вместо этого KASAN разделяет резервное пространство между несколькими
отображениями. Он выделяет резервную страницу, когда отображение в пространстве
vmalloc использует определённую страницу теневой области. Эта страница может
позже совместно использоваться другими vmalloc-отображениями.

KASAN перехватывает вызовы инфраструктуры vmap, чтобы лениво очищать
неиспользуемую теневую память.

Чтобы избежать сложностей, связанных с перестановкой отображений, KASAN ожидает,
что часть теневой области, покрывающая пространство vmalloc, не будет покрыта
ранней теневой страницей (early shadow page), а будет оставлена неотображённой.
Это потребует изменений в коде, специфичном для архитектуры.

Это позволяет поддерживать ``VMAP_STACK`` на x86 и может упростить поддержку
архитектур, не имеющих фиксированной области модулей.

Для разработчиков
-----------------

Игнорирование обращений
~~~~~~~~~~~~~~~~~~~~~~~~

Программные режимы KASAN используют инструментирование компилятором для вставки
проверок корректности. Такое инструментирование может быть несовместимо с
некоторыми частями ядра и поэтому должно быть отключено.

Другие части ядра могут обращаться к метаданным выделенных объектов. Обычно KASAN
обнаруживает такие обращения и сообщает о них, но в некоторых случаях (например,
в аллокаторах памяти) эти обращения корректны.

Для программных режимов KASAN, чтобы отключить инструментирование для конкретного
файла или каталога, добавьте аннотацию ``KASAN_SANITIZE`` в соответствующий
Makefile ядра:

- Для отдельного файла (например, main.o)::

    KASAN_SANITIZE_main.o := n

- Для всех файлов в одном каталоге::

    KASAN_SANITIZE := n

Для программных режимов KASAN, чтобы отключить инструментирование на уровне
отдельной функции, используйте специфичный для KASAN атрибут функции
``__no_sanitize_address`` или общий ``noinstr``.

Обратите внимание, что отключение инструментирования компилятором (на уровне
отдельного файла или отдельной функции) заставляет KASAN игнорировать обращения,
которые происходят непосредственно в этом коде, для программных режимов KASAN.
Это не помогает, когда обращения происходят опосредованно (через вызовы
инструментированных функций) или с Hardware Tag-Based KASAN, который не использует
инструментирование компилятором.

Для программных режимов KASAN, чтобы отключить отчёты KASAN в части кода ядра для
текущей задачи, оберните эту часть кода секцией
``kasan_disable_current()``/``kasan_enable_current()``. Это также отключает отчёты
для опосредованных обращений, происходящих через вызовы функций.

Для tag-based-режимов KASAN, чтобы отключить проверку обращений, используйте
``kasan_reset_tag()`` или ``page_kasan_tag_reset()``. Обратите внимание, что
временное отключение проверки обращений через ``page_kasan_tag_reset()`` требует
сохранения и восстановления постраничного тега KASAN через
``page_kasan_tag``/``page_kasan_tag_set``.

Тесты
~~~~~

Существуют тесты KASAN, позволяющие убедиться, что KASAN работает и может
обнаруживать определённые типы повреждений памяти.

Все тесты KASAN интегрированы с KUnit Test Framework и могут быть включены через
``CONFIG_KASAN_KUNIT_TEST``. Тесты можно запускать и частично проверять
автоматически несколькими разными способами; см. инструкции ниже.

Каждый тест KASAN печатает один из нескольких отчётов KASAN, если обнаружена
ошибка. Затем тест печатает свой номер и статус.

Когда тест проходит::

        ok 28 - kmalloc_double_kzfree

Когда тест не проходит из-за неудавшегося ``kmalloc``::

        # kmalloc_large_oob_right: ASSERTION FAILED at mm/kasan/kasan_test.c:245
        Expected ptr is not null, but is
        not ok 5 - kmalloc_large_oob_right

Когда тест не проходит из-за отсутствующего отчёта KASAN::

        # kmalloc_double_kzfree: EXPECTATION FAILED at mm/kasan/kasan_test.c:709
        KASAN failure expected in "kfree_sensitive(ptr)", but none occurred
        not ok 28 - kmalloc_double_kzfree


В конце печатается совокупный статус всех тестов KASAN. При успехе::

        ok 1 - kasan

Или, если один из тестов не прошёл::

        not ok 1 - kasan

Есть несколько способов запустить тесты KASAN.

1. Загружаемый модуль

   С включённым ``CONFIG_KUNIT`` тесты можно собрать как загружаемый модуль и
   запустить, загрузив ``kasan_test.ko`` с помощью ``insmod`` или ``modprobe``.

2. Встроенный

   С ``CONFIG_KUNIT``, собранным встроенным образом, тесты также можно собрать
   встроенными. В этом случае тесты будут запускаться при загрузке как
   late-init-вызов.

3. С помощью kunit_tool

   С ``CONFIG_KUNIT`` и ``CONFIG_KASAN_KUNIT_TEST``, собранными встроенным
   образом, также возможно использовать ``kunit_tool``, чтобы увидеть результаты
   тестов KUnit в более удобочитаемом виде. Это не будет печатать отчёты KASAN
   тех тестов, которые прошли.
   См. `документацию KUnit <https://www.kernel.org/doc/html/latest/dev-tools/kunit/index.html>`_
   для более актуальной информации о ``kunit_tool``.

.. _KUnit: https://www.kernel.org/doc/html/latest/dev-tools/kunit/index.html
