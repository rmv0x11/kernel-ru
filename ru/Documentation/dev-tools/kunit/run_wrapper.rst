==================================
Запуск тестов с помощью kunit_tool
==================================

Тесты KUnit можно запускать либо с помощью kunit_tool, либо вручную,
а затем использовать kunit_tool для разбора результатов. Запуск тестов
вручную описан в: Documentation/dev-tools/kunit/run_manual.rst.
Пока мы способны собрать ядро, мы можем запускать KUnit.

kunit_tool — это Python-скрипт, который настраивает и собирает ядро, запускает
тесты и форматирует их результаты.

Команда запуска:

.. code-block::

	./tools/testing/kunit/kunit.py run

Мы должны увидеть следующее:

.. code-block::

	Configuring KUnit Kernel ...
	Building KUnit kernel...
	Starting KUnit kernel...

Возможно, нам понадобятся следующие опции:

.. code-block::

	./tools/testing/kunit/kunit.py run --timeout=30 --jobs=`nproc --all`

- ``--timeout`` задаёт максимальное время выполнения тестов.
- ``--jobs`` задаёт количество потоков для сборки ядра.

kunit_tool сгенерирует ``.kunitconfig`` с конфигурацией по умолчанию,
если никакого другого файла ``.kunitconfig`` не существует
(в каталоге сборки). Кроме того, он проверяет, что сгенерированный
файл ``.config`` содержит опции ``CONFIG`` из ``.kunitconfig``.
Также возможно передать kunit_tool отдельный фрагмент ``.kunitconfig``.
Это полезно, если у нас есть несколько различных групп тестов, которые
мы хотим запускать независимо, или если мы хотим использовать заранее
определённые конфигурации тестов для отдельных подсистем.

Чтобы использовать другой файл ``.kunitconfig`` (например, предоставленный
для тестирования конкретной подсистемы), передайте его как опцию:

.. code-block::

	./tools/testing/kunit/kunit.py run --kunitconfig=fs/ext4/.kunitconfig

Чтобы просмотреть флаги kunit_tool (необязательные аргументы командной строки), выполните:

.. code-block::

	./tools/testing/kunit/kunit.py run --help

Создание файла ``.kunitconfig``
===============================

Если мы хотим запустить определённый набор тестов (а не те, что перечислены
в ``defconfig`` для KUnit), мы можем указать опции Kconfig в файле
``.kunitconfig``. Файл .kunitconfig по умолчанию см. в:
https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/tools/testing/kunit/configs/default.config.
``.kunitconfig`` — это ``minconfig`` (файл .config,
сгенерированный командой ``make savedefconfig``), используемый для запуска
определённого набора тестов. Этот файл содержит обычные конфигурации ядра
с конкретными целями для тестирования. ``.kunitconfig`` также
содержит любые другие опции конфигурации, необходимые для тестов (например:
зависимости тестируемых возможностей, конфигурации, которые включают/отключают
отдельные блоки кода, конфигурации архитектуры и так далее).

Чтобы создать ``.kunitconfig`` на основе ``defconfig`` для KUnit:

.. code-block::

	cd $PATH_TO_LINUX_REPO
	cp tools/testing/kunit/configs/default.config .kunit/.kunitconfig

Затем мы можем добавить любые другие опции Kconfig. Например:

.. code-block::

	CONFIG_LIST_KUNIT_TEST=y

kunit_tool гарантирует, что все опции конфигурации из ``.kunitconfig``
заданы в ``.config`` ядра перед запуском тестов. Он выдаёт предупреждение, если мы
не включили зависимости опций.

.. note:: Удаление чего-либо из ``.kunitconfig`` не приведёт
   к пересборке ``.config file``. Конфигурация обновляется только
   в том случае, если ``.kunitconfig`` не является подмножеством ``.config``.
   Это означает, что мы можем использовать другие инструменты
   (например: ``make menuconfig``) для настройки других опций конфигурации.
   Для работы ``make menuconfig`` необходимо задать каталог сборки,
   поэтому по умолчанию используйте ``make O=.kunit menuconfig``.

Настройка, сборка и запуск тестов
=================================

Если мы хотим внести ручные изменения в процесс сборки KUnit, мы
можем выполнить часть процесса сборки KUnit независимо.
При запуске kunit_tool из ``.kunitconfig`` мы можем сгенерировать
``.config``, используя аргумент ``config``:

.. code-block::

	./tools/testing/kunit/kunit.py config

Чтобы собрать ядро KUnit из текущего ``.config``, мы можем использовать
аргумент ``build``:

.. code-block::

	./tools/testing/kunit/kunit.py build

Если мы уже собрали ядро UML со встроенными тестами KUnit, мы
можем запустить ядро и отобразить результаты тестов с помощью аргумента
``exec``:

.. code-block::

	./tools/testing/kunit/kunit.py exec

Команда ``run``, рассмотренная в разделе: **Запуск тестов с помощью kunit_tool**,
эквивалентна последовательному выполнению трёх приведённых выше команд.

Разбор результатов тестов
=========================

Вывод тестов KUnit отображает результаты в формате TAP (Test Anything Protocol).
При запуске тестов kunit_tool разбирает этот вывод и печатает
сводку. Чтобы увидеть сырые результаты тестов в формате TAP, мы можем передать
аргумент ``--raw_output``:

.. code-block::

	./tools/testing/kunit/kunit.py run --raw_output

Если у нас есть результаты KUnit в сыром формате TAP, мы можем разобрать их и
напечатать удобочитаемую сводку с помощью команды ``parse`` для
kunit_tool. Она принимает имя файла в качестве аргумента или будет читать со
стандартного ввода.

.. code-block:: bash

	# Reading from a file
	./tools/testing/kunit/kunit.py parse /var/log/dmesg
	# Reading from stdin
	dmesg | ./tools/testing/kunit/kunit.py parse

Фильтрация тестов
=================

Передав фильтр в стиле glob-шаблонов bash командам ``exec`` или ``run``,
мы можем запустить подмножество тестов, встроенных в ядро. Например:
если мы хотим запустить только тесты ресурсов KUnit, используйте:

.. code-block::

	./tools/testing/kunit/kunit.py run 'kunit-resource*'

При этом используется стандартный формат glob с символами подстановки.

Запуск тестов на QEMU
=====================

kunit_tool поддерживает запуск тестов на qemu, а также
через UML. Для запуска тестов на qemu по умолчанию требуются два флага:

- ``--arch``: выбирает коллекцию конфигураций (Kconfig, опции конфигурации qemu
  и так далее), которые позволяют запускать тесты KUnit на указанной
  архитектуре минимальным образом. Аргумент архитектуры совпадает с
  именем опции, передаваемой в переменную ``ARCH``, используемую Kbuild.
  Не все архитектуры в настоящее время поддерживают этот флаг, но мы можем использовать
  ``--qemu_config`` для его обработки. Если передано ``um`` (или этот флаг
  проигнорирован), тесты будут выполняться через UML. Не-UML-архитектуры,
  например: i386, x86_64, arm и так далее, выполняются на qemu.

  ``--arch help`` выводит список всех допустимых значений ``--arch``.

- ``--cross_compile``: задаёт тулчейн Kbuild. Передаёт тот же
  аргумент, что и передаваемый в переменную ``CROSS_COMPILE``, используемую
  Kbuild. Напоминаем, что это будет префикс для бинарных файлов тулчейна,
  таких как GCC. Например:

  - ``sparc64-linux-gnu``, если на нашей системе установлен тулчейн sparc.

  - ``$HOME/toolchains/microblaze/gcc-9.2.0-nolibc/microblaze-linux/bin/microblaze-linux``,
    если мы скачали тулчейн microblaze с сайта 0-day
    в каталог в нашем домашнем каталоге под названием toolchains.

Это означает, что для большинства архитектур запуск под qemu столь же прост, как:

.. code-block:: bash

	./tools/testing/kunit/kunit.py run --arch=x86_64

При кросс-компиляции нам, вероятно, понадобится указать другой тулчейн, например:

.. code-block:: bash

	./tools/testing/kunit/kunit.py run \
		--arch=s390 \
		--cross_compile=s390x-linux-gnu-

Если мы хотим запускать тесты KUnit на архитектуре, не поддерживаемой
флагом ``--arch``, или хотим запускать тесты KUnit на qemu с использованием
конфигурации, отличной от стандартной, то мы можем написать свой собственный ``QemuConfig``.
Эти ``QemuConfigs`` пишутся на Python. В начале файла у них есть строка импорта
``from..qemu_config import QemuArchParams``.
Файл должен содержать переменную с именем ``QEMU_ARCH``, которой присвоен
экземпляр ``QemuArchParams``. См. пример в:
``tools/testing/kunit/qemu_configs/x86_64.py``.

После того как у нас есть ``QemuConfig``, мы можем передать его в kunit_tool,
используя флаг ``--qemu_config``. При использовании этот флаг заменяет
флаг ``--arch``. Например: при использовании
``tools/testing/kunit/qemu_configs/x86_64.py`` вызов выглядит
так:

.. code-block:: bash

	./tools/testing/kunit/kunit.py run \
		--timeout=60 \
		--jobs=12 \
		--qemu_config=./tools/testing/kunit/qemu_configs/x86_64.py

Аргументы командной строки
==========================

kunit_tool имеет ряд других аргументов командной строки, которые могут
быть полезны для нашего тестового окружения. Ниже приведены наиболее часто используемые
аргументы командной строки:

- ``--help``: выводит список всех доступных опций. Чтобы вывести общие опции,
  поместите ``--help`` перед командой. Чтобы вывести опции, специфичные для этой
  команды, поместите ``--help`` после команды.

  .. note:: Разные команды (``config``, ``build``, ``run`` и т.д.)
            поддерживают разные опции.
- ``--build_dir``: задаёт каталог сборки kunit_tool. Он включает в себя
  файлы ``.kunitconfig``, ``.config`` и скомпилированное ядро.

- ``--make_options``: задаёт дополнительные опции для передачи make при
  компиляции ядра (с помощью команд ``build`` или ``run``). Например:
  чтобы включить предупреждения компилятора, мы можем передать ``--make_options W=1``.

- ``--alltests``: включает предопределённый набор опций, чтобы собрать
  как можно больше тестов.

  .. note:: Список включаемых опций можно найти в
            ``tools/testing/kunit/configs/all_tests.config``.

            Если вы хотите включить только все тесты с уже удовлетворёнными
            зависимостями, добавьте вместо этого ``CONFIG_KUNIT_ALL_TESTS=y`` в ваш
            ``.kunitconfig``.

- ``--kunitconfig``: задаёт путь или каталог файла ``.kunitconfig``.
  Например:

  - ``lib/kunit/.kunitconfig`` может быть путём к файлу.

  - ``lib/kunit`` может быть каталогом, в котором расположен файл.

  Этот файл используется для сборки и запуска с предопределённым набором тестов
  и их зависимостями. Например, для запуска тестов заданной подсистемы.

- ``--kconfig_add``: задаёт дополнительные опции конфигурации, которые будут
  добавлены в конец файла ``.kunitconfig``. Например:

  .. code-block::

	./tools/testing/kunit/kunit.py run --kconfig_add CONFIG_KASAN=y

- ``--arch``: запускает тесты на указанной архитектуре. Аргумент архитектуры
  совпадает с переменной окружения Kbuild ARCH.
  Например, i386, x86_64, arm, um и т.д. Не-UML-архитектуры выполняются на qemu.
  Значение по умолчанию — `um`.

- ``--cross_compile``: задаёт тулчейн Kbuild. Передаёт тот же
  аргумент, что и передаваемый в переменную ``CROSS_COMPILE``, используемую
  Kbuild. Это будет префикс для бинарных файлов тулчейна,
  таких как GCC. Например:

  - ``sparc64-linux-gnu-``, если на нашей системе установлен тулчейн sparc.

  - ``$HOME/toolchains/microblaze/gcc-9.2.0-nolibc/microblaze-linux/bin/microblaze-linux``,
    если мы скачали тулчейн microblaze с сайта 0-day
    по указанному пути в нашем домашнем каталоге под названием toolchains.

- ``--qemu_config``: задаёт путь к файлу, содержащему
  определение пользовательской архитектуры qemu. Это должен быть python-файл,
  содержащий объект `QemuArchParams`.

- ``--qemu_args``: задаёт дополнительные аргументы qemu, например, ``-smp 8``.

- ``--jobs``: задаёт количество заданий (команд), выполняемых одновременно.
  По умолчанию это значение равно числу ядер в вашей системе.

- ``--timeout``: задаёт максимальное число секунд, отведённое на выполнение всех тестов.
  Оно не включает время, затраченное на сборку тестов.

- ``--kernel_args``: задаёт дополнительные аргументы командной строки ядра. Может повторяться.

- ``--run_isolated``: если задан, загружает ядро для каждого отдельного набора/теста.
  Это полезно для отладки негерметичного теста — такого, который
  может пройти или не пройти в зависимости от того, что выполнялось до него.

- ``--raw_output``: если задан, генерирует неотформатированный вывод ядра. Возможные опции:

   - ``all``: чтобы просмотреть полный вывод ядра, используйте ``--raw_output=all``.

   - ``kunit``: это опция по умолчанию, фильтрующая вывод до вывода KUnit. Используйте ``--raw_output`` или ``--raw_output=kunit``.

- ``--json``: если задан, сохраняет результаты тестов в формате JSON и печатает в `stdout` или
  сохраняет в файл, если указано имя файла.

- ``--filter``: задаёт фильтры по атрибутам тестов, например, ``speed!=slow``.
  Несколько фильтров можно использовать, обернув ввод в кавычки и разделив фильтры
  запятыми. Пример: ``--filter "speed>slow, module=example"``.

- ``--filter_action``: если задано значение ``skip``, отфильтрованные тесты будут показаны как пропущенные
  в выводе вместо отсутствия вывода.

- ``--list_tests``: если задан, выводит список всех тестов, которые будут запущены.

- ``--list_tests_attr``: если задан, выводит список всех тестов, которые будут запущены, и все их
  атрибуты.

- ``--list_suites``: если задан, выводит список всех наборов, которые будут запущены.

Автодополнение командной строки
===============================

kunit_tool поставляется со скриптом автодополнения для bash:

.. code-block:: bash

	source tools/testing/kunit/kunit-completion.sh
