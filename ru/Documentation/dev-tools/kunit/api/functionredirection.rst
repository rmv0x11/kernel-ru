===========================
API перенаправления функций
===========================

Обзор
=====

При написании модульных тестов важно иметь возможность изолировать тестируемый
код от остальных частей ядра. Это обеспечивает надёжность теста (на него не будут
влиять внешние факторы), снижает зависимость от конкретного оборудования или
параметров конфигурации (упрощая запуск теста) и защищает стабильность остальной
части системы (снижая вероятность того, что специфичное для теста состояние будет
мешать работе остальной системы).

Если для некоторого кода (как правило, обобщённых структур данных, вспомогательных
функций и других «чистых функций») это тривиально, то для другого (например,
драйверов устройств, файловых систем, базовых подсистем) код тесно связан с
остальными частями ядра.

Эта связанность часто так или иначе обусловлена глобальным состоянием: будь то
глобальный список устройств, файловая система или некоторое состояние оборудования.
Тестам нужно либо тщательно управлять состоянием, изолировать и восстанавливать его,
либо они могут полностью избежать этого, заменяя доступ к этому состоянию и его
изменение «поддельным» или «фиктивным» (mock) вариантом.

Это достигается рефакторингом доступа к такому состоянию, например путём введения
слоя косвенности, который может использовать или эмулировать отдельный набор
тестового состояния. Однако такой рефакторинг имеет свою цену (а проведение
значительного рефакторинга прежде, чем появится возможность писать тесты, не
оптимально).

Более простой способ перехватить и заменить некоторые из вызовов функций — это
использовать перенаправление функций с помощью статических заглушек (static stubs).


Статические заглушки
=====================

Статические заглушки — это способ перенаправления вызовов одной функции
(«настоящей» функции) на другую функцию («замещающую» функцию).

Это работает за счёт добавления в «настоящую» функцию макроса, который проверяет,
выполняется ли тест и доступна ли замещающая функция. Если да, то вместо исходной
вызывается эта функция.

Использование статических заглушек довольно простое:

1. Добавьте макрос KUNIT_STATIC_STUB_REDIRECT() в начало «настоящей» функции.

   Это должно быть первым оператором в функции, после всех объявлений переменных.
   KUNIT_STATIC_STUB_REDIRECT() принимает имя функции, за которым следуют все
   аргументы, переданные настоящей функции.

   Например:

   .. code-block:: c

	void send_data_to_hardware(const char *str)
	{
		KUNIT_STATIC_STUB_REDIRECT(send_data_to_hardware, str);
		/* real implementation */
	}

2. Напишите одну или несколько замещающих функций.

   Эти функции должны иметь ту же сигнатуру, что и настоящая функция. В случае,
   если им нужно получить доступ к специфичному для теста состоянию или изменить
   его, они могут использовать kunit_get_current_test() для получения указателя
   struct kunit. Затем его можно передать в макросы ожиданий/утверждений или
   использовать для поиска ресурсов KUnit.

   Например:

   .. code-block:: c

	void fake_send_data_to_hardware(const char *str)
	{
		struct kunit *test = kunit_get_current_test();
		KUNIT_EXPECT_STREQ(test, str, "Hello World!");
	}

3. Активируйте статическую заглушку из вашего теста.

   Внутри теста перенаправление можно включить с помощью
   kunit_activate_static_stub(), которая принимает указатель struct kunit,
   настоящую функцию и замещающую функцию. Вы можете вызывать её несколько раз с
   разными замещающими функциями, чтобы подменять реализации функции.

   В нашем примере это будет так:

   .. code-block:: c

	kunit_activate_static_stub(test,
				   send_data_to_hardware,
				   fake_send_data_to_hardware);

4. Вызовите (возможно, косвенно) настоящую функцию.

   После того как перенаправление активировано, любой вызов настоящей функции
   будет вместо этого вызывать замещающую функцию. Такие вызовы могут быть скрыты
   глубоко в реализации другой функции, но должны происходить из kthread теста.

   Например:

   .. code-block:: c

	send_data_to_hardware("Hello World!"); /* Succeeds */
	send_data_to_hardware("Something else"); /* Fails the test. */

5. (Необязательно) Отключите заглушку.

   Когда она больше не нужна, отключите перенаправление (и тем самым возобновите
   исходное поведение «настоящей» функции) с помощью
   kunit_deactivate_static_stub(). В противном случае оно будет автоматически
   отключено при завершении теста.

   Например:

   .. code-block:: c

	kunit_deactivate_static_stub(test, send_data_to_hardware);


Также эти замещающие функции можно использовать для проверки того, вызывается ли
функция вообще, например:

.. code-block:: c

	void send_data_to_hardware(const char *str)
	{
		KUNIT_STATIC_STUB_REDIRECT(send_data_to_hardware, str);
		/* real implementation */
	}

	/* In test file */
	int times_called = 0;
	void fake_send_data_to_hardware(const char *str)
	{
		times_called++;
	}
	...
	/* In the test case, redirect calls for the duration of the test */
	kunit_activate_static_stub(test, send_data_to_hardware, fake_send_data_to_hardware);

	send_data_to_hardware("hello");
	KUNIT_EXPECT_EQ(test, times_called, 1);

	/* Can also deactivate the stub early, if wanted */
	kunit_deactivate_static_stub(test, send_data_to_hardware);

	send_data_to_hardware("hello again");
	KUNIT_EXPECT_EQ(test, times_called, 1);
