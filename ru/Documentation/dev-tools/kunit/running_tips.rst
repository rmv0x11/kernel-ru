============================================
Советы по запуску тестов KUnit
============================================

Использование ``kunit.py run`` («инструмент kunit»)
===================================================

Запуск из любого каталога
-------------------------

Бывает удобно создать bash-функцию вроде такой:

.. code-block:: bash

	function run_kunit() {
	  ( cd "$(git rev-parse --show-toplevel)" && ./tools/testing/kunit/kunit.py run "$@" )
	}

.. note::
	Ранние версии ``kunit.py`` (до 5.6) не работали, если их не запускали из
	корня дерева ядра, поэтому здесь используются подоболочка и ``cd``.

Запуск подмножества тестов
--------------------------

``kunit.py run`` принимает необязательный аргумент-glob для фильтрации тестов.
Формат — ``"<suite_glob>[.test_glob]"``.

Допустим, мы хотим запустить тесты sysctl, сделать это можно так:

.. code-block:: bash

	$ echo -e 'CONFIG_KUNIT=y\nCONFIG_KUNIT_ALL_TESTS=y' > .kunit/.kunitconfig
	$ ./tools/testing/kunit/kunit.py run 'sysctl*'

Можно сузить выборку только до тестов «write»:

.. code-block:: bash

	$ echo -e 'CONFIG_KUNIT=y\nCONFIG_KUNIT_ALL_TESTS=y' > .kunit/.kunitconfig
	$ ./tools/testing/kunit/kunit.py run 'sysctl*.*write*'

При таком подходе мы расплачиваемся сборкой большего числа тестов, чем нужно, но
это проще, чем возиться с файлами ``.kunitconfig`` или комментировать
``kunit_suite``.

Однако если требуется определить набор тестов менее спонтанным образом, пригодится
следующий совет.

Определение набора тестов
-------------------------

``kunit.py run`` (как и ``build``, и ``config``) поддерживает флаг
``--kunitconfig``. Поэтому, если у вас есть набор тестов, который нужно регулярно
запускать (особенно если у них есть другие зависимости), вы можете создать для них
отдельный ``.kunitconfig``.

Например, у kunit есть такой для своих тестов:

.. code-block:: bash

	$ ./tools/testing/kunit/kunit.py run --kunitconfig=lib/kunit/.kunitconfig

Кроме того, если вы придерживаетесь соглашения об именовании файла
``.kunitconfig``, можно просто передать каталог, например:

.. code-block:: bash

	$ ./tools/testing/kunit/kunit.py run --kunitconfig=lib/kunit

.. note::
	Это относительно новая возможность (5.12+), поэтому у нас пока нет
	соглашений о том, какие файлы следует фиксировать в репозитории, а какие
	просто держать локально. Решать вам и вашему сопровождающему (maintainer),
	достаточно ли полезна конфигурация, чтобы её отправлять (и, следовательно,
	затем сопровождать).

.. note::
	Иметь фрагменты ``.kunitconfig`` одновременно в родительском и дочернем
	каталогах сомнительно. Обсуждается добавление в эти файлы инструкции «import»,
	чтобы можно было иметь конфигурацию верхнего уровня, запускающую тесты из всех
	дочерних каталогов. Но это означало бы, что файлы ``.kunitconfig`` перестанут
	быть просто простыми фрагментами .config.

	Альтернативой могло бы стать автоматическое рекурсивное объединение конфигураций
	инструментом kunit, но тесты теоретически могут зависеть от несовместимых
	опций, поэтому корректно обработать это было бы непросто.

Задание параметров командной строки ядра
----------------------------------------

Для передачи произвольных аргументов ядру можно использовать ``--kernel_args``,
например:

.. code-block:: bash

	$ ./tools/testing/kunit/kunit.py run --kernel_args=param=42 --kernel_args=param2=false


Генерация отчётов о покрытии кода под UML
-----------------------------------------

.. note::
	TODO(brendanhiggins@google.com): Существуют различные проблемы с UML и
	версиями gcc 7 и выше. Вы, скорее всего, столкнётесь с отсутствующими файлами
	``.gcda`` или с ошибками компиляции.

Это отличается от «обычного» способа получения информации о покрытии, который
описан в Documentation/dev-tools/gcov.rst.

Вместо включения ``CONFIG_GCOV_KERNEL=y`` мы можем задать следующие опции:

.. code-block:: none

	CONFIG_DEBUG_KERNEL=y
	CONFIG_DEBUG_INFO=y
	CONFIG_DEBUG_INFO_DWARF_TOOLCHAIN_DEFAULT=y
	CONFIG_GCOV=y


Собирая всё это в последовательность команд, которую можно скопировать и вставить:

.. code-block:: bash

	# Дописать опции покрытия к текущей конфигурации
	$ ./tools/testing/kunit/kunit.py run --kunitconfig=.kunit/ --kunitconfig=tools/testing/kunit/configs/coverage_uml.config
	# Извлечь информацию о покрытии из каталога сборки (.kunit/)
	$ lcov -t "my_kunit_tests" -o coverage.info -c -d .kunit/

	# Дальше процесс такой же, как и с CONFIG_GCOV_KERNEL=y
	# Например, можно сгенерировать HTML-отчёт во временном каталоге так:
	$ genhtml -o /tmp/coverage_html coverage.info


Если установленная у вас версия gcc не работает, можно подправить шаги:

.. code-block:: bash

	$ ./tools/testing/kunit/kunit.py run --make_options=CC=/usr/bin/gcc-6
	$ lcov -t "my_kunit_tests" -o coverage.info -c -d .kunit/ --gcov-tool=/usr/bin/gcov-6

Кроме того, можно использовать тулчейны (toolchain) на основе LLVM:

.. code-block:: bash

	# Сборка с LLVM и добавление опций покрытия к текущей конфигурации
	$ ./tools/testing/kunit/kunit.py run --make_options LLVM=1 --kunitconfig=.kunit/ --kunitconfig=tools/testing/kunit/configs/coverage_uml.config
	$ llvm-profdata merge -sparse default.profraw -o default.profdata
	$ llvm-cov export --format=lcov .kunit/vmlinux -instr-profile default.profdata > coverage.info
	# Файл coverage.info имеет формат, совместимый с lcov, и его можно использовать, например, для генерации HTML-отчёта
	$ genhtml -o /tmp/coverage_html coverage.info


Запуск тестов вручную
=====================

Запуск тестов без использования ``kunit.py run`` также является важным сценарием.
В настоящее время это единственный вариант, если вы хотите тестировать на
архитектурах, отличных от UML.

Поскольку запуск тестов под UML довольно прост (сконфигурировать и скомпилировать
ядро, запустить двоичный файл ``./linux``), этот раздел сосредоточен на
тестировании архитектур, отличных от UML.


Запуск встроенных тестов
------------------------

При установке тестов в ``=y`` тесты будут выполняться как часть загрузки и
печатать результаты в dmesg в формате TAP. Поэтому вам нужно лишь добавить свои
тесты в ``.config``, собрать и загрузить ядро как обычно.

Так, если бы мы скомпилировали наше ядро с:

.. code-block:: none

	CONFIG_KUNIT=y
	CONFIG_KUNIT_EXAMPLE_TEST=y

то увидели бы в dmesg вывод вроде этого, сигнализирующий, что тест выполнился и
прошёл:

.. code-block:: none

	TAP version 14
	1..1
	    # Subtest: example
	    1..1
	    # example_simple_test: initializing
	    ok 1 - example_simple_test
	ok 1 - example

Запуск тестов в виде модулей
----------------------------

В зависимости от тестов вы можете собирать их как загружаемые модули.

Например, мы бы изменили опции конфигурации, приведённые ранее, на

.. code-block:: none

	CONFIG_KUNIT=y
	CONFIG_KUNIT_EXAMPLE_TEST=m

Затем, загрузившись в наше ядро, мы можем запустить тест через

.. code-block:: none

	$ modprobe kunit-example-test

После этого он напечатает вывод TAP в stdout.

.. note::
	``modprobe`` *не* вернёт ненулевой код завершения, если какой-либо тест
	завершился неудачно (по состоянию на 5.13). А вот ``kunit.py parse`` вернёт,
	см. ниже.

.. note::
	Вы также можете задать ``CONFIG_KUNIT=m``, однако при этом некоторые
	возможности не будут работать, и потому некоторые тесты могут сломаться. В
	идеале тесты должны указывать в своих ``Kconfig``, что они зависят от
	``KUNIT=y``, но это крайний случай, о котором большинство авторов тестов не
	задумываются.
	По состоянию на 5.13 единственное отличие в том, что ``current->kunit_test``
	не будет существовать.

Красивый вывод результатов
--------------------------

Вы можете использовать ``kunit.py parse`` для разбора dmesg в поисках вывода
тестов и печати результатов в том же привычном формате, что и ``kunit.py run``.

.. code-block:: bash

	$ ./tools/testing/kunit/kunit.py parse /var/log/dmesg


Получение результатов по каждому набору тестов
----------------------------------------------

Независимо от того, как вы запускаете тесты, вы можете включить
``CONFIG_KUNIT_DEBUGFS``, чтобы предоставлять результаты по каждому набору в
формате TAP:

.. code-block:: none

	CONFIG_KUNIT=y
	CONFIG_KUNIT_EXAMPLE_TEST=m
	CONFIG_KUNIT_DEBUGFS=y

Результаты для каждого набора будут доступны по пути
``/sys/kernel/debug/kunit/<suite>/results``.
Итак, используя нашу примерную конфигурацию:

.. code-block:: bash

	$ modprobe kunit-example-test > /dev/null
	$ cat /sys/kernel/debug/kunit/example/results
	... <TAP output> ...

	# После выгрузки модуля соответствующие файлы исчезнут
	$ modprobe -r kunit-example-test
	$ cat /sys/kernel/debug/kunit/example/results
	/sys/kernel/debug/kunit/example/results: No such file or directory

Генерация отчётов о покрытии кода
---------------------------------

Подробности о том, как это сделать, см. в Documentation/dev-tools/gcov.rst.

Единственный смутно специфичный для KUnit совет здесь — что вам, вероятно, стоит
собирать свои тесты как модули. Так вы сможете изолировать покрытие от тестов от
покрытия другого кода, выполняемого во время загрузки, например:

.. code-block:: bash

	# Сбросить счётчики покрытия перед запуском теста.
	$ echo 0 > /sys/kernel/debug/gcov/reset
	$ modprobe kunit-example-test


Атрибуты тестов и фильтрация
============================

Наборы тестов и отдельные тесты можно помечать атрибутами тестов, такими как
скорость теста. Эти атрибуты позже будут печататься в выводе теста и могут
использоваться для фильтрации выполнения тестов.

Назначение атрибутов тестам
---------------------------

Тесты помечаются атрибутом путём включения объекта ``kunit_attributes`` в
определение теста.

Тесты можно помечать с помощью макроса ``KUNIT_CASE_ATTR(test_name, attributes)``
для определения теста вместо ``KUNIT_CASE(test_name)``.

.. code-block:: c

	static const struct kunit_attributes example_attr = {
		.speed = KUNIT_VERY_SLOW,
	};

	static struct kunit_case example_test_cases[] = {
		KUNIT_CASE_ATTR(example_test, example_attr),
	};

.. note::
	Чтобы пометить тест как медленный, можно также использовать
	``KUNIT_CASE_SLOW(test_name)``. Это удобный макрос, так как атрибут «slow»
	используется чаще всего.

Наборы тестов можно помечать атрибутом, задав поле «attr» в определении набора.

.. code-block:: c

	static const struct kunit_attributes example_attr = {
		.speed = KUNIT_VERY_SLOW,
	};

	static struct kunit_suite example_test_suite = {
		...,
		.attr = example_attr,
	};

.. note::
	Не все атрибуты нужно задавать в объекте ``kunit_attributes``. Незаданные
	атрибуты останутся неинициализированными и будут вести себя так, как если бы
	атрибут был установлен в 0 или NULL. Таким образом, если атрибут установлен в
	0, он трактуется как незаданный. Эти незаданные атрибуты не будут
	сообщаться и могут выступать значением по умолчанию для целей фильтрации.

Сообщение об атрибутах
----------------------

Когда пользователь запускает тесты, атрибуты будут присутствовать в необработанном
выводе ядра (в формате KTAP). Обратите внимание, что атрибуты по умолчанию скрыты в
выводе kunit.py для всех проходящих тестов, но к необработанному выводу ядра можно
получить доступ с помощью флага ``--raw_output``. Вот пример того, как атрибуты
тестов для отдельных тестов будут форматироваться в выводе ядра:

.. code-block:: none

	# example_test.speed: slow
	ok 1 example_test

Вот пример того, как атрибуты тестов для наборов тестов будут форматироваться в
выводе ядра:

.. code-block:: none

	  KTAP version 2
	  # Subtest: example_suite
	  # module: kunit_example_test
	  1..3
	  ...
	ok 1 example_suite

Кроме того, пользователи могут вывести полный отчёт об атрибутах тестов с их
атрибутами, используя флаг командной строки ``--list_tests_attr``:

.. code-block:: bash

	kunit.py run "example" --list_tests_attr

.. note::
	Доступ к этому отчёту можно получить при запуске KUnit вручную, передав
	module_param ``kunit.action=list_attr``.

Фильтрация
----------

Пользователи могут фильтровать тесты с помощью флага командной строки
``--filter`` при запуске тестов. В качестве примера:

.. code-block:: bash

	kunit.py run --filter speed=slow


Вы также можете использовать в фильтрах следующие операции: «<», «>», «<=», «>=»,
«!=» и «=». Пример:

.. code-block:: bash

	kunit.py run --filter "speed>slow"

В этом примере будут запущены все тесты со скоростью выше, чем slow. Обратите
внимание, что символы < и > часто интерпретируются оболочкой, поэтому их может
потребоваться заключить в кавычки или экранировать, как показано выше.

Кроме того, можно использовать несколько фильтров одновременно. Просто разделите
фильтры запятыми. Пример:

.. code-block:: bash

	kunit.py run --filter "speed>slow, module=kunit_example_test"

.. note::
	Вы можете использовать эту возможность фильтрации при запуске KUnit вручную,
	передав фильтр как параметр модуля: ``kunit.filter="speed>slow, speed<=normal"``.

Отфильтрованные тесты не будут выполняться и не появятся в выводе тестов. Вы
можете использовать флаг ``--filter_action=skip``, чтобы вместо этого пропускать
отфильтрованные тесты. Эти тесты будут показаны в выводе тестов, но не будут
выполняться. Чтобы использовать эту возможность при запуске KUnit вручную,
используйте параметр модуля ``kunit.filter_action=skip``.

Правила процедуры фильтрации
----------------------------

Поскольку и наборы, и отдельные тесты могут иметь атрибуты, при фильтрации могут
возникать конфликты между атрибутами. Процесс фильтрации следует этим правилам:

- Фильтрация всегда работает на уровне отдельного теста.

- Если у теста задан атрибут, то фильтрация выполняется по значению теста.

- В противном случае значение откатывается к значению набора.

- Если не задано ни то, ни другое, у атрибута есть глобальное значение «по
  умолчанию», которое и используется.

Список текущих атрибутов
------------------------

``speed``

Этот атрибут указывает скорость выполнения теста (насколько тест медленный или
быстрый).

Этот атрибут сохраняется как enum со следующими категориями: «normal», «slow» или
«very_slow». Предполагаемая скорость по умолчанию для тестов — «normal». Это
означает, что тест занимает относительно ничтожное количество времени (менее
1 секунды) независимо от машины, на которой он выполняется. Любой тест медленнее
этого может быть помечен как «slow» или «very_slow».

Макрос ``KUNIT_CASE_SLOW(test_name)`` можно легко использовать, чтобы задать
скорость теста как «slow».

``module``

Этот атрибут указывает имя модуля, связанного с тестом.

Этот атрибут автоматически сохраняется как строка и печатается для каждого набора.
Тесты также можно фильтровать по этому атрибуту.

``is_init``

Этот атрибут указывает, использует ли тест данные или функции инициализации (init).

Этот атрибут автоматически сохраняется как булево значение, и тесты также можно
фильтровать по этому атрибуту.
