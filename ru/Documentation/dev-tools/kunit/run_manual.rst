=================================
Запуск тестов без kunit_tool
=================================

Если мы не хотим использовать kunit_tool (например: мы хотим интегрироваться
с другими системами или запускать тесты на реальном оборудовании), мы можем
включить KUnit в любое ядро, считать результаты и разобрать их вручную.

.. note:: KUnit не предназначен для использования в производственной системе.
          Не исключено, что тесты могут снизить стабильность или безопасность
          системы.

Настройка ядра
==============

Тесты KUnit могут запускаться без kunit_tool. Это может быть полезно, если:

- У нас уже есть готовая конфигурация ядра, которую нужно протестировать.
- Требуется запуск на реальном оборудовании (или с использованием эмулятора/VM,
  не поддерживаемого kunit_tool).
- Есть желание интегрироваться с какими-либо существующими системами тестирования.

KUnit настраивается с помощью опции ``CONFIG_KUNIT``, а отдельные тесты также
можно собрать, включив соответствующие опции конфигурации в нашем
``.config``. Тесты KUnit обычно (но не всегда) имеют опции конфигурации,
оканчивающиеся на ``_KUNIT_TEST``. Большинство тестов можно собрать либо как
модуль, либо встроить в ядро.

.. note ::

	Мы можем включить опцию конфигурации ``KUNIT_ALL_TESTS``, чтобы
	автоматически включить все тесты с удовлетворёнными зависимостями. Это
	хороший способ быстро протестировать всё, что применимо к текущей
	конфигурации.

	KUnit можно включить или отключить во время загрузки, и это поведение
	управляется параметром ядра kunit.enable.
	По умолчанию kunit.enable установлен в 1, поскольку KUNIT_DEFAULT_ENABLED
	включён по умолчанию. Чтобы убедиться, что тесты выполняются как ожидается,
	проверьте, что во время загрузки kunit.enable=1.

После того как мы собрали наше ядро (и/или модули), запустить тесты несложно.
Если тесты встроены, они запустятся автоматически при загрузке ядра. Результаты
будут записаны в журнал ядра (``dmesg``) в формате TAP.

Если тесты собраны как модули, они запустятся при загрузке модуля.

.. code-block :: bash

	# modprobe example-test

Результаты появятся в формате TAP в ``dmesg``.

debugfs
=======

Доступ к KUnit можно получить из пространства пользователя (userspace) через
файловую систему debugfs (подробнее о debugfs см. в
Documentation/filesystems/debugfs.rst).

Если включена опция ``CONFIG_KUNIT_DEBUGFS``, файловая система debugfs KUnit
монтируется в /sys/kernel/debug/kunit. Эту файловую систему можно использовать
для выполнения следующих действий.

Получение результатов тестов
============================

Для получения результатов тестов KUnit можно использовать debugfs. Результаты
тестов доступны из файловой системы debugfs в следующем файле, доступном только
для чтения:

.. code-block :: bash

	/sys/kernel/debug/kunit/<test_suite>/results

Результаты тестов выводятся в документе KTAP. Обратите внимание, что этот
документ отделён от журнала ядра и потому может иметь иную нумерацию наборов
тестов.

Запуск тестов после загрузки ядра
=================================

Файловую систему debugfs можно использовать для запуска встроенных тестов после
загрузки. Чтобы запустить набор тестов, можно воспользоваться следующей командой
для записи в файл ``/sys/kernel/debug/kunit/<test_suite>/run``:

.. code-block :: bash

	echo "any string" > /sys/kernel/debugfs/kunit/<test_suite>/run

В результате набор тестов запускается, и результаты выводятся в журнал ядра.

Однако эта возможность недоступна для наборов KUnit, использующих init-данные,
поскольку init-данные могут быть отброшены после загрузки ядра. Наборы KUnit,
использующие init-данные, должны определяться с помощью макроса
kunit_test_init_section_suites().

Кроме того, эту возможность нельзя использовать для параллельного запуска тестов.
Вместо этого тест будет ожидать своего запуска до тех пор, пока другие тесты не
завершатся успешно или неуспешно.

.. note ::

	Авторам тестов: для использования этой возможности тестам потребуется
	корректно инициализировать и/или очищать любые данные, чтобы тест корректно
	выполнялся во второй раз.
