==================================
Стиль и номенклатура тестов
==================================

Чтобы поиск, написание и использование KUnit-тестов были максимально простыми,
настоятельно рекомендуется именовать и писать их в соответствии с приведёнными
ниже рекомендациями. Хотя писать KUnit-тесты, не следующие этим правилам,
возможно, такие тесты могут нарушить работу некоторых инструментов, могут
конфликтовать с другими тестами и могут не запускаться автоматически системами
тестирования.

Отклоняться от этих рекомендаций рекомендуется только в следующих случаях:

1. Перенос на KUnit тестов, которые уже известны под существующим именем.
2. Написание тестов, которые при автоматическом запуске вызвали бы серьёзные
   проблемы. Например, недетерминированно дающих ложноположительные или
   ложноотрицательные результаты либо выполняющихся продолжительное время.

Подсистемы, наборы и тесты
==========================

Чтобы тесты было легко находить, они группируются в наборы и подсистемы. Набор
тестов — это группа тестов, проверяющих связанную область ядра. Подсистема — это
множество наборов тестов, проверяющих различные части подсистемы ядра или
драйвера.

Подсистемы
----------

Каждый набор тестов должен принадлежать какой-либо подсистеме. Подсистема — это
совокупность одного или нескольких наборов KUnit-тестов, которые проверяют один и
тот же драйвер или часть ядра. Тестовая подсистема должна соответствовать одному
модулю ядра. Если тестируемый код не может быть скомпилирован как модуль, во
многих случаях подсистема должна соответствовать каталогу в дереве исходного кода
или записи в файле ``MAINTAINERS``. Если вы не уверены, следуйте соглашениям,
заданным тестами в аналогичных областях.

Тестовые подсистемы следует именовать по тестируемому коду: либо по модулю (там,
где это возможно), либо по тестируемому каталогу или файлам. Тестовые подсистемы
следует именовать так, чтобы при необходимости избежать неоднозначности.

Если имя тестовой подсистемы состоит из нескольких компонентов, их следует
разделять символами подчёркивания. *Не* включайте «test» или «kunit»
непосредственно в имя подсистемы, если только мы действительно не тестируем
другие тесты или сам фреймворк kunit. Например, подсистемы могут называться так:

``ext4``
  Совпадает с именем модуля и файловой системы.
``apparmor``
  Совпадает с именем модуля и именем LSM.
``kasan``
  Общепринятое имя инструмента, заметная часть пути ``mm/kasan``
``snd_hda_codec_hdmi``
  Состоит из нескольких компонентов (``snd``, ``hda``, ``codec``, ``hdmi``),
  разделённых символами подчёркивания. Совпадает с именем модуля.

Избегайте имён, подобных приведённым в примерах ниже:

``linear-ranges``
  В именах для разделения слов следует использовать символы подчёркивания, а не
  дефисы. Предпочтительнее ``linear_ranges``.
``qos-kunit-test``
  В этом имени следует использовать символы подчёркивания и не должно быть суффикса
  «kunit-test». ``qos`` также является неоднозначным именем подсистемы, поскольку
  несколько частей ядра имеют подсистему ``qos``. ``power_qos`` было бы лучшим
  именем.
``pc_parallel_port``
  Соответствующее имя модуля — ``parport_pc``, поэтому эту подсистему также
  следует назвать ``parport_pc``.

.. note::
        KUnit API и инструменты не имеют явного представления о подсистемах. Это
        способ категоризации наборов тестов и именования модулей, обеспечивающий
        простой и последовательный для человека способ находить и запускать
        тесты. В будущем это может измениться.

Наборы
------

KUnit-тесты группируются в наборы тестов, которые охватывают конкретную область
тестируемой функциональности. Наборы тестов могут иметь общий код инициализации и
завершения, который выполняется для всех тестов в наборе. Не все подсистемы нужно
разбивать на несколько наборов тестов (например, простые драйверы).

Наборы тестов именуются по подсистеме, частью которой они являются. Если
подсистема содержит несколько наборов, к имени подсистемы следует добавить
конкретную тестируемую область, отделив её символом подчёркивания.

В случае если в пределах подсистемы используется несколько типов тестов на KUnit
(например, как модульные тесты, так и интеграционные тесты), их следует разместить
в отдельных наборах, указав тип теста последним элементом в имени набора. Если эти
тесты в действительности не присутствуют, избегайте использования ``_test``,
``_unittest`` и подобного в имени набора.

Полное имя набора тестов (включая имя подсистемы) должно быть указано в поле
``.name`` структуры ``kunit_suite`` и служит основой для имени модуля. Например,
наборы тестов могут включать:

``ext4_inode``
  Часть подсистемы ``ext4``, тестирующая область ``inode``.
``kunit_try_catch``
  Часть самой реализации ``kunit``, тестирующая область ``try_catch``.
``apparmor_property_entry``
  Часть подсистемы ``apparmor``, тестирующая область ``property_entry``.
``kasan``
  Подсистема ``kasan`` имеет только один набор, поэтому имя набора совпадает с
  именем подсистемы.

Избегайте имён, например:

``ext4_ext4_inode``
  Нет причин указывать подсистему дважды.
``property_entry``
  Имя набора неоднозначно без имени подсистемы.
``kasan_integration_test``
  Поскольку в подсистеме ``kasan`` есть только один набор, его следует называть
  просто ``kasan``. Не добавляйте избыточно ``integration_test``. Это должен быть
  отдельный набор тестов. Например, если добавляются модульные тесты, то такой
  набор можно назвать ``kasan_unittest`` или подобным образом.

Тестовые случаи
----------------

Отдельные тесты состоят из одной функции, которая тестирует ограниченный путь
выполнения, свойство или функцию. В выводе тестов результаты отдельного теста
будут отображаться как подтесты результатов набора.

Тесты следует именовать по тому, что они тестируют. Часто это имя тестируемой
функции с описанием тестируемого входа или пути выполнения. Поскольку тесты
являются функциями на C, их следует именовать и писать в соответствии со стилем
кодирования ядра.

.. note::
        Поскольку тесты сами являются функциями, их имена не могут конфликтовать с
        другими идентификаторами C в ядре. Это может потребовать некоторой
        изобретательности в именовании. Хорошей идеей будет сделать ваши тестовые
        функции `static`, чтобы не засорять глобальное пространство имён.

Примеры имён тестов:

``unpack_u32_with_null_name``
  Тестирует функцию ``unpack_u32``, когда передаётся имя NULL.
``test_list_splice``
  Тестирует макрос ``list_splice``. Имеет префикс ``test_``, чтобы избежать
  конфликта имён с самим макросом.


Если потребуется сослаться на тест вне контекста его набора тестов,
*полностью квалифицированное* имя теста должно состоять из имени набора, за
которым следует имя теста, разделённых двоеточием (т. е. ``suite:test``).

Записи Kconfig для тестов
=========================

Каждый набор тестов должен быть привязан к записи Kconfig.

Эта запись Kconfig должна:

* быть названа ``CONFIG_<name>_KUNIT_TEST``, где <name> — имя набора тестов.
* располагаться либо рядом с записями config для тестируемого драйвера/подсистемы,
  либо в разделе [Kernel Hacking]->[Kernel Testing and Coverage]
* зависеть от ``CONFIG_KUNIT``.
* быть видимой только если ``CONFIG_KUNIT_ALL_TESTS`` не включён.
* иметь значение по умолчанию ``CONFIG_KUNIT_ALL_TESTS``.
* содержать краткое описание KUnit в тексте справки.

Если мы не можем выполнить указанные выше условия (например, тест невозможно
собрать как модуль), записи Kconfig для тестов должны быть tristate.

Например, запись Kconfig может выглядеть так:

.. code-block:: none

	config FOO_KUNIT_TEST
		tristate "KUnit test for foo" if !KUNIT_ALL_TESTS
		depends on KUNIT
		default KUNIT_ALL_TESTS
		help
		  This builds unit tests for foo.

		  For more information on KUnit and unit tests in general,
		  please refer to the KUnit documentation in Documentation/dev-tools/kunit/.

		  If unsure, say N.


Имена файлов тестов и модулей
=============================

KUnit-тесты часто компилируются как отдельный модуль. Чтобы избежать конфликтов с
обычными модулями, KUnit-модули следует именовать по набору тестов, за которым
следует ``_kunit`` (например, если «foobar» — основной модуль, то «foobar_kunit» —
KUnit-модуль теста).

Файлы исходного кода тестов, независимо от того, компилируются ли они как отдельный
модуль или через ``#include`` в другом файле исходного кода, лучше всего хранить в
подкаталоге ``tests/``, чтобы не конфликтовать с другими файлами исходного кода
(например, для автодополнения по табуляции).

Обратите внимание, что суффикс ``_test`` также использовался в некоторых
существующих тестах. Суффикс ``_kunit`` предпочтительнее, так как он делает
различие между KUnit- и не-KUnit-тестами более ясным.

Поэтому в типичном случае называйте файл, содержащий набор тестов,
``tests/<suite>_kunit.c``. Каталог ``tests`` следует размещать на том же уровне,
что и тестируемый код. Например, тесты для ``lib/string.c`` находятся в
``lib/tests/string_kunit.c``.

Если имя набора содержит часть или всё имя родительского каталога теста, может
иметь смысл изменить имя файла исходного кода, чтобы уменьшить избыточность.
Например, набор ``foo_firmware`` мог бы находиться в файле
``foo/tests/firmware_kunit.c``.
