==================
Архитектура KUnit
==================

Архитектура KUnit разделена на две части:

- `Внутриядерный фреймворк тестирования`_
- `kunit_tool (инструмент тестирования из командной строки)`_

Внутриядерный фреймворк тестирования
====================================

Библиотека тестирования ядра поддерживает тесты KUnit, написанные на C с
использованием KUnit. Эти тесты KUnit представляют собой код ядра. KUnit
выполняет следующие задачи:

- Организует тесты
- Сообщает о результатах тестов
- Предоставляет вспомогательные средства для тестирования

Тестовые случаи
---------------

Тестовый случай — это фундаментальная единица в KUnit. Тестовые случаи KUnit
организованы в наборы. Тестовый случай KUnit — это функция с сигнатурой типа
``void (*)(struct kunit *test)``. Эти функции тестовых случаев обёрнуты в
структуру с именем struct kunit_case.

.. note:
	``generate_params`` является необязательным для непараметризованных тестов.

Каждый тестовый случай KUnit получает объект контекста ``struct kunit``, который
отслеживает выполняющийся тест. Макросы утверждений KUnit и другие вспомогательные
средства KUnit используют объект контекста ``struct kunit``. В качестве исключения
существуют два поля:

- ``->priv``: функции настройки могут использовать его для хранения произвольных
  пользовательских данных теста.

- ``->param_value``: оно содержит значение параметра, которое можно
  получить в параметризованных тестах.

Наборы тестов
-------------

Набор KUnit включает в себя коллекцию тестовых случаев. Наборы KUnit
представлены структурой ``struct kunit_suite``. Например:

.. code-block:: c

	static struct kunit_case example_test_cases[] = {
		KUNIT_CASE(example_test_foo),
		KUNIT_CASE(example_test_bar),
		KUNIT_CASE(example_test_baz),
		{}
	};

	static struct kunit_suite example_test_suite = {
		.name = "example",
		.init = example_test_init,
		.exit = example_test_exit,
		.test_cases = example_test_cases,
	};
	kunit_test_suite(example_test_suite);

В приведённом выше примере набор тестов ``example_test_suite`` запускает
тестовые случаи ``example_test_foo``, ``example_test_bar`` и
``example_test_baz``. Перед запуском теста вызывается ``example_test_init``,
а после запуска теста вызывается ``example_test_exit``.
Вызов ``kunit_test_suite(example_test_suite)`` регистрирует набор тестов
во фреймворке тестирования KUnit.

Исполнитель
-----------

Исполнитель KUnit может перечислять и запускать встроенные тесты KUnit при
загрузке. Наборы тестов хранятся в секции компоновщика
с именем ``.kunit_test_suites``. Код см. в определении макроса ``KUNIT_TABLE()``
в
`include/asm-generic/vmlinux.lds.h <https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/include/asm-generic/vmlinux.lds.h?h=v6.0#n950>`_.
Секция компоновщика состоит из массива указателей на
``struct kunit_suite`` и заполняется макросом ``kunit_test_suites()``.
Исполнитель KUnit перебирает массив секции компоновщика, чтобы
запустить все тесты, скомпилированные в ядро.

.. kernel-figure:: kunit_suitememorydiagram.svg
	:alt:	KUnit Suite Memory

	Диаграмма памяти наборов KUnit

При загрузке ядра исполнитель KUnit использует начальный и конечный адреса
этой секции, чтобы перебрать и запустить все тесты. Реализацию
исполнителя см. в
`lib/kunit/executor.c <https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/lib/kunit/executor.c>`_.
При сборке в виде модуля макрос ``kunit_test_suites()`` определяет
функцию ``module_init()``, которая запускает все тесты в единице
компиляции вместо использования исполнителя.

В тестах KUnit некоторые классы ошибок не влияют на другие тесты
или части ядра — каждый случай KUnit выполняется в отдельном контексте
потока. См. функцию ``kunit_try_catch_run()`` в
`lib/kunit/try-catch.c <https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/lib/kunit/try-catch.c?h=v5.15#n58>`_.

Макросы утверждений
-------------------
Тесты KUnit проверяют состояние, используя ожидания/утверждения.
Все ожидания/утверждения имеют формат:
``KUNIT_{EXPECT|ASSERT}_<op>[_MSG](kunit, property[, message])``

- ``{EXPECT|ASSERT}`` определяет, является ли проверка утверждением или
  ожиданием.
  В случае сбоя ход тестирования различается следующим образом:

	- Для ожиданий тест помечается как проваленный, а сбой записывается в журнал.

	- Проваленные утверждения, напротив, приводят к немедленному
	  завершению тестового случая.

		- Утверждения вызывают функцию:
		  ``void __noreturn __kunit_abort(struct kunit *)``.

		- ``__kunit_abort`` вызывает функцию:
		  ``void __noreturn kunit_try_catch_throw(struct kunit_try_catch *try_catch)``.

		- ``kunit_try_catch_throw`` вызывает функцию:
		  ``void kthread_complete_and_exit(struct completion *, long) __noreturn;``
		  и завершает специальный контекст потока.

- ``<op>`` обозначает проверку с опциями: ``TRUE`` (переданное свойство
  имеет логическое значение «true»), ``EQ`` (два переданных свойства
  равны), ``NOT_ERR_OR_NULL`` (переданный указатель не равен null и не
  содержит значение «err»).

- ``[_MSG]`` выводит пользовательское сообщение при сбое.

Сообщение о результатах тестов
------------------------------
KUnit выводит результаты тестов в формате KTAP. KTAP основан на TAP14, см.
Documentation/dev-tools/ktap.rst.
KTAP работает с KUnit и Kselftest. Исполнитель KUnit выводит результаты KTAP в
dmesg и debugfs (если настроено).

Параметризованные тесты
-----------------------

Каждый параметризованный тест KUnit связан с коллекцией
параметров. Тест вызывается несколько раз, по одному разу для каждого значения
параметра, и параметр сохраняется в поле ``param_value``.
Тестовый случай включает макрос KUNIT_CASE_PARAM(), который принимает
функцию-генератор. Функции-генератору передаётся предыдущий параметр,
и она возвращает следующий параметр. Он также включает макрос для генерации
типовых генераторов на основе массивов.

kunit_tool (инструмент тестирования из командной строки)
========================================================

``kunit_tool`` — это скрипт на Python, который находится в ``tools/testing/kunit/kunit.py``.
Он используется для конфигурирования, сборки, выполнения, разбора результатов тестов
и запуска всех предыдущих команд в правильном порядке (то есть конфигурирование, сборка, выполнение и разбор).
У вас есть два варианта запуска тестов KUnit: либо собрать ядро с включённым KUnit
и разобрать результаты вручную (см.
Documentation/dev-tools/kunit/run_manual.rst), либо использовать ``kunit_tool``
(см. Documentation/dev-tools/kunit/run_wrapper.rst).

- Команда ``configure`` генерирует ``.config`` ядра из
  файла ``.kunitconfig`` (и любых архитектурно-специфичных опций).
  Скрипты на Python, доступные в папке ``qemu_configs``
  (например, ``tools/testing/kunit/qemu configs/powerpc.py``), содержат
  дополнительные опции конфигурации для конкретных архитектур.
  Она разбирает как существующий файл ``.config``, так и файл ``.kunitconfig``,
  чтобы убедиться, что ``.config`` является надмножеством ``.kunitconfig``.
  Если это не так, она объединит оба файла и запустит ``make olddefconfig``, чтобы перегенерировать
  файл ``.config``. Затем она проверяет, стал ли ``.config`` надмножеством.
  Это подтверждает, что все зависимости Kconfig правильно указаны в
  файле ``.kunitconfig``. Скрипт ``kunit_config.py`` содержит код для разбора
  Kconfig. Код, который запускает ``make olddefconfig``, является частью
  скрипта ``kunit_kernel.py``. Вы можете вызвать эту команду через:
  ``./tools/testing/kunit/kunit.py config`` и
  сгенерировать файл ``.config``.
- ``build`` запускает ``make`` в дереве ядра с необходимыми опциями
  (зависит от архитектуры и некоторых опций, например: build_dir)
  и сообщает обо всех ошибках.
  Чтобы собрать ядро KUnit из текущего ``.config``, вы можете использовать
  аргумент ``build``: ``./tools/testing/kunit/kunit.py build``.
- Команда ``exec`` выполняет ядро с результатами либо напрямую (используя
  конфигурацию User-mode Linux), либо через эмулятор, такой
  как QEMU. Она считывает результаты из журнала, используя стандартный
  вывод (stdout), и передаёт их в ``parse`` для разбора.
  Если вы уже собрали ядро со встроенными тестами KUnit,
  вы можете запустить ядро и отобразить результаты тестов с помощью аргумента
  ``exec``: ``./tools/testing/kunit/kunit.py exec``.
- ``parse`` извлекает вывод KTAP из журнала ядра, разбирает
  результаты тестов и выводит сводку. Для проваленных тестов будет включён
  любой диагностический вывод.
