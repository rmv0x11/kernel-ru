Kernel Concurrency Sanitizer (KCSAN)
====================================

Kernel Concurrency Sanitizer (KCSAN) — это динамический детектор гонок, который
опирается на инструментирование во время компиляции и использует подход на основе
точек наблюдения (watchpoint) с выборкой для обнаружения гонок. Основное
назначение KCSAN — обнаруживать `Гонки данных`_.

Использование
-------------

KCSAN поддерживается как GCC, так и Clang. Для GCC требуется версия 11 или
новее, для Clang также требуется версия 11 или новее.

Чтобы включить KCSAN, сконфигурируйте ядро с::

    CONFIG_KCSAN = y

KCSAN предоставляет несколько других параметров конфигурации для настройки
поведения (подробнее см. соответствующий справочный текст в ``lib/Kconfig.kcsan``).

Отчёты об ошибках
~~~~~~~~~~~~~~~~~

Типичный отчёт о гонке данных выглядит так::

    ==================================================================
    BUG: KCSAN: data-race in test_kernel_read / test_kernel_write

    write to 0xffffffffc009a628 of 8 bytes by task 487 on cpu 0:
     test_kernel_write+0x1d/0x30
     access_thread+0x89/0xd0
     kthread+0x23e/0x260
     ret_from_fork+0x22/0x30

    read to 0xffffffffc009a628 of 8 bytes by task 488 on cpu 6:
     test_kernel_read+0x10/0x20
     access_thread+0x89/0xd0
     kthread+0x23e/0x260
     ret_from_fork+0x22/0x30

    value changed: 0x00000000000009a6 -> 0x00000000000009b2

    Reported by Kernel Concurrency Sanitizer on:
    CPU: 6 PID: 488 Comm: access_thread Not tainted 5.12.0-rc2+ #1
    Hardware name: QEMU Standard PC (i440FX + PIIX, 1996), BIOS 1.14.0-2 04/01/2014
    ==================================================================

Заголовок отчёта содержит краткую сводку функций, вовлечённых в гонку. За ним
следуют типы доступа и трассировки стека двух потоков, вовлечённых в гонку
данных. Если KCSAN также наблюдал изменение значения, то наблюдавшиеся старое и
новое значения показываются в строке «value changed» соответственно.

Другой, менее распространённый тип отчёта о гонке данных выглядит так::

    ==================================================================
    BUG: KCSAN: data-race in test_kernel_rmw_array+0x71/0xd0

    race at unknown origin, with read to 0xffffffffc009bdb0 of 8 bytes by task 515 on cpu 2:
     test_kernel_rmw_array+0x71/0xd0
     access_thread+0x89/0xd0
     kthread+0x23e/0x260
     ret_from_fork+0x22/0x30

    value changed: 0x0000000000002328 -> 0x0000000000002329

    Reported by Kernel Concurrency Sanitizer on:
    CPU: 2 PID: 515 Comm: access_thread Not tainted 5.12.0-rc2+ #1
    Hardware name: QEMU Standard PC (i440FX + PIIX, 1996), BIOS 1.14.0-2 04/01/2014
    ==================================================================

Такой отчёт генерируется, когда не удалось определить другой участвующий в гонке
поток, но наличие гонки было выведено из того, что значение данных в наблюдаемой
ячейке памяти изменилось. В таких отчётах всегда присутствует строка «value
changed». Распространённая причина отчётов такого типа — отсутствие
инструментирования в участвующем в гонке потоке, но они также могут возникать,
например, из-за доступа по DMA. Такие отчёты показываются только если задано
``CONFIG_KCSAN_REPORT_RACE_UNKNOWN_ORIGIN=y``, что включено по умолчанию.

Выборочный анализ
~~~~~~~~~~~~~~~~~

Может быть желательно отключить обнаружение гонок данных для определённых
доступов, функций, единиц компиляции или целых подсистем.  Для статического
занесения в чёрный список доступны следующие параметры:

* KCSAN понимает аннотацию ``data_race(expr)``, которая сообщает KCSAN, что
  любые гонки данных из-за доступов в ``expr`` следует игнорировать, а
  результирующее поведение при возникновении гонки данных считается безопасным.
  Подробнее см. `"Marking Shared-Memory Accesses" in the LKMM`_.

* Аналогично ``data_race(...)``, квалификатор типа ``__data_racy`` может
  использоваться, чтобы задокументировать, что все гонки данных из-за доступов
  к переменной являются преднамеренными и должны игнорироваться KCSAN::

    struct foo {
        ...
        int __data_racy stats_counter;
        ...
    };

* Отключить обнаружение гонок данных для целых функций можно с помощью атрибута
  функции ``__no_kcsan``::

    __no_kcsan
    void foo(void) {
        ...

  Чтобы динамически ограничить, для каких функций генерируются отчёты, см.
  возможность чёрного/белого списка `Интерфейс DebugFS`_.

* Чтобы отключить обнаружение гонок данных для конкретной единицы компиляции,
  добавьте в ``Makefile``::

    KCSAN_SANITIZE_file.o := n

* Чтобы отключить обнаружение гонок данных для всех единиц компиляции,
  перечисленных в ``Makefile``, добавьте в соответствующий ``Makefile``::

    KCSAN_SANITIZE := n

.. _"Marking Shared-Memory Accesses" in the LKMM: https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/tools/memory-model/Documentation/access-marking.txt

Кроме того, можно указать KCSAN показывать или скрывать целые классы гонок
данных в зависимости от предпочтений. Это можно изменить с помощью следующих
параметров Kconfig:

* ``CONFIG_KCSAN_REPORT_VALUE_CHANGE_ONLY``: если включено и через точку
  наблюдения наблюдается конфликтующая запись, но было замечено, что значение
  данных в ячейке памяти осталось неизменным, не сообщать о гонке данных.

* ``CONFIG_KCSAN_ASSUME_PLAIN_WRITES_ATOMIC``: считать обычные выровненные
  записи размером до машинного слова атомарными по умолчанию. Предполагается,
  что такие записи не подвержены небезопасным оптимизациям компилятора,
  приводящим к гонкам данных. Этот параметр заставляет KCSAN не сообщать о
  гонках данных из-за конфликтов, где единственными обычными доступами являются
  выровненные записи размером до машинного слова.

* ``CONFIG_KCSAN_PERMISSIVE``: включить дополнительные разрешающие правила для
  игнорирования определённых классов распространённых гонок данных. В отличие от
  приведённых выше, эти правила более сложны и учитывают шаблоны изменения
  значений, тип доступа и адрес. Этот параметр зависит от
  ``CONFIG_KCSAN_REPORT_VALUE_CHANGE_ONLY=y``. Подробнее см.
  ``kernel/kcsan/permissive.h``. Тестировщикам и сопровождающим (maintainer),
  которые сосредоточены только на отчётах из конкретных подсистем, а не всего
  ядра, рекомендуется отключить этот параметр.

Чтобы использовать максимально строгие правила, выберите
``CONFIG_KCSAN_STRICT=y``, который настраивает KCSAN максимально близко
следовать модели согласованности памяти ядра Linux (LKMM).

Интерфейс DebugFS
~~~~~~~~~~~~~~~~~

Файл ``/sys/kernel/debug/kcsan`` предоставляет следующий интерфейс:

* Чтение ``/sys/kernel/debug/kcsan`` возвращает различную статистику времени
  выполнения.

* Запись ``on`` или ``off`` в ``/sys/kernel/debug/kcsan`` позволяет включать или
  выключать KCSAN соответственно.

* Запись ``!some_func_name`` в ``/sys/kernel/debug/kcsan`` добавляет
  ``some_func_name`` в список фильтрации отчётов, который (по умолчанию)
  заносит в чёрный список отчёты о гонках данных, где один из верхних кадров
  стека является функцией из этого списка.

* Запись ``blacklist`` или ``whitelist`` в ``/sys/kernel/debug/kcsan``
  изменяет поведение фильтрации отчётов. Например, возможность чёрного списка
  можно использовать, чтобы заглушить часто возникающие гонки данных;
  возможность белого списка может помочь с воспроизведением и тестированием
  исправлений.

Настройка производительности
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Основные параметры, влияющие на общую производительность KCSAN и способность
обнаруживать ошибки, доступны как аргументы командной строки ядра, чьи значения
по умолчанию также можно изменить через соответствующие параметры Kconfig.

* ``kcsan.skip_watch`` (``CONFIG_KCSAN_SKIP_WATCH``): количество операций с
  памятью на каждый CPU, которые нужно пропустить, прежде чем будет установлена
  следующая точка наблюдения. Более частая установка точек наблюдения приводит к
  увеличению вероятности наблюдения гонок. Этот параметр оказывает наиболее
  значительное влияние на общую производительность системы и способность
  обнаруживать гонки.

* ``kcsan.udelay_task`` (``CONFIG_KCSAN_UDELAY_TASK``): для задач — задержка в
  микросекундах для приостановки выполнения после установки точки наблюдения.
  Бо́льшие значения приводят к увеличению окна, в котором мы можем наблюдать
  гонку.

* ``kcsan.udelay_interrupt`` (``CONFIG_KCSAN_UDELAY_INTERRUPT``): для прерываний
  — задержка в микросекундах для приостановки выполнения после установки точки
  наблюдения. У прерываний более жёсткие требования к задержке (latency), и их
  задержка, как правило, должна быть меньше выбранной для задач.

Их можно подстраивать во время выполнения через ``/sys/module/kcsan/parameters/``.

Гонки данных
------------

В рамках выполнения два доступа к памяти образуют *гонку данных* (data race),
если они *конфликтуют*, происходят одновременно в разных потоках, и хотя бы один
из них является *обычным доступом* (plain access); они *конфликтуют*, если оба
обращаются к одной и той же ячейке памяти, и хотя бы один из них является
записью. Более подробное обсуждение и определение см. в `"Plain Accesses and
Data Races" in the LKMM`_.

.. _"Plain Accesses and Data Races" in the LKMM: https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/tools/memory-model/Documentation/explanation.txt?id=8f6629c004b193d23612641c3607e785819e97ab#n2164

Связь с моделью согласованности памяти ядра Linux (LKMM)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

LKMM определяет правила распространения и упорядочивания различных операций с
памятью, что даёт разработчикам возможность рассуждать о конкурентном коде. В
конечном счёте это позволяет определить возможные варианты выполнения
конкурентного кода и то, свободен ли этот код от гонок данных.

KCSAN знает о *маркированных атомарных операциях* (``READ_ONCE``, ``WRITE_ONCE``,
``atomic_*`` и т. д.) и о подмножестве гарантий упорядочивания, подразумеваемых
барьерами памяти. С ``CONFIG_KCSAN_WEAK_MEMORY=y`` KCSAN моделирует буферизацию
загрузок и сохранений и может обнаруживать отсутствующие ``smp_mb()``,
``smp_wmb()``, ``smp_rmb()``, ``smp_store_release()``, а также все операции
``atomic_*`` с эквивалентными подразумеваемыми барьерами.

Заметьте, KCSAN сообщит не обо всех гонках данных, вызванных отсутствием
упорядочивания памяти, в частности там, где барьер памяти потребовался бы для
запрета переупорядочивания последующей операции с памятью до барьера. Поэтому
разработчикам следует тщательно учитывать требуемые требования к упорядочиванию
памяти, которые остаются непроверенными.

Обнаружение гонок помимо гонок данных
-------------------------------------

Для кода со сложным конкурентным дизайном ошибки типа «состояние гонки» не всегда
проявляются как гонки данных. Состояния гонки возникают, если одновременно
выполняемые операции приводят к неожиданному поведению системы. С другой стороны,
гонки данных определены на уровне языка C. Следующие макросы можно использовать
для проверки свойств конкурентного кода, где ошибки не проявлялись бы как гонки
данных.

.. kernel-doc:: include/linux/kcsan-checks.h
    :functions: ASSERT_EXCLUSIVE_WRITER ASSERT_EXCLUSIVE_WRITER_SCOPED
                ASSERT_EXCLUSIVE_ACCESS ASSERT_EXCLUSIVE_ACCESS_SCOPED
                ASSERT_EXCLUSIVE_BITS

Детали реализации
-----------------

KCSAN полагается на наблюдение того, что два доступа происходят одновременно.
Принципиально важно, что мы хотим (a) увеличить шансы наблюдения гонок (особенно
гонок, которые проявляются редко) и (b) иметь возможность фактически наблюдать
их. Цели (a) мы можем достичь, внедряя различные задержки, а (b) — используя
адресные точки наблюдения (или точки останова).

Если мы намеренно приостанавливаем доступ к памяти, пока для его адреса
установлена точка наблюдения, и затем наблюдаем срабатывание этой точки
наблюдения, значит, два доступа к одному адресу только что вошли в гонку.
С использованием аппаратных точек наблюдения этот подход применён в `DataCollider
<http://usenix.org/legacy/events/osdi10/tech/full_papers/Erickson.pdf>`_.
В отличие от DataCollider, KCSAN не использует аппаратные точки наблюдения, а
вместо этого полагается на инструментирование компилятором и «программные точки
наблюдения».

В KCSAN точки наблюдения реализованы с использованием эффективного кодирования,
которое хранит тип доступа, размер и адрес в значении типа long; преимущества
использования «программных точек наблюдения» — переносимость и большая гибкость.
Затем KCSAN полагается на то, что компилятор инструментирует обычные доступы. Для
каждого инструментированного обычного доступа:

1. Проверяется, существует ли совпадающая точка наблюдения; если да и хотя бы
   один доступ является записью, значит, мы столкнулись с участвующим в гонке
   доступом.

2. Периодически, если совпадающей точки наблюдения не существует, устанавливается
   точка наблюдения и выполнение приостанавливается на небольшую рандомизированную
   задержку.

3. Также проверяется значение данных перед задержкой и перепроверяется значение
   данных после задержки; если значения не совпадают, мы делаем вывод о наличии
   гонки неизвестного происхождения.

Чтобы обнаруживать гонки данных между обычными и маркированными доступами, KCSAN
также аннотирует маркированные доступы, но только для проверки того, существует
ли точка наблюдения; то есть KCSAN никогда не устанавливает точку наблюдения на
маркированных доступах. Никогда не устанавливая точки наблюдения для
маркированных операций, KCSAN, если все доступы к переменной, к которой
обращаются одновременно, правильно маркированы, никогда не сработает по точке
наблюдения и, следовательно, никогда не сообщит об этих доступах.

Моделирование слабой памяти
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Подход KCSAN к обнаружению гонок данных, вызванных отсутствием барьеров памяти,
основан на моделировании переупорядочивания доступов (с
``CONFIG_KCSAN_WEAK_MEMORY=y``). Каждый обычный доступ к памяти, для которого
установлена точка наблюдения, также отбирается для имитируемого переупорядочивания
в пределах области его функции (не более 1 доступа «в полёте»).

После того как доступ выбран для переупорядочивания, он проверяется относительно
каждого другого доступа до конца области функции. Если встречается подходящий
барьер памяти, доступ перестаёт рассматриваться для имитируемого
переупорядочивания.

Когда результат операции с памятью должен быть упорядочен барьером, KCSAN может
затем обнаружить гонки данных, где конфликт возникает только в результате
отсутствия барьера. Рассмотрим пример::

    int x, flag;
    void T1(void)
    {
        x = 1;                  // data race!
        WRITE_ONCE(flag, 1);    // correct: smp_store_release(&flag, 1)
    }
    void T2(void)
    {
        while (!READ_ONCE(flag));   // correct: smp_load_acquire(&flag)
        ... = x;                    // data race!
    }

Когда включено моделирование слабой памяти, KCSAN может рассматривать ``x`` в
``T1`` для имитируемого переупорядочивания. После записи ``flag`` ``x`` снова
проверяется на одновременные доступы: поскольку ``T2`` может продолжить
выполнение после записи ``flag``, обнаруживается гонка данных. При наличии
правильных барьеров ``x`` не рассматривался бы для переупорядочивания после
надлежащего release-сохранения ``flag``, и никакая гонка данных не была бы
обнаружена.

Преднамеренные компромиссы в сложности, а также практические ограничения
означают, что может быть обнаружено лишь подмножество гонок данных, вызванных
отсутствием барьеров памяти. С имеющейся на данный момент поддержкой компилятора
реализация ограничена моделированием эффектов «буферизации» (задержки доступов),
поскольку среда выполнения не может «предвыбирать» (prefetch) доступы. Также
вспомните, что точки наблюдения устанавливаются только для обычных доступов —
единственного типа доступа, для которого KCSAN имитирует переупорядочивание. Это
означает, что переупорядочивание маркированных доступов не моделируется.

Следствием вышесказанного является то, что операции acquire не требуют
инструментирования барьеров (нет предвыборки). Кроме того, маркированные доступы,
вводящие адресные зависимости или зависимости по управлению, не требуют особой
обработки (маркированный доступ не может быть переупорядочен, а более поздние
зависимые доступы не могут быть предвыбраны).

Ключевые свойства
~~~~~~~~~~~~~~~~~

1. **Накладные расходы по памяти:**  Общие накладные расходы по памяти составляют
   всего несколько MiB в зависимости от конфигурации. Текущая реализация
   использует небольшой массив значений типа long для кодирования информации о
   точках наблюдения, что пренебрежимо мало.

2. **Накладные расходы на производительность:** Среда выполнения KCSAN
   стремится быть минимальной, используя эффективное кодирование точек
   наблюдения, которое не требует захвата каких-либо разделяемых блокировок на
   быстром пути (fast-path). Для загрузки ядра на системе с 8 CPU:

   - замедление в 5.0x с конфигурацией KCSAN по умолчанию;
   - замедление в 2.8x только из-за накладных расходов быстрого пути среды
     выполнения (задайте очень большое ``KCSAN_SKIP_WATCH`` и снимите
     ``KCSAN_SKIP_WATCH_RANDOMIZE``).

3. **Накладные расходы на аннотации:** За пределами среды выполнения KCSAN
   требуется минимум аннотаций. В результате накладные расходы на сопровождение
   минимальны по мере развития ядра.

4. **Обнаружение участвующих в гонке записей от устройств:** Благодаря проверке
   значений данных при установке точек наблюдения могут быть обнаружены также и
   участвующие в гонке записи от устройств.

5. **Упорядочивание памяти:** KCSAN знает лишь о подмножестве правил
   упорядочивания LKMM; это может привести к пропущенным гонкам данных
   (ложноотрицательным результатам).

6. **Точность анализа:** Для наблюдаемых вариантов выполнения из-за использования
   стратегии выборки анализ является *несостоятельным* (unsound) (возможны
   ложноотрицательные результаты), но стремится быть полным (нет
   ложноположительных результатов).

Рассмотренные альтернативы
--------------------------

Альтернативный подход к обнаружению гонок данных для ядра можно найти в
`Kernel Thread Sanitizer (KTSAN)
<https://github.com/google/kernel-sanitizers/blob/master/KTSAN.md>`_.
KTSAN — это детектор гонок данных на основе отношения happens-before, который
явно устанавливает порядок happens-before между операциями с памятью, который
затем можно использовать для определения гонок данных в том смысле, как они
определены в разделе `Гонки данных`_.

Чтобы построить корректное отношение happens-before, KTSAN должен знать все
правила упорядочивания LKMM и примитивы синхронизации. К сожалению, любое
упущение приводит к большому количеству ложноположительных результатов, что
особенно вредно в контексте ядра, которое включает многочисленные собственные
механизмы синхронизации. Для отслеживания отношения happens-before реализация
KTSAN требует метаданных для каждой ячейки памяти (теневая память, shadow
memory), что для каждой страницы соответствует 4 страницам теневой памяти и
может выливаться в накладные расходы в десятки GiB на большой системе.
