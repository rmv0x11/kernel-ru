===============================
Kernel Memory Sanitizer (KMSAN)
===============================

KMSAN — это динамический детектор ошибок, предназначенный для поиска
использований неинициализированных значений. Он основан на инструментации
компилятором и весьма похож на пользовательский инструмент `MemorySanitizer tool`_.

Важно отметить, что KMSAN не предназначен для использования в продакшене,
поскольку он резко увеличивает потребление памяти ядром и замедляет работу
всей системы.

Использование
=============

Сборка ядра
-----------

Чтобы собрать ядро с KMSAN, вам понадобится свежий Clang (14.0.6+).
Инструкции по сборке Clang приведены в `LLVM documentation`_.

Теперь сконфигурируйте и соберите ядро с включённым CONFIG_KMSAN.

В отчёте сообщается, что локальная переменная ``uninit`` была создана
неинициализированной в ``do_uninit_local_array()``. Третья трассировка стека
соответствует месту, где эта переменная была создана.

Первая трассировка стека показывает, где было использовано неинициализированное
значение (в ``test_uninit_kmsan_check_memory()``). Инструмент показывает байты,
которые остались неинициализированными в локальной переменной, а также стек,
где значение было скопировано в другую область памяти перед использованием.

KMSAN сообщает об использовании неинициализированного значения ``v`` в следующих
случаях:

 - в условии, например ``if (v) { ... }``;
 - при индексировании или разыменовании указателя, например ``array[v]`` или ``*v``;
 - когда оно копируется в пространство пользователя или в оборудование, например
   ``copy_to_user(..., &v, ...)``;
 - когда оно передаётся в функцию в качестве аргумента и включён параметр
   ``CONFIG_KMSAN_CHECK_PARAM_RETVAL`` (см. ниже).

Упомянутые случаи (за исключением копирования данных в пространство пользователя
или в оборудование, что является проблемой безопасности) с точки зрения
стандарта C11 считаются неопределённым поведением.

Отключение инструментации
-------------------------

Функцию можно пометить атрибутом ``__no_kmsan_checks``. В этом случае KMSAN
игнорирует неинициализированные значения в этой функции и помечает её вывод как
инициализированный. В результате пользователь не будет получать отчёты KMSAN,
относящиеся к этой функции.

Ещё один атрибут функции, поддерживаемый KMSAN, — это ``__no_sanitize_memory``.
Применение этого атрибута к функции приведёт к тому, что KMSAN не будет её
инструментировать, что может быть полезно, если мы не хотим, чтобы компилятор
вмешивался в некоторый низкоуровневый код (например, помеченный ``noinstr``,
который неявно добавляет ``__no_sanitize_memory``).

Однако это имеет свою цену: размещения на стеке из таких функций будут иметь
некорректные значения shadow/origin, что, вероятно, приведёт к ложным
срабатываниям. Функции, вызываемые из неинструментированного кода, также могут
получать некорректные метаданные для своих параметров.

Как правило, избегайте явного использования ``__no_sanitize_memory``.

Также возможно отключить KMSAN для отдельного файла (например, main.o)::

  KMSAN_SANITIZE_main.o := n

или для целого каталога::

  KMSAN_SANITIZE := n

в Makefile. Считайте это применением ``__no_sanitize_memory`` к каждой функции
в файле или каталоге. Большинству пользователей KMSAN_SANITIZE не понадобится,
если только их код не ломается из-за KMSAN (например, исполняется на ранней
стадии загрузки).

Проверки KMSAN также можно временно отключить для текущей задачи с помощью
вызовов ``kmsan_disable_current()`` и ``kmsan_enable_current()``. Каждому вызову
``kmsan_enable_current()`` должен предшествовать вызов
``kmsan_disable_current()``; эти пары вызовов могут быть вложенными. С этими
вызовами следует обращаться осторожно, сохраняя области короткими и предпочитая
по возможности другие способы отключения инструментации.

Поддержка
=========

Чтобы KMSAN работал, ядро должно быть собрано с помощью Clang, который на данный
момент является единственным компилятором, поддерживающим KMSAN. Проход
инструментации ядра основан на пользовательском инструменте `MemorySanitizer tool`_.

Библиотека времени выполнения на данный момент поддерживает только x86_64.

Как работает KMSAN
==================

Теневая память KMSAN
--------------------

KMSAN сопоставляет каждому байту памяти ядра байт метаданных (также называемый
теневым байтом). Бит в теневом байте устанавливается, если соответствующий бит
байта памяти ядра неинициализирован. Пометка памяти как неинициализированной
(т. е. установка её теневых байтов в ``0xff``) называется отравлением
(poisoning), а пометка её как инициализированной (установка теневых байтов в
``0x00``) называется снятием отравления (unpoisoning).

Когда на стеке выделяется новая переменная, по умолчанию она отравляется кодом
инструментации, вставленным компилятором (если только это не переменная стека,
которая инициализируется немедленно). Любое новое выделение в куче, выполненное
без ``__GFP_ZERO``, также отравляется.

Инструментация компилятором также отслеживает теневые значения по мере их
использования в коде. Когда это необходимо, код инструментации вызывает
библиотеку времени выполнения в ``mm/kmsan/``, чтобы сохранить теневые значения.

Теневое значение базового или составного типа — это массив байтов той же длины.
Когда в память записывается константное значение, с этой памяти снимается
отравление. Когда значение читается из памяти, его теневая память также
извлекается и распространяется во все операции, использующие это значение. Для
каждой инструкции, принимающей одно или несколько значений, компилятор
генерирует код, вычисляющий тень результата в зависимости от этих значений и их
теней.

Пример::

  int a = 0xff;  // i.e. 0x000000ff
  int b;
  int c = a | b;

В этом случае тень ``a`` равна ``0``, тень ``b`` равна ``0xffffffff``, тень
``c`` равна ``0xffffff00``. Это означает, что верхние три байта ``c``
неинициализированы, а младший байт инициализирован.

Отслеживание происхождения (origin)
-----------------------------------

С каждыми четырьмя байтами памяти ядра также сопоставлено так называемое
происхождение (origin). Это происхождение описывает точку в исполнении
программы, в которой было создано неинициализированное значение. Каждое
происхождение связано либо с полным стеком выделения (для памяти, выделенной в
куче), либо с функцией, содержащей неинициализированную переменную (для
локальных переменных).

Когда неинициализированная переменная выделяется на стеке или в куче, создаётся
новое значение происхождения, и происхождение этой переменной заполняется этим
значением. Когда значение читается из памяти, его происхождение также читается и
хранится вместе с тенью. Для каждой инструкции, принимающей одно или несколько
значений, происхождение результата равно одному из происхождений,
соответствующих любому из неинициализированных входных данных. Если в память
записывается отравленное значение, его происхождение также записывается в
соответствующее хранилище.

Пример 1::

  int a = 42;
  int b;
  int c = a + b;

В этом случае происхождение ``b`` генерируется при входе в функцию и
сохраняется в происхождение ``c`` непосредственно перед тем, как результат
сложения записывается в память.

Несколько переменных могут разделять один и тот же адрес происхождения, если они
хранятся в одном и том же четырёхбайтовом фрагменте. В этом случае каждая запись
в любую из переменных обновляет происхождение для всех них. В этом случае нам
приходится жертвовать точностью, поскольку хранение происхождений для отдельных
битов (и даже байтов) было бы слишком затратным.

Пример 2::

  int combine(short a, short b) {
    union ret_t {
      int i;
      short s[2];
    } ret;
    ret.s[0] = a;
    ret.s[1] = b;
    return ret.i;
  }

Если ``a`` инициализирована, а ``b`` нет, то тень результата будет равна
0xffff0000, а происхождение результата будет происхождением ``b``. ``ret.s[0]``
будет иметь то же происхождение, но оно никогда не будет использовано, поскольку
эта переменная инициализирована.

Если оба аргумента функции неинициализированы, сохраняется только происхождение
второго аргумента.

Сцепление происхождений (origin chaining)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Чтобы облегчить отладку, KMSAN создаёт новое происхождение для каждой записи
неинициализированного значения в память. Новое происхождение ссылается как на
свой стек создания, так и на предыдущее происхождение, которое имело значение.
Это может приводить к повышенному потреблению памяти, поэтому мы ограничиваем
длину цепочек происхождений в библиотеке времени выполнения.

API инструментации Clang
------------------------

Проход инструментации Clang вставляет в код ядра вызовы функций, определённых в
``mm/kmsan/nstrumentation.c``.

Манипуляции с тенью
~~~~~~~~~~~~~~~~~~~~

Для каждого обращения к памяти компилятор генерирует вызов функции, которая
возвращает пару указателей на теневой адрес и адрес происхождения для заданной
памяти::

  typedef struct {
    void *shadow, *origin;
  } shadow_origin_ptr_t

  shadow_origin_ptr_t __msan_metadata_ptr_for_load_{1,2,4,8}(void *addr)
  shadow_origin_ptr_t __msan_metadata_ptr_for_store_{1,2,4,8}(void *addr)
  shadow_origin_ptr_t __msan_metadata_ptr_for_load_n(void *addr, uintptr_t size)
  shadow_origin_ptr_t __msan_metadata_ptr_for_store_n(void *addr, uintptr_t size)

Имя функции зависит от размера обращения к памяти.

Компилятор гарантирует, что для каждого загруженного значения его теневое
значение и значение происхождения читаются из памяти. Когда значение
сохраняется в память, его тень и происхождение также сохраняются с помощью
указателей на метаданные.

Обработка локальных переменных
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Специальная функция используется для создания нового значения происхождения для
локальной переменной и установки происхождения этой переменной в это значение::

  void __msan_poison_alloca(void *addr, uintptr_t size, char *descr)

Доступ к данным конкретной задачи
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

В начале каждой инструментированной функции KMSAN вставляет вызов
``__msan_get_context_state()``::

  kmsan_context_state *__msan_get_context_state(void)

``kmsan_context_state`` объявлена в ``include/linux/kmsan.h``::

  struct kmsan_context_state {
    char param_tls[KMSAN_PARAM_SIZE];
    char retval_tls[KMSAN_RETVAL_SIZE];
    char va_arg_tls[KMSAN_PARAM_SIZE];
    char va_arg_origin_tls[KMSAN_PARAM_SIZE];
    u64 va_arg_overflow_size_tls;
    char param_origin_tls[KMSAN_PARAM_SIZE];
    depot_stack_handle_t retval_origin_tls;
  };

Эта структура используется KMSAN для передачи теней и происхождений параметров
между инструментированными функциями (если только параметры не проверяются
немедленно с помощью ``CONFIG_KMSAN_CHECK_PARAM_RETVAL``).

Передача неинициализированных значений в функции
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Инструментация MemorySanitizer в Clang имеет опцию
``-fsanitize-memory-param-retval``, которая заставляет компилятор проверять
параметры функций, передаваемые по значению, а также возвращаемые значения
функций.

Эта опция управляется параметром ``CONFIG_KMSAN_CHECK_PARAM_RETVAL``, который
включён по умолчанию, чтобы KMSAN сообщал о неинициализированных значениях
раньше. Подробности приведены в `LKML discussion`_.

Из-за того, как реализованы проверки в LLVM (они применяются только к
параметрам, помеченным ``noundef``), не все параметры гарантированно
проверяются, поэтому мы не можем отказаться от хранилища метаданных в
``kmsan_context_state``.

Строковые функции
~~~~~~~~~~~~~~~~~~

Компилятор заменяет вызовы ``memcpy()``/``memmove()``/``memset()`` следующими
функциями. Эти функции также вызываются при инициализации или копировании
структур данных, гарантируя, что теневые значения и значения происхождения
копируются вместе с данными::

  void *__msan_memcpy(void *dst, void *src, uintptr_t n)
  void *__msan_memmove(void *dst, void *src, uintptr_t n)
  void *__msan_memset(void *dst, int c, uintptr_t n)

Сообщение об ошибках
~~~~~~~~~~~~~~~~~~~~~

Для каждого использования значения компилятор генерирует теневую проверку,
которая вызывает ``__msan_warning()`` в случае, если значение отравлено::

  void __msan_warning(u32 origin)

``__msan_warning()`` заставляет библиотеку времени выполнения KMSAN напечатать
отчёт об ошибке.

Инструментация встроенного ассемблера
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

KMSAN инструментирует каждый вывод встроенного ассемблера вызовом::

  void __msan_instrument_asm_store(void *addr, uintptr_t size)

, который снимает отравление с области памяти.

Этот подход может маскировать некоторые ошибки, но он также помогает избежать
большого числа ложных срабатываний в побитовых операциях, атомарных операциях
и т. п.

Иногда указатели, передаваемые во встроенный ассемблер, не указывают на
корректную память. В таких случаях они игнорируются во время выполнения.


Библиотека времени выполнения
------------------------------

Код находится в ``mm/kmsan/``.

Состояние KMSAN для конкретной задачи
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Каждый task_struct имеет связанное с ним состояние задачи KMSAN, которое хранит
контекст KMSAN (см. выше) и счётчик для конкретной задачи, запрещающий отчёты
KMSAN::

  struct kmsan_context {
    ...
    unsigned int depth;
    struct kmsan_context_state cstate;
    ...
  }

  struct task_struct {
    ...
    struct kmsan_context kmsan;
    ...
  }

Контексты KMSAN
~~~~~~~~~~~~~~~

При выполнении в контексте задачи ядра KMSAN использует
``current->kmsan.cstate`` для хранения метаданных параметров функций и
возвращаемых значений.

Но в случае, когда ядро выполняется в контексте прерывания, softirq или NMI,
где ``current`` недоступен, KMSAN переключается на состояние прерывания,
выделенное для каждого CPU (per-cpu)::

  DEFINE_PER_CPU(struct kmsan_ctx, kmsan_percpu_ctx);

Выделение метаданных
~~~~~~~~~~~~~~~~~~~~~

В ядре есть несколько мест, для которых хранятся метаданные.

1. Каждый экземпляр ``struct page`` содержит два указателя на свои теневые
страницы и страницы происхождения::

  struct page {
    ...
    struct page *shadow, *origin;
    ...
  };

Во время загрузки ядро выделяет теневые страницы и страницы происхождения для
каждой доступной страницы ядра. Это делается довольно поздно, когда адресное
пространство ядра уже фрагментировано, поэтому обычные страницы данных могут
произвольно чередоваться со страницами метаданных.

Это означает, что в общем случае для двух смежных страниц памяти их теневые
страницы и страницы происхождения могут не быть смежными. Следовательно, если
обращение к памяти пересекает границу блока памяти, обращения к теневой памяти и
памяти происхождения могут потенциально повредить другие страницы или прочитать
из них некорректные значения.

На практике смежные страницы памяти, возвращённые одним и тем же вызовом
``alloc_pages()``, будут иметь смежные метаданные, тогда как если эти страницы
принадлежат двум разным выделениям, их страницы метаданных могут быть
фрагментированы.

Для данных ядра (``.data``, ``.bss`` и т. п.) и областей памяти percpu также
нет никаких гарантий смежности метаданных.

В случае, когда ``__msan_metadata_ptr_for_XXX_YYY()`` попадает на границу между
двумя страницами с несмежными метаданными, она возвращает указатели на фиктивные
области shadow/origin::

  char dummy_load_page[PAGE_SIZE] __attribute__((aligned(PAGE_SIZE)));
  char dummy_store_page[PAGE_SIZE] __attribute__((aligned(PAGE_SIZE)));

``dummy_load_page`` инициализируется нулями, поэтому чтения из неё всегда дают
нули. Все записи в ``dummy_store_page`` игнорируются.

2. Для памяти vmalloc и модулей существует прямое отображение между диапазоном
памяти, его тенью и происхождением. KMSAN уменьшает область vmalloc на 3/4,
делая доступной для ``vmalloc()`` только первую четверть. Вторая четверть
области vmalloc содержит теневую память для первой четверти, третья хранит
происхождения. Небольшая часть четвёртой четверти содержит тень и происхождения
для модулей ядра. Подробности приведены в
``arch/x86/include/asm/pgtable_64_types.h``.

Когда массив страниц отображается в непрерывное виртуальное адресное
пространство, его теневые страницы и страницы происхождения аналогичным образом
отображаются в непрерывные области.
