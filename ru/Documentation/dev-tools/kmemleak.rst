Детектор утечек памяти ядра
===========================

Kmemleak предоставляет способ обнаружения возможных утечек памяти ядра
способом, похожим на `сборщик мусора с трассировкой
<https://en.wikipedia.org/wiki/Tracing_garbage_collection>`_,
с тем отличием, что объекты-сироты не освобождаются, а только
сообщаются через /sys/kernel/debug/kmemleak. Похожий метод используется
инструментом Valgrind (``memcheck --leak-check``) для обнаружения утечек памяти в
приложениях пространства пользователя.

Использование
-------------

Должна быть включена опция CONFIG_DEBUG_KMEMLEAK в разделе «Kernel hacking». Поток
ядра сканирует память каждые 10 минут (по умолчанию) и выводит
число новых найденных объектов без ссылок. Если ``debugfs`` ещё не
смонтирована, смонтируйте её так::

  # mount -t debugfs nodev /sys/kernel/debug/

Чтобы вывести подробную информацию обо всех возможных просканированных утечках памяти::

  # cat /sys/kernel/debug/kmemleak

Чтобы запустить промежуточное сканирование памяти::

  # echo scan > /sys/kernel/debug/kmemleak

Чтобы очистить список всех текущих возможных утечек памяти::

  # echo clear > /sys/kernel/debug/kmemleak

Затем новые утечки появятся при повторном чтении ``/sys/kernel/debug/kmemleak``.

Обратите внимание, что объекты-сироты перечисляются в порядке их выделения,
и один объект в начале списка может привести к тому, что другие последующие
объекты будут сообщены как сироты.

Параметры сканирования памяти могут быть изменены во время выполнения путём записи в
файл ``/sys/kernel/debug/kmemleak``. Поддерживаются следующие параметры:

- off
    отключить kmemleak (необратимо)
- stack=on
    включить сканирование стеков задач (по умолчанию)
- stack=off
    отключить сканирование стеков задач
- scan=on
    запустить поток автоматического сканирования памяти (по умолчанию)
- scan=off
    остановить поток автоматического сканирования памяти
- scan=<secs>
    задать период автоматического сканирования памяти в секундах
    (по умолчанию 600, 0 — остановить автоматическое сканирование)
- scan
    запустить сканирование памяти
- clear
    очистить список текущих подозреваемых утечек памяти; выполняется путём
    пометки всех текущих сообщённых объектов без ссылок серым цветом,
    либо освобождением всех объектов kmemleak, если kmemleak был отключён.
- dump=<addr>
    вывести информацию об объекте, найденном по адресу <addr>

Kmemleak также может быть отключён во время загрузки путём передачи ``kmemleak=off`` в
командной строке ядра.

Память может быть выделена или освобождена до инициализации kmemleak, и
эти действия сохраняются в раннем буфере журнала. Размер этого буфера
настраивается через опцию CONFIG_DEBUG_KMEMLEAK_MEM_POOL_SIZE.

Если включена опция CONFIG_DEBUG_KMEMLEAK_DEFAULT_OFF, то kmemleak
отключён по умолчанию. Передача ``kmemleak=on`` в командной строке
ядра включает эту функцию.

Если вы получаете ошибки вида «Error while writing to stdout» или «write_loop:
Invalid argument», убедитесь, что kmemleak включён правильно.

Базовый алгоритм
----------------

Выделения памяти через :c:func:`kmalloc`, :c:func:`vmalloc`,
:c:func:`kmem_cache_alloc` и
им подобные трассируются, и указатели вместе с дополнительной
информацией, такой как размер и трассировка стека, сохраняются в rbtree.
Соответствующие вызовы функций освобождения отслеживаются, и указатели
удаляются из структур данных kmemleak.

Выделенный блок памяти считается сиротой, если ни одного указателя на его
начальный адрес или на какое-либо место внутри блока не удаётся найти при
сканировании памяти (включая сохранённые регистры). Это означает, что
у ядра может не быть способа передать адрес выделенного
блока функции освобождения, и поэтому блок считается
утечкой памяти.

Шаги алгоритма сканирования:

  1. пометить все объекты белым цветом (оставшиеся белые объекты позже будут
     считаться сиротами)
  2. сканировать память, начиная с секции данных и стеков, сверяя
     значения с адресами, сохранёнными в rbtree. Если
     найден указатель на белый объект, объект добавляется в
     серый список
  3. сканировать серые объекты на предмет совпадающих адресов (некоторые белые
     объекты могут стать серыми и быть добавлены в конец серого списка), пока
     набор серых не будет исчерпан
  4. оставшиеся белые объекты считаются сиротами и сообщаются через
     /sys/kernel/debug/kmemleak

Некоторые выделенные блоки памяти имеют указатели, сохранённые во внутренних
структурах данных ядра, и они не могут быть обнаружены как сироты. Чтобы
избежать этого, kmemleak также может сохранять число значений, указывающих на
адрес внутри диапазона адресов блока, которые должны быть найдены, чтобы
блок не считался утечкой. Один из примеров — __vmalloc().

Тестирование конкретных участков с помощью kmemleak
---------------------------------------------------

При начальной загрузке вывод страницы /sys/kernel/debug/kmemleak может быть
довольно обширным. Это также может иметь место, если у вас очень
проблемный код при разработке. Чтобы обойти эти ситуации, вы можете использовать
команду «clear», чтобы очистить все сообщённые объекты без ссылок из
вывода /sys/kernel/debug/kmemleak. Выполнив «scan» после «clear»,
вы можете найти новые объекты без ссылок; это должно помочь при тестировании
конкретных участков кода.

Чтобы протестировать критический участок по требованию с чистым kmemleak, выполните::

  # echo clear > /sys/kernel/debug/kmemleak
  ... протестируйте ваше ядро или модули ...
  # echo scan > /sys/kernel/debug/kmemleak

Затем, как обычно, получите отчёт командой::

  # cat /sys/kernel/debug/kmemleak

Освобождение внутренних объектов kmemleak
-----------------------------------------

Чтобы разрешить доступ к ранее найденным утечкам памяти после того, как kmemleak был
отключён пользователем или из-за фатальной ошибки, внутренние объекты kmemleak
не освобождаются при отключении kmemleak, и эти объекты могут занимать
большую часть физической памяти.

В этой ситуации вы можете освободить память командой::

  # echo clear > /sys/kernel/debug/kmemleak

API Kmemleak
------------

Прототипы функций смотрите в заголовочном файле include/linux/kmemleak.h.

- ``kmemleak_init``		 - инициализировать kmemleak
- ``kmemleak_alloc``		 - уведомить о выделении блока памяти
- ``kmemleak_alloc_percpu``	 - уведомить о выделении percpu-блока памяти
- ``kmemleak_vmalloc``		 - уведомить о выделении памяти через vmalloc()
- ``kmemleak_free``		 - уведомить об освобождении блока памяти
- ``kmemleak_free_part``	 - уведомить о частичном освобождении блока памяти
- ``kmemleak_free_percpu``	 - уведомить об освобождении percpu-блока памяти
- ``kmemleak_update_trace``	 - обновить трассировку стека выделения объекта
- ``kmemleak_not_leak``	 - пометить объект как не являющийся утечкой
- ``kmemleak_transient_leak``	 - пометить объект как временную утечку
- ``kmemleak_ignore``		 - не сканировать и не сообщать об объекте как об утечке
- ``kmemleak_scan_area``	 - добавить области сканирования внутри блока памяти
- ``kmemleak_no_scan``	 - не сканировать блок памяти
- ``kmemleak_erase``		 - стереть старое значение в переменной-указателе
- ``kmemleak_alloc_recursive`` - как kmemleak_alloc, но проверяет рекурсивность
- ``kmemleak_free_recursive``	 - как kmemleak_free, но проверяет рекурсивность

Следующие функции принимают физический адрес в качестве указателя на объект
и выполняют соответствующее действие только в том случае, если адрес имеет отображение
в lowmem:

- ``kmemleak_alloc_phys``
- ``kmemleak_free_part_phys``
- ``kmemleak_ignore_phys``

Работа с ложноположительными и ложноотрицательными результатами
---------------------------------------------------------------

Ложноотрицательные результаты — это настоящие утечки памяти (объекты-сироты), о которых
не сообщается kmemleak, потому что найденные при сканировании памяти значения
указывают на такие объекты. Чтобы уменьшить число ложноотрицательных результатов, kmemleak
предоставляет функции kmemleak_ignore, kmemleak_scan_area, kmemleak_no_scan и
kmemleak_erase (см. выше). Стеки задач также увеличивают
число ложноотрицательных результатов, и их сканирование по умолчанию не включено.

Ложноположительные результаты — это объекты, ошибочно сообщённые как утечки памяти
(сироты). Для объектов, заведомо не являющихся утечками, kmemleak предоставляет
функцию kmemleak_not_leak. Функцию kmemleak_ignore также можно использовать, если
известно, что блок памяти не содержит других указателей; тогда он больше не
будет сканироваться.

Некоторые из сообщённых утечек являются лишь временными, особенно в SMP-системах,
из-за указателей, временно сохранённых в регистрах или стеках CPU.
Kmemleak определяет MSECS_MIN_AGE (по умолчанию 1000), представляющий
минимальный возраст объекта, чтобы он был сообщён как утечка памяти.

Ограничения и недостатки
------------------------

Главный недостаток — снижение производительности выделения и освобождения
памяти. Чтобы избежать других издержек, сканирование памяти выполняется только
при чтении файла /sys/kernel/debug/kmemleak. В любом случае этот инструмент
предназначен для отладки, где производительность может быть не самым
важным требованием.

Чтобы алгоритм оставался простым, kmemleak ищет значения, указывающие на любой
адрес внутри диапазона адресов блока. Это может привести к увеличению
числа ложноотрицательных результатов. Тем не менее вероятно, что настоящая утечка памяти
рано или поздно станет видимой.

Другой источник ложноотрицательных результатов — это данные, хранящиеся в
значениях, не являющихся указателями. В будущей версии kmemleak мог бы сканировать только
поля-указатели в выделенных структурах. Эта возможность решила бы многие из
описанных выше случаев ложноотрицательных результатов.

Инструмент может сообщать о ложноположительных результатах. Это случаи, когда выделенный
блок не нуждается в освобождении (некоторые случаи в функциях init_call),
указатель вычисляется другими методами, нежели обычный макрос container_of,
или указатель хранится в месте, не сканируемом kmemleak.

Выделения страниц и ioremap не отслеживаются.

Тестирование с помощью kmemleak-test
------------------------------------

Чтобы проверить, всё ли у вас настроено для использования kmemleak, вы можете использовать модуль
kmemleak-test — модуль, который намеренно допускает утечку памяти. Установите CONFIG_SAMPLE_KMEMLEAK
как модуль (его нельзя использовать как встроенный) и загрузите ядро с включённым
kmemleak. Загрузите модуль и выполните сканирование командой::

        # modprobe kmemleak-test
        # echo scan > /sys/kernel/debug/kmemleak

Обратите внимание, что вы можете не получить результаты мгновенно или при первом сканировании. Когда
kmemleak получает результаты, он запишет в журнал ``kmemleak: <count of leaks> new suspected
memory leaks``. Затем прочитайте файл, чтобы увидеть их::
