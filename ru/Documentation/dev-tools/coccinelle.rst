Coccinelle
==========

Coccinelle — это инструмент для сопоставления с образцом (pattern matching) и
преобразования текста, который находит множество применений в разработке ядра,
включая наложение сложных, охватывающих всё дерево исходников патчей и обнаружение
проблемных шаблонов программирования.

Получение Coccinelle
--------------------

Семантические патчи, входящие в состав ядра, используют возможности и опции,
предоставляемые Coccinelle версии 1.0.0-rc11 и выше.
Использование более ранних версий завершится неудачей, поскольку имена опций,
используемые файлами Coccinelle и coccicheck, были обновлены.

Coccinelle доступен через менеджер пакетов
многих дистрибутивов, например:

 - Debian
 - Fedora
 - Ubuntu
 - OpenSUSE
 - Arch Linux
 - NetBSD
 - FreeBSD

Некоторые дистрибутивные пакеты устарели, и рекомендуется
использовать последнюю версию, выпущенную на домашней странице Coccinelle по адресу
http://coccinelle.lip6.fr/

Или с Github по адресу:

https://github.com/coccinelle/coccinelle

Получив его, выполните следующие команды::

        ./autogen
        ./configure
        make

от имени обычного пользователя и установите его командой::

        sudo make install

Более подробные инструкции по установке для сборки из исходных кодов можно
найти по адресу:

https://github.com/coccinelle/coccinelle/blob/master/install.txt

Использование Coccinelle с ядром Linux
--------------------------------------

В Makefile верхнего уровня определена цель, специфичная для Coccinelle.
Эта цель называется ``coccicheck`` и вызывает интерфейс ``coccicheck``
в каталоге ``scripts``.

Определены четыре базовых режима: ``patch``, ``report``, ``context`` и
``org``. Используемый режим задаётся установкой переменной MODE с помощью
``MODE=<mode>``.

- ``patch`` предлагает исправление, когда это возможно.

- ``report`` генерирует список в следующем формате:
  file:line:column-column: message

- ``context`` подсвечивает интересующие строки и их контекст в
  diff-подобном стиле. Интересующие строки обозначаются символом ``-``.

- ``org`` генерирует отчёт в формате Org mode редактора Emacs.

Обратите внимание, что не все семантические патчи реализуют все режимы. Для
удобства использования Coccinelle режимом по умолчанию является "report".

Два других режима предоставляют некоторые распространённые комбинации этих режимов.

- ``chain`` пробует предыдущие режимы в указанном выше порядке, пока один из них не сработает.

- ``rep+ctxt`` последовательно запускает режим report и режим context.
  Его следует использовать с опцией C (описанной далее),
  которая проверяет код на уровне отдельных файлов.

Примеры
~~~~~~~

Чтобы сформировать отчёт для каждого семантического патча, выполните следующую команду::

		make coccicheck MODE=report

Чтобы получить патчи, выполните::

		make coccicheck MODE=patch


Цель coccicheck применяет каждый доступный семантический патч из
подкаталогов ``scripts/coccinelle`` ко всему ядру Linux.

Для каждого семантического патча предлагается сообщение коммита. Оно даёт
описание проблемы, проверяемой семантическим патчем, и
включает ссылку на Coccinelle.

Как и любой статический анализатор кода, Coccinelle выдаёт ложные
срабатывания. Поэтому отчёты необходимо тщательно проверять, а патчи —
рецензировать.

Чтобы включить подробные сообщения, установите переменную V=, например::

   make coccicheck MODE=report V=1

По умолчанию coccicheck выводит отладочные журналы в stdout и перенаправляет stderr в
/dev/null. Из-за этого вывод coccicheck может быть трудночитаемым и непонятным.
Вместо этого отладочные сообщения и сообщения об ошибках можно записывать в отладочный файл,
установив переменную ``DEBUG_FILE``::

    make coccicheck MODE=report DEBUG_FILE="cocci.log"

Coccinelle не может перезаписать отладочный файл. Чтобы не удалять журнальный
файл каждый раз, можно включить в имя отладочного файла дату и время::

    make coccicheck MODE=report DEBUG_FILE="cocci-$(date -Iseconds).log"

Параллелизация Coccinelle
-------------------------

По умолчанию coccicheck старается выполняться настолько параллельно, насколько это возможно. Чтобы изменить
степень параллелизма, установите переменную J=. Например, чтобы запустить на 4 CPU::

   make coccicheck MODE=report J=4

Начиная с Coccinelle 1.0.2, Coccinelle использует для параллелизации Ocaml parmap;
если поддержка этого обнаружена, вы получите выгоду от параллелизации parmap.

Когда parmap включён, coccicheck включает динамическую балансировку нагрузки, используя
аргумент ``--chunksize 1``. Это обеспечивает подачу работы потокам
по одной, так что мы избегаем ситуации, когда большая часть работы выполняется лишь
несколькими потоками. При динамической балансировке нагрузки, если поток завершается рано, мы продолжаем
подавать ему больше работы.

Когда parmap включён, если в Coccinelle происходит ошибка, это
значение ошибки распространяется обратно, и возвращаемое значение команды ``make coccicheck``
фиксирует это возвращаемое значение.

Использование Coccinelle с одним семантическим патчем
-----------------------------------------------------

Необязательную переменную make COCCI можно использовать для проверки одного
семантического патча. В этом случае переменную нужно инициализировать
именем применяемого семантического патча.

Например::

	make coccicheck COCCI=<my_SP.cocci> MODE=patch

или::

	make coccicheck COCCI=<my_SP.cocci> MODE=report


Управление тем, какие файлы обрабатываются Coccinelle
-----------------------------------------------------

По умолчанию проверяется всё дерево исходных кодов ядра.

Чтобы применить Coccinelle к конкретному каталогу, можно использовать ``M=``.
Например, чтобы проверить drivers/net/wireless/, можно написать::

    make coccicheck M=drivers/net/wireless/

Чтобы применять Coccinelle на уровне файлов, а не на уровне каталогов,
makefile использует переменную C для выбора файлов, с которыми работать.
Эту переменную можно использовать для запуска скриптов для всего ядра, для
конкретного каталога или для отдельного файла.

Например, чтобы проверить drivers/bluetooth/bfusb.c, переменной C
передаётся значение 1, чтобы проверять файлы, которые, по мнению make,
нуждаются в компиляции.::

    make C=1 CHECK=scripts/coccicheck drivers/bluetooth/bfusb.o

Значение 2 передаётся переменной C, чтобы проверять файлы независимо от
того, нуждаются они в компиляции или нет.::

    make C=2 CHECK=scripts/coccicheck drivers/bluetooth/bfusb.o

В этих режимах, работающих на уровне файлов, не отображается информация
о семантических патчах и не предлагается сообщение коммита.

Это по умолчанию запускает каждый семантический патч из scripts/coccinelle.
Переменную COCCI дополнительно можно использовать для применения только одного
семантического патча, как показано в предыдущем разделе.

Режим "report" используется по умолчанию. Вы можете выбрать другой с помощью
переменной MODE, объяснённой выше.

Отладка SmPL-патчей Coccinelle
------------------------------

Использование coccicheck предпочтительно, так как оно предоставляет в командной строке spatch
include-опции, соответствующие опциям, используемым при компиляции ядра.
Узнать, какие это опции, можно с помощью V=1; затем можно
вручную запустить Coccinelle с добавленными отладочными опциями.

Более простой подход к отладке запуска Coccinelle на SmPL-патчах — попросить
coccicheck перенаправить stderr в отладочный файл. Как упоминалось в примерах, по
умолчанию stderr перенаправляется в /dev/null; если вы хотите захватить stderr, вы
можете указать coccicheck опцию ``DEBUG_FILE="file.txt"``. Например::

    rm -f cocci.err
    make coccicheck COCCI=scripts/coccinelle/free/kfree.cocci MODE=report DEBUG_FILE=cocci.err
    cat cocci.err

Вы можете использовать SPFLAGS для добавления отладочных флагов; например, при отладке
вы можете захотеть добавить в SPFLAGS оба флага ``--profile --show-trying``. Например,
вы можете захотеть использовать::

    rm -f err.log
    export COCCI=scripts/coccinelle/misc/irqf_oneshot.cocci
    make coccicheck DEBUG_FILE="err.log" MODE=report SPFLAGS="--profile --show-trying" M=./drivers/mfd

Теперь err.log будет содержать информацию профилирования, а stdout будет
предоставлять некоторую информацию о ходе работы по мере того, как Coccinelle
продвигается в работе.

ПРИМЕЧАНИЕ:

Поддержка DEBUG_FILE доступна только при использовании coccinelle >= 1.0.2.

В настоящее время поддержка DEBUG_FILE доступна только для проверки каталогов, а
не отдельных файлов. Это связано с тем, что проверка одного файла требует двукратного
вызова spatch, из-за чего DEBUG_FILE устанавливается оба раза в одно и то же значение,
что приводит к ошибке.

Поддержка .cocciconfig
----------------------

Coccinelle поддерживает чтение .cocciconfig для опций Coccinelle по умолчанию, которые
должны использоваться при каждом запуске spatch. Порядок старшинства
переменных для .cocciconfig следующий:

- Сначала обрабатывается домашний каталог вашего текущего пользователя
- Затем обрабатывается каталог, из которого вызывается spatch
- В последнюю очередь обрабатывается каталог, указанный опцией ``--dir``, если она используется

``make coccicheck`` также поддерживает использование целей M=. Если вы не указываете
ни одной цели M=, предполагается, что вы хотите нацелиться на всё ядро.
Скрипт coccicheck ядра содержит::

    OPTIONS="--dir $srcroot $COCCIINCLUDE"

Здесь $srcroot ссылается на каталог исходных кодов цели: он указывает на
каталог исходных кодов внешнего модуля, когда используется M=, и в противном случае — на каталог
исходных кодов ядра. Третье правило гарантирует, что spatch читает .cocciconfig из
целевого каталога, позволяя внешним модулям иметь собственный файл .cocciconfig.

Если вы не используете цель coccicheck ядра, придерживайтесь приведённого выше порядка
старшинства логики чтения .cocciconfig. Если вы используете цель coccicheck ядра,
переопределяйте любые настройки .coccicheck ядра с помощью SPFLAGS.

Мы помогаем Coccinelle при использовании с Linux набором разумных значений по
умолчанию для Linux в нашем собственном .cocciconfig для Linux. Это подсказывает coccinelle,
что для запросов ``git grep`` через coccigrep можно использовать git. Таймаута в 200
секунд на данный момент должно быть достаточно.

Опции, подхватываемые coccinelle при чтении .cocciconfig, не появляются
в виде аргументов процессов spatch, запущенных в вашей системе. Чтобы подтвердить, какие
опции будут использоваться Coccinelle, запустите::

      spatch --print-options-only

Вы можете переопределить их собственной предпочитаемой опцией индекса с помощью SPFLAGS. Обратите
внимание, что при наличии конфликтующих опций Coccinelle отдаёт предпочтение
последним переданным опциям. Использовать idutils через .cocciconfig возможно, однако,
учитывая порядок старшинства, которому следует Coccinelle, поскольку теперь ядро
несёт собственный .cocciconfig, при желании использовать idutils вам потребуется применять SPFLAGS.
См. раздел "Дополнительные флаги" ниже для подробностей о том, как использовать
idutils.

Дополнительные флаги
--------------------

Дополнительные флаги можно передавать в spatch через переменную SPFLAGS.
Это работает, поскольку Coccinelle уважает последние флаги,
переданные ему, когда опции конфликтуют. ::

    make SPFLAGS=--use-glimpse coccicheck

Coccinelle также поддерживает idutils, но требует coccinelle >= 1.0.6.
Когда ID-файл не указан, coccinelle предполагает, что файл вашей базы данных ID
находится в файле .id-utils.index на верхнем уровне ядра. Coccinelle
несёт скрипт scripts/idutils_index.sh, который создаёт базу данных командой::

    mkid -i C --output .id-utils.index

Если у вас другое имя файла базы данных, вы также можете просто создать символьную ссылку с этим
именем. ::

    make SPFLAGS=--use-idutils coccicheck

Либо вы можете явно указать имя файла базы данных, например::

    make SPFLAGS="--use-idutils /full-path/to/ID" coccicheck

См. ``spatch --help``, чтобы узнать больше об опциях spatch.

Обратите внимание, что опции ``--use-glimpse`` и ``--use-idutils``
требуют внешних инструментов для индексации кода. Поэтому ни одна из них
не активна по умолчанию. Однако, проиндексировав код одним
из этих инструментов и в зависимости от используемого cocci-файла,
spatch мог бы обрабатывать всю кодовую базу быстрее.

Опции, специфичные для SmPL-патча
---------------------------------

SmPL-патчи могут иметь собственные требования к опциям, передаваемым
в Coccinelle. Опции, специфичные для SmPL-патча, можно предоставить,
указав их в начале SmPL-патча, например::

	// Options: --no-includes --include-headers

Требования SmPL-патча к Coccinelle
----------------------------------

По мере добавления возможностей в Coccinelle некоторые более продвинутые SmPL-патчи
могут требовать более новых версий Coccinelle. Если SmPL-патч требует
минимальной версии Coccinelle, это можно указать следующим образом,
например, если требуется как минимум Coccinelle >= 1.0.5::

	// Requires: 1.0.5

Предложение новых семантических патчей
--------------------------------------

Новые семантические патчи могут предлагаться и отправляться разработчиками
ядра. Для ясности их следует организовывать в
подкаталогах ``scripts/coccinelle/``.


Подробное описание режима ``report``
------------------------------------

``report`` генерирует список в следующем формате::

  file:line:column-column: message

Пример
~~~~~~

Запуск::

	make coccicheck MODE=report COCCI=scripts/coccinelle/api/err_cast.cocci

выполнит следующую часть SmPL-скрипта::

   <smpl>
   @r depends on !context && !patch && (org || report)@
   expression x;
   position p;
   @@

     ERR_PTR@p(PTR_ERR(x))

   @script:python depends on report@
   p << r.p;
   x << r.x;
   @@

   msg="ERR_CAST can be used with %s" % (x)
   coccilib.report.print_report(p[0], msg)
   </smpl>

Этот фрагмент SmPL генерирует записи на стандартном выводе, как
показано ниже::

    /home/user/linux/crypto/ctr.c:188:9-16: ERR_CAST can be used with alg
    /home/user/linux/crypto/authenc.c:619:9-16: ERR_CAST can be used with auth
    /home/user/linux/crypto/xts.c:227:9-16: ERR_CAST can be used with alg


Подробное описание режима ``patch``
-----------------------------------

Когда режим ``patch`` доступен, он предлагает исправление для каждой выявленной
проблемы.

Пример
~~~~~~

Запуск::

	make coccicheck MODE=patch COCCI=scripts/coccinelle/api/err_cast.cocci

выполнит следующую часть SmPL-скрипта::

    <smpl>
    @ depends on !context && patch && !org && !report @
    expression x;
    @@

    - ERR_PTR(PTR_ERR(x))
    + ERR_CAST(x)
    </smpl>

Этот фрагмент SmPL генерирует фрагменты патча (hunks) на стандартном выводе, как
показано ниже::

    diff -u -p a/crypto/ctr.c b/crypto/ctr.c
    --- a/crypto/ctr.c 2010-05-26 10:49:38.000000000 +0200
    +++ b/crypto/ctr.c 2010-06-03 23:44:49.000000000 +0200
    @@ -185,7 +185,7 @@ static struct crypto_instance *crypto_ct
 	alg = crypto_attr_alg(tb[1], CRYPTO_ALG_TYPE_CIPHER,
 				  CRYPTO_ALG_TYPE_MASK);
 	if (IS_ERR(alg))
    -		return ERR_PTR(PTR_ERR(alg));
    +		return ERR_CAST(alg);

 	/* Block size must be >= 4 bytes. */
 	err = -EINVAL;

Подробное описание режима ``context``
-------------------------------------

``context`` подсвечивает интересующие строки и их контекст
в diff-подобном стиле.

      **ПРИМЕЧАНИЕ**: Генерируемый diff-подобный вывод НЕ является применимым патчем.
      Назначение режима ``context`` — подсветить важные строки
      (помеченные минусом, ``-``) и дать некоторое количество окружающих
      строк контекста вокруг них. Этот вывод можно использовать с режимом diff
      редактора Emacs для рецензирования кода.

Пример
~~~~~~

Запуск::

	make coccicheck MODE=context COCCI=scripts/coccinelle/api/err_cast.cocci

выполнит следующую часть SmPL-скрипта::

    <smpl>
    @ depends on context && !patch && !org && !report@
    expression x;
    @@

    * ERR_PTR(PTR_ERR(x))
    </smpl>

Этот фрагмент SmPL генерирует фрагменты diff (hunks) на стандартном выводе, как
показано ниже::

    diff -u -p /home/user/linux/crypto/ctr.c /tmp/nothing
    --- /home/user/linux/crypto/ctr.c	2010-05-26 10:49:38.000000000 +0200
    +++ /tmp/nothing
    @@ -185,7 +185,6 @@ static struct crypto_instance *crypto_ct
 	alg = crypto_attr_alg(tb[1], CRYPTO_ALG_TYPE_CIPHER,
 				  CRYPTO_ALG_TYPE_MASK);
 	if (IS_ERR(alg))
    -		return ERR_PTR(PTR_ERR(alg));

 	/* Block size must be >= 4 bytes. */
 	err = -EINVAL;

Подробное описание режима ``org``
---------------------------------

``org`` генерирует отчёт в формате Org mode редактора Emacs.

Пример
~~~~~~

Запуск::

	make coccicheck MODE=org COCCI=scripts/coccinelle/api/err_cast.cocci

выполнит следующую часть SmPL-скрипта::

    <smpl>
    @r depends on !context && !patch && (org || report)@
    expression x;
    position p;
    @@

      ERR_PTR@p(PTR_ERR(x))

    @script:python depends on org@
    p << r.p;
    x << r.x;
    @@

    msg="ERR_CAST can be used with %s" % (x)
    msg_safe=msg.replace("[","@(").replace("]",")")
    coccilib.org.print_todo(p[0], msg_safe)
    </smpl>

Этот фрагмент SmPL генерирует записи Org на стандартном выводе, как
