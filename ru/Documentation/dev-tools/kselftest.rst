==========================
Самотесты ядра Linux
==========================

Ядро содержит набор «самотестов» (self tests) в каталоге tools/testing/selftests/.
Они задуманы как небольшие тесты, проверяющие отдельные пути исполнения кода в
ядре. Тесты предназначены для запуска после сборки, установки и загрузки ядра.

Kselftest из mainline можно запускать на более старых стабильных ядрах. Запуск
тестов из mainline обеспечивает наилучшее покрытие. Несколько тестовых стендов
прогоняют набор kselftest из mainline на стабильных релизах. Причина в том, что
когда добавляется новый тест для проверки существующего кода ради регрессионного
тестирования некоторой ошибки, мы должны иметь возможность запустить этот тест на
более старом ядре. Поэтому важно сохранять код, который всё ещё способен
тестировать более старое ядро, и обеспечивать корректный пропуск теста на более
новых релизах.

Дополнительную информацию о фреймворке Kselftest и о том, как писать новые тесты с
его использованием, можно найти в вики Kselftest:

https://kselftest.wiki.kernel.org/

На некоторых системах тесты горячего подключения (hot-plug) могут зависать
навсегда, ожидая, когда CPU и память будут готовы к отключению. Для запуска
полного набора тестов горячего подключения создана отдельная цель hot-plug. В
режиме по умолчанию тесты hot-plug выполняются в безопасном режиме с ограниченной
областью. В ограниченном режиме тест cpu-hotplug запускается на одном CPU вместо
всех CPU, поддерживающих горячее подключение, а тест memory hotplug запускается на
2% памяти, поддерживающей горячее подключение, вместо 10%.

kselftest выполняется как процесс в пространстве пользователя.  Для тестов, которые
можно написать/запускать в пространстве пользователя, может быть удобен `Test Harness`_.
Для тестов, которые нужно выполнять в пространстве ядра, может быть удобен `Test Module`_.

Запуск самотестов (тесты hotplug выполняются в ограниченном режиме)
===================================================================

Чтобы собрать тесты::

  $ make headers
  $ make -C tools/testing/selftests

Чтобы запустить тесты::

  $ make -C tools/testing/selftests run_tests

Чтобы собрать и запустить тесты одной командой, используйте::

  $ make kselftest

Учтите, что некоторым тестам потребуются права root.

Kselftest поддерживает сохранение выходных файлов в отдельном каталоге с
последующим запуском тестов. Для размещения выходных файлов в отдельном каталоге
поддерживаются два синтаксиса. В обоих случаях рабочим каталогом должен быть корень
исходников ядра. Это относится и к разделу «Запуск подмножества самотестов» ниже.

Чтобы собрать и сохранить выходные файлы в отдельном каталоге с помощью O= ::

  $ make O=/tmp/kselftest kselftest

Чтобы собрать и сохранить выходные файлы в отдельном каталоге с помощью KBUILD_OUTPUT ::

  $ export KBUILD_OUTPUT=/tmp/kselftest; make kselftest

Присваивание O= имеет приоритет над переменной окружения KBUILD_OUTPUT.

Приведённые выше команды по умолчанию запускают тесты и печатают полный отчёт о
прохождении/непрохождении. Kselftest поддерживает опцию «summary», чтобы облегчить
понимание результатов тестов. Подробные результаты каждого отдельного теста при
указании опции summary можно найти в файле(ах) /tmp/testname. Это относится и к
разделу «Запуск подмножества самотестов» ниже.

Чтобы запустить kselftest с включённой опцией summary ::

  $ make summary=1 kselftest

Запуск подмножества самотестов
==============================

Вы можете использовать переменную «TARGETS» в командной строке make, чтобы указать
один тест для запуска или список тестов для запуска.

Чтобы запустить только тесты, нацеленные на одну подсистему::

  $ make -C tools/testing/selftests TARGETS=ptrace run_tests

Вы можете указать несколько тестов для сборки и запуска::

  $  make TARGETS="size timers" kselftest

Чтобы собрать и сохранить выходные файлы в отдельном каталоге с помощью O= ::

  $ make O=/tmp/kselftest TARGETS="size timers" kselftest

Чтобы собрать и сохранить выходные файлы в отдельном каталоге с помощью KBUILD_OUTPUT ::

  $ export KBUILD_OUTPUT=/tmp/kselftest; make TARGETS="size timers" kselftest

Кроме того, вы можете использовать переменную «SKIP_TARGETS» в командной строке
make, чтобы указать одну или несколько целей для исключения из списка TARGETS.

Чтобы запустить все тесты, кроме одной подсистемы::

  $ make -C tools/testing/selftests SKIP_TARGETS=ptrace run_tests

Вы можете указать несколько тестов для пропуска::

  $  make SKIP_TARGETS="size timers" kselftest

Вы также можете указать ограниченный список тестов для совместного запуска вместе с
отдельным списком пропуска::

  $  make TARGETS="breakpoints size timers" SKIP_TARGETS=size kselftest

Список всех возможных целей см. в Makefile верхнего уровня
tools/testing/selftests/Makefile.

Запуск полного набора самотестов hotplug
========================================

Чтобы собрать тесты hotplug::

  $ make -C tools/testing/selftests hotplug

Чтобы запустить тесты hotplug::

  $ make -C tools/testing/selftests run_hotplug

Учтите, что некоторым тестам потребуются права root.


Установка самотестов
====================

Вы можете использовать цель «install» команды «make» (которая вызывает инструмент
`kselftest_install.sh`), чтобы установить самотесты в расположение по умолчанию
(`tools/testing/selftests/kselftest_install`) или в заданное пользователем
расположение через переменную «make» `INSTALL_PATH`.

Чтобы установить самотесты в расположение по умолчанию::

   $ make -C tools/testing/selftests install

Чтобы установить самотесты в заданное пользователем расположение::

   $ make -C tools/testing/selftests install INSTALL_PATH=/some/other/path

Запуск установленных самотестов
===============================

В каталоге установки, а также в tarball'е Kselftest находится скрипт с именем
`run_kselftest.sh` для запуска тестов.

Чтобы запустить установленные Kselftest'ы, достаточно сделать следующее. Учтите,
что некоторым тестам потребуются права root::

   $ cd kselftest_install
   $ ./run_kselftest.sh

Чтобы увидеть список доступных тестов, можно использовать опцию `-l`::

   $ ./run_kselftest.sh -l

Опцию `-c` можно использовать для запуска всех тестов из тестовой коллекции, а
опцию `-t` — для конкретных отдельных тестов. Любую из них можно использовать
несколько раз::

   $ ./run_kselftest.sh -c size -c seccomp -t timers:posix_timers -t timer:nanosleep

О прочих возможностях см. вывод справки скрипта, доступный с опцией `-h`.

Таймаут для самотестов
======================

Самотесты рассчитаны на быстрое выполнение, поэтому для каждого теста используется
таймаут по умолчанию в 45 секунд. Тесты могут переопределить таймаут по умолчанию,
добавив файл settings в своём каталоге и задав там переменную timeout равной
желаемому верхнему пределу времени для теста. Лишь немногие тесты переопределяют
таймаут значением выше 45 секунд, и самотесты стремятся сохранять это так.
Таймауты в самотестах не считаются фатальными, поскольку система, в которой
выполняется тест, может меняться, и это также может изменить ожидаемое время
выполнения теста. Если вы управляете системами, на которых будут выполняться тесты,
вы можете настроить запускатель тестов на этих системах на использование большего
или меньшего таймаута в командной строке, как с аргументом `-o` или
`--override-timeout`. Например, чтобы использовать 165 секунд вместо этого, следует
использовать::

   $ ./run_kselftest.sh --override-timeout 165

Вы можете посмотреть вывод TAP, чтобы понять, столкнулись ли вы с таймаутом.
Запускатели тестов, которые знают, что тест должен выполняться в пределах
определённого времени, могут затем по желанию трактовать эти таймауты как
фатальные.

Упаковка самотестов
===================

В некоторых случаях желательна упаковка, например когда тесты нужно запускать на
другой системе. Чтобы упаковать самотесты, выполните::

   $ make -C tools/testing/selftests gen_tar

Это создаёт tarball в каталоге `INSTALL_PATH/kselftest-packages`. По умолчанию
используется формат `.gz`. Формат сжатия tar можно переопределить, указав
переменную make `FORMAT`. Поддерживается любое значение, распознаваемое опцией
`tar's auto-compress`_, например::

    $ make -C tools/testing/selftests gen_tar FORMAT=.xz

`make gen_tar` вызывает `make install`, поэтому вы можете использовать его для
упаковки подмножества тестов, применяя переменные, указанные в разделе
`Запуск подмножества самотестов`_::

    $ make -C tools/testing/selftests gen_tar TARGETS="size" FORMAT=.xz

.. _tar's auto-compress: https://www.gnu.org/software/tar/manual/html_node/gzip.html#auto_002dcompress

Внесение новых тестов
=====================

В общем случае правила для самотестов таковы

 * Делайте столько, сколько можете, если вы не root;

 * Не выполняйтесь слишком долго;

 * Не ломайте сборку ни на одной архитектуре, и

 * Не приводите к сбою команды верхнего уровня «make run_tests», если ваша
   функциональность не сконфигурирована.

 * Вывод тестов должен соответствовать стандарту TAP, чтобы обеспечить высокое
   качество тестирования и фиксировать сбои/ошибки с конкретными деталями.
   Заголовочные файлы kselftest.h и kselftest_harness.h предоставляют обёртки для
   вывода результатов тестов. Эти обёртки следует использовать для сообщений о
   прохождении (pass), непрохождении (fail), выходе (exit) и пропуске (skip).
   CI-системы могут легко разбирать сообщения вывода TAP для определения
   результатов тестов.

Внесение новых тестов (подробности)
===================================

 * В своём Makefile используйте средства из lib.mk, подключив его, вместо того
   чтобы изобретать колесо. При необходимости указывайте флаги и флаги генерации
   бинарников перед подключением lib.mk. ::

    CFLAGS = $(KHDR_INCLUDES)
    TEST_GEN_PROGS := close_range_test
    include ../lib.mk

 * Используйте TEST_GEN_XXX, если такие бинарники или файлы генерируются во время
   компиляции.

   TEST_PROGS, TEST_GEN_PROGS означают, что это исполняемый файл, тестируемый по
   умолчанию.

   TEST_GEN_MODS_DIR должен использоваться тестами, которым требуется собрать
   модули перед началом теста. Переменная будет содержать имя каталога с модулями.

   TEST_CUSTOM_PROGS должен использоваться тестами, которым требуются особые
   правила сборки и которые препятствуют использованию общих правил сборки.

   TEST_PROGS предназначены для тестовых shell-скриптов. Убедитесь, что у
   shell-скрипта установлен бит exec. Иначе run_tests из lib.mk сгенерирует
   предупреждение.

   TEST_CUSTOM_PROGS и TEST_PROGS будут запускаться общим run_tests.

   TEST_PROGS_EXTENDED, TEST_GEN_PROGS_EXTENDED означают, что это исполняемый файл,
   который не тестируется по умолчанию.

   TEST_FILES, TEST_GEN_FILES означают, что это файл, используемый тестом.

   TEST_INCLUDES похож на TEST_FILES, он перечисляет файлы, которые должны быть
   включены при экспорте или установке тестов, со следующими отличиями:

    * символические ссылки на файлы в других каталогах сохраняются
    * при копировании файлов в выходной каталог сохраняется часть путей ниже
      tools/testing/selftests/

   TEST_INCLUDES предназначен для перечисления зависимостей, расположенных в других
   каталогах иерархии selftests.

 * Сначала используйте заголовочные файлы из исходников ядра и/или git-репозитория,
   и лишь затем системные заголовочные файлы.  Чтобы иметь возможность находить
   регрессии, основное внимание следует уделять заголовочным файлам релиза ядра, а
   не заголовочным файлам, установленным дистрибутивом в системе. Используйте
   KHDR_INCLUDES в Makefile, чтобы подключать заголовочные файлы из исходников ядра.

 * Если тесту требуются включённые определённые параметры конфигурации ядра,
   добавьте файл config в каталоге теста, чтобы включить их.

   например: tools/testing/selftests/android/config

 * Создайте файл .gitignore в каталоге теста и добавьте в него все генерируемые
   объекты.

 * Добавьте имя нового теста в TARGETS в selftests/Makefile::

    TARGETS += android

 * Все изменения должны проходить::

    kselftest-{all,install,clean,gen_tar}
    kselftest-{all,install,clean,gen_tar} O=abo_path
    kselftest-{all,install,clean,gen_tar} O=rel_path
    make -C tools/testing/selftests {all,install,clean,gen_tar}
    make -C tools/testing/selftests {all,install,clean,gen_tar} O=abs_path
    make -C tools/testing/selftests {all,install,clean,gen_tar} O=rel_path

Test Module
===========

Kselftest тестирует ядро из пространства пользователя.  Иногда что-то нужно
тестировать изнутри ядра, и один из способов сделать это — создать тестовый модуль.
Мы можем привязать модуль к фреймворку kselftest, используя запускатель тестов в
виде shell-скрипта.  ``kselftest/module.sh`` предназначен для облегчения этого
процесса.  Также предоставляется заголовочный файл для помощи в написании модулей
ядра, предназначенных для использования с kselftest:

- ``tools/testing/selftests/kselftest_module.h``
- ``tools/testing/selftests/kselftest/module.sh``

Учтите, что тестовые модули должны помечать ядро (taint) флагом TAINT_TEST. Это
произойдёт автоматически для модулей, находящихся в каталоге ``tools/testing/``,
или для модулей, использующих заголовочный файл ``kselftest_module.h``,
упомянутый выше. В противном случае вам потребуется добавить
``MODULE_INFO(test, "Y")`` в исходный код вашего модуля. Самотесты, которые не
загружают модули, как правило, не должны помечать ядро, но в случаях, когда
загружается не тестовый модуль, TEST_TAINT можно применить из пространства
пользователя, записав в ``/proc/sys/kernel/tainted``.

Как использовать
----------------

Здесь мы показываем типичные шаги для создания тестового модуля и привязки его к
kselftest.  В качестве примера мы используем kselftest'ы для lib/.

1. Создайте тестовый модуль

2. Создайте тестовый скрипт, который будет запускать (загружать/выгружать) модуль,
   например ``tools/testing/selftests/lib/bitmap.sh``

3. Добавьте строку в файл config, например ``tools/testing/selftests/lib/config``

4. Добавьте тестовый скрипт в makefile, например ``tools/testing/selftests/lib/Makefile``

5. Проверьте, что это работает:

.. code-block:: sh

   # Assumes you have booted a fresh build of this kernel tree
   cd /path/to/linux/tree
   make kselftest-merge
   make modules
   sudo make modules_install
   make TARGETS=lib kselftest

Пример модуля
-------------

Минимальный тестовый модуль может выглядеть так:

.. code-block:: c

   // SPDX-License-Identifier: GPL-2.0+

   #define pr_fmt(fmt) KBUILD_MODNAME ": " fmt

   #include "../tools/testing/selftests/kselftest_module.h"

   KSTM_MODULE_GLOBALS();

   /*
    * Kernel module for testing the foobinator
    */

   static int __init test_function()
   {
           ...
   }

   static void __init selftest(void)
   {
           KSTM_CHECK_ZERO(do_test_case("", 0));
   }

   KSTM_MODULE_LOADERS(test_foo);
   MODULE_AUTHOR("John Developer <jd@fooman.org>");
   MODULE_LICENSE("GPL");
   MODULE_INFO(test, "Y");

Пример тестового скрипта
------------------------

.. code-block:: sh

    #!/bin/bash
    # SPDX-License-Identifier: GPL-2.0+
    $(dirname $0)/../kselftest/module.sh "foo" test_foo


Test Harness
============

Файл kselftest_harness.h содержит полезные вспомогательные средства для построения
тестов.  Test harness предназначен для тестирования в пространстве пользователя; о
тестировании в пространстве ядра см. `Test Module`_ выше.

В качестве примера можно использовать тесты из
tools/testing/selftests/seccomp/seccomp_bpf.c.
