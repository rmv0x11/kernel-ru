==========
Checkpatch
==========

Checkpatch (scripts/checkpatch.pl) — это perl-скрипт, который проверяет патчи
на тривиальные нарушения стиля и при желании исправляет их.  Checkpatch также
может быть запущен на содержимом файлов и без дерева исходников ядра.

Checkpatch не всегда прав. Ваше суждение имеет приоритет над сообщениями
checkpatch.  Если ваш код выглядит лучше с этими нарушениями, то, вероятно,
его лучше оставить как есть.


Опции
=====

В этом разделе описаны опции, с которыми может быть запущен checkpatch.

Использование::

  ./scripts/checkpatch.pl [OPTION]... [FILE]...

Доступные опции:

 - -q,  --quiet

   Включить тихий режим.

 - -v,  --verbose
   Включить подробный режим.  Выводятся дополнительные подробные описания
   проверок, чтобы предоставить информацию о том, почему показано данное
   конкретное сообщение.

 - --no-tree

   Запустить checkpatch без дерева исходников ядра.

 - --no-signoff

   Отключить проверку строки 'Signed-off-by'.  Sign-off — это простая строка
   в конце пояснения к патчу, которая удостоверяет, что вы написали его или
   иным образом имеете право передавать его как патч с открытым исходным кодом.

   Пример::

	 Signed-off-by: Random J Developer <random@developer.example.org>

   Установка этого флага фактически прекращает выдачу сообщения об отсутствии
   строки signed-off-by в контексте патча.

 - --patch

   Рассматривать FILE как патч.  Это опция по умолчанию, и её не нужно
   указывать явно.

 - --emacs

   Установить вывод в формате окна компиляции emacs.  Это позволяет
   пользователям emacs переходить от ошибки в окне компиляции напрямую к
   проблемной строке в патче.

 - --terse

   Выводить только одну строку на отчёт.

 - --showfile

   Показывать позицию в файле diff вместо позиции во входном файле.

 - -g,  --git

   Рассматривать FILE как одиночный коммит или диапазон git-ревизий.

   Одиночный коммит с помощью:

   - <rev>
   - <rev>^
   - <rev>~n

   Несколько коммитов с помощью:

   - <rev1>..<rev2>
   - <rev1>...<rev2>
   - <rev>-<count>

 - -f,  --file

   Рассматривать FILE как обычный файл исходного кода.  Эту опцию необходимо
   использовать при запуске checkpatch на файлах исходного кода в ядре.

 - --subjective,  --strict

   Включить более строгие проверки в checkpatch.  По умолчанию проверки,
   выдаваемые как CHECK, не активируются.  Используйте этот флаг для активации
   проверок CHECK.

 - --list-types

   Каждое сообщение, выдаваемое checkpatch, имеет связанный с ним TYPE.
   Добавьте этот флаг, чтобы отобразить все типы в checkpatch.

   Обратите внимание, что когда этот флаг активен, checkpatch не читает входной
   FILE и не выдаёт сообщений.  Выводится только список типов в checkpatch.

 - --types TYPE(,TYPE2...)

   Отображать только сообщения с заданными типами.

   Пример::

     ./scripts/checkpatch.pl mypatch.patch --types EMAIL_SUBJECT,BRACES

 - --ignore TYPE(,TYPE2...)

   Checkpatch не будет выдавать сообщения для указанных типов.

   Пример::

     ./scripts/checkpatch.pl mypatch.patch --ignore EMAIL_SUBJECT,BRACES

 - --show-types

   По умолчанию checkpatch не отображает тип, связанный с сообщениями.
   Установите этот флаг, чтобы показывать тип сообщения в выводе.

 - --max-line-length=n

   Установить максимальную длину строки (по умолчанию 100).  Если строка
   превышает указанную длину, выдаётся сообщение LONG_LINE.


   Уровень сообщения различается для контекстов патча и файла.  Для патчей
   выдаётся WARNING.  В то время как для файлов выдаётся более мягкая CHECK.
   Поэтому для контекстов файла необходимо также включить флаг --strict.

 - --min-conf-desc-length=n

   Установить минимальную длину описания записи Kconfig; если короче —
   предупредить.

 - --tab-size=n

   Установить количество пробелов для табуляции (по умолчанию 8).

 - --root=PATH

   PATH к корню дерева исходников ядра.

   Эту опцию необходимо указывать при вызове checkpatch вне корня ядра.

 - --no-summary

   Подавить пофайловую сводку.

 - --mailback

   Выдавать отчёт только в случае Warnings или Errors.  Более мягкие Checks
   из этого исключаются.

 - --summary-file

   Включать имя файла в сводку.

 - --debug KEY=[0|1]

   Включить/выключить отладку KEY, где KEY — одно из 'values', 'possible',
   'type' и 'attr' (по умолчанию всё выключено).

 - --fix

   Это ЭКСПЕРИМЕНТАЛЬНАЯ функциональность.  Если существуют исправимые ошибки,
   создаётся файл <inputfile>.EXPERIMENTAL-checkpatch-fixes, в котором
   автоматически исправляемые ошибки исправлены.

 - --fix-inplace

   ЭКСПЕРИМЕНТАЛЬНАЯ — аналогична --fix, но входной файл перезаписывается
   исправлениями.

   НЕ ИСПОЛЬЗУЙТЕ этот флаг, если вы не абсолютно уверены и у вас нет
   резервной копии.

 - --ignore-perl-version

   Отменить проверку версии perl.  После включения этого флага могут возникнуть
   ошибки времени выполнения, если версия perl не соответствует указанному
   минимуму.

 - --codespell

   Использовать словарь codespell для проверки орфографических ошибок.

 - --codespellfile

   Использовать указанный файл codespell.
   По умолчанию '/usr/share/codespell/dictionary.txt'.

 - --typedefsfile

   Читать дополнительные типы из этого файла.

 - --color[=WHEN]

   Использовать цвета 'always', 'never' или только когда вывод направлен в
   терминал ('auto').
   По умолчанию 'auto'.

 - --kconfig-prefix=WORD

   Использовать WORD как префикс для символов Kconfig (по умолчанию `CONFIG_`).

 - -h, --help, --version

   Отобразить текст справки.

Уровни сообщений
================

Сообщения в checkpatch делятся на три уровня. Уровни сообщений в checkpatch
обозначают серьёзность ошибки. Это:

 - ERROR

   Это самый строгий уровень.  Сообщения типа ERROR следует воспринимать
   серьёзно, так как они обозначают вещи, которые с очень высокой вероятностью
   неверны.

 - WARNING

   Это следующий по строгости уровень.  Сообщения типа WARNING требуют более
   внимательного рассмотрения.  Но это мягче, чем ERROR.

 - CHECK

   Это самый мягкий уровень.  Это вещи, которые могут потребовать некоторого
   обдумывания.

Описания типов
==============

Этот раздел содержит описание всех типов сообщений в checkpatch.

.. Types in this section are also parsed by checkpatch.
.. The types are grouped into subsections based on use.


Стиль выделения памяти
----------------------

  **ALLOC_ARRAY_ARGS**
    Первым аргументом для kcalloc или kmalloc_array должно быть
    количество элементов.  sizeof() в качестве первого аргумента, как правило,
    неверен.

    См.: https://www.kernel.org/doc/html/latest/core-api/memory-allocation.html

  **ALLOC_SIZEOF_STRUCT**
    Стиль выделения памяти плох.  В целом для семейства
    функций выделения памяти, использующих sizeof() для получения размера
    памяти, конструкции вида::

      p = alloc(sizeof(struct foo), ...)

    должны быть::

      p = alloc(sizeof(*p), ...)

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#allocating-memory

  **ALLOC_WITH_MULTIPLY**
    Предпочитайте kmalloc_array/kcalloc вместо kmalloc/kzalloc с
    умножением на sizeof.

    См.: https://www.kernel.org/doc/html/latest/core-api/memory-allocation.html


Использование API
-----------------

  **ARCH_DEFINES**
    Определений, специфичных для архитектуры, следует избегать везде, где
    это возможно.

  **ARCH_INCLUDE_LINUX**
    Всякий раз, когда включается asm/file.h и существует linux/file.h, может
    быть выполнено преобразование, когда linux/file.h включает asm/file.h.
    Однако это не всегда так (см. signal.h).
    Этот тип сообщения выдаётся только для включений из arch/.

  **AVOID_BUG**
    BUG() или BUG_ON() следует полностью избегать.
    Вместо этого используйте WARN() и WARN_ON() и обрабатывайте "невозможное"
    условие ошибки как можно более корректно.

    См.: https://www.kernel.org/doc/html/latest/process/deprecated.html#bug-and-bug-on

  **CONSIDER_KSTRTO**
    Функции simple_strtol(), simple_strtoll(), simple_strtoul() и
    simple_strtoull() явно игнорируют переполнения, что может привести к
    неожиданным результатам в вызывающем коде.  Соответствующие функции
    kstrtol(), kstrtoll(), kstrtoul() и kstrtoull() обычно являются
    правильной заменой.

    См.: https://www.kernel.org/doc/html/latest/process/deprecated.html#simple-strtol-simple-strtoll-simple-strtoul-simple-strtoull

  **CONSTANT_CONVERSION**
    Использование формы __constant_<foo> не рекомендуется для следующих функций::

      __constant_cpu_to_be[x]
      __constant_cpu_to_le[x]
      __constant_be[x]_to_cpu
      __constant_le[x]_to_cpu
      __constant_htons
      __constant_ntohs

    Использование любой из них вне include/uapi/ нежелательно, так как
    использование функции без __constant_ идентично, когда аргумент является
    константой.

    В системах с обратным порядком байтов (big endian) макросы вида
    __constant_cpu_to_be32(x) и cpu_to_be32(x) разворачиваются в одно и то же
    выражение::

      #define __constant_cpu_to_be32(x) ((__force __be32)(__u32)(x))
      #define __cpu_to_be32(x)          ((__force __be32)(__u32)(x))

    В системах с прямым порядком байтов (little endian) макросы
    __constant_cpu_to_be32(x) и cpu_to_be32(x) разворачиваются в
    __constant_swab32 и __swab32.  __swab32 имеет проверку
    __builtin_constant_p::

      #define __swab32(x)				\
        (__builtin_constant_p((__u32)(x)) ?	\
        ___constant_swab32(x) :			\
        __fswab32(x))

    Так что в конечном счёте они имеют особый случай для констант.
    Аналогична ситуация со всеми макросами в списке.  Таким образом,
    использование форм __constant_... неоправданно многословно и
    нежелательно вне include/uapi.

    См.: https://lore.kernel.org/lkml/1400106425.12666.6.camel@joe-AO725/

  **DEPRECATED_API**
    Обнаружено использование устаревшего RCU API.  Рекомендуется заменять
    старые "вкусовые" RCU API их новыми ванильными RCU-аналогами.

    Полный список доступных RCU API можно посмотреть в документации ядра.

    См.: https://www.kernel.org/doc/html/latest/RCU/whatisRCU.html#full-list-of-rcu-apis

  **DEVICE_ATTR_FUNCTIONS**
    Имена функций, используемые в DEVICE_ATTR, необычны.
    Обычно функции store и show используются как <attr>_store и
    <attr>_show, где <attr> — это именованная переменная атрибута устройства.

    Рассмотрите следующие примеры::

      static DEVICE_ATTR(type, 0444, type_show, NULL);
      static DEVICE_ATTR(power, 0644, power_show, power_store);

    Имена функций предпочтительно должны следовать приведённому выше шаблону.

    См.: https://www.kernel.org/doc/html/latest/driver-api/driver-model/device.html#attributes

  **DEVICE_ATTR_RO**
    Вспомогательный макрос DEVICE_ATTR_RO(name) может использоваться вместо
    DEVICE_ATTR(name, 0444, name_show, NULL);

    Обратите внимание, что макрос автоматически добавляет _show к именованной
    переменной атрибута устройства для метода show.

    См.: https://www.kernel.org/doc/html/latest/driver-api/driver-model/device.html#attributes

  **DEVICE_ATTR_RW**
    Вспомогательный макрос DEVICE_ATTR_RW(name) может использоваться вместо
    DEVICE_ATTR(name, 0644, name_show, name_store);

    Обратите внимание, что макрос автоматически добавляет _show и _store к
    именованной переменной атрибута устройства для методов show и store.

    См.: https://www.kernel.org/doc/html/latest/driver-api/driver-model/device.html#attributes

  **DEVICE_ATTR_WO**
    Вспомогательный макрос DEVICE_AATR_WO(name) может использоваться вместо
    DEVICE_ATTR(name, 0200, NULL, name_store);

    Обратите внимание, что макрос автоматически добавляет _store к
    именованной переменной атрибута устройства для метода store.

    См.: https://www.kernel.org/doc/html/latest/driver-api/driver-model/device.html#attributes

  **DUPLICATED_SYSCTL_CONST**
    Коммит d91bff3011cf ("proc/sysctl: add shared variables for range
    check") добавил несколько разделяемых const-переменных для использования
    вместо локальной копии в каждом файле исходного кода.

    Рассмотрите возможность замены значения проверки диапазона sysctl на
    разделяемое в include/linux/sysctl.h.  Может использоваться следующая
    схема преобразования::

      &zero     ->  SYSCTL_ZERO
      &one      ->  SYSCTL_ONE
      &int_max  ->  SYSCTL_INT_MAX

    См.:

      1. https://lore.kernel.org/lkml/20190430180111.10688-1-mcroce@redhat.com/
      2. https://lore.kernel.org/lkml/20190531131422.14970-1-mcroce@redhat.com/

  **ENOSYS**
    ENOSYS означает, что был вызван несуществующий системный вызов.
    Ранее он ошибочно использовался для таких вещей, как недопустимые операции
    над в остальном допустимыми syscall.  Этого следует избегать в новом коде.

    См.: https://lore.kernel.org/lkml/5eb299021dec23c1a48fa7d9f2c8b794e967766d.1408730669.git.luto@amacapital.net/

  **ENOTSUPP**
    ENOTSUPP не является стандартным кодом ошибки, и его следует избегать в
    новых патчах.
    Вместо него следует использовать EOPNOTSUPP.

    См.: https://lore.kernel.org/netdev/20200510182252.GA411829@lunn.ch/

  **EXPORT_SYMBOL**
    EXPORT_SYMBOL должен непосредственно следовать за экспортируемым символом.

  **IN_ATOMIC**
    in_atomic() не предназначен для использования в драйверах, поэтому любое
    такое использование сообщается как ERROR.
    Также in_atomic() часто используется для определения, разрешён ли сон,
    но в этой модели использования он ненадёжен.  Поэтому его использование
    настоятельно не рекомендуется.

    Однако in_atomic() допустим для использования в ядре core.

    См.: https://lore.kernel.org/lkml/20080320201723.b87b3732.akpm@linux-foundation.org/

  **LOCKDEP**
    Класс lockdep_no_validate был добавлен как временная мера для
    предотвращения предупреждений при преобразовании device->sem в
    device->mutex.
    Его не следует использовать для каких-либо других целей.

    См.: https://lore.kernel.org/lkml/1268959062.9440.467.camel@laptop/

  **MALFORMED_INCLUDE**
    Оператор #include имеет некорректный путь.  Это произошло
    потому, что автор случайно включил двойной слеш "//" в имя пути.

  **USE_LOCKDEP**
    Аннотации lockdep_assert_held() следует предпочитать утверждениям,
    основанным на spin_is_locked()

    См.: https://www.kernel.org/doc/html/latest/locking/lockdep-design.html#annotations

  **UAPI_INCLUDE**
    Никакие операторы #include в include/uapi не должны использовать путь uapi/.

  **USLEEP_RANGE**
    usleep_range() следует предпочитать вместо udelay(). Правильный способ
    использования usleep_range() описан в документации ядра.


Комментарии
-----------

  **BLOCK_COMMENT_STYLE**
    Стиль комментариев некорректен.  Предпочтительный стиль для многострочных
    комментариев::

      /*
       * This is the preferred style
       * for multi line comments.
       */

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#commenting

  **C99_COMMENTS**
    Однострочные комментарии в стиле C99 (//) не следует использовать.
    Вместо этого предпочитайте стиль блочных комментариев.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#commenting

  **DATA_RACE**
    Применения data_race() должны иметь комментарий, документирующий
    обоснование того, почему это было сочтено безопасным.

    См.: https://lore.kernel.org/lkml/20200401101714.44781-1-elver@google.com/

  **FSF_MAILING_ADDRESS**
    Сопровождающие ядра отклоняют новые экземпляры шаблонного абзаца GPL,
    указывающего людям писать в FSF за копией GPL, так как
    FSF переезжал в прошлом и может сделать это снова.
    Поэтому не пишите абзацы о письме на почтовый адрес Free Software
    Foundation.

    См.: https://lore.kernel.org/lkml/20131006222342.GT19510@leaf/

  **UNCOMMENTED_RGMII_MODE**
    Исторически режимы RGMII PHY, указанные в Device Trees, использовались
    непоследовательно, часто ссылаясь на использование задержек на стороне PHY,
    а не описывая плату.

    Режимы PHY "rgmii", "rgmii-rxid" и "rgmii-txid" требуют, чтобы сигнал
    тактирования был задержан на печатной плате; эту необычную конфигурацию
    следует описать в комментарии. Если они не описаны (что означает, что
    задержка реализована внутри MAC или PHY), правильным режимом PHY является
    "rgmii-id".

Сообщение коммита
-----------------

  **BAD_SIGN_OFF**
    Строка signed-off-by не соответствует стандартам, указанным сообществом.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#developer-s-certificate-of-origin-1-1

  **BAD_STABLE_ADDRESS_STYLE**
    Формат email для stable некорректен.
    Некоторые допустимые варианты адреса stable::

      1. stable@vger.kernel.org
      2. stable@kernel.org

    Для добавления информации о версии следует использовать следующий стиль
    комментария::

      stable@vger.kernel.org # version info

  **COMMIT_COMMENT_SYMBOL**
    Строки журнала коммита, начинающиеся с '#', игнорируются git как
    комментарии.  Чтобы решить эту проблему, достаточно добавления одного
    пробела перед строкой журнала.

  **COMMIT_MESSAGE**
    В патче отсутствует описание коммита.  Следует добавить краткое описание
    изменений, внесённых патчем.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#describe-your-changes

  **EMAIL_SUBJECT**
    Указание инструмента, который нашёл проблему, не очень полезно в строке
    темы.  Хорошая строка темы кратко описывает изменение, которое привносит
    патч.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#describe-your-changes

  **FROM_SIGN_OFF_MISMATCH**
    Email автора не совпадает с тем, что в строке(ах) Signed-off-by:.
    Иногда это может быть вызвано неправильно сконфигурированным
    почтовым клиентом.

    Это сообщение выдаётся по любой из следующих причин::

      - Имена в email не совпадают.
      - Адреса email не совпадают.
      - Субадреса email не совпадают.
      - Комментарии email не совпадают.

  **MISSING_SIGN_OFF**
    В патче отсутствует строка Signed-off-by.  Строку signed-off-by
    следует добавлять в соответствии с Developer's certificate of
    Origin.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#sign-your-work-the-developer-s-certificate-of-origin

  **NO_AUTHOR_SIGN_OFF**
    Автор патча не подписал патч (sign off).  Требуется, чтобы в конце
    пояснения к патчу присутствовала простая строка sign off, обозначающая,
    что автор написал его или иным образом имеет права передавать его как
    патч с открытым исходным кодом.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#sign-your-work-the-developer-s-certificate-of-origin

  **DIFF_IN_COMMIT_MSG**
    Избегайте наличия содержимого diff в сообщении коммита.
    Это вызывает проблемы, когда кто-то пытается применить файл, содержащий
    как changelog, так и diff, потому что patch(1) пытается применить diff,
    который он нашёл в changelog.

    См.: https://lore.kernel.org/lkml/20150611134006.9df79a893e3636019ad2759e@linux-foundation.org/

  **GERRIT_CHANGE_ID**
    Чтобы быть подхваченным gerrit, нижний колонтитул сообщения коммита может
    иметь Change-Id вида::

      Change-Id: Ic8aaa0728a43936cd4c6e1ed590e01ba8f0fbf5b
      Signed-off-by: A. U. Thor <author@example.com>

    Строку Change-Id необходимо удалить перед отправкой.

  **GIT_COMMIT_ID**
    Правильный способ ссылаться на идентификатор коммита таков:
    commit <12+ chars of sha1> ("<title line>")

    Примером может быть::

      Commit e21d2170f36602ae2708 ("video: remove unnecessary
      platform_set_drvdata()") removed the unnecessary
      platform_set_drvdata(), but left the variable "dev" unused,
      delete it.

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#describe-your-changes

  **BAD_FIXES_TAG**
    Тег Fixes: некорректен или не следует соглашениям сообщества.
    Это может произойти, если тег был разбит на несколько строк (например,
    когда вставлен в почтовую программу с включённым переносом слов).

    См.: https://www.kernel.org/doc/html/latest/process/submitting-patches.html#describe-your-changes

  **BAD_COMMIT_SEPARATOR**
    Разделитель коммита — это одна строка с 3 дефисами.
    Соответствие регулярному выражению '^---$'
    Строки, которые начинаются с 3 дефисов и имеют больше содержимого на той же
    строке, могут запутать инструменты, применяющие патчи.

Стиль сравнений
---------------

  **ASSIGN_IN_IF**
    Не используйте присваивания в условии if.
    Пример::

      if ((foo = bar(...)) < BAZ) {

    следует писать как::

      foo = bar(...);
      if (foo < BAZ) {

  **BOOL_COMPARISON**
    Сравнения A с true и false лучше писать
    как A и !A.

    См.: https://lore.kernel.org/lkml/1365563834.27174.12.camel@joe-AO722/

  **COMPARISON_TO_NULL**
    Сравнения с NULL в форме (foo == NULL) или (foo != NULL)
    лучше писать как (!foo) и (foo).

  **CONSTANT_COMPARISON**
    Сравнений с константой или идентификатором в верхнем регистре с левой
    стороны проверки следует избегать.


Отступы и переносы строк
------------------------

  **CODE_INDENT**
    Для отступов кода следует использовать табуляции вместо пробелов.
    Вне комментариев, документации и Kconfig
    пробелы никогда не используются для отступов.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#indentation

  **DEEP_INDENTATION**
    Отступ из 6 или более табуляций обычно указывает на чрезмерно отступленный
    код.

    Рекомендуется рефакторить чрезмерные отступы операторов
    if/else/for/do/while/switch.

    См.: https://lore.kernel.org/lkml/1328311239.21255.24.camel@joe2Laptop/

  **SWITCH_CASE_INDENT_LEVEL**
    switch должен быть на том же отступе, что и case.
    Пример::

      switch (suffix) {
      case 'G':
      case 'g':
              mem <<= 30;
              break;
      case 'M':
      case 'm':
              mem <<= 20;
              break;
      case 'K':
      case 'k':
              mem <<= 10;
              fallthrough;
      default:
              break;
      }

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#indentation

  **LONG_LINE**
    Строка превысила указанную максимальную длину.
    Чтобы использовать другую максимальную длину строки, при вызове checkpatch
    может быть добавлена опция --max-line-length=n.

    Ранее длина строки по умолчанию составляла 80 столбцов.  Коммит bdc48fa11e46
    ("checkpatch/coding-style: deprecate 80-column warning") увеличил
    лимит до 100 столбцов.  Это тоже не жёсткий лимит, и
    предпочтительно оставаться в пределах 80 столбцов всякий раз, когда это
    возможно.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#breaking-long-lines-and-strings

  **LONG_LINE_STRING**
    Строка начинается раньше, но выходит за пределы максимальной длины строки.
    Чтобы использовать другую максимальную длину строки, при вызове checkpatch
    может быть добавлена опция --max-line-length=n.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#breaking-long-lines-and-strings

  **LONG_LINE_COMMENT**
    Комментарий начинается раньше, но выходит за пределы максимальной длины
    строки.
    Чтобы использовать другую максимальную длину строки, при вызове checkpatch
    может быть добавлена опция --max-line-length=n.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#breaking-long-lines-and-strings

  **SPLIT_STRING**
    Строки в кавычках, которые появляются как сообщения в пространстве
    пользователя (userspace) и могут быть найдены через grep, не следует
    разбивать на несколько строк.

    См.: https://lore.kernel.org/lkml/20120203052727.GA15035@leaf/

  **MULTILINE_DEREFERENCE**
    Одиночный разыменовывающий идентификатор, растянутый на несколько строк,
    как::

      struct_identifier->member[index].
      member = <foo>;

    в целом трудно отслеживать. Это легко может привести к опечаткам и тем
    самым делает код уязвимым для ошибок.

    Если исправление многострочного разыменования приводит к нарушению
    лимита в 80 столбцов, то либо перепишите код более простым способом, либо,
    если начальная часть разыменовывающего идентификатора одинакова и
    используется в нескольких местах, сохраните её во временной переменной и
    используйте эту временную переменную во всех местах. Например, если есть
    два разыменовывающих идентификатора::

      member1->member2->member3.foo1;
      member1->member2->member3.foo2;

    то сохраните часть member1->member2->member3 во временной переменной.
    Это не только помогает избежать нарушения лимита в 80 столбцов, но и
    уменьшает размер программы за счёт удаления ненужных разыменований.

    Но если ни один из вышеперечисленных методов не работает, то проигнорируйте
    нарушение лимита в 80 столбцов, потому что разыменовывающий идентификатор
    гораздо легче читать на одной строке.

  **TRAILING_STATEMENTS**
    Завершающие операторы (например, после любого условного оператора) должны
    быть на следующей строке.
    Операторы, такие как::

      if (x == y) break;

    должны быть::

      if (x == y)
              break;


Макросы, атрибуты и символы
---------------------------

  **ARRAY_SIZE**
    Макрос ARRAY_SIZE(foo) следует предпочитать вместо
    sizeof(foo)/sizeof(foo[0]) для нахождения количества элементов в
    массиве.

    Макрос определён в include/linux/array_size.h::

      #define ARRAY_SIZE(x) (sizeof(x) / sizeof((x)[0]))

  **AVOID_EXTERNS**
    Прототипы функций не нужно объявлять как extern в .h-файлах.
    Это подразумевается компилятором и является излишним.

  **AVOID_L_PREFIX**
    Имён локальных символов с префиксом `.L` следует избегать,
    так как это имеет особое значение для ассемблера; запись символа не
    будет внесена в таблицу символов.  Это может помешать `objtool`
    генерировать корректную информацию о раскрутке стека.

    Символы со связыванием STB_LOCAL по-прежнему могут использоваться, и
    имена локальных символов с префиксом `.L` по-прежнему в целом пригодны
    для использования внутри функции, но имена локальных символов с префиксом
    `.L` не следует использовать для обозначения начала или конца областей
    кода через `SYM_CODE_START_LOCAL`/`SYM_CODE_END`

  **BIT_MACRO**
    Определения вида: 1 << <digit> могли бы быть BIT(digit).
    Макрос BIT() определён через include/linux/bits.h::

      #define BIT(nr)         (1UL << (nr))

  **CONST_READ_MOSTLY**
    Когда переменная помечена аннотацией __read_mostly, это
    сигнал компилятору о том, что обращения к переменной будут в основном
    чтениями и редко (но НЕ никогда) записью.

    const __read_mostly не имеет никакого смысла, так как const-данные уже
    доступны только для чтения.  Поэтому аннотацию __read_mostly следует
    удалить.

  **DATE_TIME**
    В целом желательно, чтобы сборка одного и того же исходного кода с
    одним и тем же набором инструментов была воспроизводимой, то есть вывод
    всегда был в точности одинаковым.

    Ядро *не* использует макросы ``__DATE__`` и ``__TIME__``
    и включает предупреждения, если они используются, так как они могут
    привести к недетерминированным сборкам.

    См.: https://www.kernel.org/doc/html/latest/kbuild/reproducible-builds.html#timestamps

  **DEFINE_ARCH_HAS**
    Шаблоны ARCH_HAS_xyz и ARCH_HAVE_xyz неверны.

    Для крупных концептуальных функций вместо этого используйте символы
    Kconfig.  А для более мелких вещей, где у нас есть запасные функции для
    совместимости, но мы хотим, чтобы архитектуры могли переопределять их
    оптимизированными, нам следует либо использовать слабые функции
    (уместно в некоторых случаях), либо символ, который их защищает, должен
    быть тем же символом, который мы используем.

    См.: https://lore.kernel.org/lkml/CA+55aFycQ9XJvEOsiM3txHL5bjUc8CeKWJNR_H+MiicaddB42Q@mail.gmail.com/

  **DO_WHILE_MACRO_WITH_TRAILING_SEMICOLON**
    Макросы do {} while(0) не должны иметь завершающую точку с запятой.

  **INIT_ATTRIBUTE**
    Const-определения init должны использовать __initconst вместо
    __initdata.

    Аналогично init-определения без const требуют отдельного
    использования const.

  **INLINE_LOCATION**
    Ключевое слово inline должно располагаться между классом хранения и типом.

    Например, следующий фрагмент::

      inline static int example_function(void)
      {
              ...
      }

    должен быть::

      static inline int example_function(void)
      {
              ...
      }

  **MISPLACED_INIT**
    Можно использовать маркеры секций для переменных таким образом,
    который gcc не понимает (или, по крайней мере, не так, как
    задумывал разработчик)::

      static struct __initdata samsung_pll_clock exynos4_plls[nr_plls] = {

    не помещает exynos4_plls в секцию .initdata. Маркер __initdata
    может находиться практически где угодно на строке, кроме как сразу после
    "struct". Предпочтительное расположение — перед знаком "=", если он есть,
    или перед завершающей ";" в противном случае.

    См.: https://lore.kernel.org/lkml/1377655732.3619.19.camel@joe-AO722/

  **MULTISTATEMENT_MACRO_USE_DO_WHILE**
    Макросы с несколькими операторами должны быть заключены в
    блок do - while.  То же должно быть и для макросов,
    начинающихся с `if`, чтобы избежать логических дефектов::

      #define macrofun(a, b, c)                 \
        do {                                    \
                if (a == 5)                     \
                        do_this(b, c);          \
        } while (0)

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#macros-enums-and-rtl

  **PREFER_FALLTHROUGH**
    Используйте псевдо-ключевое слово `fallthrough;` вместо
    комментариев вида `/* fallthrough */`.

  **TRAILING_SEMICOLON**
    Определение макроса не должно заканчиваться точкой с запятой. Стиль
    вызова макроса должен быть согласован с вызовами функций.
    Это может предотвратить любые неожиданные пути выполнения кода::

      #define MAC do_something;

    Если этот макрос используется внутри оператора if else, как::

      if (some_condition)
              MAC;

      else
              do_something;

    Тогда возникнет ошибка компиляции, потому что при разворачивании макроса
    появляются две завершающие точки с запятой, так что ветка else
    осиротеет.

    См.: https://lore.kernel.org/lkml/1399671106.2912.21.camel@joe-AO725/

  **MACRO_ARG_UNUSED**
    Если функциеподобные макросы не используют параметр, это может привести
    к предупреждению при сборке. Мы выступаем за использование статических
    встраиваемых (static inline) функций для замены таких макросов.
    Например, для макроса вроде приведённого ниже::

      #define test(a) do { } while (0)

    возникло бы предупреждение вроде приведённого ниже::

      WARNING: Argument 'a' is not used in function-like macro.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#macros-enums-and-rtl

  **SINGLE_STATEMENT_DO_WHILE_MACRO**
    Для многооператорных макросов необходимо использовать цикл do-while,
    чтобы избежать непредсказуемых путей выполнения кода. Цикл do-while
    помогает сгруппировать несколько операторов в один, чтобы
    функциеподобный макрос можно было использовать только как функцию.

    Но для одинооператорных макросов использовать цикл do-while
    нет необходимости. Хотя код синтаксически корректен, использование
    цикла do-while избыточно. Поэтому удалите цикл do-while для
    одинооператорных макросов.

  **WEAK_DECLARATION**
    Использование слабых объявлений вроде __attribute__((weak)) или __weak
    может иметь непреднамеренные дефекты компоновки.  Избегайте их
    использования.


Функции и переменные
--------------------

  **CAMELCASE**
    Избегайте идентификаторов в стиле CamelCase.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#naming

  **CONST_CONST**
    Использование `const <type> const *` обычно подразумевается записать как
    `const <type> * const`.

  **CONST_STRUCT**
    Использование const, как правило, хорошая идея.  Checkpatch читает
    список часто используемых структур, которые всегда или
    почти всегда являются константными.

    Существующий список структур можно посмотреть в
    `scripts/const_structs.checkpatch`.

    См.: https://lore.kernel.org/lkml/alpine.DEB.2.10.1608281509480.3321@hadrien/

  **EMBEDDED_FUNCTION_NAME**
    Встроенные имена функций менее уместно использовать, так как
    рефакторинг может вызвать переименование функции.  Предпочитайте
    использование "%s", __func__ вместо встроенных имён функций.

    Обратите внимание, что это не работает с опцией checkpatch -f (--file),
    так как зависит от контекста патча, предоставляющего имя функции.

  **FUNCTION_ARGUMENTS**
    Это предупреждение выдаётся по любой из следующих причин:

      1. Аргументы для объявления функции не следуют за
         именем идентификатора.  Пример::

           void foo
           (int bar, int baz)

         Это следует исправить на::

           void foo(int bar, int baz)

      2. Некоторые аргументы для определения функции не
         имеют имени идентификатора.  Пример::

           void foo(int)

         Все аргументы должны иметь имена идентификаторов.

  **FUNCTION_WITHOUT_ARGS**
    Объявления функций без аргументов вроде::

      int foo()

    должны быть::

      int foo(void)

  **GLOBAL_INITIALISERS**
    Глобальные переменные не следует инициализировать явно
    0 (или NULL, false и т.д.).  Ваш компилятор (или, скорее, ваш
    загрузчик, который отвечает за обнуление соответствующих
    секций) автоматически делает это за вас.

  **INITIALISED_STATIC**
    Статические переменные не следует инициализировать явно нулём.
    Ваш компилятор (или, скорее, ваш загрузчик) автоматически делает
    это за вас.

  **MULTIPLE_ASSIGNMENTS**
    Несколько присваиваний на одной строке делают код неоправданно
    усложнённым. Поэтому на одной строке присваивайте значение только
    одной переменной, это делает код более читаемым и помогает избежать
    опечаток.

  **RETURN_PARENTHESES**
    return не является функцией и как таковой не нуждается в скобках::

      return (bar);

    может быть просто::

      return bar;

  **UNINITIALIZED_PTR_WITH_FREE**
    Указатели с атрибутом __free должны объявляться в месте использования
    и инициализироваться (см. include/linux/cleanup.h). В этом случае
    правило об объявлениях в начале функции может быть смягчено. Если этого
    не сделать, это может привести к неопределённому поведению, так как
    память, присвоенная указателю (мусор, в случае если она не
    инициализирована), освобождается автоматически, когда
    указатель выходит из области видимости.

    См. также: https://lore.kernel.org/lkml/58fd478f408a34b578ee8d949c5c4b4da4d4f41d.camel@HansenPartnership.com/

    Пример::

      type var __free(free_func);
      ... // var not used, but, in future someone might add a return here
      var = malloc(var_size);
      ...

    следует инициализировать как::

      ...
      type var __free(free_func) = malloc(var_size);
      ...


Права доступа
-------------

  **DEVICE_ATTR_PERMS**
    Права доступа, используемые в DEVICE_ATTR, необычны.
    Обычно используются только три права доступа — 0644 (RW), 0444 (RO)
    и 0200 (WO).

    См.: https://www.kernel.org/doc/html/latest/filesystems/sysfs.html#attributes

  **EXECUTE_PERMISSIONS**
    Нет причин для того, чтобы файлы исходного кода были исполняемыми.  Бит
    исполнения можно безопасно удалить.

  **EXPORTED_WORLD_WRITABLE**
    Экспортирование файлов sysfs/debugfs, доступных для записи всем, обычно
    плохая вещь.
    Когда это делается произвольно, они могут привнести серьёзные уязвимости
    безопасности.
    В прошлом некоторые из уязвимостей debugfs, по-видимому, позволяли
    любому локальному пользователю записывать произвольные значения в регистры
    устройства — ситуация, из которой едва ли можно ожидать чего-то хорошего.

    См.: https://lore.kernel.org/linux-arm-kernel/cover.1296818921.git.segoon@openwall.com/

  **NON_OCTAL_PERMISSIONS**
    Биты прав доступа должны использовать 4-значные восьмеричные права
    (вроде 0700 или 0444).
    Избегайте использования любой другой системы счисления, например десятичной.

  **SYMBOLIC_PERMS**
    Биты прав доступа в восьмеричной форме более читаемы и понятны,
    чем их символьные аналоги, потому что многие инструменты командной
    строки используют эту нотацию. Опытные разработчики ядра использовали
    эти традиционные Unix-биты прав доступа десятилетиями, поэтому им
    легче понимать восьмеричную нотацию, чем символьные макросы.
    Например, труднее прочитать S_IWUSR|S_IRUGO, чем 0644, что
    скорее затемняет намерение разработчика, чем проясняет его.

    См.: https://lore.kernel.org/lkml/CA+55aFw5v23T-zvDZp-MmD_EYxF8WbafwwB59934FV7g21uMGQ@mail.gmail.com/


Пробелы и скобки
----------------

  **ASSIGNMENT_CONTINUATIONS**
    Операторы присваивания не следует писать в начале
    строки, они должны следовать за операндом на предыдущей строке.

  **BRACES**
    Расположение фигурных скобок стилистически некорректно.
    Предпочтительный способ — поставить открывающую фигурную скобку последней
    на строке и поставить закрывающую фигурную скобку первой::

      if (x is true) {
              we do y
      }

    Это применимо ко всем нефункциональным блокам.
    Однако есть один особый случай, а именно функции: у них
    открывающая фигурная скобка находится в начале следующей строки, таким образом::

      int function(int x)
      {
              body of function
      }

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#placing-braces-and-spaces

  **BRACKET_SPACE**
    Пробел перед открывающей скобкой '[' запрещён.
    Есть некоторые исключения:

    1. С типом слева::

        int [] a;

    2. В начале строки для инициализаторов срезов::

        [0...10] = 5,

    3. Внутри фигурной скобки::

        = { [0...10] = 5 }

  **CONCATENATED_STRING**
    Конкатенируемые элементы должны иметь пробел между собой.
    Пример::

      printk(KERN_INFO"bar");

    следует::

      printk(KERN_INFO "bar");

  **ELSE_AFTER_BRACE**
    `else {` должно следовать за закрывающей скобкой блока `}` на той же строке.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#placing-braces-and-spaces

  **LINE_SPACING**
    Вертикальное пространство тратится впустую, учитывая ограниченное
    количество строк, которое может отобразить окно редактора, когда
    используется несколько пустых строк.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#spaces

  **OPEN_BRACE**
    Открывающая фигурная скобка должна следовать за определениями функций на
    следующей строке.  Для любого нефункционального блока она должна быть на той же строке,
    что и последняя конструкция.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#placing-braces-and-spaces

  **POINTER_LOCATION**
    При использовании данных-указателей или функции, возвращающей тип-указатель,
    предпочтительное использование * — рядом с именем данных или именем функции,
    а не рядом с именем типа.
    Примеры::

      char *linux_banner;
      unsigned long long memparse(char *ptr, char **retptr);
      char *match_strdup(substring_t *s);

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#spaces

  **SPACING**
    Стиль пробелов, используемый в исходниках ядра, описан в документации ядра.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#spaces

  **TRAILING_WHITESPACE**
    Завершающие пробелы всегда следует удалять.
    Некоторые редакторы подсвечивают завершающие пробелы и вызывают визуальные
    отвлечения при редактировании файлов.

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#spaces

  **UNNECESSARY_PARENTHESES**
    Скобки не требуются в следующих случаях:

      1. Использование указателей на функции::

          (foo->bar)();

        могло бы быть::

          foo->bar();

      2. Сравнения в if::

          if ((foo->bar) && (foo->baz))
          if ((foo == bar))

        могло бы быть::

          if (foo->bar && foo->baz)
          if (foo == bar)

      3. Взятие адреса/разыменование одиночных L-значений::

          &(foo->bar)
          *(foo->bar)

        могло бы быть::

          &foo->bar
          *foo->bar

  **WHILE_AFTER_BRACE**
    while должно следовать за закрывающей скобкой на той же строке::

      do {
              ...
      } while(something);

    См.: https://www.kernel.org/doc/html/latest/process/coding-style.html#placing-braces-and-spaces


Прочее
------

  **CONFIG_DESCRIPTION**
    Символы Kconfig должны иметь текст справки, который полностью описывает
    их.

  **CORRUPTED_PATCH**
    Патч, по-видимому, повреждён или строки перенесены.
    Пожалуйста, перегенерируйте файл патча перед отправкой его сопровождающему.

  **CVS_KEYWORD**
    Поскольку linux перешёл на git, маркеры CVS больше не используются.
    Поэтому ключевые слова в стиле CVS ($Id$, $Revision$, $Log$) не следует
    добавлять.

  **DEFAULT_NO_BREAK**
    Случай default в switch иногда пишется как "default:;".  Это может
    привести к дефектности новых случаев, добавленных ниже default.

    После пустого оператора default следует добавить "break;", чтобы избежать
    нежелательного fallthrough.

  **DOS_LINE_ENDINGS**
    Для патчей в формате DOS в конце строки есть дополнительные символы ^M.
    Их следует удалить.

  **DT_SCHEMA_BINDING_PATCH**
    Привязки DT (DT bindings) перешли на формат на основе json-schema вместо
    свободного текста.

    См.: https://www.kernel.org/doc/html/latest/devicetree/bindings/writing-schema.html

  **DT_SPLIT_BINDING_PATCH**
    Привязки Devicetree должны быть отдельным патчем.  Это потому, что
    привязки логически независимы от реализации драйвера,
    у них другой сопровождающий (даже если они часто
    применяются через то же дерево), и это даёт более чистую историю в
    дереве только для DT, создаваемом с помощью git-filter-branch.

    См.: https://www.kernel.org/doc/html/latest/devicetree/bindings/submitting-patches.html#i-for-patch-submitters

  **EMBEDDED_FILENAME**
    Встраивание полного пути имени файла внутрь файла не особенно
    полезно, так как часто путь перемещается и становится некорректным.

  **FILE_PATH_CHANGES**
    Всякий раз, когда файлы добавляются, перемещаются или удаляются, шаблоны
    в файле MAINTAINERS могут рассинхронизироваться или устареть.

    Поэтому в этих случаях MAINTAINERS может потребовать обновления.

  **MEMSET**
    Использование memset, по-видимому, некорректно.  Это может быть вызвано
    неправильно упорядоченными параметрами.  Пожалуйста, перепроверьте
    использование.

  **NOT_UNIFIED_DIFF**
    Файл патча, по-видимому, не находится в формате unified-diff.  Пожалуйста,
    перегенерируйте файл патча перед отправкой его сопровождающему.

  **PLACEHOLDER_USE**
    Обнаруживает необработанный текст-заполнитель, оставленный в сопроводительных письмах или заголовках/журналах коммитов.
    Распространённые заполнители включают строки вроде::

      *** SUBJECT HERE ***
      *** BLURB HERE ***

    Обычно они происходят из автоматически сгенерированных шаблонов. Замените их подходящими
    темой и описанием перед отправкой.

  **PRINTF_0XDECIMAL**
    Префикс 0x с десятичным выводом дефектен и должен быть исправлен.

  **SPDX_LICENSE_TAG**
    В файле исходного кода отсутствует или имеет неправильную форму тег
    идентификатора SPDX.
    Ядро Linux требует точного идентификатора SPDX во всех файлах исходного кода,
    и это подробно задокументировано в документации ядра.

    См.: https://www.kernel.org/doc/html/latest/process/license-rules.html

  **TYPO_SPELLING**
    Некоторые слова могли быть написаны с ошибкой.  Рассмотрите возможность их
    проверки.
