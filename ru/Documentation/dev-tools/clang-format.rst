.. _clangformat:

clang-format
============

``clang-format`` — это инструмент для форматирования кода на C/C++/... в
соответствии с набором правил и эвристик. Как и большинство инструментов, он
не идеален и не покрывает все возможные случаи, но достаточно хорош, чтобы
быть полезным.

``clang-format`` может использоваться для нескольких целей:

  - Быстро переформатировать блок кода в соответствии со стилем ядра. Особенно
    полезно при перемещении кода, а также при выравнивании/сортировке.
    См. clangformatreformat_.

  - Выявлять стилистические ошибки, опечатки и возможные улучшения в файлах,
    которые вы сопровождаете, в патчах, которые вы рецензируете, в diff'ах
    и т.п. См. clangformatreview_.

  - Помогать вам следовать правилам стиля написания кода — особенно полезно для
    тех, кто только начинает заниматься разработкой ядра или одновременно
    работает над несколькими проектами с разными стилями кода.

Файл его конфигурации — ``.clang-format`` в корне дерева ядра. Содержащиеся
там правила пытаются приблизиться к наиболее распространённому стилю
написания кода ядра. Они также стараются как можно точнее следовать
:ref:`Documentation/process/coding-style.rst <codingstyle>`. Поскольку не всё
ядро придерживается единого стиля, возможно, вы захотите подправить значения
по умолчанию для конкретной подсистемы или каталога. Для этого вы можете
переопределить значения по умолчанию, написав ещё один файл ``.clang-format``
в подкаталоге.

Сам инструмент уже давно включён в репозитории популярных дистрибутивов
Linux. Поищите ``clang-format`` в своих репозиториях. В противном случае вы
можете либо скачать готовые двоичные файлы LLVM/clang, либо собрать исходный
код, взяв его здесь:

    https://releases.llvm.org/download.html

Дополнительную информацию об инструменте смотрите здесь:

    https://clang.llvm.org/docs/ClangFormat.html

    https://clang.llvm.org/docs/ClangFormatStyleOptions.html


.. _clangformatreview:

Рецензирование файлов и патчей на предмет стиля написания кода
--------------------------------------------------------------

Запустив инструмент в режиме inline, вы можете проверить целые подсистемы,
каталоги или отдельные файлы на наличие стилистических ошибок, опечаток или
возможностей для улучшения.

Для этого вы можете выполнить что-то вроде::

    # Make sure your working directory is clean!
    clang-format -i kernel/*.[ch]

А затем посмотреть на git diff.

Подсчёт строк в таком diff также полезен для улучшения/настройки опций стиля
в файле конфигурации, а также для тестирования новых
возможностей/версий ``clang-format``.

``clang-format`` также поддерживает чтение унифицированных diff'ов, поэтому вы
можете легко рецензировать патчи и git diff'ы. Смотрите документацию здесь:

    https://clang.llvm.org/docs/ClangFormat.html#script-for-patch-reformatting

Чтобы ``clang-format`` не форматировал какую-то часть файла, вы можете
сделать так::

    int formatted_code;
    // clang-format off
        void    unformatted_code  ;
    // clang-format on
    void formatted_code_again;

Хотя может возникнуть соблазн использовать это, чтобы всегда держать файл
синхронизированным с ``clang-format`` — особенно если вы пишете новые файлы
или являетесь сопровождающим — обратите внимание, что у людей могут быть
запущены разные версии ``clang-format`` или его может вообще не быть. Поэтому
вам, вероятно, стоит воздержаться от использования этого в исходниках ядра —
по крайней мере до тех пор, пока мы не увидим, станет ли ``clang-format``
общепринятым.


.. _clangformatreformat:

Переформатирование блоков кода
------------------------------

Используя интеграцию с вашим текстовым редактором, вы можете переформатировать
произвольные блоки (выделения) кода одним нажатием клавиши. Это особенно
полезно при перемещении кода, для сложного кода с глубокими отступами, для
многострочных макросов (и выравнивания их обратных слешей) и т.п.

Помните, что в тех случаях, когда инструмент сработал не оптимально, вы всегда
можете подправить изменения после. Но в качестве первого приближения это
может быть очень полезно.

Существуют интеграции для многих популярных текстовых редакторов. Для
некоторых из них, таких как vim, emacs, BBEdit и Visual Studio, вы найдёте
встроенную поддержку. Инструкции читайте в соответствующем разделе здесь:

    https://clang.llvm.org/docs/ClangFormat.html

Для Atom, Eclipse, Sublime Text, Visual Studio Code, XCode и других редакторов
и IDE вы должны без труда найти готовые к использованию плагины.

Для этого сценария использования рассмотрите возможность применения
вторичного ``.clang-format``, чтобы вы могли подправить несколько опций.
См. clangformatextra_.


.. _clangformatmissing:

Отсутствующая поддержка
-----------------------

В ``clang-format`` отсутствует поддержка некоторых вещей, которые часто
встречаются в коде ядра. Их легко запомнить, поэтому если вы используете
инструмент регулярно, вы быстро научитесь избегать/игнорировать их.

В частности, вот некоторые очень распространённые из них, которые вы заметите:

  - Выровненные блоки однострочных ``#defines``, например::

        #define TRACING_MAP_BITS_DEFAULT       11
        #define TRACING_MAP_BITS_MAX           17
        #define TRACING_MAP_BITS_MIN           7

    против::

        #define TRACING_MAP_BITS_DEFAULT 11
        #define TRACING_MAP_BITS_MAX 17
        #define TRACING_MAP_BITS_MIN 7

  - Выровненные именованные инициализаторы (designated initializers),
    например::

        static const struct file_operations uprobe_events_ops = {
                .owner          = THIS_MODULE,
                .open           = probes_open,
                .read           = seq_read,
                .llseek         = seq_lseek,
                .release        = seq_release,
                .write          = probes_write,
        };

    против::

        static const struct file_operations uprobe_events_ops = {
                .owner = THIS_MODULE,
                .open = probes_open,
                .read = seq_read,
                .llseek = seq_lseek,
                .release = seq_release,
                .write = probes_write,
        };


.. _clangformatextra:

Дополнительные возможности/опции
--------------------------------

Некоторые возможности/опции стиля не включены по умолчанию в файле
конфигурации, чтобы минимизировать различия между выводом и текущим кодом.
Иными словами, чтобы сделать различие как можно меньше, что делает
рецензирование стиля целого файла, а также diff'ов и патчей максимально
простым.

В других случаях (например, для определённых подсистем/каталогов/файлов)
стиль ядра может отличаться, и включение некоторых из этих опций может лучше
приблизиться к стилю там.

Например:

  - Выравнивание присваиваний (``AlignConsecutiveAssignments``).

  - Выравнивание объявлений (``AlignConsecutiveDeclarations``).

  - Перекомпоновка текста в комментариях (``ReflowComments``).

  - Сортировка ``#includes`` (``SortIncludes``).

Они обычно полезны для переформатирования блоков, а не целого файла. Возможно,
вы захотите создать ещё один файл ``.clang-format`` и использовать его из
вашего редактора/IDE.
