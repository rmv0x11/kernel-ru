==================================================================
Неточный логический анализатор для Linux на основе GPIO
==================================================================

:Author: Wolfram Sang

Введение
========

В этом документе кратко описывается, как запустить встроенный в ядро неточный
логический анализатор на основе GPIO, работающий на изолированном CPU.

Неточный логический анализатор задействует несколько линий GPIO в режиме входа
на системе, чтобы быстро опрашивать (sample) эти цифровые линии, что, при
выполнении критерия Найквиста, даст временной ряд с приблизительными формами
сигналов в том виде, в каком они присутствовали на этих линиях. Один из
способов его применения — анализ внешнего трафика, подключённого к этим линиям
GPIO проводами (т.е. цифровыми щупами), при котором анализатор выступает в роли
обычного логического анализатора.

Ещё одна возможность — подслушивание (snoop) встроенной в чип периферии, если
ячейки ввода-вывода этой периферии можно использовать в режиме входа GPIO
одновременно с тем, как они используются в качестве входов или выходов для самой
периферии. Это означает, что можно, например, подслушивать трафик I2C вообще
без проводов (если ваше оборудование это поддерживает). В подсистеме управления
выводами (pin control) такие контроллеры выводов называются «нестрогими»
(non-strict): определённый вывод может одновременно использоваться с
определённой периферией и в качестве входной линии GPIO.

Обратите внимание, что это анализатор для крайних случаев, на работу которого
могут влиять задержки (latency), недетерминированные пути выполнения кода и
немаскируемые прерывания. Он называется «неточным» (sloppy) не просто так. Тем
не менее, например при удалённой разработке, он может быть полезен для получения
первого представления и помощи в дальнейшей отладке.

Настройка
=========

В вашем ядре должны быть включены CONFIG_DEBUG_FS и CONFIG_CPUSETS. В идеале
ваша рабочая среда больше нигде не использует cpusets — тогда изолировать ядро
CPU проще всего. Если cpusets вам всё же нужны, проверьте, что вспомогательный
скрипт неточного логического анализатора не конфликтует с другими вашими
настройками.

Сообщите ядру, какие GPIO используются в качестве щупов. Для системы на основе
Device Tree нужно использовать следующие привязки (binding). Поскольку эти
привязки предназначены только для отладки, официальной схемы для них нет::

    i2c-analyzer {
            compatible = "gpio-sloppy-logic-analyzer";
            probe-gpios = <&gpio6 21 GPIO_OPEN_DRAIN>, <&gpio6 4 GPIO_OPEN_DRAIN>;
            probe-names = "SCL", "SDA";
    };

Обратите внимание, что для каждого указанного GPIO необходимо задать имя. В
настоящее время поддерживается максимум 8 щупов. 32, по всей видимости, тоже
возможны, но пока не реализованы.

Использование
=============

Логический анализатор настраивается через файлы в debugfs. Тем не менее
настоятельно рекомендуется не использовать их напрямую, а пользоваться скриптом
``tools/gpio/gpio-sloppy-logic-analyzer``. Помимо более тщательной проверки
параметров, он изолирует ядро CPU, благодаря чему во время измерений будет
наименьшее количество помех.

У скрипта есть опция справки, поясняющая параметры. Для приведённого выше
фрагмента DT, который анализирует шину I2C на частоте 400 кГц на плате Renesas
Salvator-XS, используются следующие настройки. Изолированным CPU должен быть
CPU1, потому что это большое ядро в конфигурации big.LITTLE. Поскольку CPU1
используется по умолчанию, этот параметр задавать не нужно. Скорость шины
составляет 400 кГц. Таким образом, теорема о дискретизации гласит, что опрашивать
нужно как минимум на частоте 800 кГц. Однако спадающие фронты обоих сигналов в
условии старта I2C происходят быстрее, поэтому нужна более высокая частота
опроса, например ``-s 1500000`` для 1,5 МГц. Кроме того, мы не хотим начинать
опрос сразу же, а хотим дождаться условия старта на свободной шине. Поэтому нужно
задать триггер на спадающий фронт SDA при удержании SCL в высоком уровне, т.е.
``-t 1H+2F``. Последнее — это длительность; предположим здесь 15 мс, что даёт
параметр ``-d 15000``. Итак, всё вместе::

    gpio-sloppy-logic-analyzer -s 1500000 -t 1H+2F -d 15000

Обратите внимание, что процесс вернёт вас к приглашению командной строки, но
дочерний процесс всё ещё ведёт опрос в фоновом режиме. Пока он не завершится, вы
не найдёте файл результата в текущем или указанном каталоге. Для приведённого
выше примера затем нужно будет инициировать обмен данными по I2C::

    i2cdetect -y -r <your bus number>

Результатом является файл .sr, предназначенный для использования с PulseView или
sigrok-cli из свободного проекта `sigrok`_. Это zip-файл, который также содержит
двоичные данные выборок (sample), которые могут быть обработаны другим
программным обеспечением. Имя файла состоит из имени экземпляра логического
анализатора плюс временной метки от начала эпохи.

.. _sigrok: https://sigrok.org/
