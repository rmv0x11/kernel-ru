=====================================
Сборки в контейнерах
=====================================

Инструмент ``container`` можно использовать для запуска любой команды в дереве
исходного кода ядра внутри контейнера.  Это упрощает воспроизведение сборок на
различных платформах, например, когда тестовый бот сообщил о проблеме, для
которой требуется определённая версия компилятора или внешний набор тестов.
Хотя это уже могут делать пользователи, знакомые с контейнерами, наличие
выделенного инструмента в дереве ядра снижает порог входа, решая
распространённые проблемы раз и навсегда (например, управление
идентификаторами пользователей).  Это также упрощает обмен точной командной
строкой, приводящей к конкретному результату.  Основным сценарием
использования, скорее всего, будут сборки ядра, но запустить можно практически
что угодно: KUnit, checkpatch и т. п. при условии, что доступен подходящий
образ.


Параметры
=========

Синтаксис командной строки::

  scripts/container -i IMAGE [OPTION]... CMD...

Доступные параметры:

``-e, --env-file ENV_FILE``

    Путь к файлу окружения для загрузки в контейнер.

``-g, --gid GID``

    Идентификатор группы для использования внутри контейнера.

``-i, --image IMAGE``

    Имя образа контейнера (обязательно).

``-r, --runtime RUNTIME``

    Имя среды выполнения контейнера.  Поддерживаемые среды выполнения:
    ``docker``, ``podman``.

    Если не указано, будет использована первая найденная в системе,
    т. е. Podman, если он присутствует, иначе Docker.

``-s, --shell``

    Запустить контейнер в интерактивной оболочке.

``-u, --uid UID``

    Идентификатор пользователя для использования внутри контейнера.

    Если параметр ``-g`` не указан, идентификатор пользователя будет также
    использован в качестве идентификатора группы.

``-v, --verbose``

    Включить подробный вывод.

``-h, --help``

    Показать справочное сообщение и выйти.


Использование
=============

Выбор используемого образа полностью остаётся за пользователем, а аргументы
``CMD`` передаются напрямую как произвольная командная строка для запуска в
контейнере.  Инструмент позаботится о монтировании дерева исходного кода в
качестве текущего рабочего каталога и при необходимости скорректирует
идентификаторы пользователя и группы.

Образ контейнера, который обычно включает тулчейн компилятора (toolchain),
предоставляется пользователем и выбирается с помощью параметра ``-i``.  Среду
выполнения контейнера можно выбрать с помощью параметра ``-r``, который может
принимать значение ``docker`` или ``podman``.  Если ничего не указано, будет
использована первая найденная в системе, при этом приоритет отдаётся Podman.
Поддержка других сред выполнения может быть добавлена позже в зависимости от их
популярности среди пользователей.

По умолчанию команды выполняются неинтерактивно.  Пользователь может прервать
работающий контейнер сигналом SIGINT (Ctrl-C).  Для интерактивного выполнения
команд с TTY можно использовать параметр ``--shell`` или ``-s``.  Тогда сигналы
будут получаться напрямую оболочкой, а не родительским процессом ``container``.
Чтобы выйти из интерактивной оболочки, используйте Ctrl-D или ``exit``.

.. note::

   Единственное требование к хосту помимо среды выполнения контейнера — это
   Python 3.10 или новее.

.. note::

   Сборки вне дерева (out-of-tree) пока поддерживаются не полностью.  Тем не
   менее параметр ``O=`` уже можно использовать с относительным путём внутри
   дерева исходного кода, чтобы хранить раздельные результаты сборки.  Обходной
   путь для сборки вне дерева — использовать ``mount --bind``, см. раздел с
   примерами далее.


Переменные окружения
====================

Переменные окружения не передаются в контейнер, поэтому они должны быть либо
определены в самом образе, либо через параметр ``-e`` с помощью файла
окружения.  В некоторых случаях имеет больше смысла определить их в
Containerfile, используемом для создания образа.  Например, образ тулчейна
компилятора только на основе Clang может иметь определение ``LLVM=1``.

Локальный файл окружения более полезен для специфичных для пользователя
переменных, добавляемых в процессе разработки.  Он передаётся среде выполнения
контейнера как есть, поэтому его формат может различаться.  Как правило, он
будет выглядеть как вывод ``env``.  Например::

  INSTALL_MOD_STRIP=1
  SOME_RANDOM_TEXT=One upon a time

Также обратите внимание, что параметры ``make`` по-прежнему можно передавать в
командной строке, поэтому, хотя так сделать нельзя, поскольку первым аргументом
должен быть исполняемый файл::

  scripts/container -i docker.io/tuxmake/korg-clang LLVM=1 make  # won't work

вот так сработает::

  scripts/container -i docker.io/tuxmake/korg-clang make LLVM=1


Идентификаторы пользователей
============================

Это область, в которой поведение будет немного различаться в зависимости от
среды выполнения контейнера.  Цель — выполнять команды от имени пользователя,
вызывающего инструмент.  В Podman создаётся пространство имён для отображения
идентификатора текущего пользователя в другой идентификатор в контейнере (1000
по умолчанию).  В Docker, хотя это также возможно в недавних версиях, требуется
включить специальную функциональность в демоне, поэтому здесь она не
используется ради простоты.  Вместо этого контейнер запускается напрямую с
идентификатором текущего пользователя.  В обоих случаях это обеспечит одинаковые
права доступа к файлам дерева исходного кода ядра, смонтированному как том.
Единственное различие в том, что при использовании Docker без пространства имён
идентификатор пользователя может не совпадать с идентификатором по умолчанию,
заданным в образе.

Допустим, мы используем образ, в котором настроен пользователь по умолчанию с
идентификатором 1000, а текущий пользователь, вызывающий инструмент
``container``, имеет идентификатор 1234.  Дерево исходного кода ядра было
выгружено этим же пользователем, поэтому файлы принадлежат пользователю 1234.  В
Podman контейнер будет работать с идентификатором пользователя 1000 с
отображением на идентификатор 1234, так что файлы из смонтированного тома будут
выглядеть внутри контейнера как принадлежащие идентификатору 1000.  В Docker без
пространства имён контейнер будет работать с идентификатором пользователя 1234,
который может получить доступ к файлам в томе, но не в домашнем каталоге
пользователя 1000.  Это не должно быть проблемой при выполнении команд только в
дереве ядра, но стоит подчеркнуть это здесь, поскольку это может иметь значение в
особых граничных случаях.

.. note::

   Режим `совместимости с Docker
   <https://podman-desktop.io/docs/migrating-from-docker/managing-docker-compatibility>`__
   в Podman для запуска команд ``docker`` поверх бэкенда Podman более сложен и
   пока поддерживается не полностью.  По этой причине Podman будет иметь
   приоритет, если в системе доступны обе среды выполнения.


Примеры
=======

Проект TuxMake предоставляет множество предварительно собранных образов
контейнеров, доступных на `Docker Hub <https://hub.docker.com/u/tuxmake>`__.
Вот кратчайший пример сборки ядра с использованием образа TuxMake Clang::

  scripts/container -i docker.io/tuxmake/korg-clang -- make LLVM=1 defconfig
  scripts/container -i docker.io/tuxmake/korg-clang -- make LLVM=1 -j$(nproc)

.. note::

   При запуске команды с параметрами внутри контейнера её следует отделять
   двойным дефисом ``--``, чтобы не путать их с параметрами инструмента
   ``container``.  Для простых команд без параметров двойные дефисы строго не
   требуются, например::

     scripts/container -i docker.io/tuxmake/korg-clang make mrproper

Чтобы запустить ``checkpatch.pl`` в каталоге ``patches`` с универсальным образом
Perl::

  scripts/container -i perl:slim-trixie scripts/checkpatch.pl patches/*

В качестве альтернативы образам TuxMake в приведённых ниже примерах
используются образы ``kernel.org``, которые основаны на `тулчейнах компилятора
kernel.org <https://mirrors.edge.kernel.org/pub/tools/>`__.  Они (пока) не
доступны официально ни в одном публичном реестре, но пользователи могут вместо
этого собрать их самостоятельно локально, используя этот `экспериментальный
репозиторий <https://gitlab.com/gtucker/korg-containers>`__, выполнив ``make
PREFIX=kernel.org/``.

Чтобы собрать только ``bzImage`` с использованием Clang::

  scripts/container -i kernel.org/clang -- make bzImage -j$(nproc)

То же самое с GCC 15 в качестве конкретного тега версии::

  scripts/container -i kernel.org/gcc:15 -- make bzImage -j$(nproc)

Для сборки вне дерева трюк состоит в том, чтобы привязочно смонтировать
(bind-mount) каталог назначения к относительному пути внутри дерева исходного
кода::

  mkdir -p $HOME/tmp/my-kernel-build
  mkdir -p build
  sudo mount --bind $HOME/tmp/my-kernel-build build
  scripts/container -i kernel.org/gcc -- make mrproper
  scripts/container -i kernel.org/gcc -- make O=build defconfig
  scripts/container -i kernel.org/gcc -- make O=build -j$(nproc)

Чтобы запустить KUnit в интерактивной оболочке и получить полный вывод::

  scripts/container -s -i kernel.org/gcc:kunit -- \
      tools/testing/kunit/kunit.py \
          run \
          --arch=x86_64 \
          --cross_compile=x86_64-linux-

Чтобы просто запустить интерактивную оболочку::

  scripts/container -si kernel.org/gcc bash

Чтобы собрать документацию в формате HTML, для чего требуется образ ``kdocs``,
собранный с помощью ``make PREFIX=kernel.org/ extra``, так как это не тулчейн
компилятора::

  scripts/container -i kernel.org/kdocs make htmldocs
