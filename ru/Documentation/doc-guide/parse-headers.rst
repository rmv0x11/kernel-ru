============================================
Включение заголовочных файлов uAPI
============================================

Иногда полезно включить заголовочные файлы и примеры кода на C, чтобы
описать API пространства пользователя (userspace) и сгенерировать
перекрёстные ссылки между кодом и документацией. Добавление перекрёстных
ссылок для файлов API пространства пользователя даёт дополнительное
преимущество: Sphinx будет генерировать предупреждения, если символ не
найден в документации. Это помогает поддерживать документацию uAPI в
синхронизации с изменениями ядра.
:ref:`parse_headers.py <parse_headers>` предоставляет способ генерировать
такие перекрёстные ссылки. Его необходимо вызывать через Makefile во время
сборки документации. Пример того, как использовать его внутри дерева ядра,
см. в ``Documentation/userspace-api/media/Makefile``.

.. _parse_headers:

tools/docs/parse_headers.py
^^^^^^^^^^^^^^^^^^^^^^^^^^^

NAME
****

parse_headers.py - разбирает файл на C, чтобы идентифицировать функции,
структуры, перечисления (enum) и определения (define) и создать перекрёстные
ссылки в книге Sphinx.

USAGE
*****

parse-headers.py [-h] [-d] [-t] ``FILE_IN`` ``FILE_OUT`` ``FILE_RULES``

SYNOPSIS
********

Преобразует заголовочный или исходный файл на C ``FILE_IN`` в текст
ReStructured Text, включаемый через блок ..parsed-literal, с перекрёстными
ссылками на файлы документации, которые описывают API. Принимает
необязательный файл ``FILE_RULES``, описывающий, какие элементы будут либо
проигнорированы, либо указывать на нестандартный тип/имя ссылки.

Результат записывается в ``FILE_OUT``.

Он способен идентифицировать ``define``, ``struct``, ``typedef``, ``enum``
и ``symbol`` перечисления, создавая перекрёстные ссылки для всех из них.

Он также способен отличать ``#define``, используемые для задания
специфичных для Linux макросов, применяемых для определения ``ioctl``.

Необязательный ``FILE_RULES`` содержит набор правил вида::

    ignore ioctl VIDIOC_ENUM_FMT
    replace ioctl VIDIOC_DQBUF vidioc_qbuf
    replace define V4L2_EVENT_MD_FL_HAVE_FRAME_SEQ :c:type:`v4l2_event_motion_det`

POSITIONAL ARGUMENTS
********************

  ``FILE_IN``
      Входной файл на C
  ``FILE_OUT``
      Выходной файл RST
  ``FILE_RULES``
      Файл исключений (необязательный)

OPTIONS
*******

  ``-h``, ``--help``
      показать справочное сообщение и выйти
  ``-d``, ``--debug``
      Повысить уровень отладки. Можно использовать несколько раз
  ``-t``, ``--toc``
      вместо дословного блока вывести в файле RST таблицу содержания (TOC)


DESCRIPTION
***********

Создаёт обогащённую версию заголовочного файла ядра с перекрёстными ссылками
на каждый тип структуры данных C из ``FILE_IN``, форматируя её с использованием
нотации reStructuredText, либо как есть, либо как оглавление.

Принимает необязательный ``FILE_RULES``, который описывает, какие элементы
будут либо проигнорированы, либо указывать на нестандартную ссылку, и при
желании задаёт пространство имён C, которое будет использоваться.

Он предназначен для того, чтобы обеспечить более полную документацию, в
которой заголовки uAPI будут создавать перекрёстные ссылки на код.

Результат записывается в ``FILE_OUT``.

``FILE_RULES`` может содержать три типа выражений:
**ignore**, **replace** и **namespace**.

По умолчанию он создаёт правила для всех символов и определений, но также
позволяет разобрать файл исключений. Такой файл содержит набор правил,
использующих приведённый ниже синтаксис:

1. Правила игнорирования:

    ignore *type* *symbol*

Удаляет символ из генерации ссылок.

2. Правила замены:

    replace *type* *old_symbol* *new_reference*

    Заменяет *old_symbol* на *new_reference*.
    *new_reference* может быть:

    - Простым именем символа;
    - Полной ссылкой Sphinx.

3. Правила пространства имён

    namespace *namespace*

    Задаёт пространство имён C *namespace*, которое будет использоваться при
    генерации перекрёстных ссылок. Может быть переопределено правилами замены.

В правилах ignore и replace *type* может быть:

    - ioctl:
        для определений вида ``_IO*``, например, для определений ioctl

    - define:
        для прочих определений

    - symbol:
        для символов, определённых внутри перечислений;

    - typedef:
        для typedef;

    - enum:
        для имени неанонимного перечисления;

    - struct:
        для структур.


EXAMPLES
********

- Проигнорировать определение ``_VIDEODEV2_H`` в ``FILE_IN``::

    ignore define _VIDEODEV2_H

- Для структуры данных вроде такого перечисления::

    enum foo { BAR1, BAR2, PRIVATE };

  Он не будет генерировать перекрёстные ссылки для ``PRIVATE``::

    ignore symbol PRIVATE

  Для той же структуры вместо создания отдельной перекрёстной ссылки на каждый
  символ сделать так, чтобы все они указывали на тип C ``enum foo``::

    replace symbol BAR1 :c:type:\`foo\`
    replace symbol BAR2 :c:type:\`foo\`


- Использовать пространство имён C ``MC`` для всех символов в ``FILE_IN``::

    namespace MC
