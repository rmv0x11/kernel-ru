.. _sphinxdoc:

============================================================
Использование Sphinx для документации ядра
============================================================

Ядро Linux использует `Sphinx`_ для генерации красивой документации из
файлов в формате `reStructuredText`_, расположенных в ``Documentation``. Чтобы
собрать документацию в форматах HTML или PDF, используйте ``make htmldocs`` или
``make pdfdocs``. Сгенерированная документация помещается в
``Documentation/output``.

.. _Sphinx: http://www.sphinx-doc.org/
.. _reStructuredText: http://docutils.sourceforge.net/rst.html

Файлы reStructuredText могут содержать директивы для включения структурированных
комментариев документации, или kernel-doc комментариев, из исходных файлов.
Обычно они используются для описания функций, типов и устройства кода.
Комментарии kernel-doc имеют некоторую особую структуру и форматирование, но в
остальном они также обрабатываются как reStructuredText.

Наконец, по всему каталогу ``Documentation`` разбросаны тысячи файлов
документации в виде обычного текста. Некоторые из них со временем, вероятно,
будут преобразованы в reStructuredText, но большинство останется в виде обычного
текста.

Установка Sphinx
================

Разметка ReST, используемая в настоящее время файлами в Documentation/,
предназначена для сборки с помощью ``Sphinx`` версии 3.4.3 или выше.

Существует скрипт, который проверяет требования Sphinx. Подробнее см.
:ref:`sphinx-pre-install`.

В большинство дистрибутивов входит Sphinx, но его тулчейн (toolchain) хрупок,
и нередко его обновление или обновление каких-либо других пакетов Python на вашей
машине приводит к поломке сборки документации.

Способ избежать этого — использовать версию, отличную от той, что поставляется
с вашим дистрибутивом. Для этого рекомендуется установить Sphinx внутри
виртуального окружения, используя ``virtualenv-3`` или ``virtualenv`` в
зависимости от того, как ваш дистрибутив упаковал Python 3.

Подводя итог: если вы хотите установить последнюю версию Sphinx, вам следует
выполнить::

       $ virtualenv sphinx_latest
       $ . sphinx_latest/bin/activate
       (sphinx_latest) $ pip install -r Documentation/sphinx/requirements.txt

После выполнения ``. sphinx_latest/bin/activate`` приглашение командной строки
изменится, чтобы показать, что вы используете новое окружение. Если вы
открываете новую командную оболочку, перед сборкой документации необходимо
снова выполнить эту команду, чтобы вновь войти в виртуальное окружение.

Вывод изображений
-----------------

Система сборки документации ядра содержит расширение, которое обрабатывает
изображения в форматах GraphViz и SVG (см. :ref:`sphinx_kfigure`).

Чтобы оно работало, необходимо установить пакеты GraphViz и ImageMagick. Если
эти пакеты не установлены, система сборки всё равно соберёт документацию, но не
включит в вывод никаких изображений.

Сборка PDF и LaTeX
------------------

Такие сборки в настоящее время поддерживаются только с Sphinx версий 2.4 и выше.

Для вывода в PDF и LaTeX вам также понадобится ``XeLaTeX`` версии 3.14159265.

В зависимости от дистрибутива вам также может потребоваться установить ряд
пакетов ``texlive``, которые предоставляют минимальный набор функциональности,
необходимый для работы ``XeLaTeX``.

Математические выражения в HTML
-------------------------------

Некоторые страницы ReST содержат математические выражения. Из-за особенностей
работы Sphinx эти выражения записываются с использованием нотации LaTeX.
Существует два варианта рендеринга математических выражений в выводе html.
Первый — это расширение под названием `imgmath`_, которое преобразует
математические выражения в изображения и встраивает их в html-страницы.
Второй — это расширение под названием `mathjax`_, которое делегирует рендеринг
математики веб-браузерам, поддерживающим JavaScript.
Первый был единственным вариантом для документации ядра версий до 6.1, и он
требует довольно большого числа пакетов texlive, включая, помимо прочего,
amsfonts и amsmath.

Начиная с релиза ядра 6.1, html-страницы с математическими выражениями можно
собирать без установки каких-либо пакетов texlive. Подробнее см.
`Выбор рендерера математики`_.

.. _imgmath: https://www.sphinx-doc.org/en/master/usage/extensions/math.html#module-sphinx.ext.imgmath
.. _mathjax: https://www.sphinx-doc.org/en/master/usage/extensions/math.html#module-sphinx.ext.mathjax

Проверка зависимостей Sphinx
----------------------------

Существует скрипт, который автоматически проверяет зависимости Sphinx. Если он
сможет распознать ваш дистрибутив, он также подскажет параметры командной строки
для установки в вашем дистрибутиве::

	$ ./tools/docs/sphinx-pre-install
	Checking if the needed tools for Fedora release 26 (Twenty Six) are available
	Warning: better to also install "texlive-luatex85".
	You should run:

		sudo dnf install -y texlive-luatex85
		/usr/bin/virtualenv sphinx_2.4.4
		. sphinx_2.4.4/bin/activate
		pip install -r Documentation/sphinx/requirements.txt

	Can't build as 1 mandatory dependency is missing at ./tools/docs/sphinx-pre-install line 468.

По умолчанию он проверяет все требования как для html, так и для PDF, включая
требования для изображений, математических выражений и сборки LaTeX, и
предполагает, что будет использоваться виртуальное окружение Python. Требования,
необходимые для сборки html, считаются обязательными; остальные —
необязательными.

Он поддерживает два необязательных параметра:

``--no-pdf``
	Отключить проверки для PDF;

``--no-virtualenv``
	Использовать пакеты ОС для Sphinx вместо виртуального окружения Python.

Установка минимальной версии Sphinx
-----------------------------------

При изменении системы сборки Sphinx важно убедиться, что минимальная версия
по-прежнему будет поддерживаться. В наши дни делать это на современных
дистрибутивах становится всё сложнее, поскольку установка невозможна на
Python 3.13 и выше.

Тестирование с наименьшей поддерживаемой версией Python, как определено в
Documentation/process/changes.rst, можно выполнить, создав с ней venv и
установив минимальные требования с помощью::

	/usr/bin/python3.9 -m venv sphinx_min
	. sphinx_min/bin/activate
	pip install -r Documentation/sphinx/min_requirements.txt

Более всестороннее тестирование можно выполнить с помощью:

	tools/docs/test_doc_build.py

Такой скрипт создаёт по одному venv Python для каждой поддерживаемой версии,
при необходимости собирая документацию для диапазона версий Sphinx.


Сборка Sphinx
=============

Обычный способ сгенерировать документацию — выполнить ``make htmldocs`` или
``make pdfdocs``. Доступны также другие форматы: см. раздел документации в выводе
``make help``. Сгенерированная документация помещается в специфичные для
формата подкаталоги в ``Documentation/output``.

Чтобы сгенерировать документацию, очевидно, должен быть установлен Sphinx
(``sphinx-build``).  Для вывода в PDF вам также понадобятся ``XeLaTeX`` и
``convert(1)`` из ImageMagick (https://www.imagemagick.org).\ [#ink]_ Всё это
широко доступно и упаковано в дистрибутивах.

Чтобы передать дополнительные параметры в Sphinx, можно использовать make-переменную
``SPHINXOPTS``. Например, используйте ``make SPHINXOPTS=-v htmldocs``, чтобы
получить более подробный вывод.

Также можно передать дополнительный накладываемый файл DOCS_CSS, чтобы
настроить макет html, используя make-переменную ``DOCS_CSS``.

По умолчанию для сборки документации в HTML используется тема «Alabaster»;
эта тема входит в комплект Sphinx и не нуждается в отдельной установке.
Тему Sphinx можно переопределить с помощью make-переменной ``DOCS_THEME``.

.. note::

   Некоторые предпочитают использовать тему RTD для вывода html.
   В зависимости от версии Sphinx её следует установить отдельно с помощью
   ``pip install sphinx_rtd_theme``.

Существует ещё одна make-переменная ``SPHINXDIRS``, которая полезна при тестовой
сборке части документации.  Например, вы можете собрать документы в
``Documentation/doc-guide``, выполнив ``make SPHINXDIRS=doc-guide htmldocs``.
Раздел документации в выводе ``make help`` покажет вам список подкаталогов,
которые можно указать.

Чтобы удалить сгенерированную документацию, выполните ``make cleandocs``.

.. [#ink] Наличие также ``inkscape(1)`` из Inkscape (https://inkscape.org)
	  улучшило бы качество изображений, встраиваемых в PDF-документы,
	  особенно для релизов ядра 5.18 и более поздних.

Выбор рендерера математики
--------------------------

Начиная с релиза ядра 6.1, mathjax работает как резервный рендерер математики
для вывода html.\ [#sph1_8]_

Рендерер математики выбирается в зависимости от доступных команд, как показано
ниже:

.. table:: Math Renderer Choices for HTML

    ============= ================= ============
    Math renderer Required commands Image format
    ============= ================= ============
    imgmath       latex, dvipng     PNG (raster)
    mathjax
    ============= ================= ============

Выбор можно переопределить, установив переменную окружения ``SPHINX_IMGMATH``,
как показано ниже:

.. table:: Effect of Setting ``SPHINX_IMGMATH``

    ====================== ========
    Setting                Renderer
    ====================== ========
    ``SPHINX_IMGMATH=yes`` imgmath
    ``SPHINX_IMGMATH=no``  mathjax
    ====================== ========

.. [#sph1_8] Резервный рендерер математики требует Sphinx >=1.8.


Написание документации
=======================

Добавление новой документации может быть простым:

1. Добавьте новый файл ``.rst`` где-нибудь в ``Documentation``.
2. Сошлитесь на него из главного `TOC tree`_ Sphinx в ``Documentation/index.rst``.

.. _TOC tree: http://www.sphinx-doc.org/en/stable/markup/toctree.html

Этого обычно достаточно для простой документации (вроде той, что вы читаете
прямо сейчас), но для более крупных документов может быть целесообразно создать
подкаталог (или использовать существующий). Например, документация графической
подсистемы расположена в ``Documentation/gpu``, разбита на несколько файлов
``.rst`` и имеет отдельный ``index.rst`` (со своим собственным ``toctree``), на
который ссылается главный индекс.

См. документацию по `Sphinx`_ и `reStructuredText`_ о том, что можно делать с их
помощью. В частности, `reStructuredText Primer`_ от Sphinx — хорошее место для
начала знакомства с reStructuredText. Существуют также некоторые `Sphinx specific
markup constructs`_.

.. _reStructuredText Primer: http://www.sphinx-doc.org/en/stable/rest.html
.. _Sphinx specific markup constructs: http://www.sphinx-doc.org/en/stable/markup/index.html

Специальные рекомендации для документации ядра
----------------------------------------------

Вот некоторые специальные рекомендации для документации ядра:

* Пожалуйста, не увлекайтесь разметкой reStructuredText. Сохраняйте её
  простой. По большей части документация должна быть обычным текстом с
  достаточным единообразием форматирования, чтобы её можно было преобразовать
  в другие форматы.

* Пожалуйста, делайте изменения форматирования минимальными при преобразовании
  существующей документации в reStructuredText.

* Также обновляйте содержание, а не только форматирование, при преобразовании
  документации.

* Пожалуйста, придерживайтесь следующего порядка оформления заголовков:

  1. ``=`` с надчёркиванием для заголовка документа::

       ==============
       Document title
       ==============

  2. ``=`` для глав::

       Chapters
       ========

  3. ``-`` для разделов::

       Section
       -------

  4. ``~`` для подразделов::

       Subsection
       ~~~~~~~~~~

  Хотя RST не предписывает определённого порядка («Вместо навязывания
  фиксированного числа и порядка стилей оформления заголовков разделов,
  принудительным становится тот порядок, в котором они встречаются»), одинаковость
  более высоких уровней в целом облегчает чтение документов.

* Для вставки блоков текста фиксированной ширины (для примеров кода, примеров
  сценариев использования и т.п.) используйте ``::`` для всего, что на самом деле
  не выигрывает от подсветки синтаксиса, особенно для коротких фрагментов.
  Используйте ``.. code-block:: <language>`` для более длинных блоков кода,
  которые выигрывают от подсветки. Для короткого фрагмента кода, встроенного в
  текст, используйте \`\`.


Домен C
-------

**Домен C Sphinx** (name c) подходит для документирования C API. Например,
прототип функции:

.. code-block:: rst

    .. c:function:: int ioctl( int fd, int request )

Домен C в kernel-doc имеет некоторые дополнительные возможности. Например, вы
можете *переименовать* имя ссылки функции с распространённым именем вроде
``open`` или ``ioctl``:

.. code-block:: rst

     .. c:function:: int ioctl( int fd, int request )
        :name: VIDIOC_LOG_STATUS

Имя функции (например, ioctl) остаётся в выводе, но имя ссылки изменяется с
``ioctl`` на ``VIDIOC_LOG_STATUS``. Запись в индексе для этой функции также
изменяется на ``VIDIOC_LOG_STATUS``.

Обратите внимание, что нет необходимости использовать ``c:func:`` для генерации
перекрёстных ссылок на документацию функций.  Благодаря некоторой магии
расширения Sphinx система сборки документации автоматически превратит ссылку на
``function()`` в перекрёстную ссылку, если для данного имени функции существует
запись в индексе.  Если вы видите использование ``c:func:`` в документе ядра,
не стесняйтесь его удалить.

Таблицы
-------

ReStructuredText предоставляет несколько вариантов синтаксиса таблиц. Стиль ядра
для таблиц — предпочитать синтаксис *simple table* или *grid table*. Подробнее
см. `reStructuredText user reference for table syntax`_.

.. _reStructuredText user reference for table syntax:
   https://docutils.sourceforge.io/docs/user/rst/quickref.html#tables

Таблицы-списки
~~~~~~~~~~~~~~

Форматы list-table могут быть полезны для таблиц, которые непросто разместить в
обычных форматах ASCII-графики Sphinx.  Однако эти форматы почти невозможно
понять читателям документов в виде обычного текста, и их следует избегать при
отсутствии веских обоснований для их использования.

``flat-table`` — это двухуровневый список, похожий на ``list-table``, с
некоторыми дополнительными возможностями:

* объединение столбцов (column-span): с помощью роли ``cspan`` ячейку можно
  расширить на дополнительные столбцы

* объединение строк (row-span): с помощью роли ``rspan`` ячейку можно расширить
  на дополнительные строки

* автоматическое расширение крайней правой ячейки строки таблицы на
  недостающие ячейки в правой части этой строки таблицы.  С помощью опции
  ``:fill-cells:`` это поведение можно изменить с *auto span* на *auto fill*,
  при котором вместо расширения последней ячейки автоматически вставляются
  (пустые) ячейки.

опции:

* ``:header-rows:``   [int] количество строк заголовка
* ``:stub-columns:``  [int] количество столбцов-заглушек
* ``:widths:``        [[int] [int] ... ] ширины столбцов
* ``:fill-cells:``    вместо автоматического расширения на недостающие ячейки вставлять недостающие ячейки

роли:

* ``:cspan:`` [int] дополнительные столбцы (*morecols*)
* ``:rspan:`` [int] дополнительные строки (*morerows*)

Пример ниже показывает, как использовать эту разметку.  Первый уровень
многоуровневого списка — это *table-row*. В *table-row* допускается только одна
разметка — список ячеек этой *table-row*. Исключениями являются *комментарии*
( ``..`` ) и *цели* (например, ссылка на ``:ref:`last row <last row>``` /
:ref:`last row <last row>`).

.. code-block:: rst

   .. flat-table:: table title
      :widths: 2 1 1 3

      * - head col 1
        - head col 2
        - head col 3
        - head col 4

      * - row 1
        - field 1.1
        - field 1.2 with autospan

      * - row 2
        - field 2.1
        - :rspan:`1` :cspan:`1` field 2.2 - 3.3

      * .. _`last row`:

        - row 3

Отрендерено как:

   .. flat-table:: table title
      :widths: 2 1 1 3

      * - head col 1
        - head col 2
        - head col 3
        - head col 4

      * - row 1
        - field 1.1
        - field 1.2 with autospan

      * - row 2
        - field 2.1
        - :rspan:`1` :cspan:`1` field 2.2 - 3.3

      * .. _`last row`:

        - row 3

Перекрёстные ссылки
-------------------

Перекрёстную ссылку с одной страницы документации на другую можно сделать просто,
написав путь к файлу документа, без специального синтаксиса. Путь может быть как
абсолютным, так и относительным. Для абсолютных путей начинайте его с
"Documentation/". Например, чтобы сослаться на эту страницу, все следующие
варианты допустимы в зависимости от каталога текущего документа (обратите
внимание, что расширение ``.rst`` обязательно)::

    See Documentation/doc-guide/sphinx.rst. This always works.
    Take a look at sphinx.rst, which is at this same directory.
    Read ../sphinx.rst, which is one directory above.

Если вы хотите, чтобы у ссылки был отрендеренный текст, отличный от заголовка
документа, вам нужно использовать роль ``doc`` Sphinx. Например::

    See :doc:`my custom link text for document sphinx <sphinx>`.

В большинстве случаев предпочтителен первый вариант, поскольку он чище и больше
подходит для тех, кто читает исходные файлы. Если вам встретится использование
``:doc:``, которое не добавляет никакой ценности, не стесняйтесь преобразовать
его в просто путь к документу.

Информацию о перекрёстных ссылках на функции или типы kernel-doc см. в
Documentation/doc-guide/kernel-doc.rst.

Ссылки на коммиты
~~~~~~~~~~~~~~~~~

Ссылки на git-коммиты автоматически становятся гиперссылками при условии, что
они записаны в одном из следующих форматов::

    commit 72bf4f1767f0
    commit 72bf4f1767f0 ("net: do not leave an empty skb in write queue")

Рисунки и изображения
=====================

Если вы хотите добавить изображение, следует использовать директивы
``kernel-figure`` и ``kernel-image``. Например, чтобы вставить рисунок с
масштабируемым форматом изображения, используйте SVG (:ref:`svg_image_example`)::

    .. kernel-figure::  svg_image.svg
       :alt:    simple SVG image

       SVG image example

.. _svg_image_example:

.. kernel-figure::  svg_image.svg
   :alt:    simple SVG image

   SVG image example

Директива kernel figure (и image) поддерживает файлы в формате **DOT**, см.

* DOT: http://graphviz.org/pdf/dotguide.pdf
* Graphviz: http://www.graphviz.org/content/dot-language

Простой пример (:ref:`hello_dot_file`)::

     :alt:    hello world

     DOT's hello world example

.. kernel-figure::  hello.dot

Встроенная *render*-разметка (или языки), такая как **DOT** от Graphviz,
предоставляется директивами ``kernel-render``.::

  .. kernel-render:: DOT
     :alt: foobar digraph
     :caption: Embedded **DOT** (Graphviz) code

     digraph foo {
      "bar" -> "baz";
     }

Как именно это будет отрендерено, зависит от установленных инструментов. Если
установлен Graphviz, вы увидите векторное изображение. Если нет, исходная
разметка вставляется как *literal-block* (:ref:`hello_dot_render`).

.. _hello_dot_render:

.. kernel-render:: DOT
   :alt: foobar digraph
   :caption: Embedded **DOT** (Graphviz) code

   digraph foo {
      "bar" -> "baz";
   }

Директива *render* имеет все опции, известные из директивы *figure*, плюс опцию
``caption``.  Если ``caption`` имеет значение, вставляется узел *figure*. Если
нет — вставляется узел *image*. ``caption`` также необходим, если вы хотите
сослаться на него (:ref:`hello_svg_render`).

Встроенный **SVG**::

  .. kernel-render:: SVG
     :caption: Embedded **SVG** markup
     :alt: so-nw-arrow

     <?xml version="1.0" encoding="UTF-8"?>
     <svg xmlns="http://www.w3.org/2000/svg" version="1.1" ...>
        ...
     </svg>

.. kernel-render:: SVG
   :caption: Embedded **SVG** markup
   :alt: so-nw-arrow

   </svg>
