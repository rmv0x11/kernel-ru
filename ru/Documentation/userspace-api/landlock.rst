=====================================================
Landlock: непривилегированное управление доступом
=====================================================

:Author: Mickaël Salaün
:Date: March 2026

Цель Landlock — предоставить возможность ограничивать неявные права (ambient
rights, например глобальный доступ к файловой системе или к сети) для набора
процессов. Поскольку Landlock является стекируемым LSM, он позволяет создавать
безопасные песочницы (sandbox) в виде новых уровней безопасности в дополнение
к существующим системным механизмам управления доступом. Ожидается, что такого
рода песочница поможет снизить влияние ошибок или неожиданного/вредоносного
поведения приложений пространства пользователя на безопасность. Landlock даёт
любому процессу, в том числе непривилегированному, возможность безопасно
ограничить самого себя.

Можно быстро убедиться, что Landlock включён в работающей системе, выполнив поиск
строки "landlock: Up and running" в журналах ядра (от имени root):
``dmesg | grep landlock || journalctl -kb -g landlock`` .
Разработчики также могут легко проверить поддержку Landlock с помощью
:ref:`соответствующего системного вызова <landlock_abi_versions>`.
Если Landlock в данный момент не поддерживается, необходимо
:ref:`правильно сконфигурировать ядро <kernel_support>`.

Правила Landlock
================

Правило Landlock описывает действие над объектом, которое процесс намеревается
выполнить. Набор правил объединяется в набор правил (ruleset), который затем
может ограничивать применяющий его поток и его будущих потомков.

Существуют два типа правил:

Правила файловой системы
    Для таких правил объектом является иерархия файлов,
    а связанные с ней действия файловой системы определяются с помощью
    `filesystem access rights`.

Сетевые правила (начиная с ABI v4)
    Для таких правил объектом является TCP-порт,
    а связанные действия определяются с помощью `network access rights`.

Определение и применение политики безопасности
----------------------------------------------

Сначала необходимо определить набор правил, который будет содержать наши правила.

В этом примере набор правил будет содержать правила, которые разрешают только
действия чтения файловой системы и установление определённого TCP-соединения.
Действия записи в файловую систему и прочие TCP-действия будут запрещены.

Затем набор правил должен обрабатывать оба этих вида действий. Это требуется для
обратной и прямой совместимости (т.е. ядро и пространство пользователя могут не
знать о поддерживаемых ограничениях друг друга), отсюда и необходимость явно
указывать запрещённые по умолчанию права доступа.

.. code-block:: c

    struct landlock_ruleset_attr ruleset_attr = {
        .handled_access_fs =
            LANDLOCK_ACCESS_FS_EXECUTE |
            LANDLOCK_ACCESS_FS_WRITE_FILE |
            LANDLOCK_ACCESS_FS_READ_FILE |
            LANDLOCK_ACCESS_FS_READ_DIR |
            LANDLOCK_ACCESS_FS_REMOVE_DIR |
            LANDLOCK_ACCESS_FS_REMOVE_FILE |
            LANDLOCK_ACCESS_FS_MAKE_CHAR |
            LANDLOCK_ACCESS_FS_MAKE_DIR |
            LANDLOCK_ACCESS_FS_MAKE_REG |
            LANDLOCK_ACCESS_FS_MAKE_SOCK |
            LANDLOCK_ACCESS_FS_MAKE_FIFO |
            LANDLOCK_ACCESS_FS_MAKE_BLOCK |
            LANDLOCK_ACCESS_FS_MAKE_SYM |
            LANDLOCK_ACCESS_FS_REFER |
            LANDLOCK_ACCESS_FS_TRUNCATE |
            LANDLOCK_ACCESS_FS_IOCTL_DEV,
        .handled_access_net =
            LANDLOCK_ACCESS_NET_BIND_TCP |
            LANDLOCK_ACCESS_NET_CONNECT_TCP,
        .scoped =
            LANDLOCK_SCOPE_ABSTRACT_UNIX_SOCKET |
            LANDLOCK_SCOPE_SIGNAL,
    };

Поскольку мы можем не знать, на какой версии ядра будет выполняться приложение,
безопаснее придерживаться подхода «максимально возможной защиты» (best-effort).
Действительно, нам следует стараться защитить пользователей настолько, насколько
это возможно, независимо от того, какое ядро они используют.

Чтобы быть совместимыми с более старыми версиями Linux, мы определяем доступную
версию ABI Landlock и используем только доступное подмножество прав доступа:

.. code-block:: c

    int abi;

    abi = landlock_create_ruleset(NULL, 0, LANDLOCK_CREATE_RULESET_VERSION);
    if (abi < 0) {
        /* Degrades gracefully if Landlock is not handled. */
        perror("The running kernel does not enable to use Landlock");
        return 0;
    }
    switch (abi) {
    case 1:
        /* Removes LANDLOCK_ACCESS_FS_REFER for ABI < 2 */
        ruleset_attr.handled_access_fs &= ~LANDLOCK_ACCESS_FS_REFER;
        __attribute__((fallthrough));
    case 2:
        /* Removes LANDLOCK_ACCESS_FS_TRUNCATE for ABI < 3 */
        ruleset_attr.handled_access_fs &= ~LANDLOCK_ACCESS_FS_TRUNCATE;
        __attribute__((fallthrough));
    case 3:
        /* Removes network support for ABI < 4 */
        ruleset_attr.handled_access_net &=
            ~(LANDLOCK_ACCESS_NET_BIND_TCP |
              LANDLOCK_ACCESS_NET_CONNECT_TCP);
        __attribute__((fallthrough));
    case 4:
        /* Removes LANDLOCK_ACCESS_FS_IOCTL_DEV for ABI < 5 */
        ruleset_attr.handled_access_fs &= ~LANDLOCK_ACCESS_FS_IOCTL_DEV;
        __attribute__((fallthrough));
    case 5:
        /* Removes LANDLOCK_SCOPE_* for ABI < 6 */
        ruleset_attr.scoped &= ~(LANDLOCK_SCOPE_ABSTRACT_UNIX_SOCKET |
                                 LANDLOCK_SCOPE_SIGNAL);
    }

Это позволяет создать включающий (inclusive) набор правил, который будет
содержать наши правила.

.. code-block:: c

    int ruleset_fd;

    ruleset_fd = landlock_create_ruleset(&ruleset_attr, sizeof(ruleset_attr), 0);
    if (ruleset_fd < 0) {
        perror("Failed to create a ruleset");
        return 1;
    }

Теперь мы можем добавить новое правило в этот набор правил благодаря
возвращённому файловому дескриптору, который ссылается на этот набор правил.
Правило разрешит чтение и выполнение в иерархии файлов ``/usr``. Без другого
правила действия записи будут затем запрещены набором правил. Чтобы добавить
``/usr`` в набор правил, мы открываем его с флагом ``O_PATH`` и заполняем
&struct landlock_path_beneath_attr этим файловым дескриптором.

.. code-block:: c

    int err;
    struct landlock_path_beneath_attr path_beneath = {
        .allowed_access =
            LANDLOCK_ACCESS_FS_EXECUTE |
            LANDLOCK_ACCESS_FS_READ_FILE |
            LANDLOCK_ACCESS_FS_READ_DIR,
    };

    path_beneath.parent_fd = open("/usr", O_PATH | O_CLOEXEC);
    if (path_beneath.parent_fd < 0) {
        perror("Failed to open file");
        close(ruleset_fd);
        return 1;
    }
    err = landlock_add_rule(ruleset_fd, LANDLOCK_RULE_PATH_BENEATH,
                            &path_beneath, 0);
    close(path_beneath.parent_fd);
    if (err) {
        perror("Failed to update ruleset");
        close(ruleset_fd);
        return 1;
    }

Также может потребоваться создавать правила по той же логике, что описана для
создания набора правил, отфильтровывая права доступа в соответствии с версией
ABI Landlock. В данном примере это не требуется, поскольку все запрашиваемые
права ``allowed_access`` уже доступны в ABI 1.

Для управления доступом к сети мы можем добавить набор правил, разрешающих
использование номера порта для определённого действия: HTTPS-соединений.

.. code-block:: c

    struct landlock_net_port_attr net_port = {
        .allowed_access = LANDLOCK_ACCESS_NET_CONNECT_TCP,
        .port = 443,
    };

    err = landlock_add_rule(ruleset_fd, LANDLOCK_RULE_NET_PORT,
                            &net_port, 0);

При передаче ненулевого аргумента ``flags`` в ``landlock_restrict_self()``
необходима аналогичная проверка обратной совместимости для флагов ограничения
(доступные флаги см. в документации sys_landlock_restrict_self()):

.. code-block:: c

    __u32 restrict_flags =
        LANDLOCK_RESTRICT_SELF_LOG_NEW_EXEC_ON |
        LANDLOCK_RESTRICT_SELF_TSYNC;
    switch (abi) {
    case 1 ... 6:
        /* Removes logging flags for ABI < 7 */
        restrict_flags &= ~(LANDLOCK_RESTRICT_SELF_LOG_SAME_EXEC_OFF |
                            LANDLOCK_RESTRICT_SELF_LOG_NEW_EXEC_ON |
                            LANDLOCK_RESTRICT_SELF_LOG_SUBDOMAINS_OFF);
        __attribute__((fallthrough));
    case 7:
        /*
         * Removes multithreaded enforcement flag for ABI < 8
         *
         * WARNING: Without this flag, calling landlock_restrict_self(2) is
         * only equivalent if the calling process is single-threaded. Below
         * ABI v8 (and as of ABI v8, when not using this flag), a Landlock
         * policy would only be enforced for the calling thread and its
         * children (and not for all threads, including parents and siblings).
         */
        restrict_flags &= ~LANDLOCK_RESTRICT_SELF_TSYNC;
    }

Следующим шагом является запрет текущему потоку получать дополнительные
привилегии (например, через SUID-бинарник). Теперь у нас есть набор правил, в
котором первое правило разрешает доступ на чтение и выполнение к ``/usr``,
запрещая все прочие обрабатываемые виды доступа к файловой системе, и второе
правило, разрешающее HTTPS-соединения.

.. code-block:: c

    if (prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0)) {
        perror("Failed to restrict privileges");
        close(ruleset_fd);
        return 1;
    }

Текущий поток теперь готов поместить себя в песочницу с этим набором правил.

.. code-block:: c

    if (landlock_restrict_self(ruleset_fd, restrict_flags)) {
        perror("Failed to enforce ruleset");
        close(ruleset_fd);
        return 1;
    }
    close(ruleset_fd);

Если системный вызов ``landlock_restrict_self`` завершается успешно, текущий
поток теперь ограничен, и эта политика будет применяться также ко всем
создаваемым им впоследствии потомкам. После того как поток помещён в песочницу
Landlock, его политику безопасности нельзя снять; разрешается только добавление
новых ограничений. Эти потоки теперь находятся в новом домене Landlock,
который является объединением родительского домена (если таковой был) с новым
набором правил.

Полный рабочий код можно найти в `samples/landlock/sandboxer.c`_.

Рекомендуемые практики
----------------------

Рекомендуется задавать права доступа как можно ближе к листьям иерархии файлов.
Например, лучше иметь ``~/doc/`` в качестве иерархии только для чтения и
``~/tmp/`` в качестве иерархии для чтения и записи, чем ``~/`` в качестве
иерархии только для чтения и ``~/tmp/`` в качестве иерархии для чтения и записи.
Следование этой рекомендуемой практике приводит к самодостаточным иерархиям,
которые не зависят от своего расположения (т.е. от родительских каталогов). Это
особенно важно, когда мы хотим разрешить создание ссылок или переименование.
Действительно, согласованные права доступа для каждого каталога позволяют менять
расположение таких каталогов без зависимости от прав доступа каталога назначения
(за исключением тех, которые требуются для этой операции, см. документацию
``LANDLOCK_ACCESS_FS_REFER``).

Наличие самодостаточных иерархий также помогает ужать необходимые права доступа
до минимального набора данных. Это также помогает избежать каталогов-«стоков»
(sinkhole), т.е. каталогов, в которые можно создавать ссылки на данные, но из
которых ссылаться нельзя. Однако это зависит от организации данных, которая может
не контролироваться разработчиками. В этом случае предоставление доступа на
чтение и запись к ``~/tmp/`` вместо доступа только на запись потенциально
позволит переместить ``~/tmp/`` в нечитаемый каталог и при этом сохранить
возможность перечислять содержимое ``~/tmp/``.

Уровни прав доступа к путям файлов
----------------------------------

Каждый раз, когда поток применяет к себе набор правил, он обновляет свой домен
Landlock новым уровнем политики. Эта дополнительная политика накладывается
поверх любых других наборов правил, которые потенциально уже ограничивают этот
поток. Поток в песочнице может затем безопасно добавить себе новые ограничения с
помощью нового применённого набора правил.

Один уровень политики предоставляет доступ к пути файла, если хотя бы одно из его
правил, встреченных на этом пути, предоставляет данный доступ. Поток в песочнице
может получить доступ к пути файла только если все применённые к нему уровни
политики предоставляют этот доступ, равно как и все остальные системные
механизмы управления доступом (например, DAC файловой системы, политики других
LSM и т.д.).

Bind-монтирование и OverlayFS
-----------------------------

Landlock позволяет ограничивать доступ к иерархиям файлов, что означает, что эти
права доступа могут распространяться через bind-монтирование (см.
Documentation/filesystems/sharedsubtree.rst), но не через
Documentation/filesystems/overlayfs.rst.

Bind-монтирование зеркально отображает исходную иерархию файлов в место
назначения. Иерархия назначения затем состоит из в точности тех же самых файлов,
к которым могут быть привязаны правила Landlock — либо через исходный путь, либо
через путь назначения. Эти правила ограничивают доступ, когда они встречаются на
пути, что означает, что они могут ограничивать доступ к нескольким иерархиям
файлов одновременно, независимо от того, являются эти иерархии результатом
bind-монтирования или нет.

Точка монтирования OverlayFS состоит из верхнего и нижнего уровней (layers). Эти
уровни объединяются в каталог слияния, и этот объединённый каталог становится
доступным в точке монтирования. Эта иерархия слияния может включать файлы из
верхнего и нижнего уровней, но изменения, выполняемые в иерархии слияния,
отражаются только на верхнем уровне. С точки зрения политики Landlock все уровни
OverlayFS и иерархии слияния являются самостоятельными и каждый содержит свой
собственный набор файлов и каталогов, что отличается от bind-монтирования.
Политика, ограничивающая уровень OverlayFS, не будет ограничивать получившуюся
объединённую иерархию, и наоборот. Поэтому пользователям Landlock следует думать
только об иерархиях файлов, к которым они хотят разрешить доступ, независимо от
нижележащей файловой системы.

Наследование
------------

Каждый новый поток, возникающий в результате :manpage:`clone(2)`, наследует
ограничения домена Landlock от своего родителя. Это аналогично наследованию
seccomp (см. Documentation/userspace-api/seccomp_filter.rst) или любому другому
LSM, работающему с :manpage:`credentials(7)` задачи. Например, поток одного
процесса может применить правила Landlock к самому себе, но они не будут
автоматически применены к другим родственным (sibling) потокам (в отличие от
изменений учётных данных потоков POSIX, см. :manpage:`nptl(7)`).

Когда поток помещает себя в песочницу, мы получаем гарантию того, что
соответствующая политика безопасности будет оставаться применённой ко всем
потомкам этого потока. Это позволяет создавать самостоятельные и модульные
политики безопасности для каждого приложения, которые будут автоматически
компоноваться между собой в соответствии с политиками их родителей во время
выполнения.

Ограничения ptrace
------------------

Процесс в песочнице имеет меньше привилегий, чем процесс не в песочнице, и
поэтому должен подвергаться дополнительным ограничениям при манипулировании
другим процессом. Чтобы процессу в песочнице было разрешено использовать
:manpage:`ptrace(2)` и связанные системные вызовы над целевым процессом, такой
процесс должен иметь надмножество прав доступа целевого процесса, что означает,
что трассируемый (tracee) должен находиться в поддомене трассировщика (tracer).

Ограничение области действия IPC
--------------------------------

Аналогично неявным `Ограничения ptrace`_, мы можем захотеть дополнительно
ограничить взаимодействия между песочницами. Поэтому во время создания набора
правил каждый домен Landlock может ограничивать область действия (scope) для
определённых операций, так что эти операции могут затрагивать только процессы в
том же домене Landlock или во вложенном домене Landlock (эта «область»).

Операции, для которых может быть ограничена область действия:

``LANDLOCK_SCOPE_SIGNAL``
    Это ограничивает отправку сигналов только теми целевыми процессами, которые
    выполняются в том же или во вложенном домене Landlock.

``LANDLOCK_SCOPE_ABSTRACT_UNIX_SOCKET``
    Это ограничивает множество абстрактных сокетов :manpage:`unix(7)`, к которым
    мы можем выполнить :manpage:`connect(2)`, теми адресами сокетов, которые были
    созданы процессом в том же или во вложенном домене Landlock.

    Вызов :manpage:`sendto(2)` на неподключённом датаграммном сокете трактуется
    так, как если бы он выполнял неявный :manpage:`connect(2)`, и будет
    заблокирован, если удалённая сторона не происходит из того же или из
    вложенного домена Landlock.

    Вызов :manpage:`sendto(2)` на сокете, который был ранее подключён, не будет
    ограничен. Это работает как для датаграммных, так и для потоковых сокетов.

Ограничение области действия IPC не поддерживает исключения через
:manpage:`landlock_add_rule(2)`. Если для операции ограничена область действия в
пределах домена, нельзя добавить никаких правил, разрешающих доступ к ресурсам
или процессам вне этой области.

Усечение файлов
---------------

Операции, охватываемые ``LANDLOCK_ACCESS_FS_WRITE_FILE`` и
``LANDLOCK_ACCESS_FS_TRUNCATE``, обе изменяют содержимое файла и иногда
перекрываются неинтуитивным образом. Рекомендуется всегда указывать оба этих
права вместе.

Особенно неожиданным примером является :manpage:`creat(2)`. Название наводит на
мысль, что этот системный вызов требует прав на создание и запись файлов. Однако
он также требует права на усечение, если уже существует файл с тем же именем.

Следует также отметить, что усечение файлов не требует права
``LANDLOCK_ACCESS_FS_WRITE_FILE``. Помимо системного вызова
:manpage:`truncate(2)`, это также можно сделать через :manpage:`open(2)` с
флагами ``O_RDONLY | O_TRUNC``.

Право на усечение связано с открытым файлом (см. ниже).

Права, связанные с файловыми дескрипторами
------------------------------------------

При открытии файла наличие прав ``LANDLOCK_ACCESS_FS_TRUNCATE`` и
``LANDLOCK_ACCESS_FS_IOCTL_DEV`` связывается с вновь созданным файловым
дескриптором и будет использоваться для последующих попыток усечения и ioctl с
помощью :manpage:`ftruncate(2)` и :manpage:`ioctl(2)`. Это поведение аналогично
открытию файла для чтения или записи, когда разрешения проверяются во время
:manpage:`open(2)`, но не во время последующих вызовов :manpage:`read(2)` и
:manpage:`write(2)`.

Как следствие, возможно, что процесс имеет несколько открытых файловых
дескрипторов, ссылающихся на один и тот же файл, но Landlock применяет к ним
разные ограничения при работе с этими файловыми дескрипторами. Это может
произойти, когда применяется набор правил Landlock, а процесс сохраняет файловые
дескрипторы, открытые как до, так и после применения. Также возможно передавать
такие файловые дескрипторы между процессами с сохранением их свойств Landlock,
даже когда у некоторых из задействованных процессов нет применённого набора
правил Landlock.

Совместимость
=============

Обратная и прямая совместимость
-------------------------------

Landlock спроектирован так, чтобы быть совместимым с прошлыми и будущими версиями
ядра. Это достигается благодаря атрибутам системных вызовов и связанным с ними
битовым флагам, в частности ``handled_access_fs`` набора правил. Явное указание
обрабатываемых прав доступа позволяет ядру и пространству пользователя иметь чёткий
контракт друг с другом. Это требуется для того, чтобы гарантировать, что
помещение в песочницу не станет более строгим при обновлении системы, что могло
бы нарушить работу приложений.

Разработчики могут подписаться на `список рассылки Landlock
<https://subspace.kernel.org/lists.linux.dev.html>`_, чтобы осознанно обновлять и
тестировать свои приложения с использованием новейших доступных возможностей. В
интересах пользователей и поскольку они могут использовать разные версии ядра,
настоятельно рекомендуется придерживаться подхода «максимально возможной защиты»
(best-effort), проверяя версию ABI Landlock во время выполнения и применяя только
поддерживаемые возможности.

.. _landlock_abi_versions:

Версии ABI Landlock
-------------------

Версию ABI Landlock можно прочитать с помощью системного вызова
sys_landlock_create_ruleset():

.. code-block:: c

    int abi;

    abi = landlock_create_ruleset(NULL, 0, LANDLOCK_CREATE_RULESET_VERSION);
    if (abi < 0) {
        switch (errno) {
        case ENOSYS:
            printf("Landlock is not supported by the current kernel.\n");
            break;
        case EOPNOTSUPP:
            printf("Landlock is currently disabled.\n");
            break;
        }
        return 0;
    }
    if (abi >= 2) {
        printf("Landlock supports LANDLOCK_ACCESS_FS_REFER.\n");
    }

Все интерфейсы ядра Landlock поддерживаются начиная с первой версии ABI, если
иное явно не отмечено в их документации.

Errata Landlock
---------------

В дополнение к версиям ABI Landlock предоставляет механизм errata для
отслеживания исправлений проблем, которые могут влиять на обратную совместимость
или требовать осведомлённости со стороны пространства пользователя. Битовую
маску errata можно запросить с помощью:

.. code-block:: c

    int errata;

    errata = landlock_create_ruleset(NULL, 0, LANDLOCK_CREATE_RULESET_ERRATA);
    if (errata < 0) {
        /* Landlock not available or disabled */
        return 0;
    }

Возвращаемое значение представляет собой битовую маску, где каждый бит
соответствует определённому erratum. Если бит N установлен
(``errata & (1 << (N - 1))``), то erratum N был исправлен в работающем ядре.

.. warning::

   **Большинству приложений НЕ следует проверять errata.** В 99,9 % случаев
   проверка errata излишня, увеличивает сложность кода и может потенциально
   снизить уровень защиты при неправильном использовании. Например, отключение
   песочницы, когда erratum не исправлен, может оставить систему менее
   защищённой, чем при использовании защиты Landlock по принципу «максимально
   возможной защиты». В случае сомнений игнорируйте errata.

.. kernel-doc:: security/landlock/errata/abi-4.h
    :doc: erratum_1

.. kernel-doc:: security/landlock/errata/abi-6.h
    :doc: erratum_2

.. kernel-doc:: security/landlock/errata/abi-1.h
    :doc: erratum_3

Как проверять errata
~~~~~~~~~~~~~~~~~~~~~

Если вы определили, что вашему приложению необходимо проверять конкретные errata,
используйте следующий шаблон:

.. code-block:: c

    int errata = landlock_create_ruleset(NULL, 0, LANDLOCK_CREATE_RULESET_ERRATA);
    if (errata >= 0) {
        /* Check for specific erratum (1-indexed) */
        if (errata & (1 << (erratum_number - 1))) {
            /* Erratum N is fixed in this kernel */
        } else {
            /* Erratum N is NOT fixed - consider implications for your use case */
        }
    }

**Важно:** проверяйте errata только если ваше приложение специально полагается на
поведение, изменившееся в результате исправления. Исправления, как правило,
делают Landlock менее ограничивающим или более корректным, а не более
ограничивающим.

Текущие ограничения
===================

Изменение топологии файловой системы
------------------------------------

Потоки, помещённые в песочницу с ограничениями файловой системы, не могут изменять
топологию файловой системы — ни через :manpage:`mount(2)`, ни через
:manpage:`pivot_root(2)`. Однако вызовы :manpage:`chroot(2)` не запрещаются.

Специальные файловые системы
----------------------------

Доступ к обычным файлам и каталогам может быть ограничен Landlock в соответствии с
обрабатываемыми видами доступа набора правил. Однако файлы, которые не относятся к
видимой пользователю файловой системе (например, pipe, socket), но всё же могут
быть доступны через ``/proc/<pid>/fd/*``, в настоящее время нельзя явно ограничить.
Аналогично, некоторые специальные файловые системы ядра, такие как nsfs, доступ к
которым возможен через ``/proc/<pid>/ns/*``, в настоящее время нельзя явно
ограничить. Однако благодаря `Ограничения ptrace`_ доступ к таким
чувствительным файлам ``/proc`` автоматически ограничивается в соответствии с
иерархиями доменов. Будущее развитие Landlock всё же может позволить явно
ограничивать такие пути с помощью специальных флагов набора правил.

Уровни наборов правил
---------------------

Существует ограничение в 16 уровней наложенных друг на друга наборов правил. Это
может быть проблемой для задачи, желающей применить новый набор правил в дополнение
к своим 16 унаследованным наборам правил. Как только этот предел достигнут,
sys_landlock_restrict_self() возвращает E2BIG. Поэтому настоятельно рекомендуется
аккуратно выстраивать наборы правил один раз за время жизни потока, особенно для
приложений, способных запускать другие приложения, которые также могут захотеть
поместить себя в песочницу (например, оболочки, менеджеры контейнеров и т.д.).

Использование памяти
--------------------

Память ядра, выделяемая для создания наборов правил, учитывается и может быть
ограничена с помощью Documentation/admin-guide/cgroup-v1/memory.rst.

Поддержка IOCTL
---------------

Право ``LANDLOCK_ACCESS_FS_IOCTL_DEV`` ограничивает использование
:manpage:`ioctl(2)`, но применяется только к *вновь открытым* файлам устройств.
Это, в частности, означает, что уже существующие файловые дескрипторы, такие как
stdin, stdout и stderr, не затрагиваются.

Пользователям следует знать, что устройства TTY традиционно позволяли управлять
другими процессами на том же TTY с помощью команд IOCTL ``TIOCSTI`` и
``TIOCLINUX``. Обе они требуют ``CAP_SYS_ADMIN`` в современных системах Linux, но
для ``TIOCSTI`` это поведение конфигурируемо.

Поэтому в более старых системах рекомендуется закрывать унаследованные файловые
дескрипторы TTY или, по возможности, повторно открывать их из
``/proc/self/fd/*`` без права ``LANDLOCK_ACCESS_FS_IOCTL_DEV``.

Поддержка IOCTL в Landlock на данный момент крупнозернистая (coarse-grained), но
в будущем может стать более мелкозернистой. До тех пор пользователям советуется
устанавливать необходимые им гарантии через иерархию файлов, разрешая право
``LANDLOCK_ACCESS_FS_IOCTL_DEV`` только на тех файлах, где это действительно
требуется.

Прежние ограничения
===================

Переименование и создание ссылок на файлы (ABI < 2)
---------------------------------------------------

Поскольку Landlock нацелен на непривилегированные механизмы управления доступом,
он должен корректно обрабатывать композицию правил. Такое свойство также
подразумевает вложенность правил. Корректная обработка нескольких уровней наборов
правил, каждый из которых способен ограничивать доступ к файлам, также
подразумевает наследование ограничений набора правил от родителя к его иерархии.
Поскольку файлы идентифицируются и ограничиваются своей иерархией, перемещение или
создание ссылки на файл из одного каталога в другой подразумевает распространение
ограничений иерархии или ограничение этих действий в соответствии с потенциально
утраченными ограничениями. Чтобы защититься от повышения привилегий через
переименование или создание ссылок, а также для простоты, Landlock ранее
ограничивал создание ссылок и переименование пределами одного каталога. Начиная с
версии ABI 2 Landlock теперь возможно безопасно управлять переименованием и
созданием ссылок благодаря новому праву доступа ``LANDLOCK_ACCESS_FS_REFER``.

Усечение файлов (ABI < 3)
-------------------------

Усечение файлов нельзя было запретить до третьей версии ABI Landlock, поэтому оно
всегда разрешено при использовании ядра, поддерживающего только первую или вторую
версию ABI.

Начиная с версии ABI 3 Landlock теперь возможно безопасно управлять усечением
благодаря новому праву доступа ``LANDLOCK_ACCESS_FS_TRUNCATE``.

TCP bind и connect (ABI < 4)
----------------------------

Начиная с версии ABI 4 Landlock теперь возможно ограничивать действия TCP bind и
connect только набором разрешённых портов благодаря новым правам доступа
``LANDLOCK_ACCESS_NET_BIND_TCP`` и ``LANDLOCK_ACCESS_NET_CONNECT_TCP``.

IOCTL для устройств (ABI < 5)
-----------------------------

Операции IOCTL нельзя было запретить до пятой версии ABI Landlock, поэтому
:manpage:`ioctl(2)` всегда разрешён при использовании ядра, поддерживающего только
более раннюю версию ABI.

Начиная с версии ABI 5 Landlock возможно ограничивать использование
:manpage:`ioctl(2)` на символьных и блочных устройствах с помощью нового права
``LANDLOCK_ACCESS_FS_IOCTL_DEV``.

Абстрактный UNIX-сокет (ABI < 6)
--------------------------------

Начиная с версии ABI 6 Landlock возможно ограничивать соединения с абстрактным
сокетом :manpage:`unix(7)`, устанавливая ``LANDLOCK_SCOPE_ABSTRACT_UNIX_SOCKET``
в атрибут набора правил ``scoped``.

Сигналы (ABI < 6)
-----------------

Начиная с версии ABI 6 Landlock возможно ограничивать отправку
:manpage:`signal(7)`, устанавливая ``LANDLOCK_SCOPE_SIGNAL`` в атрибут набора
правил ``scoped``.

Журналирование (ABI < 7)
------------------------

Начиная с версии ABI 7 Landlock возможно управлять журналированием событий аудита
Landlock с помощью флагов ``LANDLOCK_RESTRICT_SELF_LOG_SAME_EXEC_OFF``,
``LANDLOCK_RESTRICT_SELF_LOG_NEW_EXEC_ON`` и
``LANDLOCK_RESTRICT_SELF_LOG_SUBDOMAINS_OFF``, передаваемых в
sys_landlock_restrict_self(). Более подробно об аудите см.
Documentation/admin-guide/LSM/landlock.rst.

Синхронизация потоков (ABI < 8)
-------------------------------

Начиная с версии ABI 8 Landlock теперь возможно применять наборы правил Landlock
ко всем потокам вызывающего процесса с помощью флага
``LANDLOCK_RESTRICT_SELF_TSYNC``, передаваемого в
sys_landlock_restrict_self().

Поддержка ядром
===============

Конфигурация на этапе сборки
----------------------------

Landlock был впервые представлен в Linux 5.13, но он должен быть сконфигурирован
на этапе сборки с помощью ``CONFIG_SECURITY_LANDLOCK=y``. Landlock также должен
быть включён на этапе загрузки, как и другие модули безопасности. Список модулей
безопасности, включённых по умолчанию, задаётся с помощью ``CONFIG_LSM``. Тогда
конфигурация ядра должна содержать ``CONFIG_LSM=landlock,[...]``, где ``[...]`` —
список других потенциально полезных для работающей системы модулей безопасности
(см. справку по ``CONFIG_LSM``).

Конфигурация на этапе загрузки
------------------------------

Если в работающем ядре нет ``landlock`` в ``CONFIG_LSM``, то мы можем включить
Landlock, добавив ``lsm=landlock,[...]`` в
Documentation/admin-guide/kernel-parameters.rst в конфигурации загрузчика.

Например, если текущая встроенная конфигурация такова:

.. code-block:: console

    $ zgrep -h "^CONFIG_LSM=" "/boot/config-$(uname -r)" /proc/config.gz 2>/dev/null
    CONFIG_LSM="lockdown,yama,integrity,apparmor"

...и если cmdline также не содержит ``landlock``:

.. code-block:: console

    $ sed -n 's/.*\(\<lsm=\S\+\).*/\1/p' /proc/cmdline
    lsm=lockdown,yama,integrity,apparmor

...то нам следует сконфигурировать загрузчик так, чтобы задать cmdline,
расширяющий список ``lsm`` префиксом ``landlock,``::

  lsm=landlock,lockdown,yama,integrity,apparmor

После перезагрузки мы можем проверить, что Landlock запущен и работает, посмотрев
в журналы ядра:

.. code-block:: console

    # dmesg | grep landlock || journalctl -kb -g landlock
    [    0.000000] Command line: [...] lsm=landlock,lockdown,yama,integrity,apparmor
    [    0.000000] Kernel command line: [...] lsm=landlock,lockdown,yama,integrity,apparmor
    [    0.000000] LSM: initializing lsm=lockdown,capability,landlock,yama,integrity,apparmor
    [    0.000000] landlock: Up and running.

Ядро может быть сконфигурировано на этапе сборки так, чтобы всегда загружать LSM
``lockdown`` и ``capability``. В этом случае эти LSM также появятся в начале строки
журнала ``LSM: initializing``, даже если они не сконфигурированы в загрузчике.

Поддержка сети
--------------

Чтобы иметь возможность явно разрешать TCP-операции (например, добавляя сетевое
правило с ``LANDLOCK_ACCESS_NET_BIND_TCP``), ядро должно поддерживать TCP
(``CONFIG_INET=y``). В противном случае sys_landlock_add_rule() возвращает ошибку
``EAFNOSUPPORT``, которую можно безопасно игнорировать, поскольку такого рода
TCP-операция и так невозможна.

Вопросы и ответы
================

Как насчёт менеджеров песочниц в пространстве пользователя?
-----------------------------------------------------------

Использование процессов пространства пользователя для применения ограничений на
ресурсы ядра может приводить к состояниям гонки (race condition) или
несогласованным оценкам (т.е. `Incorrect mirroring of
the OS code and state
<https://www.ndss-symposium.org/ndss2003/traps-and-pitfalls-practical-problems-system-call-interposition-based-security-tools/>`_).

Как насчёт пространств имён и контейнеров?
------------------------------------------

Пространства имён могут помочь в создании песочниц, но они не предназначены для
управления доступом и потому лишены полезных для такого сценария использования
возможностей (например, нет мелкозернистых ограничений). Более того, их сложность
может приводить к проблемам с безопасностью, особенно когда ими могут
манипулировать недоверенные процессы (см. `Controlling access to user namespaces
<https://lwn.net/Articles/673597/>`_).

Как отключить записи аудита Landlock?
-------------------------------------

Возможно, вы захотите настроить фильтры, как описано здесь:
Documentation/admin-guide/LSM/landlock.rst
