Интерфейс GPIO Sysfs для пространства пользователя
==================================================

.. warning::
   Этот API устарел и заменён на chardev.rst, а документация ABI перенесена
   в Documentation/ABI/obsolete/sysfs-gpio.

   В новых разработках следует использовать chardev.rst, а уже существующие
   разработки рекомендуется по возможности скорее перевести на новый API,
   поскольку в будущем этот API будет удалён.

   Данный интерфейс будет продолжать поддерживаться в течение переходного
   периода, но новые возможности будут добавляться только в новый API.

Устаревший ABI sysfs
--------------------
Платформы, использующие фреймворк реализации «gpiolib», могут настроить
пользовательский интерфейс sysfs к GPIO. Он отличается от интерфейса debugfs
тем, что предоставляет управление направлением и значением GPIO, а не просто
показывает сводку состояния gpio. Кроме того, он может присутствовать на
рабочих системах без поддержки отладки.

При наличии соответствующей документации на оборудование системы пространство
пользователя может, например, узнать, что GPIO #23 управляет линией защиты от
записи, используемой для защиты сегментов загрузчика во флеш-памяти. Процедуры
обновления системы могут требовать временного снятия этой защиты: сначала
импортировать GPIO, затем изменить его состояние выхода, затем обновить код и
снова включить защиту от записи. При обычном использовании GPIO #23 никогда не
затрагивался бы, и ядру не было бы нужды о нём знать.

Опять же, в зависимости от наличия соответствующей документации на оборудование,
на некоторых системах GPIO из пространства пользователя может использоваться для
определения данных конфигурации системы, о которых стандартные ядра ничего не
знают. А для некоторых задач простые драйверы GPIO в пространстве пользователя
могут оказаться всем, что в действительности нужно системе.

.. note::
   НЕ злоупотребляйте sysfs для управления оборудованием, для которого есть
   надлежащие драйверы ядра. Пожалуйста, прочитайте
   Documentation/driver-api/gpio/drivers-on-gpio.rst, чтобы не изобретать
   заново ядерные велосипеды в пространстве пользователя.

   Я СЕРЬЁЗНО. ПРАВДА.

Пути в Sysfs
------------
В /sys/class/gpio есть три вида записей:

   -	управляющие интерфейсы, используемые для получения контроля над GPIO из
	пространства пользователя;

   -	сами GPIO; и

   -	контроллеры GPIO (экземпляры «gpio_chip»).

Это в дополнение к стандартным файлам, включая символьную ссылку «device».

Управляющие интерфейсы доступны только для записи:

    /sys/class/gpio/

	«export» ...
		Пространство пользователя может попросить ядро экспортировать
		контроль над GPIO в пространство пользователя, записав его номер
		в этот файл.

		Пример:  «echo 19 > export» создаст узел «gpio19» для GPIO #19,
		если он не запрошен кодом ядра.

	«unexport» ...
		Отменяет эффект экспорта в пространство пользователя.

		Пример:  «echo 19 > unexport» удалит узел «gpio19»,
		экспортированный с помощью файла «export».

Сигналы GPIO имеют пути вида /sys/class/gpio/gpio42/ (для GPIO #42)
и обладают следующими атрибутами чтения/записи:

    /sys/class/gpio/gpioN/

	«direction» ...
		читается как «in» или «out». Обычно это значение можно
		записывать. Запись «out» по умолчанию инициализирует значение
		как низкое (low). Для обеспечения работы без сбоев (glitch free)
		можно записать значения «low» и «high», чтобы настроить GPIO как
		выход с соответствующим начальным значением.

		Обратите внимание, что этот атрибут *не будет существовать*, если
		ядро не поддерживает изменение направления GPIO или если он был
		экспортирован кодом ядра, который явно не разрешил пространству
		пользователя переконфигурировать направление этого GPIO.

	«value» ...
		читается как 0 (неактивно) или 1 (активно). Если GPIO настроен
		как выход, это значение можно записывать; любое ненулевое значение
		трактуется как активное.

		Если вывод может быть настроен как генерирующий прерывания и если
		он был настроен на генерацию прерываний (см. описание «edge»), вы
		можете выполнять poll(2) для этого файла, и poll(2) будет
		возвращать управление всякий раз, когда срабатывает прерывание.
		Если вы используете poll(2), установите события POLLPRI и POLLERR.
		Если вы используете select(2), установите файловый дескриптор в
		exceptfds. После возврата из poll(2) используйте pread(2) для
		чтения значения по смещению ноль. Как вариант, либо выполните
		lseek(2) к началу файла sysfs и прочитайте новое значение, либо
		закройте файл и снова откройте его для чтения значения.

	«edge» ...
		читается как «none», «rising», «falling» или «both». Запишите
		эти строки, чтобы выбрать фронт(ы) сигнала, при которых poll(2)
		для файла «value» будет возвращать управление.

		Этот файл существует только в том случае, если вывод может быть
		настроен как входной вывод, генерирующий прерывания.

	«active_low» ...
		читается как 0 (false) или 1 (true). Запишите любое ненулевое
		значение, чтобы инвертировать атрибут value как при чтении, так и
		при записи. Существующая и последующая поддержка poll(2),
		настраиваемая через атрибут edge для фронтов «rising» и «falling»,
		будет следовать этой настройке.

Контроллеры GPIO имеют пути вида /sys/class/gpio/gpiochip42/ (для контроллера,
реализующего GPIO начиная с #42) и обладают следующими атрибутами только для
чтения:

    /sys/class/gpio/gpiochipN/

	«base» ...
		то же, что и N, первый GPIO, управляемый этим чипом

	«label» ...
		предоставляется для диагностики (не всегда уникален)

	«ngpio» ...
		сколько GPIO он управляет (от N до N + ngpio - 1)

Документация на плату должна в большинстве случаев описывать, какие GPIO для
каких целей используются. Однако эти номера не всегда стабильны; GPIO на дочерней
плате могут отличаться в зависимости от используемой базовой платы или других
плат в стеке. В таких случаях вам может потребоваться использовать узлы gpiochip
(возможно, в сочетании со схемами), чтобы определить правильный номер GPIO для
заданного сигнала.


Экспорт из кода ядра
--------------------
Код ядра может явно управлять экспортом GPIO, которые уже были запрошены с
помощью gpio_request()::

	/* export the GPIO to userspace */
	int gpiod_export(struct gpio_desc *desc, bool direction_may_change);

	/* reverse gpiod_export() */
	void gpiod_unexport(struct gpio_desc *desc);

	/* create a sysfs link to an exported GPIO node */
	int gpiod_export_link(struct device *dev, const char *name,
		      struct gpio_desc *desc);

После того как драйвер ядра запросил GPIO, он может быть сделан доступным в
интерфейсе sysfs только через gpiod_export(). Драйвер может управлять тем, может
ли меняться направление сигнала. Это помогает драйверам предотвращать случайное
искажение пользовательским кодом важного состояния системы.

Такой явный экспорт может помочь при отладке (упрощая некоторые виды
экспериментов) или может предоставить всегда присутствующий интерфейс, который
подходит для документирования в составе пакета поддержки платы.

После того как GPIO был экспортирован, gpiod_export_link() позволяет создавать
символьные ссылки из других мест sysfs на узел GPIO в sysfs. Драйверы могут
использовать это, чтобы предоставить интерфейс под своим собственным устройством
в sysfs с описательным именем.
