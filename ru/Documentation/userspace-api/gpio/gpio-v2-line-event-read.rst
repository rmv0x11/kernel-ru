***********************
GPIO_V2_LINE_EVENT_READ
***********************

Имя
===

GPIO_V2_LINE_EVENT_READ - Чтение событий обнаружения фронтов для линий из запроса.

Синопсис
========

``int read(int req_fd, void *buf, size_t count)``

Аргументы
=========

``req_fd``
    Файловый дескриптор символьного устройства GPIO, возвращаемый в
    :c:type:`request.fd<gpio_v2_line_request>` посредством gpio-v2-get-line-ioctl.rst.

``buf``
    Буфер для хранения :c:type:`событий<gpio_v2_line_event>`.

``count``
    Число доступных байтов в ``buf``, которое должно быть не меньше
    размера :c:type:`gpio_v2_line_event`.

Описание
========

Чтение событий обнаружения фронтов для линий из запроса.

Обнаружение фронтов должно быть включено для входной линии с помощью
``GPIO_V2_LINE_FLAG_EDGE_RISING`` или ``GPIO_V2_LINE_FLAG_EDGE_FALLING``, либо
обоих флагов сразу. После этого события фронтов генерируются всякий раз, когда на
входной линии обнаруживаются прерывания по фронту.

Фронты определяются в терминах изменений логического значения линии, так что
переход из неактивного состояния в активное является нарастающим фронтом. Если
установлен ``GPIO_V2_LINE_FLAG_ACTIVE_LOW``, то логическая полярность
противоположна физической полярности, и тогда ``GPIO_V2_LINE_FLAG_EDGE_RISING``
соответствует спадающему физическому фронту.

Ядро захватывает события фронтов и проставляет им метки времени как можно ближе к
моменту их возникновения и сохраняет их в буфере, откуда они могут быть прочитаны
пространством пользователя в удобный для него момент с помощью `read()`.

События, прочитанные из буфера, всегда находятся в том же порядке, в котором они
были обнаружены ядром, в том числе когда одним запросом отслеживается несколько
линий.

Размер буфера событий ядра фиксируется в момент создания запроса линии и может
быть задан через
:c:type:`request.event_buffer_size<gpio_v2_line_request>`.
Размер по умолчанию в 16 раз превышает число запрошенных линий.

Буфер может переполниться, если всплески событий возникают быстрее, чем они
читаются пространством пользователя. Если происходит переполнение, то самое
старое буферизированное событие отбрасывается. Переполнение может быть обнаружено
из пространства пользователя путём отслеживания порядковых номеров событий.

Чтобы минимизировать число вызовов, требуемых для копирования событий из ядра в
пространство пользователя, `read()` поддерживает копирование нескольких событий.
Число копируемых событий равно наименьшему из числа доступных в буфере ядра и
числа, которое поместится в буфере пространства пользователя (``buf``).

Изменение флагов обнаружения фронтов с помощью gpio-v2-line-set-config-ioctl.rst
не удаляет и не изменяет события, уже содержащиеся в буфере событий ядра.

`read()` блокируется, если ни одно событие не доступно и для ``req_fd`` не был
установлен флаг **O_NONBLOCK**.

Наличие события можно проверить, проверив, доступен ли ``req_fd`` для чтения, с
помощью `poll()` или эквивалентного механизма.

Возвращаемое значение
=====================

В случае успеха — число прочитанных байтов, которое будет кратно размеру события
:c:type:`gpio_v2_line_event`.

В случае ошибки — -1, и переменная ``errno`` устанавливается соответствующим
образом. Распространённые коды ошибок описаны в error-codes.rst.
