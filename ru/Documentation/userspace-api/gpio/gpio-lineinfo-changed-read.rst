.. _GPIO_LINEINFO_CHANGED_READ:

**************************
GPIO_LINEINFO_CHANGED_READ
**************************

.. warning::
    Этот ioctl является частью chardev_v1.rst и считается устаревшим в пользу
    gpio-v2-lineinfo-changed-read.rst.

Имя
===

GPIO_LINEINFO_CHANGED_READ - Чтение из микросхемы событий изменения информации
о линии для отслеживаемых линий.

Краткое описание
================

``int read(int chip_fd, void *buf, size_t count)``

Аргументы
=========

``chip_fd``
    Файловый дескриптор символьного устройства GPIO, возвращаемый `open()`.

``buf``
    Буфер, содержащий :c:type:`события<gpioline_info_changed>`.

``count``
    Количество байт, доступных в ``buf``; оно должно быть не меньше размера
    события :c:type:`gpioline_info_changed`.

Описание
========

Чтение из микросхемы событий изменения информации о линии для отслеживаемых линий.

.. note::
    Отслеживание изменений информации о линии обычно не требуется и, как правило,
    выполняется только компонентом мониторинга системы.

    Эти события относятся к изменениям состояния запроса или конфигурации линии,
    а не её значения. Используйте gpio-lineevent-data-read.rst для получения
    событий при изменении значения линии.

Чтобы генерировались события изменения информации, линия должна отслеживаться
с помощью gpio-get-lineinfo-watch-ioctl.rst.  В дальнейшем запрос, освобождение
или переконфигурирование линии будет приводить к генерации события изменения
информации.

Ядро проставляет события временными метками в момент их возникновения и сохраняет
их в буфере, откуда они могут быть прочитаны пространством пользователя в удобный
момент с помощью `read()`.

Размер буфера событий ядра фиксирован и составляет 32 события на каждый ``chip_fd``.

Буфер может переполниться, если всплески событий происходят быстрее, чем они
читаются пространством пользователя. При возникновении переполнения наиболее
свежее событие отбрасывается. Переполнение невозможно обнаружить из пространства
пользователя.

События, прочитанные из буфера, всегда идут в том же порядке, в котором они были
обнаружены ядром, в том числе когда через один ``chip_fd`` отслеживается несколько
линий.

Чтобы минимизировать количество вызовов, необходимых для копирования событий из
ядра в пространство пользователя, `read()` поддерживает копирование нескольких
событий. Количество копируемых событий равно меньшему из двух значений: числа
событий, доступных в буфере ядра, и числа событий, которые поместятся в буфер
пространства пользователя (``buf``).

Вызов `read()` будет блокироваться, если ни одно событие недоступно и для
``chip_fd`` не установлен флаг **O_NONBLOCK**.

Наличие события можно проверить, проверив, что ``chip_fd`` доступен для чтения,
с помощью `poll()` или эквивалентного механизма.

Впервые добавлено в 5.7.

Возвращаемое значение
=====================

В случае успеха — количество прочитанных байт, которое будет кратно размеру
события :c:type:`gpioline_info_changed`.

В случае ошибки — -1, а переменной ``errno`` присваивается соответствующее значение.
Распространённые коды ошибок описаны в error-codes.rst.
