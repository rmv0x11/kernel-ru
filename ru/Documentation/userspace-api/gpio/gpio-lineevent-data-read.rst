************************
GPIO_LINEEVENT_DATA_READ
************************

.. warning::
    Этот ioctl является частью chardev_v1.rst и считается устаревшим в пользу
    gpio-v2-line-event-read.rst.

Имя
===

GPIO_LINEEVENT_DATA_READ - чтение событий обнаружения фронтов из события линии.

Синопсис
========

``int read(int event_fd, void *buf, size_t count)``

Аргументы
=========

``event_fd``
    Файловый дескриптор символьного устройства GPIO, возвращаемый в
    :c:type:`request.fd<gpioevent_request>` функцией gpio-get-lineevent-ioctl.rst.

``buf``
    Буфер для хранения :c:type:`событий<gpioevent_data>`.

``count``
    Количество байт, доступных в ``buf``; оно должно быть не менее размера
    :c:type:`gpioevent_data`.

Описание
========

Чтение событий обнаружения фронтов для линии из события линии.

Для входной линии должно быть включено обнаружение фронтов с помощью либо
``GPIOEVENT_REQUEST_RISING_EDGE``, либо ``GPIOEVENT_REQUEST_FALLING_EDGE``,
либо обоих. После этого события фронтов генерируются всякий раз, когда на
входной линии обнаруживаются прерывания по фронту.

Фронты определяются в терминах изменений логического значения линии, поэтому
переход из неактивного состояния в активное является нарастающим фронтом. Если
установлен ``GPIOHANDLE_REQUEST_ACTIVE_LOW``, то логическая полярность
противоположна физической, и ``GPIOEVENT_REQUEST_RISING_EDGE`` тогда
соответствует спадающему физическому фронту.

Ядро захватывает и проставляет временные метки событиям фронтов как можно ближе
к моменту их возникновения и сохраняет их в буфере, из которого они могут быть
прочитаны пространством пользователя в удобный для него момент с помощью `read()`.

Источником тактового сигнала для :c:type:`event.timestamp<gpioevent_data>`
является ``CLOCK_MONOTONIC``, за исключением ядер старше Linux 5.7, когда
использовался ``CLOCK_REALTIME``. В :c:type:`gpioevent_data` нет указания на то,
какой источник тактового сигнала используется, его необходимо определять либо по
версии ядра, либо по проверкам разумности самой временной метки.

События, прочитанные из буфера, всегда следуют в том же порядке, в каком они
были обнаружены ядром.

Размер буфера событий ядра фиксирован и составляет 16 событий.

Буфер может переполниться, если всплески событий происходят быстрее, чем они
читаются пространством пользователя. Если происходит переполнение, то самое
недавнее событие отбрасывается. Переполнение не может быть обнаружено из
пространства пользователя.

Чтобы минимизировать количество вызовов, необходимых для копирования событий из
ядра в пространство пользователя, `read()` поддерживает копирование нескольких
событий. Количество скопированных событий равно меньшему из числа доступных в
буфере ядра и числа, которое поместится в буфере пространства пользователя
(``buf``).

`read()` будет блокироваться, если ни одно событие не доступно и для ``event_fd``
не установлен флаг **O_NONBLOCK**.

Наличие события можно проверить, проверив, что ``event_fd`` доступен для чтения,
с помощью `poll()` или эквивалента.

Возвращаемое значение
=====================

В случае успеха — количество прочитанных байт, которое будет кратно размеру
события :c:type:`gpio_lineevent_data`.

В случае ошибки — -1, а переменной ``errno`` присваивается соответствующее
значение. Распространённые коды ошибок описаны в error-codes.rst.
