**********************
GPIO_V2_GET_LINE_IOCTL
**********************

Имя
===

GPIO_V2_GET_LINE_IOCTL - Запрос линии или линий у ядра.

Синопсис
========

.. c:macro:: GPIO_V2_GET_LINE_IOCTL

``int ioctl(int chip_fd, GPIO_V2_GET_LINE_IOCTL, struct gpio_v2_line_request *request)``

Аргументы
=========

``chip_fd``
    Файловый дескриптор символьного устройства GPIO, возвращённый `open()`.

``request``
    :c:type:`line_request<gpio_v2_line_request>`, задающий запрашиваемые линии
    и их конфигурацию.

Описание
========

В случае успеха запрашивающий процесс получает эксклюзивный доступ к значению
линии, доступ на запись к конфигурации линии и может получать события при
обнаружении фронтов на линии; всё это подробнее описано в
:ref:`gpio-v2-line-request`.

В одном запросе линий может быть запрошено несколько линий, и операции запроса
выполняются ядром над запрошенными линиями настолько атомарно, насколько это
возможно. Например, gpio-v2-line-get-values-ioctl.rst прочитает все запрошенные
линии за один раз.

Состояние линии, включая значение выходных линий, гарантированно остаётся таким,
как было запрошено, до тех пор пока не будет закрыт возвращённый файловый
дескриптор. После закрытия файлового дескриптора состояние линии становится
неконтролируемым с точки зрения пространства пользователя и может вернуться к
своему состоянию по умолчанию.

Запрос линии, уже находящейся в использовании, является ошибкой (**EBUSY**).

Закрытие ``chip_fd`` не влияет на существующие запросы линий.

.. _gpio-v2-get-line-config-rules:

Правила конфигурации
--------------------

Для любой заданной запрошенной линии применяются следующие правила конфигурации:

Флаги направления, ``GPIO_V2_LINE_FLAG_INPUT`` и
``GPIO_V2_LINE_FLAG_OUTPUT``, нельзя комбинировать. Если не установлен ни один из
них, то единственным другим флагом, который может быть установлен, является
``GPIO_V2_LINE_FLAG_ACTIVE_LOW``, и линия запрашивается «как есть», чтобы
позволить чтение значения линии без изменения её электрической конфигурации.

Флаги управления выходом, ``GPIO_V2_LINE_FLAG_OPEN_xxx``, требуют установки
``GPIO_V2_LINE_FLAG_OUTPUT``.
Может быть установлен только один флаг управления выходом.
Если не установлен ни один, то линия считается двухтактной (push-pull).

Может быть установлен только один флаг смещения, ``GPIO_V2_LINE_FLAG_BIAS_xxx``,
и он требует, чтобы был также установлен флаг направления.
Если не установлено ни одного флага смещения, то конфигурация смещения не изменяется.

Флаги фронта, ``GPIO_V2_LINE_FLAG_EDGE_xxx``, требуют установки
``GPIO_V2_LINE_FLAG_INPUT`` и могут комбинироваться для обнаружения как нарастающих,
так и спадающих фронтов.  Запрос обнаружения фронтов у линии, которая его не
поддерживает, является ошибкой (**ENXIO**).

Может быть установлен только один флаг тактовых импульсов событий,
``GPIO_V2_LINE_FLAG_EVENT_CLOCK_xxx``.
Если не установлено ни одного, то для тактовых импульсов событий по умолчанию
используется ``CLOCK_MONOTONIC``.
Флаг ``GPIO_V2_LINE_FLAG_EVENT_CLOCK_HTE`` требует поддерживающего оборудования
и ядра с установленным ``CONFIG_HTE``.  Запрос HTE у устройства, которое его
не поддерживает, является ошибкой (**EOPNOTSUPP**).

Атрибут :c:type:`debounce_period_us<gpio_v2_line_attribute>` может применяться
только к линиям с установленным ``GPIO_V2_LINE_FLAG_INPUT``. Когда он установлен,
устранение дребезга применяется как к значениям, возвращаемым
gpio-v2-line-get-values-ioctl.rst, так и к фронтам, возвращаемым
gpio-v2-line-event-read.rst.  Если оно не поддерживается напрямую оборудованием,
устранение дребезга эмулируется ядром программно.  Запрос устранения дребезга у
линии, которая не поддерживает ни устранение дребезга в оборудовании, ни
прерывания, необходимые для программной эмуляции, является ошибкой
(**ENXIO**).

Запрос недопустимой конфигурации является ошибкой (**EINVAL**).

.. _gpio-v2-get-line-config-support:

Поддержка конфигурации
----------------------

Там, где запрошенная конфигурация не поддерживается напрямую нижележащим
оборудованием и драйвером, ядро применяет один из следующих подходов:

 - отклонить запрос
 - эмулировать функциональность программно
 - трактовать функциональность по принципу наилучшего усилия (best effort)

Применяемый подход зависит от того, может ли функциональность быть разумно
эмулирована программно, и от влияния на оборудование и пространство пользователя,
если функциональность не поддерживается.
Подход, применяемый для каждой функциональности, следующий:

==============   ===========
Feature          Approach
==============   ===========
Bias             best effort
Debounce         emulate
Direction        reject
Drive            emulate
Edge Detection   reject
==============   ===========

Смещение трактуется по принципу наилучшего усилия, чтобы позволить пространству
пользователя применять одинаковую конфигурацию как для платформ, поддерживающих
внутреннее смещение, так и для тех, что требуют внешнего смещения.
В худшем случае линия оказывается в плавающем состоянии вместо ожидаемого смещения.

Устранение дребезга эмулируется применением фильтра к аппаратным прерываниям на
линии.
Событие фронта генерируется после того, как фронт обнаружен и линия остаётся
стабильной в течение периода устранения дребезга.
Временная метка события соответствует концу периода устранения дребезга.

Управление выходом эмулируется переключением линии на вход, когда линия не должна
активно управляться.

Обнаружение фронтов требует поддержки прерываний и отклоняется, если она не
поддерживается. Эмуляция путём опроса (polling) всё же может выполняться из
пространства пользователя.

Во всех случаях конфигурация, сообщаемая gpio-v2-get-lineinfo-ioctl.rst,
является запрошенной конфигурацией, а не результирующей аппаратной конфигурацией.
Пространство пользователя не может определить, поддерживается ли функциональность
аппаратно, эмулируется ли она или трактуется по принципу наилучшего усилия.

Возвращаемое значение
=====================

В случае успеха — 0, и :c:type:`request.fd<gpio_v2_line_request>` содержит
файловый дескриптор для запроса.

В случае ошибки — -1, и переменной ``errno`` присваивается соответствующее
значение.
Распространённые коды ошибок описаны в error-codes.rst.
