==========================================
Выделение dma-buf с помощью куч (heaps)
==========================================

Dma-buf Heaps — это механизм, позволяющий пространству пользователя (userspace)
выделять объекты dma-buf. Обычно они используются для выделения буферов из
определённого пула выделения или для совместного использования буферов разными
фреймворками.

Кучи (heaps)
============

Куча представляет конкретный аллокатор. В настоящее время ядро Linux поддерживает
следующие кучи:

 - Куча ``system`` выделяет виртуально непрерывные, кэшируемые буферы.

 - Куча ``default_cma_region`` выделяет физически непрерывные, кэшируемые
   буферы. Присутствует только при наличии области CMA. Обычно такая
   область создаётся либо через командную строку ядра с помощью
   параметра ``cma``, либо через узел Device-Tree, описывающий область
   памяти, с установленным свойством ``linux,cma-default``, либо через
   Kconfig-параметры ``CMA_SIZE_MBYTES`` или ``CMA_SIZE_PERCENTAGE``. До
   Linux 6.17 её имя не было стабильным и в зависимости от платформы
   могло называться ``reserved``, ``linux,cma`` или ``default-pool``.

 - Для каждой повторно используемой области в дереве устройств (device
   tree) с признаком совместимости ``shared-dma-pool`` создаётся куча, в
   качестве имени которой используется полное имя узла дерева устройств.
   Семантика буферов идентична ``default-cma-region``.

Соглашение об именовании
========================

Имя кучи ``dma-buf`` должно удовлетворять ряду ограничений:

- Имя должно быть стабильным и не должно меняться от одной версии к другой.
  Пространство пользователя идентифицирует кучи по их именам, поэтому, если
  имена изменятся, мы, скорее всего, внесём регрессии.

- Имя должно описывать область памяти, из которой куча будет выделять память, и
  должно однозначно идентифицировать её в пределах данной платформы. Поскольку
  приложения пространства пользователя используют имя кучи в качестве
  различительного признака, при наличии нескольких куч они должны иметь
  возможность надёжно определить, какую кучу они хотят использовать.

- Имя не должно упоминать детали реализации, такие как аллокатор. Драйвер кучи
  со временем будет меняться, и детали реализации на момент его появления могут
  оказаться неактуальными в будущем.

- Имя должно описывать свойства буферов, которые будут выделяться. Это упростит
  идентификацию кучи для пространства пользователя. Такими свойствами являются:

  - ``contiguous`` для физически непрерывных буферов;

  - ``protected`` для зашифрованных буферов, недоступных ОС;

- Имя может описывать предполагаемое использование. Это упростит идентификацию
  кучи для приложений пространства пользователя и пользователей.

Например, для платформы с зарезервированной областью памяти, расположенной
по адресу ОЗУ 0x42000000, предназначенной для выделения видеобуферов кадра
(framebuffers), физически непрерывной и обслуживаемой аллокатором CMA ядра,
хорошими именами были бы ``memory@42000000-contiguous`` или ``video@42000000``,
а вот ``cma-video`` — нет.
