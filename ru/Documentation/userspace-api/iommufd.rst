=======
IOMMUFD
=======

:Author: Jason Gunthorpe
:Author: Kevin Tian

Обзор
=====

IOMMUFD — это пользовательский API для управления подсистемой IOMMU в части
управления таблицами страниц ввода-вывода из пространства пользователя с
помощью файловых дескрипторов. Он задуман как универсальный и пригодный для
использования любым драйвером, который хочет предоставить DMA в пространство
пользователя. Ожидается, что со временем такие драйверы откажутся от любой
внутренней логики IOMMU, которую они, возможно, уже/исторически реализуют
(например, vfio_iommu_type1.c).

Как минимум, iommufd обеспечивает универсальную поддержку управления адресными
пространствами ввода-вывода и таблицами страниц ввода-вывода для всех IOMMU, при
этом архитектура оставляет место для добавления неуниверсальных возможностей,
обслуживающих конкретную аппаратную функциональность.

В данном контексте заглавные буквы (IOMMUFD) относятся к подсистеме, тогда как
строчные (iommufd) относятся к файловым дескрипторам, создаваемым через
/dev/iommu для использования пространством пользователя.

Ключевые понятия
================

Объекты, видимые пользователю
-----------------------------

Пространству пользователя доступны следующие объекты IOMMUFD:

- IOMMUFD_OBJ_IOAS — представляет адресное пространство ввода-вывода (IOAS) и
  позволяет отображать/снимать отображение (map/unmap) памяти пространства
  пользователя в диапазоны виртуальных адресов ввода-вывода (IOVA).

  IOAS является функциональной заменой VFIO container и, подобно VFIO container,
  копирует отображение IOVA в список iommu_domains, хранящихся внутри него.

- IOMMUFD_OBJ_DEVICE — представляет устройство, которое привязано к iommufd
  внешним драйвером.

- IOMMUFD_OBJ_HWPT_PAGING — представляет реальную аппаратную таблицу страниц
  ввода-вывода (то есть одну struct iommu_domain), управляемую драйвером iommu.
  «PAGING» в первую очередь указывает на то, что HWPT такого типа должен быть
  связан с IOAS. Это также указывает на то, что он подкреплён iommu_domain с
  флагом возможностей __IOMMU_DOMAIN_PAGING. Это может быть либо домен
  UNMANAGED stage-1 для устройства, работающего в пространстве пользователя,
  либо родительский домен stage-2 для вложенности (nesting parent), для
  отображений из физических адресов уровня гостя в физические адреса уровня
  хоста.

  У IOAS есть список HWPT_PAGING, которые совместно используют одно и то же
  отображение IOVA, и он будет синхронизировать своё отображение с каждым
  входящим в него HWPT_PAGING.

- IOMMUFD_OBJ_HWPT_NESTED — представляет реальную аппаратную таблицу страниц
  ввода-вывода (то есть одну struct iommu_domain), управляемую пространством
  пользователя (например, гостевой ОС). «NESTED» указывает на то, что HWPT
  такого типа должен быть связан с HWPT_PAGING. Это также указывает на то, что
  он подкреплён iommu_domain типа IOMMU_DOMAIN_NESTED. Это должен быть домен
  stage-1 для устройства, работающего в пространстве пользователя (например, в
  гостевой ВМ с включённой возможностью вложенной трансляции IOMMU). Поэтому он
  должен создаваться с указанием конкретного родительского домена stage-2 для
  вложенности, с которым его нужно связать. Эта вложенная таблица страниц
  stage-1, управляемая пространством пользователя, обычно содержит отображения
  из виртуальных адресов ввода-вывода уровня гостя в физические адреса уровня
  гостя.

- IOMMUFD_FAULT — представляет программную очередь для HWPT, сообщающую об
  ошибках страниц ввода-вывода с помощью PRI (Page Request Interface) аппаратуры
  IOMMU. Этот объект очереди предоставляет пространству пользователя FD для
  опроса событий ошибок страниц, а также для реагирования на эти события.
  Сначала необходимо создать объект FAULT, чтобы получить fault_id, который
  затем можно использовать для выделения HWPT с включёнными ошибками с помощью
  команды IOMMU_HWPT_ALLOC, установив бит IOMMU_HWPT_FAULT_ID_VALID в его поле
  flags.

- IOMMUFD_OBJ_VIOMMU — представляет срез физического экземпляра IOMMU,
  передаваемый ВМ или совместно используемый с ней. Это могут быть некоторые
  аппаратно-ускоренные возможности виртуализации, а также некоторые программные
  ресурсы, используемые ВМ. Например:

  * Пространство имён безопасности для идентификаторов, принадлежащих гостю,
    например управляемых гостем тегов кэша
  * Отчётность о событиях, не связанных с устройством, например об ошибках
    очереди инвалидации
  * Доступ к общей родительской таблице страниц для вложенности через несколько
    физических IOMMU
  * Виртуализация различных идентификаторов платформ, например RID и других
  * Доставка паравиртуализованной инвалидации
  * Напрямую назначаемые очереди инвалидации
  * Напрямую назначаемые прерывания

  Такой объект vIOMMU обычно имеет доступ к родительской таблице страниц для
  вложенности, чтобы поддерживать некоторые аппаратно-ускоренные возможности
  виртуализации. Поэтому объект vIOMMU должен создаваться с указанием объекта
  HWPT_PAGING — родителя для вложенности, и тогда он инкапсулирует этот объект
  HWPT_PAGING. Таким образом, объект vIOMMU можно использовать для выделения
  объекта HWPT_NESTED вместо инкапсулированного HWPT_PAGING.

  .. note::

     Название «vIOMMU» не обязательно идентично виртуализованному IOMMU в ВМ. У
     ВМ может быть один огромный виртуализованный IOMMU, работающий на машине с
     несколькими физическими IOMMU, и в этом случае VMM будет распределять
     запросы или конфигурации от этого единственного экземпляра
     виртуализованного IOMMU на несколько объектов vIOMMU, созданных для
     отдельных срезов различных физических IOMMU. Иными словами, объект vIOMMU
     всегда является представлением одного физического IOMMU, но не обязательно
     виртуализованного IOMMU. Для VMM, которым нужны все возможности
     виртуализации физических IOMMU, рекомендуется создавать столько же
     виртуализованных IOMMU, сколько имеется физических IOMMU, чтобы
     пробрасываемые устройства подключались к собственным виртуализованным
     IOMMU, подкреплённым соответствующими объектами vIOMMU; в этом случае
     гостевая ОС будет выполнять «распределение» естественным образом, без
     перехватов со стороны VMM.

- IOMMUFD_OBJ_VDEVICE — представляет виртуальное устройство для
  IOMMUFD_OBJ_DEVICE относительно IOMMUFD_OBJ_VIOMMU. Это виртуальное устройство
  содержит виртуальную информацию или атрибуты устройства (связанные с vIOMMU) в
  ВМ. Непосредственным примером vDATA может служить виртуальный идентификатор
  устройства на vIOMMU — уникальный идентификатор, который VMM назначает
  устройству для канала/порта трансляции vIOMMU, например vSID у ARM SMMUv3,
  vDeviceID у AMD IOMMU и vRID у Intel VT-d для Context Table. Через этот объект
  может также передаваться некоторая потенциальная информация о расширенной
  безопасности, например уровень безопасности или информация о realm в
  архитектуре конфиденциальных вычислений (Confidential Compute Architecture).
  VMM следует создавать объект vDEVICE для передачи всей информации об устройстве
  в ВМ при подключении устройства к vIOMMU; это отдельный вызов ioctl,
  отличающийся от присоединения того же устройства к HWPT_PAGING, которым владеет
  vIOMMU.

- IOMMUFD_OBJ_VEVENTQ — представляет программную очередь для vIOMMU,
  предназначенную для сообщения его событий, таких как ошибки трансляции,
  возникшие на вложенном уровне stage-1 (исключая ошибки страниц ввода-вывода,
  которые должны проходить через IOMMUFD_OBJ_FAULT), и аппаратно-специфичных
  событий. Этот объект очереди предоставляет пространству пользователя FD для
  опроса/чтения событий vIOMMU. Сначала необходимо создать объект vIOMMU, чтобы
  получить его viommu_id, который затем можно использовать для выделения
  vEVENTQ. Каждый vIOMMU может поддерживать несколько типов vEVENTS, но
  ограничен одной vEVENTQ на каждый тип vEVENTQ.

- IOMMUFD_OBJ_HW_QUEUE — представляет аппаратно-ускоренную очередь как
  подмножество возможностей виртуализации IOMMU, позволяющую аппаратуре IOMMU
  напрямую читать или записывать память виртуальной очереди, принадлежащую
  гостевой ОС. Эта возможность аппаратного ускорения может позволить ВМ работать
  с аппаратурой IOMMU напрямую без VM Exit, чтобы снизить накладные расходы от
  hypercall'ов. Вместе с объектом HW QUEUE iommufd предоставляет пространству
  пользователя интерфейс mmap, чтобы VMM мог отобразить (mmap) физическую область
  MMIO из физического адресного пространства хоста в физическое адресное
  пространство гостя, позволяя гостевой ОС напрямую управлять выделенной
  HW QUEUE. Таким образом, при выделении HW QUEUE VMM должен запросить пару
  значений mmap (offset/length) и передать их в точности в системный вызов mmap
  через его аргументы offset и length.

Все видимые пользователю объекты уничтожаются через uAPI IOMMU_DESTROY.

1. IOMMUFD_OBJ_IOAS создаётся через uAPI IOMMU_IOAS_ALLOC. Один iommufd может
   содержать несколько объектов IOAS. IOAS — наиболее универсальный объект, и он
   не предоставляет интерфейсов, специфичных для отдельных драйверов IOMMU. Все
   операции над IOAS должны выполняться одинаково над каждым из находящихся в
   нём iommu_domains.

2. IOMMUFD_OBJ_DEVICE создаётся, когда внешний драйвер вызывает kAPI IOMMUFD
   для привязки устройства к iommufd. Ожидается, что драйвер реализует набор
   ioctl'ов, позволяющих пространству пользователя инициировать операцию
   привязки. Успешное завершение этой операции устанавливает требуемое владение
   DMA (DMA ownership) над устройством. Драйвер также должен установить флаг
   driver_managed_dma и не должен обращаться к устройству, пока эта операция не
   завершится успешно.

3. IOMMUFD_OBJ_HWPT_PAGING может быть создан двумя способами:

   * IOMMUFD_OBJ_HWPT_PAGING создаётся автоматически, когда внешний драйвер
     вызывает kAPI IOMMUFD для присоединения привязанного устройства к IOAS.
     Аналогично, uAPI внешнего драйвера позволяет пространству пользователя
     инициировать операцию присоединения. Если в списке HWPT_PAGING данного
     IOAS существует совместимый входящий объект HWPT_PAGING, он будет
     переиспользован. В противном случае будет создан новый HWPT_PAGING,
     представляющий iommu_domain пространству пользователя, который затем будет
     добавлен в список. Успешное завершение этой операции устанавливает связи
     между IOAS, устройством и iommu_domain. После её завершения устройство
     сможет выполнять DMA.

   * IOMMUFD_OBJ_HWPT_PAGING можно создать вручную через uAPI IOMMU_HWPT_ALLOC,
     предоставив ioas_id через @pt_id для связывания нового HWPT_PAGING с
     соответствующим объектом IOAS. Преимущество такого ручного выделения в том,
     что оно позволяет задавать флаги выделения (определённые в enum
     iommufd_hwpt_alloc_flags); например, оно выделяет родительский HWPT_PAGING
     для вложенности, если установлен флаг IOMMU_HWPT_ALLOC_NEST_PARENT.

4. IOMMUFD_OBJ_HWPT_NESTED можно создать только вручную через uAPI
   IOMMU_HWPT_ALLOC, предоставив hwpt_id или viommu_id объекта vIOMMU,
   инкапсулирующего родительский HWPT_PAGING для вложенности, через @pt_id для
   связывания нового объекта HWPT_NESTED с соответствующим объектом HWPT_PAGING.
   Связываемый объект HWPT_PAGING должен быть родителем для вложенности, ранее
   выделенным вручную через тот же uAPI с флагом IOMMU_HWPT_ALLOC_NEST_PARENT,
   иначе выделение завершится неудачно. Выделение будет дополнительно проверено
   драйвером IOMMU, чтобы убедиться, что родительский домен для вложенности и
   выделяемый вложенный домен совместимы. Успешное завершение этой операции
   устанавливает связи между IOAS, устройством и iommu_domains. После её
   завершения устройство сможет выполнять DMA посредством двухэтапной (2-stage)
   трансляции, она же вложенная трансляция (nested translation). Обратите
   внимание, что один и тот же родитель для вложенности может выделить (и затем
   связать с собой) несколько объектов HWPT_NESTED.

   .. note::

      Как ручной IOMMUFD_OBJ_HWPT_PAGING, так и IOMMUFD_OBJ_HWPT_NESTED
      создаются через один и тот же uAPI IOMMU_HWPT_ALLOC. Разница состоит в
      типе объекта, передаваемого через поле @pt_id структуры
      iommufd_hwpt_alloc.

5. IOMMUFD_OBJ_VIOMMU можно создать только вручную через uAPI
   IOMMU_VIOMMU_ALLOC, предоставив dev_id (для физического IOMMU устройства,
   который будет подкреплять vIOMMU) и hwpt_id (для связывания vIOMMU с
   родительским HWPT_PAGING для вложенности). Ядро iommufd свяжет объект vIOMMU
   со struct iommu_device, за которой находится struct device. А драйвер IOMMU
   может реализовать операцию viommu_alloc для выделения собственной структуры
   данных vIOMMU, встраивающей структуру уровня ядра iommufd_viommu и некоторые
   специфичные для драйвера данные. При необходимости драйвер может также
   настроить свою аппаратную возможность виртуализации для этого vIOMMU (и тем
   самым для ВМ). Успешное завершение этой операции устанавливает связи между
   объектом vIOMMU и HWPT_PAGING, после чего этот объект vIOMMU можно
   использовать как родительский объект для вложенности при выделении объекта
   HWPT_NESTED, описанного выше.

6. IOMMUFD_OBJ_VDEVICE можно создать только вручную через uAPI
   IOMMU_VDEVICE_ALLOC, предоставив viommu_id для объекта iommufd_viommu и
   dev_id для объекта iommufd_device. Объект vDEVICE будет связью между этими
   двумя родительскими объектами. Через uAPI также задаётся ещё один @virt_id,
   предоставляющий ядру iommufd индекс для хранения объекта vDEVICE в массиве
   vDEVICE для каждого vIOMMU. При необходимости драйвер IOMMU может реализовать
   операцию vdevce_alloc для инициализации своей аппаратуры под возможность
   виртуализации, связанную с vDEVICE. Успешное завершение этой операции
   устанавливает связи между vIOMMU и устройством.

Устройство может привязаться только к одному iommufd из-за заявленного владения
DMA и присоединиться не более чем к одному объекту IOAS (поддержки PASID пока
нет).

Структуры данных ядра
---------------------

Видимые пользователю объекты подкреплены следующими структурами данных:

- iommufd_ioas для IOMMUFD_OBJ_IOAS.
- iommufd_device для IOMMUFD_OBJ_DEVICE.
- iommufd_hwpt_paging для IOMMUFD_OBJ_HWPT_PAGING.
- iommufd_hwpt_nested для IOMMUFD_OBJ_HWPT_NESTED.
- iommufd_fault для IOMMUFD_OBJ_FAULT.
- iommufd_viommu для IOMMUFD_OBJ_VIOMMU.
- iommufd_vdevice для IOMMUFD_OBJ_VDEVICE.
- iommufd_veventq для IOMMUFD_OBJ_VEVENTQ.
- iommufd_hw_queue для IOMMUFD_OBJ_HW_QUEUE.

Несколько терминов, возникающих при рассмотрении этих структур данных:

- Automatic domain — относится к домену iommu, создаваемому автоматически при
  присоединении устройства к объекту IOAS. Это совместимо с семантикой VFIO
  type1.

- Manual domain — относится к домену iommu, назначенному пользователем в
  качестве целевой таблицы страниц, к которой будет присоединяться устройство.
  Хотя в настоящее время нет uAPI для прямого создания такого домена, структура
  данных и алгоритмы готовы к обработке этого сценария использования.

- In-kernel user — относится к чему-то вроде VFIO mdev, использующему интерфейс
  доступа IOMMUFD для доступа к IOAS. Это начинается с создания объекта
  iommufd_access, аналогичного привязке домена, которую выполняло бы физическое
  устройство. Затем объект доступа позволит преобразовывать диапазоны IOVA в
  списки struct page * или выполнять прямое чтение/запись по IOVA.

iommufd_ioas служит структурой данных метаданных для управления тем, как
диапазоны IOVA отображаются на страницы памяти, и состоит из:

- struct io_pagetable, хранящей отображение IOVA
- struct iopt_area, представляющих заполненные части IOVA
- struct iopt_pages, представляющей хранилище PFN
- struct iommu_domain, представляющей таблицу страниц ввода-вывода в IOMMU
- struct iopt_pages_access, представляющей внутриядерных пользователей PFN
- struct xarray pinned_pfns, хранящей список страниц, закреплённых
  внутриядерными пользователями

Каждый iopt_pages представляет логический линейный массив полных PFN. PFN в
конечном счёте выводятся из VA пространства пользователя через mm_struct. После
закрепления PFN хранятся в IOPTE некоторого iommu_domain либо внутри xarray
pinned_pfns, если они были закреплены через iommufd_access.

PFN приходится копировать между всеми сочетаниями мест хранения в зависимости от
того, какие домены присутствуют и какого рода внутриядерные пользователи
«программного доступа» («software access») существуют. Этот механизм гарантирует,
что страница закрепляется только один раз.

io_pagetable состоит из iopt_areas, указывающих на iopt_pages, вместе со списком
iommu_domains, отражающих отображение IOVA на PFN.

Несколько io_pagetable через свои iopt_area могут совместно использовать один
iopt_pages, что позволяет избежать многократного закрепления и двойного учёта
потребления страниц.

iommufd_ioas можно совместно использовать между подсистемами, например VFIO и
VDPA, при условии, что устройства, управляемые разными подсистемами, привязаны к
одному и тому же iommufd.

API ядра IOMMUFD
================

kAPI IOMMUFD ориентирован на устройства (device-centric), а связанные с группами
хитрости управляются за кулисами. Это позволяет внешним драйверам, вызывающим
такой kAPI, реализовать простой ориентированный на устройства uAPI для
подключения своего устройства к iommufd, не навязывая явно семантику групп в
своём uAPI, как это делает VFIO.

VFIO и IOMMUFD
--------------

Подключить устройство VFIO к iommufd можно двумя способами.

Первый — это совместимый с VFIO способ, реализующий напрямую IOCTL'ы контейнера
/dev/vfio/vfio путём их отображения в операции io_pagetable. Это позволяет
использовать iommufd в устаревших приложениях VFIO, создав символическую ссылку
с /dev/vfio/vfio на /dev/iommufd либо расширив VFIO для выполнения SET_CONTAINER
с использованием iommufd вместо дескриптора контейнера.

Второй подход напрямую расширяет VFIO для поддержки нового набора
ориентированных на устройства пользовательских API на основе упомянутого выше
API ядра IOMMUFD. Он требует изменений в пространстве пользователя, но лучше
соответствует семантике API IOMMUFD и упрощает поддержку новых возможностей
iommufd по сравнению с первым подходом.

В настоящее время оба подхода всё ещё находятся в разработке.

Остаётся ещё несколько пробелов, которые нужно устранить, чтобы догнать VFIO
type1, как задокументировано в iommufd_vfio_check_extension().
