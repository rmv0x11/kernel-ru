========================================================
OpenCAPI (Open Coherent Accelerator Processor Interface)
========================================================

OpenCAPI — это интерфейс между процессорами и ускорителями. Он нацелен
на низкую задержку (latency) и высокую пропускную способность.

Спецификация была разработана консорциумом OpenCAPI Consortium и теперь
доступна в `Compute Express Link Consortium
<https://computeexpresslink.org/resource/opencapi-specification-archive/>`_.

Он позволяет ускорителю (которым может быть FPGA, ASIC, ...) когерентно
обращаться к памяти хоста, используя виртуальные адреса. Устройство
OpenCAPI также может иметь собственную память, к которой можно
обращаться со стороны хоста.

OpenCAPI известен в linux как «ocxl» — открытая, не зависящая от
процессора эволюция «cxl» (драйвера для интерфейса IBM CAPI на
powerpc), который был назван так, чтобы избежать путаницы с подсистемой
ISDN CAPI.


Высокоуровневый обзор
=====================

OpenCAPI определяет канальный уровень (Data Link Layer, DL) и
транзакционный уровень (Transaction Layer, TL), которые должны быть
реализованы поверх физического канала. Любой процессор или устройство,
реализующие DL и TL, могут начать совместное использование памяти.

::

  +-----------+                         +-------------+
  |           |                         |             |
  |           |                         | Accelerated |
  | Processor |                         |  Function   |
  |           |  +--------+             |    Unit     |  +--------+
  |           |--| Memory |             |    (AFU)    |--| Memory |
  |           |  +--------+             |             |  +--------+
  +-----------+                         +-------------+
       |                                       |
  +-----------+                         +-------------+
  |    TL     |                         |    TLX      |
  +-----------+                         +-------------+
       |                                       |
  +-----------+                         +-------------+
  |    DL     |                         |    DLX      |
  +-----------+                         +-------------+
       |                                       |
       |                   PHY                 |
       +---------------------------------------+



Обнаружение устройств
=====================

OpenCAPI опирается на PCI-подобное конфигурационное пространство,
реализованное на устройстве. Поэтому хост может обнаруживать AFU,
запрашивая конфигурационное пространство.

Устройства OpenCAPI в Linux рассматриваются как устройства PCI (с
несколькими оговорками). Предполагается, что прошивка (firmware)
абстрагирует аппаратное обеспечение так, как если бы это был канал PCI.
Повторно используется значительная часть существующей инфраструктуры
PCI: устройства сканируются, а BAR назначаются в ходе стандартного
перечисления PCI. Поэтому для просмотра доступных устройств можно
использовать команды наподобие «lspci».

Конфигурационное пространство определяет AFU, которые могут быть найдены
на физическом адаптере, например его имя, со сколькими контекстами
памяти он может работать, размер его областей MMIO, ...



MMIO
====

OpenCAPI определяет для каждого AFU две области MMIO:

* глобальную область MMIO, с регистрами, относящимися ко всему AFU;
* область MMIO на каждый процесс, которая имеет фиксированный размер для
  каждого контекста.



Прерывания AFU
==============

OpenCAPI включает возможность для AFU отправить прерывание процессу
хоста. Это делается через «intrp_req», определённый в транзакционном
уровне (Transaction Layer), с указанием 64-битного дескриптора объекта,
который определяет прерывание.

Драйвер позволяет процессу выделить прерывание и получить его 64-битный
дескриптор объекта, который может быть передан AFU.



Символьные устройства
=====================

Драйвер создаёт по одному символьному устройству на каждый AFU,
найденный на физическом устройстве. Физическое устройство может иметь
несколько функций, и каждая функция может иметь несколько AFU. Однако на
момент написания это тестировалось только с устройствами, экспортирующими
лишь один AFU.

Символьные устройства можно найти в /dev/ocxl/, они именуются как:
/dev/ocxl/<AFU name>.<location>.<index>

где <AFU name> — это имя длиной не более 20 символов, как оно
указано в конфигурационном пространстве AFU.
<location> добавляется драйвером и помогает различать устройства,
когда в системе присутствует более одного экземпляра одного и того же
устройства OpenCAPI.
<index> также служит для различения AFU в маловероятном случае, когда
устройство несёт несколько копий одного и того же AFU.



Класс sysfs
===========

Для устройств, представляющих AFU, добавляется класс ocxl. См.
/sys/class/ocxl. Структура описана в
Documentation/ABI/testing/sysfs-class-ocxl



Пользовательский API
====================

open
----

На основе определения AFU, найденного в конфигурационном пространстве,
AFU может поддерживать работу с более чем одним контекстом памяти; в
этом случае соответствующее символьное устройство может быть открыто
несколько раз разными процессами.


ioctl
-----

OCXL_IOCTL_ATTACH:

  Присоединить контекст памяти вызывающего процесса к AFU, чтобы AFU мог
  обращаться к его памяти.

OCXL_IOCTL_IRQ_ALLOC:

  Выделить прерывание AFU и вернуть идентификатор.

OCXL_IOCTL_IRQ_FREE:

  Освободить ранее выделенное прерывание AFU.

OCXL_IOCTL_IRQ_SET_FD:

  Связать event fd с прерыванием AFU, чтобы пользовательский процесс мог
  быть уведомлён, когда AFU отправляет прерывание.

OCXL_IOCTL_GET_METADATA:

  Получить с карты конфигурационную информацию, такую как размер
  областей MMIO, версия AFU и PASID для текущего контекста.

OCXL_IOCTL_ENABLE_P9_WAIT:

  Позволяет AFU пробудить поток пространства пользователя, выполняющий
  «wait». Возвращает в пространство пользователя информацию, позволяющую
  ему настроить AFU. Обратите внимание, что это доступно только на
  POWER9.

OCXL_IOCTL_GET_FEATURES:

  Сообщает, какие функции CPU, влияющие на OpenCAPI, доступны для
  использования из пространства пользователя.


mmap
----

Процесс может выполнить mmap области MMIO на каждый процесс для
взаимодействия с AFU.
