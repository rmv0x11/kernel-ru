******************************
Различия между V4L и V4L2
******************************

API Video For Linux был впервые представлен в Linux 2.1, чтобы объединить и
заменить различные интерфейсы, связанные с ТВ- и радиоустройствами, которые
независимо разрабатывались авторами драйверов в предыдущие годы. Начиная с
Linux 2.5 значительно улучшенный API V4L2 заменяет API V4L. Поддержка старых
вызовов V4L была удалена из ядра, однако библиотека :ref:`libv4l`
поддерживает преобразование системного вызова API V4L в вызов V4L2.

Открытие и закрытие устройств
=============================

Из соображений совместимости имена файлов символьных устройств, рекомендуемые
для устройств захвата видео, наложения (overlay), радио и захвата сырых vbi в
V4L2, не изменились по сравнению с теми, что использовались в V4L. Они
перечислены в :ref:`devices` и ниже в :ref:`v4l-dev`.

Устройства телетекста (диапазон младших номеров 192-223) были удалены в V4L2 и
больше не существуют. Оборудования для обработки чистого телетекста больше нет.
Вместо него используется сырой или нарезанный (sliced) VBI.

Модуль ``videodev`` из V4L автоматически назначает младшие номера драйверам в
порядке загрузки, в зависимости от типа зарегистрированного устройства. Мы
рекомендуем, чтобы драйверы V4L2 по умолчанию регистрировали устройства с теми
же номерами, но системный администратор может назначать произвольные младшие
номера через параметры модуля драйвера. Старший номер устройства остаётся 81.

V4L запрещает (или раньше запрещал) множественные открытия файла устройства.
Драйверы V4L2 *могут* поддерживать множественные открытия, см. :ref:`open` для
подробностей и последствий.

Драйверы V4L отвечают на ioctl-вызовы V4L2 кодом ошибки ``EINVAL``.

Запрос возможностей
===================

Вызов ioctl ``VIDIOCGCAP`` из V4L эквивалентен вызову
:ref:`VIDIOC_QUERYCAP` из V4L2.

Поле ``name`` в структуре ``video_capability`` стало
``card`` в структуре :c:type:`v4l2_capability`, ``type``
было заменено на ``capabilities``. Обратите внимание, что V4L2 не различает
типы устройств подобным образом; лучше представлять себе базовые устройства
ввода видео, вывода видео и радио, поддерживающие набор связанных функций,
таких как захват видео, наложение видео и захват VBI. Вводное описание см. в
:ref:`open`.

Поле ``audios`` было заменено флагом ``capabilities``
``V4L2_CAP_AUDIO``, который указывает, *есть ли* у устройства какие-либо
аудиовходы или выходы. Чтобы определить их количество, приложения могут
перечислить аудиовходы с помощью ioctl :ref:`VIDIOC_G_AUDIO <VIDIOC_G_AUDIO>`.
Аудио-ioctl описаны в :ref:`audio`.

Поля ``maxwidth``, ``maxheight``, ``minwidth`` и ``minheight``
были удалены. Вызов ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>` или
:ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` с желаемыми
размерами возвращает ближайший возможный размер с учётом
текущего видеостандарта, ограничений обрезки и масштабирования.

Источники видео
===============

V4L предоставляет ioctl ``VIDIOCGCHAN`` и ``VIDIOCSCHAN``, использующие структуру
``video_channel``, для перечисления видеовходов устройства V4L.
Эквивалентные ioctl V4L2 — это
:ref:`VIDIOC_ENUMINPUT`,
:ref:`VIDIOC_G_INPUT <VIDIOC_G_INPUT>` и
:ref:`VIDIOC_S_INPUT <VIDIOC_G_INPUT>`, использующие структуру
:c:type:`v4l2_input`, как описано в :ref:`video`.

Поле ``channel``, подсчитывающее входы, было переименовано в ``index``, а
типы видеовходов были переименованы следующим образом:


.. flat-table::
    :header-rows:  1
    :stub-columns: 0

    * - ``type`` структуры ``video_channel``
      - ``type`` структуры :c:type:`v4l2_input`
    * - ``VIDEO_TYPE_TV``
      - ``V4L2_INPUT_TYPE_TUNER``
    * - ``VIDEO_TYPE_CAMERA``
      - ``V4L2_INPUT_TYPE_CAMERA``

В отличие от поля ``tuners``, выражавшего количество тюнеров для данного
входа, V4L2 предполагает, что каждый видеовход подключён не более чем к одному
тюнеру. Однако тюнер может иметь более одного входа, т. е. ВЧ-разъёмов, а
устройство может иметь несколько тюнеров. Порядковый номер тюнера,
связанного со входом, если он есть, хранится в поле ``tuner``
структуры :c:type:`v4l2_input`. Перечисление тюнеров
обсуждается в :ref:`tuner`.

Избыточный флаг ``VIDEO_VC_TUNER`` был убран. Видеовходы,
связанные с тюнером, имеют тип ``V4L2_INPUT_TYPE_TUNER``. Флаг
``VIDEO_VC_AUDIO`` был заменён полем ``audioset``. V4L2
рассматривает устройства, имеющие до 32 аудиовходов. Каждый установленный бит в
поле ``audioset`` представляет один аудиовход, с которым сочетается данный
видеовход. Сведения об аудиовходах и о том, как переключаться между ними,
см. в :ref:`audio`.

Поле ``norm``, описывавшее поддерживаемые видеостандарты, было заменено
полем ``std``. В спецификации V4L упоминается флаг ``VIDEO_VC_NORM``,
указывающий, можно ли изменить стандарт. Этот флаг был добавлен позднее
вместе с полем ``norm`` и к настоящему моменту удалён. В V4L2 применяется
похожий, хотя и более всеобъемлющий подход к видеостандартам, дополнительные
сведения см. в :ref:`standard`.

Настройка тюнера
================

Вызовы ioctl ``VIDIOCGTUNER`` и ``VIDIOCSTUNER`` из V4L и структура
``video_tuner`` могут использоваться для перечисления тюнеров
ТВ- или радиоустройства V4L. Эквивалентные ioctl V4L2 — это
:ref:`VIDIOC_G_TUNER <VIDIOC_G_TUNER>` и
:ref:`VIDIOC_S_TUNER <VIDIOC_G_TUNER>`, использующие структуру
:c:type:`v4l2_tuner`. Тюнеры рассматриваются в :ref:`tuner`.

Поле ``tuner``, подсчитывающее тюнеры, было переименовано в ``index``. Поля
``name``, ``rangelow`` и ``rangehigh`` остались без изменений.

Флаги ``VIDEO_TUNER_PAL``, ``VIDEO_TUNER_NTSC`` и ``VIDEO_TUNER_SECAM``,
указывавшие поддерживаемые видеостандарты, были убраны. Эта
информация теперь содержится в связанной структуре
:c:type:`v4l2_input`. Для флага ``VIDEO_TUNER_NORM``, указывавшего,
можно ли переключать видеостандарт, замены не существует. Поле ``mode``
для выбора другого видеостандарта было
заменено целым новым набором ioctl и структур, описанных в
:ref:`standard`. Ввиду его повсеместного распространения стоит упомянуть, что
драйвер BTTV поддерживает несколько стандартов в дополнение к обычным
``VIDEO_MODE_PAL`` (0), ``VIDEO_MODE_NTSC``, ``VIDEO_MODE_SECAM`` и
``VIDEO_MODE_AUTO`` (3). А именно N/PAL Argentina, M/PAL, N/PAL и NTSC
Japan с номерами 3-6 (sic).

Флаг ``VIDEO_TUNER_STEREO_ON``, указывавший стереоприём, стал
``V4L2_TUNER_SUB_STEREO`` в поле ``rxsubchans``. Это поле также
позволяет обнаруживать монофонический и двуязычный звук, см.
определение структуры :c:type:`v4l2_tuner` для подробностей.
В настоящее время для флагов ``VIDEO_TUNER_RDS_ON`` и
``VIDEO_TUNER_MBS_ON`` замены не существует.

Флаг ``VIDEO_TUNER_LOW`` был переименован в ``V4L2_TUNER_CAP_LOW`` в
поле ``capability`` структуры :c:type:`v4l2_tuner`.

Вызовы ioctl ``VIDIOCGFREQ`` и ``VIDIOCSFREQ`` для изменения частоты тюнера
были переименованы в
:ref:`VIDIOC_G_FREQUENCY <VIDIOC_G_FREQUENCY>` и
:ref:`VIDIOC_S_FREQUENCY <VIDIOC_G_FREQUENCY>`. Они принимают указатель
на структуру :c:type:`v4l2_frequency` вместо
целого числа типа unsigned long.

Свойства изображения
====================

В V4L2 нет эквивалента ioctl ``VIDIOCGPICT`` и ``VIDIOCSPICT``
и структуры ``video_picture``. Следующие поля были
заменены элементами управления (controls) V4L2, доступными через
ioctl :ref:`VIDIOC_QUERYCTRL`,
:ref:`VIDIOC_G_CTRL <VIDIOC_G_CTRL>` и
:ref:`VIDIOC_S_CTRL <VIDIOC_G_CTRL>`:


.. flat-table::
    :header-rows:  1
    :stub-columns: 0

    * - структура ``video_picture``
      - идентификатор элемента управления (control ID) V4L2
    * - ``brightness``
      - ``V4L2_CID_BRIGHTNESS``
    * - ``hue``
      - ``V4L2_CID_HUE``
    * - ``colour``
      - ``V4L2_CID_SATURATION``
    * - ``contrast``
      - ``V4L2_CID_CONTRAST``
    * - ``whiteness``
      - ``V4L2_CID_WHITENESS``

Предполагается, что элементы управления изображением в V4L изменяются в
диапазоне от 0 до 65535 без какого-либо конкретного значения сброса. API V4L2
допускает произвольные пределы и значения по умолчанию, которые можно запросить
с помощью ioctl :ref:`VIDIOC_QUERYCTRL`. Общие
сведения об элементах управления см. в :ref:`control`.

подразумевается выбранным форматом изображения. V4L2 не предоставляет такую
информацию явно, исходя из того, что приложения, распознающие формат, знают о
глубине изображения, а остальным её знать не нужно. Поле ``palette`` перешло

Аудио
=====

Вызовы ioctl ``VIDIOCGAUDIO`` и ``VIDIOCSAUDIO`` и структура
``video_audio`` используются для перечисления аудиовходов
устройства V4L. Эквивалентные ioctl V4L2 — это
:ref:`VIDIOC_G_AUDIO <VIDIOC_G_AUDIO>` и
:ref:`VIDIOC_S_AUDIO <VIDIOC_G_AUDIO>`, использующие структуру
:c:type:`v4l2_audio`, как описано в :ref:`audio`.

Поле ``audio`` («номер канала»), подсчитывавшее аудиовходы, было переименовано
в ``index``.

В ``VIDIOCSAUDIO`` поле ``mode`` выбирает *один* из режимов аудиодемодуляции
``VIDEO_SOUND_MONO``, ``VIDEO_SOUND_STEREO``, ``VIDEO_SOUND_LANG1`` или
``VIDEO_SOUND_LANG2``. Когда текущий аудиостандарт — BTSC,
``VIDEO_SOUND_LANG2`` относится к SAP, а
``VIDEO_SOUND_LANG1`` не имеет смысла. Кроме того, что не задокументировано в
спецификации V4L, нет способа запросить выбранный режим. В
``VIDIOCGAUDIO`` драйвер возвращает в этом поле *фактически принимаемые*
аудиопрограммы. В API V4L2 эта информация хранится в
полях ``rxsubchans`` и ``audmode`` структуры :c:type:`v4l2_tuner`
соответственно. Дополнительные сведения о тюнерах см. в :ref:`tuner`.
Применительно к аудиорежимам структура
:c:type:`v4l2_audio` также сообщает, является ли это моно- или
стереовходом, независимо от того, является ли источником тюнер.

Следующие поля были заменены элементами управления V4L2, доступными через
ioctl :ref:`VIDIOC_QUERYCTRL`,
:ref:`VIDIOC_G_CTRL <VIDIOC_G_CTRL>` и
:ref:`VIDIOC_S_CTRL <VIDIOC_G_CTRL>`:


.. flat-table::
    :header-rows:  1
    :stub-columns: 0

    * - структура ``video_audio``
      - идентификатор элемента управления (control ID) V4L2
    * - ``volume``
      - ``V4L2_CID_AUDIO_VOLUME``
    * - ``bass``
      - ``V4L2_CID_AUDIO_BASS``
    * - ``treble``
      - ``V4L2_CID_AUDIO_TREBLE``
    * - ``balance``
      - ``V4L2_CID_AUDIO_BALANCE``

Чтобы определить, какие из этих элементов управления поддерживаются драйвером,
V4L предоставляет ``flags`` ``VIDEO_AUDIO_VOLUME``, ``VIDEO_AUDIO_BASS``,
``VIDEO_AUDIO_TREBLE`` и ``VIDEO_AUDIO_BALANCE``. В API V4L2
ioctl :ref:`VIDIOC_QUERYCTRL` сообщает, поддерживается ли
соответствующий элемент управления. Соответственно, флаги ``VIDEO_AUDIO_MUTABLE``
и ``VIDEO_AUDIO_MUTE`` были заменены булевым
элементом управления ``V4L2_CID_AUDIO_MUTE``.

Все элементы управления V4L2 имеют атрибут ``step``, заменяющий поле
``step`` структуры ``video_audio``. Предполагается, что аудиоэлементы управления
V4L изменяются в диапазоне от 0 до 65535 без какого-либо конкретного значения
сброса. API V4L2 допускает произвольные пределы и значения по умолчанию, которые
можно запросить с помощью ioctl :ref:`VIDIOC_QUERYCTRL`. Общие
сведения об элементах управления см. в :ref:`control`.

Наложение кадрового буфера
==========================

ioctl V4L2, эквивалентные ``VIDIOCGFBUF`` и ``VIDIOCSFBUF``, — это
:ref:`VIDIOC_G_FBUF <VIDIOC_G_FBUF>` и
:ref:`VIDIOC_S_FBUF <VIDIOC_G_FBUF>`. Поле ``base`` структуры
``video_buffer`` осталось без изменений, за исключением того, что V4L2
определяет флаг для указания неразрушающих наложений вместо указателя
``NULL``. Все остальные поля перешли в подструктуру ``fmt`` типа
:c:type:`v4l2_pix_format` структуры :c:type:`v4l2_framebuffer`. Поле ``depth``
было заменено полем ``pixelformat``. Список форматов RGB и
соответствующих им глубин цвета см. в :ref:`pixfmt-rgb`.

Вместо специальных ioctl ``VIDIOCGWIN`` и ``VIDIOCSWIN`` V4L2
использует ioctl согласования формата данных общего назначения
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Они принимают в качестве аргумента указатель
на структуру :c:type:`v4l2_format`. Здесь используется член ``win``
объединения ``fmt`` — структура
:c:type:`v4l2_window`.

Поля ``x``, ``y``, ``width`` и ``height`` структуры
``video_window`` перешли в подструктуру ``w`` типа
:c:type:`v4l2_rect` структуры :c:type:`v4l2_window`. Поля ``chromakey``,
``clips`` и ``clipcount`` остались без изменений. Структура
``video_clip`` была переименована в структуру
:c:type:`v4l2_clip`, также содержащую структуру
:c:type:`v4l2_rect`, но семантика осталась прежней.

Флаг ``VIDEO_WINDOW_INTERLACE`` был убран. Вместо этого приложения
должны установить поле ``field`` в ``V4L2_FIELD_ANY`` или
``V4L2_FIELD_INTERLACED``. Флаг ``VIDEO_WINDOW_CHROMAKEY`` перешёл
в структуру :c:type:`v4l2_framebuffer` под новым
именем ``V4L2_FBUF_FLAG_CHROMAKEY``.

В V4L сохранение указателя на битовую карту в ``clips`` и установка ``clipcount``
в ``VIDEO_CLIP_BITMAP`` (-1) запрашивают отсечение по битовой карте, используя
битовую карту фиксированного размера 1024 × 625 бит. Структура :c:type:`v4l2_window`
имеет для этой цели отдельное поле-указатель ``bitmap``, а размер битовой карты
определяется значениями ``w.width`` и ``w.height``.

ioctl ``VIDIOCCAPTURE`` для включения или отключения наложения был переименован в
:ref:`VIDIOC_OVERLAY`.

Обрезка
=======

Для захвата только подобласти полного изображения V4L определяет
ioctl ``VIDIOCGCAPTURE`` и ``VIDIOCSCAPTURE``, использующие структуру
``video_capture``. Эквивалентные ioctl V4L2 — это
:ref:`VIDIOC_G_CROP <VIDIOC_G_CROP>` и
:ref:`VIDIOC_S_CROP <VIDIOC_G_CROP>`, использующие структуру
:c:type:`v4l2_crop`, и связанный с ними
ioctl :ref:`VIDIOC_CROPCAP`. Это довольно
сложная тема, см. :ref:`crop` для подробностей.

Поля ``x``, ``y``, ``width`` и ``height`` перешли в подструктуру ``c`` типа
:c:type:`v4l2_rect` структуры :c:type:`v4l2_crop`. Поле ``decimation`` было
убрано. В API V4L2 коэффициент масштабирования подразумевается размером
прямоугольника обрезки и размером захваченного или накладываемого изображения.

Флаги ``VIDEO_CAPTURE_ODD`` и ``VIDEO_CAPTURE_EVEN`` для захвата
только нечётного или чётного поля соответственно были заменены
``V4L2_FIELD_TOP`` и ``V4L2_FIELD_BOTTOM`` в поле с именем
``field`` структур :c:type:`v4l2_pix_format` и
:c:type:`v4l2_window`. Эти структуры используются для
выбора формата захвата или наложения с помощью
ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`.

Чтение изображений, отображение памяти
======================================

Захват методом чтения
----------------------

Между чтением изображений с устройства V4L или
V4L2 с помощью функции :c:func:`read()` нет существенной разницы, однако драйверы
V4L2 не обязаны поддерживать этот метод ввода-вывода. Приложения могут
определить, доступна ли эта функция, с помощью
ioctl :ref:`VIDIOC_QUERYCAP`. Все устройства V4L2,
обменивающиеся данными с приложениями, должны поддерживать
функции :c:func:`select()` и :c:func:`poll()`.

Для выбора формата и размера изображения V4L предоставляет ioctl ``VIDIOCSPICT`` и
``VIDIOCSWIN``. V4L2 использует ioctl согласования формата данных общего
назначения :ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Они принимают в качестве аргумента указатель
на структуру :c:type:`v4l2_format`, здесь используется структура
:c:type:`v4l2_pix_format` с именем ``pix`` её
объединения ``fmt``.

Дополнительные сведения об интерфейсе чтения V4L2 см. в :ref:`rw`.

Захват с использованием отображения памяти
------------------------------------------

Приложения могут читать с устройств V4L, отображая буферы в памяти
устройства, или, что более распространено, просто буферы, выделенные в
доступной для DMA системной памяти, в своё адресное пространство. Это позволяет
избежать накладных расходов на копирование данных метода чтения. V4L2 также
поддерживает отображение памяти, с некоторыми различиями.


.. flat-table::
    :header-rows:  1
    :stub-columns: 0

    * - V4L
      - V4L2
    * -
      - Формат изображения должен быть выбран до выделения буферов,
	с помощью ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Если
	формат не выбран, драйвер может использовать последний формат, возможно,
	запрошенный другим приложением.
    * - Приложения не могут изменять количество буферов. Оно встроено
	в драйвер, если только у него нет параметра модуля для изменения этого
	количества при загрузке модуля драйвера.
      - ioctl :ref:`VIDIOC_REQBUFS` выделяет
	желаемое количество буферов, это обязательный шаг в
	последовательности инициализации.
    * - Драйверы отображают все буферы как один непрерывный диапазон памяти.
	ioctl ``VIDIOCGMBUF`` доступен для запроса количества буферов,
	смещения каждого буфера от начала виртуального файла и
	общего объёма используемой памяти, которые можно использовать в качестве аргументов
	функции :c:func:`mmap()`.
      - Буферы отображаются по отдельности. Смещение и размер каждого
	буфера можно определить с помощью
	ioctl :ref:`VIDIOC_QUERYBUF`.
    * - ioctl ``VIDIOCMCAPTURE`` подготавливает буфер для захвата. Он
	также определяет формат изображения для этого буфера. ioctl
	возвращается немедленно, в конечном итоге с кодом ошибки ``EAGAIN``, если не
	был обнаружен видеосигнал. Когда драйвер поддерживает более
	одного буфера, приложения могут вызывать ioctl несколько раз и тем самым
	иметь несколько ожидающих запросов на захват.

	ioctl ``VIDIOCSYNC`` приостанавливает выполнение до тех пор, пока определённый
	буфер не будет заполнен.
      - Драйверы поддерживают входящую и исходящую очереди.
	:ref:`VIDIOC_QBUF` ставит любой пустой буфер во
	входящую очередь. Заполненные буферы извлекаются из исходящей
	очереди с помощью ioctl :ref:`VIDIOC_DQBUF <VIDIOC_QBUF>`. Чтобы дождаться,
	пока заполненные буферы станут доступны, можно использовать эту функцию,
	:c:func:`select()` или :c:func:`poll()`. ioctl :ref:`VIDIOC_STREAMON`
	должен быть вызван один раз после постановки в очередь одного или нескольких буферов, чтобы начать
	захват. Его аналог
	:ref:`VIDIOC_STREAMOFF <VIDIOC_STREAMON>` останавливает захват и
	извлекает все буферы из обеих очередей. Приложения могут запросить
	состояние сигнала, если оно известно, с помощью
	ioctl :ref:`VIDIOC_ENUMINPUT`.

Более подробное обсуждение отображения памяти и примеры см. в
:ref:`mmap`.

Чтение сырых данных VBI
=======================

Изначально API V4L не определял интерфейс захвата сырого VBI, только
файл устройства ``/dev/vbi`` был зарезервирован для этой цели. Единственным
драйвером, поддерживавшим этот интерфейс, был драйвер BTTV, де-факто определивший
интерфейс VBI в V4L. Чтение с устройства даёт сырое изображение VBI
со следующими параметрами:


.. flat-table::
    :header-rows:  1
    :stub-columns: 0

    * - структура :c:type:`v4l2_vbi_format`
      - V4L, драйвер BTTV
    * - sampling_rate
      - 28636363 Гц NTSC (или любой другой 525-строчный стандарт); 35468950 Гц PAL
	и SECAM (625-строчные стандарты)
    * - offset
      - ?
    * - samples_per_line
      - 2048
    * - sample_format
      - V4L2_PIX_FMT_GREY. Последние четыре байта (целое число в порядке
	байтов машины) содержат счётчик кадров.
    * - start[]
      - 10, 273 NTSC; 22, 335 PAL и SECAM
    * - count[]
      - 16, 16 [#f9]_
    * - flags
      - 0

Не задокументированные в спецификации V4L, в Linux 2.3 были добавлены ioctl
``VIDIOCGVBIFMT`` и ``VIDIOCSVBIFMT``, использующие структуру
``vbi_format``, для определения параметров изображения VBI.
Эти ioctl лишь частично совместимы с интерфейсом VBI V4L2,
описанным в :ref:`raw-vbi`.

Поля ``offset`` не существует, ``sample_format`` предположительно должно быть
``VIDEO_PALETTE_RAW``, что эквивалентно ``V4L2_PIX_FMT_GREY``. Остальные
поля, вероятно, эквивалентны полям структуры
:c:type:`v4l2_vbi_format`.

По всей видимости, эти ioctl реализует только драйвер Zoran (ZR 36120). Их
семантика отличается от заданной для V4L2 двумя способами. Параметры
сбрасываются при :c:func:`open()`, и ``VIDIOCSVBIFMT`` всегда возвращает код
ошибки ``EINVAL``, если параметры недопустимы.

Прочее
======

В V4L2 нет эквивалента ioctl ``VIDIOCGUNIT``. Приложения могут
найти устройство VBI, связанное с устройством захвата видео (или наоборот),
повторно открыв устройство и запросив данные VBI. Подробности см. в
:ref:`open`.

Замены для ``VIDIOCKEY`` и функций V4L для программирования микрокода
не существует. Новый интерфейс для устройств сжатия и воспроизведения MPEG
задокументирован в :ref:`extended-controls`.

.. [#f1]
   Согласно Documentation/admin-guide/devices.rst, они должны быть символическими ссылками
   на ``/dev/video0``. Обратите внимание, что исходный интерфейс bttv не
   совместим с V4L или V4L2.

.. [#f2]
   Согласно ``Documentation/admin-guide/devices.rst``, символическая ссылка на
   ``/dev/radio0``.

.. [#f3]

   форматов V4L2.

.. [#f4]

   драйверы могут интерпретировать их в соответствии с порядком байтов машины. V4L2
   определяет варианты little-endian, big-endian и с переставленными red/blue. Для
   подробностей см. :ref:`pixfmt-rgb`.

.. [#f5]
   ``VIDEO_PALETTE_YUV422`` и ``VIDEO_PALETTE_YUYV`` — это одни и те же

.. [#f6]

.. [#f7]
   V4L объясняет это так: «RAW capture (BT848)»
