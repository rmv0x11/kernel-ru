***************************************************************************
Обрезка, вставка и масштабирование изображения — программный интерфейс CROP
***************************************************************************

.. note::

   Программный интерфейс CROP в основном вытеснен более новым
   :ref:`интерфейсом SELECTION <selection-api>`. В большинстве случаев
   следует предпочитать новый интерфейс, за исключением определения
   соотношения сторон пикселя, которое реализуется через
   :ref:`VIDIOC_CROPCAP <VIDIOC_CROPCAP>` и не имеет эквивалента в
   интерфейсе SELECTION. Сравнение двух интерфейсов см. в
   :ref:`selection-vs-crop`.

Некоторые устройства захвата видео могут выбирать часть изображения и
уменьшать или увеличивать её до изображения произвольного размера. Эти
возможности мы называем обрезкой и масштабированием. Некоторые устройства
вывода видео могут масштабировать изображение в большую или меньшую сторону и
вставлять его в видеосигнал с произвольным смещением по строке развёртки и по
горизонтали.

Приложения могут использовать следующий интерфейс для выбора области в
видеосигнале, запроса области по умолчанию и аппаратных ограничений.

.. note::

   Несмотря на свои названия, ioctl-вызовы :ref:`VIDIOC_CROPCAP <VIDIOC_CROPCAP>`,
   :ref:`VIDIOC_G_CROP <VIDIOC_G_CROP>` и :ref:`VIDIOC_S_CROP
   <VIDIOC_G_CROP>` применимы как к устройствам ввода, так и к устройствам
   вывода.

Для масштабирования требуются источник и приёмник. На устройстве захвата
видео или устройстве наложения (overlay) источником является видеосигнал, а
ioctl-вызовы обрезки определяют область, которая фактически выбирается.
Приёмником являются изображения, считываемые приложением или накладываемые на
графический экран. Их размер (и положение для наложения) согласуется с помощью
ioctl-вызовов :ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`.

На устройстве вывода видео источником являются изображения, передаваемые
приложением, и их размер снова согласуется с помощью ioctl-вызовов
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`,
либо может быть закодирован в сжатом видеопотоке. Приёмником является
видеосигнал, а ioctl-вызовы обрезки определяют область, в которую вставляются
изображения.

Прямоугольники источника и приёмника определены, даже если устройство не
поддерживает масштабирование или ioctl-вызовы :ref:`VIDIOC_G_CROP <VIDIOC_G_CROP>` и
:ref:`VIDIOC_S_CROP <VIDIOC_G_CROP>`. В этом случае их размер (и положение,
там где это применимо) будет фиксированным.

.. note::

   Все устройства захвата и вывода, поддерживающие интерфейс CROP или
   SELECTION, также поддерживают ioctl-вызов
   :ref:`VIDIOC_CROPCAP <VIDIOC_CROPCAP>`.

Структуры обрезки
=================


.. _crop-scale:

.. kernel-figure:: crop.svg
    :alt:    crop.svg
    :align:  center

    Обрезка, вставка и масштабирование изображения

    Процесс обрезки, вставки и масштабирования



Для устройств захвата координаты верхнего левого угла, ширина и высота
области, которую можно выбирать, задаются подструктурой ``bounds`` структуры
:c:type:`v4l2_cropcap`, возвращаемой ioctl-вызовом
:ref:`VIDIOC_CROPCAP <VIDIOC_CROPCAP>`. Для поддержки широкого диапазона
оборудования данная спецификация не определяет ни начала координат, ни единиц
измерения. Однако по соглашению драйверы должны вести горизонтальный отсчёт
немасштабированных отсчётов относительно 0H (передний фронт импульса
горизонтальной синхронизации, см. :ref:`vbi-hsync`). По вертикали — номера
строк ITU-R первого поля (нумерацию строк ITU R-525 см. для :ref:`525 строк
<vbi-525>` и для :ref:`625 строк <vbi-625>`), умноженные на два, если драйвер
может захватывать оба поля.

Верхний левый угол, ширина и высота прямоугольника источника, то есть области,
которая фактически выбирается, задаются структурой
:c:type:`v4l2_crop` в той же системе координат, что и
структура :c:type:`v4l2_cropcap`. Приложения могут использовать ioctl-вызовы
:ref:`VIDIOC_G_CROP <VIDIOC_G_CROP>` и :ref:`VIDIOC_S_CROP <VIDIOC_G_CROP>`
для чтения и установки этого прямоугольника. Он должен полностью лежать в
пределах границ захвата, и драйвер может дополнительно скорректировать
запрошенный размер и/или положение в соответствии с аппаратными ограничениями.

Каждое устройство захвата имеет прямоугольник источника по умолчанию,
задаваемый подструктурой ``defrect`` структуры
:c:type:`v4l2_cropcap`. Центр этого прямоугольника
должен совпадать с центром активной области изображения видеосигнала и
покрывать то, что автор драйвера считает полным изображением.
Драйверы должны сбрасывать прямоугольник источника в значение по умолчанию при
первой загрузке драйвера, но не позднее.

Для устройств вывода эти структуры и ioctl-вызовы используются соответствующим
образом, определяя прямоугольник *приёмника*, в который изображения будут
вставляться в видеосигнал.


Корректировки масштабирования
=============================

Видеооборудование может иметь различные ограничения на обрезку, вставку и
масштабирование. Оно может только увеличивать или только уменьшать масштаб,
поддерживать только дискретные коэффициенты масштабирования или иметь
различные возможности масштабирования по горизонтали и по вертикали. Также оно
может вообще не поддерживать масштабирование. В то же время прямоугольник
структуры :c:type:`v4l2_crop` может требовать выравнивания,
а оба прямоугольника, источника и приёмника, могут иметь произвольные верхние и
нижние пределы размера. В частности, максимальные ``width`` и ``height`` в
структуре :c:type:`v4l2_crop` могут быть меньше области ``bounds`` структуры
:c:type:`v4l2_cropcap`. Поэтому, как обычно, от драйверов ожидается, что они
скорректируют запрошенные параметры и вернут фактически выбранные значения.

Приложения могут сначала изменить прямоугольник источника или приёмника, так
как они могут предпочитать определённый размер изображения или определённую
область в видеосигнале. Если драйверу приходится корректировать оба, чтобы
удовлетворить аппаратным ограничениям, приоритет должен иметь последний
запрошенный прямоугольник, и драйверу предпочтительно следует корректировать
противоположный. Однако ioctl-вызов :ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` не
должен изменять состояние драйвера и поэтому корректирует только запрошенный
прямоугольник.

Предположим, что масштабирование на устройстве захвата видео ограничено
коэффициентом 1:1 или 2:1 в любом направлении, а размер целевого изображения
должен быть кратен 16 × 16 пикселей. Прямоугольник обрезки источника
установлен в значения по умолчанию, которые в этом примере также являются
верхним пределом, — 640 × 400 пикселей со смещением 0, 0. Приложение
запрашивает размер изображения 300 × 225 пикселей, предполагая, что видео
будет соответственно уменьшено от «полного изображения». Драйвер устанавливает
размер изображения в ближайшие возможные значения 304 × 224, затем выбирает
прямоугольник обрезки, ближайший к запрошенному размеру, то есть 608 × 224
(224 × 2:1 превысило бы предел 400). Смещение 0, 0 по-прежнему допустимо и
поэтому не изменяется. Учитывая прямоугольник обрезки по умолчанию,
сообщаемый :ref:`VIDIOC_CROPCAP <VIDIOC_CROPCAP>`, приложение может легко
предложить другое смещение, чтобы отцентрировать прямоугольник обрезки.

Теперь приложение может настаивать на покрытии области с соотношением сторон
изображения, более близким к исходному запросу, поэтому оно запрашивает
прямоугольник обрезки 608 × 456 пикселей. Текущие коэффициенты масштабирования
ограничивают обрезку до 640 × 384, поэтому драйвер возвращает размер обрезки
608 × 384 и корректирует размер изображения до ближайшего возможного 304 × 192.


Примеры
=======

Прямоугольники источника и приёмника должны оставаться неизменными при закрытии
и повторном открытии устройства, чтобы перенаправление данных в устройство или
из него работало без специальной подготовки. Более продвинутые приложения
должны убедиться, что параметры подходят, перед началом операций ввода-вывода.

.. note::

   В следующих двух примерах предполагается устройство захвата видео;
   замените ``V4L2_BUF_TYPE_VIDEO_CAPTURE`` для других типов устройств.

Пример: сброс параметров обрезки
================================

.. code-block:: c

    struct v4l2_cropcap cropcap;
    struct v4l2_crop crop;

    memset (&cropcap, 0, sizeof (cropcap));
    cropcap.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;

    if (-1 == ioctl (fd, VIDIOC_CROPCAP, &cropcap)) {
	perror ("VIDIOC_CROPCAP");
	exit (EXIT_FAILURE);
    }

    memset (&crop, 0, sizeof (crop));
    crop.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
    crop.c = cropcap.defrect;

    /* Ignore if cropping is not supported (EINVAL). */

    if (-1 == ioctl (fd, VIDIOC_S_CROP, &crop)
	&& errno != EINVAL) {
	perror ("VIDIOC_S_CROP");
	exit (EXIT_FAILURE);
    }


Пример: простое уменьшение масштаба
===================================

.. code-block:: c

    struct v4l2_cropcap cropcap;
    struct v4l2_format format;

    reset_cropping_parameters ();

    /* Scale down to 1/4 size of full picture. */

    memset (&format, 0, sizeof (format)); /* defaults */

    format.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;

    format.fmt.pix.width = cropcap.defrect.width >> 1;
    format.fmt.pix.height = cropcap.defrect.height >> 1;
    format.fmt.pix.pixelformat = V4L2_PIX_FMT_YUYV;

    if (-1 == ioctl (fd, VIDIOC_S_FMT, &format)) {
	perror ("VIDIOC_S_FORMAT");
	exit (EXIT_FAILURE);
    }

    /* We could check the actual image size now, the actual scaling factor
       or if the driver can scale at all. */

Пример: выбор области вывода
============================

.. note:: В этом примере предполагается устройство вывода.

.. code-block:: c

    struct v4l2_cropcap cropcap;
    struct v4l2_crop crop;

    memset (&cropcap, 0, sizeof (cropcap));
    cropcap.type = V4L2_BUF_TYPE_VIDEO_OUTPUT;

    if (-1 == ioctl (fd, VIDIOC_CROPCAP;, &cropcap)) {
	perror ("VIDIOC_CROPCAP");
	exit (EXIT_FAILURE);
    }

    memset (&crop, 0, sizeof (crop));

    crop.type = V4L2_BUF_TYPE_VIDEO_OUTPUT;
    crop.c = cropcap.defrect;

    /* Scale the width and height to 50 % of their original size
       and center the output. */

    crop.c.width /= 2;
    crop.c.height /= 2;
    crop.c.left += crop.c.width / 2;
    crop.c.top += crop.c.height / 2;

    /* Ignore if cropping is not supported (EINVAL). */

    if (-1 == ioctl (fd, VIDIOC_S_CROP, &crop)
	&& errno != EINVAL) {
	perror ("VIDIOC_S_CROP");
	exit (EXIT_FAILURE);
    }

Пример: текущий коэффициент масштабирования и соотношение сторон пикселя
========================================================================

.. note:: В этом примере предполагается устройство захвата видео.

.. code-block:: c

    struct v4l2_cropcap cropcap;
    struct v4l2_crop crop;
    struct v4l2_format format;
    double hscale, vscale;
    double aspect;
    int dwidth, dheight;

    memset (&cropcap, 0, sizeof (cropcap));
    cropcap.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;

    if (-1 == ioctl (fd, VIDIOC_CROPCAP, &cropcap)) {
	perror ("VIDIOC_CROPCAP");
	exit (EXIT_FAILURE);
    }

    memset (&crop, 0, sizeof (crop));
    crop.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;

    if (-1 == ioctl (fd, VIDIOC_G_CROP, &crop)) {
	if (errno != EINVAL) {
	    perror ("VIDIOC_G_CROP");
	    exit (EXIT_FAILURE);
	}

	/* Cropping not supported. */
	crop.c = cropcap.defrect;
    }

    memset (&format, 0, sizeof (format));
    format.fmt.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;

    if (-1 == ioctl (fd, VIDIOC_G_FMT, &format)) {
	perror ("VIDIOC_G_FMT");
	exit (EXIT_FAILURE);
    }

    /* The scaling applied by the driver. */

    hscale = format.fmt.pix.width / (double) crop.c.width;
    vscale = format.fmt.pix.height / (double) crop.c.height;

    aspect = cropcap.pixelaspect.numerator /
	 (double) cropcap.pixelaspect.denominator;
    aspect = aspect * hscale / vscale;

    /* Devices following ITU-R BT.601 do not capture
       square pixels. For playback on a computer monitor
       we should scale the images to this size. */

    dwidth = format.fmt.pix.width / aspect;
    dheight = format.fmt.pix.height;
