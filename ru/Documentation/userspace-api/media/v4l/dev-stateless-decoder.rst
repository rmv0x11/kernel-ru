*********************************************************************
Интерфейс stateless-видеодекодера memory-to-memory
*********************************************************************

Stateless-декодер — это декодер, работающий без сохранения какого-либо состояния
между обработанными кадрами. Это означает, что каждый кадр декодируется независимо
от любых предыдущих и будущих кадров, и что клиент отвечает за поддержание
состояния декодирования и передачу его декодеру при каждом запросе на
декодирование. Это противоположно интерфейсу stateful-видеодекодера,
где аппаратура и драйвер поддерживают состояние декодирования, а от клиента
требуется лишь предоставлять необработанный закодированный поток и извлекать
декодированные кадры в порядке отображения.

В этом разделе описывается, как пространство пользователя («клиент») должно
взаимодействовать со stateless-декодерами, чтобы успешно декодировать
закодированный поток. По сравнению со stateful-кодеками последовательность
действий декодера/клиента проще, но ценой этой простоты становится
дополнительная сложность на стороне клиента, который отвечает за поддержание
согласованного состояния декодирования.

Stateless-декодеры используют :ref:`media-request-api`. Stateless-декодер
должен предоставлять возможность ``V4L2_BUF_CAP_SUPPORTS_REQUESTS`` на своей
очереди ``OUTPUT``, когда вызываются :c:func:`VIDIOC_REQBUFS` или
:c:func:`VIDIOC_CREATE_BUFS`.

В зависимости от закодированных форматов, поддерживаемых декодером, один
декодированный кадр может быть результатом нескольких запросов на декодирование
(например, потоки H.264 с несколькими слайсами на кадр). Декодеры, поддерживающие
такие форматы, должны также предоставлять возможность
``V4L2_BUF_CAP_SUPPORTS_M2M_HOLD_CAPTURE_BUF`` на своей очереди ``OUTPUT``.

Запрос возможностей
===================

1. Чтобы перечислить набор закодированных форматов, поддерживаемых декодером,
   клиент вызывает :c:func:`VIDIOC_ENUM_FMT` на очереди ``OUTPUT``.

   * Драйвер должен всегда возвращать полный набор поддерживаемых форматов
     ``OUTPUT`` независимо от формата, установленного в данный момент на очереди
     ``CAPTURE``.

   * Одновременно драйвер должен ограничивать набор значений, возвращаемых
     специфичными для кодека элементами управления возможностями (такими как
     профили H.264), набором, фактически поддерживаемым аппаратурой.

2. Чтобы перечислить набор поддерживаемых необработанных (raw) форматов, клиент
   вызывает :c:func:`VIDIOC_ENUM_FMT` на очереди ``CAPTURE``.

   * Драйвер должен возвращать только форматы, поддерживаемые для формата,
     активного в данный момент на очереди ``OUTPUT``.

   * В зависимости от установленного в данный момент формата ``OUTPUT`` набор
     поддерживаемых необработанных форматов может зависеть от значения некоторых
     элементов управления, специфичных для кодека.
     Клиент отвечает за то, чтобы эти элементы управления были установлены
     до запроса очереди ``CAPTURE``. Невыполнение этого приведёт к использованию
     значений по умолчанию для этих элементов управления и к возвращению такого
     набора форматов, который может оказаться непригодным для медиаданных,
     которые клиент пытается декодировать.

3. Клиент может использовать :c:func:`VIDIOC_ENUM_FRAMESIZES`, чтобы определить
   поддерживаемые разрешения для заданного формата, передавая желаемый формат
   пикселей в поле ``pixel_format`` структуры :c:type:`v4l2_frmsizeenum`.

4. Поддерживаемые профили и уровни для текущего формата ``OUTPUT``, если они
   применимы, можно запросить с помощью соответствующих им элементов управления
   через :c:func:`VIDIOC_QUERYCTRL`.

Инициализация
=============

1. Установите закодированный формат на очереди ``OUTPUT`` с помощью
   :c:func:`VIDIOC_S_FMT`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

     ``pixelformat``
         закодированный формат пикселей.

     ``width``, ``height``
         закодированные ширина и высота, разобранные из потока.

     прочие поля
         следуют стандартной семантике.

   .. note::

      Изменение формата ``OUTPUT`` может изменить установленный в данный момент
      формат ``CAPTURE``. Драйвер выведет новый формат ``CAPTURE`` из
      устанавливаемого формата ``OUTPUT``, включая разрешение, параметры
      колориметрии и т. д. Если клиенту нужен конкретный формат ``CAPTURE``,
      он должен скорректировать его впоследствии.

2. Вызовите :c:func:`VIDIOC_S_EXT_CTRLS`, чтобы установить все элементы
   управления (разобранные заголовки и т. д.), требуемые форматом ``OUTPUT``
   для перечисления форматов ``CAPTURE``.

3. Вызовите :c:func:`VIDIOC_G_FMT` для очереди ``CAPTURE``, чтобы получить формат
   для буферов назначения, разобранных/декодированных из потока байтов.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

   * **Возвращаемые поля:**

     ``width``, ``height``
         разрешение буфера кадра для декодированных кадров.

     ``pixelformat``
         формат пикселей для декодированных кадров.

     ``num_planes`` (только для ``type`` _MPLANE)
         количество плоскостей для pixelformat.

     ``sizeimage``, ``bytesperline``
         согласно стандартной семантике; соответствуют формату буфера кадра.

   .. note::

      Значением ``pixelformat`` может быть любой формат пикселей, поддерживаемый
      для формата ``OUTPUT``, исходя из возможностей аппаратуры. Рекомендуется,
      чтобы драйвер выбирал предпочтительный/оптимальный формат для текущей
      конфигурации. Например, формат YUV может быть предпочтительнее формата RGB,
      если для RGB потребовался бы дополнительный шаг преобразования.

4. *[необязательно]* Перечислите форматы ``CAPTURE`` с помощью
   :c:func:`VIDIOC_ENUM_FMT` на очереди ``CAPTURE``. Клиент может использовать
   этот ioctl, чтобы узнать, какие альтернативные необработанные форматы
   поддерживаются для текущего формата ``OUTPUT``, и выбрать один из них с
   помощью :c:func:`VIDIOC_S_FMT`.

   .. note::

      Драйвер будет возвращать только форматы, поддерживаемые для выбранного в
      данный момент формата ``OUTPUT`` и установленных в данный момент элементов
      управления, даже если декодер в целом может поддерживать больше форматов.

      Например, декодер может поддерживать форматы YUV и RGB для разрешений
      1920x1088 и ниже, но только YUV для более высоких разрешений (из-за
      аппаратных ограничений). После установки разрешения 1920x1088 или ниже
      в качестве формата ``OUTPUT`` вызов :c:func:`VIDIOC_ENUM_FMT` может вернуть
      набор форматов пикселей YUV и RGB, но после установки разрешения выше
      1920x1088 драйвер не будет возвращать форматы пикселей RGB, поскольку они
      не поддерживаются для этого разрешения.

5. *[необязательно]* Выберите формат ``CAPTURE``, отличный от предложенного, с
   помощью :c:func:`VIDIOC_S_FMT` на очереди ``CAPTURE``. Клиент может выбрать
   формат, отличный от выбранного/предложенного драйвером в
   :c:func:`VIDIOC_G_FMT`.

    * **Обязательные поля:**

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``pixelformat``
          необработанный формат пикселей.

      ``width``, ``height``
         разрешение буфера кадра декодированного потока; обычно не изменяется по
         сравнению с возвращённым :c:func:`VIDIOC_G_FMT`, но может отличаться,
         если аппаратура поддерживает композицию и/или масштабирование.

   После выполнения этого шага клиент должен снова выполнить шаг 3, чтобы
   получить актуальную информацию о размере и компоновке буферов.

6. Выделите буферы источника (потока байтов) с помощью :c:func:`VIDIOC_REQBUFS`
   на очереди ``OUTPUT``.

    * **Обязательные поля:**

      ``count``
          запрошенное количество буферов для выделения; больше нуля.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

      ``memory``
          следует стандартной семантике.

    * **Возвращаемые поля:**

      ``count``
          фактическое количество выделенных буферов.

    * При необходимости драйвер скорректирует ``count`` так, чтобы он был равен
      или больше минимума из требуемого количества буферов ``OUTPUT`` для
      заданного формата и запрошенного количества. Клиент должен проверить это
      значение после возврата из ioctl, чтобы получить фактическое количество
      выделенных буферов.

7. Выделите буферы назначения (необработанного формата) с помощью
   :c:func:`VIDIOC_REQBUFS` на очереди ``CAPTURE``.

    * **Обязательные поля:**

      ``count``
          запрошенное количество буферов для выделения; больше нуля. Клиент
          отвечает за определение минимального количества буферов, требуемого
          для корректного декодирования потока (учитывая, например, опорные
          кадры), и за передачу равного или большего числа.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``memory``
          следует стандартной семантике. ``V4L2_MEMORY_USERPTR`` не
          поддерживается для буферов ``CAPTURE``.

    * **Возвращаемые поля:**

      ``count``
          скорректировано до выделенного количества буферов, в случае если кодеку
          требуется больше буферов, чем запрошено.

    * Драйвер должен скорректировать count до минимума из требуемого количества
      буферов ``CAPTURE`` для текущего формата, конфигурации потока и
      запрошенного количества. Клиент должен проверить это значение после
      возврата из ioctl, чтобы получить количество выделенных буферов.

8. Выделите запросы (вероятно, по одному на каждый буфер ``OUTPUT``) с помощью
    :c:func:`MEDIA_IOC_REQUEST_ALLOC` на медиаустройстве.

9. Запустите потоковую передачу на обеих очередях ``OUTPUT`` и ``CAPTURE`` с
    помощью :c:func:`VIDIOC_STREAMON`.

Декодирование
=============

Для каждого кадра клиент отвечает за отправку как минимум одного запроса, к
которому прикреплено следующее:

* Объём закодированных данных, ожидаемый кодеком для его текущей конфигурации,
  в виде буфера, отправленного в очередь ``OUTPUT``. Обычно это соответствует
  объёму закодированных данных, равному одному кадру, но некоторые форматы могут
  допускать (или требовать) разные объёмы на единицу.
* Все метаданные, необходимые для декодирования отправленных закодированных
  данных, в виде элементов управления, относящихся к декодируемому формату.

Объём данных и содержимое исходного буфера ``OUTPUT``, а также элементы
управления, которые должны быть установлены в запросе, зависят от активного
закодированного формата пикселей и могут зависеть от специфичных для кодека
расширенных элементов управления, как указано в документации к каждому формату.

Если есть вероятность, что для получения декодированного кадра потребуется один
или несколько запросов на декодирование после текущего, то клиент должен
установить флаг ``V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF`` на буфере ``OUTPUT``.
В результате (возможно, частично) декодированный буфер ``CAPTURE`` не будет
сделан доступным для извлечения и будет повторно использован для следующего
запроса на декодирование, если метка времени следующего буфера ``OUTPUT`` не
изменилась.

Таким образом, типичный кадр будет декодироваться с использованием следующей
последовательности:

1. Поставьте в очередь буфер ``OUTPUT``, содержащий одну единицу закодированных
   данных потока байтов для запроса на декодирование, с помощью
   :c:func:`VIDIOC_QBUF`.

    * **Обязательные поля:**

      ``index``
          индекс буфера, помещаемого в очередь.

      ``type``
          тип буфера.

      ``bytesused``
          количество байтов, занятых закодированным кадром данных в буфере.

      ``flags``
          флаг ``V4L2_BUF_FLAG_REQUEST_FD`` должен быть установлен. Кроме того,
          если мы не уверены, что текущий запрос на декодирование является
          последним, необходимым для получения полностью декодированного кадра,
          то должен быть также установлен флаг
          ``V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF``.

      ``request_fd``
          должен быть установлен в файловый дескриптор запроса на декодирование.

      ``timestamp``
          должна быть установлена в уникальное для каждого кадра значение. Это
          значение будет распространено в буфер декодированного кадра и может
          также использоваться, чтобы применять этот кадр как опорный для
          другого. При использовании нескольких запросов на декодирование на один
          кадр метки времени всех буферов ``OUTPUT`` для данного кадра должны
          быть идентичными. Если метка времени изменится, то удерживаемый в
          данный момент буфер ``CAPTURE`` будет сделан доступным для извлечения, а
          текущий запрос будет работать с новым буфером ``CAPTURE``.

2. Установите специфичные для кодека элементы управления для запроса на
   декодирование с помощью :c:func:`VIDIOC_S_EXT_CTRLS`.

    * **Обязательные поля:**

      ``which``
          должно быть ``V4L2_CTRL_WHICH_REQUEST_VAL``.

      ``request_fd``
          должен быть установлен в файловый дескриптор запроса на декодирование.

      прочие поля
          прочие поля устанавливаются как обычно при установке элементов
          управления. Массив ``controls`` должен содержать все специфичные для
          кодека элементы управления, требуемые для декодирования кадра.

   .. note::

      Можно задавать элементы управления в разных вызовах
      :c:func:`VIDIOC_S_EXT_CTRLS` или перезаписывать ранее установленный элемент
      управления, при условии что ``request_fd`` и ``which`` установлены
      правильно. Учитываться будет состояние элементов управления на момент
      отправки запроса.

   .. note::

      Порядок, в котором выполняются шаги 1 и 2, можно менять местами.

3. Отправьте запрос, вызвав :c:func:`MEDIA_REQUEST_IOC_QUEUE` на FD запроса.

    Если запрос отправлен без буфера ``OUTPUT`` или если некоторые из требуемых
    элементов управления отсутствуют в запросе, то
    :c:func:`MEDIA_REQUEST_IOC_QUEUE` вернёт ``-ENOENT``. Если в очередь
    помещено более одного буфера ``OUTPUT``, то он вернёт ``-EINVAL``.
    Возврат ненулевого значения из :c:func:`MEDIA_REQUEST_IOC_QUEUE` означает,
    что для этого запроса не будет произведён буфер ``CAPTURE``.

Буферы ``CAPTURE`` не должны быть частью запроса и помещаются в очередь
независимо. Они возвращаются в порядке декодирования (т. е. в том же порядке, в
котором закодированные кадры отправлялись в очередь ``OUTPUT``).

Об ошибках декодирования во время выполнения сигнализируется тем, что извлечённые
буферы ``CAPTURE`` несут флаг ``V4L2_BUF_FLAG_ERROR``. Если декодированный
опорный кадр имеет ошибку, то у всех последующих декодированных кадров, которые
на него ссылаются, также установлен флаг ``V4L2_BUF_FLAG_ERROR``, хотя декодер
всё равно будет пытаться производить (вероятно, повреждённые) кадры.

Управление буферами во время декодирования
==========================================
В отличие от stateful-декодеров, stateless-декодер не выполняет никакого
управления буферами: он лишь гарантирует, что извлечённые буферы ``CAPTURE``
могут использоваться клиентом до тех пор, пока они снова не помещены в очередь.
Слово «используются» здесь охватывает использование буфера для композиции или
отображения.

Извлечённый буфер захвата может также использоваться как опорный кадр для другого
буфера.

Кадр задаётся как опорный путём преобразования его метки времени в наносекунды и
сохранения её в соответствующем члене специфичной для кодека структуры элемента
управления. Для выполнения этого преобразования необходимо использовать функцию
:c:func:`v4l2_timeval_to_ns`. Метку времени кадра можно использовать для ссылки
на него, как только все его единицы закодированных данных успешно отправлены в
очередь ``OUTPUT``.

Декодированный буфер, содержащий опорный кадр, не должен повторно использоваться
в качестве цели декодирования до тех пор, пока не будут декодированы все кадры,
ссылающиеся на него. Самый безопасный способ добиться этого — воздержаться от
помещения опорного буфера в очередь до тех пор, пока не будут извлечены все
декодированные кадры, ссылающиеся на него. Однако если драйвер может
гарантировать, что буферы, помещённые в очередь ``CAPTURE``, обрабатываются в
порядке постановки в очередь, то пространство пользователя может воспользоваться
этой гарантией и поместить опорный буфер в очередь при выполнении следующих
условий:

1. Все запросы для кадров, на которые влияет опорный кадр, помещены в очередь, и

2. Помещено в очередь достаточное количество буферов ``CAPTURE``, чтобы покрыть
   все декодированные ссылающиеся кадры.

При помещении в очередь запроса на декодирование драйвер увеличит счётчик ссылок
всех ресурсов, связанных с опорными кадрами. Это означает, что клиент может,
например, закрыть файловые дескрипторы DMABUF буферов опорных кадров, если они
ему больше не понадобятся.

Перемотка (seeking)
===================
Чтобы выполнить перемотку, клиенту нужно лишь отправить запросы, используя
входные буферы, соответствующие новой позиции в потоке. Однако он должен
учитывать, что разрешение могло измениться, и в этом случае следовать
последовательности динамического изменения разрешения. Также, в зависимости от
используемого кодека, параметры изображения (например, SPS/PPS для H.264) могли
измениться, и клиент отвечает за то, чтобы декодеру было отправлено корректное
состояние.

После этого клиент волен игнорировать любой возвращённый буфер ``CAPTURE``,
относящийся к позиции до перемотки.

Приостановка (pausing)
======================

Чтобы приостановиться, клиент может просто прекратить помещать буферы в очередь
``OUTPUT``. Без данных исходного потока байтов нет данных для обработки, и кодек
останется в простое.

Динамическое изменение разрешения
=================================

Если клиент обнаруживает изменение разрешения в потоке, ему потребуется снова
выполнить последовательность инициализации с новым разрешением:

1. Если последний отправленный запрос привёл к удержанию буфера ``CAPTURE`` за
   счёт использования флага ``V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF``, то последний
   кадр недоступен в очереди ``CAPTURE``. В этом случае должна быть отправлена
   команда ``V4L2_DEC_CMD_FLUSH``. Это заставит драйвер извлечь удерживаемый
   буфер ``CAPTURE``.

2. Дождитесь завершения всех отправленных запросов и извлеките соответствующие
   выходные буферы.

3. Вызовите :c:func:`VIDIOC_STREAMOFF` на обеих очередях ``OUTPUT`` и
   ``CAPTURE``.

4. Освободите все буферы ``CAPTURE``, вызвав :c:func:`VIDIOC_REQBUFS` на очереди
   ``CAPTURE`` с количеством буферов, равным нулю.

5. Снова выполните последовательность инициализации (за вычетом выделения буферов
   ``OUTPUT``) с новым разрешением, установленным на очереди ``OUTPUT``.
   Обратите внимание, что из-за ограничений на разрешение на очереди ``CAPTURE``
   может потребоваться выбрать другой формат.

Слив (drain)
============

Если последний отправленный запрос привёл к удержанию буфера ``CAPTURE`` за счёт
использования флага ``V4L2_BUF_FLAG_M2M_HOLD_CAPTURE_BUF``, то последний кадр
недоступен в очереди ``CAPTURE``. В этом случае должна быть отправлена команда
``V4L2_DEC_CMD_FLUSH``. Это заставит драйвер извлечь удерживаемый буфер
``CAPTURE``.

После этого, чтобы слить поток на stateless-декодере, клиенту нужно лишь
дождаться завершения всех отправленных запросов.
