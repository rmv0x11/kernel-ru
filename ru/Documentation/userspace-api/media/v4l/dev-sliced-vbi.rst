***************************************
Интерфейс данных Sliced VBI
***************************************

VBI расшифровывается как Vertical Blanking Interval (интервал вертикального
гашения) — это промежуток в последовательности строк аналогового видеосигнала.
Во время VBI информация изображения не передаётся, что даёт некоторое время на
возврат электронного луча телевизора с электронно-лучевой трубкой в верхнюю
часть экрана.

Устройства sliced VBI используют аппаратные средства для демодуляции данных,
передаваемых в VBI. Драйверы V4L2 *не* должны делать это программно, см. также
:ref:`интерфейс raw VBI <raw-vbi>`. Данные передаются короткими пакетами
фиксированного размера, каждый из которых покрывает одну строку развёртки.
Число пакетов на видеокадр является переменным.

Доступ к устройствам захвата и вывода sliced VBI осуществляется через те же
специальные символьные файлы, что и для устройств raw VBI. Когда драйвер
поддерживает оба интерфейса, функцией по умолчанию для устройства ``/dev/vbi``
является захват или вывод *raw* VBI, а функция sliced VBI доступна только после
вызова ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`, как определено ниже. Аналогично
устройство ``/dev/video`` может поддерживать API sliced VBI, однако функцией по
умолчанию здесь является захват или вывод видео. Для одновременной передачи
данных raw и sliced VBI должны использоваться разные файловые дескрипторы, если
это поддерживается драйвером.

Опрос возможностей
==================

Устройства, поддерживающие API захвата или вывода sliced VBI, устанавливают
соответственно флаг ``V4L2_CAP_SLICED_VBI_CAPTURE`` или
``V4L2_CAP_SLICED_VBI_OUTPUT`` в поле ``capabilities`` структуры
:c:type:`v4l2_capability`, возвращаемой ioctl :ref:`VIDIOC_QUERYCAP`. Должен
поддерживаться хотя бы один из :ref:`методов ввода-вывода <io>` — чтение/запись
или потоковый. Устройства sliced VBI могут иметь тюнер или модулятор.

Вспомогательные функции
=======================

Устройства sliced VBI должны поддерживать ioctl :ref:`видеовхода или вывода <video>`
и :ref:`тюнера или модулятора <tuner>`, если они обладают этими возможностями,
и могут поддерживать ioctl :ref:`управления <control>`. Ioctl
:ref:`видеостандарта <standard>` предоставляют сведения, жизненно важные для
программирования устройства sliced VBI, поэтому они должны поддерживаться.

.. _sliced-vbi-format-negotiation:

Согласование формата Sliced VBI
===============================

Чтобы выяснить, какие сервисы данных поддерживаются аппаратурой, приложения
могут вызвать ioctl
:ref:`VIDIOC_G_SLICED_VBI_CAP <VIDIOC_G_SLICED_VBI_CAP>`. Все драйверы,
реализующие интерфейс sliced VBI, должны поддерживать этот ioctl. Результаты
могут отличаться от результатов ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`, когда
ограничено число строк VBI, которые аппаратура может захватывать или выводить за
кадр, либо число сервисов, которые она может идентифицировать на заданной строке.
Например, на строке 16 PAL аппаратура может уметь искать сигнал VPS или Teletext,
но не оба одновременно.

Чтобы определить выбранные в настоящий момент сервисы, приложения устанавливают
поле ``type`` структуры :c:type:`v4l2_format` в
``V4L2_BUF_TYPE_SLICED_VBI_CAPTURE`` или
``V4L2_BUF_TYPE_SLICED_VBI_OUTPUT``, и ioctl
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` заполняет член ``fmt.sliced`` — структуру
:c:type:`v4l2_sliced_vbi_format`.

Приложения могут запросить другие параметры, инициализировав или изменив член
``fmt.sliced`` и вызвав ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>` с указателем на
структуру :c:type:`v4l2_format`.

API sliced VBI сложнее, чем API raw VBI, поскольку аппаратуре необходимо
сообщить, какой сервис VBI следует ожидать на каждой строке развёртки. Не все
сервисы могут поддерживаться аппаратурой на всех строках (это особенно верно для
вывода VBI, где Teletext часто не поддерживается, а другие сервисы могут
вставляться только в одну конкретную строку). Однако во многих случаях
достаточно просто установить поле ``service_set`` в требуемые сервисы и позволить
драйверу заполнить массив ``service_lines`` в соответствии с возможностями
аппаратуры. Только если требуется более точное управление, программист должен
задать массив ``service_lines`` явно.

Ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>` изменяет параметры в соответствии с
возможностями аппаратуры. Когда драйвер выделяет ресурсы в этот момент, он может
вернуть код ошибки ``EBUSY``, если требуемые ресурсы временно недоступны. Другими
точками выделения ресурсов, которые могут вернуть ``EBUSY``, могут быть ioctl
:ref:`VIDIOC_STREAMON` и первый вызов :c:func:`read()`, :c:func:`write()` и
:c:func:`select()`.

Чтение и запись данных sliced VBI
=================================

Один вызов :c:func:`read()` или :c:func:`write()`
должен передавать все данные, принадлежащие одному видеокадру. Это массив
структур :c:type:`v4l2_sliced_vbi_data` с одним или
несколькими элементами и общим размером, не превышающим ``io_size`` байтов.
Аналогично в режиме потокового ввода-вывода один буфер размером ``io_size``
байтов должен содержать данные одного видеокадра. Поле ``id`` неиспользуемых
элементов структуры :c:type:`v4l2_sliced_vbi_data` должно быть нулевым.

    * - __u32
      - ``id``
      - Флаг из :ref:`vbi-services`, идентифицирующий тип данных в
	этом пакете. Должен быть установлен только один бит. Когда ``id``
	захваченного пакета равен нулю, пакет пуст, а содержимое
	других полей не определено. Приложения должны игнорировать пустые
	пакеты. Когда ``id`` пакета для вывода равен нулю,
	содержимое поля ``data`` не определено, и драйвер
	больше не должен вставлять данные на запрошенные ``field`` и ``line``.
    * - __u32
      - ``field``
      - Номер видеополя, из которого эти данные были захвачены или в которое
	они должны быть вставлены. ``0`` для первого поля, ``1`` для второго
	поля.
    * - __u32
      - ``line``
      - Номер строки поля (в отличие от кадра), из которой эти данные были
	захвачены или в которую они должны быть вставлены. См. :ref:`vbi-525` и
	:ref:`vbi-625` для допустимых значений. Устройства захвата sliced VBI
	могут установить номер строки всех пакетов в ``0``, если аппаратура не
	может надёжно идентифицировать строки развёртки. Номер поля всегда
	должен быть допустимым.
    * - __u32
      - ``reserved``
      - Это поле зарезервировано для будущих расширений. Приложения и
	драйверы должны установить его в ноль.
    * - __u8
      - ``data``\ [48]
      - Полезная нагрузка пакета. См. :ref:`vbi-services` для содержимого и
	числа байтов, передаваемых для каждого типа данных. Содержимое байтов
	заполнения в конце этого массива не определено, драйверы и
	приложения должны их игнорировать.

Пакеты всегда передаются в порядке возрастания номеров строк, без
повторяющихся номеров строк. Функция :c:func:`write()` и
ioctl :ref:`VIDIOC_QBUF` должны вернуть код ошибки ``EINVAL``,
когда приложения нарушают это правило. Они также должны вернуть код ошибки
EINVAL, когда приложения передают неправильный номер поля или строки,
либо комбинацию ``field``, ``line`` и ``id``, которая не была согласована с
ioctl :ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` или
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Когда номера строк
неизвестны, драйвер должен передавать пакеты в порядке передачи. Драйвер
может вставлять пустые пакеты с ``id``, установленным в ноль, в любое место
массива пакетов.

Чтобы обеспечить синхронизацию и отличить от выпадения кадров, когда
захваченный кадр не несёт ни один из запрошенных сервисов данных, драйверы
должны передать один или несколько пустых пакетов. Когда приложению не удаётся
передать данные VBI вовремя для вывода, драйвер должен снова вывести последний
пакет VPS и WSS, и отключить вывод данных Closed Caption и Teletext,
либо выводить данные, которые игнорируются декодерами Closed Caption и Teletext.

Устройство sliced VBI может поддерживать :ref:`чтение/запись <rw>` и/или
потоковый ввод-вывод (:ref:`отображение в память <mmap>` и/или
:ref:`пользовательский указатель <userp>`). Последний несёт возможность
синхронизации видео- и VBI-данных с помощью временных меток буферов.

Данные Sliced VBI в потоках MPEG
================================

Если устройство может производить выходной поток MPEG, оно может быть способно
предоставлять
:ref:`согласованные сервисы sliced VBI <sliced-vbi-format-negotiation>`
в виде данных, встроенных в поток MPEG. Пользователи или приложения управляют
этой вставкой данных sliced VBI с помощью элемента управления
:ref:`V4L2_CID_MPEG_STREAM_VBI_FMT <v4l2-mpeg-stream-vbi-fmt>`.

Если драйвер не предоставляет элемент управления
:ref:`V4L2_CID_MPEG_STREAM_VBI_FMT <v4l2-mpeg-stream-vbi-fmt>`
или позволяет устанавливать этот элемент управления только в
:ref:`V4L2_MPEG_STREAM_VBI_FMT_NONE <v4l2-mpeg-stream-vbi-fmt>`,
то устройство не может встраивать данные sliced VBI в поток MPEG.

Элемент управления
:ref:`V4L2_CID_MPEG_STREAM_VBI_FMT <v4l2-mpeg-stream-vbi-fmt>`
неявно не настраивает драйвер устройства ни на захват, ни на прекращение
захвата данных sliced VBI. Этот элемент управления лишь указывает встраивать
данные sliced VBI в поток MPEG, если приложение согласовало захват сервиса
sliced VBI.

Может также быть так, что устройство может встраивать данные sliced VBI только
в определённые типы потоков MPEG: например, в MPEG-2 PS, но не в MPEG-2 TS. В
этой ситуации, если запрошена вставка данных sliced VBI, данные sliced VBI будут
встраиваться в те типы потоков MPEG, где это поддерживается, и без уведомления
опускаться для тех типов потоков MPEG, где вставка данных sliced VBI устройством
не поддерживается.

В следующих подразделах задаётся формат встроенных данных sliced VBI.

Формат встроенных в поток MPEG данных Sliced VBI: NONE
------------------------------------------------------

Встроенный формат sliced VBI
:ref:`V4L2_MPEG_STREAM_VBI_FMT_NONE <v4l2-mpeg-stream-vbi-fmt>`
должен интерпретироваться драйверами как команда прекратить встраивание
данных sliced VBI в потоки MPEG. Ни устройство, ни драйвер не должны вставлять
«пустые» встроенные пакеты данных sliced VBI в поток MPEG, когда установлен этот
формат. Для этого формата никакие структуры данных потока MPEG не задаются.

Формат встроенных в поток MPEG данных Sliced VBI: IVTV
------------------------------------------------------

Встроенный формат sliced VBI
:ref:`V4L2_MPEG_STREAM_VBI_FMT_IVTV <v4l2-mpeg-stream-vbi-fmt>`,
когда поддерживается, указывает драйверу встраивать до 36 строк данных sliced
VBI на кадр в пакет MPEG-2 *Private Stream 1 PES*, инкапсулированный в MPEG-2
*Program Pack* в потоке MPEG.

*Исторический контекст*: данная спецификация формата происходит от
пользовательского встроенного формата данных sliced VBI, использовавшегося
драйвером ``ivtv``. Этот формат уже был неформально описан в исходных кодах ядра
в файле ``Documentation/userspace-api/media/drivers/cx2341x-uapi.rst`` .
Максимальный размер полезной нагрузки и другие аспекты этого формата обусловлены
возможностями и ограничениями MPEG-декодера CX23415 в отношении извлечения,
декодирования и отображения данных sliced VBI, встроенных в поток MPEG.

Использование этого формата *не* является исключительной принадлежностью
драйвера ``ivtv`` и *не* исключительно для устройств CX2341x, поскольку вставка
пакетов данных sliced VBI в поток MPEG реализуется программно в драйвере.
По меньшей мере драйвер ``cx18`` также предоставляет вставку данных sliced VBI в
MPEG-2 PS в этом формате.

Следующие определения задают полезную нагрузку пакетов MPEG-2 *Private
Stream 1 PES*, которые содержат данные sliced VBI, когда установлен
:ref:`V4L2_MPEG_STREAM_VBI_FMT_IVTV <v4l2-mpeg-stream-vbi-fmt>`.
(Заголовок пакета MPEG-2 *Private Stream 1 PES* и заголовок инкапсулирующего
MPEG-2 *Program Pack* здесь подробно не рассматриваются. Сведения об этих
заголовках пакетов см. в спецификациях MPEG-2.)

Полезная нагрузка пакетов MPEG-2 *Private Stream 1 PES*, которые содержат
данные sliced VBI, задаётся структурой
:c:type:`v4l2_mpeg_vbi_fmt_ivtv`. Полезная
нагрузка имеет переменную длину, зависящую от фактического числа строк данных
sliced VBI, присутствующих в видеокадре. Полезная нагрузка может быть дополнена
в конце неуказанными байтами-заполнителями для выравнивания её конца по границе
4 байтов. Полезная нагрузка никогда не должна превышать 1552 байта (2 поля
по 18 строк/поле с 43 байтами данных/строку и 4-байтовым магическим числом).

.. tabularcolumns:: |p{4.6cm}|p{2.0cm}|p{10.7cm}|

.. flat-table::
    :header-rows:  0
    :stub-columns: 0
    :widths:       1 1 2

    * - __le32
      - ``linemask``\ [2]
      - Битовые маски, указывающие присутствующие строки сервисов VBI. Эти
	значения ``linemask`` хранятся в формате little endian в
	потоке MPEG. Ниже приведены некоторые ориентировочные позиции битов
	``linemask`` с соответствующими им номерами строк VBI и видеополями.
	b\ :sub:`0` обозначает младший значащий бит значения ``linemask``:


	::

	    linemask[0] b0:     line  6  first field
	    linemask[0] b17:    line 23  first field
	    linemask[0] b18:    line  6  second field
	    linemask[0] b31:    line 19  second field
	    linemask[1] b0:     line 20  second field
	    linemask[1] b3:     line 23  second field
	    linemask[1] b4-b31: unused and set to 0
    * - struct
	:c:type:`v4l2_mpeg_vbi_itv0_line`
      - ``line``\ [35]
      - Это массив переменной длины, содержащий от 1 до 35 строк
	данных sliced VBI. Присутствующие строки данных sliced VBI соответствуют
	битам, установленным в массиве ``linemask``, начиная с b\ :sub:`0`
	``linemask``\ [0] вплоть до b\ :sub:`31` ``linemask``\ [0],
	и с b\ :sub:`0` ``linemask``\ [1] вплоть до b\ :sub:`3`
	``linemask``\ [1]. ``line``\ [0] соответствует первому
	найденному установленному биту в массиве ``linemask``, ``line``\ [1] соответствует
	второму найденному установленному биту в массиве ``linemask`` и т. д. Если ни один
	бит массива ``linemask`` не установлен, то ``line``\ [0] может содержать
	одну строку неуказанных данных, которую приложения должны
	игнорировать.
