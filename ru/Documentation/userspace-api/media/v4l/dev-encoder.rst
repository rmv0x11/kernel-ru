******************************************************************************
Интерфейс stateful-видеокодера типа «память-в-память» (Memory-to-Memory)
******************************************************************************

Stateful-видеокодер принимает кадры необработанного (raw) видео в порядке
отображения и кодирует их в поток байтов. Он формирует завершённые фрагменты
потока байтов, включая все метаданные, заголовки и т. п. Полученный поток байтов
не требует какой-либо дальнейшей постобработки со стороны клиента.

Выполнение программной обработки потока, генерации заголовков и т. п. в драйвере
ради поддержки этого интерфейса крайне не рекомендуется. Если подобные операции
необходимы, настоятельно рекомендуется использовать интерфейс stateless-видеокодера
(Stateless Video Encoder Interface, в разработке).

Соглашения и обозначения, используемые в этом документе
=======================================================

1. Общие правила V4L2 API применяются, если в этом документе не указано иное.

2. Значение слов «must», «may», «should» и т. п. соответствует `RFC
   2119 <https://tools.ietf.org/html/rfc2119>`_.

3. Все шаги, не помеченные как «необязательные», являются обязательными.

4. :c:func:`VIDIOC_G_EXT_CTRLS` и :c:func:`VIDIOC_S_EXT_CTRLS` могут
   использоваться взаимозаменяемо с :c:func:`VIDIOC_G_CTRL` и
   :c:func:`VIDIOC_S_CTRL`, если не указано иное.

5. Одноплоскостной (single-planar) API (см. :ref:`planar-apis`) и применимые
   структуры могут использоваться взаимозаменяемо с многоплоскостным
   (multi-planar) API, если не указано иное, в зависимости от возможностей
   кодера и в соответствии с общими рекомендациями V4L2.

6. i = [a..b]: последовательность целых чисел от a до b включительно, т. е. i =
   [0..2]: i = 0, 1, 2.

7. Для заданного буфера A очереди ``OUTPUT`` запись A' обозначает буфер очереди
   ``CAPTURE``, содержащий данные, полученные в результате обработки буфера A.

Опрос возможностей
==================

1. Чтобы перечислить набор кодированных форматов, поддерживаемых кодером,
   клиент может вызвать :c:func:`VIDIOC_ENUM_FMT` для ``CAPTURE``.

   * Будет возвращён полный набор поддерживаемых форматов независимо от
     формата, установленного на ``OUTPUT``.

2. Чтобы перечислить набор поддерживаемых необработанных (raw) форматов, клиент
   может вызвать :c:func:`VIDIOC_ENUM_FMT` для ``OUTPUT``.

   * Будут возвращены только форматы, поддерживаемые для формата, активного в
     данный момент на ``CAPTURE``.

   * Чтобы перечислить необработанные форматы, поддерживаемые для заданного
     кодированного формата, клиент должен сначала установить этот кодированный
     формат на ``CAPTURE``, а затем перечислить форматы на ``OUTPUT``.

3. Клиент может использовать :c:func:`VIDIOC_ENUM_FRAMESIZES` для определения
   поддерживаемых разрешений для заданного формата, передавая нужный формат
   пикселей в :c:type:`v4l2_frmsizeenum` ``pixel_format``.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMESIZES` для кодированного
     формата пикселей, будут включать все возможные кодированные разрешения,
     поддерживаемые кодером для заданного кодированного формата пикселей.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMESIZES` для необработанного
     формата пикселей, будут включать все возможные разрешения буфера кадра,
     поддерживаемые кодером для заданного необработанного формата пикселей и
     кодированного формата, установленного в данный момент на ``CAPTURE``.

4. Клиент может использовать :c:func:`VIDIOC_ENUM_FRAMEINTERVALS` для определения
   поддерживаемых интервалов между кадрами для заданного формата и разрешения,
   передавая нужный формат пикселей в :c:type:`v4l2_frmivalenum` ``pixel_format``
   и разрешение в :c:type:`v4l2_frmivalenum` ``width`` и :c:type:`v4l2_frmivalenum`
   ``height``.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMEINTERVALS` для кодированного
     формата пикселей и кодированного разрешения, будут включать все возможные
     интервалы между кадрами, поддерживаемые кодером для заданного кодированного
     формата пикселей и разрешения.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMEINTERVALS` для необработанного
     формата пикселей и разрешения, будут включать все возможные интервалы между
     кадрами, поддерживаемые кодером для заданного необработанного формата пикселей
     и разрешения, а также для кодированного формата, кодированного разрешения и
     кодированного интервала между кадрами, установленных в данный момент на
     ``CAPTURE``.

   * Поддержка :c:func:`VIDIOC_ENUM_FRAMEINTERVALS` необязательна. Если она не
     реализована, то нет никаких особых ограничений, кроме ограничений самого
     кодека.

5. Поддерживаемые профили и уровни для кодированного формата, установленного в
   данный момент на ``CAPTURE``, если применимо, можно запросить через
   соответствующие элементы управления с помощью :c:func:`VIDIOC_QUERYCTRL`.

6. Любые дополнительные возможности кодера можно обнаружить, запрашивая
   соответствующие им элементы управления.

Инициализация
=============

1. Установите кодированный формат на очереди ``CAPTURE`` с помощью
   :c:func:`VIDIOC_S_FMT`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``CAPTURE``.

     ``pixelformat``
         кодированный формат, который требуется получить.

     ``sizeimage``
         желаемый размер буферов ``CAPTURE``; кодер может скорректировать его
         в соответствии с требованиями оборудования.

     ``width``, ``height``
         игнорируются (только для чтения).

     другие поля
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``sizeimage``
         скорректированный размер буферов ``CAPTURE``.

     ``width``, ``height``
         кодированный размер, выбранный кодером на основе текущего состояния,
         например формата ``OUTPUT``, прямоугольников выборки (selection) и т. п.
         (только для чтения).

   .. important::

      Изменение формата ``CAPTURE`` может изменить установленный в данный момент
      формат ``OUTPUT``. То, как определяется новый формат ``OUTPUT``, остаётся
      на усмотрение кодера, и клиент должен впоследствии убедиться, что он
      соответствует его потребностям.

2. **Необязательно.** Перечислите поддерживаемые форматы ``OUTPUT`` (необработанные
   форматы источника) для выбранного кодированного формата с помощью
   :c:func:`VIDIOC_ENUM_FMT`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT``.

     другие поля
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``pixelformat``
         необработанный формат, поддерживаемый для кодированного формата,
         выбранного в данный момент на очереди ``CAPTURE``.

     другие поля
         следуют стандартной семантике.

3. Установите необработанный формат источника на очереди ``OUTPUT`` с помощью
   :c:func:`VIDIOC_S_FMT`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT``.

     ``pixelformat``
         необработанный формат источника.

     ``width``, ``height``
         разрешение источника.

     другие поля
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``width``, ``height``
         могут быть скорректированы в соответствии с минимальными, максимальными
         значениями и требованиями к выравниванию кодера, как того требуют
         выбранные в данный момент форматы, согласно сведениям, сообщаемым
         :c:func:`VIDIOC_ENUM_FRAMESIZES`.

     другие поля
         следуют стандартной семантике.

   * Установка формата ``OUTPUT`` сбросит прямоугольники выборки (selection) к их
     значениям по умолчанию на основе нового разрешения, как описано на следующем
     шаге.

4. Установите необработанный интервал между кадрами на очереди ``OUTPUT`` с
   помощью :c:func:`VIDIOC_S_PARM`. Это также устанавливает кодированный интервал
   между кадрами на очереди ``CAPTURE`` в то же значение.

   * **Обязательные поля:**

     ``type``
	 enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT``.

     ``parm.output``
	 установите все поля, кроме ``parm.output.timeperframe``, в 0.

     ``parm.output.timeperframe``
	 желаемый интервал между кадрами; кодер может скорректировать его в
	 соответствии с требованиями оборудования.

   * **Возвращаемые поля:**

     ``parm.output.timeperframe``
	 скорректированный интервал между кадрами.

   .. important::

      Изменение интервала между кадрами ``OUTPUT`` *также* задаёт частоту кадров,
      которую кодер использует для кодирования видео. Так, установка интервала
      между кадрами в 1/24 (или 24 кадра в секунду) даст кодированный
      видеопоток, который можно воспроизводить с этой скоростью. Интервал между
      кадрами для очереди ``OUTPUT`` является лишь подсказкой, приложение может
      предоставлять необработанные кадры с другой частотой. Драйвер может
      использовать его для помощи в планировании работы нескольких кодеров,
      выполняющихся параллельно.

      На следующем шаге интервал между кадрами ``CAPTURE`` можно при желании
      изменить на другое значение. Это полезно для офлайн-кодирования, когда
      кодированный интервал между кадрами может отличаться от частоты, с которой
      поставляются необработанные кадры.

   .. important::

      ``timeperframe`` оперирует *кадрами*, а не полями. Поэтому для чересстрочных
      (interlaced) форматов это время для двух полей, поскольку кадр состоит из
      верхнего и нижнего поля.

   .. note::

      По историческим причинам изменение интервала между кадрами ``OUTPUT`` также
      изменяет кодированный интервал между кадрами на очереди ``CAPTURE``. В
      идеале это были бы независимые настройки, но это нарушило бы существующий API.

5. **Необязательно.** Установите кодированный интервал между кадрами на очереди
   ``CAPTURE`` с помощью :c:func:`VIDIOC_S_PARM`. Это необходимо только в том
   случае, если кодированный интервал между кадрами отличается от необработанного
   интервала между кадрами, что типично для офлайн-кодирования. Поддержка этой
   возможности сигнализируется флагом формата
   :ref:`V4L2_FMT_FLAG_ENC_CAP_FRAME_INTERVAL <fmtdesc-flags>`.

   * **Обязательные поля:**

     ``type``
	 enum ``V4L2_BUF_TYPE_*``, соответствующий ``CAPTURE``.

     ``parm.capture``
	 установите все поля, кроме ``parm.capture.timeperframe``, в 0.

     ``parm.capture.timeperframe``
	 желаемый кодированный интервал между кадрами; кодер может скорректировать
	 его в соответствии с требованиями оборудования.

   * **Возвращаемые поля:**

     ``parm.capture.timeperframe``
	 скорректированный интервал между кадрами.

   .. important::

      Изменение интервала между кадрами ``CAPTURE`` задаёт частоту кадров для
      кодированного видео. Оно *не* задаёт частоту, с которой буферы поступают
      в очередь ``CAPTURE``; она зависит от того, насколько быстр кодер и
      насколько быстро необработанные кадры помещаются в очередь ``OUTPUT``.

   .. important::

      ``timeperframe`` оперирует *кадрами*, а не полями. Поэтому для чересстрочных
      (interlaced) форматов это время для двух полей, поскольку кадр состоит из
      верхнего и нижнего поля.

   .. note::

      Не все драйверы поддерживают эту функциональность; в этом случае просто
      установите желаемый кодированный интервал между кадрами для очереди
      ``OUTPUT``.

      Однако драйверы, способные планировать работу нескольких кодеров на основе
      интервала между кадрами ``OUTPUT``, должны поддерживать эту необязательную
      возможность.

6. **Необязательно.** Установите видимое разрешение для метаданных потока с
   помощью :c:func:`VIDIOC_S_SELECTION` на очереди ``OUTPUT``, если требуется,
   чтобы оно отличалось от полного разрешения OUTPUT.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT``.

     ``target``
         установите в ``V4L2_SEL_TGT_CROP``.

     ``r.left``, ``r.top``, ``r.width``, ``r.height``
         видимый прямоугольник; он должен помещаться внутри прямоугольника
         `V4L2_SEL_TGT_CROP_BOUNDS` и может быть скорректирован в соответствии с
         ограничениями кодека и оборудования.

   * **Возвращаемые поля:**

     ``r.left``, ``r.top``, ``r.width``, ``r.height``
         видимый прямоугольник, скорректированный кодером.

   * На ``OUTPUT`` поддерживаются следующие цели выборки (selection targets):

     ``V4L2_SEL_TGT_CROP_BOUNDS``
         равен полному кадру источника, соответствующему активному формату
         ``OUTPUT``.

     ``V4L2_SEL_TGT_CROP_DEFAULT``
         равен ``V4L2_SEL_TGT_CROP_BOUNDS``.

     ``V4L2_SEL_TGT_CROP``
         прямоугольник внутри буфера источника, кодируемый в поток ``CAPTURE``;
         по умолчанию равен ``V4L2_SEL_TGT_CROP_DEFAULT``.

         .. note::

            Распространённый сценарий использования этой цели выборки — кодирование
            исходного видео с разрешением, не кратным размеру макроблока, например
            распространённое разрешение 1920x1080 может требовать выравнивания
            буферов источника до 1920x1088 для кодеков с размером макроблока 16x16.
            Чтобы избежать кодирования заполнения (padding), клиенту нужно явно
            настроить эту цель выборки в 1920x1080.

   .. warning::

      Кодер может скорректировать прямоугольники обрезки/компоновки (crop/compose)
      к ближайшим поддерживаемым, чтобы удовлетворить требованиям кодека и
      оборудования. Клиенту нужно проверить скорректированный прямоугольник,
      возвращаемый :c:func:`VIDIOC_S_SELECTION`.

7. Выделите буферы для обеих очередей ``OUTPUT`` и ``CAPTURE`` с помощью
   :c:func:`VIDIOC_REQBUFS`. Это может быть выполнено в любом порядке.

   * **Обязательные поля:**

     ``count``
         запрашиваемое число буферов для выделения; больше нуля.

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT`` или
         ``CAPTURE``.

     другие поля
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``count``
          фактическое число выделенных буферов.

   .. warning::

      Фактическое число выделенных буферов может отличаться от заданного значения
      ``count``. Клиент должен проверить обновлённое значение ``count`` после
      возврата из вызова.

   .. note::

      Чтобы выделить больше, чем минимальное число буферов OUTPUT (для глубины
      конвейера), клиент может запросить элемент управления
      ``V4L2_CID_MIN_BUFFERS_FOR_OUTPUT``, чтобы получить минимально необходимое
      число буферов, и передать полученное значение плюс число дополнительно
      необходимых буферов в поле ``count`` функции :c:func:`VIDIOC_REQBUFS`.

   В качестве альтернативы для более полного контроля над выделением буферов
   можно использовать :c:func:`VIDIOC_CREATE_BUFS`.

   * **Обязательные поля:**

     ``count``
         запрашиваемое число буферов для выделения; больше нуля.

     ``type``
         enum ``V4L2_BUF_TYPE_*``, соответствующий ``OUTPUT``.

     другие поля
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``count``
         скорректировано до числа выделенных буферов.

8. Начните потоковую передачу на обеих очередях ``OUTPUT`` и ``CAPTURE`` с
   помощью :c:func:`VIDIOC_STREAMON`. Это может быть выполнено в любом порядке.
   Фактический процесс кодирования начинается, когда обе очереди начинают
   потоковую передачу.

.. note::

   Если клиент останавливает очередь ``CAPTURE`` в процессе кодирования, а затем
   снова её запускает, кодер начнёт генерировать поток, независимый от потока,
   сгенерированного до остановки. Точные ограничения зависят от кодированного
   формата, но могут включать следующие следствия:

   * кодированные кадры, произведённые после перезапуска, не должны ссылаться на
     какие-либо кадры, произведённые до остановки, например никаких долгосрочных
     ссылок (long term references) для H.264/HEVC,

   * любые заголовки, которые должны быть включены в самодостаточный поток,
     должны быть произведены заново, например SPS и PPS для H.264/HEVC.

Кодирование
===========

Это состояние достигается после успешного завершения последовательности
`Инициализация`. В этом состоянии клиент помещает и извлекает буферы в обеих
очередях с помощью :c:func:`VIDIOC_QBUF` и :c:func:`VIDIOC_DQBUF`, следуя
стандартной семантике.

Содержимое кодированных буферов ``CAPTURE`` зависит от активного кодированного
формата пикселей и может зависеть от расширенных элементов управления, специфичных
для кодека, как указано в документации каждого формата.

Обе очереди работают независимо, следуя стандартному поведению очередей буферов
V4L2 и устройств типа «память-в-память». Кроме того, порядок кодированных кадров,
извлекаемых из очереди ``CAPTURE``, может отличаться от порядка помещения
необработанных кадров в очередь ``OUTPUT`` из-за свойств выбранного кодированного
формата, например переупорядочивания кадров.

Клиент не должен предполагать какую-либо прямую связь между буферами ``CAPTURE``
и ``OUTPUT`` и какой-либо определённый момент готовности буферов к извлечению.
В частности:

* буфер, помещённый в ``OUTPUT``, может привести к появлению более чем одного
  буфера на ``CAPTURE`` (например, если возврат кодированного кадра позволил
  кодеру вернуть кадр, предшествовавший ему при отображении, но следовавший за
  ним в порядке декодирования; впрочем, для этого могут быть и другие причины),

* буфер, помещённый в ``OUTPUT``, может привести к появлению буфера на
  ``CAPTURE`` позже в процессе кодирования и/или после обработки дальнейших
  буферов ``OUTPUT``, либо быть возвращён не по порядку, например если
  используется переупорядочивание отображения,

* буферы могут становиться доступными на очереди ``CAPTURE`` без помещения
  дополнительных буферов в ``OUTPUT`` (например, во время дренажа (drain) или
  ``EOS``) из-за буферов ``OUTPUT``, помещённых в прошлом, результаты кодирования
  которых становятся доступны только позднее, в силу особенностей процесса
  кодирования,

* буферы, помещённые в ``OUTPUT``, могут не становиться доступными для извлечения
  мгновенно после их кодирования в соответствующий буфер ``CAPTURE``, например
  если кодеру нужно использовать кадр в качестве ссылки для кодирования дальнейших
  кадров.

.. note::

   Чтобы позволить сопоставлять кодированные буферы ``CAPTURE`` с буферами
   ``OUTPUT``, из которых они произошли, клиент может задать поле ``timestamp``
   структуры :c:type:`v4l2_buffer` при помещении буфера ``OUTPUT``. Буфер(ы)
   ``CAPTURE``, полученные в результате кодирования этого буфера ``OUTPUT``, будут
   иметь поле ``timestamp``, установленное в то же значение, при извлечении.

   Помимо прямого случая, когда один буфер ``OUTPUT`` производит один буфер
   ``CAPTURE``, определены следующие случаи:

   * один буфер ``OUTPUT`` генерирует несколько буферов ``CAPTURE``: одна и та же
     метка времени ``OUTPUT`` будет скопирована в несколько буферов ``CAPTURE``,

   * порядок кодирования отличается от порядка представления (т. е. буферы
     ``CAPTURE`` следуют не по порядку по сравнению с буферами ``OUTPUT``):
     метки времени ``CAPTURE`` не сохранят порядок меток времени ``OUTPUT``.

.. note::

   Чтобы позволить клиенту различать типы кадров (ключевые кадры, промежуточные
   кадры; точный список типов зависит от кодированного формата), буферы
   ``CAPTURE`` при извлечении будут иметь соответствующие биты флагов,
   установленные в их структуре :c:type:`v4l2_buffer`. См. документацию по
   :c:type:`v4l2_buffer` и по каждому кодированному формату пикселей для точного
   списка флагов и их значений.

Если возникнет ошибка кодирования, она будет сообщена клиенту с уровнем
детализации, зависящим от возможностей кодера. В частности:

* буфер ``CAPTURE`` (если он есть), содержащий результаты неудавшейся операции
  кодирования, будет возвращён с установленным флагом ``V4L2_BUF_FLAG_ERROR``,

* если кодер способен точно сообщить буфер(ы) ``OUTPUT``, вызвавшие ошибку, такой
  буфер(ы) будет возвращён с установленным флагом ``V4L2_BUF_FLAG_ERROR``.

.. note::

   Если буфер ``CAPTURE`` слишком мал, то он просто возвращается с установленным
   флагом ``V4L2_BUF_FLAG_ERROR``. Требуется дополнительная работа, чтобы
   определить, что эта ошибка возникла из-за слишком малого буфера, и обеспечить
   поддержку освобождения существующих буферов, которые оказались слишком малыми.

В случае фатального сбоя, не позволяющего продолжить кодирование, любые
дальнейшие операции с соответствующим файловым дескриптором кодера будут
возвращать код ошибки -EIO. Клиент может закрыть файловый дескриптор и открыть
новый или, в качестве альтернативы, повторно инициализировать экземпляр,
остановив потоковую передачу на обеих очередях, освободив все буферы и снова
выполнив последовательность инициализации.

Изменение параметров кодирования
================================

Клиенту разрешается использовать :c:func:`VIDIOC_S_CTRL` для изменения параметров
кодера в любое время. Доступность параметров специфична для кодера, и клиент
должен опросить кодер, чтобы выяснить набор доступных элементов управления.

Возможность изменения каждого параметра во время кодирования специфична для
кодера, в соответствии со стандартной семантикой интерфейса элементов управления
V4L2. Клиент может попытаться установить элемент управления во время кодирования,
и если операция завершится с кодом ошибки -EBUSY, для разрешения изменения
конфигурации необходимо остановить очередь ``CAPTURE``. Чтобы сделать это, он
может следовать последовательности `Дренаж` во избежание потери уже
помещённых/кодированных кадров.

Момент применения обновлений параметров специфичен для кодера, в соответствии со
стандартной семантикой интерфейса элементов управления V4L2. Если клиенту нужно
применить параметры точно на конкретном кадре, следует рассмотреть использование
Request API (:ref:`media-request-api`), если он поддерживается кодером.

Дренаж
======

Чтобы гарантировать, что все помещённые буферы ``OUTPUT`` были обработаны, а
связанные буферы ``CAPTURE`` переданы клиенту, клиент должен следовать описанной
ниже последовательности дренажа. После завершения последовательности дренажа
клиент получил все кодированные кадры для всех буферов ``OUTPUT``, помещённых до
начала последовательности.

1. Начните последовательность дренажа, выполнив :c:func:`VIDIOC_ENCODER_CMD`.

   * **Обязательные поля:**

     ``cmd``
         установите в ``V4L2_ENC_CMD_STOP``.

     ``flags``
         установите в 0.

     ``pts``
         установите в 0.

   .. warning::

      Последовательность может быть инициирована только если обе очереди
      ``OUTPUT`` и ``CAPTURE`` ведут потоковую передачу. По соображениям
      совместимости вызов :c:func:`VIDIOC_ENCODER_CMD` не завершится ошибкой,
      даже если какая-либо из очередей не ведёт потоковую передачу, но при этом
      он не инициирует последовательность `Дренаж`, и потому описанные ниже шаги
      будут неприменимы.

2. Любые буферы ``OUTPUT``, помещённые клиентом до выполнения
   :c:func:`VIDIOC_ENCODER_CMD`, будут обработаны и кодированы как обычно. Клиент
   должен продолжать обрабатывать обе очереди независимо, аналогично обычной
   операции кодирования. Это включает:

   * помещение и извлечение буферов ``CAPTURE`` до извлечения буфера, помеченного
     флагом ``V4L2_BUF_FLAG_LAST``,

     .. warning::

        Последний буфер может быть пустым (с :c:type:`v4l2_buffer`
        ``bytesused`` = 0), и в этом случае он должен игнорироваться клиентом,
        так как не содержит кодированного кадра.

     .. note::

        Любая попытка извлечь дополнительные буферы ``CAPTURE`` сверх буфера,
        помеченного ``V4L2_BUF_FLAG_LAST``, приведёт к ошибке -EPIPE от
        :c:func:`VIDIOC_DQBUF`.

   * извлечение обработанных буферов ``OUTPUT`` до извлечения всех буферов,
     помещённых до команды ``V4L2_ENC_CMD_STOP``,

   * извлечение события ``V4L2_EVENT_EOS``, если клиент подписан на него.

   .. note::

      Для обратной совместимости кодер будет сигнализировать событие
      ``V4L2_EVENT_EOS``, когда последний кадр был кодирован и все кадры готовы к
      извлечению. Это устаревшее поведение, и клиент не должен полагаться на него.
      Вместо этого следует использовать флаг буфера ``V4L2_BUF_FLAG_LAST``.

3. Как только все буферы ``OUTPUT``, помещённые до вызова ``V4L2_ENC_CMD_STOP``,
   извлечены и последний буфер ``CAPTURE`` извлечён, кодер останавливается и будет
   принимать, но не обрабатывать любые вновь помещённые буферы ``OUTPUT``, пока
   клиент не выполнит одну из следующих операций:

   * ``V4L2_ENC_CMD_START`` — кодер не будет сброшен и возобновит работу как
     обычно, со всем состоянием, имевшимся до дренажа,

   * пару :c:func:`VIDIOC_STREAMOFF` и :c:func:`VIDIOC_STREAMON` на очереди
     ``CAPTURE`` — кодер будет сброшен (см. последовательность `Сброс`) и затем
     возобновит кодирование,

   * пару :c:func:`VIDIOC_STREAMOFF` и :c:func:`VIDIOC_STREAMON` на очереди
     ``OUTPUT`` — кодер возобновит работу как обычно, однако любые кадры
     источника, помещённые в очередь ``OUTPUT`` между ``V4L2_ENC_CMD_STOP`` и
     :c:func:`VIDIOC_STREAMOFF`, будут отброшены.

.. note::

   После того как последовательность дренажа инициирована, клиент должен довести
   её до завершения, как описано в шагах выше, если только он не прерывает процесс
   выполнением :c:func:`VIDIOC_STREAMOFF` на любой из очередей ``OUTPUT`` или
   ``CAPTURE``. Клиенту не разрешается снова выполнять ``V4L2_ENC_CMD_START`` или
   ``V4L2_ENC_CMD_STOP``, пока последовательность дренажа выполняется; при попытке
   они завершатся с кодом ошибки -EBUSY.

   Для справки, обработка различных пограничных случаев описана ниже:

   * В случае отсутствия буфера в очереди ``OUTPUT`` на момент выполнения команды
     ``V4L2_ENC_CMD_STOP`` последовательность дренажа завершается немедленно, и
     кодер возвращает пустой буфер ``CAPTURE`` с установленным флагом
     ``V4L2_BUF_FLAG_LAST``.

   * В случае отсутствия буфера в очереди ``CAPTURE`` на момент завершения
     последовательности дренажа в следующий раз, когда клиент поместит буфер
     ``CAPTURE``, он будет возвращён сразу же как пустой буфер с установленным
     флагом ``V4L2_BUF_FLAG_LAST``.

   * Если :c:func:`VIDIOC_STREAMOFF` вызывается на очереди ``CAPTURE`` в середине
     последовательности дренажа, последовательность дренажа отменяется, и все
     буферы ``CAPTURE`` неявно возвращаются клиенту.

   * Если :c:func:`VIDIOC_STREAMOFF` вызывается на очереди ``OUTPUT`` в середине
     последовательности дренажа, последовательность дренажа завершается немедленно,
     и следующий буфер ``CAPTURE`` будет возвращён пустым с установленным флагом
     ``V4L2_BUF_FLAG_LAST``.

   Хотя это и необязательно, доступность команд кодера можно запросить с помощью
   :c:func:`VIDIOC_TRY_ENCODER_CMD`.

Сброс
=====

Клиент может захотеть запросить у кодера повторную инициализацию кодирования,
чтобы последующие данные потока стали независимыми от данных потока,
сгенерированных ранее. В зависимости от кодированного формата это может
подразумевать, что:

* кодированные кадры, произведённые после перезапуска, не должны ссылаться на
  какие-либо кадры, произведённые до остановки, например никаких долгосрочных
  ссылок (long term references) для H.264/HEVC,

* любые заголовки, которые должны быть включены в самодостаточный поток, должны
  быть произведены заново, например SPS и PPS для H.264/HEVC.

Это может быть достигнуто выполнением последовательности сброса.

1. Выполните последовательность `Дренаж`, чтобы гарантировать, что всё
   незавершённое кодирование завершено и соответствующие буферы извлечены.

2. Остановите потоковую передачу на очереди ``CAPTURE`` с помощью
   :c:func:`VIDIOC_STREAMOFF`. Это вернёт все помещённые в данный момент буферы
   ``CAPTURE`` клиенту без действительных данных кадров.

3. Начните потоковую передачу на очереди ``CAPTURE`` с помощью
   :c:func:`VIDIOC_STREAMON` и продолжайте обычную последовательность кодирования.
   Кодированные кадры, формируемые в буферы ``CAPTURE`` с этого момента, будут
   содержать самодостаточный поток, который можно декодировать без необходимости
   в кадрах, кодированных до последовательности сброса, начиная с первого буфера
   ``OUTPUT``, помещённого после выполнения `V4L2_ENC_CMD_STOP` последовательности
   `Дренаж`.

Эта последовательность может также использоваться для изменения параметров
кодирования для кодеров, не имеющих возможности изменять параметры на лету.

Точки фиксации
==============

Установка форматов и выделение буферов запускают изменения в поведении кодера.

1. Установка формата на очереди ``CAPTURE`` может изменить набор форматов,
   поддерживаемых/объявляемых на очереди ``OUTPUT``. В частности, это также
   означает, что формат ``OUTPUT`` может быть сброшен, и клиент не должен
   полагаться на сохранение ранее установленного формата.

2. Перечисление форматов на очереди ``OUTPUT`` всегда возвращает только форматы,
   поддерживаемые для текущего формата ``CAPTURE``.

3. Установка формата на очереди ``OUTPUT`` не изменяет список форматов, доступных
   на очереди ``CAPTURE``. Попытка установить формат ``OUTPUT``, который не
   поддерживается для выбранного в данный момент формата ``CAPTURE``, приведёт к
   тому, что кодер скорректирует запрошенный формат ``OUTPUT`` к поддерживаемому.

4. Перечисление форматов на очереди ``CAPTURE`` всегда возвращает полный набор
   поддерживаемых кодированных форматов, независимо от текущего формата ``OUTPUT``.

5. Пока буферы выделены на любой из очередей ``OUTPUT`` или ``CAPTURE``, клиент не
   должен изменять формат на очереди ``CAPTURE``. Драйверы будут возвращать код
   ошибки -EBUSY на любую такую попытку изменения формата.

Подводя итог, установка форматов и выделение всегда должны начинаться с очереди
``CAPTURE``, и очередь ``CAPTURE`` является главной (master), управляющей набором
поддерживаемых форматов для очереди ``OUTPUT``.
