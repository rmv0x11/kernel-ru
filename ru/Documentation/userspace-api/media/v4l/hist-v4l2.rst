***********************
Изменения в V4L2 API
***********************

Вскоре после добавления V4L API в ядро его раскритиковали как слишком
негибкий. В августе 1998 года Bill Dirks предложил ряд улучшений и начал
работу над документацией, примерами драйверов и приложений. С помощью
других добровольцев это в конечном итоге превратилось в V4L2 API — не просто
расширение, а замену V4L API. Однако потребовалось ещё четыре года и два
стабильных релиза ядра, прежде чем новый API был наконец принят для включения
в ядро в его нынешнем виде.

Ранние версии
=============

1998-08-20: Первая версия.

1998-08-27: Была введена функция :c:func:`select()`.

1998-09-10: Новый интерфейс видеостандартов.

1998-09-18: ioctl ``VIDIOC_NONCAP`` был заменён в остальном бессмысленным
флагом ``O_TRUNC`` функции :c:func:`open()`, и были определены псевдонимы
``O_NONCAP`` и ``O_NOIO``. Приложения могут установить этот флаг, если они
намерены обращаться только к элементам управления, в отличие от приложений
захвата, которым требуется монопольный доступ. Идентификаторы
``VIDEO_STD_XXX`` теперь являются порядковыми номерами, а не флагами, а
вспомогательная функция ``video_std_construct()`` принимает аргументы id и
transmission.

:ref:`VIDIOC_G_INPUT <VIDIOC_G_INPUT>` — в

Некоторые существенные изменения в struct v4l2_capability.

1998-11-14: ``V4L2_PIX_FMT_RGB24`` изменён на ``V4L2_PIX_FMT_BGR24``, а
``V4L2_PIX_FMT_RGB32`` изменён на ``V4L2_PIX_FMT_BGR32``. Аудиоконтролы

1998-11-28: Изменились несколько символов ioctl. Были добавлены интерфейсы для

V4L2 версии 0.20 (1999-09-10)
=============================

Версия 0.20 внесла ряд изменений, которые были *несовместимы с предыдущими*
версиями 0.19 и более ранними. Целью этих изменений было упрощение API при
одновременном повышении его расширяемости и следовании общим соглашениям API
драйверов Linux.

1. Исправлены некоторые опечатки в символах ``V4L2_FMT_FLAG``. struct
   v4l2_clip была изменена для совместимости с v4l. (1999-08-30)

2. Добавлен ``V4L2_TUNER_SUB_LANG1``. (1999-09-05)

3. Все команды ioctl(), использовавшие целочисленный аргумент, теперь
   принимают указатель на целое число. Там, где это имеет смысл, ioctl будут
   возвращать фактическое новое значение в целом числе, на которое указывает
   аргумент, — распространённое соглашение в V4L2 API. Затронутые ioctl:
   VIDIOC_PREVIEW, VIDIOC_STREAMON, VIDIOC_STREAMOFF, VIDIOC_S_FREQ,
   VIDIOC_S_INPUT, VIDIOC_S_OUTPUT, VIDIOC_S_EFFECT. Например

   .. code-block:: c

       err = ioctl (fd, VIDIOC_XXX, V4L2_XXX);

   становится

   .. code-block:: c

       int a = V4L2_XXX; err = ioctl(fd, VIDIOC_XXX, &a);

4. Все различные команды получения и установки формата были сведены в один
   ioctl :ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и
   :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`, принимающий в качестве параметра
   объединение и поле type, выбирающее член объединения. Цель — упростить API,
   устранив несколько ioctl, и разрешить новые и частные для драйвера потоки
   данных без добавления новых ioctl.

   Это изменение делает устаревшими следующие ioctl: ``VIDIOC_S_INFMT``,
   ``VIDIOC_G_INFMT``, ``VIDIOC_S_OUTFMT``, ``VIDIOC_G_OUTFMT``,
   ``VIDIOC_S_VBIFMT`` и ``VIDIOC_G_VBIFMT``. Структура формата изображения
   struct v4l2_format была переименована в struct v4l2_pix_format, тогда как
   struct v4l2_format теперь является охватывающей структурой для всех
   согласований форматов.

5. Аналогично изменениям выше, ioctl ``VIDIOC_G_PARM`` и ``VIDIOC_S_PARM``
   были объединены с ``VIDIOC_G_OUTPARM`` и ``VIDIOC_S_OUTPARM``. Поле
   ``type`` в новой struct v4l2_streamparm выбирает соответствующий член
   объединения.

   Это изменение делает устаревшими ioctl ``VIDIOC_G_OUTPARM`` и
   ``VIDIOC_S_OUTPARM``.

6. Перечисление элементов управления было упрощено, введены два новых флага
   элементов управления и один удалён. Поле ``catname`` было заменено полем
   ``group``.

   Драйверы теперь могут помечать неподдерживаемые и временно недоступные
   элементы управления флагами ``V4L2_CTRL_FLAG_DISABLED`` и
   ``V4L2_CTRL_FLAG_GRABBED`` соответственно. Имя ``group`` указывает на,
   возможно, более узкую классификацию, чем ``category``. Иными словами, в
   пределах одной категории может быть несколько групп. Элементы управления
   внутри группы обычно отрисовываются внутри группового блока (group box).
   Элементы управления в разных категориях могут иметь большее разделение или
   даже отображаться в отдельных окнах.

7. Поле ``timestamp`` структуры struct v4l2_buffer было изменено на 64-битное
   целое число, содержащее время выборки или вывода кадра в наносекундах.
   Кроме того, временные метки будут указываться в абсолютном системном
   времени, а не начинаться с нуля в начале потока. Имя типа данных для
   временных меток — stamp_t, определённое как знаковое 64-битное целое число.
   Устройства вывода не должны отправлять буфер, пока не наступит время,
   указанное в поле timestamp. Я хотел бы последовать примеру SGI и принять
   систему временных меток мультимедиа, подобную их UST (Unadjusted System
   Time). См.
   http://web.archive.org/web/\*/http://reality.sgi.com
   /cpirazzi_engr/lg/time/intro.html. UST использует временные метки, которые
   являются 64-битными знаковыми целыми числами (а не struct timeval) и
   задаются в наносекундах. Часы UST стартуют с нуля при загрузке системы и
   работают непрерывно и равномерно. Переполнение UST наступает чуть более чем
   через 292 года. Установить часы UST невозможно. Обычные часы времени суток
   Linux могут периодически изменяться, что привело бы к ошибкам, если бы они
   использовались для проставления временных меток мультимедийному потоку.
   Настоящие часы в стиле UST потребуют некоторой поддержки в ядре, которой там
   пока нет. Но в предвидении этого я изменю поле timestamp на 64-битное целое
   число и изменю функцию v4l2_masterclock_gettime() (используемую только
   драйверами), чтобы она возвращала 64-битное целое число.

8. В struct v4l2_buffer было добавлено поле ``sequence``. Поле ``sequence``
   считает захваченные кадры; оно игнорируется устройствами вывода. Когда
   драйвер захвата отбрасывает кадр, порядковый номер этого кадра пропускается.

V4L2 версии 0.20 2000-11-23
===========================

В интерфейс raw VBI был внесён ряд изменений.

1. В спецификацию V4L2 API были добавлены рисунки, поясняющие схему нумерации
   строк. Поля ``start``\ [0] и ``start``\ [1] больше не считают номера строк
   начиная с нуля. Обоснование: a) Предыдущее определение было неясным. b)
   Значения ``start``\ [] являются порядковыми числами. c) Нет смысла изобретать
   новую схему нумерации строк. Теперь мы используем номер строки, как он
   определён ITU-R, и точка. Совместимость: добавьте единицу к значениям start.
   Приложения, зависящие от предыдущей семантики, могут работать некорректно.

2. Ограничение "count[0] > 0 and count[1] > 0" было ослаблено до
   "(count[0] + count[1]) > 0". Обоснование: драйверы могут выделять ресурсы
   с гранулярностью строки развёртки, а некоторые сервисы данных передаются
   только в первом поле. Комментарий о том, что оба значения ``count`` обычно
   будут равны, вводит в заблуждение, бесполезен и был удалён. Это изменение
   *нарушает совместимость* с более ранними версиями: драйверы могут возвращать
   ``EINVAL``, приложения могут работать некорректно.

3. Драйверам снова разрешено возвращать отрицательные (неизвестные) значения
   start, как предлагалось ранее. Почему эта возможность была отброшена,
   неясно. Это изменение может *нарушить совместимость* с приложениями,
   зависящими от того, что значения start положительны. Использование кодов
   ошибок ``EBUSY`` и ``EINVAL`` с ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`
   было уточнено. Код ошибки ``EBUSY`` был наконец задокументирован, а также
   поле ``reserved2``, которое ранее упоминалось только в заголовочном файле
   ``videodev.h``.

4. Были добавлены новые типы буферов ``V4L2_TYPE_VBI_INPUT`` и
   ``V4L2_TYPE_VBI_OUTPUT``. Первый является псевдонимом для старого
   ``V4L2_TYPE_VBI``, второй отсутствовал в файле ``videodev.h``.

V4L2 в Linux 2.5.46, 2002-10
============================

Примерно в октябре-ноябре 2002 года, перед объявленной заморозкой
функциональности Linux 2.5, API был пересмотрен с учётом опыта работы с V4L2
0.20. Эта безымянная версия была в итоге влита в Linux 2.5.46.

1.  Как указано в :ref:`related`, драйверы должны делать связанные функции
    устройства доступными под всеми младшими номерами устройств.

2.  Функция :c:func:`open()` требует режима доступа ``O_RDWR`` независимо от
    типа устройства. Все драйверы V4L2, обменивающиеся данными с приложениями,
    должны поддерживать флаг ``O_NONBLOCK``. Флаг ``O_NOIO`` — символ V4L2,
    который служил псевдонимом бессмысленного ``O_TRUNC`` для обозначения
    доступа без обмена данными (приложения панели управления), — был отброшен.
    Драйверы должны оставаться в "режиме панели" до тех пор, пока приложение не
    попытается инициировать обмен данными, см. :ref:`open`.

3.  Структура struct v4l2_capability существенно изменилась. Обратите внимание,
    что также изменился размер структуры, который закодирован в коде запроса
    ioctl, поэтому более старые устройства V4L2 будут отвечать кодом ошибки
    ``EINVAL`` на новый ioctl :ref:`VIDIOC_QUERYCAP`.

    Появились новые поля для идентификации драйвера, новая функция устройства
    RDS ``V4L2_CAP_RDS_CAPTURE``, флаг ``V4L2_CAP_AUDIO`` указывает, имеет ли
    устройство какие-либо аудиоразъёмы, можно пометить ещё одну возможность
    ввода-вывода V4L2_CAP_ASYNCIO. В ответ на эти изменения поле ``type`` стало
    набором битов и было объединено с полем ``flags``. ``V4L2_FLAG_TUNER`` был
    переименован в ``V4L2_CAP_TUNER``, ``V4L2_CAP_VIDEO_OVERLAY`` заменил
    ``V4L2_FLAG_PREVIEW``, а ``V4L2_CAP_VBI_CAPTURE`` и
    ``V4L2_CAP_VBI_OUTPUT`` заменили ``V4L2_FLAG_DATA_SERVICE``.
    ``V4L2_FLAG_READ`` и ``V4L2_FLAG_WRITE`` были объединены в
    ``V4L2_CAP_READWRITE``.

    Избыточные поля ``inputs``, ``outputs`` и ``audios`` были удалены. Эти
    свойства можно определить, как описано в :ref:`video` и :ref:`audio`.

    Несколько изменчивые и потому едва ли полезные поля ``maxwidth``,
    ``maxheight``, ``minwidth``, ``minheight``, ``maxframerate`` были удалены.
    Эта информация доступна, как описано в :ref:`format` и :ref:`standard`.

    ``V4L2_FLAG_SELECT`` был удалён. Мы считаем, что функция select()
    достаточно важна, чтобы требовать её поддержки во всех драйверах V4L2,
    обменивающихся данными с приложениями. Избыточный флаг
    ``V4L2_FLAG_MONOCHROME`` был удалён, эта информация доступна, как описано в
    :ref:`format`.

4.  В struct v4l2_input поле ``assoc_audio``, поле ``capability`` и его
    единственный флаг ``V4L2_INPUT_CAP_AUDIO`` были заменены новым полем
    ``audioset``. Вместо связывания одного видеовхода с одним аудиовходом это
    поле сообщает обо всех аудиовходах, с которыми сочетается данный видеовход.

    Новые поля — ``tuner`` (обращающее прежнюю связь от тюнеров к видеовходам),
    ``std`` и ``status``.

    Соответственно struct v4l2_output потеряла поля ``capability`` и
    ``assoc_audio``. Вместо них были добавлены ``audioset``, ``modulator`` и
    ``std``.

5.  Поле ``audio`` структуры struct v4l2_audio было переименовано в ``index``
    для согласованности с другими структурами. Был добавлен новый флаг
    возможности ``V4L2_AUDCAP_STEREO``, указывающий, поддерживает ли данный
    аудиовход стереозвук. ``V4L2_AUDCAP_EFFECTS`` и соответствующие флаги
    ``V4L2_AUDMODE`` были удалены. Это легко реализуется с помощью элементов
    управления. (Однако то же относится к AVL, который всё ещё присутствует.)

    Снова для согласованности поле ``audio`` структуры struct v4l2_audioout
    было переименовано в ``index``.

6.  Поле ``input`` структуры struct v4l2_tuner было заменено полем ``index``,
    что позволяет иметь устройства с несколькими тюнерами. Связь между
    видеовходами и тюнерами теперь обращена: входы указывают на свой тюнер.
    Подструктура ``std`` стала простым набором (подробнее об этом ниже) и
    переместилась в struct v4l2_input. Было добавлено поле ``type``.

    Соответственно в struct v4l2_modulator поле ``output`` было заменено полем
    ``index``.

    В struct v4l2_frequency поле ``port`` было заменено полем ``tuner``,
    содержащим номер соответствующего индекса тюнера или модулятора. Было
    добавлено поле ``type`` тюнера, а поле ``reserved`` стало больше для будущих
    расширений (в частности, спутниковых тюнеров).

7.  Идея полностью прозрачных видеостандартов была отброшена. Опыт показал, что
    приложения должны уметь работать с видеостандартами не только для того,
    чтобы предъявить пользователю меню. Вместо перечисления поддерживаемых
    стандартов с помощью ioctl приложения теперь могут ссылаться на стандарты
    по :ref:`v4l2_std_id <v4l2-std-id>` и символам, определённым в заголовочном
    файле ``videodev2.h``. Подробности см. в :ref:`standard`.
    :ref:`VIDIOC_G_STD <VIDIOC_G_STD>` и :ref:`VIDIOC_S_STD <VIDIOC_G_STD>`
    теперь принимают указатель на этот тип в качестве аргумента.
    :ref:`VIDIOC_QUERYSTD` был добавлен для автоопределения принимаемого
    стандарта, если оборудование обладает такой возможностью. В struct
    v4l2_standard было добавлено поле ``index`` для :ref:`VIDIOC_ENUMSTD`. Было
    добавлено поле типа :ref:`v4l2_std_id <v4l2-std-id>` с именем ``id`` как
    машиночитаемый идентификатор, также заменяющий поле ``transmission``.
    Вводящее в заблуждение поле ``framerate`` было переименовано в
    ``frameperiod``. Теперь устаревшая информация ``colorstandard``,
    изначально нужная для различения вариаций стандартов, была удалена.

    Структура ``v4l2_enumstd`` прекратила своё существование.
    :ref:`VIDIOC_ENUMSTD` теперь принимает указатель непосредственно на struct
    v4l2_standard. Информация о том, какие стандарты поддерживаются конкретным
    видеовходом или видеовыходом, переместилась в поля struct v4l2_input и
    struct v4l2_output с именем ``std`` соответственно.

8.  Поля ``category`` и ``group`` структуры
    :ref:`v4l2_queryctrl <v4l2-queryctrl>` не прижились и/или были реализованы
    не так, как ожидалось, и поэтому были удалены.

9.  Был добавлен ioctl :ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` для согласования

    В struct v4l2_format объединение ``fmt`` было расширено для включения

    ioctl ``VIDIOC_G_WIN`` и ``VIDIOC_S_WIN`` для подготовки к видеооверлею

13. В struct v4l2_buffer поле ``type`` было корректно определено как
    enum v4l2_buf_type. Типы буферов изменились, как упомянуто выше. Было
    добавлено поле ``field`` типа enum v4l2_field, чтобы указывать, содержит ли
    буфер верхнее или нижнее поле. Старые флаги поля были удалены. Поскольку
    несмещаемые системные часы (unadjusted system time clock) в ядро добавлены
    не были, как планировалось, поле ``timestamp`` изменилось обратно с типа
    stamp_t, беззнакового 64-битного целого числа, выражающего время выборки в
    наносекундах, на struct timeval. С добавлением второго метода отображения
    памяти поле ``offset`` переместилось в объединение ``m``, и было добавлено
    новое поле ``memory`` типа enum v4l2_memory для различения методов
    ввода-вывода. Подробности см. в :ref:`io`.

    Флаг ``V4L2_BUF_REQ_CONTIG`` использовался слоем совместимости V4L; после
    изменений в этом коде он стал не нужен. Флаг ``V4L2_BUF_ATTR_DEVICEMEM``
    указывал бы, был ли буфер действительно выделен в памяти устройства, а не в
    доступной для DMA системной памяти. Он был едва полезен и потому был удалён.

    механизма синхронизации, был заменён единственным указателем. Флаги
    ``V4L2_FBUF_CAP_SCALEUP`` и ``V4L2_FBUF_CAP_SCALEDOWN`` были удалены.
    Приложения могут определить эту возможность точнее с помощью нового
    интерфейса обрезки и масштабирования. Флаг ``V4L2_FBUF_CAP_CLIPPING`` был
    заменён на ``V4L2_FBUF_CAP_LIST_CLIPPING`` и
    ``V4L2_FBUF_CAP_BITMAP_CLIPPING``.

15. В struct v4l2_clip поля ``x``, ``y``, ``width`` и ``height`` переместились
    в подструктуру ``c`` типа struct v4l2_rect. Поля ``x`` и ``y`` были
    переименованы в ``left`` и ``top``, т. е. смещения относительно зависящего
    от контекста начала координат.

16. В struct v4l2_window поля ``x``, ``y``, ``width`` и ``height``
    переместились в подструктуру ``w`` как выше. Было добавлено поле ``field``
    типа enum v4l2_field для различения полевого и кадрового (чересстрочного)
    оверлея.

17. Интерфейс цифрового зума, включая struct ``v4l2_zoomcap``, struct
    ``v4l2_zoom``, ``V4L2_ZOOM_NONCAP`` и ``V4L2_ZOOM_WHILESTREAMING``, был

    struct v4l2_cropcap и struct v4l2_crop были переопределены для этой цели.
    Подробности см. в :ref:`crop`.

18. В struct v4l2_vbi_format поле ``SAMPLE_FORMAT`` теперь содержит

    видеоизображений, а ``V4L2_PIX_FMT_GREY`` заменяет определение
    ``V4L2_VBI_SF_UBYTE``. Поле ``reserved`` было расширено.

19. В struct v4l2_captureparm тип поля ``timeperframe`` изменился с unsigned
    long на struct v4l2_fract. Это позволяет точно выражать кратные частоте
    кадров NTSC-M 30000 / 1001. Было добавлено новое поле ``readbuffers`` для
    управления поведением драйвера в режиме ввода-вывода чтения.

    Аналогичные изменения были внесены в struct v4l2_outputparm.

20. Структура ``v4l2_performance`` и ioctl ``VIDIOC_G_PERF`` были отброшены. За
    исключением случаев использования
    :ref:`метода ввода-вывода чтения/записи <rw>`, который в любом случае
    ограничен, эта информация уже доступна приложениям.

21. Пример преобразования из цветового пространства RGB в YCbCr в старой
    документации V4L2 был неточным; это было исправлено в :ref:`pixfmt`.

V4L2 2003-06-19
===============

1. Для радиоустройств был добавлен новый флаг возможности ``V4L2_CAP_RADIO``.
   До этого изменения радиоустройства идентифицировались исключительно по
   наличию ровно одного тюнера, чьё поле type читается как
   ``V4L2_TUNER_RADIO``.

2. Был добавлен необязательный механизм приоритетов доступа драйвера,
   подробности см. в :ref:`app-pri`.

3. Интерфейс аудиовхода и аудиовыхода был признан неполным.

   Ранее ioctl :ref:`VIDIOC_G_AUDIO <VIDIOC_G_AUDIO>` перечислял доступные
   аудиовходы. ioctl для определения текущего аудиовхода, если с текущим
   видеовходом сочетается более одного, не существовал. Поэтому
   ``VIDIOC_G_AUDIO`` был переименован в ``VIDIOC_G_AUDIO_OLD``; этот ioctl был
   удалён в ядре 2.6.39. Был добавлен ioctl :ref:`VIDIOC_ENUMAUDIO` для
   перечисления аудиовходов, тогда как :ref:`VIDIOC_G_AUDIO <VIDIOC_G_AUDIO>`
   теперь сообщает о текущем аудиовходе.

   Те же изменения были внесены в
   :ref:`VIDIOC_G_AUDOUT <VIDIOC_G_AUDOUT>` и
   :ref:`VIDIOC_ENUMAUDOUT <VIDIOC_ENUMAUDOUT>`.

   До дальнейших указаний модуль "videodev" будет автоматически транслировать
   между старыми и новыми ioctl, но драйверы и приложения необходимо обновить,
   чтобы они снова успешно компилировались.

4. ioctl :ref:`VIDIOC_OVERLAY` был неверно определён с параметром
   запись-чтение. Он был изменён на только-запись, тогда как версия
   запись-чтение была переименована в ``VIDIOC_OVERLAY_OLD``. Старый ioctl был
   удалён в ядре 2.6.39. До дальнейших указаний модуль ядра "videodev" будет
   автоматически транслировать в новую версию, поэтому драйверы нужно
   перекомпилировать, но не приложения.

5. В :ref:`overlay` неверно утверждалось, что прямоугольники обрезки
   определяют области, где видео видно. Правильно то, что прямоугольники
   обрезки определяют области, где *не* должно отображаться видео, благодаря
   чему видна графическая поверхность.

6. ioctl :ref:`VIDIOC_S_PARM <VIDIOC_G_PARM>` и
   :ref:`VIDIOC_S_CTRL <VIDIOC_G_CTRL>` были определены с параметром
   только-запись, что несогласуется с другими ioctl, изменяющими свой аргумент.
   Они были изменены на запись-чтение, тогда как к версиям только-запись был
   добавлен суффикс ``_OLD``. Старые ioctl были удалены в ядре 2.6.39. Драйверы
   и приложения, предполагающие постоянный параметр, нуждаются в обновлении.

Расширение X Video
------------------

Расширение X Video (сокращённо XVideo или просто Xv) — это расширение системы
X Window, реализованное, например, проектом XFree86. Его область применения
схожа с V4L2 — это API для устройств видеозахвата и видеовывода для клиентов X.
Xv позволяет приложениям отображать «живое» видео в окне, отправлять содержимое
окна на телевизионный выход и захватывать или выводить неподвижные изображения
в XPixmaps [#f1]_. Благодаря своей реализации XFree86 делает это расширение
доступным во многих операционных системах и архитектурах.

Поскольку драйвер встроен в X-сервер, Xv имеет ряд преимуществ перед
:ref:`интерфейсом видеооверлея <overlay>` V4L2. Драйвер может легко определить
цель оверлея, т. е. видимую графическую память или закадровые буферы для
деструктивного оверлея. Он может программировать RAMDAC для недеструктивного
оверлея, масштабирования или color-keying либо функции обрезки оборудования
видеозахвата, всегда синхронно с операциями рисования или перемещением окон
либо изменением порядка их наложения.

Чтобы объединить преимущества Xv и V4L, в XFree86 и XOrg существует
специальный драйвер Xv, который просто программирует любое найденное им
устройство Video4Linux, способное на оверлей. Чтобы включить его,
``/etc/X11/XF86Config`` должен содержать следующие строки:

::

    Section "Module"
	Load "v4l"
    EndSection

Начиная с XFree86 4.2, этот драйвер по-прежнему поддерживает только ioctl V4L,
однако он должен прекрасно работать со всеми устройствами V4L2 через слой
обратной совместимости V4L2. Поскольку V4L2 допускает несколько открытий, можно
(если это поддерживается драйвером V4L2) захватывать видео, пока клиент X
запросил видеооверлей. Действуют ограничения на одновременный захват и оверлей,
обсуждаемые в :ref:`overlay`.

Лишь косвенно связанное с V4L2, XFree86 расширило Xv для поддержки аппаратного
преобразования YUV в RGB и масштабирования для более быстрого воспроизведения
видео, а также добавило интерфейс к оборудованию декодирования MPEG-2. Этот API
полезен для отображения изображений, захваченных с помощью устройств V4L2.

Цифровое видео
--------------

V4L2 не поддерживает цифровое наземное, кабельное или спутниковое вещание.
Существует отдельный проект, ориентированный на цифровые приёмники. Его
домашнюю страницу можно найти по адресу
`https://linuxtv.org <https://linuxtv.org>`__. Linux DVB API не имеет
связи с V4L2 API, за исключением того, что драйверы для гибридного
оборудования могут поддерживать оба.

Устаревшие элементы API
=======================

Следующие элементы V4L2 API были вытеснены новыми интерфейсами и не должны
реализовываться в новых драйверах.

-  ioctl ``VIDIOC_G_MPEGCOMP`` и ``VIDIOC_S_MPEGCOMP``. Используйте расширенные
   элементы управления, :ref:`extended-controls`.

-  ioctl VIDIOC_G_DV_PRESET, VIDIOC_S_DV_PRESET, VIDIOC_ENUM_DV_PRESETS и
   VIDIOC_QUERY_DV_PRESET. Используйте DV Timings API (:ref:`dv-timings`).

-  ioctl ``VIDIOC_SUBDEV_G_CROP`` и ``VIDIOC_SUBDEV_S_CROP``. Используйте
   ``VIDIOC_SUBDEV_G_SELECTION`` и ``VIDIOC_SUBDEV_S_SELECTION``,
   :ref:`VIDIOC_SUBDEV_G_SELECTION`.

   Это не реализовано в XFree86.
