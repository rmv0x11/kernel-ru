*****************************
Аудиовходы и аудиовыходы
*****************************

Аудиовходы и аудиовыходы — это физические разъёмы устройства. Устройства
захвата видео имеют входы, устройства вывода — выходы, в количестве от
нуля и более каждого типа. Радиоустройства не имеют ни аудиовходов, ни
аудиовыходов. У них есть ровно один тюнер, который, по сути, *является*
источником аудио, однако этот API связывает тюнеры только с видеовходами
или видеовыходами, а у радиоустройств их нет. [#f1]_ Разъём на ТВ-плате,
предназначенный для возврата принятого аудиосигнала на звуковую плату, не
считается аудиовыходом.

Аудио- и видеовходы и выходы связаны между собой. Выбор источника видео
также выбирает источник аудио. Это наиболее очевидно, когда источником
видео и аудио является тюнер. Кроме того, аудиоразъёмы могут сочетаться с
более чем одним видеовходом или видеовыходом. Если предположить, что
существуют два композитных видеовхода и два аудиовхода, то возможно до
четырёх допустимых комбинаций. Связь видео- и аудиоразъёмов определяется
в поле ``audioset`` соответствующей структуры
:c:type:`v4l2_input` или структуры
:c:type:`v4l2_output`, где каждый бит представляет порядковый
номер, начиная с нуля, одного аудиовхода или аудиовыхода.

Чтобы узнать о количестве и атрибутах доступных входов и выходов,
приложения могут перечислить их с помощью ioctl
:ref:`VIDIOC_ENUMAUDIO` и
:ref:`VIDIOC_ENUMAUDOUT <VIDIOC_ENUMAUDOUT>` соответственно.
Структура :c:type:`v4l2_audio`, возвращаемая ioctl
:ref:`VIDIOC_ENUMAUDIO`, также содержит информацию о
состоянии сигнала, применимую при запросе текущего аудиовхода.

ioctl :ref:`VIDIOC_G_AUDIO <VIDIOC_G_AUDIO>` и
:ref:`VIDIOC_G_AUDOUT <VIDIOC_G_AUDOUT>` сообщают о текущем
аудиовходе и аудиовыходе соответственно.

.. note::

   Обратите внимание, что, в отличие от :ref:`VIDIOC_G_INPUT <VIDIOC_G_INPUT>` и
   :ref:`VIDIOC_G_OUTPUT <VIDIOC_G_OUTPUT>`, эти ioctl возвращают
   структуру, как это делают :ref:`VIDIOC_ENUMAUDIO` и
   :ref:`VIDIOC_ENUMAUDOUT <VIDIOC_ENUMAUDOUT>`, а не просто индекс.

Чтобы выбрать аудиовход и изменить его свойства, приложения вызывают ioctl
:ref:`VIDIOC_S_AUDIO <VIDIOC_G_AUDIO>`. Чтобы выбрать аудиовыход
(который в настоящее время не имеет изменяемых свойств), приложения вызывают
ioctl :ref:`VIDIOC_S_AUDOUT <VIDIOC_G_AUDOUT>`.

Драйверы должны реализовывать все ioctl аудиовхода, когда устройство имеет
несколько выбираемых аудиовходов, и все ioctl аудиовыхода, когда
устройство имеет несколько выбираемых аудиовыходов. Когда устройство имеет
какие-либо аудиовходы или аудиовыходы, драйвер должен установить флаг ``V4L2_CAP_AUDIO``
в структуре :c:type:`v4l2_capability`, возвращаемой
ioctl :ref:`VIDIOC_QUERYCAP`.


Пример: информация о текущем аудиовходе
=======================================

.. code-block:: c

    struct v4l2_audio audio;

    memset(&audio, 0, sizeof(audio));

    if (-1 == ioctl(fd, VIDIOC_G_AUDIO, &audio)) {
	perror("VIDIOC_G_AUDIO");
	exit(EXIT_FAILURE);
    }

    printf("Current input: %s\\n", audio.name);


Пример: переключение на первый аудиовход
========================================

.. code-block:: c

    struct v4l2_audio audio;

    memset(&audio, 0, sizeof(audio)); /* clear audio.mode, audio.reserved */

    audio.index = 0;

    if (-1 == ioctl(fd, VIDIOC_S_AUDIO, &audio)) {
	perror("VIDIOC_S_AUDIO");
	exit(EXIT_FAILURE);
    }

.. [#f1]
   На самом деле структура :c:type:`v4l2_audio` должна была бы иметь
   поле ``tuner``, как структура :c:type:`v4l2_input`, что не
   только сделало бы API более согласованным, но и позволило бы
   радиоустройствам иметь несколько тюнеров.
