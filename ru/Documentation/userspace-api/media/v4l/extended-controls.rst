************************************
API расширенных элементов управления
************************************


Введение
========

Механизм элементов управления, как он был задуман изначально, предназначался
для пользовательских настроек (яркость, насыщенность и т. д.). Однако
оказалось, что это очень удобная модель для реализации более сложных API
драйверов, где каждый драйвер реализует лишь подмножество более крупного API.

API кодирования MPEG стал движущей силой при проектировании и реализации этого
механизма расширенных элементов управления: стандарт MPEG довольно велик, и
поддерживаемые в настоящее время аппаратные MPEG-кодеры каждый реализуют лишь
подмножество этого стандарта. Более того, многие параметры, относящиеся к тому,
как видео кодируется в поток MPEG, специфичны для микросхемы MPEG-кодирования,
поскольку стандарт MPEG определяет только формат итогового потока MPEG, но не
то, как видео фактически кодируется в этот формат.

К сожалению, исходному API элементов управления недоставало некоторых функций,
необходимых для этих новых применений, поэтому он был расширен до (не слишком
оригинально названного) API расширенных элементов управления.

Хотя API кодирования MPEG был первой попыткой использовать API расширенных
элементов управления, в настоящее время существуют и другие классы расширенных
элементов управления, такие как элементы управления камерой (Camera Controls) и
элементы управления FM-передатчиком (FM Transmitter Controls). API расширенных
элементов управления, а также все классы расширенных элементов управления
описаны в следующем далее тексте.


API расширенного элемента управления
=====================================

Доступны три новых ioctl:
:ref:`VIDIOC_G_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>`,
:ref:`VIDIOC_S_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>` и
:ref:`VIDIOC_TRY_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>`. Эти ioctl действуют
на массивы элементов управления (в отличие от ioctl
:ref:`VIDIOC_G_CTRL <VIDIOC_G_CTRL>` и
:ref:`VIDIOC_S_CTRL <VIDIOC_G_CTRL>`, которые действуют на один
элемент управления). Это необходимо, поскольку часто требуется атомарно изменять
сразу несколько элементов управления.

Каждый из новых ioctl ожидает указатель на структуру
:c:type:`v4l2_ext_controls`. Эта структура
содержит указатель на массив элементов управления, количество элементов
управления в этом массиве и класс элементов управления. Классы элементов
управления используются для группировки схожих элементов управления в один
класс. Например, класс элементов управления
``V4L2_CTRL_CLASS_USER`` содержит все пользовательские элементы управления
(т. е. все элементы управления, которые также можно задать с помощью старого
ioctl :ref:`VIDIOC_S_CTRL <VIDIOC_G_CTRL>`). Класс элементов управления
``V4L2_CTRL_CLASS_CODEC`` содержит элементы управления, относящиеся к кодекам.

Все элементы управления в массиве элементов управления должны принадлежать
указанному классу элементов управления. Если это не так, возвращается ошибка.

Также можно использовать пустой массив элементов управления (``count`` == 0),
чтобы проверить, поддерживается ли указанный класс элементов управления.

Массив элементов управления — это массив структур
:c:type:`v4l2_ext_control`. Структура
:c:type:`v4l2_ext_control` очень похожа на
структуру :c:type:`v4l2_control`, за исключением того, что
она также позволяет передавать 64-битные значения и указатели.

Поскольку структура :c:type:`v4l2_ext_control` поддерживает
указатели, теперь также возможны элементы управления составных типов,
таких как N-мерные массивы и/или структуры. Чтобы действительно
иметь возможность видеть такие составные элементы управления, при перечислении
элементов управления необходимо указать
``V4L2_CTRL_FLAG_NEXT_COMPOUND``. Иными словами, эти элементы управления
составных типов следует использовать только программно.

Поскольку таким составным элементам управления нужно раскрывать о себе больше
информации, чем это возможно с помощью :ref:`VIDIOC_QUERYCTRL <VIDIOC_QUERYCTRL>`,
был добавлен ioctl :ref:`VIDIOC_QUERY_EXT_CTRL <VIDIOC_QUERYCTRL>`. В
частности, этот ioctl сообщает размерности N-мерного массива, если
этот элемент управления состоит более чем из одного элемента.

.. note::

   #. Важно осознавать, что из-за гибкости элементов управления необходимо
      проверять, действительно ли элемент управления, который вы хотите задать,
      поддерживается драйвером и каков допустимый диапазон значений. Поэтому
      используйте :ref:`VIDIOC_QUERYCTRL`, чтобы проверить это.

   #. Возможно, что некоторые индексы меню в элементе управления
      типа ``V4L2_CTRL_TYPE_MENU`` могут не поддерживаться (``VIDIOC_QUERYMENU``
      вернёт ошибку). Хорошим примером является список поддерживаемых
      MPEG-аудиобитрейтов. Некоторые драйверы поддерживают только один или два
      битрейта, другие поддерживают более широкий диапазон.

Все элементы управления используют машинный порядок байтов.


Перечисление расширенных элементов управления
==============================================

Рекомендуемый способ перечисления расширенных элементов управления — это
использование :ref:`VIDIOC_QUERYCTRL` в сочетании с
флагом ``V4L2_CTRL_FLAG_NEXT_CTRL``:


.. code-block:: c

    struct v4l2_queryctrl qctrl;

    qctrl.id = V4L2_CTRL_FLAG_NEXT_CTRL;
    while (0 == ioctl (fd, VIDIOC_QUERYCTRL, &qctrl)) {
	/* ... */
	qctrl.id |= V4L2_CTRL_FLAG_NEXT_CTRL;
    }

Начальный идентификатор элемента управления задаётся равным 0, объединённым по
ИЛИ с флагом ``V4L2_CTRL_FLAG_NEXT_CTRL``. Ioctl ``VIDIOC_QUERYCTRL``
вернёт первый элемент управления с идентификатором больше указанного. Когда
такие элементы управления не найдены, возвращается ошибка.

Если вы хотите получить все элементы управления в определённом классе элементов
управления, то можно задать начальное значение ``qctrl.id`` равным классу
элементов управления и добавить дополнительную проверку для выхода из цикла,
когда найден элемент управления другого класса элементов управления:


.. code-block:: c

    qctrl.id = V4L2_CTRL_CLASS_CODEC | V4L2_CTRL_FLAG_NEXT_CTRL;
    while (0 == ioctl(fd, VIDIOC_QUERYCTRL, &qctrl)) {
	if (V4L2_CTRL_ID2CLASS(qctrl.id) != V4L2_CTRL_CLASS_CODEC)
	    break;
	/* ... */
	qctrl.id |= V4L2_CTRL_FLAG_NEXT_CTRL;
    }

32-битное значение ``qctrl.id`` подразделяется на три битовых диапазона:
старшие 4 бита зарезервированы для флагов (например, ``V4L2_CTRL_FLAG_NEXT_CTRL``)
и фактически не являются частью идентификатора. Оставшиеся 28 бит образуют
идентификатор элемента управления, из которого старшие 12 бит определяют класс
элемента управления, а младшие 16 бит идентифицируют элемент управления внутри
класса элементов управления. Гарантируется, что эти последние 16 бит всегда
ненулевые для элементов управления. Диапазон 0x1000 и выше зарезервирован для
специфичных для драйвера элементов управления. Макрос ``V4L2_CTRL_ID2CLASS(id)``
возвращает идентификатор класса элемента управления на основе идентификатора
элемента управления.

Если драйвер не поддерживает расширенные элементы управления, то
``VIDIOC_QUERYCTRL`` завершится с ошибкой при использовании в сочетании с
``V4L2_CTRL_FLAG_NEXT_CTRL``. В этом случае следует использовать старый метод
перечисления элементов управления (см. :ref:`enum_all_controls`). Но если он
поддерживается, то гарантируется перечисление всех элементов управления,
включая частные элементы управления драйвера.


Создание панелей управления
============================

Можно создавать панели управления для графического пользовательского интерфейса,
где пользователь может выбирать различные элементы управления. По сути, вам
придётся перебирать все элементы управления с помощью метода, описанного выше.
Каждый класс элементов управления начинается с элемента управления типа
``V4L2_CTRL_TYPE_CTRL_CLASS``. ``VIDIOC_QUERYCTRL`` вернёт имя
этого класса элементов управления, которое может использоваться в качестве
заголовка вкладки внутри панели управления.

Поле flags структуры :ref:`v4l2_queryctrl <v4l2-queryctrl>` также
содержит подсказки о поведении элемента управления. Дополнительные сведения см.
в документации :ref:`VIDIOC_QUERYCTRL`.
