*************************************************************
Интерфейс stateful-видеодекодера типа memory-to-memory
*************************************************************

Stateful-видеодекодер принимает на вход целостные фрагменты потока байтов
(например, поток H.264/HEVC в формате Annex-B, «сырой» поток VP8/9) и
декодирует их в «сырые» кадры видео в порядке отображения. Предполагается, что
для обработки этих буферов декодеру не требуется никакой дополнительной
информации от клиента.

Выполнение программного разбора, обработки и т. п. потока в драйвере ради
поддержки этого интерфейса настоятельно не рекомендуется. Если такие операции
необходимы, настоятельно советуется использовать интерфейс stateless-видеодекодера
(в разработке).

Соглашения и обозначения, используемые в этом документе
=======================================================

1. Если в этом документе не указано иное, действуют общие правила API V4L2.

2. Значение слов «must», «may», «should» и т. п. соответствует `RFC
   2119 <https://tools.ietf.org/html/rfc2119>`_.

3. Все шаги, не помеченные как «optional» (необязательные), являются обязательными.

4. :c:func:`VIDIOC_G_EXT_CTRLS` и :c:func:`VIDIOC_S_EXT_CTRLS` могут
   использоваться взаимозаменяемо с :c:func:`VIDIOC_G_CTRL` и
   :c:func:`VIDIOC_S_CTRL`, если не указано иное.

5. Одноплоскостной (single-planar) API (см. :ref:`planar-apis`) и применимые к
   нему структуры могут использоваться взаимозаменяемо с многоплоскостным
   (multi-planar) API, если не указано иное, в зависимости от возможностей
   декодера и в соответствии с общими рекомендациями V4L2.

6. i = [a..b]: последовательность целых чисел от a до b включительно, т. е. i =
   [0..2]: i = 0, 1, 2.

7. Для заданного буфера A очереди ``OUTPUT`` обозначение A' представляет буфер
   очереди ``CAPTURE``, содержащий данные, полученные в результате обработки
   буфера A.

Глоссарий
=========

CAPTURE
   очередь буферов-приёмников; для декодеров — очередь буферов, содержащих
   декодированные кадры; для энкодеров — очередь буферов, содержащих
   кодированный поток байтов; ``V4L2_BUF_TYPE_VIDEO_CAPTURE`` или
   ``V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE``; данные захватываются из аппаратуры в
   буферы ``CAPTURE``.

client
   приложение, взаимодействующее с декодером или энкодером, реализующим
   данный интерфейс.

coded format
   формат кодированного/сжатого потока байтов видео (например, H.264, VP8 и
   т. п.); см. также: raw format.

coded height
   высота для заданного coded resolution.

coded resolution
   разрешение потока в пикселях, выровненное по требованиям кодека и
   аппаратуры; обычно это visible resolution, округлённое вверх до полных
   макроблоков; см. также: visible resolution.

coded width
   ширина для заданного coded resolution.

coding tree unit
   единица обработки кодека HEVC (соответствует макроблочным единицам в
   H.264, VP8, VP9),
   может использовать блочные структуры размером до 64×64 пикселей.
   Хорошо справляется с разбиением изображения на структуры переменного размера.

decode order
   порядок, в котором декодируются кадры; может отличаться от display order,
   если coded format включает возможность переупорядочивания кадров; для
   декодеров буферы ``OUTPUT`` должны помещаться клиентом в очередь в порядке
   decode order; для энкодеров буферы ``CAPTURE`` должны возвращаться энкодером
   в порядке decode order.

destination
   данные, полученные в результате процесса декодирования; см. ``CAPTURE``.

display order
   порядок, в котором должны отображаться кадры; для энкодеров буферы
   ``OUTPUT`` должны помещаться клиентом в очередь в порядке display order; для
   декодеров буферы ``CAPTURE`` должны возвращаться декодером в порядке
   display order.

DPB
   Decoded Picture Buffer; термин H.264/HEVC, обозначающий буфер, в котором
   хранится декодированный «сырой» кадр, доступный для использования в качестве
   опорного на дальнейших этапах декодирования.

EOS
   end of stream (конец потока).

IDR
   Instantaneous Decoder Refresh; разновидность ключевого кадра (keyframe) в
   потоке, закодированном H.264/HEVC, которая очищает список более ранних
   опорных кадров (DPB).

keyframe
   кодированный кадр, который не ссылается на кадры, декодированные ранее, т. е.
   может быть декодирован полностью самостоятельно.

macroblock
   единица обработки в форматах сжатия изображений и видео, основанных на
   линейных блочных преобразованиях (например, H.264, VP8, VP9); зависит от
   кодека, но для большинства популярных кодеков размер составляет 16x16
   отсчётов (пикселей). Кодек HEVC использует несколько более гибкую единицу
   обработки, называемую coding tree unit (CTU).

OUTPUT
   очередь буферов-источников; для декодеров — очередь буферов, содержащих
   кодированный поток байтов; для энкодеров — очередь буферов, содержащих
   «сырые» кадры; ``V4L2_BUF_TYPE_VIDEO_OUTPUT`` или
   ``V4L2_BUF_TYPE_VIDEO_OUTPUT_MPLANE``; аппаратура получает данные из
   буферов ``OUTPUT``.

PPS
   Picture Parameter Set; разновидность сущности метаданных в потоке байтов H.264/HEVC.

raw format
   несжатый формат, содержащий «сырые» пиксельные данные (например, форматы YUV, RGB).

resume point
   точка в потоке байтов, с которой можно начать/продолжить декодирование без
   наличия какого-либо предыдущего состояния/данных, например: ключевой кадр
   (keyframe) (VP8/VP9) или последовательность SPS/PPS/IDR (H.264/HEVC);
   resume point необходим для начала декодирования нового потока или для
   возобновления декодирования после перемотки (seek).

source
   данные, подаваемые на декодер или энкодер; см. ``OUTPUT``.

source height
   высота в пикселях для заданного source resolution; актуально только для энкодеров.

source resolution
   разрешение в пикселях исходных кадров, подаваемых на энкодер и подлежащих
   дальнейшей обрезке до границ visible resolution; актуально только для
   энкодеров.

source width
   ширина в пикселях для заданного source resolution; актуально только для энкодеров.

SPS
   Sequence Parameter Set; разновидность сущности метаданных в потоке байтов H.264/HEVC.

stream metadata
   дополнительная (невизуальная) информация, содержащаяся внутри кодированного
   потока байтов; например: coded resolution, visible resolution, профиль кодека.

visible height
   высота для заданного visible resolution; высота отображения.

visible resolution
   разрешение потока для видимого изображения, в пикселях, используемое для
   целей отображения; должно быть меньше или равно coded resolution;
   разрешение отображения.

visible width
   ширина для заданного visible resolution; ширина отображения.

Запрос возможностей
===================

1. Чтобы перечислить набор coded format, поддерживаемых декодером, клиент
   может вызвать :c:func:`VIDIOC_ENUM_FMT` на ``OUTPUT``.

   * Будет возвращён полный набор поддерживаемых форматов, независимо от
     формата, установленного на ``CAPTURE``.
   * Проверьте поле flags структуры :c:type:`v4l2_fmtdesc` для получения
     дополнительной информации о возможностях декодера применительно к каждому
     coded format. В частности, о том, имеет ли декодер полноценный анализатор
     (parser) потока байтов и поддерживает ли декодер динамическое изменение
     разрешения.

2. Чтобы перечислить набор поддерживаемых raw format, клиент может вызвать
   :c:func:`VIDIOC_ENUM_FMT` на ``CAPTURE``.

   * Будут возвращены только форматы, поддерживаемые для формата, активного в
     данный момент на ``OUTPUT``.

   * Чтобы перечислить raw format, поддерживаемые для заданного coded format,
     клиент должен сначала установить этот coded format на ``OUTPUT``, а затем
     перечислить форматы на ``CAPTURE``.

3. Клиент может использовать :c:func:`VIDIOC_ENUM_FRAMESIZES` для определения
   поддерживаемых разрешений для заданного формата, передавая желаемый
   пиксельный формат в :c:type:`v4l2_frmsizeenum` ``pixel_format``.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMESIZES` для кодированного
     пиксельного формата, будут включать все возможные coded resolution,
     поддерживаемые декодером для заданного кодированного пиксельного формата.

   * Значения, возвращаемые :c:func:`VIDIOC_ENUM_FRAMESIZES` для «сырого»
     пиксельного формата, будут включать все возможные разрешения буфера кадра
     (frame buffer), поддерживаемые декодером для заданного «сырого» пиксельного
     формата и coded format, установленного в данный момент на ``OUTPUT``.

4. Поддерживаемые профили и уровни для coded format, установленного в данный
   момент на ``OUTPUT``, если применимо, можно запросить с помощью
   соответствующих элементов управления (controls) через
   :c:func:`VIDIOC_QUERYCTRL`.

Инициализация
=============

1. Установите coded format на ``OUTPUT`` через :c:func:`VIDIOC_S_FMT`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

     ``pixelformat``
         кодированный пиксельный формат.

     ``width``, ``height``
         coded resolution потока; требуется только в том случае, если оно не
         может быть разобрано из потока для заданного coded format; иначе декодер
         будет использовать это разрешение как разрешение-заполнитель, которое,
         скорее всего, изменится, как только декодер сможет разобрать
         фактическое coded resolution из потока.

     ``sizeimage``
         желаемый размер буферов ``OUTPUT``; декодер может скорректировать его в
         соответствии с требованиями аппаратуры.

     other fields
         следуют стандартной семантике.

   * **Возвращаемые поля:**

     ``sizeimage``
         скорректированный размер буферов ``OUTPUT``.

   * Формат ``CAPTURE`` будет мгновенно обновлён соответствующим разрешением
     буфера кадра на основе ширины и высоты, возвращённых
     :c:func:`VIDIOC_S_FMT`.
     Однако для coded format, включающих информацию о разрешении потока,
     после того как декодер завершит разбор этой информации из потока, он
     обновит формат ``CAPTURE`` новыми значениями и подаст событие об изменении
     источника (source change), независимо от того, совпадают ли они со
     значениями, установленными клиентом, или нет.

   .. important::

      Изменение формата ``OUTPUT`` может изменить установленный в данный момент
      формат ``CAPTURE``. То, как определяется новый формат ``CAPTURE``,
      зависит от декодера, и клиент должен впоследствии убедиться, что он
      соответствует его потребностям.

2.  Выделите буферы-источники (потока байтов) через :c:func:`VIDIOC_REQBUFS` на
    ``OUTPUT``.

    * **Обязательные поля:**

      ``count``
          запрашиваемое количество выделяемых буферов; больше нуля.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

      ``memory``
          следует стандартной семантике.

    * **Возвращаемые поля:**

      ``count``
          фактическое количество выделенных буферов.

    .. warning::

       Фактическое количество выделенных буферов может отличаться от заданного
       значения ``count``. Клиент должен проверить обновлённое значение
       ``count`` после возврата из вызова.

    Как вариант, для большего контроля над выделением буферов можно
    использовать :c:func:`VIDIOC_CREATE_BUFS` на очереди ``OUTPUT``.

    * **Обязательные поля:**

      ``count``
          запрашиваемое количество выделяемых буферов; больше нуля.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

      ``memory``
          следует стандартной семантике.

      ``format``
          следует стандартной семантике.

    * **Возвращаемые поля:**

      ``count``
          скорректировано до количества выделенных буферов.

    .. warning::

       Фактическое количество выделенных буферов может отличаться от заданного
       значения ``count``. Клиент должен проверить обновлённое значение
       ``count`` после возврата из вызова.

3.  Запустите потоковую передачу на очереди ``OUTPUT`` через :c:func:`VIDIOC_STREAMON`.

4.  **Этот шаг применим только к coded format, содержащим информацию о
    разрешении в потоке.** Продолжайте помещать в очередь / извлекать из очереди
    буферы потока байтов в/из очереди ``OUTPUT`` через :c:func:`VIDIOC_QBUF` и
    :c:func:`VIDIOC_DQBUF`. Буферы будут обработаны и возвращены клиенту по
    порядку, пока не будут найдены метаданные, требуемые для настройки очереди
    ``CAPTURE``. Об этом сигнализирует декодер, отправляя событие
    ``V4L2_EVENT_SOURCE_CHANGE`` с полем ``changes``, установленным в
    ``V4L2_EVENT_SRC_CH_RESOLUTION``.

    * Не является ошибкой, если первый буфер не содержит достаточно данных для
      того, чтобы это произошло. Обработка буферов будет продолжаться до тех
      пор, пока требуется больше данных.

    * Если данные в буфере, который вызывает это событие, требуются для
      декодирования первого кадра, такой буфер не будет возвращён клиенту, пока
      не завершится последовательность инициализации и не будет декодирован кадр.

    * Если клиент не установил coded resolution потока самостоятельно, вызов
      :c:func:`VIDIOC_G_FMT`, :c:func:`VIDIOC_S_FMT`,
      :c:func:`VIDIOC_TRY_FMT` или :c:func:`VIDIOC_REQBUFS` на очереди
      ``CAPTURE`` не вернёт реальные значения для потока до тех пор, пока не
      будет подано событие ``V4L2_EVENT_SOURCE_CHANGE`` с полем ``changes``,
      установленным в ``V4L2_EVENT_SRC_CH_RESOLUTION``.

    .. important::

       Любой запрос клиента, выданный после того, как декодер поместил событие в
       очередь, вернёт значения, применимые к только что разобранному потоку,
       включая форматы очередей, прямоугольники выбора (selection rectangles) и
       элементы управления.

    .. note::

       Клиент, способный получать параметры потока из потока байтов
       самостоятельно, может попытаться установить ширину и высоту формата
       ``OUTPUT`` в ненулевые значения, соответствующие coded size потока,
       пропустить этот шаг и продолжить с последовательности `Capture Setup`_.
       Однако он не должен полагаться ни на какие запросы к драйверу
       относительно параметров потока, таких как прямоугольники выбора и
       элементы управления, поскольку декодер ещё не разобрал их из потока. Если
       значения, заданные клиентом, не совпадают с разобранными декодером, будет
       запущена `Dynamic Resolution Change`_, чтобы перенастроить их.

    .. note::

       На этом этапе декодированные кадры не производятся.

5.  Продолжите с последовательности `Capture Setup`_.

Capture Setup
=============

1.  Вызовите :c:func:`VIDIOC_G_FMT` на очереди ``CAPTURE``, чтобы получить
    формат для буферов-приёмников, разобранных/декодированных из потока байтов.

    * **Обязательные поля:**

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

    * **Возвращаемые поля:**

      ``width``, ``height``
          разрешение буфера кадра для декодированных кадров.

      ``pixelformat``
          пиксельный формат для декодированных кадров.

      ``num_planes`` (только для ``type`` вида _MPLANE)
          количество плоскостей для pixelformat.

      ``sizeimage``, ``bytesperline``
          согласно стандартной семантике; соответствуют формату буфера кадра.

    .. note::

       Значение ``pixelformat`` может быть любым пиксельным форматом,
       поддерживаемым декодером для текущего потока. Декодер должен выбрать
       предпочтительный/оптимальный формат для конфигурации по умолчанию.
       Например, формат YUV может быть предпочтительнее формата RGB, если для
       последнего потребовался бы дополнительный шаг преобразования.

2.  **Необязательно.** Получите visible resolution через
    :c:func:`VIDIOC_G_SELECTION`.

    * **Обязательные поля:**

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``target``
          установлен в ``V4L2_SEL_TGT_COMPOSE``.

    * **Возвращаемые поля:**

      ``r.left``, ``r.top``, ``r.width``, ``r.height``
          видимый прямоугольник; он должен помещаться в пределах разрешения
          буфера кадра, возвращённого :c:func:`VIDIOC_G_FMT` на ``CAPTURE``.

    * На ``CAPTURE`` поддерживаются следующие цели выбора (selection targets):

      ``V4L2_SEL_TGT_CROP_BOUNDS``
          соответствует coded resolution потока.

      ``V4L2_SEL_TGT_CROP_DEFAULT``
          прямоугольник, покрывающий ту часть буфера ``CAPTURE``, которая
          содержит осмысленные данные изображения (видимую область); ширина и
          высота будут равны visible resolution потока.

      ``V4L2_SEL_TGT_CROP``
          прямоугольник в пределах coded resolution, который будет выведен в
          ``CAPTURE``; по умолчанию равен ``V4L2_SEL_TGT_CROP_DEFAULT``;
          доступен только для чтения на аппаратуре без дополнительных
          возможностей композиции/масштабирования.

      ``V4L2_SEL_TGT_COMPOSE_BOUNDS``
          максимальный прямоугольник в пределах буфера ``CAPTURE``, в который
          может быть скомпонован обрезанный кадр; равен ``V4L2_SEL_TGT_CROP``,
          если аппаратура не поддерживает композицию/масштабирование.

      ``V4L2_SEL_TGT_COMPOSE_DEFAULT``
          равен ``V4L2_SEL_TGT_CROP``.

      ``V4L2_SEL_TGT_COMPOSE``
          прямоугольник внутри буфера ``CAPTURE``, в который записывается
          обрезанный кадр; по умолчанию равен ``V4L2_SEL_TGT_COMPOSE_DEFAULT``;
          доступен только для чтения на аппаратуре без дополнительных
          возможностей композиции/масштабирования.

      ``V4L2_SEL_TGT_COMPOSE_PADDED``
          прямоугольник внутри буфера ``CAPTURE``, который перезаписывается
          аппаратурой; равен ``V4L2_SEL_TGT_COMPOSE``, если аппаратура не
          записывает дополняющие (padding) пиксели.

    .. warning::

       Эти значения гарантированно осмысленны только после того, как декодер
       успешно разобрал stream metadata. Клиент не должен полагаться на этот
       запрос до того, как это произойдёт.

3.  **Необязательно.** Перечислите форматы ``CAPTURE`` через
    :c:func:`VIDIOC_ENUM_FMT` на очереди ``CAPTURE``. Как только информация о
    потоке разобрана и известна, клиент может использовать этот ioctl, чтобы
    выяснить, какие raw format поддерживаются для заданного потока, и выбрать
    один из них через :c:func:`VIDIOC_S_FMT`.

    .. important::

       Декодер вернёт только форматы, поддерживаемые для установленного в данный
       момент coded format, согласно формату ``OUTPUT`` и/или stream metadata,
       разобранным в этой последовательности инициализации, даже если декодер в
       целом может поддерживать больше форматов. Иными словами, возвращаемый
       набор будет подмножеством начального запроса, упомянутого в разделе
       `Запрос возможностей`_.

       Например, декодер может поддерживать форматы YUV и RGB для разрешений
       1920x1088 и ниже, но только YUV для более высоких разрешений (из-за
       аппаратных ограничений). После разбора разрешения 1920x1088 или ниже
       :c:func:`VIDIOC_ENUM_FMT` может вернуть набор пиксельных форматов YUV и
       RGB, но после разбора разрешения выше 1920x1088 декодер не вернёт RGB,
       не поддерживаемый для этого разрешения.

       Однако последующее событие изменения разрешения, запущенное после
       обнаружения изменения разрешения внутри того же потока, может
       переключить поток на более низкое разрешение, и в этом случае
       :c:func:`VIDIOC_ENUM_FMT` снова вернёт форматы RGB.

4.  **Необязательно.** Установите формат ``CAPTURE`` через :c:func:`VIDIOC_S_FMT`
    на очереди ``CAPTURE``. Клиент может выбрать формат, отличный от
    выбранного/предложенного декодером в :c:func:`VIDIOC_G_FMT`.

    * **Обязательные поля:**

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``pixelformat``
          «сырой» пиксельный формат.

      ``width``, ``height``
         разрешение буфера кадра декодированного потока; обычно не отличается от
	 того, что было возвращено :c:func:`VIDIOC_G_FMT`, но может отличаться,
	 если аппаратура поддерживает композицию и/или масштабирование.

   * Установка формата ``CAPTURE`` сбросит прямоугольники выбора для композиции
     к их значениям по умолчанию, основанным на новом разрешении, как описано на
     предыдущем шаге.

5. **Необязательно.** Установите прямоугольник композиции через
   :c:func:`VIDIOC_S_SELECTION` на очереди ``CAPTURE``, если это желательно и
   если декодер обладает возможностями композиции и/или масштабирования.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

     ``target``
         установлен в ``V4L2_SEL_TGT_COMPOSE``.

     ``r.left``, ``r.top``, ``r.width``, ``r.height``
         прямоугольник внутри буфера ``CAPTURE``, в который записывается
         обрезанный кадр; по умолчанию равен ``V4L2_SEL_TGT_COMPOSE_DEFAULT``;
         доступен только для чтения на аппаратуре без дополнительных
         возможностей композиции/масштабирования.

   * **Возвращаемые поля:**

     ``r.left``, ``r.top``, ``r.width``, ``r.height``
         видимый прямоугольник; он должен помещаться в пределах разрешения
         буфера кадра, возвращённого :c:func:`VIDIOC_G_FMT` на ``CAPTURE``.

   .. warning::

      Декодер может скорректировать прямоугольник композиции до ближайшего
      поддерживаемого, чтобы удовлетворить требованиям кодека и аппаратуры.
      Клиенту необходимо проверить скорректированный прямоугольник, возвращённый
      :c:func:`VIDIOC_S_SELECTION`.

6.  Если выполнены все следующие условия, клиент может мгновенно возобновить
    декодирование:

    * ``sizeimage`` нового формата (определённого на предыдущих шагах) меньше или
      равен размеру выделенных в данный момент буферов,

    * количество выделенных в данный момент буферов больше или равно
      минимальному количеству буферов, полученному на предыдущих шагах. Чтобы
      выполнить это требование, клиент может использовать
      :c:func:`VIDIOC_CREATE_BUFS` для добавления новых буферов.

    В этом случае остальные шаги не применяются, и клиент может возобновить
    декодирование одним из следующих действий:

    * если очередь ``CAPTURE`` ведёт потоковую передачу, вызвать
      :c:func:`VIDIOC_DECODER_CMD` с командой ``V4L2_DEC_CMD_START``,

    * если очередь ``CAPTURE`` не ведёт потоковую передачу, вызвать
      :c:func:`VIDIOC_STREAMON` на очереди ``CAPTURE``.

    Однако если клиент намерен изменить набор буферов, чтобы снизить
    использование памяти или по любым другим причинам, этого можно достичь,
    следуя приведённым ниже шагам.

7.  **Если очередь** ``CAPTURE`` **ведёт потоковую передачу,** продолжайте
    помещать в очередь и извлекать из очереди буферы на очереди ``CAPTURE`` до
    тех пор, пока не будет извлечён буфер, помеченный флагом
    ``V4L2_BUF_FLAG_LAST``.

8.  **Если очередь** ``CAPTURE`` **ведёт потоковую передачу,** вызовите
    :c:func:`VIDIOC_STREAMOFF` на очереди ``CAPTURE``, чтобы остановить
    потоковую передачу.

    .. warning::

       Очередь ``OUTPUT`` должна оставаться в режиме потоковой передачи. Вызов
       :c:func:`VIDIOC_STREAMOFF` на ней прервал бы последовательность и запустил
       перемотку (seek).

9.  **Если для очереди** ``CAPTURE`` **выделены буферы,** освободите буферы
    ``CAPTURE`` с помощью :c:func:`VIDIOC_REQBUFS`.

    * **Обязательные поля:**

      ``count``
          установлен в 0.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``memory``
          следует стандартной семантике.

10. Выделите буферы ``CAPTURE`` через :c:func:`VIDIOC_REQBUFS` на очереди
    ``CAPTURE``.

    * **Обязательные поля:**

      ``count``
          запрашиваемое количество выделяемых буферов; больше нуля.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``memory``
          следует стандартной семантике.

    * **Возвращаемые поля:**

      ``count``
          фактическое количество выделенных буферов.

    .. warning::

       Фактическое количество выделенных буферов может отличаться от заданного
       значения ``count``. Клиент должен проверить обновлённое значение
       ``count`` после возврата из вызова.

    .. note::

       Чтобы выделить больше буферов, чем минимальное количество (для глубины
       конвейера), клиент может запросить элемент управления
       ``V4L2_CID_MIN_BUFFERS_FOR_CAPTURE``, чтобы получить минимальное
       требуемое количество буферов, и передать полученное значение плюс
       количество необходимых дополнительных буферов в поле ``count`` для
       :c:func:`VIDIOC_REQBUFS`.

    Как вариант, для большего контроля над выделением буферов можно
    использовать :c:func:`VIDIOC_CREATE_BUFS` на очереди ``CAPTURE``. Например,
    выделяя буферы большего размера, чем текущий формат ``CAPTURE``, можно
    учесть будущие изменения разрешения.

    * **Обязательные поля:**

      ``count``
          запрашиваемое количество выделяемых буферов; больше нуля.

      ``type``
          enum ``V4L2_BUF_TYPE_*``, подходящий для ``CAPTURE``.

      ``memory``
          следует стандартной семантике.

      ``format``
          формат, представляющий максимальное разрешение буфера кадра, которое
          должно быть учтено вновь выделяемыми буферами.

    * **Возвращаемые поля:**

      ``count``
          скорректировано до количества выделенных буферов.

    .. warning::

        Фактическое количество выделенных буферов может отличаться от заданного
        значения ``count``. Клиент должен проверить обновлённое значение
        ``count`` после возврата из вызова.

    .. note::

       Чтобы выделить буферы для формата, отличного от разобранного из stream
       metadata, клиент должен поступить следующим образом, до того как будет
       инициирован разбор метаданных:

       * установить ширину и высоту формата ``OUTPUT`` в желаемое coded
         resolution, чтобы декодер настроил формат ``CAPTURE`` надлежащим образом,

       * запросить формат ``CAPTURE`` с помощью :c:func:`VIDIOC_G_FMT` и
         сохранить его до этого шага.

       Полученный в запросе формат может быть затем использован с
       :c:func:`VIDIOC_CREATE_BUFS` на этом шаге для выделения буферов.

11. Вызовите :c:func:`VIDIOC_STREAMON` на очереди ``CAPTURE``, чтобы начать
    декодирование кадров.

Декодирование
=============

Это состояние достигается после успешного завершения последовательности
`Capture Setup`_. В этом состоянии клиент помещает в очередь и извлекает из
очереди буферы для обеих очередей через :c:func:`VIDIOC_QBUF` и
:c:func:`VIDIOC_DQBUF`, следуя стандартной семантике.

Содержимое буферов-источников ``OUTPUT`` зависит от активного кодированного
пиксельного формата и может зависеть от специфичных для кодека расширенных
элементов управления (extended controls), как указано в документации каждого
формата.

Обе очереди работают независимо, следуя стандартному поведению очередей буферов
V4L2 и устройств memory-to-memory. Кроме того, порядок декодированных кадров,
извлекаемых из очереди ``CAPTURE``, может отличаться от порядка помещения в
очередь кодированных кадров в очередь ``OUTPUT`` из-за свойств выбранного coded
format, например переупорядочивания кадров.

Клиент не должен предполагать никакой прямой связи между буферами ``CAPTURE``
и ``OUTPUT`` и какой-либо конкретной временной привязки появления буферов,
доступных для извлечения. В частности:

* буфер, помещённый в очередь ``OUTPUT``, может не привести к появлению ни
  одного буфера на ``CAPTURE`` (например, если он не содержит кодированных
  данных или если в нём присутствуют только синтаксические структуры
  метаданных),

* буфер, помещённый в очередь ``OUTPUT``, может привести к появлению более чем
  одного буфера на ``CAPTURE`` (если кодированные данные содержали более одного
  кадра или если возврат декодированного кадра позволил декодеру вернуть кадр,
  который предшествовал ему в порядке декодирования, но следовал за ним в
  порядке отображения),

* буфер, помещённый в очередь ``OUTPUT``, может привести к появлению буфера на
  ``CAPTURE`` позднее в процессе декодирования, и/или после обработки
  дальнейших буферов ``OUTPUT``, либо быть возвращён не по порядку, например,
  если используется переупорядочивание для отображения,

* буферы могут стать доступными на очереди ``CAPTURE`` без помещения
  дополнительных буферов в очередь ``OUTPUT`` (например, во время drain или
  ``EOS``) из-за буферов ``OUTPUT``, помещённых в очередь в прошлом, результаты
  декодирования которых доступны лишь позднее, в силу специфики процесса
  декодирования.

.. note::

   Чтобы можно было сопоставить декодированные буферы ``CAPTURE`` с буферами
   ``OUTPUT``, из которых они получены, клиент может установить поле
   ``timestamp`` структуры :c:type:`v4l2_buffer` при помещении буфера
   ``OUTPUT`` в очередь. Буфер(ы) ``CAPTURE``, полученные в результате
   декодирования этого буфера ``OUTPUT``, при извлечении будут иметь поле
   ``timestamp``, установленное в то же значение.

   В дополнение к простому случаю, когда один буфер ``OUTPUT`` производит один
   буфер ``CAPTURE``, определены следующие случаи:

   * один буфер ``OUTPUT`` порождает несколько буферов ``CAPTURE``: одна и та же
     метка времени ``OUTPUT`` будет скопирована в несколько буферов ``CAPTURE``.

   * несколько буферов ``OUTPUT`` порождают один буфер ``CAPTURE``: будет
     скопирована метка времени буфера ``OUTPUT``, помещённого в очередь первым.

   * порядок декодирования отличается от порядка отображения (т. е. буферы
     ``CAPTURE`` идут не по порядку по сравнению с буферами ``OUTPUT``): метки
     времени ``CAPTURE`` не сохранят порядок меток времени ``OUTPUT``.

.. note::

   Память, лежащая в основе буферов ``CAPTURE``, которые используются как
   опорные кадры потоком, может считываться аппаратурой даже после того, как они
   извлечены из очереди. Следовательно, клиент должен избегать записи в эту
   память, пока очередь ``CAPTURE`` ведёт потоковую передачу. Несоблюдение этого
   может привести к повреждению декодированных кадров.

   Аналогично, при использовании типа памяти, отличного от ``V4L2_MEMORY_MMAP``,
   клиент должен убедиться, что каждый буфер ``CAPTURE`` всегда помещается в
   очередь с одной и той же лежащей в основе памятью на протяжении всего времени,
   пока очередь ``CAPTURE`` ведёт потоковую передачу. Причина этого в том, что
   индексы буферов V4L2 могут использоваться драйверами для идентификации
   кадров. Таким образом, если лежащая в основе память опорного кадра будет
   представлена под другим идентификатором буфера, драйвер может неверно его
   идентифицировать и декодировать в него новый кадр, пока он ещё используется,
   что приведёт к повреждению последующих кадров.

Во время декодирования декодер может инициировать одну из специальных
последовательностей, перечисленных ниже. Эти последовательности приводят к
тому, что декодер возвращает все буферы ``CAPTURE``, которые были получены из
всех буферов ``OUTPUT``, обработанных до начала последовательности. Последний из
буферов будет иметь установленный флаг ``V4L2_BUF_FLAG_LAST``. Чтобы определить,
какой последовательности следовать, клиент должен проверить, имеется ли какое-либо
ожидающее событие, и:

* если ожидает событие ``V4L2_EVENT_SOURCE_CHANGE`` с полем ``changes``,
  установленным в ``V4L2_EVENT_SRC_CH_RESOLUTION``, необходимо выполнить
  последовательность `Dynamic Resolution Change`_,

* если ожидает событие ``V4L2_EVENT_EOS``, необходимо выполнить
  последовательность `End of Stream`_.

Некоторые из последовательностей могут перемежаться друг с другом, и их нужно
обрабатывать по мере возникновения. Точный порядок действий задокументирован
для каждой последовательности.

В случае возникновения ошибки декодирования о ней будет сообщено клиенту с
уровнем детализации, зависящим от возможностей декодера. В частности:

* буфер CAPTURE, содержащий результаты неудачной операции декодирования, будет
  возвращён с установленным флагом V4L2_BUF_FLAG_ERROR,

* если декодер способен точно сообщить, какой буфер OUTPUT вызвал ошибку, такой
  буфер будет возвращён с установленным флагом V4L2_BUF_FLAG_ERROR.

В случае фатального сбоя, не позволяющего продолжить декодирование, любые
дальнейшие операции с соответствующим файловым дескриптором декодера будут
возвращать код ошибки -EIO. Клиент может закрыть файловый дескриптор и открыть
новый или, как вариант, повторно инициализировать экземпляр, остановив потоковую
передачу на обеих очередях, освободив все буферы и выполнив последовательность
Инициализации заново.

Перемотка (seek)
================

Перемотка (seek) управляется очередью ``OUTPUT``, поскольку она является
источником кодированных данных. Перемотка не требует какой-либо конкретной
операции на очереди ``CAPTURE``, но та может быть затронута в рамках обычной
работы декодера.

1. Остановите очередь ``OUTPUT``, чтобы начать последовательность перемотки,
   через :c:func:`VIDIOC_STREAMOFF`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

   * Декодер сбросит все ожидающие буферы ``OUTPUT``, и они должны
     рассматриваться как возвращённые клиенту (согласно стандартной семантике).

2. Перезапустите очередь ``OUTPUT`` через :c:func:`VIDIOC_STREAMON`.

   * **Обязательные поля:**

     ``type``
         enum ``V4L2_BUF_TYPE_*``, подходящий для ``OUTPUT``.

   * Декодер начнёт принимать новые буферы потока байтов-источника после
     возврата из вызова.

3. Начните помещать в очередь ``OUTPUT`` буферы, содержащие кодированные данные
   после перемотки, пока не будет найдена подходящая resume point.

   .. note::

      Не требуется начинать помещение в очередь кодированных данных точно с
      resume point (например, с SPS или ключевого кадра). Любые помещённые в
      очередь буферы ``OUTPUT`` будут обработаны и возвращены клиенту, пока не
      будет найдена подходящая resume point. Во время поиска resume point
      декодер не должен производить никаких декодированных кадров в буферы
      ``CAPTURE``.

      Известно, что некоторая аппаратура неправильно обрабатывает перемотку к
      точке, не являющейся resume point. Такая операция может привести к тому,
      что на очереди ``CAPTURE`` станет доступно неопределённое количество
      повреждённых декодированных кадров. Драйверы должны гарантировать, что не
      произойдёт фатальных ошибок декодирования или сбоев, и реализовать любую
      необходимую обработку и обходные решения для проблем аппаратуры, связанных
      с операциями перемотки.

   .. warning::

      В случае кодека H.264/HEVC клиент должен позаботиться о том, чтобы не
      перематывать через изменение SPS/PPS. Даже если целевой кадр является
      ключевым, устаревшие SPS/PPS внутри состояния декодера приведут к
      неопределённым результатам при декодировании. Хотя декодер должен
      обработать этот случай без сбоя или фатальной ошибки декодирования, клиент
      не должен ожидать осмысленного результата декодирования.

      Если аппаратура способна обнаруживать такие повреждённые декодированные
      кадры, то соответствующие буферы будут возвращены клиенту с установленным
      V4L2_BUF_FLAG_ERROR. См. раздел `Декодирование`_ для дальнейшего
      описания сообщения об ошибках декодирования.

4. После того как найдена resume point, декодер начнёт возвращать буферы
   ``CAPTURE``, содержащие декодированные кадры.

.. important::

   Перемотка может привести к инициированию последовательности `Dynamic
   Resolution Change`_ из-за того, что цель перемотки имеет параметры
   декодирования, отличные от той части потока, что была декодирована до
   перемотки. Эту последовательность необходимо обрабатывать в рамках обычной
   работы декодера.

.. warning::

   Не определено, когда очередь ``CAPTURE`` начнёт производить буферы,
   содержащие декодированные данные из буферов ``OUTPUT``, помещённых в очередь
   после перемотки, поскольку она работает независимо от очереди ``OUTPUT``.

   Декодер может вернуть некоторое количество оставшихся буферов ``CAPTURE``,
   содержащих декодированные кадры, происходящие из буферов ``OUTPUT``,
   помещённых в очередь до выполнения последовательности перемотки.

   Операция ``VIDIOC_STREAMOFF`` отбрасывает любые оставшиеся помещённые в
   очередь буферы ``OUTPUT``, что означает, что не для всех буферов ``OUTPUT``,
   помещённых в очередь до последовательности перемотки, могут быть произведены
   соответствующие буферы ``CAPTURE``. Например, для следующей
   последовательности операций на очереди ``OUTPUT``:

     QBUF(A), QBUF(B), STREAMOFF(), STREAMON(), QBUF(G), QBUF(H),

   допустим любой из следующих результатов на очереди ``CAPTURE``:

     {A', B', G', H'}, {A', G', H'}, {G', H'}.

   Чтобы определить буфер CAPTURE, содержащий первый декодированный кадр после
   перемотки, клиент может наблюдать за метками времени, чтобы сопоставить буферы
   CAPTURE и OUTPUT, или использовать V4L2_DEC_CMD_STOP и V4L2_DEC_CMD_START для
   опустошения (drain) декодера.

.. note::

   Чтобы добиться мгновенной перемотки, клиент может также перезапустить
   потоковую передачу на очереди ``CAPTURE``, чтобы отбросить декодированные, но
   ещё не извлечённые буферы.

Dynamic Resolution Change
=========================

Потоки, включающие метаданные о разрешении в потоке байтов, могут потребовать
переключения на другое разрешение во время декодирования.

.. note::

   Не все декодеры способны обнаруживать изменения разрешения. Те, что
   способны, устанавливают флаг ``V4L2_FMT_FLAG_DYN_RESOLUTION`` для coded
   format при вызове :c:func:`VIDIOC_ENUM_FMT`.

Последовательность начинается, когда декодер обнаруживает кодированный кадр с
одним или несколькими из следующих параметров, отличающихся от установленных
ранее (и отражённых соответствующими запросами):

* coded resolution (ширина и высота ``OUTPUT``),

* visible resolution (прямоугольники выбора),

* минимальное количество буферов, необходимых для декодирования,

* изменилась битовая глубина (bit-depth) потока битов,

* изменилось цветовое пространство (colorspace) потока битов, но это не требует
  перевыделения буферов.

Всякий раз, когда это происходит, декодер должен действовать следующим образом:

1.  После обнаружения изменения разрешения в потоке декодер отправляет событие
    ``V4L2_EVENT_SOURCE_CHANGE`` с полем ``changes``, установленным в
    ``V4L2_EVENT_SRC_CH_RESOLUTION``.

    .. important::

       Любой запрос клиента, выданный после того, как декодер поместил событие в
       очередь, вернёт значения, применимые к потоку после изменения разрешения,
       включая форматы очередей, прямоугольники выбора и элементы управления.

2.  Затем декодер обработает и декодирует все оставшиеся буферы с момента до
    точки изменения разрешения.

    * Последний буфер с момента до изменения должен быть помечен флагом
      ``V4L2_BUF_FLAG_LAST``, аналогично последовательности `Drain`_ выше.

    .. warning::

       Последний буфер может быть пустым (с :c:type:`v4l2_buffer` ``bytesused``
       = 0), и в этом случае он должен игнорироваться клиентом, поскольку не
       содержит декодированного кадра.

    .. note::

       Любая попытка извлечь больше буферов ``CAPTURE`` за пределами буфера,
       помеченного ``V4L2_BUF_FLAG_LAST``, приведёт к ошибке -EPIPE от
       :c:func:`VIDIOC_DQBUF`.

Клиент должен продолжить последовательность, как описано ниже, чтобы продолжить
процесс декодирования.

1.  Извлеките событие изменения источника (source change).

    .. important::

       Изменение источника запускает неявное опустошение (drain) декодера,
       аналогичное явной последовательности `Drain`_. После его завершения
       декодер останавливается. Процесс декодирования необходимо возобновить
       либо парой вызовов :c:func:`VIDIOC_STREAMOFF` и
       :c:func:`VIDIOC_STREAMON` на очереди ``CAPTURE``, либо вызовом
       :c:func:`VIDIOC_DECODER_CMD` с командой ``V4L2_DEC_CMD_START``.

2.  Продолжите с последовательности `Capture Setup`_.

.. note::

   Во время последовательности изменения разрешения очередь ``OUTPUT`` должна
   оставаться в режиме потоковой передачи. Вызов :c:func:`VIDIOC_STREAMOFF` на
   очереди ``OUTPUT`` прервал бы последовательность и инициировал перемотку.

   В принципе очередь ``OUTPUT`` работает отдельно от очереди ``CAPTURE``, и это
   остаётся верным на протяжении всей последовательности изменения разрешения.

   Для наилучшей производительности и простоты клиенту следует продолжать
   помещать в очередь / извлекать из очереди буферы в/из очереди ``OUTPUT`` даже
   во время обработки этой последовательности.

Drain
=====

Чтобы гарантировать, что все помещённые в очередь буферы ``OUTPUT`` были
обработаны и связанные с ними буферы ``CAPTURE`` переданы клиенту, клиент должен
следовать последовательности опустошения (drain), описанной ниже. После
завершения последовательности опустошения клиент получил все декодированные
кадры для всех буферов ``OUTPUT``, помещённых в очередь до начала
последовательности.

1. Начните опустошение, выдав :c:func:`VIDIOC_DECODER_CMD`.

   * **Обязательные поля:**

     ``cmd``
         установлен в ``V4L2_DEC_CMD_STOP``.

     ``flags``
         установлен в 0.

     ``pts``
         установлен в 0.

   .. warning::

      Последовательность может быть инициирована только в том случае, если обе
      очереди, ``OUTPUT`` и ``CAPTURE``, ведут потоковую передачу. По причинам
      совместимости вызов :c:func:`VIDIOC_DECODER_CMD` не завершится ошибкой,
      даже если какая-либо из очередей не ведёт потоковую передачу, но при этом
      он не инициирует последовательность `Drain`_, и поэтому описанные ниже
      шаги будут неприменимы.

2. Любые буферы ``OUTPUT``, помещённые в очередь клиентом до того, как был выдан
   :c:func:`VIDIOC_DECODER_CMD`, будут обработаны и декодированы обычным
   образом. Клиент должен продолжать обрабатывать обе очереди независимо,
   аналогично обычной операции декодирования. Это включает:

   * обработку любых операций, запущенных в результате обработки этих буферов,
     таких как последовательность `Dynamic Resolution Change`_, перед
     продолжением последовательности опустошения,

   * помещение в очередь и извлечение из очереди буферов ``CAPTURE``, пока не
     будет извлечён буфер, помеченный флагом ``V4L2_BUF_FLAG_LAST``,

     .. warning::

        Последний буфер может быть пустым (с :c:type:`v4l2_buffer`
        ``bytesused`` = 0), и в этом случае он должен игнорироваться клиентом,
        поскольку не содержит декодированного кадра.

     .. note::

        Любая попытка извлечь больше буферов ``CAPTURE`` за пределами буфера,
        помеченного ``V4L2_BUF_FLAG_LAST``, приведёт к ошибке -EPIPE от
        :c:func:`VIDIOC_DQBUF`.

   * извлечение из очереди обработанных буферов ``OUTPUT``, пока не будут
     извлечены все буферы, помещённые в очередь до команды ``V4L2_DEC_CMD_STOP``,

   * извлечение из очереди события ``V4L2_EVENT_EOS``, если клиент подписался на него.

   .. note::

      Для обратной совместимости декодер подаёт событие ``V4L2_EVENT_EOS``,
      когда декодирован последний кадр и все кадры готовы к извлечению из
      очереди. Это устаревшее поведение, и клиент не должен полагаться на него.
      Вместо этого следует использовать флаг буфера ``V4L2_BUF_FLAG_LAST``.

3. Как только все буферы ``OUTPUT``, помещённые в очередь до вызова
   ``V4L2_DEC_CMD_STOP``, извлечены, и извлечён последний буфер ``CAPTURE``,
   декодер останавливается и будет принимать, но не обрабатывать, любые вновь
   помещённые в очередь буферы ``OUTPUT``, пока клиент не выдаст одну из
   следующих операций:

   * ``V4L2_DEC_CMD_START`` — декодер не будет сброшен и возобновит работу в
     обычном режиме со всем состоянием, имевшимся до опустошения,

   * пару :c:func:`VIDIOC_STREAMOFF` и :c:func:`VIDIOC_STREAMON` на очереди
     ``CAPTURE`` — декодер возобновит работу в обычном режиме, однако любые
     буферы ``CAPTURE``, всё ещё находящиеся в очереди, будут возвращены
     клиенту,

   * пару :c:func:`VIDIOC_STREAMOFF` и :c:func:`VIDIOC_STREAMON` на очереди
     ``OUTPUT`` — любые ожидающие буферы-источники будут возвращены клиенту, и
     будет запущена последовательность `Перемотка (seek)`_.

.. note::

   Как только последовательность опустошения инициирована, клиент должен довести
   её до завершения, как описано в приведённых выше шагах, если только он не
   прервёт процесс, выдав :c:func:`VIDIOC_STREAMOFF` на любой из очередей
   ``OUTPUT`` или ``CAPTURE``. Клиенту не разрешается снова выдавать
   ``V4L2_DEC_CMD_START`` или ``V4L2_DEC_CMD_STOP``, пока выполняется
   последовательность опустошения, и при попытке они завершатся с кодом ошибки
   -EBUSY.

   Хотя это и не обязательно, доступность команд декодера может быть запрошена с
   помощью :c:func:`VIDIOC_TRY_DECODER_CMD`.

End of Stream
=============

Если декодер встречает в потоке маркировку конца потока, декодер инициирует
последовательность `Drain`_, которую клиент должен обработать, как описано
выше, пропустив начальный :c:func:`VIDIOC_DECODER_CMD`.

Точки фиксации (commit points)
==============================

Установка форматов и выделение буферов запускают изменения в поведении декодера.

1. Установка формата на очереди ``OUTPUT`` может изменить набор форматов,
   поддерживаемых/объявляемых на очереди ``CAPTURE``. В частности, это также
   означает, что формат ``CAPTURE`` может быть сброшен, и клиент не должен
   полагаться на сохранение ранее установленного формата.

2. Перечисление форматов на очереди ``CAPTURE`` всегда возвращает только
   форматы, поддерживаемые для текущего формата ``OUTPUT``.

3. Установка формата на очереди ``CAPTURE`` не изменяет список форматов,
   доступных на очереди ``OUTPUT``. Попытка установить формат ``CAPTURE``,
   который не поддерживается для выбранного в данный момент формата ``OUTPUT``,
   приведёт к тому, что декодер скорректирует запрошенный формат ``CAPTURE`` до
   поддерживаемого.

4. Перечисление форматов на очереди ``OUTPUT`` всегда возвращает полный набор
   поддерживаемых coded format, независимо от текущего формата ``CAPTURE``.

5. Пока на любой из очередей, ``OUTPUT`` или ``CAPTURE``, выделены буферы,
   клиент не должен изменять формат на очереди ``OUTPUT``. Драйверы будут
   возвращать код ошибки -EBUSY при любой такой попытке изменения формата.

Подводя итог, установка форматов и выделение всегда должны начинаться с очереди
``OUTPUT``, и очередь ``OUTPUT`` является ведущей (master), которая определяет
набор поддерживаемых форматов для очереди ``CAPTURE``.
