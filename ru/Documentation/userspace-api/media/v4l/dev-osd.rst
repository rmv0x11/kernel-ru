*************************************************
Интерфейс наложения вывода видео (Output Overlay)
*************************************************

**Также известен как экранное меню (On-Screen Display, OSD)**

Некоторые устройства вывода видео могут накладывать изображение из
фреймбуфера на исходящий видеосигнал. Приложения могут настроить такое
наложение с помощью этого интерфейса, который заимствует структуры и
ioctl интерфейса :ref:`Video Overlay <overlay>`.

Функция OSD доступна через тот же символьный специальный файл, что и
функция :ref:`Video Output <capture>`.

.. note::

   Функцией по умолчанию для такого устройства ``/dev/video`` является
   захват или вывод видео. Функция OSD становится доступна только после
   вызова ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`.


Запрос возможностей
===================

Устройства, поддерживающие интерфейс *Video Output Overlay*, устанавливают
флаг ``V4L2_CAP_VIDEO_OUTPUT_OVERLAY`` в поле ``capabilities`` структуры
:c:type:`v4l2_capability`, возвращаемой ioctl
:ref:`VIDIOC_QUERYCAP`.


Фреймбуфер
==========

В отличие от интерфейса *Video Overlay*, фреймбуфер обычно реализован на
ТВ-карте, а не на графической карте. В Linux он доступен как
устройство-фреймбуфер (``/dev/fbN``). Имея устройство V4L2, приложения
могут найти соответствующее устройство-фреймбуфер, вызвав ioctl
:ref:`VIDIOC_G_FBUF <VIDIOC_G_FBUF>`. Среди прочей информации он
возвращает физический адрес фреймбуфера в поле ``base`` структуры
:c:type:`v4l2_framebuffer`.
Ioctl устройства-фреймбуфера ``FBIOGET_FSCREENINFO`` возвращает тот же
адрес в поле ``smem_start`` структуры
:c:type:`fb_fix_screeninfo`. Ioctl ``FBIOGET_FSCREENINFO``
и структура :c:type:`fb_fix_screeninfo` определены в
заголовочном файле ``linux/fb.h``.

Ширина и высота фреймбуфера зависят от текущего видеостандарта. Драйвер
V4L2 может отклонять попытки изменить видеостандарт (или любой другой
ioctl, который подразумевал бы изменение размера фреймбуфера) с кодом
ошибки ``EBUSY`` до тех пор, пока все приложения не закроют
устройство-фреймбуфер.

Пример: поиск устройства-фреймбуфера для OSD
--------------------------------------------

.. code-block:: c

    #include <linux/fb.h>

    struct v4l2_framebuffer fbuf;
    unsigned int i;
    int fb_fd;

    if (-1 == ioctl(fd, VIDIOC_G_FBUF, &fbuf)) {
	perror("VIDIOC_G_FBUF");
	exit(EXIT_FAILURE);
    }

    for (i = 0; i < 30; i++) {
	char dev_name[16];
	struct fb_fix_screeninfo si;

	snprintf(dev_name, sizeof(dev_name), "/dev/fb%u", i);

	fb_fd = open(dev_name, O_RDWR);
	if (-1 == fb_fd) {
	    switch (errno) {
	    case ENOENT: /* no such file */
	    case ENXIO:  /* no driver */
		continue;

	    default:
		perror("open");
		exit(EXIT_FAILURE);
	    }
	}

	if (0 == ioctl(fb_fd, FBIOGET_FSCREENINFO, &si)) {
	    if (si.smem_start == (unsigned long)fbuf.base)
		break;
	} else {
	    /* Apparently not a framebuffer device. */
	}

	close(fb_fd);
	fb_fd = -1;
    }

    /* fb_fd is the file descriptor of the framebuffer device
       for the video output overlay, or -1 if no device was found. */


Окно наложения и масштабирование
================================

Наложение управляется исходным и целевым прямоугольниками. Исходный
прямоугольник выбирает часть изображения фреймбуфера, которая будет
наложена, а целевой прямоугольник — область в исходящем видеосигнале, где
появится изображение. Драйверы могут поддерживать или не поддерживать
масштабирование, а также произвольные размеры и положения этих
прямоугольников. Кроме того, драйверы могут поддерживать любой (или ни
один) из методов отсечения/наложения, определённых для интерфейса
:ref:`Video Overlay <overlay>`.

Структура :c:type:`v4l2_window` определяет размер исходного
прямоугольника, его положение в фреймбуфере и метод
отсечения/наложения, используемый для наложения. Чтобы получить текущие
параметры, приложения устанавливают поле ``type`` структуры
:c:type:`v4l2_format` в
``V4L2_BUF_TYPE_VIDEO_OUTPUT_OVERLAY`` и вызывают ioctl
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>`. Драйвер заполняет
подструктуру :c:type:`v4l2_window` с именем ``win``. Извлечь ранее
запрограммированный список отсечения или битовую карту невозможно.

Чтобы запрограммировать исходный прямоугольник, приложения устанавливают
поле ``type`` структуры :c:type:`v4l2_format` в
``V4L2_BUF_TYPE_VIDEO_OUTPUT_OVERLAY``, инициализируют подструктуру
``win`` и вызывают ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`.
Драйвер корректирует параметры в соответствии с аппаратными ограничениями
и возвращает фактические параметры так же, как это делает
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>`. Как и :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`,
ioctl :ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` можно использовать, чтобы узнать
о возможностях драйвера, фактически не изменяя его состояние. В отличие от
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`, это также работает после того, как наложение было включено.

Структура :c:type:`v4l2_crop` определяет размер и положение
целевого прямоугольника. Коэффициент масштабирования наложения неявно
задаётся шириной и высотой, указанными в структурах :c:type:`v4l2_window`
и :c:type:`v4l2_crop`. API кадрирования применяется к
устройствам *Video Output* и *Video Output Overlay* так же, как и к
устройствам *Video Capture* и *Video Overlay*, лишь меняя на
противоположное направление потока данных. Дополнительные сведения см. в :ref:`crop`.


Включение наложения
===================

В V4L2 нет ioctl для включения или отключения наложения, однако интерфейс
фреймбуфера драйвера может поддерживать ioctl ``FBIOBLANK``.
