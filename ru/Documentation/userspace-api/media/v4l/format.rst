Согласование формата данных
============================

Различные устройства обмениваются с приложениями разными видами данных,
например видеоизображениями, необработанными или нарезанными данными VBI,
датаграммами RDS. Даже в пределах одного вида возможно множество различных
форматов, в частности существует изобилие форматов изображений. Хотя драйверы
обязаны предоставлять формат по умолчанию, а выбор сохраняется при закрытии и
повторном открытии устройства, приложениям следует всегда согласовывать формат
данных перед началом обмена данными. Согласование означает, что приложение
запрашивает определённый формат, а драйвер выбирает и сообщает наилучшее, что
оборудование способно сделать для удовлетворения запроса. Разумеется, приложения
могут и просто запросить текущий выбор.

Существует единый механизм для согласования всех форматов данных с помощью
агрегирующей структуры :c:type:`v4l2_format` и ioctl
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Дополнительно ioctl
:ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` может использоваться для проверки того,
что оборудование *могло бы* сделать, без фактического выбора нового формата
данных. Форматы данных, поддерживаемые API V4L2, описаны в соответствующем
разделе об устройствах в :ref:`devices`. Более подробно о форматах изображений
см. :ref:`pixfmt`.

Ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>` является важным поворотным моментом в
последовательности инициализации. До этого момента несколько панельных
приложений могут одновременно обращаться к одному и тому же устройству, чтобы
выбрать текущий вход, изменить элементы управления или модифицировать другие
свойства. Первый :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`
назначает логический поток (видеоданные, данные VBI и т. д.) исключительно
одному файловому дескриптору.

Исключительно означает, что никакое другое приложение, точнее никакой другой
файловый дескриптор, не может захватить этот поток или изменить свойства
устройства несовместимо с согласованными параметрами. Изменение видеостандарта,
например, когда новый стандарт использует другое число строк развёртки, может
сделать недействительным выбранный формат изображения. Поэтому только файловый
дескриптор, владеющий потоком, может вносить изменения, делающие его
недействительным. Соответственно, несколько файловых дескрипторов, захвативших
разные логические потоки, не дают друг другу вмешиваться в свои настройки. Когда,
например, видеоналожение (video overlay) вот-вот начнётся или уже выполняется,
одновременный видеозахват может быть ограничен той же областью кадрирования и
размером изображения.

Когда приложения опускают ioctl :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`, его побочные
эффекты блокировки подразумеваются следующим шагом — выбором метода ввода-вывода
с помощью ioctl :ref:`VIDIOC_REQBUFS` или неявно
при первом вызове :c:func:`read()` или
:c:func:`write()`.

Как правило, файловому дескриптору может быть назначен только один логический
поток; исключение составляют драйверы, разрешающие одновременный видеозахват и
наложение с использованием одного и того же файлового дескриптора для
совместимости с V4L и более ранними версиями V4L2. Переключение логического
потока или возврат в «панельный режим» возможны путём закрытия и повторного
открытия устройства. Драйверы *могут* поддерживать переключение с помощью
:ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`.

Все драйверы, обменивающиеся данными с приложениями, обязаны поддерживать ioctl
:ref:`VIDIOC_G_FMT <VIDIOC_G_FMT>` и :ref:`VIDIOC_S_FMT <VIDIOC_G_FMT>`. Реализация
:ref:`VIDIOC_TRY_FMT <VIDIOC_G_FMT>` настоятельно рекомендуется, но необязательна.

Перечисление форматов изображений
==================================

Помимо обобщённых функций согласования формата, доступен специальный ioctl для
перечисления всех форматов изображений, поддерживаемых устройствами
видеозахвата, наложения или вывода. [#f1]_

Ioctl :ref:`VIDIOC_ENUM_FMT` обязан поддерживаться
всеми драйверами, обменивающимися данными изображений с приложениями.

.. important::

    Драйверы не должны преобразовывать форматы изображений в пространстве ядра.
    Они должны перечислять только форматы, непосредственно поддерживаемые
    оборудованием. При необходимости авторам драйверов следует опубликовать
    пример процедуры или библиотеки преобразования для интеграции в приложения.

.. [#f1]
   Перечисление форматов, о которых приложение не имеет априорного знания
   (иначе оно могло бы явно запросить их и не нуждалось бы в перечислении),
   кажется бесполезным, однако существуют приложения, выступающие посредником
   между драйверами и собственно видеоприложениями, для которых это полезно.
