Request API
===========

Request API спроектирован так, чтобы позволить V4L2 удовлетворять требованиям
современных устройств (stateless-кодеков, сложных конвейеров камер, ...) и API
(Android Codec v2). Одно из таких требований — способность устройств,
принадлежащих одному конвейеру, перенастраиваться и тесно взаимодействовать на
покадровой основе. Другое — поддержка stateless-кодеков, которым для
эффективной работы необходимо применять элементы управления (controls) к
конкретным кадрам (так называемые «покадровые элементы управления», per-frame
controls).

Хотя первоначальным сценарием использования был V4L2, его можно распространить
и на другие подсистемы, если они используют media controller.

Поддержка этих возможностей без Request API не всегда возможна, а если и
возможна, то крайне неэффективна: пространству пользователя пришлось бы сбросить
всю активность на медиаконвейере, перенастроить его под следующий кадр,
поставить в очередь буферы, которые должны быть обработаны с этой конфигурацией,
и ждать, пока они все не станут доступны для извлечения из очереди, прежде чем
рассматривать следующий кадр. Это сводит на нет смысл наличия очередей буферов,
поскольку на практике в каждый момент времени в очереди находился бы только один
буфер.

Request API позволяет связать конкретную конфигурацию конвейера (топологию media
controller + конфигурацию для каждой медиасущности) с конкретными буферами. Это
позволяет пространству пользователя заранее планировать несколько задач
(«запросов», requests) с различными конфигурациями, зная, что конфигурация будет
применена тогда, когда это необходимо для получения ожидаемого результата.
Значения конфигурации на момент завершения запроса также доступны для чтения.

Общее использование
-------------------

Request API расширяет Media Controller API и взаимодействует со
специфичными для подсистем API для поддержки использования запросов. На уровне
Media Controller запросы выделяются из соответствующего узла устройства Media
Controller. Затем их жизненный цикл управляется через файловые дескрипторы
запросов непрозрачным образом. Данные конфигурации, дескрипторы буферов и
результаты обработки, хранящиеся в запросах, доступны через специфичные для
подсистем API, расширенные для поддержки запросов, — такие как API V4L2,
принимающие явный параметр ``request_fd``.

Выделение запроса
-----------------

Пространство пользователя выделяет запросы, используя
:ref:`MEDIA_IOC_REQUEST_ALLOC` для узла устройства media. Это возвращает
файловый дескриптор, представляющий запрос. Как правило, выделяется несколько
таких запросов.

Подготовка запроса
------------------

Стандартные ioctl-ы V4L2 затем могут получать файловый дескриптор запроса, чтобы
выразить тот факт, что данный ioctl является частью указанного запроса и не
должен применяться немедленно. См. :ref:`MEDIA_IOC_REQUEST_ALLOC` для перечня
ioctl-ов, поддерживающих это. Конфигурации, заданные с параметром ``request_fd``,
сохраняются вместо немедленного применения, а буферы, поставленные в очередь
запроса, не попадают в обычную очередь буферов до тех пор, пока сам запрос не
будет поставлен в очередь.

Отправка запроса
----------------

После того как заданы конфигурация и буферы запроса, его можно поставить в
очередь, вызвав :ref:`MEDIA_REQUEST_IOC_QUEUE` для файлового дескриптора
запроса. Запрос должен содержать хотя бы один буфер, иначе возвращается
``ENOENT``. Запрос, поставленный в очередь, больше не может быть изменён.

.. caution::
   Для :ref:`устройств типа «память-в-память» <mem2mem>` запросы можно
   использовать только для буферов вывода (output), но не для буферов захвата
   (capture). Попытка добавить буфер захвата в запрос приведёт к ошибке
   ``EBADR``.

Если запрос содержит конфигурации для нескольких сущностей, отдельные драйверы
могут синхронизироваться так, чтобы топология запрошенного конвейера была
применена до обработки буферов. Драйверы media controller реализуют это по
принципу наилучших усилий (best effort), поскольку идеальная атомарность может
быть невозможна из-за аппаратных ограничений.

.. caution::

   Не допускается смешивать постановку в очередь запросов с прямой постановкой в
   очередь буферов: какой бы метод ни был использован первым, он закрепляется до
   тех пор, пока не будет вызван :ref:`VIDIOC_STREAMOFF <VIDIOC_STREAMON>` или
   устройство не будет :ref:`закрыто <func-close>`. Попытки напрямую поставить в
   очередь буфер, когда ранее буфер был поставлен в очередь через запрос, либо
   наоборот, приведут к ошибке ``EBUSY``.

Элементы управления по-прежнему могут устанавливаться без запроса и применяются
немедленно, независимо от того, используется запрос или нет.

.. caution::

   Установка одного и того же элемента управления как через запрос, так и
   напрямую может привести к неопределённому поведению!

Пространство пользователя может вызвать :c:func:`poll()` для файлового
дескриптора запроса, чтобы дождаться завершения запроса. Запрос считается
завершённым, как только все связанные с ним буферы доступны для извлечения из
очереди и все связанные элементы управления обновлены значениями на момент
завершения. Обратите внимание, что пространству пользователя не нужно ждать
завершения запроса, чтобы извлечь его буферы из очереди: буферы, которые
становятся доступными на полпути выполнения запроса, могут быть извлечены из
очереди независимо от состояния запроса.

Завершённый запрос содержит состояние устройства после его выполнения.
Пространство пользователя может запросить это состояние, вызвав
:ref:`ioctl VIDIOC_G_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>` с файловым дескриптором
запроса. Вызов :ref:`ioctl VIDIOC_G_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>` для запроса,
который был поставлен в очередь, но ещё не завершён, вернёт ``EBUSY``, поскольку
значения элементов управления могут быть изменены драйвером в любой момент, пока
запрос находится в обработке.

.. _media-request-life-time:

Повторное использование и уничтожение
-------------------------------------

Наконец, завершённый запрос можно либо отбросить, либо использовать повторно.
Вызов :c:func:`close()` для файлового дескриптора запроса сделает этот файловый
дескриптор непригодным для использования, и запрос будет освобождён, как только
он перестанет использоваться ядром. То есть, если запрос поставлен в очередь, а
затем файловый дескриптор закрыт, то он не будет освобождён, пока драйвер не
завершит запрос.

:ref:`MEDIA_REQUEST_IOC_REINIT` очистит состояние запроса и снова сделает его
доступным. Эта операция не сохраняет никакого состояния: запрос выглядит так, как
будто был только что выделен.

Пример для устройства-кодека
----------------------------

Для сценариев использования, таких как :ref:`кодеки <mem2mem>`, Request API
можно использовать для связывания конкретных элементов управления, которые
драйвер должен применить для буфера OUTPUT, что позволяет пространству
пользователя заранее ставить в очередь множество таких буферов. Он также может
воспользоваться способностью запросов фиксировать состояние элементов управления
при завершении запроса, чтобы считать обратно информацию, которая может
измениться.

Выраженное в коде, после получения запроса пространство пользователя может
назначить ему элементы управления и один буфер OUTPUT:

.. code-block:: c

	struct v4l2_buffer buf;
	struct v4l2_ext_controls ctrls;
	int req_fd;
	...
	if (ioctl(media_fd, MEDIA_IOC_REQUEST_ALLOC, &req_fd))
		return errno;
	...
	ctrls.which = V4L2_CTRL_WHICH_REQUEST_VAL;
	ctrls.request_fd = req_fd;
	if (ioctl(codec_fd, VIDIOC_S_EXT_CTRLS, &ctrls))
		return errno;
	...
	buf.type = V4L2_BUF_TYPE_VIDEO_OUTPUT;
	buf.flags |= V4L2_BUF_FLAG_REQUEST_FD;
	buf.request_fd = req_fd;
	if (ioctl(codec_fd, VIDIOC_QBUF, &buf))
		return errno;

Обратите внимание, что не допускается использование Request API для буферов
CAPTURE, поскольку там нет покадровых настроек, о которых можно было бы сообщать.

После того как запрос полностью подготовлен, его можно поставить в очередь
драйвера:

.. code-block:: c

	if (ioctl(req_fd, MEDIA_REQUEST_IOC_QUEUE))
		return errno;

Затем пространство пользователя может либо дождаться завершения запроса, вызвав
poll() для его файлового дескриптора, либо начать извлекать буферы CAPTURE из
очереди. Скорее всего, оно захочет получить буферы CAPTURE как можно скорее, и
это можно сделать с помощью обычного :ref:`VIDIOC_DQBUF <VIDIOC_QBUF>`:

.. code-block:: c

	struct v4l2_buffer buf;

	memset(&buf, 0, sizeof(buf));
	buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
	if (ioctl(codec_fd, VIDIOC_DQBUF, &buf))
		return errno;

Обратите внимание, что в этом примере для простоты предполагается, что на каждый
буфер OUTPUT приходится один буфер CAPTURE, но это не обязательно так.

Затем, убедившись, что запрос завершён, через опрос (polling) файлового
дескриптора запроса, мы можем запросить значения элементов управления на момент
его завершения с помощью вызова :ref:`VIDIOC_G_EXT_CTRLS <VIDIOC_G_EXT_CTRLS>`.
Это особенно полезно для изменчивых (volatile) элементов управления, значения
которых мы хотим запросить сразу после того, как создан буфер захвата.

.. code-block:: c

	struct pollfd pfd = { .events = POLLPRI, .fd = req_fd };
	poll(&pfd, 1, -1);
	...
	ctrls.which = V4L2_CTRL_WHICH_REQUEST_VAL;
	ctrls.request_fd = req_fd;
	if (ioctl(codec_fd, VIDIOC_G_EXT_CTRLS, &ctrls))
		return errno;

Как только запрос нам больше не нужен, мы можем либо переработать его для
повторного использования с помощью :ref:`MEDIA_REQUEST_IOC_REINIT`...

.. code-block:: c

	if (ioctl(req_fd, MEDIA_REQUEST_IOC_REINIT))
		return errno;

... либо закрыть его файловый дескриптор, чтобы полностью от него избавиться.

.. code-block:: c

	close(req_fd);

Пример для простого устройства захвата
--------------------------------------

С простым устройством захвата запросы можно использовать для указания элементов
управления, которые нужно применить для заданного буфера CAPTURE.

.. code-block:: c

	struct v4l2_buffer buf;
	struct v4l2_ext_controls ctrls;
	int req_fd;
	...
	if (ioctl(media_fd, MEDIA_IOC_REQUEST_ALLOC, &req_fd))
		return errno;
	...
	ctrls.which = V4L2_CTRL_WHICH_REQUEST_VAL;
	ctrls.request_fd = req_fd;
	if (ioctl(camera_fd, VIDIOC_S_EXT_CTRLS, &ctrls))
		return errno;
	...
	buf.type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
	buf.flags |= V4L2_BUF_FLAG_REQUEST_FD;
	buf.request_fd = req_fd;
	if (ioctl(camera_fd, VIDIOC_QBUF, &buf))
		return errno;

После того как запрос полностью подготовлен, его можно поставить в очередь
драйвера:

.. code-block:: c

	if (ioctl(req_fd, MEDIA_REQUEST_IOC_QUEUE))
		return errno;

Затем пространство пользователя может извлекать буферы из очереди, ждать
завершения запроса, запрашивать элементы управления и перерабатывать запрос, как
в примере с M2M выше.
