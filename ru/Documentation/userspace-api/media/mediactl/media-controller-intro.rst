Введение
========

Медиаустройства всё чаще объединяют несколько связанных функций. Многие
USB-камеры содержат микрофоны, оборудование для захвата видео может также
выводить видео, а интерфейсы камер SoC дополнительно выполняют операции
типа «память-в-память», подобные видеокодекам.

Независимые функции, даже когда они реализованы в одном и том же аппаратном
обеспечении, могут быть смоделированы как отдельные устройства. USB-камера
с микрофоном будет представлена приложениям из пространства пользователя как
устройства захвата V4L2 и ALSA. Взаимосвязи между этими устройствами (при
использовании веб-камеры конечному пользователю не должно требоваться вручную
выбирать связанный с ней USB-микрофон), хотя драйверы и не предоставляют их
приложениям напрямую, обычно можно извлечь из sysfs.

С появлением всё более и более продвинутых устройств SoC текущий подход
перестаёт масштабироваться. Топологии устройств становятся всё более сложными
и не всегда могут быть представлены древовидной структурой. Аппаратные блоки
совместно используются разными функциями, создавая зависимости между, казалось
бы, не связанными друг с другом устройствами.

API абстракции ядра, такие как V4L2 и ALSA, предоставляют приложениям средства
для доступа к параметрам аппаратного обеспечения. Поскольку более новое
оборудование предоставляет всё большее число таких параметров, драйверам
приходится угадывать, что на самом деле требуется приложениям, на основе
ограниченной информации, тем самым реализуя политики, которые относятся
к пространству пользователя.

API медиаконтроллера нацелен на решение этих проблем.
