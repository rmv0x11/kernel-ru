***********************************
ioctls CEC_RECEIVE and CEC_TRANSMIT
***********************************

Имя
===

CEC_RECEIVE, CEC_TRANSMIT - Принять или передать CEC-сообщение

Синопсис
========

.. c:macro:: CEC_RECEIVE

``int ioctl(int fd, CEC_RECEIVE, struct cec_msg *argp)``

.. c:macro:: CEC_TRANSMIT

``int ioctl(int fd, CEC_TRANSMIT, struct cec_msg *argp)``

Аргументы
=========

``fd``
    Файловый дескриптор, возвращённый :c:func:`open()`.

``argp``
    Указатель на struct cec_msg.

Описание
========

Чтобы принять CEC-сообщение, приложение должно заполнить поле
``timeout`` структуры :c:type:`cec_msg` и передать её
:ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>`.
Если файловый дескриптор находится в неблокирующем режиме и нет ожидающих
принятых сообщений, то вызов вернёт -1 и установит errno в код ошибки
``EAGAIN``. Если файловый дескриптор находится в блокирующем режиме, ``timeout``
ненулевой и в течение ``timeout`` миллисекунд ни одно сообщение не пришло, то
вызов вернёт -1 и установит errno в код ошибки ``ETIMEDOUT``.

Принятое сообщение может быть:

1. сообщением, принятым от другого CEC-устройства (поле ``sequence`` будет
   равно 0, ``tx_status`` будет равно 0, а ``rx_status`` будет ненулевым).
2. результатом передачи более ранней неблокирующей передачи (поле ``sequence``
   будет ненулевым, ``tx_status`` будет ненулевым, а ``rx_status``
   будет равно 0).
3. ответом на более раннюю неблокирующую передачу (поле ``sequence`` будет
   ненулевым, ``tx_status`` будет равно 0, а ``rx_status`` будет ненулевым).

Чтобы отправить CEC-сообщение, приложение должно заполнить структуру
:c:type:`cec_msg` и передать её :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>`.
:ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>` доступен только если
установлен ``CEC_CAP_TRANSMIT``. Если в очереди передачи больше нет места,
то вызов вернёт -1 и установит errno в код ошибки ``EBUSY``.
В очереди передачи достаточно места для 18 сообщений (примерно 1 секунда
двухбайтовых сообщений). Обратите внимание, что фреймворк CEC ядра также будет
отвечать на core-сообщения (см. :ref:`cec-core-processing`), поэтому полностью
заполнять очередь передачи — не лучшая идея.

Если файловый дескриптор находится в неблокирующем режиме, то передача
вернёт 0, а результат передачи будет доступен через
:ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>` после завершения передачи.
Если для неблокирующей передачи было также указано ожидание ответа, то
ответ придёт в более позднем сообщении. Поле ``sequence`` может
использоваться для сопоставления как результатов передачи, так и ответов с исходной
передачей.

Обычно вызов :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>`, когда физический
адрес недействителен (например, из-за отключения), возвращает ``ENONET``.

Однако спецификация CEC разрешает отправку сообщений от «Unregistered» к
«TV», когда физический адрес недействителен, поскольку некоторые телевизоры подтягивают
вывод определения горячего подключения (hotplug detect) разъёма HDMI к низкому уровню,
когда переходят в режим ожидания или при переключении на другой вход.

Когда вывод определения горячего подключения переходит на низкий уровень, EDID исчезает,
а с ним и физический адрес, но кабель по-прежнему подключён, и CEC по-прежнему работает.
Для обнаружения/пробуждения устройства разрешено отправлять сообщения poll и «Image/Text
View On» от инициатора 0xf («Unregistered») к получателю 0 («TV»).

    * - __u32
      - ``timeout``
      - Тайм-аут в миллисекундах. Это время, в течение которого устройство будет ждать
	приёма сообщения, прежде чем истечёт тайм-аут. Если оно установлено в 0,
	то при вызове через :ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>` ожидание будет бесконечным.
	Если оно равно 0 и вызвано через :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>`,
	то оно будет заменено на 1000, если ``reply`` ненулевой, или
	проигнорировано, если ``reply`` равен 0.
    * - __u32
      - ``sequence``
      - Ненулевой порядковый номер автоматически назначается фреймворком CEC
	для всех передаваемых сообщений. Он используется фреймворком CEC, когда тот ставит
	результат передачи в очередь для неблокирующей передачи. Это позволяет приложению
	сопоставить принятое сообщение с исходной передачей.

	Кроме того, если неблокирующая передача будет ожидать ответа (т. е. ``timeout``
	был не 0), то поле ``sequence`` ответа будет установлено в значение sequence
	исходной передачи. Это позволяет приложению сопоставить
	принятое сообщение с исходной передачей.
    * - __u32
      - ``flags``
      - Флаги. См. :ref:`cec-msg-flags` для списка доступных флагов.
    * - __u8
      - ``msg[16]``
      - Полезная нагрузка сообщения. Для :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>` это поле заполняется
	приложением. Драйвер заполнит его для :ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>`.
	Для :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>` оно будет заполнено драйвером
	полезной нагрузкой сообщения-ответа, если был задан ``timeout``.
    * - __u8
      - ``reply``
      - Ждать, пока на это сообщение не придёт ответ. Если ``reply`` равен 0 и
	``timeout`` равен 0, то не ждать ответа, а вернуться после
	передачи сообщения. Игнорируется :ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>`.
	Случай, когда ``reply`` равен 0 (это опкод сообщения Feature Abort)
	и ``timeout`` ненулевой, специально разрешён, чтобы сделать
	возможным отправку сообщения и ожидание ответа Feature Abort в течение до
	``timeout`` миллисекунд. В этом случае ``rx_status`` будет установлен
	либо в :ref:`CEC_RX_STATUS_TIMEOUT <CEC-RX-STATUS-TIMEOUT>`, либо в
	:ref:`CEC_RX_STATUS_FEATURE_ABORT <CEC-RX-STATUS-FEATURE-ABORT>`.

	Если передаваемое сообщение — ``CEC_MSG_INITIATE_ARC``, то значения ``reply``
	``CEC_MSG_REPORT_ARC_INITIATED`` и ``CEC_MSG_REPORT_ARC_TERMINATED``
	обрабатываются иначе: любое из значений будет соответствовать обоим возможным ответам.
	Причина в том, что сообщение ``CEC_MSG_INITIATE_ARC`` — единственное CEC-сообщение,
	имеющее два возможных ответа помимо Feature Abort.
	Поле ``reply`` будет обновлено фактическим ответом, чтобы оно было
	синхронизировано с содержимым принятого сообщения.
    * - __u8
      - ``rx_status``

	:ref:`cec-rx-status` для возможных значений статуса.
    * - __u8

	При вызове :ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>` в неблокирующем режиме

	Обратите внимание, что при этом предполагается, что байт после Vendor ID — это

Возвращаемое значение
=====================

При успехе возвращается 0, при ошибке -1, и переменная ``errno`` устанавливается
соответствующим образом. Общие коды ошибок описаны в главе
:ref:`Generic Error Codes <gen-errors>`.

:ref:`ioctl CEC_RECEIVE <CEC_RECEIVE>` может возвращать следующие
коды ошибок:

EAGAIN
    В очереди приёма нет сообщений, а файловый дескриптор находится в неблокирующем режиме.

ETIMEDOUT
    Истёк ``timeout`` во время ожидания сообщения.

ERESTARTSYS
    Ожидание сообщения было прервано (например, нажатием Ctrl-C).

:ref:`ioctl CEC_TRANSMIT <CEC_TRANSMIT>` может возвращать следующие
коды ошибок:

ENOTTY
    Возможность ``CEC_CAP_TRANSMIT`` не была установлена, поэтому этот ioctl не поддерживается.

EPERM
    CEC-адаптер не сконфигурирован, т. е. :ref:`ioctl CEC_ADAP_S_LOG_ADDRS <CEC_ADAP_S_LOG_ADDRS>`
    ни разу не вызывался, или ``CEC_MSG_FL_RAW`` был использован из процесса, который
    не имел привилегии ``CAP_SYS_RAWIO``.

ENONET
    CEC-адаптер не сконфигурирован, т. е. :ref:`ioctl CEC_ADAP_S_LOG_ADDRS <CEC_ADAP_S_LOG_ADDRS>`
    был вызван, но физический адрес недействителен, поэтому логический адрес не был получен.
    В этом случае делается исключение для передач от инициатора 0xf («Unregistered»)
    к получателю 0 («TV»). В этом случае передача пройдёт как обычно.

EBUSY
    Другой файловый дескриптор находится в эксклюзивном режиме follower или initiator, либо файловый дескриптор
    находится в режиме ``CEC_MODE_NO_INITIATOR``. Это также возвращается, если очередь
    передачи заполнена.

EINVAL
    Содержимое структуры :c:type:`cec_msg` недействительно.

ERESTARTSYS
    Ожидание успешной передачи было прервано (например, нажатием Ctrl-C).
