Внедрение ошибок в CEC Pin Framework
====================================

CEC Pin Framework — это базовый CEC-фреймворк для CEC-оборудования, которое
обладает только низкоуровневой поддержкой шины CEC. Большинство современного
оборудования имеет высокоуровневую поддержку CEC, при которой оборудование
само управляет шиной CEC, но некоторые старые устройства не настолько
продвинуты. Вместе с тем этот фреймворк также позволяет подключить вывод CEC
к GPIO, например, на Raspberry Pi, и тем самым вы получаете CEC-адаптер.

Особенно интересным это делает то, что, поскольку мы полностью контролируем
шину, легко реализовать внедрение ошибок. Это идеально подходит для проверки
того, насколько хорошо CEC-адаптеры справляются с ошибочными ситуациями.

В настоящее время этот фреймворк поддерживают только драйвер cec-gpio (когда
линия CEC напрямую подключена к подтянутой к питанию линии GPIO) и
drm-драйвер AllWinner A10/A20.

Если включён параметр ``CONFIG_CEC_PIN_ERROR_INJ``, то внедрение ошибок
становится доступным через debugfs. В частности, в каталоге
``/sys/kernel/debug/cec/cecX/`` теперь появляется файл ``error-inj``.

.. note::

    Команды внедрения ошибок не являются стабильным ABI и могут измениться в
    будущем.

С помощью ``cat error-inj`` можно увидеть как доступные команды, так и текущее
состояние внедрения ошибок::

	$ cat /sys/kernel/debug/cec/cec0/error-inj
	# Clear error injections:
	#   clear          clear all rx and tx error injections
	#   rx-clear       clear all rx error injections
	#   tx-clear       clear all tx error injections
	#   <op> clear     clear all rx and tx error injections for <op>
	#   <op> rx-clear  clear all rx error injections for <op>
	#   <op> tx-clear  clear all tx error injections for <op>
	#
	# RX error injection settings:
	#   rx-no-low-drive                    do not generate low-drive pulses
	#
	# RX error injection:
	#   <op>[,<mode>] rx-nack              NACK the message instead of sending an ACK
	#   <op>[,<mode>] rx-low-drive <bit>   force a low-drive condition at this bit position
	#   <op>[,<mode>] rx-add-byte          add a spurious byte to the received CEC message
	#   <op>[,<mode>] rx-remove-byte       remove the last byte from the received CEC message
	#    any[,<mode>] rx-arb-lost [<poll>] generate a POLL message to trigger an arbitration lost
	#
	# TX error injection settings:
	#   tx-ignore-nack-until-eom           ignore early NACKs until EOM
	#   tx-custom-low-usecs <usecs>        define the 'low' time for the custom pulse
	#   tx-custom-high-usecs <usecs>       define the 'high' time for the custom pulse
	#   tx-custom-pulse                    transmit the custom pulse once the bus is idle
	#   tx-glitch-low-usecs <usecs>        define the 'low' time for the glitch pulse
	#   tx-glitch-high-usecs <usecs>       define the 'high' time for the glitch pulse
	#   tx-glitch-falling-edge             send the glitch pulse after every falling edge
	#   tx-glitch-rising-edge              send the glitch pulse after every rising edge
	#
	# TX error injection:
	#   <op>[,<mode>] tx-no-eom            don't set the EOM bit
	#   <op>[,<mode>] tx-early-eom         set the EOM bit one byte too soon
	#   <op>[,<mode>] tx-add-bytes <num>   append <num> (1-255) spurious bytes to the message
	#   <op>[,<mode>] tx-remove-byte       drop the last byte from the message
	#   <op>[,<mode>] tx-short-bit <bit>   make this bit shorter than allowed
	#   <op>[,<mode>] tx-long-bit <bit>    make this bit longer than allowed
	#   <op>[,<mode>] tx-custom-bit <bit>  send the custom pulse instead of this bit
	#   <op>[,<mode>] tx-short-start       send a start pulse that's too short
	#   <op>[,<mode>] tx-long-start        send a start pulse that's too long
	#   <op>[,<mode>] tx-custom-start      send the custom pulse instead of the start pulse
	#   <op>[,<mode>] tx-last-bit <bit>    stop sending after this bit
	#   <op>[,<mode>] tx-low-drive <bit>   force a low-drive condition at this bit position
	#
	# <op>       CEC message opcode (0-255) or 'any'
	# <mode>     'once' (default), 'always', 'toggle' or 'off'
	# <bit>      CEC message bit (0-159)
	#            10 bits per 'byte': bits 0-7: data, bit 8: EOM, bit 9: ACK
	# <poll>     CEC poll message used to test arbitration lost (0x00-0xff, default 0x0f)
	# <usecs>    microseconds (0-10000000, default 1000)

	clear

Команды внедрения ошибок можно записать в ``error-inj`` с помощью
``echo 'cmd' >error-inj`` или ``cat cmd.txt >error-inj``. Вывод ``cat error-inj``
содержит текущие команды внедрения ошибок. Этот вывод можно сохранить в файл и
позже использовать его как входные данные для ``error-inj``.

Базовый синтаксис
-----------------

Ведущие пробелы и табуляции игнорируются. Если следующий символ — ``#`` или
достигнут конец строки, то вся строка игнорируется. В противном случае ожидается
команда.

Команды внедрения ошибок делятся на две основные группы: относящиеся к приёму
CEC-сообщений и относящиеся к передаче CEC-сообщений. Кроме того, есть команды
для очистки существующих команд внедрения ошибок и для создания
пользовательских импульсов на шине CEC.

Большинство команд внедрения ошибок могут выполняться для конкретных опкодов CEC
или для всех опкодов (``any``). У каждой команды также есть «режим» (mode),
который может быть ``off`` (можно использовать для отключения существующей
команды внедрения ошибок), ``once`` (по умолчанию), который вызовет внедрение
ошибки только один раз для следующего принятого или переданного сообщения,
``always`` — для постоянного срабатывания внедрения ошибки и ``toggle`` —
для переключения внедрения ошибки между включённым и выключенным состоянием при
каждой передаче или приёме.

Так, «``any rx-nack``» вызовет NACK для следующего принятого CEC-сообщения,
«``any,always rx-nack``» вызовет NACK для всех принятых CEC-сообщений, а
«``0x82,toggle rx-nack``» вызовет NACK только в том случае, если было принято
сообщение Active Source, и сделает это только для каждого второго принятого
сообщения.

После того как ошибка была внедрена в режиме ``once``, команда внедрения ошибки
очищается автоматически, поэтому ``once`` — это одноразовое действие.

Все сочетания ``<op>`` и команд внедрения ошибок могут сосуществовать. Так что
вот это вполне допустимо::

	0x9e tx-add-bytes 1
	0x9e tx-early-eom
	0x9f tx-add-bytes 2
	any rx-nack

Все четыре команды внедрения ошибок будут активны одновременно.

Однако если задано то же сочетание ``<op>`` и команды, но с другими
аргументами::

	0x9e tx-add-bytes 1
	0x9e tx-add-bytes 2

то вторая перезапишет первую.

Очистка внедрений ошибок
------------------------

``clear``
    Очистить все внедрения ошибок.

``rx-clear``
    Очистить все внедрения ошибок приёма.

``tx-clear``
    Очистить все внедрения ошибок передачи.

``<op> clear``
    Очистить все внедрения ошибок для заданного опкода.

``<op> rx-clear``
    Очистить все внедрения ошибок приёма для заданного опкода.

``<op> tx-clear``
    Очистить все внедрения ошибок передачи для заданного опкода.

Приём сообщений
---------------

``<op>[,<mode>] rx-nack``
    Выдавать NACK для широковещательных сообщений и сообщений, адресованных
    этому CEC-адаптеру. Каждый байт сообщения будет получать NACK на случай,
    если передающая сторона продолжит передачу после того, как первый байт
    получил NACK.

``<op>[,<mode>] rx-low-drive <bit>``
    Принудительно создать состояние Low Drive в этой позиции бита. Если ``<op>``
    задаёт конкретный опкод CEC, то позиция бита должна быть не меньше 18,
    иначе опкод ещё не был принят. Это проверяет, может ли передающая сторона
    корректно обработать состояние Low Drive и корректно сообщить об ошибке.
    Обратите внимание, что Low Drive в первых 4 битах может также быть
    интерпретирован передающей стороной как состояние Arbitration Lost. Это
    зависит от реализации.

``<op>[,<mode>] rx-add-byte``
    Добавить ложный байт 0x55 к принятому CEC-сообщению при условии, что
    сообщение имело длину 15 байт или менее. Это полезно для проверки
    высокоуровневого протокола, поскольку ложные байты должны игнорироваться.

``<op>[,<mode>] rx-remove-byte``
    Удалить последний байт из принятого CEC-сообщения при условии, что оно
    было длиной не менее 2 байт. Это полезно для проверки высокоуровневого
    протокола, поскольку слишком короткие сообщения должны игнорироваться.

``<op>[,<mode>] rx-arb-lost <poll>``
    Сгенерировать сообщение POLL, чтобы вызвать состояние Arbitration Lost.
    Эта команда допускается только для значений ``<op>``, равных ``next`` или
    ``all``. Как только будет принят стартовый бит, CEC-адаптер переключится в
    режим передачи и передаст сообщение POLL. По умолчанию это 0x0f, но его
    также можно задать явно через аргумент ``<poll>``.

    Эту команду можно использовать для проверки состояния Arbitration Lost в
    удалённой передающей стороне CEC. Арбитраж происходит, когда два
    CEC-адаптера начинают отправлять сообщение одновременно. В этом случае
    побеждает инициатор с наибольшим числом ведущих нулей, а другая передающая
    сторона вынуждена прекратить передачу («Arbitration Lost»). Это очень
    трудно проверить иначе, кроме как с помощью этой команды внедрения ошибок.

    Это не работает, если удалённая передающая сторона CEC имеет логический
    адрес 0 («TV»), поскольку она всегда будет побеждать.

``rx-no-low-drive``
    Приёмник будет игнорировать ситуации, которые обычно вызвали бы импульс
    Low Drive (3,6 мс). Обычно это делается, если при приёме сообщения
    обнаруживается ложный импульс, и это сигнализирует передающей стороне о том,
    что сообщение необходимо передать повторно, поскольку приёмник запутался.
    Отключение этого полезно для проверки того, как другие CEC-устройства
    обрабатывают сбои, путём обеспечения того, что мы не будем тем, кто
    генерирует Low Drive.

Передача сообщений
------------------

``tx-ignore-nack-until-eom``
    Эта настройка изменяет поведение при передаче CEC-сообщений. Обычно, как
    только приёмник выдаёт NACK на байт, передача прекращается, но спецификация
    также допускает, что всё сообщение передаётся полностью и только в конце
    передающая сторона смотрит на бит ACK. Это не рекомендуемое поведение,
    поскольку нет смысла занимать шину CEC дольше, чем это строго необходимо.
    Особенно учитывая, насколько медленна эта шина.

    Эту настройку можно использовать для проверки того, насколько хорошо
    приёмник справляется с передающими сторонами, которые игнорируют NACK до
    самого конца сообщения.

``<op>[,<mode>] tx-no-eom``
    Не устанавливать бит EOM. Обычно у последнего байта сообщения установлен
    бит EOM (End-Of-Message). С этой командой передача просто прекратится, так
    и не отправив EOM. Это можно использовать для проверки того, как приёмник
    обрабатывает такой случай. Обычно у приёмников есть тайм-аут, по истечении
    которого они возвращаются в состояние Idle.

``<op>[,<mode>] tx-early-eom``
    Установить бит EOM на один байт раньше. Это, очевидно, работает только для
    сообщений из двух и более байт. Бит EOM будет установлен у предпоследнего
    байта, а не у последнего. В этом случае приёмник должен проигнорировать
    последний байт. Поскольку получившееся сообщение по той же причине, скорее
    всего, окажется слишком коротким, обычно игнорируется всё сообщение целиком.
    После передачи последнего байта приёмник должен находиться в состоянии Idle.

``<op>[,<mode>] tx-add-bytes <num>``
    Добавить в конец сообщения ``<num>`` (1-255) ложных байт. Дополнительные
    байты имеют значение, равное позиции байта в сообщении. Так что если вы
    передаёте двухбайтовое сообщение (например, сообщение Get CEC Version) и
    добавляете 2 байта, то полное сообщение, принятое удалённым CEC-адаптером,
    будет ``0x40 0x9f 0x02 0x03``.

    Эту команду можно использовать для проверки переполнений буфера в приёмнике.
    Например, что он делает, когда принимает больше максимального размера
    сообщения в 16 байт.

``<op>[,<mode>] tx-remove-byte``
    Отбросить последний байт сообщения при условии, что сообщение имеет длину
    не менее двух байт. Приёмник должен игнорировать слишком короткие сообщения.

``<op>[,<mode>] tx-short-bit <bit>``
    Сделать период этого бита короче допустимого. Позиция бита не может быть
    битом Ack. Если ``<op>`` задаёт конкретный опкод CEC, то позиция бита должна
    быть не меньше 18, иначе опкод ещё не был принят. Обычно период бита данных
    составляет от 2,05 до 2,75 миллисекунды. С этой командой период этого бита
    равен 1,8 миллисекунды; это достигается за счёт сокращения времени, в
    течение которого шина CEC находится в высоком состоянии. Этот период бита
    меньше допустимого, и приёмник должен ответить состоянием Low Drive.

    Эта команда игнорируется для битов со значением 0 в позициях с 0 по 3. Это
    связано с тем, что приёмник также ищет состояние Arbitration Lost в этих
    первых четырёх битах, и не определено, что произойдёт, если он увидит
    слишком короткий бит 0.

``<op>[,<mode>] tx-long-bit <bit>``
    Сделать период этого бита длиннее допустимого. Позиция бита не может быть
    битом Ack. Если ``<op>`` задаёт конкретный опкод CEC, то позиция бита должна
    быть не меньше 18, иначе опкод ещё не был принят. Обычно период бита данных
    составляет от 2,05 до 2,75 миллисекунды. С этой командой период этого бита
    равен 2,9 миллисекунды; это достигается за счёт увеличения времени, в
    течение которого шина CEC находится в высоком состоянии.

    Хотя этот период бита длиннее допустимого, не определено, что сделает
    приёмник. Он может просто принять его, а может выйти по тайм-ауту и
    вернуться в состояние Idle. К сожалению, спецификация CEC об этом умалчивает.

    Эта команда игнорируется для битов со значением 0 в позициях с 0 по 3. Это
    связано с тем, что приёмник также ищет состояние Arbitration Lost в этих
    первых четырёх битах, и не определено, что произойдёт, если он увидит
    слишком длинный бит 0.

``<op>[,<mode>] tx-short-start``
    Сделать период этого стартового бита короче допустимого. Обычно период
    стартового бита составляет от 4,3 до 4,7 миллисекунды. С этой командой
    период стартового бита равен 4,1 миллисекунды; это достигается за счёт
    сокращения времени, в течение которого шина CEC находится в высоком
    состоянии. Этот период стартового бита меньше допустимого, и при его
    обнаружении приёмник должен вернуться в состояние Idle.

``<op>[,<mode>] tx-long-start``
    Сделать период этого стартового бита длиннее допустимого. Обычно период
    стартового бита составляет от 4,3 до 4,7 миллисекунды. С этой командой
    период стартового бита равен 5 миллисекундам; это достигается за счёт
    увеличения времени, в течение которого шина CEC находится в высоком
    состоянии. Этот период стартового бита больше допустимого, и при его
    обнаружении приёмник должен вернуться в состояние Idle.

    Хотя этот период стартового бита длиннее допустимого, не определено, что
    сделает приёмник. Он может просто принять его, а может выйти по тайм-ауту и
    вернуться в состояние Idle. К сожалению, спецификация CEC об этом умалчивает.

``<op>[,<mode>] tx-last-bit <bit>``
    Просто прекратить передачу после этого бита. Если ``<op>`` задаёт конкретный
    опкод CEC, то позиция бита должна быть не меньше 18, иначе опкод ещё не был
    принят. Эту команду можно использовать для проверки того, как приёмник
    реагирует, когда сообщение просто внезапно прекращается. Он должен выйти по
    тайм-ауту и вернуться в состояние Idle.

``<op>[,<mode>] tx-low-drive <bit>``
    Принудительно создать состояние Low Drive в этой позиции бита. Если ``<op>``
    задаёт конкретный опкод CEC, то позиция бита должна быть не меньше 18, иначе
    опкод ещё не был принят. Это можно использовать для проверки того, как
    приёмник обрабатывает состояния Low Drive. Обратите внимание, что если это
    происходит в позициях битов 0-3, приёмник может интерпретировать это как
    состояние Arbitration Lost. Это зависит от реализации.

Пользовательские импульсы
--------------------------

``tx-custom-low-usecs <usecs>``
    Это задаёт длительность в микросекундах, в течение которой пользовательский
    импульс удерживает линию CEC в низком состоянии. По умолчанию — 1000
    микросекунд.

``tx-custom-high-usecs <usecs>``
    Это задаёт длительность в микросекундах, в течение которой пользовательский
    импульс удерживает линию CEC в высоком состоянии (если только за это время
    другой CEC-адаптер не притянет её к низкому уровню). По умолчанию — 1000
    микросекунд. Полный период пользовательского импульса равен
    ``tx-custom-low-usecs + tx-custom-high-usecs``.

``<op>[,<mode>] tx-custom-bit <bit>``
    Отправить пользовательский бит вместо обычного бита данных. Позиция бита не
    может быть битом Ack. Если ``<op>`` задаёт конкретный опкод CEC, то позиция
    бита должна быть не меньше 18, иначе опкод ещё не был принят.

``<op>[,<mode>] tx-custom-start``
    Отправить пользовательский бит вместо обычного стартового бита.

``tx-custom-pulse``
    Передать один пользовательский импульс, как только шина CEC освободится.

Сбойные импульсы (glitch)
-------------------------

Это эмулирует то, что происходит, если сигнал на линии CEC наблюдает ложные
импульсы. Обычно это происходит после спадающего или нарастающего фронта, где
имеется кратковременное колебание напряжения, которое, если CEC-оборудование не
выполняет деглитчинг, может быть воспринято как ложный импульс и может вызвать
состояние Low Drive или повредить данные.

``tx-glitch-low-usecs <usecs>``
    Это задаёт длительность в микросекундах, в течение которой сбойный импульс
    удерживает линию CEC в низком состоянии. По умолчанию — 1 микросекунда.
    Диапазон — 0-100 микросекунд. Если 0, то сбойный импульс генерироваться не
    будет.

``tx-glitch-high-usecs <usecs>``
    Это задаёт длительность в микросекундах, в течение которой сбойный импульс
    удерживает линию CEC в высоком состоянии (если только за это время другой
    CEC-адаптер не притянет её к низкому уровню). По умолчанию — 1 микросекунда.
    Диапазон — 0-100 микросекунд. Если 0, то сбойный импульс генерироваться не
    будет. Полный период сбойного импульса равен
    ``tx-custom-low-usecs + tx-custom-high-usecs``.

``tx-glitch-falling-edge``
    Отправить сбойный импульс сразу после спадающего фронта.

``tx-glitch-rising-edge``
    Отправить сбойный импульс сразу после нарастающего фронта.
