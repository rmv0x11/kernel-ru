=================================
Версионирование модулей DWARF
=================================

Введение
========

Когда включён CONFIG_MODVERSIONS, версии символов для модулей
обычно вычисляются из препроцессированного исходного кода с помощью
утилиты **genksyms**.  Однако это несовместимо с языками вроде
Rust, где исходный код содержит недостаточно информации о
получающемся ABI. Если выбраны CONFIG_GENDWARFKSYMS (и CONFIG_DEBUG_INFO),
вместо неё для вычисления версий символов используется **gendwarfksyms**,
который опирается на отладочную информацию DWARF, содержащую необходимые
сведения об итоговом ABI модуля.

Зависимости
-----------

gendwarfksyms зависит от библиотек libelf, libdw и zlib.

Вот несколько примеров того, как установить эти зависимости:

* Arch Linux и производные::

	sudo pacman --needed -S libelf zlib

* Debian, Ubuntu и производные::

	sudo apt install libelf-dev libdw-dev zlib1g-dev

* Fedora и производные::

	sudo dnf install elfutils-libelf-devel elfutils-devel zlib-devel

* openSUSE и производные::

	sudo zypper install libelf-devel libdw-devel zlib-devel

Использование
-------------

gendwarfksyms принимает список объектных файлов в командной строке и
список имён символов (по одному на строку) на стандартном вводе::

	Usage: gendwarfksyms [options] elf-object-file ... < symbol-list

	Options:
	  -d, --debug          Print debugging information
	      --dump-dies      Dump DWARF DIE contents
	      --dump-die-map   Print debugging information about die_map changes
	      --dump-types     Dump type strings
	      --dump-versions  Dump expanded type strings used for symbol versions
	  -s, --stable         Support kABI stability features
	  -T, --symtypes file  Write a symtypes file
	  -h, --help           Print this message


Доступность информации о типах
==============================

Хотя символы обычно экспортируются в той же единице трансляции (TU),
где они определены, единице трансляции вполне допустимо экспортировать
внешние символы. Например, так поступают при вычислении версий символов
для экспортов в отдельном ассемблерном коде.

Чтобы гарантировать, что компилятор выдаст необходимую информацию о типах
DWARF в той единице трансляции, где символы действительно экспортируются,
gendwarfksyms добавляет указатель на экспортируемые символы в макрос
`EXPORT_SYMBOL()` с помощью следующего макроса::

	#define __GENDWARFKSYMS_EXPORT(sym)				\
		static typeof(sym) *__gendwarfksyms_ptr_##sym __used	\
			__section(".discard.gendwarfksyms") = &sym;


Когда указатель на символ обнаружен в DWARF, gendwarfksyms может использовать
его тип для вычисления версий символов, даже если сам символ определён
в другом месте. Ожидается, что имя указателя на символ начинается с
`__gendwarfksyms_ptr_`, за которым следует имя экспортируемого символа.

Формат вывода symtypes
======================

Подобно genksyms, gendwarfksyms поддерживает запись файла symtypes
для каждого обработанного объекта; такой файл содержит типы для
экспортируемых символов и каждый ссылающийся тип, который использовался при
вычислении версий символов. Эти файлы могут быть полезны при попытке
определить, что именно привело к изменению версий символов между сборками.
Чтобы генерировать файлы symtypes во время сборки ядра, установите
`KBUILD_SYMTYPES=1`.

В соответствии с существующим форматом первый столбец каждой строки содержит
либо ссылку на тип, либо имя символа. Ссылки на типы имеют
однобуквенный префикс, за которым следуют "#" и имя типа. Поддерживаются
четыре типа ссылок::

	e#<type> = enum
	s#<type> = struct
	t#<type> = typedef
	u#<type> = union

Имена типов, содержащие пробелы, заключаются в одинарные кавычки, например::

	s#'core::result::Result<u8, core::num::error::ParseIntError>'

Остальная часть строки содержит строку типа. В отличие от genksyms, который
выдаёт строки типов в стиле C, gendwarfksyms использует тот же простой
разобранный формат DWARF, что и **--dump-dies**, но со ссылками на типы
вместо полностью развёрнутых строк.

Поддержание стабильного kABI
============================

Сопровождающим дистрибутивов часто нужна возможность вносить ABI-совместимые
изменения в структуры данных ядра из-за LTS-обновлений или бэкпортов. Использование
традиционного `#ifndef __GENKSYMS__` для скрытия этих изменений от
версионирования символов не сработает при обработке объектных файлов. Чтобы
поддержать этот сценарий использования, gendwarfksyms предоставляет средства
стабильности kABI, предназначенные для скрытия изменений, которые не влияют на
ABI, при вычислении версий. Все эти средства активируются флагом командной строки
**--stable** и не используются в ядре mainline. Чтобы задействовать средства
стабильности во время сборки ядра, установите `KBUILD_GENDWARFKSYMS_STABLE=1`.

Примеры использования этих средств приведены в каталоге
**scripts/gendwarfksyms/examples**, включая вспомогательные макросы
для аннотирования исходного кода. Обратите внимание, что, поскольку эти средства
используются только для преобразования входных данных версионирования символов,
пользователь сам отвечает за то, чтобы его изменения действительно не нарушали ABI.

Правила kABI
------------

Правила kABI позволяют дистрибутивам тонко настраивать определённые части
вывода gendwarfksyms и тем самым управлять тем, как вычисляются версии
символов. Эти правила определяются в секции
`.discard.gendwarfksyms.kabi_rules` объектного файла и
состоят из простых строк, завершающихся нулём, со следующей структурой::

	version\0type\0target\0value\0

Эта последовательность строк повторяется столько раз, сколько необходимо, чтобы
выразить все правила. Поля таковы:

- `version`: Обеспечивает обратную совместимость для будущих изменений
  структуры. В настоящее время ожидается значение "1".
- `type`: Указывает тип применяемого правила.
- `target`: Задаёт цель правила, обычно полностью
  квалифицированное имя записи отладочной информации DWARF (Debugging Information Entry, DIE).
- `value`: Предоставляет данные, специфичные для правила.

Например, следующие вспомогательные макросы можно использовать для задания правил
в исходном коде::

	#define ___KABI_RULE(hint, target, value)                            \
		static const char __PASTE(__gendwarfksyms_rule_,             \
					  __COUNTER__)[] __used __aligned(1) \
			__section(".discard.gendwarfksyms.kabi_rules") =     \
				"1\0" #hint "\0" target "\0" value

	#define __KABI_RULE(hint, target, value) \
		___KABI_RULE(hint, #target, #value)


В настоящее время поддерживаются только правила, рассмотренные в этом разделе, но
формат достаточно расширяем, чтобы по мере необходимости можно было добавлять
дальнейшие правила.

Управление видимостью определений
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Объявление может превратиться в полное определение, когда в единицу трансляции
подтягиваются дополнительные включаемые файлы. Это меняет версии любого
символа, который ссылается на данный тип, даже если ABI остаётся неизменным.
Поскольку отказаться от включаемых файлов без нарушения сборки бывает невозможно,
правило `declonly` можно использовать, чтобы задать тип как существующий только
в виде объявления (declaration-only), даже если отладочная информация содержит
полное определение.

Ожидается, что поля правила будут такими:

- `type`: "declonly"
- `target`: Полностью квалифицированное имя целевой структуры данных
  (как показано в выводе **--dump-dies**).
- `value`: Это поле игнорируется.

С помощью макроса `__KABI_RULE` это правило можно определить так::

	#define KABI_DECLONLY(fqn) __KABI_RULE(declonly, fqn, )

Пример использования::

	struct s {
		/* definition */
	};

	KABI_DECLONLY(s);

Добавление перечислителей
~~~~~~~~~~~~~~~~~~~~~~~~~~

Для перечислений все перечислители и их значения учитываются при вычислении
версий символов, что становится проблемой, если позже нам потребуется добавить
новые перечислители без изменения версий символов. Правило `enumerator_ignore`
позволяет нам скрыть именованные перечислители из входных данных.

Ожидается, что поля правила будут такими:

- `type`: "enumerator_ignore"
- `target`: Полностью квалифицированное имя целевого перечисления
  (как показано в выводе **--dump-dies**) и имя поля
  перечислителя, разделённые пробелом.
- `value`: Это поле игнорируется.

С помощью макроса `__KABI_RULE` это правило можно определить так::

	#define KABI_ENUMERATOR_IGNORE(fqn, field) \
		__KABI_RULE(enumerator_ignore, fqn field, )

Пример использования::

	enum e {
		A, B, C, D,
	};

	KABI_ENUMERATOR_IGNORE(e, B);
	KABI_ENUMERATOR_IGNORE(e, C);

Если перечисление дополнительно содержит конечный маркер и новые значения нужно
добавить в середину, нам может потребоваться использовать старое значение для
последнего перечислителя при вычислении версий. Правило `enumerator_value`
позволяет нам переопределить значение перечислителя для вычисления версий:

- `type`: "enumerator_value"
- `target`: Полностью квалифицированное имя целевого перечисления
  (как показано в выводе **--dump-dies**) и имя поля
  перечислителя, разделённые пробелом.
- `value`: Целочисленное значение, используемое для поля.

С помощью макроса `__KABI_RULE` это правило можно определить так::

	#define KABI_ENUMERATOR_VALUE(fqn, field, value) \
		__KABI_RULE(enumerator_value, fqn field, value)

Пример использования::

	enum e {
		A, B, C, LAST,
	};

	KABI_ENUMERATOR_IGNORE(e, C);
	KABI_ENUMERATOR_VALUE(e, LAST, 2);

Управление изменениями размера структуры
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Структура данных может быть частично непрозрачной для модулей, если её выделение
обрабатывается основным ядром, а модулям нужен доступ лишь к некоторым из её
членов. В этой ситуации можно добавлять новые члены в конец структуры,
не нарушая ABI, при условии, что компоновка исходных членов остаётся
неизменной.

Чтобы добавить новые члены, мы можем скрыть их от версионирования символов, как
описано в разделе :ref:`Скрытие членов <hiding_members>`, но мы не можем
скрыть увеличение размера структуры. Правило `byte_size` позволяет нам
переопределить размер структуры, используемый для версионирования символов.

Ожидается, что поля правила будут такими:

- `type`: "byte_size"
- `target`: Полностью квалифицированное имя целевой структуры данных
  (как показано в выводе **--dump-dies**).
- `value`: Положительное десятичное число, указывающее размер структуры
  в байтах.

С помощью макроса `__KABI_RULE` это правило можно определить так::

	#define KABI_BYTE_SIZE(fqn, value) \
		__KABI_RULE(byte_size, fqn, value)

Пример использования::

	struct s {
		/* Unchanged original members */
		unsigned long a;
		void *p;

		/* Appended new members */
		KABI_IGNORE(0, unsigned long n);
	};

	KABI_BYTE_SIZE(s, 16);

Переопределение строк типов
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

В редких ситуациях, когда дистрибутивам приходится вносить значительные изменения в
структуры данных, которые в остальном непрозрачны, но непреднамеренно попали
в опубликованный ABI, поддержание стабильных версий символов с помощью более
точечных правил kABI может стать утомительным. Правило `type_string` позволяет нам
переопределить полную строку типа для типа или символа и даже добавить
типы для версионирования, которых больше нет в ядре.

Ожидается, что поля правила будут такими:

- `type`: "type_string"
- `target`: Полностью квалифицированное имя целевой структуры данных
  (как показано в выводе **--dump-dies**) или символа.
- `value`: Корректная строка типа (как показано в выводе **--symtypes**)),
  которую следует использовать вместо реального типа.

С помощью макроса `__KABI_RULE` это правило можно определить так::

	#define KABI_TYPE_STRING(type, str) \
		___KABI_RULE("type_string", type, str)

Пример использования::

	/* Override type for a structure */
	KABI_TYPE_STRING("s#s",
		"structure_type s { "
			"member base_type int byte_size(4) "
				"encoding(5) n "
			"data_member_location(0) "
		"} byte_size(8)");

	/* Override type for a symbol */
	KABI_TYPE_STRING("my_symbol", "variable s#s");

Правило `type_string` следует использовать только как крайнее средство, если
поддержание стабильных версий символов не может быть разумно достигнуто другими
способами. Переопределение строки типа повышает риск того, что реальные
нарушения ABI останутся незамеченными, поскольку оно скрывает все изменения типа.

Добавление членов структуры
----------------------------

Пожалуй, самое распространённое ABI-совместимое изменение — это добавление члена в
структуру данных ядра. Когда изменения структуры предвидятся,
сопровождающие дистрибутивов могут заранее зарезервировать место в
структуре и задействовать его позже, не нарушая ABI. Если
изменения нужны для структур данных без зарезервированного места, вместо этого
потенциально можно использовать существующие дыры выравнивания. Хотя для
такого рода изменений можно было бы добавить правила kABI, использование
объединений обычно является более естественным методом. В этом разделе описана
поддержка в gendwarfksyms использования зарезервированного места в структурах данных
и скрытия членов, которые не меняют ABI, при вычислении версий символов.

Резервирование места и замена членов
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Место обычно резервируют для последующего использования, добавляя целочисленные типы или
массивы в конец структуры данных, но можно использовать любой тип. Каждый
зарезервированный член нуждается в уникальном имени, но, поскольку фактическое назначение обычно
неизвестно в момент резервирования места, для удобства имена, которые
начинаются с `__kabi_`, не учитываются при вычислении версий символов::

	struct s {
		long a;
		long __kabi_reserved_0; /* reserved for future use */
	};

Зарезервированное место можно задействовать, обернув член в
объединение, которое включает исходный тип и заменяющий член::

	struct s {
		long a;
		union {
			long __kabi_reserved_0; /* original type */
			struct b b; /* replaced field */
		};
	};

Если при резервировании места использовалась схема именования `__kabi_`, имя
первого члена объединения должно начинаться с `__kabi_reserved`. Это
гарантирует, что при вычислении версий используется исходный тип, но имя
снова не учитывается. Остальная часть объединения игнорируется.

Если мы заменяем член, который не следует этому соглашению об именовании,
нам также нужно сохранить исходное имя, чтобы избежать изменения версий,
что можно сделать, изменив имя первого члена объединения так, чтобы оно начиналось с
`__kabi_renamed`, за которым следует исходное имя.

Примеры включают макросы `KABI_(RESERVE|USE|REPLACE)*`, которые помогают
упростить процесс, а также гарантируют, что заменяющий член корректно
выровнен и его размер не превысит зарезервированное место.

.. _hiding_members:

Скрытие членов
~~~~~~~~~~~~~~~

Предсказать, какие структуры потребуют изменений в течение срока поддержки,
не всегда возможно, и в таком случае может прийтись прибегнуть
к размещению новых членов в существующих дырах выравнивания::

	struct s {
		int a;
		/* a 4-byte alignment hole */
		unsigned long b;
	};


Хотя это не изменит размер структуры данных, нужно иметь возможность
скрыть добавленные члены от версионирования символов. Подобно
зарезервированным полям, этого можно достичь, обернув добавленный
член в объединение, у которого имя одного из полей начинается с
`__kabi_ignored`::

	struct s {
		int a;
		union {
			char __kabi_ignored_0;
			int n;
		};
		unsigned long b;
	};

С **--stable** обе версии дают одинаковую версию символа. Примеры
включают макрос `KABI_IGNORE` для упрощения кода.
