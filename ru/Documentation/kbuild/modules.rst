================================
Сборка внешних модулей
================================

Этот документ описывает, как собрать модуль ядра, находящийся вне дерева исходников (out-of-tree).

Введение
========

«kbuild» — это система сборки, используемая ядром Linux. Модули обязаны использовать
kbuild, чтобы оставаться совместимыми с изменениями в инфраструктуре сборки и
получать правильные флаги для компилятора. Предоставляется функциональность для сборки модулей
как внутри дерева (in-tree), так и вне дерева (out-of-tree). Способ сборки
в обоих случаях схож, и все модули изначально разрабатываются и собираются
вне дерева.

В этом документе рассматривается информация, ориентированная на разработчиков, заинтересованных
в сборке модулей вне дерева (или «внешних»). Автор
внешнего модуля должен предоставить makefile, который скрывает большую часть
сложности, чтобы для сборки модуля достаточно было ввести «make». Это
легко достижимо, и полный пример будет представлен в
разделе `Создание файла Kbuild для внешнего модуля`_.


Как собирать внешние модули
===========================

Чтобы собирать внешние модули, у вас должно быть доступно заранее собранное ядро,
содержащее файлы конфигурации и заголовочные файлы, используемые при сборке.
Кроме того, ядро должно быть собрано с включённой поддержкой модулей. Если вы
используете ядро из дистрибутива, то ваш дистрибутив предоставляет пакет для запущенного
у вас ядра.

Альтернатива — использовать цель «make» «modules_prepare». Это
гарантирует, что ядро содержит требуемую информацию. Эта цель
существует исключительно как простой способ подготовить дерево исходников ядра к
сборке внешних модулей.

ПРИМЕЧАНИЕ: «modules_prepare» не будет собирать Module.symvers, даже если
задан CONFIG_MODVERSIONS; следовательно, для работы версионирования модулей нужно
выполнить полную сборку ядра.

Синтаксис команды
-----------------

	Команда для сборки внешнего модуля::

		$ make -C <path_to_kernel_dir> M=$PWD

	Система kbuild понимает, что собирается внешний модуль,
	благодаря опции «M=<dir>», переданной в команде.

	Чтобы собрать модуль для запущенного ядра, используйте::

		$ make -C /lib/modules/`uname -r`/build M=$PWD

	Затем, чтобы установить только что собранный модуль (модули), добавьте к команде
	цель «modules_install»::

		$ make -C /lib/modules/`uname -r`/build M=$PWD modules_install

	Начиная с Linux 6.13, вместо опции -C можно использовать опцию -f. Это
	позволит избежать ненужной смены рабочего каталога. Внешний
	модуль будет выведен в каталог, из которого вы вызываете make.

		$ make -f /lib/modules/`uname -r`/build/Makefile M=$PWD

Опции
-----

	($KDIR указывает на путь к каталогу исходников ядра или на путь
	к каталогу вывода ядра, если ядро собиралось в отдельном
	каталоге сборки.)

	При желании вы можете передать опцию MO=, если хотите собрать модули в
	отдельном каталоге.

	make -C $KDIR M=$PWD [MO=$BUILD_DIR]

	-C $KDIR
		Каталог, содержащий ядро и соответствующие артефакты сборки,
		используемые для сборки внешнего модуля.
		При выполнении «make» фактически перейдёт в указанный каталог
		и вернётся обратно по завершении.

	M=$PWD
		Сообщает kbuild, что собирается внешний модуль.
		Значение, переданное в «M», — это абсолютный путь к
		каталогу, где расположен внешний модуль (файл kbuild).

	MO=$BUILD_DIR
		Задаёт отдельный каталог вывода для внешнего модуля.

Цели
----

	При сборке внешнего модуля доступно только подмножество
	целей «make».

	make -C $KDIR M=$PWD [target]

	По умолчанию будет собран модуль (модули), расположенный в текущем
	каталоге, поэтому цель указывать не требуется. Все
	выходные файлы также будут сгенерированы в этом каталоге. Никаких
	попыток обновить исходники ядра не предпринимается, и предварительным
	условием является успешное выполнение «make» для
	ядра.

	modules
		Цель по умолчанию для внешних модулей. Она имеет ту
		же функциональность, как если бы цель не указывалась. См.
		описание выше.

	modules_install
		Установить внешний модуль (модули). Местоположением по умолчанию является
		/lib/modules/<kernel_release>/updates/, но к нему может
		быть добавлен префикс с помощью INSTALL_MOD_PATH (рассматривается в разделе
		`Установка модулей`_).

	clean
		Удалить все сгенерированные файлы только в каталоге модуля.

	help
		Перечислить доступные цели для внешних модулей.

Сборка отдельных файлов
-----------------------

	Можно собирать отдельные файлы, входящие в состав модуля.
	Это одинаково хорошо работает для ядра, для модуля и даже для
	внешних модулей.

	Пример (модуль foo.ko состоит из bar.o и baz.o)::

		make -C $KDIR M=$PWD bar.lst
		make -C $KDIR M=$PWD baz.o
		make -C $KDIR M=$PWD foo.ko
		make -C $KDIR M=$PWD ./


Создание файла Kbuild для внешнего модуля
=========================================

В предыдущем разделе мы рассмотрели команду для сборки модуля для
запущенного ядра. Однако модуль фактически не собирается, потому что
требуется файл сборки. В этом файле будет содержаться имя
собираемого модуля (модулей) вместе со списком необходимых исходных
файлов. Файл может быть таким же простым, как одна строка::

	obj-m := <module_name>.o

Система kbuild соберёт <module_name>.o из <module_name>.c
и после компоновки получит модуль ядра <module_name>.ko.
Приведённую выше строку можно поместить либо в файл «Kbuild», либо в «Makefile».
Когда модуль собирается из нескольких исходников, нужна дополнительная
строка, перечисляющая файлы::

	<module_name>-y := <src1>.o <src2>.o ...

ПРИМЕЧАНИЕ: Дальнейшая документация, описывающая синтаксис, используемый kbuild,
находится в Documentation/kbuild/makefiles.rst.

Приведённые ниже примеры демонстрируют, как создать файл сборки для
модуля 8123.ko, который собирается из следующих файлов::

	8123_if.c
	8123_if.h
	8123_pci.c

Общий Makefile
--------------

	Внешний модуль всегда включает makefile-обёртку, которая
	поддерживает сборку модуля с помощью «make» без аргументов.
	Эта цель не используется kbuild; она нужна лишь для удобства.
	Может быть включена и дополнительная функциональность, например тестовые цели,
	но её следует отфильтровывать от kbuild из-за возможных конфликтов
	имён.

	Пример 1::

		--> filename: Makefile
		ifneq ($(KERNELRELEASE),)
		# kbuild part of makefile
		obj-m  := 8123.o
		8123-y := 8123_if.o 8123_pci.o

		else
		# normal makefile
		KDIR ?= /lib/modules/`uname -r`/build

		default:
			$(MAKE) -C $(KDIR) M=$$PWD

		endif

	Проверка KERNELRELEASE используется для разделения двух частей
	makefile. В примере kbuild увидит только два
	присваивания, тогда как «make» увидит всё, кроме этих
	двух присваиваний. Это связано с двумя проходами по файлу:
	первый проход выполняет экземпляр «make», запущенный из
	командной строки; второй проход выполняет система kbuild,
	которая инициируется параметризованным «make» в цели default.

Раздельные файлы Kbuild и Makefile
----------------------------------

	Kbuild сначала ищет файл с именем «Kbuild», и, если он не
	найден, затем ищет «Makefile». Использование файла «Kbuild»
	позволяет нам разделить «Makefile» из примера 1 на два файла:

	Пример 2::

		--> filename: Kbuild
		obj-m  := 8123.o
		8123-y := 8123_if.o 8123_pci.o

		--> filename: Makefile
		KDIR ?= /lib/modules/`uname -r`/build

		default:
			$(MAKE) -C $(KDIR) M=$$PWD

	Разделение в примере 2 сомнительно из-за простоты
	каждого файла; однако некоторые внешние модули используют makefile,
	состоящие из нескольких сотен строк, и здесь действительно
	выгодно отделить часть kbuild от остального.

	Linux 6.13 и более поздние версии поддерживают другой способ. Makefile внешнего модуля
	может напрямую включать Makefile ядра вместо вызова вложенного Make.

	Пример 3::

		--> filename: Kbuild
		obj-m  := 8123.o
		8123-y := 8123_if.o 8123_pci.o

		--> filename: Makefile
		KDIR ?= /lib/modules/$(shell uname -r)/build
		export KBUILD_EXTMOD := $(realpath $(dir $(lastword $(MAKEFILE_LIST))))
		include $(KDIR)/Makefile


Сборка нескольких модулей
-------------------------

	kbuild поддерживает сборку нескольких модулей из одного файла
	сборки. Например, если бы вы хотели собрать два модуля, foo.ko
	и bar.ko, строки kbuild были бы такими::

		obj-m := foo.o bar.o
		foo-y := <foo_srcs>
		bar-y := <bar_srcs>

	Вот так просто!


Заголовочные файлы
==================

Внутри ядра заголовочные файлы хранятся в стандартных местах
согласно следующему правилу:

	* Если заголовочный файл описывает только внутренний интерфейс
	  модуля, то этот файл помещается в тот же каталог, что и исходные
	  файлы.
	* Если заголовочный файл описывает интерфейс, используемый другими частями
	  ядра, которые расположены в других каталогах, то
	  этот файл помещается в include/linux/.

	  ПРИМЕЧАНИЕ:
	      Из этого правила есть два заметных исключения: более крупные
	      подсистемы имеют свой собственный каталог в include/, например
	      include/scsi; а специфичные для архитектуры заголовки расположены
	      в arch/$(SRCARCH)/include/.

Заголовки ядра
--------------

	Чтобы включить заголовочный файл, расположенный в include/linux/, просто
	используйте::

		#include <linux/module.h>

	kbuild добавит компилятору опции, чтобы поиск выполнялся в соответствующих
	каталогах.

Один подкаталог
---------------

	Внешние модули обычно размещают заголовочные файлы в отдельном
	каталоге include/, где находятся их исходники, хотя это
	и не обычный стиль ядра. Чтобы сообщить kbuild об этом
	каталоге, используйте либо ccflags-y, либо CFLAGS_<filename>.o.

	Используя пример из раздела 3, если бы мы переместили 8123_if.h в
	подкаталог с именем include, итоговый файл kbuild
	выглядел бы так::

		--> filename: Kbuild
		obj-m := 8123.o

		ccflags-y := -I $(src)/include
		8123-y := 8123_if.o 8123_pci.o

Несколько подкаталогов
----------------------

	kbuild может работать с файлами, разбросанными по нескольким каталогам.
	Рассмотрим следующий пример::

		.
		|__ src
		|   |__ complex_main.c
		|   |__ hal
		|	|__ hardwareif.c
		|	|__ include
		|	    |__ hardwareif.h
		|__ include
			|__ complex.h

	Чтобы собрать модуль complex.ko, нам тогда нужен следующий
	файл kbuild::

		--> filename: Kbuild
		obj-m := complex.o
		complex-y := src/complex_main.o
		complex-y += src/hal/hardwareif.o

		ccflags-y := -I$(src)/include
		ccflags-y += -I$(src)/src/hal/include

	Как видите, kbuild умеет работать с объектными файлами, расположенными
	в других каталогах. Хитрость в том, чтобы указать каталог
	относительно местоположения файла kbuild. Тем не менее, это
	НЕ рекомендуемая практика.

	Для заголовочных файлов kbuild необходимо явно указать, где
	искать. Когда kbuild выполняется, текущим каталогом всегда является
	корень дерева ядра (аргумент опции «-C»), и поэтому требуется
	абсолютный путь. $(src) предоставляет абсолютный путь, указывая
	на каталог, где расположен выполняющийся в данный момент файл kbuild.


Установка модулей
=================

Модули, включённые в состав ядра, устанавливаются в
каталог:

	/lib/modules/$(KERNELRELEASE)/kernel/

А внешние модули устанавливаются в:

	/lib/modules/$(KERNELRELEASE)/updates/

INSTALL_MOD_PATH
----------------

	Выше приведены каталоги по умолчанию, но, как всегда, возможен некоторый уровень
	настройки. К пути установки можно добавить
	префикс с помощью переменной INSTALL_MOD_PATH::

		$ make INSTALL_MOD_PATH=/frodo modules_install
		=> Install dir: /frodo/lib/modules/$(KERNELRELEASE)/kernel/

	INSTALL_MOD_PATH может быть задана как обычная переменная окружения или,
	как показано выше, может быть указана в командной строке при
	вызове «make». Это действует при установке как модулей внутри дерева,
	так и модулей вне дерева.

INSTALL_MOD_DIR
---------------

	Внешние модули по умолчанию устанавливаются в каталог в
	/lib/modules/$(KERNELRELEASE)/updates/, но вы можете захотеть
	разместить модули, относящиеся к определённой функциональности, в отдельном
	каталоге. Для этого используйте INSTALL_MOD_DIR, чтобы указать
	альтернативное имя вместо «updates»::

		$ make INSTALL_MOD_DIR=gandalf -C $KDIR \
		       M=$PWD modules_install
		=> Install dir: /lib/modules/$(KERNELRELEASE)/gandalf/


Версионирование модулей
=======================

Версионирование модулей включается тегом CONFIG_MODVERSIONS и используется
как простая проверка согласованности ABI. Создаётся значение CRC полного прототипа
экспортируемого символа. Когда модуль загружается/используется, значения
CRC, содержащиеся в ядре, сравниваются с аналогичными значениями в
модуле; если они не равны, ядро отказывается загружать
модуль.

Module.symvers содержит список всех экспортируемых символов из сборки ядра.

Символы из ядра (vmlinux + модули)
----------------------------------

	Во время сборки ядра будет сгенерирован файл с именем Module.symvers.
	Module.symvers содержит все экспортируемые символы из
	ядра и скомпилированных модулей. Для каждого символа также
	сохраняется соответствующее значение CRC.

	Синтаксис файла Module.symvers::

		<CRC>       <Symbol>         <Module>                         <Export Type>     <Namespace>

		0xe1cc2a05  usb_stor_suspend drivers/usb/storage/usb-storage  EXPORT_SYMBOL_GPL USB_STORAGE

	Поля разделяются символами табуляции, и значения могут быть пустыми (например,
	если для экспортируемого символа не определено пространство имён).

	Для сборки ядра без включённого CONFIG_MODVERSIONS значение CRC
	будет равно 0x00000000.

	Module.symvers служит двум целям:

	1) Он перечисляет все экспортируемые символы из vmlinux и всех модулей.
	2) Он перечисляет CRC, если включён CONFIG_MODVERSIONS.

Форматы информации о версиях
----------------------------

	Экспортируемые символы хранят информацию в секциях __ksymtab или __ksymtab_gpl.
	Имена символов и пространства имён хранятся в __ksymtab_strings
	в формате, похожем на таблицу строк, используемую для ELF. Если
	включён CONFIG_MODVERSIONS, значения CRC, соответствующие экспортируемым
	символам, будут добавлены в __kcrctab или __kcrctab_gpl.

	Если включён CONFIG_BASIC_MODVERSIONS (по умолчанию вместе с
	CONFIG_MODVERSIONS), импортируемые символы будут иметь имя своего символа и
	CRC, сохранённые в секции __versions импортирующего модуля. Этот
	режим поддерживает только символы длиной до 64 байт.

	Если включён CONFIG_EXTENDED_MODVERSIONS (требуется для одновременного включения
	CONFIG_MODVERSIONS и CONFIG_RUST), импортируемые символы будут
	иметь имя своего символа, записанное в секции __version_ext_names
	в виде последовательности конкатенированных, завершающихся нулём строк. Значения CRC для
	этих символов будут записаны в секции __version_ext_crcs.

Символы и внешние модули
------------------------

	При сборке внешнего модуля системе сборки нужен доступ
	к символам из ядра, чтобы проверить, что все внешние символы
	определены. Это делается на шаге MODPOST. modpost получает
	символы, читая Module.symvers из дерева исходников
	ядра. Во время шага MODPOST будет записан новый файл Module.symvers,
	содержащий все экспортируемые символы из этого внешнего модуля.

Символы из другого внешнего модуля
----------------------------------

	Иногда внешний модуль использует экспортируемые символы из
	другого внешнего модуля. Kbuild нужно полное знание
	всех символов, чтобы не выдавать предупреждения о неопределённых
	символах. Для этой ситуации существуют два решения.

	ПРИМЕЧАНИЕ: Метод с файлом kbuild верхнего уровня рекомендуется,
	но может быть непрактичным в определённых ситуациях.

	Использовать файл kbuild верхнего уровня
		Если у вас есть два модуля, foo.ko и bar.ko, где
		foo.ko нужны символы из bar.ko, вы можете использовать
		общий файл kbuild верхнего уровня, чтобы оба модуля
		компилировались в одной сборке. Рассмотрим следующую
		структуру каталогов::

			./foo/ <= contains foo.ko
			./bar/ <= contains bar.ko

		Файл kbuild верхнего уровня тогда выглядел бы так::

			#./Kbuild (or ./Makefile):
				obj-m := foo/ bar/

		И выполнение::

			$ make -C $KDIR M=$PWD

		сделает тогда ожидаемое и скомпилирует оба модуля с
		полным знанием символов из любого из модулей.

	Использовать переменную «make» KBUILD_EXTRA_SYMBOLS
		Если добавить файл kbuild верхнего уровня непрактично,
		вы можете присвоить переменной KBUILD_EXTRA_SYMBOLS в вашем
		файле сборки разделённый пробелами список файлов.
		Эти файлы будут загружены modpost во время
		инициализации его таблиц символов.


Советы и приёмы
===============

Проверка CONFIG_FOO_BAR
-----------------------

	Модулям часто нужно проверять определённые опции `CONFIG_`, чтобы
	решить, включена ли в модуль конкретная функциональность. В
	kbuild это делается прямой ссылкой на переменную `CONFIG_`::

		#fs/ext2/Makefile
		obj-$(CONFIG_EXT2_FS) += ext2.o

		ext2-y := balloc.o bitmap.o dir.o
		ext2-$(CONFIG_EXT2_FS_XATTR) += xattr.o
