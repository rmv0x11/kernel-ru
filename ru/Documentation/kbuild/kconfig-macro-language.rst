=========================
Макроязык Kconfig
=========================

Концепция
---------

Базовая идея была навеяна Make. Если присмотреться к Make, можно заметить как бы
два языка в одном. Один язык описывает графы зависимостей, состоящие из целей
(targets) и предпосылок (prerequisites). Другой — это макроязык для выполнения
текстовой подстановки.

Между двумя языковыми стадиями есть чёткое разграничение. Например, можно
написать makefile следующим образом::

    APP := foo
    SRC := foo.c
    CC := gcc

    $(APP): $(SRC)
            $(CC) -o $(APP) $(SRC)

Макроязык заменяет ссылки на переменные их раскрытой формой и обрабатывает всё
так, как если бы исходный файл был подан на вход следующим образом::

    foo: foo.c
            gcc -o foo foo.c

Затем Make анализирует граф зависимостей и определяет цели, которые нужно
обновить.

Идея весьма схожа в Kconfig — можно описать файл Kconfig вот так::

    CC := gcc

    config CC_HAS_FOO
            def_bool $(shell, $(srctree)/scripts/gcc-check-foo.sh $(CC))

Макроязык в Kconfig обрабатывает исходный файл в следующее промежуточное
представление::

    config CC_HAS_FOO
            def_bool y

Затем Kconfig переходит к стадии вычисления, чтобы разрешить межсимвольные
зависимости, как описано в kconfig-language.rst.


Переменные
----------

Как и в Make, переменная в Kconfig работает как макропеременная. Макропеременная
раскрывается «на месте», давая текстовую строку, которая затем может быть
раскрыта далее. Чтобы получить значение переменной, заключите её имя в $( ).
Круглые скобки обязательны даже для однобуквенных имён переменных; $X является
синтаксической ошибкой. Форма с фигурными скобками, как ${CC}, также не
поддерживается.

Существует два типа переменных: просто раскрываемые переменные (simply expanded
variables) и рекурсивно раскрываемые переменные (recursively expanded
variables).

Просто раскрываемая переменная определяется с помощью оператора присваивания :=.
Её правая часть раскрывается немедленно при чтении строки из файла Kconfig.

Рекурсивно раскрываемая переменная определяется с помощью оператора присваивания
=. Её правая часть просто сохраняется как значение переменной без какого-либо
раскрытия. Вместо этого раскрытие выполняется в момент использования переменной.

Есть ещё один тип оператора присваивания; += используется для добавления текста
к переменной. Правая часть += раскрывается немедленно, если левая часть
изначально была определена как простая переменная. В противном случае её
вычисление откладывается.

Ссылка на переменную может принимать параметры в следующей форме::

  $(name,arg1,arg2,arg3)

Параметризованную ссылку можно рассматривать как функцию (точнее, «определённую
пользователем функцию» в противоположность «встроенной функции», перечисленной
ниже).

Полезные функции должны раскрываться в момент использования, поскольку одна и та
же функция раскрывается по-разному, если ей передаются разные параметры. Поэтому
определённая пользователем функция определяется с помощью оператора присваивания
=. На параметры ссылаются внутри тела определения через $(1), $(2) и т.д.

На самом деле рекурсивно раскрываемые переменные и определённые пользователем
функции внутренне устроены одинаково. (Иными словами, «переменная» — это
«функция с нулём аргументов».) Когда мы говорим «переменная» в широком смысле,
это понятие включает и «определённую пользователем функцию».


Встроенные функции
-------------------

Как и Make, Kconfig предоставляет несколько встроенных функций. Каждая функция
принимает определённое число аргументов.

В Make каждая встроенная функция принимает хотя бы один аргумент. Kconfig
допускает ноль аргументов для встроенных функций, таких как $(filename),
$(lineno). Их можно было бы считать «встроенными переменными», но в конечном
счёте это лишь вопрос того, как мы их называем. Будем здесь говорить «встроенная
функция», подразумевая нативно поддерживаемую функциональность.

В настоящее время Kconfig поддерживает следующие встроенные функции.

 - $(shell,command)

  Функция «shell» принимает единственный аргумент, который раскрывается и
  передаётся подоболочке (subshell) на исполнение. Затем читается стандартный
  вывод команды и возвращается как значение функции. Каждый перевод строки в
  выводе заменяется пробелом. Любые завершающие переводы строк удаляются.
  Стандартный поток ошибок не возвращается, как и любой код завершения
  программы.

 - $(info,text)

  Функция «info» принимает единственный аргумент и печатает его в stdout.
  Она вычисляется в пустую строку.

 - $(warning-if,condition,text)

  Функция «warning-if» принимает два аргумента. Если часть condition равна «y»,
  часть text отправляется в stderr. Текст предваряется именем текущего файла
  Kconfig и номером текущей строки.

 - $(error-if,condition,text)

  Функция «error-if» аналогична «warning-if», но она немедленно прекращает
  разбор, если часть condition равна «y».

 - $(filename)

  Функция 'filename' не принимает аргументов, и $(filename) раскрывается в имя
  разбираемого в данный момент файла.

 - $(lineno)

  Функция 'lineno' не принимает аргументов, и $(lineno) раскрывается в номер
  разбираемой в данный момент строки.


Make и Kconfig
--------------

Kconfig перенимает Make-подобный макроязык, но синтаксис вызова функций немного
отличается.

Вызов функции в Make выглядит так::

  $(func-name arg1,arg2,arg3)

Имя функции и первый аргумент разделяются как минимум одним пробельным символом.
Затем из первого аргумента обрезаются ведущие пробелы, тогда как пробелы в
остальных аргументах сохраняются. Чтобы начать первый параметр с пробелов,
нужно прибегнуть к своего рода уловке. Например, если вы хотите, чтобы функция
«info» напечатала "  hello", вы можете написать следующим образом::

  empty :=
  space := $(empty) $(empty)
  $(info $(space)$(space)hello)

Kconfig использует в качестве разделителей только запятые и сохраняет все
пробелы в вызове функции. Некоторые предпочитают ставить пробел после каждого
разделителя-запятой::

  $(func-name, arg1, arg2, arg3)

В этом случае «func-name» получит " arg1", " arg2", " arg3". Наличие ведущих
пробелов может иметь значение в зависимости от функции. То же относится и к
Make — например, $(subst .c, .o, $(sources)) является типичной ошибкой; он
заменяет ".c" на " .o".

В Make определённая пользователем функция вызывается с помощью встроенной
функции 'call', вот так::

    $(call my-func,arg1,arg2,arg3)

Kconfig вызывает определённые пользователем функции и встроенные функции
одинаково. Опускание 'call' делает синтаксис короче.

В Make некоторые функции трактуют запятые буквально, а не как разделители
аргументов. Например, $(shell echo hello, world) запускает команду
"echo hello, world". Аналогично, $(info hello, world) печатает "hello, world" в
stdout. Можно сказать, что это *полезная* несогласованность.

В Kconfig ради более простой реализации и грамматической согласованности
запятые, появляющиеся в контексте $( ), всегда являются разделителями. Это
означает, что::

  $(shell, echo hello, world)

является ошибкой, поскольку передаёт два параметра туда, где функция 'shell'
принимает только один. Чтобы передать запятые в аргументах, можно
воспользоваться следующей уловкой::

  comma := ,
  $(shell, echo hello$(comma) world)


Предостережения
---------------

Переменная (или функция) не может быть раскрыта через границы токенов. Поэтому
нельзя использовать переменную как сокращение для выражения, состоящего из
нескольких токенов. Следующее работает::

    RANGE_MIN := 1
    RANGE_MAX := 3

    config FOO
            int "foo"
            range $(RANGE_MIN) $(RANGE_MAX)

Но следующее не работает::

    RANGES := 1 3

    config FOO
            int "foo"
            range $(RANGES)

Переменная не может быть раскрыта в какое-либо ключевое слово Kconfig. Следующее
не работает::

    MY_TYPE := tristate

    config FOO
            $(MY_TYPE) "foo"
            default y

Очевидно из устройства, что $(shell command) раскрывается на фазе текстовой
подстановки. Передавать символы в функцию 'shell' нельзя.

Следующее работает не так, как ожидается::

    config ENDIAN_FLAG
            string
            default "-mbig-endian" if CPU_BIG_ENDIAN
            default "-mlittle-endian" if CPU_LITTLE_ENDIAN

    config CC_HAS_ENDIAN_FLAG
            def_bool $(shell $(srctree)/scripts/gcc-check-flag ENDIAN_FLAG)

Вместо этого можно поступить следующим образом, чтобы любой вызов функции был
раскрыт статически::

    config CC_HAS_ENDIAN_FLAG
            bool
            default $(shell $(srctree)/scripts/gcc-check-flag -mbig-endian) if CPU_BIG_ENDIAN
            default $(shell $(srctree)/scripts/gcc-check-flag -mlittle-endian) if CPU_LITTLE_ENDIAN
