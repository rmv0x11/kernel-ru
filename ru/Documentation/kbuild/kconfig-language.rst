================
Язык Kconfig
================

Введение
--------

База данных конфигурации — это набор параметров конфигурации,
организованных в древовидную структуру::

	+- Code maturity level options
	|  +- Prompt for development and/or incomplete code/drivers
	+- General setup
	|  +- Networking support
	|  +- System V IPC
	|  +- BSD Process Accounting
	|  +- Sysctl support
	+- Loadable module support
	|  +- Enable loadable module support
	|     +- Set version information on all module symbols
	|     +- Kernel module loader
	+- ...

У каждой записи есть свои собственные зависимости. Эти зависимости
используются для определения видимости записи. Любая дочерняя запись
видима только тогда, когда видима её родительская запись.

Записи меню
-----------

Большинство записей определяют параметр конфигурации; все остальные записи
помогают их организовать. Отдельный параметр конфигурации определяется так::

  config MODVERSIONS
	bool "Set version information on all module symbols"
	depends on MODULES
	help
	  Usually, modules have to be recompiled whenever you switch to a new
	  kernel.  ...

Каждая строка начинается с ключевого слова и может сопровождаться несколькими
аргументами.  "config" начинает новую запись config. Следующие строки
определяют атрибуты для этого параметра конфигурации. Атрибутами могут быть тип
параметра конфигурации, приглашение ввода, зависимости, текст справки и
значения по умолчанию. Параметр конфигурации может быть определён несколько раз
с одним и тем же именем, но каждое определение может иметь только одно
приглашение ввода, а тип не должен конфликтовать.

Атрибуты меню
-------------

Запись меню может иметь ряд атрибутов. Не все из них применимы везде
(см. синтаксис).

- определение типа: "bool"/"tristate"/"string"/"hex"/"int"

  Каждый параметр конфигурации должен иметь тип. Существует всего два базовых
  типа: tristate и string; остальные типы основаны на этих двух. Определение
  типа опционально принимает приглашение ввода, так что эти два примера
  эквивалентны::

	bool "Networking support"

  и::

	bool
	prompt "Networking support"

- приглашение ввода: "prompt" <prompt> ["if" <expr>]

  Каждая запись меню может иметь не более одного приглашения, которое
  используется для отображения пользователю. Опционально с помощью "if" можно
  добавить зависимости только для этого приглашения. Если приглашение
  отсутствует, параметр конфигурации является невидимым символом, то есть его
  значение не может быть изменено пользователем напрямую (например, путём
  изменения значения в ``.config``), и параметр не будет отображаться ни в одном
  из меню конфигурации. Его значение можно задать только через "default" и
  "select" (см. ниже).

- значение по умолчанию: "default" <expr> ["if" <expr>]

  Параметр конфигурации может иметь любое количество значений по умолчанию. Если
  видимо несколько значений по умолчанию, активным является только первое
  определённое. Значения по умолчанию не ограничены той записью меню, где они
  определены. Это означает, что значение по умолчанию может быть определено
  где-то ещё или переопределено более ранним определением.
  Значение по умолчанию присваивается символу конфигурации только в том случае,
  если пользователь не задал другое значение (через приглашение ввода выше).
  Если приглашение ввода видимо, значение по умолчанию представляется
  пользователю и может быть им переопределено.
  Опционально с помощью "if" можно добавить зависимости только для этого значения
  по умолчанию.

 Значение по умолчанию намеренно по умолчанию равно 'n', чтобы избежать
 разрастания сборки. За редкими исключениями, новые параметры конфигурации не
 должны это изменять. Замысел в том, чтобы "make oldconfig" добавлял в
 конфигурацию как можно меньше от релиза к релизу.

 Примечание:
	К тому, что заслуживает "default y/m", относятся:

	a) Новый параметр Kconfig для чего-то, что раньше всегда собиралось,
	   должен иметь "default y".

	b) Новый сторожевой (gatekeeping) параметр Kconfig, который
	   скрывает/показывает другие параметры Kconfig (но не генерирует
	   собственного кода), должен иметь "default y", чтобы люди видели те
	   другие параметры.

	c) Поведение поддрайвера или аналогичные параметры для драйвера, который
	   имеет "default n". Это позволяет вам предоставить разумные значения по
	   умолчанию.

	d) Оборудование или инфраструктура, которые ожидают все, например CONFIG_NET
	   или CONFIG_BLOCK. Это редкие исключения.

- определение типа + значение по умолчанию::

	"def_bool"/"def_tristate" <expr> ["if" <expr>]

  Это сокращённая запись для определения типа плюс значение.
  Опционально с помощью "if" можно добавить зависимости для этого значения по
  умолчанию.

- зависимости: "depends on" <expr> ["if" <expr>]

  Это определяет зависимость для данной записи меню. Если определено несколько
  зависимостей, они объединяются с помощью '&&'. Зависимости применяются ко всем
  остальным параметрам в пределах этой записи меню (которые также принимают
  выражение "if"), так что эти два примера эквивалентны::

	bool "foo" if BAR
	default y if BAR

  и::

	depends on BAR
	bool "foo"
	default y

  Само определение зависимости может быть условным, если добавить "if", за
  которым следует выражение. Например::

    config FOO
	tristate
	depends on BAR if BAZ

  что означает, что FOO ограничивается значением BAR только в том случае, если
  также установлен BAZ.

- обратные зависимости: "select" <symbol> ["if" <expr>]

  В то время как обычные зависимости снижают верхнюю границу символа (см. ниже),
  обратные зависимости можно использовать для принудительного задания нижней
  границы другого символа. Значение текущего символа меню используется как
  минимальное значение, до которого может быть установлен <symbol>. Если
  <symbol> выбирается несколько раз, граница устанавливается равной наибольшему
  выбору.
  Обратные зависимости могут использоваться только с символами boolean или
  tristate.

  Примечание:
	select следует использовать с осторожностью. select принудительно
	установит символ в значение без посещения его зависимостей.
	Злоупотребляя select, вы можете выбрать символ FOO, даже
	если FOO зависит от BAR, который не установлен.
	В общем случае используйте select только для невидимых символов
	(без приглашений нигде) и для символов без зависимостей.
	Это ограничит полезность, но, с другой стороны, позволит избежать
	недопустимых конфигураций повсюду.

	Если за "select" <symbol> следует "if" <expr>, <symbol> будет
	выбран как логическое И значения текущего символа меню
	и <expr>. Это означает, что нижняя граница может быть понижена из-за
	наличия "if" <expr>. Такое поведение может показаться странным, но мы на
	него полагаемся. (Будущее этого поведения не определено.)

- слабые обратные зависимости: "imply" <symbol> ["if" <expr>]

  Это похоже на "select", поскольку оно обеспечивает нижнюю границу для другого
  символа, за исключением того, что значение «подразумеваемого» (implied)
  символа всё ещё может быть установлено в n из-за прямой зависимости или с
  помощью видимого приглашения.

  Дан следующий пример::

    config FOO
	tristate "foo"
	imply BAZ

    config BAZ
	tristate "baz"
	depends on BAR

  Возможны следующие значения:

	===		===		=============	==============
	FOO		BAR		BAZ's default	choice for BAZ
	===		===		=============	==============
	n		y		n		N/m/y
	m		y		m		M/y/n
	y		y		y		Y/m/n
	n		m		n		N/m
	m		m		m		M/n
	y		m		m		M/n
	y		n		*		N
	===		===		=============	==============

  Это полезно, например, при наличии нескольких драйверов, которые хотят указать
  свою способность подключаться к вторичной подсистеме, позволяя при этом
  пользователю исключить эту подсистему из конфигурации без необходимости также
  отключать эти драйверы.

  Примечание: Если функциональность, предоставляемая BAZ, крайне желательна для
  FOO, FOO должен подразумевать (imply) не только BAZ, но и его зависимость BAR::

    config FOO
	tristate "foo"
	imply BAR
	imply BAZ

  Примечание: Если за "imply" <symbol> следует "if" <expr>, значением по
  умолчанию <symbol> будет логическое И значения текущего символа меню и <expr>.
  (Будущее этого поведения не определено.)

- ограничение отображения меню: "visible if" <expr>

  Этот атрибут применим только к блокам меню; если условие
  ложно, блок меню не отображается пользователю (символы,
  содержащиеся в нём, тем не менее всё ещё могут быть выбраны другими символами).
  Это аналогично условному атрибуту "prompt" для отдельных записей
  меню. Значение "visible" по умолчанию равно true.

- числовые диапазоны: "range" <symbol> <symbol> ["if" <expr>]

  Это позволяет ограничить диапазон возможных вводимых значений для символов int
  и hex. Пользователь может ввести только значение, которое больше
  или равно первому символу и меньше или равно второму
  символу.

- текст справки: "help"

  Это определяет текст справки. Конец текста справки определяется
  уровнем отступа, то есть он заканчивается на первой строке, которая имеет
  меньший отступ, чем первая строка текста справки.

- атрибут модуля: "modules"
  Это объявляет символ предназначенным для использования в качестве символа
  MODULES, который включает третье, модульное, состояние для всех символов
  конфигурации.
  Не более чем у одного символа может быть установлен параметр "modules".

- переходный атрибут: "transitional"
  Это объявляет символ переходным, то есть он должен обрабатываться
  во время конфигурации, но опускаться во вновь записываемых файлах .config.
  Переходные символы полезны для обратной совместимости при миграциях параметров
  конфигурации — они позволяют olddefconfig обрабатывать существующие файлы
  .config, гарантируя при этом, что старый параметр не появится в новых
  конфигурациях.

  Переходный символ:
  - Не имеет приглашения (не виден пользователям в меню)
  - Обрабатывается обычным образом во время конфигурации (значения читаются и используются)
  - Может быть указан в выражениях default других символов
  - Не записывается в новые файлы .config
  - Не может иметь никаких других свойств (это сквозной (pass-through) параметр)

  Пример миграции из OLD_NAME в NEW_NAME::

    config NEW_NAME
	bool "New option name"
	default OLD_NAME
	help
	  This replaces the old CONFIG_OLD_NAME option.

    config OLD_NAME
	bool
	transitional
	help
	  Transitional config for OLD_NAME to NEW_NAME migration.

  При такой настройке существующие файлы .config с "CONFIG_OLD_NAME=y" приведут
  к установке "CONFIG_NEW_NAME=y", в то время как CONFIG_OLD_NAME будет
  опущен во вновь записываемых файлах .config.

Зависимости меню
----------------

Зависимости определяют видимость записи меню и также могут уменьшать
диапазон ввода для символов tristate. Логика tristate, используемая в
выражениях, использует на одно состояние больше, чем обычная булева логика, для
выражения состояния модуля. Выражения зависимостей имеют следующий синтаксис::

  <expr> ::= <symbol>                           (1)
           <symbol> '=' <symbol>                (2)
           <symbol> '!=' <symbol>               (3)
           <symbol1> '<' <symbol2>              (4)
           <symbol1> '>' <symbol2>              (4)
           <symbol1> '<=' <symbol2>             (4)
           <symbol1> '>=' <symbol2>             (4)
           '(' <expr> ')'                       (5)
           '!' <expr>                           (6)
           <expr> '&&' <expr>                   (7)
           <expr> '||' <expr>                   (8)

Выражения перечислены в порядке убывания приоритета.

(1) Преобразует символ в выражение. Символы boolean и tristate
    просто преобразуются в соответствующие значения выражений. Все
    остальные типы символов дают 'n'.
(2) Если значения обоих символов равны, возвращает 'y',
    в противном случае 'n'.
(3) Если значения обоих символов равны, возвращает 'n',
    в противном случае 'y'.
(4) Если значение <symbol1> соответственно меньше, больше, меньше-или-равно
    или больше-или-равно значению <symbol2>, возвращает 'y',
    в противном случае 'n'.
(5) Возвращает значение выражения. Используется для переопределения приоритета.
(6) Возвращает результат (2-/expr/).
(7) Возвращает результат min(/expr/, /expr/).
(8) Возвращает результат max(/expr/, /expr/).

Выражение может иметь значение 'n', 'm' или 'y' (или 0, 1, 2
соответственно для вычислений). Запись меню становится видимой, когда её
выражение вычисляется в 'm' или 'y'.

Существует два типа символов: константные и неконстантные символы.
Неконстантные символы — наиболее распространённые, они определяются с помощью
оператора 'config'. Неконстантные символы состоят целиком из буквенно-цифровых
символов или знаков подчёркивания.
Константные символы являются только частью выражений. Константные символы
всегда заключены в одинарные или двойные кавычки. Внутри кавычек разрешён любой
другой символ, а сами кавычки можно экранировать с помощью '\'.

Структура меню
--------------

Положение записи меню в дереве определяется двумя способами. Во-первых,
оно может быть задано явно::

  menu "Network device support"
	depends on NET

  config NETDEVICES
	...

  endmenu

Все записи внутри блока "menu" ... "endmenu" становятся подменю
"Network device support". Все вложенные записи наследуют зависимости от
записи меню; например, это означает, что зависимость "NET" добавляется к списку
зависимостей параметра конфигурации NETDEVICES.

Другой способ генерации структуры меню осуществляется путём анализа
зависимостей. Если запись меню каким-либо образом зависит от предыдущей записи,
её можно сделать подменю этой записи. Во-первых, предыдущий (родительский)
символ должен быть частью списка зависимостей, а затем должно быть истинным одно
из этих двух условий:

- дочерняя запись должна становиться невидимой, если родитель установлен в 'n'
- дочерняя запись должна быть видимой только тогда, когда видим родитель::

    config MODULES
	bool "Enable loadable module support"

    config MODVERSIONS
	bool "Set version information on all module symbols"
	depends on MODULES

    comment "module support disabled"
	depends on !MODULES

MODVERSIONS напрямую зависит от MODULES, это означает, что он видим только если
MODULES отличается от 'n'. Комментарий, с другой стороны, видим только
тогда, когда MODULES установлен в 'n'.


Синтаксис Kconfig
-----------------

Конфигурационный файл описывает последовательность записей меню, где каждая
строка начинается с ключевого слова (кроме текстов справки). Следующие ключевые
слова завершают запись меню:

- config
- menuconfig
- choice/endchoice
- comment
- menu/endmenu
- if/endif
- source

Первые пять также начинают определение записи меню.

config::

	"config" <symbol>
	<config options>

Это определяет символ конфигурации <symbol> и принимает любой из перечисленных
выше атрибутов в качестве параметров.

menuconfig::

	"menuconfig" <symbol>
	<config options>

Это похоже на простую запись config выше, но дополнительно даёт
подсказку фронтендам, что все подпараметры должны отображаться как
отдельный список параметров. Чтобы все подпараметры действительно
отображались под записью menuconfig, а не вне её, каждый элемент
из списка <config options> должен зависеть от символа menuconfig.
На практике это достигается использованием одной из двух следующих конструкций::

  (1):
  menuconfig M
  if M
      config C1
      config C2
  endif

  (2):
  menuconfig M
  config C1
      depends on M
  config C2
      depends on M

В следующих примерах (3) и (4) C1 и C2 по-прежнему имеют зависимость от M,
но больше не будут появляться под menuconfig M из-за
C0, который не зависит от M::

  (3):
  menuconfig M
      config C0
  if M
      config C1
      config C2
  endif

  (4):
  menuconfig M
  config C0
  config C1
      depends on M
  config C2
      depends on M

choices::

	"choice"
	<choice options>
	<choice block>
	"endchoice"

Это определяет группу выбора (choice) и принимает в качестве параметров атрибуты
"prompt", "default", "depends on" и "help".

Choice позволяет выбрать только одну запись config.

comment::

	"comment" <prompt>
	<comment options>

Это определяет комментарий, который отображается пользователю в процессе
конфигурации, а также выводится в выходные файлы. Единственными
возможными параметрами являются зависимости.

menu::

	"menu" <prompt>
	<menu options>
	<menu block>
	"endmenu"

Это определяет блок меню, см. "Menu structure" выше для получения
дополнительной информации. Единственными возможными параметрами являются
зависимости и атрибуты "visible".

if::

	"if" <expr>
	<if block>
	"endif"

Это определяет блок if. Выражение зависимости <expr> добавляется
ко всем заключённым в него записям меню.

source::

	"source" <prompt>

Это читает указанный конфигурационный файл. Этот файл всегда разбирается.

mainmenu::

	"mainmenu" <prompt>

Это задаёт строку заголовка конфигурационной программы, если конфигурационная
программа решит её использовать. Этот оператор следует размещать в начале
конфигурации, перед любым другим оператором.

Комментарий '#' в исходном файле Kconfig:

Символ '#' без кавычек в любом месте строки исходного файла обозначает
начало комментария исходного файла.  Остаток этой строки
является комментарием.


Советы по Kconfig
-----------------
Это набор советов по Kconfig, большинство из которых не очевидны с
первого взгляда и большинство из которых стали идиомами в ряде файлов
Kconfig.

Добавление общих функциональностей и настройка их использования
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Это распространённая идиома — реализовать функцию/функциональность, которая
актуальна для некоторых архитектур, но не для всех.
Рекомендуемый способ сделать это — использовать конфигурационную переменную с
именем HAVE_*, которая определена в общем файле Kconfig и выбирается
соответствующими архитектурами.
Примером является обобщённая функциональность IOMAP.

В lib/Kconfig мы бы увидели::

  # Generic IOMAP is used to ...
  config HAVE_GENERIC_IOMAP

  config GENERIC_IOMAP
	depends on HAVE_GENERIC_IOMAP && FOO

А в lib/Makefile мы бы увидели::

	obj-$(CONFIG_GENERIC_IOMAP) += iomap.o

Для каждой архитектуры, использующей обобщённую функциональность IOMAP, мы бы увидели::

  config X86
	select ...
	select HAVE_GENERIC_IOMAP
	select ...

Примечание: мы используем существующий параметр конфигурации и избегаем создания
нового конфигурационного параметра для выбора HAVE_GENERIC_IOMAP.

Примечание: использование внутренней конфигурационной переменной
HAVE_GENERIC_IOMAP введено для преодоления ограничения select, который
принудительно устанавливает параметр конфигурации в 'y' независимо от
зависимостей.
Зависимости перенесены на символ GENERIC_IOMAP, и мы избегаем
ситуации, когда select принудительно делает символ равным 'y'.

Добавление функциональностей, требующих поддержки компилятора
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Существует несколько функциональностей, которым нужна поддержка компилятора.
Рекомендуемый способ описать зависимость от возможности компилятора — это
использовать "depends on", за которым следует тестовый макрос::

  config STACKPROTECTOR
	bool "Stack Protector buffer overflow detection"
	depends on $(cc-option,-fstack-protector)
	...

Если вам нужно открыть возможность компилятора для makefile и/или исходных файлов
на C, рекомендуемым префиксом для параметра конфигурации является `CC_HAS_`::

  config CC_HAS_FOO
	def_bool $(success,$(srctree)/scripts/cc-check-foo.sh $(CC))

Сборка только в виде модуля
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Чтобы ограничить сборку компонента только модулем, дополните его символ
конфигурации "depends on m".  Например::

  config FOO
	depends on BAR && m

ограничивает FOO модулем (=m) или отключённым состоянием (=n).

Тестирование компиляцией
~~~~~~~~~~~~~~~~~~~~~~~~~
Если символ конфигурации имеет зависимость, но код, управляемый этим символом
конфигурации, всё равно может быть скомпилирован, когда зависимость не
удовлетворена, рекомендуется увеличить покрытие сборки путём добавления к
зависимости предложения "|| COMPILE_TEST". Это особенно полезно для драйверов
более экзотического оборудования, поскольку
это позволяет системам непрерывной интеграции выполнять тестовую компиляцию кода
на более распространённой системе и таким образом обнаруживать ошибки.
Обратите внимание, что компилируемый для теста код должен избегать аварийного
завершения при запуске на системе, где зависимость не удовлетворена.

Зависимости от архитектуры и платформы
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
Из-за наличия заглушек большинство драйверов теперь может компилироваться на
большинстве архитектур. Однако это не означает, что имеет смысл иметь все
драйверы доступными везде, так как реальное оборудование может существовать
только на определённых архитектурах и платформах. Это особенно верно для
IP-ядер на SoC, которые могут быть ограничены конкретным вендором или семейством
SoC.

Чтобы не спрашивать пользователя о драйверах, которые невозможно использовать на
системе(ах), для которых пользователь компилирует ядро, и если это имеет смысл,
символы конфигурации, управляющие компиляцией драйвера, должны содержать
корректные зависимости, ограничивающие видимость символа (надмножеством)
платформ(ы), на которых может использоваться драйвер. Зависимость может быть
зависимостью от архитектуры (например, ARM) или
платформы (например, ARCH_OMAP4). Это упрощает жизнь не только владельцам
конфигураций дистрибутивов, но и каждому отдельному разработчику или
пользователю, который конфигурирует ядро.

Такую зависимость можно ослабить, скомбинировав её с правилом тестирования
компиляцией выше, что приведёт к:

  config FOO
	bool "Support for foo hardware"
	depends on ARCH_FOO_VENDOR || COMPILE_TEST

Необязательные зависимости
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Некоторые драйверы способны опционально использовать функциональность из другого
модуля или собираться без ошибок при отключённом этом модуле, но вызывают сбой
компоновки при попытке использовать этот загружаемый модуль из встроенного
драйвера.

Рекомендуемый способ выразить эту необязательную зависимость в логике Kconfig
использует условную форму::

  config FOO
	tristate "Support for foo hardware"
	depends on BAR if BAR

Этот слегка контринтуитивный стиль также широко используется::

  config FOO
	tristate "Support for foo hardware"
	depends on BAR || !BAR

Это означает, что либо существует зависимость от BAR, которая запрещает
комбинацию FOO=y с BAR=m, либо BAR полностью отключён. Модуль BAR
должен предоставлять все заглушки для случая !BAR.

Для более формализованного подхода, если есть несколько драйверов, имеющих
одну и ту же зависимость, можно использовать вспомогательный символ, например::

  config FOO
	tristate "Support for foo hardware"
	depends on BAR_OPTIONAL

  config BAR_OPTIONAL
	def_tristate BAR || !BAR

Гораздо менее предпочтительный способ выразить необязательную зависимость — это
IS_REACHABLE() в коде модуля, полезный, например, когда модуль BAR не предоставляет
заглушки для !BAR::

	foo_init()
	{
		if (IS_REACHABLE(CONFIG_BAR))
			bar_register(&foo);
		...
	}

IS_REACHABLE() в целом не рекомендуется, потому что код будет молча
отброшен, когда CONFIG_BAR=m, а этот код встроен. Это не то, что обычно ожидают
пользователи при включении BAR в виде модуля.

Ограничения Kconfig, связанные с рекурсивными зависимостями
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Если вы столкнулись с ошибкой Kconfig: "recursive dependency detected", вы
наткнулись на проблему рекурсивной зависимости в Kconfig; рекурсивную зависимость
можно кратко описать как циклическую зависимость. Инструментам kconfig нужно
обеспечить, чтобы файлы Kconfig соответствовали заданным требованиям конфигурации.
Чтобы это сделать,
kconfig должен определить значения, возможные для всех символов Kconfig; в
настоящее время это невозможно, если между двумя или более символами Kconfig есть
циклическая связь. Подробнее см. подраздел "Simple
Kconfig recursive issue" ниже. Kconfig не выполняет разрешения рекурсивных
зависимостей; это имеет несколько последствий для авторов файлов Kconfig.
Сначала мы объясним, почему эта проблема существует, а затем приведём пример
технического ограничения, которое это накладывает на разработчиков Kconfig.
Энергичным разработчикам, желающим попытаться устранить это ограничение, следует
прочитать следующие подразделы.

Практические решения проблемы рекурсии в kconfig
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

У разработчиков, столкнувшихся с проблемой рекурсии в Kconfig, есть два варианта
в их распоряжении. Мы документируем их ниже, а также предоставляем список
исторических проблем, решённых с помощью этих различных решений.

  a) Удалить любые лишние "select FOO" или "depends on FOO"
  b) Согласовать семантику зависимостей:

	b1) Заменить все "select FOO" на "depends on FOO" или,

	b2) Заменить все "depends on FOO" на "select FOO"

Решение a) можно протестировать с помощью образцового файла Kconfig
Documentation/kbuild/Kconfig.recursion-issue-01 путём удаления
"select CORE" из CORE_BELL_A_ADVANCED, так как это уже неявно подразумевается,
поскольку CORE_BELL_A зависит от CORE. Иногда может быть невозможно удалить
некоторые критерии зависимости, в таких случаях вы можете работать с решением b).

Два разных решения для b) можно протестировать в образцовом файле Kconfig
Documentation/kbuild/Kconfig.recursion-issue-02.

Будущая работа над kconfig
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Работа над kconfig приветствуется в обеих областях — как по прояснению семантики,
так и по оценке использования полноценного SAT-решателя для него. Полноценный
SAT-решатель может быть желателен для обеспечения более сложных отображений
и/или запросов зависимостей,
например, одним из возможных вариантов использования SAT-решателя могло бы быть
устранение текущих известных проблем рекурсивных зависимостей. Неизвестно,
устранит ли это такие проблемы, но такая оценка желательна. Если поддержка
полноценного SAT-решателя окажется слишком сложной или если он не сможет
устранить проблемы рекурсивных зависимостей,
Kconfig должен иметь хотя бы ясную и чётко определённую семантику, которая также
устраняет и документирует ограничения или требования, например такие, которые
связаны с рекурсивными зависимостями.

Дальнейшая работа в обеих этих областях приветствуется в Kconfig. Мы подробнее
разбираем обе из них в следующих двух подразделах.

Семантика Kconfig
~~~~~~~~~~~~~~~~~~

Использование Kconfig широко, Linux теперь лишь один из пользователей Kconfig:
одно исследование выполнило широкий анализ использования Kconfig в 12 проектах [0]_.
Несмотря на его широкое применение и хотя этот документ проделывает приемлемую
работу по документированию базового синтаксиса Kconfig, более точное определение
семантики Kconfig приветствуется. Один проект вывел семантику Kconfig путём
использования конфигуратора xconfig [1]_. Следует проделать работу по
подтверждению того, что выведенная семантика соответствует нашим
предполагаемым целям проектирования Kconfig.
Другой проект формализовал денотационную семантику основного подмножества
языка Kconfig [10]_.

Наличие чётко определённой семантики может быть полезным для инструментов
практической оценки зависимостей, например, одним таким случаем была работа по
выражению в булевой абстракции выведенной семантики Kconfig для
трансляции логики Kconfig в булевы формулы и запуска на них SAT-решателя, чтобы
найти мёртвый код / функциональности (всегда неактивные); с помощью этой
методологии в Linux было найдено 114 мёртвых функциональностей [1]_ (Section 8:
Threats to validity).
Инструмент kismet, основанный на семантике из [10]_, находит злоупотребления
обратными зависимостями и привёл к десяткам зафиксированных исправлений в файлах
Kconfig Linux [11]_.

Подтверждение этого может оказаться полезным, так как Kconfig является одним из
ведущих промышленных языков моделирования вариативности [1]_ [2]_. Его изучение
помогло бы оценить практические применения таких языков; ранее их использование
было только теоретическим, а реальные требования были недостаточно понятны. Тем
не менее, на текущий момент для вывода семантики из языков моделирования
вариативности, таких как Kconfig, использовались только методы обратной
инженерии [3]_.

Полноценный SAT-решатель для Kconfig
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Хотя SAT-решатели [4]_ ещё не использовались Kconfig напрямую, как отмечено
в предыдущем подразделе, тем не менее была проделана работа по выражению в булевой
абстракции выведенной семантики Kconfig для трансляции логики Kconfig в
булевы формулы и запуска на них SAT-решателя [5]_. Другим известным родственным
проектом является CADOS [6]_ (бывший VAMOS [7]_) и инструменты, главным образом
undertaker [8]_, который
был впервые представлен в [9]_.  Основная концепция undertaker состоит в том,
чтобы извлекать модели вариативности из Kconfig и помещать их вместе с
пропозициональной формулой, извлечённой из #ifdef CPP и правил сборки, в
SAT-решатель, чтобы находить мёртвый код, мёртвые файлы и мёртвые символы. Если
использование SAT-решателя в Kconfig желательно, одним из подходов было бы
оценить, как можно перепрофилировать такие наработки для Kconfig. Со стороны
менторов существующих проектов есть достаточный интерес не только в том, чтобы
помочь советом, как интегрировать эту работу в апстрим,
но и в том, чтобы помочь поддерживать её в долгосрочной перспективе.
Заинтересованным разработчикам следует посетить:

https://kernelnewbies.org/KernelProjects/kconfig-sat
