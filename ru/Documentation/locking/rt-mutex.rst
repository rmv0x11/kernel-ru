=======================================
Подсистема RT-mutex с поддержкой PI
=======================================

RT-mutex'ы с наследованием приоритета (priority inheritance) используются для
поддержки PI-futex'ов, которые включают атрибуты наследования приоритета
pthread_mutex_t (PTHREAD_PRIO_INHERIT). [Подробнее о PI-futex'ах см. в
Documentation/locking/pi-futex.rst.]

Эта технология была разработана в дереве -rt и оптимизирована для поддержки
pthread_mutex.

Основные принципы:
------------------

RT-mutex'ы расширяют семантику простых мьютексов (mutex) протоколом
наследования приоритета.

Владелец rt-mutex'а с низким приоритетом наследует приоритет ожидающего с более
высоким приоритетом до тех пор, пока rt-mutex не будет освобождён. Если временно
повышенный в приоритете владелец сам блокируется на rt-mutex'е, он распространяет
повышение приоритета на владельца того другого rt_mutex'а, на котором он
заблокировался. Повышение приоритета немедленно снимается, как только rt_mutex
был разблокирован.

Такой подход позволяет нам сократить время блокировки высокоприоритетных задач на
мьютексах, защищающих разделяемые ресурсы. Наследование приоритета — не панацея для
плохо спроектированных приложений, но оно позволяет хорошо спроектированным
приложениям использовать блокировки в пространстве пользователя (userspace) в
критических частях высокоприоритетного потока, не теряя детерминизма.

Постановка ожидающих в дерево ожидающих rtmutex'а выполняется в порядке приоритета.
Для одинаковых приоритетов выбирается порядок FIFO. Для каждого rtmutex'а в дерево
приоритетных ожидающих владельца ставится только ожидающий с наивысшим приоритетом.
Это дерево также упорядочивается по приоритету. Всякий раз, когда ожидающий с
наивысшим приоритетом у задачи меняется (например, у него истёк тайм-аут или он
получил сигнал), приоритет задачи-владельца перенастраивается. Постановка в очередь
по приоритету обрабатывается через "pi_waiters".

RT-mutex'ы оптимизированы для операций быстрого пути (fastpath) и не несут
накладных расходов на внутреннюю блокировку при захвате неоспариваемого мьютекса
или разблокировке мьютекса без ожидающих. Оптимизированные операции быстрого пути
требуют поддержки cmpxchg. [Если она недоступна, то используется внутренний
спинлок (spinlock) rt-mutex'а.]

Состояние rt-mutex'а отслеживается через поле owner структуры rt-mutex'а:

lock->owner хранит указатель task_struct владельца. Бит 0 используется для
отслеживания состояния "у блокировки есть ожидающие":

 ============ ======= ================================================
 owner        bit0    Notes
 ============ ======= ================================================
 NULL         0       блокировка свободна (возможен быстрый захват)
 NULL         1       блокировка свободна и имеет ожидающих, и
		      ожидающий с наивысшим приоритетом собирается взять
		      блокировку [1]_
 taskpointer  0       блокировка удерживается (возможно быстрое
		      освобождение)
 taskpointer  1       блокировка удерживается и имеет ожидающих [2]_
 ============ ======= ================================================

Быстрые атомарные захват и освобождение на основе сравнения с обменом
(compare exchange) возможны только тогда, когда бит 0 поля lock->owner равен 0.

.. [1] Это также может быть переходным состоянием при захвате блокировки, когда
       удерживается ->wait_lock. Чтобы предотвратить любой cmpxchg быстрого пути к
       блокировке, нам нужно установить bit0 перед обращением к блокировке, и в этот
       короткий промежуток времени owner может быть NULL, поэтому это может быть
       переходным состоянием.

.. [2] Существует короткий промежуток времени, когда бит 0 установлен, но
       ожидающих нет. Это может произойти при захвате блокировки на медленном пути.
       Чтобы предотвратить cmpxchg для владельца, освобождающего блокировку, нам
       нужно установить этот бит перед обращением к блокировке.

Кстати, технически по-прежнему существует "Pending Owner", просто он больше так не
называется. Pending owner оказывается top_waiter'ом блокировки, у которой нет
владельца и который был разбужен, чтобы захватить блокировку.
