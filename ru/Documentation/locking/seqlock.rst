==========================================================
Счётчики последовательности и последовательные блокировки
==========================================================

Введение
========

Счётчики последовательности (sequence counters) — это механизм
согласованности «читатель — писатель» с безблокировочными читателями
(циклы повторных попыток только для чтения) и без голодания писателей.
Они применяются для данных, которые записываются редко (например,
системное время), когда читателю нужен согласованный набор информации и
он готов повторить попытку, если эта информация изменится.

Набор данных согласован, когда счётчик последовательности в начале
критической секции чтения чётный и то же значение счётчика
последовательности считывается снова в конце критической секции. Данные
из набора должны быть скопированы внутри критической секции чтения. Если
счётчик последовательности изменился между началом и концом критической
секции, читатель должен повторить попытку.

Писатели увеличивают счётчик последовательности в начале и в конце своей
критической секции. После начала критической секции счётчик
последовательности нечётный и сигнализирует читателям, что обновление
выполняется. В конце критической секции записи счётчик
последовательности снова становится чётным, что позволяет читателям
продвигаться вперёд.

Критическая секция записи счётчика последовательности никогда не должна
вытесняться или прерываться секциями чтения. Иначе читатель будет
крутиться в ожидании в течение всего тика планировщика из-за нечётного
значения счётчика последовательности и прерванного писателя. Если этот
читатель принадлежит к классу планирования реального времени, он может
крутиться вечно, и ядро войдёт в livelock.

Этот механизм нельзя использовать, если защищаемые данные содержат
указатели, так как писатель может сделать недействительным указатель, по
которому следует читатель.


.. _seqcount_t:

Счётчики последовательности (``seqcount_t``)
============================================

Это базовый механизм счёта, который не защищает от нескольких писателей.
Поэтому критические секции записи должны быть сериализованы внешней
блокировкой.

Если примитив сериализации записи неявно не запрещает вытеснение,
вытеснение необходимо явно запретить перед входом в секцию записи. Если
секция чтения может вызываться из контекстов hardirq или softirq,
прерывания или нижние половины (bottom halves) также должны быть
соответственно запрещены перед входом в секцию записи.

Если требуется автоматически удовлетворять требования счётчика
последовательности по сериализации писателей и невытесняемости,
используйте вместо этого :ref:`seqlock_t`.

Инициализация::

	/* dynamic */
	seqcount_t foo_seqcount;
	seqcount_init(&foo_seqcount);

	/* static */
	static seqcount_t foo_seqcount = SEQCNT_ZERO(foo_seqcount);

	/* C99 struct init */
	struct {
		.seq   = SEQCNT_ZERO(foo.seq),
	} foo;

Путь записи::

	/* Serialized context with disabled preemption */

	write_seqcount_begin(&foo_seqcount);

	/* ... [[write-side critical section]] ... */

	write_seqcount_end(&foo_seqcount);

Путь чтения::

	do {
		seq = read_seqcount_begin(&foo_seqcount);

		/* ... [[read-side critical section]] ... */

	} while (read_seqcount_retry(&foo_seqcount, seq));


.. _seqcount_locktype_t:

Счётчики последовательности с ассоциированными блокировками (``seqcount_LOCKNAME_t``)
-------------------------------------------------------------------------------------

Как обсуждалось в :ref:`seqcount_t`, критические секции записи счётчика
последовательности должны быть сериализованы и невытесняемы. Этот
вариант счётчиков последовательности ассоциирует блокировку,
используемую для сериализации писателей, во время инициализации, что
позволяет lockdep проверять, что критические секции записи правильно
сериализованы.

Эта ассоциация блокировки является NOOP, если lockdep отключён, и не
требует ни памяти, ни накладных расходов во время выполнения. Если
lockdep включён, указатель на блокировку сохраняется в struct seqcount,
и утверждения lockdep «блокировка удерживается» внедряются в начало
критической секции записи для проверки того, что она надлежащим образом
защищена.

Для типов блокировок, которые неявно не запрещают вытеснение, защита от
вытеснения обеспечивается в функции записи.

Определены следующие счётчики последовательности с ассоциированными
блокировками:

  - ``seqcount_spinlock_t``
  - ``seqcount_raw_spinlock_t``
  - ``seqcount_rwlock_t``
  - ``seqcount_mutex_t``
  - ``seqcount_ww_mutex_t``

API чтения и записи счётчика последовательности может принимать либо
обычный seqcount_t, либо любой из вариантов seqcount_LOCKNAME_t,
приведённых выше.

Инициализация (замените "LOCKNAME" на одну из поддерживаемых блокировок)::

	/* dynamic */
	seqcount_LOCKNAME_t foo_seqcount;
	seqcount_LOCKNAME_init(&foo_seqcount, &lock);

	/* static */
	static seqcount_LOCKNAME_t foo_seqcount =
		SEQCNT_LOCKNAME_ZERO(foo_seqcount, &lock);

	/* C99 struct init */
	struct {
		.seq   = SEQCNT_LOCKNAME_ZERO(foo.seq, &lock),
	} foo;

Путь записи: такой же, как в :ref:`seqcount_t`, при выполнении из
контекста с захваченной ассоциированной блокировкой сериализации записи.

Путь чтения: такой же, как в :ref:`seqcount_t`.


.. _seqcount_latch_t:

Защёлкивающие счётчики последовательности (``seqcount_latch_t``)
----------------------------------------------------------------

Защёлкивающие счётчики последовательности (latch sequence counters) —
это механизм управления конкурентным доступом с множеством версий
(multiversion concurrency control), в котором чётное/нечётное значение
встроенного счётчика seqcount_t используется для переключения между
двумя копиями защищаемых данных. Это позволяет пути чтения счётчика
последовательности безопасно прерывать собственную критическую секцию
записи.

Используйте seqcount_latch_t, когда секции записи не могут быть защищены
от прерывания читателями. Это типично, когда секция чтения может
вызываться из обработчиков NMI.

Дополнительную информацию см. в `write_seqcount_latch()`.


.. _seqlock_t:

Последовательные блокировки (``seqlock_t``)
===========================================

Содержит механизм :ref:`seqcount_t`, обсуждённый ранее, плюс встроенный
спинлок (spinlock) для сериализации писателей и невытесняемости.

Если секция чтения может вызываться из контекста hardirq или softirq,
используйте варианты функций записи, которые соответственно запрещают
прерывания или нижние половины (bottom halves).

Инициализация::

	/* dynamic */
	seqlock_t foo_seqlock;
	seqlock_init(&foo_seqlock);

	/* static */
	static DEFINE_SEQLOCK(foo_seqlock);

	/* C99 struct init */
	struct {
		.seql   = __SEQLOCK_UNLOCKED(foo.seql)
	} foo;

Путь записи::

	write_seqlock(&foo_seqlock);

	/* ... [[write-side critical section]] ... */

	write_sequnlock(&foo_seqlock);

Путь чтения, три категории:

1. Обычные читатели последовательности, которые никогда не блокируют
   писателя, но должны повторить попытку, если писатель выполняется,
   обнаружив изменение в номере последовательности. Писатели не ждут
   читателя последовательности::

	do {
		seq = read_seqbegin(&foo_seqlock);

		/* ... [[read-side critical section]] ... */

	} while (read_seqretry(&foo_seqlock, seq));

2. Блокирующие читатели, которые будут ждать, если выполняется писатель
   или другой блокирующий читатель. Выполняющийся блокирующий читатель
   также не даст писателю войти в свою критическую секцию. Эта
   блокировка чтения является эксклюзивной. В отличие от rwlock_t, её
   может захватить только один блокирующий читатель::

	read_seqlock_excl(&foo_seqlock);

	/* ... [[read-side critical section]] ... */

	read_sequnlock_excl(&foo_seqlock);

3. Условный безблокировочный читатель (как в п. 1) либо блокирующий
   читатель (как в п. 2), в зависимости от переданного маркера. Это
   используется, чтобы избежать голодания безблокировочных читателей
   (слишком много циклов повторных попыток) в случае резкого всплеска
   активности записи. Сначала пробуется безблокировочное чтение
   (передаётся чётный маркер). Если эта попытка завершается неудачей
   (счётчик последовательности не совпадает), сделайте маркер нечётным
   для следующей итерации, безблокировочное чтение преобразуется в
   полное блокирующее чтение, и цикл повторных попыток становится не
   нужен, например::

	/* marker; even initialization */
	int seq = 1;
	do {
		seq++; /* 2 on the 1st/lockless path, otherwise odd */
		read_seqbegin_or_lock(&foo_seqlock, &seq);

		/* ... [[read-side critical section]] ... */

	} while (need_seqretry(&foo_seqlock, seq));
	done_seqretry(&foo_seqlock, seq);
