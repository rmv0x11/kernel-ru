================
Futex Requeue PI
================

Переочередь (requeue) задач с не-PI futex на PI futex требует
особой обработки, чтобы гарантировать, что лежащий в основе rt_mutex
никогда не остаётся без владельца, если у него есть ожидающие; в противном случае
это нарушило бы логику PI-повышения (PI boosting) [см. rt-mutex-design.rst]. Для краткости
это действие далее по тексту называется «requeue_pi».
Наследование приоритетов (priority inheritance) по всему документу
сокращается как «PI».

Мотивация
---------

Без requeue_pi реализация в glibc функции
pthread_cond_broadcast() вынуждена прибегать к пробуждению всех задач, ожидающих
на pthread_condvar, предоставляя им самим разбираться, какая задача
получит право выполняться первой, в классической схеме «гремящего стада» (thundering-herd). Идеальная
реализация пробуждала бы ожидающего с наивысшим приоритетом и оставляла
остальных естественному пробуждению, заложенному в разблокировку мьютекса,
связанного с condvar.

Рассмотрим упрощённые вызовы glibc::

	/* caller must lock mutex */
	pthread_cond_wait(cond, mutex)
	{
		lock(cond->__data.__lock);
		unlock(mutex);
		do {
		unlock(cond->__data.__lock);
		futex_wait(cond->__data.__futex);
		lock(cond->__data.__lock);
		} while(...)
		unlock(cond->__data.__lock);
		lock(mutex);
	}

	pthread_cond_broadcast(cond)
	{
		lock(cond->__data.__lock);
		unlock(cond->__data.__lock);
		futex_requeue(cond->data.__futex, cond->mutex);
	}

После того как pthread_cond_broadcast() переочередит задачи, у cond->mutex
появляются ожидающие. Обратите внимание, что pthread_cond_wait() пытается заблокировать
мьютекс только после того, как вернётся в пространство пользователя. Это оставит
лежащий в основе rt_mutex с ожидающими, но без владельца, нарушив
упомянутые ранее алгоритмы PI-повышения.

Чтобы поддерживать PI-осведомлённые pthread_condvar, ядро должно
уметь переочередить задачи на PI futexes. Эта поддержка подразумевает, что
при успешном системном вызове futex_wait вызывающий вернётся в
пространство пользователя, уже удерживая PI futex. Реализация в glibc
была бы изменена следующим образом::


	/* caller must lock mutex */
	pthread_cond_wait_pi(cond, mutex)
	{
		lock(cond->__data.__lock);
		unlock(mutex);
		do {
		unlock(cond->__data.__lock);
		futex_wait_requeue_pi(cond->__data.__futex);
		lock(cond->__data.__lock);
		} while(...)
		unlock(cond->__data.__lock);
		/* the kernel acquired the mutex for us */
	}

	pthread_cond_broadcast_pi(cond)
	{
		lock(cond->__data.__lock);
		unlock(cond->__data.__lock);
		futex_requeue_pi(cond->data.__futex, cond->mutex);
	}

Фактическая реализация в glibc, скорее всего, будет проверять наличие PI и вносить
необходимые изменения внутри существующих вызовов, а не создавать новые
вызовы для случаев PI. Аналогичные изменения нужны для
pthread_cond_timedwait() и pthread_cond_signal().

Реализация
----------

Чтобы гарантировать, что у rt_mutex есть владелец, если у него есть ожидающие,
необходимо, чтобы и код переочереди, и код ожидания
могли захватить rt_mutex перед возвратом в пространство пользователя.
Код переочереди не может просто разбудить ожидающего и оставить ему
захват rt_mutex, поскольку это открыло бы окно гонки между
возвратом вызова переочереди в пространство пользователя и пробуждением ожидающего и
началом его выполнения. Это особенно верно в неоспариваемом случае.

Решение использует две новые вспомогательные процедуры rt_mutex,
rt_mutex_start_proxy_lock() и rt_mutex_finish_proxy_lock(), которые
позволяют коду переочереди захватить неоспариваемый rt_mutex от имени
ожидающего и поставить ожидающего в очередь на оспариваемый rt_mutex.
Два новых системных вызова обеспечивают интерфейс ядро<->пользователь к
requeue_pi: FUTEX_WAIT_REQUEUE_PI и FUTEX_CMP_REQUEUE_PI.

FUTEX_WAIT_REQUEUE_PI вызывается ожидающим (pthread_cond_wait()
и pthread_cond_timedwait()), чтобы заблокироваться на исходном futex и ждать
переочереди на PI-осведомлённый futex. Реализация является
результатом высокоскоростного столкновения между futex_wait() и
futex_lock_pi() с некоторой дополнительной логикой для проверки добавочных
сценариев пробуждения.

FUTEX_CMP_REQUEUE_PI вызывается пробуждающим
(pthread_cond_broadcast() и pthread_cond_signal()), чтобы переочередить и
возможно разбудить ожидающие задачи. Внутри этот системный вызов
по-прежнему обрабатывается futex_requeue (путём передачи requeue_pi=1). Перед
переочередью futex_requeue() пытается захватить целевой для переочереди
PI futex от имени верхнего ожидающего. Если это удаётся, этот ожидающий
пробуждается. Затем futex_requeue() приступает к переочереди оставшихся
nr_wake+nr_requeue задач на PI futex, вызывая
rt_mutex_start_proxy_lock() перед каждой переочередью, чтобы подготовить
задачу как ожидающего на лежащем в основе rt_mutex. Возможно, что
блокировка может быть захвачена и на этом этапе; если так, следующий
ожидающий пробуждается, чтобы завершить захват блокировки.

FUTEX_CMP_REQUEUE_PI принимает nr_wake и nr_requeue в качестве аргументов, но
по-настоящему важна лишь их сумма. futex_requeue() разбудит или
переочередит до nr_wake + nr_requeue задач. Он разбудит ровно столько
задач, для скольких сможет захватить блокировку, что в большинстве случаев
должно быть 0, поскольку хорошая практика программирования диктует, чтобы вызывающий
либо pthread_cond_broadcast(), либо pthread_cond_signal() захватывал
мьютекс перед выполнением вызова. FUTEX_CMP_REQUEUE_PI требует, чтобы
nr_wake=1. nr_requeue должен быть INT_MAX для broadcast и 0 для
signal.
