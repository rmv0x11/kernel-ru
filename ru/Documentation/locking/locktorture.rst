============================================================
Работа теста нагрузочного тестирования блокировок ядра
============================================================

CONFIG_LOCK_TORTURE_TEST
========================

Параметр конфигурации CONFIG_LOCK_TORTURE_TEST предоставляет модуль ядра,
который выполняет нагрузочные (torture) тесты базовых примитивов блокировки
ядра. Модуль ядра «locktorture» при необходимости может быть собран
постфактум для работающего ядра, которое требуется протестировать. Тесты
периодически выводят сообщения о состоянии через printk(), которые можно
просмотреть через dmesg (например, выполнив grep по строке «torture»). Тест
запускается при загрузке модуля и останавливается при его выгрузке. Эта
программа основана на том, как нагрузочно тестируется RCU, а именно
через rcutorture.

Этот нагрузочный тест заключается в создании некоторого числа потоков ядра,
которые захватывают блокировку и удерживают её в течение определённого
промежутка времени, тем самым имитируя различное поведение критических
секций. Степень конкуренции (contention) за блокировку можно имитировать
либо увеличением времени удержания этой критической секции, либо созданием
большего числа kthread, либо и тем и другим.


Параметры модуля
================

Этот модуль имеет следующие параметры:


Специфичные для locktorture
---------------------------

nwriters_stress
		  Число потоков ядра, которые будут нагружать монопольное
		  владение блокировкой (писатели). Значение по умолчанию вдвое
		  превышает число CPU в сети (online).

nreaders_stress
		  Число потоков ядра, которые будут нагружать совместное
		  владение блокировкой (читатели). По умолчанию совпадает с
		  числом блокировок писателей. Если пользователь не указал
		  nwriters_stress, то число читателей и писателей будет равно
		  числу CPU в сети (online).

torture_type
		  Тип блокировки для нагрузочного тестирования. По умолчанию
		  нагрузочному тестированию подвергаются только спинлоки. Этот
		  модуль может нагрузочно тестировать следующие блокировки,
		  задаваемые строковыми значениями:

		     - "lock_busted":
				Имитирует реализацию блокировки с ошибкой.

		     - "spin_lock":
				Пары spin_lock() и spin_unlock().

		     - "spin_lock_irq":
				Пары spin_lock_irq() и spin_unlock_irq().

		     - "rw_lock":
				Пары read/write lock() и unlock() для rwlock.

		     - "rw_lock_irq":
				Пары read/write lock_irq() и unlock_irq()
				для rwlock.

		     - "mutex_lock":
				Пары mutex_lock() и mutex_unlock().

		     - "rtmutex_lock":
				Пары rtmutex_lock() и rtmutex_unlock().
				Ядро должно иметь CONFIG_RT_MUTEXES=y.

		     - "rwsem_lock":
				Пары read/write down() и up() для семафора.


Каркас нагрузочного тестирования (RCU + блокировки)
---------------------------------------------------

shutdown_secs
		  Число секунд работы теста до его завершения и выключения
		  питания системы. По умолчанию равно нулю, что отключает
		  завершение теста и выключение системы. Эта возможность полезна
		  для автоматизированного тестирования.

onoff_interval
		  Число секунд между каждой попыткой выполнить случайно выбранную
		  операцию горячего подключения CPU (CPU-hotplug). По умолчанию
		  равно нулю, что отключает горячее подключение CPU. В ядрах с
		  CONFIG_HOTPLUG_CPU=n locktorture молча откажется выполнять
		  какие-либо операции горячего подключения CPU независимо от
		  того, какое значение указано для onoff_interval.

onoff_holdoff
		  Число секунд ожидания перед запуском операций горячего
		  подключения CPU (CPU-hotplug). Обычно это используется только
		  когда locktorture встроен в ядро и запущен автоматически во
		  время загрузки, в этом случае оно полезно для того, чтобы не
		  сбивать с толку код времени загрузки появлением и исчезновением
		  CPU. Этот параметр полезен только если включён
		  CONFIG_HOTPLUG_CPU.

stat_interval
		  Число секунд между связанными со статистикой вызовами printk().
		  По умолчанию locktorture сообщает статистику каждые 60 секунд.
		  Установка интервала в ноль приводит к тому, что статистика
		  выводится -только- при выгрузке модуля.

stutter
		  Промежуток времени работы теста перед паузой на такой же
		  промежуток времени. По умолчанию равно "stutter=5", то есть
		  работа и пауза выполняются (приблизительно) с пятисекундными
		  интервалами. Указание "stutter=0" приводит к тому, что тест
		  работает непрерывно без пауз.

shuffle_interval
		  Число секунд, в течение которых тестовые потоки удерживаются
		  привязанными (affinitized) к определённому подмножеству CPU,
		  по умолчанию 3 секунды. Используется совместно с
		  test_no_idle_hz.

verbose
		  Включает подробный отладочный вывод через printk(). Включено
		  по умолчанию. Эта дополнительная информация в основном связана
		  с высокоуровневыми ошибками и отчётами от основного каркаса
		  «torture».


Статистика
==========

Статистика выводится в следующем формате::

  spin_lock-torture: Writes:  Total: 93746064  Max/Min: 0/0   Fail: 0
     (A)		    (B)		   (C)		  (D)	       (E)

  (A): Тип блокировки, подвергаемой нагрузочному тестированию — параметр torture_type.

  (B): Число захватов блокировки писателями. Если речь идёт о примитиве
       чтения/записи, выводится вторая строка статистики «Reads».

  (C): Число случаев захвата блокировки.

  (D): Минимальное и максимальное число случаев, когда потокам не удалось
       захватить блокировку.

  (E): Значения true/false в зависимости от того, были ли ошибки при захвате
       блокировки. Это значение должно быть положительным -только- при наличии
       ошибки в реализации примитива блокировки. В противном случае захват
       блокировки никогда не должен завершаться неудачей (т.е. spin_lock()).
       Разумеется, то же относится к (C) выше. Фиктивным примером этого
       является тип "lock_busted".

Использование
=============

Для нагрузочного тестирования блокировок можно использовать следующий скрипт::

	#!/bin/sh

	modprobe locktorture
	sleep 3600
	rmmod locktorture
	dmesg | grep torture:

Вывод можно вручную проверить на наличие флага ошибки "!!!". Разумеется,
можно создать более сложный скрипт, который автоматически проверял бы
наличие таких ошибок. Команда "rmmod" вынуждает вывести через printk()
индикацию "SUCCESS", "FAILURE" или "RCU_HOTPLUG". Первые две говорят сами
за себя, а последняя указывает на то, что, хотя сбоев блокировок не было,
были обнаружены проблемы с горячим подключением CPU.
