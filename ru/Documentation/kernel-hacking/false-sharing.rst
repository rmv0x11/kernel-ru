==================================
Ложное разделение (False Sharing)
==================================

Что такое ложное разделение
===========================
Ложное разделение (false sharing) связано с механизмом кэша, который
поддерживает когерентность данных одной строки кэша (cache line),
хранящейся в кэшах нескольких CPU; академическое определение приведено
в [1]_. Рассмотрим структуру со счётчиком ссылок и строкой::

	struct foo {
		refcount_t refcount;
		...
		char name[16];
	} ____cacheline_internodealigned_in_smp;

Поля 'refcount'(A) и 'name'(B) _разделяют_ одну строку кэша, как показано
ниже::

                +-----------+                     +-----------+
                |   CPU 0   |                     |   CPU 1   |
                +-----------+                     +-----------+
               /                                        |
              /                                         |
             V                                          V
         +----------------------+             +----------------------+
         | A      B             | Cache 0     | A       B            | Cache 1
         +----------------------+             +----------------------+
                             |                  |
  ---------------------------+------------------+-----------------------------
                             |                  |
                           +----------------------+
                           |                      |
                           +----------------------+
              Main Memory  | A       B            |
                           +----------------------+

Поле 'refcount' модифицируется часто, а 'name' задаётся один раз при
создании объекта и больше никогда не изменяется. Когда множество CPU
обращаются к 'foo' одновременно, при этом 'refcount' часто инкрементируется
только одним CPU, а 'name' читается другими CPU, всем этим читающим CPU
приходится снова и снова перезагружать всю строку кэша из-за «разделения»,
хотя 'name' никогда не меняется.

Существует множество реальных случаев деградации производительности,
вызванной ложным разделением. Один из них — это rw_semaphore 'mmap_lock'
внутри структуры mm_struct, изменение раскладки строки кэша которой
вызвало деградацию; Линус проанализировал это в [2]_.

Есть два ключевых фактора вредоносного ложного разделения:

* Глобальные данные, к которым обращаются (которые разделяют) многие CPU
* В конкурентных обращениях к данным присутствует хотя бы одна операция
  записи: случаи запись/запись или запись/чтение.

Разделение может происходить как от совершенно несвязанных компонентов
ядра, так и от разных путей кода одного и того же компонента ядра.


Подводные камни ложного разделения
==================================
В прошлом, когда платформа имела только один или несколько CPU, «горячие»
поля данных можно было намеренно помещать в одну строку кэша, чтобы они
оставались «горячими» в кэше и экономили строки кэша/TLB, например
блокировка и защищаемые ею данные. Но для современных больших систем
с сотнями CPU это может не работать, когда блокировка сильно
конкурентна, поскольку CPU-владелец блокировки может писать в данные,
в то время как другие CPU заняты ожиданием блокировки на спине (spinning).

Если взглянуть на прошлые случаи, можно выделить несколько часто
встречающихся паттернов ложного разделения:

* блокировка (spinlock/mutex/semaphore) и защищаемые ею данные намеренно
  помещены в одну строку кэша.
* глобальные данные собраны вместе в одной строке кэша. У некоторых
  подсистем ядра много глобальных параметров небольшого размера (4 байта),
  которые легко сгруппировать вместе и поместить в одну строку кэша.
* поля данных большой структуры данных случайно оказались рядом, оставшись
  незамеченными (строка кэша обычно составляет 64 байта или более), как,
  например, у структуры 'mem_cgroup'.

В следующем разделе «mitigation» приведены реальные примеры.

Ложное разделение легко может возникнуть, если его специально не
проверять, и для нагрузок, критичных к производительности, имеет смысл
запускать специальные инструменты для обнаружения ложного разделения,
влияющего на производительность, и оптимизировать соответствующим
образом.


Как обнаружить и проанализировать ложное разделение
===================================================
perf record/report/stat широко используются для настройки
производительности, и как только обнаружены «горячие точки» (hotspots),
для дальнейшего обнаружения и точного определения возможных структур
данных с ложным разделением можно использовать инструменты вроде
'perf-c2c' и 'pahole'. 'addr2line' также хорош для декодирования
указателя инструкции, когда имеется несколько уровней inline-функций.

perf-c2c может захватывать строки кэша с наибольшим числом попаданий
ложного разделения, декодированные функции (номер строки файла),
обращающиеся к этой строке кэша, и внутреннее смещение данных.
Простые команды::

  $ perf c2c record -ag sleep 3
  $ perf c2c report --call-graph none -k vmlinux

will-it-scale perf выдаёт примерно такой отчёт::

до строки кэша. Пользователи могут сопоставить смещение из вывода
perf-c2c с декодированием pahole, чтобы найти точные поля данных. Для
глобальных данных пользователи могут искать адрес данных в System.map.


Возможные способы смягчения
===========================
Ложное разделение не всегда требует смягчения. Способы смягчения ложного
разделения должны уравновешивать прирост производительности со сложностью
и потреблением пространства. Иногда более низкая производительность
приемлема, и нет необходимости сверх-оптимизировать каждую редко
используемую структуру данных или «холодный» путь данных.

Случаи, когда ложное разделение вредит производительности, наблюдаются
тем чаще, чем больше число ядер. Из-за этих пагубных эффектов было
предложено и принято множество патчей в самых разных подсистемах
(таких как сетевая подсистема и управление памятью). Вот некоторые
распространённые способы смягчения (с примерами):

* Выделить «горячие» глобальные данные в их собственную отдельную строку
  кэша, даже если это просто тип 'short'. Недостаток — большее потребление
  памяти, строк кэша и записей TLB.

  - Commit 91b6d3256356 ("net: cache align tcp_memory_allocated, tcp_sockets_allocated")

* Реорганизовать структуру данных, разделив интерферирующие поля по разным
  строкам кэша. Один из недостатков в том, что это может привести к новому
  ложному разделению других полей.

  - Commit 802f1d522d5f ("mm: page_counter: re-layout structure to reduce false sharing")

* По возможности заменить «запись» на «чтение», особенно в циклах.
  Например, для некоторой глобальной переменной используйте
  сравнение(чтение)-затем-запись вместо безусловной записи. Например,
  используйте::

	if (!test_bit(XXX))
		set_bit(XXX);

  вместо непосредственного "set_bit(XXX);", аналогично для данных
  типа atomic_t::

	if (atomic_read(XXX) == AAA)
		atomic_set(XXX, BBB);

  - Commit 7b1002f7cfe5 ("bcache: fixup bcache_dev_sectors_dirty_add() multithreaded CPU false sharing")
  - Commit 292648ac5cf1 ("mm: gup: allow FOLL_PIN to scale in SMP")

* По возможности превратить «горячие» глобальные данные в «per-cpu данные
  + глобальные данные» или разумно увеличить порог синхронизации per-cpu
  данных с глобальными данными, чтобы уменьшить или отложить «запись»
  в эти глобальные данные.

  - Commit 520f897a3554 ("ext4: use percpu_counters for extent_status cache hits/misses")
  - Commit 56f3547bfa4d ("mm: adjust vm_committed_as_batch according to vm overcommit policy")

Безусловно, все способы смягчения должны быть тщательно проверены, чтобы
не вызвать побочных эффектов. Чтобы не вносить ложное разделение при
написании кода, лучше:

* Помнить о границах строк кэша
* Группировать преимущественно read-only поля вместе
* Группировать вместе то, что записывается в одно и то же время
* Разделять часто читаемые и часто записываемые поля по разным строкам
  кэша.

и лучше добавить комментарий, в котором указывается, что учтено ложное
разделение.

Стоит отметить, что иногда даже после того, как серьёзное ложное
разделение обнаружено и устранено, производительность может всё равно
не дать заметного улучшения, поскольку «горячая точка» перемещается
в новое место.


Прочее
======

ситуацию с разделением строк кэша между полями данных.
