=========================
Загрузка ARM Linux
=========================

Author:	Russell King

Date  : 18 May 2002

Эта документация актуальна для версии 2.4.18-rmk6 и более поздних.

Чтобы загрузить ARM Linux, требуется boot loader — небольшая программа,
которая выполняется до основного ядра. От boot loader ожидается, что он
инициализирует различные устройства и в конечном итоге вызовет ядро Linux,
передав ему информацию.

По сути, boot loader должен обеспечивать (как минимум) следующее:

1. Настройку и инициализацию RAM.
2. Инициализацию одного последовательного порта.
3. Определение типа машины.
4. Настройку тегированного списка ядра.
5. Загрузку initramfs.
6. Вызов образа ядра.


1. Настройка и инициализация RAM
--------------------------------

Existing boot loaders:
	MANDATORY
New boot loaders:
	MANDATORY

От boot loader ожидается, что он найдёт и инициализирует всю RAM, которую
ядро будет использовать для хранения энергозависимых данных в системе. Он
выполняет это машинно-зависимым образом. (Он может использовать внутренние
алгоритмы для автоматического обнаружения и определения объёма всей RAM,
либо использовать знание о RAM в данной машине, либо любой другой метод,
который разработчик boot loader сочтёт подходящим.)


2. Инициализация одного последовательного порта
-----------------------------------------------

Existing boot loaders:
	OPTIONAL, RECOMMENDED
New boot loaders:
	OPTIONAL, RECOMMENDED

Boot loader должен инициализировать и включить один последовательный порт
на целевой машине. Это позволяет драйверу последовательного порта ядра
автоматически определить, какой последовательный порт ему следует
использовать в качестве консоли ядра (обычно используется для целей отладки
или для связи с целевой машиной).

В качестве альтернативы boot loader может передать ядру соответствующую
опцию 'console=' через тегированные списки, указав порт и параметры формата
последовательного порта, как описано в

       Documentation/admin-guide/kernel-parameters.rst.


3. Определение типа машины
--------------------------

Existing boot loaders:
	OPTIONAL
New boot loaders:
	MANDATORY except for DT-only platforms

Boot loader должен каким-либо способом определить тип машины, на которой он
работает. Является ли это жёстко заданным значением или некоторым алгоритмом,
анализирующим подключённое оборудование, — выходит за рамки данного документа.
Boot loader должен в конечном счёте быть способен предоставить ядру значение
MACH_TYPE_xxx (см. linux/arch/arm/tools/mach-types). Оно должно быть передано
ядру в регистре r1.

Для платформ, использующих только DT (DT-only), тип машины будет определяться
по device tree. Установите тип машины во все единицы (~0). Это не является
строго обязательным, но гарантирует, что он не совпадёт ни с одним из
существующих типов.

4. Настройка данных загрузки
----------------------------

Existing boot loaders:
	OPTIONAL, HIGHLY RECOMMENDED
New boot loaders:
	MANDATORY

Boot loader должен предоставить либо тегированный список, либо образ dtb для
передачи ядру данных конфигурации. Физический адрес данных загрузки передаётся
ядру в регистре r2.

4a. Настройка тегированного списка ядра
---------------------------------------

Boot loader должен создать и инициализировать тегированный список ядра.
Корректный тегированный список начинается с ATAG_CORE и заканчивается
ATAG_NONE. Тег ATAG_CORE может быть как пустым, так и непустым. У пустого
тега ATAG_CORE поле размера установлено в '2' (0x00000002). У ATAG_NONE
поле размера должно быть установлено в ноль.

В списке может размещаться любое количество тегов. Не определено, дополняет ли
повторяющийся тег информацию, переносимую предыдущим тегом, или же полностью
заменяет её; одни теги ведут себя первым образом, другие — вторым.

Boot loader должен передавать как минимум размер и расположение системной
памяти, а также расположение корневой файловой системы. Поэтому минимальный
тегированный список должен выглядеть так::

		+-----------+
  base ->	| ATAG_CORE |  |
		+-----------+  |
		| ATAG_MEM  |  | increasing address
		+-----------+  |
		| ATAG_NONE |  |
		+-----------+  v

Тегированный список должен храниться в системной RAM.

Тегированный список должен быть размещён в области памяти, которую не
перезапишут ни декомпрессор ядра, ни программа 'bootp' для initrd.
Рекомендуемое размещение — в первых 16 KiB RAM.

4b. Настройка device tree
-------------------------

Boot loader должен загрузить образ device tree (dtb) в системную RAM по
адресу, выровненному по 64 битам, и инициализировать его данными загрузки.
Формат dtb описан по адресу https://www.devicetree.org/specifications/.
Ядро будет искать магическое значение dtb 0xd00dfeed по физическому адресу
dtb, чтобы определить, был ли передан dtb вместо тегированного списка.

Boot loader должен передавать как минимум размер и расположение системной
памяти, а также расположение корневой файловой системы. Dtb должен быть
размещён в области памяти, которую не перезапишет декомпрессор ядра, оставаясь
при этом в пределах области, которая будет покрыта отображением нижней памяти
(low-memory mapping) ядра.

Безопасное расположение — сразу выше границы 128 MiB от начала RAM.

5. Загрузка initramfs.
----------------------

Existing boot loaders:
	OPTIONAL
New boot loaders:
	OPTIONAL

Если используется initramfs, то, как и dtb, он должен быть размещён в области
памяти, которую не перезапишет декомпрессор ядра, оставаясь при этом в
пределах области, которая будет покрыта отображением нижней памяти
(low-memory mapping) ядра.

Безопасное расположение — сразу выше блоба device tree, который сам будет
загружен сразу выше границы 128 MiB от начала RAM, как рекомендовано выше.

6. Вызов образа ядра
--------------------

Existing boot loaders:
	MANDATORY
New boot loaders:
	MANDATORY

Существуют два варианта вызова zImage ядра. Если zImage хранится во flash и
корректно скомпонован для запуска из flash, то для boot loader допустимо
вызвать zImage во flash напрямую.

zImage также может быть размещён в системной RAM и вызван оттуда. Ядро должно
быть размещено в первых 128 MiB RAM. Рекомендуется загружать его выше 32 MiB,
чтобы избежать необходимости перемещения перед декомпрессией, что сделает
процесс загрузки немного быстрее.

При загрузке «сырого» ядра (не zImage) ограничения жёстче. В этом случае ядро
должно быть загружено в систему по смещению, равному TEXT_OFFSET - PAGE_OFFSET.

В любом случае должны быть выполнены следующие условия:

- Перевести в состояние покоя все устройства, способные к DMA, чтобы память не
  была повреждена ложными сетевыми пакетами или данными с диска. Это сэкономит
  вам многие часы отладки.

- Настройки регистров CPU

  - r0 = 0,
  - r1 = номер типа машины, определённый в пункте (3) выше.
  - r2 = физический адрес тегированного списка в системной RAM, либо
    физический адрес блока device tree (dtb) в системной RAM

- Режим CPU

  Все формы прерываний должны быть отключены (IRQ и FIQ)

  Для CPU, которые не включают расширения виртуализации ARM, CPU должен
  находиться в режиме SVC. (Существует специальное исключение для Angel)

  CPU, которые включают поддержку расширений виртуализации, могут входить в
  режим HYP, чтобы позволить ядру в полной мере использовать эти расширения.
  Это рекомендуемый метод загрузки для таких CPU, если только средства
  виртуализации уже не используются предустановленным гипервизором.

  Если ядро по какой-либо причине не входит в режим HYP, оно должно входить
  в режим SVC.

- Кэши, MMU

  MMU должен быть выключен.

  Кэш инструкций может быть как включён, так и выключен.

  Кэш данных должен быть выключен.

  Если ядро входит в режиме HYP, перечисленные выше требования применяются к
  конфигурации режима HYP в дополнение к обычной конфигурации PL1
  (привилегированные режимы ядра). Кроме того, все ловушки (traps) в
  гипервизор должны быть отключены, а доступ PL1 должен быть предоставлен для
  всех периферийных устройств и ресурсов CPU, для которых это архитектурно
  возможно. За исключением входа в режим HYP, конфигурация системы должна быть
  такой, чтобы ядро, не включающее поддержку расширений виртуализации, могло
  корректно загрузиться без дополнительной помощи.

- От boot loader ожидается, что он вызовет образ ядра, перейдя
  непосредственно к первой инструкции образа ядра.

  На CPU, поддерживающих набор инструкций ARM, вход должен осуществляться
  в состоянии ARM, даже для ядра Thumb-2.

  На CPU, поддерживающих только набор инструкций Thumb, таких как CPU класса
  Cortex-M, вход должен осуществляться в состоянии Thumb.
