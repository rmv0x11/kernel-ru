============================
Процесс загрузки Chromebook
============================

Большинство современных устройств Chromebook, использующих device tree, применяют
открытый загрузчик depthcharge_. Depthcharge_ ожидает, что ОС упакована в виде `FIT
Image`_, который содержит образ ОС, а также набор device tree. Именно depthcharge_
отвечает за выбор правильного device tree из `FIT Image`_ и его передачу ОС.

Схема, по которой depthcharge_ выбирает device tree, учитывает три переменные:

- Имя платы (board name), задаваемое во время компиляции depthcharge_. Далее это $(BOARD).
- Номер ревизии платы (board revision number), определяемый во время выполнения
  (возможно, путём чтения GPIO-перемычек, возможно, каким-то другим способом). Далее это $(REV).
- Номер SKU, считываемый из GPIO-перемычек во время загрузки. Далее это $(SKU).

Для современных устройств Chromebook depthcharge_ создаёт список соответствий,
который выглядит следующим образом:

- google,$(BOARD)-rev$(REV)-sku$(SKU)
- google,$(BOARD)-rev$(REV)
- google,$(BOARD)-sku$(SKU)
- google,$(BOARD)

Обратите внимание, что некоторые более старые устройства Chromebook используют
несколько иной список, который может не включать сопоставление по SKU или может
расставлять приоритеты SKU/rev иначе.

Обратите внимание, что для некоторых плат может существовать дополнительная
специфичная для платы логика, добавляющая в список дополнительные значения
compatible, но это встречается нечасто.

Depthcharge_ просматривает все device tree в `FIT Image`_, пытаясь найти тот,
который соответствует наиболее специфичному значению compatible. Затем он
просматривает все device tree в `FIT Image`_, пытаясь найти тот, который
соответствует *второму по специфичности* значению compatible, и так далее.

При поиске device tree depthcharge_ не имеет значения, в каком месте массива
корневых строк compatible конкретного device tree находится строка compatible.
В качестве примера: если мы находимся на плате «lazor», ревизия 4, SKU 0, и у нас
есть два device tree:

- "google,lazor-rev5-sku0", "google,lazor-rev4-sku0", "qcom,sc7180"
- "google,lazor", "qcom,sc7180"

тогда depthcharge_ выберет первый device tree, даже несмотря на то, что
"google,lazor-rev4-sku0" является вторым значением compatible, перечисленным в этом
device tree. Это связано с тем, что данное значение compatible более специфично, чем
"google,lazor".

Следует отметить, что у depthcharge_ нет какой-либо логики, пытающейся подобрать
«близкие» ревизии платы или SKU. То есть, если depthcharge_ знает, что находится на
«rev4» платы, но device tree для «rev4» отсутствует, то depthcharge_ *не* станет
искать device tree для «rev3».

В общем случае при внесении любых значимых изменений в плату номер ревизии платы
увеличивается, даже если ни одно из этих изменений не требует отражения в device tree.
Поэтому довольно часто можно встретить device tree с несколькими ревизиями.

Следует отметить, что с учётом описанной выше системы, которую использует
depthcharge_, наибольшая гибкость достигается, если device tree, поддерживающий
самые новые ревизии платы, опускает строки compatible вида "-rev{REV}". Когда это
сделано, то при появлении новой ревизии платы и попытке запустить на ней старое
программное обеспечение мы по крайней мере выберем самый новый device tree, о котором
нам известно.
