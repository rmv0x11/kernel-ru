=========================
ARM Linux 2.6 и новее
=========================

    Актуальные обновления см. на <ftp://ftp.arm.linux.org.uk/pub/armlinux>.

Компиляция ядра
---------------

  Чтобы скомпилировать ARM Linux, вам понадобится компилятор, способный
  генерировать код ARM ELF с расширениями GNU.  Известно, что GCC 3.3 —
  хороший компилятор.  К счастью, гадать не придётся.  Ядро сообщит об
  ошибке, если ваш компилятор является известным нарушителем.

  Чтобы собрать ARM Linux нативно, вам не придётся менять строку ARCH = в
  Makefile верхнего уровня.  Однако, если у вас не установлены по умолчанию
  инструменты ARM Linux ELF, тогда следует изменить строку CROSS_COMPILE,
  как описано ниже.

  Если вы хотите выполнить кросс-компиляцию, измените следующие строки в
  make-файле верхнего уровня::

    ARCH = <whatever>

  на::

    ARCH = arm

  и::

    CROSS_COMPILE=

  на::

    CROSS_COMPILE=<your-path-to-your-compiler-without-gcc>

  например::

    CROSS_COMPILE=arm-linux-

  Выполните 'make config', а затем 'make Image', чтобы собрать ядро
  (arch/arm/boot/Image).  Сжатый образ можно собрать, выполнив
  'make zImage' вместо 'make Image'.


Отчёты об ошибках и прочее
--------------------------

  Пожалуйста, отправляйте патчи в систему патчей.  Подробнее см.
  http://www.arm.linux.org.uk/developer/patches/info.php Всегда включайте
  некоторое пояснение о том, что делает патч и зачем он нужен.

  Отчёты об ошибках следует отправлять на linux-arm-kernel@lists.arm.linux.org.uk
  или через веб-форму по адресу
  http://www.arm.linux.org.uk/developer/

  При отправке отчётов об ошибках убедитесь, что они содержат всю значимую
  информацию, например сообщения ядра, которые были выведены до/во время
  проблемы, что вы делали, и т.д.


Заголовочные файлы (include files)
----------------------------------

  Под include/asm-arm создано несколько новых каталогов include, которые
  существуют для уменьшения беспорядка в каталоге верхнего уровня.  Эти
  каталоги и их назначение перечислены ниже:

  ============= ==========================================================
   `arch-*`	специфичные для машины/платформы заголовочные файлы
   `hardware`	внутренние для драйверов ARM-специфичные структуры данных/определения
   `mach`	описания интерфейсов от обобщённого ARM к конкретной машине
   `proc-*`	зависящие от процессора заголовочные файлы (сейчас всего две
		категории)
  ============= ==========================================================


Поддержка машин/платформ
------------------------

  Дерево ARM содержит поддержку множества различных типов машин.  Чтобы
  продолжать поддерживать эти различия, стало необходимым разделить
  специфичные для машины части по каталогам.  Для этого используется
  категория машины, которая служит для выбора того, какие каталоги и файлы
  будут включены (мы будем использовать $(MACHINE) для обозначения этой
  категории).

  С этой целью у нас теперь есть каталоги arch/arm/mach-$(MACHINE), которые
  предназначены для размещения файлов конкретной машины, не относящихся к
  драйверам (например, PCI, управление памятью, определения архитектуры и
  т.д.).  Для всех будущих машин должен существовать соответствующий каталог
  arch/arm/mach-$(MACHINE)/include/mach.


Модули
------

  Хотя модуляризация поддерживается (и обязательна для эмулятора FP), каждый
  модуль на машине ARM2/ARM250/ARM3 при загрузке будет занимать память
  вплоть до следующей границы в 32k из-за размера страниц.  Поэтому стоит ли
  модуляризация на этих машинах того?

  Однако машины ARM6 и старше позволяют модулям занимать память кратно 4k, и
  поэтому Acorn RiscPC и другие архитектуры, использующие эти процессоры,
  могут с пользой применять модуляризацию.


Файлы образов ADFS
------------------

  Вы можете получить доступ к файлам образов на ваших разделах ADFS,
  смонтировав раздел ADFS, а затем используя драйвер loopback-устройства.  У
  вас должен быть установлен losetup.

  Обратите внимание, что DOS-разделы PCEmulator имеют таблицу разделов в
  начале, и поэтому вам придётся передать '-o offset' для losetup.


Просьба к разработчикам
-----------------------

  При написании драйверов устройств, которые включают отдельный файл на
  ассемблере, включайте его вместе с C-файлом, а не в каталог arch/arm/lib.
  Это позволяет компилировать драйвер как загружаемый модуль без
  необходимости компилировать половину кода в образ ядра.

  В целом старайтесь избегать использования ассемблера, если только он
  действительно не необходим.  Это значительно усложняет перенос драйверов
  на другое оборудование.


Жёсткие диски ST506
-------------------

  Контроллеры жёстких дисков ST506, похоже, работают нормально (хотя и
  немного медленно).  На данный момент они будут работать только с
  контроллерами на материнской плате A4x0, но чтобы это заработало с Podule,
  требуется лишь, чтобы кто-то с podule добавил в исходники адреса для маски
  IRQ и базы HDC.

  По состоянию на 31/3/96 это работает с двумя дисками (вы должны установить
  ADFS `*configure` harddrive в значение 2).  У меня есть внутренний диск на
  20MB и здоровенный внешний диск 5.25" FH на 64MB (кому вообще может
  понадобиться больше :-) ).

  Я только что получил с него 240K/s (dd с bs=128k); это примерно половина
  того, что получает RiscOS; но это чертовски лучше, чем 50K/s, которые я
  получал на прошлой неделе :-)

  Известный баг: ошибки данных на диске могут вызвать зависание; включая
  случаи, когда контроллер исправил ошибку с помощью ECC.  (Возможно, ТОЛЬКО
  в этом случае... хм).


Флоппи 1772
-----------

`CONFIG_MACH_` and `CONFIG_ARCH_`
---------------------------------
  В 2003 году было внесено изменение в имена макросов для новых машин.
  Исторически `CONFIG_ARCH_` использовался для самой архитектуры,
  например SA1100, а также для реализаций архитектуры, например Assabet.
  Было решено изменить макросы реализаций на `CONFIG_MACH_` для ясности.
  Более того, ретроактивная правка не была сделана, поскольку это усложнило
  бы наложение патчей.

  Предыдущие регистрации можно найти в сети.

    <http://www.arm.linux.org.uk/developer/machines/>

Точка входа в ядро (head.S)
---------------------------
  Изначальный вход в ядро осуществляется через head.S, который использует
  машинно-независимый код.  Машина выбирается значением 'r1' при входе,
  которое должно оставаться уникальным.

  Из-за большого количества машин, для которых предусмотрен порт Linux на
  ARM, у нас есть метод управления этим, который гарантирует, что мы не
  будем в итоге дублировать большие объёмы кода.

  Мы группируем код поддержки машин (или платформ) в классы машин.  Класс
  обычно основан на одном или нескольких устройствах «система на кристалле»
  (system on a chip) и служит естественным контейнером для самих реализаций.
  Этим классам выделяются каталоги — arch/arm/mach-<class> — которые
  содержат исходные файлы и include/mach/ для поддержки класса машин.

  Например, класс SA1100 основан на устройствах SoC SA1100 и SA1110 и
  содержит код для поддержки того, как используются встроенные и внешние
  устройства, или того, как устройство настраивается, и обеспечивает эту
  специфичную для машины «индивидуальность».

  Для платформ, поддерживающих device tree (DT), выбор машины управляется во
  время выполнения путём передачи ядру blob-файла device tree.  Во время
  компиляции должна быть выбрана поддержка типа машины.  Это позволяет
  использовать одну мультиплатформенную сборку ядра для нескольких типов
  машин.

  Для платформ, не использующих device tree, этот выбор машины управляется
  идентификатором типа машины, который служит методом выбора кода как во
  время выполнения, так и во время компиляции.  Вы можете зарегистрировать
  новую машину через веб-сайт по адресу:

    <http://www.arm.linux.org.uk/developer/machines/>

  Примечание: пожалуйста, не регистрируйте тип машины для платформ,
  работающих только с DT.  Если ваша платформа работает только с DT, вам не
  нужен зарегистрированный тип машины.
