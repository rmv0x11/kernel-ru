=============================================================================================
Использование физического DMA, предоставляемого FireWire-контроллерами OHCI-1394, для отладки
=============================================================================================

Введение
--------

По сути все используемые сегодня FireWire-контроллеры совместимы со
спецификацией OHCI-1394, которая определяет контроллер как мастер шины PCI,
использующий DMA для разгрузки CPU от передачи данных и имеющий
«Physical Response Unit», который выполняет определённые запросы посредством
DMA в режиме мастера PCI-шины после применения фильтров, заданных драйвером
OHCI-1394.

После надлежащей настройки удалённые машины могут отправлять такие запросы,
чтобы попросить контроллер OHCI-1394 выполнить запросы на чтение и запись в
физической памяти системы и, для запросов на чтение, отправить результат
чтения физической памяти обратно запрашивающей стороне.

Благодаря этому можно отлаживать проблемы, читая интересующие области памяти,
такие как буферы вроде буфера printk или таблицы процессов.

Также возможно получение полного дампа памяти системы по FireWire со скоростью
передачи данных порядка 10 МБ/с и более.

С большинством FireWire-контроллеров доступ к памяти ограничен нижними 4 ГБ
физического адресного пространства. Это может быть проблемой на машинах, где
память расположена преимущественно выше этого предела, но это редко является
проблемой на более распространённом оборудовании, таком как x86, x86-64 и PowerPC.

По крайней мере контроллеры LSI FW643e и FW643e2, как известно, поддерживают
доступ к физическим адресам выше 4 ГБ, но в настоящее время эта возможность не
включена в Linux.

Вместе с ранней инициализацией контроллера OHCI-1394 для отладки этот механизм
оказался крайне полезным для изучения длинных отладочных логов в буфере printk
при отладке проблем ранней загрузки в таких областях, как ACPI, где система не
загружается, а другие средства отладки (последовательный порт) либо недоступны
(ноутбуки), либо слишком медленны для обширной отладочной информации (как в случае ACPI).

Драйверы
--------

Драйвер firewire-ohci в drivers/firewire по умолчанию использует фильтруемый
физический DMA, что более безопасно, но не подходит для удалённой отладки.
Передайте драйверу параметр remote_dma=1, чтобы получить нефильтруемый физический DMA.

Поскольку драйвер firewire-ohci зависит от завершения перечисления PCI, для x86
была реализована процедура инициализации, которая выполняется довольно рано. Эта
процедура выполняется задолго до того, как может быть вызвана console_init(),
то есть до появления буфера printk на консоли.

Чтобы активировать её, включите CONFIG_PROVIDE_OHCI1394_DMA_INIT (меню Kernel hacking:
Remote debugging over FireWire early on boot) и передайте перекомпилированному ядру
при загрузке параметр "ohci1394_dma=early".

Инструменты
-----------

firescope — изначально разработана Benjamin Herrenschmidt, Andi Kleen перенёс её
с PowerPC на x86 и x86_64 и добавил функциональность; теперь firescope можно
использовать для просмотра буфера printk удалённой машины, даже с обновлением в реальном времени.

Bernhard Kaindl усовершенствовал firescope, добавив поддержку доступа к 64-битным
машинам из 32-битной firescope и наоборот:
- http://v3.sk/~lkundrak/firescope/

а также он реализовал быстрый дамп системы (альфа-версия — прочтите README.txt):
- http://halobates.de/firewire/firedump-0.1.tar.bz2

Существует также gdb-прокси для firewire, который позволяет использовать gdb для
доступа к данным, на которые можно ссылаться по символам, найденным gdb в vmlinux:
- http://halobates.de/firewire/fireproxy-0.33.tar.bz2

Новейшая версия этого gdb-прокси (fireproxy-0.34) может взаимодействовать (пока
нестабильно) с kgdb через модуль связи на основе памяти (kgdbom).

Приступая к работе
------------------

Спецификация OHCI-1394 предписывает, что контроллер OHCI-1394 должен отключать
весь физический DMA при каждом сбросе шины.

Это означает, что если вы хотите отладить проблему в состоянии системы, когда
прерывания отключены и не происходит опроса контроллера OHCI-1394 на предмет
сбросов шины, вы должны установить все кабельные соединения FireWire и полностью
инициализировать всё оборудование FireWire __до__ того, как система перейдёт в
такое состояние.

Пошаговые инструкции по использованию firescope с ранней инициализацией OHCI:

1) Убедитесь, что ваше оборудование поддерживается:

   Загрузите модуль firewire-ohci и проверьте логи ядра.
   Вы должны увидеть строку, похожую на::

     firewire_ohci 0000:15:00.1: added OHCI v1.0 device as card 2, 4 IR + 4 IT
     ... contexts, quirks 0x11

   при загрузке драйвера. Если у вас нет поддерживаемого контроллера, доступно
   множество карт PCI, CardBus и даже некоторых Express, полностью совместимых
   со спецификацией OHCI-1394. Если карта не требует драйвера для операционных
   систем Windows, она, скорее всего, совместима. Только специализированные
   магазины имеют несовместимые карты, они основаны на чипах TI PCILynx и
   требуют драйверов для операционных систем Windows.

   Упомянутое сообщение в логе ядра содержит строку "physUB", если контроллер
   реализует доступный для записи регистр Physical Upper Bound. Это требуется для
   физического DMA выше 4 ГБ (но пока не используется в Linux).

2) Установите рабочее кабельное соединение FireWire:

   Подойдёт любой кабель FireWire, если он обеспечивает электрически и механически
   стабильное соединение и имеет подходящие разъёмы (существуют малые 4-контактные
   и большие 6-контактные порты FireWire).

   Если драйвер запущен на обеих машинах, вы должны увидеть строку вроде::

     firewire_core 0000:15:00.1: created device fw1: GUID 00061b0020105917, S400

   на обеих машинах в логе ядра, когда кабель подключён и соединяет две машины.

3) Проверьте физический DMA с помощью firescope:

   На отладочном хосте убедитесь, что /dev/fw* доступен,
   затем запустите firescope::

	$ firescope
	Port 0 (/dev/fw1) opened, 2 nodes detected

	FireScope
	---------
	Target : <unspecified>
	Gen    : 1
	[Ctrl-T] choose target
	[Ctrl-H] this menu
	[Ctrl-Q] quit

    ------> Press Ctrl-T now, the output should be similar to:

	2 nodes available, local node is: 0
	 0: ffc0, uuid: 00000000 00000000 [LOCAL]
	 1: ffc1, uuid: 00279000 ba4bb801

   Помимо узла [LOCAL], он должен показать ещё один узел без сообщения об ошибке.

4) Подготовьтесь к отладке с ранней инициализацией OHCI-1394:

   4.1) Компиляция и установка ядра на отладочную цель

   Скомпилируйте отлаживаемое ядро с включённым CONFIG_PROVIDE_OHCI1394_DMA_INIT
   (Kernel hacking: Provide code for enabling DMA over FireWire early on boot)
   и установите его на отлаживаемую машину (отладочную цель).

   4.2) Перенесите System.map отлаживаемого ядра на отладочный хост

   Скопируйте System.map отлаживаемого ядра на отладочный хост (хост, который
   подключён к отлаживаемой машине через кабель FireWire).

5) Получение содержимого буфера printk:

   При подключённом кабеле FireWire и загруженном драйвере OHCI-1394 на отладочном
   хосте перезагрузите отлаживаемую машину, загрузив ядро, в котором включён
   CONFIG_PROVIDE_OHCI1394_DMA_INIT, с параметром ohci1394_dma=early.

   Затем на отладочном хосте запустите firescope, например с использованием -A::

	firescope -A System.map-of-debug-target-kernel

   Примечание: -A автоматически подключается к первому нелокальному узлу. Это
   работает надёжно, только если по FireWire соединены лишь две машины.

   После подключения к отладочной цели нажмите Ctrl-D, чтобы просмотреть полный
   буфер printk, или Ctrl-U, чтобы войти в режим автоматического обновления и
   получить обновляемое в реальном времени представление недавних сообщений ядра,
   записанных на отладочной цели.

   Вызовите "firescope -h", чтобы получить больше информации о параметрах firescope.
