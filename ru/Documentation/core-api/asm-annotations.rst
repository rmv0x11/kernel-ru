Аннотации в ассемблере
======================

Copyright (c) 2017-2019 Jiri Slaby

Этот документ описывает новые макросы для аннотации данных и кода на ассемблере.
В частности, он содержит информацию о ``SYM_FUNC_START``, ``SYM_FUNC_END``,
``SYM_CODE_START`` и им подобных.

Обоснование
-----------
Некоторый код, например точки входа (entries), трамплины (trampolines) или код
загрузки, должен быть написан на ассемблере. Как и в C, такой код группируется
в функции и сопровождается данными. Стандартные ассемблеры не заставляют
пользователей точно помечать эти фрагменты как код, данные или хотя бы указывать
их длину. Тем не менее ассемблеры предоставляют разработчикам такие аннотации,
чтобы помогать отладчикам по всему ассемблерному коду. Помимо этого, разработчики
также хотят помечать некоторые функции как *global*, чтобы они были видимы за
пределами своих единиц трансляции.

Со временем ядро Linux переняло макросы из различных проектов (таких как
``binutils``), чтобы упростить такие аннотации. Поэтому по историческим причинам
разработчики использовали в ассемблере ``ENTRY``, ``END``, ``ENDPROC`` и другие
аннотации. Из-за отсутствия их документации эти макросы в некоторых местах
используются в довольно неправильных контекстах. Очевидно, что ``ENTRY``
предназначался для обозначения начала глобальных символов (будь то данные или
код). ``END`` использовался для обозначения конца данных или конца специальных
функций с *нестандартным* соглашением о вызовах. В противоположность этому
``ENDPROC`` должен аннотировать только концы *стандартных* функций.

При правильном использовании эти макросы помогают ассемблерам генерировать
хороший объект с корректно установленными размерами и типами. Например, результат
для ``arch/x86/lib/putuser.S``::

   Num:    Value          Size Type    Bind   Vis      Ndx Name
    25: 0000000000000000    33 FUNC    GLOBAL DEFAULT    1 __put_user_1
    29: 0000000000000030    37 FUNC    GLOBAL DEFAULT    1 __put_user_2
    32: 0000000000000060    36 FUNC    GLOBAL DEFAULT    1 __put_user_4
    35: 0000000000000090    37 FUNC    GLOBAL DEFAULT    1 __put_user_8

Это важно не только для целей отладки. Когда есть надлежащим образом
аннотированные объекты, подобные этому, на них можно запускать инструменты для
генерации более полезной информации. В частности, на надлежащим образом
аннотированных объектах можно запускать ``objtool``, чтобы проверить и при
необходимости исправить объект. В настоящее время ``objtool`` может сообщать об
отсутствующей установке/уничтожении указателя кадра (frame pointer) в функциях.
Он также может автоматически генерировать аннотации для механизма раскрутки
стека ORC (Documentation/arch/x86/orc-unwinder.rst)
для большей части кода. Оба эти свойства особенно важны для поддержки надёжных
трассировок стека, которые, в свою очередь, необходимы для «живого»
патчинга ядра (live patching)
(Documentation/livepatch/livepatch.rst).

Оговорки и обсуждение
---------------------
Как можно заметить, ранее существовало всего три макроса. Этого действительно
недостаточно, чтобы покрыть все сочетания случаев:

* стандартная/нестандартная функция
* код/данные
* глобальный/локальный символ

Состоялось обсуждение (discussion_), и вместо расширения существующих макросов
``ENTRY/END*`` было решено ввести совершенно новые макросы::

    So how about using macro names that actually show the purpose, instead
    of importing all the crappy, historic, essentially randomly chosen
    debug symbol macro names from the binutils and older kernels?

.. _discussion: https://lore.kernel.org/r/20170217104757.28588-1-jslaby@suse.cz

Описание макросов
-----------------

Новые макросы снабжены префиксом ``SYM_`` и могут быть разделены на три основные
группы:

1. ``SYM_FUNC_*`` -- для аннотации C-подобных функций. Имеются в виду функции со
   стандартными соглашениями о вызовах C. Например, на x86 это означает, что стек
   содержит адрес возврата в заранее определённом месте и возврат из функции может
   произойти стандартным образом. Когда указатели кадров (frame pointers)
   включены, сохранение/восстановление указателя кадра также должно происходить
   в начале/конце функции соответственно.

   Проверяющие инструменты, такие как ``objtool``, должны гарантировать, что
   помеченные таким образом функции соответствуют этим правилам. Инструменты также
   могут легко автоматически аннотировать эти функции отладочной информацией
   (например, *данными ORC*).

2. ``SYM_CODE_*`` -- специальные функции, вызываемые со специальным стеком. Будь
   то обработчики прерываний со специальным содержимым стека, трамплины или
   функции запуска.

   Проверяющие инструменты по большей части пропускают проверку таких функций. Но
   некоторая отладочная информация всё же может быть сгенерирована автоматически.
   Для корректных отладочных данных этому коду нужны подсказки, такие как
   ``UNWIND_HINT_REGS``, предоставляемые разработчиками.

3. ``SYM_DATA*`` -- очевидно, данные, принадлежащие секциям ``.data``, а не
   ``.text``. Данные не содержат инструкций, поэтому инструменты должны
   обрабатывать их особым образом: они не должны трактовать байты как инструкции
   и не должны назначать им какую-либо отладочную информацию.

Макросы инструкций
~~~~~~~~~~~~~~~~~~~
Этот раздел охватывает ``SYM_FUNC_*`` и ``SYM_CODE_*``, перечисленные выше.

``objtool`` требует, чтобы весь код содержался в ELF-символе. Имена символов,
имеющие префикс ``.L``, не порождают записей в таблице символов. Символы с
префиксом ``.L`` можно использовать внутри области кода, но их следует избегать
для обозначения диапазона кода через аннотации ``SYM_*_START/END``.

* ``SYM_FUNC_START`` и ``SYM_FUNC_START_LOCAL`` должны быть **наиболее частыми
  пометками**. Они используются для функций со стандартными соглашениями о
  вызовах -- глобальных и локальных. Как и в C, обе они выравнивают функции на
  специфичное для архитектуры число байтов ``__ALIGN``. Существуют также варианты
  ``_NOALIGN`` для особых случаев, когда разработчики не хотят этого неявного
  выравнивания.

  Пометки ``SYM_FUNC_START_WEAK`` и ``SYM_FUNC_START_WEAK_NOALIGN`` также
  предлагаются как ассемблерный аналог атрибута *weak*, известного из C.

  Все они **должны** сопровождаться ``SYM_FUNC_END``. Во-первых, это помечает
  последовательность инструкций как функцию и вычисляет её размер для
  генерируемого объектного файла. Во-вторых, это также облегчает проверку и
  обработку таких объектных файлов, поскольку инструменты могут тривиально найти
  точные границы функции.

  Поэтому в большинстве случаев разработчикам следует писать что-то вроде
  следующего примера, имея, разумеется, какие-то ассемблерные инструкции между
  макросами::

    SYM_FUNC_START(memset)
        ... asm insns ...
    SYM_FUNC_END(memset)

  По сути, такой вид аннотации соответствует ныне устаревшим макросам ``ENTRY``
  и ``ENDPROC``.

* ``SYM_FUNC_ALIAS``, ``SYM_FUNC_ALIAS_LOCAL`` и ``SYM_FUNC_ALIAS_WEAK`` можно
  использовать для определения нескольких имён функции. Типичное использование::

    SYM_FUNC_START(__memset)
        ... asm insns ...
    SYN_FUNC_END(__memset)
    SYM_FUNC_ALIAS(memset, __memset)

  В этом примере можно вызвать ``__memset`` или ``memset`` с одинаковым
  результатом, за исключением того, что отладочная информация для инструкций
  генерируется в объектный файл только один раз -- для случая без ``ALIAS``.

* ``SYM_CODE_START`` и ``SYM_CODE_START_LOCAL`` следует использовать только в
  особых случаях -- если вы знаете, что делаете. Это используется исключительно
  для обработчиков прерываний и им подобных, где соглашение о вызовах не C-шное.
  Варианты ``_NOALIGN`` тоже существуют. Использование такое же, как для категории
  ``FUNC`` выше::

    SYM_CODE_START_LOCAL(bad_put_user)
        ... asm insns ...
    SYM_CODE_END(bad_put_user)

  Опять же, каждый ``SYM_CODE_START*`` **должен** сопровождаться ``SYM_CODE_END``.

  В некоторой степени эта категория соответствует устаревшим ``ENTRY`` и ``END``.
  За исключением того, что у ``END`` было также несколько других значений.

* ``SYM_INNER_LABEL*`` используется для обозначения метки внутри какого-либо
  ``SYM_{CODE,FUNC}_START`` и ``SYM_{CODE,FUNC}_END``. Они очень похожи на метки
  C, за исключением того, что их можно сделать глобальными. Пример использования::

    SYM_CODE_START(ftrace_caller)
        /* save_mcount_regs fills in first two parameters */
        ...

    SYM_INNER_LABEL(ftrace_caller_op_ptr, SYM_L_GLOBAL)
        /* Load the ftrace_ops into the 3rd parameter */
        ...

    SYM_INNER_LABEL(ftrace_call, SYM_L_GLOBAL)
        call ftrace_stub
        ...
        retq
    SYM_CODE_END(ftrace_caller)

Макросы данных
~~~~~~~~~~~~~~
Аналогично инструкциям, существует пара макросов для описания данных в
ассемблере.

* ``SYM_DATA_START`` и ``SYM_DATA_START_LOCAL`` помечают начало некоторых данных
  и должны использоваться в сочетании либо с ``SYM_DATA_END``, либо с
  ``SYM_DATA_END_LABEL``. Последний также добавляет метку в конце, так что люди
  могут использовать ``lstack`` и (локальный) ``lstack_end`` в следующем
  примере::

    SYM_DATA_START_LOCAL(lstack)
        .skip 4096
    SYM_DATA_END_LABEL(lstack, SYM_L_LOCAL, lstack_end)

* ``SYM_DATA`` и ``SYM_DATA_LOCAL`` -- это варианты для простых, в основном
  однострочных данных::

    SYM_DATA(HEAP,     .long rm_heap)
    SYM_DATA(heap_end, .long rm_stack)

  В конечном счёте они внутренне разворачиваются в ``SYM_DATA_START`` с
  ``SYM_DATA_END``.

Вспомогательные макросы
~~~~~~~~~~~~~~~~~~~~~~~~
Все вышеперечисленное в итоге сводится к некоторому вызову ``SYM_START``,
``SYM_END`` или ``SYM_ENTRY``. Обычно разработчикам следует избегать их
использования.

Кроме того, в приведённых выше примерах можно было видеть ``SYM_L_LOCAL``.
Существуют также ``SYM_L_GLOBAL`` и ``SYM_L_WEAK``. Все они предназначены для
обозначения компоновки (linkage) символа, помеченного ими. Они используются либо
в вариантах ``_LABEL`` ранее упомянутых макросов, либо в ``SYM_START``.


Переопределение макросов
~~~~~~~~~~~~~~~~~~~~~~~~~
Архитектура также может переопределить любой из макросов в своём собственном
``asm/linkage.h``, включая макросы, задающие тип символа (``SYM_T_FUNC``,
``SYM_T_OBJECT`` и ``SYM_T_NONE``). Поскольку каждый макрос, описанный в этом
файле, окружён ``#ifdef`` + ``#endif``, достаточно определить эти макросы
по-другому в упомянутом архитектурно-зависимом заголовочном файле.
