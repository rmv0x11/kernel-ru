Обработка входа/выхода для исключений, прерываний, системных вызовов и KVM
==========================================================================

Все переходы между доменами выполнения требуют обновления состояния, которое
подчиняется строгим ограничениям на порядок. Обновление состояния требуется
для следующего:

  * Lockdep
  * RCU / отслеживание контекста (context tracking)
  * Счётчик вытеснения (preemption counter)
  * Трассировка
  * Учёт времени

Порядок обновления зависит от типа перехода и поясняется ниже в разделах по
типам переходов: `Системные вызовы`_, `KVM`_, `Прерывания и обычные
исключения`_, `NMI и NMI-подобные исключения`_.

Неинструментируемый код — noinstr
---------------------------------

Большинство средств инструментирования зависят от RCU, поэтому инструментирование
запрещено для кода входа до того, как RCU начинает наблюдение, и для кода выхода
после того, как RCU прекращает наблюдение. Кроме того, многие архитектуры обязаны
сохранять и восстанавливать состояние регистров, а это означает, что (например)
точка останова в коде входа точки останова перезаписала бы отладочные регистры
исходной точки останова.

Такой код должен быть помечен атрибутом 'noinstr', что помещает этот код в
специальную секцию, недоступную для средств инструментирования и отладки.
Некоторые функции инструментируемы частично; это обрабатывается так: их помечают
noinstr и используют instrumentation_begin() и instrumentation_end() для
обозначения инструментируемых участков кода:

.. code-block:: c

  noinstr void entry(void)
  {
  	handle_entry();     // <-- must be 'noinstr' or '__always_inline'
	...

	instrumentation_begin();
	handle_context();   // <-- instrumentable code
	instrumentation_end();

	...
	handle_exit();      // <-- must be 'noinstr' or '__always_inline'
  }

Это позволяет проверять ограничения 'noinstr' с помощью objtool на поддерживаемых
архитектурах.

Вызов неинструментируемых функций из инструментируемого контекста не имеет
ограничений и полезен для защиты, например, переключения состояния, которое
привело бы к сбою при инструментировании.

Все секции неинструментируемого кода входа/выхода до и после переходов состояния
RCU должны выполняться с отключёнными прерываниями.

Системные вызовы
----------------

Код входа системного вызова начинается с ассемблерного кода и вызывает
низкоуровневый C-код после установки низкоуровневого специфичного для архитектуры
состояния и кадров стека. Этот низкоуровневый C-код не должен инструментироваться.
Типичная функция обработки системного вызова, вызываемая из низкоуровневого
ассемблерного кода, выглядит так:

.. code-block:: c

  noinstr void syscall(struct pt_regs *regs, int nr)
  {
	arch_syscall_enter(regs);
	nr = syscall_enter_from_user_mode(regs, nr);

	instrumentation_begin();
	if (!invoke_syscall(regs, nr) && nr != -1)
	 	result_reg(regs) = __sys_ni_syscall(regs);
	instrumentation_end();

	syscall_exit_to_user_mode(regs);
  }

syscall_enter_from_user_mode() сначала вызывает enter_from_user_mode(), которая
устанавливает состояние в следующем порядке:

  * Lockdep
  * RCU / отслеживание контекста (context tracking)
  * Трассировка

и затем вызывает различные функции работы входа, такие как ptrace, seccomp, audit,
трассировка системных вызовов и т. д. После завершения всего этого может быть
вызвана инструментируемая функция invoke_syscall. Затем инструментируемая секция
кода завершается, после чего вызывается syscall_exit_to_user_mode().

syscall_exit_to_user_mode() обрабатывает всю работу, которую необходимо выполнить
перед возвратом в пространство пользователя, такую как трассировка, audit, сигналы,
работа задачи (task work) и т. д. После этого она вызывает exit_to_user_mode(),
которая снова обрабатывает переход состояния в обратном порядке:

  * Трассировка
  * RCU / отслеживание контекста (context tracking)
  * Lockdep

syscall_enter_from_user_mode() и syscall_exit_to_user_mode() также доступны как
мелкозернистые подфункции в случаях, когда код архитектуры вынужден выполнять
дополнительную работу между различными шагами. В таких случаях он обязан
гарантировать, что enter_from_user_mode() вызывается первой на входе, а
exit_to_user_mode() вызывается последней на выходе.

Не вкладывайте системные вызовы. Вложенные системные вызовы заставят RCU и/или
отслеживание контекста вывести предупреждение.

KVM
---

Вход в гостевой режим или выход из него очень похожи на системные вызовы. С точки
зрения хост-ядра CPU уходит в пространство пользователя при входе в гостя и
возвращается в ядро при выходе.

guest_state_enter_irqoff() — это специфичный для KVM вариант exit_to_user_mode(),
а guest_state_exit_irqoff() — это вариант enter_from_user_mode() для KVM. Операции
с состоянием имеют тот же порядок.

Обработка работы задачи (task work) для гостя выполняется отдельно на границе
цикла vcpu_run() через xfer_to_guest_mode_handle_work(), которая является
подмножеством работы, обрабатываемой при возврате в пространство пользователя.

Не вкладывайте переходы входа/выхода KVM, поскольку это бессмысленно.

Прерывания и обычные исключения
-------------------------------

Обработка входа и выхода прерываний несколько сложнее, чем переходы системных
вызовов и KVM.

Если прерывание возникает, пока CPU исполняется в пространстве пользователя,
обработка входа и выхода точно такая же, как для системных вызовов.

Если прерывание возникает, пока CPU исполняется в пространстве ядра, обработка
входа и выхода немного отличается. Состояние RCU обновляется только тогда, когда
прерывание возникает в контексте задачи простоя (idle task) данного CPU. В
противном случае RCU уже будет вести наблюдение. Lockdep и трассировка должны
обновляться безусловно.

irqentry_enter() и irqentry_exit() предоставляют реализацию для этого.

Специфичная для архитектуры часть выглядит похоже на обработку системного вызова:

.. code-block:: c

  noinstr void interrupt(struct pt_regs *regs, int nr)
  {
	arch_interrupt_enter(regs);
	state = irqentry_enter(regs);

	instrumentation_begin();

	irq_enter_rcu();
	invoke_irq_handler(regs, nr);
	irq_exit_rcu();

	instrumentation_end();

	irqentry_exit(regs, state);
  }

Обратите внимание, что вызов фактического обработчика прерывания находится внутри
пары irq_enter_rcu() и irq_exit_rcu().

irq_enter_rcu() обновляет счётчик вытеснения, благодаря чему in_hardirq()
возвращает true, обрабатывает состояние тика NOHZ и учёт времени прерываний. Это
означает, что вплоть до точки, где вызывается irq_enter_rcu(), in_hardirq()
возвращает false.

irq_exit_rcu() обрабатывает учёт времени прерываний, отменяет обновление счётчика
вытеснения и в конечном итоге обрабатывает программные прерывания (soft
interrupts) и состояние тика NOHZ.

Теоретически счётчик вытеснения мог бы обновляться в irqentry_enter(). На
практике откладывание этого обновления до irq_enter_rcu() позволяет
трассировать код счётчика вытеснения, одновременно сохраняя симметрию с
irq_exit_rcu() и irqentry_exit(), которые описаны в следующем абзаце.
Единственный недостаток в том, что ранний код входа вплоть до irq_enter_rcu()
должен учитывать, что счётчик вытеснения ещё не обновлён состоянием
HARDIRQ_OFFSET.

Обратите внимание, что irq_exit_rcu() должна удалить HARDIRQ_OFFSET из счётчика
вытеснения до того, как она обработает программные прерывания, обработчики которых
должны выполняться в контексте BH, а не в контексте с отключёнными прерываниями.
Кроме того, irqentry_exit() может выполнить переключение планировщика (schedule),
что также требует, чтобы HARDIRQ_OFFSET был удалён из счётчика вытеснения.

Хотя ожидается, что обработчики прерываний выполняются с отключёнными локальными
прерываниями, вложение прерываний — обычное дело с точки зрения входа/выхода.
Например, обработка softirq происходит внутри блока irqentry_{enter,exit}() с
включёнными локальными прерываниями. Кроме того, хотя это и редкость, ничто не
мешает обработчику прерывания снова включить прерывания.

Коду входа/выхода прерывания, строго говоря, не нужно обрабатывать повторную
входимость (reentrancy), поскольку он выполняется с отключёнными локальными
прерываниями. Но NMI могут произойти в любой момент, а значительная часть кода
входа является общей для этих двух случаев.

NMI и NMI-подобные исключения
-----------------------------

NMI и NMI-подобные исключения (machine checks, двойные сбои, отладочные
прерывания и т. д.) могут возникнуть в любом контексте и обязаны быть особенно
аккуратны с состоянием.

Изменения состояния для отладочных исключений и исключений machine-check зависят
от того, произошли ли эти исключения в пространстве пользователя (точки останова
или точки наблюдения) или в режиме ядра (патчинг кода). Из пространства
пользователя они трактуются как прерывания, а из режима ядра — как NMI.

NMI и другие NMI-подобные исключения обрабатывают переходы состояния, не
различая происхождение из режима пользователя и режима ядра.

Обновление состояния на входе обрабатывается в irqentry_nmi_enter(), которая
обновляет состояние в следующем порядке:

  * Счётчик вытеснения (preemption counter)
  * Lockdep
  * RCU / отслеживание контекста (context tracking)
  * Трассировка

Парная функция выхода irqentry_nmi_exit() выполняет обратную операцию в обратном
порядке.

Обратите внимание, что обновление счётчика вытеснения должно быть первой
операцией на входе и последней операцией на выходе. Причина в том, что и lockdep,
и RCU полагаются на то, что in_nmi() возвращает true в этом случае. Изменение
счётчика вытеснения в случае входа/выхода NMI не должно трассироваться.

Специфичный для архитектуры код выглядит так:

.. code-block:: c

  noinstr void nmi(struct pt_regs *regs)
  {
	arch_nmi_enter(regs);
	state = irqentry_nmi_enter(regs);

	instrumentation_begin();
	nmi_handler(regs);
	instrumentation_end();

	irqentry_nmi_exit(regs);
  }

а для, например, отладочного исключения он может выглядеть так:

.. code-block:: c

  noinstr void debug(struct pt_regs *regs)
  {
	arch_nmi_enter(regs);

	debug_regs = save_debug_regs();

	if (user_mode(regs)) {
		state = irqentry_enter(regs);

		instrumentation_begin();
		user_mode_debug_handler(regs, debug_regs);
		instrumentation_end();

		irqentry_exit(regs, state);
  	} else {
  		state = irqentry_nmi_enter(regs);

		instrumentation_begin();
		kernel_mode_debug_handler(regs, debug_regs);
		instrumentation_end();

		irqentry_nmi_exit(regs, state);
	}
  }

Не существует объединённой функции irqentry_nmi_if_kernel(), поскольку
вышеприведённое нельзя обработать способом, не зависящим от типа исключения.

NMI могут произойти в любом контексте. Например, NMI-подобное исключение,
сработавшее во время обработки NMI. Поэтому код входа NMI должен быть повторно
входимым (reentrant), а обновления состояния должны обрабатывать вложенность.
