======================================
Обслуживание (Housekeeping)
======================================


Изоляция CPU (CPU Isolation) уводит работу ядра, которая иначе могла бы
выполняться на любом CPU. Цель связанных с ней возможностей — снизить джиттер
ОС (OS jitter), который не способны выдержать некоторые экстремальные нагрузки,
например в ряде сценариев DPDK.

Работа ядра, уводимая изоляцией CPU, обычно описывается термином «обслуживание»
(housekeeping), поскольку она включает базовую работу по очистке, ведению
статистики и связанным с ней действиям, освобождению памяти, разнообразным
отложенным операциям и т. д.

Иногда обслуживание — это просто несвязанная (unbound) работа (несвязанные
очереди работ, несвязанные таймеры, ...), которая легко назначается на
неизолированные CPU. Но иногда обслуживание привязано к конкретному CPU и требует
изощрённых приёмов, чтобы быть выгруженным на неизолированные CPU (RCU_NOCB,
удалённый тик планировщика и т. д.).

Таким образом, обслуживающий CPU можно рассматривать как противоположность
изолированному CPU. Это просто CPU, который может выполнять работу по
обслуживанию. В любой момент времени всегда должен существовать как минимум один
работающий (online) обслуживающий CPU. CPU, которые не изолированы,
автоматически назначаются как обслуживающие.

В настоящее время обслуживание разделено на четыре возможности, описываемые
``enum hk_type type``:

1.	HK_TYPE_DOMAIN соответствует работе, уводимой изоляцией доменов
	планировщика, выполняемой через загрузочный параметр ``isolcpus=domain``
	или изолированные разделы cpuset в cgroup v2. Сюда входят балансировка
	нагрузки планировщика, несвязанные очереди работ и таймеры.

2.	HK_TYPE_KERNEL_NOISE соответствует работе, уводимой изоляцией тика,
	выполняемой через загрузочные параметры ``nohz_full=`` или
	``isolcpus=nohz``. Сюда входят удалённый тик планировщика, vmstat и
	сторожевой таймер блокировок (lockup watchdog).

3.	HK_TYPE_MANAGED_IRQ соответствует обработчикам IRQ, уводимым изоляцией
	управляемых IRQ, выполняемой через ``isolcpus=managed_irq``.

4.	HK_TYPE_DOMAIN_BOOT соответствует работе, уводимой изоляцией доменов
	планировщика, выполняемой только через ``isolcpus=domain``. Она аналогична
	HK_TYPE_DOMAIN, за исключением того, что игнорирует изоляцию, выполняемую
	cpuset.


Маски CPU обслуживания
=================================

Маски CPU обслуживания включают CPU, которые могут выполнять работу, уводимую
соответствующей возможностью изоляции. Эти маски CPU возвращаются следующей
функцией::

	const struct cpumask *housekeeping_cpumask(enum hk_type type)

По умолчанию, если не используются ни ``nohz_full=``, ни ``isolcpus``, ни
изолированные разделы cpuset, что покрывает большинство сценариев, эта функция
возвращает cpu_possible_mask.

В противном случае функция возвращает маску CPU, дополнительную к возможности
изоляции. Например:

При isolcpus=domain,7 следующее вернёт маску со всеми возможными CPU, кроме 7::

	housekeeping_cpumask(HK_TYPE_DOMAIN)

Аналогично, при nohz_full=5,6 следующее вернёт маску со всеми возможными CPU,
кроме 5,6::

	housekeeping_cpumask(HK_TYPE_KERNEL_NOISE)


Синхронизация с cpuset
=================================

Cpuset может изменять маску CPU обслуживания HK_TYPE_DOMAIN при создании,
изменении или удалении изолированного раздела.

Поэтому пользователи маски CPU HK_TYPE_DOMAIN должны обеспечить правильную
синхронизацию с cpuset, чтобы гарантировать следующее:

1.	Снимок маски CPU остаётся согласованным.

2.	На вновь сделанном изолированным CPU не ставится в очередь никакая работа
	по обслуживанию.

3.	Ожидающая работа по обслуживанию, которая была поставлена в очередь на
	неизолированный CPU, только что ставший изолированным через cpuset, должна
	быть сброшена (flushed) до того, как соответствующий созданный/изменённый
	изолированный раздел станет доступен пространству пользователя (userspace).

Эта синхронизация поддерживается схемой на основе RCU. Сторона обновления cpuset
ожидает истечения льготного периода (grace period) RCU после обновления маски
CPU HK_TYPE_DOMAIN и перед сбросом ожидающих работ. На стороне чтения необходимо
позаботиться о том, чтобы выбор цели обслуживания и постановка работы в очередь
выполнялись в пределах одной и той же критической секции чтения RCU.

Типичный пример структуры выглядел бы так на стороне обновления
(``housekeeping_update()``)::

	rcu_assign_pointer(housekeeping_cpumasks[type], trial);
	synchronize_rcu();
	flush_workqueue(example_workqueue);

А затем на стороне чтения::

	rcu_read_lock();
	cpu = housekeeping_any_cpu(HK_TYPE_DOMAIN);
	queue_work_on(cpu, example_workqueue, work);
	rcu_read_unlock();
