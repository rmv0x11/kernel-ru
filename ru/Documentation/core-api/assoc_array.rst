===============================================
Универсальная реализация ассоциативного массива
===============================================

Обзор
=====

Данная реализация ассоциативного массива представляет собой контейнер объектов
со следующими свойствами:

1. Объекты являются непрозрачными указателями.  Реализации безразлично, куда они
   указывают (если вообще куда-то) и на что они указывают (если вообще на что-то).

   .. note::

      Указатели на объекты _обязаны_ иметь нулевой младший значащий бит.

2. Объектам не требуется содержать блоки связности для использования массивом.  Это
   позволяет объекту одновременно находиться в нескольких массивах.
   Вместо этого массив состоит из блоков метаданных, указывающих на объекты.

3. Для нахождения объектов в массиве им требуются индексные ключи.

4. Индексные ключи должны быть уникальными.  Вставка объекта с тем же ключом, что и у
   уже находящегося в массиве, приведёт к замене старого объекта.

5. Индексные ключи могут иметь любую длину и могут быть разной длины.

6. Индексные ключи должны кодировать длину как можно раньше, прежде чем проявится
   любая обусловленная длиной вариативность.

7. Индексные ключи могут включать хеш, чтобы рассеивать объекты по всему массиву.

8. По массиву можно итерироваться.  Объекты не обязательно будут выходить в
   порядке ключей.

9. По массиву можно итерироваться во время его изменения, при условии, что
   итератор удерживает RCU readlock.  Однако заметьте, что при таких
   обстоятельствах некоторые объекты могут встречаться более одного раза.  Если это
   проблема, итератору следует устанавливать блокировку против изменений.  Однако объекты
   не будут пропущены, если только они не удалены.

10. Объекты в массиве можно искать по их индексному ключу.

11. Объекты можно искать во время изменения массива, при условии, что
    RCU readlock удерживается потоком, выполняющим поиск.

Реализация внутренне использует дерево из узлов по 16 указателей, индексируемых
на каждом уровне ниблами из индексного ключа тем же способом, что и в radix-дереве
(radix tree).  Для повышения эффективности использования памяти могут размещаться
сокращения (shortcuts), позволяющие пропустить то, что иначе было бы серией
узлов с единственным занятым слотом.  Кроме того, узлы упаковывают листовые
указатели на объекты в свободное место узла, не создавая дополнительного ветвления
до тех пор, пока объект не потребуется добавить в полный узел.


Публичный API
=============

Публичный API можно найти в ``<linux/assoc_array.h>``.  Ассоциативный
массив имеет корень в следующей структуре::

    struct assoc_array {
            ...
    };

Код выбирается включением ``CONFIG_ASSOCIATIVE_ARRAY`` с помощью::

    ./script/config -e ASSOCIATIVE_ARRAY


Сценарий редактирования
-----------------------

Функции вставки и удаления создают «сценарий редактирования» (edit script), который
впоследствии можно применить, чтобы внести изменения без риска получить ``ENOMEM``. В нём
сохраняются предварительно выделенные блоки метаданных, которые будут установлены во
внутреннем дереве, и отслеживаются блоки метаданных, которые будут удалены из дерева при
применении сценария.

Он также используется для отслеживания мёртвых блоков и мёртвых объектов после
применения сценария, чтобы их можно было освободить позднее.  Освобождение выполняется
после прохождения RCU grace period — что позволяет функциям доступа
продолжать работу под RCU read lock.

Снаружи API сценарий выглядит как указатель типа::

    struct assoc_array_edit;

Есть две функции для работы со сценарием:

1. Применить сценарий редактирования::

    void assoc_array_apply_edit(struct assoc_array_edit *edit);

   Она выполнит функции редактирования, вставляя различные барьеры записи,
   чтобы позволить продолжаться доступам под RCU read lock.  Затем сценарий редактирования
   будет передан в ``call_rcu()`` для освобождения его самого и всего мёртвого, на что он
   указывает.

2. Отменить сценарий редактирования::

    void assoc_array_cancel_edit(struct assoc_array_edit *edit);

   Это немедленно освобождает сценарий редактирования и всю предварительно выделенную память. Если
   это была вставка, новый объект *не* освобождается этой функцией,
   а должен быть освобождён вызывающей стороной.

Эти функции гарантированно не завершаются с ошибкой.


Таблица операций
----------------

Различные функции принимают таблицу операций::

    struct assoc_array_ops {
            ...
    };

Она указывает на ряд методов, все из которых должны быть предоставлены:

1. Получить фрагмент индексного ключа из данных вызывающей стороны::

    unsigned long (*get_key_chunk)(const void *index_key, int level);

   Эта функция должна возвращать фрагмент предоставленного вызывающей стороной индексного ключа,
   начиная с *битовой* позиции, заданной аргументом level.  Аргумент level будет кратен
   ``ASSOC_ARRAY_KEY_CHUNK_SIZE``, и функция должна возвращать
   ``ASSOC_ARRAY_KEY_CHUNK_SIZE bits``.  Ошибка невозможна.


2. Получить фрагмент индексного ключа объекта::

    unsigned long (*get_object_key_chunk)(const void *object, int level);

   Как и предыдущая функция, но получает свои данные из объекта в массиве,
   а не из предоставленного вызывающей стороной индексного ключа.


3. Проверить, является ли это искомым объектом::

    bool (*compare_object)(const void *object, const void *index_key);

   Сравнивает объект с индексным ключом и возвращает ``true``, если он совпадает,
   и ``false``, если нет.


4. Вычислить различие индексных ключей двух объектов::

    int (*diff_objects)(const void *object, const void *index_key);

   Возвращает битовую позицию, в которой индексный ключ указанного объекта
   отличается от заданного индексного ключа, или -1, если они одинаковы.


5. Освободить объект::

    void (*free_object)(void *object);

   Освобождает указанный объект.  Заметьте, что эта функция может быть вызвана через RCU grace period
   после вызова ``assoc_array_apply_edit()``, поэтому ``synchronize_rcu()`` может
   потребоваться при выгрузке модуля.


Функции манипулирования
-----------------------

Существует ряд функций для манипулирования ассоциативным массивом:

1. Инициализировать ассоциативный массив::

    void assoc_array_init(struct assoc_array *array);

   Это инициализирует базовую структуру для ассоциативного массива.  Не может завершиться с ошибкой.


2. Вставить/заменить объект в ассоциативном массиве::

    struct assoc_array_edit *
    assoc_array_insert(struct assoc_array *array,
                       const struct assoc_array_ops *ops,
                       const void *index_key,
                       void *object);

   Это вставляет заданный объект в массив.  Заметьте, что младший
   значащий бит указателя должен быть нулевым, так как он используется внутренне для
   маркировки типа указателей.

   Если для этого ключа уже существует объект, то он будет заменён
   новым объектом, а старый будет автоматически освобождён.

   Аргумент ``index_key`` должен содержать информацию об индексном ключе и
   передаётся методам в таблице ops при их вызове.

   Эта функция не вносит никаких изменений в сам массив, а вместо этого возвращает
   сценарий редактирования, который должен быть применён.  ``-ENOMEM`` возвращается в случае
   ошибки нехватки памяти.

   Вызывающая сторона должна устанавливать эксклюзивную блокировку против других модификаторов массива.


3. Удалить объект из ассоциативного массива::

    struct assoc_array_edit *
    assoc_array_delete(struct assoc_array *array,
                       const struct assoc_array_ops *ops,
                       const void *index_key);

   Это удаляет из массива объект, который соответствует указанным данным.

   Аргумент ``index_key`` должен содержать информацию об индексном ключе и
   передаётся методам в таблице ops при их вызове.

   Эта функция не вносит никаких изменений в сам массив, а вместо этого возвращает
   сценарий редактирования, который должен быть применён.  ``-ENOMEM`` возвращается в случае
   ошибки нехватки памяти.  ``NULL`` будет возвращено, если указанный объект
   не найден в массиве.

Вызывающая сторона должна устанавливать эксклюзивную блокировку против других модификаторов массива.


4. Удалить все объекты из ассоциативного массива::

    struct assoc_array_edit *
    assoc_array_clear(struct assoc_array *array,
                      const struct assoc_array_ops *ops);

   Это удаляет все объекты из ассоциативного массива и оставляет его
   полностью пустым.

   Эта функция не вносит никаких изменений в сам массив, а вместо этого возвращает
   сценарий редактирования, который должен быть применён.  ``-ENOMEM`` возвращается в случае
   ошибки нехватки памяти.

   Вызывающая сторона должна устанавливать эксклюзивную блокировку против других модификаторов массива.


5. Уничтожить ассоциативный массив, удалив все объекты::

    void assoc_array_destroy(struct assoc_array *array,
                             const struct assoc_array_ops *ops);

   Это уничтожает содержимое ассоциативного массива и оставляет его
   полностью пустым.  Недопустимо, чтобы другой поток обходил
   массив под RCU read lock одновременно с тем, как эта функция
   его уничтожает, так как при освобождении памяти не выполняется отложенного RCU —
   а это потребовало бы выделения памяти.

   Вызывающая сторона должна устанавливать эксклюзивную блокировку против других модификаторов и сторон, обращающихся
   к массиву.


6. Выполнить сборку мусора ассоциативного массива::

    int assoc_array_gc(struct assoc_array *array,
                       const struct assoc_array_ops *ops,
                       bool (*iterator)(void *object, void *iterator_data),
                       void *iterator_data);

   Это итерируется по объектам в ассоциативном массиве и передаёт каждый из них
   в ``iterator()``.  Если ``iterator()`` возвращает ``true``, объект сохраняется.
   Если он возвращает ``false``, объект будет освобождён.  Если функция ``iterator()``
   возвращает ``true``, она должна перед возвратом выполнить любое подходящее
   увеличение счётчика ссылок объекта.

   Внутреннее дерево будет упаковано, если это возможно, в рамках итерации,
   чтобы сократить число узлов в нём.

   ``iterator_data`` передаётся напрямую в ``iterator()`` и в остальном
   игнорируется функцией.

   Функция вернёт ``0`` в случае успеха и ``-ENOMEM``, если памяти было
   недостаточно.

   Возможно, чтобы другие потоки итерировались по массиву или искали в нём под
   RCU read lock, пока эта функция выполняется.  Вызывающая сторона должна
   устанавливать эксклюзивную блокировку против других модификаторов массива.


Функции доступа
---------------

Существует две функции для доступа к ассоциативному массиву:

1. Итерироваться по всем объектам в ассоциативном массиве::

    int assoc_array_iterate(const struct assoc_array *array,
                            int (*iterator)(const void *object,
                                            void *iterator_data),
                            void *iterator_data);

   Это передаёт каждый объект массива в функцию обратного вызова iterator.
   ``iterator_data`` — это приватные данные для этой функции.

   Это можно использовать на массиве одновременно с тем, как массив
   изменяется, при условии, что удерживается RCU read lock.  При таких обстоятельствах
   возможно, что функция итерации увидит некоторые объекты дважды.  Если
   это проблема, то изменения следует блокировать.  Однако алгоритм
   итерации не должен пропускать никакие объекты.

   Функция вернёт ``0``, если в массиве не было объектов, иначе она
   вернёт результат последней вызванной функции итерации.  Итерация
   немедленно прекращается, если любой вызов функции итерации приводит к
   ненулевому возврату.


2. Найти объект в ассоциативном массиве::

    void *assoc_array_find(const struct assoc_array *array,
                           const struct assoc_array_ops *ops,
                           const void *index_key);

   Это проходит через внутреннее дерево массива напрямую к объекту,
   указанному индексным ключом.

   Это можно использовать на массиве одновременно с тем, как массив
   изменяется, при условии, что удерживается RCU read lock.

   Функция вернёт объект, если он найден (и установит ``*_type`` в тип
   объекта), или вернёт ``NULL``, если объект не был найден.


Форма индексного ключа
----------------------

Индексный ключ может иметь любую форму, но поскольку алгоритмам не сообщается, какова
длина ключа, настоятельно рекомендуется, чтобы индексный ключ включал свою длину
очень рано, прежде чем любая обусловленная длиной вариативность повлияет на
сравнения.

Это приведёт к тому, что листья с ключами разной длины будут рассеиваться друг
от друга — а листья с ключами одинаковой длины будут собираться вместе.

Также рекомендуется, чтобы индексный ключ начинался с хеша остальной части
ключа для максимального рассеивания по всему пространству ключей.

Чем лучше рассеивание, тем шире и ниже будет внутреннее дерево.

Плохое рассеивание не является слишком большой проблемой, так как существуют сокращения и узлы
могут содержать смеси листьев и указателей на метаданные.

Индексный ключ читается фрагментами по машинному слову.  Каждый фрагмент разделяется на
один нибл (4 бита) на уровень, так что на 32-битном CPU этого хватает на 8 уровней, а
на 64-битном CPU — на 16 уровней.  Если рассеивание не совсем плохое, маловероятно,
что потребуется использовать более одного слова любого конкретного индексного ключа.


Внутреннее устройство
=====================

Структура данных ассоциативного массива имеет внутреннее дерево.  Это дерево
строится из двух типов блоков метаданных: узлов (nodes) и сокращений (shortcuts).

Узел — это массив слотов.  Каждый слот может содержать одну из четырёх вещей:

* Указатель NULL, означающий, что слот пуст.
* Указатель на объект (лист).
* Указатель на узел следующего уровня.
* Указатель на сокращение.


Базовая компоновка внутреннего дерева
-------------------------------------

Не учитывая сокращения на данный момент, узлы образуют многоуровневое дерево.  Пространство
индексных ключей строго разделяется узлами дерева, и узлы располагаются на
фиксированных уровнях.  Например::

 Level: 0               1               2               3
        =============== =============== =============== ===============
                                                        NODE D
                        NODE B          NODE C  +------>+---+
                +------>+---+   +------>+---+   |       | 0 |
        NODE A  |       | 0 |   |       | 0 |   |       +---+
        +---+   |       +---+   |       +---+   |       :   :
        | 0 |   |       :   :   |       :   :   |       +---+
        +---+   |       +---+   |       +---+   |       | f |
        | 1 |---+       | 3 |---+       | 7 |---+       +---+
        +---+           +---+           +---+
        :   :           :   :           | 8 |---+
        +---+           +---+           +---+   |       NODE E
        | e |---+       | f |           :   :   +------>+---+
        +---+   |       +---+           +---+           | 0 |
        | f |   |                       | f |           +---+
        +---+   |                       +---+           :   :
                |       NODE F                          +---+
                +------>+---+                           | f |
                        | 0 |           NODE G          +---+
                        +---+   +------>+---+
                        :   :   |       | 0 |
                        +---+   |       +---+
                        | 6 |---+       :   :
                        +---+           +---+
                        :   :           | f |
                        +---+           +---+
                        | f |
                        +---+

В приведённом выше примере имеется 7 узлов (A-G), каждый с 16 слотами (0-f).
Предполагая отсутствие в дереве других узлов метаданных, пространство ключей делится
так:

    ===========     ====
    KEY PREFIX      NODE
    ===========     ====
    137*            D
    138*            E
    13[0-69-f]*     C
    1[0-24-f]*      B
    e6*             G
    e[0-57-f]*      F
    [02-df]*        A
    ===========     ====

Так, например, ключи со следующими примерными индексными ключами будут найдены в
соответствующих узлах:

    =============== ======= ====
    INDEX KEY       PREFIX  NODE
    =============== ======= ====
    13694892892489  13      C
    13795289025897  137     D
    13889dde88793   138     E
    138bbb89003093  138     E
    1394879524789   12      C
    1458952489      1       B
    9431809de993ba  \-      A
    b4542910809cd   \-      A
    e5284310def98   e       F
    e68428974237    e6      G
    e7fffcbd443     e       F
    f3842239082     \-      A
    =============== ======= ====

Для экономии памяти, если узел способен вместить все листья в своей части
пространства ключей, то узел будет содержать все эти листья и не будет иметь никаких
указателей на метаданные — даже если некоторые из этих листьев хотели бы находиться в одном слоте.

Узел может содержать неоднородную смесь листьев и указателей на метаданные.
Указатели на метаданные должны находиться в слотах, соответствующих их подразделениям
пространства ключей.  Листья могут находиться в любом слоте, не занятом указателем на метаданные.
Гарантируется, что ни один из листьев в узле не будет соответствовать слоту, занятому
указателем на метаданные.  Если указатель на метаданные присутствует, любой лист, чей ключ соответствует
префиксу ключа метаданных, должен находиться в поддереве, на которое указывает этот указатель на метаданные.

В приведённом выше списке индексных ключей узел A будет содержать:

    ====    =============== ==================
    SLOT    CONTENT         INDEX KEY (PREFIX)
    ====    =============== ==================
    1       PTR TO NODE B   1*
    any     LEAF            9431809de993ba
    any     LEAF            b4542910809cd
    e       PTR TO NODE F   e*
    any     LEAF            f3842239082
    ====    =============== ==================

Сокращения
----------

Сокращения — это записи метаданных, перепрыгивающие через участок пространства ключей.  Сокращение
является заменой серии узлов с единственным занятым слотом, восходящей через
уровни.  Сокращения существуют для экономии памяти и ускорения обхода.

Возможно, чтобы корнем дерева было сокращение — скажем, например,
дерево содержит как минимум 17 узлов, все с префиксом ключа ``1111``.  Алгоритм
вставки вставит сокращение, чтобы одним прыжком пропустить пространство ключей ``1111``
и добраться до четвёртого уровня, где они действительно становятся различными.


Расщепление и схлопывание узлов
-------------------------------

Каждый узел имеет максимальную ёмкость в 16 листьев и указателей на метаданные.  Если
алгоритм вставки обнаруживает, что пытается вставить в узел 17-й объект,
этот узел будет расщеплён так, что как минимум два листа, имеющие общий
сегмент ключа на этом уровне, окажутся в отдельном узле, имеющем корень в этом слоте для
этого общего сегмента ключа.

Если листья в полном узле и вставляемый лист
достаточно похожи, то в дерево будет вставлено сокращение.

Когда число объектов в поддереве, имеющем корень в узле, падает до 16 или
менее, то поддерево будет схлопнуто до одного узла — и это будет
распространяться к корню, если это возможно.


Нерекурсивная итерация
----------------------

Каждый узел и сокращение содержат обратный указатель на своего родителя и номер
слота в этом родителе, который указывает на него.  Нерекурсивная итерация использует их, чтобы
продвигаться по дереву к корню, переходя к родительскому узлу, слоту N + 1, чтобы
гарантировать продвижение без необходимости в стеке.

Однако обратные указатели делают одновременное изменение и итерацию непростыми.


Одновременное изменение и итерация
----------------------------------

Необходимо рассмотреть ряд случаев:

1. Простая вставка/замена.  Это связано с простой заменой указателя на NULL или старого
   соответствующего листа указателем на новый лист после барьера.
   В остальном блоки метаданных не меняются.  Старый лист не будет освобождён
   до прохождения RCU grace period.

2. Простое удаление.  Это связано лишь с очисткой старого соответствующего листа.  В остальном
   блоки метаданных не меняются.  Старый лист не будет освобождён до
   прохождения RCU grace period.

3. Вставка, заменяющая часть поддерева, в которое мы ещё не вошли.  Это
   может включать замену части этого поддерева — но это не повлияет
   на итерацию, так как мы ещё не достигли указателя на него, а
   блоки предков не заменяются (их компоновка не меняется).

4. Вставка, заменяющая узлы, которые мы активно обрабатываем.  Это не
   проблема, так как мы прошли заякоривающий указатель и не переключимся на
   новую компоновку, пока не последуем по обратным указателям — а к этому моменту мы
   уже изучили листья в заменённом узле (мы итерируемся по всем
   листьям в узле, прежде чем последовать по любому из его указателей на метаданные).

   Однако мы можем повторно увидеть некоторые листья, выделенные в новую
   ветвь, которая находится в слоте дальше того, на котором мы были.

5. Вставка, заменяющая узлы, зависимую ветвь которых мы обрабатываем.
   Это не повлияет на нас, пока мы не последуем по обратным указателям.  Аналогично (4).

6. Удаление, схлопывающее ветвь под нами.  Это не влияет на нас, потому что
   обратные указатели вернут нас к родителю нового узла прежде, чем мы
   смогли бы увидеть новый узел.  Всё схлопнутое поддерево выбрасывается
   неизменным — и по-прежнему будет иметь корень в том же слоте, поэтому мы не должны
   обработать его второй раз, так как вернёмся к слоту + 1.

.. note::

   При некоторых обстоятельствах нам нужно одновременно изменить указатель на
   родителя и указатель родительского слота на узле (скажем, например, мы
   вставили перед ним другой узел и подняли его на уровень выше).  Мы не можем
   сделать это без блокировки против чтения — поэтому нам приходится заменять и этот узел.

   Однако, когда мы превращаем сокращение в узел, это не проблема,
   так как сокращения имеют только один слот, и поэтому номер родительского слота не используется
   при обходе назад по нему.  Это означает, что допустимо сначала изменить
   номер слота — при условии, что используются подходящие барьеры, чтобы гарантировать,
   что номер родительского слота читается после обратного указателя.

Устаревшие блоки и листья освобождаются после прохождения RCU grace period,
поэтому до тех пор, пока любой, выполняющий проход или итерацию, удерживает RCU read lock,
старая надстройка не должна пропадать у них из-под рук.
