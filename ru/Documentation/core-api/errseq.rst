=========================
Тип данных errseq_t
=========================

errseq_t — это способ записывать ошибки в одном месте и позволять любому
количеству «подписчиков» определять, изменилось ли значение с того момента,
когда оно было опрошено в прошлый раз.

Первоначальный сценарий использования — отслеживание ошибок для системных
вызовов синхронизации файлов (fsync, fdatasync, msync и sync_file_range),
но он может быть применим и в других ситуациях.

Реализуется как беззнаковое 32-битное значение. Младшие биты предназначены
для хранения кода ошибки (от 1 до MAX_ERRNO). Старшие биты используются как
счётчик. Это сделано с помощью атомарных операций, а не блокировки, чтобы эти
функции можно было вызывать из любого контекста.

Обратите внимание, что существует риск коллизий, если новые ошибки
записываются часто, поскольку у нас так мало битов для использования в
качестве счётчика.

Чтобы смягчить это, бит между значением ошибки и счётчиком используется как
флаг, сообщающий, было ли значение опрошено с момента записи нового значения.
Это позволяет нам не увеличивать счётчик, если никто не опрашивал его с
последнего момента записи ошибки.

Таким образом мы получаем значение, которое выглядит примерно так:

+--------------------------------------+----+------------------------+
| 31..13                               | 12 | 11..0                  |
+--------------------------------------+----+------------------------+
| counter                              | SF | errno                  |
+--------------------------------------+----+------------------------+

Общая идея заключается в том, что «наблюдатели» опрашивают значение errseq_t и
хранят его как текущий курсор. Это значение затем можно использовать, чтобы
определить, произошли ли какие-либо новые ошибки с момента того опроса, и
атомарно записать состояние на момент проверки. Это позволяет нам записывать
ошибки в одном месте, а затем иметь некоторое количество «наблюдателей»,
которые могут определить, изменилось ли значение с момента их последней
проверки.

Новый errseq_t всегда должен быть обнулён. Значение errseq_t из одних нулей —
это особый (но распространённый) случай, когда ошибки ещё ни разу не было.
Таким образом, нулевое значение служит «эпохой», если требуется узнать, была
ли вообще установлена какая-либо ошибка с момента его первой инициализации.

Использование API
==================

Позвольте рассказать вам историю об одном рабочем дроне. В целом он хороший
работник, но в компании немного... перебор с руководством. Сегодня ему
приходится отчитываться перед 77 начальниками, а завтра приезжает «большой
босс» из другого города, и он наверняка тоже устроит бедняге проверку.

Все они дают ему работу — так много, что он не может уследить, кто что ему
поручил, но это не такая уж большая проблема. Начальники просто хотят знать,
когда он закончит всю работу, которую они ему дали к этому моменту, и не
допустил ли он каких-либо ошибок с тех пор, как они спрашивали в последний раз.

Он мог допустить ошибку в работе, которую они на самом деле ему не поручали,
но он не способен отслеживать вещи на таком уровне детализации, всё, что он
может вспомнить, — это самую последнюю допущенную им ошибку.

Вот наше представление worker_drone::

        struct worker_drone {
                errseq_t        wd_err; /* for recording errors */
        };

Каждый день worker_drone начинает с чистого листа::

        struct worker_drone wd;

        wd.wd_err = (errseq_t)0;

Приходят начальники и получают начальное показание на день. Их не интересует
ничего, что произошло до начала их смены::

        struct supervisor {
                errseq_t        s_wd_err; /* private "cursor" for wd_err */
                spinlock_t      s_wd_err_lock; /* protects s_wd_err */
        }

        struct supervisor       su;

        su.s_wd_err = errseq_sample(&wd.wd_err);
        spin_lock_init(&su.s_wd_err_lock);

Теперь они начинают давать ему задачи. Каждые несколько минут они просят его
завершить всю работу, которую они ему дали к этому моменту. Затем они
спрашивают, не допустил ли он каких-либо ошибок в чём-либо из этого::

        spin_lock(&su.su_wd_err_lock);
        err = errseq_check_and_advance(&wd.wd_err, &su.s_wd_err);
        spin_unlock(&su.su_wd_err_lock);

До этого момента это просто всё время возвращает 0.

Так вот, владельцы этой компании весьма скупы и выдали ему некачественное
оборудование для выполнения его работы. Время от времени оно даёт сбой, и он
допускает ошибку. Он тяжело вздыхает и помечает это::

        errseq_set(&wd.wd_err, -EIO);

...и затем возвращается к работе. Начальники в конце концов снова опрашивают,
и каждый из них получает ошибку при следующей проверке. Последующие вызовы
будут возвращать 0, пока не будет записана другая ошибка, после чего она
сообщается каждому из них по одному разу.

Обратите внимание, что начальники не могут определить, сколько ошибок он
допустил, а только то, была ли допущена ошибка с момента их последней проверки,
и последнее записанное значение.

Время от времени большой босс приходит с выборочной проверкой и просит
работника выполнить разовое задание для него. Он не наблюдает за работником
постоянно, как начальники, но ему нужно знать, произошла ли ошибка, пока
обрабатывалось его задание.

Он может просто опросить текущий errseq_t у работника, а затем использовать
это, чтобы определить, произошла ли ошибка позднее::

        errseq_t since = errseq_sample(&wd.wd_err);
        /* submit some work and wait for it to complete */
        err = errseq_check(&wd.wd_err, since);

Поскольку после этого момента он просто отбросит «since», ему не нужно
продвигать его здесь. Ему также не нужны никакие блокировки, поскольку оно не
используется никем другим.

Сериализация обновлений курсора errseq_t
=========================================

Обратите внимание, что API errseq_t не защищает курсор errseq_t во время
операции check_and_advance. Атомарно обрабатывается только канонический код
ошибки. В ситуации, когда более одной задачи может использовать один и тот же
курсор errseq_t одновременно, важно сериализовать обновления этого курсора.

Если этого не сделать, то возможно, что курсор пойдёт назад, и в этом случае
одна и та же ошибка может быть сообщена более одного раза.

Из-за этого часто выгодно сначала выполнить errseq_check, чтобы проверить, не
изменилось ли что-нибудь, и только затем выполнить errseq_check_and_advance
после взятия блокировки. Например::

        if (errseq_check(&wd.wd_err, READ_ONCE(su.s_wd_err)) {
                /* su.s_wd_err is protected by s_wd_err_lock */
                spin_lock(&su.s_wd_err_lock);
                err = errseq_check_and_advance(&wd.wd_err, &su.s_wd_err);
                spin_unlock(&su.s_wd_err_lock);
        }

Это позволяет избежать спинлока в распространённом случае, когда ничего не
изменилось с последнего момента проверки.
