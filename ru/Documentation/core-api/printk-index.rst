==============
Индекс printk
==============

Существует множество способов наблюдать за состоянием системы. Одним из важных
источников информации является системный журнал. Он предоставляет большое
количество информации, включая более или менее важные предупреждения и сообщения
об ошибках.

Существуют инструменты мониторинга, которые фильтруют записываемые в журнал
сообщения и выполняют действия на их основе.

Сообщения ядра эволюционируют вместе с кодом. Как следствие, конкретные сообщения
ядра не являются KABI и никогда ими не будут!

Это создаёт огромную сложность для сопровождения мониторов системного журнала.
Это требует знания того, какие сообщения были изменены в конкретной версии ядра
и почему. Поиск этих изменений в исходном коде потребовал бы нетривиальных
парсеров. Кроме того, потребовалось бы сопоставление исходного кода с бинарным
ядром, что не всегда тривиально. Различные изменения могут быть бэкпортированы.
На разных наблюдаемых системах могут использоваться различные версии ядра.

Именно здесь может оказаться полезной функциональность индекса printk. Она
предоставляет дамп форматов printk, используемых по всему исходному коду,
который применяется для ядра и модулей на работающей системе. Доступ к нему
осуществляется во время выполнения через debugfs.

Индекс printk помогает находить изменения в форматах сообщений. Также он помогает
отслеживать строки обратно к исходному коду ядра и связанному коммиту.


Пользовательский интерфейс
==========================

Индекс форматов printk разбит на отдельные файлы. Файлы названы в соответствии
с бинарными файлами, в которые встроены форматы printk. Всегда присутствует
"vmlinux" и опционально также модули, например::

   /sys/kernel/debug/printk/index/vmlinux
   /sys/kernel/debug/printk/index/ext4
   /sys/kernel/debug/printk/index/scsi_mod

Обратите внимание, что показываются только загруженные модули. Кроме того,
форматы printk из модуля могут появляться в "vmlinux", когда модуль встроен.

Содержимое навеяно интерфейсом динамической отладки и выглядит так::

   $> head -1 /sys/kernel/debug/printk/index/vmlinux; shuf -n 5 vmlinux
   # <level[,flags]> filename:line function "format"
   <5> block/blk-settings.c:661 disk_stack_limits "%s: Warning: Device %s is misaligned\n"
   <4> kernel/trace/trace.c:8296 trace_create_file "Could not create tracefs '%s' entry\n"
   <6> arch/x86/kernel/hpet.c:144 _hpet_print_config "hpet: %s(%d):\n"
   <6> init/do_mounts.c:605 prepare_namespace "Waiting for root device %s...\n"
   <6> drivers/acpi/osl.c:1410 acpi_no_auto_serialize_setup "ACPI: auto-serialization disabled\n"

, где значение следующее:

   - :level: значение уровня логирования: 0-7 для конкретной важности, -1 по
	умолчанию, 'c' для непрерывной строки без явного уровня логирования
   - :flags: необязательные флаги: в настоящее время только 'c' для KERN_CONT
   - :filename\:line: имя исходного файла и номер строки соответствующего вызова
	printk(). Обратите внимание, что существует множество обёрток, например,
	pr_warn(), pr_warn_once(), dev_warn().
   - :function: имя функции, где используется вызов printk().
   - :format: строка формата

Дополнительная информация немного усложняет поиск различий между разными
ядрами. Особенно часто может меняться номер строки. С другой стороны, она сильно
помогает подтвердить, что это та же самая строка, или найти коммит, который
ответственен за возможные изменения.


printk() не является стабильным KABI
=====================================

Некоторые разработчики опасаются, что экспорт всех этих деталей реализации в
пространство пользователя превратит конкретные вызовы printk() в KABI.

Но всё ровно наоборот. Вызовы printk() _не_ должны быть KABI. И индекс printk
помогает инструментам пространства пользователя справляться с этим.


Обёртки printk, специфичные для подсистем
=========================================

Индекс printk генерируется с использованием дополнительных метаданных, которые
хранятся в выделенной .elf-секции ".printk_index". Это достигается с помощью
макрообёрток, выполняющих __printk_index_emit() вместе с реальным вызовом
printk(). Та же техника используется и для метаданных, применяемых
функциональностью динамической отладки.

Метаданные сохраняются для конкретного сообщения только тогда, когда оно
выводится с использованием этих специальных обёрток. Это реализовано для
часто используемых вызовов printk(), включая, например, pr_warn() или
pr_once().

Дополнительные изменения необходимы для различных обёрток, специфичных для
подсистем, которые вызывают исходный printk() через общую вспомогательную
функцию. Им требуются собственные обёртки, добавляющие __printk_index_emit().

На данный момент обновлены лишь немногие обёртки, специфичные для подсистем,
например, dev_printk(). Как следствие, форматы printk из некоторых подсистем
могут отсутствовать в индексе printk.


Префикс, специфичный для подсистемы
===================================

Макрос pr_fmt() позволяет определить префикс, который выводится перед строкой,
генерируемой соответствующими вызовами printk().

Обёртки, специфичные для подсистем, обычно добавляют ещё более сложные префиксы.

Эти префиксы могут быть сохранены в метаданные индекса printk с помощью
необязательного параметра __printk_index_emit(). Интерфейс debugfs затем может
показывать форматы printk, включая эти префиксы. Например, drivers/acpi/osl.c
содержит::

  #define pr_fmt(fmt) "ACPI: OSL: " fmt

  static int __init acpi_no_auto_serialize_setup(char *str)
  {
	acpi_gbl_auto_serialize_methods = FALSE;
	pr_info("Auto-serialization disabled\n");

	return 1;
  }

Это приводит к следующей записи индекса printk::

  <6> drivers/acpi/osl.c:1410 acpi_no_auto_serialize_setup "ACPI: auto-serialization disabled\n"

Это помогает сопоставлять сообщения из реального журнала с индексом printk. Затем
имя исходного файла, номер строки и имя функции могут быть использованы для
сопоставления строки с исходным кодом.
