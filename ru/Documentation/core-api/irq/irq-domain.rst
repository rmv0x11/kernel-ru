=============================================================
Библиотека отображения номеров прерываний irq_domain
=============================================================

Текущая архитектура ядра Linux использует единое большое числовое
пространство, в котором каждому отдельному источнику IRQ присваивается
уникальный номер. Это просто, когда есть только один контроллер прерываний.
Но в системах с несколькими контроллерами прерываний ядро должно
гарантировать, что каждому из них выделяются непересекающиеся диапазоны
номеров Linux IRQ.

Число контроллеров прерываний, зарегистрированных как уникальные irqchip,
имеет тенденцию к росту. Например, субдрайверы различного рода, такие как
контроллеры GPIO, избегают повторной реализации механизмов обратного вызова,
идентичных тем, что есть в ядре подсистемы IRQ, моделируя свои обработчики
прерываний как irqchip. То есть, по сути, создавая каскад контроллеров
прерываний.

Так, в прошлом номера IRQ можно было выбирать так, чтобы они совпадали с
аппаратной линией IRQ в корневом контроллере прерываний (то есть с
компонентом, который фактически выставляет линию прерывания на CPU).
В наше время этот номер — просто номер, и он не имеет никакого отношения к
аппаратным номерам прерываний.

По этой причине нам нужен механизм, позволяющий отделить локальные для
контроллера номера прерываний, называемые аппаратными IRQ, от номеров
Linux IRQ.

API-функции irq_alloc_desc*() и irq_free_desc*() обеспечивают выделение
номеров IRQ, но не предоставляют никакой поддержки обратного отображения
локального для контроллера номера IRQ (hwirq) в пространство номеров
Linux IRQ.

Библиотека irq_domain добавляет отображение между номерами hwirq и IRQ
поверх API irq_alloc_desc*(). Использование irq_domain для управления
отображением предпочтительнее, чем реализация драйверами контроллеров
прерываний собственной схемы обратного отображения вручную.

irq_domain также реализует трансляцию из абстрактной struct irq_fwspec
в номера hwirq (на данный момент Device Tree, узел firmware вне DT, ACPI
GSI и программный узел) и может быть легко расширена для поддержки других
источников данных о топологии IRQ. Реализация выполнена без какого-либо
дополнительного платформенно-зависимого кода поддержки.

Использование irq_domain
========================
struct irq_domain можно определить как контроллер домена IRQ. То есть он
обрабатывает отображение между аппаратными и виртуальными номерами
прерываний для заданного домена прерываний. Структура домена обычно
создаётся кодом PIC для конкретного экземпляра PIC (хотя один домен может
охватывать более одного PIC, если у них плоская числовая модель).
Именно обратные вызовы домена отвечают за установку irq_chip для
заданного irq_desc после того, как он был отображён.

Код и структуры данных хоста используют указатель fwnode_handle для
идентификации домена. В некоторых случаях, и для сохранения совместимости
на уровне исходного кода, этот указатель fwnode «повышается» до DT
device_node. Для тех инфраструктур firmware, которые не предоставляют
уникального идентификатора для контроллера прерываний, код irq_domain
предлагает аллокатор fwnode.

Драйвер контроллера прерываний создаёт и регистрирует struct irq_domain,
вызывая одну из функций irq_domain_create_*() (у каждого метода
отображения своя функция-аллокатор, об этом позже). В случае успеха
функция вернёт указатель на struct irq_domain. Вызывающая сторона должна
передать функции-аллокатору указатель struct irq_domain_ops.

В большинстве случаев irq_domain в начале будет пустым, без каких-либо
отображений между номерами hwirq и IRQ.  Отображения добавляются в
irq_domain вызовом irq_create_mapping(), который принимает в качестве
аргументов irq_domain и номер hwirq. Если отображение для hwirq ещё не
существует, irq_create_mapping() выделяет новый Linux irq_desc, связывает
его с hwirq и вызывает обратный вызов :c:member:`irq_domain_ops.map()`.
В нём драйвер может выполнить любую необходимую аппаратную настройку.

После того как отображение установлено, его можно получить или
использовать различными способами:

- irq_resolve_mapping() возвращает указатель на структуру irq_desc для
  заданного домена и номера hwirq, или NULL, если отображения не было.
- irq_find_mapping() возвращает номер Linux IRQ для заданного домена и
  номера hwirq, или 0, если отображения не было
- generic_handle_domain_irq() обрабатывает прерывание, описываемое
  доменом и номером hwirq

Обратите внимание, что обращения к irq_domain должны происходить в
контекстах, совместимых с критической секцией на стороне чтения RCU.

Функция irq_create_mapping() должна быть вызвана *хотя бы один раз* перед
любым вызовом irq_find_mapping(), иначе дескриптор не будет выделен.

Если у драйвера есть номер Linux IRQ или указатель irq_data, и ему нужно
узнать связанный с ним номер hwirq (например, в обратных вызовах irq_chip),
то его можно получить напрямую из :c:member:`irq_data.hwirq`.

Типы отображений irq_domain
===========================

Существует несколько механизмов обратного отображения из hwirq в Linux
IRQ, и каждый механизм использует свою функцию выделения. Какой тип
обратного отображения следует использовать, зависит от сценария
применения.  Каждый из типов обратного отображения описан ниже:

Линейное
--------

::

	irq_domain_create_linear()

Линейное обратное отображение поддерживает таблицу фиксированного
размера, индексируемую номером hwirq.  Когда hwirq отображается, для него
выделяется irq_desc, а номер IRQ сохраняется в таблице.

Линейное отображение — хороший выбор, когда максимальное число hwirq
фиксировано и относительно невелико (~ < 256).  Преимущества этого
отображения — поиск номеров IRQ за фиксированное время и то, что
irq_desc выделяются только для используемых IRQ.  Недостаток в том, что
таблица должна быть такой же большой, как наибольший возможный номер hwirq.

Большинству драйверов следует использовать линейное отображение.

Дерево
------

::

	irq_domain_create_tree()

irq_domain поддерживает отображение в виде дерева (radix tree) из номеров
hwirq в Linux IRQ.  Когда hwirq отображается, выделяется irq_desc, а
hwirq используется как ключ поиска для radix tree.

Древовидное отображение — хороший выбор, если номер hwirq может быть
очень большим, поскольку при этом не нужно выделять таблицу размером с
наибольший номер hwirq.  Недостаток в том, что поиск номера IRQ по hwirq
зависит от того, сколько записей в таблице.

Это отображение должно понадобиться очень немногим драйверам.

Без отображения
---------------

::

	irq_domain_create_nomap()

Отображение «без отображения» (No Map) следует использовать, когда номер
hwirq программируется в самом оборудовании.  В этом случае лучше всего
запрограммировать номер Linux IRQ непосредственно в оборудование, так
чтобы отображение не требовалось.  Вызов irq_create_direct_mapping()
выделит номер Linux IRQ и вызовет обратный вызов .map(), чтобы драйвер
мог запрограммировать номер Linux IRQ в оборудование.

Большинство драйверов не могут использовать это отображение, и теперь
оно поставлено в зависимость от параметра CONFIG_IRQ_DOMAIN_NOMAP.
Пожалуйста, воздержитесь от введения новых пользователей этого API.

Legacy
------

::

	irq_domain_create_simple()
	irq_domain_create_legacy()

Отображение Legacy — особый случай для драйверов, у которых уже выделен
диапазон irq_desc для hwirq.  Оно используется, когда драйвер нельзя сразу
перевести на использование линейного отображения.  Например, многие файлы
поддержки плат встраиваемых систем используют набор #define для номеров
IRQ, которые передаются в регистрации struct device.  В этом случае номера
Linux IRQ нельзя назначать динамически, и следует использовать отображение
Legacy.

Как следует из названия, функции \*_legacy() являются устаревшими и
существуют лишь для облегчения поддержки очень старых платформ. Новых
пользователей добавлять не следует. То же касается функций \*_simple(),
когда их использование приводит к поведению legacy.

Отображение Legacy предполагает, что для контроллера уже выделен
непрерывный диапазон номеров IRQ и что номер IRQ можно вычислить,
прибавив фиксированное смещение к номеру hwirq, и наоборот.  Недостаток в
том, что это требует от контроллера прерываний управлять выделениями IRQ
и требует, чтобы irq_desc был выделен для каждого hwirq, даже если он не
используется.

Отображение Legacy следует использовать только в том случае, если
необходимо поддерживать фиксированные отображения IRQ.  Например,
контроллеры ISA использовали бы отображение Legacy для отображения Linux
IRQ 0-15, чтобы существующие драйверы ISA получали правильные номера IRQ.

Большинству пользователей legacy-отображений следует использовать
irq_domain_create_simple(), которая будет использовать домен legacy
только в том случае, если диапазон IRQ предоставлен системой, а в
противном случае использует линейное отображение домена. Семантика этого
вызова такова, что если диапазон IRQ указан, то дескрипторы будут
выделяться для него на лету, а если диапазон не указан, вызов перейдёт к
irq_domain_create_linear(), что означает, что *никакие* дескрипторы IRQ
выделяться не будут.

Типичный сценарий применения простых (simple) доменов — когда провайдер
irqchip поддерживает как динамическое, так и статическое назначение IRQ.

Чтобы избежать ситуации, когда используется линейный домен и при этом не
выделяется ни одного дескриптора, очень важно убедиться, что драйвер,
использующий вызов простого домена, вызывает irq_create_mapping() перед
любым irq_find_mapping(), поскольку последний действительно будет
работать в случае статического назначения IRQ.

Иерархический домен IRQ
-----------------------

На некоторых архитектурах в доставке прерывания от устройства к целевому
CPU может участвовать несколько контроллеров прерываний.
Рассмотрим типичный путь доставки прерывания на платформах x86::

  Device --> IOAPIC -> Interrupt remapping Controller -> Local APIC -> CPU

Здесь задействованы три контроллера прерываний:

1) контроллер IOAPIC
2) контроллер переотображения прерываний (interrupt remapping)
3) контроллер Local APIC

Чтобы поддержать такую аппаратную топологию и привести программную
архитектуру в соответствие с аппаратной, для каждого контроллера прерываний
строится структура данных irq_domain, и эти irq_domain организуются в
иерархию. При построении иерархии irq_domain ближайший к устройству
irq_domain является дочерним, а ближайший к CPU irq_domain — родительским.
Так, для приведённого выше примера будет построена иерархическая структура,
показанная ниже::

	CPU Vector irq_domain (root irq_domain to manage CPU vectors)
		^
		|
	Interrupt Remapping irq_domain (manage irq_remapping entries)
		^
		|
	IOAPIC irq_domain (manage IOAPIC delivery entries/pins)

Существует четыре основных интерфейса для использования иерархического
irq_domain:

1) irq_domain_alloc_irqs(): выделить дескрипторы IRQ и связанные с
   контроллером прерываний ресурсы для доставки этих прерываний.
2) irq_domain_free_irqs(): освободить дескрипторы IRQ и связанные с
   контроллером прерываний ресурсы, относящиеся к этим прерываниям.
3) irq_domain_activate_irq(): активировать оборудование контроллера
   прерываний для доставки прерывания.
4) irq_domain_deactivate_irq(): деактивировать оборудование контроллера
   прерываний, чтобы прекратить доставку прерывания.

Для поддержки иерархического irq_domain необходимо следующее:

1) Поле :c:member:`parent` в struct irq_domain используется для хранения
   информации об иерархии irq_domain.
2) Поле :c:member:`parent_data` в struct irq_data используется для
   построения иерархии irq_data, соответствующей иерархии irq_domains.
   irq_data используется для хранения указателя irq_domain и аппаратного
   номера IRQ.
3) Обратные вызовы :c:member:`alloc()`, :c:member:`free()` и другие в
   struct irq_domain_ops для поддержки операций иерархического irq_domain.

При наличии поддержки иерархического irq_domain и иерархического irq_data
для каждого контроллера прерываний строится структура irq_domain, а для
каждого irq_domain, связанного с IRQ, выделяется структура irq_data.

Чтобы драйвер контроллера прерываний поддерживал иерархический irq_domain,
ему нужно:

1) Реализовать irq_domain_ops.alloc() и irq_domain_ops.free()
2) Опционально реализовать irq_domain_ops.activate() и
   irq_domain_ops.deactivate().
3) Опционально реализовать irq_chip для управления оборудованием
   контроллера прерываний.
4) Реализовывать irq_domain_ops.map() и irq_domain_ops.unmap() не нужно.
   Они не используются с иерархическим irq_domain.

Обратите внимание, что иерархический irq_domain ни в коей мере не является
специфичным для x86 и активно используется для поддержки других
архитектур, таких как ARM, ARM64 и т. д.

Стек irq_chip
~~~~~~~~~~~~~~

Теперь мы могли бы пойти ещё на шаг дальше и поддержать многоуровневый
(иерархический) irq_chip. То есть с каждым irq_data вдоль иерархии
связывается irq_chip. Дочерний irq_chip может реализовать требуемое
действие сам или во взаимодействии со своим родительским irq_chip.

При многоуровневом irq_chip драйвер контроллера прерываний должен иметь
дело только с оборудованием, управляемым им самим, и может при
необходимости запрашивать услуги у своего родительского irq_chip. Так мы
могли бы добиться гораздо более чистой программной архитектуры.
