==================================
Трассировка состояния IRQ-флагов
==================================

:Author: started by Ingo Molnar <mingo@redhat.com>

Функциональность «irq-flags tracing» (трассировки IRQ-флагов) «трассирует»
состояние hardirq и softirq в том смысле, что она даёт заинтересованным
подсистемам возможность получать уведомления о каждом событии
hardirqs-off/hardirqs-on, softirqs-off/softirqs-on, которое происходит в ядре.

CONFIG_TRACE_IRQFLAGS_SUPPORT необходим для того, чтобы общий код отладки
блокировок предлагал CONFIG_PROVE_SPIN_LOCKING и CONFIG_PROVE_RW_LOCKING. В
противном случае на архитектуре будут предлагаться только
CONFIG_PROVE_MUTEX_LOCKING и CONFIG_PROVE_RWSEM_LOCKING — это API блокировок,
которые не используются в контексте IRQ. (единственное исключение для rwsem
обходится отдельно)

Поддержка этого со стороны архитектуры определённо не относится к «тривиальной»
категории, поскольку множество низкоуровневого ассемблерного кода работает с
изменениями состояния IRQ-флагов. Но архитектуру можно сделать поддерживающей
трассировку IRQ-флагов довольно прямолинейным и безрисковым способом.

Архитектурам, которые хотят поддерживать это, сначала нужно внести пару
организационных изменений в код:

- добавить и включить TRACE_IRQFLAGS_SUPPORT в Kconfig-файле своего уровня
  архитектуры

а затем необходимо также внести пару функциональных изменений, чтобы
реализовать поддержку трассировки IRQ-флагов:

- в низкоуровневый код входа добавить (условно компилируемые) вызовы функций
  trace_hardirqs_off()/trace_hardirqs_on(). Валидатор блокировок внимательно
  следит за тем, совпадает ли «реальное» состояние IRQ-флагов с «виртуальным»
  состоянием IRQ-флагов, и громко жалуется (и отключает сам себя), если эти два
  состояния не совпадают. Обычно большая часть времени при реализации поддержки
  трассировки IRQ-флагов на архитектуре тратится именно в этом состоянии:
  посмотреть на жалобу lockdep, попытаться выяснить, какой ассемблерный код мы
  ещё не покрыли, исправить и повторить. Как только система загрузилась и
  работает без жалобы lockdep в функциях трассировки IRQ-флагов, поддержка со
  стороны архитектуры завершена.
- если архитектура имеет немаскируемые прерывания, то их необходимо исключить из
  механизма трассировки IRQ [и валидации блокировок] с помощью
  lockdep_off()/lockdep_on().

В целом нет никакого риска от наличия в архитектуре неполной реализации
трассировки IRQ-флагов: lockdep обнаружит это и отключит сам себя. То есть
валидатор блокировок по-прежнему будет надёжным. Не должно быть сбоев из-за
ошибок трассировки IRQ. (за исключением случая, когда изменения в ассемблере
ломают другой код, модифицируя условия или регистры, которые не должны
изменяться)
