============================================
Сброс кэша и TLB в Linux
============================================

:Author: David S. Miller <davem@redhat.com>

Этот документ описывает интерфейсы сброса кэша/tlb, вызываемые
подсистемой VM ядра Linux.  Он перечисляет каждый интерфейс,
описывает его предназначение и какой побочный эффект ожидается
после вызова интерфейса.

Описываемые ниже побочные эффекты сформулированы для однопроцессорной
реализации и для того, что должно происходить на этом единственном процессоре.  Случаи
SMP представляют собой простое расширение в том смысле, что вы просто расширяете
определение так, чтобы побочный эффект для конкретного интерфейса происходил
на всех процессорах в системе.  Не позволяйте этому напугать вас и заставить
думать, что сброс кэша/tlb в SMP обязательно настолько неэффективен; на самом деле
это область, где возможно множество оптимизаций.  Например,
если можно доказать, что пользовательское адресное пространство никогда не исполнялось
на каком-либо cpu (см. mm_cpumask()), для этого адресного пространства на этом cpu
выполнять сброс не нужно.

Сначала — интерфейсы сброса TLB, поскольку они самые простые.  В Linux
«TLB» абстрагирован как нечто, что cpu использует для кэширования
преобразований виртуальных-->физических адресов, полученных из программных
таблиц страниц.  Это означает, что если программные таблицы страниц изменяются, в этом
кэше «TLB» могут существовать устаревшие преобразования.
Поэтому при изменениях программных таблиц страниц ядро вызовет
один из следующих методов сброса _после_ того, как изменения таблиц страниц
произошли:

1) ``void flush_tlb_all(void)``

	Самый радикальный сброс из всех.  После работы этого интерфейса
	cpu станет видна любая предыдущая модификация таблиц страниц,
	какой бы она ни была.

	Обычно он вызывается при изменении таблиц страниц ядра,
	поскольку такие преобразования по своей природе «глобальны».

2) ``void flush_tlb_mm(struct mm_struct *mm)``

	Этот интерфейс сбрасывает из TLB всё пользовательское адресное
	пространство.  После работы этот интерфейс должен гарантировать, что
	cpu станут видны любые предыдущие модификации таблиц страниц для адресного
	пространства 'mm'.  То есть после работы в TLB не будет
	записей для 'mm'.

	Этот интерфейс используется для обработки операций над таблицами страниц
	целого адресного пространства, таких как происходящие во время
	fork и exec.

3) ``void flush_tlb_range(struct vm_area_struct *vma,
   unsigned long start, unsigned long end)``

	Здесь мы сбрасываем из TLB определённый диапазон преобразований
	(пользовательских) виртуальных адресов.  После работы этот
	интерфейс должен гарантировать, что cpu станут видны любые предыдущие модификации
	таблиц страниц для адресного пространства 'vma->vm_mm' в диапазоне
	от 'start' до 'end-1'.  То есть после
	работы в TLB не будет записей для 'mm' для
	виртуальных адресов в диапазоне от 'start' до 'end-1'.

	«vma» — это резервное хранилище (backing store), используемое для данной области.
	Главным образом это используется для операций типа munmap().

	Интерфейс предоставлен в надежде, что порт сможет найти
	достаточно эффективный метод удаления из TLB нескольких преобразований
	размером со страницу, вместо того чтобы ядро
	вызывало flush_tlb_page (см. ниже) для каждой записи, которая может быть
	модифицирована.

4) ``void flush_tlb_page(struct vm_area_struct *vma, unsigned long addr)``

	На этот раз нам нужно удалить из TLB преобразование
	размером PAGE_SIZE.  'vma' — это резервная структура, используемая
	Linux для отслеживания отображённых через mmap областей процесса; адресное
	пространство доступно через vma->vm_mm.  Кроме того, можно
	проверить (vma->vm_flags & VM_EXEC), чтобы узнать, является ли эта область
	исполняемой (и, следовательно, могла бы находиться в «instruction TLB» в
	конфигурациях типа split-tlb).

	После работы этот интерфейс должен гарантировать, что cpu станет видна любая предыдущая
	модификация таблиц страниц для адресного пространства 'vma->vm_mm' для
	пользовательского виртуального адреса 'addr'.  То
	есть после работы в TLB не будет записей для
	'vma->vm_mm' для виртуального адреса 'addr'.

	Это используется главным образом во время обработки ошибок (fault).

5) ``void update_mmu_cache_range(struct vm_fault *vmf,
   struct vm_area_struct *vma, unsigned long address, pte_t *ptep,
   unsigned int nr)``

	В конце каждой ошибки страницы (page fault) вызывается эта процедура, чтобы сообщить
	архитектурно-зависимому коду, что в программных таблицах страниц
	теперь существуют преобразования для адресного пространства "vma->vm_mm"
	по виртуальному адресу "address" для "nr" последовательных страниц.

	Эта процедура также вызывается в различных других местах, которые передают
	NULL в качестве "vmf".

	Порт может использовать эту информацию любым угодным ему образом.
	Например, он мог бы использовать это событие для предварительной загрузки преобразований
	TLB в конфигурациях с программно управляемым TLB.
	Порт sparc64 в настоящее время так и делает.

Далее у нас есть интерфейсы сброса кэша.  В общем случае, когда Linux
изменяет существующее отображение виртуальный-->физический на новое значение,
последовательность будет иметь одну из следующих форм::

	1) flush_cache_mm(mm);
	   change_all_page_tables_of(mm);
	   flush_tlb_mm(mm);

	2) flush_cache_range(vma, start, end);
	   change_range_of_page_tables(mm, start, end);
	   flush_tlb_range(vma, start, end);

	3) flush_cache_page(vma, addr, pfn);
	   set_pte(pte_pointer, new_pte_val);
	   flush_tlb_page(vma, addr);

Сброс на уровне кэша всегда выполняется первым, поскольку это позволяет
нам корректно обрабатывать системы, кэши которых являются строгими и требуют
существования преобразования виртуальный-->физический для виртуального адреса,
когда этот виртуальный адрес сбрасывается из кэша.  Cpu HyperSparc —
один из таких cpu с этим свойством.

Приведённым ниже процедурам сброса кэша необходимо заниматься сбросом кэша
лишь в той мере, в какой это необходимо для конкретного cpu.  По большей части
эти процедуры должны быть реализованы для cpu, имеющих виртуально
индексированные кэши, которые необходимо сбрасывать при изменении или удалении
преобразований виртуальный-->физический.  Так, например, физически
индексированные физически тегированные кэши процессоров IA32 не нуждаются в
реализации этих интерфейсов, поскольку эти кэши полностью синхронизированы
и не зависят от информации о преобразованиях.

Вот эти процедуры, одна за другой:

1) ``void flush_cache_mm(struct mm_struct *mm)``

	Этот интерфейс сбрасывает из кэшей всё пользовательское адресное
	пространство.  То есть после работы не будет ни одной строки кэша,
	связанной с 'mm'.

	Этот интерфейс используется для обработки операций над таблицами страниц
	целого адресного пространства, таких как происходящие во время exit и exec.

2) ``void flush_cache_dup_mm(struct mm_struct *mm)``

	Этот интерфейс сбрасывает из кэшей всё пользовательское адресное
	пространство.  То есть после работы не будет ни одной строки кэша,
	связанной с 'mm'.

	Этот интерфейс используется для обработки операций над таблицами страниц
	целого адресного пространства, таких как происходящие во время fork.

	Этот вариант отделён от flush_cache_mm, чтобы позволить некоторые
	оптимизации для кэшей VIPT.

3) ``void flush_cache_range(struct vm_area_struct *vma,
   unsigned long start, unsigned long end)``

	Здесь мы сбрасываем из кэша определённый диапазон (пользовательских)
	виртуальных адресов.  После работы в кэше не будет
	записей для 'vma->vm_mm' для виртуальных адресов в
	диапазоне от 'start' до 'end-1'.

	«vma» — это резервное хранилище (backing store), используемое для данной области.
	Главным образом это используется для операций типа munmap().

	Интерфейс предоставлен в надежде, что порт сможет найти
	достаточно эффективный метод удаления из кэша нескольких областей
	размером со страницу, вместо того чтобы ядро
	вызывало flush_cache_page (см. ниже) для каждой записи, которая может быть
	модифицирована.

4) ``void flush_cache_page(struct vm_area_struct *vma, unsigned long addr, unsigned long pfn)``

	На этот раз нам нужно удалить из кэша диапазон размером
	PAGE_SIZE.  'vma' — это резервная структура, используемая
	Linux для отслеживания отображённых через mmap областей процесса; адресное
	пространство доступно через vma->vm_mm.  Кроме того, можно
	проверить (vma->vm_flags & VM_EXEC), чтобы узнать, является ли эта область
	исполняемой (и, следовательно, могла бы находиться в «instruction cache» в
	кэш-конфигурациях типа «Harvard»).

	'pfn' указывает физический фрейм страницы (сдвиньте это значение
	влево на PAGE_SHIFT, чтобы получить физический адрес), в который преобразуется
	'addr'.  Именно это отображение следует удалить из
	кэша.

	После работы в кэше не будет записей для
	'vma->vm_mm' для виртуального адреса 'addr', который преобразуется
	в 'pfn'.

	Это используется главным образом во время обработки ошибок (fault).

5) ``void flush_cache_kmaps(void)``

	Эту процедуру необходимо реализовать только если платформа использует
	highmem.  Она будет вызвана непосредственно перед тем, как все kmap'ы
	будут инвалидированы.

	После работы в кэше не будет записей для
	диапазона виртуальных адресов ядра от PKMAP_ADDR(0) до
	PKMAP_ADDR(LAST_PKMAP).

	Эту процедуру следует реализовать в asm/highmem.h

6) ``void flush_cache_vmap(unsigned long start, unsigned long end)``
   ``void flush_cache_vunmap(unsigned long start, unsigned long end)``

	Здесь, в этих двух интерфейсах, мы сбрасываем из кэша определённый
	диапазон (ядерных) виртуальных адресов.  После работы
	в кэше не будет записей для адресного пространства ядра
	для виртуальных адресов в диапазоне от 'start' до 'end-1'.

	Первая из этих двух процедур вызывается после того, как vmap_range()
	установил записи таблицы страниц.  Вторая вызывается
	перед тем, как vunmap_range() удалит записи таблицы страниц.

Существует ещё целый класс проблем кэша cpu, для корректной обработки
которых в настоящее время требуется совершенно другой набор интерфейсов.
Наибольшая проблема — это виртуальное наложение (virtual aliasing) в кэше данных
процессора.

Подвержен ли ваш порт виртуальному наложению в своём D-cache?
Что ж, если ваш D-cache виртуально индексирован, по размеру больше
PAGE_SIZE и не предотвращает одновременное существование нескольких строк кэша для одного
и того же физического адреса, то у вас есть эта проблема.

Если у вашего D-cache есть эта проблема, сначала правильно определите SHMLBA в
asm/shmparam.h; оно должно быть по сути размером вашего виртуально
адресуемого D-cache (или, если размер переменный, максимально возможным
размером).  Эта настройка заставит слой SYSv IPC разрешать пользовательским
процессам отображать через mmap разделяемую память только по адресам, кратным
этому значению.

.. note::

  Это не исправляет разделяемые mmap'ы; ознакомьтесь с портом sparc64 для
  одного из способов решить это (в частности, SPARC_FLAG_MMAPSHARED).

Далее вам нужно решить проблему наложения D-cache для всех
остальных случаев.  Пожалуйста, имейте в виду тот факт, что для данной страницы,
отображённой в некоторое пользовательское адресное пространство, всегда существует ещё хотя бы одно
отображение — отображение ядром в его линейном отображении, начинающемся с
PAGE_OFFSET.  Поэтому сразу же, как только первый пользователь отобразит данную
физическую страницу в своё адресное пространство, тем самым возникает потенциальная
возможность проблемы наложения D-cache, поскольку ядро уже
отображает эту страницу по своему виртуальному адресу.

  ``void copy_user_page(void *to, void *from, unsigned long addr, struct page *page)``
  ``void clear_user_page(void *to, unsigned long addr, struct page *page)``

	Эти две процедуры сохраняют данные в пользовательских анонимных или COW
	страницах.  Они позволяют порту эффективно избегать проблем наложения D-cache
	между userspace и ядром.

	Например, порт может временно отобразить 'from' и 'to' на
	виртуальные адреса ядра во время копирования.  Виртуальный адрес
	для этих двух страниц выбирается таким образом, чтобы инструкции
	load/store ядра обращались к виртуальным адресам, имеющим
	тот же «цвет» (color), что и пользовательское отображение страницы.  Sparc64,
	например, использует эту технику.

	Параметр 'addr' сообщает виртуальный адрес, по которому
	пользователь в конечном итоге будет иметь эту страницу отображённой, а параметр
	'page' даёт указатель на struct page целевой страницы.

	Если наложение D-cache не является проблемой, эти две процедуры могут
	просто напрямую вызывать memcpy/memset и не делать ничего больше.

  ``void flush_dcache_folio(struct folio *folio)``

        Эту процедуру необходимо вызывать, когда:

	  a) ядро записало в страницу, находящуюся в страничном кэше (page cache)
	     и/или в high memory
	  b) ядро собирается читать из страницы страничного кэша, и потенциально
	     существуют разделяемые/доступные на запись пользовательские отображения этой страницы.  Обратите
	     внимание, что {get,pin}_user_pages{_fast} уже вызывают flush_dcache_folio
	     для любой страницы, найденной в пользовательском адресном пространстве, и поэтому код
	     драйверов редко должен учитывать это.

	.. note::

	      Эту процедуру необходимо вызывать только для страниц страничного кэша,
	      которые потенциально могут когда-либо быть отображены в адресное
	      пространство пользовательского процесса.  Так, например, коду слоя VFS,
	      обрабатывающему символические ссылки vfs в страничном кэше, не нужно вызывать
	      этот интерфейс вообще.

	Фраза «ядро записывает в страницу страничного кэша» означает, в частности,
	что ядро исполняет инструкции store, загрязняющие (dirty) данные в этой
	странице в ядерном виртуальном отображении этой страницы.  Здесь важно
	выполнить сброс для обработки наложения D-cache, чтобы убедиться, что эти записи ядра
	видны пользовательским отображениям этой страницы.

	Сопутствующий случай столь же важен: если есть пользователи, имеющие
	разделяемые+доступные на запись отображения этого файла, мы должны убедиться, что чтения ядром
	этих страниц увидят самые последние записи, выполненные пользователем.

	Если наложение D-cache не является проблемой, эту процедуру можно просто определить
	как nop на этой архитектуре.

        В folio->flags (PG_arch_1) выделен бит как «приватный для архитектуры»
	(architecture private).  Ядро гарантирует, что для страниц страничного кэша оно
	будет сбрасывать этот бит, когда такая страница впервые попадает в страничный кэш.

	Это позволяет реализовать эти интерфейсы гораздо более
	эффективно.  Это позволяет «отложить» (возможно, на неопределённый срок)
	фактический сброс, если в настоящий момент нет пользовательских процессов, отображающих эту
	страницу.  Пример того, как это сделать, см. в реализациях flush_dcache_folio и
	update_mmu_cache_range для sparc64.

	Идея в следующем: сначала во время flush_dcache_folio(), если
	folio_flush_mapping() возвращает отображение, а mapping_mapped() для этого
	отображения возвращает %false, просто пометить бит приватного для архитектуры
	флага страницы.  Позже, в update_mmu_cache_range(), производится проверка
	этого бита флага, и если он установлен, выполняется сброс и бит флага
	сбрасывается.

	.. important::

				Часто важно, если вы откладываете сброс,
				чтобы фактический сброс произошёл на том же CPU,
				который выполнял записи cpu в страницу, сделавшие её
				грязной (dirty).  Снова см. sparc64 для примеров того,
				как иметь с этим дело.

  ``void copy_to_user_page(struct vm_area_struct *vma, struct page *page,
  unsigned long user_vaddr, void *dst, void *src, int len)``
  ``void copy_from_user_page(struct vm_area_struct *vma, struct page *page,
  unsigned long user_vaddr, void *dst, void *src, int len)``

	Когда ядру нужно скопировать произвольные данные в произвольные
	пользовательские страницы и обратно (например, для ptrace()), оно использует
	эти две процедуры.

	Любой необходимый сброс кэша или другие операции обеспечения когерентности,
	которые должны произойти, должны происходить здесь.  Если кэш инструкций
	процессора не отслеживает (snoop) записи cpu, весьма
	вероятно, что вам потребуется сбросить кэш инструкций
	для copy_to_user_page().

  ``void flush_anon_page(struct vm_area_struct *vma, struct page *page,
  unsigned long vmaddr)``

  	Когда ядру нужно получить доступ к содержимому анонимной
	страницы, оно вызывает эту функцию (в настоящее время только
	get_user_pages()).  Примечание: flush_dcache_folio() намеренно
	не работает для анонимной страницы.  Реализация по умолчанию
	является nop (и должна оставаться таковой для всех когерентных
	архитектур).  Для некогерентных архитектур она должна сбрасывать
	кэш страницы по адресу vmaddr.

  ``void flush_icache_range(unsigned long start, unsigned long end)``

  	Когда ядро записывает в адреса, из которых оно будет исполнять
	код (например, при загрузке модулей), вызывается эта функция.

	Если icache не отслеживает (snoop) записи, то этой процедуре потребуется
	сбросить его.

  ``void flush_icache_page(struct vm_area_struct *vma, struct page *page)``

	Вся функциональность flush_icache_page может быть реализована в
	flush_dcache_folio и update_mmu_cache_range. В будущем есть надежда
	полностью удалить этот интерфейс.

Последняя категория API предназначена для ввода-вывода в намеренно наложенные
(aliased) диапазоны адресов внутри ядра.  Такие наложения настраиваются с помощью
API vmap/vmalloc.  Поскольку ввод-вывод ядра идёт через физические страницы, подсистема
ввода-вывода предполагает, что пользовательское отображение и отображение с ядерным смещением
являются единственными наложениями.  Это неверно для наложений vmap, поэтому всё, что
в ядре пытается выполнять ввод-вывод в области vmap, должно вручную управлять
когерентностью.  Оно должно делать это, сбрасывая диапазон vmap перед выполнением
ввода-вывода и инвалидируя его после возврата из ввода-вывода.

  ``void flush_kernel_vmap_range(void *vaddr, int size)``

       сбрасывает кэш ядра для заданного диапазона виртуальных адресов в
       области vmap.  Это нужно для того, чтобы убедиться, что любые данные, которые ядро
       модифицировало в диапазоне vmap, становятся видны физической
       странице.  Замысел в том, чтобы сделать эту область безопасной для выполнения ввода-вывода.
       Обратите внимание, что этот API *не* сбрасывает заодно наложение через отображение со смещением
       для этой области.

  ``void invalidate_kernel_vmap_range(void *vaddr, int size) invalidates``

       инвалидирует кэш для заданного диапазона виртуальных адресов в
       области vmap, что не даёт процессору сделать кэш устаревшим за счёт
       спекулятивного чтения данных во время выполнения ввода-вывода в физические
       страницы.  Это необходимо только для чтений данных в
       область vmap.
