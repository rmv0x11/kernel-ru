============================================
API refcount_t в сравнении с atomic_t
============================================

.. contents:: :local:

Введение
========

Цель API refcount_t — предоставить минимальный API для реализации
счётчиков ссылок на объекты. Хотя обобщённая, не зависящая от архитектуры
реализация из lib/refcount.c использует под капотом атомарные операции,
существует ряд различий между некоторыми из функций ``refcount_*()`` и
``atomic_*()`` в части гарантий упорядочивания обращений к памяти.
Данный документ описывает эти различия и приводит соответствующие примеры,
чтобы помочь сопровождающим (maintainers) проверить свой код на корректность
относительно изменений в этих гарантиях упорядочивания обращений к памяти.

Термины, используемые в этом документе, стараются следовать формальной модели
LKMM, определённой в tools/memory-model/Documentation/explanation.txt.

Документы memory-barriers.txt и atomic_t.txt дают более широкий контекст по
упорядочиванию обращений к памяти в целом и для атомарных операций в частности.

Релевантные типы упорядочивания обращений к памяти
==================================================

.. note:: Следующий раздел охватывает лишь некоторые из типов упорядочивания
   обращений к памяти, которые релевантны для атомиков и счётчиков ссылок
   и используются в этом документе. Для гораздо более полной картины
   обратитесь к документу memory-barriers.txt.

При отсутствии каких-либо гарантий упорядочивания обращений к памяти (т. е.
при полной неупорядоченности) атомики и счётчики ссылок обеспечивают только
атомарность и отношение программного порядка (program order, po) (на одном и
том же CPU). Это гарантирует, что каждая операция ``atomic_*()`` и
``refcount_*()`` атомарна и что инструкции выполняются в программном порядке на
одном CPU. Это реализуется с помощью READ_ONCE()/WRITE_ONCE() и примитивов
compare-and-swap.

Строгое (полное) упорядочивание обращений к памяти гарантирует, что все
предшествующие операции загрузки и сохранения (все po-более ранние инструкции)
на одном и том же CPU завершены до того, как на этом же CPU будет выполнена
любая po-более поздняя инструкция. Оно также гарантирует, что все po-более
ранние операции сохранения на этом же CPU и все распространённые операции
сохранения с других CPU должны распространиться на все остальные CPU прежде, чем
на исходном CPU будет выполнена любая po-более поздняя инструкция (свойство
A-кумулятивности). Это реализуется с помощью smp_mb().

Упорядочивание обращений к памяти типа RELEASE гарантирует, что все
предшествующие операции загрузки и сохранения (все po-более ранние инструкции)
на одном и том же CPU завершены до этой операции. Оно также гарантирует, что
все po-более ранние операции сохранения на этом же CPU и все распространённые
операции сохранения с других CPU должны распространиться на все остальные CPU
до операции release (свойство A-кумулятивности). Это реализуется с помощью
smp_store_release().

Упорядочивание обращений к памяти типа ACQUIRE гарантирует, что все последующие
операции загрузки и сохранения (все po-более поздние инструкции) на одном и том
же CPU завершаются после операции acquire. Оно также гарантирует, что все
po-более поздние операции сохранения на этом же CPU должны распространиться на
все остальные CPU после выполнения операции acquire. Это реализуется с помощью
smp_acquire__after_ctrl_dep().

Управляющая зависимость (при успехе) для счётчиков ссылок гарантирует, что если
ссылка на объект была успешно получена (произошёл инкремент или добавление к
счётчику ссылок, функция вернула true), то дальнейшие операции сохранения
упорядочены относительно этой операции. Управляющие зависимости для операций
сохранения реализуются не с помощью каких-либо явных барьеров, а опираются на
то, что CPU не выполняет спекуляцию для операций сохранения. Это отношение
действует только в пределах одного CPU и не даёт никаких гарантий для других CPU.


Сравнение функций
=================

случай 1) — операции, не относящиеся к «Read/Modify/Write» (RMW)
----------------------------------------------------------------

Изменения функций:

 * atomic_set() --> refcount_set()
 * atomic_read() --> refcount_read()

Изменения гарантий упорядочивания обращений к памяти:

 * нет (обе полностью неупорядочены)


случай 2) — операции, не относящиеся к «Read/Modify/Write» (RMW), с упорядочиванием release
-------------------------------------------------------------------------------------------

Изменения функций:

 * atomic_set_release() --> refcount_set_release()

Изменения гарантий упорядочивания обращений к памяти:

 * нет (обе обеспечивают упорядочивание RELEASE)


случай 3) — операции на основе инкремента, не возвращающие значение
-------------------------------------------------------------------

Изменения функций:

 * atomic_inc() --> refcount_inc()
 * atomic_add() --> refcount_add()

Изменения гарантий упорядочивания обращений к памяти:

 * нет (обе полностью неупорядочены)

случай 4) — RMW-операции на основе декремента, не возвращающие значение
-----------------------------------------------------------------------

Изменения функций:

 * atomic_dec() --> refcount_dec()

Изменения гарантий упорядочивания обращений к памяти:

 * полностью неупорядочено --> упорядочивание RELEASE


случай 5) — RMW-операции на основе инкремента, возвращающие значение
--------------------------------------------------------------------

Изменения функций:

 * atomic_inc_not_zero() --> refcount_inc_not_zero()
 * нет атомарного аналога --> refcount_add_not_zero()

Изменения гарантий упорядочивания обращений к памяти:

 * полностью упорядочено --> управляющая зависимость при успехе для операций сохранения

.. note:: Здесь мы фактически исходим из того, что необходимое упорядочивание
   обеспечивается в результате получения указателя на объект!


случай 6) — RMW-операции на основе инкремента с упорядочиванием acquire, возвращающие значение
----------------------------------------------------------------------------------------------

Изменения функций:

 * atomic_inc_not_zero() --> refcount_inc_not_zero_acquire()
 * нет атомарного аналога --> refcount_add_not_zero_acquire()

Изменения гарантий упорядочивания обращений к памяти:

 * полностью упорядочено --> упорядочивание ACQUIRE при успехе


случай 7) — обобщённые RMW-операции на основе декремента dec/sub, возвращающие значение
---------------------------------------------------------------------------------------

Изменения функций:

 * atomic_dec_and_test() --> refcount_dec_and_test()
 * atomic_sub_and_test() --> refcount_sub_and_test()

Изменения гарантий упорядочивания обращений к памяти:

 * полностью упорядочено --> упорядочивание RELEASE + упорядочивание ACQUIRE при успехе


случай 8) прочие RMW-операции на основе декремента, возвращающие значение
-------------------------------------------------------------------------

Изменения функций:

 * нет атомарного аналога --> refcount_dec_if_one()
 * ``atomic_add_unless(&var, -1, 1)`` --> ``refcount_dec_not_one(&var)``

Изменения гарантий упорядочивания обращений к памяти:

 * полностью упорядочено --> упорядочивание RELEASE + управляющая зависимость

.. note:: atomic_add_unless() обеспечивает полное упорядочивание только при успехе.


случай 9) — RMW на основе блокировки
------------------------------------

Изменения функций:

 * atomic_dec_and_lock() --> refcount_dec_and_lock()
 * atomic_dec_and_mutex_lock() --> refcount_dec_and_mutex_lock()

Изменения гарантий упорядочивания обращений к памяти:

 * полностью упорядочено --> упорядочивание RELEASE + управляющая зависимость + удержание
   spin_lock() при успехе
