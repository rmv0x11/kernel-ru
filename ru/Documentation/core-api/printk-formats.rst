Как правильно использовать спецификаторы формата printk
==============================================================

.. _printk-specifiers:

:Author: Randy Dunlap <rdunlap@infradead.org>
:Author: Andrew Murray <amurray@mpc-data.co.uk>


Целочисленные типы
==================

::

	If variable is of Type,		use printk format specifier:
	------------------------------------------------------------
		signed char		%d or %hhx
		unsigned char		%u or %x
		char			%u or %x
		short int		%d or %hx
		unsigned short int	%u or %x
		int			%d or %x
		unsigned int		%u or %x
		long			%ld or %lx
		unsigned long		%lu or %lx
		long long		%lld or %llx
		unsigned long long	%llu or %llx
		size_t			%zu or %zx
		ssize_t			%zd or %zx
		s8			%d or %hhx
		u8			%u or %x
		s16			%d or %hx
		u16			%u or %x
		s32			%d or %x
		u32			%u or %x
		s64			%lld or %llx
		u64			%llu or %llx


Если размер <type> зависит от архитектуры (например, cycles_t, tcflag_t) или
зависит от опции конфигурации (например, blk_status_t), используйте спецификатор
формата для наибольшего возможного типа и явно приводите значение к нему.

Пример::

	printk("test: latency: %llu cycles\n", (unsigned long long)time);

Напоминание: sizeof() возвращает тип size_t.

Реализация printf в ядре не поддерживает %n. Форматы для чисел с плавающей запятой
(%e, %f, %g, %a) также не распознаются по очевидным причинам. Использование любого
неподдерживаемого спецификатора или квалификатора длины приводит к WARN и раннему
возврату из vsnprintf().

Типы указателей
===============

Значение «сырого» указателя можно вывести с помощью %p, при этом адрес будет
хэширован перед выводом. Ядро также поддерживает расширенные спецификаторы для
вывода указателей различных типов.

Некоторые из расширенных спецификаторов выводят данные по заданному адресу вместо
вывода самого адреса. В этом случае вместо недоступной информации могут быть
выведены следующие сообщения об ошибках::

	(null)	 data on plain NULL address
	(efault) data on invalid address
	(einval) invalid data on a valid address

Обычные указатели
------------------

::

	%p	abcdef12 or 00000000abcdef12

Указатели, выводимые без расширения спецификатора (то есть «голым» %p),
хэшируются, чтобы предотвратить утечку информации о компоновке памяти ядра. Это
даёт дополнительное преимущество — предоставление уникального идентификатора. На
64-битных машинах первые 32 бита обнуляются. Ядро будет выводить ``(ptrval)`` до
тех пор, пока не накопит достаточно энтропии.

Когда это возможно, используйте специализированные модификаторы, такие как %pS или
%pB (описанные ниже), чтобы избежать необходимости предоставлять нехэшированный
адрес, который придётся интерпретировать постфактум. Если это невозможно, а цель
вывода адреса — предоставить больше информации для отладки, используйте %p и
загружайте ядро с параметром ``no_hash_pointers`` во время отладки, что приведёт к
выводу всех адресов %p без изменений. Если вы *действительно* всегда хотите видеть
неизменённый адрес, см. %px ниже.

Если (и только если) вы выводите адреса как содержимое виртуального файла,
например, в procfs или sysfs (используя, например, seq_printf(), а не printk()),
читаемого процессом из пространства пользователя, используйте модификатор %pK,
описанный ниже, вместо %p или %px.

Указатели на ошибку (Error Pointers)
------------------------------------

::

	%pe	-ENOSPC

Для вывода указателей на ошибку (то есть указателей, для которых IS_ERR() истинно)
в виде символического имени ошибки. Значения ошибок, для которых неизвестно
символическое имя, выводятся в десятичном виде, тогда как значение, не являющееся
ERR_PTR и переданное в качестве аргумента %pe, обрабатывается как обычный %p.

Указатели на символы/функции
----------------------------

::

	%pS	versatile_init+0x0/0x110
	%ps	versatile_init
	%pSR	versatile_init+0x9/0x110
		(with __builtin_extract_return_addr() translation)
	%pB	prev_fn_of_versatile_init+0x88/0x88


Спецификаторы ``S`` и ``s`` используются для вывода указателя в символическом
формате. Они дают имя символа со смещениями (S) или без них (s). Если KALLSYMS
отключены, то вместо этого выводится адрес символа.

Спецификатор ``B`` даёт имя символа со смещениями и должен использоваться при
выводе трассировок стека (stack backtraces). Спецификатор учитывает эффект
оптимизаций компилятора, которые могут происходить при использовании хвостовых
вызовов (tail-calls), помеченных атрибутом noreturn компилятора GCC.

Если указатель находится внутри модуля, после имени символа выводится имя модуля и,
при необходимости, build ID — для этого к концу спецификатора добавляется
дополнительный ``b``.

::

	%pS	versatile_init+0x0/0x110 [module_name]
	%pSb	versatile_init+0x0/0x110 [module_name ed5019fdf5e53be37cb1ba7899292d7e143b259e]
	%pSRb	versatile_init+0x9/0x110 [module_name ed5019fdf5e53be37cb1ba7899292d7e143b259e]
		(with __builtin_extract_return_addr() translation)
	%pBb	prev_fn_of_versatile_init+0x88/0x88 [module_name ed5019fdf5e53be37cb1ba7899292d7e143b259e]

Зондированные указатели из BPF / трассировки
--------------------------------------------

::

	%pks	kernel string
	%pus	user string

Спецификаторы ``k`` и ``u`` используются для вывода ранее зондированной памяти либо
из памяти ядра (k), либо из памяти пользователя (u). Следующий за ними спецификатор
``s`` приводит к выводу строки. При непосредственном использовании в обычном
vsnprintf() аннотации (k) и (u) игнорируются, однако при использовании, например, из
bpf_trace_printk() в BPF, происходит чтение памяти, на которую указывает указатель,
без возникновения исключения (faulting).

Указатели ядра
--------------

::

	%pK	01234567 or 0123456789abcdef

Для вывода указателей ядра, которые должны быть скрыты от непривилегированных
пользователей. Поведение %pK зависит от sysctl kptr_restrict — подробнее см.
Documentation/admin-guide/sysctl/kernel.rst.

Этот модификатор предназначен *только* для случая, когда формируется содержимое
файла, читаемого из пространства пользователя, например, из procfs или sysfs, но не
для dmesg. Обратитесь, пожалуйста, к разделу про %p выше, где обсуждается, как
управлять хэшированием указателей в printk().

Неизменённые адреса
-------------------

::

	%px	01234567 or 0123456789abcdef

Для вывода указателей, когда вы *действительно* хотите вывести адрес. Прежде чем
выводить указатели с помощью %px, подумайте, пожалуйста, не приводит ли это к утечке
конфиденциальной информации о компоновке памяти ядра. %px функционально эквивалентен
%lx (или %lu). %px предпочтительнее, потому что его удобнее однозначно искать через
grep. Если в будущем нам понадобится изменить способ, которым ядро обрабатывает
вывод указателей, мы будем лучше подготовлены для поиска мест вызова.

Прежде чем использовать %px, подумайте, не достаточно ли использовать %p вместе с
включением параметра ядра ``no_hash_pointers`` во время сеансов отладки (см.
описание %p выше). Одним из допустимых сценариев для %px может быть вывод информации
непосредственно перед паникой (panic), что в любом случае предотвращает
использование конфиденциальной информации злоумышленником, и при использовании %px
не будет необходимости воспроизводить панику с no_hash_pointers.

Разности указателей
-------------------

::

	%td	2560
	%tx	a00

Для вывода разностей указателей используйте модификатор %t для ptrdiff_t.

Пример::

	printk("test: difference between pointers: %td\n", ptr2 - ptr1);

Структуры Resource
------------------

::

	%pr	[mem 0x60000000-0x6fffffff flags 0x2200] or
		[mem 0x60000000 flags 0x2200] or
		[mem 0x0000000060000000-0x000000006fffffff flags 0x2200]
		[mem 0x0000000060000000 flags 0x2200]
	%pR	[mem 0x60000000-0x6fffffff pref] or
		[mem 0x60000000 pref] or
		[mem 0x0000000060000000-0x000000006fffffff pref]
		[mem 0x0000000060000000 pref]

Для вывода структур resource. Спецификаторы ``R`` и ``r`` дают выведенный ресурс с
декодированным полем флагов (R) или без него (r).  Если start равно end, выводится
только значение start.

Передаётся по ссылке.

Для вывода структуры range.  struct range содержит произвольный диапазон значений

«Сырой» буфер как экранированная строка
---------------------------------------

::

	%*pE[achnops]

Для вывода «сырого» буфера в виде экранированной строки. Для следующего буфера::

		1b 62 20 5c 43 07 22 90 0d 5d

Несколько примеров показывают, как будет выполнено преобразование (без окружающих
кавычек)::

		%*pE		"\eb \C\a"\220\r]"
		%*pEhp		"\x1bb \C\x07"\x90\x0d]"
		%*pEa		"\e\142\040\\\103\a\042\220\r\135"

Правила преобразования применяются в соответствии с необязательной комбинацией
флагов (подробности см. в документации ядра для :c:func:`string_escape_mem`):

	- a - ESCAPE_ANY
	- c - ESCAPE_SPECIAL
	- h - ESCAPE_HEX
	- n - ESCAPE_NULL
	- o - ESCAPE_OCTAL
	- p - ESCAPE_NP
	- s - ESCAPE_SPACE

По умолчанию используется ESCAPE_ANY_NP.

ESCAPE_ANY_NP — разумный выбор для многих случаев, в частности для вывода SSID.

Если ширина поля опущена, экранируется только 1 байт.

«Сырой» буфер как шестнадцатеричная строка
------------------------------------------

::

	%*ph	00 01 02  ...  3f
	%*phC	00:01:02: ... :3f
	%*phD	00-01-02- ... -3f
	%*phN	000102 ... 3f

Для вывода небольших буферов (длиной до 64 байт) в виде шестнадцатеричной строки с
определённым разделителем. Для буферов большего размера рассмотрите использование
:c:func:`print_hex_dump`.

Адреса MAC/FDDI
---------------

::

	%pM	00:01:02:03:04:05
	%pMR	05:04:03:02:01:00
	%pMF	00-01-02-03-04-05
	%pm	000102030405
	%pmR	050403020100

Для вывода 6-байтовых адресов MAC/FDDI в шестнадцатеричной нотации. Спецификаторы
``M`` и ``m`` дают выведенный адрес с разделителями байтов (M) или без них (m).
Разделитель байтов по умолчанию — двоеточие (:).

Что касается адресов FDDI, после спецификатора ``M`` может использоваться
спецификатор ``F``, чтобы использовать в качестве разделителей дефисы (-) вместо
разделителя по умолчанию.

Для адресов Bluetooth после спецификатора ``M`` следует использовать спецификатор
``R``, чтобы использовать обратный порядок байтов, удобный для визуальной
интерпретации адресов Bluetooth, которые представлены в порядке little endian.

Передаётся по ссылке.

Адреса IPv4
-----------

::

	%pI4	1.2.3.4
	%pi4	001.002.003.004
	%p[Ii]4[hnbl]

Для вывода десятичных адресов IPv4, разделённых точками. Спецификаторы ``I4`` и
``i4`` дают выведенный адрес с ведущими нулями (i4) или без них (I4).

Дополнительные спецификаторы ``h``, ``n``, ``b`` и ``l`` используются для задания
адресов в порядке host, network, big или little endian соответственно. Если
спецификатор не указан, используется порядок по умолчанию — network/big endian.

Передаётся по ссылке.

Адреса IPv6
-----------

::

	%pI6	0001:0002:0003:0004:0005:0006:0007:0008
	%pi6	00010002000300040005000600070008
	%pI6c	1:2:3:4:5:6:7:8

Для вывода 16-битных шестнадцатеричных адресов IPv6 в сетевом порядке байтов.
Спецификаторы ``I6`` и ``i6`` дают выведенный адрес с разделителями-двоеточиями (I6)
или без них (i6). Ведущие нули используются всегда.

Дополнительный спецификатор ``c`` может использоваться со спецификатором ``I`` для
вывода сжатого адреса IPv6, как описано в
https://tools.ietf.org/html/rfc5952

Передаётся по ссылке.

Адреса IPv4/IPv6 (универсальные, с портом, flowinfo, scope)
-----------------------------------------------------------

::

	%pIS	1.2.3.4		or 0001:0002:0003:0004:0005:0006:0007:0008
	%piS	001.002.003.004	or 00010002000300040005000600070008
	%pISc	1.2.3.4		or 1:2:3:4:5:6:7:8
	%pISpc	1.2.3.4:12345	or [1:2:3:4:5:6:7:8]:12345
	%p[Ii]S[pfschnbl]

Для вывода IP-адреса без необходимости различать, относится ли он к типу AF_INET
или AF_INET6. Этому спецификатору формата можно передать указатель на корректную
структуру sockaddr, задаваемую через ``IS`` или ``iS``.

Дополнительные спецификаторы ``p``, ``f`` и ``s`` используются для задания порта
(IPv4, IPv6), flowinfo (IPv6) и scope (IPv6). Порты имеют префикс ``:``, flowinfo —
``/``, а scope — ``%``, за каждым из которых следует фактическое значение.

В случае адреса IPv6 используется сжатый адрес IPv6, как описано в
https://tools.ietf.org/html/rfc5952, если задан дополнительный спецификатор ``c``.
Адрес IPv6 окружается ``[``, ``]`` в случае дополнительных спецификаторов ``p``,
``f`` или ``s``, как предложено в
https://tools.ietf.org/html/draft-ietf-6man-text-addr-representation-07

В случае адресов IPv4 также могут использоваться дополнительные спецификаторы
``h``, ``n``, ``b`` и ``l``, и они игнорируются в случае адреса IPv6.

Передаётся по ссылке.

Дополнительные примеры::

	%pISfc		1.2.3.4		or [1:2:3:4:5:6:7:8]/123456789
	%pISsc		1.2.3.4		or [1:2:3:4:5:6:7:8]%1234567890
	%pISpfc		1.2.3.4:12345	or [1:2:3:4:5:6:7:8]:12345/123456789

Адреса UUID/GUID
----------------

::

	%pUb	00010203-0405-0607-0809-0a0b0c0d0e0f
	%pUB	00010203-0405-0607-0809-0A0B0C0D0E0F
	%pUl	03020100-0504-0706-0809-0a0b0c0e0e0f
	%pUL	03020100-0504-0706-0809-0A0B0C0E0E0F

Для вывода 16-байтовых адресов UUID/GUID. Дополнительные спецификаторы ``l``,
``L``, ``b`` и ``B`` используются для задания порядка little endian в
шестнадцатеричной нотации в нижнем (l) или верхнем регистре (L) — и порядка big
endian в шестнадцатеричной нотации в нижнем (b) или верхнем регистре (B).

Если дополнительные спецификаторы не используются, будет выведен порядок по
умолчанию — big endian с шестнадцатеричной нотацией в нижнем регистре.

Передаётся по ссылке.

Имена dentry
------------

::

	%pd{,2,3,4}
	%pD{,2,3,4}

Для вывода имени dentry; если возникает гонка с :c:func:`d_move`, имя может быть
смесью старого и нового, но это не вызовет oops.  %pd dentry — более безопасный
эквивалент %s dentry->d_name.name, который мы использовали раньше; %pd<n> выводит
``n`` последних компонентов.  %pD делает то же самое для struct file.

Передаётся по ссылке.

Имена block_device
------------------

::

	%pg	sda, sda1 or loop0p1

Для вывода имени указателей block_device.

struct va_format
----------------

::

	%pV

Для вывода структур struct va_format. Они содержат строку формата и va_list
следующим образом::

	struct va_format {
		const char *fmt;
		va_list *va;
	};

Реализует «рекурсивный vsnprintf».

Не используйте эту возможность без какого-либо механизма проверки корректности
строки формата и аргументов va_list.

Передаётся по ссылке.

Узлы дерева устройств (device tree)
-----------------------------------

::

	%pOF[fnpPcCF]


Для вывода структур узлов дерева устройств. Поведение по умолчанию эквивалентно
%pOFf.

	- f - device node full_name
	- n - device node name
	- p - device node phandle
	- P - device node path spec (name + @unit)
	- F - device node flags
	- c - major compatible string
	- C - full compatible string

Разделитель при использовании нескольких аргументов — ':'

Примеры::

	%pOF	/foo/bar@0			- Node full name
	%pOFf	/foo/bar@0			- Same as above
	%pOFfp	/foo/bar@0:10			- Node full name + phandle
	%pOFfcF	/foo/bar@0:foo,device:--P-	- Node full name +
	                                          major compatible string +
						  node flags
							D - dynamic
							d - detached
							P - Populated
							B - Populated bus

Передаётся по ссылке.

Дескрипторы fwnode
------------------

::

	%pfw[fP]

Для вывода информации об fwnode_handle. По умолчанию выводится полное имя узла,
включая путь. Модификаторы функционально эквивалентны %pOF выше.

	- f - full name of the node, including the path
	- P - the name of the node including an address (if there is one)

Примеры (ACPI)::

	%pfwf	\_SB.PCI0.CIO2.port@1.endpoint@0	- Full node name
	%pfwP	endpoint@0				- Node name

Примеры (OF)::

	%pfwf	/ocp@68000000/i2c@48072000/camera@10/port/endpoint - Full name
	%pfwP	endpoint				- Node name

Время и дата
------------

::

	%pt[RT]			YYYY-mm-ddTHH:MM:SS
	%pt[RT]s		YYYY-mm-dd HH:MM:SS
	%pt[RT]d		YYYY-mm-dd
	%pt[RT]t		HH:MM:SS
	%ptSp			<seconds>.<nanoseconds>
	%pt[RST][dt][r][s]

Для вывода даты и времени, представленных::

	R  content of struct rtc_time
	S  content of struct timespec64
	T  time64_t type

в человекочитаемом формате.

По умолчанию год будет увеличен на 1900, а месяц — на 1. Используйте %pt[RT]r (raw),
чтобы подавить это поведение.

%pt[RT]s (space) переопределяет разделитель ISO 8601, используя ' ' (пробел) вместо
'T' (заглавная T) между датой и временем. Он не будет иметь эффекта, если дата или
время опущены.

%ptSp эквивалентен %lld.%09ld для содержимого структуры struct timespec64. Когда
заданы другие спецификаторы, он становится соответствующим эквивалентом
%ptT[dt][r][s].%09ld. Другими словами, секунды выводятся в человекочитаемом формате,
за которым следует точка и наносекунды.

Передаётся по ссылке.

struct clk
----------

::

	%pC	pll1

Для вывода структур struct clk. %pC выводит имя тактового сигнала (Common Clock
Framework) или уникальный 32-битный ID (унаследованная clock framework).

Передаётся по ссылке.

bitmap и его производные, такие как cpumask и nodemask
------------------------------------------------------

::

	%*pb	0779
	%*pbl	0,3-6,8-10

Для вывода bitmap и его производных, таких как cpumask и nodemask, %*pb выводит
bitmap с шириной поля, равной числу битов, а %*pbl выводит bitmap в виде списка
диапазонов с шириной поля, равной числу битов.

Ширина поля передаётся по значению, bitmap передаётся по ссылке. Доступны
вспомогательные макросы cpumask_pr_args() и nodemask_pr_args() для облегчения вывода
cpumask и nodemask.

Битовые поля флагов, такие как флаги страниц и gfp_flags
--------------------------------------------------------

::

	%pGp	0x17ffffc0002036(referenced|uptodate|lru|active|private|node=0|zone=2|lastcpupid=0x1fffff)
	%pGg	GFP_USER|GFP_DMA32|GFP_NOWARN
	%pGv	read|exec|mayread|maywrite|mayexec|denywrite

Для вывода битовых полей флагов в виде набора символических констант, которые
составили бы данное значение. Тип флагов задаётся третьим символом. В настоящее
время поддерживаются:

        - p - [p]age flags, ожидает значение типа (``unsigned long *``)
        - v - [v]ma_flags, ожидает значение типа (``unsigned long *``)
        - g - [g]fp_flags, ожидает значение типа (``gfp_t *``)

Имена флагов и порядок их вывода зависят от конкретного типа.

Обратите внимание, что этот формат не следует использовать напрямую в части
:c:func:`TP_printk()` трассировочной точки (tracepoint). Вместо этого используйте
функции show_*_flags() из <trace/events/mmflags.h>.

Передаётся по ссылке.

Код FourCC V4L2 и DRM (формат пикселей)
---------------------------------------

::

	%p4cc

Выводит код FourCC, используемый V4L2 или DRM, включая порядок байтов формата и его
числовое значение в шестнадцатеричном виде.

Передаётся по ссылке.

Примеры::

	%p4cc	BG12 little-endian (0x32314742)
	%p4cc	Y10  little-endian (0x20303159)
	%p4cc	NV12 big-endian (0xb231564e)

Универсальный код FourCC
------------------------

::
	%p4c[h[R]lb]	gP00 (0x67503030)

Выводит универсальный код FourCC — как в виде ASCII-символов, так и в виде его
числового значения в шестнадцатеричном виде.

Универсальный код FourCC всегда выводится в формате big-endian, со старшего байта.
Это противоположно FourCC для V4L/DRM.

Дополнительные спецификаторы ``h``, ``hR``, ``l`` и ``b`` определяют, какой порядок
байтов используется для загрузки сохранённых байтов. Данные могут интерпретироваться
с использованием порядка байтов хоста, обратного порядка байтов хоста, little-endian
или big-endian.

Передаётся по ссылке.

Примеры для машины с порядком байтов little-endian, при заданном &(u32)0x67503030::

	%p4ch	gP00 (0x67503030)
	%p4chR	00Pg (0x30305067)
	%p4cl	gP00 (0x67503030)
	%p4cb	00Pg (0x30305067)

Примеры для машины с порядком байтов big-endian, при заданном &(u32)0x67503030::

	%p4ch	gP00 (0x67503030)
	%p4chR	00Pg (0x30305067)
	%p4cl	00Pg (0x30305067)
	%p4cb	gP00 (0x67503030)
