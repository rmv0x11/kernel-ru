==========================================
Механизм параллельного выполнения padata
==========================================

:Date: May 2020

Padata — это механизм, с помощью которого ядро может распределять задания для
параллельного выполнения на нескольких CPU, при необходимости сохраняя их
порядок.

Изначально он был разработан для IPsec, которому необходимо выполнять
шифрование и расшифровку большого количества пакетов без их переупорядочивания.
В настоящее время это единственный потребитель поддержки сериализованных
заданий в padata.

Padata также поддерживает многопоточные задания, равномерно разделяя задание и
при этом балансируя нагрузку и координируя работу между потоками.

Выполнение сериализованных заданий
==================================

Инициализация
-------------

Первый шаг в использовании padata для выполнения сериализованных заданий —
настройка структуры padata_instance для общего управления тем, как должны
выполняться задания::

    #include <linux/padata.h>

    struct padata_instance *padata_alloc(const char *name);

'name' просто идентифицирует экземпляр.

Затем завершите инициализацию padata, выделив padata_shell::

   struct padata_shell *padata_alloc_shell(struct padata_instance *pinst);

padata_shell используется для отправки задания в padata и позволяет независимо
сериализовать серию таких заданий. С padata_instance может быть связан один или
несколько padata_shell, каждый из которых обслуживает отдельную серию заданий.

Изменение cpumask'ов
--------------------

CPU, используемые для выполнения заданий, можно изменить двумя способами:
программно с помощью padata_set_cpumask() или через sysfs. Первый способ
определяется так::

    int padata_set_cpumask(struct padata_instance *pinst, int cpumask_type,
			   cpumask_var_t cpumask);

Здесь cpumask_type — это PADATA_CPU_PARALLEL или PADATA_CPU_SERIAL, где
параллельный cpumask описывает, какие процессоры будут использоваться для
параллельного выполнения заданий, отправленных в этот экземпляр, а
последовательный cpumask определяет, какие процессоры разрешено использовать в
качестве процессора обратного вызова сериализации. cpumask задаёт новый cpumask
для использования.

Для cpumask'ов экземпляра могут существовать файлы sysfs. Например, файлы
pcrypt находятся в /sys/kernel/pcrypt/<instance-name>. В каталоге экземпляра
есть два файла, parallel_cpumask и serial_cpumask, и любой из cpumask'ов можно
изменить, записав битовую маску в файл с помощью echo, например::

    echo f > /sys/kernel/pcrypt/pencrypt/parallel_cpumask

Чтение одного из этих файлов показывает заданный пользователем cpumask, который
может отличаться от «пригодного» (usable) cpumask.

Padata внутренне поддерживает две пары cpumask'ов: заданные пользователем
cpumask'и и «пригодные» (usable) cpumask'и. (Каждая пара состоит из
параллельного и последовательного cpumask.) Заданные пользователем cpumask'и по
умолчанию при выделении экземпляра включают все возможные CPU и могут быть
изменены, как описано выше. Пригодные cpumask'и всегда являются подмножеством
заданных пользователем cpumask'ов и содержат только подключённые (online) CPU
из заданных пользователем масок; именно эти cpumask'и padata фактически
использует. Поэтому допустимо передать padata cpumask, содержащий отключённые
(offline) CPU. Как только отключённый CPU из заданного пользователем cpumask
становится подключённым, padata начнёт его использовать.

Изменение CPU-масок — затратные операции, поэтому не следует выполнять их
слишком часто.

Выполнение задания
------------------

Собственно отправка работы в экземпляр padata требует создания структуры
padata_priv, которая представляет одно задание::

    struct padata_priv {
        /* Other stuff here... */
	void                    (*parallel)(struct padata_priv *padata);
	void                    (*serial)(struct padata_priv *padata);
    };

Эта структура почти наверняка будет встроена в некоторую более крупную
структуру, специфичную для выполняемой работы. Большинство её полей являются
приватными для padata, но при инициализации структуру следует обнулить, а также
предоставить функции parallel() и serial(). Эти функции будут вызываться в
процессе выполнения работы, как мы вскоре увидим.

Отправка задания выполняется с помощью::

    int padata_do_parallel(struct padata_shell *ps,
		           struct padata_priv *padata, int *cb_cpu);

Структуры ps и padata должны быть настроены так, как описано выше; cb_cpu
указывает на предпочтительный CPU, который будет использоваться для финального
обратного вызова по завершении задания; он должен входить в CPU-маску текущего
экземпляра (если нет, указатель cb_cpu обновляется так, чтобы указывать на
фактически выбранный CPU). Возвращаемое значение padata_do_parallel() равно
нулю при успехе, что означает, что задание выполняется. -EBUSY означает, что
кто-то где-то ещё работает с CPU-маской экземпляра, а -EINVAL — это жалоба на
то, что cb_cpu отсутствует в последовательном cpumask, что нет подключённых CPU
в параллельном или последовательном cpumask, либо что экземпляр остановлен.

Каждое задание, отправленное в padata_do_parallel(), в свою очередь, будет
передано ровно в один вызов упомянутой выше функции parallel() на одном CPU,
поэтому настоящий параллелизм достигается отправкой нескольких заданий.
parallel() выполняется с отключёнными программными прерываниями и поэтому не
может засыпать. Функция parallel() получает указатель на структуру padata_priv
в качестве единственного параметра; информация о фактической работе, которую
нужно выполнить, вероятно, получается с помощью container_of() для нахождения
охватывающей структуры.

Обратите внимание, что parallel() не имеет возвращаемого значения; подсистема
padata предполагает, что parallel() возьмёт ответственность за задание начиная
с этого момента. Задание не обязательно должно быть завершено в течение этого
вызова, но если parallel() оставляет незавершённую работу, она должна быть
готова к тому, что её снова вызовут с новым заданием прежде, чем завершится
предыдущее.

Сериализация заданий
--------------------

Когда задание действительно завершается, parallel() (или какая-либо иная
функция, которая фактически завершает работу) должна сообщить об этом padata
вызовом::

    void padata_do_serial(struct padata_priv *padata);

В некоторый момент в будущем padata_do_serial() инициирует вызов функции
serial() в структуре padata_priv. Этот вызов произойдёт на CPU, запрошенном в
первоначальном вызове padata_do_parallel(); он, как и прежде, выполняется с
отключёнными локальными программными прерываниями.
Обратите внимание, что этот вызов может быть отложен на некоторое время,
поскольку код padata тщательно следит за тем, чтобы задания завершались в том
порядке, в котором они были отправлены.

Уничтожение
-----------

Очистка экземпляра padata, как и следует ожидать, включает вызов двух функций
освобождения, соответствующих выделению, в обратном порядке::

    void padata_free_shell(struct padata_shell *ps);
    void padata_free(struct padata_instance *pinst);

Пользователь несёт ответственность за то, чтобы все незавершённые задания были
завершены до вызова любой из приведённых выше функций.

Выполнение многопоточных заданий
================================

Многопоточное задание имеет основной поток и ноль или более вспомогательных
потоков, причём основной поток участвует в задании, а затем ожидает, пока все
вспомогательные потоки не завершатся. padata разбивает задание на единицы,
называемые чанками (chunk), где чанк — это часть задания, которую один поток
завершает за один вызов функции потока.

Чтобы запустить многопоточное задание, пользователю нужно сделать три вещи.
Во-первых, описать задание, определив структуру padata_mt_job, которая описана
в разделе Interface. Сюда входит указатель на функцию потока, которую padata
будет вызывать каждый раз, когда назначает чанк задания потоку. Затем определить
функцию потока, которая принимает три аргумента: ``start``, ``end`` и ``arg``,
где первые два ограничивают диапазон, с которым работает поток, а последний —
указатель на общее состояние задания, если оно есть. Подготовить общее
состояние, которое обычно размещается в стеке основного потока. Наконец, вызвать
padata_do_multithreaded(), которая вернёт управление по завершении задания.
