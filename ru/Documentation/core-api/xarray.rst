======
XArray
======

:Author: Matthew Wilcox

Обзор
=====

XArray — это абстрактный тип данных, который ведёт себя как очень большой
массив указателей. Он удовлетворяет многим из тех же потребностей, что и
хеш или обычный массив с изменяемым размером. В отличие от хеша, он позволяет
осмысленно переходить к следующей или предыдущей записи эффективным по
отношению к кэшу способом. В отличие от массива с изменяемым размером, для
увеличения массива не требуется копировать данные или менять отображения MMU.
Он эффективнее по памяти, лучше распараллеливается и более дружелюбен к кэшу,
чем двусвязный список. Он использует преимущества RCU для выполнения поиска
без блокировки.

Реализация XArray эффективна, когда используемые индексы плотно
сгруппированы; хеширование объекта и использование хеша в качестве индекса
не даст хорошей производительности. XArray оптимизирован для небольших
индексов, но при этом сохраняет хорошую производительность и с большими
индексами. Если ваш индекс может быть больше ``ULONG_MAX``, то XArray —
не тот тип данных, который вам нужен. Самый важный пользователь XArray —
это страничный кэш (page cache).

Обычные указатели можно хранить в XArray напрямую. Они должны быть выровнены
по 4 байтам, что справедливо для любого указателя, возвращаемого kmalloc()
и alloc_page(). Это неверно для произвольных указателей пространства
пользователя, а также для указателей на функции. Вы можете хранить указатели
на статически выделенные объекты, если выравнивание этих объектов составляет
не менее 4.

В XArray также можно хранить целые числа от 0 до ``LONG_MAX``. Сначала их
нужно преобразовать в запись с помощью xa_mk_value().
Когда вы извлекаете запись из XArray, вы можете проверить, является ли она
записью-значением, вызвав xa_is_value(), и преобразовать её обратно в
целое число, вызвав xa_to_value().

Некоторые пользователи хотят помечать тегами указатели, которые они хранят
в XArray. Вы можете вызвать xa_tag_pointer() для создания записи с тегом,
xa_untag_pointer() для превращения помеченной тегом записи обратно в
непомеченный указатель и xa_pointer_tag() для извлечения тега записи.
Помеченные тегом указатели используют те же биты, что применяются для
отличия записей-значений от обычных указателей, поэтому для каждого
конкретного XArray вы должны решить, что вы хотите хранить — записи-значения
или помеченные тегом указатели.

XArray не поддерживает хранение указателей IS_ERR(), поскольку некоторые из
них конфликтуют с записями-значениями или внутренними записями.

Необычная особенность XArray — возможность создавать записи, которые занимают
диапазон индексов. После сохранения поиск по любому индексу из диапазона
вернёт ту же запись, что и поиск по любому другому индексу из диапазона.
Сохранение по любому индексу сохранит запись по всем из них. Записи с
несколькими индексами (multi-index) можно явно разбить на меньшие записи.
Снятие (с помощью xa_erase() или xa_store() с ``NULL``) любой записи
заставит XArray забыть об этом диапазоне.

Обычный API
===========

Начните с инициализации XArray — либо с помощью DEFINE_XARRAY()
для статически выделенных XArray, либо с помощью xa_init() для
динамически выделенных. Свежеинициализированный XArray содержит указатель
``NULL`` по каждому индексу.

Затем вы можете устанавливать записи с помощью xa_store() и получать записи
с помощью xa_load(). xa_store() перезапишет любую запись новой записью и
вернёт предыдущую запись, хранившуюся по этому индексу. Вы можете снять
записи с помощью xa_erase() или установив запись в ``NULL`` с помощью
xa_store(). Нет разницы между записью, в которую никогда не сохраняли,
и записью, которая была стёрта с помощью xa_erase(); запись, в которую
последним сохранили ``NULL``, также эквивалентна, за исключением случая,
когда XArray был инициализирован с ``XA_FLAGS_ALLOC``.

Вы можете условно заменить запись по индексу с помощью
xa_cmpxchg(). Как и cmpxchg(), она завершится успешно только в том случае,
если запись по этому индексу имеет «старое» значение. Она также возвращает
запись, которая была по этому индексу; если она возвращает ту же запись,
что была передана как «старая», значит, xa_cmpxchg() завершилась успешно.

Если вы хотите сохранить новую запись по индексу только в том случае, когда
текущая запись по этому индексу равна ``NULL``, вы можете использовать
xa_insert(), которая возвращает ``-EBUSY``, если запись не пуста.

Вы можете скопировать записи из XArray в обычный массив, вызвав
xa_extract(). Или вы можете перебрать имеющиеся записи в XArray,
вызвав xa_for_each(), xa_for_each_start() или xa_for_each_range().
Возможно, вам будет удобнее использовать xa_find() или xa_find_after()
для перехода к следующей имеющейся записи в XArray.

Вызов xa_store_range() сохраняет одну и ту же запись в диапазоне
индексов. Если вы это сделаете, некоторые из прочих операций будут вести себя
несколько необычно. Например, пометка записи по одному индексу может привести
к тому, что запись окажется помеченной по некоторым, но не по всем остальным
индексам. Сохранение по одному индексу может привести к изменению записи,
извлекаемой по некоторым, но не по всем остальным индексам.

Иногда нужно гарантировать, что последующий вызов xa_store()
не потребует выделения памяти. Функция xa_reserve() сохранит
зарезервированную запись по указанному индексу. Пользователи обычного API
увидят эту запись как содержащую ``NULL``. Если вам не нужно использовать
зарезервированную запись, вы можете вызвать xa_release()
для удаления неиспользуемой записи. Если за это время другой пользователь
сохранил в эту запись что-то, xa_release() не сделает ничего; если же
вместо этого вы хотите, чтобы запись стала ``NULL``, вам следует
использовать xa_erase(). Использование xa_insert() для
зарезервированной записи завершится неудачей.

Если все записи в массиве равны ``NULL``, функция xa_empty()
вернёт ``true``.

Наконец, вы можете удалить все записи из XArray, вызвав
xa_destroy(). Если записи XArray — это указатели, возможно, вы захотите
сначала освободить записи. Сделать это можно, перебрав все имеющиеся
записи в XArray с помощью итератора xa_for_each().

Поисковые метки
---------------

С каждой записью в массиве связаны три бита, называемые метками (marks).
Каждая метка может быть установлена или сброшена независимо от других. Вы
можете перебирать помеченные записи с помощью итератора xa_for_each_marked().

Вы можете узнать, установлена ли метка на записи, с помощью
xa_get_mark(). Если запись не равна ``NULL``, вы можете установить на ней
метку с помощью xa_set_mark() и удалить метку с записи, вызвав
xa_clear_mark(). Вы можете спросить, установлена ли определённая метка
на какой-либо записи в XArray, вызвав xa_marked(). Стирание записи из
XArray приводит к сбросу всех меток, связанных с этой записью.

Установка или сброс метки на любом индексе записи с несколькими индексами
затронет все индексы, охваченные этой записью. Запрос метки по любому
индексу вернёт один и тот же результат.

Нет способа перебрать записи, которые не помечены; структура данных не
позволяет реализовать это эффективно. В настоящее время нет итераторов
для поиска по логическим комбинациям битов (например, перебрать все записи,
у которых установлены и ``XA_MARK_1``, и ``XA_MARK_2``, либо перебрать все
записи, у которых установлены ``XA_MARK_0`` или ``XA_MARK_2``). При наличии
заинтересованного пользователя их можно было бы добавить.

Выделяющие XArray
-----------------

Если вы используете DEFINE_XARRAY_ALLOC() для определения XArray или
инициализируете его, передав ``XA_FLAGS_ALLOC`` в xa_init_flags(),
XArray начинает отслеживать, используются записи или нет.

Вы можете вызвать xa_alloc() для сохранения записи по неиспользуемому
индексу в XArray. Если вам нужно изменять массив из контекста прерывания,
вы можете использовать xa_alloc_bh() или xa_alloc_irq() для отключения
прерываний на время выделения ID.

Использование xa_store(), xa_cmpxchg() или xa_insert() также
пометит запись как выделенную. В отличие от обычного XArray, сохранение
``NULL`` пометит запись как используемую, подобно xa_reserve().
Чтобы освободить запись, используйте xa_erase() (или xa_release(),
если вы хотите освободить запись только в случае, когда она равна ``NULL``).

По умолчанию самая нижняя свободная запись выделяется начиная с 0. Если вы
хотите выделять записи начиная с 1, эффективнее использовать
DEFINE_XARRAY_ALLOC1() или ``XA_FLAGS_ALLOC1``. Если вы хотите
выделять ID вплоть до максимального значения, а затем заворачивать обратно
к самому нижнему свободному ID, вы можете использовать xa_alloc_cyclic().

Вы не можете использовать ``XA_MARK_0`` с выделяющим XArray, поскольку эта
метка используется для отслеживания того, свободна запись или нет. Остальные
метки доступны для вашего использования.

Выделение памяти
----------------

Функции xa_store(), xa_cmpxchg(), xa_alloc(),
xa_reserve() и xa_insert() принимают параметр gfp_t
на случай, если XArray потребуется выделить память для хранения этой записи.
Если запись удаляется, выделение памяти выполнять не нужно, и указанные
флаги GFP будут проигнорированы.

Возможна ситуация, когда память выделить нельзя, особенно если вы передаёте
ограничивающий набор флагов GFP. В этом случае функции возвращают
специальное значение, которое можно превратить в errno с помощью xa_err().
Если вам не нужно точно знать, какая именно ошибка произошла, использование
xa_is_err() немного эффективнее.

Блокировка
----------

При использовании обычного API вам не нужно беспокоиться о блокировке.
XArray использует RCU и внутренний спинлок (spinlock) для синхронизации
доступа:

Блокировка не нужна:
 * xa_empty()
 * xa_marked()

Берёт RCU read lock:
 * xa_load()
 * xa_for_each()
 * xa_for_each_start()
 * xa_for_each_range()
 * xa_find()
 * xa_find_after()
 * xa_extract()
 * xa_get_mark()

Берёт xa_lock внутри себя:
 * xa_store()
 * xa_store_bh()
 * xa_store_irq()
 * xa_insert()
 * xa_insert_bh()
 * xa_insert_irq()
 * xa_erase()
 * xa_erase_bh()
 * xa_erase_irq()
 * xa_cmpxchg()
 * xa_cmpxchg_bh()
 * xa_cmpxchg_irq()
 * xa_store_range()
 * xa_alloc()
 * xa_alloc_bh()
 * xa_alloc_irq()
 * xa_reserve()
 * xa_reserve_bh()
 * xa_reserve_irq()
 * xa_destroy()
 * xa_set_mark()
 * xa_clear_mark()

Предполагает, что xa_lock удерживается на входе:
 * __xa_store()
 * __xa_insert()
 * __xa_erase()
 * __xa_cmpxchg()
 * __xa_alloc()
 * __xa_set_mark()
 * __xa_clear_mark()

Если вы хотите воспользоваться блокировкой для защиты структур данных,
которые вы храните в XArray, вы можете вызвать xa_lock()
перед вызовом xa_load(), затем взять счётчик ссылок на найденный
объект перед вызовом xa_unlock(). Это предотвратит удаление объекта
из массива операциями сохранения в промежутке между поиском объекта и
увеличением refcount. Вы также можете использовать RCU, чтобы избежать
разыменования освобождённой памяти, но объяснение этого выходит за рамки
данного документа.

XArray не отключает прерывания или softirq во время изменения
массива. Читать XArray из контекста прерывания или softirq безопасно,
поскольку блокировка RCU обеспечивает достаточную защиту.

Если, например, вы хотите сохранять записи в XArray в контексте процесса,
а затем стирать их в контексте softirq, вы можете сделать это так::

    void foo_init(struct foo *foo)
    {
        xa_init_flags(&foo->array, XA_FLAGS_LOCK_BH);
    }

    int foo_store(struct foo *foo, unsigned long index, void *entry)
    {
        int err;

        xa_lock_bh(&foo->array);
        err = xa_err(__xa_store(&foo->array, index, entry, GFP_KERNEL));
        if (!err)
            foo->count++;
        xa_unlock_bh(&foo->array);
        return err;
    }

    /* foo_erase() is only called from softirq context */
    void foo_erase(struct foo *foo, unsigned long index)
    {
        xa_lock(&foo->array);
        __xa_erase(&foo->array, index);
        foo->count--;
        xa_unlock(&foo->array);
    }

Если вы собираетесь изменять XArray из контекста прерывания или softirq,
вам нужно инициализировать массив с помощью xa_init_flags(), передав
``XA_FLAGS_LOCK_IRQ`` или ``XA_FLAGS_LOCK_BH``.

Приведённый выше пример также демонстрирует распространённый шаблон, когда
требуется расширить охват xa_lock на стороне сохранения, чтобы защитить
некоторую статистику, связанную с массивом.

Совместное использование XArray с контекстом прерывания также возможно —
либо с помощью xa_lock_irqsave() и в обработчике прерывания, и в контексте
процесса, либо с помощью xa_lock_irq() в контексте процесса и xa_lock()
в обработчике прерывания. Для некоторых более распространённых шаблонов
существуют вспомогательные функции, такие как xa_store_bh(), xa_store_irq(),
xa_erase_bh(), xa_erase_irq(), xa_cmpxchg_bh()
и xa_cmpxchg_irq().

Иногда вам нужно защитить доступ к XArray с помощью мьютекса (mutex),
потому что эта блокировка располагается выше другого мьютекса в иерархии
блокировок. Это не даёт вам права использовать функции вроде __xa_erase()
без взятия xa_lock; xa_lock используется для валидации lockdep и в будущем
будет использоваться для других целей.

Функции __xa_set_mark() и __xa_clear_mark() также
доступны для ситуаций, когда вы ищете запись и хотите атомарно
установить или сбросить метку. В этом случае может быть эффективнее
использовать продвинутый API, поскольку это избавит вас от двукратного
обхода дерева.

Продвинутый API
===============

Продвинутый API предлагает большую гибкость и лучшую производительность
ценой интерфейса, который может быть сложнее в использовании и имеет меньше
средств защиты. Продвинутый API не выполняет блокировку за вас, и от вас
требуется использовать xa_lock при изменении массива. Вы можете выбрать,
использовать ли xa_lock или RCU lock при выполнении операций чтения над
массивом. Вы можете смешивать продвинутые и обычные операции над одним и тем
же массивом; более того, обычный API реализован через продвинутый API.
Продвинутый API доступен только модулям с GPL-совместимой лицензией.

Продвинутый API построен вокруг xa_state. Это непрозрачная структура
данных, которую вы объявляете на стеке с помощью макроса XA_STATE().
Этот макрос инициализирует xa_state, готовый к началу обхода
XArray. Он используется в качестве курсора для поддержания позиции в XArray
и позволяет вам компоновать различные операции вместе без необходимости
начинать каждый раз сверху. Содержимое xa_state защищено
rcu_read_lock() или xas_lock(). Если вам нужно отпустить ту из
этих блокировок, которая защищает ваше состояние и дерево, вы должны вызвать
xas_pause(), чтобы будущие вызовы не полагались на те части состояния,
которые были оставлены незащищёнными.

xa_state также используется для хранения ошибок. Вы можете вызвать
xas_error() для извлечения ошибки. Все операции перед началом работы
проверяют, находится ли xa_state в состоянии ошибки, поэтому вам не нужно
проверять наличие ошибки после каждого вызова; вы можете выполнить несколько
вызовов подряд и проверить только в удобный момент. Единственные
ошибки, которые в настоящее время генерируются самим кодом XArray, — это
``ENOMEM`` и ``EINVAL``, но он поддерживает произвольные ошибки на случай,
если вы захотите вызвать xas_set_err() самостоятельно.

Если xa_state удерживает ошибку ``ENOMEM``, вызов xas_nomem()
попытается выделить больше памяти с помощью указанных флагов gfp и
закэшировать её в xa_state для следующей попытки. Идея в том, что вы берёте
xa_lock, пытаетесь выполнить операцию и отпускаете блокировку. Операция
пытается выделить память, удерживая блокировку, но это с большей вероятностью
завершится неудачей. После того как вы отпустили блокировку, xas_nomem()
может приложить больше усилий для выделения памяти. Она вернёт ``true``,
если повтор операции имеет смысл (т. е. что произошла ошибка памяти *и*
было выделено больше памяти). Если ранее была выделена память, и эта
память не использовалась, и ошибки нет (или есть некоторая ошибка, отличная
от ``ENOMEM``), то она освободит ранее выделенную память.

Внутренние записи
-----------------

XArray резервирует некоторые записи для собственных нужд. Они никогда не
выставляются наружу через обычный API, но при использовании продвинутого API
их можно увидеть. Обычно лучший способ обработать их — передать их
в xas_retry() и повторить операцию, если он возвращает ``true``.

.. flat-table::
   :widths: 1 1 6

   * - Имя
     - Проверка
     - Назначение

   * - Node
     - xa_is_node()
     - Узел XArray. Может быть виден при использовании xa_state с
       несколькими индексами.

   * - Sibling
     - xa_is_sibling()
     - Неканоническая запись для записи с несколькими индексами. Значение
       указывает, в каком слоте этого узла находится каноническая запись.

   * - Retry
     - xa_is_retry()
     - Эта запись в данный момент изменяется потоком, удерживающим
       xa_lock. Узел, содержащий эту запись, может быть освобождён в конце
       этого периода RCU. Вам следует перезапустить поиск с головы
       массива.

   * - Zero
     - xa_is_zero()
     - Нулевые записи (zero) выглядят как ``NULL`` через обычный API, но
       занимают запись в XArray, которую можно использовать для резервирования
       индекса на будущее. Это используется выделяющими XArray для выделенных
       записей, равных ``NULL``.

В будущем могут быть добавлены другие внутренние записи. Насколько это
возможно, они будут обрабатываться xas_retry().

Дополнительная функциональность
-------------------------------

Функция xas_create_range() выделяет всю необходимую память
для хранения каждой записи в диапазоне. Она установит ENOMEM в xa_state,
если не сможет выделить память.

Вы можете использовать xas_init_marks() для сброса меток на записи
в их состояние по умолчанию. Обычно это означает, что все метки сброшены,
если только XArray не помечен ``XA_FLAGS_TRACK_FREE``, в этом случае метка 0
установлена, а все остальные метки сброшены. Замена одной записи на другую
с помощью xas_store() не сбросит метки на этой записи; если вы хотите
сбросить метки, вам следует сделать это явно.

xas_load() пройдёт по xa_state настолько близко к записи,
насколько сможет. Если вы знаете, что xa_state уже пройден до
записи, и вам нужно проверить, что запись не изменилась, вы можете
использовать xas_reload(), чтобы сэкономить вызов функции.

Если вам нужно перейти к другому индексу в XArray, вызовите
xas_set(). Это сбрасывает курсор к вершине дерева, что
обычно приводит к тому, что следующая операция проведёт курсор к нужному
месту в дереве. Если вы хотите перейти к следующему или предыдущему индексу,
вызовите xas_next() или xas_prev(). Установка индекса
не проводит курсор по массиву, поэтому не требует удержания блокировки,
тогда как переход к следующему или предыдущему индексу требует.

Вы можете искать следующую имеющуюся запись с помощью xas_find(). Это
эквивалент и xa_find(), и xa_find_after();
если курсор был проведён до записи, то будет найдена следующая
запись после той, на которую сейчас указывает курсор. Если нет, будет
возвращена запись по индексу xa_state. Использование xas_next_entry()
для перехода к следующей имеющейся записи вместо xas_find() в большинстве
случаев сэкономит вызов функции ценой генерации большего количества
встраиваемого (inline) кода.

Функция xas_find_marked() аналогична. Если xa_state не был
пройден, она вернёт запись по индексу xa_state, если она помечена.
В противном случае она вернёт первую помеченную запись после
записи, на которую указывает xa_state. Функция xas_next_marked()
является эквивалентом xas_next_entry().

При переборе диапазона XArray с помощью xas_for_each()
или xas_for_each_marked() может потребоваться временно остановить
итерацию. Для этой цели существует функция xas_pause().
После того как вы выполнили необходимую работу и хотите продолжить, xa_state
находится в подходящем состоянии для продолжения итерации после записи,
которую вы обработали последней. Если на время итерации у вас отключены
прерывания, то хорошим тоном будет приостанавливать итерацию и вновь включать
прерывания каждые ``XA_CHECK_SCHED`` записей.

Функции xas_get_mark(), xas_set_mark() и xas_clear_mark() требуют, чтобы
курсор xa_state был перемещён в соответствующее место в
XArray; они не сделают ничего, если вы вызвали xas_pause() или xas_set()
непосредственно перед ними.

Вы можете вызвать xas_set_update(), чтобы функция обратного вызова
вызывалась каждый раз, когда XArray обновляет узел. Это используется кодом
workingset страничного кэша для поддержания его списка узлов, которые
содержат только теневые (shadow) записи.

Записи с несколькими индексами
------------------------------

XArray обладает способностью связывать несколько индексов вместе так, чтобы
операции над одним индексом затрагивали все индексы. Например, сохранение
по любому индексу изменит значение записи, извлекаемой по любому индексу.
Установка или сброс метки на любом индексе установит или сбросит метку
на каждом связанном вместе индексе. Текущая реализация
позволяет связывать вместе только диапазоны, являющиеся выровненными
степенями двойки; например, индексы 64-127 можно связать вместе, а 2-6 —
нельзя. Это может сэкономить существенные объёмы памяти; например, связывание
512 записей вместе сэкономит более 4 КБ.

Вы можете создать запись с несколькими индексами с помощью XA_STATE_ORDER()
или xas_set_order(), за которыми следует вызов xas_store().
Вызов xas_load() с xa_state с несколькими индексами проведёт
xa_state к нужному месту в дереве, но возвращаемое значение не
имеет смысла и может оказаться внутренней записью или ``NULL`` даже тогда,
когда в диапазоне есть сохранённая запись. Вызов xas_find_conflict()
вернёт первую запись в диапазоне или ``NULL``, если в диапазоне нет
записей. Итератор xas_for_each_conflict() будет
перебирать каждую запись, перекрывающуюся с указанным диапазоном.

Если xas_load() встречает запись с несколькими индексами, xa_index
в xa_state не изменится. При переборе XArray
или вызове xas_find(), если начальный индекс находится в середине
записи с несколькими индексами, он не будет изменён. Последующие вызовы
или итерации переместят индекс к первому индексу диапазона.
Каждая запись будет возвращена только один раз, независимо от того, сколько
индексов она занимает.

Использование xas_next() или xas_prev() с xa_state с несколькими индексами
не поддерживается. Использование любой из этих функций на записи с несколькими
индексами выявит sibling-записи; вызывающая сторона должна их пропускать.

Сохранение ``NULL`` в любой индекс записи с несколькими индексами установит
запись по каждому индексу в ``NULL`` и расторгнет связь. Запись с несколькими
индексами можно разбить на записи, занимающие меньшие диапазоны, вызвав
xas_split_alloc() без удержания xa_lock, после чего взять блокировку
и вызвать xas_split() либо вызвать xas_try_split() с xa_lock. Разница
между xas_split_alloc()+xas_split() и xas_try_alloc() в том,
что xas_split_alloc() + xas_split() разбивают запись с исходного
порядка до нового порядка за один раз равномерно, тогда как xas_try_split()
итеративно разбивает запись, содержащую индекс, неравномерно.
Например, чтобы разбить запись порядка 9, которая занимает 2^(9-6)=8 слотов,
при условии, что ``XA_CHUNK_SHIFT`` равен 6, xas_split_alloc() + xas_split()
требуют 8 xa_node. xas_try_split() разбивает запись порядка 9 на
2 записи порядка 8, затем разбивает одну запись порядка 8, на основе заданного
индекса, на 2 записи порядка 7, ..., и разбивает одну запись порядка 1 на
2 записи порядка 0. Когда разбивается запись порядка 6 и требуется новый
xa_node, xas_try_split() попытается выделить его, если это возможно.
В результате xas_try_split() потребовался бы лишь 1 xa_node вместо 8.
