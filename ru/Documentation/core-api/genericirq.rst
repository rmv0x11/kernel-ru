=================================================
Универсальная обработка IRQ в Linux
=================================================

:Copyright: |copy| 2005-2010: Thomas Gleixner
:Copyright: |copy| 2005-2006:  Ingo Molnar

Введение
========

Универсальный слой обработки прерываний разработан, чтобы предоставить
полную абстракцию обработки прерываний для драйверов устройств. Он способен
обслуживать все различные типы аппаратных контроллеров прерываний. Драйверы
устройств используют функции универсального API для запроса, включения,
отключения и освобождения прерываний. Драйверам не нужно знать какие-либо
подробности об аппаратуре прерываний, поэтому они могут использоваться на
разных платформах без изменений в коде.

Эта документация предоставлена разработчикам, которые хотят реализовать
подсистему прерываний для своей архитектуры с помощью универсального слоя
обработки IRQ.

Обоснование
===========

Изначальная реализация обработки прерываний в Linux использует
супер-обработчик __do_IRQ(), который способен иметь дело с любым типом
логики прерываний.

Первоначально Russell King выделил различные типы обработчиков, чтобы
построить достаточно универсальный набор для реализации обработчика
прерываний ARM в Linux 2.5/2.6. Он различал:

-  Уровневый тип (Level type)

-  Фронтовой тип (Edge type)

-  Простой тип (Simple type)

В ходе реализации мы выделили ещё один тип:

-  Тип Fast EOI

В мире SMP супер-обработчика __do_IRQ() был выделен ещё один тип:

-  Тип Per CPU

Такая раздельная реализация высокоуровневых обработчиков IRQ позволяет нам
оптимизировать поток обработки прерывания для каждого конкретного типа
прерывания. Это снижает сложность в данном конкретном пути кода и позволяет
оптимизированную обработку заданного типа.

Изначальная общая реализация IRQ использовала структуры hw_interrupt_type
и их обратные вызовы ``->ack``, ``->end`` [и т. д.], чтобы различать
управление потоком в супер-обработчике. Это приводит к смешению логики
потока и низкоуровневой аппаратной логики, а также к ненужному дублированию
кода: например, в i386 есть типы IRQ ``ioapic_level_irq`` и
``ioapic_edge_irq``, которые имеют много общих низкоуровневых деталей, но
различаются обработкой потока.

Более естественная абстракция — это чёткое разделение «потока irq» и
«деталей чипа».

Анализ реализаций подсистем IRQ нескольких архитектур показывает, что
большинство из них могут использовать универсальный набор методов «потока
irq» и нуждаются лишь в добавлении кода, специфичного для уровня чипа.
Разделение также ценно для (под)архитектур, которым требуются специфические
особенности в самом потоке IRQ, но не в деталях чипа, — и тем самым
обеспечивает более прозрачную структуру подсистемы IRQ.

Каждому дескриптору прерывания назначается собственный высокоуровневый
обработчик потока, которым обычно является одна из универсальных реализаций.
(Такая реализация высокоуровневого обработчика потока также упрощает
предоставление демультиплексирующих обработчиков, которые встречаются на
встраиваемых платформах различных архитектур.)

Разделение делает универсальный слой обработки прерываний более гибким и
расширяемым. Например, (под)архитектура может использовать универсальную
реализацию потока IRQ для прерываний «уровневого типа» и добавить
специфичную для (под)архитектуры реализацию «фронтового типа».

Чтобы облегчить переход к новой модели и предотвратить поломку существующих
реализаций, супер-обработчик __do_IRQ() всё ещё доступен. Это на время
приводит к своего рода дуальности. Со временем новая модель должна
использоваться во всё большем числе архитектур, поскольку она позволяет
создавать меньшие и более чистые подсистемы IRQ. Она уже три года является
устаревшей и вот-вот будет удалена.

Слои абстракции
===============

В коде прерываний есть три основных уровня абстракции:

1. Высокоуровневый драйверный API

2. Высокоуровневые обработчики потока IRQ

3. Инкапсуляция аппаратуры на уровне чипа

Поток управления прерываниями
-----------------------------

Каждое прерывание описывается структурой дескриптора прерывания irq_desc.
На прерывание ссылается числовое значение типа 'unsigned int', которое
выбирает соответствующую структуру описания прерывания в массиве структур
дескрипторов. Структура дескриптора содержит информацию о состоянии и
указатели на метод потока прерывания и структуру чипа прерывания, которые
назначены этому прерыванию.

Всякий раз, когда срабатывает прерывание, низкоуровневый код архитектуры
вызывает универсальный код прерываний, вызывая desc->handle_irq(). Эта
высокоуровневая функция обработки IRQ использует только примитивы
desc->irq_data.chip, на которые ссылается назначенная структура дескриптора
чипа.

Высокоуровневый драйверный API
------------------------------

Высокоуровневый драйверный API состоит из следующих функций:

-  request_irq()

-  request_threaded_irq()

-  free_irq()

-  disable_irq()

-  enable_irq()

-  disable_irq_nosync() (только SMP)

-  synchronize_irq() (только SMP)

-  irq_set_irq_type()

-  irq_set_irq_wake()

-  irq_set_handler_data()

-  irq_set_chip()

-  irq_set_chip_data()

Подробности см. в автогенерируемой документации функций.

Высокоуровневые обработчики потока IRQ
--------------------------------------

Универсальный слой предоставляет набор предопределённых методов потока irq:

-  handle_level_irq()

-  handle_edge_irq()

-  handle_fasteoi_irq()

-  handle_simple_irq()

-  handle_percpu_irq()

-  handle_edge_eoi_irq()

-  handle_bad_irq()

Обработчики потока прерываний (либо предопределённые, либо специфичные для
архитектуры) назначаются конкретным прерываниям архитектурой либо во время
загрузки, либо во время инициализации устройства.

Реализации потока по умолчанию
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Вспомогательные функции
^^^^^^^^^^^^^^^^^^^^^^^

Вспомогательные функции вызывают примитивы чипа и используются реализациями
потока по умолчанию. Реализованы следующие вспомогательные функции
(упрощённая выдержка)::

    default_enable(struct irq_data *data)
    {
        desc->irq_data.chip->irq_unmask(data);
    }

    default_disable(struct irq_data *data)
    {
        if (!delay_disable(data))
            desc->irq_data.chip->irq_mask(data);
    }

    default_ack(struct irq_data *data)
    {
        chip->irq_ack(data);
    }

    default_mask_ack(struct irq_data *data)
    {
        if (chip->irq_mask_ack) {
            chip->irq_mask_ack(data);
        } else {
            chip->irq_mask(data);
            chip->irq_ack(data);
        }
    }

    noop(struct irq_data *data)
    {
    }



Реализации обработчиков потока по умолчанию
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Обработчик потока IRQ по умолчанию для уровневого типа
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_level_irq предоставляет универсальную реализацию для прерываний,
срабатывающих по уровню.

Реализуется следующий поток управления (упрощённая выдержка)::

    desc->irq_data.chip->irq_mask_ack();
    handle_irq_event(desc->action);
    desc->irq_data.chip->irq_unmask();


Обработчик потока IRQ по умолчанию для Fast EOI
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_fasteoi_irq предоставляет универсальную реализацию для прерываний,
которым нужен только EOI в конце обработчика.

Реализуется следующий поток управления (упрощённая выдержка)::

    handle_irq_event(desc->action);
    desc->irq_data.chip->irq_eoi();


Обработчик потока IRQ по умолчанию для фронтового типа
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_edge_irq предоставляет универсальную реализацию для прерываний,
срабатывающих по фронту.

Реализуется следующий поток управления (упрощённая выдержка)::

    if (desc->status & running) {
        desc->irq_data.chip->irq_mask_ack();
        desc->status |= pending | masked;
        return;
    }
    desc->irq_data.chip->irq_ack();
    desc->status |= running;
    do {
        if (desc->status & masked)
            desc->irq_data.chip->irq_unmask();
        desc->status &= ~pending;
        handle_irq_event(desc->action);
    } while (desc->status & pending);
    desc->status &= ~running;


Обработчик потока IRQ по умолчанию для простого типа
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_simple_irq предоставляет универсальную реализацию для простых
прерываний.

.. note::

   Простой обработчик потока не вызывает никаких примитивов обработчика/чипа.

Реализуется следующий поток управления (упрощённая выдержка)::

    handle_irq_event(desc->action);


Обработчик потока по умолчанию для per CPU
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_percpu_irq предоставляет универсальную реализацию для прерываний
per CPU.

Прерывания per CPU доступны только на SMP, и обработчик предоставляет
упрощённую версию без блокировок.

Реализуется следующий поток управления (упрощённая выдержка)::

    if (desc->irq_data.chip->irq_ack)
        desc->irq_data.chip->irq_ack();
    handle_irq_event(desc->action);
    if (desc->irq_data.chip->irq_eoi)
        desc->irq_data.chip->irq_eoi();


Обработчик потока IRQ EOI Edge
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_edge_eoi_irq предоставляет извращённую разновидность фронтового
обработчика, которая используется исключительно для того, чтобы укротить
сильно искалеченный контроллер irq на powerpc/cell.

Обработчик потока IRQ для плохих прерываний
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

handle_bad_irq используется для ложных прерываний, которым не назначен
реальный обработчик..

Особенности и оптимизации
~~~~~~~~~~~~~~~~~~~~~~~~~~~

Универсальные функции предназначены для «чистых» архитектур и чипов, у
которых нет специфичных для платформы особенностей обработки IRQ. Если
архитектуре нужно реализовать особенности на уровне «потока», то она может
сделать это, переопределив высокоуровневый обработчик потока irq.

Отложенное отключение прерывания
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Эта выбираемая для каждого прерывания возможность, которая была введена
Russell King в реализации прерываний ARM, не маскирует прерывание на
аппаратном уровне при вызове disable_irq(). Прерывание остаётся включённым
и маскируется в обработчике потока, когда происходит событие прерывания.
Это предотвращает потерю фронтовых прерываний на аппаратуре, которая не
сохраняет событие фронтового прерывания, пока прерывание отключено на
аппаратном уровне. Когда прерывание приходит при установленном флаге
IRQ_DISABLED, прерывание маскируется на аппаратном уровне, а бит
IRQ_PENDING устанавливается. Когда прерывание повторно включается через
enable_irq(), бит pending проверяется, и если он установлен, прерывание
повторно отправляется либо аппаратно, либо через программный механизм
повторной отправки. (Необходимо включить CONFIG_HARDIRQS_SW_RESEND, если
вы хотите использовать возможность отложенного отключения прерывания, а ваша
аппаратура не способна повторно инициировать прерывание.) Отложенное
отключение прерывания не настраивается.

Инкапсуляция аппаратуры на уровне чипа
--------------------------------------

Структура аппаратного дескриптора уровня чипа :c:type:`irq_chip` содержит все
непосредственно относящиеся к чипу функции, которые могут использоваться
реализациями потока irq.

-  ``irq_ack``

-  ``irq_mask_ack`` - Необязательно, рекомендуется для производительности

-  ``irq_mask``

-  ``irq_unmask``

-  ``irq_eoi`` - Необязательно, требуется для обработчиков потока EOI

-  ``irq_retrigger`` - Необязательно

-  ``irq_set_type`` - Необязательно

-  ``irq_set_wake`` - Необязательно

Эти примитивы строго означают именно то, что они говорят: ack означает ACK,
маскирование означает маскирование линии IRQ и т. д. Использование этих
базовых единиц низкоуровневой функциональности остаётся на усмотрение
обработчика(ов) потока.

Точка входа __do_IRQ
====================

Изначальная реализация __do_IRQ() была альтернативной точкой входа для всех
типов прерываний. Её больше не существует.

Этот обработчик оказался непригодным для всей аппаратуры прерываний и поэтому
был переработан с разделением функциональности для прерываний
edge/level/simple/percpu. Это не только функциональная оптимизация. Это также
укорачивает пути кода для прерываний.

Блокировки на SMP
=================

Блокировка регистров чипа остаётся на усмотрение архитектуры, которая
определяет примитивы чипа. Структура для каждого irq защищается через
desc->lock универсальным слоем.

Универсальный чип прерываний
============================

Чтобы избежать копий идентичных реализаций чипов IRQ, ядро предоставляет
настраиваемую реализацию универсального чипа прерываний. Разработчикам
следует внимательно проверить, подходит ли универсальный чип под их нужды,
прежде чем самостоятельно реализовывать ту же функциональность немного
иначе.
