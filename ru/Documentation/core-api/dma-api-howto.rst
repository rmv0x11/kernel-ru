Это руководство для разработчиков драйверов устройств о том, как использовать
DMA API, с примерами псевдокода.  Краткое описание API см. в

Адреса CPU и DMA
================

В DMA API задействовано несколько видов адресов, и важно понимать различия
между ними.

Ядро обычно использует виртуальные адреса.  Любой адрес, возвращаемый
kmalloc(), vmalloc() и подобными интерфейсами, является виртуальным адресом и
может храниться в ``void *``.

Система виртуальной памяти (TLB, таблицы страниц и т. д.) транслирует
виртуальные адреса в физические адреса CPU, которые хранятся как "phys_addr_t"
или "resource_size_t".  Ядро управляет ресурсами устройства, такими как
регистры, в виде физических адресов.  Это адреса в /proc/iomem.  Физический
адрес не пригоден для непосредственного использования драйвером; драйвер должен
использовать ioremap(), чтобы отобразить пространство и получить виртуальный
адрес.

Устройства ввода-вывода используют третий вид адресов: "адрес шины" (bus
address).  Если у устройства есть регистры по MMIO-адресу или если оно
выполняет DMA для чтения или записи системной памяти, то адреса, используемые
устройством, являются адресами шины.  В некоторых системах адреса шины
идентичны физическим адресам CPU, но в общем случае это не так.  IOMMU и
хост-мосты (host bridges) могут создавать произвольные отображения между
физическими адресами и адресами шины.

С точки зрения устройства, DMA использует пространство адресов шины, но оно
может быть ограничено подмножеством этого пространства.  Например, даже если
система поддерживает 64-битные адреса для основной памяти и PCI BAR, она может
использовать IOMMU, так что устройствам нужно использовать только 32-битные
DMA-адреса.

Вот схема и несколько примеров::

               CPU                  CPU                  Bus
             Virtual              Physical             Address
             Address              Address               Space
              Space                Space

            +-------+             +------+             +------+
            |       |             |MMIO  |   Offset    |      |
            |       |  Virtual    |Space |   applied   |      |
          C +-------+ --------> B +------+ ----------> +------+ A
            |       |  mapping    |      |   by host   |      |
  +-----+   |       |             |      |   bridge    |      |   +--------+
  |     |   |       |             +------+             |      |   |        |
  | CPU |   |       |             | RAM  |             |      |   | Device |
  |     |   |       |             |      |             |      |   |        |
  +-----+   +-------+             +------+             +------+   +--------+
            |       |  Virtual    |Buffer|   Mapping   |      |
          X +-------+ --------> Y +------+ <---------- +------+ Z
            |       |  mapping    | RAM  |   by IOMMU
            |       |             |      |
            |       |             |      |
            +-------+             +------+

В процессе перечисления (enumeration) ядро узнаёт об устройствах
ввода-вывода, их MMIO-пространстве и хост-мостах, которые соединяют их с
системой.  Например, если у PCI-устройства есть BAR, ядро считывает адрес шины
(A) из BAR и преобразует его в физический адрес CPU (B).  Адрес B сохраняется в
struct resource и обычно отображается через /proc/iomem.  Когда драйвер
захватывает устройство, он, как правило, использует ioremap(), чтобы отобразить
физический адрес B на виртуальный адрес (C).  Затем он может использовать,
например, ioread32(C), чтобы обращаться к регистрам устройства по адресу шины A.

Если устройство поддерживает DMA, драйвер создаёт буфер с помощью kmalloc() или
подобного интерфейса, который возвращает виртуальный адрес (X).  Система
виртуальной памяти отображает X на физический адрес (Y) в системной RAM.
Драйвер может использовать виртуальный адрес X для доступа к буферу, но само
устройство этого делать не может, потому что DMA не проходит через систему
виртуальной памяти CPU.

В некоторых простых системах устройство может выполнять DMA напрямую по
физическому адресу Y.  Но во многих других есть аппаратный IOMMU, который
транслирует DMA-адреса в физические адреса, например, он транслирует Z в Y.  Это
одна из причин существования DMA API: драйвер может передать виртуальный адрес X
интерфейсу вроде dma_map_single(), который настраивает любое необходимое
отображение IOMMU и возвращает DMA-адрес Z.  Затем драйвер указывает устройству
выполнять DMA по адресу Z, а IOMMU отображает его на буфер по адресу Y в
системной RAM.

Чтобы Linux мог использовать динамическое отображение DMA, ему нужна некоторая
помощь от драйверов, а именно: он должен учитывать, что DMA-адреса должны быть
отображены только на время их фактического использования и сняты с отображения
после завершения передачи DMA.

Приведённый ниже API будет работать, разумеется, даже на платформах, где такого
оборудования нет.

Обратите внимание, что DMA API работает с любой шиной независимо от
архитектуры лежащего в основе микропроцессора. Следует использовать DMA API, а
не специфичный для конкретной шины DMA API, то есть использовать интерфейсы
dma_map_*(), а не интерфейсы pci_map_*().

Прежде всего следует убедиться, что::

	#include <linux/dma-mapping.h>

присутствует в вашем драйвере, поскольку он предоставляет определение
dma_addr_t.  Этот тип может хранить любой допустимый для платформы DMA-адрес и
должен использоваться везде, где вы храните DMA-адрес, возвращённый функциями
отображения DMA.

Какая память пригодна для DMA?
==============================

Первое, что вам необходимо знать, — это какую память ядра можно использовать со
средствами отображения DMA.  По этому поводу существовал неписаный набор правил,
и данный текст представляет собой попытку наконец-то их записать.

Если вы получили память через постраничный аллокатор (page allocator)
(т. е. __get_free_page*()) или через универсальные аллокаторы памяти
(т. е. kmalloc() или kmem_cache_alloc()), то вы можете выполнять DMA в эту
память и из неё, используя адреса, возвращённые этими функциями.

В частности, это означает, что вы _не_ можете использовать память/адреса,
возвращённые vmalloc(), для DMA.  Можно выполнять DMA в _нижележащую_ память,
отображённую в область vmalloc(), но для этого требуется обход таблиц страниц
для получения физических адресов с последующей трансляцией каждой из этих
страниц обратно в адрес ядра с помощью чего-то вроде __va().  [ EDIT: Update
this when we integrate Gerd Knorr's generic code which does this. ]

Это правило также означает, что для DMA нельзя использовать ни адреса образа
ядра (элементы сегментов data/text/bss), ни адреса образа модуля, ни адреса
стека.  Все они могут быть отображены в совершенно другое место, нежели
остальная физическая память.  Даже если бы эти классы памяти физически могли
работать с DMA, вам пришлось бы обеспечить, чтобы буферы ввода-вывода были
выровнены по границе линии кэша (cacheline-aligned).  Без этого вы столкнулись
бы с проблемами совместного использования линии кэша (повреждение данных) на
CPU с DMA-некогерентными кэшами.  (CPU мог бы записать в одно слово, DMA
записало бы в другое слово в той же линии кэша, и одно из них могло бы быть
перезаписано.)

Кроме того, это означает, что вы не можете взять результат вызова kmap() и
выполнять DMA в него и из него.  Это аналогично vmalloc().

Что насчёт буферов блочного ввода-вывода и сетевых буферов?  Подсистемы
блочного ввода-вывода и сети гарантируют, что используемые ими буферы пригодны
для DMA из них и в них.

Аннотации __dma_from_device_group_begin/end
===========================================

Как пояснялось ранее, когда структура содержит буфер DMA_FROM_DEVICE /
DMA_BIDIRECTIONAL (устройство пишет в память) рядом с полями, в которые пишет
CPU, совместное использование линии кэша между DMA-буфером и полями, в которые
пишет CPU, может привести к повреждению данных на CPU с DMA-некогерентными
кэшами.

Макросы
``__dma_from_device_group_begin(GROUP)/__dma_from_device_group_end(GROUP)``
обеспечивают надлежащее выравнивание, чтобы предотвратить это::

	struct my_device {
		spinlock_t lock1;
		__dma_from_device_group_begin();
		char dma_buffer1[16];
		char dma_buffer2[16];
		__dma_from_device_group_end();
		spinlock_t lock2;
	};

Чтобы изолировать DMA-буфер от соседних полей, используйте
``__dma_from_device_group_begin(GROUP)`` перед первым полем DMA-буфера и
``__dma_from_device_group_end(GROUP)`` после последнего поля DMA-буфера (с тем
же именем GROUP). Это защищает от совместного использования линии кэша как
начало, так и конец буфера.

Параметр GROUP — это необязательный идентификатор, который именует группу
DMA-буферов (на случай, если их несколько в одной и той же структуре)::

	struct my_device {
		spinlock_t lock1;
		__dma_from_device_group_begin(buffer1);
		char dma_buffer1[16];
		__dma_from_device_group_end(buffer1);
		spinlock_t lock2;
		__dma_from_device_group_begin(buffer2);
		char dma_buffer2[16];
		__dma_from_device_group_end(buffer2);
	};

На кэш-когерентных платформах эти макросы разворачиваются в маркеры массива
нулевой длины.  На некогерентных платформах они также обеспечивают минимальное
выравнивание DMA, которое может достигать 128 байт.

.. note::

        Допускается (хотя это и несколько ненадёжно) включать в группу
        дополнительные поля, не предназначенные для DMA от устройства (чтобы
        упаковать структуру плотнее), — но только при условии, что CPU не пишет
        в эти поля, пока какие-либо поля в группе отображены для DMA_FROM_DEVICE
        или DMA_BIDIRECTIONAL.

Возможности адресации DMA
=========================

По умолчанию ядро предполагает, что ваше устройство может адресовать 32 бита
DMA-адресации.  Для устройства, способного работать с 64 битами, это значение
нужно увеличить, а для устройства с ограничениями — уменьшить.

Особое замечание о PCI: спецификация PCI-X требует, чтобы PCI-X-устройства
поддерживали 64-битную адресацию (DAC) для всех транзакций.  И как минимум одна
платформа (SGI SN2) требует 64-битных когерентных выделений для корректной
работы, когда шина ввода-вывода находится в режиме PCI-X.

Для корректной работы вы должны установить маску DMA, чтобы сообщить ядру о
возможностях DMA-адресации вашего устройства.

Это делается вызовом dma_set_mask_and_coherent()::

	int dma_set_mask_and_coherent(struct device *dev, u64 mask);

который установит маску одновременно и для потокового (streaming), и для
когерентного API.  Если у вас есть особые требования, то вместо этого можно
использовать два следующих отдельных вызова:

	Настройка потоковых отображений выполняется вызовом dma_set_mask()::

		int dma_set_mask(struct device *dev, u64 mask);

	Настройка когерентных выделений выполняется вызовом
	dma_set_coherent_mask()::

		int dma_set_coherent_mask(struct device *dev, u64 mask);

Здесь dev — указатель на структуру device вашего устройства, а mask — битовая
маска, описывающая, какие биты адреса поддерживает ваше устройство.  Часто
структура device вашего устройства встроена в специфичную для шины структуру
device вашего устройства.  Например, &pdev->dev — это указатель на структуру
device PCI-устройства (pdev — указатель на структуру PCI-устройства вашего
устройства).

Эти вызовы обычно возвращают ноль, указывая, что ваше устройство может
надлежащим образом выполнять DMA на данной машине с предоставленной вами маской
адреса, но они могут вернуть ошибку, если маска слишком мала, чтобы быть
поддерживаемой в данной системе.  Если возвращено ненулевое значение, ваше
устройство не может надлежащим образом выполнять DMA на этой платформе, и
попытка это сделать приведёт к неопределённому поведению.  Вы не должны
использовать DMA на этом устройстве, если только семейство функций
dma_set_mask не вернуло успех.

Это означает, что в случае неудачи у вас есть два варианта:

1) Использовать какой-либо режим передачи данных без DMA, если это возможно.
2) Проигнорировать это устройство и не инициализировать его.

Рекомендуется, чтобы ваш драйвер выводил сообщение ядра KERN_WARNING, когда
установка маски DMA завершается неудачно.  Таким образом, если пользователь
вашего драйвера сообщит, что производительность плохая или что устройство вообще
не обнаруживается, вы можете попросить у него сообщения ядра, чтобы выяснить
точную причину.

Устройство с 24-битной адресацией сделало бы что-то вроде этого::

	if (dma_set_mask_and_coherent(dev, DMA_BIT_MASK(24))) {
		dev_warn(dev, "mydev: No suitable DMA available\n");
		goto ignore_this_device;
	}

Стандартное устройство с 64-битной адресацией сделало бы что-то вроде этого::

	dma_set_mask_and_coherent(dev, DMA_BIT_MASK(64))

dma_set_mask_and_coherent() никогда не возвращает неудачу при DMA_BIT_MASK(64).
Типичный код ошибки вроде::

	/* Wrong code */
	if (dma_set_mask_and_coherent(dev, DMA_BIT_MASK(64)))
		dma_set_mask_and_coherent(dev, DMA_BIT_MASK(32))

dma_set_mask_and_coherent() никогда не вернёт неудачу при значении больше 32.
Поэтому типичный код вроде::

	/* Recommended code */
	if (support_64bit)
		dma_set_mask_and_coherent(dev, DMA_BIT_MASK(64));
	else
		dma_set_mask_and_coherent(dev, DMA_BIT_MASK(32));

Если устройство поддерживает только 32-битную адресацию для дескрипторов в
когерентных выделениях, но поддерживает полные 64 бита для потоковых
отображений, это выглядело бы так::

	if (dma_set_mask(dev, DMA_BIT_MASK(64))) {
		dev_warn(dev, "mydev: No suitable DMA available\n");
		goto ignore_this_device;
	}

Когерентная маска всегда сможет установить такую же или меньшую маску, чем
потоковая маска. Однако в редком случае, когда драйвер устройства использует
только когерентные выделения, пришлось бы проверять возвращаемое значение
dma_set_coherent_mask().

Наконец, если ваше устройство способно управлять только младшими 24 битами
адреса, вы могли бы сделать что-то вроде::

	if (dma_set_mask(dev, DMA_BIT_MASK(24))) {
		dev_warn(dev, "mydev: 24-bit DMA addressing not available\n");
		goto ignore_this_device;
	}

Когда dma_set_mask() или dma_set_mask_and_coherent() завершается успешно и
возвращает ноль, ядро сохраняет предоставленную вами маску.  Ядро будет
использовать эту информацию позже, когда вы будете создавать отображения DMA.

Существует случай, о котором нам известно в настоящее время и который стоит
упомянуть в этой документации.  Если ваше устройство поддерживает несколько
функций (например, звуковая карта предоставляет функции воспроизведения и
записи) и эти различные функции имеют _разные_ ограничения DMA-адресации, вы,
возможно, захотите проверить каждую маску и предоставить только ту
функциональность, с которой машина может справиться.  Важно, чтобы последний
вызов dma_set_mask() был для наиболее специфичной маски.

Вот псевдокод, показывающий, как это можно сделать::

	#define PLAYBACK_ADDRESS_BITS	DMA_BIT_MASK(32)
	#define RECORD_ADDRESS_BITS	DMA_BIT_MASK(24)

	struct my_sound_card *card;
	struct device *dev;

	...
	if (!dma_set_mask(dev, PLAYBACK_ADDRESS_BITS)) {
		card->playback_enabled = 1;
	} else {
		card->playback_enabled = 0;
		dev_warn(dev, "%s: Playback disabled due to DMA limitations\n",
		       card->name);
	}
	if (!dma_set_mask(dev, RECORD_ADDRESS_BITS)) {
		card->record_enabled = 1;
	} else {
		card->record_enabled = 0;
		dev_warn(dev, "%s: Record disabled due to DMA limitations\n",
		       card->name);
	}

В качестве примера здесь использована звуковая карта, потому что этот тип
PCI-устройств, по-видимому, изобилует ISA-чипами, снабжёнными PCI-фронтендом, и
потому сохраняющими ограничения ISA в виде 16 МБ DMA-адресации.

Типы отображений DMA
====================

Существует два типа отображений DMA:

- Когерентные отображения DMA (coherent), которые обычно отображаются при
  инициализации драйвера, снимаются с отображения в конце и для которых
  оборудование должно гарантировать, что устройство и CPU могут обращаться к
  данным параллельно и будут видеть изменения, сделанные друг другом, без
  какого-либо явного программного сброса (flushing).

  Воспринимайте "coherent" как "синхронный".

  Текущее поведение по умолчанию — возвращать когерентную память в младших
  32 битах пространства DMA.  Однако для совместимости в будущем вам следует
  устанавливать когерентную маску, даже если это значение по умолчанию вполне
  подходит для вашего драйвера.

  Хорошие примеры того, для чего использовать когерентные отображения:

	- Дескрипторы DMA-кольца сетевой карты.
	- Структуры данных команд почтового ящика (mailbox) SCSI-адаптера.
	- Микрокод прошивки устройства, исполняемый из основной памяти.

  Инвариант, который требуется во всех этих примерах, состоит в том, что любая
  запись CPU в память немедленно видна устройству, и наоборот.  Когерентные
  отображения это гарантируют.

  .. important::

	     Когерентная память DMA не отменяет необходимости в правильных
	     барьерах памяти.  CPU может переупорядочивать записи в когерентную
	     память так же, как и в обычную.  Пример: если для устройства важно
	     увидеть, что первое слово дескриптора обновлено раньше второго, вы
	     должны сделать что-то вроде::

		desc->word0 = address;
		wmb();
		desc->word1 = DESC_VALID;

             чтобы получить корректное поведение на всех платформах.

	     Кроме того, на некоторых платформах вашему драйверу может
	     понадобиться сбрасывать буферы записи CPU во многом так же, как ему
	     нужно сбрасывать буферы записи в PCI-мостах (например, путём чтения
	     значения регистра после его записи).

- Потоковые отображения DMA (streaming), которые обычно отображаются для одной
  передачи DMA, снимаются с отображения сразу после неё (если вы не используете
  dma_sync_* ниже) и для которых оборудование может оптимизироваться под
  последовательные обращения.

  Воспринимайте "streaming" как "асинхронный" или "вне домена когерентности".

  Хорошие примеры того, для чего использовать потоковые отображения:

	- Сетевые буферы, передаваемые/принимаемые устройством.
	- Буферы файловой системы, записываемые/читаемые SCSI-устройством.

  Интерфейсы для использования этого типа отображения были спроектированы
  таким образом, чтобы реализация могла выполнять любые оптимизации
  производительности, которые позволяет оборудование.  С этой целью при
  использовании таких отображений вы должны явно указывать, что именно хотите,
  чтобы произошло.

Ни один из типов отображений DMA не имеет ограничений на выравнивание,
исходящих от нижележащей шины, хотя у некоторых устройств такие ограничения
могут быть.  Кроме того, системы с кэшами, не являющимися DMA-когерентными,
будут работать лучше, когда нижележащие буферы не делят линии кэша с другими
данными.


Использование когерентных отображений DMA
=========================================

Чтобы выделить и отобразить большие (порядка PAGE_SIZE) когерентные области
DMA, вам следует сделать::

	dma_addr_t dma_handle;

	cpu_addr = dma_alloc_coherent(dev, size, &dma_handle, gfp);

где device — это ``struct device *``. Этот вызов может выполняться в контексте
прерывания с флагом GFP_ATOMIC.

Size — это длина области, которую вы хотите выделить, в байтах.

Эта функция выделит RAM для данной области, так что она действует аналогично
__get_free_pages() (но принимает size вместо порядка страниц).  Если вашему
драйверу нужны области размером меньше страницы, вы можете предпочесть
использовать интерфейс dma_pool, описанный ниже.

Интерфейсы когерентного отображения DMA по умолчанию возвращают DMA-адрес,
адресуемый 32 битами.  Даже если устройство указывает (через маску DMA), что оно
может адресовать старшие 32 бита, когерентное выделение будет возвращать
DMA-адреса больше 32 бит только в том случае, если когерентная маска DMA была
явно изменена через dma_set_coherent_mask().  Это верно и для интерфейса
dma_pool.

dma_alloc_coherent() возвращает два значения: виртуальный адрес, который вы
можете использовать для доступа к памяти со стороны CPU, и dma_handle, который
вы передаёте карте.

Гарантируется, что и виртуальный адрес CPU, и DMA-адрес выровнены по наименьшему
порядку PAGE_SIZE, который больше или равен запрошенному размеру.  Этот инвариант
существует (например) для того, чтобы гарантировать, что если вы выделяете блок,
который меньше или равен 64 килобайтам, то протяжённость полученного вами буфера
не пересечёт границу 64 КБ.

Чтобы снять с отображения и освободить такую область DMA, вы вызываете::

	dma_free_coherent(dev, size, cpu_addr, dma_handle);

где dev, size те же, что и в вызове выше, а cpu_addr и dma_handle — значения,
которые dma_alloc_coherent() вернула вам.  Эту функцию нельзя вызывать в
контексте прерывания.

Если вашему драйверу нужно много небольших областей памяти, вы можете написать
собственный код для разбиения страниц, возвращаемых dma_alloc_coherent(), либо
можете использовать для этого API dma_pool.  dma_pool похож на kmem_cache, но он
использует dma_alloc_coherent(), а не __get_free_pages().  Кроме того, он
учитывает распространённые аппаратные ограничения на выравнивание, например то,
что головы очередей (queue heads) должны быть выровнены по границам в N байт.

Создайте dma_pool так::

	struct dma_pool *pool;

	pool = dma_pool_create(name, dev, size, align, boundary);

"name" предназначен для диагностики (как имя kmem_cache); dev и size — как
выше.  Аппаратное требование устройства к выравниванию для данного типа данных —
это "align" (которое выражается в байтах и должно быть степенью двойки).  Если у
вашего устройства нет ограничений на пересечение границ, передайте 0 для
boundary; передача 4096 означает, что память, выделенная из этого пула, не
должна пересекать границы в 4 КБ (но в таком случае, возможно, лучше
использовать непосредственно dma_alloc_coherent()).

Выделяйте память из пула DMA так::

	cpu_addr = dma_pool_alloc(pool, flags, &dma_handle);

flags равны GFP_KERNEL, если допускается блокирование (не in_interrupt и не
удерживаются SMP-блокировки), иначе GFP_ATOMIC.  Подобно dma_alloc_coherent(),
этот вызов возвращает два значения, cpu_addr и dma_handle.

Освобождайте память, выделенную из dma_pool, так::

	dma_pool_free(pool, cpu_addr, dma_handle);

где pool — то, что вы передали в dma_pool_alloc(), а cpu_addr и dma_handle —
значения, которые вернула dma_pool_alloc(). Эту функцию можно вызывать в
контексте прерывания.

Уничтожьте dma_pool вызовом::

	dma_pool_destroy(pool);

Убедитесь, что вы вызвали dma_pool_free() для всей памяти, выделенной из пула,
прежде чем уничтожать пул. Эту функцию нельзя вызывать в контексте прерывания.

Направление DMA
===============

Интерфейсы, описанные в последующих частях этого документа, принимают аргумент
направления DMA, который является целым числом и принимает одно из следующих
значений::

 DMA_BIDIRECTIONAL
 DMA_TO_DEVICE
 DMA_FROM_DEVICE
 DMA_NONE

Вам следует указывать точное направление DMA, если оно вам известно.

DMA_TO_DEVICE означает "из основной памяти в устройство"
DMA_FROM_DEVICE означает "из устройства в основную память"
Это направление, в котором перемещаются данные во время передачи DMA.

Вам _настоятельно_ рекомендуется указывать его настолько точно, насколько это
возможно.

Если вы абсолютно не можете знать направление передачи DMA, укажите
DMA_BIDIRECTIONAL.  Это означает, что DMA может идти в любом направлении.
Платформа гарантирует, что вы вправе указать это и что это будет работать, но
это может, например, стоить производительности.

Значение DMA_NONE предназначено для отладки.  Его можно держать в структуре
данных до того, как вы узнаете точное направление, и это поможет отлавливать
случаи, когда ваша логика отслеживания направления не настроила всё правильно.

Ещё одно преимущество точного указания этого значения (помимо потенциальных
специфичных для платформы оптимизаций на его основе) — отладка.  Некоторые
платформы фактически имеют булев флаг разрешения на запись, которым можно
помечать отображения DMA, во многом подобно защитам страниц в адресном
пространстве пользовательской программы.  Такие платформы могут сообщать (и
сообщают) об ошибках в журналах ядра, когда оборудование DMA-контроллера
обнаруживает нарушение настройки разрешения.

Только потоковые отображения указывают направление; когерентные отображения
неявно имеют настройку атрибута направления DMA_BIDIRECTIONAL.

Подсистема SCSI сообщает вам направление для использования в члене
'sc_data_direction' SCSI-команды, над которой работает ваш драйвер.

Для сетевых драйверов всё довольно просто.  Передаваемые пакеты
отображайте/снимайте с отображения со спецификатором направления
DMA_TO_DEVICE.  Для принимаемых пакетов — ровно наоборот:
отображайте/снимайте с отображения со спецификатором направления
DMA_FROM_DEVICE.

Использование потоковых отображений DMA
=======================================

Функции потокового отображения DMA можно вызывать из контекста прерывания.
Существует две версии каждой функции map/unmap: одна отображает/снимает с
отображения одну область памяти, а другая отображает/снимает с отображения
scatterlist.

Чтобы отобразить одну область, вы делаете::

	struct device *dev = &my_dev->dev;
	dma_addr_t dma_handle;
	void *addr = buffer->ptr;
	size_t size = buffer->len;

	dma_handle = dma_map_single(dev, addr, size, direction);
	if (dma_mapping_error(dev, dma_handle)) {
		/*
		 * reduce current DMA mapping usage,
		 * delay and try again later or
		 * reset driver.
		 */
		goto map_error_handling;
	}

а чтобы снять её с отображения::

	dma_unmap_single(dev, dma_handle, size, direction);

Вам следует вызывать dma_mapping_error(), потому что dma_map_single() может
завершиться неудачей и вернуть ошибку.  Это гарантирует, что код отображения
будет работать корректно на всех реализациях DMA без какой-либо зависимости от
особенностей нижележащей реализации. Использование возвращённого адреса без
проверки на ошибки может привести к сбоям, варьирующимся от паники до молчаливого
повреждения данных.  То же относится и к dma_map_page().

Вам следует вызывать dma_unmap_single(), когда активность DMA завершена,
например, из прерывания, которое сообщило вам, что передача DMA завершена.

Использование таких указателей CPU для одиночных отображений имеет недостаток:
вы не можете таким образом ссылаться на память HIGHMEM.  Поэтому существует пара
интерфейсов map/unmap, аналогичная dma_{map,unmap}_single().  Эти интерфейсы
оперируют парами страница/смещение вместо указателей CPU.  В частности::

	struct device *dev = &my_dev->dev;
	dma_addr_t dma_handle;
	struct page *page = buffer->page;
	unsigned long offset = buffer->offset;
	size_t size = buffer->len;

	dma_handle = dma_map_page(dev, page, offset, size, direction);
	if (dma_mapping_error(dev, dma_handle)) {
		/*
		 * reduce current DMA mapping usage,
		 * delay and try again later or
		 * reset driver.
		 */
		goto map_error_handling;
	}

	...

	dma_unmap_page(dev, dma_handle, size, direction);

Здесь "offset" означает байтовое смещение внутри заданной страницы.

Вам следует вызывать dma_mapping_error(), потому что dma_map_page() может
завершиться неудачей и вернуть ошибку, как описано в обсуждении
dma_map_single().

Вам следует вызывать dma_unmap_page(), когда активность DMA завершена,
например, из прерывания, которое сообщило вам, что передача DMA завершена.

При работе со scatterlists вы отображаете область, собранную из нескольких
областей, так::

	int i, count = dma_map_sg(dev, sglist, nents, direction);
	struct scatterlist *sg;

	for_each_sg(sglist, sg, count, i) {
		hw_address[i] = sg_dma_address(sg);
		hw_len[i] = sg_dma_len(sg);
	}

где nents — число элементов в sglist.

Реализация вправе объединять несколько последовательных элементов sglist в один
(например, если отображение DMA выполняется с гранулярностью PAGE_SIZE, любые
последовательные элементы sglist могут быть объединены в один при условии, что
первый заканчивается, а второй начинается на границе страницы — на самом деле
это огромное преимущество для карт, которые либо вообще не умеют делать
scatter-gather, либо имеют очень ограниченное число элементов scatter-gather) и
возвращает фактическое число элементов sg, в которые она их отобразила. В случае
неудачи возвращается 0.

Затем вам следует выполнить цикл count раз (примечание: это может быть меньше,
чем nents раз) и использовать макросы sg_dma_address() и sg_dma_len() там, где
вы ранее обращались к sg->address и sg->length, как показано выше.

Чтобы снять с отображения scatterlist, просто вызовите::

	dma_unmap_sg(dev, sglist, nents, direction);

И снова убедитесь, что активность DMA уже завершена.

.. note::

	Аргумент 'nents' для вызова dma_unmap_sg должен быть *тем же*, который
	вы передали в вызов dma_map_sg, это *НЕ* должно быть значение 'count',
	_возвращённое_ вызовом dma_map_sg.

Каждому вызову dma_map_{single,sg}() должен соответствовать парный вызов
dma_unmap_{single,sg}(), потому что пространство DMA-адресов — это общий
ресурс, и вы можете сделать машину непригодной к использованию, израсходовав
все DMA-адреса.

Если вам нужно использовать одну и ту же потоковую область DMA несколько раз и
обращаться к данным между передачами DMA, буфер нужно правильно
синхронизировать, чтобы CPU и устройство видели наиболее актуальную и
корректную копию DMA-буфера.

Итак, сначала просто отобразите его с помощью dma_map_{single,sg}(), и после
каждой передачи DMA вызывайте либо::

	dma_sync_single_for_cpu(dev, dma_handle, size, direction);

либо::

	dma_sync_sg_for_cpu(dev, sglist, nents, direction);

по обстоятельствам.

Затем, если вы хотите снова дать устройству доступ к области DMA, завершите
обращение к данным со стороны CPU, а потом, прежде чем фактически передать буфер
оборудованию, вызовите либо::

	dma_sync_single_for_device(dev, dma_handle, size, direction);

либо::

	dma_sync_sg_for_device(dev, sglist, nents, direction);

по обстоятельствам.

.. note::

	      Аргумент 'nents' для dma_sync_sg_for_cpu() и
	      dma_sync_sg_for_device() должен быть тем же, что передан в
	      dma_map_sg(). Это _НЕ_ значение count, возвращённое
	      dma_map_sg().

После последней передачи DMA вызовите одну из функций снятия отображения DMA
dma_unmap_{single,sg}(). Если вы не обращаетесь к данным с момента первого
вызова dma_map_*() и до dma_unmap_*(), то вам вообще не нужно вызывать функции
dma_sync_*().

Вот псевдокод, показывающий ситуацию, в которой вам потребовалось бы
использовать интерфейсы dma_sync_*()::

	my_card_setup_receive_buffer(struct my_card *cp, char *buffer, int len)
	{
		dma_addr_t mapping;

		mapping = dma_map_single(cp->dev, buffer, len, DMA_FROM_DEVICE);
		if (dma_mapping_error(cp->dev, mapping)) {
			/*
			 * reduce current DMA mapping usage,
			 * delay and try again later or
			 * reset driver.
			 */
			goto map_error_handling;
		}

		cp->rx_buf = buffer;
		cp->rx_len = len;
		cp->rx_dma = mapping;

		give_rx_buf_to_card(cp);
	}

	...

	my_card_interrupt_handler(int irq, void *devid, struct pt_regs *regs)
	{
		struct my_card *cp = devid;

		...
		if (read_card_status(cp) == RX_BUF_TRANSFERRED) {
			struct my_card_header *hp;

			/* Examine the header to see if we wish
			 * to accept the data.  But synchronize
			 * the DMA transfer with the CPU first
			 * so that we see updated contents.
			 */
			dma_sync_single_for_cpu(&cp->dev, cp->rx_dma,
						cp->rx_len,
						DMA_FROM_DEVICE);

			/* Now it is safe to examine the buffer. */
			hp = (struct my_card_header *) cp->rx_buf;
			if (header_is_ok(hp)) {
				dma_unmap_single(&cp->dev, cp->rx_dma, cp->rx_len,
						 DMA_FROM_DEVICE);
				pass_to_upper_layers(cp->rx_buf);
				make_and_setup_new_rx_buf(cp);
			} else {
				/* CPU should not write to
				 * DMA_FROM_DEVICE-mapped area,
				 * so dma_sync_single_for_device() is
				 * not needed here. It would be required
				 * for DMA_BIDIRECTIONAL mapping if
				 * the memory was modified.
				 */
				give_rx_buf_to_card(cp);
			}
		}
	}

Обработка ошибок
================

На некоторых архитектурах пространство DMA-адресов ограничено, и неудачу
выделения можно определить:

- проверяя, возвращает ли dma_alloc_coherent() значение NULL или возвращает ли
  dma_map_sg значение 0

- проверяя dma_addr_t, возвращённый из dma_map_single() и dma_map_page(), с
  помощью dma_mapping_error()::

	dma_addr_t dma_handle;

	dma_handle = dma_map_single(dev, addr, size, direction);
	if (dma_mapping_error(dev, dma_handle)) {
		/*
		 * reduce current DMA mapping usage,
		 * delay and try again later or
		 * reset driver.
		 */
		goto map_error_handling;
	}

- снимая с отображения уже отображённые страницы, когда ошибка отображения
  возникает в середине попытки отобразить несколько страниц. Эти примеры
  применимы и к dma_map_page().

Пример 1::

	dma_addr_t dma_handle1;
	dma_addr_t dma_handle2;

	dma_handle1 = dma_map_single(dev, addr, size, direction);
	if (dma_mapping_error(dev, dma_handle1)) {
		/*
		 * reduce current DMA mapping usage,
		 * delay and try again later or
		 * reset driver.
		 */
		goto map_error_handling1;
	}
	dma_handle2 = dma_map_single(dev, addr, size, direction);
	if (dma_mapping_error(dev, dma_handle2)) {
		/*
		 * reduce current DMA mapping usage,
		 * delay and try again later or
		 * reset driver.
		 */
		goto map_error_handling2;
	}

	...

	map_error_handling2:
		dma_unmap_single(dma_handle1);
	map_error_handling1:

Пример 2::

	/*
	 * if buffers are allocated in a loop, unmap all mapped buffers when
	 * mapping error is detected in the middle
	 */

	dma_addr_t dma_addr;
	dma_addr_t array[DMA_BUFFERS];
	int save_index = 0;

	for (i = 0; i < DMA_BUFFERS; i++) {

		...

		dma_addr = dma_map_single(dev, addr, size, direction);
		if (dma_mapping_error(dev, dma_addr)) {
			/*
			 * reduce current DMA mapping usage,
			 * delay and try again later or
			 * reset driver.
			 */
			goto map_error_handling;
		}
		array[i].dma_addr = dma_addr;
		save_index++;
	}

	...

	map_error_handling:

	for (i = 0; i < save_index; i++) {

		...

		dma_unmap_single(array[i].dma_addr);
	}

Сетевые драйверы должны вызывать dev_kfree_skb(), чтобы освободить буфер сокета
(socket buffer), и возвращать NETDEV_TX_OK, если отображение DMA завершается
неудачей в хуке передачи (ndo_start_xmit). Это означает, что в случае неудачи
буфер сокета просто отбрасывается.

SCSI-драйверы должны возвращать SCSI_MLQUEUE_HOST_BUSY, если отображение DMA
завершается неудачей в хуке queuecommand. Это означает, что подсистема SCSI
передаёт команду драйверу снова позже.

Оптимизация расхода пространства состояния снятия отображения
=============================================================

На многих платформах dma_unmap_{single,page}() — это попросту nop.  Поэтому
отслеживание адреса и длины отображения — пустая трата места.  Вместо того
чтобы заполнять ваши драйверы ifdef-ами и тому подобным, чтобы "обойти" это
(что свело бы на нет весь смысл переносимого API), предоставляются следующие
средства.

На самом деле, вместо того чтобы описывать макросы один за другим, мы
преобразуем некоторый пример кода.

1) Используйте DEFINE_DMA_UNMAP_{ADDR,LEN} в структурах сохранения состояния.
   Пример, до::

	struct ring_state {
		struct sk_buff *skb;
		dma_addr_t mapping;
		__u32 len;
	};

   после::

	struct ring_state {
		struct sk_buff *skb;
		DEFINE_DMA_UNMAP_ADDR(mapping);
		DEFINE_DMA_UNMAP_LEN(len);
	};

2) Используйте dma_unmap_{addr,len}_set() для установки этих значений.
   Пример, до::

	ringp->mapping = FOO;
	ringp->len = BAR;

   после::

	dma_unmap_addr_set(ringp, mapping, FOO);
	dma_unmap_len_set(ringp, len, BAR);

3) Используйте dma_unmap_{addr,len}() для доступа к этим значениям.
   Пример, до::

	dma_unmap_single(dev, ringp->mapping, ringp->len,
			 DMA_FROM_DEVICE);

   после::

	dma_unmap_single(dev,
			 dma_unmap_addr(ringp, mapping),
			 dma_unmap_len(ringp, len),
			 DMA_FROM_DEVICE);

Это действительно должно быть самоочевидно.  Мы рассматриваем ADDR и LEN
по отдельности, потому что реализации может быть достаточно только адреса для
выполнения операции снятия отображения.

Вопросы, связанные с платформой
===============================

Если вы просто пишете драйверы для Linux и не сопровождаете порт архитектуры
для ядра, вы можете спокойно перейти вниз к разделу "Заключение".

1) Требования к struct scatterlist.

   Вам нужно включить CONFIG_NEED_SG_DMA_LENGTH, если архитектура поддерживает
   IOMMU (включая программный IOMMU).

2) ARCH_DMA_MINALIGN

   Архитектуры должны гарантировать, что буфер, выделенный kmalloc, безопасен
   для DMA. Драйверы и подсистемы полагаются на это. Если архитектура не
   полностью DMA-когерентна (т. е. оборудование не гарантирует, что данные в
   кэше CPU идентичны данным в основной памяти), необходимо установить
   ARCH_DMA_MINALIGN, чтобы аллокатор памяти гарантировал, что буфер,
   выделенный kmalloc, не делит линию кэша с другими.  В качестве примера см.
   arch/arm/include/asm/cache.h.

   Обратите внимание, что ARCH_DMA_MINALIGN касается ограничений на выравнивание
   памяти для DMA. Вам не нужно беспокоиться об ограничениях на выравнивание
   данных, связанных с архитектурой (например, об ограничениях на выравнивание
   64-битных объектов).
