==============================
Горячее подключение памяти
==============================

Уведомитель событий горячего подключения памяти
================================================

События горячего подключения отправляются в очередь уведомлений.

Уведомитель памяти
------------------

В ``include/linux/memory.h`` определено шесть типов уведомлений:

MEM_GOING_ONLINE
  Генерируется перед тем, как новая память становится доступной, чтобы дать
  возможность подсистемам подготовиться к работе с памятью. Аллокатор страниц
  по-прежнему не может выделять память из новой области.

MEM_CANCEL_ONLINE
  Генерируется, если MEM_GOING_ONLINE завершается неудачей.

MEM_ONLINE
  Генерируется, когда память была успешно введена в работу (online). Обработчик
  может выделять страницы из новой памяти.

MEM_GOING_OFFLINE
  Генерируется, чтобы начать процесс вывода памяти из работы (offlining).
  Выделение из этой памяти больше невозможно, но часть памяти, подлежащей
  выводу, всё ещё используется. Обработчик можно использовать для освобождения
  памяти, известной подсистеме, из указанного блока памяти.

MEM_CANCEL_OFFLINE
  Генерируется, если MEM_GOING_OFFLINE завершается неудачей. Память снова
  доступна из блока памяти, который мы пытались вывести из работы.

MEM_OFFLINE
  Генерируется после завершения вывода памяти из работы.

Процедуру-обработчик можно зарегистрировать вызовом::

  hotplug_memory_notifier(callback_func, priority)

Функции-обработчики с более высокими значениями priority вызываются раньше
функций-обработчиков с более низкими значениями.

Функция-обработчик должна иметь следующий прототип::

  int callback_func(
    struct notifier_block *self, unsigned long action, void *arg);

Первый аргумент функции-обработчика (self) — это указатель на блок цепочки
уведомителя, который указывает на саму функцию-обработчик.
Второй аргумент (action) — один из типов событий, описанных выше.
Третий аргумент (arg) передаёт указатель на struct memory_notify::

	struct memory_notify {
		unsigned long start_pfn;
		unsigned long nr_pages;
	}

- start_pfn — это start_pfn вводимой/выводимой памяти.
- nr_pages — это число страниц вводимой/выводимой памяти.

Возможно получить уведомление MEM_CANCEL_ONLINE, не получив перед этим
уведомления MEM_GOING_ONLINE; то же относится к MEM_CANCEL_OFFLINE и
MEM_GOING_OFFLINE.
Это может произойти, когда сбой даёт потребитель: в таком случае мы прерываем
цепочку вызовов и прекращаем вызывать остальных потребителей уведомителя.
Поэтому важно, чтобы пользователи memory_notify не делали никаких допущений и
были готовы обрабатывать такие случаи.

Процедура-обработчик должна возвращать одно из значений
NOTIFY_DONE, NOTIFY_OK, NOTIFY_BAD, NOTIFY_STOP,
определённых в ``include/linux/notifier.h``

NOTIFY_DONE и NOTIFY_OK не влияют на дальнейшую обработку.

NOTIFY_BAD используется как ответ на действие MEM_GOING_ONLINE, MEM_GOING_OFFLINE,
MEM_ONLINE или MEM_OFFLINE для отмены горячего подключения. Он прекращает
дальнейшую обработку очереди уведомлений.

NOTIFY_STOP прекращает дальнейшую обработку очереди уведомлений.

Уведомитель узла NUMA
---------------------

В ``include/linux/node.h`` определено шесть типов уведомлений:

NODE_ADDING_FIRST_MEMORY
 Генерируется перед тем, как память становится доступной этому узлу впервые.

NODE_CANCEL_ADDING_FIRST_MEMORY
 Генерируется, если NODE_ADDING_FIRST_MEMORY завершается неудачей.

NODE_ADDED_FIRST_MEMORY
 Генерируется, когда память стала доступной этому узлу впервые.

NODE_REMOVING_LAST_MEMORY
 Генерируется, когда последняя память, доступная этому узлу, вот-вот будет выведена из работы.

NODE_CANCEL_REMOVING_LAST_MEMORY
 Генерируется, когда NODE_CANCEL_REMOVING_LAST_MEMORY завершается неудачей.

NODE_REMOVED_LAST_MEMORY
 Генерируется, когда последняя память, доступная этому узлу, была выведена из работы.

Процедуру-обработчик можно зарегистрировать вызовом::

  hotplug_node_notifier(callback_func, priority)

Функции-обработчики с более высокими значениями priority вызываются раньше
функций-обработчиков с более низкими значениями.

Функция-обработчик должна иметь следующий прототип::

  int callback_func(

    struct notifier_block *self, unsigned long action, void *arg);

Первый аргумент функции-обработчика (self) — это указатель на блок цепочки
уведомителя, который указывает на саму функцию-обработчик.
Второй аргумент (action) — один из типов событий, описанных выше.
Третий аргумент (arg) передаёт указатель на struct node_notify::

        struct node_notify {
                int nid;
        }

- nid — это узел, в который мы добавляем или из которого удаляем память.

Возможно получить уведомление NODE_CANCEL_ADDING_FIRST_MEMORY, не получив перед
этим уведомления NODE_ADDING_FIRST_MEMORY; то же относится к
NODE_CANCEL_REMOVING_LAST_MEMORY и NODE_REMOVING_LAST_MEMORY.
Это может произойти, когда сбой даёт потребитель: в таком случае мы прерываем
цепочку вызовов и прекращаем вызывать остальных потребителей уведомителя.
Поэтому важно, чтобы пользователи node_notify не делали никаких допущений и
были готовы обрабатывать такие случаи.

Процедура-обработчик должна возвращать одно из значений
NOTIFY_DONE, NOTIFY_OK, NOTIFY_BAD, NOTIFY_STOP,
определённых в ``include/linux/notifier.h``

NOTIFY_DONE и NOTIFY_OK не влияют на дальнейшую обработку.

NOTIFY_BAD используется как ответ на действие NODE_ADDING_FIRST_MEMORY,
NODE_REMOVING_LAST_MEMORY, NODE_ADDED_FIRST_MEMORY или
NODE_REMOVED_LAST_MEMORY для отмены горячего подключения.
Он прекращает дальнейшую обработку очереди уведомлений.

NOTIFY_STOP прекращает дальнейшую обработку очереди уведомлений.

Обратите внимание, что мы не должны допускать сбой для NODE_ADDED_FIRST_MEMORY /
NODE_REMOVED_FIRST_MEMORY, поскольку код memory_hotplug на этом этапе уже не
может выполнить откат.

Внутреннее устройство блокировок
================================

При добавлении/удалении памяти, использующей устройства блоков памяти
(то есть обычного ОЗУ), следует удерживать device_hotplug_lock, чтобы:

- синхронизироваться с запросами на ввод/вывод (online/offline) (например, через
  sysfs). Так устройства блоков памяти могут быть доступны (атрибуты
  .online/.state) из пространства пользователя только после того, как память
  была полностью добавлена. А при удалении памяти мы знаем, что никто не
  находится в критических секциях.
- синхронизироваться с горячим подключением CPU и подобным (например, важно
  для ACPI и PPC)

В частности, существует возможная инверсия блокировок, которой избегают с
помощью device_hotplug_lock, когда при добавлении памяти пространство
пользователя пытается ввести эту память в работу быстрее, чем ожидается:

- device_online() сначала захватывает device_lock(), затем
  mem_hotplug_lock
- add_memory_resource() сначала захватывает mem_hotplug_lock, затем
  device_lock() (при создании устройств, во время bus_add_device()).

Поскольку устройство видно пространству пользователя ещё до захвата
device_lock(), это может привести к инверсии блокировок.

Ввод/вывод (onlining/offlining) памяти следует выполнять через device_online()/
device_offline() — чтобы гарантировать корректную синхронизацию с действиями
через sysfs. Рекомендуется удерживать device_hotplug_lock (например, чтобы
защитить online_type).

При добавлении/удалении/вводе/выводе памяти или добавлении/удалении
гетерогенной/устройственной памяти всегда следует удерживать mem_hotplug_lock в
режиме записи, чтобы сериализовать горячее подключение памяти (например, доступ
к глобальным/зонным переменным).

Кроме того, mem_hotplug_lock (в отличие от device_hotplug_lock) в режиме чтения
позволяет реализовать достаточно эффективные get_online_mems/put_online_mems,
так что код, обращающийся к памяти, может защититься от исчезновения этой памяти.
