===============
DMA и swiotlb
===============

swiotlb — это аллокатор буферов памяти, используемый слоем DMA ядра Linux. Обычно
он применяется, когда устройство, выполняющее DMA, не может напрямую обратиться к
целевому буферу памяти из-за аппаратных ограничений или иных требований. В таком
случае слой DMA вызывает swiotlb для выделения временного буфера памяти,
удовлетворяющего этим ограничениям. DMA выполняется в/из этого временного буфера
памяти, а CPU копирует данные между временным буфером и исходным целевым буфером
памяти. Этот подход в общем случае называется «bounce buffering», а временный
буфер памяти называется «bounce buffer».

Драйверы устройств не взаимодействуют со swiotlb напрямую. Вместо этого драйверы
сообщают слою DMA об атрибутах DMA управляемых ими устройств и используют обычные
API DMA map, unmap и sync при программировании устройства для выполнения DMA. Эти
API применяют атрибуты DMA устройства и общесистемные настройки ядра для
определения того, необходима ли bounce buffering. Если необходима, слой DMA
управляет выделением, освобождением и синхронизацией («sync») bounce-буферов.
Поскольку атрибуты DMA задаются для каждого устройства, в системе одни устройства
могут использовать bounce buffering, а другие — нет.

Так как CPU копирует данные между bounce-буфером и исходным целевым буфером памяти,
выполнение bounce buffering медленнее, чем выполнение DMA напрямую в исходный буфер
памяти, и потребляет больше ресурсов CPU. Поэтому она используется только тогда,
когда это необходимо для обеспечения функциональности DMA.

Сценарии использования
-----------------------
swiotlb изначально была создана для обработки DMA для устройств с ограничениями
адресации. По мере того как размеры физической памяти росли выше 4 GiB, некоторые
устройства могли предоставлять только 32-битные адреса DMA. Выделяя память
bounce-буфера ниже отметки 4 GiB, эти устройства с ограничениями адресации
по-прежнему могли работать и выполнять DMA.

В последнее время виртуальные машины Confidential Computing (CoCo) по умолчанию
имеют зашифрованную память гостевой VM, и эта память недоступна для хостового
гипервизора и VMM. Чтобы хост мог выполнять I/O от имени гостя, I/O должен быть
направлен в незашифрованную гостевую память. CoCo VM устанавливают общесистемную
опцию ядра, принудительно заставляющую весь DMA I/O использовать bounce-буферы,
и память bounce-буфера настраивается как незашифрованная. Хост выполняет DMA I/O
в/из памяти bounce-буфера, а слой DMA ядра Linux выполняет операции «sync», чтобы
заставить CPU копировать данные в/из исходного целевого буфера памяти. Копирование
CPU выступает мостом между незашифрованной и зашифрованной памятью. Такое
использование bounce-буферов позволяет драйверам устройств «просто работать» в
CoCo VM, без каких-либо модификаций, необходимых для обработки сложности
шифрования памяти.

Возникают и другие пограничные сценарии для bounce-буферов. Например, когда
отображения IOMMU настраиваются для операции DMA в/из устройства, которое
считается «недоверенным» («untrusted»), устройству следует предоставлять доступ
только к памяти, содержащей передаваемые данные. Но если эта память занимает лишь
часть гранулы IOMMU, другие части гранулы могут содержать несвязанные данные ядра.
Поскольку управление доступом IOMMU осуществляется на уровне гранулы, недоверенное
устройство может получить доступ к этим несвязанным данным ядра. Эта проблема
решается через bounce buffering операции DMA и обеспечение того, что неиспользуемые
части bounce-буферов не содержат каких-либо несвязанных данных ядра.

Основная функциональность
--------------------------
Основными API swiotlb являются swiotlb_tbl_map_single() и
swiotlb_tbl_unmap_single(). API «map» выделяет bounce-буфер заданного размера в
байтах и возвращает физический адрес буфера. Память буфера является физически
непрерывной. Предполагается, что слой DMA отображает физический адрес памяти в
DMA-адрес и возвращает DMA-адрес драйверу для программирования в устройство. Если
операция DMA задаёт несколько сегментов буфера памяти, для каждого сегмента должен
быть выделен отдельный bounce-буфер. swiotlb_tbl_map_single() всегда выполняет
операцию «sync» (т. е. копирование CPU), чтобы инициализировать bounce-буфер в
соответствии с содержимым исходного буфера.

swiotlb_tbl_unmap_single() выполняет обратное действие. Если операция DMA могла
обновить память bounce-буфера и не установлен DMA_ATTR_SKIP_CPU_SYNC, unmap
выполняет операцию «sync», чтобы заставить CPU скопировать данные из bounce-буфера
обратно в исходный буфер. Затем память bounce-буфера освобождается.

swiotlb также предоставляет API «sync», соответствующие API dma_sync_*(), которые
драйвер может использовать, когда управление буфером переходит между CPU и
устройством. API «sync» swiotlb вызывают копирование CPU данных между исходным
буфером и bounce-буфером. Как и API dma_sync_*(), API «sync» swiotlb поддерживают
выполнение частичной синхронизации, при которой в/из исходного буфера копируется
лишь подмножество bounce-буфера.

Ограничения основной функциональности
--------------------------------------
API map/unmap/sync swiotlb должны работать без блокировки, так как они вызываются
соответствующими API DMA, которые могут выполняться в контекстах, не допускающих
блокировку. Поэтому пул памяти по умолчанию для выделений swiotlb должен быть
предварительно выделен во время загрузки (но см. `Dynamic swiotlb`_ ниже).
Поскольку выделения swiotlb должны быть физически непрерывными, весь пул памяти по
умолчанию выделяется как единый непрерывный блок.

Необходимость предварительно выделять пул swiotlb по умолчанию создаёт компромисс
на этапе загрузки. Пул должен быть достаточно большим, чтобы гарантировать, что
запросы на bounce-буфер всегда могут быть удовлетворены, поскольку требование
неблокирования означает, что запросы не могут ждать освобождения места. Но большой
пул потенциально расходует память впустую, так как эта предварительно выделенная
память недоступна для других нужд в системе. Этот компромисс особенно остро стоит в
CoCo VM, использующих bounce-буферы для всего DMA I/O. Эти VM используют эвристику
для установки размера пула по умолчанию в ~6% памяти, с максимумом 1 GiB, что
потенциально может очень расточительно расходовать память. И наоборот, эвристика
может выдать размер, который недостаточен, в зависимости от паттернов I/O нагрузки
в VM. Функциональность dynamic swiotlb, описанная ниже, может помочь, но имеет
ограничения. Более эффективное управление размером пула памяти swiotlb по умолчанию
остаётся открытым вопросом.

Одно выделение из swiotlb ограничено IO_TLB_SIZE * IO_TLB_SEGSIZE байтами, что при
текущих определениях составляет 256 KiB. Когда настройки DMA устройства таковы, что
устройство может использовать swiotlb, максимальный размер сегмента DMA должен быть
ограничен этими 256 KiB. Это значение сообщается коду ядра более высокого уровня
через dma_map_mapping_size() и swiotlb_max_mapping_size(). Если код более высокого
уровня не учитывает этот предел, он может делать слишком большие для swiotlb запросы
и получать ошибку «swiotlb full».

Ключевой настройкой DMA устройства является «min_align_mask», которая представляет
собой степень 2 минус 1, так что установлено некоторое количество младших битов,
либо она может быть равна нулю. Выделения swiotlb гарантируют, что эти биты
min_align_mask физического адреса bounce-буфера совпадают с теми же битами в адресе
исходного буфера. Когда min_align_mask ненулевая, она может порождать «alignment
offset» (смещение выравнивания) в адресе bounce-буфера, что слегка уменьшает
максимальный размер выделения. Это потенциальное смещение выравнивания отражается в
значении, возвращаемом swiotlb_max_mapping_size(), которое может проявляться в таких
местах, как /sys/block/<device>/queue/max_sectors_kb. Например, если устройство не
использует swiotlb, max_sectors_kb может быть 512 KiB или больше. Если устройство
может использовать swiotlb, max_sectors_kb будет 256 KiB. Когда min_align_mask
ненулевая, max_sectors_kb может быть ещё меньше, например 252 KiB.

swiotlb_tbl_map_single() также принимает параметр «alloc_align_mask». Этот параметр
задаёт, что выделение пространства bounce-буфера должно начинаться по физическому
адресу с битами alloc_align_mask, установленными в ноль. Но фактический bounce-буфер
может начинаться по большему адресу, если min_align_mask ненулевая. Поэтому может
существовать пространство pre-padding, выделяемое до начала bounce-буфера.
Аналогично, конец bounce-буфера округляется вверх до границы alloc_align_mask, что
потенциально приводит к пространству post-padding. Любое пространство pre-padding или
post-padding не инициализируется кодом swiotlb. Параметр «alloc_align_mask»
используется кодом IOMMU при отображении для недоверенных устройств. Он
устанавливается в размер гранулы - 1, так что bounce-буфер выделяется целиком из
гранул, не используемых ни для каких других целей.

Концепции структур данных
--------------------------
Память, используемая для bounce-буферов swiotlb, выделяется из общей системной
памяти в виде одного или нескольких «пулов». Пул по умолчанию выделяется во время
загрузки системы с размером по умолчанию 64 MiB. Размер пула по умолчанию может
быть изменён с помощью параметра загрузочной строки ядра «swiotlb=». Размер по
умолчанию также может корректироваться из-за других условий, таких как выполнение в
CoCo VM, как описано выше. Если включён CONFIG_SWIOTLB_DYNAMIC, дополнительные пулы
могут быть выделены позже в течение жизни системы. Каждый пул должен представлять
собой непрерывный диапазон физической памяти. Пул по умолчанию выделяется ниже
отметки физического адреса 4 GiB, поэтому он работает для устройств, способных
адресовать только 32 бита физической памяти (если только архитектурно-специфичный
код не предоставляет флаг SWIOTLB_ANY). В CoCo VM память пула должна быть
расшифрована перед использованием swiotlb.

Каждый пул делится на «слоты» («slots») размером IO_TLB_SIZE, что при текущих
определениях составляет 2 KiB. IO_TLB_SEGSIZE непрерывных слотов (128 слотов)
составляют то, что можно назвать «набором слотов» («slot set»). Когда выделяется
bounce-буфер, он занимает один или несколько непрерывных слотов. Слот никогда не
разделяется между несколькими bounce-буферами. Кроме того, bounce-буфер должен
выделяться из единственного набора слотов, что приводит к тому, что максимальный
размер bounce-буфера составляет IO_TLB_SIZE * IO_TLB_SEGSIZE. Несколько меньших
bounce-буферов могут сосуществовать в одном наборе слотов, если могут быть
удовлетворены ограничения по выравниванию и размеру.

Слоты также группируются в «области» («areas»), с ограничением, что набор слотов
существует целиком в единственной области. Каждая область имеет собственный
спинлок (spinlock), который должен удерживаться для манипуляций со слотами в этой
области. Разделение на области позволяет избежать конкуренции за единственный
глобальный спинлок, когда swiotlb интенсивно используется, например в CoCo VM.
Число областей по умолчанию равно числу CPU в системе для максимального
параллелизма, но поскольку область не может быть меньше IO_TLB_SEGSIZE слотов,
может потребоваться назначить несколько CPU одной и той же области. Число областей
также может быть задано через параметр загрузки ядра «swiotlb=».

При выделении bounce-буфера, если в области, связанной с вызывающим CPU,
недостаточно свободного места, последовательно пробуются области, связанные с
другими CPU. Для каждой пробуемой области спинлок области должен быть получен перед
попыткой выделения, поэтому может возникнуть конкуренция, если swiotlb в целом
относительно загружена. Но запрос на выделение не завершается неудачей, если только
во всех областях не окажется недостаточно свободного места.

IO_TLB_SIZE, IO_TLB_SEGSIZE и число областей должны быть степенями 2, так как код
использует сдвиг и битовое маскирование для выполнения многих вычислений. Число
областей при необходимости округляется вверх до степени 2, чтобы удовлетворить
этому требованию.

Пул по умолчанию выделяется с выравниванием PAGE_SIZE. Если аргумент
alloc_align_mask для swiotlb_tbl_map_single() задаёт большее выравнивание, один или
несколько начальных слотов в каждом наборе слотов могут не удовлетворять критерию
alloc_align_mask. Поскольку выделение bounce-буфера не может пересекать границу
набора слотов, устранение этих начальных слотов фактически уменьшает максимальный
размер bounce-буфера. В настоящее время проблемы нет, так как alloc_align_mask
устанавливается на основе размера гранулы IOMMU, а гранулы не могут быть больше
PAGE_SIZE. Но если это изменится в будущем, начальное выделение пула, возможно,
придётся выполнять с выравниванием, большим PAGE_SIZE.

Dynamic swiotlb
---------------
Когда включён CONFIG_SWIOTLB_DYNAMIC, swiotlb может выполнять расширение по
требованию объёма памяти, доступной для выделения в качестве bounce-буферов. Если
запрос на bounce-буфер завершается неудачей из-за нехватки доступного места,
запускается асинхронная фоновая задача для выделения памяти из общей системной
памяти и превращения её в пул swiotlb. Создание дополнительного пула должно
выполняться асинхронно, поскольку выделение памяти может блокироваться, а, как
отмечено выше, запросам swiotlb не разрешается блокироваться. После запуска фоновой
задачи запрос на bounce-буфер создаёт «transient pool» (временный пул), чтобы
избежать возврата ошибки «swiotlb full». Transient pool имеет размер запроса на
bounce-буфер и удаляется, когда bounce-буфер освобождается. Память для этого
transient pool берётся из атомарного пула общей системной памяти, чтобы создание не
блокировалось. Создание transient pool имеет относительно высокую стоимость,
особенно в CoCo VM, где память должна быть расшифрована, поэтому оно выполняется
только как временная мера, пока фоновая задача не сможет добавить ещё один
не-transient пул.

Добавление динамического пула имеет ограничения. Как и в случае пула по умолчанию,
память должна быть физически непрерывной, поэтому размер ограничен MAX_PAGE_ORDER
страницами (например, 4 MiB на типичной системе x86). Из-за фрагментации памяти
выделение максимального размера может быть недоступно. Аллокатор динамического пула
пробует меньшие размеры, пока не добьётся успеха, но с минимальным размером 1 MiB.
При достаточной фрагментации системной памяти динамическое добавление пула может не
удаться вовсе.

Число областей в динамическом пуле может отличаться от числа областей в пуле по
умолчанию. Поскольку размер нового пула обычно составляет максимум несколько MiB,
число областей, скорее всего, будет меньше. Например, при размере нового пула 4 MiB
и минимальном размере области 256 KiB может быть создано только 16 областей. Если в
системе более 16 CPU, несколько CPU должны разделять одну область, создавая больше
конкуренции за блокировку.

Новые пулы, добавленные через dynamic swiotlb, связываются вместе в линейный список.
Код swiotlb часто должен искать пул, содержащий конкретный физический адрес swiotlb,
поэтому этот поиск является линейным и не производительным при большом числе
динамических пулов. Структуры данных можно было бы улучшить для более быстрого
поиска.

В целом dynamic swiotlb работает лучше всего для небольших конфигураций с
относительно малым числом CPU. Она позволяет пулу swiotlb по умолчанию быть меньше,
чтобы память не расходовалась впустую, а динамические пулы делают доступным больше
места при необходимости (пока фрагментация не является препятствием). Она менее
полезна для крупных CoCo VM.

Подробности структур данных
----------------------------
swiotlb управляется четырьмя основными структурами данных: io_tlb_mem, io_tlb_pool,
io_tlb_area и io_tlb_slot. io_tlb_mem описывает аллокатор памяти swiotlb, который
включает пул памяти по умолчанию и любые динамические или transient пулы, связанные
с ним. Ограниченная статистика использования swiotlb хранится для каждого аллокатора
памяти и сохраняется в этой структуре данных. Эта статистика доступна в
/sys/kernel/debug/swiotlb, когда установлен CONFIG_DEBUG_FS.

io_tlb_pool описывает пул памяти, либо пул по умолчанию, либо динамический пул, либо
transient пул. Описание включает начальный и конечный адреса памяти в пуле,
указатель на массив структур io_tlb_area и указатель на массив структур io_tlb_slot,
которые связаны с пулом.

io_tlb_area описывает область. Основным полем является спинлок, используемый для
сериализации доступа к слотам в области. Массив io_tlb_area для пула имеет запись
для каждой области и доступен с использованием индекса области, отсчитываемого от 0,
производного от идентификатора вызывающего процессора. Области существуют
исключительно для того, чтобы обеспечить параллельный доступ к swiotlb с нескольких
CPU.

io_tlb_slot описывает отдельный слот памяти в пуле размером IO_TLB_SIZE (в настоящее
время 2 KiB). Массив io_tlb_slot индексируется по индексу слота, вычисляемому из
адреса bounce-буфера относительно начального адреса памяти пула. Размер struct
io_tlb_slot составляет 24 байта, поэтому накладные расходы составляют около 1% от
размера слота.

Массив io_tlb_slot спроектирован для удовлетворения нескольких требований. Во-первых,
API DMA и соответствующие API swiotlb используют адрес bounce-буфера в качестве
идентификатора bounce-буфера. Этот адрес возвращается функцией
swiotlb_tbl_map_single(), а затем передаётся в качестве аргумента функциям
swiotlb_tbl_unmap_single() и swiotlb_sync_*().  Адрес исходного буфера памяти,
очевидно, должен передаваться в качестве аргумента в swiotlb_tbl_map_single(), но он
не передаётся в другие API. Следовательно, структуры данных swiotlb должны сохранять
адрес исходного буфера памяти, чтобы его можно было использовать при выполнении
операций sync. Этот исходный адрес сохраняется в массиве io_tlb_slot.

Во-вторых, массив io_tlb_slot должен обрабатывать запросы на частичную
синхронизацию. В таких случаях аргументом для swiotlb_sync_*() является не адрес
начала bounce-буфера, а адрес где-то в середине bounce-буфера, и адрес начала
bounce-буфера не известен коду swiotlb. Но код swiotlb должен быть способен вычислить
соответствующий адрес исходного буфера памяти, чтобы выполнить копирование CPU,
предписанное «sync». Поэтому скорректированный адрес исходного буфера памяти
заносится в struct io_tlb_slot для каждого слота, занятого bounce-буфером.
Скорректированный «alloc_size» bounce-буфера также записывается в каждую struct
io_tlb_slot, чтобы можно было выполнить проверку корректности размера операции
«sync». Поле «alloc_size» не используется, кроме как для проверки корректности.

В-третьих, массив io_tlb_slot используется для отслеживания доступных слотов. Поле
«list» в struct io_tlb_slot записывает, сколько непрерывных доступных слотов
существует, начиная с этого слота. «0» указывает, что слот занят. Значение «1»
указывает, что доступен только текущий слот. Значение «2» указывает, что доступны
текущий слот и следующий слот, и т. д. Максимальное значение — IO_TLB_SEGSIZE,
которое может появляться в первом слоте набора слотов и указывает, что доступен весь
набор слотов. Эти значения используются при поиске доступных слотов для использования
под новый bounce-буфер. Они обновляются при выделении нового bounce-буфера и при
освобождении bounce-буфера. Во время создания пула поле «list» инициализируется
значениями от IO_TLB_SEGSIZE вниз до 1 для слотов в каждом наборе слотов.

В-четвёртых, массив io_tlb_slot отслеживает любые «padding slots» (слоты-заполнители),
выделенные для удовлетворения требований alloc_align_mask, описанных выше. Когда
swiotlb_tbl_map_single() выделяет пространство bounce-буфера для удовлетворения
требований alloc_align_mask, она может выделить пространство pre-padding в пределах
нуля или более слотов. Но когда swiotlb_tbl_unmap_single() вызывается с адресом
bounce-буфера, значение alloc_align_mask, управлявшее выделением, а следовательно, и
выделение любых padding-слотов, не известно. Поле «pad_slots» записывает число
padding-слотов, чтобы swiotlb_tbl_unmap_single() могла их освободить. Значение
«pad_slots» записывается только в первом не-padding слоте, выделенном bounce-буферу.

Restricted pools
----------------
Механизм swiotlb также используется для «restricted pools» — пулов памяти,
отдельных от пула swiotlb по умолчанию и предназначенных для использования DMA
конкретным устройством. Restricted pools обеспечивают определённый уровень защиты
памяти DMA в системах с ограниченными аппаратными возможностями защиты, таких как
системы без IOMMU. Такое использование задаётся записями DeviceTree и требует, чтобы
был установлен CONFIG_DMA_RESTRICTED_POOL. Каждый restricted pool основан на
собственной структуре данных io_tlb_mem, которая независима от основной io_tlb_mem
swiotlb.

Restricted pools добавляют API swiotlb_alloc() и swiotlb_free(), которые вызываются
из API dma_alloc_*() и dma_free_*(). API swiotlb_alloc/free() выделяют/освобождают
слоты из/в restricted pool напрямую и не проходят через
swiotlb_tbl_map/unmap_single().
