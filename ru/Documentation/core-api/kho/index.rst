==========================
Подсистема Kexec Handover
==========================

Обзор
=====

Kexec HandOver (KHO) — это механизм, позволяющий Linux сохранять области
памяти, которые могут содержать сериализованные состояния системы, при
переходе через kexec.

KHO использует :ref:`сплющенное дерево устройств (flattened device tree, FDT) <kho_fdt>`
для передачи информации о сохраняемом состоянии от ядра, работающего до kexec,
к ядру, работающему после kexec, а также :ref:`области резервной памяти <kho_scratch>`
для обеспечения целостности сохраняемой памяти.

.. _kho_fdt:

KHO FDT
=======
Каждый KHO-kexec несёт специфичный для KHO блоб сплющенного дерева устройств
(FDT), который описывает сохраняемое состояние. FDT включает свойства,
описывающие сохраняемые области памяти, и узлы, в которых хранится
специфичное для подсистем состояние.

Сохраняемые области памяти содержат либо сериализованные состояния подсистем,
либо данные в памяти, которые не должны изменяться при переходе через kexec.
После KHO подсистемы могут извлечь и восстановить сохранённое состояние из
KHO FDT.

Подсистемы, участвующие в KHO, могут определять собственный формат для
сериализации и сохранения состояния.

KHO FDT и структуры, определённые подсистемами, образуют ABI между ядрами до
kexec и после kexec. Этот ABI определён заголовочными файлами в каталоге
``include/linux/kho/abi``.

.. _kho_scratch:

Области резервной памяти
========================

Чтобы загрузиться в kexec, нам необходим физически непрерывный диапазон
памяти, не содержащий переданной памяти. Затем kexec размещает целевое ядро и
initrd в этой области. Новое ядро использует эту область исключительно для
выделений памяти до и во время загрузки вплоть до инициализации страничного
аллокатора.

Мы гарантируем, что у нас всегда есть такие области, посредством областей
резервной памяти: при первой загрузке KHO выделяет несколько физически
непрерывных областей памяти. Поскольку после kexec эти области будут
использоваться ранними выделениями памяти, имеется по одной области резервной
памяти на каждый узел NUMA, плюс одна область резервной памяти для
удовлетворения запросов на выделение, которым не требуется привязка к
конкретному узлу NUMA.
По умолчанию размер области резервной памяти вычисляется на основе объёма
памяти, выделенной во время загрузки. Для явного определения размера областей
резервной памяти может использоваться параметр командной строки ядра
``kho_scratch``.
Области резервной памяти объявляются как CMA при инициализации страничного
аллокатора, чтобы их память могла использоваться на протяжении всего времени
работы системы. CMA даёт нам гарантию, что в эту область не попадут страницы
передачи, поскольку страницы передачи должны располагаться в статическом
физическом расположении памяти, а CMA обеспечивает, что внутри могут
находиться только перемещаемые страницы.

После KHO-kexec мы игнорируем параметр командной строки ядра ``kho_scratch``
и вместо этого повторно используем ту же самую область, которая была изначально
выделена. Это позволяет нам рекурсивно выполнять любое количество KHO-kexec'ов.
Поскольку мы использовали эту область для выделений памяти при загрузке и в
качестве целевой памяти для блобов kexec, некоторые части этой области памяти
могут быть зарезервированы. Эти резервирования не имеют значения для
следующего KHO, потому что kexec может перезаписать даже исходное ядро.

.. _kho-finalization-phase:

Фаза финализации KHO
====================

Чтобы задействовать загрузчик файлов kexec на основе пространства пользователя,
ядро должно быть способно предоставить FDT, описывающий текущее состояние ядра,
до выполнения собственно kexec. Процесс генерации этого FDT называется
сериализацией. Когда FDT сгенерирован, некоторые свойства системы могут стать
неизменяемыми, поскольку они уже записаны в FDT. Это состояние называется фазой
финализации KHO.
