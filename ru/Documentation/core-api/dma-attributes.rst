==============
Атрибуты DMA
==============

Этот документ описывает семантику атрибутов DMA, которые определены
в linux/dma-mapping.h.

DMA_ATTR_WEAK_ORDERING
----------------------

DMA_ATTR_WEAK_ORDERING указывает, что чтения и записи в отображение
могут быть слабо упорядочены, то есть чтения и записи могут обгонять друг друга.

Поскольку реализация DMA_ATTR_WEAK_ORDERING для платформ необязательна,
те из них, что её не реализуют, просто игнорируют этот атрибут и проявляют
поведение по умолчанию.

DMA_ATTR_WRITE_COMBINE
----------------------

DMA_ATTR_WRITE_COMBINE указывает, что записи в отображение могут
буферизоваться для повышения производительности.

Поскольку реализация DMA_ATTR_WRITE_COMBINE для платформ необязательна,
те из них, что её не реализуют, просто игнорируют этот атрибут и проявляют
поведение по умолчанию.

DMA_ATTR_NO_KERNEL_MAPPING
--------------------------

DMA_ATTR_NO_KERNEL_MAPPING позволяет платформе избежать создания виртуального
отображения в ядре для выделенного буфера. На некоторых архитектурах создание
такого отображения является нетривиальной задачей и потребляет весьма ограниченные
ресурсы (например, виртуальное адресное пространство ядра или согласованное
адресное пространство dma). Буферы, выделенные с этим атрибутом, можно передать
в пространство пользователя только вызовом dma_mmap_attrs(). Используя этот API,
вы гарантируете, что не будете разыменовывать указатель, возвращаемый
dma_alloc_attr(). Вы можете рассматривать его как cookie, который необходимо
передавать в dma_mmap_attrs() и dma_free_attrs(). Убедитесь, что этот атрибут также
установлен при каждом из этих вызовов.

Поскольку реализация DMA_ATTR_NO_KERNEL_MAPPING для платформ необязательна,
те из них, что её не реализуют, просто игнорируют этот атрибут и проявляют
поведение по умолчанию.

DMA_ATTR_SKIP_CPU_SYNC
----------------------

По умолчанию семейство функций dma_map_{single,page,sg} переносит заданный буфер
из домена CPU в домен устройства. В некоторых продвинутых сценариях использования
может потребоваться совместное использование буфера несколькими устройствами. Это
требует создания отдельного отображения для каждого устройства и обычно выполняется
вызовом функции dma_map_{single,page,sg} более одного раза для заданного буфера
с указателем на каждое устройство, участвующее в совместном использовании буфера.
Первый вызов переносит буфер из домена «CPU» в домен «устройства», что синхронизирует
кэши CPU для заданной области (обычно это означает, что кэш был сброшен или
инвалидирован в зависимости от направления dma). Однако последующие вызовы
dma_map_{single,page,sg}() для других устройств будут выполнять ровно ту же операцию
синхронизации над кэшем CPU. Синхронизация кэша CPU может быть затратной по времени
операцией, особенно если буферы большие, поэтому настоятельно рекомендуется избегать
её, если это возможно. DMA_ATTR_SKIP_CPU_SYNC позволяет коду платформы пропустить
синхронизацию кэша CPU для заданного буфера в предположении, что он уже был перенесён
в домен «устройства». Этот атрибут также можно использовать для семейства функций
dma_unmap_{single,page,sg}, чтобы заставить буфер оставаться в домене устройства
после освобождения его отображения. Используйте этот атрибут с осторожностью!

DMA_ATTR_FORCE_CONTIGUOUS
-------------------------

По умолчанию подсистеме DMA-mapping разрешено собирать буфер, выделенный функцией
dma_alloc_attrs(), из отдельных страниц, если он может быть отображён как непрерывный
блок в dma-адресное пространство устройства. Указание этого атрибута заставляет
выделенный буфер быть непрерывным также и в физической памяти.

DMA_ATTR_ALLOC_SINGLE_PAGES
---------------------------

Это подсказка подсистеме DMA-mapping о том, что, вероятно, не стоит тратить время
на попытки выделить память способом, дающим лучшую эффективность TLB (иначе говоря,
не стоит пытаться строить отображение из более крупных страниц). Возможно, вы захотите
указать этот атрибут, если:

- Вы знаете, что обращения к этой памяти не будут вызывать пробуксовку TLB.
  Вы можете знать, что обращения, вероятно, будут последовательными, или
  что они не последовательны, но маловероятно, что вы будете метаться
  между множеством адресов, которые, скорее всего, находятся в разных физических
  страницах.
- Вы знаете, что штраф за промахи TLB при обращении к памяти будет
  достаточно малым, чтобы быть несущественным. Если вы выполняете
  тяжёлую операцию вроде расшифровки или распаковки, это может быть
  именно тот случай.
- Вы знаете, что отображение DMA довольно кратковременное. Если вы ожидаете,
  что отображение будет иметь короткое время жизни, то, возможно, стоит
  оптимизировать выделение (избегая формирования больших страниц) вместо того,
  чтобы получать незначительный выигрыш в производительности от больших страниц.

Установка этой подсказки не гарантирует, что вы не получите огромные страницы, но
означает, что мы не будем стараться изо всех сил, чтобы их получить.

.. note:: На данный момент DMA_ATTR_ALLOC_SINGLE_PAGES реализован только на ARM,
	  хотя патчи для ARM64, вероятно, скоро будут опубликованы.

DMA_ATTR_NO_WARN
----------------

Это сообщает подсистеме DMA-mapping о необходимости подавлять отчёты об ошибках
выделения (подобно __GFP_NOWARN).

На некоторых архитектурах ошибки выделения сообщаются в системные журналы в виде
сообщений об ошибках. Хотя это может помочь выявить и отладить проблемы, для драйверов,
которые обрабатывают сбои (например, повторяют попытку позже), они не представляют
проблемы и фактически могут заполонить системные журналы сообщениями об ошибках,
которые вовсе не являются проблемой, в зависимости от реализации механизма повторных
попыток.

Итак, это даёт драйверам способ избежать таких сообщений об ошибках при вызовах,
где ошибки выделения не являются проблемой и не должны засорять журналы.

.. note:: На данный момент DMA_ATTR_NO_WARN реализован только на PowerPC.

DMA_ATTR_PRIVILEGED
-------------------

Некоторые продвинутые периферийные устройства, такие как удалённые процессоры и GPU,
выполняют обращения к буферам DMA как в привилегированном режиме «супервизора», так
и в непривилегированном «пользовательском» режиме. Этот атрибут используется, чтобы
указать подсистеме DMA-mapping, что буфер полностью доступен на повышенном уровне
привилегий (и в идеале недоступен или, как минимум, доступен только для чтения на
менее привилегированных уровнях).

DMA_ATTR_MMIO
-------------

Этот атрибут указывает, что физический адрес не является обычной системной памятью.
Он не может использоваться с функциями kmap*()/phys_to_virt()/phys_to_page(), он может
быть некэшируемым, и доступ с помощью инструкций загрузки/сохранения CPU может быть
недопустим.

Обычно это используется для описания адресов MMIO или иных некэшируемых адресов
регистров. При DMA-отображении адреса такого рода мы называем операцию Peer to Peer,
поскольку одно устройство выполняет DMA в другое устройство. Для устройств PCI
необходимо использовать API p2pdma, чтобы определить, уместен ли DMA_ATTR_MMIO.

Для архитектур, требующих сброса кэша для согласованности DMA, DMA_ATTR_MMIO не будет
выполнять никакого сброса кэша. Предоставленный адрес никогда не должен отображаться
как кэшируемый в CPU.

DMA_ATTR_DEBUGGING_IGNORE_CACHELINES
------------------------------------

Этот атрибут указывает, что строки кэша CPU могут перекрываться для буферов,
отображённых с DMA_FROM_DEVICE или DMA_BIDIRECTIONAL.

Такое перекрытие может возникнуть, когда вызывающие отображают несколько небольших
буферов, размещённых в пределах одной строки кэша. В этом случае вызывающие должны
гарантировать, что CPU не загрязнит эти строки кэша после установления отображений.
При соблюдении этого условия несколько буферов могут безопасно использовать общую
строку кэша без риска повреждения данных.

Все отображения, использующие общую строку кэша, должны устанавливать этот атрибут,
чтобы подавить предупреждения отладки DMA о перекрывающихся отображениях.

DMA_ATTR_REQUIRE_COHERENT
-------------------------

Запросы DMA-отображения с DMA_ATTR_REQUIRE_COHERENT завершаются неудачей в любой
системе, где требуется SWIOTLB или управление кэшем. Это следует использовать только
для поддержки uAPI-архитектур, которым требуется непрерывная аппаратная согласованность
DMA с процессами пространства пользователя, например RDMA и DRM. Как минимум,
отображаемая память должна быть памятью пространства пользователя из pin_user_pages()
или аналогичной функции.

Драйверам при построении своих uAPI следует рассмотреть использование dma_mmap_pages()
вместо этого интерфейса, когда это возможно.

Его никогда нельзя использовать во внутриядерном драйвере, который работает только
с памятью ядра.
