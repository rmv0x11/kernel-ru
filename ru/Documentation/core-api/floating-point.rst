API работы с плавающей точкой
=============================

Коду ядра обычно запрещено использовать регистры или инструкции плавающей точки
(FP, floating-point), включая типы данных C float и double. Это правило снижает
накладные расходы на системные вызовы, поскольку ядру не нужно сохранять и
восстанавливать состояние регистров плавающей точки пространства пользователя.

Тем не менее иногда драйверам или библиотечным функциям может потребоваться
включать FP-код. Это поддерживается за счёт изоляции функций, содержащих FP-код,
в отдельной единице трансляции (отдельном файле исходного кода) и
сохранения/восстановления состояния FP-регистров вокруг вызовов этих функций.
Так создаются «критические секции» использования плавающей точки.

Причина такой изоляции — не дать компилятору сгенерировать код, обращающийся к
FP-регистрам за пределами этих критических секций. Компиляторы иногда используют
FP-регистры для оптимизации встраиваемого ``memcpy`` или присваивания
переменных, поскольку регистры плавающей точки могут быть шире регистров общего
назначения.

Возможность использования кода плавающей точки внутри ядра зависит от
архитектуры. Кроме того, поскольку одно и то же ядро может быть сконфигурировано
для поддержки платформ как с модулем плавающей точки, так и без него, доступность
FPU должна проверяться как во время сборки, так и во время выполнения.

Несколько архитектур реализуют универсальный API плавающей точки ядра из
``linux/fpu.h``, как описано ниже. Некоторые другие архитектуры реализуют свои
собственные уникальные API, которые документируются отдельно.

API времени сборки
-------------------

Код плавающей точки может быть собран, если включена опция
``ARCH_HAS_KERNEL_FPU_SUPPORT``. Для кода на C такой код должен быть размещён в
отдельном файле, и для этого файла необходимо скорректировать флаги компиляции по
следующему шаблону::

    CFLAGS_foo.o += $(CC_FLAGS_FPU)
    CFLAGS_REMOVE_foo.o += $(CC_FLAGS_NO_FPU)

Предполагается, что архитектуры определяют одну или обе эти переменные в своём
Makefile верхнего уровня по мере необходимости. Например::

    CC_FLAGS_FPU := -mhard-float

или::

    CC_FLAGS_NO_FPU := -msoft-float

Предполагается, что обычный код ядра использует эквивалент ``CC_FLAGS_NO_FPU``.

API времени выполнения
----------------------

API времени выполнения предоставляется в ``linux/fpu.h``. Этот заголовочный файл
нельзя включать из файлов, реализующих FP-код (тех, чьи флаги компиляции
скорректированы, как описано выше). Вместо этого его следует включать при
определении критических секций плавающей точки.

.. c:function:: bool kernel_fpu_available( void )

        Эта функция сообщает, можно ли использовать код плавающей точки на данном
        CPU или платформе. Не предполагается, что возвращаемое этой функцией
        значение будет меняться во время выполнения, поэтому её достаточно
        вызвать один раз, а не перед каждой критической секцией.

.. c:function:: void kernel_fpu_begin( void )
                void kernel_fpu_end( void )

        Эти функции создают критическую секцию плавающей точки. Вызов
        ``kernel_fpu_begin()`` допустим только после того, как предыдущий вызов
        ``kernel_fpu_available()`` вернул ``true``. Гарантируется лишь, что эти
        функции можно вызывать из контекста процесса (вытесняемого или
        невытесняемого).

        Внутри критических секций вытеснение может быть отключено, поэтому их
        размер следует минимизировать. Они *не* обязаны быть реентерабельными.
        Если вызывающий код предполагает вложение критических секций, он должен
        реализовать собственный подсчёт ссылок.
