=====================================================
Семантика и поведение локальных атомарных операций
=====================================================

:Author: Mathieu Desnoyers


Этот документ объясняет назначение локальных атомарных операций, способ их
реализации для любой заданной архитектуры и показывает, как их правильно
использовать. Он также акцентирует внимание на мерах предосторожности, которые
необходимо соблюдать при чтении этих локальных переменных с разных CPU, когда
порядок записей в память имеет значение.

.. note::

    Обратите внимание, что операции на основе ``local_t`` не рекомендуются для
    общего использования в ядре. Вместо них используйте операции ``this_cpu``,
    если только нет действительно особой необходимости. Большинство случаев
    использования ``local_t`` в ядре были заменены операциями ``this_cpu``.
    Операции ``this_cpu`` объединяют релокацию с семантикой, подобной
    ``local_t``, в одной инструкции и дают более компактный и быстрее
    исполняемый код.


Назначение локальных атомарных операций
=======================================

Локальные атомарные операции предназначены для предоставления быстрых и
высоко реентерабельных счётчиков для каждого CPU. Они минимизируют
производительностные затраты стандартных атомарных операций за счёт удаления
префикса LOCK и барьеров памяти, обычно требуемых для синхронизации между CPU.

Наличие быстрых атомарных счётчиков для каждого CPU интересно во многих случаях:
это не требует отключения прерываний для защиты от обработчиков прерываний и
позволяет иметь согласованные счётчики в обработчиках NMI. Это особенно полезно
для целей трассировки и для различных счётчиков мониторинга производительности.

Локальные атомарные операции гарантируют атомарность модификации переменной
только в отношении того CPU, которому принадлежат данные. Поэтому необходимо
позаботиться о том, чтобы запись в данные ``local_t`` производил только один
CPU. Это достигается использованием данных для каждого CPU (per cpu) и
обеспечением того, что мы изменяем их из контекста, безопасного по отношению к
вытеснению. Однако чтение данных ``local_t`` разрешено с любого CPU: при этом
будет казаться, что они записываются не по порядку относительно других записей
в память, выполняемых CPU-владельцем.


Реализация для заданной архитектуры
===================================

Это можно сделать путём незначительной модификации стандартных атомарных
операций: следует оставить только их UP-вариант. Обычно это означает удаление
префикса LOCK (на i386 и x86_64) и любого барьера SMP-синхронизации. Если
архитектура не имеет различного поведения для SMP и UP, достаточно включить
``asm-generic/local.h`` в файл ``local.h`` вашей архитектуры.

Тип ``local_t`` определён как непрозрачный ``signed long`` путём встраивания
``atomic_long_t`` внутрь структуры. Это сделано так, чтобы приведение из этого
типа к ``long`` завершалось неудачей. Определение выглядит так::

    typedef struct { atomic_long_t a; } local_t;


Правила, которым нужно следовать при использовании локальных атомарных операций
===============================================================================

* Переменные, затрагиваемые локальными операциями, должны быть переменными для
  каждого CPU (per cpu).
* Запись в эти переменные должен производить *только* CPU-владелец этих
  переменных.
* Этот CPU может использовать локальные операции из любого контекста (process,
  irq, softirq, nmi, ...) для обновления своих переменных ``local_t``.
* Вытеснение (или прерывания) должно быть отключено при использовании локальных
  операций в контексте процесса, чтобы гарантировать, что процесс не будет
  мигрирован на другой CPU между получением переменной для данного CPU и
  выполнением собственно локальной операции.
* При использовании локальных операций в контексте прерывания на mainline-ядре
  никаких особых мер предпринимать не нужно, поскольку они будут выполняться на
  локальном CPU с уже отключённым вытеснением. Тем не менее я рекомендую всё же
  явно отключать вытеснение, чтобы убедиться, что это по-прежнему будет
  корректно работать на -rt-ядрах.
* Чтение локальной переменной CPU предоставит текущую копию этой переменной.
* Чтение этих переменных можно выполнять с любого CPU, поскольку обновления
  выровненных переменных типа «``long``» всегда атомарны. Так как CPU-писатель
  не выполняет синхронизации памяти, при чтении переменных *другого* CPU может
  быть прочитана устаревшая копия переменной.


Как использовать локальные атомарные операции
==============================================

::

    #include <linux/percpu.h>
    #include <asm/local.h>

    static DEFINE_PER_CPU(local_t, counters) = LOCAL_INIT(0);


Подсчёт
=======

Подсчёт ведётся по всем битам значения signed long.

В вытесняемом контексте используйте ``get_cpu_var()`` и ``put_cpu_var()`` вокруг
локальных атомарных операций: это гарантирует, что вытеснение отключено вокруг
доступа на запись к переменной для данного CPU. Например::

    local_inc(&get_cpu_var(counters));
    put_cpu_var(counters);

Если вы уже находитесь в контексте, безопасном по отношению к вытеснению, вы
можете использовать ``this_cpu_ptr()``::

    local_inc(this_cpu_ptr(&counters));



Чтение счётчиков
================

Эти локальные счётчики могут быть прочитаны с чужих CPU для суммирования
значения. Обратите внимание, что данные, видимые через local_read с разных CPU,
следует считать неупорядоченными относительно других записей в память,
происходящих на CPU, которому принадлежат данные::

    long sum = 0;
    for_each_online_cpu(cpu)
            sum += local_read(&per_cpu(counters, cpu));

Если вы хотите использовать удалённый local_read для синхронизации доступа к
ресурсу между CPU, необходимо использовать явные барьеры памяти ``smp_wmb()`` и
``smp_rmb()`` соответственно на CPU-писателе и CPU-читателе. Это было бы так,
если бы вы использовали переменную ``local_t`` как счётчик байтов, записанных в
буфер: должен быть ``smp_wmb()`` между записью в буфер и инкрементом счётчика, а
также ``smp_rmb()`` между чтением счётчика и чтением буфера.


Ниже приведён пример модуля, реализующего базовый счётчик для каждого CPU с
помощью ``local.h``::

    /* test-local.c
     *
     * Sample module for local.h usage.
     */


    #include <asm/local.h>
    #include <linux/module.h>
    #include <linux/timer.h>

    static DEFINE_PER_CPU(local_t, counters) = LOCAL_INIT(0);

    static struct timer_list test_timer;

    /* IPI called on each CPU. */
    static void test_each(void *info)
    {
            /* Increment the counter from a non preemptible context */
            printk("Increment on cpu %d\n", smp_processor_id());
            local_inc(this_cpu_ptr(&counters));

            /* This is what incrementing the variable would look like within a
             * preemptible context (it disables preemption) :
             *
             * local_inc(&get_cpu_var(counters));
             * put_cpu_var(counters);
             */
    }

    static void do_test_timer(unsigned long data)
    {
            int cpu;

            /* Increment the counters */
            on_each_cpu(test_each, NULL, 1);
            /* Read all the counters */
            printk("Counters read from CPU %d\n", smp_processor_id());
            for_each_online_cpu(cpu) {
                    printk("Read : CPU %d, count %ld\n", cpu,
                            local_read(&per_cpu(counters, cpu)));
            }
            mod_timer(&test_timer, jiffies + 1000);
    }

    static int __init test_init(void)
    {
            /* initialize the timer that will increment the counter */
            timer_setup(&test_timer, do_test_timer, 0);
            mod_timer(&test_timer, jiffies + 1);

            return 0;
    }

    static void __exit test_exit(void)
    {
            timer_shutdown_sync(&test_timer);
    }

    module_init(test_init);
    module_exit(test_exit);

    MODULE_LICENSE("GPL");
    MODULE_AUTHOR("Mathieu Desnoyers");
    MODULE_DESCRIPTION("Local Atomic Ops");
