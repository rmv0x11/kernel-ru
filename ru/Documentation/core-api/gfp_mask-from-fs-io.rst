==========================================
Маски GFP, используемые в контексте FS/IO
==========================================

:Date: May, 2018
:Author: Michal Hocko <mhocko@kernel.org>

Введение
========

Пути выполнения кода в стеках файловой системы и ввода-вывода должны быть
осторожны при выделении памяти, чтобы предотвратить рекурсивные взаимоблокировки
(deadlock), вызванные тем, что прямое освобождение памяти (direct memory reclaim)
обращается обратно к путям FS или IO и блокируется на уже удерживаемых ресурсах
(например, на блокировках — чаще всего тех, что используются для контекста
транзакции).

Традиционный способ избежать этой проблемы взаимоблокировки — сбросить __GFP_FS,
либо __GFP_IO (обратите внимание, что последнее подразумевает также сброс
первого), в маске gfp при вызове аллокатора. В качестве сокращения можно
использовать GFP_NOFS, либо GFP_NOIO. Однако выяснилось, что приведённый выше
подход привёл к злоупотреблениям, когда ограниченная маска gfp используется «на
всякий случай», без более глубокого осмысления, что ведёт к проблемам, поскольку
чрезмерное использование GFP_NOFS/GFP_NOIO может приводить к избыточному
освобождению памяти (memory over-reclaim) или к другим проблемам освобождения
памяти.

Новый API
=========

Начиная с 4.12, у нас есть универсальный API областей видимости (scope API) как
для контекста NOFS, так и для контекста NOIO: ``memalloc_nofs_save``,
``memalloc_nofs_restore``, а также ``memalloc_noio_save``,
``memalloc_noio_restore``, которые позволяют пометить область видимости как
критическую секцию с точки зрения файловой системы или ввода-вывода. Любое
выделение памяти из этой области будет неявно сбрасывать __GFP_FS, либо __GFP_IO,
из заданной маски, так что никакое выделение памяти не сможет рекурсивно вернуться
обратно в FS/IO.

.. kernel-doc:: include/linux/sched/mm.h
   :functions: memalloc_nofs_save memalloc_nofs_restore
.. kernel-doc:: include/linux/sched/mm.h
   :functions: memalloc_noio_save memalloc_noio_restore

Код FS/IO затем просто вызывает соответствующую функцию save перед началом любой
критической секции по отношению к освобождению памяти — например, перед
блокировкой, разделяемой с контекстом освобождения памяти, или когда вложение
контекста транзакции было бы возможно через освобождение памяти. Функцию restore
следует вызывать по окончании критической секции. Всё это в идеале вместе с
пояснением, что представляет собой контекст освобождения памяти, для облегчения
сопровождения.

Обратите внимание, что правильное парное использование функций save/restore
допускает вложенность, поэтому безопасно вызывать ``memalloc_noio_save`` или,
соответственно, ``memalloc_noio_restore`` из уже существующей области NOIO или
NOFS.

Как насчёт __vmalloc(GFP_NOFS)
==============================

Начиная с v5.17, и в частности после коммита 451769ebb7e79 ("mm/vmalloc:
alloc GFP_NO{FS,IO} for vmalloc"), GFP_NOFS/GFP_NOIO теперь поддерживаются в
``[k]vmalloc`` за счёт неявного использования scope API.

В более ранних ядрах ``vmalloc`` не поддерживал семантику GFP_NOFS, поскольку
глубоко внутри аллокатора были жёстко зашитые выделения GFP_KERNEL. Это означает,
что вызов ``vmalloc`` с GFP_NOFS/GFP_NOIO почти всегда был ошибкой.

В идеальном мире верхние уровни должны уже помечать опасные контексты, и поэтому
никаких особых мер не требуется, а ``vmalloc`` можно вызывать без каких-либо
проблем. Иногда, если контекст не вполне ясен или присутствуют нарушения
разделения на уровни, рекомендуемый способ обойти это (на ядрах до v5.17) —
обернуть ``vmalloc`` с помощью scope API, добавив комментарий с объяснением
проблемы.
