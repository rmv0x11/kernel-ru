=========================================
Заметки о Netlink для разработчиков ядра
=========================================

Общие рекомендации
===================

Перечисления атрибутов
----------------------

В более старых семействах часто определяются «нулевые» атрибуты и команды
со значением ``0`` и именем ``unspec``. Это поддерживается (``type: unused``),
но в новых семействах этого следует избегать. Значения enum ``unspec``
на практике не используются, поэтому просто задайте значение первого атрибута равным ``1``.

Перечисления сообщений
----------------------

Используйте одни и те же идентификаторы команд для запросов и ответов. Это упрощает
их сопоставление, а пространство идентификаторов у нас достаточно велико.

Используйте отдельные идентификаторы команд для уведомлений. Это упрощает
отделение уведомлений от ответов (и предоставление их пользовательскому
приложению через API, отличный от API ответов).

Отвечайте на запросы
--------------------

Более старые семейства отвечают не на все команды, особенно на команды NEW / ADD.
Пользователь узнаёт только о том, удалась операция или нет, посредством ACK.
Старайтесь найти полезные данные для возврата. После того как команда добавлена,
то, отвечает ли она полным сообщением или только ACK, является частью uAPI и
не может быть изменено. Лучше склоняться в пользу того, чтобы давать ответ.

В частности, команды NEW и ADD должны отвечать информацией, идентифицирующей
созданный объект, например идентификатором выделенного объекта (без необходимости
прибегать к использованию ``NLM_F_ECHO``).

NLM_F_ECHO
----------

Обязательно передавайте информацию о запросе в genl_notify(), чтобы ``NLM_F_ECHO``
возымел эффект.  Это полезно для программ, которым требуется точная обратная связь
от ядра (например, в целях логирования).

Поддержка консистентности дампа
-------------------------------

Если при итерации по объектам во время дампа возможен пропуск объектов или их
повторение — обязательно сообщайте о неконсистентности дампа с помощью ``NLM_F_DUMP_INTR``.
Обычно это реализуется поддержанием идентификатора поколения (generation id) для
структуры и его записью в поле ``seq`` структуры netlink_callback.

Спецификация Netlink
====================

Документация по тем частям спецификации Netlink, которые относятся только
к пространству ядра.

Globals
-------

kernel-policy
~~~~~~~~~~~~~

Определяет, является ли политика валидации ядра ``global``, т.е. одинаковой для всех
операций семейства, определяемой для каждой операции индивидуально — ``per-op``,
или отдельно для каждой операции и типа операции (do или dump) — ``split``.
Новые семейства должны использовать ``per-op`` (по умолчанию), чтобы иметь возможность
сузить набор атрибутов, принимаемых конкретной командой.

checks
------

Документация для подразделов ``checks`` в спецификациях атрибутов.

unterminated-ok
~~~~~~~~~~~~~~~

Принимать строки без нуль-терминатора (только для устаревших семейств).
Переключает тип политики с ``NLA_NUL_STRING`` на ``NLA_STRING``.

max-len
~~~~~~~

Определяет максимальную длину для бинарного или строкового атрибута (соответствует
полю ``len`` структуры nla_policy). Для строковых атрибутов завершающий нулевой символ
не учитывается в ``max-len``.

Поле может быть либо целочисленным литеральным значением, либо именем определённой
константы. Строковые типы могут уменьшать константу на единицу
(т.е. указывать ``max-len: CONST - 1``), чтобы зарезервировать место для завершающего
символа, поэтому реализациям следует распознавать такой шаблон.

min-len
~~~~~~~

Аналогично ``max-len``, но определяет минимальную длину.
