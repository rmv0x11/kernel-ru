==============================
Пространства имён для символов
==============================

Этот документ описывает, как использовать пространства имён для символов
(Symbol Namespaces) для структурирования экспортируемой поверхности символов
ядра, экспортируемых с помощью семейства макросов EXPORT_SYMBOL().

Введение
========

Пространства имён для символов были введены как средство структурирования
экспортируемой поверхности внутриядерного API. Они позволяют сопровождающим
подсистем разделять экспортируемые ими символы на отдельные пространства имён.
Это полезно как для целей документирования (вспомните о пространстве имён
SUBSYSTEM_DEBUG), так и для ограничения доступности набора символов для
использования в других частях ядра. На сегодняшний день модули, использующие
символы, экспортированные в пространства имён, обязаны импортировать это
пространство имён. В противном случае ядро, в зависимости от его конфигурации,
либо откажется загружать модуль, либо предупредит об отсутствующем импорте.

Кроме того, символы можно помещать в пространство имён модуля, строго ограничивая
то, каким модулям разрешено использовать эти символы.

Как определять пространства имён для символов
=============================================

Символы могут быть экспортированы в пространство имён различными способами. Все
они изменяют то, как EXPORT_SYMBOL и подобные ему инструментируются для создания
записей ksymtab.

Использование макросов EXPORT_SYMBOL
------------------------------------

В дополнение к макросам EXPORT_SYMBOL() и EXPORT_SYMBOL_GPL(), которые позволяют
экспортировать символы ядра в таблицу символов ядра, доступны их варианты для
экспорта символов в определённое пространство имён: EXPORT_SYMBOL_NS() и
EXPORT_SYMBOL_NS_GPL(). Они принимают один дополнительный аргумент: пространство
имён в виде строковой константы. Обратите внимание, что эта строка не должна
содержать пробельных символов.
Например, чтобы экспортировать символ ``usb_stor_suspend`` в пространство имён
``USB_STORAGE``, используйте::

	EXPORT_SYMBOL_NS(usb_stor_suspend, "USB_STORAGE");

В соответствующей записи ksymtab struct ``kernel_symbol`` поле ``namespace``
будет установлено соответствующим образом. Символ, экспортированный без
пространства имён, будет ссылаться на ``NULL``. Если ни одно пространство имён не
определено, пространство имён по умолчанию отсутствует. ``modpost`` и
kernel/module/main.c используют пространство имён во время сборки или во время
загрузки модуля соответственно.

Использование определения DEFAULT_SYMBOL_NAMESPACE
--------------------------------------------------

Определение пространств имён для всех символов подсистемы может быть весьма
многословным и трудным для сопровождения. Поэтому предоставляется определение по
умолчанию (DEFAULT_SYMBOL_NAMESPACE), которое, если оно задано, станет значением
по умолчанию для всех раскрытий макросов EXPORT_SYMBOL() и EXPORT_SYMBOL_GPL(),
не указывающих пространство имён.

Существует несколько способов задать это определение, и то, какой из них
использовать, зависит от подсистемы и предпочтений сопровождающего. Первый
вариант — определить пространство имён по умолчанию в файле ``Makefile``
подсистемы. Например, чтобы экспортировать все символы, определённые в
usb-common, в пространство имён USB_COMMON, добавьте строку вроде следующей в
drivers/usb/common/Makefile::

	ccflags-y += -DDEFAULT_SYMBOL_NAMESPACE='"USB_COMMON"'

Это повлияет на все выражения EXPORT_SYMBOL() и EXPORT_SYMBOL_GPL(). Символ,
экспортированный с помощью EXPORT_SYMBOL_NS() при наличии этого определения,
всё равно будет экспортирован в пространство имён, переданное в качестве
аргумента namespace, поскольку этот аргумент имеет приоритет над пространством
имён по умолчанию.

Второй вариант задать пространство имён по умолчанию — непосредственно в единице
компиляции в виде директивы препроцессора. В этом случае приведённый выше пример
выглядел бы так::

	#define DEFAULT_SYMBOL_NAMESPACE "USB_COMMON"

в соответствующей единице компиляции перед #include для <linux/export.h>. Обычно
оно размещается перед первой директивой #include.

Использование макроса EXPORT_SYMBOL_FOR_MODULES()
-------------------------------------------------

Символы, экспортированные с помощью этого макроса, помещаются в пространство имён
модуля. Это пространство имён нельзя импортировать. Такие экспорты доступны
только под GPL (GPL-only), так как они предназначены только для внутридеревенских
(in-tree) модулей.

Макрос принимает разделённый запятыми список имён модулей, разрешая доступ к
этому символу только этим модулям. Поддерживаются простые хвостовые глобы (tail-globs).

Например::

  EXPORT_SYMBOL_FOR_MODULES(preempt_notifier_inc, "kvm,kvm-*")

ограничит использование этого символа модулями, имена которых соответствуют
заданным шаблонам.

Как использовать символы, экспортированные в пространства имён
==============================================================

Чтобы использовать символы, экспортированные в пространства имён, модули ядра
должны явно импортировать эти пространства имён. В противном случае ядро может
отказаться загружать модуль. Код модуля обязан использовать макрос
MODULE_IMPORT_NS для тех пространств имён, из которых он использует символы.
Например, модуль, использующий приведённый выше символ usb_stor_suspend, должен
импортировать пространство имён USB_STORAGE с помощью выражения вроде::

	MODULE_IMPORT_NS("USB_STORAGE");

Это создаст в модуле тег ``modinfo`` для каждого импортированного пространства
имён. Это имеет побочный эффект: импортированные пространства имён модуля можно
просмотреть с помощью modinfo::

	$ modinfo drivers/usb/storage/ums-karma.ko
	[...]
	import_ns:      USB_STORAGE
	[...]


Рекомендуется добавлять выражение MODULE_IMPORT_NS() рядом с другими
определениями метаданных модуля, такими как MODULE_AUTHOR() или
MODULE_LICENSE().

Загрузка модулей, использующих символы из пространств имён
==========================================================

Во время загрузки модуля (например, ``insmod``) ядро проверяет каждый символ,
на который ссылается модуль, на его доступность, а также на то, было ли
импортировано модулем пространство имён, в которое этот символ мог быть
экспортирован. Поведение ядра по умолчанию — отказывать в загрузке модулей,
которые не указывают достаточных импортов. Будет записана ошибка, и загрузка
завершится неудачей с EINVAL. Чтобы разрешить загрузку модулей, не
удовлетворяющих этому предусловию, доступен параметр конфигурации: установка
MODULE_ALLOW_MISSING_NAMESPACE_IMPORTS=y разрешит загрузку в любом случае, но
выдаст предупреждение.

Автоматическое создание выражений MODULE_IMPORT_NS
==================================================

Отсутствующие импорты пространств имён легко обнаружить во время сборки. На самом
деле, modpost выдаст предупреждение, если модуль использует символ из
пространства имён, не импортировав его.
Выражения MODULE_IMPORT_NS() обычно добавляются в определённом месте (вместе с
прочими метаданными модуля). Чтобы упростить жизнь авторам модулей (и
сопровождающим подсистем), доступны скрипт и make-цель для исправления
отсутствующих импортов. Исправить отсутствующие импорты можно с помощью::

	$ make nsdeps

Типичный сценарий для авторов модулей выглядел бы так::

	- написать код, зависящий от символа из неимпортированного пространства имён
	- ``make``
	- заметить предупреждение modpost об отсутствующем импорте
	- запустить ``make nsdeps``, чтобы добавить импорт в правильное место в коде

Для сопровождающих подсистем, вводящих пространство имён, шаги очень похожи. И
снова ``make nsdeps`` в итоге добавит отсутствующие импорты пространств имён для
внутридеревенских (in-tree) модулей::

	- переместить или добавить символы в пространство имён (например, с помощью EXPORT_SYMBOL_NS())
	- ``make`` (желательно с allmodconfig, чтобы охватить все внутриядерные
	  модули)
	- заметить предупреждение modpost об отсутствующем импорте
	- запустить ``make nsdeps``, чтобы добавить импорт в правильное место в коде

Вы также можете запускать nsdeps для сборок внешних модулей. Типичное
использование::

	$ make -C <path_to_kernel_src> M=$PWD nsdeps

Примечание: он с готовностью сгенерирует выражение импорта для пространства имён
модуля, что не сработает и приведёт к ошибкам сборки и времени выполнения.
