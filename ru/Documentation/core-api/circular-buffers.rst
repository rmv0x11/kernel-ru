===================
Кольцевые буферы
===================

:Author: David Howells <dhowells@redhat.com>
:Author: Paul E. McKenney <paulmck@linux.ibm.com>


Linux предоставляет ряд возможностей, которые можно использовать для реализации
кольцевой буферизации. Существует два набора таких возможностей:

 (1) Вспомогательные функции для определения информации о буферах, размер которых
     является степенью двойки.

 (2) Барьеры памяти для случаев, когда производитель и потребитель объектов в
     буфере не хотят разделять общую блокировку.

Чтобы пользоваться этими средствами, как описано ниже, должен быть ровно один
производитель и ровно один потребитель. Несколько производителей можно обслужить,
сериализовав их, а несколько потребителей — сериализовав их.

Что такое кольцевой буфер?
==========================

Прежде всего, что такое кольцевой буфер? Кольцевой буфер — это буфер
фиксированного, конечного размера, в который ведут два индекса:

 (1) Индекс «головы» (head) — точка, в которую производитель вставляет элементы в
     буфер.

 (2) Индекс «хвоста» (tail) — точка, в которой потребитель находит следующий
     элемент в буфере.

Обычно, когда указатель хвоста равен указателю головы, буфер пуст; а буфер полон,
когда указатель головы на единицу меньше указателя хвоста.

Индекс головы увеличивается при добавлении элементов, а индекс хвоста — при их
удалении. Индекс хвоста никогда не должен обгонять индекс головы, и оба индекса
должны заворачиваться в 0 при достижении конца буфера, что тем самым позволяет
бесконечному объёму данных проходить через буфер.

Обычно все элементы имеют одинаковый единичный размер, но для использования
описанных ниже приёмов это не является строго обязательным. Индексы могут
увеличиваться более чем на 1, если в буфер должно быть включено несколько
элементов или элементы переменного размера, при условии, что ни один из индексов
не обгоняет другой. Однако разработчик должен быть внимателен, так как область
размером более одной единицы может завернуть за конец буфера и быть разбита на два
сегмента.

Измерение буферов размера степени двойки
========================================

Вычисление заполненности или оставшейся ёмкости кольцевого буфера произвольного
размера обычно является медленной операцией, требующей использования инструкции
взятия остатка от деления (деления). Однако если размер буфера является степенью
двойки, то вместо неё можно использовать гораздо более быструю инструкцию
побитового И.

Linux предоставляет набор макросов для работы с кольцевыми буферами, размер
которых является степенью двойки. Их можно использовать так::

	#include <linux/circ_buf.h>

Эти макросы таковы:

 (#) Измерить оставшуюся ёмкость буфера::

	CIRC_SPACE(head_index, tail_index, buffer_size);

     Это возвращает объём оставшегося в буфере[1] места, в которое можно вставлять
     элементы.


 (#) Измерить максимальное непрерывное непосредственно доступное место в буфере::

	CIRC_SPACE_TO_END(head_index, tail_index, buffer_size);

     Это возвращает объём непрерывного оставшегося в буфере[1] места, в которое
     можно немедленно вставлять элементы без необходимости заворачивать обратно к
     началу буфера.


 (#) Измерить заполненность буфера::

	CIRC_CNT(head_index, tail_index, buffer_size);

     Это возвращает количество элементов, занимающих буфер в данный момент[2].


 (#) Измерить незаворачивающуюся заполненность буфера::

	CIRC_CNT_TO_END(head_index, tail_index, buffer_size);

     Это возвращает количество последовательных элементов[2], которые можно извлечь
     из буфера без необходимости заворачивать обратно к началу буфера.


Каждый из этих макросов номинально возвращает значение от 0 до buffer_size-1,
однако:

 (1) CIRC_SPACE*() предназначены для использования в производителе. Производителю
     они возвращают нижнюю границу, так как производитель управляет индексом головы,
     но потребитель может всё ещё опустошать буфер на другом CPU и перемещать индекс
     хвоста.

     Потребителю это покажет верхнюю границу, так как производитель может быть занят
     заполнением свободного места.

 (2) CIRC_CNT*() предназначены для использования в потребителе. Потребителю они
     возвращают нижнюю границу, так как потребитель управляет индексом хвоста, но
     производитель может всё ещё заполнять буфер на другом CPU и перемещать индекс
     головы.

     Производителю это покажет верхнюю границу, так как потребитель может быть занят
     опустошением буфера.

 (3) Для третьей стороны порядок, в котором становятся видимыми записи в индексы со
     стороны производителя и потребителя, не может быть гарантирован, так как они
     независимы и могут выполняться на разных CPU — поэтому результат в такой
     ситуации будет лишь догадкой и может даже оказаться отрицательным.

Использование барьеров памяти с кольцевыми буферами
===================================================

Используя барьеры памяти совместно с кольцевыми буферами, можно избежать
необходимости:

 (1) использовать единую блокировку для управления доступом к обоим концам буфера,
     что тем самым позволяет заполнять и опустошать буфер одновременно; и

 (2) использовать атомарные операции со счётчиком.

У этого есть две стороны: производитель, который заполняет буфер, и потребитель,
который его опустошает. В любой момент времени буфер должно заполнять только что-то
одно, и в любой момент времени буфер должно опустошать только что-то одно, но обе
стороны могут работать одновременно.


Производитель
-------------

Производитель будет выглядеть примерно так::

	spin_lock(&producer_lock);

	unsigned long head = buffer->head;
	/* The spin_unlock() and next spin_lock() provide needed ordering. */
	unsigned long tail = READ_ONCE(buffer->tail);

	if (CIRC_SPACE(head, tail, buffer->size) >= 1) {
		/* insert one item into the buffer */
		struct item *item = buffer[head];

		produce_item(item);

		smp_store_release(buffer->head,
				  (head + 1) & (buffer->size - 1));

		/* wake_up() will make sure that the head is committed before
		 * waking anyone up */
		wake_up(consumer);
	}

	spin_unlock(&producer_lock);

Это укажет CPU, что содержимое нового элемента должно быть записано прежде, чем
индекс головы сделает его доступным потребителю, а затем укажет CPU, что
изменённый индекс головы должен быть записан прежде, чем потребитель будет
разбужен.

Обратите внимание, что wake_up() не гарантирует никакого барьера, если только
что-то действительно не было разбужено. Поэтому мы не можем полагаться на него для
упорядочивания. Однако всегда остаётся один пустой элемент массива. Поэтому
производитель должен произвести два элемента, прежде чем он смог бы потенциально
повредить элемент, читаемый потребителем в данный момент. Поэтому пара
unlock-lock между последовательными вызовами потребителя обеспечивает необходимое
упорядочивание между чтением индекса, указывающего, что потребитель освободил
данный элемент, и записью производителя в этот же элемент.


Потребитель
-----------

Потребитель будет выглядеть примерно так::

	spin_lock(&consumer_lock);

	/* Read index before reading contents at that index. */
	unsigned long head = smp_load_acquire(buffer->head);
	unsigned long tail = buffer->tail;

	if (CIRC_CNT(head, tail, buffer->size) >= 1) {

		/* extract one item from the buffer */
		struct item *item = buffer[tail];

		consume_item(item);

		/* Finish reading descriptor before incrementing tail. */
		smp_store_release(buffer->tail,
				  (tail + 1) & (buffer->size - 1));
	}

	spin_unlock(&consumer_lock);

Это укажет CPU убедиться, что индекс актуален, прежде чем читать новый элемент, а
затем он обеспечит, что CPU завершил чтение элемента, прежде чем он запишет новый
указатель хвоста, который сотрёт этот элемент.

Обратите внимание на использование READ_ONCE() и smp_load_acquire() для чтения
противоположного индекса. Это не даёт компилятору отбросить и перезагрузить его
закэшированное значение. Это не является строго необходимым, если вы можете быть
уверены, что противоположный индекс будет использован _только_ один раз.
smp_load_acquire() дополнительно заставляет CPU упорядочивать относительно
последующих обращений к памяти. Аналогично, smp_store_release() используется в
обоих алгоритмах для записи индекса потока. Это документирует тот факт, что мы
записываем нечто, что может читаться конкурентно, не даёт компилятору разорвать
запись на части и обеспечивает упорядочивание относительно предыдущих обращений.
