=========
Workqueue
=========

:Date: September, 2010
:Author: Tejun Heo <tj@kernel.org>
:Author: Florian Mickler <florian@mickler.org>


Введение
========

Существует множество случаев, когда требуется контекст для асинхронного
выполнения процесса, и API workqueue (wq) является наиболее часто
используемым механизмом для таких случаев.

Когда требуется такой контекст асинхронного выполнения, в очередь
помещается рабочий элемент (work item), описывающий, какую функцию нужно
выполнить. Контекстом асинхронного выполнения служит независимый поток.
Эта очередь называется workqueue, а поток называется worker.

Пока в workqueue есть рабочие элементы, worker выполняет связанные с ними
функции одну за другой. Когда в workqueue не остаётся ни одного рабочего
элемента, worker переходит в состояние простоя. Когда в очередь помещается
новый рабочий элемент, worker снова начинает выполнение.


Зачем нужен Concurrency Managed Workqueue?
==========================================

В первоначальной реализации wq многопоточная (multi threaded, MT) wq имела
по одному рабочему потоку на CPU, а однопоточная (single threaded, ST) wq
имела один рабочий поток на всю систему. Одна MT wq должна была держать
столько же рабочих потоков, сколько имелось CPU. С годами в ядре появилось
много пользователей MT wq, и при непрерывном росте числа ядер CPU некоторые
системы исчерпывали стандартное пространство PID размером 32k уже на этапе
загрузки.

Хотя MT wq расходовала много ресурсов, предоставляемый уровень
параллелизма был неудовлетворительным. Это ограничение было общим как для
ST, так и для MT wq, хотя для MT оно было менее серьёзным. Каждая wq
поддерживала свой собственный отдельный пул рабочих потоков. MT wq могла
предоставить лишь один контекст выполнения на CPU, а ST wq — один на всю
систему. Рабочим элементам приходилось конкурировать за эти весьма
ограниченные контексты выполнения, что приводило к различным проблемам,
включая склонность к взаимоблокировкам (deadlock) вокруг единственного
контекста выполнения.

Противоречие между предоставляемым уровнем параллелизма и использованием
ресурсов также вынуждало пользователей идти на ненужные компромиссы:
например, libata выбирала использование ST wq для опроса PIO, мирясь с
ненужным ограничением, при котором никакие два опрашиваемых PIO не могут
выполняться одновременно. Поскольку MT wq не обеспечивает существенно
лучшего параллелизма, пользователям, которым требовался более высокий
уровень параллелизма, например async или fscache, приходилось реализовывать
собственный пул потоков.

Concurrency Managed Workqueue (cmwq) — это переработанная реализация wq,
ориентированная на достижение следующих целей.

* Сохранение совместимости с первоначальным API workqueue.

* Использование единых пулов рабочих потоков на каждый CPU, разделяемых
  всеми wq, чтобы по требованию обеспечивать гибкий уровень параллелизма
  без значительного расходования ресурсов.

* Автоматическое регулирование пула рабочих потоков и уровня параллелизма,
  чтобы пользователям API не приходилось беспокоиться о таких деталях.


Архитектура
===========

Чтобы облегчить асинхронное выполнение функций, вводится новая абстракция —
рабочий элемент (work item).

Рабочий элемент представляет собой простую структуру, которая содержит
указатель на функцию, подлежащую асинхронному выполнению. Каждый раз, когда
драйвер или подсистема хочет, чтобы некоторая функция была выполнена
асинхронно, он должен создать рабочий элемент, указывающий на эту функцию,
и поместить этот рабочий элемент в workqueue.

Рабочий элемент может выполняться либо в потоке, либо в контексте BH
(softirq).

В потоковых workqueue функции из очереди одну за другой выполняют
специальные потоки, называемые [k]worker'ами. Если в очереди нет работы,
рабочие потоки переходят в состояние простоя. Эти рабочие потоки управляются
в пулах рабочих потоков (worker-pools).

Архитектура cmwq различает обращённые к пользователю workqueue, в которые
подсистемы и драйверы помещают рабочие элементы, и внутренний механизм,
который управляет пулами рабочих потоков и обрабатывает поставленные в
очередь рабочие элементы.

Для каждого возможного CPU есть два пула рабочих потоков — один для обычных
рабочих элементов, а другой для высокоприоритетных, — а также несколько
дополнительных пулов рабочих потоков для обслуживания рабочих элементов,
поставленных в очередь несвязанных (unbound) workqueue; число таких
обеспечивающих пулов является динамическим.

Workqueue типа BH используют тот же каркас. Однако, поскольку может
существовать лишь один параллельный контекст выполнения, заботиться о
параллелизме не нужно. Каждый пул рабочих потоков BH на каждом CPU содержит
лишь один псевдо-worker, который представляет контекст выполнения BH.
Workqueue типа BH можно рассматривать как удобный интерфейс к softirq.

Подсистемы и драйверы могут по своему усмотрению создавать и помещать в
очередь рабочие элементы с помощью специальных функций API workqueue. Они
могут влиять на некоторые аспекты способа выполнения рабочих элементов,
устанавливая флаги на той workqueue, в которую они помещают рабочий элемент.
Эти флаги включают такие вещи, как привязка к CPU, ограничения параллелизма,
приоритет и прочее. Для подробного обзора обратитесь к описанию API
``alloc_workqueue()`` ниже.

Когда рабочий элемент помещается в workqueue, целевой пул рабочих потоков
определяется в соответствии с параметрами постановки в очередь и атрибутами
workqueue, и элемент добавляется в общий список работ (worklist) этого пула
рабочих потоков. Например, если это специально не переопределено, рабочий
элемент связанной (bound) workqueue будет поставлен в worklist обычного или
высокоприоритетного (highpri) пула рабочих потоков, связанного с тем CPU, на
котором выполняется инициатор (issuer).

Для любой реализации пула потоков важным вопросом является управление
уровнем параллелизма (тем, сколько контекстов выполнения активно). cmwq
старается поддерживать параллелизм на минимальном, но достаточном уровне.
Минимальном — чтобы экономить ресурсы, и достаточном — в том смысле, что
система используется на полную мощность.

Каждый пул рабочих потоков, привязанный к реальному CPU, реализует
управление параллелизмом, подключаясь к планировщику. Пул рабочих потоков
уведомляется каждый раз, когда активный worker просыпается или засыпает, и
отслеживает количество текущих способных к выполнению (runnable) рабочих
потоков. Как правило, не предполагается, что рабочие элементы будут
монополизировать CPU и потреблять много циклов. Это означает, что
оптимальным должно быть поддержание ровно достаточного параллелизма, чтобы
предотвратить простой в обработке работы. Пока на CPU есть один или более
способных к выполнению рабочих потоков, пул рабочих потоков не начинает
выполнение новой работы, но когда последний выполняющийся worker засыпает,
он немедленно планирует новый worker, чтобы CPU не простаивал, пока есть
ожидающие рабочие элементы. Это позволяет использовать минимальное число
рабочих потоков, не теряя пропускной способности выполнения.

Сохранение простаивающих рабочих потоков не обходится ни во что, кроме
пространства памяти под kthreads, поэтому cmwq удерживает простаивающие
потоки некоторое время, прежде чем уничтожить их.

Для несвязанных workqueue число обеспечивающих пулов является динамическим.
Несвязанной workqueue можно назначить пользовательские атрибуты с помощью
``apply_workqueue_attrs()``, и workqueue автоматически создаст обеспечивающие
пулы рабочих потоков, соответствующие этим атрибутам. Ответственность за
регулирование уровня параллелизма лежит на пользователях. Существует также
флаг для пометки связанной wq, чтобы она игнорировала управление
параллелизмом. За подробностями обратитесь к разделу про API.

Гарантия продвижения вперёд (forward progress) опирается на то, что рабочие
потоки могут быть созданы, когда требуется больше контекстов выполнения, что
в свою очередь гарантируется за счёт использования спасательных рабочих
потоков (rescue workers). Все рабочие элементы, которые могут использоваться
на путях кода, обрабатывающих освобождение памяти (memory reclaim), обязаны
помещаться в очередь тех wq, у которых зарезервирован спасательный рабочий
поток для выполнения под давлением памяти. В противном случае возможна
ситуация, когда пул рабочих потоков попадёт во взаимоблокировку, ожидая
освобождения контекстов выполнения.


Интерфейс прикладного программирования (API)
=============================================

``alloc_workqueue()`` выделяет wq. Первоначальные функции
``create_*workqueue()`` устарели и запланированы к удалению.
``alloc_workqueue()`` принимает три аргумента — ``@name``, ``@flags`` и
``@max_active``. ``@name`` — это имя wq, которое также используется в
качестве имени потока-спасателя (rescuer), если таковой есть.

Теперь wq не управляет ресурсами выполнения, а служит доменом для гарантии
продвижения вперёд, сброса (flush) и атрибутов рабочих элементов. ``@flags``
и ``@max_active`` управляют тем, как рабочим элементам назначаются ресурсы
выполнения, как они планируются и выполняются.


``flags``
---------

``WQ_BH``
  Workqueue типа BH можно рассматривать как удобный интерфейс к softirq.
  Workqueue типа BH всегда привязаны к конкретному CPU (per-CPU), и все
  рабочие элементы BH выполняются в контексте softirq того CPU, на котором
  они поставлены в очередь, в порядке постановки в очередь.

  Все workqueue типа BH должны иметь ``max_active`` равным 0, и
  ``WQ_HIGHPRI`` — единственный допустимый дополнительный флаг.

  Рабочие элементы BH не могут засыпать. Все остальные возможности, такие как
  отложенная постановка в очередь, сброс и отмена, поддерживаются.

``WQ_PERCPU``
  Рабочие элементы, поставленные в очередь per-cpu wq, привязаны к
  конкретному CPU. Этот флаг — правильный выбор, когда важна привязка к CPU.

  Этот флаг является дополнением к ``WQ_UNBOUND``.

``WQ_UNBOUND``
  Рабочие элементы, поставленные в очередь несвязанной wq, обслуживаются
  специальными пулами рабочих потоков, в которых размещаются worker'ы, не
  привязанные ни к какому конкретному CPU. Это заставляет wq вести себя как
  простой поставщик контекста выполнения без управления параллелизмом.
  Несвязанные пулы рабочих потоков стараются начать выполнение рабочих
  элементов как можно скорее. Несвязанная wq жертвует привязкой, но полезна
  в следующих случаях.

  * Ожидаются широкие колебания требований к уровню параллелизма, и
    использование связанной wq может в итоге привести к созданию большого
    числа в основном неиспользуемых рабочих потоков на разных CPU по мере
    того, как инициатор перемещается между разными CPU.

  * Длительно работающие нагрузки, интенсивно использующие CPU, которыми
    лучше управлять с помощью системного планировщика.

``WQ_FREEZABLE``
  Замораживаемая (freezable) wq участвует в фазе заморозки операций
  приостановки системы (system suspend). Рабочие элементы на этой wq
  опустошаются (drained), и ни один новый рабочий элемент не начинает
  выполнение до размораживания (thawed).

``WQ_MEM_RECLAIM``
  Все wq, которые могут использоваться на путях освобождения памяти,
  **ОБЯЗАНЫ** иметь установленным этот флаг. Гарантируется, что у такой wq
  будет как минимум один контекст выполнения независимо от давления памяти.

``WQ_HIGHPRI``
  Рабочие элементы highpri wq помещаются в высокоприоритетный пул рабочих
  потоков целевого CPU. Высокоприоритетные пулы рабочих потоков
  обслуживаются рабочими потоками с повышенным уровнем nice.

  Обратите внимание, что обычные и высокоприоритетные пулы рабочих потоков не
  взаимодействуют друг с другом. Каждый поддерживает свой отдельный пул
  рабочих потоков и реализует управление параллелизмом среди своих рабочих
  потоков.

``WQ_CPU_INTENSIVE``
  Рабочие элементы wq, интенсивно использующей CPU, не вносят вклад в уровень
  параллелизма. Иными словами, способные к выполнению рабочие элементы,
  интенсивно использующие CPU, не помешают другим рабочим элементам в том же
  пуле рабочих потоков начать выполнение. Это полезно для связанных рабочих
  элементов, от которых ожидается монополизация циклов CPU, так что их
  выполнение регулируется системным планировщиком.

  Хотя рабочие элементы, интенсивно использующие CPU, не вносят вклад в
  уровень параллелизма, начало их выполнения по-прежнему регулируется
  управлением параллелизмом, и способные к выполнению рабочие элементы, не
  интенсивно использующие CPU, могут задержать выполнение рабочих элементов,
  интенсивно использующих CPU.

  Этот флаг не имеет смысла для несвязанной wq.


``max_active``
--------------

``@max_active`` определяет максимальное число контекстов выполнения на CPU,
которые могут быть назначены рабочим элементам wq. Например, при
``@max_active`` равном 16 одновременно на каждом CPU могут выполняться не
более 16 рабочих элементов wq. Это всегда атрибут уровня CPU (per-CPU), даже
для несвязанных workqueue.

Максимальный предел для ``@max_active`` равен 2048, а значение по умолчанию,
используемое при указании 0, равно 1024. Эти значения выбраны достаточно
высокими, чтобы они не были ограничивающим фактором, при этом обеспечивая
защиту в случаях неуправляемого роста.

Число активных рабочих элементов wq обычно регулируется пользователями wq, а
точнее тем, сколько рабочих элементов пользователи могут поставить в очередь
одновременно. Если нет конкретной потребности в ограничении числа активных
рабочих элементов, рекомендуется указывать «0».

Некоторые пользователи зависят от строгого порядка выполнения, при котором в
любой момент времени выполняется только один рабочий элемент, а рабочие
элементы обрабатываются в порядке постановки в очередь. Хотя раньше для
достижения такого поведения использовалось сочетание ``@max_active`` равного
1 и ``WQ_UNBOUND``, теперь это не так. Вместо этого используйте
alloc_ordered_workqueue().


Примеры сценариев выполнения
============================

Следующие примеры сценариев выполнения призваны проиллюстрировать, как cmwq
ведёт себя при различных конфигурациях.

 Рабочие элементы w0, w1, w2 поставлены в очередь связанной wq q0 на одном и
 том же CPU. w0 нагружает CPU в течение 5 мс, затем спит 10 мс, затем снова
 нагружает CPU в течение 5 мс, прежде чем завершиться. w1 и w2 нагружают CPU
 в течение 5 мс, затем спят 10 мс.

Игнорируя все остальные задачи, работы и накладные расходы на обработку, и
предполагая простое планирование FIFO, ниже приведена одна сильно
упрощённая версия возможной последовательности событий с первоначальной wq.
::

 TIME IN MSECS	EVENT
 0		w0 starts and burns CPU
 5		w0 sleeps
 15		w0 wakes up and burns CPU
 20		w0 finishes
 20		w1 starts and burns CPU
 25		w1 sleeps
 35		w1 wakes up and finishes
 35		w2 starts and burns CPU
 40		w2 sleeps
 50		w2 wakes up and finishes

А с cmwq при ``@max_active`` >= 3, ::

 TIME IN MSECS	EVENT
 0		w0 starts and burns CPU
 5		w0 sleeps
 5		w1 starts and burns CPU
 10		w1 sleeps
 10		w2 starts and burns CPU
 15		w2 sleeps
 15		w0 wakes up and burns CPU
 20		w0 finishes
 20		w1 wakes up and finishes
 25		w2 wakes up and finishes

Если ``@max_active`` == 2, ::

 TIME IN MSECS	EVENT
 0		w0 starts and burns CPU
 5		w0 sleeps
 5		w1 starts and burns CPU
 10		w1 sleeps
 15		w0 wakes up and burns CPU
 20		w0 finishes
 20		w1 wakes up and finishes
 20		w2 starts and burns CPU
 25		w2 sleeps
 35		w2 wakes up and finishes

Теперь предположим, что w1 и w2 поставлены в очередь другой wq q1, у которой
установлен ``WQ_CPU_INTENSIVE``, ::

 TIME IN MSECS	EVENT
 0		w0 starts and burns CPU
 5		w0 sleeps
 5		w1 and w2 start and burn CPU
 10		w1 sleeps
 15		w2 sleeps
 15		w0 wakes up and burns CPU
 20		w0 finishes
 20		w1 wakes up and finishes
 25		w2 wakes up and finishes


Рекомендации
============

* Не забывайте использовать ``WQ_MEM_RECLAIM``, если wq может обрабатывать
  рабочие элементы, используемые во время освобождения памяти. Для каждой wq
  с установленным ``WQ_MEM_RECLAIM`` зарезервирован контекст выполнения.
  Если между несколькими рабочими элементами, используемыми во время
  освобождения памяти, есть зависимость, они должны помещаться в отдельные
  wq, каждая с ``WQ_MEM_RECLAIM``.

* Если строгий порядок не требуется, использовать ST wq нет необходимости.

* Если нет конкретной потребности, рекомендуется использовать 0 для
  @max_active. В большинстве случаев использования уровень параллелизма
  обычно остаётся значительно ниже предела по умолчанию.

* wq служит доменом для гарантии продвижения вперёд (``WQ_MEM_RECLAIM``,
  сброса и атрибутов рабочих элементов. Рабочие элементы, которые не
  задействованы в освобождении памяти, не нуждаются в сбросе в составе
  группы рабочих элементов и не требуют каких-либо специальных атрибутов,
  могут использовать одну из системных wq. Между использованием выделенной
  wq и системной wq нет разницы в характеристиках выполнения.

  Примечание: если что-то может породить более чем @max_active незавершённых
  рабочих элементов (проведите стресс-тест своих производителей), это может
  насытить системную wq и потенциально привести к взаимоблокировке. В этом
  случае следует использовать собственную выделенную workqueue, а не
  системную wq.

* Если только от рабочих элементов не ожидается потребления огромного
  количества циклов CPU, использование связанной wq обычно выгодно
  благодаря повышенному уровню локальности в операциях wq и выполнении
  рабочих элементов.


Области привязки (Affinity Scopes)
==================================

Несвязанная workqueue группирует CPU в соответствии со своей областью
привязки (affinity scope) для улучшения локальности кэша. Например, если
workqueue использует область привязки по умолчанию «cache», она будет
группировать CPU в соответствии с границами кэша последнего уровня. Рабочий
элемент, поставленный в очередь workqueue, будет назначен worker'у на одном
из CPU, разделяющих кэш последнего уровня с инициирующим CPU. После запуска
worker'у может быть разрешено или не разрешено выходить за пределы области в
зависимости от настройки ``affinity_strict`` этой области.

В настоящее время workqueue поддерживает следующие области привязки.

``default``
  Использовать область, указанную в параметре модуля
  ``workqueue.default_affinity_scope``, который всегда установлен в одну из
  приведённых ниже областей.

``cpu``
  CPU не группируются. Рабочий элемент, инициированный на одном CPU,
  обрабатывается worker'ом на том же CPU. Это заставляет несвязанные
  workqueue вести себя как per-cpu workqueue без управления параллелизмом.

``smt``
  CPU группируются в соответствии с границами SMT. Обычно это означает, что
  логические потоки каждого физического ядра CPU группируются вместе.

``cache``
  CPU группируются в соответствии с границами кэша. То, какая именно граница
  кэша используется, определяется кодом архитектуры. Во многих случаях
  используется L3. Это область привязки по умолчанию.

``numa``
  CPU группируются в соответствии с границами NUMA.

``system``
  Все CPU помещаются в одну группу. Workqueue не прилагает усилий к тому,
  чтобы обработать рабочий элемент на CPU, близком к инициирующему CPU.

Область привязки по умолчанию можно изменить с помощью параметра модуля
``workqueue.default_affinity_scope``, а область привязки конкретной workqueue
можно изменить с помощью ``apply_workqueue_attrs()``.

Если установлен ``WQ_SYSFS``, у workqueue будут следующие файлы интерфейса,
связанные с областью привязки, в её каталоге
``/sys/devices/virtual/workqueue/WQ_NAME/``.

``affinity_scope``
  Чтение показывает текущую область привязки. Запись изменяет её.

  Когда текущей областью является default, чтение этого файла также покажет
  в скобках текущую действующую область, например, ``default (cache)``.

``affinity_strict``
  По умолчанию 0, что означает, что области привязки не являются строгими.
  Когда рабочий элемент начинает выполнение, workqueue по принципу
  максимальных усилий пытается обеспечить, чтобы worker находился внутри
  своей области привязки, что называется репатриацией (repatriation). После
  запуска планировщик волен перемещать worker'а куда угодно в системе по
  своему усмотрению. Это позволяет извлекать выгоду из локальности области и
  при этом по-прежнему иметь возможность задействовать другие CPU, если это
  необходимо и доступно.

  Если установлено в 1, гарантируется, что все worker'ы области всегда
  находятся внутри области. Это может быть полезно, когда пересечение
  областей привязки имеет иные последствия, например, в плане потребления
  энергии или изоляции нагрузки. Строгую область NUMA также можно
  использовать для воспроизведения поведения workqueue в старых ядрах.


Области привязки и производительность
=====================================

Было бы идеально, если бы поведение несвязанной workqueue было оптимальным
для подавляющего большинства случаев использования без дальнейшей настройки.
К сожалению, в текущем ядре существует выраженный компромисс между
локальностью и утилизацией, требующий явных конфигураций, когда workqueue
интенсивно используются.

Более высокая локальность ведёт к более высокой эффективности, при которой
за то же число потреблённых циклов CPU выполняется больше работы. Однако
более высокая локальность также может приводить к более низкой общей
утилизации системы, если рабочие элементы недостаточно распределяются
инициаторами по областям привязки. Следующее тестирование
производительности с dm-crypt наглядно иллюстрирует этот компромисс.

Тесты проводятся на CPU с 12 ядрами / 24 потоками, распределёнными по
четырём кэшам L3 (AMD Ryzen 9 3900x). Для постоянства буст тактовой частоты
CPU отключён. ``/dev/dm-0`` — это устройство dm-crypt, созданное на NVME SSD
(Samsung 990 PRO) и открытое с помощью ``cryptsetup`` с настройками по
умолчанию.


Сценарий 1: достаточно инициаторов, и работа распределена по машине
-------------------------------------------------------------------

Используемая команда: ::

  $ fio --filename=/dev/dm-0 --direct=1 --rw=randrw --bs=32k --ioengine=libaio \
    --iodepth=64 --runtime=60 --numjobs=24 --time_based --group_reporting \
    --name=iops-test-job --verify=sha512

Имеется 24 инициатора, каждый из которых параллельно выдаёт по 64 операции
ввода-вывода. ``--verify=sha512`` заставляет ``fio`` каждый раз
генерировать и считывать обратно содержимое, из-за чего локальность
выполнения между инициатором и ``kcryptd`` имеет значение. Ниже приведены
пропускные способности чтения и утилизации CPU в зависимости от различных
настроек области привязки на ``kcryptd``, измеренные по пяти прогонам.
Пропускные способности указаны в MiBps, а утилизация CPU — в процентах.

.. list-table::
   :widths: 16 20 20
   :header-rows: 1

   * - Affinity
     - Bandwidth (MiBps)
     - CPU util (%)

   * - system
     - 1159.40 ±1.34
     - 99.31 ±0.02

   * - cache
     - 1166.40 ±0.89
     - 99.34 ±0.01

   * - cache (strict)
     - 1166.00 ±0.71
     - 99.35 ±0.01

При достаточном числе инициаторов, распределённых по системе, у «cache»,
строгой или нет, нет недостатков. Все три конфигурации насыщают всю машину,
но привязанные к кэшу опережают на 0,6% благодаря улучшенной локальности.


Сценарий 2: меньше инициаторов, работы достаточно для насыщения
---------------------------------------------------------------

Используемая команда: ::

  $ fio --filename=/dev/dm-0 --direct=1 --rw=randrw --bs=32k \
    --ioengine=libaio --iodepth=64 --runtime=60 --numjobs=8 \
    --time_based --group_reporting --name=iops-test-job --verify=sha512

Единственное отличие от предыдущего сценария — ``--numjobs=8``. Инициаторов
теперь треть, но суммарной работы всё ещё достаточно для насыщения системы.

.. list-table::
   :widths: 16 20 20
   :header-rows: 1

   * - Affinity
     - Bandwidth (MiBps)
     - CPU util (%)

   * - system
     - 1155.40 ±0.89
     - 97.41 ±0.05

   * - cache
     - 1154.40 ±1.14
     - 96.15 ±0.09

   * - cache (strict)
     - 1112.00 ±4.64
     - 93.26 ±0.35

Это более чем достаточно работы для насыщения системы. И «system», и
«cache» почти насыщают машину, но не полностью. «cache» использует меньше
CPU, но лучшая эффективность ставит его на ту же пропускную способность, что
и «system».

Восемь инициаторов, перемещающихся по области из четырёх кэшей L3,
по-прежнему позволяют «cache (strict)» в основном насыщать машину, но потеря
сохранения работы (work conservation) теперь начинает сказываться потерей
пропускной способности в 3,7%.


Сценарий 3: ещё меньше инициаторов, работы недостаточно для насыщения
---------------------------------------------------------------------

Используемая команда: ::

  $ fio --filename=/dev/dm-0 --direct=1 --rw=randrw --bs=32k \
    --ioengine=libaio --iodepth=64 --runtime=60 --numjobs=4 \
    --time_based --group_reporting --name=iops-test-job --verify=sha512

Снова единственное отличие — ``--numjobs=4``. При сокращении числа
инициаторов до четырёх работы теперь недостаточно для насыщения всей
системы, и пропускная способность становится зависимой от задержек
завершения.

.. list-table::
   :widths: 16 20 20
   :header-rows: 1

   * - Affinity
     - Bandwidth (MiBps)
     - CPU util (%)

   * - system
     - 993.60 ±1.82
     - 75.49 ±0.06

   * - cache
     - 973.40 ±1.52
     - 74.90 ±0.07

   * - cache (strict)
     - 828.20 ±4.49
     - 66.84 ±0.29

Теперь компромисс между локальностью и утилизацией стал яснее. «cache»
показывает потерю пропускной способности в 2% по сравнению с «system», а
«cache (struct)» — целых 20%.


Заключение и рекомендации
-------------------------

В вышеприведённых экспериментах преимущество в эффективности области
привязки «cache» над «system», будучи устойчивым и заметным, невелико.
Однако это влияние зависит от расстояний между областями и может быть более
выраженным в процессорах с более сложными топологиями.

Хотя потеря сохранения работы в определённых сценариях сказывается, это
намного лучше, чем «cache (strict)», и максимизация утилизации workqueue
всё равно вряд ли будет распространённым случаем. Поэтому «cache» является
областью привязки по умолчанию для несвязанных пулов.

* Поскольку нет единственного варианта, который был бы хорош для большинства
  случаев, для тех способов использования workqueue, которые могут
  потреблять значительное количество CPU, рекомендуется настраивать
  workqueue с помощью ``apply_workqueue_attrs()`` и/или включать
  ``WQ_SYSFS``.

* Несвязанная workqueue со строгой областью привязки «cpu» ведёт себя так
  же, как per-cpu workqueue типа ``WQ_CPU_INTENSIVE``. У последней нет
  реального преимущества, а несвязанная workqueue даёт намного больше
  гибкости.

* Области привязки введены в Linux v6.5. Чтобы воспроизвести прежнее
  поведение, используйте строгую область привязки «numa».

* Потеря сохранения работы в нестрогих областях привязки, вероятно, исходит
  от планировщика. Нет теоретической причины, по которой ядро не было бы в
  состоянии поступать правильно и сохранять сохранение работы в большинстве
  случаев. Поэтому возможно, что будущие улучшения планировщика сделают
  большинство этих регулировок ненужными.

Отладка
=======

Поскольку рабочие функции выполняются обобщёнными рабочими потоками,
требуется несколько приёмов, чтобы пролить свет на некорректно ведущих себя
пользователей workqueue.

Рабочие потоки отображаются в списке процессов как: ::

  root      5671  0.0  0.0      0     0 ?        S    12:07   0:00 [kworker/0:1]
  root      5672  0.0  0.0      0     0 ?        S    12:07   0:00 [kworker/1:2]
  root      5673  0.0  0.0      0     0 ?        S    12:12   0:00 [kworker/0:0]
  root      5674  0.0  0.0      0     0 ?        S    12:13   0:00 [kworker/1:0]

Если kworker'ы сходят с ума (используют слишком много CPU), есть два типа
возможных проблем:

	1. Что-то планируется в быстрой последовательности
	2. Один рабочий элемент, потребляющий много циклов CPU

Первую из них можно отследить с помощью трассировки: ::

	$ echo workqueue:workqueue_queue_work > /sys/kernel/tracing/set_event
	$ cat /sys/kernel/tracing/trace_pipe > out.txt
	(wait a few secs)
	^C

Если что-то занято в цикле постановки работы в очередь, это будет
доминировать в выводе, и нарушителя можно определить по функции рабочего
элемента.

Для второго типа проблем должна быть возможность просто проверить трассировку
стека (stack trace) нарушающего рабочего потока. ::

	$ cat /proc/THE_OFFENDING_KWORKER/stack

Функция рабочего элемента должна быть тривиально видна в трассировке стека.


Условия невозможности повторного входа (Non-reentrance)
=======================================================

Workqueue гарантирует, что рабочий элемент не может быть повторно входимым
(re-entrant), если после постановки рабочего элемента в очередь выполняются
следующие условия:

        1. Рабочая функция не была изменена.
        2. Никто не помещает рабочий элемент в другую workqueue.
        3. Рабочий элемент не был переинициализирован.

Иными словами, если приведённые выше условия выполняются, гарантируется, что
в любой момент времени рабочий элемент выполняется не более чем одним
worker'ом в масштабах всей системы.

Обратите внимание, что повторная постановка рабочего элемента в очередь (в
ту же очередь) в самой функции не нарушает этих условий, поэтому это
безопасно. В остальных случаях при нарушении этих условий внутри рабочей
функции требуется осторожность.
