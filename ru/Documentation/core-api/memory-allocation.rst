===============================
Руководство по выделению памяти
===============================

Linux предоставляет множество API для выделения памяти. Вы можете
выделять небольшие фрагменты с помощью семейств функций `kmalloc` или
`kmem_cache_alloc`, большие виртуально непрерывные области с помощью
`vmalloc` и его производных, либо напрямую запрашивать страницы у
страничного аллокатора через `alloc_pages`. Также возможно использовать
более специализированные аллокаторы, например `cma_alloc` или
`zs_malloc`.

Большинство API выделения памяти используют флаги GFP, чтобы выразить,
каким образом эту память следует выделять. Аббревиатура GFP расшифровывается
как «get free pages» — это название базовой функции выделения памяти.

Разнообразие API выделения в сочетании с многочисленными флагами GFP
делает ответ на вопрос «Как мне выделить память?» не таким уж простым,
хотя весьма вероятно, что вам следует использовать

::

  kzalloc(<size>, GFP_KERNEL);

Разумеется, бывают случаи, когда необходимо использовать другие API
выделения и иные флаги GFP.

Флаги Get Free Page
===================

Флаги GFP управляют поведением аллокаторов. Они указывают, какие зоны
памяти можно использовать, насколько усердно аллокатор должен пытаться
найти свободную память, может ли память быть доступна из пространства
пользователя и т. п. Документ
:ref:`Documentation/core-api/mm-api.rst <mm-api-gfp-flags>` предоставляет
справочную документацию по флагам GFP и их комбинациям, а здесь мы кратко
изложим рекомендуемое использование:

  * В большинстве случаев вам нужен ``GFP_KERNEL``. Память для структур
    данных ядра, память, пригодная для DMA, кэш inode — всё это и многие
    другие типы выделений могут использовать ``GFP_KERNEL``. Заметьте,
    что использование ``GFP_KERNEL`` подразумевает ``GFP_RECLAIM``, а это
    означает, что при нехватке памяти может быть запущено прямое
    высвобождение (direct reclaim); вызывающий контекст должен иметь право
    переходить в сон.
  * Если выделение выполняется из атомарного контекста, например из
    обработчика прерывания, используйте ``GFP_NOWAIT``. Этот флаг
    предотвращает прямое высвобождение, а также операции ввода-вывода и
    файловой системы. Следовательно, при нехватке памяти выделение с
    ``GFP_NOWAIT`` с большой вероятностью завершится неудачей.
    Пользователям этого флага необходимо предусмотреть подходящий
    запасной вариант, чтобы справляться с такими сбоями там, где это
    уместно.
  * Если вы считаете, что обращение к резервам памяти оправдано и что ядро
    окажется в затруднительном положении, если выделение не удастся, вы
    можете использовать ``GFP_ATOMIC``.
  * Недоверенные выделения, инициированные из пространства пользователя,
    должны подлежать учёту kmem и обязаны иметь установленный бит
    ``__GFP_ACCOUNT``. Существует удобное сокращение
    ``GFP_KERNEL_ACCOUNT`` для выделений ``GFP_KERNEL``, которые должны
    учитываться.
  * Выделения для пространства пользователя должны использовать один из
    флагов ``GFP_USER``, ``GFP_HIGHUSER`` или ``GFP_HIGHUSER_MOVABLE``.
    Чем длиннее имя флага, тем менее ограничительным он является.

    ``GFP_HIGHUSER_MOVABLE`` не требует, чтобы выделенная память была
    напрямую доступна ядру, и подразумевает, что данные являются
    перемещаемыми.

    ``GFP_HIGHUSER`` означает, что выделенная память не является
    перемещаемой, но при этом не требуется, чтобы она была напрямую
    доступна ядру. Примером может служить аппаратное выделение, которое
    отображает данные напрямую в пространство пользователя, но не имеет
    ограничений по адресации.

    ``GFP_USER`` означает, что выделенная память не является перемещаемой
    и должна быть напрямую доступна ядру.

Вы можете заметить, что довольно много выделений в существующем коде
указывают ``GFP_NOIO`` или ``GFP_NOFS``. Исторически они использовались
для предотвращения рекурсивных взаимоблокировок (deadlock), вызванных тем,
что прямое высвобождение памяти возвращалось обратно в пути FS или
ввода-вывода и блокировалось на уже удерживаемых ресурсах. Начиная с
версии 4.12 предпочтительным способом решения этой проблемы является
использование новых API областей видимости (scope), описанных в
:ref:`Documentation/core-api/gfp_mask-from-fs-io.rst <gfp_mask_from_fs_io>`.

Другими унаследованными флагами GFP являются ``GFP_DMA`` и ``GFP_DMA32``.
Они используются для гарантии того, что выделенная память будет доступна
оборудованию с ограниченными возможностями адресации. Поэтому, если вы не
пишете драйвер для устройства с такими ограничениями, избегайте
использования этих флагов. И даже при работе с оборудованием, имеющим
ограничения, предпочтительнее использовать API `dma_alloc*`.

Флаги GFP и поведение высвобождения
-----------------------------------
Выделения памяти могут запускать прямое или фоновое высвобождение, и
полезно понимать, насколько усердно страничный аллокатор будет пытаться
удовлетворить тот или иной запрос.

  * ``GFP_KERNEL & ~__GFP_RECLAIM`` — оптимистичное выделение без *какой-либо*
    попытки освободить память вообще. Самый легковесный режим, который даже
    не запускает фоновое высвобождение. Следует использовать осторожно,
    поскольку он может истощить память, и следующему пользователю придётся
    столкнуться с более агрессивным высвобождением.

  * ``GFP_KERNEL & ~__GFP_DIRECT_RECLAIM`` (или ``GFP_NOWAIT``) —
    оптимистичное выделение без какой-либо попытки освободить память из
    текущего контекста, но способное разбудить kswapd для высвобождения
    памяти, если зона находится ниже нижней отметки (low watermark). Может
    использоваться либо из атомарных контекстов, либо когда запрос является
    оптимизацией производительности и существует другой запасной вариант
    для медленного пути.

  * ``(GFP_KERNEL|__GFP_HIGH) & ~__GFP_DIRECT_RECLAIM`` (он же ``GFP_ATOMIC``) —
    неспящее выделение с дорогостоящим запасным вариантом, благодаря чему
    оно может обращаться к некоторой части резервов памяти. Обычно
    используется из контекста прерывания/нижней половины (bottom-half) с
    дорогостоящим запасным вариантом медленного пути.

  * ``GFP_KERNEL`` — разрешены как фоновое, так и прямое высвобождение, и
    используется **поведение страничного аллокатора по умолчанию**. Это
    означает, что недорогостоящие запросы на выделение в принципе не должны
    давать сбоев, однако гарантии такого поведения нет, поэтому сбои должны
    надлежащим образом проверяться вызывающими сторонами (например, в
    настоящее время жертве OOM killer'а разрешено завершаться неудачей).

  * ``GFP_KERNEL | __GFP_NORETRY`` — переопределяет поведение аллокатора по
    умолчанию, и все запросы на выделение завершаются неудачей на раннем
    этапе, вместо того чтобы вызывать разрушительное высвобождение (один
    проход высвобождения в данной реализации). OOM killer не вызывается.

  * ``GFP_KERNEL | __GFP_RETRY_MAYFAIL`` — переопределяет поведение
    аллокатора по умолчанию, и все запросы на выделение стараются изо всех
    сил. Запрос завершится неудачей, если высвобождение не может добиться
    никакого прогресса. OOM killer не будет запущен.

  * ``GFP_KERNEL | __GFP_NOFAIL`` — переопределяет поведение аллокатора по
    умолчанию, и все запросы на выделение будут бесконечно повторяться в
    цикле, пока не завершатся успешно. Это может быть действительно опасно,
    особенно для больших порядков (orders).

Выбор аллокатора памяти
==========================

Самый прямолинейный способ выделить память — использовать функцию из
семейства kmalloc(). И, чтобы быть в безопасности, лучше всего
использовать процедуры, обнуляющие память, такие как kzalloc(). Если вам
нужно выделить память под массив, существуют вспомогательные функции
kmalloc_array() и kcalloc(). Вспомогательные функции struct_size(),
array_size() и array3_size() можно использовать для безопасного расчёта
размеров объектов без переполнения.

Максимальный размер фрагмента, который можно выделить с помощью
`kmalloc`, ограничен. Фактический предел зависит от оборудования и
конфигурации ядра, но хорошей практикой является использование `kmalloc`
для объектов меньше размера страницы.

Адрес фрагмента, выделенного с помощью `kmalloc`, выровнен по меньшей мере
на ARCH_KMALLOC_MINALIGN байт. Для размеров, являющихся степенью двойки,
выравнивание также гарантированно не меньше соответствующего размера. Для
прочих размеров выравнивание гарантированно не меньше наибольшего делителя
размера, являющегося степенью двойки.

Размер фрагментов, выделенных с помощью kmalloc(), можно изменить с
помощью krealloc(). Аналогично kmalloc_array(): вспомогательная функция
для изменения размера массивов предоставляется в виде krealloc_array().

Для больших выделений вы можете использовать vmalloc() и vzalloc() либо
напрямую запрашивать страницы у страничного аллокатора. Память,
выделенная `vmalloc` и связанными функциями, не является физически
непрерывной.

Если вы не уверены, не слишком ли велик размер выделения для `kmalloc`,
можно использовать kvmalloc() и его производные. Она попытается выделить
память с помощью `kmalloc`, а если выделение не удастся, попытка будет
повторена с помощью `vmalloc`. Существуют ограничения на то, какие флаги
GFP можно использовать с `kvmalloc`; пожалуйста, см. справочную
документацию по kvmalloc_node(). Заметьте, что `kvmalloc` может вернуть
память, которая не является физически непрерывной.

Если вам нужно выделить множество идентичных объектов, вы можете
использовать слабовый кэш-аллокатор (slab cache). Кэш должен быть
настроен с помощью kmem_cache_create() или kmem_cache_create_usercopy()
перед тем, как его можно будет использовать. Вторую функцию следует
использовать, если часть кэша может быть скопирована в пространство
пользователя. После создания кэша kmem_cache_alloc() и его удобные
обёртки могут выделять память из этого кэша.

Когда выделенная память больше не нужна, её необходимо освободить.

Объекты, выделенные `kmalloc`, можно освободить с помощью `kfree` или
`kvfree`. Объекты, выделенные `kmem_cache_alloc`, можно освободить с
помощью `kmem_cache_free`, `kfree` или `kvfree`, причём два последних
варианта могут быть удобнее, поскольку не требуют указателя на
kmem_cache.

Те же правила применяются к разновидностям функций освобождения _bulk и
_rcu.

Память, выделенную `vmalloc`, можно освободить с помощью `vfree` или
`kvfree`. Память, выделенную `kvmalloc`, можно освободить с помощью
`kvfree`. Кэши, созданные `kmem_cache_create`, следует освобождать с
помощью `kmem_cache_destroy` только после того, как сначала будут
освобождены все выделенные объекты.
