============================================
Ключи защиты памяти (Memory Protection Keys)
============================================

Ключи защиты памяти (Memory Protection Keys) предоставляют механизм применения
постраничной защиты, но без необходимости изменять таблицы страниц, когда
приложение меняет домены защиты.

Pkeys Userspace (PKU) — это функциональность, которую можно найти на:
        * серверных CPU Intel, Skylake и более новых
        * клиентских CPU Intel, Tiger Lake (11th Gen Core) и более новых
        * будущих CPU AMD
        * CPU arm64, реализующих Permission Overlay Extension (FEAT_S1POE)

x86_64
======
Pkeys работают за счёт выделения 4 ранее зарезервированных бит в каждой записи
таблицы страниц под «ключ защиты», что даёт 16 возможных ключей.

Защита для каждого ключа определяется доступным пользователю регистром,
индивидуальным для каждого CPU (PKRU).  Каждый из них — это 32-битный регистр,
хранящий два бита (Access Disable и Write Disable) для каждого из 16 ключей.

Будучи регистром CPU, PKRU по своей природе локален для потока, потенциально
давая каждому потоку набор средств защиты, отличный от любого другого потока.

Существуют две инструкции (RDPKRU/WRPKRU) для чтения и записи этого регистра.
Эта функциональность доступна только в 64-битном режиме, хотя теоретически
в PAE PTE для неё есть место.  Эти разрешения применяются только при доступе
к данным и не влияют на выборку инструкций.

arm64
=====

Pkeys используют 3 бита в каждой записи таблицы страниц, чтобы закодировать
«индекс ключа защиты», что даёт 8 возможных ключей.

Защита для каждого ключа определяется системным регистром, доступным
пользователю для записи и индивидуальным для каждого CPU (POR_EL0).  Это
64-битный регистр, кодирующий overlay-разрешения на чтение, запись и
исполнение для каждого индекса ключа защиты.

Будучи регистром CPU, POR_EL0 по своей природе локален для потока,
потенциально давая каждому потоку набор средств защиты, отличный от любого
другого потока.

В отличие от x86_64, разрешения ключей защиты применяются также к выборке
инструкций.

Системные вызовы
================

Существуют 3 системных вызова, которые напрямую взаимодействуют с pkeys::

	int pkey_alloc(unsigned long flags, unsigned long init_access_rights)
	int pkey_free(int pkey);
	int pkey_mprotect(unsigned long start, size_t len,
			  unsigned long prot, int pkey);

Прежде чем pkey можно будет использовать, его необходимо сначала выделить с
помощью pkey_alloc().  Чтобы изменить разрешения на доступ к памяти, покрытой
ключом, приложение записывает значение непосредственно в специфичный для
архитектуры регистр CPU.  В этом примере это обёрнуто в C-функцию под названием
pkey_set().
::

	int real_prot = PROT_READ|PROT_WRITE;
	pkey = pkey_alloc(0, PKEY_DISABLE_WRITE);
	ptr = mmap(NULL, PAGE_SIZE, PROT_NONE, MAP_ANONYMOUS|MAP_PRIVATE, -1, 0);
	ret = pkey_mprotect(ptr, PAGE_SIZE, real_prot, pkey);
	... application runs here

Теперь, если приложению нужно обновить данные по 'ptr', оно может получить
доступ, выполнить обновление, а затем убрать своё право на запись::

	pkey_set(pkey, 0); // clear PKEY_DISABLE_WRITE
	*ptr = foo; // assign something
	pkey_set(pkey, PKEY_DISABLE_WRITE); // set PKEY_DISABLE_WRITE again

Теперь, когда оно освобождает память, оно также освободит pkey, поскольку тот
больше не используется::

	munmap(ptr, PAGE_SIZE);
	pkey_free(pkey);

.. note:: pkey_set() — это обёртка вокруг записи в регистр CPU.
          Примеры реализаций можно найти в
          tools/testing/selftests/mm/pkey-{arm64,powerpc,x86}.h

Поведение
=========

Ядро пытается сделать ключи защиты согласованными с поведением обычного
mprotect().  Например, если вы делаете так::

	mprotect(ptr, size, PROT_NONE);
	something(ptr);

то вы можете ожидать такого же эффекта с ключами защиты, делая так::

	pkey = pkey_alloc(0, PKEY_DISABLE_WRITE | PKEY_DISABLE_READ);
	pkey_mprotect(ptr, size, PROT_READ|PROT_WRITE, pkey);
	something(ptr);

Это должно быть верно как в случае, когда something() выполняет прямой доступ
к 'ptr', например::

	*ptr = foo;

так и когда ядро выполняет доступ от имени приложения, например при read()::

	read(fd, ptr, 1);

Ядро отправит SIGSEGV в обоих случаях, но si_code будет установлен в
SEGV_PKERR при нарушении ключей защиты, в отличие от SEGV_ACCERR, когда
нарушаются разрешения обычного mprotect().

Учтите, что доступы из ядра, выполняемые из kthread (например, io_uring),
будут использовать значение регистра ключа защиты по умолчанию и поэтому не
будут согласованы со значением этого регистра в пространстве пользователя или
с mprotect().
