====================================================
pin_user_pages() и связанные вызовы
====================================================

.. contents:: :local:

Обзор
=====

Этот документ описывает следующие функции::

 pin_user_pages()
 pin_user_pages_fast()
 pin_user_pages_remote()

Базовое описание FOLL_PIN
=========================

FOLL_PIN и FOLL_LONGTERM — это флаги, которые могут быть переданы семейству функций
get_user_pages*() («gup»). FOLL_PIN значительно взаимодействует и имеет взаимные
зависимости с FOLL_LONGTERM, поэтому оба рассматриваются здесь.

FOLL_PIN является внутренним для gup, то есть он не должен появляться в местах вызова
gup. Это позволяет соответствующим функциям-обёрткам (pin_user_pages*() и другим)
устанавливать правильную комбинацию этих флагов, а также проверять наличие проблем.

FOLL_LONGTERM, с другой стороны, *разрешается* устанавливать в местах вызова gup. Это
сделано для того, чтобы избежать создания большого числа функций-обёрток, покрывающих все
комбинации get*(), pin*(), FOLL_LONGTERM и прочего. Кроме того, API pin_user_pages*()
явно отличаются от API get_user_pages*(), поэтому это естественная разделительная линия и
хорошая точка для выделения отдельных вызовов-обёрток. Иными словами, используйте
pin_user_pages*() для страниц, закреплённых под DMA, и get_user_pages*() для остальных
случаев. Далее в этом документе описаны пять случаев, чтобы дополнительно прояснить эту
концепцию.

FOLL_PIN и FOLL_GET являются взаимоисключающими для данного вызова gup. Однако несколько
потоков и мест вызова могут свободно закреплять одни и те же struct page как через
FOLL_PIN, так и через FOLL_GET. Выбирать одно или другое должно лишь место вызова, а не
сама struct page.

Реализация FOLL_PIN почти такая же, как у FOLL_GET, за исключением того, что FOLL_PIN
использует другую технику подсчёта ссылок.

FOLL_PIN является предварительным условием для FOLL_LONGTERM. Иначе говоря, FOLL_LONGTERM
— это частный, более ограниченный случай FOLL_PIN.

Какие флаги устанавливает каждая обёртка
========================================

Для этих функций pin_user_pages*() FOLL_PIN объединяется по OR с любыми флагами gup,
которые предоставляет вызывающая сторона. Вызывающая сторона обязана передать ненулевой
массив struct pages*, после чего функция закрепляет страницы, увеличивая каждую на
специальное значение: GUP_PIN_COUNTING_BIAS.

Для больших folio схема GUP_PIN_COUNTING_BIAS не используется. Вместо этого для прямого
хранения pincount используется дополнительное пространство, доступное в struct folio.

Этот подход для больших folio позволяет избежать проблем с верхним пределом счётчика,
которые обсуждаются ниже. Эти ограничения серьёзно усугублялись бы огромными страницами,
поскольку каждая хвостовая страница добавляет refcount к головной странице. И в самом деле,
тестирование показало, что без отдельного поля pincount в некоторых стресс-тестах огромных
страниц наблюдались переполнения refcount.

Это также означает, что огромные страницы и большие folio не страдают от проблемы ложных
срабатываний, упоминаемой ниже.::

 Function
 --------
 pin_user_pages          FOLL_PIN is always set internally by this function.
 pin_user_pages_fast     FOLL_PIN is always set internally by this function.
 pin_user_pages_remote   FOLL_PIN is always set internally by this function.

Для этих функций get_user_pages*() флаг FOLL_GET может быть вообще не указан. Поведение
немного сложнее, чем выше. Если FOLL_GET *не* был указан, но вызывающая сторона передала
ненулевой массив struct pages*, то функция сама устанавливает FOLL_GET и закрепляет
страницы, увеличивая refcount каждой страницы на +1.::

 Function
 --------
 get_user_pages           FOLL_GET is sometimes set internally by this function.
 get_user_pages_fast      FOLL_GET is sometimes set internally by this function.
 get_user_pages_remote    FOLL_GET is sometimes set internally by this function.

Отслеживание страниц, закреплённых под dma
==========================================

Некоторые ключевые проектные ограничения и решения для отслеживания страниц,
закреплённых под dma:

* Требуется реальный счётчик ссылок для каждой struct page. Это связано с тем, что
  несколько процессов могут закреплять и откреплять страницу.

* Ложные срабатывания (сообщение о том, что страница закреплена под dma, тогда как на
  самом деле это не так) допустимы, но ложные отрицания — нет.

* Размер struct page не может быть увеличен ради этого, а все поля уже используются.

* С учётом вышесказанного мы можем перегрузить поле page->_refcount, используя, в некотором
  роде, старшие биты этого поля для счётчика закреплений под dma. «В некотором роде» означает,
  что вместо разделения page->_refcount на битовые поля мы просто прибавляем среднее-большое
  значение (GUP_PIN_COUNTING_BIAS, изначально выбранное равным 1024: 10 бит) к
  page->_refcount. Это даёт нечёткое поведение: если для страницы get_page() была вызвана
  1024 раза, то она будет выглядеть как имеющая единичный счётчик закреплений под dma.
  И снова — это допустимо.

Это также приводит к ограничениям: для счётчика, который инкрементируется по 10 бит за раз,
доступен только 31-10==21 бит.

* Из-за этого ограничения для нулевых страниц применяется специальная обработка при
  использовании FOLL_PIN.  Мы лишь делаем вид, что закрепляем нулевую страницу — мы вообще
  не изменяем её refcount или pincount (она постоянна, поэтому в этом нет необходимости).
  Функции открепления также ничего не делают с нулевой страницей.  Для вызывающей стороны
  это прозрачно.

* Вызывающие стороны должны явно запрашивать «отслеживание закрепления страниц под dma».
  Иными словами, простого вызова get_user_pages() недостаточно; необходимо использовать
  новый набор функций — pin_user_page() и связанные с ней.

FOLL_PIN, FOLL_GET, FOLL_LONGTERM: когда какие флаги использовать
=================================================================

Благодарность Яну Каре (Jan Kara), Властимилу Бабке (Vlastimil Babka) и нескольким другим
людям из -mm за описание этих категорий:

СЛУЧАЙ 1: Прямой ввод-вывод (DIO)
---------------------------------
Существуют ссылки GUP на страницы, служащие буферами DIO. Эти буферы нужны на относительно
короткое время (поэтому они не «долгосрочные»). Никакой специальной синхронизации с
folio_mkclean() или munmap() не предоставляется. Следовательно, в месте вызова нужно
установить такие флаги: ::

    FOLL_PIN

...но вместо непосредственной установки FOLL_PIN места вызова должны использовать одну из
процедур pin_user_pages*(), которые устанавливают FOLL_PIN.

СЛУЧАЙ 2: RDMA
--------------
Существуют ссылки GUP на страницы, служащие буферами DMA. Эти буферы нужны на длительное
время («долгосрочно»). Никакой специальной синхронизации с folio_mkclean() или munmap() не
предоставляется. Следовательно, в месте вызова нужно установить такие флаги: ::

    FOLL_PIN | FOLL_LONGTERM

ПРИМЕЧАНИЕ: Некоторые страницы, такие как страницы DAX, не могут быть закреплены
долгосрочными закреплениями. Это связано с тем, что у страниц DAX нет отдельного page
cache, и поэтому «закрепление» подразумевает блокировку блоков файловой системы, что (пока
ещё) не поддерживается таким образом.

.. _mmu-notifier-registration-case:

СЛУЧАЙ 3: Регистрация mmu notifier, с аппаратной обработкой страничных отказов или без неё
------------------------------------------------------------------------------------------
Драйверы устройств могут закреплять страницы через get_user_pages*() и регистрироваться на
обратные вызовы mmu notifier для диапазона памяти. Затем, получив обратный вызов notifier
«invalidate range», прекратить использование диапазона устройством и открепить страницы.
Возможны и другие схемы, например явная синхронизация с ожидающим вводом-выводом, которые
достигают примерно того же самого.

Или же, если аппаратура поддерживает воспроизводимые страничные отказы, то драйвер
устройства может вообще избежать закрепления (это идеально), следующим образом:
зарегистрироваться на обратные вызовы mmu notifier, как выше, но вместо остановки устройства
и открепления в обратном вызове просто удалить диапазон из таблиц страниц устройства.

В любом случае, пока драйвер открепляет страницы по обратному вызову mmu notifier,
обеспечивается корректная синхронизация как с файловой системой, так и с mm
(folio_mkclean(), munmap() и т. д.). Следовательно, ни один из флагов устанавливать не
нужно.

СЛУЧАЙ 4: Закрепление только для манипуляций со struct page
-----------------------------------------------------------
Если затрагиваются только данные struct page (в отличие от фактического содержимого памяти,
которое отслеживает страница), то достаточно обычных вызовов GUP, и ни один из флагов
устанавливать не нужно.

СЛУЧАЙ 5: Закрепление с целью записи данных внутри страницы
-----------------------------------------------------------
Даже если ни DMA, ни прямой ввод-вывод не задействованы, простой случай «закрепить, записать
данные страницы, открепить» может вызвать проблему. Случай 5 можно считать надмножеством
случая 1 плюс случая 2 плюс всего, что задействует этот шаблон. Иными словами, если код не
относится ни к случаю 1, ни к случаю 2, ему всё равно может потребоваться FOLL_PIN — для
шаблонов вроде этого:

Правильно (использует вызовы FOLL_PIN):
    pin_user_pages()
    запись данных внутри страниц
    unpin_user_pages()

НЕПРАВИЛЬНО (использует вызовы FOLL_GET):
    get_user_pages()
    запись данных внутри страниц
    put_page()

folio_maybe_dma_pinned(): главный смысл закрепления
===================================================

Главный смысл пометки folio как «закреплённых под DMA» или «закреплённых через gup» — это
возможность запросить «закреплена ли эта folio под DMA?». Это позволяет такому коду, как
folio_mkclean() (и коду обратной записи файловой системы в целом), принимать обоснованные
решения о том, что делать, когда folio не может быть отображена обратно из-за таких
закреплений.

Что делать в этих случаях — предмет многолетней серии обсуждений и дебатов (см. раздел
References в конце этого документа). Здесь это пункт TODO: заполнить детали, как только это
будет проработано. А пока можно с уверенностью сказать, что наличие вот этого: ::

        static inline bool folio_maybe_dma_pinned(struct folio *folio)

...является предварительным условием для решения давней проблемы gup+DMA.

Ещё один способ думать о FOLL_GET, FOLL_PIN и FOLL_LONGTERM
===========================================================

Другой способ думать об этих флагах — как о прогрессии ограничений: FOLL_GET предназначен
для манипуляций со struct page без влияния на данные, на которые ссылается struct page.
FOLL_PIN является *заменой* FOLL_GET и предназначен для краткосрочных закреплений страниц,
данные которых *будут* доступны для обращения. В этом смысле FOLL_PIN — это «более строгая»
форма закрепления. И наконец, FOLL_LONGTERM — это ещё более ограничительный случай, для
которого FOLL_PIN является предварительным условием: он предназначен для страниц, которые
будут закреплены долгосрочно и к данным которых будет производиться обращение.

Модульное тестирование
======================
Этот файл::

 tools/testing/selftests/mm/gup_test.c

содержит следующие новые вызовы для проверки новых функций-обёрток pin*():

* PIN_FAST_BENCHMARK (./gup_test -a)
* PIN_BASIC_TEST (./gup_test -b)

Вы можете отслеживать, сколько всего страниц, закреплённых под dma, было получено и
освобождено с момента загрузки системы, через две новые записи в /proc/vmstat: ::

    /proc/vmstat/nr_foll_pin_acquired
    /proc/vmstat/nr_foll_pin_released

В нормальных условиях эти два значения будут равны, если только не установлено каких-либо
долгосрочных закреплений [R]DMA или во время переходов закрепления/открепления.

* nr_foll_pin_acquired: Это число логических закреплений, полученных с момента включения
  системы. Для огромных страниц головная страница закрепляется по одному разу за каждую
  страницу (головную и каждую хвостовую) внутри огромной страницы. Это следует тому же
  поведению, которое get_user_pages() использует для огромных страниц: головная страница
  получает refcount по одному разу за каждую хвостовую или головную страницу в огромной
  странице, когда get_user_pages() применяется к огромной странице.

* nr_foll_pin_released: Число логических закреплений, освобождённых с момента включения
  системы. Обратите внимание, что страницы освобождаются (открепляются) с гранулярностью
  PAGE_SIZE, даже если изначальное закрепление было применено к огромной странице. Из-за
  поведения счётчика закреплений, описанного выше в «nr_foll_pin_acquired», учёт сходится,
  так что после выполнения вот этого::

    pin_user_pages(huge_page);
    for (each page in huge_page)
        unpin_user_page(page);

...ожидается следующее::

    nr_foll_pin_released == nr_foll_pin_acquired

(...если только баланс уже не был нарушен из-за установленного долгосрочного закрепления
RDMA.)

Прочая диагностика
==================

dump_page() была немного улучшена для обработки этих новых полей подсчёта и для более
качественного отчёта о больших folio в целом.  В частности, для больших folio сообщается
точное значение pincount.
