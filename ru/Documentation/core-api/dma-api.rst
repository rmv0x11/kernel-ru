====================================================================
Динамическое отображение DMA с использованием обобщённого устройства
====================================================================

:Author: James E.J. Bottomley <James.Bottomley@HansenPartnership.com>

Этот документ описывает DMA API.  Более мягкое введение в этот API
(с реальными примерами) см. в Documentation/core-api/dma-api-howto.rst.

Данный API разделён на две части.  Часть I описывает базовый API.
Часть II описывает расширения для поддержки машин с некогерентной
памятью.  Если вы точно не знаете, что вашему драйверу обязательно
требуется поддержка некогерентных платформ (обычно это только устаревшие
платформы), вам следует использовать только API, описанный в части I.

Часть I — DMA API
-----------------

Чтобы получить доступ к DMA API, необходимо выполнить #include <linux/dma-mapping.h>.
Он предоставляет dma_addr_t и описанные ниже интерфейсы.

Значение dma_addr_t может хранить любой корректный для данной платформы DMA-адрес.
Его можно передать устройству для использования в качестве источника или приёмника DMA.
CPU не может напрямую разыменовывать dma_addr_t, поскольку между его физическим
адресным пространством и адресным пространством DMA может существовать трансляция.

Часть Ia — Использование больших DMA-когерентных буферов
--------------------------------------------------------

::

	void *
	dma_alloc_coherent(struct device *dev, size_t size,
			   dma_addr_t *dma_handle, gfp_t flag)

Когерентная память — это память, запись в которую как устройством, так и
процессором может быть немедленно прочитана процессором или устройством
без необходимости беспокоиться об эффектах кэширования.  (Тем не менее, вам
может потребоваться убедиться, что буферы записи процессора сброшены, прежде
чем сообщить устройствам о необходимости чтения этой памяти.)

Эта подпрограмма выделяет область когерентной памяти размером <size> байт.

Она возвращает указатель на выделенную область (в виртуальном адресном
пространстве процессора) либо NULL, если выделение не удалось.

Она также возвращает <dma_handle>, который можно привести к беззнаковому целому
той же разрядности, что и шина, и передать устройству в качестве базового
DMA-адреса этой области.

Примечание: на некоторых платформах когерентная память может быть дорогостоящей,
а минимальная длина выделения может достигать размера страницы, поэтому вам следует
максимально консолидировать ваши запросы на когерентную память.
Самый простой способ сделать это — использовать вызовы dma_pool (см. ниже).

Параметр flag позволяет вызывающему указать флаги ``GFP_`` (см.
kmalloc()) для выделения (реализация может игнорировать флаги, влияющие на
расположение возвращаемой памяти, такие как GFP_DMA).

::

	void
	dma_free_coherent(struct device *dev, size_t size, void *cpu_addr,
			  dma_addr_t dma_handle)

Освобождает ранее выделенную область когерентной памяти.  Параметры dev, size и dma_handle
должны полностью совпадать с теми, что были переданы в dma_alloc_coherent().  cpu_addr должен
быть виртуальным адресом, возвращённым dma_alloc_coherent().

Обратите внимание, что в отличие от родственного вызова выделения, эта подпрограмма может
вызываться только при включённых IRQ.


Часть Ib — Использование малых DMA-когерентных буферов
------------------------------------------------------

Чтобы получить доступ к этой части DMA API, необходимо выполнить #include <linux/dmapool.h>

Многим драйверам требуется множество небольших DMA-когерентных областей памяти для
DMA-дескрипторов или буферов ввода-вывода.  Вместо того чтобы выделять память единицами
по странице или больше с помощью dma_alloc_coherent(), вы можете использовать DMA-пулы.  Они работают
во многом подобно struct kmem_cache, за исключением того, что используют DMA-когерентный аллокатор,
а не __get_free_pages().  Кроме того, они учитывают распространённые аппаратные ограничения
по выравниванию, например требование выравнивать головы очередей по границам N байт.

.. kernel-doc:: mm/dmapool.c
   :export:

.. kernel-doc:: include/linux/dmapool.h


Часть Ic — Ограничения DMA-адресации
------------------------------------

DMA-маска — это битовая маска адресуемой области для устройства. Иными словами,
если применение DMA-маски (побитовая операция AND) к DMA-адресу области памяти
не сбрасывает в адресе ни одного бита, то устройство может
выполнять DMA к этой области памяти.

Все приведённые ниже функции, устанавливающие DMA-маску, могут завершиться неудачей, если запрошенная маска
не может быть использована с устройством, или если устройство не способно выполнять DMA.

::

	int
	dma_set_mask_and_coherent(struct device *dev, u64 mask)

Обновляет одновременно потоковую и когерентную DMA-маски.

Возвращает: 0 в случае успеха и отрицательное значение ошибки в противном случае.

::

	int
	dma_set_mask(struct device *dev, u64 mask)

Обновляет только потоковую DMA-маску.

Возвращает: 0 в случае успеха и отрицательное значение ошибки в противном случае.

::

	int
	dma_set_coherent_mask(struct device *dev, u64 mask)

Обновляет только когерентную DMA-маску.

Возвращает: 0 в случае успеха и отрицательное значение ошибки в противном случае.

::

	u64
	dma_get_required_mask(struct device *dev)

Этот API возвращает маску, которая требуется платформе для
эффективной работы.  Обычно это означает, что возвращаемая маска
является минимальной, необходимой для покрытия всей памяти.  Анализ
требуемой маски даёт драйверам с переменными размерами дескрипторов
возможность при необходимости использовать дескрипторы меньшего размера.

Запрос требуемой маски не изменяет текущую маску.  Если вы
хотите воспользоваться ею, вам следует выполнить вызов dma_set_mask(),
чтобы установить маску в возвращённое значение.

::

	size_t
	dma_max_mapping_size(struct device *dev);

Возвращает максимальный размер отображения для устройства. Параметр size
функций отображения, таких как dma_map_single(), dma_map_page() и
других, не должен превышать возвращаемое значение.

::

	size_t
	dma_opt_mapping_size(struct device *dev);

Возвращает максимальный оптимальный размер отображения для устройства.

Отображение более крупных буферов в некоторых сценариях может занимать гораздо больше времени. Кроме
того, для частых короткоживущих потоковых отображений предварительное время,
затрачиваемое на отображение, может составлять ощутимую часть общего времени
жизни запроса. Поэтому, если разбиение более крупных запросов не влечёт
существенных потерь производительности, драйверам устройств рекомендуется
ограничивать суммарную длину потоковых DMA-отображений возвращаемым значением.

::

	bool
	dma_need_sync(struct device *dev, dma_addr_t dma_addr);

Возвращает %true, если для передачи владения памятью требуются вызовы
dma_sync_single_for_{device,cpu}.  Возвращает %false, если эти вызовы можно пропустить.

::

	unsigned long
	dma_get_merge_boundary(struct device *dev);

Возвращает границу слияния DMA. Если устройство не может слить ни одного DMA-адресного
сегмента, функция возвращает 0.

Часть Id — Потоковые DMA-отображения
------------------------------------

Потоковый DMA позволяет отобразить существующий буфер для DMA-передач, а затем
снять его отображение по завершении.  Не гарантируется, что функции отображения завершатся успешно, поэтому
возвращаемое значение необходимо проверять.

.. note::

	В частности, отображение может завершиться неудачей для памяти, не адресуемой
	устройством, например если она не находится в пределах DMA-маски устройства и/или
	соединительного моста шины.  Потоковые DMA-функции пытаются преодолеть такие
	ограничения адресации либо путём использования IOMMU (устройства, которое отображает
	DMA-адреса ввода-вывода на адреса физической памяти), либо путём копирования
	данных в/из bounce-буфера, если ядро сконфигурировано с
	:doc:`SWIOTLB <swiotlb>`.  Однако эти методы не всегда
	доступны, и даже если они доступны, они всё равно могут завершиться неудачей по ряду
	причин.

	Коротко говоря, драйверу устройства может потребоваться внимательно следить за тем, где буферы
	расположены в физической памяти, особенно если DMA-маска меньше 32
	бит.

::

	dma_addr_t
	dma_map_single(struct device *dev, void *cpu_addr, size_t size,
		       enum dma_data_direction direction)

Отображает участок виртуальной памяти процессора так, чтобы он стал доступен
устройству, и возвращает DMA-адрес этой памяти.

DMA API использует строго типизированный перечислитель для указания направления:

======================= =============================================
DMA_NONE		нет направления (используется для отладки)
DMA_TO_DEVICE		данные идут из памяти к устройству
DMA_FROM_DEVICE		данные идут от устройства в память
DMA_BIDIRECTIONAL	направление неизвестно
======================= =============================================

.. note::

	Непрерывное виртуальное пространство ядра может не быть непрерывным как
	физическая память.  Поскольку этот API не предоставляет никаких возможностей scatter/gather,
	он завершится неудачей, если пользователь попытается отобразить физически
	непрерывный участок памяти.  По этой причине память, отображаемую
	этим API, следует получать из источников, гарантирующих её физическую
	непрерывность (таких как kmalloc).

.. warning::

	Когерентность памяти работает на уровне гранулярности, называемом шириной
	кэш-линии.  Чтобы память, отображённая этим API, работала
	корректно, отображаемая область должна начинаться точно на границе кэш-линии
	и заканчиваться точно на ней (чтобы предотвратить совместное использование
	одной кэш-линии двумя отдельно отображёнными областями).  Поскольку размер кэш-линии
	может быть неизвестен на этапе компиляции, API не будет принудительно требовать выполнения этого
	требования.  Поэтому авторам драйверов, которые не предпринимают особых мер
	для определения размера кэш-линии во время выполнения, рекомендуется
	отображать только те виртуальные области, которые начинаются и заканчиваются на границах страниц (которые
	гарантированно также являются границами кэш-линий).

	Синхронизация DMA_TO_DEVICE должна выполняться после последнего изменения
	области памяти программным обеспечением и до того, как она будет передана
	устройству.  После того как этот примитив использован, память, покрываемая этим
	примитивом, должна рассматриваться устройством как доступная только для чтения.  Если устройство
	может в какой-либо момент производить в неё запись, следует использовать DMA_BIDIRECTIONAL (см.
	ниже).

	Синхронизация DMA_FROM_DEVICE должна выполняться до того, как драйвер
	обращается к данным, которые могли быть изменены устройством.  Эта память должна
	рассматриваться драйвером как доступная только для чтения.  Если драйверу необходимо производить запись
	в неё в какой-либо момент, следует использовать DMA_BIDIRECTIONAL (см. ниже).

	DMA_BIDIRECTIONAL требует особой обработки: это означает, что драйвер
	не уверен, была ли память изменена перед передачей
	устройству, а также не уверен, будет ли устройство также изменять её.  Таким образом,
	двунаправленную память необходимо всегда синхронизировать дважды: один раз перед тем, как
	память передаётся устройству (чтобы убедиться, что все изменения памяти
	сброшены из процессора), и один раз перед тем, как к данным можно будет
	обращаться после использования устройством (чтобы убедиться, что все кэш-линии процессора
	обновлены данными, которые мог изменить устройство).

::

	void
	dma_unmap_single(struct device *dev, dma_addr_t dma_addr, size_t size,
			 enum dma_data_direction direction)

Снимает отображение ранее отображённой области.  Все передаваемые параметры
должны быть идентичны тем, что были переданы в (и возвращены) dma_map_single().

::

	dma_addr_t
	dma_map_page(struct device *dev, struct page *page,
		     unsigned long offset, size_t size,
		     enum dma_data_direction direction)

	void
	dma_unmap_page(struct device *dev, dma_addr_t dma_address, size_t size,
		       enum dma_data_direction direction)

API для отображения и снятия отображения для страниц.  Здесь применимы все замечания и предупреждения
для других API отображения.  Кроме того, хотя параметры <offset>
и <size> предоставлены для выполнения частичного отображения страницы,
рекомендуется никогда их не использовать, если вы действительно не знаете, какова
ширина кэша.

::

	dma_addr_t
	dma_map_resource(struct device *dev, phys_addr_t phys_addr, size_t size,
			 enum dma_data_direction dir, unsigned long attrs)

	void
	dma_unmap_resource(struct device *dev, dma_addr_t addr, size_t size,
			   enum dma_data_direction dir, unsigned long attrs)

API для отображения и снятия отображения для ресурсов MMIO. Здесь применимы все замечания и
предупреждения для других API отображения. Этот API следует использовать только для
отображения ресурсов MMIO устройства, отображение RAM не допускается.

::

	int
	dma_mapping_error(struct device *dev, dma_addr_t dma_addr)

При некоторых обстоятельствах dma_map_single(), dma_map_page() и dma_map_resource()
не смогут создать отображение. Драйвер может проверить наличие этих ошибок, проверив
возвращённый DMA-адрес с помощью dma_mapping_error(). Ненулевое возвращаемое значение
означает, что отображение не удалось создать, и драйвер должен предпринять соответствующие
действия (например, уменьшить текущее использование DMA-отображений либо выждать и повторить попытку позже).

::

	int
	dma_map_sg(struct device *dev, struct scatterlist *sg,
		   int nents, enum dma_data_direction direction)

Отображает список scatter/gather для DMA. Возвращает число отображённых DMA-адресных сегментов,
которое может быть меньше переданного <nents>, если несколько последовательных
записей sglist слиты (например, при помощи IOMMU, или если некоторые соседние сегменты
просто оказались физически непрерывными).

Обратите внимание, что sg нельзя отобразить повторно, если он уже был отображён однажды.
Процессу отображения разрешено уничтожать информацию в sg.

Как и с другими интерфейсами отображения, dma_map_sg() может завершиться неудачей. Когда это
происходит, возвращается 0, и драйвер должен предпринять соответствующие действия. Критически
важно, чтобы драйвер что-то предпринял: в случае блочного драйвера
прервать запрос или даже выдать oops лучше, чем ничего не делать и
повредить файловую систему.

Со scatterlist'ами полученное отображение используется следующим образом::

	int i, count = dma_map_sg(dev, sglist, nents, direction);
	struct scatterlist *sg;

	for_each_sg(sglist, sg, count, i) {
		hw_address[i] = sg_dma_address(sg);
		hw_len[i] = sg_dma_len(sg);
	}

где nents — это число записей в sglist.

Реализация вправе слить несколько последовательных записей sglist
в одну.  Возвращаемое число — это фактическое число записей sg, в которые она их
отобразила. В случае неудачи возвращается 0.

Затем вам следует выполнить цикл count раз (примечание: это может быть меньше, чем nents раз)
и использовать макросы sg_dma_address() и sg_dma_len() там, где вы ранее
обращались к sg->address и sg->length, как показано выше.

::

	void
	dma_unmap_sg(struct device *dev, struct scatterlist *sg,
		     int nents, enum dma_data_direction direction)

Снимает отображение ранее отображённого списка scatter/gather.  Все параметры
должны быть теми же, что и переданные в API отображения
scatter/gather.

Примечание: <nents> должен быть числом, которое вы передавали, *а не* числом
возвращённых DMA-адресных записей.

::

	void
	dma_sync_single_for_cpu(struct device *dev, dma_addr_t dma_handle,
				size_t size,
				enum dma_data_direction direction)

	void
	dma_sync_single_for_device(struct device *dev, dma_addr_t dma_handle,
				   size_t size,
				   enum dma_data_direction direction)

	void
	dma_sync_sg_for_cpu(struct device *dev, struct scatterlist *sg,
			    int nents,
			    enum dma_data_direction direction)

	void
	dma_sync_sg_for_device(struct device *dev, struct scatterlist *sg,
			       int nents,
			       enum dma_data_direction direction)

Синхронизирует одиночное непрерывное отображение или отображение scatter/gather для CPU
и устройства. С API sync_sg все параметры должны быть теми же,
что переданы в API отображения sg. С API sync_single
вы можете использовать параметры dma_handle и size, которые не идентичны
переданным в API одиночного отображения, чтобы выполнить частичную синхронизацию.


.. note::

   Вы должны делать это:

   - Перед чтением значений, записанных DMA от устройства
     (используйте направление DMA_FROM_DEVICE)
   - После записи значений, которые будут записаны в устройство с помощью DMA
     (используйте направление DMA_TO_DEVICE)
   - до *и* после передачи памяти устройству, если память является
     DMA_BIDIRECTIONAL

См. также dma_map_single().

::

	dma_addr_t
	dma_map_single_attrs(struct device *dev, void *cpu_addr, size_t size,
			     enum dma_data_direction dir,
			     unsigned long attrs)

	void
	dma_unmap_single_attrs(struct device *dev, dma_addr_t dma_addr,
			       size_t size, enum dma_data_direction dir,
			       unsigned long attrs)

	int
	dma_map_sg_attrs(struct device *dev, struct scatterlist *sgl,
			 int nents, enum dma_data_direction dir,
			 unsigned long attrs)

	void
	dma_unmap_sg_attrs(struct device *dev, struct scatterlist *sgl,
			   int nents, enum dma_data_direction dir,
			   unsigned long attrs)

Четыре приведённые выше функции аналогичны соответствующим функциям
без суффиксов _attrs, за исключением того, что они передают необязательный
dma_attrs.

Интерпретация DMA-атрибутов специфична для архитектуры, и
каждый атрибут должен быть описан в
Documentation/core-api/dma-attributes.rst.

Если dma_attrs равны 0, семантика каждой из этих функций
идентична семантике соответствующей функции
без суффикса _attrs. В результате dma_map_single_attrs()
в общем случае может заменить dma_map_single() и т. д.

В качестве примера использования функций ``*_attrs`` приведём то, как
можно передать атрибут DMA_ATTR_FOO при отображении памяти
для DMA::

	#include <linux/dma-mapping.h>
	/* DMA_ATTR_FOO should be defined in linux/dma-mapping.h and
	* documented in Documentation/core-api/dma-attributes.rst */
	...

		unsigned long attr;
		attr |= DMA_ATTR_FOO;
		....
		n = dma_map_sg_attrs(dev, sg, nents, DMA_TO_DEVICE, attr);
		....

Архитектуры, для которых важен DMA_ATTR_FOO, проверяли бы его
наличие в своих реализациях подпрограмм отображения и снятия отображения,
например:::

	void whizco_dma_map_sg_attrs(struct device *dev, dma_addr_t dma_addr,
				     size_t size, enum dma_data_direction dir,
				     unsigned long attrs)
	{
		....
		if (attrs & DMA_ATTR_FOO)
			/* twizzle the frobnozzle */
		....
	}

Часть Ie — DMA-отображения на основе IOVA
-----------------------------------------

Эти API позволяют выполнять очень эффективное отображение при использовании IOMMU.  Они представляют собой
необязательный путь, требующий дополнительного кода, и рекомендуются только для драйверов,
для которых важна производительность DMA-отображения или объём памяти, используемой для хранения DMA-адресов.
Все соображения из предыдущего раздела применимы и здесь.

::

    bool dma_iova_try_alloc(struct device *dev, struct dma_iova_state *state,
		phys_addr_t phys, size_t size);

Используется для попытки выделить пространство IOVA для операции отображения.  Если она возвращает
false, этот API нельзя использовать для данного устройства, и следует использовать обычный потоковый
DMA API отображения.  Структура ``struct dma_iova_state`` выделяется
драйвером и должна сохраняться вплоть до момента снятия отображения.

::

    static inline bool dma_use_iova(struct dma_iova_state *state)

Может использоваться драйвером для проверки того, используется ли API на основе IOVA после
вызова dma_iova_try_alloc.  Это может быть полезно на пути снятия отображения.

::

    int dma_iova_link(struct device *dev, struct dma_iova_state *state,
		phys_addr_t phys, size_t offset, size_t size,
		enum dma_data_direction dir, unsigned long attrs);

Используется для привязки диапазонов к ранее выделенному IOVA.  Начало всех
вызовов dma_iova_link для данного state, кроме первого, должно быть выровнено
по границе слияния DMA, возвращаемой ``dma_get_merge_boundary())``, а
размер всех диапазонов, кроме последнего, также должен быть выровнен по границе слияния
DMA.

::

    int dma_iova_sync(struct device *dev, struct dma_iova_state *state,
		size_t offset, size_t size);

Должна вызываться для синхронизации таблиц страниц IOMMU для диапазона IOVA, отображённого одним или
несколькими вызовами ``dma_iova_link()``.

Для драйверов, использующих одноразовое отображение, все диапазоны можно снять с отображения, а
IOVA освободить, вызвав:

::

   void dma_iova_destroy(struct device *dev, struct dma_iova_state *state,
		size_t mapped_len, enum dma_data_direction dir,
                unsigned long attrs);

В качестве альтернативы драйверы могут динамически управлять пространством IOVA, снимая отображение
и отображая отдельные области.  В этом случае

::

    void dma_iova_unlink(struct device *dev, struct dma_iova_state *state,
		size_t offset, size_t size, enum dma_data_direction dir,
		unsigned long attrs);

используется для снятия отображения ранее отображённого диапазона, а

::

   void dma_iova_free(struct device *dev, struct dma_iova_state *state);

используется для освобождения пространства IOVA.  Все области должны быть сняты с отображения с помощью
``dma_iova_unlink()`` перед вызовом ``dma_iova_free()``.

Часть II — Некогерентные DMA-выделения
--------------------------------------

Эти API позволяют выделять страницы, которые гарантированно DMA-адресуемы
переданным устройством, но которые требуют явного управления владением памятью
со стороны ядра и устройства.

Если вы не понимаете, как работает когерентность кэш-линий между процессором и
устройством ввода-вывода, вам не следует использовать эту часть API.

::

	struct page *
	dma_alloc_pages(struct device *dev, size_t size, dma_addr_t *dma_handle,
			enum dma_data_direction dir, gfp_t gfp)

Эта подпрограмма выделяет область некогерентной памяти размером <size> байт.  Она
возвращает указатель на первую struct page этой области, либо NULL, если
выделение не удалось. Полученную struct page можно использовать для всего, для чего
пригодна struct page.

Она также возвращает <dma_handle>, который можно привести к беззнаковому целому
той же разрядности, что и шина, и передать устройству в качестве базового DMA-адреса
этой области.

Параметр dir указывает, читаются и/или записываются ли данные устройством,
подробности см. в dma_map_single().

Параметр gfp позволяет вызывающему указать флаги ``GFP_`` (см.
kmalloc()) для выделения, но отвергает флаги, используемые для указания зоны
памяти, такие как GFP_DMA или GFP_HIGHMEM.

Перед передачей памяти устройству необходимо вызвать dma_sync_single_for_device(),
а перед чтением памяти, записанной устройством,
dma_sync_single_for_cpu(), как и для повторно используемых потоковых
DMA-отображений.

::

	void
	dma_free_pages(struct device *dev, size_t size, struct page *page,
			dma_addr_t dma_handle, enum dma_data_direction dir)

Освобождает область памяти, ранее выделенную с помощью dma_alloc_pages().
dev, size, dma_handle и dir должны полностью совпадать с теми, что были переданы в
dma_alloc_pages().  page должен быть указателем, возвращённым dma_alloc_pages().

::

	int
	dma_mmap_pages(struct device *dev, struct vm_area_struct *vma,
		       size_t size, struct page *page)

Отображает выделение, возвращённое из dma_alloc_pages(), в пользовательское адресное пространство.
dev и size должны быть теми же, что были переданы в dma_alloc_pages().
page должен быть указателем, возвращённым dma_alloc_pages().

::

	void *
	dma_alloc_noncoherent(struct device *dev, size_t size,
			dma_addr_t *dma_handle, enum dma_data_direction dir,
			gfp_t gfp)

Эта подпрограмма представляет собой удобную обёртку вокруг dma_alloc_pages, которая возвращает
виртуальный адрес ядра для выделенной памяти вместо структуры страницы.

::

	void
	dma_free_noncoherent(struct device *dev, size_t size, void *cpu_addr,
			dma_addr_t dma_handle, enum dma_data_direction dir)

Освобождает область памяти, ранее выделенную с помощью dma_alloc_noncoherent().
dev, size, dma_handle и dir должны полностью совпадать с теми, что были переданы в
dma_alloc_noncoherent().  cpu_addr должен быть виртуальным адресом, возвращённым
dma_alloc_noncoherent().

::

	struct sg_table *
	dma_alloc_noncontiguous(struct device *dev, size_t size,
				enum dma_data_direction dir, gfp_t gfp,
				unsigned long attrs);

Эта подпрограмма выделяет <size> байт некогерентной и, возможно, непрерывной
памяти.  Она возвращает указатель на struct sg_table, описывающую выделенную
и отображённую для DMA память, либо NULL, если выделение не удалось. Полученную память
можно использовать для всего, для чего пригодна struct page, отображённая в scatterlist.

Возвращаемая sg_table гарантированно имеет 1 единственный отображённый для DMA сегмент, как
указано в sgt->nents, но может иметь несколько сегментов на стороне CPU, как
указано в sgt->orig_nents.

Параметр dir указывает, читаются и/или записываются ли данные устройством,
подробности см. в dma_map_single().

Параметр gfp позволяет вызывающему указать флаги ``GFP_`` (см.
kmalloc()) для выделения, но отвергает флаги, используемые для указания зоны
памяти, такие как GFP_DMA или GFP_HIGHMEM.

Аргумент attrs должен быть либо 0, либо DMA_ATTR_ALLOC_SINGLE_PAGES.

Перед передачей памяти устройству необходимо вызвать dma_sync_sgtable_for_device(),
а перед чтением памяти, записанной устройством,
dma_sync_sgtable_for_cpu(), как и для повторно используемых потоковых
DMA-отображений.

::

	void
	dma_free_noncontiguous(struct device *dev, size_t size,
			       struct sg_table *sgt,
			       enum dma_data_direction dir)

Освобождает память, ранее выделенную с помощью dma_alloc_noncontiguous().  dev, size
и dir должны полностью совпадать с теми, что были переданы в dma_alloc_noncontiguous().
sgt должен быть указателем, возвращённым dma_alloc_noncontiguous().

::

	void *
	dma_vmap_noncontiguous(struct device *dev, size_t size,
		struct sg_table *sgt)

Возвращает непрерывное отображение ядра для выделения, возвращённого из
dma_alloc_noncontiguous().  dev и size должны быть теми же, что были переданы в
dma_alloc_noncontiguous().  sgt должен быть указателем, возвращённым
dma_alloc_noncontiguous().

После того как непрерывное выделение отображено с помощью этой функции,
для управления когерентностью между отображением ядра, устройством и пользовательскими
отображениями (если они есть) необходимо использовать API flush_kernel_vmap_range() и invalidate_kernel_vmap_range().

::

	void
	dma_vunmap_noncontiguous(struct device *dev, void *vaddr)

Снимает отображение ядра, возвращённое dma_vmap_noncontiguous().  dev должен быть
тем же, что был передан в dma_alloc_noncontiguous().  vaddr должен быть указателем,
возвращённым dma_vmap_noncontiguous().


::

	int
	dma_mmap_noncontiguous(struct device *dev, struct vm_area_struct *vma,
			       size_t size, struct sg_table *sgt)

Отображает выделение, возвращённое из dma_alloc_noncontiguous(), в пользовательское адресное
пространство.  dev и size должны быть теми же, что были переданы в
dma_alloc_noncontiguous().  sgt должен быть указателем, возвращённым
dma_alloc_noncontiguous().

::

	int
	dma_get_cache_alignment(void)

Возвращает выравнивание кэша процессора.  Это абсолютный минимум
выравнивания *и* ширины, который необходимо соблюдать при отображении
памяти или выполнении частичных сбросов.

.. note::

	Этот API может вернуть число *больше*, чем фактическая кэш-линия,
	но он гарантирует, что одна или несколько кэш-линий точно укладываются
	в ширину, возвращаемую этим вызовом.  Оно также всегда будет степенью
	двойки для удобства выравнивания.


Часть III — Отладка использования DMA API драйверами
----------------------------------------------------

Описанный выше DMA API имеет некоторые ограничения. Например, DMA-адреса должны
освобождаться соответствующей функцией с тем же размером. С
появлением аппаратных IOMMU становится всё более важным, чтобы драйверы
не нарушали эти ограничения. В худшем случае такое нарушение может
привести к повреждению данных вплоть до уничтожения файловых систем.

Чтобы отлаживать драйверы и находить ошибки в использовании DMA API, в ядро можно
встроить проверяющий код, который будет сообщать разработчику о таких
нарушениях. Если ваша архитектура это поддерживает, вы можете выбрать опцию "Enable
debugging of DMA API usage" в конфигурации ядра. Включение этой
опции влияет на производительность. Не включайте её в производственных ядрах.

Если вы загрузите получившееся ядро, оно будет содержать код, который ведёт некоторый учёт
того, какая DMA-память была выделена для какого устройства. Если этот код обнаруживает
ошибку, он выводит предупреждающее сообщение с некоторыми подробностями в журнал ядра. Пример
предупреждающего сообщения может выглядеть так::

По умолчанию предупреждающее сообщение появляется только для первой ошибки. Все остальные
ошибки лишь молча подсчитываются. Это ограничение существует, чтобы предотвратить переполнение
журнала ядра кодом. Для поддержки отладки драйвера устройства это можно
отключить через debugfs. Подробности см. в приведённой ниже документации по интерфейсу debugfs.

Каталог debugfs для кода отладки DMA API называется dma-api/. В
этом каталоге в настоящее время можно найти следующие файлы:

=============================== ===============================================
dma-api/all_errors		Этот файл содержит числовое значение. Если это
				значение не равно нулю, код отладки
				будет выводить предупреждение в журнал ядра
				для каждой найденной ошибки. Будьте осторожны с этой
				опцией, так как она может легко переполнить журналы.

dma-api/disabled		Этот файл только для чтения содержит символ 'Y',
				если код отладки отключён. Это может
				произойти, когда исчерпана память или если он был
				отключён во время загрузки

dma-api/dump			Этот файл только для чтения содержит текущие DMA-
				отображения.

dma-api/error_count		Этот файл только для чтения и показывает общее
				число найденных ошибок.

dma-api/num_errors		Число в этом файле показывает, сколько
				предупреждений будет выведено в журнал ядра,
				прежде чем он остановится. Это число инициализируется
				единицей при загрузке системы и может быть задано записью в
				этот файл

dma-api/min_free_entries	Этот файл только для чтения можно прочитать, чтобы получить
				минимальное число свободных dma_debug_entries, которое
				аллокатор когда-либо видел. Если это значение опускается
				до нуля, код попытается увеличить
				nr_total_entries, чтобы это компенсировать.

dma-api/num_free_entries	Текущее число свободных dma_debug_entries
				в аллокаторе.

dma-api/nr_total_entries	Общее число dma_debug_entries в
				аллокаторе, как свободных, так и используемых.

dma-api/driver_filter		Вы можете записать имя драйвера в этот файл,
				чтобы ограничить отладочный вывод запросами от этого
				конкретного драйвера. Запишите пустую строку в
				этот файл, чтобы отключить фильтр и снова видеть
				все ошибки.
=============================== ===============================================

Если этот код встроен в ваше ядро, он будет включён по умолчанию.
Если вы всё же хотите загрузиться без ведения учёта, вы можете указать
'dma_debug=off' в качестве параметра загрузки. Это отключит отладку DMA API.
Обратите внимание, что вы не сможете включить её снова во время выполнения. Для этого
вам придётся перезагрузиться.

Если вы хотите видеть отладочные сообщения только для определённого драйвера устройства, вы можете
указать параметр dma_debug_driver=<drivername>. Это включит фильтр
драйвера во время загрузки. После этого отладочный код будет выводить ошибки только для этого
драйвера. Этот фильтр можно отключить или изменить позже с помощью debugfs.

Когда код отключает себя во время выполнения, это чаще всего происходит из-за того, что он исчерпал
dma_debug_entries и не смог выделить больше по требованию. При загрузке
предварительно выделяется 65536 записей — если для вас это слишком мало, загрузитесь с
'dma_debug_entries=<your_desired_number>', чтобы переопределить значение по умолчанию. Обратите
внимание, что код выделяет записи пакетами, поэтому точное число предварительно
выделенных записей может быть больше фактически запрошенного. Код
будет выводить в журнал ядра каждый раз, когда динамически выделит
столько же записей, сколько было первоначально предварительно выделено. Это призвано указать, что
может быть уместен больший размер предварительного выделения, либо, если это происходит постоянно,
что драйвер может допускать утечку отображений.

::

	void
	debug_dma_mapping_error(struct device *dev, dma_addr_t dma_addr);

Интерфейс dma-debug debug_dma_mapping_error() предназначен для отладки драйверов, которые не
проверяют ошибки DMA-отображения для адресов, возвращённых интерфейсами dma_map_single() и
dma_map_page(). Этот интерфейс сбрасывает флаг, установленный
debug_dma_map_phys(), чтобы указать, что dma_mapping_error() была вызвана
драйвером. Когда драйвер выполняет снятие отображения, debug_dma_unmap() проверяет флаг, и если
этот флаг всё ещё установлен, выводит предупреждающее сообщение, включающее трассировку вызовов,
приведшую к снятию отображения. Этот интерфейс можно вызывать из подпрограмм dma_mapping_error()
для включения отладки проверки ошибок DMA-отображения.
