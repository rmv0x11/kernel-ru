==============================
Общий механизм уведомлений
==============================

Общий механизм уведомлений построен поверх стандартного драйвера каналов (pipe)
и по сути сплайсит (splice) сообщения-уведомления из ядра в каналы, открытые
пространством пользователя (userspace).  Его можно использовать совместно с::

  * Key/keyring notifications


Буферы уведомлений можно включить через:

	"General setup"/"General notification queue"
	(CONFIG_WATCH_QUEUE)

Этот документ состоит из следующих разделов:

.. contents:: :local:


Обзор
=====

Эта возможность представлена в виде канала (pipe), открытого в особом режиме.
Внутренний кольцевой буфер канала используется для хранения сообщений,
генерируемых ядром.  Затем эти сообщения считываются через read().  Splice и
подобные операции отключены для таких каналов, поскольку при некоторых
обстоятельствах они стремятся откатить свои добавления в кольцо — что может
привести к их чередованию с сообщениями-уведомлениями.

Владелец канала должен сообщить ядру, за какими источниками он хотел бы
наблюдать через этот канал.  Только источники, подключённые к каналу, будут
вставлять в него сообщения.  Обратите внимание, что источник может быть привязан
к нескольким каналам и вставлять сообщения во все из них одновременно.

На канал также могут быть наложены фильтры, чтобы определённые типы источников и
подсобытия можно было игнорировать, если они не представляют интереса.

Сообщение будет отброшено, если в кольце нет доступного слота или если нет
доступного предварительно выделенного буфера сообщений.  В обоих этих случаях
read() вставит сообщение WATCH_META_LOSS_NOTIFICATION в выходной буфер после
того, как будет прочитано последнее сообщение, находящееся в буфере в данный
момент.

Обратите внимание, что при формировании уведомления ядро не ждёт, пока его
заберут потребители, а просто продолжает работу.  Это означает, что уведомления
могут генерироваться, пока удерживаются спинлоки (spinlock), а также защищает
ядро от бесконечного удержания из-за сбоя в пространстве пользователя.


Структура сообщения
====================

Сообщения-уведомления начинаются с короткого заголовка::

	struct watch_notification {
		__u32	type:24;
		__u32	subtype:8;
		__u32	info;
	};

"type" указывает источник записи уведомления, а "subtype" указывает тип записи
из этого источника (см. раздел Watch Sources ниже).  Тип также может быть
"WATCH_TYPE_META".  Это особый тип записи, генерируемый внутренне самой watch
queue.  Существует два подтипа:

  * WATCH_META_REMOVAL_NOTIFICATION
  * WATCH_META_LOSS_NOTIFICATION

Первый указывает, что объект, на который был установлен watch, был удалён или
уничтожен, а второй указывает, что некоторые сообщения были потеряны.

"info" указывает на целый ряд вещей, в том числе:

  * Длина сообщения в байтах, включая заголовок (маска WATCH_INFO_LENGTH и сдвиг
    на WATCH_INFO_LENGTH__SHIFT).  Это указывает размер записи, который может
    составлять от 8 до 127 байт.

  * Идентификатор watch (маска WATCH_INFO_ID и сдвиг на WATCH_INFO_ID__SHIFT).
    Это указывает на присвоенный вызывающей стороной ID watch, который может
    составлять от 0 до 255.  Несколько watch могут совместно использовать одну
    очередь, и это даёт средство их различать.

  * Поле, специфичное для типа (WATCH_INFO_TYPE_INFO).  Оно устанавливается
    производителем уведомления для обозначения некоторого смысла, специфичного
    для данного типа и подтипа.

Всё в info, кроме длины, может использоваться для фильтрации.

За заголовком может следовать дополнительная информация.  Её формат
определяется по усмотрению типа и подтипа.


Watch List (Notification Source) API
====================================

«Watch list» — это список наблюдателей, подписанных на источник уведомлений.
Список может быть привязан к объекту (скажем, к ключу или суперблоку) либо может
быть глобальным (скажем, для событий устройств).  С точки зрения пространства
пользователя на неглобальный watch list обычно ссылаются через объект, которому
он принадлежит (например, используя KEYCTL_NOTIFY и передавая ему серийный номер
ключа, чтобы наблюдать за этим конкретным ключом).

Для управления watch list предоставляются следующие функции:

  * ::

	void init_watch_list(struct watch_list *wlist,
			     void (*release_watch)(struct watch *wlist));

    Инициализировать watch list.  Если ``release_watch`` не NULL, то это
    указывает функцию, которая должна быть вызвана при уничтожении объекта
    watch_list, чтобы освободить любые ссылки, которые watch list удерживает на
    наблюдаемый объект.

  * ``void remove_watch_list(struct watch_list *wlist);``

    Это удаляет все watch, подписанные на watch_list, и освобождает их, а затем
    уничтожает сам объект watch_list.


Watch Queue (Notification Output) API
=====================================

«Watch queue» — это буфер, выделяемый приложением, в который будут записываться
записи уведомлений.  Внутреннее устройство этого полностью скрыто внутри
драйвера устройства pipe, но для установки watch необходимо получить ссылку на
него.  Ими можно управлять с помощью:

  * ``struct watch_queue *get_watch_queue(int fd);``

    Поскольку watch queue обозначаются для ядра через fd канала, реализующего
    буфер, пространство пользователя должно передать этот fd через системный
    вызов.  Это можно использовать для поиска непрозрачного указателя на watch
    queue из системного вызова.

  * ``void put_watch_queue(struct watch_queue *wqueue);``

    Это освобождает ссылку, полученную от ``get_watch_queue()``.


Watch Subscription API
======================

«Watch» — это подписка на watch list, указывающая watch queue, а следовательно и
буфер, в который должны записываться записи уведомлений.  Объект watch queue
также может нести правила фильтрации для этого объекта, установленные
пространством пользователя.  Некоторые части структуры watch могут быть
установлены драйвером::

	struct watch {
		union {
			u32		info_id;	/* ID to be OR'd in to info field */
			...
		};
		void			*private;	/* Private data for the watched object */
		u64			id;		/* Internal identifier */
		...
	};

Значение ``info_id`` должно быть 8-битным числом, полученным от пространства
пользователя и сдвинутым на WATCH_INFO_ID__SHIFT.  Оно объединяется по ИЛИ (OR)
с полем WATCH_INFO_ID структуры watch_notification::info тогда и если
уведомление записывается в связанный буфер watch queue.

Поле ``private`` — это данные драйвера, связанные с watch_list, и они очищаются
методом ``watch_list::release_watch()``.

Поле ``id`` — это ID источника.  Уведомления, отправленные с другим ID,
игнорируются.

Для управления watch предоставляются следующие функции:

  * ``void init_watch(struct watch *watch, struct watch_queue *wqueue);``

    Инициализировать объект watch, устанавливая его указатель на watch queue, с
    использованием подходящего барьера (barriering), чтобы избежать нареканий со
    стороны lockdep.

  * ``int add_watch_to_object(struct watch *watch, struct watch_list *wlist);``

    Подписать watch на watch list (источник уведомлений).  Поля структуры watch,
    устанавливаемые драйвером, должны быть установлены до вызова этой функции.

  * ::

	int remove_watch_from_object(struct watch_list *wlist,
				     struct watch_queue *wqueue,
				     u64 id, false);

    Удалить watch из watch list, при этом watch должен соответствовать указанной
    watch queue (``wqueue``) и идентификатору объекта (``id``).  В watch queue
    отправляется уведомление (``WATCH_META_REMOVAL_NOTIFICATION``), указывающее,
    что watch был удалён.

  * ``int remove_watch_from_object(struct watch_list *wlist, NULL, 0, true);``

    Удалить все watch из watch list.  Ожидается, что это будет вызвано в порядке
    подготовки к уничтожению и что к этому моменту watch list будет недоступен
    для новых watch.  В watch queue каждого подписанного watch отправляется
    уведомление (``WATCH_META_REMOVAL_NOTIFICATION``), указывающее, что watch был
    удалён.


Notification Posting API
========================

Чтобы отправить уведомление в watch list, чтобы подписанные watch могли его
увидеть, следует использовать следующую функцию::

	void post_watch_notification(struct watch_list *wlist,
				     struct watch_notification *n,
				     const struct cred *cred,
				     u64 id);

Уведомление должно быть предварительно отформатировано, и должен быть передан
указатель на заголовок (``n``).  Уведомление может быть больше этого, и его
размер в единицах слотов буфера отмечается в ``n->info & WATCH_INFO_LENGTH``.

Структура ``cred`` указывает учётные данные источника (субъекта) и передаётся в
LSM, такие как SELinux, чтобы разрешить или подавить запись заметки в каждую
отдельную очередь в соответствии с учётными данными этой очереди (объекта).

``id`` — это ID объекта-источника (например, серийный номер ключа).  Только
watch, в которых установлен тот же ID, увидят это уведомление.


Watch Sources
=============

Любой конкретный буфер может получать данные из нескольких источников.  Среди
источников:

  * WATCH_TYPE_KEY_NOTIFY

    Уведомления этого типа указывают на изменения ключей и keyring, включая
    изменения содержимого keyring или атрибутов ключей.

    Дополнительную информацию см. в Documentation/security/keys/core.rst.


Фильтрация событий
==================

После того как watch queue создана, к ней может быть применён набор фильтров для
ограничения принимаемых событий с помощью::

	struct watch_notification_filter filter = {
		...
	};
	ioctl(fd, IOC_WATCH_QUEUE_SET_FILTER, &filter)

Описание фильтра — это переменная типа::

	struct watch_notification_filter {
		__u32	nr_filters;
		__u32	__reserved;
		struct watch_notification_type_filter filters[];
	};

Где "nr_filters" — это количество фильтров в filters[], а "__reserved" должно
быть 0.  Массив "filters" содержит элементы следующего типа::

	struct watch_notification_type_filter {
		__u32	type;
		__u32	info_filter;
		__u32	info_mask;
		__u32	subtype_filter[8];
	};

Где:

  * ``type`` — это тип события, по которому производится фильтрация; должно быть
    что-то вроде "WATCH_TYPE_KEY_NOTIFY"

  * ``info_filter`` и ``info_mask`` действуют как фильтр на поле info записи
    уведомления.  Уведомление записывается в буфер только в том случае, если::

	(watch.info & info_mask) == info_filter

    Это можно использовать, например, для игнорирования событий, которые
    приходятся не точно на наблюдаемую точку в дереве монтирования.

  * ``subtype_filter`` — это битовая маска, указывающая интересующие подтипы.
    Бит 0 в subtype_filter[0] соответствует подтипу 0, бит 1 — подтипу 1 и так
    далее.

Если аргумент для ioctl() равен NULL, то фильтры будут удалены и будут проходить
все события от наблюдаемых источников.


Пример кода для пространства пользователя
=========================================

Буфер создаётся примерно следующим образом::

	pipe2(fds, O_TMPFILE);
	ioctl(fds[1], IOC_WATCH_QUEUE_SET_SIZE, 256);

Затем его можно настроить на приём уведомлений об изменениях keyring::

	keyctl(KEYCTL_WATCH_KEY, KEY_SPEC_SESSION_KEYRING, fds[1], 0x01);

Затем уведомления можно потреблять примерно следующим образом::

	static void consumer(int rfd, struct watch_queue_buffer *buf)
	{
		unsigned char buffer[128];
		ssize_t buf_len;

		while (buf_len = read(rfd, buffer, sizeof(buffer)),
		       buf_len > 0
		       ) {
			void *p = buffer;
			void *end = buffer + buf_len;
			while (p < end) {
				union {
					struct watch_notification n;
					unsigned char buf1[128];
				} n;
				size_t largest, len;

				largest = end - p;
				if (largest > 128)
					largest = 128;
				memcpy(&n, p, largest);

				len = (n->info & WATCH_INFO_LENGTH) >>
					WATCH_INFO_LENGTH__SHIFT;
				if (len == 0 || len > largest)
					return;

				switch (n.n.type) {
				case WATCH_TYPE_META:
					got_meta(&n.n);
				case WATCH_TYPE_KEY_NOTIFY:
					saw_key_change(&n.n);
					break;
				}

				p += len;
			}
		}
	}
