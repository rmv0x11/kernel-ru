Подсистема genalloc/genpool
===========================

В ядре существует несколько подсистем выделения памяти, каждая из которых
ориентирована на определённую потребность. Однако иногда разработчику ядра
требуется реализовать новый аллокатор для конкретного диапазона памяти
специального назначения; зачастую такая память расположена где-то на устройстве.
Автор драйвера для этого устройства, безусловно, может написать небольшой
аллокатор, чтобы решить задачу, но это путь к заполнению ядра десятками плохо
протестированных аллокаторов. Ещё в 2005 году Jes Sorensen извлёк один из таких
аллокаторов из драйвера sym53c8xx_2 и posted_ его в виде универсального модуля
для создания специализированных аллокаторов памяти. Этот код был влит в выпуск
2.6.13; с тех пор он был значительно изменён.

.. _posted: https://lwn.net/Articles/125842/

Код, использующий этот аллокатор, должен включать <linux/genalloc.h>.
Всё начинается с создания пула с помощью одной из функций:

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_create

.. kernel-doc:: lib/genalloc.c
   :functions: devm_gen_pool_create

Вызов gen_pool_create() создаст пул. Гранулярность выделений задаётся
параметром min_alloc_order; это число по основанию 2 (логарифм), подобное тем,
что используются аллокатором страниц, но оно относится к байтам, а не к
страницам. Так, если min_alloc_order передан равным 3, то все выделения будут
кратны восьми байтам. Увеличение min_alloc_order уменьшает объём памяти,
требуемый для отслеживания памяти в пуле. Параметр nid указывает, какой узел
NUMA должен использоваться для выделения служебных структур; он может быть равен
-1, если вызывающему это безразлично.

«Управляемый» интерфейс devm_gen_pool_create() привязывает пул к конкретному
устройству. Помимо прочего, он автоматически очистит пул при уничтожении
заданного устройства.

Пул закрывается с помощью:

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_destroy

Стоит отметить, что если из данного пула ещё остаются незавершённые выделения,
эта функция предпримет довольно крайний шаг — вызовет BUG(), что приведёт к
аварийному завершению всей системы. Вы были предупреждены.

Только что созданный пул не имеет памяти для выделения. В таком состоянии он
довольно бесполезен, поэтому одним из первых действий обычно является добавление
памяти в пул. Это можно сделать с помощью одной из функций:

.. kernel-doc:: include/linux/genalloc.h
   :functions: gen_pool_add

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_add_owner

Вызов gen_pool_add() поместит size байт памяти, начинающейся по адресу addr
(в виртуальном адресном пространстве ядра), в заданный пул, снова используя nid
в качестве идентификатора узла для вспомогательных выделений памяти. Вариант
gen_pool_add_virt() связывает с памятью явный физический адрес; это необходимо
только в том случае, если пул будет использоваться для выделений DMA.

Функции для выделения памяти из пула (и для её возврата):

.. kernel-doc:: include/linux/genalloc.h
   :functions: gen_pool_alloc

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_dma_alloc

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_free_owner

Как и следовало ожидать, gen_pool_alloc() выделит size байт из заданного пула.
Вариант gen_pool_dma_alloc() выделяет память для использования в операциях DMA,
возвращая соответствующий физический адрес в области, на которую указывает dma.
Это будет работать только в том случае, если память была добавлена с помощью
gen_pool_add_virt(). Обратите внимание, что эта функция отступает от обычного
паттерна genpool, использующего значения типа unsigned long для представления
адресов ядра; вместо этого она возвращает ``void *``.

Всё это кажется относительно простым; и действительно, некоторые разработчики
явно сочли это слишком простым. В конце концов, приведённый выше интерфейс не
даёт никакого контроля над тем, как функции выделения выбирают конкретный
участок памяти для возврата. Если такого рода контроль необходим, интерес будут
представлять следующие функции:

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_alloc_algo_owner

.. kernel-doc:: lib/genalloc.c
   :functions: gen_pool_set_algo

Выделения с помощью gen_pool_alloc_algo() задают алгоритм, который будет
использоваться для выбора выделяемой памяти; алгоритм по умолчанию можно
установить с помощью gen_pool_set_algo(). Значение data передаётся алгоритму;
большинство его игнорируют, но иногда оно бывает необходимо. Естественно, можно
написать специализированный алгоритм, но уже доступен достаточный набор готовых:

- gen_pool_first_fit — простой аллокатор по принципу «первое подходящее»; это
  алгоритм по умолчанию, если никакой другой не был указан.

- gen_pool_first_fit_align вынуждает выделение иметь определённое выравнивание
  (передаваемое через data в структуре genpool_data_align).

- gen_pool_first_fit_order_align выравнивает выделение по порядку размера. Так,
  например, выделение в 60 байт будет выровнено по 64 байтам.

- gen_pool_best_fit, как и следовало ожидать, — простой аллокатор по принципу
  «наиболее подходящее».

- gen_pool_fixed_alloc выделяет память по определённому смещению (передаваемому
  в структуре genpool_data_fixed через параметр data) внутри пула. Если
  указанная память недоступна, выделение завершается неудачей.

Существует также небольшое число других функций, в основном для таких целей,
как запрос доступного в пуле пространства или итерирование по фрагментам памяти.
Однако большинству пользователей не должно требоваться ничего сверх того, что
было описано выше. Будем надеяться, что более широкая осведомлённость об этом
модуле поможет предотвратить написание специализированных аллокаторов памяти в
будущем.
