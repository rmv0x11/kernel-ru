===================
Операции this_cpu
===================

:Author: Christoph Lameter, August 4th, 2014
:Author: Pranith Kumar, Aug 2nd, 2014

Операции this_cpu — это способ оптимизации доступа к per cpu переменным,
связанным с *текущим* исполняющимся процессором. Это достигается за счёт
использования сегментных регистров (или выделенного регистра, в котором
процессор постоянно хранит начало per cpu области для конкретного процессора).

Операции this_cpu добавляют смещение per cpu переменной к специфичному для
процессора per cpu базовому адресу и кодируют эту операцию в инструкции,
работающей с per cpu переменной.

Это означает, что между вычислением смещения и операцией над данными нет
проблем атомарности. Поэтому нет необходимости отключать вытеснение или
прерывания, чтобы гарантировать, что процессор не сменится между вычислением
адреса и операцией над данными.

Особый интерес представляют операции read-modify-write. Зачастую процессоры
имеют специальные инструкции с меньшей задержкой (latency), которые могут
работать без типичных накладных расходов на синхронизацию, но при этом всё ещё
предоставляют некоторые ослабленные гарантии атомарности. Например, x86 может
выполнять RMW (Read Modify Write) инструкции вроде inc/dec/cmpxchg без префикса
lock и связанного с ним штрафа по задержке.

Доступ к переменной без префикса lock не синхронизирован, но синхронизация и не
нужна, поскольку мы имеем дело с per cpu данными, специфичными для текущего
исполняющегося процессора. Только текущий процессор должен обращаться к этой
переменной, и поэтому нет проблем конкурентности с другими процессорами в
системе.

Обратите внимание, что доступы удалённых процессоров к per cpu области являются
исключительными ситуациями и могут влиять на производительность и/или
корректность (операции удалённой записи) локальных RMW операций через
this_cpu_*.

Основным применением операций this_cpu была оптимизация операций со счётчиками.

Определены следующие операции this_cpu() с неявной защитой от вытеснения. Эти
операции можно использовать, не беспокоясь о вытеснении и прерываниях::

	this_cpu_read(pcp)
	this_cpu_write(pcp, val)
	this_cpu_add(pcp, val)
	this_cpu_and(pcp, val)
	this_cpu_or(pcp, val)
	this_cpu_add_return(pcp, val)
	this_cpu_xchg(pcp, nval)
	this_cpu_cmpxchg(pcp, oval, nval)
	this_cpu_sub(pcp, val)
	this_cpu_inc(pcp)
	this_cpu_dec(pcp)
	this_cpu_sub_return(pcp, val)
	this_cpu_inc_return(pcp)
	this_cpu_dec_return(pcp)


Внутреннее устройство операций this_cpu
---------------------------------------

На x86 сегментные регистры fs: или gs: содержат базу per cpu области. После
этого можно просто использовать сегментное переопределение, чтобы переместить
относительный per cpu адрес в нужную per cpu область для процессора. Таким
образом, перемещение к per cpu базе кодируется в инструкции через префикс
сегментного регистра.

Например::

	DEFINE_PER_CPU(int, x);
	int z;

	z = this_cpu_read(x);

приводит к единственной инструкции::

	mov ax, gs:[x]

вместо последовательности вычисления адреса и последующей выборки по этому
адресу, которая происходит при per cpu операциях. До this_cpu_ops такая
последовательность также требовала отключения/включения вытеснения (preempt
disable/enable), чтобы предотвратить перемещение ядром потока на другой процессор
во время выполнения вычисления.

Рассмотрим следующую операцию this_cpu::

	this_cpu_inc(x)

Приведённое выше приводит к следующей единственной инструкции (без префикса
lock!)::

	inc gs:[x]

вместо следующих операций, которые требуются при отсутствии сегментного
регистра::

	int *y;
	int cpu;

	cpu = get_cpu();
	y = per_cpu_ptr(&x, cpu);
	(*y)++;
	put_cpu();

Обратите внимание, что эти операции можно использовать только с per cpu данными,
зарезервированными для конкретного процессора. Без отключения вытеснения в
окружающем коде this_cpu_inc() будет лишь гарантировать, что один из per cpu
счётчиков корректно инкрементируется. Однако нет гарантии, что ОС не переместит
процесс непосредственно до или после выполнения инструкции this_cpu. В общем
случае это означает, что значение отдельных счётчиков для каждого процессора не
имеет смысла. Сумма всех per cpu счётчиков — единственное значение, которое
представляет интерес.

Per cpu переменные используются из соображений производительности. Можно
избежать «перебрасывания» (bouncing) строк кэша, если несколько процессоров
одновременно проходят через одни и те же пути кода. Поскольку у каждого
процессора есть собственные per cpu переменные, конкурентных обновлений строк
кэша не происходит. Цена, которую приходится платить за эту оптимизацию, —
необходимость суммировать per cpu счётчики, когда требуется значение счётчика.


Специальные операции
---------------------

::

	y = this_cpu_ptr(&x)

Принимает смещение per cpu переменной (&x !) и возвращает адрес per cpu
переменной, принадлежащей текущему исполняющемуся процессору. this_cpu_ptr
позволяет избежать множества шагов, которые требует обычная
последовательность get_cpu/put_cpu. Номер процессора недоступен. Вместо этого
к per cpu смещению просто добавляется смещение локальной per cpu области.

Обратите внимание, что эту операцию можно использовать только в сегментах кода,
где допустимо применение smp_processor_id(), например там, где вытеснение
отключено. Затем указатель используется для доступа к локальным per cpu данным
в критической секции. Когда вытеснение снова включается, этот указатель обычно
больше не полезен, поскольку он может больше не указывать на per cpu данные
текущего процессора.

Особые случаи, когда имеет смысл получить per-CPU указатель в вытесняемом коде,
обрабатываются raw_cpu_ptr(), но такие сценарии использования должны учитывать
случаи, когда два разных CPU обращаются к одной и той же per cpu переменной,
которая вполне может принадлежать третьему CPU. Эти сценарии использования, как
правило, являются оптимизациями производительности. Например, SRCU реализует
пару счётчиков в виде пары per-CPU переменных, а rcu_read_lock_nmisafe()
использует raw_cpu_ptr() для получения указателя на счётчик некоторого CPU и
использует atomic_inc_long() для обработки миграции между raw_cpu_ptr() и
atomic_inc_long().

Per cpu переменные и смещения
-----------------------------

Per cpu переменные имеют *смещения* относительно начала per cpu области. Они не
имеют адресов, хотя в коде выглядят именно так. Смещения нельзя разыменовывать
напрямую. Смещение должно быть добавлено к базовому указателю per cpu области
процессора, чтобы сформировать корректный адрес.

Поэтому использование x или &x вне контекста per cpu операций некорректно и в
общем случае будет рассматриваться как разыменование NULL-указателя.

::

	DEFINE_PER_CPU(int, x);

В контексте per cpu операций приведённое выше подразумевает, что x является per
cpu переменной. Большинство операций this_cpu принимают cpu переменную.

::

	int __percpu *p = &x;

&x и, следовательно, p — это *смещение* per cpu переменной. this_cpu_ptr()
принимает смещение per cpu переменной, что делает это немного странным на вид.


Операции над полем per cpu структуры
------------------------------------

Допустим, у нас есть percpu структура::

	struct s {
		int n,m;
	};

	DEFINE_PER_CPU(struct s, p);


Операции над этими полями просты::

	this_cpu_inc(p.m)

	z = this_cpu_cmpxchg(p.m, 0, 1);


Если у нас есть смещение к struct s::

	struct s __percpu *ps = &p;

	this_cpu_dec(ps->m);

	z = this_cpu_inc_return(ps->n);


Вычисление указателя может потребовать использования this_cpu_ptr(), если мы не
используем this_cpu ops позднее для манипуляции полями::

	struct s *pp;

	pp = this_cpu_ptr(&p);

	pp->m--;

	z = pp->n++;


Разновидности this_cpu ops
--------------------------

this_cpu ops безопасны по отношению к прерываниям. Некоторые архитектуры не
поддерживают эти per cpu локальные операции. В таком случае операция должна
быть заменена кодом, который отключает прерывания, затем выполняет операции,
гарантированно атомарные, и затем снова включает прерывания. Делать так
дорого. Если есть другие причины, по которым планировщик не может сменить
процессор, на котором мы исполняемся, то нет причин отключать прерывания. Для
этой цели предоставляются следующие операции __this_cpu.

Эти операции не дают гарантий против конкурентных прерываний или вытеснения.
Если per cpu переменная не используется в контексте прерывания и планировщик не
может вытеснить, то они безопасны. Если же какие-либо прерывания всё же
происходят во время выполнения операции и если прерывание тоже изменяет
переменную, то безопасность RMW действий не может быть гарантирована::

	__this_cpu_read(pcp)
	__this_cpu_write(pcp, val)
	__this_cpu_add(pcp, val)
	__this_cpu_and(pcp, val)
	__this_cpu_or(pcp, val)
	__this_cpu_add_return(pcp, val)
	__this_cpu_xchg(pcp, nval)
	__this_cpu_cmpxchg(pcp, oval, nval)
	__this_cpu_sub(pcp, val)
	__this_cpu_inc(pcp)
	__this_cpu_dec(pcp)
	__this_cpu_sub_return(pcp, val)
	__this_cpu_inc_return(pcp)
	__this_cpu_dec_return(pcp)


Инкрементирует x и не будет прибегать к коду, отключающему прерывания, на
платформах, которые не могут обеспечить атомарность через перемещение адреса и
операцию Read-Modify-Write в одной инструкции.


&this_cpu_ptr(pp)->n vs this_cpu_ptr(&pp->n)
--------------------------------------------

Первая операция берёт смещение и формирует адрес, а затем добавляет смещение
поля n. Это может привести к двум инструкциям add, генерируемым компилятором.

Вторая сначала складывает два смещения, а затем выполняет перемещение. На мой
взгляд (IMHO), вторая форма выглядит чище и проще обходится со скобками (). Вторая
форма также согласуется с тем, как используются this_cpu_read() и подобные ему.


Удалённый доступ к per cpu данным
---------------------------------

Per cpu структуры данных спроектированы для эксклюзивного использования одним
cpu. Если вы используете переменные так, как задумано, то this_cpu_ops()
гарантированно «атомарны», поскольку ни один другой CPU не имеет доступа к этим
структурам данных.

Существуют особые случаи, когда вам может понадобиться удалённый доступ к per
cpu структурам данных. Обычно безопасно выполнять удалённый доступ на чтение, и
это часто делается для суммирования счётчиков. Удалённый доступ на запись — это
нечто, что может быть проблематичным, поскольку this_cpu ops не имеют семантики
блокировки. Удалённая запись может помешать this_cpu RMW операции.

Удалённый доступ на запись к percpu структурам данных крайне не рекомендуется,
если только это не абсолютно необходимо. Пожалуйста, рассмотрите использование
IPI для пробуждения удалённого CPU и выполнения обновления его per cpu области.

Для удалённого доступа к per-cpu структуре данных обычно используется функция
per_cpu_ptr()::


	DEFINE_PER_CPU(struct data, datap);

	struct data *p = per_cpu_ptr(&datap, cpu);

Это делает явным то, что мы готовимся к удалённому доступу к percpu области.

Также можно сделать следующее, чтобы преобразовать смещение datap в адрес::

	struct data *p = this_cpu_ptr(&datap);

но передача указателей, вычисленных через this_cpu_ptr, другим cpu необычна и
её следует избегать.

Удалённый доступ обычно предназначен только для чтения состояния per cpu данных
другого cpu. Доступ на запись может вызывать своеобразные проблемы из-за
ослабленных требований к синхронизации для операций this_cpu.

Один пример, иллюстрирующий некоторые опасения относительно операций записи, —
это следующий сценарий, который возникает из-за того, что две per cpu
переменные совместно используют строку кэша, но ослабленная синхронизация
применяется только к одному процессу, обновляющему строку кэша.

Рассмотрим следующий пример::


	struct test {
		atomic_t a;
		int b;
	};

	DEFINE_PER_CPU(struct test, onecacheline);

Есть некоторая обеспокоенность тем, что произойдёт, если поле 'a' обновляется
удалённо с одного процессора, а локальный процессор будет использовать this_cpu
ops для обновления поля b. Следует позаботиться о том, чтобы избегать таких
одновременных доступов к данным в пределах одной и той же строки кэша. Также
может потребоваться дорогостоящая синхронизация. В таких сценариях обычно
рекомендуются IPI вместо удалённой записи в per cpu область другого процессора.

Даже в случаях, когда удалённые записи редки, пожалуйста, имейте в виду, что
удалённая запись вытеснит строку кэша из процессора, который, скорее всего,
будет к ней обращаться. Если процессор пробуждается и обнаруживает отсутствие
локальной строки кэша per cpu области, это повлияет на его производительность, а
значит, и на время пробуждения.
