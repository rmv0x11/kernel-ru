===============================================================
Инфраструктура отладки времени жизни объектов (object-lifetime)
===============================================================

:Author: Thomas Gleixner

Введение
========

debugobjects — это универсальная инфраструктура для отслеживания времени
жизни объектов ядра и проверки выполняемых над ними операций.

debugobjects полезна для выявления следующих шаблонов ошибок:

-  Активация неинициализированных объектов

-  Инициализация активных объектов

-  Использование освобождённых/уничтоженных объектов

debugobjects не изменяет структуру данных реального объекта, поэтому её
можно встроить при компиляции с минимальным влиянием на время выполнения и
включать по требованию с помощью параметра командной строки ядра.

Как использовать debugobjects
=============================

Подсистема ядра должна предоставить структуру данных, описывающую тип
объекта, и добавить вызовы отладочного кода в соответствующих местах. Структуре
данных, описывающей тип объекта, как минимум необходимо имя типа объекта.
Дополнительно можно и следует предоставить функции для исправления
обнаруженных проблем, чтобы ядро могло продолжать работу, а отладочную
информацию можно было получить из работающей системы вместо жёсткой отладки
с последовательными консолями и расшифровками трассировки стека с монитора.

Отладочные вызовы, предоставляемые debugobjects:

-  debug_object_init

-  debug_object_init_on_stack

-  debug_object_activate

-  debug_object_deactivate

-  debug_object_destroy

-  debug_object_free

-  debug_object_assert_init

Каждая из этих функций принимает адрес реального объекта и указатель на
специфичную для типа объекта структуру описания для отладки.

Каждая обнаруженная ошибка регистрируется в статистике, а ограниченное
число ошибок выводится через printk, включая полную трассировку стека.

Статистика доступна через /sys/kernel/debug/debug_objects/stats.
Она предоставляет информацию о количестве предупреждений и количестве
успешных исправлений, а также информацию об использовании внутренних
объектов отслеживания и о состоянии пула внутренних объектов отслеживания.

Отладочные функции
==================

.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_init

Эта функция вызывается всякий раз, когда вызывается функция инициализации
реального объекта.

Когда реальный объект уже отслеживается debugobjects, проверяется, может ли
объект быть инициализирован. Инициализация не допускается для активных и
уничтоженных объектов. Когда debugobjects обнаруживает ошибку, она вызывает
функцию fixup_init из структуры описания типа объекта, если та предоставлена
вызывающим. Функция исправления может устранить проблему до того, как
произойдёт реальная инициализация объекта. Например, она может
деактивировать активный объект, чтобы предотвратить повреждение подсистемы.

Когда реальный объект ещё не отслеживается debugobjects, debugobjects
выделяет объект-трекер для реального объекта и устанавливает состояние
объекта-трекера в ODEBUG_STATE_INIT. Она проверяет, что объект не находится
в стеке вызывающего. Если он находится в стеке вызывающего, то выводится
ограниченное число предупреждений через printk, включая полную трассировку
стека. Вызывающий код должен использовать debug_object_init_on_stack() и
удалить объект до выхода из функции, которая его выделила. См. следующий
раздел.

.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_init_on_stack

Эта функция вызывается всякий раз, когда вызывается функция инициализации
реального объекта, расположенного в стеке.

Когда реальный объект уже отслеживается debugobjects, проверяется, может ли
объект быть инициализирован. Инициализация не допускается для активных и
уничтоженных объектов. Когда debugobjects обнаруживает ошибку, она вызывает
функцию fixup_init из структуры описания типа объекта, если та предоставлена
вызывающим. Функция исправления может устранить проблему до того, как
произойдёт реальная инициализация объекта. Например, она может
деактивировать активный объект, чтобы предотвратить повреждение подсистемы.

Когда реальный объект ещё не отслеживается debugobjects, debugobjects
выделяет объект-трекер для реального объекта и устанавливает состояние
объекта-трекера в ODEBUG_STATE_INIT. Она проверяет, что объект находится в
стеке вызывающего.

Объект, находящийся в стеке, должен быть удалён из трекера вызовом
debug_object_free() до того, как функция, выделяющая объект, вернёт
управление. В противном случае мы продолжаем отслеживать устаревшие объекты.

.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_activate

Эта функция вызывается всякий раз, когда вызывается функция активации
реального объекта.

Когда реальный объект уже отслеживается debugobjects, проверяется, может ли
объект быть активирован. Активация не допускается для активных и
уничтоженных объектов. Когда debugobjects обнаруживает ошибку, она вызывает
функцию fixup_activate из структуры описания типа объекта, если та
предоставлена вызывающим. Функция исправления может устранить проблему до
того, как произойдёт реальная активация объекта. Например, она может
деактивировать активный объект, чтобы предотвратить повреждение подсистемы.

Когда реальный объект ещё не отслеживается debugobjects, вызывается функция
fixup_activate, если она доступна. Это необходимо, чтобы разрешить законную
активацию статически выделенных и инициализированных объектов. Функция
исправления проверяет, является ли объект допустимым, и вызывает функцию
debug_objects_init() для инициализации отслеживания этого объекта.

Когда активация законна, состояние связанного объекта-трекера
устанавливается в ODEBUG_STATE_ACTIVE.


.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_deactivate

Эта функция вызывается всякий раз, когда вызывается функция деактивации
реального объекта.

Когда реальный объект отслеживается debugobjects, проверяется, может ли
объект быть деактивирован. Деактивация не допускается для неотслеживаемых
или уничтоженных объектов.

Когда деактивация законна, состояние связанного объекта-трекера
устанавливается в ODEBUG_STATE_INACTIVE.

.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_destroy

Эта функция вызывается, чтобы пометить объект уничтоженным. Это полезно для
предотвращения использования недопустимых объектов, которые всё ещё доступны
в памяти: либо статически выделенных объектов, либо объектов, которые будут
освобождены позднее.

Когда реальный объект отслеживается debugobjects, проверяется, может ли
объект быть уничтожен. Уничтожение не допускается для активных и уничтоженных
объектов. Когда debugobjects обнаруживает ошибку, она вызывает функцию
fixup_destroy из структуры описания типа объекта, если та предоставлена
вызывающим. Функция исправления может устранить проблему до того, как
произойдёт реальное уничтожение объекта. Например, она может деактивировать
активный объект, чтобы предотвратить повреждение подсистемы.

Когда уничтожение законно, состояние связанного объекта-трекера
устанавливается в ODEBUG_STATE_DESTROYED.

.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_free

Эта функция вызывается перед освобождением объекта.

Когда реальный объект отслеживается debugobjects, проверяется, может ли
объект быть освобождён. Освобождение не допускается для активных объектов.
Когда debugobjects обнаруживает ошибку, она вызывает функцию fixup_free из
структуры описания типа объекта, если та предоставлена вызывающим. Функция
исправления может устранить проблему до того, как произойдёт реальное
освобождение объекта. Например, она может деактивировать активный объект,
чтобы предотвратить повреждение подсистемы.

Обратите внимание, что debug_object_free удаляет объект из трекера.
Последующее использование объекта обнаруживается другими отладочными
проверками.


.. kernel-doc:: lib/debugobjects.c
   :functions: debug_object_assert_init

Эта функция вызывается, чтобы утвердить, что объект был инициализирован.

Когда реальный объект не отслеживается debugobjects, она вызывает
fixup_assert_init из структуры описания типа объекта, предоставленной
вызывающим, с жёстко заданным состоянием объекта ODEBUG_NOT_AVAILABLE.
Функция исправления может устранить проблему вызовом debug_object_init и
других специфичных функций инициализации.

Когда реальный объект уже отслеживается debugobjects, он игнорируется.

Функции исправления (fixup)
===========================

Структура описания типа объекта для отладки
-------------------------------------------

.. kernel-doc:: include/linux/debugobjects.h
   :internal:

fixup_init
-----------

Эта функция вызывается отладочным кодом всякий раз, когда обнаруживается
проблема в debug_object_init. Функция принимает адрес объекта и состояние,
которое в данный момент зафиксировано в трекере.

Вызывается из debug_object_init, когда состояние объекта:

-  ODEBUG_STATE_ACTIVE

Функция возвращает true, когда исправление было успешным, иначе false.
Возвращаемое значение используется для обновления статистики.

Обратите внимание, что функция должна снова вызвать функцию
debug_object_init() после устранения повреждения, чтобы сохранить
согласованность состояния.

fixup_activate
---------------

Эта функция вызывается отладочным кодом всякий раз, когда обнаруживается
проблема в debug_object_activate.

Вызывается из debug_object_activate, когда состояние объекта:

-  ODEBUG_STATE_NOTAVAILABLE

-  ODEBUG_STATE_ACTIVE

Функция возвращает true, когда исправление было успешным, иначе false.
Возвращаемое значение используется для обновления статистики.

Обратите внимание, что функция должна снова вызвать функцию
debug_object_activate() после устранения повреждения, чтобы сохранить
согласованность состояния.

Активация статически инициализированных объектов — особый случай. Когда у
debug_object_activate() нет отслеживаемого объекта для данного адреса
объекта, вызывается fixup_activate() с состоянием объекта
ODEBUG_STATE_NOTAVAILABLE. Функция исправления должна проверить, является ли
это законным случаем статически инициализированного объекта или нет. Если
это так, она вызывает debug_object_init() и debug_object_activate(), чтобы
сделать объект известным трекеру и пометить его активным. В этом случае
функция должна вернуть false, поскольку это не реальное исправление.

fixup_destroy
--------------

Эта функция вызывается отладочным кодом всякий раз, когда обнаруживается
проблема в debug_object_destroy.

Вызывается из debug_object_destroy, когда состояние объекта:

-  ODEBUG_STATE_ACTIVE

Функция возвращает true, когда исправление было успешным, иначе false.
Возвращаемое значение используется для обновления статистики.

fixup_free
-----------

Эта функция вызывается отладочным кодом всякий раз, когда обнаруживается
проблема в debug_object_free. Кроме того, она может вызываться из отладочных
проверок в kfree/vfree, когда активный объект обнаруживается проверками
корректности debug_check_no_obj_freed().

Вызывается из debug_object_free() или debug_check_no_obj_freed(), когда
состояние объекта:

-  ODEBUG_STATE_ACTIVE

Функция возвращает true, когда исправление было успешным, иначе false.
Возвращаемое значение используется для обновления статистики.

fixup_assert_init
-------------------

Эта функция вызывается отладочным кодом всякий раз, когда обнаруживается
проблема в debug_object_assert_init.

Вызывается из debug_object_assert_init() с жёстко заданным состоянием
ODEBUG_STATE_NOTAVAILABLE, когда объект не найден в отладочной корзине.

Функция возвращает true, когда исправление было успешным, иначе false.
Возвращаемое значение используется для обновления статистики.

Обратите внимание, эта функция должна обеспечить вызов debug_object_init()
перед возвратом.

Обработка статически инициализированных объектов — особый случай. Функция
исправления должна проверить, является ли это законным случаем статически
инициализированного объекта или нет. В этом случае следует вызвать только
debug_object_init(), чтобы сделать объект известным трекеру. Затем функция
должна вернуть false, поскольку это не реальное исправление.
