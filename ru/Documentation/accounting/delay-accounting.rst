================================
Учёт задержек (delay accounting)
================================

Задачи сталкиваются с задержками выполнения, когда они ожидают
освобождения какого-либо ресурса ядра; например,
готовая к выполнению задача может ждать освобождения CPU для запуска.

Функциональность поучётного учёта задержек измеряет
задержки, которые испытывает задача при

a) ожидании CPU (находясь в состоянии готовности к выполнению)
b) завершении синхронного блочного ввода-вывода, инициированного задачей
c) подгрузке страниц (swapping in)
d) освобождении памяти (memory reclaim)
e) пробуксовке (thrashing)
f) прямом уплотнении (direct compact)
g) копировании при защите от записи (write-protect copy)
h) IRQ/SOFTIRQ

и предоставляет эту статистику в пространство пользователя через
интерфейс taskstats.

Такие задержки дают обратную связь для корректной установки значений
приоритета CPU, приоритета ввода-вывода и лимита rss задачи. Большие задержки для
важных задач могут служить поводом для повышения соответствующего приоритета.

Эта функциональность, посредством использования интерфейса taskstats, также предоставляет
статистику задержек, агрегированную по всем задачам (или потокам), принадлежащим
группе потоков (соответствующей традиционному процессу Unix). Это часто
требуемая агрегация, которая более эффективно выполняется ядром.

Утилиты пространства пользователя, в частности приложения управления ресурсами, также могут
агрегировать статистику задержек в произвольные группы. Чтобы это стало возможным, статистика
задержек задачи доступна как в течение её жизни, так и при её
завершении, что обеспечивает возможность непрерывного и полного мониторинга.


Интерфейс
---------

Учёт задержек использует интерфейс taskstats, который подробно описан
в отдельном документе в этом каталоге. Taskstats возвращает в пространство пользователя
обобщённую структуру данных, соответствующую статистике по pid и по tgid.
Функциональность учёта задержек заполняет определённые поля
этой структуры. См.

     include/uapi/linux/taskstats.h

для описания полей, относящихся к учёту задержек.
Обычно это будут счётчики, возвращающие накопленную
задержку для cpu, синхронного блочного ввода-вывода, подгрузки страниц, освобождения памяти, пробуксовки страничного
кэша, прямого уплотнения, копирования при защите от записи, IRQ/SOFTIRQ и т. д.

Взятие разности двух последовательных значений заданного
счётчика (скажем, cpu_delay_total) для задачи даст задержку,
испытанную задачей при ожидании соответствующего ресурса
в этом интервале.

Когда задача завершается, записи, содержащие поучётную статистику,
отправляются в пространство пользователя без необходимости подавать команду. Если это последняя завершающаяся
задача группы потоков, статистика по tgid также отправляется. Подробнее
описано в описании интерфейса taskstats.

Утилита пространства пользователя getdelays.c в каталоге tools/accounting позволяет запускать простые
команды и отображать соответствующую статистику задержек. Она
также служит примером использования интерфейса taskstats.

Использование
-------------

Соберите ядро с::

	CONFIG_TASK_DELAY_ACCT=y
	CONFIG_TASKSTATS=y

Учёт задержек по умолчанию отключён при загрузке.
Чтобы включить его, добавьте::

   delayacct

в параметры загрузки ядра. Остальные инструкции ниже предполагают, что это
было сделано. Как вариант, используйте sysctl kernel.task_delayacct для переключения состояния
во время выполнения. Однако обратите внимание, что только задачи, запущенные после включения, будут иметь
информацию delayacct.

После того как система загрузится, используйте утилиту,
подобную getdelays.c, для доступа к задержкам,
которые испытала заданная задача или группа задач (tgid).
Утилита также позволяет выполнить заданную команду
и увидеть соответствующие задержки.

Общий формат команды getdelays::

	getdelays [-dilv] [-t tgid] [-p pid]

Получить задержки с момента загрузки системы для pid 10::

	# ./getdelays -d -p 10
	(output similar to next case)

	# ./getdelays -i -p 1
	printing IO accounting

Приведённую выше команду можно использовать с -v для получения дополнительной отладочной информации.

После запуска системы используйте `delaytop` для получения общесистемной информации о задержках,
которая включает общесистемную информацию PSI и Top-N задач с наибольшей задержкой.
Примечание: для полной функциональности поддержки PSI требуются `CONFIG_PSI=y` и `psi=1`.

`delaytop` — это интерактивный инструмент для мониторинга нагрузки системы и задержек задач.
Он поддерживает несколько вариантов сортировки, режимов отображения и управление с клавиатуры в реальном времени.

Интерактивное управление с клавиатуры во время выполнения::

	o - Select sort field (CPU, IO, IRQ, Memory, etc.)
	M - Toggle display mode (Default/Memory Verbose)
	q - Quit

Доступные поля сортировки (используйте -s/--sort или интерактивную команду)::

	cpu(c)       - CPU delay
	blkio(i)     - I/O delay
	irq(q)       - IRQ delay
	mem(m)       - Total memory delay
	swapin(s)    - Swapin delay (memory verbose mode only)
	freepages(r) - Freepages reclaim delay (memory verbose mode only)
	thrashing(t) - Thrashing delay (memory verbose mode only)
	compact(p)   - Compaction delay (memory verbose mode only)
	wpcopy(w)    - Write page copy delay (memory verbose mode only)

Примеры расширенного использования::

	# ./delaytop -s blkio
	Sorted by IO delay

	# ./delaytop -s mem -M
	Sorted by memory delay in memory verbose mode

	# ./delaytop -p pid
	Print delayacct stats

	# ./delaytop -P num
	Display the top N tasks

	# ./delaytop -n num
	Set delaytop refresh frequency (num times)

	# ./delaytop -d secs
	Specify refresh interval as secs
