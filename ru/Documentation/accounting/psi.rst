================================
PSI - Pressure Stall Information
================================

:Date: April, 2018
:Author: Johannes Weiner <hannes@cmpxchg.org>

Когда возникает конкуренция за CPU, память или устройства ввода-вывода,
рабочие нагрузки сталкиваются со всплесками задержек (latency), потерями
пропускной способности и риском срабатывания OOM-killer.

Без точной оценки такой конкуренции пользователи вынуждены либо
перестраховываться и недоиспользовать аппаратные ресурсы, либо полагаться
на удачу и часто сталкиваться с нарушениями работы, вызванными чрезмерным
overcommit.

Функциональность psi выявляет и количественно оценивает нарушения работы,
вызванные подобной нехваткой ресурсов, и их влияние во времени на сложные
рабочие нагрузки или даже на целые системы.

Наличие точной оценки потерь продуктивности, вызванных дефицитом ресурсов,
помогает пользователям подбирать рабочие нагрузки под оборудование — или
выделять оборудование в соответствии с потребностями рабочей нагрузки.

Поскольку psi агрегирует эту информацию в реальном времени, системами можно
управлять динамически, используя такие приёмы, как сброс нагрузки (load
shedding), миграция задач на другие системы или в другие центры обработки
данных, а также стратегическая приостановка или завершение низкоприоритетных
или перезапускаемых пакетных задач.

Это позволяет максимизировать использование оборудования, не жертвуя
работоспособностью рабочей нагрузки и не рискуя серьёзными нарушениями работы,
такими как срабатывание OOM-killer.

Интерфейс pressure
==================

Информация о pressure для каждого ресурса экспортируется через
соответствующий файл в /proc/pressure/ — cpu, memory и io.

Формат выглядит следующим образом::

	some avg10=0.00 avg60=0.00 avg300=0.00 total=0
	full avg10=0.00 avg60=0.00 avg300=0.00 total=0

Строка «some» обозначает долю времени, в течение которой по крайней мере
некоторые задачи приостановлены (stalled) в ожидании данного ресурса.

Строка «full» обозначает долю времени, в течение которой все незанятые
(non-idle) задачи приостановлены в ожидании данного ресурса одновременно.
В этом состоянии реальные циклы CPU расходуются впустую, и рабочая нагрузка,
проводящая в этом состоянии продолжительное время, считается находящейся в
состоянии thrashing. Это серьёзно влияет на производительность, и полезно
отличать такую ситуацию от состояния, в котором некоторые задачи приостановлены,
но CPU всё ещё выполняет полезную работу. Поэтому время, проведённое в этом
подмножестве состояния приостановки, отслеживается отдельно и экспортируется
в средних значениях «full».

CPU full не определён на уровне системы, но передаётся начиная с версии 5.13,
поэтому он устанавливается в ноль для обратной совместимости.

Соотношения (в %) отслеживаются как недавние тренды по окнам длиной в десять,
шестьдесят и триста секунд, что даёт представление как о краткосрочных событиях,
так и о среднесрочных и долгосрочных трендах. Также отслеживается и
экспортируется общее абсолютное время приостановки (в мкс), что позволяет
обнаруживать всплески задержек, которые необязательно заметно повлияли бы на
средние значения во времени, или усреднять тренды по произвольным временным
интервалам.

Мониторинг порогов pressure
===========================

Пользователи могут регистрировать триггеры и использовать poll(), чтобы
получать пробуждение при превышении pressure ресурса определённых порогов.

Триггер описывает максимальное суммарное время приостановки за определённое
временное окно, например 100 мс общего времени приостановки в пределах любого
окна 500 мс для генерации события пробуждения.

Чтобы зарегистрировать триггер, пользователь должен открыть файл интерфейса psi
в /proc/pressure/, представляющий отслеживаемый ресурс, и записать желаемый порог
и временное окно. Открытый файловый дескриптор следует использовать для ожидания
событий триггера с помощью select(), poll() или epoll(). Применяется следующий
формат::

	<some|full> <stall amount in us> <time window in us>

Например, запись «some 150000 1000000» в /proc/pressure/memory добавила бы порог
150 мс для частичной приостановки по памяти, измеряемой в пределах временного
окна 1 с. Запись «full 50000 1000000» в /proc/pressure/io добавила бы порог 50 мс
для полной приостановки по io, измеряемой в пределах временного окна 1 с.

Триггеры можно задавать для более чем одной метрики psi, и для одной и той же
метрики psi можно указать более одного триггера. Однако для каждого триггера
требуется отдельный файловый дескриптор, чтобы можно было опрашивать его отдельно
от других, поэтому для каждого триггера следует выполнять отдельный системный
вызов open(), даже при открытии одного и того же файла интерфейса psi. Операции
записи в файловый дескриптор с уже существующим триггером psi завершатся ошибкой
EBUSY.

Мониторы активируются только тогда, когда система входит в состояние приостановки
для отслеживаемой метрики psi, и деактивируются при выходе из состояния
приостановки. Пока система находится в состоянии приостановки, рост сигнала psi
отслеживается с частотой 10 раз за окно отслеживания.

Ядро принимает размеры окна в диапазоне от 500 мс до 10 с, поэтому минимальный
интервал обновления мониторинга составляет 50 мс, а максимальный — 1 с.
Минимальный предел установлен для предотвращения чрезмерно частого опроса.
Максимальный предел выбран как достаточно большое число, после которого мониторы,
скорее всего, уже не нужны и вместо них можно использовать средние значения psi.

Непривилегированные пользователи также могут создавать мониторы, с единственным
ограничением: размер окна должен быть кратен 2 с, чтобы предотвратить чрезмерное
использование ресурсов.

После активации монитор psi остаётся активным как минимум в течение
длительности одного окна отслеживания, чтобы избежать повторяющихся
активаций/деактиваций, когда система то входит, то выходит из состояния
приостановки.

Уведомления в пространство пользователя (userspace) ограничены по частоте — одно
на окно отслеживания.

Триггер будет отменён, когда файловый дескриптор, использованный для определения
триггера, будет закрыт.

Пример использования монитора в пространстве пользователя
=========================================================

::

  #include <errno.h>
  #include <fcntl.h>
  #include <stdio.h>
  #include <poll.h>
  #include <string.h>
  #include <unistd.h>

  /*
   * Monitor memory partial stall with 1s tracking window size
   * and 150ms threshold.
   */
  int main() {
	const char trig[] = "some 150000 1000000";
	struct pollfd fds;
	int n;

	fds.fd = open("/proc/pressure/memory", O_RDWR | O_NONBLOCK);
	if (fds.fd < 0) {
		printf("/proc/pressure/memory open error: %s\n",
			strerror(errno));
		return 1;
	}
	fds.events = POLLPRI;

	if (write(fds.fd, trig, strlen(trig) + 1) < 0) {
		printf("/proc/pressure/memory write error: %s\n",
			strerror(errno));
		return 1;
	}

	printf("waiting for events...\n");
	while (1) {
		n = poll(&fds, 1, -1);
		if (n < 0) {
			printf("poll error: %s\n", strerror(errno));
			return 1;
		}
		if (fds.revents & POLLERR) {
			printf("got POLLERR, event source is gone\n");
			return 0;
		}
		if (fds.revents & POLLPRI) {
			printf("event triggered!\n");
		} else {
			printf("unknown event received: 0x%x\n", fds.revents);
			return 1;
		}
	}

	return 0;
  }

Интерфейс cgroup2
=================

В системе с ядром, собранным с CONFIG_CGROUPS=y, и смонтированной файловой
системой cgroup2 информация о pressure также отслеживается для задач,
сгруппированных в cgroup. Каждый подкаталог в точке монтирования cgroupfs
содержит файлы cpu.pressure, memory.pressure и io.pressure; формат такой же,
как у файлов /proc/pressure/.

Мониторы psi для отдельных cgroup можно задавать и использовать так же, как
общесистемные.
