==============================================
Интерфейс статистики по отдельным задачам
==============================================


Taskstats — это интерфейс на основе netlink для передачи статистики по
отдельным задачам и процессам из ядра в пространство пользователя.

Taskstats был спроектирован ради следующих преимуществ:

- эффективное предоставление статистики в течение времени жизни задачи и при её завершении
- единый интерфейс для нескольких подсистем учёта
- расширяемость для использования будущими патчами учёта

Терминология
------------

«pid», «tid» и «task» используются как взаимозаменяемые и обозначают стандартную
задачу Linux, определяемую структурой struct task_struct.  Статистика по pid — это
то же самое, что статистика по задаче.

«tgid», «process» и «thread group» используются как взаимозаменяемые и обозначают
задачи, разделяющие один mm_struct, то есть традиционный Unix-процесс. Несмотря на
использование tgid, задача, являющаяся лидером группы потоков, не получает особого
обращения — процесс считается живым до тех пор, пока ему принадлежит хотя бы одна задача.

Использование
-------------

Чтобы получить статистику в течение времени жизни задачи, пространство пользователя
открывает одноадресный (unicast) сокет netlink (семейства NETLINK_GENERIC) и отправляет
команды, указывающие pid или tgid. Ответ содержит статистику по задаче (если указан pid)
или сумму статистики по всем задачам процесса (если указан tgid).

Чтобы получить статистику по завершающимся задачам, слушатель в пространстве пользователя
отправляет команду регистрации и указывает cpumask. Всякий раз, когда задача завершается на
одном из CPU из cpumask, её статистика по pid отправляется зарегистрированному слушателю.
Использование cpumask позволяет ограничить объём данных, получаемых одним слушателем, и
помогает в управлении потоком данных через интерфейс netlink; это более подробно
объясняется ниже.

Если завершающаяся задача является последним завершающимся потоком своей группы потоков,
в пространство пользователя дополнительно отправляется ещё одна запись, содержащая
статистику по tgid. Последняя содержит сумму статистики по pid для всех потоков группы
потоков, как прошлых, так и текущих.

getdelays.c — это простая утилита, демонстрирующая использование интерфейса taskstats
для вывода статистики учёта задержек. Пользователи могут регистрировать cpumask,
отправлять команды и обрабатывать ответы, прослушивать данные о завершении по tid/tgid,
записывать полученные данные в файл и осуществлять базовое управление потоком, увеличивая
размеры приёмных буферов.

Интерфейс
---------

Интерфейс между пользователем и ядром инкапсулирован в include/linux/taskstats.h

Чтобы эта документация не устаревала по мере развития интерфейса, здесь приводится только
общий обзор текущей версии. taskstats.h всегда имеет приоритет над описанием здесь.

struct taskstats — это общая структура учёта как для данных по pid, так и для данных по
tgid. Она версионируется и может расширяться каждой подсистемой учёта, добавляемой в ядро.
Поля и их семантика определены в файле taskstats.h.

Данные, которыми обмениваются пространство пользователя и ядро, представляют собой
сообщение netlink, относящееся к семейству NETLINK_GENERIC и использующее интерфейс
атрибутов netlink. Сообщения имеют формат::

    +----------+- - -+-------------+-------------------+
    | nlmsghdr | Pad |  genlmsghdr | taskstats payload |
    +----------+- - -+-------------+-------------------+


Полезная нагрузка taskstats бывает одного из следующих трёх видов:

1. Команды: отправляются от пользователя к ядру. Команды для получения данных по
   pid/tgid состоят из одного атрибута типа TASKSTATS_CMD_ATTR_PID/TGID,
   содержащего u32 pid или tgid в полезной нагрузке атрибута. pid/tgid обозначает
   задачу/процесс, для которой пространство пользователя хочет получить статистику.

   Команды для регистрации/отмены регистрации интереса к данным о завершении от набора
   CPU состоят из одного атрибута типа
   TASKSTATS_CMD_ATTR_REGISTER/DEREGISTER_CPUMASK и содержат cpumask в
   полезной нагрузке атрибута. cpumask задаётся как ASCII-строка из
   разделённых запятыми диапазонов CPU; например, чтобы прослушивать данные о завершении от CPU 1,2,3,5,7,8,
   cpumask будет равен "1-3,5,7-8". Если пространство пользователя забывает отменить
   регистрацию интереса к CPU перед закрытием прослушивающего сокета, ядро со временем
   очищает свой набор интереса. Однако ради эффективности рекомендуется явная
   отмена регистрации.

2. Ответ на команду: отправляется ядром в ответ на команду из пространства пользователя.
   Полезная нагрузка представляет собой последовательность из трёх атрибутов типа:

   a) TASKSTATS_TYPE_AGGR_PID/TGID: атрибут, не содержащий полезной нагрузки, но
      указывающий, что за pid/tgid последует некоторая статистика.

   b) TASKSTATS_TYPE_PID/TGID: атрибут, полезной нагрузкой которого является pid/tgid, чья
      статистика возвращается.

   c) TASKSTATS_TYPE_STATS: атрибут с struct taskstats в качестве полезной нагрузки. Одна и
      та же структура используется как для статистики по pid, так и для статистики по tgid.

3. Новое сообщение, отправляемое ядром всякий раз, когда задача завершается. Полезная нагрузка состоит из
   последовательности атрибутов следующего типа:

   a) TASKSTATS_TYPE_AGGR_PID: указывает, что следующие два атрибута будут pid+stats
   b) TASKSTATS_TYPE_PID: содержит pid завершающейся задачи
   c) TASKSTATS_TYPE_STATS: содержит статистику по pid завершающейся задачи
   d) TASKSTATS_TYPE_AGGR_TGID: указывает, что следующие два атрибута будут
      tgid+stats
   e) TASKSTATS_TYPE_TGID: содержит tgid процесса, которому принадлежит задача
   f) TASKSTATS_TYPE_STATS: содержит статистику по tgid для процесса завершающейся
      задачи


Статистика по tgid
------------------

Taskstats предоставляет статистику по процессам в дополнение к статистике по задачам,
поскольку управление ресурсами часто осуществляется на уровне гранулярности процесса, а
агрегирование статистики задач исключительно в пространстве пользователя неэффективно и
потенциально неточно (из-за отсутствия атомарности).

Однако ведение статистики по процессам в дополнение к статистике по задачам внутри ядра
сопряжено с накладными расходами по памяти и времени. Чтобы решить эту проблему, код
taskstats накапливает статистику каждой завершающейся задачи в структуре данных уровня
процесса. Когда завершается последняя задача процесса, накопленные данные уровня процесса
также отправляются в пространство пользователя (вместе с данными по задачам).

Когда пользователь делает запрос на получение данных по tgid, сумма по всем остальным живым
потокам группы суммируется и добавляется к накопленному итогу по ранее завершившимся
потокам той же группы потоков.

Расширение taskstats
--------------------

Существует два способа расширить интерфейс taskstats для экспорта большего числа
показателей по задачам/процессам по мере добавления в ядро в будущем патчей для их сбора:

1. Добавление новых полей в конец существующей struct taskstats. Обратная
   совместимость обеспечивается номером версии внутри
   структуры. Пространство пользователя будет использовать только те поля структуры,
   которые соответствуют используемой им версии.

2. Определение отдельных структур статистики и использование интерфейса атрибутов
   netlink для их возврата. Поскольку пространство пользователя обрабатывает каждый атрибут
   netlink независимо, оно всегда может игнорировать атрибуты, тип которых ему непонятен
   (потому что оно использует более старую версию интерфейса).


Выбор между способами 1 и 2 — это вопрос компромисса между гибкостью и
накладными расходами. Если нужно добавить лишь несколько полей, то предпочтительнее
путь 1, поскольку ядру и пространству пользователя не требуется нести накладные расходы на
обработку новых атрибутов netlink. Но если новые поля слишком сильно расширяют существующую
структуру, заставляя разнородные утилиты учёта в пространстве пользователя без необходимости
получать большие структуры, поля которых не представляют интереса, то расширение структуры
атрибутов было бы оправданным.

Управление потоком для taskstats
--------------------------------

Когда частота завершения задач становится высокой, слушатель может не успевать за частотой
отправки ядром данных о завершении по tid/tgid, что приводит к потере данных. Эта
вероятность усугубляется, когда структура taskstats расширяется, а число CPU становится
большим.

Чтобы избежать потери статистики, пространству пользователя следует выполнить одно или
несколько из следующих действий:

- увеличить размеры приёмных буферов для сокетов netlink, открытых
  слушателями для приёма данных о завершении.

- создать больше слушателей и уменьшить число CPU, прослушиваемых
  каждым слушателем. В крайнем случае на каждый CPU может приходиться по одному слушателю.
  Пользователи также могут рассмотреть установку привязки слушателя к подмножеству
  CPU, которое он прослушивает, особенно если они прослушивают всего один CPU.

Несмотря на эти меры, если пространство пользователя получает сообщения об ошибке ENOBUFS,
указывающие на переполнение приёмных буферов, ему следует принять меры для обработки
потери данных.
