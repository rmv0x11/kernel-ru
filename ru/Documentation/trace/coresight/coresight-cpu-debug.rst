==============================
Модуль отладки CPU Coresight
==============================

   :Author:   Leo Yan <leo.yan@linaro.org>
   :Date:     April 5th, 2017

Введение
--------

Модуль отладки CPU Coresight определён в архитектурном справочном руководстве
ARMv8-a (ARM DDI 0487A.k), глава «Part H: External debug»; CPU может встраивать
отладочный модуль, который в основном используется в двух режимах: self-hosted
debug и external debug. Обычно режим external debug хорошо известен как способ,
при котором внешний отладчик подключается к SoC через порт JTAG; с другой
стороны, программа может задействовать метод отладки, который опирается на режим
self-hosted debug, и данный документ посвящён именно этой части.

Отладочный модуль предоставляет расширение профилирования на основе выборок
(sample-based profiling), которое можно использовать для выборки счётчика команд
CPU, состояния безопасности (secure state), уровня исключений (exception level)
и т. д.; обычно к каждому CPU подключается один выделенный отладочный модуль.
На основе механизма self-hosted debug ядро Linux может обращаться к этим
связанным регистрам из области mmio при возникновении паники ядра. Колбэк-нотификатор
для паники ядра будет дампить связанные регистры для каждого CPU; в итоге это
полезно для вспомогательного анализа паники.


Реализация
----------

- Во время регистрации драйвера он использует EDDEVID и EDDEVID1 — два регистра
  идентификации устройства, чтобы решить, реализовано ли профилирование на основе
  выборок. На некоторых платформах эта аппаратная возможность реализована полностью
  или частично; и если эта возможность не поддерживается, то регистрация завершится
  неудачей.

- На момент написания этой документации отладочный драйвер в основном опирается на
  информацию, собранную колбэк-нотификатором паники ядра из трёх регистров выборки:
  EDPCSR, EDVIDSR и EDCIDSR: из EDPCSR мы можем получить счётчик команд; EDVIDSR
  содержит информацию о состоянии безопасности, уровне исключений, разрядности и
  т. д.; EDCIDSR — это значение контекстного идентификатора, которое содержит
  выбранное значение CONTEXTIDR_EL1.

- Драйвер поддерживает CPU, работающий как в режиме AArch64, так и в режиме AArch32.
  Соглашение об именовании регистров между ними немного различается: AArch64
  использует префикс регистров «ED» (ARM DDI 0487A.k, глава H9.1), а AArch32
  использует префикс «DBG» (ARM DDI 0487A.k, глава G5.1). Драйвер унифицирован для
  использования соглашения об именовании AArch64.

- ARMv8-a (ARM DDI 0487A.k) и ARMv7-a (ARM DDI 0406C.b) имеют разные определения
  битов регистров. Поэтому драйвер сглаживает эти два различия:

  Если PCSROffset=0b0000, то в ARMv8-a возможность EDPCSR не реализована; но
  ARMv7-a определяет «PCSR samples are offset by a value that depends on the
  instruction set state». Для ARMv7-a драйвер дополнительно проверяет, работает ли
  CPU с набором команд ARM или thumb, и калибрует значение PCSR; подробное описание
  смещения приведено в ARMv7-a ARM (ARM DDI 0406C.b), глава C11.11.34 «DBGPCSR,
  Program Counter Sampling Register».

  Если PCSROffset=0b0010, то ARMv8-a определяет «EDPCSR implemented, and samples
  have no offset applied and do not sample the instruction set state in AArch32
  state». Поэтому в ARMv8, если EDDEVID1.PCSROffset равен 0b0010 и CPU работает в
  состоянии AArch32, EDPCSR не выбирается; когда же CPU работает в состоянии AArch64,
  EDPCSR выбирается и смещение не применяется.


Домен тактирования и питания
----------------------------

Прежде чем обращаться к отладочным регистрам, мы должны убедиться, что домен
тактирования и питания включён должным образом. В ARMv8-a ARM (ARM DDI 0487A.k),
глава «H9.1 Debug registers», отладочные регистры распределены по двум доменам:
домену отладки и домену CPU.
::

                                +---------------+
                                |               |
                                |               |
                     +----------+--+            |
        dbg_clock -->|          |**|            |<-- cpu_clock
                     |    Debug |**|   CPU      |
 dbg_power_domain -->|          |**|            |<-- cpu_power_domain
                     +----------+--+            |
                                |               |
                                |               |
                                +---------------+

Для домена отладки пользователь использует DT-привязки «clocks» и «power-domains»,
чтобы указать соответствующий источник тактирования и питания для отладочной
логики. Драйвер по мере необходимости вызывает операции pm_runtime_{put|get} для
управления доменом питания отладки.

Для домена CPU различные конструкции SoC имеют разные схемы управления питанием,
и в итоге это сильно влияет на модуль external debug. Поэтому мы можем разделить
ситуацию на следующие случаи:

- В системах с разумным контроллером питания, который может корректно вести себя
  в отношении домена питания CPU, доменом питания CPU можно управлять из драйвера
  через регистр EDPRCR. Драйвер сначала записывает бит EDPRCR.COREPURQ, чтобы
  включить питание CPU, а затем записывает бит EDPRCR.CORENPDRQ для эмуляции
  отключения питания CPU. В результате это может гарантировать, что домен питания
  CPU включён должным образом в период обращения к отладочным регистрам;

- Некоторые конструкции отключают питание целого кластера, если у всех CPU в
  кластере отключено питание, — включая те части отладочных регистров, которые
  должны оставаться запитанными в домене питания отладки. В таких случаях биты в
  EDPRCR не учитываются, поэтому эти конструкции не поддерживают отладку при
  отключённом питании так, как это предполагали разработчики CoreSight / Debug.
  Это означает, что даже проверка EDPRSR потенциально способна вызвать зависание
  шины, если целевой регистр обесточен.

  В этом случае обращение к отладочным регистрам, пока они обесточены, — это
  верный путь к катастрофе; поэтому нам нужно предотвращать переход CPU в состояния
  пониженного энергопотребления во время загрузки или когда пользователь включает
  модуль во время работы. Подробную информацию об использовании для этого случая
  см. в главе «Как использовать модуль».

Как использовать модуль
-----------------------

Если вы хотите включить отладочную функциональность во время загрузки, вы можете
добавить «coresight_cpu_debug.enable=1» в параметры командной строки ядра.

Драйвер также может работать как модуль, поэтому можно включить отладку при
загрузке модуля через insmod::

  # insmod coresight_cpu_debug.ko debug=1

Если во время загрузки или загрузки модуля через insmod вы не включили отладку,
драйвер использует файловую систему debugfs, чтобы предоставить переключатель для
динамического включения или отключения отладки:

Чтобы включить её, запишите «1» в /sys/kernel/debug/coresight_cpu_debug/enable::

  # echo 1 > /sys/kernel/debug/coresight_cpu_debug/enable

Чтобы отключить её, запишите «0» в /sys/kernel/debug/coresight_cpu_debug/enable::

  # echo 0 > /sys/kernel/debug/coresight_cpu_debug/enable

Как объяснялось в главе «Домен тактирования и питания», если вы работаете на
платформе, которая имеет состояния простоя (idle states), отключающие питание
отладочной логики, а контроллер питания не может корректно обработать запрос от
EDPRCR, то вам сначала следует ограничить состояния простоя CPU, прежде чем
включать возможность отладки CPU; так можно гарантировать обращение к отладочной
логике.

Если вы хотите ограничить состояния простоя во время загрузки, вы можете
использовать «nohlt» или «cpuidle.off=1» в командной строке ядра.

Во время работы вы можете отключить состояния простоя следующими способами:

Можно отключить состояния простоя CPU с помощью подсистемы PM QoS, а именно
используя интерфейс «/dev/cpu_dma_latency» (подробности см. в
Documentation/power/pm_qos_interface.rst). Как указано в документации PM QoS,
запрошенный параметр будет действовать, пока файловый дескриптор не будет
освобождён. Например::

  # exec 3<> /dev/cpu_dma_latency; echo 0 >&3
  ...
  Do some work...
  ...
  # exec 3<>-

То же самое можно сделать и из прикладной программы.

Отключить конкретное состояние простоя конкретного CPU из cpuidle sysfs (см.
Documentation/admin-guide/pm/cpuidle.rst)::

  # echo 1 > /sys/devices/system/cpu/cpu$cpu/cpuidle/state$state/disable
