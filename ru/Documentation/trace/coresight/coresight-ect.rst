=============================================
CoreSight Embedded Cross Trigger (CTI и CTM).
=============================================

    :Author:   Mike Leach <mike.leach@linaro.org>
    :Date:     November 2019

Описание оборудования
---------------------

CoreSight Cross Trigger Interface (CTI) — это аппаратное устройство, которое
принимает отдельные входные и выходные аппаратные сигналы, называемые
триггерами, к устройствам и от устройств и соединяет их через Cross Trigger
Matrix (CTM) с другими устройствами по нумерованным каналам, чтобы
распространять события между устройствами.

например::

 0000000  in_trigs  :::::::
 0 C   0----------->:     :             +======>(other CTI channel IO)
 0  P  0<-----------:     :             v
 0   U 0  out_trigs :     : Channels  *****      :::::::
 0000000            : CTI :<=========>*CTM*<====>: CTI :---+
 #######  in_trigs  :     : (id 0-3)  *****      :::::::   v
 # ETM #----------->:     :                         ^   #######
 #     #<-----------:     :                         +---# ETR #
 ####### out_trigs  :::::::                             #######

Драйвер CTI позволяет программировать CTI для привязки триггеров к каналам.
Когда входной триггер становится активным, привязанный к нему канал также
становится активным. Любой выходной триггер, привязанный к этому каналу, тоже
станет активным. Активный канал распространяется на другие CTI через CTM,
активируя там подключённые выходные триггеры, если только это не отфильтровано
шлюзом каналов CTI (CTI channel gate).

Также можно активировать канал с помощью системного программного обеспечения,
напрямую программируя регистры в CTI.

CTI регистрируются системой для связи с CPU и/или другими устройствами CoreSight
на тракте передачи данных трассировки. Когда эти устройства включаются,
привязанные к ним CTI также включаются. По умолчанию/при включении питания у CTI
нет запрограммированных привязок триггеров/каналов, поэтому они не влияют на
систему, пока не будут явно запрограммированы.

Аппаратные соединения триггеров между CTI и устройствами определяются
реализацией, за исключением случая, когда комбинация CPU/ETM соответствует
архитектуре v8 — в этом случае соединения имеют стандартную, определённую
архитектурой схему.

Аппаратные сигналы триггеров также могут быть подключены к устройствам, не
относящимся к CoreSight (например, к UART), либо распространяться за пределы
чипа в виде аппаратных линий IO.

Все устройства CTI связаны с CTM. Во многих системах будет один эффективный CTM
(один CTM или несколько CTM, соединённых между собой), но возможно, что в
системах существуют сети CTI+CTM, не соединённые между собой через CTM. В таких
системах объявляется индекс CTM для связывания устройств CTI, соединённых через
конкретный CTM.

Файлы и каталоги Sysfs
----------------------

Устройства CTI появляются на существующей шине CoreSight рядом с другими
устройствами CoreSight::

    >$ ls /sys/bus/coresight/devices
     cti_cpu0  cti_cpu2  cti_sys0  etm0  etm2  funnel0  replicator0  tmc_etr0
     cti_cpu1  cti_cpu3  cti_sys1  etm1  etm3  funnel1  tmc_etf0     tpiu0

CTI с именами ``cti_cpu<N>`` связаны с CPU и с любым ETM, используемым этим
ядром. CTI ``cti_sys<N>`` — это CTI общей системной инфраструктуры, которые могут
быть связаны с другими устройствами CoreSight или с другим системным
оборудованием, способным генерировать или использовать сигналы триггеров.::

  >$ ls /sys/bus/coresight/devices/etm0/cti_cpu0
  channels  ctmid  enable  nr_trigger_cons mgmt  power powered  regs
  connections subsystem triggers0 triggers1  uevent

*Ключевые файловые элементы:-*
   * ``enable``: включает/выключает CTI. Чтение позволяет определить текущее
     состояние. Если здесь показано «включено» (1), но ``powered`` показывает
     «без питания» (0), то «включено» означает запрос на включение, когда
     устройство будет под питанием.
   * ``ctmid`` : связанный CTM — актуально только если в системе есть несколько
     кластеров CTI+CTM, не соединённых между собой.
   * ``nr_trigger_cons`` : общее число соединений — каталоги triggers<N>.
   * ``powered`` : чтение позволяет определить, находится ли CTI в данный момент
     под питанием.

*Подкаталоги:-*
   * ``triggers<N>``: содержит список триггеров для отдельного соединения.
   * ``channels``: содержит API каналов — основной программный интерфейс CTI.
   * ``regs``: даёт доступ к необработанным программируемым регистрам CTI.
   * ``mgmt``: стандартные управляющие регистры CoreSight.
   * ``connections``: ссылки на подключённые устройства *CoreSight*. Число ссылок
     может быть от 0 до ``nr_trigger_cons``. Фактическое число задаётся
     значением ``nr_links`` в этом каталоге.


каталоги triggers<N>
~~~~~~~~~~~~~~~~~~~~~

Информация об отдельном соединении триггера. Здесь описываются сигналы триггеров
для соединений CoreSight и не-CoreSight.

Каждый каталог triggers содержит набор параметров, описывающих триггеры
соединения.

   * ``name`` : имя соединения
   * ``in_signals`` : индексы входных сигналов триггеров, используемых в этом
     соединении.
   * ``in_types`` : функциональные типы входных сигналов.
   * ``out_signals`` : выходные сигналы триггеров для этого соединения.
   * ``out_types`` : функциональные типы выходных сигналов.

например::

    >$ ls ./cti_cpu0/triggers0/
    in_signals  in_types  name  out_signals  out_types
    >$ cat ./cti_cpu0/triggers0/name
    cpu0
    >$ cat ./cti_cpu0/triggers0/out_signals
    0-2
    >$ cat ./cti_cpu0/triggers0/out_types
    pe_edbgreq pe_dbgrestart pe_ctiirq
    >$ cat ./cti_cpu0/triggers0/in_signals
    0-1
    >$ cat ./cti_cpu0/triggers0/in_types
    pe_dbgtrigger pe_pmuirq

Если у соединения нет сигналов в триггерах «in» или «out», то соответствующие
параметры будут опущены.

Каталог API каналов
~~~~~~~~~~~~~~~~~~~~

Он предоставляет простой способ привязки триггеров к каналам без необходимости
выполнять множественные операции с регистрами, требуемые при прямом
манипулировании элементами подкаталога «regs».

Этот API предоставляется набором файлов::

   >$ ls ./cti_sys0/channels/
   chan_clear         chan_inuse      chan_xtrigs_out     trigin_attach
   chan_free          chan_pulse      chan_xtrigs_reset   trigin_detach
   chan_gate_disable  chan_set        chan_xtrigs_sel     trigout_attach
   chan_gate_enable   chan_xtrigs_in  trig_filter_enable  trigout_detach
   trigout_filtered

Большая часть обращений к этим элементам имеет вид::

  echo <chan> [<trigger>] > /<device_path>/<operation>

где необязательный <trigger> нужен только для операций trigXX_attach | detach.

например::

   >$ echo 0 1 > ./cti_sys0/channels/trigout_attach
   >$ echo 0 > ./cti_sys0/channels/chan_set

Привязывает trigout(1) к channel(0), затем активирует channel(0), порождая
состояние set на cti_sys0.trigout(1)


*Операции API*

   * ``trigin_attach, trigout_attach``: привязать канал к сигналу триггера.
   * ``trigin_detach, trigout_detach``: отвязать канал от сигнала триггера.
   * ``chan_set``: установить канал — состояние set будет распространено по CTM
     на другие подключённые устройства.
   * ``chan_clear``: сбросить канал.
   * ``chan_pulse``: установить канал на один такт тактового сигнала CoreSight.
   * ``chan_gate_enable``: операция записи устанавливает шлюз CTI на
     распространение (включение) канала на другие устройства. Эта операция
     принимает номер канала. По умолчанию при включении питания шлюз CTI включён
     для всех каналов. Чтение выводит список каналов, в настоящий момент
     включённых на шлюзе.
   * ``chan_gate_disable``: запись номера канала отключает шлюз для этого канала.
   * ``chan_inuse``: показывает каналы, в настоящий момент привязанные к
     какому-либо сигналу.
   * ``chan_free``: показывает каналы без привязанных сигналов.
   * ``chan_xtrigs_sel``: запись номера канала выбирает канал для просмотра,
     чтение показывает номер выбранного канала.
   * ``chan_xtrigs_in``: чтение показывает входные триггеры, привязанные к
     выбранному для просмотра каналу.
   * ``chan_xtrigs_out``:чтение показывает выходные триггеры, привязанные к
     выбранному для просмотра каналу.
   * ``trig_filter_enable``: по умолчанию включено; отключите, чтобы разрешить
     установку потенциально опасных выходных сигналов.
   * ``trigout_filtered``: выходные сигналы триггеров, установка которых
     запрещена, если включена фильтрация ``trig_filter_enable``. Одно из
     применений — предотвращение случайной остановки ядра сигналами
     ``EDBGREQ``.
   * ``chan_xtrigs_reset``: запись 1 очищает всё программирование каналов /
     триггеров. Сбрасывает оборудование устройства в состояние по умолчанию.


В приведённом ниже примере входной триггер с индексом 1 привязывается к каналу
2, а выходной триггер с индексом 6 — к тому же каналу. Затем с помощью
соответствующих атрибутов sysfs исследуется состояние соединений каналов /
триггеров.

Эти настройки означают, что если станет активным либо входной триггер 1, либо
канал 2, то станет активным выходной триггер 6. Затем мы включаем CTI и
используем программное управление каналами для активации канала 2. Активный
канал виден в регистре ``choutstatus``, а активный сигнал — в регистре
``trigoutstatus``. Наконец, сброс канала убирает это.

например::

   .../cti_sys0/channels# echo 2 1 > trigin_attach
   .../cti_sys0/channels# echo 2 6 > trigout_attach
   .../cti_sys0/channels# cat chan_free
   0-1,3
   .../cti_sys0/channels# cat chan_inuse
   2
   .../cti_sys0/channels# echo 2 > chan_xtrigs_sel
   .../cti_sys0/channels# cat chan_xtrigs_trigin
   1
   .../cti_sys0/channels# cat chan_xtrigs_trigout
   6
   .../cti_sys0/# echo 1 > enable
   .../cti_sys0/channels# echo 2 > chan_set
   .../cti_sys0/channels# cat ../regs/choutstatus
   0x4
   .../cti_sys0/channels# cat ../regs/trigoutstatus
   0x40
   .../cti_sys0/channels# echo 2 > chan_clear
   .../cti_sys0/channels# cat ../regs/trigoutstatus
   0x0
   .../cti_sys0/channels# cat ../regs/choutstatus
   0x0
