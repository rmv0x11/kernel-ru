================
CoreSight - Perf
================

    :Author:   Carsten Haitzler <carsten.haitzler@arm.com>
    :Date:     June 29th, 2022

Perf способен локально получать данные трассировки CoreSight и сохранять их в
выходные файлы данных perf. Впоследствии эти данные можно декодировать, чтобы
получить инструкции, которые были трассированы, для целей отладки или
профилирования. Записать такие данные можно командой perf record, например::

   perf record -e cs_etm//u testbinary

Это запустит некоторый тестовый бинарный файл (testbinary) до его завершения и
запишет файл трассировки perf.data. Этот файл будет содержать секции AUX, если
CoreSight работает корректно. Содержимое этого файла можно вывести в виде
читаемого текста командой, например::

   perf report --stdio --dump -i perf.data

Вы должны обнаружить, что некоторые секции этого файла содержат блоки данных AUX,
например::

   0x1e78 [0x30]: PERF_RECORD_AUXTRACE size: 0x11dd0  offset: 0  ref: 0x1b614fc1061b0ad1  idx: 0  tid: 531230  cpu: -1

   . ... CoreSight ETM Trace data: size 73168 bytes
           Idx:0; ID:10;   I_ASYNC : Alignment Synchronisation.
             Idx:12; ID:10;  I_TRACE_INFO : Trace Info.; INFO=0x0 { CC.0 }
             Idx:17; ID:10;  I_ADDR_L_64IS0 : Address, Long, 64 bit, IS0.; Addr=0x0000000000000000;
             Idx:26; ID:10;  I_TRACE_ON : Trace On.
             Idx:27; ID:10;  I_ADDR_CTXT_L_64IS0 : Address & Context, Long, 64 bit, IS0.; Addr=0x0000FFFFB6069140; Ctxt: AArch64,EL0, NS;
             Idx:38; ID:10;  I_ATOM_F6 : Atom format 6.; EEEEEEEEEEEEEEEEEEEEEEEE
             Idx:39; ID:10;  I_ATOM_F6 : Atom format 6.; EEEEEEEEEEEEEEEEEEEEEEEE
             Idx:40; ID:10;  I_ATOM_F6 : Atom format 6.; EEEEEEEEEEEEEEEEEEEEEEEE
             Idx:41; ID:10;  I_ATOM_F6 : Atom format 6.; EEEEEEEEEEEN
             ...

Если вы видите приведённое выше, значит ваша система трассирует данные CoreSight
корректно.

Чтобы собрать perf с поддержкой CoreSight, в каталоге tools/perf выполните::

    make CORESIGHT=1

Для сборки требуется OpenCSD. Вы можете установить пакеты дистрибутива,
обеспечивающие эту поддержку, такие как libopencsd и libopencsd-dev, либо
скачать его и собрать самостоятельно. Апстрим OpenCSD находится по адресу:

  https://github.com/Linaro/OpenCSD

Для получения полной информации о сборке perf с поддержкой CoreSight и более
подробных сведений об использовании см.:

  https://github.com/Linaro/OpenCSD/blob/master/HOWTO.md


Поддержка CoreSight в ядре
--------------------------

Вам также потребуется включить поддержку CoreSight в конфигурации вашего ядра.
Убедитесь, что она включена::

   CONFIG_CORESIGHT=y

Существуют различные другие параметры CoreSight, которые вы, вероятно, также
захотите включить, например::

   CONFIG_CORESIGHT_LINKS_AND_SINKS=y
   CONFIG_CORESIGHT_LINK_AND_SINK_TMC=y
   CONFIG_CORESIGHT_CATU=y
   CONFIG_CORESIGHT_SINK_TPIU=y
   CONFIG_CORESIGHT_SINK_ETBV10=y
   CONFIG_CORESIGHT_SOURCE_ETM4X=y
   CONFIG_CORESIGHT_CTI=y
   CONFIG_CORESIGHT_CTI_INTEGRATION_REGS=y

Дополнительную информацию см. в справке по конфигурации ядра.

Точная трассировка с приостановкой и возобновлением AUX
-------------------------------------------------------

Arm CoreSight может генерировать большой объём данных аппаратной трассировки, что
приводит к накладным расходам при записи и отвлекает пользователей при анализе
результатов профилирования. Чтобы смягчить проблему чрезмерного объёма данных
трассировки, Perf предоставляет функциональность приостановки и возобновления AUX
для точной трассировки.

Приостановку и возобновление AUX можно инициировать связанными событиями. Этими
событиями могут быть точки трассировки ftrace (включая статические и динамические
точки трассировки) или события PMU (например, событие циклов CPU PMU). Чтобы
создать сессию perf с приостановкой / возобновлением AUX, вводятся три
конфигурационных термина:

- "aux-action=start-paused": задаётся для события cs_etm PMU, чтобы запустить его
  в приостановленном состоянии.
- "aux-action=pause": с этим термином задаётся связанное событие, чтобы
  приостановить трассировку AUX.
- "aux-action=resume": с этим термином задаётся связанное событие, чтобы
  возобновить трассировку AUX.

Пример инициирования приостановки и возобновления AUX с помощью точек трассировки ftrace::

  perf record -e cs_etm/aux-action=start-paused/k,syscalls:sys_enter_openat/aux-action=resume/,syscalls:sys_exit_openat/aux-action=pause/ ls

Пример инициирования приостановки и возобновления AUX с помощью события PMU::

  perf record -a -e cs_etm/aux-action=start-paused/k \
        -e cycles/aux-action=pause,period=10000000/ \
        -e cycles/aux-action=resume,period=1050000/ -- sleep 1

Perf test - Проверка работы perf CoreSight в ядре и пространстве пользователя
-----------------------------------------------------------------------------

При запуске perf test будет выполнено множество самопроверок. Некоторые из этих
тестов охватывают CoreSight (только если он включён и на ARM64). Обычно perf test
запускают из каталога tools/perf в дереве ядра. Некоторые тесты проверяют
внутреннюю поддержку perf, например:

   Check Arm CoreSight trace data recording and synthesized samples
   Check Arm SPE trace data recording and synthesized samples

Некоторые другие фактически используют perf record и некоторые тестовые бинарные
файлы, находящиеся в tests/shell/coresight, и собирают трассировки, чтобы
убедиться в достижении минимального уровня функциональности. Скрипты, запускающие
эти тесты, находятся в том же каталоге. Все они будут выглядеть так:

   CoreSight / ASM Pure Loop
   CoreSight / Memcpy 16k 10 Threads
   CoreSight / Thread Loop 10 Threads - Check TID
   и т. д.

Эти тесты perf record не будут запущены, если бинарные файлы инструментов
отсутствуют в tests/shell/coresight/\*/, и будут пропущены. Если у вас нет
аппаратной поддержки CoreSight, то либо не собирайте perf с поддержкой CoreSight,
либо удалите эти бинарные файлы, чтобы эти тесты не завершались с ошибкой, а
вместо этого пропускались.

Эти тесты будут записывать историю результатов в текущем рабочем каталоге
(например, tools/perf), и их имена будут вида stats-\*.csv, например:

   stats-asm_pure_loop-out.csv
   stats-memcpy_thread-16k_10.csv
   ...

Эти файлы статистики записывают некоторые аспекты секций данных AUX в выходных
данных perf, подсчитывая некоторые количества определённых кодировок (простой
способ узнать, что всё работает, на очень базовом уровне). Одна из проблем с
CoreSight состоит в том, что при достаточно большом объёме данных, требующих
записи, часть из них может быть потеряна из-за того, что процессор не успевает
вовремя проснуться, чтобы считать все данные из буферов и т. п. Вы заметите, что
объём собранных данных может сильно различаться от запуска к запуску perf test.
Если вы хотите увидеть, как это меняется со временем, просто запустите perf test
несколько раз, и во все эти csv-файлы будет добавляться всё больше данных,
которые впоследствии можно изучить, построить графики и иным образом
использовать, чтобы понять, стало хуже или лучше.

Это означает, что иногда эти тесты завершаются с ошибкой, поскольку не
захватывают все необходимые данные. Речь идёт об отслеживании качества и объёма
производимых данных с течением времени и о том, чтобы видеть, когда изменения в
ядре Linux улучшают качество трассировок.

Учтите, что некоторые из этих тестов выполняются довольно долго, особенно при
обработке файла данных perf и выводе содержимого для последующего изучения того,
что находится внутри.

Изменить место хранения этих csv-логов можно, установив переменную окружения
PERF_TEST_CORESIGHT_STATDIR перед запуском perf test, например::

   export PERF_TEST_CORESIGHT_STATDIR=/var/tmp
   perf test

Они также будут сохранять полученные выходные данные perf в текущем каталоге для
последующего просмотра, например::

   perf-asm_pure_loop-out.data
   perf-memcpy_thread-16k_10.data
   ...

Изменить место хранения файлов данных perf можно, установив переменную окружения
PERF_TEST_CORESIGHT_DATADIR, например::

   PERF_TEST_CORESIGHT_DATADIR=/var/tmp
   perf test

Возможно, вы захотите установить указанные выше переменные окружения, если хотите
хранить выходные данные тестов вне текущего рабочего каталога для более
длительного хранения и изучения.
