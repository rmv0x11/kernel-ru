================================
Методы ACPI _OSI и _REV
================================

ACPI BIOS может использовать метод «Operating System Interfaces» (_OSI),
чтобы узнать, что поддерживает операционная система. Например, если AML-код
BIOS содержит _OSI("XYZ"), интерпретатор AML в ядре может вычислить этот
метод, проверить, поддерживается ли «XYZ», и ответить BIOS YES или NO.

Метод ACPI _REV возвращает «Revision of the ACPI specification
that OSPM supports» (ревизию спецификации ACPI, которую поддерживает OSPM).

В этом документе объясняется, как и почему BIOS и Linux должны использовать
эти методы. Также объясняется, как и почему они повсеместно используются
неправильно.

Как использовать _OSI
=====================

Linux работает на двух группах машин — на тех, которые были протестированы
OEM-производителем на совместимость с Linux, и на тех, которые никогда не
тестировались с Linux, но на которые Linux был установлен взамен исходной ОС
(Windows или OSX).

Бо́льшая группа — это системы, протестированные только на работу с Windows.
Более того, многие из них тестировались на работу лишь с одной конкретной
версией Windows. Поэтому, даже если BIOS использует _OSI для запроса о том,
какая версия Windows запущена, реально протестирован был только один путь
исполнения через BIOS. Опыт показывает, что прохождение непротестированных
путей через BIOS подвергает Linux целой категории ошибок BIOS.
По этой причине значения _OSI по умолчанию в Linux должны и впредь заявлять
о совместимости со всеми версиями Windows.

Но Linux на самом деле несовместим с Windows, и сообщество Linux также не раз
страдало от регрессий, когда Linux добавлял последнюю версию Windows в свой
список строк _OSI. Поэтому возможно, что в будущем дополнительные строки будут
более тщательно проверяться перед выпуском в апстрим. Но, скорее всего, в
конечном итоге все они будут добавлены.

Что делать OEM-производителю, если он хочет поддерживать Linux и Windows,
используя один и тот же образ BIOS? Зачастую им нужно делать что-то иначе для
Linux, чтобы учесть отличия Linux от Windows.

В этом случае OEM-производителю следует создать собственный код ASL, который
будет исполняться ядром Linux, и внести изменения в драйверы ядра Linux,
исполняющие этот собственный ASL. Проще всего этого добиться, введя
device specific method (_DSM), вызываемый из ядра Linux.

В прошлом ядро поддерживало нечто вроде:
_OSI("Linux-OEM-my_interface_name"),
где «OEM» нужен, если это OEM-специфичная зацепка (hook),
а «my_interface_name» описывает зацепку, которая может быть
особенностью (quirk), ошибкой или исправлением ошибки.

Однако обнаружилось, что этим злоупотребляют другие поставщики BIOS, изменяя
совершенно несвязанный код на совершенно несвязанных системах. Это побудило
провести оценку всех вариантов их использования. В ходе неё выяснилось, что они
не нужны ни по одной из изначальных причин. Поэтому ядро по умолчанию не будет
отвечать ни на какие собственные строки Linux-\*.

Это было просто. Читайте дальше, чтобы узнать, как сделать неправильно.

До _OSI был _OS
===============

ACPI 1.0 определял «_OS» как
«object that evaluates to a string that identifies the operating system»
(объект, который вычисляется в строку, идентифицирующую операционную систему).

Поток ACPI BIOS включал вычисление _OS, и интерпретатор AML в ядре возвращал
ему строку, идентифицирующую ОС:

Windows 98, SE: "Microsoft Windows"
Windows ME: "Microsoft WindowsME:Millennium Edition"
Windows NT: "Microsoft Windows NT"

Идея состояла в том, что на платформе, предназначенной для запуска нескольких
ОС, BIOS мог использовать _OS, чтобы включить устройства, которые ОС могла бы
поддерживать, либо включить особенности (quirks) или обходные пути для ошибок,
необходимые для совместимости платформы с этой уже существующей ОС.

Но у _OS были фундаментальные проблемы. Во-первых, BIOS должен был знать имя
каждой возможной версии ОС, которая на нём будет работать, и должен был знать
все особенности этих ОС. Безусловно, для BIOS было бы разумнее спрашивать у ОС
*конкретные* вещи, например «поддерживаешь ли ты конкретный интерфейс», и так
в ACPI 3.0 родился _OSI на замену _OS.

От _OS отказались, хотя даже сегодня многие BIOS ищут _OS «Microsoft Windows
NT», хотя кажется несколько надуманным, что кто-то станет устанавливать эти
старые операционные системы поверх того, что поставлялось с машиной.

Linux отвечает «Microsoft Windows NT», чтобы угодить этой идиоме BIOS.
Это *единственная* жизнеспособная стратегия, поскольку именно так поступает
современная Windows, а значит, поступив иначе, можно было бы направить BIOS по
непротестированному пути.

Рождение _OSI и немедленное злоупотребление им
==============================================

В случае _OSI строку, описывающую интерфейс, предоставляет *BIOS*,
и он спрашивает ОС: «YES/NO, совместима ли ты с этим интерфейсом?»

Например, _OSI("3.0 Thermal Model") вернёт TRUE, если ОС знает, как
работать с тепловыми расширениями, внесёнными в спецификацию ACPI 3.0.
Старая ОС, которая не знает об этих расширениях, ответит FALSE,
а новая ОС, возможно, сможет вернуть TRUE.

Для ОС-специфичного интерфейса спецификация ACPI предписывала, что BIOS и ОС
должны договориться о строке вида «Windows-interface_name».

Но произошли две скверные вещи. Во-первых, экосистема Windows стала
использовать _OSI не так, как было задумано, а как прямую замену _OS — для
идентификации версии ОС, а не поддерживаемого ОС интерфейса. Более того, прямо
с самого начала сама спецификация ACPI 3.0 закрепила это неправильное
использование в примере кода с _OSI("Windows 2001").

Это неправильное использование было перенято и сохраняется по сей день.

У Linux не оставалось иного выбора, кроме как тоже возвращать TRUE на
_OSI("Windows 2001") и его преемников. Поступить иначе означало бы практически
гарантированно сломать BIOS, который был протестирован только с тем _OSI,
возвращающим TRUE.

Эта стратегия проблематична, поскольку Linux никогда не бывает полностью
совместим с последней версией Windows, и иногда требуется больше года, чтобы
устранить несовместимости.

Не желая отставать, сообщество Linux усугубило положение, начав возвращать
TRUE на _OSI("Linux"). Это даже хуже, чем злоупотребление _OSI со стороны
Windows, поскольку «Linux» вообще не содержит никакой информации о версии.
_OSI("Linux") привёл к сбоям в работе некоторых BIOS из-за того, что авторы
BIOS использовали его в непротестированных потоках BIOS. Но некоторые
OEM-производители использовали _OSI("Linux") в протестированных потоках для
поддержки реальных возможностей Linux. В 2009 году Linux удалил _OSI("Linux")
и добавил параметр командной строки для его восстановления на устаревших
системах, которым он ещё нужен. Кроме того, для всех BIOS, которые его
вызывают, выводится предупреждение BIOS_BUG.

Ни один BIOS не должен использовать _OSI("Linux").

В результате получается стратегия для Linux по максимизации совместимости с
ACPI BIOS, протестированными на машинах с Windows. Существует реальный риск
преувеличить эту совместимость; но альтернативой нередко был катастрофический
сбой из-за того, что BIOS шёл путями, которые никогда не проверялись ни под
*какой* ОС.

Не используйте _REV
===================

С тех пор как _OSI("Linux") исчез, некоторые авторы BIOS стали использовать
_REV для поддержки различий между Linux и Windows в одном и том же BIOS.

_REV был определён в ACPI 1.0 для возврата версии ACPI, поддерживаемой ОС и
интерпретатором AML этой ОС.

Современная Windows возвращает _REV = 2. Linux использовал
ACPI_CA_SUPPORT_LEVEL, который увеличивался в зависимости от поддерживаемой
версии спецификации.

К сожалению, _REV также использовался неправильно. Например, некоторые BIOS
проверяли _REV = 3 и делали что-то для Linux, но когда Linux возвращал
_REV = 4, эта поддержка ломалась.

В ответ на эту проблему Linux, начиная с середины 2015 года, всегда возвращает
_REV = 2. Спецификация ACPI также будет обновлена, чтобы отразить, что _REV
устарел и всегда возвращает 2.

Apple Mac и _OSI("Darwin")
==========================

На платформах Apple Mac ACPI BIOS вызывает _OSI("Darwin"), чтобы определить,
работает ли машина под управлением Apple OSX.

Подобно стратегии Linux с _OSI("*Windows*"), Linux по умолчанию отвечает YES
на _OSI("Darwin"), чтобы обеспечить полный доступ к оборудованию и к
проверенным путям BIOS, которые видит OSX. Как и на платформах,
протестированных с Windows, у этой стратегии есть риски.

Начиная с Linux-3.18 ядро отвечало YES на _OSI("Darwin") для того, чтобы
включить поддержку Mac Thunderbolt. Кроме того, если ядро замечало вызов
_OSI("Darwin"), оно дополнительно отключало все _OSI("*Windows*"), чтобы не
дать плохо написанным Mac BIOS пойти по непротестированным комбинациям путей.

Изменение умолчания в Linux-3.18 вызвало регрессии энергопотребления на
ноутбуках Mac, а реализация в 3.18 не позволяла изменить умолчание через
параметр командной строки «acpi_osi=!Darwin». Linux-4.7 исправил возможность
использовать acpi_osi=!Darwin в качестве обходного решения, и мы надеемся
увидеть поддержку управления энергопотреблением Mac Thunderbolt в Linux-4.11.
