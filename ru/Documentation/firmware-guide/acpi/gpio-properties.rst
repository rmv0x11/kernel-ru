==========================================================
Свойства устройств _DSD, относящиеся к GPIO
==========================================================

С выходом ACPI 5.1 конфигурационный объект _DSD наконец-то позволяет
давать имена линиям GPIO (и не только им), возвращаемым _CRS. Раньше для
поиска соответствующего GPIO можно было использовать только целочисленный
индекс, что весьма подвержено ошибкам (например, зависит от порядка вывода
_CRS).

С помощью _DSD теперь можно запрашивать GPIO по имени вместо целочисленного
индекса, как показано в примере ASL ниже::

  // Bluetooth device with reset and shutdown GPIOs
  Device (BTH)
  {
      Name (_HID, ...)

      Name (_CRS, ResourceTemplate ()
      {
          GpioIo (Exclusive, PullUp, 0, 0, IoRestrictionOutputOnly,
                  "\\_SB.GPO0", 0, ResourceConsumer) { 15 }
          GpioIo (Exclusive, PullUp, 0, 0, IoRestrictionOutputOnly,
                  "\\_SB.GPO0", 0, ResourceConsumer) { 27, 31 }
      })

      Name (_DSD, Package ()
      {
          ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
          Package ()
          {
              Package () { "reset-gpios", Package () { ^BTH, 1, 1, 0 } },
              Package () { "shutdown-gpios", Package () { ^BTH, 0, 0, 0 } },
          }
      })
  }

Формат поддерживаемого свойства GPIO следующий::

  Package () { "name", Package () { ref, index, pin, active_low }}

ref
  Устройство, имеющее _CRS с ресурсами GpioIo()/GpioInt(); как правило,
  это само устройство (BTH в нашем случае).
index
  Индекс ресурса GpioIo()/GpioInt() в _CRS, начиная с нуля.
pin
  Вывод (pin) в ресурсе GpioIo()/GpioInt(). Обычно равен нулю.
active_low
  Если равно 1, GPIO помечается как active-low.

Поскольку ресурс ACPI GpioIo() не содержит поля, указывающего, является ли
он active-low или active-high, здесь можно использовать аргумент
"active_low". Установка его в 1 помечает GPIO как active-low.

Обратите внимание, что active_low в _DSD не имеет смысла для ресурса
GpioInt() и должен быть равен 0. У ресурса GpioInt() есть собственные
средства для его определения.

В нашем примере с Bluetooth "reset-gpios" ссылается на второй ресурс
GpioIo(), второй вывод в этом ресурсе с номером GPIO 31.

К сожалению, ресурс GpioIo() явно не задаёт начальное состояние выходного
вывода, которое драйвер должен использовать при инициализации.

Linux здесь старается руководствоваться здравым смыслом и выводит состояние
из настроек смещения (bias) и полярности. В таблице ниже показаны
ожидаемые значения:

+-------------+-------------+-----------------------------------------------+
| Pull Bias   | Polarity    | Requested...                                  |
+=============+=============+===============================================+
| Implicit                                                                  |
+-------------+-------------+-----------------------------------------------+
| **Default** | x           | AS IS (assumed firmware configured it for us) |
+-------------+-------------+-----------------------------------------------+
| Explicit                                                                  |
+-------------+-------------+-----------------------------------------------+
| **None**    | x           | AS IS (assumed firmware configured it for us) |
|             |             | with no Pull Bias                             |
+-------------+-------------+-----------------------------------------------+
| **Up**      | x (no _DSD) |                                               |
|             +-------------+ as high, assuming non-active                  |
|             | Low         |                                               |
|             +-------------+-----------------------------------------------+
|             | High        | as high, assuming active                      |
+-------------+-------------+-----------------------------------------------+
| **Down**    | x (no _DSD) |                                               |
|             +-------------+ as low, assuming non-active                   |
|             | High        |                                               |
|             +-------------+-----------------------------------------------+
|             | Low         | as low, assuming active                       |
+-------------+-------------+-----------------------------------------------+

При этом для нашего примера выше, поскольку настройка смещения задана явно
и присутствует _DSD, оба GPIO будут трактоваться как активные с высокой
полярностью, и Linux настроит выводы в этом состоянии до тех пор, пока
драйвер не перепрограммирует их иначе.

В массиве GPIO допустимо оставлять пропуски. Это полезно в таких случаях,
как с управляющими контроллерами SPI, где часть линий выбора кристалла
(chip select) может быть реализована как GPIO, а часть — как нативные
сигналы. Например, управляющий контроллер SPI может иметь линии выбора
кристалла 0 и 2, реализованные как GPIO, и 1 — как нативную::

  Package () {
      "cs-gpios",
      Package () {
          ^GPIO, 19, 0, 0, // chip select 0: GPIO
          0,               // chip select 1: native signal
          ^GPIO, 20, 0, 0, // chip select 2: GPIO
      }
  }

Обратите внимание, что исторически в ACPI отсутствуют средства задания
полярности GPIO, и поэтому ресурс SPISerialBus() определяет её на уровне
отдельного кристалла. Чтобы избежать цепочки отрицаний, полярность GPIO
считается Active High. Даже в случаях, когда задействован _DSD() (см.
пример выше), полярность GPIO CS должна определяться как Active High во
избежание неоднозначности.

Другие поддерживаемые свойства
==============================

Следующие свойства устройств, совместимые с Device Tree, также
поддерживаются свойствами устройств _DSD для контроллеров GPIO:

- gpio-hog
- output-high
- output-low
- input
- line-name

Пример::

  Name (_DSD, Package () {
      // _DSD Hierarchical Properties Extension UUID
      ToUUID("dbb8e3e6-5886-4ba6-8795-1319f52a966b"),
      Package () {
          Package () { "hog-gpio8", "G8PU" }
      }
  })

  Name (G8PU, Package () {
      ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
      Package () {
          Package () { "gpio-hog", 1 },
          Package () { "gpios", Package () { 8, 0 } },
          Package () { "output-high", 1 },
          Package () { "line-name", "gpio8-pullup" },
      }
  })

- gpio-line-names

Объявление ``gpio-line-names`` — это список строк ("имён"), который
описывает каждую линию/вывод контроллера/расширителя GPIO. Этот список,
содержащийся в пакете (package), должен быть вставлен внутрь объявления
контроллера GPIO в таблице ACPI (как правило, внутри DSDT). Список
``gpio-line-names`` должен соблюдать следующие правила (см. также примеры):

  - первое имя в списке соответствует первой линии/выводу
    контроллера/расширителя GPIO
  - имена внутри списка должны идти последовательно (никакие "пропуски" не
    допускаются)
  - список может быть неполным и может заканчиваться до последней линии
    GPIO: иными словами, заполнять все линии GPIO не обязательно
  - пустые имена допускаются (две кавычки ``""`` соответствуют пустому
    имени)
  - имена внутри одного контроллера/расширителя GPIO должны быть уникальны

Пример контроллера GPIO с 16 линиями, с неполным списком и двумя пустыми
именами::

  Package () {
      "gpio-line-names",
      Package () {
          "pin_0",
          "pin_1",
          "",
          "",
          "pin_3",
          "pin_4_push_button",
      }
  }

Во время выполнения приведённое выше объявление даёт следующий результат (с
использованием инструментов "libgpiod")::

  root@debian:~# gpioinfo gpiochip4
  gpiochip4 - 16 lines:
          line   0:      "pin_0"       unused   input  active-high
          line   1:      "pin_1"       unused   input  active-high
          line   2:      unnamed       unused   input  active-high
          line   3:      unnamed       unused   input  active-high
          line   4:      "pin_3"       unused   input  active-high
          line   5: "pin_4_push_button" unused input active-high
          line   6:      unnamed       unused   input  active-high
          line   7       unnamed       unused   input  active-high
          line   8:      unnamed       unused   input  active-high
          line   9:      unnamed       unused   input  active-high
          line  10:      unnamed       unused   input  active-high
          line  11:      unnamed       unused   input  active-high
          line  12:      unnamed       unused   input  active-high
          line  13:      unnamed       unused   input  active-high

Ещё один пример::

  Package () {
      "gpio-line-names",
      Package () {
          "SPI0_CS_N", "EXP2_INT", "MUX6_IO", "UART0_RXD",
          "MUX7_IO", "LVL_C_A1", "MUX0_IO", "SPI1_MISO",
      }
  }

Дополнительные сведения об этих свойствах см. в
Documentation/devicetree/bindings/gpio/gpio.txt.

Сопоставления ACPI GPIO, предоставляемые драйверами
===================================================

Существуют системы, в которых таблицы ACPI не содержат _DSD, но
предоставляют _CRS с ресурсами GpioIo()/GpioInt(), и драйверам устройств
всё равно нужно с ними работать.

В таких случаях доступные драйверу объекты идентификации ACPI-устройства,
_HID, _CID, _CLS, _SUB, _HRV, могут быть использованы для идентификации
устройства, и предполагается, что этого достаточно для определения смысла и
назначения всех линий GPIO, перечисленных в ресурсах GpioIo()/GpioInt(),
возвращаемых _CRS. Иными словами, предполагается, что драйвер знает, что
именно использовать из ресурсов GpioIo()/GpioInt(), как только он
идентифицировал устройство. Сделав это, он может просто назначить имена
линиям GPIO, которые он собирается использовать, и предоставить подсистеме
GPIO сопоставление между этими именами и соответствующими им ресурсами ACPI
GPIO.

Чтобы сделать это, драйвер должен определить таблицу сопоставления в виде
завершающегося NULL массива объектов struct acpi_gpio_mapping, каждый из
которых содержит имя, указатель на массив объектов с данными линий (struct
acpi_gpio_params) и размер этого массива. Каждый объект struct
acpi_gpio_params состоит из трёх полей: crs_entry_index, line_index,
active_low, представляющих, соответственно, индекс целевого ресурса
GpioIo()/GpioInt() в _CRS, начиная с нуля, индекс целевой линии в этом
ресурсе, начиная с нуля, и флаг active-low для этой линии, по аналогии с
форматом свойства _DSD GPIO, указанным выше.

Для рассмотренного ранее примера устройства Bluetooth соответствующие
структуры данных выглядели бы так::

  static const struct acpi_gpio_params reset_gpio = { 1, 1, false };
  static const struct acpi_gpio_params shutdown_gpio = { 0, 0, false };

  static const struct acpi_gpio_mapping bluetooth_acpi_gpios[] = {
      { "reset-gpios", &reset_gpio, 1 },
      { "shutdown-gpios", &shutdown_gpio, 1 },
      { }
  };

Далее таблицу сопоставления нужно передать в качестве второго аргумента в
acpi_dev_add_driver_gpios() или его managed-аналог, который зарегистрирует
её для объекта ACPI-устройства, на который указывает её первый аргумент.
Это следует делать в процедуре .probe() драйвера. При удалении драйвер
должен отменить регистрацию своей таблицы сопоставления GPIO, вызвав
acpi_dev_remove_driver_gpios() для объекта ACPI-устройства, для которого эта
таблица была ранее зарегистрирована.

Использование запасного варианта с _CRS
=======================================

Если у устройства нет _DSD или драйвер не создаёт сопоставление ACPI GPIO,
GPIO-фреймворк Linux отказывается возвращать какие-либо GPIO. Это связано с
тем, что драйвер не знает, что именно он получает. Например, если у нас
есть устройство, как показано ниже::

  Device (BTH)
  {
      Name (_HID, ...)

      Name (_CRS, ResourceTemplate () {
          GpioIo (Exclusive, PullNone, 0, 0, IoRestrictionNone,
                  "\\_SB.GPO0", 0, ResourceConsumer) { 15 }
          GpioIo (Exclusive, PullNone, 0, 0, IoRestrictionNone,
                  "\\_SB.GPO0", 0, ResourceConsumer) { 27 }
      })
  }

Драйвер может ожидать получить нужный GPIO, когда он выполняет::

  desc = gpiod_get(dev, "reset", GPIOD_OUT_LOW);
  if (IS_ERR(desc))
	...error handling...

но поскольку нет способа узнать сопоставление между "reset" и GpioIo() в
_CRS, desc будет содержать ERR_PTR(-ENOENT).

Автор драйвера может решить это, передав сопоставление явно (это
рекомендуемый способ, и он описан в главе выше).

Таблицы сопоставления ACPI GPIO не должны загрязнять драйверы, которые не
знают, какому именно устройству они служат. Это подразумевает, что таблицы
сопоставления ACPI GPIO жёстко привязаны к ACPI ID и определённым объектам
рассматриваемого устройства, перечисленным в главе выше.

Получение дескриптора GPIO
==========================

Существует два основных подхода к получению ресурса GPIO из ACPI::

  desc = gpiod_get(dev, connection_id, flags);
  desc = gpiod_get_index(dev, connection_id, index, flags);

Здесь можно рассмотреть два различных случая, т. е. когда указан
идентификатор соединения (connection ID) и когда нет.

Случай 1::

  desc = gpiod_get(dev, "non-null-connection-id", flags);
  desc = gpiod_get_index(dev, "non-null-connection-id", index, flags);

Случай 1 предполагает, что соответствующее описание ACPI-устройства должно
иметь определённые свойства устройства, и в противном случае не позволит
получить какие-либо ресурсы GPIO.

Случай 2::

  desc = gpiod_get(dev, NULL, flags);
  desc = gpiod_get_index(dev, NULL, index, flags);

Случай 2 явно указывает ядру GPIO искать ресурсы в _CRS.

Имейте в виду, что gpiod_get_index() в случаях 1 и 2, при условии что
предоставлены две версии описания ACPI-устройства и в драйвере отсутствует
сопоставление, будет возвращать разные ресурсы. Именно поэтому конкретный
драйвер должен обрабатывать их аккуратно, как объяснено в предыдущей главе.
