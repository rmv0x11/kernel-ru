Аннотация
=========
Платформы, содержащие крышку, сообщают её состояние (открыта/закрыта) OSPM
с помощью устройства крышки, управляемого control method. Чтобы реализовать это,
таблицы AML выполняют Notify(lid_device, 0x80) для уведомления OSPM каждый раз,
когда состояние крышки изменяется. Control method _LID для устройства крышки
должен быть реализован так, чтобы сообщать «текущее» состояние крышки как
«открыта» или «закрыта».

На большинстве платформ как метод _LID, так и уведомления о крышке являются
надёжными. Однако существуют исключения. Чтобы работать с такими нештатными
платформами с ошибками, следует учитывать особые ограничения и исключения.
Этот документ описывает ограничения и исключения драйвера устройства крышки
ACPI в Linux.


Ограничения возвращаемого значения control method _LID
======================================================

Control method _LID описан как возвращающий «текущее» состояние крышки.
Однако слово «текущее» имеет неоднозначность: некоторые таблицы AML с ошибками
возвращают состояние крышки на момент последнего уведомления о крышке вместо
возврата состояния крышки на момент последнего вычисления _LID. Различия не
будет, когда control method _LID вычисляется в процессе работы (runtime),
проблема заключается в его начальном возвращаемом значении. Когда таблицы AML
реализуют этот control method с кэшированным значением, начальное возвращаемое
значение, скорее всего, ненадёжно. Существуют платформы, всегда возвращающие
«закрыта» в качестве начального состояния крышки.

Ограничения уведомлений об изменении состояния крышки
=====================================================

Существуют таблицы AML с ошибками, которые никогда не уведомляют, когда
состояние устройства крышки изменяется на «открыта». Таким образом, уведомление
«открыта» не гарантируется. Но гарантируется, что таблицы AML всегда уведомляют
«закрыта», когда состояние крышки изменяется на «закрыта». Уведомление «закрыта»
обычно используется в Windows для запуска некоторых операций энергосбережения
системы. Поскольку оно полностью протестировано, оно является надёжным во всех
таблицах AML.

Исключения для пользователей драйвера устройства крышки ACPI из userspace
=========================================================================

Драйвер кнопки ACPI экспортирует состояние крышки в userspace через
следующий файл::

  /proc/acpi/button/lid/LID0/state

Этот файл фактически вызывает control method _LID, описанный выше. И, учитывая
приведённое выше объяснение, он недостаточно надёжен на некоторых платформах.
Поэтому программам в userspace не рекомендуется полагаться исключительно на
этот файл для определения фактического состояния крышки.

Драйвер кнопки ACPI генерирует следующее входное событие в userspace:
  * SW_LID

Драйвер устройства крышки ACPI реализован так, чтобы пытаться доставлять
инициированные платформой события в userspace. Однако, учитывая тот факт, что
прошивка с ошибками не может гарантировать парность событий «открыта»/«закрыта»,
драйвер кнопки ACPI использует следующие 3 режима, чтобы не вызывать проблем.

Если userspace не подготовлен к тому, чтобы игнорировать ненадёжные события
«открыта» и ненадёжное уведомление о начальном состоянии, пользователи Linux
могут использовать следующие параметры ядра для обработки возможных проблем:

A. button.lid_init_state=method:
   Когда указан этот параметр, драйвер кнопки ACPI сообщает начальное состояние
   крышки, используя возвращаемое значение control method _LID, а парность
   событий «открыта»/«закрыта» полностью зависит от реализации прошивки.

   Этот параметр можно использовать для исправления некоторых платформ, где
   возвращаемое значение control method _LID является надёжным, но уведомление
   о начальном состоянии крышки отсутствует.

   Этот параметр является поведением по умолчанию в течение периода, пока
   userspace не готов обрабатывать таблицы AML с ошибками.

B. button.lid_init_state=open:
   Когда указан этот параметр, драйвер кнопки ACPI всегда сообщает начальное
   состояние крышки как «открыта», а парность событий «открыта»/«закрыта»
   полностью зависит от реализации прошивки.

   Это может исправить некоторые платформы, где возвращаемое значение control
   method _LID ненадёжно, а уведомление о начальном состоянии крышки
   отсутствует.

Если userspace подготовлен к тому, чтобы игнорировать ненадёжные события
«открыта» и ненадёжное уведомление о начальном состоянии, пользователям Linux
следует всегда использовать следующий параметр ядра:

C. button.lid_init_state=ignore:
   Когда указан этот параметр, драйвер кнопки ACPI никогда не сообщает начальное
   состояние крышки, и реализован компенсационный механизм, гарантирующий, что
   надёжные уведомления «закрыта» всегда могут быть доставлены в userspace путём
   неизменного объединения входных событий «закрыта» в пару с дополняющими
   входными событиями «открыта». Но всё равно нет гарантии, что уведомления
   «открыта» могут быть доставлены в userspace, когда крышка действительно
   открывается, учитывая, что некоторые таблицы AML не отправляют уведомления
   «открыта» надёжно.

   В этом режиме, если всё корректно реализовано прошивкой платформы, старые
   программы userspace по-прежнему должны работать. В противном случае требуются
   новые программы userspace для работы с драйвером кнопки ACPI. Этот параметр
   будет поведением по умолчанию после того, как userspace будет готов
   обрабатывать таблицы AML с ошибками.
