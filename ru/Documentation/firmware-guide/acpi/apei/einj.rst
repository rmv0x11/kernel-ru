===========================
Инъекция ошибок APEI (EINJ)
===========================

EINJ предоставляет аппаратный механизм инъекции ошибок. Он очень полезен
для отладки и тестирования APEI, а также возможностей RAS в целом.

Сначала вам нужно проверить, поддерживает ли ваш BIOS EINJ. Для этого посмотрите
ранние загрузочные сообщения, подобные этому::

  ACPI: EINJ 0x000000007370A000 000150 (v01 INTEL           00000001 INTL 00000001)

которое показывает, что BIOS предоставляет таблицу EINJ — это и есть
механизм, посредством которого выполняется инъекция.

Также можно поискать в /sys/firmware/acpi/tables файл с именем «EINJ»,
который является другим представлением того же самого.

Отсутствие указанного выше не обязательно означает, что EINJ не
поддерживается: прежде чем сдаться, зайдите в настройки BIOS и проверьте, есть ли
у BIOS опция включения инъекции ошибок. Поищите что-то под названием WHEA
или похожее. Часто для этого предварительно нужно включить опцию поддержки
ACPI5, чтобы увидеть функциональность APEI, EINJ и т.п., поддерживаемую и
предоставляемую через меню BIOS.

Чтобы использовать EINJ, убедитесь, что следующие опции включены в вашей
конфигурации ядра::

  CONFIG_DEBUG_FS
  CONFIG_ACPI_APEI
  CONFIG_ACPI_APEI_EINJ

...и для (необязательного) включения инъекции ошибок протокола CXL задайте::

  CONFIG_ACPI_APEI_EINJ_CXL

Пользовательский интерфейс EINJ находится в <точка монтирования debugfs>/apei/einj.

К нему относятся следующие файлы:

- available_error_type

  Этот файл показывает, какие типы ошибок поддерживаются:

  ================  ===================================
  Error Type Value	Error Description
  ================  ===================================
  0x00000001        Processor Correctable
  0x00000002        Processor Uncorrectable non-fatal
  0x00000004        Processor Uncorrectable fatal
  0x00000008        Memory Correctable
  0x00000010        Memory Uncorrectable non-fatal
  0x00000020        Memory Uncorrectable fatal
  0x00000040        PCI Express Correctable
  0x00000080        PCI Express Uncorrectable non-fatal
  0x00000100        PCI Express Uncorrectable fatal
  0x00000200        Platform Correctable
  0x00000400        Platform Uncorrectable non-fatal
  0x00000800        Platform Uncorrectable fatal
  V2_0x00000001     EINJV2 Processor Error
  V2_0x00000002     EINJV2 Memory Error
  V2_0x00000004     EINJV2 PCI Express Error
  ================  ===================================

  Формат содержимого файла такой, как указано выше, за исключением того, что
  присутствуют только доступные типы ошибок.

- error_type

  Задаёт значение типа ошибки, которая будет инъецирована. Возможные типы
  ошибок определены в файле available_error_type выше.

- error_inject

  Запишите в этот файл любое целое число, чтобы запустить инъекцию ошибки.
  Убедитесь, что вы задали все необходимые параметры ошибки, то есть эта
  запись должна быть последним шагом при инъекции ошибок.

- flags

  Присутствует для версий ядра 3.13 и выше. Используется для указания того,
  какие из param{1..4} действительны и должны использоваться прошивкой во время
  инъекции. Значение представляет собой битовую маску, как указано в спецификации
  ACPI5.0 для структуры данных SET_ERROR_TYPE_WITH_ADDRESS:

    Бит 0
      Поле Processor APIC действительно (см. param3 ниже).
    Бит 1
      Адрес памяти и маска действительны (param1 и param2).
    Бит 2
      PCIe (seg,bus,dev,fn) действительны (см. param4 ниже).
    Бит 3
      Структура расширения EINJv2 действительна

  Если задано нулевое значение, имитируется устаревшее поведение, при котором
  тип инъекции задаёт лишь один установленный бит, а param1 мультиплексируется.

- param1

  Этот файл используется для задания значения первого параметра ошибки. Его
  эффект зависит от типа ошибки, заданного в error_type. Например, если тип
  ошибки относится к памяти, то param1 должен быть действительным физическим
  адресом памяти. [Если только не установлен «flag» — см. выше]

- param2

  Используется так же, как param1 выше. Например, если тип ошибки относится к
  памяти, то param2 должен быть маской физического адреса памяти.
  Linux требует гранулярность не крупнее страницы, скажем, 0xfffffffffffff000.

- param3

  Используется, когда в «flags» установлен бит 0x1, для задания идентификатора APIC

- param4
  Используется, когда в «flags» установлен бит 0x4, для задания целевого устройства PCIe

- notrigger

  Механизм инъекции ошибок — это двухшаговый процесс. Сначала инъецируется
  ошибка, затем выполняются некоторые действия для её срабатывания. Установка
  «notrigger» в 1 пропускает фазу срабатывания, что *может* позволить
  пользователю вызвать ошибку в каком-то другом контексте простым обращением к
  CPU, ячейке памяти или устройству, являющемуся целью инъекции ошибки. Будет ли
  это действительно работать, зависит от того, какие операции BIOS на самом деле
  включает в фазу срабатывания.

- component_id0 .. component_idN, component_syndrome0 .. component_syndromeN

  Эти файлы используются для задания поля «Component Array»
  структуры расширения EINJv2. Каждый из них содержит 128-битное
  шестнадцатеричное значение. Запись просто символа новой строки в любой из этих
  файлов задаёт недействительное (состоящее из всех единиц) значение.

Типы ошибок CXL поддерживаются начиная с ACPI 6.5 (при наличии порта CXL).
Пользовательский интерфейс EINJ для типов ошибок CXL находится в
<точка монтирования debugfs>/cxl. К нему относятся следующие файлы:

- einj_types:

  Предоставляет ту же функциональность, что и available_error_types выше, но
  для типов ошибок CXL

- $dport_dev/einj_inject:

  Инъецирует тип ошибки CXL в порт CXL, представленный $dport_dev,
  где $dport_dev — это имя порта CXL (обычно имя PCIe-устройства).
  Инъекции ошибок, нацеленные на порт CXL 2.0+, могут использовать устаревший
  интерфейс под <точка монтирования debugfs>/apei/einj, тогда как инъекции в
  порт CXL 1.1/1.0 должны использовать этот файл.


Версии BIOS, основанные на спецификации ACPI 4.0, имеют ограниченные
возможности в управлении тем, куда инъецируются ошибки. Ваш BIOS может
поддерживать расширение (включаемое параметром модуля param_extension=1 или
параметром командной строки загрузки einj.param_extension=1). Это позволяет
задавать адрес и маску для инъекций в память с помощью файлов param1 и param2 в
apei/einj.

Версии BIOS, основанные на спецификации ACPI 5.0, имеют больший контроль над
целью инъекции. Для ошибок, связанных с процессором (тип 0x1, 0x2
и 0x4), можно установить flags в 0x3 (param3 для бита 0, а также param1 и
param2 для бита 1), чтобы к инъецируемой сигнатуре ошибки добавлялось больше
информации. Фактически передаваемые данные таковы::

	memory_address = param1;
	memory_address_range = param2;
	apicid = param3;
	pcie_sbdf = param4;

Для ошибок памяти (тип 0x8, 0x10 и 0x20) адрес задаётся с помощью
param1 с маской в param2 (0x0 эквивалентно всем единицам). Для ошибок PCI
express (тип 0x40, 0x80 и 0x100) сегмент, шина, устройство и
функция задаются с помощью param1::

         31     24 23    16 15    11 10      8  7        0
	+-------------------------------------------------+
	| segment |   bus  | device | function | reserved |
	+-------------------------------------------------+

В любом случае, вы поняли идею; если есть сомнения, просто загляните в код в
drivers/acpi/apei/einj.c.

BIOS с ACPI 5.0 может также позволять инъецировать ошибки, специфичные для
производителя. В этом случае файл с именем vendor будет содержать
идентифицирующую информацию от BIOS, которая, как мы надеемся, позволит
приложению, желающему использовать расширение производителя, понять, что оно
выполняется на BIOS, который его поддерживает. У всех расширений производителя
установлен бит 0x80000000 в error_type. Файл vendor_flags управляет
интерпретацией param1 и param2 (1 = PROCESSOR, 2 = MEMORY, 4 = PCI). За
подробностями обращайтесь к документации производителя вашего BIOS (и ожидайте
изменений в этом API, если творчество производителей в использовании этой
возможности выйдет за пределы наших ожиданий).


Пример инъекции ошибки::

  # cd /sys/kernel/debug/apei/einj
  # cat available_error_type		# See which errors can be injected
  0x00000002	Processor Uncorrectable non-fatal
  0x00000008	Memory Correctable
  0x00000010	Memory Uncorrectable non-fatal
  # echo 0x12345000 > param1		# Set memory address for injection
  # echo 0xfffffffffffff000 > param2		# Mask - anywhere in this page
  # echo 0x8 > error_type			# Choose correctable memory error
  # echo 1 > error_inject			# Inject now

Пример инъекции ошибки EINJv2::

  # cd /sys/kernel/debug/apei/einj
  # cat available_error_type			# See which errors can be injected
  0x00000002	Processor Uncorrectable non-fatal
  0x00000008	Memory Correctable
  0x00000010	Memory Uncorrectable non-fatal
  V2_0x00000001	EINJV2 Processor Error
  V2_0x00000002	EINJV2 Memory Error

  # echo 0x12345000 > param1			# Set memory address for injection
  # echo 0xfffffffffffff000 > param2		# Range - anywhere in this page
  # echo 0x1 > component_id0			# First device ID
  # echo 0x4 > component_syndrome0		# First error syndrome
  # echo 0x2 > component_id1			# Second device ID
  # echo 0x4 > component_syndrome1		# Second error syndrome
  # echo '' > component_id2			# Mark id2 invalid to terminate list
  # echo V2_0x2 > error_type			# Choose EINJv2 memory error
  # echo 0xa > flags				# set flags to indicate EINJv2
  # echo 1 > error_inject			# Inject now

Пример инъекции ошибки CXL при $dport_dev=0000:e0:01.1::

    # cd /sys/kernel/debug/cxl/
    # ls
    0000:e0:01.1 0000:0c:00.0
    # cat einj_types                # See which errors can be injected
	0x00008000  CXL.mem Protocol Correctable
	0x00010000  CXL.mem Protocol Uncorrectable non-fatal
	0x00020000  CXL.mem Protocol Uncorrectable fatal
    # cd 0000:e0:01.1               # Navigate to dport to inject into
    # echo 0x8000 > einj_inject     # Inject error

Особые замечания об инъекции в анклавы SGX:

Для включения инъекции SGX может быть отдельная опция в настройках BIOS.

Процесс инъекции состоит в установке некоторого специального триггера контроллера
памяти, который инъецирует ошибку при следующей записи по целевому
адресу. Однако аппаратура препятствует доступу любого программного обеспечения
вне анклава SGX к страницам анклава (даже в режиме BIOS SMM).

Может использоваться следующая последовательность:
  1) Определить физический адрес страницы анклава
  2) Использовать режим «notrigger=1» для инъекции (это настроит
     адрес инъекции, но фактически не выполнит инъекцию)
  3) Войти в анклав
  4) Сохранить данные по виртуальному адресу, соответствующему физическому адресу из шага 1
  5) Выполнить CLFLUSH для этого виртуального адреса
  6) Сделать задержку (спин) на 250 мс
  7) Прочитать по виртуальному адресу. Это вызовет срабатывание ошибки

За дополнительной информацией об EINJ обращайтесь к спецификации ACPI
версии 4.0, раздел 17.5, и ACPI 5.0, раздел 18.6.
