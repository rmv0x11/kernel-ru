Аннотация
=========
Подсистема ACPI ядра Linux преобразует объекты пространства имён ACPI в дерево
устройств Linux в каталоге /sys/devices/LNXSYSTM:00 и обновляет его при
получении событий уведомлений о горячем подключении (hotplug) ACPI.  Для каждого
объекта устройства в этой иерархии существует соответствующая символическая
ссылка в /sys/bus/acpi/devices.

Этот документ иллюстрирует структуру дерева устройств ACPI.

Блоки определений ACPI
======================

Прошивка (firmware) ACPI настраивает RSDP (Root System Description Pointer) в
адресном пространстве системной памяти, указывающий на XSDT (Extended System
Description Table).  XSDT всегда указывает на FADT (Fixed ACPI
Description Table) своей первой записью; данные внутри FADT
включают различные записи фиксированной длины, которые описывают фиксированные
возможности ACPI данного оборудования.  FADT содержит указатель на DSDT
(Differentiated System Description Table).  XSDT также содержит
записи, указывающие на, возможно, несколько SSDT (Secondary System
Description Table).

Данные DSDT и SSDT организованы в структуры данных, называемые блоками определений,
которые содержат определения различных объектов, включая управляющие
методы ACPI, закодированные в AML (ACPI Machine Language).  Блок данных
DSDT вместе с содержимым SSDT представляет иерархическую
структуру данных, называемую пространством имён ACPI, чья топология отражает
структуру нижележащей аппаратной платформы.

Описанные выше отношения между системными таблицами определений ACPI
проиллюстрированы на следующей диаграмме::

   +---------+    +-------+    +--------+    +------------------------+
   |  RSDP   | +->| XSDT  | +->|  FADT  |    |  +-------------------+ |
   +---------+ |  +-------+ |  +--------+  +-|->|       DSDT        | |
   | Pointer | |  | Entry |-+  | ...... |  | |  +-------------------+ |
   +---------+ |  +-------+    | X_DSDT |--+ |  | Definition Blocks | |
   | Pointer |-+  | ..... |    | ...... |    |  +-------------------+ |
   +---------+    +-------+    +--------+    |  +-------------------+ |
                  | Entry |------------------|->|       SSDT        | |
                  +- - - -+                  |  +-------------------| |
                  | Entry | - - - - - - - -+ |  | Definition Blocks | |
                  +- - - -+                | |  +-------------------+ |
                                           | |  +- - - - - - - - - -+ |
                                           +-|->|       SSDT        | |
                                             |  +-------------------+ |
                                             |  | Definition Blocks | |
                                             |  +- - - - - - - - - -+ |
                                             +------------------------+
                                                          |
                                             OSPM Loading |
                                                         \|/
                                                   +----------------+
                                                   | ACPI Namespace |
                                                   +----------------+

                  Figure 1. ACPI Definition Blocks

.. note:: RSDP также может содержать указатель на RSDT (Root System
   Description Table).  Платформы предоставляют RSDT для обеспечения
   совместимости с операционными системами ACPI 1.0.  Предполагается, что ОС
   будет использовать XSDT, если он присутствует.


Пример пространства имён ACPI
=============================

Все блоки определений загружаются в единое пространство имён.  Пространство имён
представляет собой иерархию объектов, идентифицируемых по именам и путям.
Следующие соглашения об именовании применяются к именам объектов в пространстве
имён ACPI:

   1. Все имена имеют длину 32 бита.
   2. Первый байт имени должен быть одним из 'A' - 'Z', '_'.
   3. Каждый из оставшихся байтов имени должен быть одним из 'A' - 'Z', '0'
      - '9', '_'.
   4. Имена, начинающиеся с '_', зарезервированы спецификацией ACPI.
   5. Символ '\' представляет корень пространства имён (т.е. имена
      с префиксом '\' являются относительными к корню пространства имён).
   6. Символ '^' представляет родителя текущего узла пространства имён
      (т.е. имена с префиксом '^' являются относительными к родителю
      текущего узла пространства имён).

Объекты устройств ACPI в Linux
==============================

Базовая подсистема ACPI ядра Linux создаёт объекты struct acpi_device
для объектов пространства имён ACPI, представляющих устройства, ресурсы питания,
процессоры, тепловые зоны.  Эти объекты экспортируются в пространство пользователя
(userspace) через sysfs в виде каталогов в поддереве /sys/devices/LNXSYSTM:00.  Формат
их имён — <bus_id:instance>, где 'bus_id' ссылается на
представление данного объекта в пространстве имён ACPI, а 'instance' используется
для различения разных объектов с одним и тем же 'bus_id' (это
двузначное десятичное представление беззнакового целого числа).

Значение 'bus_id' зависит от типа объекта, имени которого оно является
частью, как перечислено в таблице ниже::

                +---+-----------------+-------+----------+
                |   | Object/Feature  | Table | bus_id   |
                +---+-----------------+-------+----------+
                | N | Root            | xSDT  | LNXSYSTM |
                +---+-----------------+-------+----------+
                | N | Device          | xSDT  | _HID     |
                +---+-----------------+-------+----------+
                | N | Processor       | xSDT  | LNXCPU   |
                +---+-----------------+-------+----------+
                | N | ThermalZone     | xSDT  | LNXTHERM |
                +---+-----------------+-------+----------+
                | N | PowerResource   | xSDT  | LNXPOWER |
                +---+-----------------+-------+----------+
                | N | Other Devices   | xSDT  | device   |
                +---+-----------------+-------+----------+
                | F | PWR_BUTTON      | FADT  | LNXPWRBN |
                +---+-----------------+-------+----------+
                | F | SLP_BUTTON      | FADT  | LNXSLPBN |
                +---+-----------------+-------+----------+
                | M | Video Extension | xSDT  | LNXVIDEO |
                +---+-----------------+-------+----------+
                | M | ATA Controller  | xSDT  | LNXIOBAY |
                +---+-----------------+-------+----------+
                | M | Docking Station | xSDT  | LNXDOCK  |
                +---+-----------------+-------+----------+

                 Table 1. ACPI Namespace Objects Mapping

Следующие правила применяются при создании объектов struct acpi_device на
основе содержимого системных таблиц описания ACPI (как
обозначено буквой в первом столбце и обозначением во
втором столбце таблицы выше):

   N:
      Источником объекта является узел пространства имён ACPI (как обозначено
      типом именованного объекта во втором столбце).  В этом случае каталог
      объекта в sysfs будет содержать атрибут 'path', значением которого является
      полный путь к узлу от корня пространства имён.
   F:
      Объект struct acpi_device создаётся для фиксированной аппаратной
      возможности (как обозначено именем флага фиксированной возможности во втором
      столбце), поэтому его каталог в sysfs не будет содержать атрибута
      'path'.
   M:
      Объект struct acpi_device создаётся для узла пространства имён ACPI
      с определёнными управляющими методами (как обозначено типом
      определённого ACPI устройства во втором столбце).  Атрибут 'path', содержащий
      его путь в пространстве имён, будет присутствовать в его каталоге sysfs.  Например,
      если для узла пространства имён ACPI присутствует метод _BCL, для него будет
      создан объект struct acpi_device с 'bus_id', равным LNXVIDEO.

Третий столбец приведённой выше таблицы указывает, какие системные таблицы
описания ACPI содержат информацию, используемую для создания
объектов struct acpi_device, представленных данной строкой (xSDT означает DSDT
или SSDT).

Четвёртый столбец приведённой выше таблицы указывает правило генерации 'bus_id'
объекта struct acpi_device:

   _HID:
      _HID в последнем столбце таблицы означает, что bus_id объекта
      выводится из идентификационных объектов _HID/_CID, присутствующих под
      соответствующим узлом пространства имён ACPI. Каталог sysfs объекта
      будет тогда содержать атрибуты 'hid' и 'modalias', которые можно
      использовать для получения _HID и _CID этого объекта.
   LNXxxxxx:
      Атрибут 'modalias' также присутствует у объектов struct acpi_device,
      имеющих bus_id вида "LNXxxxxx" (псевдоустройства), в
      каковых случаях он содержит сам строку bus_id.
   device:
      'device' в последнем столбце таблицы указывает, что bus_id объекта
      не может быть определён из _HID/_CID соответствующего
      узла пространства имён ACPI, хотя этот объект представляет устройство (например,
      это может быть устройство PCI с определённым _ADR и без _HID
      или _CID).  В этом случае строка 'device' будет использована в качестве
      bus_id объекта.


Связующий слой физических устройств ACPI в Linux
================================================

Объекты устройств ACPI (т.е. struct acpi_device) могут быть связаны с другими
объектами в иерархии устройств Linux, которые представляют «физические» устройства
(например, устройства на шине PCI).  Если это происходит, это означает, что
объект устройства ACPI является «компаньоном» устройства, иным образом
представленного другим способом, и используется (1) для предоставления конфигурационной
информации об этом устройстве, которую невозможно получить другими средствами,
и (2) для выполнения специфических действий с устройством с помощью его управляющих
методов ACPI.  Один объект устройства ACPI может быть связан таким образом с
несколькими «физическими» устройствами.

Если объект устройства ACPI связан с «физическим» устройством, его каталог sysfs
содержит символическую ссылку "physical_node" на каталог sysfs
целевого объекта устройства.  В свою очередь, каталог sysfs целевого устройства
будет тогда содержать символическую ссылку "firmware_node" на
каталог sysfs объекта-компаньона устройства ACPI.
Механизм связывания опирается на идентификацию устройств, предоставляемую
пространством имён ACPI.  Например, если есть объект пространства имён ACPI,
представляющий устройство PCI (т.е. объект устройства под объектом пространства имён ACPI,
представляющим мост PCI), чей _ADR возвращает 0x00020000, а номер
шины родительского моста PCI равен 0, каталог sysfs,
представляющий объект struct acpi_device, созданный для этого объекта
пространства имён ACPI, будет содержать символическую ссылку 'physical_node' на каталог
sysfs /sys/devices/pci0000:00/0000:00:02:0/
соответствующего устройства PCI.

Механизм связывания обычно специфичен для конкретной шины.  Ядро его
реализации находится в файле drivers/acpi/glue.c, но существуют
дополнительные части, зависящие от рассматриваемых типов шин, расположенные
в других местах.  Например, специфичная для PCI часть находится в
drivers/pci/pci-acpi.c.

   1. 'object' — это имя каталога объекта в sysfs.

      объекта пространства имён ACPI, возвращаемый атрибутом sysfs 'path'
      объекта.

      объекта (как описано ранее в этом документе).
