==============================
Видеорасширения ACPI
==============================

Этот драйвер реализует ACPI Extensions For Display Adapters для
интегрированных графических устройств на материнской плате, как описано в
спецификации ACPI 2.0 Specification, Appendix B, позволяя выполнять некоторые
базовые операции управления, такие как задание видеоустройства POST, получение
информации EDID или настройка видеовыхода и т.д. Обратите внимание, что это
только эталонная (ref.) реализация. Она может работать или не работать с вашим
интегрированным видеоустройством.

Видеодрайвер ACPI выполняет 3 задачи, связанные с управлением подсветкой.

Экспорт интерфейса sysfs для управления уровнем подсветки из пространства пользователя
======================================================================================

Если в таблице ACPI есть видеоустройство, и в командной строке ядра отсутствует
параметр acpi_backlight=vendor, драйвер зарегистрирует устройство подсветки и
установит для него необходимую структуру операций подсветки для управления через
интерфейс sysfs. Для каждого зарегистрированного class device будет создан
каталог с именем acpi_videoX в /sys/class/backlight.

Стандартное определение интерфейса sysfs для подсветки находится здесь:
Documentation/ABI/stable/sysfs-class-backlight.

А видеодрайвер ACPI делает следующее:

actual_brightness:
  при чтении вычисляется управляющий метод _BQC для
  получения уровня яркости, который, по мнению прошивки, установлен;
bl_power:
  не реализовано, вместо этого будет установлена текущая яркость;
brightness:
  при записи запускается управляющий метод _BCM для установки запрошенного уровня яркости;
max_brightness:
  выводится из пакета _BCL (см. ниже);
type:
  firmware

Обратите внимание, что драйвер подсветки видео ACPI всегда использует индекс для
brightness, actual_brightness и max_brightness. Так, если у нас есть
следующий пакет _BCL::

	Method (_BCL, 0, NotSerialized)
	{
		Return (Package (0x0C)
		{
			0x64,
			0x32,
			0x0A,
			0x14,
			0x1E,
			0x28,
			0x32,
			0x3C,
			0x46,
			0x50,
			0x5A,
			0x64
		})
	}

Первые два уровня предназначены для случаев, когда ноутбук работает от сети или
от батареи, и в настоящее время в Linux не используются. Остальные 10 уровней —
это поддерживаемые уровни, из которых мы можем выбирать. Применимые значения
индекса — от 0 (что соответствует значению яркости 0x0A) до 9 (что соответствует
значению яркости 0x64) включительно. Каждое из этих значений индекса
рассматривается как индикатор «уровня яркости». Таким образом, с точки зрения
пространства пользователя диапазон доступных уровней яркости — от 0 до 9
(max_brightness) включительно.

Уведомление пространства пользователя о событии горячей клавиши
===============================================================

Как правило, для уведомления о событии горячей клавиши есть два случая:

i) На некоторых ноутбуках при нажатии пользователем горячей клавиши генерируется
   скан-код, который отправляется в пространство пользователя через устройство
   ввода, создаваемое драйвером клавиатуры, как событие ввода типа клавиши; при
   правильном переназначении в пространстве пользователя появится следующий
   код клавиши::

	EV_KEY, KEY_BRIGHTNESSUP
	EV_KEY, KEY_BRIGHTNESSDOWN
	etc.

В этом случае видеодрайверу ACPI ничего делать не нужно (фактически
он даже не знает, что это произошло).

ii) На некоторых ноутбуках нажатие горячей клавиши не генерирует
    скан-код, вместо этого прошивка уведомляет узел ACPI видеоустройства
    о событии. Значение события определено в спецификации ACPI. Видеодрайвер
    ACPI сгенерирует событие ввода типа клавиши в соответствии с полученным
    значением уведомления и отправит событие в пространство пользователя через
    созданное им устройство ввода:

	=======  ==================
	событие  код клавиши
	=======  ==================
	0x86     KEY_BRIGHTNESSUP
	0x87     KEY_BRIGHTNESSDOWN
	etc.
	=======  ==================

поэтому теперь это приведёт к тому же эффекту, что и в случае i).

После того как инструмент пространства пользователя получает это событие, он может
изменить уровень подсветки через интерфейс sysfs.

Изменение уровня подсветки в ядре
=================================

Это работает для машин, охватываемых случаем ii) в Разделе 2. После того как
драйвер получил уведомление, он установит уровень подсветки соответствующим
образом. Это не влияет на отправку события в пространство пользователя — они
всегда отправляются в пространство пользователя независимо от того, управляет ли
видеомодуль уровнем подсветки напрямую. Это поведение можно контролировать через
параметр модуля brightness_switch_enabled, как описано в
admin-guide/kernel-parameters.rst. Рекомендуется отключить это поведение, как
только запускается окружение GUI и хочет получить полный контроль над уровнем
подсветки.
