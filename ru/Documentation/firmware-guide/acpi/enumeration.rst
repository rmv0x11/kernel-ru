===================================
Перечисление устройств на базе ACPI
===================================

В ACPI 5 был введён набор новых ресурсов (UartTSerialBus, I2cSerialBus,
SpiSerialBus, GpioIo и GpioInt), которые можно использовать при перечислении
ведомых (slave) устройств за контроллерами последовательных шин.

Кроме того, мы начинаем сталкиваться с периферийными устройствами,
интегрированными в SoC/чипсет, которые появляются только в пространстве имён
ACPI. Обычно это устройства, доступ к которым осуществляется через регистры,
отображённые в память (memory-mapped).

Чтобы поддержать это и при этом максимально переиспользовать существующие
драйверы, мы решили сделать следующее:

  - Устройства, не имеющие ресурса соединителя шины (bus connector), представляются
    как платформенные устройства (platform devices).

  - Устройства за реальными шинами, где присутствует ресурс соединителя,
    представляются как struct spi_device или struct i2c_client. Обратите
    внимание, что стандартные UART не являются шинами, поэтому struct uart_device
    не существует, хотя некоторые из них могут быть представлены через struct serdev_device.

Поскольку и ACPI, и Device Tree представляют дерево устройств (и их
ресурсов), эта реализация максимально следует подходу Device Tree.

Реализация ACPI перечисляет устройства за шинами (platform, SPI,
I2C, а в некоторых случаях UART), создаёт физические устройства и привязывает их
к их ACPI-дескриптору (handle) в пространстве имён ACPI.

Это означает, что когда ACPI_HANDLE(dev) возвращает не NULL, устройство было
перечислено из пространства имён ACPI. Этот дескриптор можно использовать для извлечения
прочей специфичной для устройства конфигурации. Ниже приведён пример этого.

Поддержка платформенной шины
============================

Поскольку для представления устройств, не подключённых ни к какой физической шине,
мы используем платформенные устройства, нам нужно лишь реализовать платформенный
драйвер для устройства и добавить поддерживаемые ACPI ID. Если этот же IP-блок
используется на какой-либо другой, не-ACPI платформе, драйвер может заработать
сразу же либо потребует незначительных изменений.

Добавление поддержки ACPI в существующий драйвер должно быть довольно
несложным. Вот простейший пример::

	static const struct acpi_device_id mydrv_acpi_match[] = {
		/* ACPI IDs here */
		{ }
	};
	MODULE_DEVICE_TABLE(acpi, mydrv_acpi_match);

	static struct platform_driver my_driver = {
		...
		.driver = {
			.acpi_match_table = mydrv_acpi_match,
		},
	};

Если драйверу нужно выполнить более сложную инициализацию, например получить и
настроить GPIO, он может получить свой ACPI-дескриптор и извлечь эту информацию
из таблиц ACPI.

Объекты устройств ACPI
======================

Вообще говоря, в системе, где ACPI используется как интерфейс между прошивкой
платформы и ОС, существуют две категории устройств: устройства, которые можно
обнаружить и перечислить нативно, через протокол, определённый для конкретной
шины, на которой они находятся (например, конфигурационное пространство в PCI),
без помощи прошивки платформы, и устройства, которые должны быть описаны прошивкой
платформы, чтобы их можно было обнаружить. При этом для любого устройства,
известного прошивке платформы, независимо от того, к какой категории оно относится,
в пространстве имён ACPI может присутствовать соответствующий объект устройства
ACPI, и в этом случае ядро Linux создаст на его основе объект struct acpi_device
для этого устройства.

Эти объекты struct acpi_device никогда не используются для привязки драйверов к
нативно обнаруживаемым устройствам, поскольку те представлены другими типами
объектов устройств (например, struct pci_dev для устройств PCI), к которым
привязываются драйверы устройств (соответствующий объект struct acpi_device при
этом используется как дополнительный источник информации о конфигурации данного
устройства). Более того, основной код перечисления устройств ACPI создаёт объекты
struct platform_device для большинства устройств, обнаруживаемых и перечисляемых с
помощью прошивки платформы, и к этим объектам платформенных устройств могут
привязываться платформенные драйверы по прямой аналогии со случаем нативно
перечисляемых устройств. Поэтому логически непоследовательно и, как следствие,
в общем случае недопустимо привязывать драйверы к объектам struct acpi_device,
включая драйверы для устройств, которые обнаруживаются с помощью прошивки платформы.

Исторически для некоторых устройств, перечисляемых с помощью прошивки платформы,
реализовывались ACPI-драйверы, которые привязывались напрямую к объектам
struct acpi_device, но это не рекомендуется для каких-либо новых драйверов. Как
объяснялось выше, для таких устройств, как правило, создаются объекты платформенных
устройств (за немногими исключениями, которые здесь несущественны), и поэтому для
их обработки следует использовать платформенные драйверы, даже если соответствующие
объекты устройств ACPI являются в этом случае единственным источником информации о
конфигурации устройства.

Для каждого устройства, имеющего соответствующий объект struct acpi_device,
указатель на него возвращается макросом ACPI_COMPANION(), поэтому таким способом
всегда можно получить доступ к информации о конфигурации устройства, хранящейся
в объекте устройства ACPI. Соответственно, struct acpi_device можно рассматривать
как часть интерфейса между ядром и пространством имён ACPI, тогда как объекты
устройств других типов (например, struct pci_dev или struct platform_device)
используются для взаимодействия с остальной частью системы.

Поддержка DMA
=============

Контроллеры DMA, перечисляемые через ACPI, должны быть зарегистрированы в системе,
чтобы предоставить обобщённый доступ к своим ресурсам. Например, драйвер, который
хотел бы быть доступным для ведомых устройств через обобщённый вызов API
dma_request_chan(), должен зарегистрировать себя в конце функции probe вот
так::

	err = devm_acpi_dma_controller_register(dev, xlate_func, dw);
	/* Handle the error if it's not a case of !CONFIG_ACPI */

и реализовать собственную функцию xlate, если это необходимо (обычно достаточно
acpi_dma_simple_xlate()), которая преобразует ресурс FixedDMA, предоставленный
struct acpi_dma_spec, в соответствующий канал DMA. Фрагмент кода для этого случая
мог бы выглядеть так::

	#ifdef CONFIG_ACPI
	struct filter_args {
		/* Provide necessary information for the filter_func */
		...
	};

	static bool filter_func(struct dma_chan *chan, void *param)
	{
		/* Choose the proper channel */
		...
	}

	static struct dma_chan *xlate_func(struct acpi_dma_spec *dma_spec,
			struct acpi_dma *adma)
	{
		dma_cap_mask_t cap;
		struct filter_args args;

		/* Prepare arguments for filter_func */
		...
		return dma_request_channel(cap, filter_func, &args);
	}
	#else
	static struct dma_chan *xlate_func(struct acpi_dma_spec *dma_spec,
			struct acpi_dma *adma)
	{
		return NULL;
	}
	#endif

dma_request_chan() будет вызывать xlate_func() для каждого зарегистрированного
контроллера DMA. В функции xlate должен быть выбран подходящий канал на основе
информации из struct acpi_dma_spec и свойств контроллера, предоставленных
struct acpi_dma.

Клиенты должны вызывать dma_request_chan() со строковым параметром, который
соответствует конкретному ресурсу FixedDMA. По умолчанию "tx" означает первый
элемент массива ресурсов FixedDMA, "rx" — второй элемент. В таблице ниже показана
компоновка::

	Device (I2C0)
	{
		...
		Method (_CRS, 0, NotSerialized)
		{
			Name (DBUF, ResourceTemplate ()
			{
				FixedDMA (0x0018, 0x0004, Width32bit, _Y48)
				FixedDMA (0x0019, 0x0005, Width32bit, )
			})
		...
		}
	}

Итак, FixedDMA с линией запроса 0x0018 в этом примере — это "tx", а следующий —
"rx".

В надёжных (robust) случаях клиенту, к сожалению, приходится вызывать
acpi_dma_request_slave_chan_by_index() напрямую и, следовательно, выбирать
конкретный ресурс FixedDMA по его индексу.

Именованные прерывания
======================

Драйверы, перечисляемые через ACPI, могут иметь в таблице ACPI имена для прерываний,
которые можно использовать для получения номера IRQ в драйвере.

Имя прерывания может быть указано в _DSD как 'interrupt-names'. Имена должны быть
перечислены в виде массива строк, которые будут сопоставлены ресурсу Interrupt()
в таблице ACPI в соответствии с его индексом.

В таблице ниже показан пример его использования::

    Device (DEV0) {
        ...
        Name (_CRS, ResourceTemplate() {
            ...
            Interrupt (ResourceConsumer, Level, ActiveHigh, Exclusive) {
                0x20,
                0x24
            }
        })

        Name (_DSD, Package () {
            ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
            Package () {
                Package () { "interrupt-names", Package () { "default", "alert" } },
            }
        ...
        })
    }

Имя прерывания 'default' будет соответствовать 0x20 в ресурсе Interrupt(),
а 'alert' — 0x24. Обратите внимание, что сопоставляется только ресурс Interrupt(),
а не GpioInt() или подобный.

Драйвер может вызвать функцию fwnode_irq_get_byname() с fwnode и именем
прерывания в качестве аргументов, чтобы получить соответствующий номер IRQ.

Поддержка последовательной шины SPI
===================================

Ведомые устройства за шиной SPI имеют присоединённый к ним ресурс SpiSerialBus.
Он автоматически извлекается ядром SPI, и ведомые устройства перечисляются, как
только драйвер шины вызывает spi_register_master().

Вот как может выглядеть пространство имён ACPI для ведомого устройства SPI::

	Device (EEP0)
	{
		Name (_ADR, 1)
		Name (_CID, Package () {
			"ATML0025",
			"AT25",
		})
		...
		Method (_CRS, 0, NotSerialized)
		{
			SPISerialBus(1, PolarityLow, FourWireMode, 8,
				ControllerInitiated, 1000000, ClockPolarityLow,
				ClockPhaseFirst, "\\_SB.PCI0.SPI1",)
		}
		...

Драйверам устройств SPI нужно лишь добавить ACPI ID аналогично тому, как это
делается для драйверов платформенных устройств. Ниже приведён пример, в котором мы
добавляем поддержку ACPI к драйверу SPI eeprom at25 (он предназначен для приведённого
выше фрагмента ACPI)::

	static const struct acpi_device_id at25_acpi_match[] = {
		{ "AT25", 0 },
		{ }
	};
	MODULE_DEVICE_TABLE(acpi, at25_acpi_match);

	static struct spi_driver at25_driver = {
		.driver = {
			...
			.acpi_match_table = at25_acpi_match,
		},
	};

Обратите внимание, что этому драйверу на самом деле нужно больше информации,
например размер страницы eeprom и т. д. Эта информация может быть передана через
метод _DSD вот так::

	Device (EEP0)
	{
		...
		Name (_DSD, Package ()
		{
			ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
			Package ()
			{
				Package () { "size", 1024 },
				Package () { "pagesize", 32 },
				Package () { "address-width", 16 },
			}
		})
	}

Тогда драйвер SPI at25 может получить эту конфигурацию, вызывая API свойств
устройства во время фазы ->probe() вот так::

	err = device_property_read_u32(dev, "size", &size);
	if (err)
		...error handling...

	err = device_property_read_u32(dev, "pagesize", &page_size);
	if (err)
		...error handling...

	err = device_property_read_u32(dev, "address-width", &addr_width);
	if (err)
		...error handling...

Поддержка последовательной шины I2C
===================================

Ведомым устройствам за контроллером шины I2C нужно лишь добавить ACPI ID, как и в
случае драйверов платформенных устройств и SPI. Ядро I2C автоматически перечисляет
любые ведомые устройства за устройством-контроллером, как только зарегистрирован
адаптер.

Ниже приведён пример того, как добавить поддержку ACPI к существующему драйверу
ввода mpu3050::

	static const struct acpi_device_id mpu3050_acpi_match[] = {
		{ "MPU3050", 0 },
		{ }
	};
	MODULE_DEVICE_TABLE(acpi, mpu3050_acpi_match);

	static struct i2c_driver mpu3050_i2c_driver = {
		.driver	= {
			.name	= "mpu3050",
			.pm	= &mpu3050_pm,
			.of_match_table = mpu3050_of_match,
			.acpi_match_table = mpu3050_acpi_match,
		},
		.probe		= mpu3050_probe,
		.remove		= mpu3050_remove,
		.id_table	= mpu3050_ids,
	};
	module_i2c_driver(mpu3050_i2c_driver);

Ссылка на устройство PWM
========================

Иногда устройство может быть потребителем канала PWM. Очевидно, ОС хотела бы знать,
какого именно. Чтобы предоставить это сопоставление, было введено специальное
свойство, а именно::

    Device (DEV)
    {
        Name (_DSD, Package ()
        {
            ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
            Package () {
                Package () { "compatible", Package () { "pwm-leds" } },
                Package () { "label", "alarm-led" },
                Package () { "pwms",
                    Package () {
                        "\\_SB.PCI0.PWM",  // <PWM device reference>
                        0,                 // <PWM index>
                        600000000,         // <PWM period>
                        0,                 // <PWM flags>
                    }
                }
            }
        })
        ...
    }

В приведённом выше примере драйвер светодиода на базе PWM ссылается на канал PWM 0
устройства \_SB.PCI0.PWM с начальной настройкой периода, равной 600 мс (обратите
внимание, что значение задано в наносекундах).

Поддержка GPIO
==============

В ACPI 5 было введено два новых ресурса для описания соединений GPIO: GpioIo
и GpioInt. Эти ресурсы можно использовать для передачи драйверу номеров GPIO,
используемых устройством. ACPI 5.1 расширил это с помощью _DSD (Device
Specific Data), что среди прочего позволило именовать GPIO.

Например::

	Device (DEV)
	{
		Method (_CRS, 0, NotSerialized)
		{
			Name (SBUF, ResourceTemplate()
			{
				// Used to power on/off the device
				GpioIo (Exclusive, PullNone, 0, 0, IoRestrictionOutputOnly,
					"\\_SB.PCI0.GPI0", 0, ResourceConsumer) { 85 }

				// Interrupt for the device
				GpioInt (Edge, ActiveHigh, ExclusiveAndWake, PullNone, 0,
					 "\\_SB.PCI0.GPI0", 0, ResourceConsumer) { 88 }
			}

			Return (SBUF)
		}

		// ACPI 5.1 _DSD used for naming the GPIOs
		Name (_DSD, Package ()
		{
			ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
			Package ()
			{
				Package () { "power-gpios", Package () { ^DEV, 0, 0, 0 } },
				Package () { "irq-gpios", Package () { ^DEV, 1, 0, 0 } },
			}
		})
		...
	}

Эти номера GPIO задаются относительно контроллера, а путь "\\_SB.PCI0.GPI0"
указывает путь к контроллеру. Чтобы использовать эти GPIO в Linux,
нам нужно транслировать их в соответствующие дескрипторы GPIO Linux.

Для этого существует стандартный API GPIO, и он описан в
Documentation/admin-guide/gpio/.

В приведённом выше примере мы можем получить соответствующие два дескриптора GPIO
с помощью такого кода::

	#include <linux/gpio/consumer.h>
	...

	struct gpio_desc *irq_desc, *power_desc;

	irq_desc = gpiod_get(dev, "irq");
	if (IS_ERR(irq_desc))
		/* handle error */

	power_desc = gpiod_get(dev, "power");
	if (IS_ERR(power_desc))
		/* handle error */

	/* Now we can use the GPIO descriptors */

Существуют также версии этих функций с префиксом devm_*, которые освобождают
дескрипторы, как только устройство освобождается.

Дополнительную информацию о привязке _DSD, связанной с GPIO, см. в
Documentation/firmware-guide/acpi/gpio-properties.rst.

Поддержка RS-485
================

ACPI _DSD (Device Specific Data) можно использовать для описания возможности
RS-485 у UART.

Например::

	Device (DEV)
	{
		...

		// ACPI 5.1 _DSD used for RS-485 capabilities
		Name (_DSD, Package ()
		{
			ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
			Package ()
			{
				Package () {"rs485-rts-active-low", Zero},
				Package () {"rs485-rx-active-high", Zero},
				Package () {"rs485-rx-during-tx", Zero},
			}
		})
		...

Устройства MFD
==============

Устройства MFD регистрируют своих потомков как платформенные устройства. Для
дочерних устройств должен существовать ACPI-дескриптор, который они могут
использовать для ссылки на относящиеся к ним части пространства имён ACPI. В
подсистеме MFD Linux мы предоставляем два способа:

  - Потомки разделяют ACPI-дескриптор родителя.
  - Ячейка MFD может указать ACPI id устройства.

В первом случае драйверам MFD не нужно ничего делать. У результирующего дочернего
платформенного устройства поле ACPI_COMPANION() будет установлено так, чтобы оно
указывало на родительское устройство.

Если в пространстве имён ACPI есть устройство, которое мы можем сопоставить, используя
ACPI id или ACPI adr, ячейку следует задать так::

	static struct mfd_cell_acpi_match my_subdevice_cell_acpi_match = {
		.pnpid = "XYZ0001",
		.adr = 0,
	};

	static struct mfd_cell my_subdevice_cell = {
		.name = "my_subdevice",
		/* set the resources relative to the parent */
		.acpi_match = &my_subdevice_cell_acpi_match,
	};

ACPI id "XYZ0001" затем используется для поиска устройства ACPI непосредственно под
устройством MFD, и если оно найдено, это устройство-компаньон ACPI привязывается к
результирующему дочернему платформенному устройству.

ID устройства-ссылки на пространство имён Device Tree
=====================================================

Протокол Device Tree использует идентификацию устройств на основе свойства
"compatible", значение которого представляет собой строку или массив строк,
распознаваемых как идентификаторы устройств драйверами и ядром драйверов. Множество
всех этих строк можно рассматривать как пространство имён идентификации устройств,
аналогичное пространству имён ID устройств ACPI/PNP. Следовательно, в принципе не
должно быть необходимости выделять новый (и, возможно, избыточный) ACPI/PNP ID
устройства для устройства, у которого уже есть идентификационная строка в
пространстве имён Device Tree (DT), особенно если этот ID нужен лишь для того,
чтобы указать, что данное устройство совместимо с другим, для которого,
предположительно, в ядре уже есть подходящий драйвер.

В ACPI объект идентификации устройства, называемый _CID (Compatible ID),
используется для перечисления ID устройств, с которыми совместимо данное, но эти ID
должны принадлежать одному из пространств имён, предписанных спецификацией ACPI (см.
раздел 6.1.2 ACPI 6.0 для подробностей), а пространство имён DT не является одним из
них. Более того, спецификация требует, чтобы для всех объектов ACPI, представляющих
устройства, присутствовал либо объект идентификации _HID, либо _ADR (раздел 6.1 ACPI
6.0). Для не-перечисляемых типов шин этим объектом должен быть _HID, и его значением
тоже должен быть ID устройства из одного из пространств имён, предписанных спецификацией.

Специальный ID устройства-ссылки на пространство имён DT, PRP0001, предоставляет
средство использовать существующую DT-совместимую идентификацию устройств в ACPI и
одновременно удовлетворять приведённым выше требованиям, вытекающим из спецификации
ACPI. А именно, если _HID возвращает PRP0001, подсистема ACPI будет искать свойство
"compatible" в _DSD объекта устройства и использовать значение этого свойства для
идентификации соответствующего устройства по аналогии с исходным алгоритмом
идентификации устройств DT. Если свойство "compatible" отсутствует или его значение
недействительно, устройство не будет перечислено подсистемой ACPI. В противном случае
оно будет перечислено автоматически как платформенное устройство (за исключением
случая, когда от устройства к его родителю присутствует связь I2C или SPI, и тогда
ядро ACPI оставит перечисление устройства драйверу родителя), и идентификационные
строки из значения свойства "compatible" будут использованы для поиска драйвера
устройства наряду с ID устройств, перечисленными _CID (если они присутствуют).

Аналогично, если PRP0001 присутствует в списке ID устройств, возвращаемом _CID,
идентификационные строки, перечисленные значением свойства "compatible" (если оно
присутствует и действительно), будут использованы для поиска драйвера, подходящего
устройству, но в этом случае их относительный приоритет по отношению к другим ID
устройств, перечисленным _HID и _CID, зависит от позиции PRP0001 в возвращаемом
пакете _CID. В частности, ID устройств, возвращаемые _HID и предшествующие PRP0001 в
возвращаемом пакете _CID, будут проверены первыми. Также в этом случае тип шины, к
которой будет перечислено устройство, зависит от ID устройства, возвращаемого _HID.

Например, следующий образец ACPI можно использовать для перечисления датчика
температуры I2C типа lm75 и сопоставления его с драйвером с помощью ссылки на
пространство имён Device Tree::

	Device (TMP0)
	{
		Name (_HID, "PRP0001")
		Name (_DSD, Package () {
			ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
			Package () {
				Package () { "compatible", "ti,tmp75" },
			}
		})
		Method (_CRS, 0, Serialized)
		{
			Name (SBUF, ResourceTemplate ()
			{
				I2cSerialBusV2 (0x48, ControllerInitiated,
					400000, AddressingMode7Bit,
					"\\_SB.PCI0.I2C1", 0x00,
					ResourceConsumer, , Exclusive,)
			})
			Return (SBUF)
		}
	}

Допустимо определять объекты устройств с _HID, возвращающим PRP0001, и без
свойства "compatible" в _DSD или _CID, при условии что один из их предков
предоставляет _DSD с действительным свойством "compatible". Такие объекты
устройств затем просто рассматриваются как дополнительные «блоки», предоставляющие
иерархическую информацию о конфигурации драйверу составного устройства-предка.

Однако PRP0001 может быть возвращён из _HID или _CID объекта устройства только в том
случае, если все свойства, возвращаемые связанным с ним _DSD (либо _DSD самого
объекта устройства, либо _DSD его предка в описанном выше случае «составного
устройства»), могут быть использованы в среде ACPI. В противном случае сам _DSD
рассматривается как недействительный, и поэтому возвращаемое им свойство
"compatible" не имеет смысла.

Дополнительную информацию см. в
Documentation/firmware-guide/acpi/DSD-properties-rules.rst.

Представление иерархии PCI
==========================

Иногда может быть полезно перечислить устройство PCI, зная его позицию на
шине PCI.

Например, в некоторых системах используются устройства PCI, припаянные прямо на
материнской плате, в фиксированной позиции (ethernet, Wi-Fi, последовательные порты
и т. д.). В этих условиях можно ссылаться на эти устройства PCI, зная их позицию в
топологии шины PCI.

Чтобы идентифицировать устройство PCI, требуется полное иерархическое описание, от
корневого порта (root port) чипсета до конечного устройства, через все промежуточные
мосты/коммутаторы (bridges/switches) платы.

Например, предположим, у нас есть система с последовательным портом PCIe,
Exar XR17V3521, припаянным на основной плате. Эта микросхема UART также включает
16 GPIO, и мы хотим добавить свойство ``gpio-line-names`` [1]_ для этих контактов.
В этом случае вывод ``lspci`` для этого компонента таков::

	07:00.0 Serial controller: Exar Corp. XR17V3521 Dual PCIe UART (rev 03)

Полный вывод ``lspci`` (вручную сокращённый по длине) таков::

	00:00.0 Host bridge: Intel Corp... Host Bridge (rev 0d)
	...
	00:13.0 PCI bridge: Intel Corp... PCI Express Port A #1 (rev fd)
	00:13.1 PCI bridge: Intel Corp... PCI Express Port A #2 (rev fd)
	00:13.2 PCI bridge: Intel Corp... PCI Express Port A #3 (rev fd)
	00:14.0 PCI bridge: Intel Corp... PCI Express Port B #1 (rev fd)
	00:14.1 PCI bridge: Intel Corp... PCI Express Port B #2 (rev fd)
	...
	05:00.0 PCI bridge: Pericom Semiconductor Device 2404 (rev 05)
	06:01.0 PCI bridge: Pericom Semiconductor Device 2404 (rev 05)
	06:02.0 PCI bridge: Pericom Semiconductor Device 2404 (rev 05)
	06:03.0 PCI bridge: Pericom Semiconductor Device 2404 (rev 05)
	07:00.0 Serial controller: Exar Corp. XR17V3521 Dual PCIe UART (rev 03) <-- Exar
	...

Топология шины такова::

	-[0000:00]-+-00.0
	           ...
	           +-13.0-[01]----00.0
	           +-13.1-[02]----00.0
	           +-13.2-[03]--
	           +-14.0-[04]----00.0
	           +-14.1-[05-09]----00.0-[06-09]--+-01.0-[07]----00.0 <-- Exar
	           |                               +-02.0-[08]----00.0
	           |                               \-03.0-[09]--
	           ...
	           \-1f.1

Чтобы описать это устройство Exar на шине PCI, мы должны начать с ACPI-имени
моста чипсета (также называемого «корневым портом», root port) с адресом::

	Bus: 0 - Device: 14 - Function: 1

Чтобы найти эту информацию, необходимо дизассемблировать таблицы ACPI BIOS,
в частности DSDT (см. также [2]_)::

	mkdir ~/tables/
	cd ~/tables/
	acpidump > acpidump
	acpixtract -a acpidump
	iasl -e ssdt?.* -d dsdt.dat

Теперь в dsdt.dsl мы должны найти устройство, чей адрес связан с
0x14 (device) и 0x01 (function). В этом случае мы можем найти следующее
устройство::

	Scope (_SB.PCI0)
	{
	... other definitions follow ...
		Device (RP02)
		{
			Method (_ADR, 0, NotSerialized)  // _ADR: Address
			{
				If ((RPA2 != Zero))
				{
					Return (RPA2) /* \RPA2 */
				}
				Else
				{
					Return (0x00140001)
				}
			}
	... other definitions follow ...

и метод _ADR [3]_ возвращает именно ту пару device/function, которую
мы ищем. С этой информацией и анализируя приведённый выше вывод ``lspci``
(как список устройств, так и дерево устройств), мы можем написать следующее
ACPI-описание для Exar PCIe UART, также добавив список имён его линий GPIO::

	Scope (_SB.PCI0.RP02)
	{
		Device (BRG1) //Bridge
		{
			Name (_ADR, 0x0000)

			Device (BRG2) //Bridge
			{
				Name (_ADR, 0x00010000)

				Device (EXAR)
				{
					Name (_ADR, 0x0000)

					Name (_DSD, Package ()
					{
						ToUUID("daffd814-6eba-4d8c-8a91-bc9bbf4aa301"),
						Package ()
						{
							Package ()
							{
								"gpio-line-names",
								Package ()
								{
									"mode_232",
									"mode_422",
									"mode_485",
									"misc_1",
									"misc_2",
									"misc_3",
									"",
									"",
									"aux_1",
									"aux_2",
									"aux_3",
								}
							}
						}
					})
				}
			}
		}
	}

Расположение "_SB.PCI0.RP02" получено в результате приведённого выше исследования
таблицы dsdt.dsl, тогда как имена устройств "BRG1", "BRG2" и "EXAR"
создаются при анализе позиции Exar UART в топологии шины PCI.
