=============================================
Правила использования свойств устройств _DSD
=============================================

Свойства, наборы свойств и подмножества свойств
===============================================

Конфигурационный объект _DSD (Device Specific Data), появившийся в ACPI 5.1,
позволяет предоставлять конфигурационные данные устройства любого типа через
пространство имён ACPI.  В принципе формат данных может быть произвольным, но он
должен идентифицироваться UUID, который должен распознаваться драйвером,
обрабатывающим вывод _DSD.  Однако существуют обобщённые UUID, определённые для
_DSD и распознаваемые подсистемой ACPI в ядре Linux, которая автоматически
обрабатывает связанные с ними пакеты данных и делает эти данные доступными
драйверам устройств как «свойства устройства» (device properties).

Свойство устройства — это элемент данных, состоящий из строкового ключа и
связанного с ним значения (определённого типа).

В контексте ACPI _DSD это элемент подпакета, следующего за обобщённым UUID
свойств устройства (Device Properties UUID) в возвращаемом пакете _DSD, как
указано в разделе «Well-Known _DSD UUIDs and Data Structure Formats»,
подраздел «Device Properties UUID» документа _DSD (Device Specific Data)
Implementation Guide [1]_.

Его также можно рассматривать как определение ключа и связанного с ним типа
данных, которые могут быть возвращены _DSD в подпакете Device Properties UUID
для данного устройства.

Набор свойств — это совокупность свойств, применимых к аппаратной сущности,
такой как устройство.  В контексте ACPI _DSD это множество всех свойств, которые
могут быть возвращены в подпакете Device Properties UUID для рассматриваемого
устройства.

Подмножества свойств — это вложенные совокупности свойств.  Каждое из них
связано с дополнительным ключом (именем), позволяющим ссылаться на подмножество
как на целое (и рассматривать его как отдельную сущность).  Каноническое
представление подмножеств свойств реализуется посредством механизма, указанного
в разделе «Well-Known _DSD UUIDs and Data Structure Formats», подраздел
«Hierarchical Data Extension UUID» документа _DSD (Device Specific Data)
Implementation Guide [1]_.

Наборы свойств могут быть иерархическими.  То есть набор свойств может содержать
несколько подмножеств свойств, каждое из которых, в свою очередь, может
содержать собственные подмножества свойств, и так далее.

Общее правило допустимости наборов свойств
===========================================

Допустимые наборы свойств должны следовать рекомендациям, приведённым в документе
определения Device Properties UUID [1].

Свойства _DSD предназначены для использования в дополнение к существующим
механизмам, определённым спецификацией ACPI, а не вместо них.  Поэтому, как
правило, их следует использовать только в том случае, если спецификация ACPI не
предусматривает прямых средств для обработки соответствующего сценария
использования.  Как правило, недопустимо возвращать из _DSD в пакетах данных,
связанных с Device Properties UUID, наборы свойств, не следующие этому правилу.

Дополнительные соображения
--------------------------

Бывают случаи, когда, даже если приведённое выше общее правило в принципе
соблюдается, набор свойств всё равно нельзя считать допустимым.

Например, это относится к свойствам устройства, которые могут заставить код ядра
(будь то драйвер устройства или библиотека/подсистема) обращаться к оборудованию
способом, который может привести к конфликту с AML-методами в пространстве имён
ACPI.  В частности, это может произойти, если код ядра использует свойства
устройства для управления оборудованием, которое обычно контролируется методами
ACPI, связанными с управлением питанием, такими как _PSx и _DSW (для объектов
устройств) или _ON и _OFF (для объектов ресурсов питания), либо методами ACPI
отключения/включения устройства, такими как _DIS и _SRS.

Во всех случаях, когда код ядра может выполнить нечто, что введёт AML в
заблуждение в результате использования свойств устройства, рассматриваемые
свойства устройства непригодны для среды ACPI и, следовательно, не могут
принадлежать к допустимому набору свойств.

Наборы свойств и привязки Device Tree
=====================================

Часто бывает полезно заставить _DSD возвращать наборы свойств, следующие
привязкам (binding) Device Tree.

Однако в таких случаях в первую очередь необходимо учитывать приведённые выше
соображения о допустимости, и следует избегать возврата из _DSD недопустимых
наборов свойств.  По этой причине может оказаться невозможным заставить _DSD
возвращать набор свойств, дословно и полностью следующий заданной привязке DT.
Тем не менее, ради повторного использования кода может иметь смысл предоставить
как можно больше конфигурационных данных в виде свойств устройства и дополнить
это ACPI-специфичным механизмом, подходящим для рассматриваемого сценария
использования.

В любом случае не следует ожидать, что наборы свойств, дословно следующие
привязкам DT, будут автоматически работать в среде ACPI независимо от их
содержимого.
