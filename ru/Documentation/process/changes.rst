.. _changes:

Минимальные требования для компиляции ядра
+++++++++++++++++++++++++++++++++++++++++++

Введение
========

Этот документ предназначен для предоставления списка минимальных версий
программного обеспечения, необходимого для запуска текущей версии ядра.

Этот документ изначально основан на моём файле «Changes» для ядер 2.0.x,
и поэтому благодарность за него причитается тем же людям, что и за тот файл
(Jared Mauch, Axel Boldt, Alessandro Sigala и бесчисленным другим пользователям
по всей Сети).

Текущие минимальные требования
******************************

Обновитесь как минимум до указанных версий программного обеспечения, прежде чем
думать, что вы столкнулись с ошибкой!  Если вы не уверены, какую версию вы сейчас
используете, предлагаемая команда должна это сообщить.

И снова имейте в виду, что этот список предполагает, что вы уже фактически
работаете на ядре Linux.  Кроме того, не все инструменты необходимы на всех
системах; очевидно, что если у вас, например, нет оборудования PC Card,
вам, вероятно, не нужно заботиться о pcmciautils.

Компиляция ядра
***************

GCC
---

Требования к версии gcc могут различаться в зависимости от типа CPU в вашем
компьютере.

Clang/LLVM (необязательно)
--------------------------

Для сборки ядер поддерживается последний официальный релиз clang и утилит LLVM
(согласно `releases.llvm.org <https://releases.llvm.org>`_). Работа более старых
релизов не гарантируется, и мы можем удалять из ядра обходные решения, которые
использовались для поддержки более старых версий. Дополнительные сведения см.
в :ref:`Сборка Linux с помощью Clang/LLVM <kbuild_llvm>`.

Rust (необязательно)
--------------------

Требуется недавняя версия компилятора Rust.

Инструкции о том, как удовлетворить требования к сборке для поддержки Rust,
см. в Documentation/rust/quick-start.rst. В частности, цель ``Makefile``
``rustavailable`` полезна для проверки того, почему тулчейн Rust может быть
не обнаружен.

bindgen (необязательно)
-----------------------

``bindgen`` используется для генерации Rust-привязок к C-части ядра.
Он зависит от ``libclang``.

Make
----

Для сборки ядра вам понадобится GNU make 4.0 или новее.

Bash
----

Для сборки ядра используются некоторые bash-скрипты.
Требуется Bash 4.2 или новее.

Binutils
--------

Для сборки ядра требуется Binutils 2.30 или новее.

pkg-config
----------

Система сборки, начиная с 4.18, требует pkg-config для проверки установленных
инструментов kconfig и для определения настроек флагов, используемых в
'make {g,x}config'.  Ранее pkg-config использовался, но не
проверялся и не документировался.

Flex
----

Начиная с Linux 4.16, система сборки генерирует лексические анализаторы
во время сборки.  Для этого требуется flex 2.5.35 или новее.


Bison
-----

Начиная с Linux 4.16, система сборки генерирует синтаксические анализаторы
во время сборки.  Для этого требуется bison 2.0 или новее.

pahole
------

Начиная с Linux 5.2, если выбран CONFIG_DEBUG_INFO_BTF, система сборки
генерирует BTF (BPF Type Format) из DWARF в vmlinux, а немного позже — также
из модулей ядра.  Для этого требуется pahole v1.22 или новее.

Его можно найти в пакетах дистрибутивов 'dwarves' или 'pahole' либо по адресу
https://fedorapeople.org/~acme/dwarves/.

Perl
----

Для сборки ядра вам понадобится perl 5 и следующие модули: ``Getopt::Long``,
``Getopt::Std``, ``File::Basename`` и ``File::Find``.

Python
------

Он требуется для нескольких параметров конфигурации: для конфигураций
по умолчанию arm/arm64, CONFIG_LTO_CLANG, некоторых необязательных конфигураций
DRM, инструмента kernel-doc и сборки документации (Sphinx), среди прочего.

BC
--

Для сборки ядер 3.10 и выше вам понадобится bc


OpenSSL
-------

Подписывание модулей и обработка внешних сертификатов используют программу
OpenSSL и крипто-библиотеку для создания ключей и генерации подписей.

Если включено подписывание модулей, для сборки ядер 3.7 и выше вам понадобится
openssl.  Также вам понадобятся пакеты разработки openssl для сборки ядер 4.3
и выше.

Tar
---

GNU tar требуется, если вы хотите включить доступ к заголовкам ядра через sysfs
(CONFIG_IKHEADERS).

gtags / GNU GLOBAL (необязательно)
----------------------------------

Сборка ядра требует GNU GLOBAL версии 6.6.5 или новее для генерации
файлов тегов через ``make gtags``.  Это связано с использованием им флага gtags
``-C (--directory)``.

mkimage
-------

Этот инструмент используется при сборке Flat Image Tree (FIT), часто
применяемого на платформах ARM. Инструмент доступен через пакет
``u-boot-tools`` или может быть собран из исходного кода U-Boot. См.
инструкции по адресу
https://docs.u-boot.org/en/latest/build/tools.html#building-tools-for-linux

GNU AWK
-------

GNU AWK требуется, если вы хотите, чтобы сборки ядра генерировали данные
о диапазонах адресов для встроенных модулей (CONFIG_BUILTIN_MODULE_RANGES).

Системные утилиты
*****************

Архитектурные изменения
-----------------------

DevFS была признана устаревшей в пользу udev
(https://www.kernel.org/pub/linux/utils/kernel/hotplug/)

Поддержка 32-битных UID теперь реализована.  Веселитесь!

Документация Linux для функций переходит на встроенную
документацию через специально отформатированные комментарии рядом с их
определениями в исходном коде.  Эти комментарии можно объединять с ReST-файлами
в каталоге Documentation/ для создания обогащённой документации, которую затем
можно преобразовать в файлы PostScript, HTML, LaTex, ePUB и PDF.
Чтобы преобразовать из формата ReST в формат по вашему выбору, вам понадобится
Sphinx.

Util-linux
----------

Новые версии util-linux предоставляют поддержку ``fdisk`` для дисков большего
размера, поддерживают новые параметры монтирования, распознают больше
поддерживаемых типов разделов и тому подобные приятные вещи.
Вероятно, вам стоит обновиться.

Ksymoops
--------

Если случается немыслимое и ваше ядро выдаёт oops, вам может понадобиться
инструмент ksymoops для его расшифровки, но в большинстве случаев он не нужен.
Обычно предпочтительнее собирать ядро с ``CONFIG_KALLSYMS``, чтобы
оно выдавало читаемые дампы, которые можно использовать как есть (это также
даёт лучший вывод, чем ksymoops).  Если по какой-то причине ваше ядро
не собрано с ``CONFIG_KALLSYMS`` и у вас нет возможности пересобрать его и
воспроизвести Oops с этим параметром, то вы всё ещё можете расшифровать этот Oops
с помощью ksymoops.

Mkinitrd
--------

Эти изменения в структуре дерева файлов ``/lib/modules`` также требуют,
чтобы mkinitrd был обновлён.

E2fsprogs
---------

Последняя версия ``e2fsprogs`` исправляет несколько ошибок в fsck и
debugfs.  Очевидно, что обновиться — хорошая идея.

JFSutils
--------

Пакет ``jfsutils`` содержит утилиты для файловой системы.
Доступны следующие утилиты:

- ``fsck.jfs`` — инициирует воспроизведение журнала транзакций, а также проверяет
  и восстанавливает раздел, отформатированный в JFS.

- ``mkfs.jfs`` — создаёт раздел, отформатированный в JFS.

- в этом пакете также доступны другие утилиты файловой системы.

Xfsprogs
--------

Последняя версия ``xfsprogs`` содержит утилиты ``mkfs.xfs``, ``xfs_db`` и
``xfs_repair``, среди прочих, для файловой системы XFS.  Она не зависит
от архитектуры, и любая версия от 2.0.0 и далее должна
корректно работать с этой версией кода ядра XFS (2.6.0 или
новее рекомендуется из-за некоторых существенных улучшений).

PCMCIAutils
-----------

PCMCIAutils заменяет ``pcmcia-cs``. Он правильно настраивает
сокеты PCMCIA при запуске системы и загружает соответствующие модули
для 16-битных устройств PCMCIA, если ядро модульное и используется
подсистема hotplug.

Quota-tools
-----------

Поддержка 32-битных uid и gid требуется, если вы хотите использовать
более новый формат квот версии 2.  Quota-tools версии 3.07 и
новее имеет эту поддержку.  Используйте рекомендуемую версию или новее
из таблицы выше.

Микрокод Intel IA32
-------------------

Был добавлен драйвер, позволяющий обновлять микрокод Intel IA32,
доступный как обычное (misc) символьное устройство.  Если вы не используете
udev, вам может понадобиться выполнить от имени root::

  mkdir /dev/cpu
  mknod /dev/cpu/microcode c 10 184
  chmod 0644 /dev/cpu/microcode

прежде чем вы сможете это использовать.  Вероятно, вы также захотите
получить утилиту пространства пользователя microcode_ctl для использования с этим.

udev
----

``udev`` — это приложение пространства пользователя для динамического заполнения
``/dev`` только записями для фактически присутствующих устройств. ``udev`` заменяет
базовую функциональность devfs, при этом обеспечивая постоянное именование
устройств.

FUSE
----

Требуется libfuse 2.4.0 или новее.  Абсолютный минимум — 2.3.0, но параметры
монтирования ``direct_io`` и ``kernel_cache`` работать не будут.

Сеть
****

Общие изменения
---------------

Если у вас есть продвинутые потребности в настройке сети, вам, вероятно, стоит
рассмотреть использование сетевых инструментов из ip-route2.

Фильтрация пакетов / NAT
------------------------
Код фильтрации пакетов и NAT использует те же инструменты, что и предыдущая
серия ядер 2.4.x (iptables).  Он по-прежнему включает модули обратной
совместимости для ipchains в стиле 2.2.x и ipfwadm в стиле 2.0.x.

PPP
---

Драйвер PPP был реструктурирован для поддержки multilink и для того,
чтобы он мог работать поверх разнообразных слоёв среды передачи.  Если вы
используете PPP, обновите pppd как минимум до 2.4.0.

Если вы не используете udev, у вас должен быть файл устройства /dev/ppp,
который можно создать командой (от имени root)::

  mknod /dev/ppp c 108 0

от имени root.

NFS-utils
---------

В древних (2.4 и более ранних) ядрах сервер nfs должен был знать
о любом клиенте, который рассчитывал получить доступ к файлам через NFS.  Эта
информация передавалась ядру утилитой ``mountd``, когда клиент
монтировал файловую систему, или утилитой ``exportfs`` при запуске системы.
exportfs брала информацию об активных клиентах из ``/var/lib/nfs/rmtab``.

Этот подход довольно хрупок, так как он зависит от корректности rmtab,
что не всегда легко, особенно при попытке реализовать
отказоустойчивость (fail-over).  Даже когда система работает хорошо, ``rmtab``
страдает от накопления множества старых записей, которые никогда не удаляются.

В современных ядрах у нас есть возможность заставить ядро сообщать mountd,
когда оно получает запрос от неизвестного хоста, а mountd может передать
соответствующую информацию об экспорте ядру.  Это устраняет
зависимость от ``rmtab`` и означает, что ядру нужно знать только о
текущих активных клиентах.

Чтобы включить эту новую функциональность, вам нужно выполнить::

  mount -t nfsd nfsd /proc/fs/nfsd

перед запуском exportfs или mountd.  Рекомендуется защищать все NFS-сервисы
от широкого интернета с помощью межсетевого экрана там, где это
возможно.

mcelog
------

На ядрах x86 утилита mcelog нужна для обработки и журналирования событий
machine check, когда включён ``CONFIG_X86_MCE``. События machine check — это ошибки,
сообщаемые CPU. Их обработка настоятельно рекомендуется.
