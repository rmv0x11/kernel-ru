.. _submitchecklist:

==============================================================
Контрольный список отправки патчей для ядра Linux
==============================================================

Ниже приведён ряд базовых действий, которые разработчикам стоит выполнять,
если они хотят, чтобы их патчи для ядра принимались быстрее.

Всё это дополняет документацию, приведённую в
:ref:`Documentation/process/submitting-patches.rst <submittingpatches>`
и в других местах относительно отправки патчей для ядра Linux.

Проверьте свой код
==================

1) Если вы пользуетесь какой-либо возможностью, то делайте #include того файла,
   который её определяет/объявляет. Не полагайтесь на то, что другие заголовочные
   файлы подтянут те, которые используете вы.

2) Проверьте свой патч на соответствие общему стилю, как подробно описано в
   :ref:`Documentation/process/coding-style.rst <codingstyle>`.

3) Все барьеры памяти {например, ``barrier()``, ``rmb()``, ``wmb()``} нуждаются в
   комментарии в исходном коде, который поясняет логику того, что они делают
   и зачем.

Проверьте изменения в Kconfig
=============================

1) Любые новые или изменённые опции ``CONFIG`` не должны портить меню конфигурации
   и по умолчанию должны быть выключены, если только они не удовлетворяют критериям
   исключения, описанным в
   ``Documentation/kbuild/kconfig-language.rst`` Menu attributes: default value.

2) Все новые опции ``Kconfig`` снабжены текстом справки.

3) Изменения тщательно проверены в отношении значимых комбинаций ``Kconfig``.
   Этого очень трудно добиться правильно с помощью тестирования — здесь
   окупается умственная работа.

Предоставьте документацию
=========================

1) Включайте :ref:`kernel-doc <kernel_doc>` для документирования глобальных API
   ядра. (Не требуется для статических функций, но там это тоже допустимо.)

2) Все новые элементы ``/proc`` документированы в ``Documentation/``

3) Все новые параметры загрузки ядра документированы в
   ``Documentation/admin-guide/kernel-parameters.rst``.

4) Все новые параметры модулей документированы с помощью ``MODULE_PARM_DESC()``

5) Все новые интерфейсы пространства пользователя (userspace) документированы
   в ``Documentation/ABI/``.
   Дополнительную информацию см. в Documentation/admin-guide/abi.rst (или
   ``Documentation/ABI/README``).
   Патчи, которые изменяют интерфейсы пространства пользователя, следует
   отправлять в копии (CC) на linux-api@vger.kernel.org.

6) Если патч добавляет какие-либо ioctl, то также обновите
   ``Documentation/userspace-api/ioctl/ioctl-number.rst``.

Проверьте свой код инструментами
================================

1) Проверьте на тривиальные нарушения с помощью программы проверки стиля патчей
   перед отправкой (``scripts/checkpatch.pl``).
   Вы должны быть в состоянии обосновать все нарушения, которые остаются
   в вашем патче.

2) Проверьте, что код проходит sparse без замечаний.

3) Используйте ``make checkstack`` и устраните все найденные им проблемы.
   Обратите внимание, что ``checkstack`` не указывает на проблемы явно,
   но любая функция, использующая более 512 байт на стеке, является
   кандидатом на изменение.

Соберите свой код
=================

1) Собирается без замечаний:

  a) с применимыми или изменёнными опциями ``CONFIG`` ``=y``, ``=m`` и
     ``=n``. Без предупреждений/ошибок ``gcc``, без предупреждений/ошибок
     компоновщика.

  b) Проходит ``allnoconfig``, ``allmodconfig``

  c) Успешно собирается при использовании ``O=builddir``

  d) Любые изменения в Documentation/ собираются успешно без новых
     предупреждений/ошибок. Используйте ``make htmldocs`` или ``make pdfdocs``,
     чтобы проверить сборку и устранить любые проблемы.

2) Собирается на нескольких архитектурах CPU с помощью локальных средств
   кросс-компиляции или какой-либо другой сборочной фермы.
   Обратите внимание, что тестирование на архитектурах с разным размером слова
   (32- и 64-битных) и разным порядком байтов (big- и little-) эффективно
   для выявления различных проблем переносимости, вызванных ложными
   предположениями о представимом диапазоне величин, выравнивании данных
   или порядке байтов, среди прочего.

3) Вновь добавленный код был скомпилирован с помощью ``gcc -W`` (используйте
   ``make KCFLAGS=-W``). Это сгенерирует много шума, но полезно
   для поиска ошибок вроде "warning: comparison between signed and unsigned".

4) Если ваш изменённый исходный код зависит от каких-либо API или возможностей
   ядра, связанных со следующими символами ``Kconfig``, или использует их,
   то протестируйте несколько сборок с отключёнными соответствующими
   символами ``Kconfig`` и/или ``=m`` (если такая опция доступна) [не все из них
   одновременно, а в различных/случайных комбинациях]:

   ``CONFIG_SMP``, ``CONFIG_SYSFS``, ``CONFIG_PROC_FS``, ``CONFIG_INPUT``,
   ``CONFIG_PCI``, ``CONFIG_BLOCK``, ``CONFIG_PM``, ``CONFIG_MAGIC_SYSRQ``,
   ``CONFIG_NET``, ``CONFIG_INET=n`` (но последний с ``CONFIG_NET=y``).

Протестируйте свой код
======================

1) Протестировано с одновременно включёнными ``CONFIG_PREEMPT``,
   ``CONFIG_DEBUG_PREEMPT``, ``CONFIG_SLUB_DEBUG``, ``CONFIG_DEBUG_PAGEALLOC``,
   ``CONFIG_DEBUG_MUTEXES``, ``CONFIG_DEBUG_SPINLOCK``,
   ``CONFIG_DEBUG_ATOMIC_SLEEP``, ``CONFIG_PROVE_RCU`` и
   ``CONFIG_DEBUG_OBJECTS_RCU_HEAD``.

2) Протестировано при сборке и во время выполнения с ``CONFIG_SMP`` и
   ``CONFIG_PREEMPT`` и без них.

3) Все пути выполнения кода были задействованы со всеми включёнными
   возможностями lockdep.

4) Проверено с внедрением как минимум отказов выделения slab и страниц.
   См. ``Documentation/fault-injection/``.
   Если новый код значителен по объёму, может быть уместно добавление
   специфичного для подсистемы внедрения отказов.

5) Протестировано с самым свежим тегом linux-next, чтобы убедиться, что код
   по-прежнему работает со всеми остальными патчами в очереди и различными
   изменениями в VM, VFS и других подсистемах.
