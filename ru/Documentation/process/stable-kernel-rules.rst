.. _stable_kernel_rules:

Всё, что вы когда-либо хотели знать о релизах Linux -stable
===========================================================

Правила о том, какого рода патчи принимаются в дерево «-stable», а какие — нет:

- Сам патч или эквивалентное исправление уже должны присутствовать в Linux mainline (upstream).
- Он должен быть очевидно корректным и протестированным.
- Он не может быть больше 100 строк, считая контекст.
- Он должен следовать правилам
  :ref:`Documentation/process/submitting-patches.rst <submittingpatches>`.
- Он должен либо исправлять реальную ошибку, которая беспокоит людей, либо просто
  добавлять идентификатор устройства (device ID). Уточняя первый случай:

  - Он исправляет проблему вроде oops, зависания, повреждения данных, реальной проблемы
    безопасности, аппаратной особенности (hardware quirk), ошибки сборки (но не для
    вещей, помеченных CONFIG_BROKEN) или какой-либо проблемы из разряда «ой, это
    нехорошо».
  - Серьёзные проблемы, о которых сообщил пользователь дистрибутивного ядра, также
    могут быть рассмотрены, если они исправляют заметную проблему производительности
    или интерактивности. Поскольку такие исправления не столь очевидны и имеют более
    высокий риск незаметной регрессии, они должны отправляться только сопровождающим
    (maintainer) дистрибутивного ядра и включать дополнение со ссылкой на запись в
    bugzilla, если она существует, и дополнительной информацией о влиянии, видимом
    для пользователя.
  - Никаких вещей типа «это могло бы быть проблемой...», как, например, «теоретическое
    состояние гонки (race condition)», если только не приведено также объяснение того,
    как эту ошибку можно использовать (exploit).
  - Никаких «тривиальных» исправлений без пользы для пользователей (исправления
    орфографии, чистка пробелов и т. п.).


Процедура отправки патчей в дерево -stable
------------------------------------------

.. note::

   Патчи безопасности не должны обрабатываться (исключительно) через процесс ревью
   -stable; вместо этого следует следовать процедурам, описанным в
   :ref:`Documentation/process/security-bugs.rst <securitybugs>`.

Существует три варианта отправки изменения в деревья -stable:

1. Добавить «stable tag» (тег stable) в описание патча, который вы затем отправляете
   для включения в mainline.
2. Попросить команду stable подобрать патч, уже попавший в mainline.
3. Отправить команде stable патч, эквивалентный изменению, уже находящемуся в mainline.

Разделы ниже описывают каждый из вариантов более подробно.

:ref:`option_1` **настоятельно** предпочтителен — он самый простой и наиболее
распространённый. :ref:`option_2` в основном предназначен для изменений, бэкпорт которых
не рассматривался во время отправки. :ref:`option_3` — это альтернатива двум предыдущим
вариантам для случаев, когда патч, попавший в mainline, нуждается в корректировках, чтобы
примениться в более старых сериях (например, из-за изменений API).

При использовании варианта 2 или 3 вы можете попросить включить ваше изменение в
конкретные серии stable. Делая это, убедитесь, что исправление или его эквивалент
применимо, отправлено или уже присутствует во всех более новых деревьях stable, которые
ещё поддерживаются. Это призвано предотвратить регрессии, с которыми пользователи могли бы
позже столкнуться при обновлении, если, например, исправление, слитое в 5.19-rc1, было бы
бэкпортировано в 5.10.y, но не в 5.15.y.

.. _option_1:

Вариант 1
*********

Чтобы патч, который вы отправляете для включения в mainline, позже был автоматически
подобран для деревьев stable, добавьте этот тег в области sign-off::

  Cc: stable@vger.kernel.org

Используйте ``Cc: stable@kernel.org`` вместо этого при исправлении неопубликованных
уязвимостей: это снижает вероятность случайного раскрытия исправления публике через
'git send-email', поскольку письма, отправленные на этот адрес, никуда не доставляются.

После того как патч попадёт в mainline, он будет применён к дереву stable без каких-либо
дополнительных действий со стороны автора или сопровождающего подсистемы.

Чтобы передать команде stable дополнительные инструкции, используйте встроенный
комментарий в стиле shell для передачи произвольных или предопределённых заметок:

* Укажите любые дополнительные предварительные условия (prerequisites) патча для
  cherry-pick::

    Cc: <stable@vger.kernel.org> # 3.3.x: a1f84a3: sched: Check for idle
    Cc: <stable@vger.kernel.org> # 3.3.x: 1b9508f: sched: Rate-limit newidle
    Cc: <stable@vger.kernel.org> # 3.3.x: fd21073: sched: Fix affinity logic
    Cc: <stable@vger.kernel.org> # 3.3.x
    Signed-off-by: Ingo Molnar <mingo@elte.hu>

  Эта последовательность тегов имеет следующий смысл::

    git cherry-pick a1f84a3
    git cherry-pick 1b9508f
    git cherry-pick fd21073
    git cherry-pick <this commit>

  Обратите внимание, что для серии патчей вам не нужно перечислять в качестве
  предварительных условий патчи, присутствующие в самой серии. Например, если у вас есть
  следующая серия патчей::

    patch1
    patch2

  где patch2 зависит от patch1, вам не нужно перечислять patch1 как предварительное
  условие для patch2, если вы уже пометили patch1 для включения в stable.

* Укажите предварительные условия по версии ядра::

    Cc: <stable@vger.kernel.org> # 3.3.x

  Этот тег имеет смысл::

    git cherry-pick <this commit>

  Для каждого дерева «-stable», начиная с указанной версии.

  Обратите внимание, что такое тегирование излишне, если команда stable может вывести
  соответствующие версии из тегов Fixes:.

* Отложите подбор патчей::

    Cc: <stable@vger.kernel.org> # after -rc3

* Укажите известные проблемы::

    Cc: <stable@vger.kernel.org> # see patch description, needs adjustments for <= 6.3

Кроме того, существует вариант тега stable, который вы можете использовать, чтобы
инструменты бэкпортирования команды stable (например, AUTOSEL или скрипты, которые ищут
коммиты, содержащие тег 'Fixes:') игнорировали изменение::

     Cc: <stable+noautosel@kernel.org> # reason goes here, and must be present

.. _option_2:

Вариант 2
*********

Если патч уже был слит в mainline, отправьте письмо на stable@vger.kernel.org, содержащее
тему патча, идентификатор коммита (commit ID), почему, по вашему мнению, его следует
применить, и в какие версии ядра вы хотите его применить.

.. _option_3:

Вариант 3
*********

Отправьте патч, после проверки того, что он следует приведённым выше правилам, на
stable@vger.kernel.org и укажите версии ядра, в которые вы хотите его применить. Делая
это, вы должны указать идентификатор upstream-коммита (commit ID) в журнале изменений
(changelog) вашей отправки отдельной строкой над текстом коммита, вот так::

  commit <sha1> upstream.

Или, альтернативно::

  [ Upstream commit <sha1> ]

Если отправленный патч отклоняется от исходного upstream-патча (например, потому что его
пришлось скорректировать под более старый API), это должно быть очень чётко
задокументировано и обосновано в описании патча.


После отправки
--------------

Отправитель получит ACK, когда патч будет принят в очередь, или NAK, если патч отклонён.
Этот ответ может занять несколько дней, в соответствии с расписанием членов команды
stable.

В случае принятия патч будет добавлен в очередь -stable для ревью другими разработчиками
и соответствующим сопровождающим подсистемы.


Цикл ревью
----------

- Когда сопровождающие -stable решают провести цикл ревью, патчи отправляются комитету
  ревью (review committee) и сопровождающему затронутой области патча (если только
  отправитель не является сопровождающим этой области), а также с CC: в список рассылки
  linux-kernel.
- У комитета ревью есть 48 часов, в течение которых нужно дать ACK или NAK патчу.
- Если патч отклонён членом комитета или члены linux-kernel возражают против патча,
  поднимая вопросы, которые сопровождающие и члены не осознали, патч будет удалён из
  очереди.
- Патчи, получившие ACK, будут опубликованы снова как часть кандидата на релиз (-rc) для
  тестирования разработчиками и тестировщиками.
- Обычно делается только один релиз -rc, однако, если есть какие-либо нерешённые
  проблемы, некоторые патчи могут быть изменены или удалены, либо в очередь могут быть
  поставлены дополнительные патчи. Затем выпускаются и тестируются дополнительные релизы
  -rc, пока не перестанут обнаруживаться проблемы.
- Реагировать на релизы -rc можно в списке рассылки, отправив письмо «Tested-by:» с любой
  желаемой информацией о тестировании. Теги «Tested-by:» будут собраны и добавлены в
  коммит релиза.
- В конце цикла ревью будет выпущен новый релиз -stable, содержащий все поставленные в
  очередь и протестированные патчи.
- Патчи безопасности будут приниматься в дерево -stable напрямую от команды безопасности
  ядра и не будут проходить обычный цикл ревью. Свяжитесь с командой безопасности ядра для
  получения дополнительных сведений об этой процедуре.


Деревья
-------

- Очереди патчей как для завершённых версий, так и для версий в процессе работы можно
  найти по адресу:

    https://git.kernel.org/pub/scm/linux/kernel/git/stable/stable-queue.git

- Финализированные и помеченные тегами релизы всех stable-ядер можно найти в отдельных
  ветках для каждой версии по адресу:

    https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git

- Кандидат на релиз (release candidate) всех версий stable-ядер можно найти по адресу:

    https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux-stable-rc.git/

  .. warning::
     Дерево -stable-rc — это снимок дерева stable-queue в определённый момент времени, и
     оно будет часто меняться, следовательно, будет часто перебазироваться (rebase). Его
     следует использовать только в целях тестирования (например, для потребления
     CI-системами).


Комитет ревью
-------------

- Он состоит из ряда разработчиков ядра, которые добровольно вызвались на эту задачу, и
  нескольких, которые не вызывались.
