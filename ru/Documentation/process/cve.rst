====
CVE
====

Номера Common Vulnerabilities and Exposure (CVE®) были разработаны как
однозначный способ идентифицировать, определять и каталогизировать публично
раскрытые уязвимости безопасности. Со временем их полезность для проекта ядра
снизилась, и номера CVE очень часто присваивались ненадлежащим образом и по
ненадлежащим причинам. Из-за этого сообщество разработчиков ядра, как правило,
старалось их избегать. Однако сочетание продолжающегося давления с целью
присвоения CVE и других видов идентификаторов безопасности, а также непрекращающиеся
злоупотребления со стороны отдельных лиц и компаний вне сообщества ядра ясно
показали, что сообщество ядра должно контролировать эти присвоения.

Команда разработчиков ядра Linux обладает возможностью присваивать CVE для
потенциальных проблем безопасности ядра Linux. Это присвоение независимо от
:doc:`обычного процесса сообщения об ошибках безопасности ядра Linux<../process/security-bugs>`.

Список всех присвоенных CVE для ядра Linux можно найти в архивах списка рассылки
linux-cve, как показано на https://lore.kernel.org/linux-cve-announce/. Чтобы
получать уведомления о присвоенных CVE, пожалуйста, `подпишитесь
<https://subspace.kernel.org/subscribing.html>`_ на этот список рассылки.

Процесс
=======

В рамках обычного процесса стабильных выпусков изменения ядра, которые
потенциально являются проблемами безопасности, выявляются разработчиками,
ответственными за присвоение номеров CVE, и им автоматически присваиваются
номера CVE. Эти присвоения публикуются в списке рассылки linux-cve-announce в
виде анонсов на регулярной основе.

Обратите внимание: из-за того уровня системы, на котором находится ядро Linux,
почти любая ошибка может быть использована для компрометации безопасности ядра,
но возможность эксплуатации зачастую неочевидна на момент исправления ошибки.
Из-за этого команда присвоения CVE проявляет чрезмерную осторожность и присваивает
номера CVE любому исправлению ошибок, которое она выявляет. Это объясняет
кажущееся большим количество CVE, выпускаемых командой ядра Linux.

Если команда присвоения CVE упустила конкретное исправление, для которого, по
мнению какого-либо пользователя, следовало присвоить CVE, пожалуйста, напишите ей
по адресу <cve@kernel.org>, и команда будет работать с вами над этим. Обратите
внимание, что никакие потенциальные проблемы безопасности не должны отправляться
на этот адрес — он предназначен ТОЛЬКО для присвоения CVE для исправлений, которые
уже находятся в выпущенных деревьях ядра. Если вы считаете, что обнаружили
неисправленную проблему безопасности, пожалуйста, следуйте :doc:`обычному процессу
сообщения об ошибках безопасности ядра Linux<../process/security-bugs>`.

Никакие CVE не будут автоматически присваиваться для неисправленных проблем
безопасности в ядре Linux; присвоение будет автоматически происходить только
после того, как исправление станет доступным и применённым к стабильному дереву
ядра, и оно будет отслеживаться по идентификатору git-коммита исходного
исправления. Если кто-либо желает получить CVE до того, как проблема будет решена
коммитом, пожалуйста, свяжитесь с командой присвоения CVE ядра по адресу
<cve@kernel.org>, чтобы получить идентификатор из её набора зарезервированных
идентификаторов.

Никакие CVE не будут присваиваться для любой проблемы, найденной в версии ядра,
которая в настоящее время не поддерживается активно командой Stable/LTS ядра.
Список ветвей ядра, поддерживаемых в настоящее время, можно найти на
https://kernel.org/releases.html

Оспаривание присвоенных CVE
===========================

Право оспаривать или изменять присвоенный CVE для конкретного изменения ядра
принадлежит исключительно сопровождающим (maintainers) соответствующей затронутой
подсистемы. Этот принцип обеспечивает высокую степень точности и ответственности
при сообщении об уязвимостях. Только те лица, которые обладают глубокой
экспертизой и доскональным знанием подсистемы, могут эффективно оценить
обоснованность и масштаб сообщённой уязвимости и определить её надлежащее
обозначение CVE. Любая попытка изменить или оспорить CVE вне рамок этого
установленного полномочия может привести к путанице, неточным сообщениям и, в
конечном счёте, к скомпрометированным системам.

Недействительные CVE
====================

Если проблема безопасности найдена в ядре Linux, которое поддерживается
дистрибутивом Linux только из-за изменений, внесённых этим дистрибутивом, или
из-за того, что дистрибутив поддерживает версию ядра, которая больше не входит в
число поддерживаемых выпусков kernel.org, то CVE не может быть присвоен командой
CVE ядра Linux и должен быть запрошен у самого этого дистрибутива Linux.

Любой CVE, присвоенный ядру Linux для активно поддерживаемой версии ядра
какой-либо группой, отличной от команды присвоения CVE ядра, не должен
рассматриваться как действительный CVE. Пожалуйста, уведомите команду присвоения
CVE ядра по адресу <cve@kernel.org>, чтобы она могла поработать над аннулированием
таких записей через процесс исправления CNA.

Применимость конкретных CVE
===========================

Поскольку ядро Linux может использоваться многими различными способами, со многими
различными способами доступа к нему внешними пользователями или вообще без доступа,
применимость любого конкретного CVE определяется пользователем Linux, а не командой
присвоения CVE. Пожалуйста, не обращайтесь к нам с попыткой определить применимость
какого-либо конкретного CVE.

Кроме того, поскольку дерево исходного кода очень велико, и любая отдельная система
использует лишь небольшое подмножество дерева исходного кода, любым пользователям
Linux следует учитывать, что большое количество присвоенных CVE не релевантно для
их систем.

Короче говоря, мы не знаем вашего сценария использования и не знаем, какие части
ядра вы используете, поэтому у нас нет способа определить, релевантен ли конкретный
CVE для вашей системы.

Как всегда, лучше всего брать все выпущенные изменения ядра, поскольку они
тестируются вместе как единое целое многими членами сообщества, а не как отдельные
выборочно перенесённые (cherry-picked) изменения. Также обратите внимание, что для
многих ошибок решение общей проблемы находится не в одном изменении, а в сумме
многих исправлений, накладывающихся друг на друга. В идеале CVE будут присвоены
всем исправлениям для всех проблем, но иногда мы не сможем заметить исправления,
поэтому исходите из того, что некоторые изменения без присвоенного CVE могут быть
важны для применения.
