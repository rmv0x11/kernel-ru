==========================================================================
Использование kgdb, kdb и внутреннее устройство отладчика ядра
==========================================================================

:Author: Jason Wessel

Введение
========

В ядре есть два различных фронтенда отладчика (kdb и kgdb), которые
взаимодействуют с ядром отладки (debug core). Можно использовать любой из
этих фронтендов и динамически переключаться между ними, если правильно
сконфигурировать ядро на этапах компиляции и выполнения.

Kdb — это упрощённый интерфейс в стиле командной оболочки, который можно
использовать на системной консоли с клавиатурой или на последовательной
консоли. С его помощью можно исследовать память, регистры, списки процессов,
dmesg и даже устанавливать точки останова для остановки в определённом месте.
Kdb не является отладчиком уровня исходного кода, хотя вы можете устанавливать
точки останова и выполнять некоторое базовое управление ходом выполнения ядра.
Kdb в основном предназначен для проведения некоторого анализа в помощь
разработке или диагностике проблем ядра. Вы можете обращаться к некоторым
символам по имени во встроенных частях ядра или в модулях ядра, если код был
собран с ``CONFIG_KALLSYMS``.

Kgdb предназначен для использования в качестве отладчика уровня исходного кода
для ядра Linux. Он используется совместно с gdb для отладки ядра Linux.
Предполагается, что gdb можно использовать для «вторжения» (break in) в ядро,
чтобы исследовать память, переменные и просматривать информацию о стеке вызовов
аналогично тому, как разработчик приложений использовал бы gdb для отладки
приложения. Можно размещать точки останова в коде ядра и выполнять некоторый
ограниченный пошаговый проход.

Для использования kgdb требуются две машины. Одна из этих машин — машина
разработки, а другая — целевая (target) машина. Отлаживаемое ядро работает на
целевой машине. На машине разработки запускается экземпляр gdb с файлом vmlinux,
который содержит символы (не загрузочный образ, такой как bzImage, zImage,
uImage...). В gdb разработчик задаёт параметры подключения и подключается к kgdb.
Тип подключения, которое разработчик создаёт через gdb, зависит от доступности
модулей ввода-вывода kgdb, скомпилированных как встроенные или как загружаемые
модули ядра в ядре тестовой машины.

Компиляция ядра
===============

-  Чтобы включить компиляцию kdb, вы должны сначала включить kgdb.

-  Опции компиляции тестов kgdb описаны в главе о наборе тестов kgdb.

Опции конфигурации ядра для kgdb
--------------------------------

Чтобы включить ``CONFIG_KGDB``, следует заглянуть в раздел
:menuselection:`Kernel hacking --> Kernel debugging` и выбрать
:menuselection:`KGDB: kernel debugger`.

Хотя наличие символов в файле vmlinux не является жёстким требованием, gdb
обычно не очень полезен без символьных данных, поэтому вам понадобится включить
``CONFIG_DEBUG_INFO``, который в меню конфигурации называется
:menuselection:`Compile the kernel with debug info`.

Рекомендуется (но не обязательно) включить опцию ядра
``CONFIG_FRAME_POINTER``, которая в меню конфигурации называется
:menuselection:`Compile the kernel with frame pointers`. Эта опция вставляет в
скомпилированный исполняемый файл код, который сохраняет информацию о кадре стека
в регистрах или в стеке в различных точках, что позволяет отладчику, такому как
gdb, более точно строить обратные трассировки стека при отладке ядра.

Если используемая вами архитектура поддерживает опцию ядра
``CONFIG_STRICT_KERNEL_RWX``, вам стоит рассмотреть возможность её отключения.
Эта опция предотвратит использование программных точек останова, поскольку она
помечает определённые области адресного пространства памяти ядра как доступные
только для чтения. Если kgdb поддерживает их для используемой вами архитектуры,
вы можете использовать аппаратные точки останова, если хотите работать с
включённой опцией ``CONFIG_STRICT_KERNEL_RWX``; в противном случае эту опцию
нужно отключить.

Далее вам следует выбрать один или несколько драйверов ввода-вывода для связи
между отлаживающим хостом и отлаживаемой целью. Отладка на ранней стадии загрузки
требует драйвера ввода-вывода KGDB, который поддерживает раннюю отладку, и этот
драйвер должен быть встроен непосредственно в ядро. Конфигурация драйвера
ввода-вывода kgdb происходит через параметры ядра или модуля, о которых вы
можете узнать больше в разделе, описывающем параметр kgdboc.

Вот пример набора символов ``.config``, которые нужно включить или отключить для
kgdb::

  # CONFIG_STRICT_KERNEL_RWX is not set
  CONFIG_FRAME_POINTER=y
  CONFIG_KGDB=y
  CONFIG_KGDB_SERIAL_CONSOLE=y

Опции конфигурации ядра для kdb
-------------------------------

Kdb значительно сложнее, чем простой gdbstub, расположенный поверх ядра отладки
(debug core). Kdb должен реализовать командную оболочку, а также добавляет
некоторые вспомогательные функции в другие части ядра, отвечающие за вывод
интересных данных, таких как то, что вы увидели бы при запуске ``lsmod`` или
``ps``. Чтобы встроить kdb в ядро, вы выполняете те же шаги, что и для kgdb.

Основная опция конфигурации для kdb — ``CONFIG_KGDB_KDB``, которая в меню
конфигурации называется
:menuselection:`KGDB_KDB: include kdb frontend for kgdb`. В теории вы уже также
выбрали бы драйвер ввода-вывода, такой как интерфейс
``CONFIG_KGDB_SERIAL_CONSOLE``, если планируете использовать kdb на
последовательном порту, при настройке kgdb.

Если вы хотите использовать клавиатуру типа PS/2 с kdb, вы выбрали бы
``CONFIG_KDB_KEYBOARD``, который в меню конфигурации называется
:menuselection:`KGDB_KDB: keyboard as input device`. Опция
``CONFIG_KDB_KEYBOARD`` не используется ни для чего в интерфейсе gdb к kgdb.
Опция ``CONFIG_KDB_KEYBOARD`` работает только с kdb.

Вот пример набора символов ``.config`` для включения/отключения kdb::

  # CONFIG_STRICT_KERNEL_RWX is not set
  CONFIG_FRAME_POINTER=y
  CONFIG_KGDB=y
  CONFIG_KGDB_SERIAL_CONSOLE=y
  CONFIG_KGDB_KDB=y
  CONFIG_KDB_KEYBOARD=y

Аргументы загрузки отладчика ядра
=================================

В этом разделе описаны различные параметры ядра времени выполнения, которые
влияют на конфигурацию отладчика ядра. В следующей главе рассматривается
использование kdb и kgdb, а также приводятся некоторые примеры параметров
конфигурации.

Параметр ядра: kgdboc
---------------------

Драйвер kgdboc изначально был аббревиатурой, означавшей «kgdb over console»
(kgdb поверх консоли). Сегодня это основной механизм для настройки того, как
взаимодействовать от gdb к kgdb, а также устройств, которые вы хотите
использовать для работы с командной оболочкой kdb.

Для kgdb/gdb kgdboc спроектирован для работы с одним последовательным портом.
Он предназначен для покрытия случая, когда вы хотите использовать
последовательную консоль в качестве основной консоли, а также использовать её
для выполнения отладки ядра. Также можно использовать kgdb на последовательном
порту, который не назначен системной консолью. Kgdboc может быть сконфигурирован
как встроенный в ядро или как загружаемый модуль ядра. Воспользоваться
``kgdbwait`` и ранней отладкой можно только при условии, что вы встроите kgdboc
в ядро как встроенный компонент.

При желании вы можете выбрать активацию интеграции с kms (Kernel Mode Setting).
Когда вы используете kms с kgdboc и у вас есть видеодрайвер, имеющий хуки atomic
mode setting, можно входить в отладчик на графической консоли. При возобновлении
выполнения ядра предыдущий графический режим будет восстановлен. Эта интеграция
может служить полезным инструментом для помощи в диагностике аварий или
проведении анализа памяти с kdb, позволяя при этом работать полноценным
приложениям графической консоли.

аргументы kgdboc
~~~~~~~~~~~~~~~~

Использование::

	kgdboc=[kms][[,]kbd][[,]serial_device][,baud]

Порядок, перечисленный выше, должен соблюдаться, если вы используете любую из
необязательных конфигураций вместе.

Сокращения:

-  kms = Kernel Mode Setting

-  kbd = Keyboard (клавиатура)

Вы можете сконфигурировать kgdboc для использования клавиатуры и/или
последовательного устройства в зависимости от того, используете ли вы kdb и/или
kgdb, в одном из следующих сценариев. Порядок, перечисленный выше, должен
соблюдаться, если вы используете любую из необязательных конфигураций вместе.
Использование kms + только gdb, как правило, не является полезной комбинацией.

Использование загружаемого модуля или встроенного компонента
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

1. Как встроенный в ядро компонент:

   Используйте аргумент загрузки ядра::

	kgdboc=<tty-device>,[baud]

2. Как загружаемый модуль ядра:

   Используйте команду::

	modprobe kgdboc kgdboc=<tty-device>,[baud]

   Вот два примера того, как можно отформатировать строку kgdboc. Первый — для
   цели x86, использующей первый последовательный порт. Второй пример — для ARM
   Versatile AB, использующего второй последовательный порт.

   1. ``kgdboc=ttyS0,115200``

   2. ``kgdboc=ttyAMA1,115200``

Конфигурирование kgdboc во время выполнения через sysfs
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Во время выполнения вы можете включить или отключить kgdboc, записав параметры в
sysfs. Вот два примера:

1. Включить kgdboc на ttyS0::

	echo ttyS0 > /sys/module/kgdboc/parameters/kgdboc

2. Отключить kgdboc::

	echo "" > /sys/module/kgdboc/parameters/kgdboc

.. note::

   Указывать скорость в бодах не нужно, если вы настраиваете консоль на tty,
   который уже настроен или открыт.

Дополнительные примеры
^^^^^^^^^^^^^^^^^^^^^^^

Вы можете сконфигурировать kgdboc для использования клавиатуры и/или
последовательного устройства в зависимости от того, используете ли вы kdb и/или
kgdb, в одном из следующих сценариев.

1. kdb и kgdb только через последовательный порт::

	kgdboc=<serial_device>[,baud]

   Пример::

	kgdboc=ttyS0,115200

2. kdb и kgdb с клавиатурой и последовательным портом::

	kgdboc=kbd,<serial_device>[,baud]

   Пример::

	kgdboc=kbd,ttyS0,115200

3. kdb с клавиатурой::

	kgdboc=kbd

4. kdb с kernel mode setting::

	kgdboc=kms,kbd

5. kdb с kernel mode setting и kgdb через последовательный порт::

	kgdboc=kms,kbd,ttyS0,115200

.. note::

   Kgdboc не поддерживает прерывание цели через удалённый протокол gdb. Вы
   должны вручную отправить `SysRq-G`, если у вас нет прокси, разделяющего вывод
   консоли в терминальную программу. Консольный прокси имеет отдельный TCP-порт
   для отладчика и отдельный TCP-порт для «человеческой» консоли. Прокси может
   взять на себя отправку `SysRq-G` за вас.

При использовании kgdboc без прокси отладчика вы можете в итоге подключить
отладчик в одной из двух точек входа. Если исключение происходит после того, как
вы загрузили kgdboc, на консоли должно вывестись сообщение о том, что система
ожидает отладчик. В этом случае вы отключаете свою терминальную программу и затем
подключаете на её место отладчик. Если вы хотите прервать целевую систему и
принудительно войти в сеанс отладки, вам нужно подать последовательность `Sysrq`
и затем набрать букву `g`. Затем вы отключаете терминальный сеанс и подключаете
gdb. Если вам это не нравится, ваши варианты — это либо доработать gdb, чтобы он
отправлял `SysRq-G` за вас, а также при начальном подключении, либо использовать
прокси отладчика, который позволяет немодифицированному gdb выполнять отладку.

Параметр ядра: ``kgdboc_earlycon``
----------------------------------

Если вы указываете параметр ядра ``kgdboc_earlycon`` и ваш драйвер
последовательного порта регистрирует загрузочную консоль (boot console),
поддерживающую опрос (не нуждающуюся в прерываниях и реализующую неблокирующую
функцию read()), kgdb попытается работать, используя загрузочную консоль, пока
не сможет перейти к обычному драйверу tty, указанному параметром ``kgdboc``.

Обычно существует только одна загрузочная консоль (особенно реализующая функцию
read()), поэтому достаточно просто добавить ``kgdboc_earlycon`` само по себе,
чтобы это заработало. Если у вас более одной загрузочной консоли, вы можете
добавить имя загрузочной консоли для различения. Обратите внимание, что имена,
зарегистрированные через слой загрузочной консоли и через слой tty, не совпадают
для одного и того же порта.

Например, на одной плате, чтобы быть явным, вы могли бы сделать так::

   kgdboc_earlycon=qcom_geni kgdboc=ttyMSM0

Если единственной загрузочной консолью на устройстве была «qcom_geni», вы могли
бы упростить::

   kgdboc_earlycon kgdboc=ttyMSM0

Параметр ядра: ``kgdbwait``
---------------------------

Опция командной строки ядра ``kgdbwait`` заставляет kgdb ждать подключения
отладчика во время загрузки ядра. Вы можете использовать эту опцию только если
вы скомпилировали драйвер ввода-вывода kgdb в ядро и указали конфигурацию
драйвера ввода-вывода как опцию командной строки ядра. Параметр kgdbwait в
командной строке ядра должен всегда следовать за параметром конфигурации драйвера
ввода-вывода kgdb, иначе драйвер ввода-вывода не будет сконфигурирован прежде, чем
ядро попросят использовать его для ожидания.

Ядро остановится и будет ждать настолько рано, насколько это позволяют драйвер
ввода-вывода и архитектура, когда вы используете эту опцию. Если вы соберёте
драйвер ввода-вывода kgdb как загружаемый модуль ядра, kgdbwait не будет делать
ничего.

Параметр ядра: ``kgdbcon``
--------------------------

Функция ``kgdbcon`` позволяет вам видеть сообщения printk() внутри gdb, пока gdb
подключён к ядру. Kdb не использует функцию kgdbcon.

Kgdb поддерживает использование последовательного протокола gdb для отправки
сообщений консоли отладчику, когда отладчик подключён и работает. Есть два
способа активировать эту функцию.

1. Активировать с помощью опции командной строки ядра::

	kgdbcon

2. Использовать sysfs до настройки драйвера ввода-вывода::

	echo 1 > /sys/module/debug_core/parameters/kgdb_use_con

.. note::

   Если вы сделаете это после того, как сконфигурируете драйвер ввода-вывода
   kgdb, настройка не вступит в силу до следующего момента, когда ввод-вывод
   будет переконфигурирован.

.. important::

   Вы не можете использовать kgdboc + kgdbcon на tty, который является активной
   системной консолью. Пример неправильного использования::

	console=ttyS0,115200 kgdboc=ttyS0 kgdbcon

Можно использовать эту опцию с kgdboc на tty, который не является системной
консолью.

Параметр времени выполнения: ``kgdbreboot``
-------------------------------------------

Функция kgdbreboot позволяет вам изменить то, как отладчик обрабатывает
уведомление о перезагрузке. У вас есть 3 варианта поведения. Поведение по
умолчанию всегда установлено в 0.

.. tabularcolumns:: |p{0.4cm}|p{11.5cm}|p{5.6cm}|

.. flat-table::
  :widths: 1 10 8

  * - 1
    - ``echo -1 > /sys/module/debug_core/parameters/kgdbreboot``
    - Полностью игнорировать уведомление о перезагрузке.

  * - 2
    - ``echo 0 > /sys/module/debug_core/parameters/kgdbreboot``
    - Отправить сообщение об отсоединении любому подключённому клиенту отладчика.

  * - 3
    - ``echo 1 > /sys/module/debug_core/parameters/kgdbreboot``
    - Войти в отладчик при уведомлении о перезагрузке.

Параметр ядра: ``nokaslr``
--------------------------

Если используемая вами архитектура включает KASLR по умолчанию, вам стоит
рассмотреть возможность его отключения. KASLR рандомизирует виртуальный адрес,
по которому отображается образ ядра, и сбивает с толку gdb, который разрешает
адреса символов ядра из таблицы символов vmlinux.

Параметр ядра: ``rodata``
-------------------------

``CONFIG_STRICT_KERNEL_RWX`` включён по умолчанию и не виден в menuconfig на
некоторых архитектурах (например, arm64); в этом случае вы можете передать ядру
``rodata=off``.

Использование kdb
=================

Быстрый старт для kdb на последовательном порту
-----------------------------------------------

Это быстрый пример того, как использовать kdb.

1. Сконфигурируйте kgdboc при загрузке, используя параметры ядра::

	console=ttyS0,115200 kgdboc=ttyS0,115200 nokaslr

   ИЛИ

   Сконфигурируйте kgdboc после того, как ядро загрузилось; предполагая, что вы
   используете консоль последовательного порта::

	echo ttyS0 > /sys/module/kgdboc/parameters/kgdboc

2. Войдите в отладчик ядра вручную или дождавшись oops или ошибки (fault). Есть
   несколько способов войти в отладчик ядра вручную; все они связаны с
   использованием `SysRq-G`, что означает, что вы должны были включить
   ``CONFIG_MAGIC_SYSRQ=y`` в конфигурации ядра.

   -  Войдя в систему как root или с сеансом суперпользователя, вы можете
      выполнить::

	echo g > /proc/sysrq-trigger

   -  Пример с использованием minicom 2.2

      Нажмите: `CTRL-A` `f` `g`

   -  Когда вы подключились по telnet к терминальному серверу, который
      поддерживает отправку удалённого break

      Нажмите: `CTRL-]`

      Наберите: ``send break``

      Нажмите: `Enter` `g`

3. Из приглашения kdb вы можете выполнить команду ``help``, чтобы увидеть полный
   список доступных команд.

   Некоторые полезные команды в kdb включают:

   =========== =================================================================
   ``lsmod``   Показывает, где загружены модули ядра
   ``ps``      Отображает только активные процессы
   ``ps A``    Показывает все процессы
   ``summary`` Показывает информацию о версии ядра и использовании памяти
   ``bt``      Получить обратную трассировку текущего процесса через dump_stack()
   ``dmesg``   Просмотреть буфер системного журнала ядра
   ``go``      Продолжить работу системы
   =========== =================================================================

4. Когда вы закончите использовать kdb, вам нужно рассмотреть перезагрузку
   системы или использование команды ``go`` для возобновления нормального
   выполнения ядра. Если вы приостановили ядро на длительный период времени,
   приложения, зависящие от своевременной работы сети или от чего-либо,
   связанного с реальным астрономическим временем, могут быть негативно затронуты,
   поэтому вам следует учитывать это при использовании отладчика ядра.

Быстрый старт для kdb с использованием консоли, подключённой к клавиатуре
-------------------------------------------------------------------------

Это быстрый пример того, как использовать kdb с клавиатурой.

1. Сконфигурируйте kgdboc при загрузке, используя параметры ядра::

	kgdboc=kbd

   ИЛИ

   Сконфигурируйте kgdboc после того, как ядро загрузилось::

	echo kbd > /sys/module/kgdboc/parameters/kgdboc

2. Войдите в отладчик ядра вручную или дождавшись oops или ошибки (fault). Есть
   несколько способов войти в отладчик ядра вручную; все они связаны с
   использованием `SysRq-G`, что означает, что вы должны были включить
   ``CONFIG_MAGIC_SYSRQ=y`` в конфигурации ядра.

   -  Войдя в систему как root или с сеансом суперпользователя, вы можете
      выполнить::

	echo g > /proc/sysrq-trigger

   -  Пример с использованием клавиатуры ноутбука:

      Нажмите и удерживайте: `Alt`

      Нажмите и удерживайте: `Fn`

      Нажмите и отпустите клавишу с надписью: `SysRq`

      Отпустите: `Fn`

      Нажмите и отпустите: `g`

      Отпустите: `Alt`

   -  Пример с использованием 101-клавишной клавиатуры PS/2

      Нажмите и удерживайте: `Alt`

      Нажмите и отпустите клавишу с надписью: `SysRq`

      Нажмите и отпустите: `g`

      Отпустите: `Alt`

3. Теперь наберите команду kdb, такую как ``help``, ``dmesg``, ``bt`` или
   ``go``, чтобы продолжить выполнение ядра.

Использование kgdb / gdb
========================

Чтобы использовать kgdb, вы должны активировать его, передав конфигурационную
информацию одному из драйверов ввода-вывода kgdb. Если вы не передадите никакой
конфигурационной информации, kgdb вообще ничего не будет делать. Kgdb будет
активно подключаться к хукам ловушек (trap hooks) ядра только если драйвер
ввода-вывода kgdb загружен и сконфигурирован. Если вы отмените конфигурацию
драйвера ввода-вывода kgdb, kgdb отменит регистрацию всех точек подключения к
ядру.

Все драйверы ввода-вывода kgdb могут быть переконфигурированы во время выполнения,
если включены ``CONFIG_SYSFS`` и ``CONFIG_MODULES``, путём вывода (echo) новой
строки конфигурации в ``/sys/module/<driver>/parameter/<option>``. Конфигурацию
драйвера можно отменить, передав пустую строку. Вы не можете изменить конфигурацию,
пока отладчик присоединён. Обязательно отсоедините отладчик командой ``detach``,
прежде чем пытаться отменить конфигурацию драйвера ввода-вывода kgdb.

Подключение с помощью gdb к последовательному порту
---------------------------------------------------

1. Сконфигурируйте kgdboc

   Сконфигурируйте kgdboc при загрузке, используя параметры ядра::

	kgdboc=ttyS0,115200

   ИЛИ

   Сконфигурируйте kgdboc после того, как ядро загрузилось::

	echo ttyS0 > /sys/module/kgdboc/parameters/kgdboc

2. Остановите выполнение ядра (войдите в отладчик)

   Чтобы подключиться к gdb через kgdboc, ядро сначала должно быть остановлено.
   Есть несколько способов остановить ядро, включающие использование kgdbwait
   как аргумента загрузки, через `SysRq-G`, или запуск ядра до тех пор, пока оно
   не получит исключение, где оно ждёт присоединения отладчика.

   -  Войдя в систему как root или с сеансом суперпользователя, вы можете
      выполнить::

	echo g > /proc/sysrq-trigger

   -  Пример с использованием minicom 2.2

      Нажмите: `CTRL-A` `f` `g`

   -  Когда вы подключились по telnet к терминальному серверу, который
      поддерживает отправку удалённого break

      Нажмите: `CTRL-]`

      Наберите: ``send break``

      Нажмите: `Enter` `g`

3. Подключитесь из gdb

   Пример (с использованием напрямую подключённого порта)::

           % gdb ./vmlinux
           (gdb) set serial baud 115200
           (gdb) target remote /dev/ttyS0


   Пример (kgdb к терминальному серверу на TCP-порту 2012)::

           % gdb ./vmlinux
           (gdb) target remote 192.168.2.2:2012


   После подключения вы можете отлаживать ядро так же, как отлаживали бы
   прикладную программу.

   Если у вас возникают проблемы с подключением или что-то идёт серьёзно не так
   во время отладки, чаще всего вам понадобится включить подробный вывод gdb о
   его взаимодействии с целью. Вы делаете это перед подачей команды
   ``target remote``, набирая::

	set debug remote 1

Помните, что если вы продолжите в gdb и вам нужно снова «вторгнуться» (break in),
вам нужно подать ещё один `SysRq-G`. Легко создать простую точку входа, поместив
точку останова на ``sys_sync``, и тогда вы сможете запустить ``sync`` из
командной оболочки или скрипта, чтобы войти в отладчик.

Совместимость kgdb и kdb
========================

Можно динамически переключаться между kdb и kgdb. Ядро отладки (debug core)
запомнит, который из них вы использовали в последний раз, и автоматически
запустится в том же режиме.

Переключение между kdb и kgdb
-----------------------------

Переключение из kgdb в kdb
~~~~~~~~~~~~~~~~~~~~~~~~~~~

Есть два способа переключиться из kgdb в kdb: вы можете использовать gdb для
подачи обслуживающего пакета (maintenance packet), или вы можете вслепую набрать
команду ``$3#33``. Каждый раз, когда отладчик ядра останавливается в режиме kgdb,
он будет выводить сообщение ``KGDB or $3#33 for KDB``. Важно отметить, что вы
должны набрать последовательность правильно за один проход. Вы не можете нажать
backspace или delete, потому что kgdb интерпретирует это как часть потока отладки.

1. Перейдите из kgdb в kdb, набрав вслепую::

	$3#33

2. Перейдите из kgdb в kdb с помощью gdb::

	maintenance packet 3

   .. note::

     Теперь вы должны завершить (kill) gdb. Обычно вы нажимаете `CTRL-Z` и
     подаёте команду::

	kill -9 %

Переход из kdb в kgdb
~~~~~~~~~~~~~~~~~~~~~~

Есть два способа перейти из kdb в kgdb. Вы можете вручную войти в режим kgdb,
подав команду kgdb из приглашения командной оболочки kdb, или вы можете подключить
gdb, пока приглашение командной оболочки kdb активно. Командная оболочка kdb ищет
типичные первые команды, которые gdb подал бы по удалённому протоколу gdb, и если
она видит одну из этих команд, она автоматически переключается в режим kgdb.

1. Из kdb подайте команду::

	kgdb

2. В приглашении kdb отключите терминальную программу и подключите на её место
   gdb.

Запуск команд kdb из gdb
------------------------

Можно запускать ограниченный набор команд kdb из gdb, используя команду monitor
gdb. Вам не стоит выполнять какие-либо операции управления ходом выполнения или
точек останова, поскольку это может нарушить состояние отладчика ядра. Вам
следует использовать gdb для точек останова и операций управления ходом
выполнения, если у вас подключён gdb. Более полезные команды для запуска — это
такие вещи, как lsmod, dmesg, ps или, возможно, некоторые из команд информации о
памяти. Чтобы увидеть все команды kdb, вы можете запустить ``monitor help``.

Пример::

    (gdb) monitor ps
    1 idle process (state I) and
    27 sleeping system daemon (state M) processes suppressed,
    use 'ps A' to see all.
    Task Addr       Pid   Parent [*] cpu State Thread     Command

    0xc78291d0        1        0  0    0   S  0xc7829404  init
    0xc7954150      942        1  0    0   S  0xc7954384  dropbear
    0xc78789c0      944        1  0    0   S  0xc7878bf4  sh
    (gdb)

Набор тестов kgdb
=================

Когда kgdb включён в конфигурации ядра, вы также можете выбрать включение
параметра конфигурации ``KGDB_TESTS``. Его включение задействует специальный
модуль ввода-вывода kgdb, который спроектирован для тестирования внутренних
функций kgdb.

Тесты kgdb в основном предназначены для разработчиков, чтобы тестировать
внутреннее устройство kgdb, а также как инструмент для разработки новой
архитектурно-специфичной реализации kgdb. Эти тесты на самом деле не для
конечных пользователей ядра Linux. Основным источником документации было бы
изучение файла ``drivers/misc/kgdbts.c``.

Набор тестов kgdb также может быть сконфигурирован на этапе компиляции для
запуска основного набора тестов путём установки параметра конфигурации ядра
``KGDB_TESTS_ON_BOOT``. Эта конкретная опция нацелена на автоматизированное
регрессионное тестирование и не требует изменения аргументов загрузки ядра. Если
это включено, набор тестов kgdb можно отключить, указав ``kgdbts=`` как аргумент
загрузки ядра.

Внутреннее устройство отладчика ядра
====================================

Особенности архитектур
-----------------------

Отладчик ядра организован в ряд компонентов:

1. Ядро отладки (debug core)

   Ядро отладки находится в ``kernel/debugger/debug_core.c``. Оно содержит:

   -  Обобщённый обработчик исключений ОС, который включает синхронизацию
      процессоров в остановленное состояние на многопроцессорной системе.

   -  API для взаимодействия с драйверами ввода-вывода kgdb

   -  API для выполнения вызовов архитектурно-специфичной реализации kgdb

   -  Логику для выполнения безопасного чтения и записи в память при
      использовании отладчика

   -  Полную реализацию программных точек останова, если она не переопределена
      архитектурой

   -  API для вызова фронтенда kdb или kgdb к ядру отладки.

   -  Структуры и API обратных вызовов для atomic kernel mode setting.

      .. note:: kgdboc — это место, где вызываются обратные вызовы kms.

2. Архитектурно-специфичная реализация kgdb

   Эта реализация обычно находится в ``arch/*/kernel/kgdb.c``. Например,
   ``arch/x86/kernel/kgdb.c`` содержит специфику для реализации аппаратных точек
   останова, а также инициализацию для динамической регистрации и отмены
   регистрации обработчиков ловушек (trap handlers) на этой архитектуре.
   Архитектурно-специфичная часть реализует:

   -  содержит архитектурно-специфичный перехватчик ловушек, который вызывает
      kgdb_handle_exception() для запуска работы kgdb

   -  преобразование в и из gdb-специфичного формата пакетов в struct pt_regs

   -  Регистрацию и отмену регистрации архитектурно-специфичных хуков ловушек

   -  Любую специальную обработку исключений и очистку

   -  Обработку исключений NMI и очистку

   -  (необязательно) Аппаратные точки останова

3. Фронтенд gdbstub (он же kgdb)

   Gdbstub расположен в ``kernel/debug/gdbstub.c``. Он содержит:

   -  Всю логику для реализации последовательного протокола gdb

4. Фронтенд kdb

   Командная оболочка отладчика kdb разбита на ряд компонентов. Ядро kdb
   расположено в kernel/debug/kdb. Есть ряд вспомогательных функций в некоторых
   из других компонентов ядра, чтобы сделать возможным для kdb исследование и
   отчёт информации о ядре без захвата блокировок, которые могли бы вызвать
   взаимоблокировку (deadlock) ядра. Ядро kdb реализует следующую
   функциональность.

   -  Простую командную оболочку

   -  Базовый набор команд kdb

   -  API регистрации для регистрации дополнительных команд командной оболочки
      kdb.

      -  Хорошим примером самодостаточного модуля kdb является команда
         ``ftdump`` для дампа буфера ftrace. См.:
         ``kernel/trace/trace_kdb.c``

      -  В качестве примера того, как динамически регистрировать новую команду
         kdb, вы можете собрать модуль ядра kdb_hello.ko из
         ``samples/kdb/kdb_hello.c``. Чтобы собрать этот пример, вы можете
         установить ``CONFIG_SAMPLES=y`` и ``CONFIG_SAMPLE_KDB=m`` в вашей
         конфигурации ядра. Позже запустите ``modprobe kdb_hello``, и в
         следующий раз, когда вы войдёте в командную оболочку kdb, вы сможете
         выполнить команду ``hello``.

   -  Реализацию kdb_printf(), которая выдаёт сообщения непосредственно драйверам
      ввода-вывода, минуя журнал ядра.

   -  Управление программными / аппаратными точками останова для командной
      оболочки kdb

5. Драйвер ввода-вывода kgdb

   Каждый драйвер ввода-вывода kgdb должен предоставить реализацию для
   следующего:

   -  конфигурация через встроенный компонент или модуль

   -  динамическая конфигурация и вызовы регистрации хуков kgdb

   -  интерфейс чтения и записи символов

   -  Обработчик очистки для отмены конфигурации из ядра kgdb

   -  (необязательно) Методология ранней отладки

   Любой данный драйвер ввода-вывода kgdb должен работать очень тесно с
   оборудованием и должен делать это таким образом, который не включает
   прерывания и не изменяет другие части контекста системы без их полного
   восстановления. Ядро kgdb будет многократно «опрашивать» (poll) драйвер
   ввода-вывода kgdb на наличие символов, когда ему нужен ввод. Ожидается, что
   драйвер ввода-вывода вернётся немедленно, если нет доступных данных. Это
   позволяет в будущем затрагивать (touch) оборудование сторожевого таймера
   (watchdog) таким образом, чтобы целевая система не перезагружалась, когда они
   включены.

Если вы намерены добавить архитектурно-специфичную поддержку kgdb для новой
архитектуры, архитектура должна определить ``HAVE_ARCH_KGDB`` в
архитектурно-специфичном файле Kconfig. Это включит kgdb для архитектуры, и в
этот момент вы должны создать архитектурно-специфичную реализацию kgdb.

Есть несколько флагов, которые должны быть установлены на каждой архитектуре в
их файле ``asm/kgdb.h``. Это:

-  ``NUMREGBYTES``:
     Размер в байтах всех регистров, чтобы мы могли убедиться, что все они
     поместятся в пакет.

-  ``BUFMAX``:
     Размер в байтах буфера, в который GDB будет читать. Он должен быть больше,
     чем NUMREGBYTES.

-  ``CACHE_FLUSH_IS_SAFE``:
     Установите в 1, если всегда безопасно вызывать flush_cache_range или
     flush_icache_range. На некоторых архитектурах эти функции может быть
     небезопасно вызывать на SMP, поскольку мы удерживаем другие CPU в режиме
     ожидания (holding pattern).

Есть также следующие функции для общего бэкенда, находящегося в
``kernel/kgdb.c``, которые должны быть предоставлены архитектурно-специфичным
бэкендом, если только они не помечены как (необязательно), в случае чего может
использоваться функция по умолчанию, если архитектуре не нужно предоставлять
специфичную реализацию.

Внутреннее устройство kgdboc
----------------------------

kgdboc и uart-ы
~~~~~~~~~~~~~~~~

Драйвер kgdboc на самом деле является очень тонким драйвером, который опирается
на то, что нижележащий низкоуровневый драйвер оборудования имеет «хуки опроса»
(polling hooks), к которым присоединён драйвер tty. В первоначальной реализации
kgdboc serial_core был изменён, чтобы предоставить низкоуровневый хук UART для
выполнения чтения и записи одиночного символа в режиме опроса (polled mode),
находясь в атомарном контексте. Когда kgdb делает запрос ввода-вывода к отладчику,
kgdboc вызывает обратный вызов в serial core, который, в свою очередь, использует
обратный вызов в драйвере UART.

При использовании kgdboc с UART драйвер UART должен реализовать два обратных
вызова в struct uart_ops.
Пример из ``drivers/8250.c``::


    #ifdef CONFIG_CONSOLE_POLL
        .poll_get_char = serial8250_get_poll_char,
        .poll_put_char = serial8250_put_poll_char,
    #endif


Любая специфика реализации, связанная с созданием драйвера опроса, использует
``#ifdef CONFIG_CONSOLE_POLL``, как показано выше. Имейте в виду, что хуки опроса
должны быть реализованы таким образом, чтобы их можно было вызывать из атомарного
контекста, и они должны при возврате восстанавливать состояние микросхемы UART
так, чтобы система могла вернуться к нормальной работе, когда отладчик
отсоединяется. Вам нужно быть очень осторожными с любого рода блокировкой, которую
вы рассматриваете, потому что ошибка здесь, скорее всего, будет означать нажатие
кнопки сброса (reset).

kgdboc и клавиатуры
~~~~~~~~~~~~~~~~~~~~

Драйвер kgdboc содержит логику для конфигурации связи с присоединённой
клавиатурой. Инфраструктура клавиатуры компилируется в ядро, только когда в
конфигурации ядра установлено ``CONFIG_KDB_KEYBOARD=y``.

Основной драйвер опрашиваемой клавиатуры для клавиатур типа PS/2 находится в
``drivers/char/kdb_keyboard.c``. Этот драйвер подключается к ядру отладки, когда
kgdboc заполняет обратный вызов в массиве, называемом
:c:expr:`kdb_poll_funcs[]`. kdb_get_kbd_char() — это функция верхнего уровня,
которая опрашивает оборудование на наличие ввода одиночного символа.

kgdboc и kms
~~~~~~~~~~~~

Драйвер kgdboc содержит логику для запроса графического дисплея переключиться в
текстовый контекст, когда вы используете ``kgdboc=kms,kbd``, при условии, что у
вас есть видеодрайвер, который имеет консоль на кадровом буфере (frame buffer
console) и поддержку atomic kernel mode setting.

Каждый раз, когда выполняется вход в отладчик ядра, он вызывает
kgdboc_pre_exp_handler(), который, в свою очередь, вызывает con_debug_enter()
в слое виртуальной консоли. При возобновлении выполнения ядра отладчик ядра
вызывает kgdboc_post_exp_handler(), который, в свою очередь, вызывает
con_debug_leave().
