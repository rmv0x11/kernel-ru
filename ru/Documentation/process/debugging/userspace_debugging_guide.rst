==========================================================
Советы по отладке из пространства пользователя (userspace)
==========================================================

Этот документ содержит краткий обзор распространённых инструментов для отладки
ядра Linux из пространства пользователя (userspace).
Советы по отладке, ориентированные на разработчиков драйверов, смотрите
:doc:`здесь
</process/debugging/driver_development_debugging_guide>`.
Общие советы по отладке смотрите в :doc:`документе с общими советами
</process/debugging/index>`.

.. contents::
    :depth: 3

В следующих разделах описаны доступные инструменты.

Dynamic debug
-------------

Механизм, позволяющий фильтровать то, что попадает в журнал ядра, отключая или
включая сообщения журнала.

Необходимое условие: ``CONFIG_DYNAMIC_DEBUG``

Dynamic debug способен воздействовать только на:

- pr_debug()
- dev_dbg()
- print_hex_dump_debug()
- print_hex_dump_bytes()

Поэтому применимость этого инструмента на данный момент довольно ограничена,
так как нет единого правила для добавления отладочных вывода в кодовую базу, что
приводит к разнообразию способов, которыми эти вывода реализованы.

Также обратите внимание, что большинство отладочных операторов реализованы как
вариация dprintk(), которые приходится активировать через параметр в
соответствующем модуле; dynamic debug не может выполнить этот шаг за вас.

Вот один пример, который включает все доступные pr_debug() внутри файла::

  $ alias ddcmd='echo $* > /proc/dynamic_debug/control'
  $ ddcmd '-p; file v4l2-h264.c +p'
  $ grep =p /proc/dynamic_debug/control
   drivers/media/v4l2-core/v4l2-h264.c:372 [v4l2_h264]print_ref_list_b =p
   "ref_pic_list_b%u (cur_poc %u%c) %s"
   drivers/media/v4l2-core/v4l2-h264.c:333 [v4l2_h264]print_ref_list_p =p
   "ref_pic_list_p (cur_poc %u%c) %s\n"

**Когда следует использовать его вместо Ftrace?**

- Когда код содержит один из допустимых операторов вывода (см. выше) или когда
  вы добавили несколько операторов pr_debug() во время разработки
- Когда время выполнения не является проблемой, то есть если несколько
  операторов pr_debug() в коде не вызывают задержек
- Когда для вас важнее получить конкретные сообщения журнала, чем отследить
  схему вызовов функции

Полную документацию смотрите в :doc:`/admin-guide/dynamic-debug-howto`

Ftrace
------

Необходимое условие: ``CONFIG_DYNAMIC_FTRACE``

Этот инструмент использует файловую систему tracefs для управляющих файлов и
файлов вывода. Эта файловая система монтируется как каталог ``tracing``,
который можно найти либо в ``/sys/kernel/``, либо в ``/sys/debug/kernel/``.

Некоторые из наиболее важных для отладки операций:

- Вы можете выполнить трассировку функции, добавив имя функции в файл
  ``set_ftrace_filter`` (который принимает любое имя функции, найденное в файле
  ``available_filter_functions``), либо вы можете специально отключить
  определённые функции, добавив их имена в файл ``set_ftrace_notrace``
  (подробнее: :ref:`trace/ftrace:dynamic ftrace`).
- Чтобы выяснить, откуда происходят вызовы, вы можете активировать опцию
  ``func_stack_trace`` в ``options/func_stack_trace``.
- Трассировка дочерних функций вызова функции и отображение возвращаемых
  значений возможны путём добавления нужной функции в файл
  ``set_graph_function`` (требует конфигурации ``FUNCTION_GRAPH_RETVAL``);
  подробнее:
  :ref:`trace/ftrace:dynamic ftrace with the function graph tracer`.

Полную документацию по Ftrace смотрите в :doc:`/trace/ftrace`

Либо вы можете также выполнять трассировку конкретных событий, :ref:`используя
трассировку событий <trace/events:2. using event tracing>`, которая может быть
определена так, как описано здесь:
:ref:`Создание собственной точки трассировки Ftrace
<process/debugging/driver_development_debugging_guide:ftrace>`.

Полную документацию по трассировке событий Ftrace смотрите в :doc:`/trace/events`

Чтение журнала ftrace
~~~~~~~~~~~~~~~~~~~~~~

Файл ``trace`` можно читать так же, как и любой другой файл (``cat``, ``tail``,
``head``, ``vim`` и т.д.); размер файла ограничен параметром
``buffer_size_kb`` (``echo 1000 > buffer_size_kb``). Файл
:ref:`trace/ftrace:trace_pipe` ведёт себя аналогично файлу ``trace``, но при
каждом чтении из этого файла его содержимое потребляется.

Perf и альтернативы
-------------------

Упомянутые выше инструменты предоставляют способы исследовать код ядра,
результаты, значения переменных и т.д. Иногда сначала нужно выяснить, где
искать, и в таких случаях набор инструментов для отслеживания
производительности поможет вам очертить проблему.

Зачем выполнять анализ производительности?
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Анализ производительности — это хороший первый шаг, в частности, когда:

- вы не можете определить проблему
- вы не знаете, где она возникает
- работающую систему нельзя прерывать, либо это удалённая система, в которую вы
  не можете установить новый модуль/ядро

Как выполнить простой анализ с помощью инструментов Linux?
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

В начале анализа производительности можно начать с обычных инструментов, таких
как:

- ``top`` / ``htop`` / ``atop`` (*получить обзор загрузки системы, увидеть
  всплески у конкретных процессов*)
- ``mpstat -P ALL`` (*посмотреть распределение нагрузки между CPU*)
- ``iostat -x`` (*наблюдать за загрузкой и производительностью устройств ввода
  и вывода*)
- ``vmstat`` (*обзор использования памяти в системе*)
- ``pidstat`` (*аналогично* ``vmstat``\ *, но по каждому процессу, чтобы
  сузить до целевого*)
- ``strace -tp $PID`` (*как только вы узнаете процесс, можно выяснить, как он
  взаимодействует с ядром*)

Это должно помочь достаточно сузить области, на которые стоит обратить внимание.

Perfetto
~~~~~~~~

Набор инструментов для измерения и анализа того, насколько хорошо работают
приложения и системы. Его можно использовать, чтобы:

* выявлять узкие места
* оптимизировать код
* заставить программное обеспечение работать быстрее и эффективнее.

**В чём разница между perfetto и perf?**

* perf — это инструмент, являющийся частью ядра Linux и специализированный под
  него, имеющий пользовательский интерфейс командной строки (CLI).
* perfetto — кроссплатформенный стек для анализа производительности, имеющий
  расширенную функциональность в пространстве пользователя (userspace) и
  предоставляющий веб-интерфейс пользователя (WEB).

Полную документацию смотрите по адресу `<https://perfetto.dev/docs/>`__

Инструменты анализа kernel panic
--------------------------------

  Для захвата дампа аварийного завершения (crash dump) используйте ``Kdump`` и
  ``Kexec``. Ниже вы найдёте несколько советов по анализу данных.

  Полную документацию смотрите в :doc:`/admin-guide/kdump/kdump`

  Чтобы найти соответствующую строку в коде, можно использовать `faddr2line
  <https://elixir.bootlin.com/linux/v6.11.6/source/scripts/faddr2line>`__;
  обратите внимание, что для этого нужно включить ``CONFIG_DEBUG_INFO``.

  Альтернативой использованию ``faddr2line`` является использование ``objdump``
  (и его производных для различных платформ, таких как
  ``aarch64-linux-gnu-objdump``). Возьмём в качестве примера эту строку:

  ``[  +0.000240]  rkvdec_device_run+0x50/0x138 [rockchip_vdec]``.

  Мы можем найти соответствующую строку кода, выполнив::

    [  +0.000240]  rkvdec_device_run+0x50/0x138 [rockchip_vdec]

  я могу взять ``0x50`` в качестве смещения, которое нужно прибавить к базовому

    0000000000000ac8 <rkvdec_device_run>:

  Результат: ``0xac8 + 0x50 = 0xb18``.
