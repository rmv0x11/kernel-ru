==========================================
Советы по отладке при разработке драйверов
==========================================

Этот документ служит общей отправной точкой и справочником по отладке
драйверов устройств.
Хотя данное руководство сосредоточено на отладке, которая требует
перекомпиляции модуля/ядра, :doc:`руководство по отладке в пространстве
пользователя </process/debugging/userspace_debugging_guide>` проведёт
вас по таким инструментам, как dynamic debug, ftrace и другим средствам,
полезным для отладки проблем и поведения.
Общие советы по отладке см. в :doc:`документе с общими советами
</process/debugging/index>`.

.. contents::
    :depth: 3

В следующих разделах показаны доступные инструменты.

printk() и его аналоги
----------------------

Это производные от printf() с различными местами назначения вывода и с
поддержкой динамического включения и выключения либо без таковой.

Простой printk()
~~~~~~~~~~~~~~~~~

Классика; может с большим эффектом использоваться для быстрой и грубой
разработки новых модулей или для извлечения любых необходимых данных при
устранении неполадок.

Предварительное условие: ``CONFIG_PRINTK`` (обычно включён по умолчанию)

**Достоинства**:

- Не нужно ничего изучать, прост в использовании
- Легко изменить точно под ваши нужды (форматирование данных (см.:
  :doc:`/core-api/printk-formats`), видимость в журнале)
- Может вызывать задержки в исполнении кода (полезно для подтверждения того,
  является ли тайминг фактором)

**Недостатки**:

- Требует пересборки ядра/модуля
- Может вызывать задержки в исполнении кода (из-за чего проблемы могут стать
  невоспроизводимыми)

Полную документацию см. в :doc:`/core-api/printk-basics`

Trace_printk
~~~~~~~~~~~~

Предварительное условие: ``CONFIG_DYNAMIC_FTRACE`` и ``#include <linux/ftrace.h>``

Этот вариант чуть менее удобен в использовании, чем printk(), поскольку
сообщения придётся читать из файла трассировки (см.: :ref:`read_ftrace_log`
вместо журнала ядра, но он очень полезен, когда printk() добавляет нежелательные
задержки в исполнение кода, из-за чего проблемы становятся нестабильными или
скрытыми.)

Если обработка этого по-прежнему вызывает проблемы с таймингом, то можно
попробовать trace_puts().

Полную документацию см. в trace_printk()

dev_dbg
~~~~~~~

Оператор печати, на который можно нацелиться при помощи
:ref:`process/debugging/userspace_debugging_guide:dynamic debug` и который
содержит дополнительную информацию об устройстве, используемом в данном контексте.

**Когда уместно оставлять отладочную печать в коде?**

Постоянные отладочные операторы должны быть полезны разработчику при устранении
неполадок в работе драйвера. Оценка этого — в большей степени искусство, чем
наука, но некоторые рекомендации приведены в :ref:`рекомендациях по стилю
кодирования <process/coding-style:13) Печать сообщений ядра>`. Почти во всех
случаях отладочные операторы не следует отправлять в апстрим, поскольку
работающий драйвер должен быть «тихим».

Собственный printk
~~~~~~~~~~~~~~~~~~~

Пример::

  #define core_dbg(fmt, arg...) do { \
	  if (core_debug) \
		  printk(KERN_DEBUG pr_fmt("core: " fmt), ## arg); \
	  } while (0)

**Когда стоит так поступать?**

Лучше просто использовать pr_debug(), который позже можно включать/выключать с
помощью dynamic debug. Кроме того, многие драйверы активируют такие операторы
печати через переменную вроде ``core_debug``, задаваемую параметром модуля.
Однако параметры модуля `больше не рекомендуются
<https://lore.kernel.org/all/2024032757-surcharge-grime-d3dd@gregkh>`_.

Ftrace
------

Создание собственной точки трассировки Ftrace
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Точка трассировки добавляет в ваш код перехватчик, который будет вызываться и
логироваться, когда точка трассировки включена. Это можно использовать,
например, для трассировки попадания в условную ветвь или для дампа внутреннего
состояния в определённых местах потока выполнения кода во время сеанса отладки.

Вот базовое описание того, :ref:`как реализовать новые точки трассировки
<trace/tracepoints:usage>`.

Полную документацию по трассировке событий см. в :doc:`/trace/events`

Полную документацию по Ftrace см. в :doc:`/trace/ftrace`

DebugFS
-------

Предварительное условие: ``CONFIG_DEBUG_FS` & `#include <linux/debugfs.h>``

DebugFS отличается от других подходов к отладке тем, что не записывает сообщения
в журнал ядра и не добавляет трассировки в код. Вместо этого он позволяет
разработчику работать с набором файлов.
С помощью этих файлов можно либо хранить значения переменных, либо делать дампы
регистров/памяти, либо сделать эти файлы доступными для записи и изменять
значения/настройки в драйвере.

Возможные варианты использования, среди прочего:

- Хранить значения регистров
- Отслеживать переменные
- Хранить ошибки
- Хранить настройки
- Переключать настройку, например включение/выключение отладки
- Инъекция ошибок

Это особенно полезно, когда размер дампа данных было бы трудно осмыслить в рамках
общего журнала ядра (например, при дампе сырых данных битового потока) или когда
вас не интересуют все значения всё время, но при этом есть возможность их
проверить.

Общая идея такова:

- Создать каталог во время probe (``struct dentry *parent =
  debugfs_create_dir("my_driver", NULL);``)
- Создать файл (``debugfs_create_u32("my_value", 444, parent, &my_variable);``)

  - В этом примере файл находится в
    ``/sys/kernel/debug/my_driver/my_value`` (с правами на чтение для
    пользователя/группы/всех)
  - любое чтение файла вернёт текущее содержимое переменной
    ``my_variable``

- Очистить каталог при удалении устройства
  (``debugfs_remove(parent);``)

Полную документацию см. в :doc:`/filesystems/debugfs`.

KASAN, UBSAN, lockdep и другие средства проверки ошибок
-------------------------------------------------------

KASAN (Kernel Address Sanitizer)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Предварительное условие: ``CONFIG_KASAN``

KASAN — это динамический детектор ошибок памяти, который помогает находить
ошибки use-after-free и выхода за границы (out-of-bounds). Он использует
инструментирование на этапе компиляции для проверки каждого доступа к памяти.

Полную документацию см. в :doc:`/dev-tools/kasan`.

UBSAN (Undefined Behavior Sanitizer)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

UBSAN опирается на инструментирование компилятором и проверки во время
выполнения для обнаружения неопределённого поведения. Он предназначен для поиска
разнообразных проблем, включая переполнение знакового целого, выход индекса
массива за границы и другие.

Полную документацию см. в :doc:`/dev-tools/ubsan`

lockdep (Lock Dependency Validator)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Предварительное условие: ``CONFIG_DEBUG_LOCKDEP``

lockdep — это валидатор зависимостей блокировок во время выполнения, который
обнаруживает потенциальные взаимоблокировки (deadlock) и другие проблемы,
связанные с блокировками, в ядре.
Он отслеживает захваты и освобождения блокировок, строя граф зависимостей,
который анализируется на предмет потенциальных взаимоблокировок.
lockdep особенно полезен для проверки корректности порядка взятия блокировок в
ядре.

PSI (Pressure stall information tracking)
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Предварительное условие: ``CONFIG_PSI``

PSI — это инструмент измерения для выявления чрезмерного переподписания
аппаратных ресурсов, которое может вызывать сбои производительности или даже
OOM-завершения процессов.

device coredump
---------------

Предварительное условие: ``CONFIG_DEV_COREDUMP`` и ``#include <linux/devcoredump.h>``

Предоставляет драйверу инфраструктуру для передачи произвольных данных в
пространство пользователя.
Чаще всего используется совместно с udev или аналогичным приложением
пространства пользователя для прослушивания uevents ядра, которые сигнализируют,
что дамп готов. У udev есть правила для копирования этого файла куда-либо для
долговременного хранения и анализа, поскольку по умолчанию данные дампа
автоматически удаляются по истечении заданных по умолчанию 5 минут. Эти данные
анализируются с помощью специфичных для драйвера инструментов или GDB.

device coredump можно создать с помощью области vmalloc, с методами
чтения/освобождения или в виде списка scatter/gather.

Пример реализации можно найти в:
`drivers/media/platform/qcom/venus/core.c
<https://elixir.bootlin.com/linux/v6.11.6/source/drivers/media/platform/qcom/venus/core.c#L30>`__,
в слое Bluetooth HCI, в нескольких беспроводных драйверах и в нескольких
DRM-драйверах.
