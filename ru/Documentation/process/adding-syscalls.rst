
.. _addsyscalls:

Добавление нового системного вызова
===================================

Этот документ описывает, что требуется для добавления нового системного вызова
в ядро Linux, помимо обычных рекомендаций по отправке изменений, изложенных в
:ref:`Documentation/process/submitting-patches.rst <submittingpatches>`.


Альтернативы системным вызовам
------------------------------

Первое, о чём стоит подумать при добавлении нового системного вызова, — не
подойдёт ли вместо него одна из альтернатив. Хотя системные вызовы являются
наиболее традиционными и наиболее очевидными точками взаимодействия между
пространством пользователя (userspace) и ядром, существуют и другие
возможности — выбирайте то, что лучше всего подходит для вашего интерфейса.

 - Если задействованные операции можно представить в виде объекта, похожего на
   файловую систему, то имеет больше смысла создать новую файловую систему или
   устройство. Это также упрощает инкапсуляцию новой функциональности в модуле
   ядра, а не требование встраивать её в основное ядро.

     - Если новая функциональность включает операции, в которых ядро уведомляет
       пространство пользователя о том, что что-то произошло, то возврат нового
       файлового дескриптора для соответствующего объекта позволяет
       пространству пользователя использовать ``poll``/``select``/``epoll`` для
       получения такого уведомления.
     - Однако операции, которые не отображаются на операции вида
       :manpage:`read(2)`/:manpage:`write(2)`,
       приходится реализовывать как запросы :manpage:`ioctl(2)`, что может
       привести к несколько непрозрачному API.

 - Если вы просто предоставляете доступ к системной информации времени
   выполнения, более уместным может оказаться новый узел в sysfs
   (см. ``Documentation/filesystems/sysfs.rst``) или в файловой системе
   ``/proc``. Однако доступ к этим механизмам требует, чтобы соответствующая
   файловая система была смонтирована, что не всегда так (например, в
   изолированном по пространствам имён / песочнице / chroot-окружении).
   Избегайте добавления какого-либо API в debugfs, так как это не считается
   «производственным» (production) интерфейсом для пространства пользователя.
 - Если операция специфична для конкретного файла или файлового дескриптора, то
   более уместной может оказаться дополнительная командная опция
   :manpage:`fcntl(2)`. Однако
   :manpage:`fcntl(2)` — это мультиплексирующий системный вызов, скрывающий
   большую сложность, поэтому
   этот вариант лучше подходит для случаев, когда новая функция тесно
   аналогична существующей функциональности :manpage:`fcntl(2)`, либо когда
   новая функциональность очень проста
   (например, получение/установка простого флага, связанного с файловым
   дескриптором).
 - Если операция специфична для конкретной задачи или процесса, то более
   уместной может оказаться дополнительная командная опция :manpage:`prctl(2)`.
   Как и
   :manpage:`fcntl(2)`, этот системный вызов является сложным мультиплексором,
   поэтому его лучше зарезервировать для близких аналогов существующих команд
   ``prctl()`` или для получения/установки простого флага, связанного с
   процессом.


Проектирование API: планирование расширяемости
----------------------------------------------

Новый системный вызов становится частью API ядра и должен поддерживаться
неограниченно долго. Поэтому очень хорошей идеей является явное обсуждение
интерфейса в списке рассылки ядра, и важно планировать будущие расширения
интерфейса.

(Таблица системных вызовов изобилует историческими примерами, где этого не было
сделано, вместе с соответствующими дополнительными системными вызовами —
``eventfd``/``eventfd2``, ``dup2``/``dup3``, ``inotify_init``/``inotify_init1``,
``pipe``/``pipe2``, ``renameat``/``renameat2`` — поэтому
учитесь на истории ядра и планируйте расширения с самого начала.)

Для более простых системных вызовов, принимающих лишь пару аргументов,
предпочтительным способом обеспечить будущую расширяемость является включение
в системный вызов аргумента флагов. Чтобы убедиться, что программы из
пространства пользователя могут безопасно использовать флаги между версиями
ядра, проверяйте, не содержит ли значение флагов какие-либо неизвестные флаги,
и отклоняйте системный вызов (с ``EINVAL``), если это так::

    if (flags & ~(THING_FLAG1 | THING_FLAG2 | THING_FLAG3))
        return -EINVAL;

(Если никакие значения флагов ещё не используются, проверяйте, что аргумент
флагов равен нулю.)

Для более сложных системных вызовов, включающих большее число аргументов,
предпочтительно инкапсулировать большинство аргументов в структуру, которая
передаётся по указателю. Такая структура может справиться с будущим расширением
путём включения в неё аргумента размера::

    struct xyzzy_params {
        u32 size; /* userspace sets p->size = sizeof(struct xyzzy_params) */
        u32 param_1;
        u64 param_2;
        u64 param_3;
    };

Пока любое впоследствии добавленное поле, скажем ``param_4``, спроектировано так,
что нулевое значение даёт прежнее поведение, это позволяет обрабатывать
несовпадение версий в обоих направлениях:

 - Чтобы справиться с более новой программой из пространства пользователя,
   вызывающей более старое ядро, код ядра должен проверять, что любая память за
   пределами ожидаемого им размера структуры равна нулю (по сути, проверяя, что
   ``param_4 == 0``).
 - Чтобы справиться с более старой программой из пространства пользователя,
   вызывающей более новое ядро, код ядра может дополнить нулями меньший
   экземпляр структуры (по сути, устанавливая ``param_4 = 0``).

Пример такого подхода см. в :manpage:`perf_event_open(2)` и функции
``perf_copy_attr()`` (в ``kernel/events/core.c``).


Проектирование API: прочие соображения
--------------------------------------

Если ваш новый системный вызов позволяет пространству пользователя ссылаться на
объект ядра, в качестве дескриптора (handle) этого объекта следует использовать
файловый дескриптор — не изобретайте новый тип дескриптора объекта в
пространстве пользователя, когда в ядре уже есть механизмы и чётко определённая
семантика использования файловых дескрипторов.

Если ваш новый системный вызов xyzzy(2) действительно возвращает новый файловый
дескриптор, то аргумент флагов должен включать значение, эквивалентное
установке ``O_CLOEXEC`` на новом FD. Это даёт пространству пользователя
возможность закрыть временное окно между ``xyzzy()`` и вызовом
``fcntl(fd, F_SETFD, FD_CLOEXEC)``, в котором неожиданные ``fork()`` и
``execve()`` в другом потоке могли бы утечь дескриптор в
запущенную через exec программу. (Однако сопротивляйтесь искушению повторно
использовать фактическое значение константы ``O_CLOEXEC``, поскольку оно
зависит от архитектуры и является частью пространства нумерации флагов ``O_*``,
которое уже довольно заполнено.)

Если ваш системный вызов возвращает новый файловый дескриптор, вам также следует
обдумать, что означает использование семейства системных вызовов
:manpage:`poll(2)` на этом файловом
дескрипторе. Перевод файлового дескриптора в состояние готовности к чтению или
записи — это обычный способ для ядра указать пространству пользователя, что на
соответствующем объекте ядра произошло событие.

Если ваш новый системный вызов xyzzy(2) включает аргумент имени файла::

    int sys_xyzzy(const char __user *path, ..., unsigned int flags);

вам также следует обдумать, не будет ли более уместной версия xyzzyat(2)::

    int sys_xyzzyat(int dfd, const char __user *path, ..., unsigned int flags);

Это даёт больше гибкости в том, как пространство пользователя указывает
рассматриваемый файл; в частности, это позволяет пространству пользователя
запросить функциональность для уже открытого файлового дескриптора с помощью
флага ``AT_EMPTY_PATH``, по сути бесплатно предоставляя операцию fxyzzy(3)::

 - xyzzyat(AT_FDCWD, path, ..., 0) эквивалентно xyzzy(path,...)
 - xyzzyat(fd, "", ..., AT_EMPTY_PATH) эквивалентно fxyzzy(fd, ...)

(Более подробно об обосновании вызовов \*at() см. man-страницу
:manpage:`openat(2)`; пример AT_EMPTY_PATH см. на man-странице
:manpage:`fstatat(2)`.)

Если ваш новый системный вызов xyzzy(2) включает параметр, описывающий
смещение внутри файла, сделайте его тип ``loff_t``, чтобы 64-битные смещения
могли поддерживаться даже на 32-битных архитектурах.

Если ваш новый системный вызов xyzzy(2) включает привилегированную
функциональность, она должна управляться соответствующим битом возможностей
(capability) Linux (проверяемым вызовом
``capable()``), как описано на man-странице :manpage:`capabilities(7)`.
Выбирайте существующий бит возможности, управляющий связанной
функциональностью, но старайтесь избегать объединения множества лишь смутно
связанных функций под одним битом, так как это противоречит цели возможностей —
разделению полномочий root. В частности, избегайте добавления новых вариантов
использования уже излишне общей возможности ``CAP_SYS_ADMIN``.

Если ваш новый системный вызов xyzzy(2) манипулирует процессом, отличным от
вызывающего процесса, он должен быть ограничен (с помощью вызова
``ptrace_may_access()``) так, чтобы манипулировать целевым процессом мог только
вызывающий процесс с теми же правами, что и целевой процесс, или с необходимыми
возможностями.

Наконец, имейте в виду, что некоторым архитектурам, отличным от x86, проще, если
параметры системного вызова, которые явно являются 64-битными, приходятся на
нечётные по номеру аргументы (т. е. параметр 1, 3, 5), что позволяет
использовать смежные пары 32-битных
регистров. (Это соображение неприменимо, если аргументы являются частью
структуры, передаваемой по указателю.)


Предложение API
---------------

Чтобы новые системные вызовы было легко рецензировать, лучше всего разбить набор
патчей на отдельные фрагменты. Они должны включать как минимум следующие
элементы в виде отдельных коммитов (каждый из которых далее описывается ниже):

 - Основная реализация системного вызова вместе с прототипами,
   обобщённой нумерацией, изменениями Kconfig и резервной заглушкой реализации.
 - Подключение нового системного вызова для одной конкретной архитектуры, обычно
   x86 (включая все из x86_64, x86_32 и x32).
 - Демонстрация использования нового системного вызова в пространстве
   пользователя через самопроверку (selftest) в ``tools/testing/selftests/``.
 - Черновик man-страницы для нового системного вызова, либо как обычный текст в
   сопроводительном письме, либо как патч к (отдельному) репозиторию man-страниц.

Предложения новых системных вызовов, как и любое изменение API ядра, всегда
должны отправляться копией (cc) на linux-api@vger.kernel.org.


Обобщённая реализация системного вызова
---------------------------------------

Главная точка входа для вашего нового системного вызова xyzzy(2) будет называться
``sys_xyzzy()``, но вы добавляете эту точку входа с помощью соответствующего
макроса ``SYSCALL_DEFINEn()``, а не явно. Буква «n» обозначает
число аргументов системного вызова, и макрос принимает в качестве аргументов имя
системного вызова, за которым следуют пары (тип, имя) для параметров. Использование
этого макроса позволяет сделать метаданные о новом системном вызове доступными
для других инструментов.

Новой точке входа также нужен соответствующий прототип функции в
``include/linux/syscalls.h``, помеченный как asmlinkage, чтобы соответствовать
тому, как вызываются системные вызовы::

    asmlinkage long sys_xyzzy(...);

Некоторые архитектуры (например, x86) имеют собственные специфичные для
архитектуры таблицы системных вызовов, но несколько других архитектур разделяют
обобщённую таблицу системных вызовов. Добавьте ваш новый системный вызов в
обобщённый список, добавив запись в список в
``include/uapi/asm-generic/unistd.h``::

    #define __NR_xyzzy 292
    __SYSCALL(__NR_xyzzy, sys_xyzzy)

Также обновите счётчик __NR_syscalls, чтобы отразить дополнительный системный
вызов, и учтите, что если несколько новых системных вызовов добавляются в одном
и том же окне слияния (merge window), номер вашего нового системного вызова
может быть скорректирован для разрешения конфликтов.

Файл ``kernel/sys_ni.c`` предоставляет резервную заглушку реализации каждого
системного вызова, возвращающую ``-ENOSYS``. Добавьте ваш новый системный вызов
и сюда::

    COND_SYSCALL(xyzzy);

Ваша новая функциональность ядра и системный вызов, который ею управляет, как
правило, должны быть необязательными, поэтому добавьте для неё опцию ``CONFIG``
(обычно в ``init/Kconfig``). Как обычно для новых опций ``CONFIG``:

 - Включите описание новой функциональности и системного вызова, управляемого
   этой опцией.
 - Сделайте опцию зависящей от EXPERT, если её следует скрывать от обычных
   пользователей.
 - Сделайте любые новые исходные файлы, реализующие функцию, зависящими от опции
   CONFIG в Makefile (например, ``obj-$(CONFIG_XYZZY_SYSCALL) += xyzzy.o``).
 - Дважды проверьте, что ядро по-прежнему собирается с выключенной новой опцией
   CONFIG.

Подытоживая, вам нужен коммит, который включает:

 - опцию ``CONFIG`` для новой функции, обычно в ``init/Kconfig``
 - ``SYSCALL_DEFINEn(xyzzy, ...)`` для точки входа
 - соответствующий прототип в ``include/linux/syscalls.h``
 - обобщённую запись в таблице в ``include/uapi/asm-generic/unistd.h``
 - резервную заглушку в ``kernel/sys_ni.c``


.. _syscall_generic_6_11:

Начиная с 6.11
~~~~~~~~~~~~~~

Начиная с версии ядра 6.11, общая реализация системного вызова для
следующих архитектур больше не требует изменений в
``include/uapi/asm-generic/unistd.h``:

 - arc
 - arm64
 - csky
 - hexagon
 - loongarch
 - nios2
 - openrisc
 - riscv

Вместо этого вам нужно обновить ``scripts/syscall.tbl`` и, если применимо,
скорректировать ``arch/*/kernel/Makefile.syscalls``.

Поскольку ``scripts/syscall.tbl`` служит общей таблицей системных вызовов для
нескольких архитектур, в этой таблице требуется новая запись::

    468   common   xyzzy     sys_xyzzy

Учтите, что добавление записи в ``scripts/syscall.tbl`` с ABI «common»
затрагивает также все архитектуры, которые разделяют эту таблицу. Для более
ограниченных или специфичных для архитектуры изменений рассмотрите
использование специфичного для архитектуры ABI или определение нового.

Если вводится новый ABI, скажем ``xyz``, то соответствующие обновления следует
внести и в ``arch/*/kernel/Makefile.syscalls``::

    syscall_abis_{32,64} += xyz (...)

Подытоживая, вам нужен коммит, который включает:

 - опцию ``CONFIG`` для новой функции, обычно в ``init/Kconfig``
 - ``SYSCALL_DEFINEn(xyzzy, ...)`` для точки входа
 - соответствующий прототип в ``include/linux/syscalls.h``
 - новую запись в ``scripts/syscall.tbl``
 - (при необходимости) обновления Makefile в ``arch/*/kernel/Makefile.syscalls``
 - резервную заглушку в ``kernel/sys_ni.c``


Реализация системного вызова для x86
------------------------------------

Чтобы подключить ваш новый системный вызов для платформ x86, вам нужно обновить
главные таблицы системных вызовов. Предполагая, что ваш новый системный вызов не
является чем-то особенным (см. ниже), это включает запись «common» (для x86_64 и
x32) в arch/x86/entry/syscalls/syscall_64.tbl::

    333   common   xyzzy     sys_xyzzy

и запись «i386» в ``arch/x86/entry/syscalls/syscall_32.tbl``::

    380   i386     xyzzy     sys_xyzzy

И снова эти номера могут быть изменены, если возникнут конфликты в
соответствующем окне слияния.


Системные вызовы совместимости (обобщённые)
-------------------------------------------

Для большинства системных вызовов одна и та же 64-битная реализация может быть
вызвана даже тогда, когда сама программа из пространства пользователя является
32-битной; даже если параметры системного вызова включают явный указатель, это
обрабатывается прозрачно.

Однако существует пара ситуаций, когда нужен слой совместимости, чтобы
справиться с различиями в размерах между 32-битным и 64-битным.

Первая — если 64-битное ядро также поддерживает 32-битные программы из
пространства пользователя и поэтому должно разбирать области памяти
(``__user``), которые могут содержать либо 32-битные, либо
64-битные значения. В частности, это нужно всякий раз, когда аргумент системного
вызова является:

 - указателем на указатель
 - указателем на структуру, содержащую указатель (например, ``struct iovec __user *``)
 - указателем на целочисленный тип переменного размера (``time_t``, ``off_t``,
   ``long``, ...)
 - указателем на структуру, содержащую целочисленный тип переменного размера.

Вторая ситуация, требующая слоя совместимости, — если один из
аргументов системного вызова имеет тип, явно являющийся 64-битным даже на
32-битной архитектуре, например ``loff_t`` или ``__u64``. В этом случае
значение, которое поступает в 64-битное ядро от 32-битного приложения, будет
разбито на два 32-битных значения, которые затем нужно заново собрать в слое
совместимости.

(Учтите, что аргумент системного вызова, являющийся указателем на явный 64-битный
тип, **не** нуждается в слое совместимости; например, аргументы
:manpage:`splice(2)` типа ``loff_t __user *`` не вызывают необходимости в
системном вызове ``compat_``.)

Версия системного вызова для совместимости называется ``compat_sys_xyzzy()``
и добавляется с помощью макроса ``COMPAT_SYSCALL_DEFINEn()``, по аналогии с
SYSCALL_DEFINEn. Эта версия реализации выполняется как часть 64-битного
ядра, но ожидает получения 32-битных значений параметров и делает всё, что
необходимо для работы с ними. (Как правило, версия ``compat_sys_`` преобразует
значения в 64-битные версии и либо вызывает версию ``sys_``, либо обе они
вызывают общую внутреннюю функцию реализации.)

Точке входа compat также нужен соответствующий прототип функции в
``include/linux/compat.h``, помеченный как asmlinkage, чтобы соответствовать
тому, как вызываются системные вызовы::

    asmlinkage long compat_sys_xyzzy(...);

Если системный вызов включает структуру, которая по-разному размещается в памяти
на 32-битных и 64-битных системах, скажем ``struct xyzzy_args``, то заголовочный
файл include/linux/compat.h должен также включать compat-версию структуры (``struct
compat_xyzzy_args``), где каждое поле переменного размера имеет соответствующий
тип ``compat_``, который соответствует типу в ``struct xyzzy_args``. Подпрограмма
``compat_sys_xyzzy()`` может затем использовать эту структуру ``compat_`` для
разбора аргументов 32-битного вызова.

Например, если есть поля::

    struct xyzzy_args {
        const char __user *ptr;
        __kernel_long_t varying_val;
        u64 fixed_val;
        /* ... */
    };

в struct xyzzy_args, то struct compat_xyzzy_args имела бы::

    struct compat_xyzzy_args {
        compat_uptr_t ptr;
        compat_long_t varying_val;
        u64 fixed_val;
        /* ... */
    };

Обобщённый список системных вызовов также нуждается в корректировке, чтобы
учесть compat-версию; запись в ``include/uapi/asm-generic/unistd.h`` должна
использовать ``__SC_COMP``, а не ``__SYSCALL``::

    #define __NR_xyzzy 292
    __SC_COMP(__NR_xyzzy, sys_xyzzy, compat_sys_xyzzy)

Подытоживая, вам нужно:

 - ``COMPAT_SYSCALL_DEFINEn(xyzzy, ...)`` для точки входа compat
 - соответствующий прототип в ``include/linux/compat.h``
 - (при необходимости) структуру 32-битного отображения в ``include/linux/compat.h``
 - экземпляр ``__SC_COMP``, а не ``__SYSCALL``, в
   ``include/uapi/asm-generic/unistd.h``


Начиная с 6.11
~~~~~~~~~~~~~~

Это применимо ко всем архитектурам, перечисленным в :ref:`Начиная с 6.11<syscall_generic_6_11>`
в разделе «Обобщённая реализация системного вызова», за исключением arm64. См.

на 64-битном ядре, должна попадать в точку входа compat::

    468   common     xyzzy     sys_xyzzy    compat_sys_xyzzy

Подытоживая, вам нужно:

 - ``COMPAT_SYSCALL_DEFINEn(xyzzy, ...)`` для точки входа compat
 - соответствующий прототип в ``include/linux/compat.h``
 - изменение записи в ``scripts/syscall.tbl`` для включения дополнительного
   столбца «compat»
 - (при необходимости) структуру 32-битного отображения в ``include/linux/compat.h``

Системные вызовы совместимости (arm64)
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

На arm64 существует выделенная таблица системных вызовов для системных вызовов
совместимости, нацеленных на 32-битное (AArch32) пространство пользователя:
``arch/arm64/tools/syscall_32.tbl``.
Вам нужно добавить в эту таблицу дополнительную строку, указывающую точку входа
compat::

    468   common     xyzzy     sys_xyzzy    compat_sys_xyzzy


Системные вызовы совместимости (x86)
------------------------------------

Чтобы подключить архитектуру x86 для системного вызова с версией совместимости,
записи в таблицах системных вызовов нужно скорректировать.

Во-первых, запись в ``arch/x86/entry/syscalls/syscall_32.tbl`` получает
дополнительный столбец, указывающий, что 32-битная программа из пространства
пользователя, работающая на 64-битном ядре, должна попадать в точку входа
compat::

    380   i386     xyzzy     sys_xyzzy    __ia32_compat_sys_xyzzy

Во-вторых, вам нужно разобраться, что должно происходить для версии нового
системного вызова под ABI x32. Здесь есть выбор: размещение аргументов
должно либо соответствовать 64-битной версии, либо 32-битной версии.

Если задействован указатель на указатель, решение простое: x32 — это
ILP32, поэтому размещение должно соответствовать 32-битной версии, и запись в
``arch/x86/entry/syscalls/syscall_64.tbl`` разделяется так, чтобы программы x32
попадали в обёртку совместимости::

    333   64       xyzzy     sys_xyzzy
    ...
    555   x32      xyzzy     __x32_compat_sys_xyzzy

Если указатели не задействованы, то предпочтительно повторно использовать
64-битный системный вызов для ABI x32 (и, как следствие, запись в
arch/x86/entry/syscalls/syscall_64.tbl остаётся неизменной).

В любом случае вам следует проверить, что типы, задействованные в размещении
ваших аргументов, действительно отображаются точно из x32 (-mx32) в
32-битные (-m32) или 64-битные (-m64) эквиваленты.


Системные вызовы, возвращающие управление в другое место
--------------------------------------------------------

Для большинства системных вызовов, как только системный вызов завершается,
пользовательская программа продолжает выполнение ровно с того места, где
остановилась, — со следующей инструкции, с тем же стеком и большей частью
регистров такими же, как до системного вызова,
и с тем же виртуальным пространством памяти.

Однако несколько системных вызовов поступают иначе. Они могут возвращать
управление в другое место (``rt_sigreturn``) или изменять пространство памяти
(``fork``/``vfork``/``clone``) или даже архитектуру (``execve``/``execveat``)
программы.

Чтобы это учесть, реализации системного вызова в ядре может потребоваться
сохранять и восстанавливать дополнительные регистры в стеке ядра, что
обеспечивает полный контроль над тем, где и как продолжится выполнение после
системного вызова.

Это специфично для архитектуры, но обычно включает определение ассемблерных
точек входа, которые сохраняют/восстанавливают дополнительные регистры и
вызывают реальную точку входа системного вызова.

Для x86_64 это реализуется как точка входа ``stub_xyzzy`` в
``arch/x86/entry/entry_64.S``, и запись в таблице системных вызовов
(``arch/x86/entry/syscalls/syscall_64.tbl``) корректируется в соответствии с этим::

    333   common   xyzzy     stub_xyzzy

Эквивалент для 32-битных программ, работающих на 64-битном ядре, обычно
называется ``stub32_xyzzy`` и реализуется в ``arch/x86/entry/entry_64_compat.S``,
с соответствующей корректировкой таблицы системных вызовов в
``arch/x86/entry/syscalls/syscall_32.tbl``::

    380   i386     xyzzy     sys_xyzzy    stub32_xyzzy

Если системному вызову нужен слой совместимости (как в предыдущем разделе),
то версия ``stub32_`` должна вызывать версию ``compat_sys_``
системного вызова, а не нативную 64-битную версию. Кроме того, если реализация
ABI x32 не является общей с версией x86_64, то её таблице системных вызовов
также потребуется вызывать заглушку, которая обращается к версии ``compat_sys_``.

Для полноты также хорошо настроить отображение, чтобы user-mode Linux
по-прежнему работал — его таблица системных вызовов будет ссылаться на
stub_xyzzy, но сборка UML не включает реализацию
``arch/x86/entry/entry_64.S`` (поскольку UML
симулирует регистры и т. п.). Исправление этого настолько просто, как
добавление #define в
``arch/x86/um/sys_call_table_64.c``::

    #define stub_xyzzy sys_xyzzy


Прочие подробности
------------------

Большая часть ядра обрабатывает системные вызовы обобщённым образом, но есть
отдельные исключения, которые могут нуждаться в обновлении для вашего
конкретного системного вызова.

Подсистема аудита — один из таких особых случаев; она включает (специфичные для
архитектуры) функции, которые классифицируют некоторые особые типы системных
вызовов — в частности, открытие файла (``open``/``openat``), выполнение
программы (``execve``/``exeveat``) или
операции мультиплексора сокетов (``socketcall``). Если ваш новый системный
вызов аналогичен одному из этих, то систему аудита следует обновить.

В более общем смысле, если существует системный вызов, аналогичный вашему
новому системному вызову, стоит выполнить grep по всему ядру для существующего
системного вызова, чтобы проверить, нет ли других особых случаев.


Тестирование
------------

Новый системный вызов, очевидно, должен быть протестирован; также полезно
предоставить рецензентам демонстрацию того, как программы из пространства
пользователя будут использовать этот системный вызов. Хороший способ
объединить эти цели — включить простую программу самопроверки (self-test)
в новый каталог внутри ``tools/testing/selftests/``.

Для нового системного вызова, очевидно, не будет функции-обёртки в libc, и
поэтому тесту потребуется вызвать его с помощью ``syscall()``; кроме того, если
системный вызов включает новую видимую из пространства пользователя структуру,
для компиляции теста нужно будет установить соответствующий заголовок.

Убедитесь, что самопроверка успешно выполняется на всех поддерживаемых
архитектурах. Например, проверьте, что она работает при компиляции как программа
с ABI x86_64 (-m64), x86_32 (-m32)
и x32 (-mx32).

Для более обширного и тщательного тестирования новой функциональности вам также
следует рассмотреть добавление тестов в Linux Test Project или в проект
xfstests для изменений, связанных с файловыми системами.

 - https://linux-test-project.github.io/
 - git://git.kernel.org/pub/scm/fs/xfs/xfstests-dev.git


Man-страница
------------

Все новые системные вызовы должны сопровождаться полной man-страницей, в идеале
с использованием разметки groff, но подойдёт и обычный текст. Если используется
groff, полезно включить предварительно отрендеренную ASCII-версию man-страницы в
сопроводительное письмо набора патчей, для удобства рецензентов.

Man-страницу следует отправить копией (cc) на linux-man@vger.kernel.org
Более подробно см. https://www.kernel.org/doc/man-pages/patches.html


Не вызывайте системные вызовы внутри ядра
-----------------------------------------

Системные вызовы, как указано выше, являются точками взаимодействия между
пространством пользователя и ядром. Поэтому функции системных вызовов, такие как
``sys_xyzzy()`` или
``compat_sys_xyzzy()``, должны вызываться только из пространства пользователя
через таблицу системных вызовов, но не из других мест ядра. Если
функциональность системного вызова полезна для использования внутри ядра,
должна разделяться между старым и новым системным вызовом или должна разделяться
между системным вызовом и его вариантом совместимости, её следует реализовать
посредством «вспомогательной» (helper) функции (такой как
``ksys_xyzzy()``). Эта функция ядра может затем вызываться внутри
заглушки системного вызова (``sys_xyzzy()``), заглушки системного вызова
совместимости (``compat_sys_xyzzy()``) и/или другого кода ядра.

По крайней мере на 64-битном x86 начиная с v4.17 будет жёстким требованием не
вызывать функции системных вызовов внутри ядра. Здесь используется иное
соглашение о вызове для системных вызовов, где ``struct pt_regs`` декодируется
на лету в обёртке системного вызова, которая затем передаёт обработку
фактической функции системного вызова.
Это означает, что во время входа в системный вызов передаются только те
параметры, которые действительно нужны для конкретного
системного вызова, вместо постоянного заполнения шести регистров CPU
произвольным содержимым из пространства пользователя (что может вызвать
серьёзные проблемы далее по цепочке вызовов).

Более того, правила доступа к данным могут различаться для данных ядра и
пользовательских данных. Это ещё одна причина, по которой вызов ``sys_xyzzy()``
в целом является плохой идеей.

Исключения из этого правила допускаются только в специфичных для архитектуры
переопределениях, специфичных для архитектуры обёртках совместимости или другом
коде в arch/.
