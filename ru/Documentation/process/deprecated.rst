=================================================================================
Устаревшие интерфейсы, языковые средства, атрибуты и соглашения
=================================================================================

В идеальном мире можно было бы преобразовать все случаи использования
некоторого устаревшего API в новый API и полностью удалить старый API
за один цикл разработки. Однако из-за размера ядра, иерархии сопровождения
и сроков не всегда возможно выполнять такие преобразования сразу. Это означает,
что новые случаи использования могут проникать в ядро, пока старые удаляются,
лишь увеличивая объём работы по удалению API. Чтобы информировать разработчиков
о том, что было объявлено устаревшим и почему, был создан этот список — место,
на которое можно указать, когда предлагается включить в ядро использование
устаревших вещей.

__deprecated
------------
Хотя этот атрибут визуально помечает интерфейс как устаревший,
он `больше не выдаёт предупреждений во время сборки
<https://git.kernel.org/linus/771c035372a036f83353eef46dbb829780330234>`_,
потому что одна из неизменных целей ядра — собираться без предупреждений,
а никто фактически ничего не делал для удаления этих устаревших интерфейсов.
Хотя использование `__deprecated` удобно, чтобы пометить старый API в
заголовочном файле, это не полное решение. Такие интерфейсы должны быть либо
полностью удалены из ядра, либо добавлены в этот файл, чтобы отговорить других
от их использования в будущем.

BUG() и BUG_ON()
----------------
Используйте вместо них WARN() и WARN_ON() и обрабатывайте «невозможное»
ошибочное состояние максимально аккуратно. Хотя семейство API BUG()
изначально проектировалось как assert для «невозможной ситуации» и для
«безопасного» завершения потока ядра, оказывается, что они просто слишком
рискованны. (Например: «В каком порядке нужно освобождать блокировки? Были ли
восстановлены различные состояния?») Очень часто использование BUG() приводит
к дестабилизации системы или к её полному отказу, что делает невозможной
отладку или даже получение пригодных отчётов о сбое. У Линуса есть `очень сильные
<https://lore.kernel.org/lkml/CA+55aFy6jNLsywVYdGp83AMrXBo_P-pkjkphPGrO=82SPKCpLQ@mail.gmail.com/>`_
чувства `по этому поводу
<https://lore.kernel.org/lkml/CAHk-=whDHsbK3HTOpTF=ue_o04onRwTEaK_ZoJp_fjbqq4+=Jw@mail.gmail.com/>`_.

Обратите внимание, что семейство WARN() следует использовать только для ситуаций,
«ожидаемо недостижимых». Если вы хотите предупредить о ситуациях «достижимых,
но нежелательных», пожалуйста, используйте семейство функций pr_warn().
Владельцы систем могут установить sysctl *panic_on_warn*, чтобы их системы
не продолжали работать при возникновении «недостижимых» условий. (Например,
см. коммиты вроде `этого
<https://git.kernel.org/linus/d4689846881d160a4d12a514e991a740bcb5d65a>`_.)

открыто закодированная арифметика в аргументах аллокатора
---------------------------------------------------------
Динамические вычисления размера (особенно умножение) не следует выполнять
в аргументах функций аллокатора памяти (или подобных) из-за риска их
переполнения. Это может привести к тому, что значения «обернутся», и будет
выделено меньше памяти, чем ожидал вызывающий. Использование таких выделений
может привести к линейным переполнениям памяти кучи и другим некорректным
поведениям. (Одно исключение — литеральные значения, для которых компилятор
может выдать предупреждение, если они могут переполниться. Однако в этих
случаях предпочтительный способ — переработать код, как предложено ниже,
чтобы избежать открыто закодированной арифметики.)

Например, не используйте ``count * size`` в качестве аргумента, как здесь::

	foo = kmalloc(count * size, GFP_KERNEL);

Вместо этого следует использовать двухфакторную форму аллокатора::

	foo = kmalloc_array(count, size, GFP_KERNEL);

В частности, kmalloc() можно заменить на kmalloc_array(), а
kzalloc() — на kcalloc().

Если двухфакторная форма недоступна, следует использовать вспомогательные
функции с насыщением при переполнении::

	bar = dma_alloc_coherent(dev, array_size(count, size), &dma, GFP_KERNEL);

Ещё один распространённый случай, которого следует избегать, — вычисление
размера структуры с завершающим массивом других структур, как здесь::

	header = kzalloc(sizeof(*header) + count * sizeof(*header->item),
			 GFP_KERNEL);

Вместо этого используйте вспомогательную функцию::

	header = kzalloc(struct_size(header, item, count), GFP_KERNEL);

.. note:: Если вы используете struct_size() для структуры, содержащей массив
        нулевой длины или массив из одного элемента в качестве завершающего
        члена-массива, пожалуйста, переработайте такое использование массива
        и переключитесь вместо этого на `гибкий член-массив
        <#zero-length-and-one-element-arrays>`_.

Для других вычислений, пожалуйста, комбинируйте использование вспомогательных
функций size_mul(), size_add() и size_sub(). Например, в случае::

	foo = krealloc(current_size + chunk_size * (count - 3), GFP_KERNEL);

Вместо этого используйте вспомогательные функции::

	foo = krealloc(size_add(current_size,
				size_mul(chunk_size,
					 size_sub(count, 3))), GFP_KERNEL);

Дополнительные сведения см. также в array3_size() и flex_array_size(),
а также в связанном семействе функций check_mul_overflow(), check_add_overflow(),
check_sub_overflow() и check_shl_overflow().

simple_strtol(), simple_strtoll(), simple_strtoul(), simple_strtoull()
----------------------------------------------------------------------
Функции simple_strtol(), simple_strtoll(),
simple_strtoul() и simple_strtoull()
явно игнорируют переполнения, что может привести к неожиданным результатам
у вызывающих. Соответствующие функции kstrtol(), kstrtoll(),
kstrtoul() и kstrtoull(), как правило, являются
правильной заменой, хотя следует учитывать, что они требуют, чтобы строка
завершалась символом NUL или символом новой строки.

strcpy()
--------
strcpy() не выполняет проверки границ буфера назначения. Это может привести
к линейным переполнениям за концом буфера, что ведёт к самым разным некорректным
поведениям. Хотя `CONFIG_FORTIFY_SOURCE=y` и различные флаги компилятора помогают
снизить риск использования этой функции, нет веской причины добавлять новые
случаи её использования. Безопасной заменой является strscpy(), хотя следует
проявлять осторожность во всех случаях, где использовалось возвращаемое значение
strcpy(), поскольку strscpy() возвращает не указатель на буфер назначения,
а количество скопированных байтов, не равных NUL (или отрицательный errno при
усечении).

strncpy() на NUL-терминированных строках
----------------------------------------
Использование strncpy() не гарантирует, что буфер назначения будет
NUL-терминирован. Это может привести к различным линейным переполнениям чтения
и другим некорректным поведениям из-за отсутствующего завершающего символа.
Кроме того, она дополняет буфер назначения символами NUL, если содержимое
источника короче размера буфера назначения, что может быть ненужным снижением
производительности для вызывающих, использующих только NUL-терминированные строки.

Когда требуется, чтобы буфер назначения был NUL-терминирован, заменой является
strscpy(), хотя следует проявлять осторожность во всех случаях, где использовалось
возвращаемое значение strncpy(), поскольку strscpy() возвращает не указатель на
буфер назначения, а количество скопированных байтов, не равных NUL (или
отрицательный errno при усечении). Любые случаи, где всё ещё необходимо дополнение
символами NUL, должны вместо этого использовать strscpy_pad().

Если вызывающий использует строки, не терминированные NUL, следует использовать
strtomem(), а буферы назначения должны быть помечены атрибутом `__nonstring
<https://gcc.gnu.org/onlinedocs/gcc/Common-Variable-Attributes.html>`_,
чтобы избежать будущих предупреждений компилятора. Для случаев, где всё ещё
необходимо дополнение символами NUL, можно использовать strtomem_pad().

strlcpy()
---------
strlcpy() сначала читает весь буфер-источник целиком (поскольку возвращаемое
значение должно совпадать со значением strlen()). Это чтение может превысить
ограничение размера назначения. Это одновременно неэффективно и может привести
к линейным переполнениям чтения, если строка-источник не NUL-терминирована.
Безопасной заменой является strscpy(), хотя следует проявлять осторожность во
всех случаях, где используется возвращаемое значение strlcpy(), поскольку
strscpy() при усечении возвращает отрицательные значения errno.

спецификатор формата %p
-----------------------
Традиционно использование "%p" в строках формата приводило к регулярным
уязвимостям раскрытия адресов в dmesg, proc, sysfs и т.п. Чтобы не оставлять их
эксплуатируемыми, все случаи использования "%p" в ядре печатаются как
хешированное значение, что делает их непригодными для адресации. Новые случаи
использования "%p" не следует добавлять в ядро. Для текстовых адресов
использование "%pS", вероятно, лучше, поскольку оно выдаёт более полезное имя
символа. Почти для всего остального просто не добавляйте "%p" вообще.

Перефразируя текущее `руководство <https://lore.kernel.org/lkml/CA+55aFwQEd_d40g4mUCSsVRZzrFPUJt74vc6PPpb675hYNXcKw@mail.gmail.com/>`_ Линуса:

- Если хешированное значение "%p" бессмысленно, спросите себя, важен ли сам
  указатель. Может быть, его следует удалить полностью?
- Если вы действительно считаете, что истинное значение указателя важно, то
  почему некоторое состояние системы или уровень привилегий пользователя
  считается «особым»? Если вы думаете, что можете обосновать это (в комментариях
  и в журнале коммита) достаточно хорошо, чтобы выдержать критический разбор
  Линуса, то, возможно, вы можете использовать "%px", вместе с тем убедившись,
  что у вас разумные права доступа.

Если вы отлаживаете что-то, где хеширование "%p" вызывает проблемы,
вы можете временно загрузиться с флагом отладки "`no_hash_pointers
<https://git.kernel.org/linus/5ead723a20e0447bc7db33dc3070b420e5f80aa6>`_".

Массивы переменной длины (VLA)
------------------------------
Использование стековых VLA порождает гораздо худший машинный код, чем стековые
массивы статического размера. Хотя эти нетривиальные `проблемы производительности
<https://git.kernel.org/linus/02361bc77888>`_ сами по себе являются достаточной
причиной для устранения VLA, они также представляют угрозу безопасности.
Динамический рост стекового массива может превысить оставшуюся память в сегменте
стека. Это может привести к сбою, возможной перезаписи важного содержимого в конце
стека (при сборке без `CONFIG_THREAD_INFO_IN_TASK=y`) или к перезаписи памяти,
смежной со стеком (при сборке без `CONFIG_VMAP_STACK=y`)

Неявное проваливание (fall-through) в switch/case
-------------------------------------------------
Язык C позволяет случаям switch проваливаться в следующий случай,
когда в конце случая отсутствует оператор "break". Это, однако,
вносит неоднозначность в код, поскольку не всегда ясно, является ли отсутствующий
break намеренным или ошибкой. Например, не очевидно лишь из взгляда
на код, спроектирован ли `STATE_ONE` намеренно так, чтобы проваливаться
в `STATE_TWO`::

	switch (value) {
	case STATE_ONE:
		do_something();
	case STATE_TWO:
		do_other();
		break;
	default:
		WARN("unknown state");
	}

Поскольку имелся длинный список изъянов `из-за отсутствующих операторов "break"
<https://cwe.mitre.org/data/definitions/484.html>`_, мы больше не допускаем
неявного проваливания. Чтобы идентифицировать намеренные случаи проваливания,
мы приняли псевдоключевое слово-макрос "fallthrough", которое
раскрывается в расширение gcc `__attribute__((__fallthrough__))
<https://gcc.gnu.org/onlinedocs/gcc/Statement-Attributes.html>`_.
(Когда синтаксис C17/C18 `[[fallthrough]]` будет более широко поддерживаться
компиляторами C, статическими анализаторами и IDE, мы сможем перейти к
использованию этого синтаксиса для псевдоключевого слова-макроса.)

Все блоки switch/case должны заканчиваться одним из:

* break;
* fallthrough;
* continue;
* goto <label>;
* return [expression];

Массивы нулевой длины и массивы из одного элемента
--------------------------------------------------
В ядре регулярно возникает потребность предоставить способ объявить наличие
динамически изменяемого набора завершающих элементов в структуре. Код ядра
должен всегда использовать `«гибкие члены-массивы» <https://en.wikipedia.org/wiki/Flexible_array_member>`_
для этих случаев. Более старый стиль с массивами из одного элемента или нулевой
длины больше не следует использовать.

В более старом коде на C динамически изменяемые завершающие элементы делались
путём указания массива из одного элемента в конце структуры::

        struct something {
                size_t count;
                struct foo items[1];
        };

Это приводило к хрупким вычислениям размера через sizeof() (которому пришлось бы
вычитать размер единственного завершающего элемента, чтобы получить корректный
размер «заголовка»). Было введено `расширение GNU C <https://gcc.gnu.org/onlinedocs/gcc/Zero-Length.html>`_,
позволяющее использовать массивы нулевой длины, чтобы избежать такого рода
проблем с размером::

        struct something {
                size_t count;
                struct foo items[0];
        };

Но это привело к другим проблемам и не решило некоторых проблем, общих для
обоих стилей, например невозможности обнаружить, когда такой массив случайно
используется _не_ в конце структуры (что могло произойти напрямую или когда
такая структура находилась в объединениях, структурах из структур и т.п.).

В C99 были введены «гибкие члены-массивы», в которых полностью отсутствует
числовой размер при объявлении массива::

        struct something {
                size_t count;
                struct foo items[];
        };

Именно так ядро ожидает объявления динамически изменяемых завершающих элементов.
Это позволяет компилятору генерировать ошибки, когда гибкий массив располагается
не последним в структуре, что помогает предотвратить непреднамеренное внесение
в кодовую базу некоторых ошибок `неопределённого поведения
<https://git.kernel.org/linus/76497732932f15e7323dc805e8ea8dc11bb587cf>`_.
Это также позволяет компилятору корректно анализировать размеры массивов (через
sizeof(), `CONFIG_FORTIFY_SOURCE` и `CONFIG_UBSAN_BOUNDS`). Например,
нет механизма, который предупреждал бы нас о том, что следующее применение
оператора sizeof() к массиву нулевой длины всегда даёт ноль::

        struct something {
                size_t count;
                struct foo items[0];
        };

        struct something *instance;

        instance = kmalloc(struct_size(instance, items, count), GFP_KERNEL);
        instance->count = count;

        size = sizeof(instance->items) * instance->count;
        memcpy(instance->items, source, size);

В последней строке кода выше ``size`` оказывается равным ``zero``, в то время как
можно было бы подумать, что оно представляет общий размер в байтах динамической
памяти, недавно выделенной для завершающего массива ``items``. Вот пара примеров
этой проблемы: `ссылка 1
<https://git.kernel.org/linus/f2cd32a443da694ac4e28fbf4ac6f9d5cc63a539>`_,
`ссылка 2
<https://git.kernel.org/linus/ab91c2a89f86be2898cee208d492816ec238b2cf>`_.
Вместо этого, `гибкие члены-массивы имеют неполный тип, и поэтому оператор sizeof()
не может быть к ним применён <https://gcc.gnu.org/onlinedocs/gcc/Zero-Length.html>`_,
так что любое неправильное использование таких операторов будет сразу замечено
во время сборки.

В отношении массивов из одного элемента нужно остро осознавать, что `такие массивы
занимают по меньшей мере столько же места, сколько один объект данного типа
<https://gcc.gnu.org/onlinedocs/gcc/Zero-Length.html>`_,
поэтому они вносят вклад в размер охватывающей структуры. Это чревато
ошибками всякий раз, когда люди хотят вычислить общий размер динамической памяти
для выделения под структуру, содержащую массив такого рода в качестве члена::

        struct something {
                size_t count;
                struct foo items[1];
        };

        struct something *instance;

        instance = kmalloc(struct_size(instance, items, count - 1), GFP_KERNEL);
        instance->count = count;

        size = sizeof(instance->items) * instance->count;
        memcpy(instance->items, source, size);

В примере выше нам приходилось помнить о вычислении ``count - 1`` при использовании
вспомогательной функции struct_size(), иначе мы бы --непреднамеренно-- выделили
память на один объект ``items`` больше, чем нужно. Самый чистый и наименее
подверженный ошибкам способ реализовать это — использование `гибкого члена-массива`
вместе с вспомогательными функциями struct_size() и flex_array_size()::

        struct something {
                size_t count;
                struct foo items[];
        };

        struct something *instance;

        instance = kmalloc(struct_size(instance, items, count), GFP_KERNEL);
        instance->count = count;

        memcpy(instance->items, source, flex_array_size(instance, items, instance->count));

Есть два особых случая замены, в которых необходимо использовать вспомогательную
функцию DECLARE_FLEX_ARRAY(). (Обратите внимание, что она называется
__DECLARE_FLEX_ARRAY() для использования в заголовках UAPI.) Эти случаи — когда
гибкий массив либо находится в структуре в одиночестве, либо является частью
объединения. Они запрещены спецификацией C99, но без какой-либо технической
причины (что видно как по существующему использованию таких массивов в этих
местах, так и по обходному решению, которое использует DECLARE_FLEX_ARRAY()).
Например, чтобы преобразовать это::

	struct something {
		...
		union {
			struct type1 one[0];
			struct type2 two[0];
		};
	};

Необходимо использовать вспомогательную функцию::

	struct something {
		...
		union {
			DECLARE_FLEX_ARRAY(struct type1, one);
			DECLARE_FLEX_ARRAY(struct type2, two);
		};
	};

Открыто закодированные присваивания kmalloc для объектов struct
---------------------------------------------------------------
Выполнение открыто закодированных присваиваний при выделении памяти через
семейство kmalloc() не позволяет ядру (и компилятору) проанализировать тип
переменной, которой выполняется присваивание, что ограничивает любой связанный
с этим самоанализ, способный помочь с выравниванием, «обёртыванием» или
дополнительной защитой. Семейство макросов kmalloc_obj() обеспечивает такой
самоанализ, который можно использовать для распространённых шаблонов кода при
выделении одиночных объектов, массивов и гибких объектов. Например, эти
открыто закодированные присваивания::

	ptr = kmalloc(sizeof(*ptr), gfp);
	ptr = kzalloc(sizeof(*ptr), gfp);
	ptr = kmalloc_array(count, sizeof(*ptr), gfp);
	ptr = kcalloc(count, sizeof(*ptr), gfp);
	ptr = kmalloc(struct_size(ptr, flex_member, count), gfp);
	ptr = kmalloc(sizeof(struct foo, gfp);

становятся, соответственно::

	ptr = kmalloc_obj(*ptr, gfp);
	ptr = kzalloc_obj(*ptr, gfp);
	ptr = kmalloc_objs(*ptr, count, gfp);
	ptr = kzalloc_objs(*ptr, count, gfp);
	ptr = kmalloc_flex(*ptr, flex_member, count, gfp);
	__auto_type ptr = kmalloc_obj(struct foo, gfp);

Если `ptr->flex_member` аннотирован с помощью __counted_by(), выделение
автоматически завершится неудачей, если `count` больше максимального
представимого значения, которое может быть сохранено в члене-счётчике,
связанном с `flex_member`.
