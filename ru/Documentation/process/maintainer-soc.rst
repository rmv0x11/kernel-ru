==============
Подсистема SoC
==============

Обзор
-----

Подсистема SoC — это место агрегации кода, специфичного для SoC.
Основные компоненты подсистемы:

* devicetree (DTS) для 32- и 64-битных ARM и RISC-V
* файлы плат для 32-битного ARM (arch/arm/mach*)
* defconfig для 32- и 64-битных ARM
* специфичные для SoC драйверы для разных архитектур, в частности для 32- и
  64-битных ARM, RISC-V и Loongarch

Эти «специфичные для SoC драйверы» не включают драйверы clock, GPIO и т. п.,
у которых есть собственные сопровождающие верхнего уровня. Каталог drivers/soc/
в целом предназначен для внутриядерных драйверов, которые используются другими
драйверами для предоставления специфичной для SoC функциональности, например для
идентификации ревизии SoC или взаимодействия с доменами питания.

Подсистема SoC также служит промежуточным местом для изменений в drivers/bus,
drivers/firmware, drivers/reset и drivers/memory. Добавление новых платформ или
удаление существующих часто проходит через дерево SoC в виде отдельной ветки,
охватывающей несколько подсистем.

Основное дерево SoC расположено на git.kernel.org:
  https://git.kernel.org/pub/scm/linux/kernel/git/soc/soc.git/

Сопровождающие
--------------

Очевидно, что это весьма широкий круг тем, который не способен сопровождать ни
один человек и даже небольшая группа людей. Вместо этого подсистема SoC состоит
из множества субсопровождающих (сопровождающих платформ), каждый из которых
заботится об отдельных платформах и подкаталогах драйверов.
В этом контексте «платформа» обычно обозначает серию SoC от конкретного
вендора, например серию Tegra SoC от Nvidia. Многие субсопровождающие работают
на уровне вендора, отвечая за несколько продуктовых линеек. По ряду причин,
включая поглощения и различные бизнес-подразделения внутри компании, здесь
наблюдаются значительные различия. Различные субсопровождающие задокументированы
в файле MAINTAINERS.

У большинства этих субсопровождающих есть собственные деревья, где они готовят
патчи, отправляя pull request в основное дерево SoC. Эти деревья обычно, но не
всегда, перечислены в MAINTAINERS.

Однако дерево SoC не является местом для изменений кода, специфичных для
архитектуры. У каждой архитектуры есть свои сопровождающие, которые отвечают за
архитектурные детали, ошибки (errata) в CPU и тому подобное.

Отправка патчей для конкретного SoC
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Все типичные патчи, связанные с платформой, следует отправлять через
субсопровождающих SoC (специфичных для платформы сопровождающих). Это включает
также изменения в defconfig для конкретной платформы или общих defconfig.
Обратите внимание, что scripts/get_maintainer.pl может не предоставить корректные
адреса для общего defconfig, поэтому игнорируйте его вывод и составьте CC-список
вручную на основе файла MAINTAINERS либо используйте что-то вроде
``scripts/get_maintainer.pl -f drivers/soc/FOO/``).

Отправка патчей основным сопровождающим SoC
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

С основными сопровождающими SoC можно связаться через псевдоним soc@kernel.org
только в следующих случаях:

1. Специфичных для платформы сопровождающих нет.

2. Специфичные для платформы сопровождающие не отвечают.

3. Внедрение совершенно новой платформы SoC. Такую работу по новому SoC следует
   сначала отправить в общие списки рассылки, указанные scripts/get_maintainer.pl,
   для рассмотрения сообществом. После положительного рассмотрения сообществом
   работу следует отправить на soc@kernel.org одним набором патчей, содержащим
   новую запись arch/foo/Kconfig, файлы DTS, запись в файле MAINTAINERS и,
   необязательно, начальные драйверы с их привязками (binding) Devicetree. Запись
   в файле MAINTAINERS должна перечислять новых специфичных для платформы
   сопровождающих, которые отныне будут отвечать за обработку патчей для этой
   платформы.

Обратите внимание, что soc@kernel.org обычно не является местом для обсуждения
патчей, поэтому работа, отправленная на этот адрес, уже должна считаться
приемлемой для сообщества.

Информация для (новых) субсопровождающих
----------------------------------------

По мере появления новых платформ они часто приводят за собой новых
субсопровождающих, многие из которых работают на производителя кремния
(silicon vendor) и могут быть незнакомы с процессом.

Стабильность ABI Devicetree
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Пожалуй, одна из самых важных вещей, которую стоит подчеркнуть, — это то, что
dt-bindings документируют ABI между devicetree и ядром.
Пожалуйста, прочитайте Documentation/devicetree/bindings/ABI.rst.

Если в DTS вносятся изменения, несовместимые со старыми ядрами, патч для DTS не
следует применять до тех пор, пока не применён драйвер, или спустя подходящее
время после этого. Самое важное: любые несовместимые изменения должны быть чётко
указаны в описании патча и pull request вместе с ожидаемым влиянием на
существующих пользователей, таких как загрузчики (bootloaders) или другие
операционные системы.

Зависимости ветки драйвера
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Распространённой проблемой является синхронизация изменений между драйверами
устройств и файлами devicetree. Даже если изменение совместимо в обоих
направлениях, это может потребовать согласования того, как изменения вливаются
через различные деревья сопровождающих.

Обычно ветка, включающая изменение драйвера, также включает соответствующее
изменение в описании привязки (binding) devicetree, чтобы убедиться, что они
действительно совместимы. Это означает, что ветка devicetree может в итоге
вызывать предупреждения на шаге ``make dtbs_check``. Если изменение devicetree
зависит от отсутствующих дополнений в файле заголовка в include/dt-bindings/,
оно завершится неудачей на шаге ``make dtbs`` и не будет влито.

Есть несколько способов справиться с этим:

* Избегайте определения пользовательских макросов в include/dt-bindings/ для
  аппаратных констант, которые можно вывести из datasheet, — макросы привязок в
  файлах заголовков следует использовать лишь как последнее средство, если нет
  естественного способа определить привязку

* Используйте в файле devicetree литеральные значения вместо макросов, даже когда
  заголовок необходим, и заменяйте их именованным представлением в следующем
  релизе

* Откладывайте изменения devicetree до релиза, следующего за тем, в котором уже
  влиты привязка и драйвер

* Изменяйте привязки в общей неизменяемой ветке, которая используется как основа
  как для изменения драйвера, так и для изменений devicetree

* Добавляйте дублирующие определения в файле devicetree, защищённые секцией
  #ifndef, удаляя их в более позднем релизе

Соглашение об именовании Devicetree
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Общая схема именования файлов devicetree выглядит следующим образом. Аспекты
платформы, которые задаются на уровне SoC, например ядра CPU, содержатся в файле
с именем $soc.dtsi, например jh7100.dtsi. Детали интеграции, которые будут
варьироваться от платы к плате, описываются в $soc-$board.dts. Примером этого
является jh7100-beaglev-starlight.dts. Часто многие платы являются вариациями на
одну тему, и нередко существуют промежуточные файлы, такие как jh7100-common.dtsi,
которые располагаются между файлами $soc.dtsi и $soc-$board.dts и содержат
описания общего оборудования.

У некоторых платформ также есть System on Modules, содержащие SoC, которые затем
интегрируются в несколько различных плат. Для таких платформ типичны
$soc-$som.dtsi и $soc-$som-$board.dts.

Каталоги обычно называются по имени вендора SoC на момент его включения, что
приводит к появлению в дереве некоторых исторических имён каталогов.

Валидация файлов Devicetree
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

``make dtbs_check`` можно использовать для проверки того, что файлы devicetree
соответствуют dt-bindings, описывающим ABI. Пожалуйста, прочитайте раздел
"Running checks" в Documentation/devicetree/bindings/writing-schema.rst для
получения дополнительной информации о валидации devicetree.

Для новых платформ или дополнений к существующим ``make dtbs_check`` не должен
добавлять никаких новых предупреждений. Для RISC-V и Samsung SoC требуется, чтобы
``make dtbs_check W=1`` не добавлял никаких новых предупреждений.
Если есть какие-либо сомнения относительно изменения devicetree, свяжитесь с
сопровождающими devicetree.

Ветки и pull request
~~~~~~~~~~~~~~~~~~~~~~

Точно так же, как основное дерево SoC имеет несколько веток, ожидается, что
субсопровождающие будут поступать так же. Изменения драйверов, defconfig и
devicetree должны быть разделены на отдельные ветки и появляться в отдельных
pull request к сопровождающим SoC. Каждая ветка должна быть пригодна к
использованию сама по себе и избегать регрессий, возникающих из-за зависимостей
от других веток.

Небольшие наборы патчей также можно отправлять отдельными письмами на
soc@kernel.org, сгруппированными в те же категории.

Если изменения не укладываются в обычные шаблоны, могут существовать
дополнительные ветки верхнего уровня, например для общесистемной переработки
(treewide rework) или для добавления новых платформ SoC, включая файлы dts и
драйверы.

Ветки с большим количеством изменений могут выиграть от разбиения на отдельные
тематические ветки, даже если в итоге они вливаются в одну и ту же ветку дерева
SoC. Примером здесь была бы одна ветка для исправлений предупреждений devicetree,
одна для переработки и одна для вновь добавленных плат.

Ещё один распространённый способ разбить изменения — отправить ранний pull
request с большей частью изменений в какой-то момент между rc1 и rc4, дополнив
его одним или несколькими меньшими pull request ближе к концу цикла, которые
могут добавить поздние изменения или устранить проблемы, выявленные при
тестировании первого набора.

Хотя для поздних pull request нет крайнего срока, по мере приближения к окну
слияния (merge window) полезно отправлять только небольшие ветки.

Pull request с исправлениями ошибок для текущего релиза можно отправлять в любое
время, но и здесь несколько меньших веток лучше, чем попытка объединить слишком
много патчей в один pull request.

Строка темы pull request должна начинаться с "[GIT PULL]" и создаваться с
использованием подписанного тега, а не ветки. Этот тег должен содержать краткое
описание, суммирующее изменения в pull request. Для получения более подробной
информации об отправке pull request, пожалуйста, см.
Documentation/maintainer/pull-requests.rst.
