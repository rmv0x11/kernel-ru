.. _codingstyle:

Стиль кодирования ядра Linux
============================

Это краткий документ, описывающий предпочтительный стиль кодирования для
ядра Linux.  Стиль кодирования — вещь очень личная, и я не буду **навязывать**
свои взгляды никому, но это то, чего следует придерживаться во всём, что я
должен иметь возможность сопровождать, и я предпочёл бы этот стиль и для
большинства других вещей.  Прошу по крайней мере рассмотреть изложенные здесь
соображения.

Прежде всего, я бы предложил распечатать копию стандартов кодирования GNU
и НЕ читать её.  Сожгите её, это великолепный символический жест.

В общем, приступим:


1) Отступы
----------

Табуляция — это 8 символов, и, следовательно, отступы также составляют
8 символов.  Существуют еретические движения, которые пытаются сделать отступы
глубиной 4 (или даже 2!) символа, и это сродни попытке определить значение числа
ПИ равным 3.

Обоснование: вся идея отступов состоит в том, чтобы чётко обозначить, где
начинается и заканчивается блок управления.  Особенно когда вы смотрели в
экран 20 часов подряд, вы обнаружите, что гораздо легче увидеть, как работают
отступы, если они большие.

Теперь некоторые люди станут утверждать, что отступы в 8 символов отодвигают
код слишком далеко вправо и затрудняют его чтение на 80-символьном экране
терминала.  Ответ на это таков: если вам нужно более 3 уровней отступа, вы
в любом случае в беде и должны исправить свою программу.

Короче говоря, отступы в 8 символов облегчают чтение и дают дополнительное
преимущество — предупреждают вас, когда вы слишком глубоко вкладываете свои
функции.  Прислушайтесь к этому предупреждению.

Предпочтительный способ облегчить множественные уровни отступов в операторе
switch — это выровнять ``switch`` и подчинённые ему метки ``case`` в одной
колонке вместо ``двойного отступа`` для меток ``case``.  Например:

.. code-block:: c

	switch (suffix) {
	case 'G':
	case 'g':
		mem <<= 30;
		break;
	case 'M':
	case 'm':
		mem <<= 20;
		break;
	case 'K':
	case 'k':
		mem <<= 10;
		fallthrough;
	default:
		break;
	}

Не размещайте несколько операторов в одной строке, если только вам не нужно
что-то скрыть:

.. code-block:: c

	if (condition) do_this;
	  do_something_everytime;

Не используйте запятые, чтобы избежать применения фигурных скобок:

.. code-block:: c

	if (condition)
		do_this(), do_that();

Всегда используйте фигурные скобки для нескольких операторов:

.. code-block:: c

	if (condition) {
		do_this();
		do_that();
	}

Также не размещайте несколько присваиваний в одной строке.  Стиль кодирования
ядра предельно прост.  Избегайте хитроумных выражений.


Вне комментариев, документации и за исключением Kconfig пробелы никогда не
используются для отступов, и приведённый выше пример намеренно сломан.

Заведите приличный редактор и не оставляйте пробельных символов в конце строк.


2) Перенос длинных строк и строковых литералов
----------------------------------------------

Стиль кодирования — это всё о читаемости и сопровождаемости при использовании
общедоступных инструментов.

Предпочтительное ограничение длины одной строки — 80 колонок.

Операторы длиннее 80 колонок следует разбивать на разумные части, если только
превышение 80 колонок существенно не повышает читаемость и не скрывает
информацию.

Потомки всегда существенно короче родителя и располагаются существенно правее.
Очень часто используемый стиль — выравнивать потомков по открывающей скобке
функции.

Те же правила применяются к заголовкам функций с длинным списком аргументов.

Однако никогда не разбивайте видимые пользователю строки, такие как сообщения
printk, поскольку это нарушает возможность найти их через grep.


3) Расстановка фигурных скобок и пробелов
-----------------------------------------

Другой вопрос, который всегда возникает при оформлении кода на C, — это
расстановка фигурных скобок.  В отличие от размера отступа, существует мало
технических причин выбирать одну стратегию расстановки вместо другой, но
предпочтительный способ, как было показано нам пророками Керниганом и Ритчи, —
ставить открывающую фигурную скобку последней на строке, а закрывающую — первой,
вот так:

.. code-block:: c

	if (x is true) {
		we do y
	}

Это применимо ко всем нефункциональным блокам операторов (if, switch, for,
while, do).  Например:

.. code-block:: c

	switch (action) {
	case KOBJ_ADD:
		return "add";
	case KOBJ_REMOVE:
		return "remove";
	case KOBJ_CHANGE:
		return "change";
	default:
		return NULL;
	}

Однако есть один особый случай, а именно функции: у них открывающая фигурная
скобка стоит в начале следующей строки, вот так:

.. code-block:: c

	int function(int x)
	{
		body of function
	}

Еретики по всему миру утверждали, что эта несогласованность ... ну ...
несогласованна, но все здравомыслящие люди знают, что (a) K&R **правы** и
(b) K&R правы.  К тому же функции всё равно особенные (их нельзя вкладывать
друг в друга в C).

Обратите внимание, что закрывающая фигурная скобка пуста на отдельной строке,
**за исключением** случаев, когда за ней следует продолжение того же оператора,
то есть ``while`` в do-операторе или ``else`` в if-операторе, вот так:

.. code-block:: c

	do {
		body of do-loop
	} while (condition);

и

.. code-block:: c

	if (x == y) {
		..
	} else if (x > y) {
		...
	} else {
		....
	}

Обоснование: K&R.

Также обратите внимание, что такая расстановка фигурных скобок минимизирует
число пустых (или почти пустых) строк без какой-либо потери читаемости.  Таким
образом, поскольку запас новых строк на вашем экране не является возобновляемым
ресурсом (вспомните здесь 25-строчные экраны терминалов), у вас остаётся больше
пустых строк для размещения комментариев.

Не используйте без необходимости фигурные скобки там, где достаточно одного
оператора.

.. code-block:: c

	if (condition)
		action();

и

.. code-block:: c

	if (condition)
		do_this();
	else
		do_that();

Это не применяется, если лишь одна ветвь условного оператора является
единственным оператором; в последнем случае используйте фигурные скобки в обеих
ветвях:

.. code-block:: c

	if (condition) {
		do_this();
		do_that();
	} else {
		otherwise();
	}

Также используйте фигурные скобки, когда цикл содержит более одного простого
оператора:

.. code-block:: c

	while (condition) {
		if (test)
			do_something();
	}

3.1) Пробелы
************

Стиль ядра Linux в отношении использования пробелов зависит (в основном) от
того, что используется — функция или ключевое слово.  Используйте пробел после
(большинства) ключевых слов.  Заметные исключения — sizeof, typeof, alignof и
__attribute__, которые несколько похожи на функции (и в Linux обычно
используются со скобками, хотя в языке они не требуются, как в случае:
``sizeof info`` после того, как объявлено ``struct fileinfo info;``).

Поэтому используйте пробел после этих ключевых слов::

	if, switch, case, for, do, while

но не с sizeof, typeof, alignof или __attribute__.  Например,

.. code-block:: c


	s = sizeof(struct file);

Не добавляйте пробелы вокруг (внутри) выражений в скобках.  Этот пример
**плох**:

.. code-block:: c


	s = sizeof( struct file );

При объявлении указательных данных или функции, возвращающей указательный тип,
предпочтительно использование ``*`` рядом с именем данных или именем функции, а
не рядом с именем типа.  Примеры:

.. code-block:: c


	char *linux_banner;
	unsigned long long memparse(char *ptr, char **retptr);
	char *match_strdup(substring_t *s);

Используйте один пробел вокруг (с каждой стороны) большинства бинарных и
тернарных операторов, таких как любой из этих::

	=  +  -  <  >  *  /  %  |  &  ^  <=  >=  ==  !=  ?  :

но без пробела после унарных операторов::

	&  *  +  -  ~  !  sizeof  typeof  alignof  __attribute__  defined

без пробела перед постфиксными унарными операторами инкремента и декремента::

	++  --

без пробела после префиксных унарных операторов инкремента и декремента::

	++  --

и без пробела вокруг операторов доступа к членам структуры ``.`` и ``->``.

Не оставляйте конечных пробельных символов в концах строк.  Некоторые редакторы
с ``умными`` отступами вставляют пробельные символы в начале новых строк по мере
необходимости, так что вы можете сразу начать набирать следующую строку кода.
Однако некоторые такие редакторы не удаляют пробельные символы, если вы в итоге
не размещаете там строку кода, например если вы оставляете пустую строку.  В
результате у вас получаются строки, содержащие конечные пробельные символы.

Git предупредит вас о патчах, которые вводят конечные пробельные символы, и при
желании может удалить эти конечные пробельные символы за вас; однако при
применении серии патчей это может привести к сбою более поздних патчей в серии
из-за изменения их строк контекста.


4) Именование
-------------

C — спартанский язык, и ваши соглашения об именовании должны следовать тому же.
В отличие от программистов на Modula-2 и Pascal, программисты на C не используют
милые имена вроде ThisVariableIsATemporaryCounter. Программист на C назвал бы эту
переменную ``tmp``, что гораздо легче писать и нисколько не сложнее понять.

ОДНАКО, хотя имена в смешанном регистре не одобряются, описательные имена для
глобальных переменных обязательны.  Назвать глобальную функцию ``foo`` — это
расстрельное преступление.

ГЛОБАЛЬНЫЕ переменные (которые следует использовать только если они вам
**действительно** нужны) должны иметь описательные имена, как и глобальные
функции.  Если у вас есть функция, подсчитывающая число активных пользователей,
вам следует назвать её ``count_active_users()`` или подобным образом, и вам
**не** следует называть её ``cntusr()``.

Кодирование типа функции в её имени (так называемая венгерская нотация) —
идиотизм: компилятор всё равно знает типы и может их проверить, а это лишь
сбивает программиста с толку.

ЛОКАЛЬНЫЕ имена переменных должны быть короткими и по делу.  Если у вас есть
какой-то случайный целочисленный счётчик цикла, его, вероятно, следует назвать
``i``.  Называть его ``loop_counter`` непродуктивно, если нет шансов, что его
неправильно поймут.  Аналогично ``tmp`` может быть практически любым типом
переменной, используемой для хранения временного значения.

Если вы боитесь перепутать имена своих локальных переменных, у вас есть другая
проблема, которая называется синдромом дисбаланса гормона роста функций.
См. главу 6 (Функции).

Для имён символов и документации избегайте введения нового использования
'master / slave' (или 'slave' независимо от 'master') и 'blacklist /
whitelist'.

Рекомендуемые замены для 'master / slave':
    '{primary,main} / {secondary,replica,subordinate}'
    '{initiator,requester} / {target,responder}'
    '{controller,host} / {device,worker,proxy}'
    'leader / follower'
    'director / performer'

Рекомендуемые замены для 'blacklist/whitelist':
    'denylist / allowlist'
    'blocklist / passlist'

Исключения для введения нового использования — это поддержка ABI/API
пространства пользователя или обновление кода для существующей (по состоянию на
2020 год) спецификации аппаратного обеспечения или протокола, которая предписывает
эти термины. Для новых спецификаций по возможности переводите употребление
терминологии из спецификации в стандарт кодирования ядра.

5) Typedef'ы
------------

Пожалуйста, не используйте вещи вроде ``vps_t``.
Использовать typedef для структур и указателей — это **ошибка**. Когда вы видите

.. code-block:: c


	vps_t a;

в исходном коде, что это означает?
Напротив, если там написано

.. code-block:: c

	struct virtual_container *a;

вы действительно можете сказать, что такое ``a``.

Многие люди думают, что typedef'ы ``помогают читаемости``. Это не так. Они
полезны только для:

 (a) полностью непрозрачных объектов (где typedef активно используется для того,
     чтобы **скрыть**, чем является объект).

     Пример: ``pte_t`` и т. п. непрозрачные объекты, к которым можно обращаться
     только с помощью соответствующих функций доступа.

     .. note::

       Непрозрачность и ``функции доступа`` сами по себе не благо.
       Причина, по которой они у нас есть для таких вещей, как pte_t и т. п.,
       состоит в том, что там действительно абсолютно **ноль** переносимо
       доступной информации.

 (b) Ясных целочисленных типов, где абстракция **помогает** избежать путаницы,
     является ли это ``int`` или ``long``.

     u8/u16/u32 — совершенно нормальные typedef'ы, хотя они лучше вписываются в
     категорию (d), чем сюда.

     .. note::

       Опять же — для этого должна быть **причина**. Если что-то является
       ``unsigned long``, то нет причины делать

	typedef unsigned long myflags_t;

     но если есть ясная причина, почему при определённых обстоятельствах это
     может быть ``unsigned int``, а при других конфигурациях может быть
     ``unsigned long``, то непременно используйте typedef.

 (c) когда вы используете sparse, чтобы буквально создать **новый** тип для
     проверки типов.

 (d) Новых типов, идентичных стандартным типам C99, в определённых
     исключительных обстоятельствах.

     Хотя глазам и мозгу потребовалось бы лишь немного времени, чтобы
     привыкнуть к стандартным типам вроде ``uint32_t``, некоторые люди всё
     равно возражают против их использования.

     Поэтому специфичные для Linux типы ``u8/u16/u32/u64`` и их знаковые
     эквиваленты, которые идентичны стандартным типам, разрешены — хотя они и
     не являются обязательными в вашем собственном новом коде.

     При редактировании существующего кода, который уже использует тот или иной
     набор типов, вам следует придерживаться существующего выбора в этом коде.

 (e) Типов, безопасных для использования в пространстве пользователя.

     В определённых структурах, которые видимы из пространства пользователя, мы
     не можем требовать типы C99 и не можем использовать форму ``u32``, описанную
     выше. Поэтому мы используем __u32 и подобные типы во всех структурах,
     которые разделяются с пространством пользователя.

Возможно, есть и другие случаи, но правило должно по сути состоять в том, чтобы
НИКОГДА И НИ ЗА ЧТО не использовать typedef, если вы не можете чётко сопоставить
один из этих случаев.

В общем случае указатель или структура, имеющая элементы, к которым разумно
обращаться напрямую, **никогда** не должны быть typedef'ом.


6) Функции
----------

Функции должны быть короткими и милыми и делать лишь одну вещь.  Они должны
помещаться на одном-двух экранах текста (размер экрана по ISO/ANSI — 80x24, как
все мы знаем) и делать одну вещь и делать её хорошо.

Максимальная длина функции обратно пропорциональна сложности и уровню отступа
этой функции.  Так, если у вас есть концептуально простая функция, представляющая
собой просто один длинный (но простой) case-оператор, где вам приходится делать
множество мелких вещей для большого числа разных случаев, длинная функция —
это нормально.

Однако если у вас есть сложная функция и вы подозреваете, что не самый
одарённый первокурсник старшей школы может даже не понять, в чём вообще суть
функции, вам следует тем более строго придерживаться максимальных пределов.
Используйте вспомогательные функции с описательными именами (вы можете попросить
компилятор встроить их, если считаете, что это критично для производительности,
и он, вероятно, справится с этим лучше, чем справились бы вы).

Ещё одна мера для функции — число локальных переменных.  Их не должно быть
больше 5–10, иначе вы делаете что-то не так.  Переосмыслите функцию и разбейте
её на меньшие части.  Человеческий мозг в целом легко удерживает примерно
7 разных вещей, всё, что сверх этого, и он запутывается.  Вы знаете, что вы
гениальны, но, возможно, вам захочется понять, что вы сделали, через 2 недели.

В исходных файлах разделяйте функции одной пустой строкой.  Если функция
экспортируется, макрос **EXPORT** для неё должен следовать сразу после строки
с закрывающей фигурной скобкой функции.  Например:

.. code-block:: c

	int system_is_up(void)
	{
		return system_state == SYSTEM_RUNNING;
	}
	EXPORT_SYMBOL(system_is_up);

6.1) Прототипы функций
**********************

В прототипах функций включайте имена параметров вместе с их типами данных.
Хотя язык C этого не требует, в Linux это предпочтительно, поскольку это
простой способ добавить ценную информацию для читателя.

Не используйте ключевое слово ``extern`` с объявлениями функций, поскольку это
удлиняет строки и не является строго необходимым.

При написании прототипов функций, пожалуйста, придерживайтесь `регулярного
порядка элементов
<https://lore.kernel.org/mm-commits/CAHk-=wiOCLRny5aifWNhr621kYrJwhfURsa0vFPeUEm8mF0ufg@mail.gmail.com/>`_.
Например, используя такой пример объявления функции::

 __init void * __must_check action(enum magic value, size_t size, u8 count,
				   char *fmt, ...) __printf(4, 5) __malloc;

Предпочтительный порядок элементов для прототипа функции таков:

- класс хранения (ниже — ``static __always_inline``, отметим, что
  ``__always_inline`` технически является атрибутом, но обрабатывается как
  ``inline``)
- атрибуты класса хранения (здесь — ``__init`` -- т. е. объявления секций, но
  также и такие вещи, как ``__cold``)
- тип возвращаемого значения (здесь — ``void *``)
- атрибуты типа возвращаемого значения (здесь — ``__must_check``)
- имя функции (здесь — ``action``)
- параметры функции (здесь — ``(enum magic value, size_t size, u8 count, char *fmt, ...)``,
  отметим, что имена параметров должны всегда включаться)
- атрибуты параметров функции (здесь — ``__printf(4, 5)``)
- атрибуты поведения функции (здесь — ``__malloc``)

Обратите внимание, что для **определения** функции (т. е. собственно тела
функции) компилятор не допускает атрибуты параметров функции после параметров
функции. В этих случаях они должны идти после атрибутов класса хранения
(например, обратите внимание на изменённое положение ``__printf(4, 5)`` ниже по
сравнению с примером **объявления** выше)::

 static __always_inline __init __printf(4, 5) void * __must_check action(enum magic value,
		size_t size, u8 count, char *fmt, ...) __malloc
 {
	...
 }

7) Централизованный выход из функций
------------------------------------

Хотя некоторые люди и считают его устаревшим, эквивалент оператора goto часто
используется компиляторами в форме инструкции безусловного перехода.

Оператор goto оказывается удобным, когда функция выходит из нескольких мест и
нужно выполнить некоторую общую работу, например очистку.  Если очистка не
нужна, то просто возвращайтесь напрямую.

Выбирайте имена меток, которые говорят, что делает goto или зачем существует
goto.  Примером хорошего имени могло бы быть ``out_free_buffer:``, если goto
освобождает ``buffer``.  Избегайте использования имён в стиле GW-BASIC, таких
как ``err1:`` и ``err2:``, поскольку вам пришлось бы перенумеровывать их, если
вы когда-либо добавите или удалите пути выхода, и они всё равно затрудняют
проверку корректности.

Обоснование использования goto таково:

- безусловные операторы легче понимать и отслеживать
- уменьшается вложенность
- предотвращаются ошибки из-за необновления отдельных точек выхода при внесении
  изменений
- избавляет компилятор от работы по оптимизации удаления избыточного кода ;)

.. code-block:: c

	int fun(int a)
	{
		int result = 0;
		char *buffer;

		buffer = kmalloc(SIZE, GFP_KERNEL);
		if (!buffer)
			return -ENOMEM;

		if (condition1) {
			while (loop1) {
				...
			}
			result = 1;
			goto out_free_buffer;
		}
		...
	out_free_buffer:
		kfree(buffer);
		return result;
	}

Распространённый тип ошибок, о котором следует знать, — это ``ошибки одной
метки err``, которые выглядят вот так:

.. code-block:: c

	err:
		kfree(foo->bar);
		kfree(foo);
		return ret;

Ошибка в этом коде состоит в том, что на некоторых путях выхода ``foo`` равно
NULL.  Обычно исправлением для этого является разбиение на две метки ошибок
``err_free_bar:`` и ``err_free_foo:``:

.. code-block:: c

	err_free_bar:
		kfree(foo->bar);
	err_free_foo:
		kfree(foo);
		return ret;

В идеале вам следует имитировать ошибки, чтобы протестировать все пути выхода.


8) Комментирование
------------------

Комментарии — это хорошо, но есть и опасность чрезмерного комментирования.
НИКОГДА не пытайтесь объяснить в комментарии, КАК работает ваш код: гораздо
лучше написать код так, чтобы **работа** была очевидна, а объяснять плохо
написанный код — это пустая трата времени.

В общем, вы хотите, чтобы ваши комментарии говорили, ЧТО делает ваш код, а не
КАК.  Также старайтесь избегать размещения комментариев внутри тела функции:
если функция настолько сложна, что вам нужно отдельно комментировать её части,
вам, вероятно, стоит на время вернуться к главе 6.  Вы можете делать небольшие
комментарии, чтобы отметить или предупредить о чём-то особенно умном (или
уродливом), но старайтесь избегать излишеств.  Вместо этого размещайте
комментарии в начале функции, рассказывая людям, что она делает, и, возможно,
ПОЧЕМУ она это делает.

При комментировании функций API ядра, пожалуйста, используйте формат kernel-doc.
Подробности см. в файлах :ref:`Documentation/doc-guide/ <doc_guide>` и
``tools/docs/kernel-doc``. Обратите внимание, что опасность чрезмерного
комментирования в равной мере применима и к комментариям kernel-doc. Не
добавляйте шаблонные kernel-doc, которые просто повторяют то, что очевидно из
сигнатуры функции.

Предпочтительный стиль для длинных (многострочных) комментариев таков:

.. code-block:: c

	/*
	 * This is the preferred style for multi-line
	 * comments in the Linux kernel source code.
	 * Please use it consistently.
	 *
	 * Description:  A column of asterisks on the left side,
	 * with beginning and ending almost-blank lines.
	 */

Также важно комментировать данные, будь то базовые типы или производные типы.
С этой целью используйте лишь одно объявление данных на строку (без запятых для
множественных объявлений данных).  Это оставляет вам место для небольшого
комментария к каждому элементу, объясняющего его использование.


9) Вы устроили из этого беспорядок
----------------------------------

Это нормально, мы все так делаем.  Вам, вероятно, ваш давний помощник —
пользователь Unix сказал, что ``GNU emacs`` автоматически форматирует исходники
на C за вас, и вы заметили, что да, он это делает, но используемые им значения
по умолчанию менее чем желательны (на самом деле они хуже случайного набора —
бесконечное число обезьян, набирающих в GNU emacs, никогда не создало бы хорошей
программы).

Итак, вы можете либо избавиться от GNU emacs, либо изменить его, чтобы он
использовал более разумные значения.  Чтобы сделать последнее, вы можете
поместить следующее в ваш файл .emacs:

.. code-block:: elisp

  (defun c-lineup-arglist-tabs-only (ignored)
    "Line up argument lists by tabs, not spaces"
    (let* ((anchor (c-langelem-pos c-syntactic-element))
           (column (c-langelem-2nd-pos c-syntactic-element))
           (offset (- (1+ column) anchor))
           (steps (floor offset c-basic-offset)))
      (* (max steps 1)
         c-basic-offset)))

  (dir-locals-set-class-variables
   'linux-kernel
   '((c-mode . (
          (c-basic-offset . 8)
          (c-label-minimum-indentation . 0)
          (c-offsets-alist . (
                  (arglist-close         . c-lineup-arglist-tabs-only)
                  (arglist-cont-nonempty .
                      (c-lineup-gcc-asm-reg c-lineup-arglist-tabs-only))
                  (arglist-intro         . +)
                  (brace-list-intro      . +)
                  (c                     . c-lineup-C-comments)
                  (case-label            . 0)
                  (comment-intro         . c-lineup-comment)
                  (cpp-define-intro      . +)
                  (cpp-macro             . -1000)
                  (cpp-macro-cont        . +)
                  (defun-block-intro     . +)
                  (else-clause           . 0)
                  (func-decl-cont        . +)
                  (inclass               . +)
                  (inher-cont            . c-lineup-multi-inher)
                  (knr-argdecl-intro     . 0)
                  (label                 . -1000)
                  (statement             . 0)
                  (statement-block-intro . +)
                  (statement-case-intro  . +)
                  (statement-cont        . +)
                  (substatement          . +)
                  ))
          (indent-tabs-mode . t)
          (show-trailing-whitespace . t)
          ))))

  (dir-locals-set-directory-class
   (expand-file-name "~/src/linux-trees")
   'linux-kernel)

Это заставит emacs лучше работать со стилем кодирования ядра для файлов на C,
расположенных в ``~/src/linux-trees``.

Но даже если вам не удастся заставить emacs делать разумное форматирование, не
всё потеряно: используйте ``indent``.

Теперь, опять же, у GNU indent те же безмозглые настройки, что и у GNU emacs,
поэтому вам нужно дать ему несколько опций командной строки.  Однако это не так
уж плохо, потому что даже создатели GNU indent признают авторитет K&R
(люди из GNU не злы, они просто серьёзно заблуждаются в этом вопросе), так что
вы просто даёте indent опции ``-kr -i8`` (означает ``K&R, 8 character indents``)
или используете ``scripts/Lindent``, который делает отступы в новейшем стиле.

У ``indent`` множество опций, и особенно когда дело доходит до переформатирования
комментариев, вы, возможно, захотите взглянуть на man-страницу.  Но помните:
``indent`` не является лекарством от плохого программирования.

Обратите внимание, что вы также можете использовать инструмент ``clang-format``,
чтобы помочь себе с этими правилами, быстро автоматически переформатировать части
вашего кода и проверять файлы целиком, чтобы выявить ошибки стиля кодирования,
опечатки и возможные улучшения. Он также удобен для сортировки ``#includes``,
для выравнивания переменных/макросов, для перекомпоновки текста и других
подобных задач.  Подробности см. в файле
:ref:`Documentation/dev-tools/clang-format.rst <clangformat>`.

Некоторые базовые настройки редактора, такие как отступы и окончания строк,
будут установлены автоматически, если вы используете редактор, совместимый с
EditorConfig. Дополнительную информацию см. на официальном сайте EditorConfig:
https://editorconfig.org/

10) Конфигурационные файлы Kconfig
----------------------------------

Для всех конфигурационных файлов Kconfig* по всему дереву исходников отступы
несколько отличаются.  Строки под определением ``config`` имеют отступ в одну
табуляцию, в то время как текст справки имеет дополнительный отступ в два
пробела.  Пример::

  config AUDIT
	bool "Auditing support"
	depends on NET
	help
	  Enable auditing infrastructure that can be used with another
	  kernel subsystem, such as SELinux (which requires this for
	  logging of avc messages output).  Does not do system-call
	  auditing without CONFIG_AUDITSYSCALL.

Серьёзно опасные возможности (такие как поддержка записи для определённых
файловых систем) должны заметно сообщать об этом в строке своего приглашения::

  config ADFS_FS_RW
	bool "ADFS write support (DANGEROUS)"
	depends on ADFS_FS
	...

Полную документацию по конфигурационным файлам см. в файле
Documentation/kbuild/kconfig-language.rst.


11) Структуры данных
--------------------

Структуры данных, которые имеют видимость за пределами однопоточной среды, в
которой они создаются и уничтожаются, всегда должны иметь счётчики ссылок.  В
ядре сборки мусора не существует (а вне ядра сборка мусора медленна и
неэффективна), что означает, что вы абсолютно **обязаны** подсчитывать ссылки
для всех своих использований.

Подсчёт ссылок означает, что вы можете избежать блокировок и позволяет
нескольким пользователям иметь параллельный доступ к структуре данных — и не
беспокоиться о том, что структура внезапно исчезнет из-под них только потому,
что они заснули или делали что-то ещё в течение некоторого времени.

Обратите внимание, что блокировка **не** является заменой подсчёта ссылок.
Блокировка используется для поддержания согласованности структур данных, в то
время как подсчёт ссылок — это техника управления памятью.  Обычно нужны оба, и
их не следует путать друг с другом.

Многие структуры данных действительно могут иметь два уровня подсчёта ссылок,
когда есть пользователи разных ``классов``.  Счётчик подкласса подсчитывает
число пользователей подкласса и уменьшает глобальный счётчик лишь один раз,
когда счётчик подкласса доходит до нуля.

Примеры такого рода ``многоуровневого подсчёта ссылок`` можно найти в управлении
памятью (``struct mm_struct``: mm_users и mm_count) и в коде файловой системы
(``struct super_block``: s_count и s_active).

Помните: если другой поток может найти вашу структуру данных, а у вас нет на неё
счётчика ссылок, у вас почти наверняка есть ошибка.


12) Макросы, Enum'ы и RTL
-------------------------

Имена макросов, определяющих константы, и меток в enum'ах пишутся заглавными
буквами.

.. code-block:: c

	#define CONSTANT 0x12345

Enum'ы предпочтительны при определении нескольких связанных констант.

Имена макросов ЗАГЛАВНЫМИ буквами приветствуются, но макросы, напоминающие
функции, могут именоваться в нижнем регистре.

В общем, встраиваемые функции предпочтительнее макросов, напоминающих функции.

Макросы с несколькими операторами следует заключать в блок do - while:

.. code-block:: c

	#define macrofun(a, b, c)			\
		do {					\
			if (a == 5)			\
				do_this(b, c);		\
		} while (0)

Функциеподобные макросы с неиспользуемыми параметрами следует заменять
статическими встраиваемыми функциями, чтобы избежать проблемы с
неиспользуемыми переменными:

.. code-block:: c

	static inline void fun(struct foo *foo)
	{
	}

В силу исторических практик многие файлы всё ещё применяют подход
«приведения к (void)» для вычисления параметров. Однако этот метод
нежелателен.  Встраиваемые функции решают проблему «выражение с побочными
эффектами вычисляется более одного раза», обходят проблемы с неиспользуемыми
переменными и почему-то в целом лучше документированы, чем макросы.

.. code-block:: c

	/*
	 * Avoid doing this whenever possible and instead opt for static
	 * inline functions
	 */
	#define macrofun(foo) do { (void) (foo); } while (0)

Вещи, которых следует избегать при использовании макросов:

1) макросы, влияющие на поток управления:

.. code-block:: c

	#define FOO(x)					\
		do {					\
			if (blah(x) < 0)		\
				return -EBUGGERED;	\
		} while (0)

— это **очень** плохая идея.  Это выглядит как вызов функции, но выходит из
``вызывающей`` функции; не ломайте внутренние парсеры тех, кто будет читать код.

2) макросы, зависящие от наличия локальной переменной с магическим именем:

.. code-block:: c

	#define FOO(val) bar(index, val)

может выглядеть как хорошая вещь, но это чертовски сбивает с толку, когда кто-то
читает код, и подвержено поломке от, казалось бы, безобидных изменений.

3) макросы с аргументами, которые используются как l-значения: FOO(x) = y;
укусит вас, если кто-то, например, превратит FOO во встраиваемую функцию.

4) забывание о приоритете: макросы, определяющие константы с помощью выражений,
должны заключать выражение в скобки. Остерегайтесь подобных проблем с макросами,
использующими параметры.

.. code-block:: c

	#define CONSTANT 0x4000
	#define CONSTEXP (CONSTANT | 3)

5) коллизии пространства имён при определении локальных переменных в макросах,
напоминающих функции:

.. code-block:: c

	#define FOO(x)				\
	({					\
		typeof(x) ret;			\
		ret = calc_ret(x);		\
		(ret);				\
	})

ret — распространённое имя для локальной переменной; __foo_ret с меньшей
вероятностью столкнётся с существующей переменной.

Руководство по cpp исчерпывающе рассматривает макросы. Руководство по
внутреннему устройству gcc также охватывает RTL, который часто используется с
языком ассемблера в ядре.


13) Печать сообщений ядра
-------------------------

Разработчики ядра любят, чтобы их считали грамотными. Следите за правописанием
сообщений ядра, чтобы произвести хорошее впечатление. Не используйте
неправильные сокращения, такие как ``dont``; вместо этого используйте ``do not``
или ``don't``. Делайте сообщения краткими, ясными и однозначными.

Сообщения ядра не обязательно завершать точкой.

Печать чисел в скобках (%d) не добавляет ценности, и её следует избегать.

В <linux/dev_printk.h> есть ряд диагностических макросов модели драйверов,
которые вам следует использовать, чтобы гарантировать, что сообщения сопоставлены
с правильным устройством и драйвером и помечены правильным уровнем:  dev_err(),
dev_warn(), dev_info() и так далее.  Для сообщений, не связанных с конкретным
устройством, <linux/printk.h> определяет pr_notice(), pr_info(), pr_warn(),
pr_err() и т. д. Когда драйверы работают правильно, они молчат, поэтому
предпочитайте использовать dev_dbg/pr_debug, если только что-то не идёт не так.

Придумывание хороших отладочных сообщений может быть весьма непростой задачей; и
когда они у вас есть, они могут оказаться огромной помощью для удалённого
устранения неполадок.  Однако печать отладочных сообщений обрабатывается иначе,
чем печать других, не отладочных сообщений.  В то время как другие функции
pr_XXX() печатают безусловно, pr_debug() этого не делает; по умолчанию он
исключается из компиляции, если только не определён DEBUG или не установлен
CONFIG_DYNAMIC_DEBUG.  То же верно и для dev_dbg(), а связанное соглашение
использует VERBOSE_DEBUG, чтобы добавить сообщения dev_vdbg() к тем, что уже
включены через DEBUG.

У многих подсистем есть отладочные опции Kconfig, чтобы включить -DDEBUG в
соответствующем Makefile; в других случаях конкретные файлы делают #define DEBUG.
А когда отладочное сообщение должно быть напечатано безусловно, например если оно
уже находится внутри связанной с отладкой секции #ifdef, можно использовать
printk(KERN_DEBUG ...).


14) Выделение памяти
--------------------

Ядро предоставляет следующие распределители памяти общего назначения:
kmalloc(), kzalloc(), kmalloc_array(), kcalloc(), vmalloc() и vzalloc().
За дополнительной информацией о них, пожалуйста, обращайтесь к документации API.
:ref:`Documentation/core-api/memory-allocation.rst
<memory_allocation>`

Предпочтительная форма для передачи размера структуры такова:

.. code-block:: c

	p = kmalloc(sizeof(*p), ...);

Альтернативная форма, где имя структуры выписано полностью, вредит читаемости и
вводит возможность ошибки, когда тип переменной-указателя изменяется, а
соответствующий sizeof, передаваемый в распределитель памяти, — нет.

Приведение возвращаемого значения, которое является указателем void, избыточно.
Преобразование из указателя void в любой другой тип указателя гарантируется
языком программирования C.

Предпочтительная форма для выделения массива такова:

.. code-block:: c

	p = kmalloc_array(n, sizeof(...), ...);

Предпочтительная форма для выделения обнулённого массива такова:

.. code-block:: c

	p = kcalloc(n, sizeof(...), ...);

Обе формы проверяют переполнение при размере выделения n * sizeof(...) и
возвращают NULL, если оно произошло.

Все эти обобщённые функции выделения выдают дамп стека при сбое, если они
используются без __GFP_NOWARN, поэтому нет смысла выдавать дополнительное
сообщение о сбое, когда возвращается NULL.

15) Болезнь встраивания
-----------------------

По-видимому, существует распространённое заблуждение, что у gcc есть магическая
опция ускорения «сделай меня быстрее» под названием ``inline``. Хотя
использование встраиваемых функций может быть уместным (например, как средство
замены макросов, см. главу 12), очень часто оно неуместно. Обильное
использование ключевого слова inline ведёт к гораздо большему ядру, что, в свою
очередь, замедляет систему в целом из-за большего объёма icache для CPU и просто
потому, что доступно меньше памяти для pagecache. Только подумайте об этом;
промах pagecache вызывает поиск на диске, который легко занимает 5 миллисекунд.
В эти 5 миллисекунд может вместиться ОЧЕНЬ МНОГО тактов CPU.

Разумное практическое правило — не ставить inline у функций, которые содержат
более 3 строк кода. Исключением из этого правила являются случаи, когда известно,
что параметр является константой времени компиляции, и в результате этой
константности вы *знаете*, что компилятор сможет оптимизировать большую часть
вашей функции на этапе компиляции. Хороший пример этого последнего случая
см. во встраиваемой функции kmalloc().

Часто люди утверждают, что добавление inline к функциям, которые являются
static и используются лишь один раз, всегда выигрышно, поскольку нет
компромисса по размеру. Хотя это технически верно, gcc способен встраивать их
автоматически без помощи, и проблема сопровождения по удалению inline, когда
появляется второй пользователь, перевешивает потенциальную ценность подсказки,
которая говорит gcc делать то, что он сделал бы в любом случае.


16) Возвращаемые значения и имена функций
-----------------------------------------

Функции могут возвращать значения многих разных видов, и одно из наиболее
распространённых — значение, указывающее, успешно ли функция выполнилась или
завершилась с ошибкой.  Такое значение может быть представлено как
целое число — код ошибки (-Exxx = ошибка, 0 = успех) или булево ``succeeded``
(0 = ошибка, ненулевое = успех).

Смешивание этих двух видов представлений — благодатный источник
трудноуловимых ошибок.  Если бы язык C включал строгое различие между целыми
числами и булевыми значениями, то компилятор находил бы эти ошибки за нас... но
он этого не делает.  Чтобы помочь предотвратить такие ошибки, всегда следуйте
этому соглашению::

	Если имя функции — это действие или повелительная команда,
	функция должна возвращать целое число — код ошибки.  Если имя —
	это предикат, функция должна возвращать булево "succeeded".

Например, ``add work`` — это команда, и функция add_work() возвращает 0 в случае
успеха или -EBUSY в случае ошибки.  Точно так же ``PCI device present`` — это
предикат, и функция pci_dev_present() возвращает 1, если ей удаётся найти
подходящее устройство, или 0, если нет.

Все EXPORT'ированные функции должны соблюдать это соглашение, как и все
публичные функции.  Приватным (static) функциям это не обязательно, но
рекомендуется, чтобы они это делали.

Функции, чьё возвращаемое значение является фактическим результатом вычисления,
а не указанием на то, было ли вычисление успешным, не подчиняются этому правилу.
Обычно они указывают на ошибку, возвращая некоторый результат вне допустимого
диапазона.  Типичными примерами были бы функции, возвращающие указатели; они
используют NULL или механизм ERR_PTR для сообщения об ошибке.


17) Использование bool
----------------------

Тип bool ядра Linux — это псевдоним для типа C99 _Bool. Значения bool могут
вычисляться только в 0 или 1, а неявное или явное преобразование в bool
автоматически преобразует значение в true или false. При использовании типов
bool конструкция !! не нужна, что устраняет класс ошибок.

При работе со значениями bool следует использовать определения true и false
вместо 1 и 0.

Типы возвращаемого значения bool и стековые переменные bool всегда нормально
использовать, когда это уместно. Использование bool поощряется для повышения
читаемости и часто является лучшим вариантом, чем 'int', для хранения булевых
значений.

Не используйте bool, если важны компоновка кэш-линии или размер значения,
поскольку его размер и выравнивание варьируются в зависимости от компилируемой
архитектуры. Структуры, оптимизированные по выравниванию и размеру, не должны
использовать bool.

Если у структуры много значений true/false, рассмотрите возможность объединить
их в битовое поле с членами по 1 биту или использовать подходящий тип
фиксированной ширины, такой как u8.

Аналогично для аргументов функций множество значений true/false можно объединить
в единый побитовый аргумент 'flags', и 'flags' часто может быть более читаемой
альтернативой, если в местах вызова есть голые константы true/false.

В остальном ограниченное использование bool в структурах и аргументах может
повысить читаемость.

18) Не изобретайте заново макросы ядра
--------------------------------------

В include/linux/ есть много заголовочных файлов, содержащих ряд макросов,
которые вам следует использовать, вместо того чтобы явно кодировать какой-то их
вариант самостоятельно.  Например, если вам нужно вычислить длину массива,
воспользуйтесь макросом

.. code-block:: c

	#define ARRAY_SIZE(x) (sizeof(x) / sizeof((x)[0]))

который определён в array_size.h.

Аналогично, если вам нужно вычислить размер некоторого члена структуры,
используйте

.. code-block:: c

	#define sizeof_field(t, f) (sizeof(((t*)0)->f))

который определён в stddef.h.

Есть также макросы min() и max(), определённые в minmax.h, которые выполняют
строгую проверку типов, если они вам нужны. Не стесняйтесь изучить заголовочные
файлы, чтобы увидеть, что ещё уже определено и что вам не следует воспроизводить
в своём коде.


19) Modeline'ы редакторов и прочий хлам
---------------------------------------

Некоторые редакторы могут интерпретировать конфигурационную информацию,
встроенную в исходные файлы, обозначенную специальными маркерами.  Например,
emacs интерпретирует строки, помеченные вот так:

.. code-block:: c

	-*- mode: c -*-

Или вот так:

.. code-block:: c

	/*
	Local Variables:
	compile-command: "gcc -DMAGIC_DEBUG_FLAG foo.c"
	End:
	*/

Vim интерпретирует маркеры, которые выглядят вот так:

.. code-block:: c

	/* vim:set sw=8 noet */

Не включайте ничего из этого в исходные файлы.  У людей есть свои собственные
персональные конфигурации редакторов, и ваши исходные файлы не должны их
переопределять.  Это включает маркеры для отступов и конфигурации режима.  Люди
могут использовать свой собственный пользовательский режим или иметь какой-то
другой магический метод, чтобы заставить отступы работать правильно.


20) Встраиваемый ассемблер
--------------------------

В архитектурно-зависимом коде вам, возможно, понадобится использовать
встраиваемый ассемблер для взаимодействия с функциональностью CPU или платформы.
Не стесняйтесь делать это, когда необходимо.  Однако не используйте встраиваемый
ассемблер без надобности, когда работу может сделать C.  Вы можете и должны
дёргать аппаратуру из C, когда это возможно.

Рассмотрите возможность написания простых вспомогательных функций, которые
оборачивают распространённые фрагменты встраиваемого ассемблера, вместо того
чтобы многократно писать их с небольшими вариациями.  Помните, что встраиваемый
ассемблер может использовать параметры C.

Большие, нетривиальные ассемблерные функции должны располагаться в .S-файлах с
соответствующими прототипами на C, определёнными в заголовочных файлах на C.
Прототипы на C для ассемблерных функций должны использовать ``asmlinkage``.

Вам, возможно, понадобится пометить ваш оператор asm как volatile, чтобы
предотвратить удаление его GCC, если GCC не замечает каких-либо побочных
эффектов.  Однако вам не всегда нужно это делать, и излишнее использование этого
может ограничить оптимизацию.

При написании одного оператора встраиваемого ассемблера, содержащего несколько
инструкций, поместите каждую инструкцию на отдельную строку в отдельной строке
в кавычках и завершите каждую строку, кроме последней, символами ``\n\t``, чтобы
правильно сделать отступ следующей инструкции в выводе ассемблера:

.. code-block:: c

	asm ("magic %reg1, #42\n\t"
	     "more_magic %reg2, %reg3"
	     : /* outputs */ : /* inputs */ : /* clobbers */);


21) Условная компиляция
-----------------------

Везде, где это возможно, не используйте препроцессорные условия (#if, #ifdef) в
.c-файлах; это делает код труднее читать, а логику труднее отслеживать.  Вместо
этого используйте такие условия в заголовочном файле, определяющем функции для
использования в этих .c-файлах, предоставляя версии-заглушки, не выполняющие
никаких действий, в случае #else, а затем безусловно вызывайте эти функции из
.c-файлов.  Компилятор избежит генерации какого-либо кода для вызовов заглушек,
производя идентичные результаты, но логику по-прежнему будет легко отслеживать.

Предпочитайте исключать из компиляции целые функции, а не части функций или
части выражений.  Вместо того чтобы помещать ifdef в выражение, вынесите часть
или всё выражение в отдельную вспомогательную функцию и примените условие к этой
функции.

Если у вас есть функция или переменная, которая потенциально может оказаться
неиспользуемой в конкретной конфигурации, и компилятор предупредил бы о том, что
её определение оказывается неиспользуемым, пометьте определение как
__maybe_unused, а не оборачивайте его в препроцессорное условие.  (Однако если
функция или переменная *всегда* оказывается неиспользуемой, удалите её.)

Внутри кода, где это возможно, используйте макрос IS_ENABLED, чтобы преобразовать
символ Kconfig в булево выражение C, и используйте его в обычном условном
операторе C:

.. code-block:: c

	if (IS_ENABLED(CONFIG_SOMETHING)) {
		...
	}

Компилятор свернёт условие путём вычисления константы и включит или исключит
блок кода точно так же, как и с #ifdef, поэтому это не добавит никаких
накладных расходов во время выполнения.  Однако этот подход всё же позволяет
компилятору C видеть код внутри блока и проверять его на корректность
(синтаксис, типы, ссылки на символы и т. д.).  Таким образом, вам всё же
придётся использовать #ifdef, если код внутри блока ссылается на символы,
которых не будет существовать, если условие не выполнено.

В конце любого нетривиального блока #if или #ifdef (более нескольких строк)
размещайте комментарий после #endif на той же строке, отмечающий использованное
условное выражение.  Например:

.. code-block:: c

	#ifdef CONFIG_SOMETHING
	...
	#endif /* CONFIG_SOMETHING */


22) Не роняйте ядро
-------------------

В общем, решение уронить ядро принадлежит пользователю, а не разработчику ядра.

Избегайте panic()
*****************

panic() следует использовать с осторожностью и в первую очередь только во время
загрузки системы.  panic() приемлем, например, когда заканчивается память во
время загрузки и нет возможности продолжать.

Используйте WARN(), а не BUG()
******************************

Не добавляйте новый код, который использует какой-либо из вариантов BUG(),
таких как BUG(), BUG_ON() или VM_BUG_ON(). Вместо этого используйте вариант
WARN*(), предпочтительно WARN_ON_ONCE(), и, возможно, с кодом восстановления.
Код восстановления не требуется, если нет разумного способа хотя бы частично
восстановиться.

«Мне лень делать обработку ошибок» — не оправдание для использования BUG().
Серьёзные внутренние повреждения без возможности продолжения по-прежнему могут
использовать BUG(), но нуждаются в хорошем обосновании.

Используйте WARN_ON_ONCE(), а не WARN() или WARN_ON()
*****************************************************

WARN_ON_ONCE() в целом предпочтительнее WARN() или WARN_ON(), потому что для
данного условия предупреждения характерно, если оно вообще возникает,
возникать многократно. Это может заполнить и зациклить журнал ядра и может даже
замедлить систему настолько, что чрезмерное журналирование превратится в свою
собственную, дополнительную проблему.

Не используйте WARN легкомысленно
*********************************

WARN*() предназначен для неожиданных ситуаций «этого-никогда-не-должно-случиться».
Макросы WARN*() не должны использоваться для чего-либо, что ожидается во время
нормальной работы. Это не утверждения пред- или постусловий, например. Ещё раз:
WARN*() не должен использоваться для условия, которое, как ожидается, легко
сработает, например в результате действий пространства пользователя.
pr_warn_once() — возможная альтернатива, если вам нужно уведомить пользователя
о проблеме.

Не беспокойтесь о пользователях panic_on_warn
*********************************************

Ещё несколько слов о panic_on_warn: помните, что ``panic_on_warn`` — это
доступная опция ядра, и что многие пользователи устанавливают эту опцию. Именно
поэтому выше есть раздел «Не используйте WARN легкомысленно». Однако
существование пользователей panic_on_warn не является веской причиной избегать
разумного использования WARN*(). Это потому, что тот, кто включает
panic_on_warn, явно попросил ядро упасть, если срабатывает WARN*(), и такие
пользователи должны быть готовы иметь дело с последствиями системы, которая
несколько более склонна падать.

Используйте BUILD_BUG_ON() для утверждений времени компиляции
*************************************************************

Использование BUILD_BUG_ON() приемлемо и поощряется, потому что это утверждение
времени компиляции, которое не имеет эффекта во время выполнения.
