Сведения о почтовых клиентах для Linux
======================================

Git
---

В наши дни большинство разработчиков используют ``git send-email`` вместо
обычных почтовых клиентов. Man-страница по этой команде весьма хороша. На
принимающей стороне сопровождающие применяют ``git am`` для наложения патчей.

Если вы только начинаете работать с ``git``, отправьте свой первый патч самому
себе. Сохраните его как обычный текст, включая все заголовки. Запустите
``git am raw_email.txt``, а затем просмотрите журнал изменений с помощью
``git log``. Когда это сработает, отправьте патч в соответствующий список (или
списки) рассылки.

Общие рекомендации
------------------

Патчи для ядра Linux отправляются по электронной почте, предпочтительно как
встроенный текст в теле письма. Некоторые сопровождающие принимают вложения,
но тогда вложения должны иметь content-type ``text/plain``. Тем не менее, к
вложениям в целом относятся неодобрительно, поскольку они усложняют
цитирование частей патча в процессе его рецензирования.

Также настоятельно рекомендуется использовать обычный текст в теле письма — как
для патчей, так и для прочих писем. Сайт https://useplaintext.email может быть
полезен для получения информации о том, как настроить ваш предпочитаемый
почтовый клиент, а также содержит перечень рекомендуемых почтовых клиентов, если
у вас ещё нет предпочтений.

Почтовые клиенты, используемые для патчей ядра Linux, должны отправлять текст
патча в неизменном виде. Например, они не должны изменять или удалять табуляции
или пробелы, даже в начале или в конце строк.

Не отправляйте патчи с ``format=flowed``. Это может вызвать неожиданные и
нежелательные переносы строк.

Не позволяйте вашему почтовому клиенту выполнять автоматический перенос слов
за вас. Это также может повредить ваш патч.

Почтовые клиенты не должны изменять кодировку символов текста. Патчи,
отправляемые по почте, должны быть только в кодировке ASCII или UTF-8.
Если вы настроите ваш почтовый клиент на отправку писем в кодировке UTF-8, вы
избежите некоторых возможных проблем с кодировкой.

Почтовые клиенты должны генерировать и сопровождать заголовки "References:" или
"In-Reply-To:", чтобы не нарушалась цепочка переписки (threading).

Копирование-вставка (copy-and-paste или cut-and-paste) обычно не работает для
патчей, поскольку табуляции преобразуются в пробелы. Использование xclipboard,
xclip и/или xcutsel может сработать, но лучше проверить это самостоятельно или
просто избегать копирования-вставки.

Не используйте подписи PGP/GPG в письмах, содержащих патчи. Это нарушает работу
многих скриптов, читающих и накладывающих патчи. (Это, по идее, можно исправить.)

Хорошей идеей будет отправить патч самому себе, сохранить полученное сообщение и
успешно наложить его с помощью 'patch', прежде чем отправлять патчи в списки
рассылки Linux.


Несколько советов по почтовым клиентам (MUA)
--------------------------------------------

Ниже приведены некоторые конкретные советы по настройке MUA для редактирования и
отправки патчей для ядра Linux. Они не претендуют на роль исчерпывающего обзора
настроек программных пакетов.


Условные обозначения:

- TUI = текстовый пользовательский интерфейс (text-based user interface)
- GUI = графический пользовательский интерфейс (graphical user interface)

Alpine (TUI)
************

Параметры конфигурации:

В разделе :menuselection:`Sending Preferences`:

- :menuselection:`Do Not Send Flowed Text` должно быть ``enabled``
- :menuselection:`Strip Whitespace Before Sending` должно быть ``disabled``

При составлении сообщения курсор следует поместить туда, где должен находиться
патч, после чего нажатие `CTRL-R` позволяет указать файл патча для вставки в
сообщение.

Claws Mail (GUI)
****************

Работает. Некоторые люди успешно используют его для патчей.

Чтобы вставить патч, используйте :menuselection:`Message-->Insert File` (`CTRL-I`)
или внешний редактор.

Если вставленный патч нужно отредактировать в окне составления Claws, параметр
"Auto wrapping" в
:menuselection:`Configuration-->Preferences-->Compose-->Wrapping` следует
отключить.

Evolution (GUI)
***************

Некоторые люди успешно используют его для патчей.

При составлении письма выберите: Preformat
  из :menuselection:`Format-->Paragraph Style-->Preformatted` (`CTRL-7`)
  либо на панели инструментов

Затем используйте:
:menuselection:`Insert-->Text File...` (`ALT-N x`)
для вставки патча.

Вы также можете выполнить ``diff -Nru old.c new.c | xclip``, выбрать
:menuselection:`Preformat`, а затем вставить средней кнопкой мыши.

Kmail (GUI)
***********

Некоторые люди успешно используют Kmail для патчей.

Настройка по умолчанию — не составлять письмо в HTML — является подходящей; не
включайте её.

При составлении письма в параметрах снимите флажок "word wrap". Единственный
недостаток в том, что любой набираемый вами в письме текст не будет переноситься
по словам, поэтому вам придётся вручную переносить текст перед патчем. Проще
всего обойти это так: составьте письмо с включённым переносом слов, затем
сохраните его как черновик. Когда вы снова откроете его из черновиков, он будет
жёстко перенесён по словам, и вы сможете снять флажок "word wrap", не теряя
существующего переноса.

Внизу письма поставьте общепринятый разделитель патча перед вставкой самого
патча: три дефиса (``---``).

Затем в пункте меню :menuselection:`Message` выберите
:menuselection:`insert file` и укажите свой патч.
В качестве дополнительного бонуса вы можете настроить панель инструментов
создания сообщения и поместить туда значок :menuselection:`insert file`.

Сделайте окно составителя достаточно широким, чтобы строки не переносились.
Начиная с KMail 1.13.5 (KDE 4.5.4), KMail будет применять перенос слов при
отправке письма, если строки переносятся в окне составителя. Отключения переноса
слов в меню Options недостаточно. Таким образом, если в вашем патче есть очень
длинные строки, вы должны сделать окно составителя очень широким перед отправкой
письма. См.: https://bugs.kde.org/show_bug.cgi?id=174034

Вы можете безопасно подписывать вложения GPG, но для патчей предпочтителен
встроенный текст, поэтому не подписывайте их GPG. Подписывание патчей, вставленных
как встроенный текст, затруднит их извлечение из 7-битной кодировки.

Если вам абсолютно необходимо отправлять патчи как вложения вместо встраивания
их в виде текста, щёлкните правой кнопкой мыши по вложению и выберите
:menuselection:`properties`, затем выделите :menuselection:`Suggest automatic display`,
чтобы вложение отображалось встроенно и было более удобным для просмотра.

При сохранении патчей, отправленных как встроенный текст, выберите письмо,
содержащее патч, в панели списка сообщений, щёлкните правой кнопкой мыши и
выберите :menuselection:`save as`. Вы можете использовать всё письмо без
изменений в качестве патча, если оно было правильно составлено. Письма
сохраняются с правами на чтение и запись только для пользователя, поэтому вам
придётся применить к ним chmod, чтобы сделать их читаемыми для группы и
остальных, если вы копируете их в другое место.

Mutt (TUI)
**********

Множество разработчиков Linux используют ``mutt``, так что он, должно быть,
работает довольно хорошо.

Mutt не поставляется с редактором, поэтому какой бы редактор вы ни использовали,
его следует применять так, чтобы не было автоматических переносов строк.
Большинство редакторов имеют параметр :menuselection:`insert file`, который
вставляет содержимое файла без изменений.

Чтобы использовать ``vim`` с mutt::

  set editor="vi"

Если вы используете xclip, введите команду::

  :set paste

перед нажатием средней кнопки или shift-insert, либо используйте::

  :r filename

если вы хотите включить патч встроенно.
(a)ttach прекрасно работает без ``set paste``.

Вы также можете генерировать патчи с помощью ``git format-patch``, а затем
использовать Mutt для их отправки::

    $ mutt -H 0001-some-bug-fix.patch

Параметры конфигурации:

Должно работать с настройками по умолчанию.
Тем не менее, хорошей идеей будет установить ``send_charset`` в::

  set send_charset="us-ascii:utf-8"

Mutt очень гибко настраивается. Вот минимальная конфигурация, чтобы начать
использовать Mutt для отправки патчей через Gmail::

  # .muttrc
  # ================  IMAP  ====================
  set imap_user = 'yourusername@gmail.com'
  set imap_pass = 'yourpassword'
  set spoolfile = imaps://imap.gmail.com/INBOX
  set folder = imaps://imap.gmail.com/
  set record="imaps://imap.gmail.com/[Gmail]/Sent Mail"
  set postponed="imaps://imap.gmail.com/[Gmail]/Drafts"
  set mbox="imaps://imap.gmail.com/[Gmail]/All Mail"

  # ================  SMTP  ====================
  set smtp_url = "smtp://username@smtp.gmail.com:587/"
  set smtp_pass = $imap_pass
  set ssl_force_tls = yes # Require encrypted connection

  # ================  Composition  ====================
  set editor = `echo \$EDITOR`
  set edit_headers = yes  # See the headers when editing
  set charset = UTF-8     # value of $LANG; also fallback for send_charset
  # Sender, email address, and sign-off line must match
  unset use_domain        # because joe@localhost is just embarrassing
  set realname = "YOUR NAME"
  set from = "username@gmail.com"
  set use_from = yes

В документации Mutt гораздо больше информации:

    https://gitlab.com/muttmua/mutt/-/wikis/UseCases/Gmail

    http://www.mutt.org/doc/manual/

Sylpheed (GUI)
**************

- Хорошо работает для встраивания текста (или использования вложений).
- Позволяет использовать внешний редактор.
- Медленно работает на больших папках.
- Не выполняет TLS SMTP-аутентификацию через соединение без SSL.
- Имеет полезную линейку в окне составления.
- При добавлении адресов в адресную книгу некорректно понимает отображаемое имя.

Thunderbird (GUI)
*****************

Thunderbird — это клон Outlook, который любит коверкать текст, но есть способы
принудить его вести себя как надо.

После внесения изменений, включая установку расширений, вам нужно перезапустить
Thunderbird.

- Разрешите использование внешнего редактора:

  Проще всего работать с Thunderbird и патчами с помощью расширений, которые
  открывают ваш любимый внешний редактор.

  Вот несколько примеров расширений, способных это делать.

  - "External Editor Revived"

    https://github.com/Frederick888/external-editor-revived

    https://addons.thunderbird.net/en-GB/thunderbird/addon/external-editor-revived/

    Оно требует установки "native messaging host".
    Пожалуйста, прочитайте вики, которую можно найти здесь:
    https://github.com/Frederick888/external-editor-revived/wiki

  - "External Editor"

    https://github.com/exteditor/exteditor

    Чтобы сделать это, загрузите и установите расширение, затем откройте окно
    :menuselection:`compose`, добавьте для него кнопку с помощью
    :menuselection:`View-->Toolbars-->Customize...`
    после чего просто нажимайте на новую кнопку, когда захотите использовать
    внешний редактор.

    Обратите внимание, что "External Editor" требует, чтобы ваш редактор не
    выполнял fork, или, другими словами, редактор не должен завершаться до
    своего закрытия. Возможно, вам придётся передать дополнительные флаги или
    изменить настройки вашего редактора. В частности, если вы используете gvim,
    то вы должны передать gvim опцию -f, поместив ``/usr/bin/gvim --nofork"``
    (если бинарный файл находится в ``/usr/bin``) в поле текстового редактора в
    настройках :menuselection:`external editor`. Если вы используете какой-то
    другой редактор, то, пожалуйста, прочитайте его руководство, чтобы узнать,
    как это сделать.

Чтобы вбить немного здравого смысла во встроенный редактор, сделайте следующее:

- Отредактируйте настройки конфигурации Thunderbird так, чтобы он не использовал
  ``format=flowed``! Перейдите в главное окно и найдите кнопку основного
  выпадающего меню.
  :menuselection:`Main Menu-->Preferences-->General-->Config Editor...`
  чтобы вызвать редактор реестра thunderbird.

  - Установите ``mailnews.send_plaintext_flowed`` в ``false``

  - Установите ``mailnews.wraplength`` с ``72`` на ``0`` **или** установите
    расширение "Toggle Line Wrap"

    https://github.com/jan-kiszka/togglelinewrap

    https://addons.thunderbird.net/thunderbird/addon/toggle-line-wrap

    чтобы управлять этим параметром реестра на лету.

- Не пишите HTML-сообщения! Перейдите в главное окно
  :menuselection:`Main Menu-->Account Settings-->youracc@server.something-->Composition & Addressing`!
  Там вы можете отключить параметр "Compose messages in HTML format".

- Открывайте сообщения только как обычный текст! Перейдите в главное окно
  :menuselection:`Main Menu-->View-->Message Body As-->Plain Text`!

***********

Gmail (Web GUI)
***************

Не работает для отправки патчей.

Веб-клиент Gmail автоматически преобразует табуляции в пробелы.

В то же время он переносит строки каждые 78 символов с переносами в стиле CRLF,
хотя проблему tab2space можно решить с помощью внешнего редактора.

Другая проблема в том, что Gmail кодирует в base64 любое сообщение, содержащее
не-ASCII символ. Сюда входят, например, европейские имена.

HacKerMaiL (TUI)
****************

HacKerMaiL (hkml) — это простой инструмент управления почтой на основе
public-inbox, который не требует подписки на списки рассылки. Он разрабатывается
и сопровождается сопровождающим DAMON и нацелен на поддержку простых рабочих
процессов разработки для DAMON и подсистем ядра в целом. Подробности см. в файле
README (https://github.com/sjp38/hackermail/blob/master/README.md).
