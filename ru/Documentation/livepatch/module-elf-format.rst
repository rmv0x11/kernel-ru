==================================
ELF-формат модулей livepatch
==================================

Этот документ описывает требования к ELF-формату, которым должны
соответствовать модули livepatch.


.. Table of Contents

.. contents:: :local:


1. Предпосылки и мотивация
==========================

Раньше для записи перемещений (relocations) livepatch требовался
отдельный архитектурно-зависимый код. Однако архитектурно-зависимый код
для записи перемещений уже существует в загрузчике модулей, поэтому
прежний подход порождал избыточный код. Поэтому вместо дублирования кода
и повторной реализации того, что загрузчик модулей уже умеет делать,
livepatch использует существующий код загрузчика модулей для выполнения
всей архитектурно-зависимой работы по перемещениям. В частности,
livepatch повторно использует функцию apply_relocate_add() из загрузчика
модулей для записи перемещений. ELF-формат модуля-патча, описанный в этом
документе, позволяет livepatch делать это. Есть надежда, что это упростит
перенос livepatch на другие архитектуры и сократит объём
архитектурно-зависимого кода, необходимого для переноса livepatch на
конкретную архитектуру.

Поскольку apply_relocate_add() требует доступа к таблице заголовков
секций модуля, таблице символов и индексам секций перемещений,
ELF-информация сохраняется для модулей livepatch (см. раздел 5). Livepatch
управляет собственными секциями перемещений и символами, которые описаны
в этом документе. ELF-константы, используемые для маркировки символов
livepatch и секций перемещений, были выбраны из OS-специфичных диапазонов
согласно определениям из glibc.

Почему livepatch нужно записывать собственные перемещения?
----------------------------------------------------------
Типичный модуль livepatch содержит исправленные версии функций, которые
могут ссылаться на неэкспортируемые глобальные символы и не включённые
локальные символы. Перемещения, ссылающиеся на такие типы символов,
нельзя оставить как есть, поскольку загрузчик модулей ядра не сможет их
разрешить и поэтому отклонит модуль livepatch. Более того, мы не можем
применять перемещения, затрагивающие модули, ещё не загруженные на момент
загрузки модуля-патча (например, патч к драйверу, который не загружен).
Раньше livepatch решал эту проблему путём встраивания специальных секций
"dynrela" (dynamic rela) в итоговый ELF-вывод модуля-патча. Используя эти
секции dynrela, livepatch мог разрешать символы с учётом их области
видимости и того, какому модулю принадлежит символ, а затем вручную
применять динамические перемещения. Однако этот подход требовал, чтобы
livepatch предоставлял архитектурно-зависимый код для записи этих
перемещений. В новом формате livepatch управляет собственными секциями
перемещений SHT_RELA вместо секций dynrela, а символы, на которые
ссылаются relas, являются специальными символами livepatch (см. разделы 2
и 3). Архитектурно-зависимый код перемещений livepatch заменён вызовом
apply_relocate_add().

2. Поле modinfo livepatch
=========================

Модули livepatch обязаны иметь атрибут modinfo "livepatch". Как это
делается, см. в образце модуля livepatch в samples/livepatch/.

Пользователи могут идентифицировать модули livepatch с помощью команды
'modinfo', проверяя наличие поля "livepatch". Это поле также используется
загрузчиком модулей ядра для идентификации модулей livepatch.

Пример:
-------

**Вывод modinfo:**

::

	% modinfo livepatch-meminfo.ko
	filename:		livepatch-meminfo.ko
	livepatch:		Y
	license:		GPL
	depends:
	vermagic:		4.3.0+ SMP mod_unload

3. Секции перемещений livepatch
===============================

Модуль livepatch управляет собственными ELF-секциями перемещений, чтобы
применять перемещения к модулям, а также к ядру (vmlinux) в нужный момент.
Например, если модуль-патч исправляет драйвер, который в данный момент не
загружен, livepatch применит соответствующую секцию (секции) перемещений
livepatch к драйверу, как только тот загрузится.

С каждым "объектом" (например, vmlinux или модулем) внутри модуля-патча
может быть связано несколько секций перемещений livepatch (например, патчи
к нескольким функциям внутри одного объекта). Существует взаимно
однозначное соответствие между секцией перемещений livepatch и целевой
секцией (обычно текстовой секцией функции), к которой применяются
перемещения. Модуль livepatch также может вообще не иметь секций
перемещений livepatch, как в случае образца модуля livepatch (см.
samples/livepatch).

Поскольку ELF-информация сохраняется для модулей livepatch (см. раздел 5),
секцию перемещений livepatch можно применить, просто передав
соответствующий индекс секции в apply_relocate_add(), которая затем
использует его для доступа к секции перемещений и применения перемещений.

Каждый символ, на который ссылается rela в секции перемещений livepatch,
является символом livepatch. Они должны быть разрешены до того, как
livepatch сможет вызвать apply_relocate_add(). Дополнительную информацию
см. в разделе 3.

3.1 Формат секции перемещений livepatch
=======================================

Секции перемещений livepatch должны быть помечены флагом секции
SHF_RELA_LIVEPATCH. Определение см. в include/uapi/linux/elf.h. Загрузчик
модулей распознаёт этот флаг и не будет применять такие секции перемещений
на момент загрузки модуля-патча. Эти секции также должны быть помечены
SHF_ALLOC, чтобы загрузчик модулей не отбрасывал их при загрузке модуля
(то есть они будут скопированы в память вместе с другими секциями
SHF_ALLOC).

Имя секции перемещений livepatch должно соответствовать следующему
формату::

  .klp.rela.objname.section_name
  ^        ^^     ^ ^          ^
  |________||_____| |__________|
     [A]      [B]        [C]

[A]
  Имя секции перемещений начинается с префикса — строки ".klp.rela."

[B]
  Имя объекта (то есть "vmlinux" или имя модуля), которому принадлежит
  секция перемещений, следует сразу после префикса.

[C]
  Фактическое имя секции, к которой применяется данная секция перемещений.

Примеры:
--------

**Имена секций перемещений livepatch:**

::

  .klp.rela.ext4.text.ext4_attr_store
  .klp.rela.vmlinux.text.cmdline_proc_show

**Вывод `readelf --sections` для модуля-патча,
исправляющего vmlinux и модули 9p, btrfs, ext4:**

::

4. Символы livepatch
====================

Символы livepatch — это символы, на которые ссылаются секции перемещений
livepatch. Это символы, к которым обращаются из новых версий функций для
исправляемых объектов и адреса которых не могут быть разрешены загрузчиком
модулей (потому что они являются локальными или неэкспортируемыми
глобальными символами). Поскольку загрузчик модулей разрешает только
экспортируемые символы, а не каждый символ, на который ссылаются новые
исправленные функции, является экспортируемым, были введены символы
livepatch. Они также используются в случаях, когда мы не можем сразу
узнать адрес символа на момент загрузки модуля-патча. Например, это
происходит, когда livepatch исправляет модуль, который ещё не загружен. В
этом случае соответствующие символы livepatch разрешаются просто при
загрузке целевого модуля. В любом случае, для любой секции перемещений
livepatch все символы livepatch, на которые ссылается эта секция, должны
быть разрешены до того, как livepatch сможет вызвать apply_relocate_add()
для этой секции перемещений.

Символы livepatch должны быть помечены SHN_LIVEPATCH, чтобы загрузчик
модулей мог их идентифицировать и игнорировать. Модули livepatch хранят
эти символы в своих таблицах символов, и таблица символов становится
доступной через module->symtab.

4.1 Таблица символов модуля livepatch
=====================================
Обычно урезанная копия таблицы символов модуля (содержащая только
"core"-символы) становится доступной через module->symtab (см.
layout_symtab() в kernel/module/kallsyms.c). Для модулей livepatch таблица
символов, копируемая в память при загрузке модуля, должна быть в точности
такой же, как таблица символов, полученная при компиляции модуля-патча.
Это связано с тем, что перемещения в каждой секции перемещений livepatch
ссылаются на свои соответствующие символы по их индексам символов, и
исходные индексы символов (а значит, и порядок в symtab) должны быть
сохранены, чтобы apply_relocate_add() мог найти нужный символ.

Например, возьмём вот такой конкретный rela из модуля livepatch::

  Relocation section '.klp.rela.btrfs.text.btrfs_feature_attr_show' at offset 0x2ba0 contains 4 entries:
      Offset             Info             Type               Symbol's Value  Symbol's Name + Addend
  000000000000001f  0000005e00000002 R_X86_64_PC32          0000000000000000 .klp.sym.vmlinux.printk,0 - 4

Этот rela ссылается на символ '.klp.sym.vmlinux.printk,0', и индекс символа
закодирован в 'Info'. Здесь его индекс символа равен 0x5e, что составляет
94 в десятичной системе, то есть он ссылается на индекс символа 94.

И в соответствующей таблице символов этого модуля-патча индекс символа 94
ссылается именно на этот символ::

  [ snip ]
  94: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT OS [0xff20] .klp.sym.vmlinux.printk,0
  [ snip ]

4.2 Формат символа livepatch
============================

Символы livepatch должны иметь индекс секции, помеченный как
SHN_LIVEPATCH, чтобы загрузчик модулей мог их идентифицировать и не
пытался их разрешать. Фактические определения см. в
include/uapi/linux/elf.h.

Имена символов livepatch должны соответствовать следующему формату::

  .klp.sym.objname.symbol_name,sympos
  ^       ^^     ^ ^         ^ ^
  |_______||_____| |_________| |
     [A]     [B]       [C]    [D]

[A]
  Имя символа начинается с префикса — строки ".klp.sym."

[B]
  Имя объекта (то есть "vmlinux" или имя модуля), которому принадлежит
  символ, следует сразу после префикса.

[C]
  Фактическое имя символа.

[D]
  Позиция символа в объекте (согласно kallsyms). Используется для
  различения дублирующихся символов внутри одного объекта. Позиция символа
  выражается численно (0, 1, 2...). Позиция уникального символа равна 0.

Примеры:
--------

**Имена символов livepatch:**

::

	.klp.sym.vmlinux.snprintf,0
	.klp.sym.vmlinux.printk,0
	.klp.sym.btrfs.btrfs_ktype,0

**Вывод `readelf --symbols` для модуля-патча:**

::

  Symbol table '.symtab' contains 127 entries:
     Num:    Value          Size Type    Bind   Vis     Ndx         Name
     [ snip ]
      73: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT OS [0xff20] .klp.sym.vmlinux.snprintf,0
      74: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT OS [0xff20] .klp.sym.vmlinux.capable,0
      75: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT OS [0xff20] .klp.sym.vmlinux.find_next_bit,0
      76: 0000000000000000     0 NOTYPE  GLOBAL DEFAULT OS [0xff20] .klp.sym.vmlinux.si_swapinfo,0
    [ snip ]                                               ^
                                                           |
                                                          [*]

[*]
  Обратите внимание, что 'Ndx' (индекс секции) для этих символов равен
  SHN_LIVEPATCH (0xff20). "OS" означает OS-специфичный.

5. Доступ к таблице символов и ELF-секциям
==========================================
Таблица символов модуля livepatch доступна через module->symtab.

Поскольку apply_relocate_add() требует доступа к заголовкам секций модуля,
таблице символов и индексам секций перемещений, ELF-информация сохраняется
для модулей livepatch и становится доступной загрузчиком модулей через
module->klp_info, который является структурой :c:type:`klp_modinfo`. Когда
модуль livepatch загружается, эта структура заполняется загрузчиком модулей.
