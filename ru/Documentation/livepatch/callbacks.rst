============================
Колбэки (un)patch
============================

Колбэки (un)patch механизма livepatch предоставляют модулям livepatch
способ выполнять функции обратного вызова при патче или снятии патча
(patch/unpatch) с объекта ядра. Их можно рассматривать как **мощную
возможность**, которая **расширяет способности livepatch**, добавляя:

  - Безопасное обновление глобальных данных

  - «Патчи» для функций init и probe

  - Патч кода, который иначе нельзя пропатчить (то есть ассемблера)

В большинстве случаев колбэки (un)patch потребуется использовать совместно
с барьерами памяти и примитивами синхронизации ядра, такими как
мьютексы/спинлоки или даже stop_machine(), чтобы избежать проблем
параллелизма.

1. Мотивация
============

Колбэки отличаются от существующих средств ядра:

  - Код init/exit модуля не выполняется при отключении и повторном
    включении патча.

  - Уведомитель модуля (module notifier) не может остановить загрузку
    модуля, который должен быть пропатчен.

Колбэки являются частью структуры klp_object, и их реализация специфична
для этого klp_object. Другие объекты livepatch могут быть пропатчены или
нет, независимо от текущего состояния целевого klp_object.

2. Типы колбэков
================

Колбэки можно зарегистрировать для следующих действий livepatch:

  * Pre-patch
                 - перед тем, как klp_object пропатчен

  * Post-patch
                 - после того, как klp_object пропатчен и активен
                   во всех задачах

  * Pre-unpatch
                 - перед тем, как с klp_object снимается патч (то есть
                   пропатченный код активен); используется для очистки
                   ресурсов колбэка post-patch

  * Post-unpatch
                 - после того, как klp_object был пропатчен, весь код
                   восстановлен и ни одна задача не выполняет пропатченный
                   код; используется для очистки ресурсов колбэка pre-patch

3. Как это работает
===================

Каждый колбэк необязателен; пропуск одного из них не препятствует указанию
любого другого. Однако ядро livepatch выполняет обработчики симметрично:
у колбэков pre-patch есть соответствующий им post-unpatch, а у колбэков
post-patch — соответствующий pre-unpatch. Колбэк unpatch будет выполнен
только в том случае, если был выполнен соответствующий ему колбэк patch.
В типичных сценариях обработчик patch, который захватывает и настраивает
ресурсы, образует пару с обработчиком unpatch, который разбирает и
освобождает эти же ресурсы.

Колбэк выполняется только в том случае, если загружен его host-объект
klp_object. Для встроенных в ядро целей vmlinux это означает, что колбэки
будут выполняться всегда при включении/отключении livepatch. Для целевых
модулей ядра колбэки будут выполняться только в том случае, если загружен
целевой модуль. Когда целевой модуль загружается или выгружается, его
колбэки будут выполнены только в том случае, если модуль livepatch включён.

Колбэк pre-patch, если он указан, должен возвращать код состояния (0 при
успехе, -ERRNO при ошибке). Код состояния с ошибкой указывает ядру
livepatch на то, что патч текущего klp_object небезопасен и текущий запрос
на патч следует остановить. (Когда колбэк pre-patch не предоставлен,
переход считается безопасным.) Если колбэк pre-patch возвращает ошибку,
загрузчик модулей ядра:

  - Откажется загрузить livepatch, если livepatch загружается после
    целевого кода.

    или:

  - Откажется загрузить модуль, если livepatch уже был успешно загружен.

Никакие колбэки post-patch, pre-unpatch или post-unpatch не будут выполнены
для данного klp_object, если объект не удалось пропатчить из-за неудачного
колбэка pre_patch или по любой другой причине.

Если переход патча отменён, ни один обработчик pre-unpatch не будет запущен
(это следует из ранее упомянутой симметрии — колбэки pre-unpatch
происходят только в том случае, если был выполнен соответствующий им колбэк
post-patch).

Если объект был успешно пропатчен, но переход патча по какой-то причине так
и не начался (например, если другой объект не удалось пропатчить), будет
вызван только колбэк post-unpatch.

4. Сценарии использования
=========================

Образцы модулей livepatch, демонстрирующие API колбэков, можно найти в
каталоге samples/livepatch/. Эти образцы были изменены для использования
в kselftests и находятся в каталоге lib/livepatch.

Обновление глобальных данных
----------------------------

Колбэк pre-patch может быть полезен для обновления глобальной переменной.
Например, коммит 75ff39ccc1bd ("tcp: make challenge acks less predictable")
изменяет глобальный sysctl, а также патчит функцию tcp_send_challenge_ack().

В этом случае, если быть особо осторожным, может иметь смысл патчить данные
*после* того, как патч завершён, с помощью колбэка post-patch, чтобы
tcp_send_challenge_ack() сначала можно было изменить так, чтобы она читала
sysctl_tcp_challenge_ack_limit с помощью READ_ONCE.

Поддержка патчей функций __init и probe
---------------------------------------

Хотя функции __init и probe нельзя пропатчить напрямую через livepatch,
аналогичные обновления, возможно, удастся реализовать с помощью колбэков
pre/post-patch.

Коммит 48900cb6af42 ("virtio-net: drop NETIF_F_FRAGLIST") изменяет способ,
которым virtnet_probe() инициализировала возможности net_device своего
драйвера. Колбэк pre/post-patch мог бы пройтись по всем таким устройствам,
внося аналогичное изменение в их значение hw_features. (Функции-клиенты
этого значения может потребоваться обновить соответствующим образом.)
