==================
Теневые переменные
==================

Теневые переменные (shadow variables) — это простой способ для модулей livepatch
ассоциировать дополнительные «теневые» данные с существующими структурами данных.
Теневые данные размещаются в памяти отдельно от родительских структур данных,
которые остаются неизменными.  Описанный в этом документе API теневых переменных
используется для выделения/добавления и удаления/освобождения теневых переменных
относительно их родителей.

Реализация вводит глобальную внутриядерную хеш-таблицу, которая ассоциирует
указатели на родительские объекты с числовым идентификатором теневых данных.
Числовой идентификатор представляет собой простую нумерацию, которая может
использоваться для описания версии, класса или типа теневой переменной и т. п.
Более конкретно, указатель на родителя служит ключом хеш-таблицы, тогда как
числовой идентификатор затем фильтрует запросы к хеш-таблице.  С одним и тем же
родительским объектом может быть связано несколько теневых переменных, но их
числовой идентификатор позволяет различать их.


1. Краткая сводка по API
========================

(Полные docbook-заметки по использованию API см. в livepatch/shadow.c.)

Хеш-таблица ссылается на все теневые переменные.  Эти ссылки хранятся и
извлекаются по паре <obj, id>.

* Структура данных теневой переменной klp_shadow инкапсулирует как
  метаданные для отслеживания, так и теневые данные:

  - метаданные

    - obj - указатель на родительский объект
    - id - идентификатор данных

  - data[] - хранилище для теневых данных

Важно отметить, что klp_shadow_alloc() и klp_shadow_get_or_alloc() по умолчанию
обнуляют переменную.  Они также позволяют вызвать пользовательскую функцию-
конструктор, когда требуется ненулевое значение. Вызывающая сторона должна
обеспечить необходимое взаимное исключение.

Обратите внимание, что конструктор вызывается под спинлоком klp_shadow_lock. Это
позволяет выполнить действия, которые должны быть выполнены только один раз при
выделении новой переменной.

* klp_shadow_get() - получить указатель на данные теневой переменной
  - поиск в хеш-таблице пары <obj, id>

* klp_shadow_alloc() - выделить и добавить новую теневую переменную
  - поиск в хеш-таблице пары <obj, id>

  - если существует

    - выдать WARN и вернуть NULL

  - если <obj, id> ещё не существует

    - выделить новую теневую переменную
    - инициализировать переменную с помощью пользовательского конструктора и данных, если они предоставлены
    - добавить <obj, id> в глобальную хеш-таблицу

* klp_shadow_get_or_alloc() - получить существующую или выделить новую теневую переменную
  - поиск в хеш-таблице пары <obj, id>

  - если существует

    - вернуть существующую теневую переменную

  - если <obj, id> ещё не существует

    - выделить новую теневую переменную
    - инициализировать переменную с помощью пользовательского конструктора и данных, если они предоставлены
    - добавить пару <obj, id> в глобальную хеш-таблицу

* klp_shadow_free() - отсоединить и освободить теневую переменную <obj, id>
  - найти и удалить ссылку <obj, id> из глобальной хеш-таблицы

    - если найдена

      - вызвать функцию-деструктор, если она определена
      - освободить теневую переменную

* klp_shadow_free_all() - отсоединить и освободить все теневые переменные <_, id>
  - найти и удалить любые ссылки <_, id> из глобальной хеш-таблицы

    - если найдены

      - вызвать функцию-деструктор, если она определена
      - освободить теневую переменную


2. Сценарии использования
=========================

(Полностью рабочие демонстрации см. в примерах модулей livepatch с теневыми
переменными в samples/livepatch/.)

Для следующих примеров сценариев использования рассмотрим коммит 1d147bfa6429
("mac80211: fix AP powersave TX vs.  wakeup race"), который добавил спинлок в
net/mac80211/sta_info.h :: struct sta_info.  Каждый пример сценария можно
рассматривать как самостоятельную реализацию данного исправления средствами
livepatch.


Согласование с жизненным циклом родителя
----------------------------------------

Если родительские структуры данных часто создаются и уничтожаются, проще всего
может оказаться согласовать время жизни их теневых переменных с теми же
функциями выделения и освобождения.  В этом случае родительская структура
данных обычно сначала выделяется, инициализируется, а затем тем или иным образом
регистрируется.  Выделение и настройку теневой переменной можно тогда
рассматривать как часть инициализации родителя, и они должны быть завершены
прежде, чем родитель «войдёт в строй» (т. е. до того, как для этой пары
<obj, id> будут сделаны какие-либо запросы к get-API теневых переменных).

Для коммита 1d147bfa6429, когда выделяется родительская структура sta_info,
выделите теневую копию указателя ps_lock, а затем инициализируйте её::

  #define PS_LOCK 1
  struct sta_info *sta_info_alloc(struct ieee80211_sub_if_data *sdata,
				  const u8 *addr, gfp_t gfp)
  {
	struct sta_info *sta;
	spinlock_t *ps_lock;

	/* Parent structure is created */
	sta = kzalloc(sizeof(*sta) + hw->sta_data_size, gfp);

	/* Attach a corresponding shadow variable, then initialize it */
	ps_lock = klp_shadow_alloc(sta, PS_LOCK, sizeof(*ps_lock), gfp,
				   NULL, NULL);
	if (!ps_lock)
		goto shadow_fail;
	spin_lock_init(ps_lock);
	...

Когда требуется ps_lock, обратитесь к API теневых переменных, чтобы получить его
для конкретной struct sta_info:::

  void ieee80211_sta_ps_deliver_wakeup(struct sta_info *sta)
  {
	spinlock_t *ps_lock;

	/* sync with ieee80211_tx_h_unicast_ps_buf */
	ps_lock = klp_shadow_get(sta, PS_LOCK);
	if (ps_lock)
		spin_lock(ps_lock);
	...

Когда родительская структура sta_info освобождается, сначала освободите теневую
переменную::

  void sta_info_free(struct ieee80211_local *local, struct sta_info *sta)
  {
	klp_shadow_free(sta, PS_LOCK, NULL);
	kfree(sta);
	...


Родительские объекты «в полёте»
-------------------------------

Иногда выделять теневые переменные одновременно с их родительскими объектами
бывает неудобно или невозможно.  Либо исправление livepatch может требовать
теневых переменных лишь для части экземпляров родительских объектов.  В этих
случаях вызов klp_shadow_get_or_alloc() можно использовать для присоединения
теневых переменных к уже находящимся «в полёте» родителям.

Для коммита 1d147bfa6429 удачным местом для выделения теневого спинлока является
ieee80211_sta_ps_deliver_wakeup()::

  int ps_lock_shadow_ctor(void *obj, void *shadow_data, void *ctor_data)
  {
	spinlock_t *lock = shadow_data;

	spin_lock_init(lock);
	return 0;
  }

  #define PS_LOCK 1
  void ieee80211_sta_ps_deliver_wakeup(struct sta_info *sta)
  {
	spinlock_t *ps_lock;

	/* sync with ieee80211_tx_h_unicast_ps_buf */
	ps_lock = klp_shadow_get_or_alloc(sta, PS_LOCK,
			sizeof(*ps_lock), GFP_ATOMIC,
			ps_lock_shadow_ctor, NULL);

	if (ps_lock)
		spin_lock(ps_lock);
	...

При таком использовании теневая переменная будет создана только при
необходимости, иначе будет использована та, что уже была создана для этой пары
<obj, id>.

Как и в предыдущем сценарии использования, теневой спинлок необходимо очищать.
Теневую переменную можно освободить непосредственно перед освобождением её
родительского объекта или даже тогда, когда сама теневая переменная больше не
требуется.


Прочие сценарии использования
-----------------------------

Теневые переменные также могут использоваться в качестве флага, указывающего, что
структура данных была выделена новым, исправленным кодом.  В этом случае не имеет
значения, какое значение данных хранит теневая переменная — само её существование
подсказывает, как обрабатывать родительский объект.
