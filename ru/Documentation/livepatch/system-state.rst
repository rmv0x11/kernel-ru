=============================
Изменения состояния системы
=============================

Некоторые пользователи крайне неохотно перезагружают систему. Это порождает
потребность предоставлять больше livepatch'ей и поддерживать определённую
совместимость между ними.

Сопровождать большее количество livepatch'ей гораздо проще с помощью
кумулятивных livepatch'ей. Каждый новый livepatch полностью заменяет любой
более старый. Он может сохранять, добавлять и даже удалять исправления. И, как
правило, любую версию livepatch'а можно безопасно заменить любой другой
благодаря возможности атомарной замены (atomic replace).

Проблемы могут возникнуть из-за теневых переменных (shadow variables) и
коллбэков. Они могут изменить поведение или состояние системы так, что станет
небезопасно возвращаться к более старому livepatch'у или к исходному коду
ядра. Кроме того, любой новый livepatch должен иметь возможность определить,
какие изменения уже были сделаны уже установленными livepatch'ами.

Именно здесь оказывается полезным отслеживание состояния системы средствами
livepatch. Оно позволяет:

  - хранить данные, необходимые для манипулирования состоянием системы и
    его восстановления

  - определять совместимость между livepatch'ами с помощью идентификатора
    изменения (change id) и версии


1. API отслеживания состояния системы livepatch
================================================

Состояние системы может быть изменено либо несколькими коллбэками livepatch'а,
либо вновь используемым кодом. Кроме того, должна быть возможность найти
изменения, сделанные уже установленными livepatch'ами.

Каждое изменённое состояние описывается структурой struct klp_state, см.
include/linux/livepatch.h.

Каждый livepatch определяет массив struct klp_states. В нём перечислены все
состояния, которые этот livepatch изменяет.

Автор livepatch'а должен определить следующие два поля для каждой
struct klp_state:

  - *id*

    - Ненулевое число, используемое для идентификации затрагиваемого
      состояния системы.

  - *version*

    - Число, описывающее вариант изменения состояния системы, который
      поддерживается данным livepatch'ем.

Состоянием можно манипулировать с помощью двух функций:

  - klp_get_state()

    - Получить struct klp_state, связанную с данным livepatch'ем и
      идентификатором состояния.

  - klp_get_prev_state()

    - Получить struct klp_state, связанную с данным идентификатором
      функциональности и уже установленными livepatch'ами.

2. Совместимость livepatch'ей
=============================

Версия состояния системы используется для предотвращения загрузки несовместимых
livepatch'ей. Проверка выполняется при включении livepatch'а. Правила таковы:

  - Любое полностью новое изменение состояния системы разрешено.

  - Изменения состояния системы с такой же или более высокой версией разрешены
    для уже изменённых состояний системы.

  - Кумулятивные livepatch'и должны обрабатывать все изменения состояния системы
    от уже установленных livepatch'ей.

  - Некумулятивным livepatch'ам разрешено затрагивать уже изменённые
    состояния системы.

3. Поддерживаемые сценарии
==========================

У livepatch'ей есть свой жизненный цикл, и то же самое справедливо для изменений
состояния системы. Каждый совместимый livepatch должен поддерживать следующие
сценарии:

  - Изменить состояние системы, когда livepatch включается, а состояние ещё
    не было изменено livepatch'ами, которые заменяются.

  - Перенять или обновить изменение состояния системы, когда оно уже было
    сделано livepatch'ем, который заменяется.

  - Восстановить исходное состояние при отключении livepatch'а.

  - Восстановить предыдущее состояние при откате перехода (transition).
    Это может быть исходное состояние системы или изменение состояния,
    сделанное livepatch'ами, которые заменялись.

  - Удалить любые уже сделанные изменения при возникновении ошибки, когда
    livepatch не может быть включён.

4. Ожидаемое использование
==========================

Состояния системы обычно изменяются коллбэками livepatch'а. Ожидаемая роль
каждого коллбэка такова:

*pre_patch()*

  - Выделить *state->data* при необходимости. Выделение может завершиться
    неудачей, и *pre_patch()* — единственный коллбэк, который может остановить
    загрузку livepatch'а. Выделение не требуется, когда данные уже
    предоставлены ранее установленными livepatch'ами.

  - Выполнить любое другое подготовительное действие, необходимое новому коду
    ещё до завершения перехода (transition).
    Например, инициализировать *state->data*.

    Само состояние системы обычно изменяется в *post_patch()*,
    когда вся система способна это обработать.

  - Убрать за собой в случае ошибки. Это может быть сделано пользовательским
    кодом или явным вызовом *post_unpatch()*.

*post_patch()*

  - Скопировать *state->data* из предыдущего livepatch'а, если они
    совместимы.

  - Выполнить фактическое изменение состояния системы. В конечном итоге
    разрешить новому коду использовать его.

  - Убедиться, что *state->data* содержит всю необходимую информацию.

  - Освободить *state->data* от заменяемых livepatch'ей, когда они
    больше не нужны.

*pre_unpatch()*

  - Не допустить, чтобы код, добавленный livepatch'ем, полагался на изменение
    состояния системы.

  - Откатить изменение состояния системы..

*post_unpatch()*

  - Различить откат перехода и отключение livepatch'а, проверив
    *klp_get_prev_state()*.

  - В случае отката перехода восстановить предыдущее состояние системы.
    Это может означать ничего не делать.

  - Удалить любые более не нужные настройки или данные.

.. note::

   *pre_unpatch()* обычно выполняет операции, симметричные *post_patch()*.
   За исключением того, что он вызывается только тогда, когда livepatch
   отключается. Поэтому ему не нужно заботиться ни о каком ранее установленном
   livepatch'е.

   *post_unpatch()* обычно выполняет операции, симметричные *pre_patch()*.
   Он также может вызываться во время отката перехода. Поэтому он должен
   обрабатывать состояние ранее установленных livepatch'ей.
